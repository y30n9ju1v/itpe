## 개요

애자일(Agile) 소프트웨어 개발은 짧은 반복 주기(스프린트)를 통해 동작하는 소프트웨어를 지속적으로 제공하고, 변화하는 요구사항에 빠르게 대응하는 개발 방법론이다. 2001년 애자일 선언(Agile Manifesto) 이후 스크럼(Scrum), 칸반(Kanban), XP(eXtreme Programming) 등 다양한 프레임워크가 발전했다.

## 핵심 내용

### 애자일 선언 4대 가치

1. 프로세스·도구보다 **개인과 상호작용**
2. 포괄적인 문서보다 **동작하는 소프트웨어**
3. 계약 협상보다 **고객과의 협력**
4. 계획을 따르기보다 **변화에 대응**

### 애자일 장점

| 장점 | 설명 |
|------|------|
| **빠른 피드백** | 스프린트마다 동작 제품 제공, 즉각적 고객 검증 |
| **변경 수용** | 요구사항 변화에 유연하게 대응 |
| **리스크 조기 발견** | 반복 개발로 초기에 문제 탐지·수정 |
| **팀 자율성** | 자기 조직화 팀, 의사결정 속도 향상 |
| **고객 만족** | 지속적 가치 제공, 고객 참여 증가 |
| **투명성** | 데일리 스크럼·번다운 차트로 진행 상황 가시화 |

### 애자일 단점

| 단점 | 설명 |
|------|------|
| **문서 부족** | 과도한 구현 중심으로 아키텍처 문서 취약 |
| **범위 관리 어려움** | 스코프 크리프(Scope Creep) 위험 |
| **대규모 팀 한계** | 소규모(5~9인) 최적화, 대형 프로젝트 조율 어려움 |
| **고객 참여 필수** | 고객 참여 부족 시 방향성 상실 |
| **장기 계획 불확실** | 전체 일정·비용 예측 어려움 |
| **팀 역량 의존** | 팀원의 높은 역량·경험 필요 |

### 애자일 vs 폭포수 비교

| 구분 | 폭포수 | 애자일 |
|------|--------|--------|
| **개발 방식** | 순차적 (단계별) | 반복·점진적 |
| **요구사항** | 초기에 완전히 정의 | 변경 허용 |
| **고객 참여** | 초기·최종 | 매 스프린트 |
| **결과물** | 최종 완성품 | 스프린트마다 동작 SW |
| **적합** | 요구사항 명확, 대형 시스템 | 빠른 변화, 혁신적 제품 |
