## 개요

정보은닉(Information Hiding)은 모듈 내부의 구현 세부사항을 외부에 노출하지 않고 인터페이스를 통해서만 접근하도록 하는 설계 원칙이다. 객체지향의 캡슐화(Encapsulation)와 밀접하게 연관되며, 모듈 간 결합도를 낮추고 변경 영향 범위를 최소화한다.

## 핵심 내용

### 정보은닉의 개념

- **제안자**: David Parnas (1972년 논문 "On the Criteria To Be Used in Decomposing Systems Into Modules")
- **핵심 아이디어**: 변경 가능성이 높은 설계 결정을 모듈 내부에 숨겨 외부 영향 차단
- **목적**: 결합도 감소, 재사용성·유지보수성 향상

### 객체지향에서의 구현

**접근 제어자 (Access Modifier)**

| 제어자 | 범위 | 설명 |
|--------|------|------|
| `private` | 클래스 내부 | 외부 완전 차단 |
| `protected` | 클래스 + 하위 클래스 | 상속 허용 |
| `public` | 모든 클래스 | 공개 인터페이스 |
| `package` | 동일 패키지 | Java 기본값 |

**캡슐화와의 관계**
- **캡슐화**: 데이터(속성)와 메서드(행위)를 하나의 단위로 묶음 — 무엇을 묶을지
- **정보은닉**: 묶인 것의 내부를 외부에서 볼 수 없게 함 — 어떻게 감출지
- 정보은닉은 캡슐화의 목적을 달성하기 위한 수단

### 정보은닉의 장점

| 장점 | 설명 |
|------|------|
| **변경 용이성** | 내부 구현 변경이 외부에 영향 없음 |
| **결합도 감소** | 모듈 간 의존성 최소화 |
| **재사용성** | 인터페이스만 알면 재사용 가능 |
| **보안성** | 내부 데이터 직접 접근 차단 |
| **테스트 용이성** | 인터페이스 기반 모킹 가능 |

### 위반 사례

- public 필드 직접 노출 (`public int balance;`)
- getter/setter 남용 — 로직 없는 단순 접근자는 정보은닉 효과 반감
- 구현 클래스를 반환 타입으로 노출 (인터페이스 타입 사용 권장)

## 실무 적용 예시

- Java Bean 패턴: private 필드 + public getter/setter
- Repository 패턴: DB 접근 로직을 인터페이스 뒤에 숨겨 구현 변경(MySQL→NoSQL) 시 호출자 코드 무변경
- API 설계: 내부 도메인 모델 대신 DTO로 외부 계약 분리
