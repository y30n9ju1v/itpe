## 개요

UML(Unified Modeling Language) 행위 다이어그램(Behavior Diagram)은 시스템의 동적 동작을 표현한다. 활동 다이어그램(Activity), 상태 다이어그램(State), 유스케이스 다이어그램(Use-Case)은 각각 프로세스 흐름, 상태 변화, 기능 요구사항을 모델링한다.

## 핵심 내용

### UML 다이어그램 분류

```
UML 다이어그램
├── 구조 다이어그램: 클래스, 객체, 컴포넌트, 배포
└── 행위 다이어그램: 유스케이스, 활동, 상태, 시퀀스, 협력
```

### 활동 다이어그램 (Activity Diagram)

**개념**: 업무 프로세스나 알고리즘의 흐름을 순서도처럼 표현. 병렬 처리·분기·합류를 표현 가능.

**주요 표기 요소**:
- 초기 노드: ● (시작)
- 활동(Action): 둥근 사각형
- 결정 노드: ◇ (조건 분기)
- 포크/조인: ━━━ (병렬 분기/합류)
- 종료 노드: ⊙ (종료)
- 스윔레인(Swimlane): 역할별 활동 구분

**활용**: 업무 프로세스 모델링, 유스케이스 상세화, 알고리즘 설계

### 상태 다이어그램 (State Diagram / State Machine Diagram)

**개념**: 객체가 생명주기 동안 거치는 상태와 상태 전이를 모델링. 이벤트로 인한 상태 변화 표현.

**주요 표기 요소**:
- 상태: 둥근 사각형
- 전이: 화살표 (이벤트[조건]/행동)
- 초기 상태: ● (검정 원)
- 최종 상태: ⊙ (이중 원)
- 히스토리 상태: H (이전 상태 기억)
- 복합 상태: 상태 내 상태

**예시** (주문 상태):
```
[초기] → 주문접수 → 결제완료 → 배송중 → 배송완료 → [종료]
                  ↘ 취소 ↗
```

**활용**: 실시간 시스템, 프로토콜 설계, UI 상태 관리

### 유스케이스 다이어그램 (Use-Case Diagram)

**개념**: 시스템과 외부 행위자(Actor) 간의 상호작용을 기능 단위로 표현. 요구사항 도출·정의에 활용.

**주요 표기 요소**:
- 액터(Actor): 사람 형태 (외부 사용자·시스템)
- 유스케이스: 타원 (시스템 기능)
- 시스템 경계: 사각형 박스
- 관계:
  - 연관(Association): ─ (액터-유스케이스)
  - 포함(Include): ─ ─ ─›«include» (반드시 포함)
  - 확장(Extend): ─ ─ ─›«extend» (조건부 포함)
  - 일반화(Generalization): ─── ▷ (상속)

**include vs extend**:
- include: A 유스케이스 실행 시 B 항상 실행 ("결제하기" includes "카드 검증")
- extend: 특정 조건에서만 B 실행 ("할인 적용"은 "결제하기"를 extend)

### 3가지 다이어그램 비교

| 구분 | 활동 다이어그램 | 상태 다이어그램 | 유스케이스 다이어그램 |
|------|--------------|--------------|-----------------|
| 표현 대상 | 프로세스 흐름 | 객체 상태 변화 | 기능 요구사항 |
| 주요 요소 | 활동, 분기, 병렬 | 상태, 전이, 이벤트 | 액터, 유스케이스 |
| 시점 | 동적 흐름 | 동적 상태 | 정적 기능 목록 |
| 활용 단계 | 설계 | 설계 | 분석·요구사항 |

## 출제 포인트

- 3가지 다이어그램을 행위 다이어그램으로 분류
- 각 다이어그램의 정의·주요 표기 요소·활용 단계 서술
- include vs extend 차이 설명
- 상태 다이어그램 예시(주문 상태 등) 제시
