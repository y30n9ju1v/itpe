## 개요

형상관리(Configuration Management)는 소프트웨어 개발 전 생명주기에 걸쳐 산출물의 변경을 체계적으로 관리하여 무결성과 추적성을 보장하는 프로세스다. 기준선(Baseline)은 정식으로 검토·승인된 형상항목의 스냅샷으로, 이후 모든 변경의 기준점이 된다.

## 핵심 내용

### 형상관리 개념

- **형상항목(CI: Configuration Item)**: 관리 대상 단위 — 소스코드, 설계문서, 테스트케이스, 빌드 스크립트 등
- **형상관리 목적**: 변경 추적, 버전 관리, 감사 가능성(Auditability), 재현 가능한 빌드 보장

### 형상관리 4대 활동

| 활동 | 내용 |
|------|------|
| **형상 식별** | 형상항목 목록 및 ID 정의, 계층 구조화 |
| **형상 통제** | 변경요청(CR) → 검토 → 승인/거부 → 구현 절차 |
| **형상 상태 기록** | 형상항목 이력·상태 문서화 (형상 관리 DB) |
| **형상 감사** | 기준선과 실제 형상 일치 여부 검증 |

### 기준선 (Baseline)

**정의**: 공식적으로 검토·승인된 특정 시점의 형상항목 집합. 이후 변경은 반드시 형상 통제 절차를 거쳐야 한다.

**기준선 유형 (개발 단계별)**

| 기준선 | 생성 시점 | 주요 산출물 |
|--------|-----------|-------------|
| **기능 기준선** | 요구사항 분석 완료 | 시스템 요구사항 명세서 (SRS) |
| **할당 기준선** | 설계 완료 | 소프트웨어 설계 명세서 |
| **제품 기준선** | 개발·시험 완료 | 소스코드, 실행파일, 테스트 결과 |

### 형상 통제 절차

```
변경요청(CR) → 영향도 분석 → CCB 검토/승인 → 구현 → 검증 → 기준선 갱신
```

- **CCB (Configuration Control Board)**: 변경 승인 의사결정 기구
- **형상 관리 도구**: Git, SVN, Perforce, Bitbucket

## 실무 적용 예시

- 전자정부 사업에서 요구사항 기준선 설정 후 요구사항 변경 시 변경관리위원회(CCB) 승인 후 반영
- DevOps 파이프라인에서 Git 태그로 릴리스 기준선 관리, CI/CD로 자동 빌드·배포

## 출제 포인트

- 형상관리 4대 활동 순서와 내용
- 기준선 3가지(기능·할당·제품)와 생성 시점
- CCB 역할과 변경 통제 절차
- Git 등 도구와의 연계
