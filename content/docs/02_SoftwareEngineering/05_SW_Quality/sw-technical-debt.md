## 개요

소프트웨어 기술 부채(Technical Debt)는 단기적 편의를 위해 최선이 아닌 설계·구현 방식을 선택함으로써 미래에 추가적인 작업(이자)이 발생하는 상태를 의미한다. Ward Cunningham이 제시한 개념으로, 금융 부채 은유를 사용한다.

## 핵심 내용

### 기술 부채의 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **코드 부채** | 복잡하고 이해하기 어려운 코드 | 중복 코드, 긴 함수, 불명확한 변수명 |
| **설계 부채** | 잘못된 아키텍처·구조 | 강결합, 순환 의존성, 레이어 위반 |
| **테스트 부채** | 불충분한 테스트 커버리지 | 단위 테스트 없음, 레거시 코드 미검증 |
| **문서 부채** | 부족하거나 구식인 문서 | 인라인 문서 없음, API 문서 미갱신 |
| **인프라 부채** | 구식 기술 스택·의존성 | 구버전 라이브러리, 지원 종료 OS |
| **보안 부채** | 취약한 코딩 패턴 누적 | 하드코딩 자격증명, 미패치 취약점 |

### 기술 부채 발생 원인 (의도적 vs 비의도적)

| 구분 | 원인 |
|------|------|
| 의도적 | 출시 일정 압박, 의식적인 단기 타협 |
| 비의도적 | 기술 역량 부족, 레거시 코드 축적, 기술 진화 |

### 관리 방법

1. **측정**: SonarQube 등 정적 분석 도구로 부채 정량화
2. **우선순위**: 비즈니스 영향도·수정 비용 기준 분류
3. **리팩토링**: 점진적 개선 (Boy Scout Rule: 항상 발견 전보다 깨끗하게)
4. **예방**: 코드 리뷰, 페어 프로그래밍, 아키텍처 원칙 준수
5. **추적**: 기술 부채 레지스트리(백로그) 관리

### 기술 부채 사분면 (Martin Fowler)

|  | 의도적 | 비의도적 |
|--|-------|---------|
| **신중한** | 출시 후 정리 계획 수립 | 더 나은 설계 발견 |
| **무모한** | 설계할 시간 없음 | 레이어가 뭔지 모름 |

## 출제 포인트

- 6가지 기술 부채 유형 암기
- Fowler의 기술 부채 사분면: 의도적/비의도적 × 신중한/무모한
- 관리 방법: 측정 → 우선순위 → 리팩토링
