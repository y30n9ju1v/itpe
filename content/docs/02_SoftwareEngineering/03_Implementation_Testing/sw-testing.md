## 개요

소프트웨어 테스트는 결함을 발견하고 제품의 품질 수준을 평가하기 위한 체계적 활동이다. ISTQB(국제소프트웨어테스팅자격위원회)의 7대 원리와 블랙박스·화이트박스 기법, 명세·구조·경험 기반 기법으로 분류된다.

## 핵심 내용

### 가. 소프트웨어 테스트 원리 (ISTQB 7대 원리)

| 원리 | 설명 |
|------|------|
| **1. 테스트는 결함의 존재를 보여줌** | 테스트로 결함을 발견할 수 있지만, 결함이 없음을 증명할 수는 없음 |
| **2. 완벽한 테스트는 불가능** | 무한한 입력 조합으로 100% 테스트는 현실적으로 불가 |
| **3. 조기 테스트** | 초기 단계 테스트일수록 결함 수정 비용 절감 (1:10:100 법칙) |
| **4. 결함 집중** | 80%의 결함이 20%의 모듈에 집중 (파레토 법칙) |
| **5. 살충제 패러독스** | 동일한 테스트 반복 시 새로운 결함 발견 불가 → 테스트 케이스 지속 갱신 |
| **6. 테스트는 컨텍스트에 의존** | 환경·목적에 따라 다른 테스트 전략 적용 |
| **7. 오류 부재의 궤변** | 모든 결함을 제거해도 요구사항 미충족 시 쓸모없음 |

### 나. 블랙박스 테스트 vs 화이트박스 테스트

| 구분 | 블랙박스 (명세 기반) | 화이트박스 (구조 기반) |
|------|---------------------|----------------------|
| **관점** | 외부 기능 (I/O) | 내부 구조 (코드) |
| **지식 필요** | 요구사항·명세만 알면 됨 | 소스코드 지식 필요 |
| **수행자** | 테스터, 사용자 | 개발자 |
| **커버리지** | 기능 커버리지 | 코드 커버리지 (구문·분기·경로) |
| **기법** | 동등 분할, 경계값, 결정 테이블 | 제어 흐름, 데이터 흐름, 구문 커버리지 |
| **발견 결함** | 명세 불일치, 기능 오작동 | 로직 오류, 미사용 코드 |

### 다. 명세기반·구조기반·경험기반 테스트 기법

**명세기반 (Specification-based)**

| 기법 | 설명 | 사례 |
|------|------|------|
| **동등 분할** | 입력을 동등한 클래스로 나눠 대표값 선택 | 나이: 유효(1~120), 음수, 121이상 |
| **경계값 분석** | 경계 지점의 값 테스트 | 1, 0, 120, 121 |
| **결정 테이블** | 조건 조합별 행동 명세 | 할인 조건 4가지 조합 테이블 |
| **상태 전이** | 상태 변화 시나리오 | 주문 상태(주문→결제→배송→완료) |
| **페어와이즈** | 2인자 조합 테스트로 케이스 수 감소 | 브라우저×OS 조합 |

**구조기반 (Structure-based)**

| 기법 | 커버리지 기준 |
|------|-------------|
| **구문 커버리지** | 모든 소스코드 구문 실행 |
| **분기 커버리지** | 모든 if/else 분기 실행 |
| **조건 커버리지** | 복합 조건의 개별 조건 T/F 실행 |
| **경로 커버리지** | 모든 독립 실행 경로 (McCabe 복잡도) |

**경험기반 (Experience-based)**

| 기법 | 설명 |
|------|------|
| **오류 추정** | 과거 경험으로 오류 발생 가능한 곳 집중 테스트 |
| **탐색적 테스트** | 동시에 테스트 설계·실행·학습 수행 |
| **체크리스트 기반** | 과거 결함 목록 기반 테스트 |

## 출제 포인트

- ISTQB 7대 원리 (특히 살충제 패러독스, 결함 집중)
- 블랙박스 vs 화이트박스 비교표
- 명세기반 5가지 기법과 각 설명
- 구조기반 커버리지 4단계 (구문→분기→조건→경로)
