## 개요

시스템 콜(System Call)은 사용자 프로그램이 운영체제 커널이 제공하는 서비스(파일 입출력, 프로세스 생성, 메모리 할당 등)를 요청하는 인터페이스다. 사용자 모드에서 커널 모드로 전환하는 유일한 합법적 경로다.

## 핵심 내용

### 시스템 콜의 필요성

- **보호 링(Protection Ring)**: CPU는 사용자 모드(Ring 3)와 커널 모드(Ring 0)를 구분
- **보안**: 사용자 프로그램이 직접 하드웨어나 커널 데이터에 접근하면 시스템 불안정·보안 위험
- **추상화**: 하드웨어 세부사항 숨기고 일관된 인터페이스 제공

### 시스템 콜 동작 과정

```
사용자 프로그램
      ↓ 시스템 콜 요청 (예: read())
소프트웨어 인터럽트 발생 (int 0x80 또는 syscall 명령)
      ↓
CPU: 사용자 모드 → 커널 모드 전환
      ↓
인터럽트 벡터 테이블에서 시스템 콜 핸들러 찾기
      ↓
커널이 요청 처리 (파일 읽기, 네트워크 등)
      ↓
결과값 반환 후 커널 모드 → 사용자 모드 복귀
      ↓
사용자 프로그램 계속 실행
```

### 시스템 콜 분류

| 분류 | 예시 시스템 콜 |
|------|--------------|
| **프로세스 제어** | fork(), exec(), exit(), wait() |
| **파일 관리** | open(), read(), write(), close(), unlink() |
| **장치 관리** | ioctl(), read(), write() (장치 파일) |
| **정보 유지** | getpid(), alarm(), sleep() |
| **통신** | pipe(), socket(), send(), recv() |

### 사용자 모드 vs 커널 모드

| 구분 | 사용자 모드 | 커널 모드 |
|------|------------|----------|
| **권한** | 제한된 CPU 명령 | 모든 CPU 명령 가능 |
| **메모리 접근** | 사용자 공간만 | 전체 메모리 |
| **하드웨어 접근** | 불가 (직접) | 가능 |
| **실행 주체** | 애플리케이션 | OS 커널 |
| **전환 방법** | 시스템 콜 발생 | 반환(return from syscall) |

### libc와 시스템 콜 관계

사용자는 보통 시스템 콜을 직접 호출하지 않고 표준 라이브러리(libc)의 래퍼 함수를 사용:
- `printf()` → `write()` 시스템 콜
- `malloc()` → `brk()`/`mmap()` 시스템 콜
- `fopen()` → `open()` 시스템 콜

## 출제 포인트

- 시스템 콜의 목적: 사용자 모드 → 커널 모드 전환의 유일한 합법적 경로
- 동작 과정: 소프트웨어 인터럽트 → 커널 처리 → 복귀
- 분류 5가지: 프로세스·파일·장치·정보·통신
- 사용자 모드와 커널 모드의 차이
