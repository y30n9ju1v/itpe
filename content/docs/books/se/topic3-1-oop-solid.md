## 해설

> **유연하고 복원력 있는 소프트웨어의 뼈대**  
> 객체지향 설계의 5대 원칙, SOLID

클라우드 네이티브와 마이크로서비스 아키텍처(MSA)가 주류가 된 현재에도, 시스템의 최소 단위인 '클래스'와 '인터페이스'가 부실하면 거대한 시스템 전체가 기술 부채의 늪에 빠지게 됩니다. 로버트 C. 마틴(Robert C. Martin)이 정립한 SOLID 원칙은 높은 응집도와 느슨한 결합도를 달성하여 변경의 파도 속에서도 부러지지 않는 유연한 코드를 만드는 영원한 지침서입니다.

### 1. 출제 배경 및 의도

오늘날 소프트웨어 공학의 가장 큰 화두 중 하나는 코파일럿(Copilot)과 같은 생성형 AI에 의한 대규모 코드 양산 환경에서의 '코드 스멜(Code Smell) 통제'입니다. AI가 작성한 코드는 겉보기에는 빠르게 동작하지만, 객체지향의 설계 원칙을 무시하면 순식간에 유지보수가 불가능한 레거시 스파게티 코드로 변모합니다.

채점관이 SOLID 원칙을 출제하는 의도는 각 원칙이 소프트웨어의 변경 비용(Change Cost)을 어떻게 낮추는지, 그리고 각 원칙이 깨졌을 때 발생하는 부작용을 실무적인 관점에서 진단하고 처방할 수 있는지를 검증하기 위함입니다.

### 2. 개념의 본질과 작동 메커니즘

SOLID 원칙의 본질은 "고객의 요구사항이 변경되어 코드를 수정해야 할 때, 이미 검증된 기존 코드의 수정을 최소화하고(격리), 새로운 기능을 안전하게 확장(유연성)하는 구조적 메커니즘"입니다.

다섯 가지 원칙은 각각 독립적으로 움직이지 않고 서로가 서로를 보완하는 유기적 관계로 작동합니다. SRP를 지켜 클래스를 잘 쪼개 놓아야 OCP를 실현할 수 있는 추상화의 토대가 마련되며, DIP를 통해 쪼개진 모듈 간의 결합도를 낮추게 됩니다.

### 3. 핵심 키워드 (Must-Have)

#### 1) SRP (Single Responsibility Principle) — 단일 책임 원칙

- **본질**: 하나의 클래스는 단 하나의 책임(기능)만 가져야 한다
- **Must-Have 키워드**: 하나의 변경 이유(Single reason to change), 높은 응집도, 액터(Actor)의 분리
- **실무적 의미**: 회계 보고서 클래스가 '데이터 포맷팅'과 'DB 자원 연결'을 동시에 한다면 두 가지 이유로 변경되므로 SRP 위반

#### 2) OCP (Open-Closed Principle) — 개방-폐쇄 원칙

- **본질**: 소프트웨어 엔티티는 확장에는 열려 있어야 하고, 변경에는 닫혀 있어야 한다
- **Must-Have 키워드**: 추상화(Interface/Abstract Class), 다형성, 코드 수정 없는 기능 추가
- **실무적 의미**: 새로운 요구사항이 들어왔을 때 기존 소스코드를 대대적으로 수정하는 대신, 인터페이스를 상속받는 새로운 클래스를 구현함으로써 기능을 확장

#### 3) LSP (Liskov Substitution Principle) — 리스코프 치환 원칙

- **본질**: 서브타입(하위 클래스)은 언제나 기반 타입(상위 클래스)을 대체할 수 있어야 한다
- **Must-Have 키워드**: 다형성 계약 준수, 상속의 올바른 사용, 직사각형-정사각형 문제(대표적 반례)
- **실무적 의미**: 부모 클래스 자리에 자식 클래스를 넣어도 프로그램의 원래 의도와 명세가 깨지지 않고 정상 동작해야 함

#### 4) ISP (Interface Segregation Principle) — 인터페이스 분리 원칙

- **본질**: 클라이언트는 자신이 사용하지 않는 메서드에 의존하도록 강제되어서는 안 된다
- **Must-Have 키워드**: 날씬한 인터페이스(Lean Interface), 인터페이스 다중 상속, 역할 중심 분리
- **실무적 의미**: 하나의 거대한 범용 인터페이스보다, 여러 개의 구체적인 인터페이스가 훨씬 우수함. 사용하지 않는 메서드의 변경 때문에 상관없는 클라이언트 모듈까지 재배포되는 참사를 방지

#### 5) DIP (Dependency Inversion Principle) — 의존성 역전 원칙

- **본질**: 고수준 모듈은 저수준 모듈에 의존해서는 안 되며, 둘 다 추상화에 의존해야 한다
- **Must-Have 키워드**: 추상화에 의존, 결합도 완화, 의존성 주입(DI: Dependency Injection)
- **실무적 의미**: 변하기 쉬운 구체적인 것(구현 클래스)에 의존하지 말고, 변하지 않는 것(인터페이스)에 의존하라는 원칙. Spring 프레임워크의 핵심 구동 원리

### 4. 맥락과 비교

SOLID 원칙은 결국 소프트웨어 엔지니어링의 대원칙인 "낮은 결합도(Low Coupling)와 높은 응집도(High Cohesion)"를 달성하기 위한 구체적인 전술 지도입니다.

| 원칙 | 주로 기여하는 설계 품질 | 깨졌을 때 발생하는 현상 | MSA로의 확장 맥락 |
|------|----------------------|----------------------|-----------------|
| SRP | 높은 응집도 | 하나의 기능을 고쳤는데 사방에서 에러가 터짐 (경직성) | 마이크로서비스의 경계 (Bounded Context) 정의 기준 |
| OCP | 낮은 결합도, 확장성 | 새로운 기능 추가 시 기존 코드의 대대적인 if-else문 수정 발생 | 플러그인 아키텍처 및 무중단 배포 환경의 근간 |
| LSP | 신뢰성, 다형성 보장 | 하위 클래스 인스턴스 사용 시 형변환 오류 및 예외 처리 남발 | 상위 계약(Contract) 기반의 컴포넌트 대체 가능성 보장 |
| ISP | 모듈 가독성, 독립 배포 | 사용하지 않는 인터페이스 메서드의 시그니처 변경으로 인한 동반 재배포 | 서비스 간 필요한 최소한의 API 인터페이스만 노출 |
| DIP | 낮은 결합도, 테스트 용이성 | 하위 모듈(예: 특정 DB)이 바뀌면 상위 모듈(비즈니스 로직)을 통째로 재작성 | Hexagonal/Clean 아키텍처에서 포트와 어댑터 구조로 발현 |

### 5. 실무 제언

**인터페이스 폭발(Interface Explosion)과 추상화의 비용**

- **챌린지**: SOLID 원칙, 특히 OCP와 DIP를 지나치게 의식한 나머지 변경 가능성이 전혀 없는 단순한 유틸리티 클래스나 CRUD 로직에까지 일일이 인터페이스를 구현하는 아키텍처 왜곡 현상이 자주 발생합니다. 이는 코드의 가독성을 저해하고 시스템을 무겁게 만드는 '오버 엔지니어링'입니다.
- **제언**: '요구사항의 가변성 비용 분석'을 선행해야 합니다. 비즈니스 규칙이 자주 바뀌는 핵심 도메인 영역(Core Domain)에는 SOLID 원칙을 엄격하게 적용하되, 변경 가능성이 극히 낮은 인프라성 유틸리티 영역은 YAGNI 원칙과의 균형을 잡아야 합니다.

**정적 코드 분석 도구를 통한 품질 거버넌스 자동화**

- **챌린지**: 코드 리뷰 단계에서 인간의 눈으로 수만 줄의 소스코드가 SOLID 원칙을 잘 지켰는지 검증하는 것은 현실적으로 불가능하며, 주관적 평가에 치우치기 쉽습니다.
- **제언**: SonarQube, ArchUnit 등의 정적 아키텍처 검증 도구를 CI/CD 파이프라인과 연계해야 합니다. "특정 패키지 내의 고수준 모듈은 하위 구현체 패키지를 직접 import 할 수 없다"와 같은 DIP 원칙 규정을 ArchUnit 테스트 코드로 자동화해 두면, SOLID 원칙 위반을 빌드 단계에서 원천 차단하는 강력한 기술 거버넌스를 확립할 수 있습니다.
