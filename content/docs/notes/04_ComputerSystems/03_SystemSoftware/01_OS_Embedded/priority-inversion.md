## 개요

우선순위 역전(Priority Inversion)은 실시간 시스템에서 낮은 우선순위 프로세스가 자원을 점유한 채 높은 우선순위 프로세스를 간접적으로 차단하고, 중간 우선순위 프로세스가 먼저 실행되는 현상이다. 1997년 Mars Pathfinder 임무에서 실제로 발생하여 시스템 리셋을 유발한 사례로 유명하다.

## 핵심 내용

### 우선순위 역전 발생 메커니즘

```
시나리오: 우선순위 H > M > L

1. L이 공유 자원(mutex) 획득
2. H가 깨어나 자원 요청 → 차단(L 대기)
3. M이 실행 가능 → H보다 먼저 실행 (우선순위 역전!)
4. M 실행 중 L은 대기 → H는 더 오래 차단
```

**발생 원인**: 뮤텍스(Mutex)/세마포어 사용 시 낮은 우선순위 프로세스가 자원 해제 전에 선점당하는 상황.

### PIP vs PCP (OPCP & IPCP) 비교

| 구분 | PIP (Priority Inheritance) | OPCP (Original Priority Ceiling) | IPCP (Immediate Priority Ceiling) |
|------|---------------------------|----------------------------------|-----------------------------------|
| **개념** | 대기 중인 최고 우선순위를 임시로 상속 | 공유자원에 Ceiling 설정, **System Ceiling보다 높아야** 자원 획득 | 자원 획득과 **동시에** 자원 상한값(Ceiling)으로 우선순위 즉시 격상 |
| **동작 메커니즘** | H가 자원 대기 시에만 L의 우선순위 동적 상승 | 대기 태스크 발생 시에만 우선순위 상승 (System Ceiling 차단 기법) | 대기 중인 태스크가 없더라도 **자원 획득 즉시 무조건 상승** |
| **교착상태(Deadlock) 방지** | 불가 (우선순위 역전만 방지) | **방지 가능** (자원 선할당 방지) | **방지 가능** |
| **Context Switch 횟수** | 보통 | 보통 | **최소화** (불필요한 스케줄링 배제) |
| **구현 복잡도** | 단순 | 복잡 (System Ceiling 실시간 추적) | 비교적 단순 (자원별 고정 상한 부여) |
| **지연 시간 예측** | 어려움 (연쇄 대기 가능) | 예측 용이 (최대 1회 블로킹 보장) | 예측 용이 (최대 1회 블로킹 보장) |
| **사용 환경** | 일반 임베디드 OS | 일반 실시간 시스템 | 항공, 철도, 자동차(AUTOSAR OS) |

### PIP 동작 예시

```
L 우선순위: 1 (낮음), H 우선순위: 10 (높음)

PIP 적용:
1. L이 mutex 획득 (우선순위 1)
2. H가 mutex 요청 → L의 우선순위를 10으로 상승
3. M(우선순위 5)이 깨어나도 L(임시 10)이 먼저 실행
4. L이 mutex 해제 → 원래 우선순위(1) 복귀 → H 실행
```

## 실무 적용 예시

- Mars Pathfinder(1997): VxWorks RTOS에서 우선순위 역전 발생 → 시스템 리셋. 원격으로 PIP(우선순위 상속) 활성화하여 해결.
- 자동차 ECU: AUTOSAR OS에서 PCP 적용으로 안전 임계 태스크 응답 시간 보장.

## 출제 포인트

- **우선순위 역전 발생 메커니즘**: H태스크가 공유 자원 때문에 L에 대기하다가, 중간 순위 M태스크에 의해 L이 선점되어 H가 오랫동안 블로킹되는 시나리오 서술
- **PIP의 동적 상속 개념** 및 단점(교착상태 미방지, 체인 블로킹)
- **PCP의 세분화**: **OPCP**(System Ceiling 기반 동적 상승) vs **IPCP**(획득 즉시 상승, 에뮬레이션 방식)의 작동 원리 및 장단점 비교
- PCP가 교착상태 방지 및 최악실행시간(WCET) 분석에 유리한 공학적 이유 서술
