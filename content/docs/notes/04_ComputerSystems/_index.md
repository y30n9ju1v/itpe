# 컴퓨터시스템: 0과 1이 세상을 움직이는 방법

소프트웨어가 아무리 정교해도 결국 어딘가에서 전기 신호가 켜지고 꺼지는 일이 일어난다. 컴퓨터시스템은 그 가장 낮은 층, 0과 1의 세계에서 시작해 CPU, 메모리, 운영체제, 가상화, 알고리즘, 인프라 아키텍처에 이르기까지 층층이 쌓인 거대한 구조다. 이 책은 그 구조를 아래에서 위로, 가장 작은 레지스터에서 수만 개의 GPU가 연결된 데이터센터까지 따라간다.

---

## 1부. CS 기초

### 1.1 RISC-V — 개방형 CPU 명령어의 새 표준

우리가 사용하는 스마트폰, 노트북, 서버에 들어 있는 CPU는 모두 특정 **명령어 집합 구조(ISA, Instruction Set Architecture)**를 따른다. ISA는 소프트웨어가 하드웨어에게 무언가를 시킬 때 사용하는 언어 규약이다. 오랫동안 이 시장은 Intel의 x86과 ARM이 양분했다. 그런데 최근 **RISC-V(리스크 파이브)**라는 새로운 ISA가 빠르게 주목받고 있다.

**RISC-V**는 버클리 대학교에서 2010년 개발한 **오픈소스 ISA**다. 누구나 무료로 사용하고 수정할 수 있다는 점이 핵심 차별점이다. x86은 Intel에 라이선스 비용을 내야 하고, ARM도 사용료가 있지만 RISC-V는 완전히 자유롭다.

RISC-V는 이름처럼 **RISC(Reduced Instruction Set Computer)** 철학을 따른다. 복잡한 명령어를 많이 두는 것보다, 단순한 명령어를 빠르게 처리하는 것이 효율적이라는 생각이다. 명령어 집합이 작고 단순하니 칩 설계도 단순해지고, 전력 소비가 줄고, 검증이 쉬워진다.

현재 RISC-V는 IoT 센서, AI 가속기, 우주 탐사 장비, 중국의 독자 프로세서 등 다양한 분야에 채택되고 있다. 미중 반도체 패권 경쟁 속에서 ARM에 대한 의존도를 줄이려는 움직임과 맞물려 RISC-V의 전략적 중요성은 더욱 커지고 있다.

## 2부. 하드웨어 시스템

### 2.1 CPU 레지스터 — 프로세서 내부의 초고속 저장소

컴퓨터가 계산을 할 때 데이터를 어딘가에 임시로 보관해야 한다. 하드디스크는 너무 느리고, RAM도 CPU에 비하면 느리다. 가장 빠른 저장 공간이 CPU 내부의 **레지스터(Register)**다.

레지스터는 CPU 내부에 있는 초소형 저장 공간으로, 수 나노초 안에 읽고 쓸 수 있다. CPU가 덧셈을 할 때 두 숫자를 레지스터에 올려두고, 계산 결과도 레지스터에 저장한 뒤, 필요하면 RAM으로 내려보낸다.

레지스터는 역할에 따라 종류가 나뉜다. **범용 레지스터(General Purpose Register)**는 산술·논리 연산에 사용되는 다목적 레지스터다. **프로그램 카운터(PC, Program Counter)**는 다음에 실행할 명령어의 메모리 주소를 가리킨다. **스택 포인터(SP)**는 스택 메모리의 현재 위치를 추적한다. **링크 레지스터(LR)**는 함수 호출 후 돌아올 주소를 저장한다.

**CPU 상태 레지스터(Status Register 또는 Flags Register)**는 조금 특별하다. 연산 결과의 특성을 비트 플래그로 기록한다. 예를 들어 덧셈 결과가 0이면 **제로 플래그(Z)**가 세워지고, 올림수가 발생하면 **캐리 플래그(C)**가 세워진다. 프로그램은 이 플래그를 보고 조건 분기(if문)를 결정한다. `if (a == b)`는 내부적으로 a-b를 계산하고 제로 플래그가 1인지 확인하는 것이다.

### 2.2 캐시 메모리 — CPU와 RAM 사이의 완충 지대

CPU가 아무리 빠르더라도 RAM에서 데이터를 가져오는 데 수백 클럭이 걸린다면 기다리는 시간이 너무 아깝다. 이 속도 차이를 메우기 위해 **캐시 메모리(Cache Memory)**가 존재한다.

캐시는 CPU와 RAM 사이에 위치한 작지만 빠른 메모리다. CPU가 데이터에 접근할 때, 먼저 캐시에서 찾는다. 있으면(**캐시 히트, Cache Hit**) 빠르게 가져오고, 없으면(**캐시 미스, Cache Miss**) RAM에서 가져와 캐시에도 저장한다.

캐시는 보통 계층으로 구성된다. **L1 캐시**는 CPU 코어 안에 있어 가장 빠르지만 용량이 작다(수십 KB). **L2 캐시**는 조금 더 크고 느리며(수백 KB~수 MB), **L3 캐시**는 여러 코어가 공유하는 더 큰 캐시(수 MB~수십 MB)다.

캐시가 효과적인 이유는 **지역성의 원리(Locality of Reference)** 때문이다. **시간 지역성(Temporal Locality)** — 방금 접근한 데이터는 곧 다시 접근할 가능성이 높다(반복문 내 변수). **공간 지역성(Spatial Locality)** — 접근한 데이터 근처의 데이터도 곧 접근할 가능성이 높다(배열 순차 접근). 캐시는 이 패턴을 활용해 미리 근처 데이터를 함께 가져와둔다(프리페칭).

### 2.3 파이프라인과 해저드 — 명령어를 조립 라인처럼 처리하다

공장의 조립 라인을 상상해보자. 자동차 한 대가 완성되기를 기다린 뒤 다음 차를 시작하는 것보다, 각 공정이 동시에 다른 차를 처리하면 훨씬 효율적이다. CPU의 **파이프라인(Pipeline)**이 바로 이 원리다.

CPU 명령어 처리는 보통 **인출(Fetch) → 해독(Decode) → 실행(Execute) → 메모리 접근(Memory) → 쓰기(Write-back)**의 단계로 이루어진다. 파이프라인은 이 단계들을 겹쳐서 실행한다. 1번 명령이 해독되는 동안 2번 명령을 인출하고, 1번이 실행되는 동안 2번을 해독하고 3번을 인출하는 식이다.

그러나 파이프라인에는 흐름을 방해하는 **해저드(Hazard)**가 있다.

**데이터 해저드(Data Hazard)**는 앞 명령의 결과를 뒤 명령이 아직 준비되지 않은 상태에서 필요로 할 때 발생한다. 해결책으로 **포워딩(Forwarding)**이나 **파이프라인 스톨(Stall)**을 사용한다.

**제어 해저드(Control Hazard)**는 분기 명령(if, goto) 때문에 발생한다. 분기 결과가 나오기 전에 미리 꺼내 놓은 명령어가 엉뚱한 것일 수 있다. **분기 예측(Branch Prediction)**으로 대응한다.

**구조 해저드(Structural Hazard)**는 여러 명령이 같은 하드웨어 자원을 동시에 필요로 할 때 발생한다. 자원을 복제하거나 스톨로 해결한다.

### 2.4 MMU — 가상 주소를 물리 주소로 바꾸는 번역기

프로그램이 메모리에 접근할 때 사용하는 주소는 **가상 주소(Virtual Address)**다. 이를 실제 RAM의 **물리 주소(Physical Address)**로 변환하는 하드웨어가 **MMU(Memory Management Unit)**다.

MMU는 CPU와 RAM 사이에 위치하며, **페이지 테이블(Page Table)**을 참조해 주소 변환을 수행한다. 모든 변환마다 페이지 테이블을 조회하면 느리기 때문에, 최근 변환 결과를 **TLB(Translation Lookaside Buffer)**라는 캐시에 저장한다. TLB에 있으면 1~2 클럭, 없으면 수백 클럭이 소요된다.

MMU는 주소 변환 외에 **메모리 보호(Memory Protection)**도 담당한다. 각 페이지에 읽기(R)·쓰기(W)·실행(X) 권한을 설정하고, 위반 시 **세그멘테이션 폴트(Segmentation Fault)**를 발생시켜 프로세스를 종료한다. 덕분에 한 프로세스가 다른 프로세스의 메모리를 침범할 수 없다.

---

### 2.5 DMA — CPU 없이도 데이터를 옮기다

하드디스크에서 파일을 읽을 때마다 CPU가 일일이 한 바이트씩 옮겨야 한다면 CPU는 다른 일을 못 할 것이다. **DMA(Direct Memory Access)**는 CPU 개입 없이 장치와 메모리 사이에서 데이터를 직접 전송하는 기술이다.

DMA 컨트롤러는 CPU로부터 "이 장치에서 이 메모리 주소로 N바이트 복사해"라는 명령을 받고 독립적으로 전송을 수행한다. 전송이 완료되면 인터럽트로 CPU에 알린다. CPU는 그동안 다른 계산을 할 수 있어 전체 시스템 효율이 크게 오른다. 네트워크 카드, SSD 컨트롤러, GPU 등 대부분의 고속 I/O 장치가 DMA를 활용한다.

---

### 2.6 메모리 종류들 — ROM, FRAM, SSD

컴퓨터의 저장 장치는 전원이 꺼져도 데이터가 유지되는지(비휘발성), 얼마나 빠른지, 얼마나 자주 쓸 수 있는지에 따라 다양하게 나뉜다.

**ROM(Read-Only Memory)**은 이름처럼 원래 읽기 전용이었지만, 현대에는 쓰기가 가능한 변형들이 일반적이다. **PROM**은 한 번만 쓸 수 있고, **EPROM**은 자외선으로 지울 수 있으며, **EEPROM**은 전기적으로 지우고 쓸 수 있다. 우리가 흔히 아는 **플래시 메모리**는 EEPROM의 일종이다. 스마트폰 펌웨어, BIOS, 부팅 코드가 ROM 계열에 저장된다.

**FRAM(Ferroelectric RAM, 강유전체 RAM)**은 특별한 소재를 사용해 DRAM의 속도와 플래시의 비휘발성을 동시에 갖춘 혁신적 메모리다. 쓰기 횟수 제한이 없고(플래시는 수만~수십만 회 제한), 전력 소비가 극히 낮아 IoT 기기, 의료 기기, 우주 탐사 장비 등 전력과 내구성이 중요한 분야에 활용된다.

**SSD(Solid State Drive)**는 플래시 메모리를 기반으로 한 저장 장치다. 기계적 부품이 없어 HDD보다 훨씬 빠르고 충격에 강하다. SSD는 데이터를 **셀(Cell)**에 저장하는데, 셀당 비트 수에 따라 SLC(1bit, 빠르고 내구성 높음), MLC(2bit), TLC(3bit, 저렴하지만 느리고 수명 짧음), QLC(4bit)로 나뉜다. 수명을 균등하게 사용하기 위한 **웨어 레벨링(Wear Leveling)**과 쓰기 전 지워야 하는 특성 때문에 복잡한 내부 관리가 이루어진다.

---

### 2.7 NPU와 병렬 컴퓨팅 — AI 시대의 새로운 두뇌

스마트폰으로 사진을 찍으면 자동으로 피부 보정이 되고, 음성 명령을 인식한다. 이런 AI 기능을 실시간으로 처리하기 위해 스마트폰 칩에는 **NPU(Neural Processing Unit)**가 탑재된다.

NPU는 딥러닝 연산, 특히 **행렬 곱셈(Matrix Multiplication)**에 특화된 프로세서다. CPU는 범용적이라 AI 연산에는 비효율적이고, GPU는 고성능이지만 전력 소비가 크다. NPU는 AI 연산만을 위해 설계된 전용 회로로, 낮은 전력으로 빠른 AI 추론을 가능하게 한다.

**병렬 컴퓨팅(Parallel Computing)**은 하나의 큰 문제를 여러 작은 문제로 나눠 동시에 처리하는 방식이다. GPU가 이의 대표 주자로, 수천 개의 작은 코어가 동시에 계산한다. **Flynn의 분류**로 병렬 구조를 정리하면, **SIMD(Single Instruction Multiple Data)**는 하나의 명령이 여러 데이터에 동시 적용(벡터 연산, GPU의 기본 원리)되고, **MIMD(Multiple Instruction Multiple Data)**는 여러 프로세서가 독립적으로 다른 명령을 수행(멀티코어 CPU)하는 방식이다.

---

### 2.8 시스템 버스 중재 — 누가 먼저 버스를 사용할까

CPU, 메모리, 그래픽카드, 네트워크 카드는 모두 데이터를 주고받기 위해 **시스템 버스(System Bus)**를 공유한다. 여러 장치가 동시에 버스를 사용하려 하면 충돌이 난다. **버스 중재(Bus Arbitration)**는 누가 버스를 사용할지 결정하는 메커니즘이다.

**중앙집중식 중재**에서는 **버스 중재자(Bus Arbiter)**가 모든 장치의 요청을 받아 우선순위에 따라 허가를 내린다. **분산식 중재**에서는 각 장치가 스스로 버스 사용 가능 여부를 판단한다. 현대 시스템에서는 PCIe처럼 점대점(Point-to-Point) 연결로 버스 충돌 문제 자체를 없애는 방향으로 발전했다.

---

### 2.9 비디오 압축 — 영상 데이터를 작게 만드는 기술

4K 영상 한 시간 분량을 압축하지 않으면 수백 GB에 달한다. **비디오 압축**은 이를 수 GB로 줄이는 기술이다.

비디오 압축의 핵심은 **중복 제거**다. **공간 중복(Spatial Redundancy)** — 한 프레임 안에서 인접한 픽셀은 비슷한 색상을 가진다. JPEG처럼 이를 DCT(이산 코사인 변환)로 압축한다. **시간 중복(Temporal Redundancy)** — 연속된 프레임에서 대부분의 영역은 변하지 않는다. 변화한 부분만 기록하는 **인터프레임(Inter-frame) 압축**이 핵심이다. I프레임(완전 이미지)을 기준으로 P프레임(앞 프레임과 차이만), B프레임(앞뒤 프레임 모두 참조)을 조합한다.

H.264, H.265(HEVC), AV1이 대표적인 비디오 코덱이다. H.265는 H.264 대비 동일 화질을 절반 용량으로 저장할 수 있고, AV1은 오픈소스로 로열티 없이 사용할 수 있어 유튜브·넷플릭스가 채택했다.

### 2.10 HW 사이징 — 얼마나 큰 서버가 필요할까

정보시스템을 구축할 때 "서버를 몇 대, 얼마나 큰 것으로 사야 하는가?"를 결정하는 것이 **HW 사이징(Hardware Sizing)**이다. 너무 작게 사면 성능이 부족하고, 너무 크게 사면 예산 낭비다.

사이징의 핵심은 **피크 트래픽 추정**이다. 평상시 사용자 수가 아닌, 가장 바쁜 시간대의 동시 접속자 수와 트랜잭션 처리량(TPS, Transactions Per Second)을 기반으로 계산한다. CPU는 피크 사용률이 70~80%를 넘지 않도록, 메모리는 사용률이 80% 이하가 되도록 여유를 둔다.

또한 **3~5년 성장률**을 감안해 초기 과투자 없이 확장 가능한 설계(Scale-out 가능 여부)를 고려한다. 클라우드 환경에서는 온디맨드로 자원을 늘릴 수 있어 초기 과잉 사이징의 필요성이 줄어들었다.

## 3부. 운영체제

### 3.1 시스템 콜 — 프로그램이 OS에 부탁하는 방법

사용자가 만든 프로그램은 직접 하드웨어에 접근할 수 없다. 파일을 읽고, 네트워크로 데이터를 보내고, 메모리를 더 할당받으려면 운영체제에 요청해야 한다. 이 요청의 공식 창구가 **시스템 콜(System Call)**이다.

CPU는 **사용자 모드(User Mode)**와 **커널 모드(Kernel Mode)** 두 가지 실행 모드가 있다. 사용자 프로그램은 사용자 모드에서 동작하며 하드웨어에 직접 접근할 수 없다. 시스템 콜이 호출되면 **소프트웨어 인터럽트**가 발생해 CPU가 커널 모드로 전환되고, OS가 요청을 처리한 뒤 결과를 반환하고 다시 사용자 모드로 복귀한다.

시스템 콜은 프로세스 제어(`fork`, `exec`), 파일 관리(`open`, `read`, `write`), 장치 관리(`ioctl`), 통신(`socket`, `send`) 등으로 분류된다. 우리가 코드에서 쓰는 `printf()`나 `malloc()` 같은 표준 라이브러리 함수들은 내부적으로 적절한 시스템 콜을 호출하는 편의 함수다.

---

### 3.2 가상 메모리 — RAM보다 큰 프로그램을 실행하는 마법

4GB RAM이 장착된 컴퓨터에서 8GB짜리 게임을 실행할 수 있을까? 가상 메모리 덕분에 가능하다. **가상 메모리(Virtual Memory)**는 실제 RAM보다 큰 메모리 공간을 프로세스에 제공하는 운영체제 기술이다.

원리는 이렇다. 프로그램이 실제로 사용하는 부분만 RAM에 올려두고, 나머지는 하드디스크의 **스왑 공간(Swap Space)**에 둔다. 프로그램이 스왑에 있는 데이터에 접근하면 **페이지 폴트(Page Fault)**가 발생하고, OS가 해당 데이터를 디스크에서 RAM으로 가져온다.

가상 메모리의 장점은 많다. RAM 용량을 초과하는 프로그램 실행이 가능하고, 각 프로세스가 독립된 주소 공간을 가져 서로를 침범하지 못한다. `fork()`로 자식 프로세스를 생성할 때 **Copy-on-Write(COW)** 기법으로 실제 메모리 복사를 최대한 지연한다.

단점도 있다. **스래싱(Thrashing)**은 가상 메모리의 최악 상황이다. RAM이 부족해 페이지 폴트가 폭발적으로 증가하면 CPU가 실제 계산은 거의 못 하고 데이터를 디스크와 RAM 사이에서 옮기는 일만 반복한다. 시스템이 거의 멈춘 것처럼 느껴지는 원인이 이것이다.

페이지 교체 알고리즘은 어떤 페이지를 스왑으로 내보낼지 결정한다. **LRU(Least Recently Used)**는 가장 오랫동안 사용되지 않은 페이지를 교체하는 방식으로 실용성이 높다. **FIFO**는 단순하지만 **벨레이디 이상현상(Belady's Anomaly)**처럼 페이지 수를 늘렸는데 오히려 폴트가 증가하는 역설이 있다.

---

### 3.3 메모리 영역 구조 — 프로그램이 메모리를 쓰는 방식

프로그램이 실행될 때 메모리는 용도별로 나뉘어 사용된다. 이 구조를 이해하면 많은 프로그래밍 오류의 원인을 파악할 수 있다.

**코드(텍스트) 영역**에는 실행 가능한 기계어 코드가 저장된다. 읽기 전용이므로 여기에 쓰려 하면 세그멘테이션 폴트가 발생한다. **데이터 영역**은 초기값이 있는 전역·정적 변수가, **BSS 영역**은 초기화되지 않은 전역·정적 변수가 저장된다.

**스택(Stack)**은 함수 호출마다 생성되는 지역 변수와 반환 주소가 저장되는 영역이다. 함수가 호출되면 스택 프레임이 쌓이고(Push), 함수가 종료되면 제거된다(Pop). 재귀 함수를 무한히 호출하면 스택이 꽉 차는 **스택 오버플로우**가 발생한다.

**힙(Heap)**은 `malloc()`·`new`로 동적으로 할당하는 영역이다. 프로그래머가 직접 할당하고 해제해야 한다. 해제를 잊으면 **메모리 누수(Memory Leak)**, 이미 해제된 메모리를 다시 사용하면 **댕글링 포인터(Dangling Pointer)** 문제가 생긴다.

---

### 3.4 동적 메모리 할당 — 힙 메모리의 숨겨진 복잡성

프로그램이 실행 중에 필요한 메모리를 그때그때 요청하는 것이 **동적 메모리 할당**이다. C에서는 `malloc()`·`free()`, C++에서는 `new`·`delete`, Java·Python에서는 런타임이 자동으로 관리한다.

힙 메모리 관리자는 큰 메모리 풀을 여러 조각으로 나눠 제공한다. 그런데 시간이 지나면 **단편화(Fragmentation)**가 발생한다. **외부 단편화**는 총 여유 메모리는 충분하지만 연속된 큰 블록이 없어 큰 요청을 수용하지 못하는 상태다. **내부 단편화**는 요청 크기보다 조금 더 큰 블록을 할당해 남는 공간이 낭비되는 상태다.

**메모리 누수(Memory Leak)**는 할당한 메모리를 해제하지 않아 점점 사용 메모리가 증가하는 버그다. 장기 실행 서버에서 특히 치명적이다. Valgrind, AddressSanitizer 같은 도구로 탐지한다. Java·Python 같은 언어의 **가비지 컬렉터(GC)**는 이 문제를 자동으로 해결하지만, GC 실행 시 멈추는(Stop-the-World) 오버헤드가 있다.

---

### 3.5 세그멘테이션 폴트 — "접근 금지" 표지판에 걸리다

프로그래밍을 하다 보면 프로그램이 갑자기 죽으면서 "Segmentation fault (core dumped)"라는 메시지를 볼 때가 있다. **세그멘테이션 폴트(Segmentation Fault)**는 프로세스가 접근 권한이 없는 메모리 영역에 접근했을 때 OS가 해당 프로세스를 강제 종료시키는 것이다.

가장 흔한 원인은 **NULL 포인터 역참조**다. `int *p = NULL; *p = 5;` — p가 아무것도 가리키지 않는데 거기에 5를 쓰려 하니 OS가 막는 것이다. **배열 범위 초과**도 흔하다. 크기 5인 배열에서 `arr[10]`에 접근하면 배열 너머의 메모리에 접근하게 된다.

보안 관점에서 **버퍼 오버플로우(Buffer Overflow)**는 악의적으로 활용될 수 있다. 함수의 반환 주소가 저장된 스택 영역을 덮어써 임의 코드를 실행하는 공격이 가능하다. 이를 방지하기 위해 **ASLR(주소 공간 랜덤화)**, **스택 카나리(Stack Canary)**, **NX(No-Execute) 비트** 같은 보안 기술이 현대 OS에 내장되어 있다.

---

### 3.6 OS 스케줄링 — CPU 시간을 어떻게 나눌까

컴퓨터에서는 수십~수백 개의 프로세스가 동시에 실행되지만, CPU 코어 수는 제한적이다. 어느 프로세스에게 CPU를 줄지 결정하는 것이 **OS 스케줄링(Scheduling)**이다.

**비선점형(Non-preemptive)** 스케줄링은 프로세스가 스스로 CPU를 놓을 때까지 기다린다. 단순하지만 한 프로세스가 오래 실행되면 다른 프로세스가 굶는다. **선점형(Preemptive)** 스케줄링은 OS가 강제로 CPU를 빼앗아 다른 프로세스에 줄 수 있다. 현대 OS의 표준 방식이다.

대표적인 알고리즘을 살펴보면, **FCFS(First Come First Served)**는 먼저 온 순서대로 처리한다. 구현이 단순하지만 긴 작업 뒤에 짧은 작업이 오래 기다리는 **콘보이 효과(Convoy Effect)**가 있다. **라운드 로빈(Round Robin)**은 각 프로세스에 일정 시간(타임 퀀텀)씩 돌아가며 CPU를 준다. 응답성이 좋아 시분할 시스템의 기본이다. **우선순위(Priority) 스케줄링**은 중요도에 따라 CPU를 배분한다. 낮은 우선순위 프로세스가 영원히 실행 못 하는 **기아 현상(Starvation)**이 생길 수 있어, **에이징(Aging)** — 대기 시간이 길어질수록 우선순위를 높이는 기법으로 해결한다.

실시간 시스템(RTOS)에서는 **EDF(Earliest Deadline First)**, **RM(Rate Monotonic)**처럼 마감 시간을 보장하는 알고리즘이 사용된다.

---

### 3.7 OS 동기화 — 공유 자원을 안전하게 사용하다

여러 스레드가 같은 데이터를 동시에 수정하면 어떤 일이 벌어질까? `counter++` 연산은 내부적으로 "읽기 → 더하기 → 쓰기" 세 단계다. 두 스레드가 동시에 이를 수행하면 둘 다 같은 값을 읽어 각자 1을 더해 쓰기 때문에, 결과가 2가 늘어야 하는데 1만 늘어난다. 이것이 **경쟁 조건(Race Condition)**이다.

이를 막기 위해 **임계 구역(Critical Section)**을 하나의 스레드만 진입할 수 있도록 보호해야 한다. 이 메커니즘이 **동기화(Synchronization)**다.

**뮤텍스(Mutex)**는 열쇠가 하나인 자물쇠다. 잠근 스레드만 열 수 있고, 다른 스레드는 열릴 때까지 기다린다. **세마포어(Semaphore)**는 카운터 기반이다. 연결 풀이 5개라면 세마포어를 5로 초기화하고, 하나 사용할 때마다 카운터를 줄이고, 반납하면 늘린다. 0이 되면 대기한다. **모니터(Monitor)**는 공유 자원과 임계 구역을 하나로 캡슐화한 고급 추상화다. Java의 `synchronized` 키워드가 대표적이다.

뮤텍스와 세마포어의 핵심 차이는 **소유권**이다. 뮤텍스는 잠근 스레드만 해제할 수 있지만, 세마포어는 다른 스레드가 Signal을 보낼 수 있다.

---

### 3.8 데드락 — 서로 기다리다 모두 멈춰버리다

서로가 상대방이 가진 것을 기다리는 상황. 교차로에서 네 방향 차가 모두 진입해 누구도 움직이지 못하는 교착 상태가 바로 **데드락(Deadlock)**이다.

데드락이 발생하려면 네 가지 조건(**Coffman 조건**)이 동시에 충족되어야 한다. **상호 배제(Mutual Exclusion)** — 자원은 한 번에 하나의 프로세스만 사용 가능하다. **점유 대기(Hold and Wait)** — 자원을 점유한 채 다른 자원을 기다린다. **비선점(No Preemption)** — 프로세스가 자원을 강제로 빼앗길 수 없다. **순환 대기(Circular Wait)** — 프로세스들이 원형으로 서로의 자원을 기다린다.

데드락 대처법은 세 가지다. **예방(Prevention)** — Coffman 조건 중 하나를 원천 제거한다(예: 모든 자원을 한 번에 요청하도록 강제). **회피(Avoidance)** — 자원 할당 전에 데드락 가능성을 검사한다(은행가 알고리즘). **탐지·복구(Detection & Recovery)** — 데드락을 허용하되, 주기적으로 탐지하고 프로세스를 강제 종료해 복구한다.

---

### 3.9 은행가 알고리즘 — 안전한 상태를 보장하는 자원 관리

**은행가 알고리즘(Banker's Algorithm)**은 에츠허르 다익스트라가 고안한 데드락 **회피** 알고리즘이다. 은행이 모든 고객이 대출을 요청해도 파산하지 않도록 여유 자금을 유지하는 것처럼, OS가 자원을 할당하기 전에 시스템이 **안전 상태(Safe State)**를 유지하는지 검사한다.

안전 상태란 모든 프로세스가 순서대로 완료될 수 있는 순서(**안전 순서열**)가 존재하는 상태다. 새로운 자원 요청이 들어오면, 가상으로 할당해보고 안전 상태가 유지되는지 확인한다. 유지되면 실제 할당하고, 그렇지 않으면 나중에 자원이 확보될 때까지 기다리게 한다.

단점은 프로세스의 최대 자원 요구량을 미리 알아야 한다는 것이다. 현실에서는 이를 정확히 알기 어려워, 주로 교육적 목적이나 특수 임베디드 환경에서 사용된다.

---

### 3.10 기아 현상과 데드락의 차이

**기아 현상(Starvation)**과 **데드락(Deadlock)**은 모두 프로세스가 자원을 얻지 못하는 상황이지만 성격이 다르다.

데드락은 관련된 모든 프로세스가 멈춘다. 자원 자체를 얻을 가능성이 전혀 없는 상태다. 반면 기아 현상은 자원을 받을 수 있는 상황이지만, 우선순위가 낮아 계속 다른 프로세스에게 밀리는 것이다. 원칙적으로 언젠가는 받을 수 있지만 현실적으로 무한정 기다리게 된다.

기아 현상의 해결책은 **에이징(Aging)**이다. 대기 시간이 길어질수록 프로세스의 우선순위를 점차 높여 결국 CPU를 받게 하는 방법이다. "기다린 만큼 보상받는" 공정성 메커니즘이다.

---

### 3.11 우선순위 역전 — 화성 탐사선에서도 발생한 실시간 OS의 함정

1997년 화성 탐사선 Mars Pathfinder가 이상하게 자꾸 재시작되는 문제가 발생했다. 원인은 **우선순위 역전(Priority Inversion)**이었다.

우선순위 역전은 이렇게 발생한다. 낮은 우선순위 프로세스 L이 공유 자원(뮤텍스)을 잡은 상태에서, 높은 우선순위 프로세스 H가 같은 자원을 요청한다. H는 L이 자원을 놓을 때까지 기다려야 한다. 그런데 중간 우선순위 프로세스 M이 실행 가능해지면 L보다 우선순위가 높은 M이 먼저 실행된다. 결과적으로 H는 본래 자신보다 낮은 M이 실행되는 동안 계속 기다리게 된다.

해결책은 **우선순위 상속 프로토콜(PIP)**과 **우선순위 한계 프로토콜(PCP)**이다. **PIP**는 H가 자원을 기다리면 L의 우선순위를 H 수준으로 임시 상승시킨다. 이러면 L이 M보다 먼저 실행되어 자원을 빨리 반납한다. **IPCP(즉각 우선순위 한계 프로토콜)**은 자원을 획득하는 즉시 해당 자원의 최대 우선순위(Ceiling)로 상승시키는 방법으로, 자동차 AUTOSAR OS에서 널리 사용된다.

---

### 3.12 IPC — 프로세스끼리 대화하는 방법

각 프로세스는 독립된 메모리 공간을 가진다. 그렇다면 서로 데이터를 어떻게 주고받을까? **IPC(Inter-Process Communication)**가 그 방법을 제공한다.

**파이프(Pipe)**는 가장 단순한 IPC다. 단방향 데이터 흐름으로, 생산자 프로세스가 쓰면 소비자 프로세스가 읽는다. 리눅스에서 `ls | grep .txt` 같은 파이프 명령이 이를 활용한다. **공유 메모리(Shared Memory)**는 여러 프로세스가 같은 메모리 영역을 공유하는 방법으로 속도가 가장 빠르지만 동기화가 필요하다. **메시지 큐(Message Queue)**는 메시지를 큐에 넣고 꺼내는 방식으로 비동기 통신에 유리하다. **소켓(Socket)**은 네트워크를 통한 통신이지만 같은 기기 내 프로세스 간에도 사용된다.

---

### 3.13 간헐 컴퓨팅 — 전력이 불안정한 환경에서 살아남기

IoT 센서, 에너지 하비스팅 장치처럼 전력이 간헐적으로 공급되는 환경에서는 프로그램 실행 도중 갑자기 전원이 끊길 수 있다. **간헐 컴퓨팅(Intermittent Computing)**은 이런 환경에서도 올바르게 동작하는 컴퓨팅 기술이다.

핵심 과제는 **체크포인트(Checkpoint)** 설계다. 중요한 실행 상태를 비휘발성 메모리(FRAM 등)에 주기적으로 저장해두고, 전력이 다시 들어왔을 때 중단된 지점부터 재개한다. **멱등성(Idempotency)** — 동일 연산을 여러 번 수행해도 결과가 같아야 한다. 전력 중단 후 재실행 시 중복 실행으로 인한 오류를 방지한다.

간헐 컴퓨팅은 우주 탐사 장비, 의료 이식형 기기, 극한 환경 모니터링 등 배터리 교체가 불가능한 응용 분야에서 매우 중요하다.

---

## 4부. 가상화

### 4.1 스토리지 가상화 — 흩어진 저장소를 하나로

기업에는 다양한 제조사의 서로 다른 스토리지 장비들이 혼재한다. 이것들을 각각 따로 관리하는 것은 비효율적이다. **스토리지 가상화(Storage Virtualization)**는 물리적으로 분산된 이기종 스토리지 장치를 하나의 논리적 풀로 통합한다.

유형은 크게 네 가지다. **블록 레벨 가상화**는 SAN(Storage Area Network) 환경에서 고성능 블록 스토리지를 논리적으로 추상화한다. 데이터베이스처럼 성능이 중요한 시스템에 적합하다. **파일 레벨 가상화**는 NAS 환경에서 여러 파일 서버를 단일 네임스페이스로 통합한다. **오브젝트 스토리지**는 AWS S3처럼 데이터를 고유 ID로 접근하는 방식으로, 대규모 비정형 데이터(이미지, 동영상, 로그)에 최적이다. **클라우드 스토리지 가상화**는 온프레미스와 클라우드를 투명하게 통합한다.

**씬 프로비저닝(Thin Provisioning)**은 실제 사용량에 따라 저장 공간을 동적으로 할당해 스토리지 활용률을 크게 높인다. **자동 계층화(Auto-Tiering)**는 자주 접근하는 데이터는 SSD에, 덜 중요한 데이터는 HDD나 테이프로 자동 이동시켜 비용과 성능을 최적화한다.

---

### 4.2 쿠버네티스 — 컨테이너의 세계를 통치하다

도커(Docker)로 컨테이너를 하나둘 실행하는 것은 쉽다. 하지만 수백~수천 개의 컨테이너를 안정적으로 운영하려면 어떻게 해야 할까? **쿠버네티스(Kubernetes, K8s)**가 그 해답이다.

쿠버네티스의 핵심 철학은 **선언적 구성(Declarative Configuration)**이다. "이 서비스를 3개의 복제본으로 실행하라"고 YAML 파일에 원하는 상태를 선언하면, K8s가 현재 상태와 비교해 자동으로 맞춰준다. 컨테이너가 죽으면 자동 재시작(**자가 치유, Self-Healing**)하고, 트래픽이 폭증하면 자동으로 컨테이너 수를 늘린다(**HPA, Horizontal Pod Autoscaler**).

쿠버네티스의 구조를 살펴보면, **Control Plane**에는 모든 API 요청을 처리하는 kube-apiserver, 클러스터 상태를 저장하는 etcd(분산 키-값 데이터베이스), 파드를 적합한 노드에 배치하는 kube-scheduler, 원하는 상태를 유지하는 kube-controller-manager가 있다. 각 **Worker Node**에는 컨테이너를 직접 관리하는 kubelet, 네트워크를 담당하는 kube-proxy, 그리고 컨테이너 런타임이 있다.

---

### 4.3 서버리스 컴퓨팅 — 서버를 생각하지 말고 함수만 생각하라

"서버를 관리하는 데 시간을 쓰지 말고 비즈니스 로직에 집중하세요." 이 약속을 실현한 것이 **서버리스 컴퓨팅(Serverless Computing)**이다. 정확히는 서버가 없는 게 아니라, 개발자가 서버를 신경 쓰지 않아도 된다는 뜻이다.

**FaaS(Function as a Service)**가 서버리스의 핵심 모델이다. AWS Lambda, Google Cloud Functions 같은 서비스에서 함수를 업로드하면, HTTP 요청이나 메시지 큐 이벤트가 발생할 때만 자동으로 실행된다. 사용한 시간(밀리초 단위)에 대해서만 과금하므로 트래픽이 없는 시간에는 비용이 0이다.

장점은 명확하다. 서버 패치·용량 계획 등의 운영 부담이 사라지고, 트래픽 급증 시 자동으로 수천 개 함수가 동시에 실행된다. 단점도 있다. **콜드 스타트(Cold Start)** — 처음 실행 시 컨테이너를 초기화하는 데 수백ms~수초 지연이 생긴다. 실행 시간 제한(AWS Lambda는 최대 15분)이 있고, 함수 간 상태 공유가 어렵다.

서버리스는 이벤트 기반 데이터 처리(이미지 업로드 → 썸네일 생성), API 백엔드, 자동화 스크립트 같은 용도에 적합하고, 지속적인 연결이 필요하거나 긴 처리 시간이 필요한 애플리케이션에는 맞지 않는다.

## 5부. 알고리즘

### 5.1 빅오 표기법 — 알고리즘의 성능을 수식으로 표현하다

"이 코드가 얼마나 빠른가요?" 이 질문에 "꽤 빠릅니다"보다 더 정확한 답이 필요할 때 사용하는 것이 **Big-O 표기법(빅오 표기법)**이다. 입력 데이터 크기(n)가 늘어날 때 연산 횟수가 어떻게 증가하는지 수식으로 표현한다.

**O(1) — 상수 시간**: 입력 크기에 관계없이 항상 동일한 시간이 걸린다. 배열의 특정 인덱스 접근, 해시 테이블 조회가 여기에 해당한다.

**O(log n) — 로그 시간**: 이진 탐색이 대표적이다. n이 두 배가 될 때 연산이 딱 한 번 더 늘어난다. 10억 개 데이터에서 이진 탐색을 하면 약 30번만 비교하면 된다.

**O(n) — 선형 시간**: 순차 탐색처럼 데이터 하나하나를 봐야 하는 경우다.

**O(n log n) — 선형 로그 시간**: 병합 정렬, 힙 정렬이 대표적이다. 효율적인 정렬 알고리즘의 이론적 하한선이다.

**O(n²) — 이차 시간**: 이중 반복문이 있는 알고리즘(버블 정렬, 삽입 정렬)이다. n이 1,000이면 연산이 백만 번, n이 10,000이면 1억 번으로 폭발적으로 증가한다.

**O(2ⁿ) — 지수 시간**: 모든 부분집합을 탐색하는 경우다. n=50만 돼도 연산 횟수가 우주의 원자 수를 넘는다. 실용적으로 사용 불가하다.

알고리즘을 선택할 때 데이터 크기를 먼저 생각하자. n이 수십이면 O(n²)도 괜찮지만, n이 수백만이라면 반드시 O(n log n) 이하의 알고리즘을 선택해야 한다.

---

### 5.2 유전 알고리즘 — 진화의 원리로 최적해를 찾다

"이 공장의 생산 스케줄을 최적화하라" — 변수가 수백 개, 제약 조건이 수천 개인 복잡한 최적화 문제를 수학적으로 푸는 것은 사실상 불가능하다. **유전 알고리즘(Genetic Algorithm)**은 다윈의 진화론을 모방해 이런 복잡한 문제를 근사적으로 풀어낸다.

핵심 아이디어는 이렇다. 가능한 해(Solution)들을 **집단(Population)**으로 유지하고, 좋은 해들을 선택해 **교배(Crossover)**하고 일부를 **돌연변이(Mutation)**시켜 더 나은 다음 세대를 만들어 간다. 세대를 거듭할수록 집단 전체의 품질이 향상된다.

알고리즘 절차는 ①초기 집단 무작위 생성 → ②적합도(Fitness) 평가 → ③우수한 개체 선택 → ④교차로 새 해 생성 → ⑤돌연변이 적용 → ②로 반복이다. **돌연변이**는 낮은 확률(1~5%)로 해의 일부를 무작위 변경한다. 이것이 지역 최적해(Local Optimum)에 갇히지 않고 더 나은 전역 최적해(Global Optimum)를 찾을 수 있게 해주는 핵심 장치다.

유전 알고리즘은 공장 스케줄링, 신경망 구조 탐색(NAS), 외판원 문제(TSP), 게임 AI 개발 등에 활용된다.

## 6부. 인프라 아키텍처

### 6.1 HBM — AI 시대의 핵심 메모리

ChatGPT 같은 거대 언어 모델(LLM)을 학습시키는 GPU에는 일반 RAM 대신 **HBM(High Bandwidth Memory, 고대역 초고속 메모리)**이 탑재된다. 왜일까?

AI 학습은 초당 수조 번의 행렬 연산을 수행하며, 이를 위해 데이터를 극도로 빠르게 메모리에서 가져와야 한다. 일반 GDDR 메모리는 버스 폭(데이터 통로)이 좁아 한계가 있다. HBM은 DRAM 다이를 **수직으로 여러 층 쌓고(3D Stacking)**, **TSV(실리콘 관통 전극)**로 층간을 연결해 버스 폭을 수천 비트로 넓혔다.

NVIDIA H100 GPU에는 HBM3가 탑재되어 3.35TB/s의 메모리 대역폭을 제공한다. 일반 GDDR6의 576GB/s와 비교하면 약 6배 차이다. 단점은 가격이 매우 비싸다는 것이다. AI 반도체 경쟁의 핵심이 GPU가 아닌 HBM 공급에 달려있다는 말이 나올 정도로 SK하이닉스·삼성·Micron이 HBM 시장에서 치열하게 경쟁하고 있다.

---

### 6.2 CXL — CPU·GPU·메모리를 하나로 연결하는 새 표준

수천 개의 GPU가 협력해 AI 모델을 학습할 때, 각 GPU의 HBM이 부족하면 어떻게 될까? **CXL(Compute Express Link)**은 이 문제의 해결사다.

CXL은 CPU, GPU, FPGA, 메모리 장치들을 **캐시 일관성(Cache Coherency)**을 유지하면서 고속으로 연결하는 개방형 인터커넥트 표준이다. PCIe 물리 계층 위에서 동작하며, 세 가지 프로토콜로 구성된다. **CXL.io**는 기본 I/O 통신, **CXL.cache**는 가속기가 CPU 메모리를 캐싱하는 데 사용, **CXL.mem**은 CPU가 가속기나 외부 메모리에 직접 접근하는 데 사용된다.

CXL 2.0에서는 **메모리 풀링(Memory Pooling)** — 공유 메모리 풀에서 여러 호스트에 필요한 만큼 동적으로 분배하는 기능이 도입됐고, CXL 3.0에서는 **메모리 공유(Memory Sharing)** — 여러 호스트가 동일 물리 메모리를 동시에 읽고 쓰는 기능이 추가됐다. AI 분산 학습에서 서버 간 데이터를 복사 없이 직접 공유할 수 있어 병목을 크게 줄일 수 있다.

---

### 6.3 AI 데이터센터 기술 — 수만 개 GPU를 연결하다

LLM 하나를 학습시키는 데 수천~수만 개의 GPU가 필요하다. 이를 수용하는 **대규모 AI 데이터센터**는 일반 데이터센터와 차원이 다른 기술 요구사항을 가진다.

**저지연 인터커넥트**가 핵심이다. GPU 수만 개가 협력하려면 마이크로초 단위로 데이터를 주고받아야 한다. **InfiniBand**, **NVIDIA NVLink/NVSwitch**, **RDMA(Remote Direct Memory Access)** 기술이 CPU를 거치지 않고 GPU 메모리끼리 직접 통신하게 한다.

네트워크는 **Spine-Leaf 아키텍처** 또는 **Fat-Tree 토폴로지**로 구성해 수십만 GPU 간 균등한 대역폭을 제공한다. 전력 밀도가 극도로 높아지면서 **액침 냉각(서버를 절연 오일에 담그기)**이나 **직접 액체 냉각(CPU/GPU에 수냉 플레이트)**이 도입된다. PUE(Power Usage Effectiveness) 1.2 이하가 목표다.

지리적으로 분산된 데이터센터를 연결하는 **DCI(Data Center Interconnect)** 기술도 중요하다. **DWDM(파장 다중화)** 광기술로 단일 광섬유에 수십~수백 Tb/s 용량을 확보한다.

---

### 6.4 FTS와 HA — 멈추지 않는 시스템을 만드는 두 가지 길

병원의 생명유지장치, 항공기 제어 시스템, 원자력 발전소 — 이런 시스템은 단 1초도 멈춰선 안 된다. **FTS(Fault Tolerant System)**는 구성 요소가 고장 나도 서비스가 중단 없이 계속되는 시스템이다. **HA(High Availability)**는 장애 발생 시 수초~수분 안에 복구해 가동률을 99.99% 이상 유지하는 설계다.

FTS와 HA의 핵심 차이는 **무중단이냐 vs 빠른 복구냐**다. FTS는 모든 구성 요소를 완전히 이중화해 한쪽이 고장 나도 나머지가 즉시 대체한다. 비용이 매우 높다. HA는 주 시스템이 장애를 감지하면 대기 시스템으로 자동 전환(Failover)한다. 전환 시간(수초)만큼의 서비스 중단이 있다.

가용성은 `MTBF / (MTBF + MTTR)` 공식으로 계산한다. MTBF(평균 무고장 시간)가 길고 MTTR(평균 복구 시간)이 짧을수록 가용성이 높아진다. "파이브 나인(99.999%)"은 연간 약 5분 다운타임만 허용하는 수준으로 FTS 대상이고, "포 나인(99.99%)"은 연간 약 52분으로 금융·이커머스 등 HA 구성의 목표다.

### 6.5 서버 이중화 — L4 스위치와 소프트웨어 방식

서버 한 대가 죽어도 서비스가 계속되려면 **서버 이중화**가 필요하다. 두 가지 대표적 접근이 있다.

**L4 스위치 기반 이중화**는 전송 계층(TCP/UDP 포트)에서 트래픽을 분산하는 하드웨어 장치를 사용한다. 외부에는 단일 **VIP(Virtual IP)**를 노출하고, 내부적으로 여러 서버에 분산한다. 서버 상태를 주기적으로 확인(**Health Check**)해 장애 서버를 자동으로 제외한다. 로드밸런싱 알고리즘으로 **라운드 로빈(순환 배분)**, **최소 연결(가장 적게 바쁜 서버 우선)**, **IP 해시(같은 클라이언트는 같은 서버로)** 등을 사용한다.

**소프트웨어 기반 이중화**는 HAProxy, Keepalived, Nginx 같은 오픈소스로 구현한다. 클라우드·컨테이너 환경에 유연하게 적용할 수 있고 비용이 저렴하다. **Keepalived + HAProxy** 조합은 VRRP 프로토콜로 마스터 장애 시 VIP를 백업으로 자동 이관하는 방식으로 널리 사용된다.

### 6.6 멀티 리전 Active-Active — 지구 어디서나 서비스가 살아있다

서울 데이터센터에 화재가 났을 때도 서비스가 계속되려면 어떻게 해야 할까? **멀티 리전 Active-Active** 구성이 답이다. 두 개 이상의 지역에 동일한 인프라를 구축하고 모두 동시에 서비스를 제공한다.

기존 **Active-Standby**는 주 사이트가 장애를 겪으면 대기 사이트로 전환하는 방식이다. 전환에 수분이 걸리고 그동안 서비스가 중단된다(RTO: 수분). **Active-Active**는 모든 사이트가 동시에 트래픽을 처리하며, 한 사이트가 장애를 겪으면 다른 사이트가 즉시 전체 트래픽을 받는다(RTO≈0, RPO≈0).

가장 어려운 과제는 **데이터 일관성**이다. 서울과 미국에서 동시에 같은 데이터를 수정하면 충돌이 생긴다. **CockroachDB, Google Spanner** 같은 글로벌 분산 데이터베이스는 전 세계에서 강한 일관성을 유지하면서 멀티 마스터 쓰기를 지원한다. **CAP 이론**에 따르면 분산 시스템에서 일관성(C), 가용성(A), 분할 내성(P) 중 두 가지만 선택 가능하다. Active-Active는 AP를 선택해 결과적 일관성을 허용하거나, Spanner처럼 CP를 선택해 강한 일관성을 유지한다.

### 6.7 공공망 백업·복구 — 배터리 화재에도 서비스는 계속된다

2023년 SK C&C 판교 데이터센터 화재, 각종 통신망 장애 — 공공 인프라는 예상치 못한 사고에 매우 취약하다. **인프라 이중화, 백업복구, BCP(업무연속성계획)**는 이런 상황을 대비하는 세 축이다.

**인프라 관리 관점**에서는 ESS·UPS 배터리의 상태를 BMS(Battery Management System)로 실시간 모니터링하고, 온도 임계값 경보와 자동 소화 시스템을 갖춰야 한다. **백업복구 관점**에서는 **RTO(Recovery Time Objective, 목표 복구 시간)**와 **RPO(Recovery Point Objective, 목표 복구 시점)**를 명확히 정의하고, 유선·무선·위성을 통한 다중 통신 경로와 자체 발전기를 확보해야 한다. **3-2-1 백업 원칙** — 3개 복사본, 2가지 다른 매체, 1개는 오프사이트 보관 — 이 데이터 보호의 기본이다.

**이중화 관점**에서는 단일 장애점(SPOF)을 제거하는 Active-Active 구성, 최소 50km 이상 지리적으로 분산된 데이터센터, SDN 기반 자동 우회 경로가 핵심이다. 평시 훈련도 중요하다. 이론상 완벽한 재해복구 계획도 실전 훈련 없이는 위기 상황에서 작동하지 않는다.

### 6.8 블록체인 네트워크 종류 — 열린 체인 vs 닫힌 체인

"블록체인은 모두가 볼 수 있는 공개 원장이다"라고만 안다면 반쪽만 아는 것이다. 블록체인은 참여 범위와 권한에 따라 세 가지로 나뉜다.

**퍼블릭 블록체인**은 누구나 참여할 수 있고 완전히 탈중앙화된다. Bitcoin, Ethereum이 대표적이다. 투명성이 높고 검열이 불가능하지만, 합의에 시간이 걸려 처리 속도가 느리다(Bitcoin은 초당 7건).

**프라이빗 블록체인**은 허가받은 참여자만 접근하고 관리자가 존재한다. 처리 속도가 빠르고(수천 TPS) 개인정보 보호가 가능하지만, 탈중앙화의 장점이 약해진다. 기업 내부 시스템에 적합하다.

**컨소시엄 블록체인**은 여러 기관이 연합해 운영하는 절충안이다. 금융기관 컨소시엄이나 공급망 참여 기업들이 공동으로 운영한다. Hyperledger Fabric, R3 Corda가 대표적이다.

**CBDC(중앙은행 디지털 화폐)**는 퍼블릭 블록체인이 아닌 중앙은행이 통제하는 허가형 블록체인을 활용한다. 완전 익명성 대신 AML(자금세탁방지)·KYC(고객확인) 준수를 위해 신원 확인이 가능한 구조를 채택한다.

## 마무리 — 하드웨어에서 인프라까지 큰 그림으로 연결하다

지금까지 컴퓨터시스템의 여섯 가지 영역을 살펴보았다. 이 영역들은 컴퓨터의 가장 작은 단위에서 시작해 점점 큰 시스템으로 확장되는 여정이다.

**CPU와 하드웨어**는 0과 1의 연산이 이루어지는 가장 작은 세계이고, **운영체제**는 그 위에서 자원을 관리하고 추상화를 제공하는 관리자다. **가상화**는 하나의 하드웨어를 여러 개처럼 쓰게 하고, **알고리즘**은 문제를 효율적으로 푸는 방법을 제공한다. 그리고 **인프라 아키텍처**는 이 모든 것을 묶어 장애에도 멈추지 않는 대규모 시스템을 만든다.

RISC-V의 개방성이 반도체 생태계를 바꾸고, HBM과 CXL이 AI 시대의 메모리 병목을 해결하며, 쿠버네티스가 수천 개 컨테이너를 통치하고, Active-Active 멀티 리전이 지구 어디서나 서비스를 살아있게 한다. 이 모든 기술이 하나의 큰 흐름으로 연결되어 오늘날 우리가 당연하게 쓰는 디지털 서비스를 만들어 낸다.

컴퓨터시스템은 가장 빠르게 변하는 분야 중 하나다. 그러나 파이프라인, 가상 메모리, 동기화, Big-O처럼 수십 년이 지나도 변하지 않는 원리가 있다. 그 변하지 않는 원리를 이해하는 것이 이 책의 목표다.
