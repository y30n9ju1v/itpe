## 개요

서버리스 컴퓨팅은 개발자가 서버 인프라 관리 없이 함수(Function) 단위의 코드를 실행하는 클라우드 실행 모델이다. 클라우드 제공자가 인프라 프로비저닝·스케일링·가용성을 자동 관리하며, 실제 실행된 컴퓨팅 자원에 대해서만 과금(Pay-per-use)한다.

## 핵심 내용

**서버리스의 핵심 특성**
1. 인프라 추상화: 서버 관리 불필요
2. 이벤트 기반 실행: 트리거(HTTP·메시지·스케줄) 발생 시 자동 실행
3. 자동 스케일링: 요청 수에 따라 0 → N개 자동 확장
4. 사용량 기반 과금: 실행 시간·메모리 기준 과금 (유휴 비용 없음)

**서비스 모델 비교**

| 모델 | 관리 범위 | 예시 |
|------|----------|------|
| IaaS | OS·미들웨어 직접 관리 | EC2, GCE |
| PaaS | 앱 코드만 관리 | Elastic Beanstalk |
| CaaS | 컨테이너 관리 | EKS, GKE |
| FaaS (서버리스) | 함수 코드만 관리 | Lambda, Cloud Functions |

**장점**

| 장점 | 설명 |
|------|------|
| 운영 부담 감소 | 인프라 패치·관리 불필요 |
| 비용 효율 | 실행 시에만 과금 |
| 자동 확장 | 급격한 트래픽 증가 대응 |
| 빠른 개발 | 인프라 고민 없이 비즈니스 로직 집중 |

**단점/한계**

| 단점 | 설명 |
|------|------|
| 콜드 스타트 지연 | 첫 호출 시 컨테이너 초기화 지연 (수백ms~수초) |
| 실행 시간 제한 | AWS Lambda 최대 15분 |
| 상태 비저장 | 함수 간 상태 공유 어려움 (외부 저장소 필요) |
| 벤더 종속 | 클라우드별 서버리스 플랫폼 차이 |
| 디버깅 어려움 | 로컬 환경과 클라우드 환경 차이 |

## 실무 적용 예시

이미지 처리 파이프라인에서 S3 업로드 이벤트 → Lambda 트리거 → 썸네일 생성·AI 분석·메타데이터 저장을 서버리스로 구성하여 트래픽 변동에 자동 대응한다.

## 출제 포인트

- 서버리스의 정의와 FaaS 모델 위치 (IaaS-PaaS-FaaS 비교)
- 장점(비용·확장성)과 단점(콜드스타트·상태비저장) 균형 서술
- 서버리스 적합 유스케이스 vs 부적합 유스케이스
