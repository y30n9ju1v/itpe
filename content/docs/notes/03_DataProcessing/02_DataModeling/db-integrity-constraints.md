## 개요

데이터베이스 무결성(Integrity) 제약 조건은 데이터의 정확성·일관성·유효성을 보장하기 위해 데이터베이스에 저장될 수 있는 데이터의 값과 관계에 부과하는 규칙이다. 잘못된 데이터 입력을 DBMS 수준에서 차단하여 데이터 품질을 유지한다.

## 핵심 내용

### 가. 데이터베이스 무결성 제약 조건 개념

- **정의**: DB에 저장되는 데이터의 의미적 정확성과 일관성을 유지하기 위해 선언하는 규칙
- **적용 시점**: INSERT·UPDATE·DELETE 연산 시 DBMS가 자동 검사
- **위반 시 처리**: 연산 거부(Reject) 또는 보상 동작(Cascade 등)

### 나. 데이터베이스 무결성 제약 조건의 종류

| 제약 조건 | 설명 | 예시 |
|----------|------|------|
| **도메인 무결성** | 컬럼 값이 정의된 도메인(데이터 타입·범위·형식)을 만족해야 함 | 나이 컬럼: 0~150 정수 |
| **개체 무결성 (Entity)** | 기본키(PK)는 NULL이 될 수 없고 유일해야 함 | 사원번호 PK = NOT NULL + UNIQUE |
| **참조 무결성 (Referential)** | 외래키(FK) 값은 참조하는 테이블의 PK 값과 일치하거나 NULL | 주문.고객번호 → 고객.고객번호 존재해야 |
| **키 무결성** | 릴레이션 내 유일한 키가 반드시 존재해야 함 | 각 튜플을 구별하는 후보키 존재 |
| **컬럼 무결성** | NOT NULL, UNIQUE, CHECK 제약 | 이메일: UNIQUE, 성별: 'M' or 'F' |
| **사용자 정의 무결성** | 비즈니스 규칙을 TRIGGER·CHECK·ASSERTION으로 구현 | 주문 수량 > 0, 입사일 ≤ 오늘 |

### 참조 무결성 위반 시 처리 옵션

| 옵션 | ON DELETE | ON UPDATE |
|------|----------|----------|
| **RESTRICT** | 참조 레코드 있으면 삭제 거부 | 동일 |
| **CASCADE** | 참조 레코드도 함께 삭제 | FK도 함께 변경 |
| **SET NULL** | FK를 NULL로 설정 | FK를 NULL로 설정 |
| **SET DEFAULT** | FK를 기본값으로 설정 | FK를 기본값으로 설정 |

### 다. 무결성 제약 조건 생성 시 고려사항

| 고려사항 | 내용 |
|---------|------|
| **성능 영향** | 제약 조건 검사는 DML 성능 저하 유발 → 핵심 제약만 선택적 적용 |
| **비즈니스 요구사항 반영** | 현업 규칙을 정확히 파악하여 제약 조건으로 구현 |
| **NULL 허용 여부** | NULL과 공백('')의 의미 구분, FK의 NULL 허용 여부 결정 |
| **순환 참조** | A→B→A 형태 참조 시 삽입 순서·CASCADE 충돌 주의 |
| **제약 조건 지연(Deferrable)** | 트랜잭션 단위로 검사 연기 가능 여부 고려 |
| **분산 환경** | 분산 DB에서 참조 무결성 관리의 복잡성 |

## 출제 포인트

- 6가지 무결성 종류: 도메인·개체·참조·키·컬럼·사용자 정의
- 참조 무결성 위반 처리: RESTRICT·CASCADE·SET NULL·SET DEFAULT
- 생성 시 고려사항: 성능 영향, 비즈니스 규칙 반영
