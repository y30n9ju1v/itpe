## 개요

데이터 늪(Data Swamp)은 데이터 레이크(Data Lake)가 적절한 거버넌스 없이 운영되어 데이터 품질·접근성·신뢰성이 저하된 상태를 말한다. 방대한 데이터가 저장되어 있지만 어디에 무엇이 있는지 알 수 없고, 실제 활용 가치가 없는 "쓸모없는 데이터 웅덩이"가 된 것이다.

## 핵심 내용

### 데이터 레이크 vs 데이터 늪

| 구분 | 데이터 레이크 | 데이터 늪 |
|------|-------------|----------|
| 데이터 품질 | 검증된 데이터 | 미검증·중복·오염 데이터 |
| 메타데이터 | 체계적 관리 | 없음 또는 부실 |
| 접근성 | 구조화된 접근 | 어디에 뭐가 있는지 모름 |
| 신뢰성 | 데이터 출처 추적 가능 | 출처 불명 |
| 활용 가치 | 높음 | 낮음 또는 없음 |

### 데이터 늪 발생 원인

1. **메타데이터 관리 부재**: 데이터 설명·출처·갱신 이력 미기록
2. **데이터 거버넌스 부재**: 데이터 오너십·품질 기준 미정의
3. **무분별한 적재**: ETL 없이 원시 데이터만 계속 축적
4. **접근 제어 부재**: 누가 무엇을 적재했는지 추적 불가
5. **데이터 계보(Lineage) 미관리**: 변환 이력 추적 불가

### 데이터 늪 탈출 방안

| 방안 | 설명 |
|------|------|
| **데이터 카탈로그 구축** | 메타데이터 체계화 (Apache Atlas, Collibra) |
| **데이터 거버넌스 체계** | 데이터 오너 지정, 품질 기준 수립 |
| **데이터 품질 관리** | 적재 시 품질 검증 자동화 |
| **데이터 관측가능성** | 실시간 파이프라인 모니터링 |
| **데이터 계보 관리** | 데이터 흐름 추적 체계 |
| **데이터 메쉬(Data Mesh)** | 도메인별 분산 소유로 책임 명확화 |

### 데이터 레이크하우스

데이터 레이크의 유연성 + 데이터 웨어하우스의 거버넌스를 결합한 아키텍처. Delta Lake, Apache Iceberg 활용.

## 출제 포인트

- 데이터 레이크와 데이터 늪의 차이를 명확히 정의
- 발생 원인 4~5가지 서술 (메타데이터·거버넌스 부재 중심)
- 탈출 방안을 데이터 카탈로그·거버넌스·관측가능성 연계하여 서술
