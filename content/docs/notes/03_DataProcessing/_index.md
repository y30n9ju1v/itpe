# 데이터처리: 데이터가 정보가 되기까지

우리는 매 순간 엄청난 양의 데이터를 생산하고 소비한다. 스마트폰으로 결제를 할 때, 쇼핑몰에서 상품을 클릭할 때, 병원에서 검사를 받을 때마다 수많은 데이터 조각이 어딘가에 저장된다. 그런데 이 날것의 데이터가 어떻게 "이 고객은 다음 달에 이탈할 가능성이 높다"거나 "이 환자는 암 초기 증상이 의심된다"는 의미 있는 정보로 탈바꿈할까?

그 답이 **데이터처리(Data Processing)**다.

데이터처리는 크게 네 가지 여정을 거친다. 데이터를 빠르게 담을 **그릇(자료구조)**을 만들고, 데이터 사이의 관계를 체계적으로 설계하는 **데이터 모델링**을 거쳐, 데이터베이스 시스템이 수백만 건의 동시 요청을 안전하게 처리하도록 하는 **DBMS 운영**을 익히고, 마지막으로 쌓인 데이터에서 패턴과 인사이트를 뽑아내는 **데이터 분석·마이닝**으로 이어진다. 이 모든 과정을 관통하는 키워드가 바로 **품질**이다.

이 장에서는 그 여정을 처음부터 차근차근 따라가 본다.

## 1부. 자료구조: 데이터를 담는 그릇

### 1.1 연결 리스트 — 메모리를 유연하게 연결하다

> 📄 자세히 보기: [연결 리스트](01_DataStructures/linked-list)

배열을 생각해 보자. 100개의 좌석이 쭉 이어진 영화관이라고 하면, 빈 자리를 찾기는 쉽지만 중간 자리를 삽입하거나 삭제하려면 옆 사람들이 모두 자리를 옮겨야 한다. **연결 리스트(Linked List)**는 그 반대다. 각 노드가 "다음 사람은 저쪽 저기"라는 주소(포인터)를 들고 있어, 중간에 사람을 끼워 넣거나 빼낼 때 포인터만 바꾸면 된다. 삽입과 삭제가 O(1)이다.

연결 리스트는 세 가지 형태가 있다.

- **단순 연결 리스트**: 다음 노드만 가리킨다. 한 방향으로만 이동한다.
- **이중 연결 리스트**: 앞뒤 양방향 포인터를 가진다. LRU 캐시처럼 앞뒤로 자유롭게 이동해야 할 때 쓴다.
- **원형 연결 리스트**: 마지막 노드가 첫 노드를 가리켜 순환 구조를 이룬다. OS의 라운드로빈 스케줄링에서 볼 수 있다.

단, 특정 위치를 바로 찾으려면 처음부터 순서대로 훑어야 하므로 임의 접근은 O(n)이다. 자주 읽고 찾는 용도라면 배열이, 자주 삽입·삭제하는 용도라면 연결 리스트가 어울린다.

### 1.2 이진 탐색 트리 — 정렬된 계층으로 검색을 빠르게

> 📄 자세히 보기: [이진 탐색 트리](01_DataStructures/binary-search-tree)

도서관에서 책을 찾을 때 "가나다순"으로 배치된 서가를 생각해 보자. 중간 책을 펼쳐서 찾는 책보다 가나다 앞이면 왼쪽, 뒤이면 오른쪽 절반만 보면 된다. 이것이 **이진 탐색 트리(BST)**의 원리다.

BST에서 각 노드는 "왼쪽 자식 < 나 < 오른쪽 자식"이라는 규칙을 지킨다. 덕분에 중위 순회(왼쪽→나→오른쪽)를 하면 자동으로 정렬된 순서로 값이 출력된다. 탐색·삽입·삭제 모두 평균 O(log n)이다.

문제는 데이터가 순서대로 들어올 때다. 1, 2, 3, 4, 5를 차례로 넣으면 트리가 한쪽으로만 기울어진 **편향 트리**가 되어 결국 O(n) 성능으로 떨어진다. 이를 해결하기 위해 자동으로 균형을 맞추는 트리들이 등장했다.

- **AVL 트리**: 좌우 높이 차가 항상 1 이하. 엄격하게 균형을 유지하지만 회전 연산이 많다.
- **Red-Black 트리**: 약간 느슨한 균형 조건. Java의 `TreeMap`이 이 구조를 쓴다.
- **B-트리**: 모든 리프 노드가 동일한 깊이에 있다. 디스크 기반 데이터베이스 인덱스의 표준이다.

### 1.3 다차원 색인구조 — 공간과 벡터를 탐색하다

> 📄 자세히 보기: [다차원 색인구조](01_DataStructures/multidimensional-index)

"서울 강남구 반경 5km 이내 병원을 찾아라"라는 질문을 일반 B-트리로 처리하면 어떨까? 위도와 경도라는 두 축의 데이터를 하나의 인덱스에 녹이기가 어렵다. **다차원 색인구조**는 이처럼 2차원 이상의 공간 데이터를 효율적으로 검색하기 위해 고안되었다.

주요 기법은 세 가지다.

**K-D 트리**: 2차원, 3차원처럼 비교적 낮은 차원의 점 데이터를 처리한다. 각 레벨에서 교대로 다른 축을 기준으로 공간을 반으로 쪼갠다. 구현이 단순하지만, 차원이 수백 개가 넘으면 성능이 급격히 떨어진다(차원의 저주).

**R-트리**: 지도 서비스처럼 폴리곤이나 선 같은 공간 객체를 다룰 때 쓴다. 객체들을 최소 경계 직사각형(MBR)으로 묶어 B-트리처럼 계층화한다. PostGIS(지리 정보 DB), 네이버 지도 등이 이 구조를 활용한다.

**HNSW**: AI 시대의 산물이다. GPT 같은 언어 모델은 단어와 문장을 수백 차원의 벡터(임베딩)로 표현한다. "이 문장과 의미가 비슷한 문서를 찾아라"는 고차원 벡터 검색이 필요한데, 이때 계층적 그래프 구조를 활용해 O(log n)의 빠른 근사 탐색을 하는 것이 HNSW다. RAG(검색 증강 생성) 시스템의 핵심 부품으로, Pinecone·Weaviate 같은 벡터 데이터베이스에 널리 쓰인다.

### 1.4 확장성 해싱 — 데이터가 늘어나도 재구축 없이

> 📄 자세히 보기: [확장성 해싱](01_DataStructures/extendible-hashing)

해시 테이블은 키를 해시 함수에 넣어 저장 위치를 바로 계산하는 자료구조다. 일반(정적) 해시 테이블의 약점은 데이터가 넘쳐날 때 전체를 다시 구축해야 한다는 점이다. **확장성 해싱(Extendible Hashing)**은 이를 해결한다.

두 가지 핵심 개념이 있다. 버킷들을 가리키는 **디렉토리**와 실제 데이터를 저장하는 **버킷**이다. 버킷이 가득 차면 해당 버킷만 두 개로 분할한다. 분할이 많아지면 디렉토리를 두 배로 늘린다(전역 깊이 증가). 전체를 재구축하는 대신 필요한 부분만 점진적으로 확장하므로 훨씬 효율적이다.

## 2부. 데이터 모델링: 관계를 설계하다

데이터를 담을 그릇을 만들었다면, 이제 데이터 사이의 관계를 체계적으로 설계해야 한다. 데이터베이스 설계의 핵심은 **중복을 없애고 무결성을 보장**하는 것이다.

### 2.1 정규화와 제4정규형 — 중복을 제거하는 단계적 여정

> 📄 자세히 보기: [정규화와 제4정규형](02_DataModeling/fourth-normal-form)

중복 데이터는 이상 현상(Anomaly)을 낳는다. "교수A가 과목B를 가르치고 연구실C를 사용한다"는 정보를 하나의 테이블 `(교수, 과목, 연구실)`에 넣으면 어떻게 될까? 교수가 두 과목을 가르치고 두 연구실을 쓴다면, 과목과 연구실의 모든 조합(2×2=4개 행)을 저장해야 하는 불필요한 중복이 생긴다.

이것이 **다치종속(Multi-Valued Dependency)** 문제다. 교수가 과목을 다치 결정하고 동시에 연구실을 다치 결정하지만, 과목과 연구실은 서로 무관하다. **제4정규형(4NF)**은 이런 다치종속을 분리하여 `(교수, 과목)`과 `(교수, 연구실)` 두 테이블로 나누는 것이다.

정규화의 단계는 1NF → 2NF → 3NF → BCNF → **4NF** → 5NF 순으로 올라간다. 각 단계마다 제거하는 이상 유형이 다르며, 4NF는 BCNF까지의 함수적 종속 처리를 끝낸 뒤 다치종속까지 제거하는 가장 정교한 단계 중 하나다.

### 2.2 반정규화 — 성능을 위해 의도적으로 중복을

> 📄 자세히 보기: [반정규화](02_DataModeling/denormalization)

정규화가 이론적 완성도라면, **반정규화(Denormalization)**는 현실의 타협이다. 완벽하게 정규화된 테이블이라도, 조회할 때마다 수십 개의 테이블을 JOIN해야 한다면 성능은 크게 저하된다.

반정규화는 읽기 성능을 위해 의도적으로 중복을 허용한다.

- **테이블 병합**: 항상 함께 조회되는 1:1 관계 테이블을 하나로 합친다.
- **컬럼 중복**: JOIN 없이 바로 쓸 수 있도록 다른 테이블의 컬럼을 복사해 넣는다.
- **파생 컬럼**: 주문 합계처럼 계산 결과를 미리 저장해 둔다.

단, 반정규화를 적용하면 중복 데이터 불일치 위험이 생긴다. 반드시 인덱스 추가나 파티셔닝 같은 다른 방법을 먼저 검토한 뒤, 최후 수단으로 선택해야 한다. 중복 데이터의 동기화는 트리거나 애플리케이션 로직으로 보장해야 한다.

### 2.3 무결성 제약 — 나쁜 데이터의 침입을 막아라

> 📄 자세히 보기: [무결성 제약](02_DataModeling/db-integrity-constraints) · [참조 무결성](02_DataModeling/referential-integrity)

아무리 잘 설계된 데이터베이스라도, 잘못된 데이터가 들어오면 무너진다. **무결성 제약 조건**은 DBMS가 데이터의 입출입 시점마다 자동으로 검사하는 규칙이다.

여섯 가지 주요 유형이 있다.

| 유형 | 의미 |
|------|------|
| **도메인 무결성** | 나이 컬럼에 음수가 들어올 수 없다 |
| **개체 무결성** | 기본키(PK)는 NULL이 될 수 없고 중복될 수 없다 |
| **참조 무결성** | 주문의 고객번호는 고객 테이블에 실제로 존재해야 한다 |
| **키 무결성** | 각 행을 구별하는 유일한 키가 반드시 있어야 한다 |
| **컬럼 무결성** | NOT NULL, UNIQUE, CHECK 등의 제약 |
| **사용자 정의 무결성** | "주문 수량은 0보다 커야 한다" 같은 업무 규칙 |

특히 **참조 무결성**은 외래키(FK) 관계의 핵심이다. 부모 데이터를 삭제할 때 자식 데이터를 어떻게 처리할지도 정해야 한다. CASCADE(함께 삭제), RESTRICT(삭제 거부), SET NULL(NULL로 변경) 중 상황에 맞게 선택한다.

## 3부. DBMS와 분산처리: 수백만 요청을 안전하게

설계가 끝났다면 이제 실제로 데이터베이스를 운영해야 한다. 현대의 데이터베이스는 수천 명의 사용자가 동시에 접근하는 환경에서도 데이터의 일관성을 보장해야 한다.

### 3.1 트랜잭션 격리 수준 — 동시성과 일관성의 줄다리기

> 📄 자세히 보기: [트랜잭션 격리 수준](03_DBMS_Distributed/transaction-isolation-level) · [팬텀 충돌](03_DBMS_Distributed/phantom-conflict)

**트랜잭션(Transaction)**은 "함께 성공하거나 함께 실패해야 하는 작업 묶음"이다. 계좌 이체를 생각하면 쉽다. A 계좌에서 돈을 빼고 B 계좌에 넣는 두 작업은 반드시 둘 다 완료되거나, 둘 다 취소되어야 한다.

문제는 수천 개의 트랜잭션이 동시에 돌아갈 때다. 서로 다른 트랜잭션이 같은 데이터를 읽고 쓰면 세 가지 이상 현상이 생길 수 있다.

**더티 리드(Dirty Read)**: T1이 아직 커밋하지 않은 값을 T2가 읽었는데, T1이 롤백해 버린다. T2는 존재하지 않는 값을 사용한 셈이다.

**반복 불가 읽기(Non-Repeatable Read)**: T1이 같은 데이터를 두 번 조회했는데 그 사이에 T2가 값을 바꿔버려 결과가 달라진다.

**팬텀 리드(Phantom Read)**: T1이 "나이 > 20" 조건으로 조회했더니 3건이었는데, T2가 새 행을 INSERT하는 바람에 같은 조건으로 재조회하니 4건이 나온다. 유령처럼 행이 생겨나는 것이다.

ANSI SQL은 이 이상 현상을 어디까지 허용할지에 따라 네 단계 격리 수준을 정의한다.

| 격리 수준 | 더티 리드 | 반복 불가 읽기 | 팬텀 리드 |
|----------|:------:|:------:|:------:|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED (Oracle 기본) | 방지 | 발생 | 발생 |
| REPEATABLE READ (MySQL 기본) | 방지 | 방지 | 발생 |
| SERIALIZABLE | 방지 | 방지 | 방지 |

격리 수준이 높을수록 안전하지만 동시에 처리할 수 있는 트랜잭션 수가 줄어든다. 금융 시스템은 REPEATABLE READ 이상, 통계·보고용 조회는 READ COMMITTED를 주로 선택한다.

팬텀 리드를 방지하려면 SERIALIZABLE이 필요하지만, MySQL InnoDB는 REPEATABLE READ에서도 **갭 잠금(Gap Lock)**으로 범위 내 신규 삽입을 막아 팬텀을 부분적으로 방지한다.

### 3.2 정적 SQL vs 동적 SQL — 성능과 유연성의 선택

> 📄 자세히 보기: [정적 SQL vs 동적 SQL](03_DBMS_Distributed/static-dynamic-sql) · [Text2SQL](03_DBMS_Distributed/text2sql)

SQL을 실행하는 방식에는 두 가지가 있다.

**정적 SQL(Static SQL)**은 코드에 SQL이 미리 고정된 방식이다. `PreparedStatement`가 대표적이다. DBMS가 컴파일 시점에 실행 계획을 미리 만들어 두므로, 같은 쿼리를 반복 실행할 때 매우 빠르다. SQL Injection 공격도 자동으로 방어된다. OLTP(온라인 거래 처리)처럼 동일한 패턴의 쿼리가 대량으로 반복되는 환경에 최적이다.

**동적 SQL(Dynamic SQL)**은 런타임에 문자열을 조합해 SQL을 만드는 방식이다. 검색 조건이 사용자 입력에 따라 달라지는 경우에 유용하다. 단, SQL 문자열에 사용자 입력을 그대로 붙이면 SQL Injection 취약점이 생긴다. 반드시 **바인딩 변수(Bind Variable)**를 사용해 값을 파라미터로 전달해야 한다.

최근에는 **Text2SQL** 기술이 등장했다. "2024년 1분기 서울 매출 상위 10개 제품은?"이라고 자연어로 물으면 LLM이 자동으로 SQL을 생성해 주는 기술이다. 비전문가도 데이터에 직접 접근할 수 있는 **데이터 민주화**의 핵심 수단으로 주목받는다. 다만 복잡한 다중 JOIN이나 서브쿼리는 아직 오류율이 높고, 생성된 SQL의 보안 검토도 필요하다.

### 3.3 인덱스 — 찾는 속도를 수백 배 높이다

> 📄 자세히 보기: [인덱스 (클러스터드/논클러스터드)](03_DBMS_Distributed/db-index-clustered-nonclustered)

백만 개의 레코드 중 하나를 찾으려면 인덱스 없이는 전부 뒤져야 한다(Full Table Scan, O(n)). **인덱스(Index)**는 특정 컬럼 값과 레코드 위치를 미리 매핑해 둔 구조로, O(log n)으로 탐색 속도를 획기적으로 높인다.

인덱스는 크게 두 종류다.

**클러스터드 인덱스(Clustered Index)**: 실제 데이터 페이지 자체가 인덱스 키 순서로 물리적으로 정렬된다. 테이블당 하나만 만들 수 있다(보통 기본키). 날짜 범위 조회처럼 연속된 값을 찾을 때 매우 빠르다.

**논클러스터드 인덱스(Non-clustered Index)**: 별도의 인덱스 구조에 "이 값은 저 위치에 있다"는 포인터를 저장한다. 하나의 테이블에 여러 개 만들 수 있다. 자주 조회되는 비PK 컬럼에 추가한다.

인덱스도 공짜가 아니다. 인덱스를 만들수록 INSERT/UPDATE/DELETE 시 인덱스도 함께 갱신해야 하므로 쓰기 성능이 저하된다. 꼭 필요한 컬럼에만 선별적으로 만들어야 한다.

### 3.4 DB 파티셔닝 — 대용량 테이블을 나누어 정복하다

> 📄 자세히 보기: [DB 파티셔닝](03_DBMS_Distributed/db-partitioning)

수십억 건의 데이터가 하나의 테이블에 있다면 관리도, 조회도 힘들다. **파티셔닝(Partitioning)**은 테이블을 논리적으로는 하나로 보이지만 물리적으로는 여러 조각으로 나누는 기법이다.

**수평 분할(행 기준)**은 "이 달 데이터는 여기, 저 달 데이터는 저기"처럼 행을 기준으로 나눈다. 방식은 네 가지다.
- **범위 분할**: 날짜·금액 범위로 나눈다. ("1월~3월", "4월~6월")
- **해시 분할**: 해시 함수 결과로 나눈다. (user_id % 4)
- **목록 분할**: 특정 값 목록으로 나눈다. ("서울", "경기", "부산")
- **복합 분할**: 두 가지 이상을 조합한다.

**수직 분할(컬럼 기준)**은 자주 접근하는 컬럼과 드물게 접근하는 컬럼을 분리한다. "사용자 ID, 이름, 이메일"처럼 자주 쓰는 컬럼과 "프로필 이미지, 소개글"처럼 드물게 쓰는 컬럼을 나눠 I/O 효율을 높인다.

여러 물리 서버에 걸쳐 수평 분할을 적용한 것이 **샤딩(Sharding)**이다. 단, 서버를 넘나드는 조인(Cross-shard Join)은 비용이 크고 재분배(Re-sharding)가 어렵다는 단점이 있다.

### 3.5 분산 DB의 투명성 — 복잡함을 숨겨라

> 📄 자세히 보기: [분산 DB의 투명성](03_DBMS_Distributed/distributed-db-transparency)

분산 데이터베이스는 데이터가 여러 서버에 흩어져 있지만, 사용자는 그 사실을 몰라도 된다. 이것이 **투명성(Transparency)**이다. 분산 DB가 제공해야 하는 투명성은 일곱 가지다.

- **위치 투명성**: 데이터가 서울 서버에 있든 부산 서버에 있든, 같은 쿼리로 접근한다.
- **분할 투명성**: 데이터가 수평·수직 분할되어 있어도 전체를 하나로 조회한다.
- **복제 투명성**: 동일 데이터가 여러 곳에 복제되어 있어도 단일 데이터처럼 읽고 쓴다.
- **병행 투명성**: 수천 개의 트랜잭션이 동시에 돌아가도 마치 순서대로 실행되는 것처럼 보장한다.
- **장애 투명성**: 일부 서버가 죽어도 서비스가 이어진다.
- **성능 투명성**: 최적 실행 경로를 시스템이 자동으로 찾는다.
- **확장 투명성**: 서버를 추가해도 애플리케이션 코드를 바꿀 필요가 없다.

이러한 투명성은 전역 스키마 관리, 분산 쿼리 최적화기, **2단계 커밋(2PC)** 같은 기술로 구현된다.

### 3.6 오픈소스 DBMS 마이그레이션 — 상용에서 오픈소스로

> 📄 자세히 보기: [오픈소스 DBMS 마이그레이션](03_DBMS_Distributed/opensource-dbms-migration)

Oracle, MS SQL 같은 상용 DBMS의 라이선스 비용은 수억 원에 달한다. 최근 PostgreSQL, MySQL, MariaDB 같은 오픈소스 DBMS가 상용 수준의 기능과 안정성을 갖추면서, 상용에서 오픈소스로 전환하는 흐름이 공공·민간 모두에서 활발하다.

전환은 쉽지 않다. Oracle의 PL/SQL을 PostgreSQL의 PL/pgSQL로 변환해야 하고, 데이터 타입이나 내장 함수의 차이를 일일이 맞춰야 한다. 변환 도구(Ora2Pg, pgloader 등)를 활용해 스키마와 데이터를 이전하고, 실제 성능이 기존과 동등한지 철저히 테스트해야 한다. 전환 후에도 Patroni(PostgreSQL 고가용성), InnoDB Cluster(MySQL 고가용성) 같은 HA 아키텍처를 갖춰야 서비스 안정성을 보장할 수 있다.

### 3.7 트랜잭션 회복 — 장애 후 데이터를 되살리다

> 📄 자세히 보기: [트랜잭션 회복](03_DBMS_Distributed/db-transaction-recovery)

서버가 갑자기 꺼졌다. 데이터는 어떻게 복구할까? 데이터베이스의 **회복 기법**은 세 가지 방식이 있다.

**로그 기반 회복(REDO/UNDO)**: 변경 내역을 로그 파일에 기록해 두고, 장애 발생 시 이를 활용한다. WAL(Write-Ahead Logging) 원칙에 따라 데이터를 바꾸기 전에 반드시 로그를 먼저 쓴다. 복구 시에는 커밋된 트랜잭션은 REDO(재실행)하고, 미완료 트랜잭션은 UNDO(취소)한다.

**체크포인트 기반 회복**: 매번 장애 시 처음부터 모든 로그를 뒤지면 너무 오래 걸린다. 주기적으로 **체크포인트**를 찍어(메모리의 변경 내용을 디스크에 강제 저장) 체크포인트 이후의 로그만 처리하면 복구 시간을 획기적으로 단축한다.

**그림자 페이징(Shadow Paging)**: 변경 전 페이지 테이블(Shadow)을 보존해 두고, 커밋 시 원자적으로 전환한다. UNDO 로그가 필요 없어 회복이 단순하지만, 페이지 테이블 관리 오버헤드와 단편화 문제가 있다.

## 4부. 데이터 마이닝: 데이터에서 가치를 캐다

수많은 데이터를 저장하고 관리하는 궁극적인 목적은 결국 그 안에서 의미를 찾는 것이다. **데이터 마이닝(Data Mining)**은 대량의 데이터에서 패턴과 지식을 자동으로 발견하는 학문이다.

### 4.1 연관 규칙 분석 — 장바구니의 비밀

> 📄 자세히 보기: [연관 규칙 분석](04_DataMining/association-rule-analysis)

마트 구매 데이터를 분석했더니 "기저귀를 사는 사람은 맥주도 함께 산다"는 패턴이 발견되었다. 이것이 **연관 규칙 분석**이다. 항목 A를 샀을 때 항목 B도 살 가능성을 수치로 표현한다.

세 가지 지표가 핵심이다.

**지지도(Support)**: "전체 거래 중 A와 B가 함께 등장하는 비율". 규칙이 얼마나 흔한지 본다.

**신뢰도(Confidence)**: "A를 샀을 때 B도 살 조건부 확률 P(B|A)". 규칙이 얼마나 정확한지 본다. 단, B 자체가 원래 자주 팔리는 물건이라면 신뢰도가 높게 나올 수 있어 단독으로는 부족하다.

**향상도(Lift)**: "A가 B 구매에 실제로 기여하는 정도". 1보다 크면 양의 연관, 1이면 독립, 1보다 작으면 음의 연관이다. 신뢰도의 편향을 보정해 준다.

이 세 지표를 이용해 최소 지지도·신뢰도 조건을 만족하는 빈발 패턴을 찾는 알고리즘이 **Apriori**다. "빈발하지 않는 항목 집합의 상위 집합도 빈발하지 않는다"는 **선험적 원리**로 탐색 공간을 대폭 줄인다.

### 4.2 군집 분석과 실루엣 계수 — 비슷한 것끼리 모으기

> 📄 자세히 보기: [실루엣 계수](04_DataMining/silhouette-coefficient)

레이블(정답) 없이 데이터를 비슷한 것끼리 그룹으로 묶는 것이 **군집화(Clustering)**다. "이 고객들은 어떤 유형으로 나눌 수 있을까?"를 답하는 데 쓴다.

K-Means 같은 군집화 알고리즘에서는 군집 수 K를 직접 지정해야 한다. K를 잘못 잡으면 의미 없는 군집이 나온다. **실루엣 계수(Silhouette Coefficient)**는 군집화 품질을 -1에서 1 사이의 값으로 평가한다.

각 데이터 포인트 i에 대해:
- a(i): 같은 군집 내 다른 점들과의 평균 거리 (작을수록 응집도가 높다)
- b(i): 가장 가까운 이웃 군집까지의 평균 거리 (클수록 분리도가 높다)

실루엣 계수 s(i) = (b(i) - a(i)) / max(a(i), b(i))

계수가 0.7 이상이면 좋은 군집, 0 이하면 잘못 분류된 것이다. K값을 바꿔 가며 실루엣 계수가 최대인 K를 선택하면 최적 군집 수를 찾을 수 있다.

### 4.3 SOM — 고차원을 2차원 지도로 펼치다

> 📄 자세히 보기: [SOM](04_DataMining/self-organizing-map)

**SOM(Self-Organizing Map, 자기조직화 지도)**은 수십 차원의 복잡한 데이터를 2차원 격자 지도 위에 펼쳐 보여 주는 비지도 신경망이다. 위상 구조를 보존한다는 것이 핵심이다. 즉, 원래 데이터 공간에서 서로 비슷한 데이터는 지도에서도 인접한 위치에 놓인다.

학습 과정은 네 단계다. ① 격자의 가중치를 초기화한다. ② 입력 데이터를 넣어 가장 가까운 격자 노드(BMU, 최우수 매칭 단위)를 찾는다. ③ BMU와 그 이웃 노드들의 가중치를 입력에 가까워지도록 갱신한다. ④ 학습률과 이웃 범위를 줄여 가며 반복한다.

SOM은 레이블 없이 학습하므로 일반 신경망(지도 학습)과 달리 역전파가 없다. 고객 세분화, 이상 탐지, 텍스트 주제 시각화 등에 활용된다.

### 4.4 벡터 데이터베이스와 HNSW/IVF — AI 검색의 엔진

> 📄 자세히 보기: [벡터 데이터베이스와 HNSW/IVF](04_DataMining/vector-database-hnsw-ivf)

ChatGPT 같은 AI에게 질문을 던지면, AI는 내부 지식 외에 관련 문서를 검색해서 답한다(RAG, 검색 증강 생성). 이때 "의미적으로 비슷한 문서"를 찾는 데 **벡터 데이터베이스**가 쓰인다.

텍스트를 수백 차원의 숫자 벡터(임베딩)로 변환한 후, 유사도가 높은 벡터를 빠르게 찾는 것이 벡터 DB의 역할이다. 수백만 개의 벡터 중 정확히 가장 가까운 것을 찾으려면 모두 비교해야 하므로(O(n)), **근사 최근접 이웃(ANN)** 알고리즘으로 속도와 정확도를 교환한다.

**HNSW(Hierarchical Navigable Small World)**: 소셜 네트워크의 "여섯 단계 분리" 이론처럼, 계층적 그래프를 이용해 상위 레이어에서 빠르게 방향을 잡고 하위 레이어에서 정밀하게 탐색한다. 빠르고 삽입이 쉽지만 메모리를 많이 쓴다. 업계 표준으로 가장 널리 쓰인다.

**IVF(Inverted File Index)**: K-Means로 벡터 공간을 미리 클러스터링해 두고, 쿼리가 들어오면 가까운 클러스터 몇 개만 탐색한다. HNSW보다 메모리 효율이 좋지만 사전 학습이 필요하고 실시간 삽입이 어렵다.

### 4.5 과적합과 과소적합 — 배움의 균형점을 찾아라

> 📄 자세히 보기: [과적합과 과소적합](04_DataMining/overfitting-underfitting)

머신러닝 모델을 학습할 때 가장 흔한 두 가지 실패가 있다.

**과적합(Overfitting)**: 모델이 훈련 데이터를 너무 열심히 외워서, 훈련 데이터에는 완벽하지만 새로운 데이터에는 형편없는 성능을 보인다. 마치 교과서 문제만 달달 외운 학생이 응용 문제 앞에서 무너지는 것과 같다. 모델이 너무 복잡하거나 훈련 데이터가 너무 적을 때 발생한다.

해결책: L1/L2 규제(가중치 크기 제한), 드롭아웃(뉴런 무작위 비활성화), 조기 종료(Early Stopping), 데이터 증강, 앙상블.

**과소적합(Underfitting)**: 모델이 너무 단순해서 훈련 데이터의 패턴조차 제대로 학습하지 못한다. 훈련 오차와 검증 오차 모두 높다.

해결책: 더 복잡한 모델 사용, 피처 엔지니어링, 규제 완화.

이 둘은 **편향-분산 트레이드오프(Bias-Variance Tradeoff)**로 설명된다. 복잡한 모델은 편향은 낮지만 분산이 높고(과적합), 단순한 모델은 편향은 높지만 분산은 낮다(과소적합). 최적 복잡도를 찾는 것이 모델링의 핵심 과제다.

### 4.6 아웃라이어 탐지 — 데이터의 이상 신호를 포착하라

> 📄 자세히 보기: [아웃라이어 탐지](04_DataMining/outlier-detection)

**아웃라이어(Outlier, 이상치)**는 다른 데이터와 현저히 다른 관측값이다. 신용카드 사기 탐지, 공장 설비 이상 감지, 의료 진단 등에서 아웃라이어 자체가 분석의 목표가 된다.

노이즈(Noise, 측정 오차)와 혼동하면 안 된다. 노이즈는 제거 대상이지만, 아웃라이어는 오히려 보존하고 집중 분석해야 할 의미 있는 신호일 수 있다.

아웃라이어는 세 가지 유형으로 분류된다.
- **전역 이상치**: 전체 데이터셋에서 명백히 이상한 값 (사람 체온 데이터에서 60°C)
- **맥락 이상치**: 특정 맥락에서만 이상한 값 (겨울철 기온 데이터에서 30°C)
- **군집 이상치**: 개별로는 정상이지만 집합적으로 이상한 패턴 (심전도의 비정상 파형)

탐지 기법으로는 Z-score/IQR 같은 통계 기반, LOF(Local Outlier Factor) 같은 밀도 기반, Isolation Forest·AutoEncoder 같은 모델 기반이 있다.

### 4.7 혼동행렬과 성능 평가 — 모델이 얼마나 잘 맞히는가

> 📄 자세히 보기: [혼동행렬과 성능 평가](04_DataMining/confusion-matrix-metrics) · [ML 성능 지표](04_DataMining/ml-performance-metrics) · [PR/ROC 곡선](04_DataMining/pr-roc-curve)

분류 모델을 만들었다면 "얼마나 잘 맞히는가"를 평가해야 한다. **혼동행렬(Confusion Matrix)**은 예측 결과를 네 칸으로 정리한다.

|  | 예측 양성 | 예측 음성 |
|--|---------|---------|
| **실제 양성** | TP (맞게 양성 판정) | FN (놓침, **2종 오류**) |
| **실제 음성** | FP (잘못 양성 판정, **1종 오류**) | TN (맞게 음성 판정) |

여기서 중요한 성능 지표 네 가지가 나온다.

- **정확도(Accuracy)**: (TP+TN)/전체. 쉽지만 클래스 불균형 데이터에서는 함정이 있다. 암 환자가 1%인 데이터에서 "모두 정상"이라고 해도 정확도 99%가 나온다.
- **정밀도(Precision)**: TP/(TP+FP). 양성이라고 판정한 것 중 실제 양성 비율. 스팸 필터처럼 FP를 줄이고 싶을 때.
- **재현율(Recall)**: TP/(TP+FN). 실제 양성 중 찾아낸 비율. 암 진단처럼 FN을 줄이고 싶을 때.
- **F1-score**: 정밀도와 재현율의 조화평균. 불균형 데이터에서 종합 성능 평가.

정밀도와 재현율은 임계값을 올리면 정밀도는 올라가고 재현율은 내려가는 **트레이드오프** 관계다.

**ROC 곡선**은 임계값 변화에 따른 재현율(TPR)과 거짓 양성률(FPR)의 관계를 시각화한다. **AUC**(곡선 아래 면적)가 1에 가까울수록 좋은 모델이다. 단, 클래스가 크게 불균형할 때는 ROC보다 **PR 곡선**(정밀도-재현율 곡선)이 더 정직한 평가를 준다. ROC는 TN이 많으면 AUC가 과대평가될 수 있기 때문이다.

### 4.8 데이터 어노테이션 — AI 학습의 원료를 만들다

> 📄 자세히 보기: [데이터 어노테이션](04_DataMining/data-annotation)

AI 모델은 레이블(정답)이 붙은 데이터로 학습한다. **데이터 어노테이션(Annotation)**은 이미지, 텍스트, 음성 등 원시 데이터에 레이블·주석을 붙이는 작업이다.

이미지 어노테이션만 해도 다양한 형태가 있다.

| 유형 | 설명 | 활용 |
|------|------|------|
| 이미지 분류 | 이미지 전체에 레이블 | 개/고양이 분류 |
| 바운딩 박스 | 객체 주위에 직사각형 | YOLO 객체 탐지 |
| 폴리곤 | 객체 외곽선을 다각형으로 | 자율주행 도로 구분 |
| 시맨틱 분할 | 픽셀별 클래스 지정 | 의료 영상 분석 |
| 키포인트 | 관절 등 특정 점 표시 | 자세 추정 |

어노테이션 품질을 보증하기 위해 여러 어노테이터의 일치도를 측정하는 **IAA(Inter-Annotator Agreement)**, 정답이 담긴 골든 셋으로 작업자를 검증하는 방법, AI가 불확실한 샘플만 선별해 어노테이션 비용을 줄이는 **액티브 러닝** 등을 활용한다.

## 5부. 데이터 품질: 신뢰할 수 있는 데이터를 만들다

"Garbage In, Garbage Out." 아무리 뛰어난 분석 기법이라도 입력 데이터가 쓰레기라면 결과도 쓰레기다.

### 5.1 AI 학습용 데이터 품질관리

> 📄 자세히 보기: [AI 학습용 데이터 품질관리](05_DataQuality/ai-training-data-quality)

AI 모델의 성능은 데이터 품질이 결정한다. AI 학습 데이터는 일반 데이터베이스와 다른 여섯 가지 특성을 가진다.

- **대규모성**: 수백만~수억 건이 필요하다.
- **다양성**: 다양한 상황을 포함해야 일반화된 모델이 나온다.
- **대표성**: 실제 배포 환경의 분포를 반영해야 한다.
- **레이블 정확성**: 어노테이션 오류가 모델 품질을 직접 깎는다.
- **균형성**: 특정 클래스가 너무 적으면 그 클래스를 잘 학습하지 못한다.
- **시의성**: 시간이 지나면 데이터 분포가 바뀐다(개념 드리프트). 최신성을 유지해야 한다.

품질관리 활동은 수집 → 전처리 → 어노테이션 → 검증 → 관리 단계로 이루어진다. 각 단계에서 결측값 처리, 이상치 제거, 다중 어노테이터 검토, 데이터 버전 관리 등을 수행한다.

### 5.2 데이터 관측가능성 — 파이프라인을 실시간으로 감시하다

> 📄 자세히 보기: [데이터 관측가능성](05_DataQuality/data-observability)

데이터 파이프라인은 생각보다 자주 고장난다. 스키마가 바뀌거나, 데이터가 갑자기 오지 않거나, 이상한 값이 섞여 들어오는 **데이터 다운타임**이 발생한다. 이로 인해 잘못된 대시보드를 보고 의사결정을 내릴 수 있다.

**데이터 관측가능성(Data Observability)**은 소프트웨어의 관측가능성(로그·메트릭·트레이스) 개념을 데이터 파이프라인에 적용한 것이다. 다섯 가지 핵심 지표를 실시간으로 모니터링한다.

| 지표 | 감시 내용 |
|------|---------|
| **신선도(Freshness)** | 데이터가 마지막으로 갱신된 시점 |
| **분포(Distribution)** | 값의 범위·분포가 정상인가 |
| **볼륨(Volume)** | 예상한 양의 데이터가 들어오는가 |
| **스키마(Schema)** | 컬럼 구조가 바뀌지 않았는가 |
| **계보(Lineage)** | 데이터가 어디서 어떻게 흘러왔는가 |

기존 데이터 품질 관리가 "저장된 데이터를 정기 점검"하는 것이라면, 데이터 관측가능성은 "파이프라인 전체를 실시간 감시"하는 것이다. Monte Carlo Data, Great Expectations, dbt 같은 도구가 이 역할을 한다.

### 5.3 데이터 늪 — 레이크가 늪으로 변하는 경위

> 📄 자세히 보기: [데이터 늪](05_DataQuality/data-swamp)

**데이터 레이크(Data Lake)**는 다양한 형태의 원시 데이터를 한 곳에 저장하는 저장소다. 저장 비용이 저렴하고 어떤 형태의 데이터도 받아들인다는 장점이 있다.

그런데 여기에 거버넌스 없이 데이터를 무차별적으로 쌓아만 두면 어떻게 될까? "어디에 무엇이 있는지 모른다", "이 데이터가 정확한지 알 수 없다", "누가 언제 넣었는지 알 방법이 없다"는 상태가 된다. 이것이 **데이터 늪(Data Swamp)**이다. 방대한 데이터가 있지만 실제로 쓸 수 있는 데이터가 없는 역설적 상태다.

탈출하려면 **데이터 카탈로그**(메타데이터를 체계적으로 관리), **데이터 거버넌스**(데이터 오너 지정, 품질 기준 수립), **데이터 계보 관리**, **데이터 관측가능성** 도구를 갖춰야 한다. 최근에는 데이터 레이크의 유연성과 데이터 웨어하우스의 거버넌스를 결합한 **데이터 레이크하우스(Lakehouse)** 아키텍처(Delta Lake, Apache Iceberg)가 주목받고 있다.

### 5.4 데이터 가치평가 — 데이터는 얼마짜리인가

> 📄 자세히 보기: [데이터 가치평가](05_DataQuality/data-valuation)

데이터도 자산이다. 그렇다면 데이터의 가치는 어떻게 매길까? **데이터 가치평가**는 데이터 거래·투자·M&A 시 데이터 자산의 경제적 가치를 산정하는 체계다.

데이터는 독특한 재화다. 여러 사람이 동시에 써도 줄어들지 않고(비경합성), 복제 비용이 거의 0이며, 데이터가 많아질수록 가치가 기하급수적으로 커진다(네트워크 효과). 하지만 오래될수록 가치가 떨어진다(시간 가치 감소).

가치평가 모델은 네 가지다.
- **비용 접근법**: 데이터를 만드는 데 얼마나 들었는가.
- **시장 접근법**: 유사한 데이터가 시장에서 얼마에 팔리는가.
- **수익 접근법**: 이 데이터로 미래에 얼마나 벌 수 있는가 (DCF 방식).
- **옵션 가치법**: 아직 실현되지 않은 미래 활용 가능성을 실물옵션으로 평가한다.

한국에서는 데이터산업진흥법(2022)을 계기로 가치평가 방법론이 제도화되어 데이터 거래소 활성화의 기반이 마련됐다.

### 5.5 공공데이터 품질인증 — 국민이 쓰는 데이터의 신뢰를 보증하다

> 📄 자세히 보기: [공공데이터 품질인증](05_DataQuality/public-data-quality-certification)

행정안전부(한국지능정보사회진흥원 위탁)는 공공기관이 보유한 데이터의 품질 수준을 심사하고 인증하는 **공공데이터 품질인증 제도**를 운영한다. 공공데이터법 제22조의2에 근거하며, 인증 유효 기간은 2년이다.

품질 심사는 여섯 개 영역을 점검한다.

| 영역 | 내용 |
|------|------|
| **정확성** | 실제 사실과 데이터 값이 일치하는가 |
| **완전성** | 필수 항목이 빠짐없이 기록되었는가 |
| **일관성** | 다른 데이터셋과 모순이 없는가 |
| **유효성** | 정의된 형식·범위에 맞는가 |
| **유일성** | 중복 데이터가 없는가 |
| **적시성** | 최신 상태를 유지하는가 |

공공데이터포털(data.go.kr)에서 CSV, JSON, XML 형태로 데이터를 개방하고 REST API로도 제공한다.

## 6부. 빅데이터 분석: 방대한 데이터를 인사이트로

### 6.1 빅데이터 시각화 — 숫자를 그림으로

> 📄 자세히 보기: [빅데이터 시각화](06_BigDataAnalytics/bigdata-visualization)

수십억 건의 데이터를 숫자로만 보면 아무도 이해하지 못한다. **빅데이터 시각화**는 데이터를 차트·그래프·지도·대시보드로 변환해 인간이 직관적으로 이해하도록 돕는다.

시각화는 목적에 따라 두 가지로 나뉜다.
- **탐색적 시각화**: 분석가가 데이터에서 패턴을 발견하는 단계. "이 데이터에서 뭔가 보이지 않나?"
- **설명적 시각화**: 발견한 인사이트를 의사결정자에게 전달하는 단계. "이래서 이런 결론입니다."

데이터 유형에 따라 적합한 차트가 다르다.

| 데이터 유형 | 적합한 차트 |
|------------|-----------|
| 범주형 비교 | 막대 차트 |
| 시계열 추이 | 선 차트 |
| 두 변수 상관관계 | 산점도 |
| 공간 데이터 | 지리 시각화(지도) |
| 계층적 비율 | 트리맵 |
| 관계·연결 구조 | 네트워크 그래프 |
| 텍스트 빈도 | 워드 클라우드 |

도구 면에서는 비즈니스 인텔리전스(Tableau, Power BI), Python 라이브러리(Matplotlib, Seaborn), 실시간 스트리밍 대시보드(Grafana, Kibana) 등이 있다. Grafana와 Kibana는 실시간 로그와 메트릭을 모니터링하는 현장에서 특히 많이 쓰인다.

빅데이터는 시각화에서도 특수한 도전이 있다. 수십억 건을 한 번에 렌더링할 수 없으므로 집계·샘플링·GPU 가속을 활용하고, 실시간 스트리밍 데이터는 Lambda/Kappa 아키텍처로 처리해 시각화한다.

## 마치며

데이터처리의 여정을 돌아보자.

자료구조로 데이터를 빠르게 담고 찾는 그릇을 만들었다. 데이터 모델링으로 중복 없이 일관된 관계를 설계했다. DBMS를 통해 수백만 동시 요청에도 데이터를 안전하게 관리하는 법을 익혔다. 데이터 마이닝으로 쌓인 데이터에서 패턴과 가치를 캐냈다. 그리고 데이터 품질과 빅데이터 시각화로 그 모든 결과를 신뢰할 수 있고 이해할 수 있는 형태로 만들었다.

데이터는 그 자체로 의미가 없다. 올바르게 수집되고, 체계적으로 저장되고, 안전하게 관리되고, 깊이 분석되고, 명확하게 시각화될 때 비로소 의사결정을 이끄는 **정보**가 된다. 이 장에서 다룬 모든 개념은 그 변환을 가능하게 하는 기술과 원칙들이다.
