## 개요

실행 중인 애플리케이션의 무중단 배포와 품질 보증을 위해 다양한 배포·테스트 전략이 사용된다. DevOps·CI/CD 파이프라인과 결합하여 서비스 중단 없이 빠르고 안전한 릴리스를 실현한다.

## 핵심 내용

### 배포 전략

**1. 블루/그린 배포 (Blue/Green Deployment)**

```
[사용자] → [로드밸런서]
                ├─ Blue (현재 운영 환경 v1.0)
                └─ Green (신규 환경 v2.0) ← 전환
```

| 항목 | 내용 |
|------|------|
| **원리** | 동일 환경 2개 유지, 트래픽을 한 번에 전환 |
| **장점** | 즉각적 롤백, 제로 다운타임 |
| **단점** | 인프라 비용 2배, 데이터베이스 스키마 호환성 필요 |
| **적합** | 빠른 롤백이 중요한 금융·커머스 시스템 |

**2. 카나리 배포 (Canary Deployment)**

```
[사용자 100%] → [v1.0 (90%)] + [v2.0 (10%)] → 검증 후 점진 확대
```

| 항목 | 내용 |
|------|------|
| **원리** | 신버전을 소수 사용자에게 먼저 배포, 점진적 확대 |
| **장점** | 실 사용자 테스트, 문제 발생 시 영향 최소화 |
| **단점** | 두 버전 동시 운영 복잡성, 모니터링 필수 |
| **적합** | 사용자 경험 변화가 큰 기능 출시 |

**3. 롤링 배포 (Rolling Deployment)**

```
인스턴스 1: v1 → v2
인스턴스 2: v1 → v2 (순차적 교체)
인스턴스 3: v1 → v2
```

| 항목 | 내용 |
|------|------|
| **원리** | 인스턴스를 순차적으로 교체 |
| **장점** | 추가 인프라 불필요, 점진적 적용 |
| **단점** | 버전 혼재 기간 발생, 롤백 복잡 |

**4. 피처 플래그 (Feature Flag)**
- 코드 배포와 기능 활성화를 분리
- 특정 사용자·그룹에게만 기능 활성화 가능
- A/B 테스트와 결합하여 활용

### 테스트 전략

**배포 파이프라인 내 테스트 계층**

```
Unit Test → Integration Test → Smoke Test → 카나리 모니터링 → 전체 배포
```

**테스트 유형별 전략**

| 테스트 | 시점 | 내용 |
|--------|------|------|
| **스모크 테스트** | 배포 직후 | 핵심 기능 작동 여부 빠른 확인 |
| **회귀 테스트** | 배포 전 | 기존 기능 깨짐 여부 자동 검증 |
| **카오스 테스트** | 운영 중 | 의도적 장애 주입으로 복원력 검증 |
| **A/B 테스트** | 카나리 배포 시 | 비즈니스 지표 기반 버전 비교 |

**블루/그린 테스트 전략**
1. Green 환경에 배포 완료
2. Smoke Test + 통합 테스트 자동 실행
3. 이상 없으면 로드밸런서 전환
4. 모니터링 후 Blue 환경 대기 유지 → 문제 시 즉시 롤백

## 출제 포인트

- 블루/그린·카나리·롤링 배포 방식 비교 (원리·장단점·적합 상황)
- CI/CD 파이프라인에서의 테스트 계층 구조
- 피처 플래그를 통한 배포와 기능 분리
- 카오스 엔지니어링의 목적
