## 개요

데이터베이스 인덱스는 검색 속도를 향상시키기 위해 특정 컬럼 값과 레코드 위치를 매핑한 데이터 구조다. 클러스터드 인덱스는 실제 데이터를 인덱스 키 순으로 물리적으로 정렬·저장하며, 논클러스터드 인덱스는 별도 인덱스 구조에 포인터를 저장한다.

## 핵심 내용

**인덱스 기본 구조**
- 대부분의 DBMS는 B+Tree 구조 사용
- 리프 노드에 실제 데이터(클러스터드) 또는 포인터(논클러스터드) 저장
- 인덱스가 없으면 Full Table Scan(O(n)), 인덱스 사용 시 O(log n)

**클러스터드 인덱스 vs 논클러스터드 인덱스**

| 구분 | 클러스터드 인덱스 | 논클러스터드 인덱스 |
|------|-----------------|-------------------|
| 물리적 정렬 | 데이터 페이지가 인덱스 키 순으로 정렬 | 별도 인덱스 구조, 데이터 순서와 무관 |
| 테이블당 개수 | 1개만 생성 가능 | 여러 개 생성 가능 |
| 리프 노드 | 실제 데이터 페이지 | 데이터 페이지 포인터(RID) |
| 범위 검색 | 빠름 (물리적 연속성) | 상대적으로 느림 |
| 저장 공간 | 추가 공간 불필요 | 별도 인덱스 저장 공간 필요 |
| INSERT 성능 | 물리적 재정렬로 느릴 수 있음 | 클러스터드보다 INSERT 빠름 |
| 주 사용 | PK (기본키), 범위 쿼리 컬럼 | 자주 조회되는 비PK 컬럼 |

**인덱스 설계 고려사항**
1. 선택도(Selectivity): 고유 값이 많을수록 인덱스 효율↑
2. 복합 인덱스: 컬럼 순서가 중요 (최좌측 접두사 규칙)
3. 인덱스 과다 생성: DML 성능 저하
4. 커버링 인덱스: 인덱스만으로 쿼리 결과 반환 (테이블 접근 불필요)

## 실무 적용 예시

주문 테이블에서 주문일자 기준 범위 조회가 빈번한 경우, 주문일자 컬럼에 클러스터드 인덱스를 설정하여 날짜 범위 검색 성능을 극대화하고, 고객ID·상품코드 등 조회 조건 컬럼에는 논클러스터드 인덱스를 추가로 설정한다.
