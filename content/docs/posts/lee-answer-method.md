# 이지은 기술사 답안 작성법 — 교시별 완전 해부

> 공감반 11기 셀프테스트 모범답안 분석을 바탕으로 정리한  
> 정보처리기술사 답안의 구조·형식·흐름 가이드

---

## 0. 답안의 물리적 레이아웃 (공통)

모든 교시에 걸쳐 **동일한 레이아웃 원칙**이 적용된다.

```
┌──┬───────────────────────────────────────────┐
│문│N)  문제 제목 / 키워드                       │
│답│)                                           │
├──┼───────────────────────────────────────────┤
│  │  1. 대주제                                  │
│  │     정 의  • 핵심 개념 서술                  │
│  │           - 보조 서술                       │
│  │     [구성도 / 분류 박스 / 표]                │
│  │     - 한 줄 결론                             │
│  │                                             │
│  │  2. 대주제                                  │
│  │    1) 소주제                                 │
│  │    2) 소주제 (표: 구분 | 항목 | 설명)         │
│  │     - 한 줄 결론                             │
│  │                                             │
│  │  3. 대주제   "끝"                             │
└──┴───────────────────────────────────────────┘
```

### 번호 체계

| 레벨 | 표기 | 예시 |
|------|------|------|
| 대단락 | `1.` `2.` `3.` | `1. MSA 구조의 통신, API Gateway` |
| 중단락 | `1)` `2)` | `1) 연관 규칙 특징` |
| 소단락 | `①` `②` `③` | `① 빈발집합 생성` |
| 나열 불릿 | `•` | `• Broker  • Adapter` |
| 설명·결론 | `-` | `- Cloud 간 보안 중개` |

> **원칙**: `•`는 항목 나열, `-`는 설명문·결론문. 절대 혼용하지 않는다.

## 1. 1교시형 — 단답형 13문항

### 1-1. 전체 구조

1교시는 문항당 배점이 낮고 수가 많다. 문제마다 **3개 대주제**로 끊어 작성한다.

```
1. [개요 / 정의 / 배경]
2. [핵심 내용 — 구성·절차·비교·분류]
3. [확장 — 활용·트렌드·비교·심화]   "끝"
```

### 1-2. 1단락 — 정의 (정 의)

- 첫 번째 대주제는 **반드시 정의(정 의)부터** 시작한다.
- 정의 문장은 **2줄 이내**로 압축. 표 안에 쓸 경우 정의 + 특징 2열로 확장 가능.
- 핵심 키워드는 `"따옴표"`로 강조한다.

**정의 작성 공식:**

```
정 의  • [핵심 기술]을 [수식어]로 [동작]하는 [상위 개념]
       - [핵심 기능 A] / [핵심 기능 B] 등을 [목적]하는 구조
```

**예시 — API Gateway:**
> MSA 등에서 서비스 간 모든 내부-외부 간 통신 시 G/W를 사용하는 구조

**예시 — EVM:**
> PV, EV 등 "금전" 기반 지도를 바탕으로 Project의 일정/비용 현황 분석 기법

정의 직후에는 반드시 **구성도 또는 분류 박스**를 삽입한다.

**원본 — API Gateway / Service Mesh 답안 (2주차 셀테, 문1):**

### 1-3. 2단락 — 핵심 내용 (구성도 + 표)

#### 구성도 유형

**① 흐름/발전도** — 기술의 세대별 발전 표현

```
[1세대] → [2세대] → [3세대] → [4세대] → [현재]
  •키1      •키2      •키3      •키4      •키5
```

예: `P2P NW → EAI 구조 → ESB P/F → API G/W → Service Mesh`

**원본 — 이기종 시스템 통합 발전 흐름도 (2주차 셀테, 문1 대주제3):**

![발전 내역 흐름도](/itpe/resources/references/이지은%20기술사%20모범답안/공감반%2011기/2주차/KakaoTalk_20230316_225505098_01.jpg)

**② 계층 구조도** — 상위/하위 관계 표현

```
         [Controller / 상위 개념]        ← Control Plane
              /           \
         [좌측]   통신   [우측]           ← Data Plane
           |                |
        [서비스]          [서비스]
```

**③ 분류 박스** — 개념을 2~4개 그룹으로 나눌 때

```
[그룹 A]              [그룹 B]
• 항목 a1             • 항목 b1
• 항목 a2             • 항목 b2
```

**④ 좌우 분리도** — 두 개념의 관계 표현

```
[개념 A]      비교/연계      [개념 B]
• 특징 1          ↔         • 특징 1
• 특징 2                    • 특징 2
```

#### 표 유형

**비교 대조표** (가장 많이 쓰임):

| 구분 | [개념 A] | [개념 B] | [개념 C] |
|------|----------|----------|----------|
| 방향 | • 하향식  | • 상향식  | • 상/하향 |
| 방식 | • 연역적  | • 귀납적  | • 복합   |
| 수행 목적 | • 원인 분석 | • 결과 분석 | • 변이 분석 |

| 구분 | 항목 | 설 명 |
|------|------|-------|
| 탐색적 기법 | ① 관찰, 참여 | - 사용 패턴 등 분석 |

### 1-4. 3단락 — 확장 (트렌드·비교·심화)

| 상황 | 3단락 유형 |
|------|------------|
| 문제점·위협이 있는 개념 | 해결방안·대응방안 무조건 추가 |
| 두 개념 비교 | 비교표 또는 퓨전도 |
| 버전·세대가 있는 기술 | 발전흐름도 (화살표 연결) |
| 윤리·사회적 이슈 | 기술적 방안 + 사회적 방안 2열 분리 |

**마무리 결론 공식:**

```
- 최근 [트렌드] 대두로 [기술 A]와 [기술 B] [사용/적용] 증대.  "끝"
- [관련 확장 기술] 등 사용 증대.  "끝"
```

## 2. 2교시형 — 서술형 2문항

2교시는 문항당 배점이 높고 깊이 있는 서술이 필요하다.  
**1단락 → 2단락 → 3단락 → (간글) → 4단락** 흐름으로 작성한다.

### 2-1. 1단락 — 개요 + 부각 배경

- 개념의 **등장 배경·발전 연혁·정의**를 포함한다.
- 발전 연혁이 있으면 **흐름도(화살표)**로 시각화한다.
- 답안 전체 구조를 미리 보여주는 **요약 박스** 또는 **마인드맵**을 삽입하면 가독성↑

**1단락 구조 예시 (멀티클라우드):**

```
1. 다수 Cloud 사용, 멀티 클라우드 개요

   정 의  - 동일 유형의 Cloud를 다양한 벤더로부터 제공받아
            복수 Cloud 운영 기술
          • Public 2개 이상  • Private 2개 이상  • Cloud 간 Fail over

   - Public 또는 Private 유형 각각 여러 개 운영.
```

### 2-2. 2단락 — 핵심 개념도 (1도 1표 원칙)

- **1개 구성도 + 1개 표** 조합이 기본.
- 구성도와 표를 그룹핑으로 연결하면 고득점 구조.
- 소주제가 2개 이상이면 `1)` `2)` 으로 나눠 각각 개념도·표 배치.

**2단락 구성 예시 (멀티클라우드 주요 기술):**

```
2. 멀티 클라우드 주요 기술
   1) 멀티 클라우드 주요 기술 구성도
      [Cloud App Orchestration] ← [IT 기업] → [Cloud Load Balancing]
                    (Public Cloud A)  통합Infra  (Public Cloud B)
                       <벤더 A>   <CSB>  <벤더 B>   CASB/CSPM

   - 멀티 클라우드는 Infra, App, 보안 등 고려.
```

**원본 — 멀티클라우드 구성도 (6주차 셀테, 문1 대주제2-1):**

![멀티클라우드 구성도](/itpe/resources/references/이지은%20기술사%20모범답안/공감반%2011기/6주차/KakaoTalk_20230413_205439326.jpg)

```
   2) 멀티 클라우드 주요 기술 상세
   | 구분   | 주요 기술           | 설 명            |
   |--------|---------------------|-----------------|
   | Infra  | • Cloud 통합 Infra  | - Infra 연계/통합 |
   |        | • CSB/Fail over     | - Cloud 서비스 중개 |
   | App    | • Orchestration     | - 기능 간 조정/배분 |
   |        | • Load Balance      | - 부하 분산/선택  |
   | 보안   | • CASB/CSPM         | - Cloud 간 보안 중개 |
   |        | • SeCaaS            | - 서비스형 보안   |
```

### 2-3. 3단락 — 비교 / 심화 분석

- 유사 개념과의 **비교표**를 작성한다.
- 박스를 사용해 핵심 차이를 시각화한다.

**비교 박스 예시 (멀티 vs 하이브리드):**

```
[멀티 Cloud]              [Hybrid Cloud]
• 동일 유형 통합           • 서로 다른 유형 통합
• 여러 벤더 사용           • 하나의 벤더 구축
• 복수 Cloud 연계          • 1개의 Cloud 구축

- 멀티는 여러 Cloud 연계, Hybrid는 한 Cloud 구축.
```

**원본 — 멀티클라우드 상세표 + 하이브리드 비교 (6주차 셀테, 문1 대주제2-2·3):**

![멀티클라우드 상세 및 하이브리드 비교](/itpe/resources/references/이지은%20기술사%20모범답안/공감반%2011기/6주차/KakaoTalk_20230413_205439326_01.jpg)

### 2-4. 4단락 — 융합 / 거버넌스 / 시사점

- 앞 단락을 기반으로 **확장·고려사항·시사점** 제시.
- **토픽 융합**: 두 기술의 단점을 서로 보완하는 융합 방안.
- **거버넌스 패턴**: 아래 집(House) 구조를 도메인만 바꿔 재활용 가능.

```
         ╱  관리 Vision / 목표  ╲
        ╱   품질 향상 / 리스크 관리  ╲
       ├──────────────────────────────┤
       │   조직    │  프로세스  │  도구  │  ← 거버넌스 요소
       ├──────────────────────────────┤
       │        표준 / 원칙 / 정책      │  ← 기반
```

> 데이터 거버넌스, AI 거버넌스, 클라우드 거버넌스 모두 이 구조로 응용 가능.

## 3. 3교시형 — 서술형 2문항 (응용·설계형)

3교시는 **설계·구축·적용** 관점의 깊이 있는 답안이 요구된다.  
2교시보다 현장 적용성과 실무 제언을 더 강조한다.

### 3-1. 1단락 — 개요 + 필요성

- **현황/문제점 → 필요성 → 개념** 순서로 1단락을 구성한다.
- "왜 이 기술이 필요한가"를 숫자나 그래프로 시각화하면 강력하다.

**1단락 구조 예시 (데이터 거버넌스):**

```
1. ISO 8000-150, 데이터 거버넌스 정의

   정 의  | 특 징
   - Master Data 관리 등    | • 원칙/조직/문화
     조직의 Data SDLC 및    | • Data SDLC 관리
     품질 등을 관리하는 원칙, | • Data 품질 관리
     조직, 문화 등 정의한 체계 | • 구조/보안 Manage

   - Meta Data를 기반으로 일반 Data도 관리.
   - ISO 8000-150의 Data 거버넌스 등 차용.
```

**원본 — 데이터 거버넌스 정의 + 구성도 (6주차 셀테, 문2 대주제1):**

![데이터 거버넌스 정의](/itpe/resources/references/이지은%20기술사%20모범답안/공감반%2011기/6주차/KakaoTalk_20230413_205439326_02.jpg)

### 3-2. 2단락 — 구성 상세 (1도 1표 + 소주제 분리)

- 소주제 `1)` `2)` 로 나눠 **구성도 → 상세 표** 패턴을 반복한다.
- 표의 구성: `구분 | 상세 요소 | 설 명` 3열 구조가 기본.

**표 예시 (데이터 거버넌스 구성):**

| 구분 | 상세 요소 | 설 명 |
|------|-----------|-------|
| 조직 | • Data 문화 | - Silo 방지, Fabric |
|     | • 관리/통제자 | - 최근 Data 관리자 |
| 원칙 | • Data Policy | - Data 관리 원칙 |
|     | • 거버넌스 원칙 | - 내부/외부 원칙 |
| 통제 | • Data SDLC | - 모니터링, 변환 통제 |
|     | • Lineage 관리 | - 시각화, 계보 관리 |
| MDM | • Meta Data | - 기준 Data 관리 |
|     | • Master Data | - ISO 8000-150 |
| 절차 | • 품질 Process | - 정착, 점시, 유형 |
|     | • 보안 Process | - DQC-S 등 기반 |

**원본 — 데이터 거버넌스 구성 상세표 (6주차 셀테, 문2 대주제2):**

![데이터 거버넌스 구성 상세](/itpe/resources/references/이지은%20기술사%20모범답안/공감반%2011기/6주차/KakaoTalk_20230413_205439326_03.jpg)

### 3-3. 3단락 — 실무 적용 / 고려사항

- **실무에서 어떻게 쓰이는가** 관점으로 서술.
- 리스크, 한계, 해결 방향을 함께 제시한다.
- 마무리는 항상 **트렌드 + 시사점**으로 끝낸다.

**마무리 공식:**

```
- [현상] 등 [관련 기술 키워드] 품질 중시, [해당 기술] 적용.  "끝"
```

### 3-4. 계산형 문제가 나올 경우 (EVM 등)

계산이 포함된 문제는 별도 구조로 작성한다.

```
1. [개념] 정의 + 기본/파생 지표 분류
2. 사례 분석
   1) 사례 조건 정리 | 사례 분석
   2) 기본 지표 계산표
      | 지 표 | 계 산 식         | 값             |
      |-------|------------------|----------------|
      | BAC   | • 총 예산         | 240,000 천원   |
      | PV    | • BAC×(일수/총기간)| = |100,000 천원| ← 박스 강조 |
      | EV    | • BAC × 수행률    | = |96,000 천원| |
      | CV    | • EV - AC         | = |-14,000 천원| |
3. 파생 지표 계산 + 해석
4. 대응 방안 (일정/비용 단축 기법)
```

> **계산 결과는 반드시 `|값|` 박스로 강조**. 채점관이 한눈에 확인할 수 있게 한다.

## 5. 4교시 공통 — 결론 문장 작성법

각 대주제의 끝, 그리고 답안 전체 마지막에 반드시 결론 문장을 쓴다.

### 대주제 결론 (중간 결론)

```
- [핵심 내용] 등 [고려/바탕으로] [행동/상태].
```

**예:**
- `- API G/W는 G/W에 통신 부하 이슈 발생.`
- `- Data 조직/문화를 기반으로 거버넌스 구성.`
- `- 요구사항 도출 시 투입 비용과 결과 등 고려.`

### 최종 결론 (트렌드 + "끝")

```
- 최근 [관련 기술] 등 [방향/현상] 증대/필요 예상.  "끝"
```

**예:**
- `- 최근 STPA, SOTIF, ETA 등 사용 증대.  "끝"`
- `- 타 Big Data 등 Data 품질 중시, 거버넌스 적용.  "끝"`
- `- 메타버스 개발, 운영, 이용 시의 윤리 이슈를 사전 숙지해 자발적 자제 위해 수립.  "끝"`

## 6. 핵심 원칙 요약

| 원칙 | 내용 |
|------|------|
| **구조화 최우선** | 번호 체계(`1.` → `1)` → `①` → `•` → `-`)를 반드시 지킨다 |
| **정의 먼저** | 모든 개념은 정의(정 의)부터 시작한다 |
| **그림 필수** | 구성도·흐름도·분류 박스를 항상 삽입한다 — 텍스트만 쓰지 않는다 |
| **표로 비교** | 2개 이상 개념은 반드시 표로 비교한다 (`구분 | 항목 | 설명` 3열) |
| **결론 마무리** | 각 대주제 끝에 `-` 결론, 마지막엔 트렌드 + `"끝"` |
| **키워드 강조** | 핵심 키워드는 밑줄(`_`)로 강조한다 |
| **계산 박스** | 계산 결과는 `|값|` 박스로 강조한다 |
| **트렌드 연결** | 마지막 항목은 반드시 "최근/향후 ~증대/필요"로 마무리한다 |

```
1교시형:  1.(정의+구성도) → 2.(핵심+표) → 3.(확장+트렌드)  "끝"
2교시형:  1.(개요+배경) → 2.(핵심:1도1표) → 3.(비교) → 간글 → 4.(융합/거버넌스)  "끝"
3교시형:  1.(개요+필요성) → 2.(구성:소주제별 1도1표) → 3.(실무+고려사항)  "끝"
4교시형:  소문항별 대주제 분리 → 위협/대응 2×2 → 사회적 4축 → 윤리 원칙  "끝"
```
