## 개요

데이터베이스 분할(Partitioning)은 대용량 테이블을 관리 가능한 단위로 나누어 조회 성능 향상, 관리 용이성 확보, 가용성 개선을 달성하는 기법이다. 분산 데이터베이스 시스템에서 데이터 배치 전략의 핵심이다.

## 핵심 내용

### 데이터베이스 분할 단위

| 분할 단위 | 설명 |
|-----------|------|
| **테이블 파티션** | 단일 논리 테이블을 물리적으로 분할 |
| **테이블스페이스** | 여러 데이터 파일을 논리적으로 그룹화 |
| **샤드(Shard)** | 여러 물리 서버에 걸쳐 데이터 분산 |
| **노드(Node)** | 분산 DB 클러스터의 독립 서버 단위 |

### 수평 분할 (Horizontal Partitioning / Sharding)

**개념**: 행(Row)을 기준으로 데이터를 나누어 여러 파티션에 저장

| 방식 | 설명 | 예시 |
|------|------|------|
| **범위 분할(Range)** | 키 값 범위로 분할 | 날짜별: 1월~3월, 4월~6월 |
| **해시 분할(Hash)** | 해시 함수 결과로 분할 | user_id % 4 |
| **목록 분할(List)** | 값 목록으로 분할 | 지역별: 서울, 경기, 부산 |
| **복합 분할(Composite)** | 두 가지 이상 방식 조합 | Range + Hash |

**장점**: 병렬 처리, 특정 파티션만 조회로 성능 향상
**단점**: 파티션 간 조인(Cross-shard Join) 비용 높음, 재분배(Re-sharding) 어려움

### 수직 분할 (Vertical Partitioning)

**개념**: 열(Column)을 기준으로 테이블을 나누어 저장

```
[원본 테이블: 사용자]
user_id | name | email | address | profile_image | bio

→ [핵심 테이블]        → [프로필 테이블]
   user_id | name | email    user_id | profile_image | bio
```

**장점**: 자주 접근하는 컬럼과 드문 컬럼 분리 → I/O 감소, 컬럼 수 줄어 캐시 효율 향상
**단점**: 조회 시 JOIN 필요, 트랜잭션 복잡도 증가

### 수평 vs 수직 분할 비교

| 구분 | 수평 분할 | 수직 분할 |
|------|-----------|-----------|
| 분할 기준 | 행(Row) | 열(Column) |
| 주요 목적 | 데이터 분산, 병렬 처리 | 접근 패턴 최적화 |
| 스키마 변화 | 동일 스키마, 다른 데이터 | 다른 스키마, 일부 데이터 |
| JOIN 필요 | 파티션 간 조인 | 항상 JOIN 필요 |
