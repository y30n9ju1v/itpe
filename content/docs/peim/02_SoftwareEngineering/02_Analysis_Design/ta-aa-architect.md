## 개요

TA(Technical Architect)는 인프라·플랫폼·네트워크 등 시스템 전반의 기술 아키텍처를 설계하며, AA(Application Architect)는 비즈니스 요구사항을 소프트웨어 구조로 변환하는 역할을 담당한다. 클라우드 네이티브 전환 프로젝트에서 두 역할의 협업이 특히 중요하다.

## 핵심 내용

### TA와 AA 역할 비교

| 구분 | TA (Technical Architect) | AA (Application Architect) |
|------|--------------------------|---------------------------|
| **범위** | 인프라, 네트워크, 보안, 플랫폼 | 애플리케이션 구조, API, 데이터 흐름 |
| **책임/목표** | 시스템 가용성·성능·보안 확보 | 비즈니스 기능의 소프트웨어 구현 |
| **주요 산출물** | 인프라 아키텍처, 네트워크 설계서, 보안 아키텍처 | 시스템 아키텍처, 컴포넌트 설계서, API 명세 |
| **관심 기술** | K8s, 네트워크, IaC, CI/CD 파이프라인 | MSA, DDD, 디자인 패턴, 프레임워크 |
| **사용자 관점** | 운영팀, DevOps | 개발팀, 비즈니스 분석가 |

### 클라우드 네이티브 전환에서의 역할

- **TA**: 컨테이너 오케스트레이션(K8s) 클러스터 설계, Service Mesh(Istio) 구성, 멀티 클라우드 네트워크, IaC(Terraform)
- **AA**: 모놀리식 → MSA 분해 전략, API Gateway 설계, 도메인 모델링(DDD), 이벤트 기반 아키텍처

### TA와 AA 협업의 중요성

클라우드 네이티브 환경에서는 인프라와 애플리케이션의 경계가 모호해져, 독립적 설계 시 다음 문제 발생:
- 컨테이너 스케일링 정책과 애플리케이션 상태 관리 충돌
- 서비스 간 통신 방식(동기/비동기) 불일치
- 보안 정책과 API 설계 미정합

### 협업 방안

1. **공동 아키텍처 리뷰 보드(ARB)**: TA·AA 공동 설계 검토
2. **ADR(Architecture Decision Record)**: 주요 결정사항 공유 문서화
3. **통합 설계 워크숍**: 스프린트 초기 공동 설계 세션
4. **공통 표준 수립**: API 표준, 로깅 표준, 보안 요구사항 통합 정의

## 출제 포인트

- TA와 AA를 범위·책임·산출물 3가지 측면 비교표 작성
- 클라우드 네이티브 맥락에서 각 역할 구체화
- 협업 방안(ARB, ADR) 포함

## 연관 개념

- MSA (마이크로서비스 아키텍처) — AA의 핵심 설계 패러다임
- DevOps/IaC — TA의 핵심 운영 방법론
- [재공학/역공학]({{< relref "/docs/peim/02_SoftwareEngineering/04_Operation_Maintenance/reengineering-reverse-engineering" >}}) — 기존 시스템을 클라우드 전환 시 연관
