## 해설

### 출제 배경 및 의도

IPv4 주소 고갈 문제를 해결하기 위해 VLSM(Variable Length Subnet Mask)과 CIDR(Classless Inter-Domain Routing)이 표준화되었다. 클래스 기반(Classful) 주소 체계의 낭비를 극복하고, 각 부서·서비스 규모에 맞는 서브넷을 효율적으로 할당하는 것이 IP 주소 설계의 핵심이다.

139회 관리 문제에서 VLSM 계산 문제를 출제한 것은 네트워크 설계 실무 역량을 직접 평가하기 위함이다. 단순 개념 이해를 넘어 큰 서브넷부터 순서대로 할당하는 VLSM 원칙과 주소 공간 낭비 최소화 계산 과정을 단계별로 명확히 서술해야 한다.

### 1. 서브네팅/수퍼네팅과 VLSM, 개요

정 의  • 서브네팅을 하나의 IP 블록을 서브넷 마스크 비트 증가로 다수의 소규모 네트워크로 분할하는 기법
       - VLSM은 서브넷마다 다른 마스크 길이를 허용하여 IP 주소 낭비를 최소화하는 확장된 서브네팅

```
[서브네팅 vs 수퍼네팅 방향]

서브네팅 (분할):
192.168.1.0/24 ──▶ 192.168.1.0/25  (128개, 마스크 비트 증가)
                    192.168.1.128/25 (128개)

수퍼네팅 (합산, CIDR):
192.168.0.0/24 ─┐
192.168.1.0/24 ─┤──▶ 192.168.0.0/23 (512개, 마스크 비트 감소)
192.168.2.0/24 ─┤──▶ 192.168.2.0/23 (512개)
192.168.3.0/24 ─┘

VLSM:
큰 블록 → [/23][/24][/25][/25] (팀 규모에 맞게 가변 분할)
```

- VLSM은 OSPF·EIGRP 등 Classless 라우팅 프로토콜에서 지원하며, Classful(RIPv1) 프로토콜은 VLSM을 미지원한다.

### 2. 수퍼네팅/서브네팅 개념 및 VLSM 설계 실습

1) 서브네팅과 수퍼네팅 비교

| 구분 | 항목 | 서브네팅 | 수퍼네팅 |
|------|------|------|------|
| 방향 | 분할/합산 | 하나 → 여러 개 분할 | 여러 개 → 하나로 합산 |
| 마스크 | 비트 변화 | 증가 (/24 → /25) | 감소 (/24 → /23) |
| 목적 | 효과 | IP 효율화, 브로드캐스트 도메인 축소 | 라우팅 테이블 집약, 경로 요약 |
| 기법 | 기술 | VLSM | CIDR (Classless Inter-Domain Routing) |
| 라우팅 프로토콜 | 지원 | OSPF, EIGRP, BGP (Classless) | BGP (경로 집약) |

- 수퍼네팅(CIDR)은 인터넷 BGP 라우팅 테이블 규모를 억제하는 경로 집약(Route Aggregation)의 핵심 기술이다.

2) VLSM 설계: 192.168.0.0/22 (4개 부서 분할)

```
[전체 주소 공간]
192.168.0.0/22 = 192.168.0.0 ~ 192.168.3.255 (1,024개 주소)

[VLSM 할당 순서: 큰 규모부터 연속 주소 공간에 순서대로 할당]

개발팀(500명):  500+2=502 → 2^9=512 → /23
  192.168.0.0/23  [.0.0 ~ .1.255]  ← 512개 사용

영업팀(250명):  250+2=252 → 2^8=256 → /24
  192.168.2.0/24  [.2.0 ~ .2.255]  ← 256개 사용

인사팀(125명):  125+2=127 → 2^7=128 → /25
  192.168.3.0/25  [.3.0 ~ .3.127]  ← 128개 사용

보안팀(125명):  125+2=127 → 2^7=128 → /25
  192.168.3.128/25 [.3.128 ~ .3.255] ← 128개 사용
  
총 사용: 512+256+128+128 = 1,024개 (낭비 없음)
```

| 구분 | 팀 | 필요 인원 | 서브넷 주소 | 서브넷 마스크 | 호스트 범위 | 사용 가능 수 |
|------|------|------|------|------|------|------|
| ① | 개발팀 | 500명 | 192.168.0.0/23 | 255.255.254.0 | .0.1 ~ .1.254 | 510개 |
| ② | 영업팀 | 250명 | 192.168.2.0/24 | 255.255.255.0 | .2.1 ~ .2.254 | 254개 |
| ③ | 인사팀 | 125명 | 192.168.3.0/25 | 255.255.255.128 | .3.1 ~ .3.126 | 126개 |
| ④ | 보안팀 | 125명 | 192.168.3.128/25 | 255.255.255.128 | .3.129 ~ .3.254 | 126개 |

- VLSM 핵심 공식: 필요 호스트수 + 2(네트워크·브로드캐스트) ≤ 2^n, 가장 작은 n 선택 → /32-n 마스크 적용

### 3. IP 주소 설계 트렌드 및 IPv6 전환 비교

| 항목 | Classful (구) | VLSM/CIDR (현) | IPv6 (차세대) |
|------|------|------|------|
| 주소 길이 | 32비트 (클래스 A/B/C) | 32비트 (가변 마스크) | 128비트 |
| 주소 효율 | 낮음 (클래스 고정) | 높음 (가변 서브넷) | 사실상 무한 |
| 라우팅 지원 | RIPv1, IGRP | OSPF, EIGRP, BGP | OSPFv3, IS-IS |
| 서브넷 설계 | 불필요 | VLSM 필수 | /48(사이트), /64(링크) 표준 |
| 주소 자동 설정 | DHCP | DHCP | SLAAC (자동) |

- IPv6 도입 후에도 기업 내부에서 IPv4 VLSM 설계 역량은 과도기 운영 및 이중 스택 환경에서 지속 요구된다.  "끝"

### 실무 제언

**미래 확장성을 고려한 여유 설계**
- **챌린지**: 개발팀 500명 기준으로 /23(510개)을 할당했으나 1년 후 1,000명으로 증가, IP 재설계 및 재할당 작업 발생
- **제언**: 초기 설계 시 예상 인원의 2배 버퍼를 포함하여 서브넷 크기 결정, 장기 IP 마스터플랜 수립으로 주소 공간 예약 관리(IP Address Management, IPAM 도구 활용)

**VLSM 라우팅 프로토콜 일관성 유지**
- **챌린지**: 일부 레거시 장비가 RIPv1(Classful) 동작으로 VLSM 서브넷 경로를 집약·오해석하여 라우팅 루프 발생
- **제언**: 전체 네트워크를 OSPFv2 또는 EIGRP로 통일, RIPv1 사용 구간은 RIPv2(서브넷 마스크 포함 광고)로 마이그레이션, 혼재 구간은 경로 재분배 정책 명확히 정의

**IPv4→IPv6 이중 스택 전환 시 주소 계획**
- **챌린지**: IPv6 도입 시 기존 IPv4 VLSM 설계와 독립적으로 IPv6 주소 체계를 새로 설계해야 하는 이중 관리 부담
- **제언**: IPv6는 사이트당 /48, 서브넷당 /64 표준 할당으로 단순화하고, IPv4 VLSM과 논리적으로 대응하는 이중 스택 주소 매핑 테이블을 IPAM에서 통합 관리
