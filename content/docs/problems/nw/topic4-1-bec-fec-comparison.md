## 해설

> **오류 발생 시 "다시 보내달라(BEC)"와 "스스로 고쳐라(FEC)" — 두 전략의 트레이드오프**

데이터 전송 중 오류는 피할 수 없습니다. 오류를 어떻게 처리할 것인가가 BEC와 FEC의 핵심 질문입니다.

### 1. 출제 배경 및 의도

무선 통신, 위성 통신, 광통신 등 다양한 매체에서 오류 제어 방식의 선택은 성능과 신뢰성에 직접적 영향을 줍니다. 위성 통신처럼 재전송이 불가능한 환경에서는 FEC가 필수이며, 안정적인 유선망에서는 BEC가 효율적입니다. 두 방식의 원리와 트레이드오프를 이해하는 것이 목표입니다.

### 2. 개념의 본질과 작동 메커니즘

#### BEC (Backward Error Correction) — 후진 오류 수정

오류를 **검출(Detection)**만 하고, 재전송을 요청하는 방식입니다. ARQ(Automatic Repeat reQuest)라고도 합니다.

```
송신자                          수신자
  │──────── 데이터 전송 ──────────▶│
  │                                │ (오류 검출: CRC 실패)
  │◀──────── NAK / 재전송 요청 ────│
  │──────── 데이터 재전송 ─────────▶│
  │                                │ (정상 수신)
  │◀──────── ACK ──────────────────│
```

#### FEC (Forward Error Correction) — 전진 오류 수정

오류 **검출 + 수정**이 가능한 여분 비트를 함께 전송하는 방식입니다. 재전송 없이 수신측에서 스스로 오류를 수정합니다.

```
송신자                          수신자
  │──── 데이터 + 여분비트(ECC) ──▶│
  │                                │ (오류 검출 + 자체 수정)
  │ (재전송 없음)                  │ (정상 복원)
```

#### BEC vs FEC 비교표

| 비교 항목 | BEC (ARQ) | FEC |
|---|---|---|
| **오류 처리** | 검출 후 재전송 요청 | 검출 + 자체 수정 |
| **재전송** | 필요 | 불필요 |
| **대역폭 효율** | 높음 (여분 비트 적음) | 낮음 (여분 비트 추가) |
| **지연** | 재전송으로 지연 발생 | 지연 없음 |
| **구현 복잡도** | 단순 | 복잡 (인코딩/디코딩) |
| **적용 환경** | 유선망, 재전송 가능 환경 | 위성통신, 실시간 스트리밍 |
| **대표 기술** | Stop-and-Wait, Go-Back-N, SR | 해밍코드, 터보코드, LDPC |

#### FEC 예시: 해밍 코드 (4비트 데이터 → 7비트 전송)

```
원본 데이터:   1 0 1 1  (4비트)
해밍코드 적용: 1 0 1 1 0 1 1  (7비트, 3비트 패리티 추가)
                     ↑ ↑ ↑
                     패리티 비트 (p1, p2, p4)

수신측: 오류 발생 위치를 스스로 계산하여 1비트 수정 가능
```

#### BEC의 ARQ 세부 방식

| ARQ 유형 | 설명 | 효율 |
|---|---|---|
| **Stop-and-Wait** | 한 번에 하나씩 전송 후 ACK 대기 | 낮음 |
| **Go-Back-N** | 슬라이딩 윈도우, 오류 패킷부터 재전송 | 중간 |
| **Selective Repeat** | 오류 패킷만 선택 재전송 | 높음 |

### 3. 핵심 키워드 (Must-Have)

- **BEC:** 오류 검출 후 재전송, ARQ, ACK/NAK
- **FEC:** 오류 검출+수정, 여분 비트, 해밍코드, 터보코드
- **Redundancy:** FEC의 여분 비트 = 오버헤드
- **RTT (Round Trip Time):** BEC에서 재전송 지연의 핵심 요소

### 4. 맥락과 비교

**혼합 방식 (Hybrid ARQ)**

현대 LTE/5G는 **HARQ(Hybrid ARQ)**를 사용합니다. FEC로 1차 오류 수정을 시도하고, 수정 불가 시 ARQ로 재전송을 요청합니다. 두 방식의 장점을 결합한 현실적 최적 해결책입니다.

```
HARQ = FEC(1차 수정 시도) + ARQ(수정 실패 시 재전송)
```

### 5. 실무 제언

5G NR(New Radio) 무선 구간에서 LDPC(Low-Density Parity Check) 코드와 Polar 코드를 FEC로 사용합니다. 데이터 채널은 LDPC, 제어 채널은 Polar 코드를 채택하여 초저지연과 높은 신뢰성을 동시에 달성합니다. 기술사 답안에서 BEC/FEC 비교표와 함께 HARQ, LTE/5G 적용 사례를 언급하면 최신 기술 트렌드까지 이해하고 있음을 보여줄 수 있습니다.
