## 해설

### 출제 배경 및 의도

데이터 전송 중 오류는 전송 매체의 물리적 특성상 완전히 피할 수 없다. 유선망에서는 전자기 간섭, 무선망에서는 페이딩·노이즈로 인해 비트 오류가 발생한다. 이를 어떻게 처리할 것인가가 BEC와 FEC의 핵심 질문이다.

위성 통신처럼 RTT(Round Trip Time)가 수백 ms에 달하거나 단방향 방송처럼 재전송이 구조적으로 불가능한 환경에서는 FEC가 필수다. 반면 유선 이더넷과 같이 낮은 오류율과 빠른 재전송이 가능한 환경에서는 BEC가 대역폭 효율 측면에서 우월하다. 현대 LTE/5G는 두 방식을 결합한 HARQ를 채택하여 실용적 최적해를 구현했다.

이 문제는 두 오류 제어 철학의 원리와 트레이드오프를 이해하고, 적합한 환경을 판단하는 능력을 평가한다.

### 1. 오류 제어(Error Control), 개요

정 의  • 전송 중 발생한 비트 오류를 검출하고, 수정하거나 재전송을 요청하여 데이터 무결성을 보장하는 기술
       - BEC는 오류 검출 후 재전송(ARQ), FEC는 여분 비트를 이용한 수신측 자체 수정으로 구분

```
[BEC (Backward Error Correction)]
송신자                          수신자
  │──────── 데이터 전송 ─────────▶│
  │                               │ (오류 검출: CRC 실패)
  │◀──────── NAK / 재전송 요청 ───│
  │──────── 데이터 재전송 ────────▶│
  │                               │ (정상 수신)
  │◀──────── ACK ─────────────────│

[FEC (Forward Error Correction)]
송신자                          수신자
  │──── 데이터 + 여분비트(ECC) ──▶│
  │                               │ (오류 검출 + 자체 수정)
  │ (재전송 없음)                 │ (정상 복원)
```

- BEC는 구현이 단순하고 대역폭 효율이 높으나 재전송 지연이 발생하고, FEC는 지연 없이 수정하나 여분 비트로 인한 오버헤드가 크다.

### 2. BEC vs FEC 핵심 비교

1) 기술 특성 비교

| 구분 | 항목 | BEC (ARQ) | FEC |
|------|------|-----------|-----|
| 오류 처리 | 방식 | 검출 후 재전송 요청 | 검출 + 자체 수정 |
| 대역폭 | 효율 | 높음 (여분 비트 적음) | 낮음 (여분 비트 추가) |
| 지연 | 발생 여부 | 재전송으로 RTT 지연 발생 | 지연 없음 |
| 구현 | 복잡도 | 단순 | 복잡 (인코딩/디코딩 연산) |
| 적용 | 환경 | 유선망, 양방향 통신 | 위성통신, 실시간 스트리밍 |
| 대표 기술 | 알고리즘 | Stop-and-Wait, GBN, SR | 해밍코드, 터보코드, LDPC |

- BEC는 재전송이 가능한 환경에서 대역폭 효율이 우수하고, FEC는 재전송이 불가능하거나 지연에 민감한 환경에서 필수적이다.

2) ARQ(BEC) 세부 방식 비교

| 구분 | 항목 | Stop-and-Wait | Go-Back-N | Selective Repeat |
|------|------|--------------|-----------|-----------------|
| 윈도우 | 크기 | 1 | N | N/2 |
| 재전송 | 범위 | 오류 1개 | 오류 이후 전체 | 오류 패킷만 |
| 수신 | 버퍼 필요 | 불필요 | 불필요 | 필요 |
| 효율 | 평가 | 낮음 | 중간 | 높음 |
| 구현 | 복잡도 | 단순 | 중간 | 복잡 |

- Selective Repeat은 수신측 버퍼 관리 복잡도가 높으나 오류 패킷만 재전송하여 대역폭 낭비를 최소화하므로 위성·고지연 링크에 적합하다.

### 3. HARQ: FEC + ARQ 혼합 방식 및 현대 적용 트렌드

```
[HARQ (Hybrid ARQ) = FEC + ARQ]
송신자           수신자
  │── 데이터 + FEC 코드 ──▶│
  │                        │ FEC로 1차 수정 시도
  │                        │ ┌─성공: ACK 전송
  │                        │ └─실패: NACK 전송
  │◀──── NACK ─────────────│
  │──── 재전송 (Incremental Redundancy) ──▶│
  │                        │ 이전 수신 데이터 + 신규 데이터 합산 복원
```

| 구분 | 항목 | 설명 |
|------|------|------|
| LTE/5G | HARQ 적용 | 물리계층에서 FEC(LDPC/Polar) + ARQ 혼합으로 초저지연 달성 |
| 5G 데이터 채널 | LDPC 코드 | 고속 데이터 채널에 적용, 높은 코딩 이득 |
| 5G 제어 채널 | Polar 코드 | 제어 채널에 적용, 짧은 코드 길이에서 최적 성능 |
| 위성통신 | 터보코드 | DVB-S2X 표준에 적용, 섀넌 한계 근접 성능 |

- LTE/5G에서 HARQ는 물리계층 재전송을 1~3회로 제한하고 이후 상위계층 ARQ(RLC)가 처리하는 계층적 오류 제어 구조로 초저지연과 높은 신뢰성을 동시에 달성한다.  "끝"

### 실무 제언

**5G 무선 구간 FEC 코드 선택 설계**
- **챌린지**: 5G NR에서 채널 유형(데이터/제어)별로 최적 FEC 코드가 달라 잘못 적용 시 성능 저하
- **제언**: 데이터 채널(PDSCH/PUSCH)에는 LDPC 코드, 제어 채널(PDCCH/PUCCH)에는 Polar 코드를 3GPP TS 38.212 규격에 따라 적용하고, 코드레이트를 채널 품질(CQI)에 따라 동적으로 조정

**위성 인터넷 서비스(LEO)의 FEC 최적화**
- **챌린지**: Starlink 등 LEO 위성의 RTT가 20~40ms로 낮아졌으나 링크 버짓이 제한되어 적절한 FEC 오버헤드 설계 필요
- **제언**: 링크 여유분(Link Margin)에 따라 FEC 코드레이트를 적응적으로 조정하는 ACM(Adaptive Coding and Modulation) 적용으로 최적 처리량 달성

**TCP 기반 파일 전송의 BEC 성능 튜닝**
- **챌린지**: 고지연 WAN 구간에서 TCP Selective Repeat의 윈도우 크기가 작아 대역폭 충분히 활용 불가
- **제언**: TCP SACK(Selective Acknowledgment) 옵션 활성화와 윈도우 스케일링(RFC 1323)으로 대역폭-지연 곱(BDP) 크기의 윈도우를 설정하여 링크 효율 극대화
