## 문제

네트워크 라우팅 기법 중 거리벡터라우팅(Distance Vector Routing)과 링크스테이트라우팅(Link State Routing) 기법을 설명하고, 두 기법의 차이점을 비교하시오.

## 해설

### 출제 배경 및 의도

라우터들이 목적지까지의 최적 경로를 어떻게 알아내는가는 인터넷 인프라의 근본 질문이다. 거리벡터(DV)는 이웃에게 소문을 전달받는 분산적 방식이고, 링크스테이트(LS)는 전체 지도를 공유하는 전역적 방식이다.

RIP, OSPF, BGP, EIGRP 등 모든 라우팅 프로토콜의 알고리즘 기반이 이 두 방식에서 출발한다. 대규모 기업 WAN, 데이터센터 패브릭, ISP 네트워크 설계 시 어떤 라우팅 프로토콜을 선택하는가는 이 개념에 대한 이해에서 결정된다.

이 문제는 두 알고리즘의 동작 원리, 카운팅 투 인피니티 문제, 수렴 속도 차이를 정확히 이해하고 실제 프로토콜과 연결 서술하는 능력을 평가한다.

### 1. 라우팅 알고리즘(Routing Algorithm), 개요

정 의  • 라우터가 목적지까지의 최적 경로를 계산하고 라우팅 테이블을 구성하는 알고리즘
       - 거리벡터는 이웃 간 라우팅 테이블 교환(Bellman-Ford), 링크스테이트는 전체 토폴로지 공유 후 독자 계산(Dijkstra)

```
[거리벡터 방식]                    [링크스테이트 방식]
 A ─1─ B ─2─ C                    A ─1─ B ─2─ C
       │                                 │
       1                                 1
       │                                 │
       D                                 D

A의 뷰: B까지 1홉 (B가 알려줌)       모든 라우터가 전체 토폴로지 보유
        C까지 3홉 (B→C 통해 전달)     각자 Dijkstra로 최단 경로 계산
        D까지 2홉 (B→D 통해 전달)     결과: 완전하고 일관된 라우팅 테이블
```

- 거리벡터는 구현이 단순하고 리소스 사용이 적으나 수렴이 느리고 루프 위험이 있으며, 링크스테이트는 빠른 수렴과 정확한 경로 계산이 가능하나 메모리·CPU 소모가 크다.

### 2. 거리벡터 vs 링크스테이트 상세 비교

1) 거리벡터 라우팅 (Distance Vector Routing)

| 구분 | 항목 | 설명 |
|------|------|------|
| 알고리즘 | Bellman-Ford | 이웃의 최소 거리 정보를 받아 자신의 최단 경로 갱신 |
| 정보 공유 | 라우팅 테이블 교환 | 이웃 라우터에게만 자신의 전체 라우팅 테이블 전송 |
| 업데이트 | 주기적 | RIP: 30초마다 전체 테이블 브로드캐스트 |
| 문제점 | 카운팅 투 인피니티 | 링크 단절 시 거리가 무한대로 증가하는 루프 발생 |
| 해결책 | Split Horizon | 경로를 배운 방향으로 같은 경로를 광고하지 않음 |
| 해결책 | Poison Reverse | 링크 단절 시 해당 경로의 거리를 ∞(16)으로 광고 |
| 대표 프로토콜 | RIPv1/v2 | UDP 520포트, 최대 홉 15, 수렴 시간 수분 |

```
[카운팅 투 인피니티 문제]
정상: A─B─C (A까지 B:1홉, C:2홉)
링크 단절: A─B 연결 끊김
B: "A까지 ∞"
C: "B를 통해 A까지 2홉 알고있어" → B에게 광고
B: "C를 통해 A까지 3홉 가능해" → 갱신
C: "B를 통해 A까지 4홉 가능해" → 갱신
... → 15홉 도달 시 무효 처리 (RIP 기준)
```

- 카운팅 투 인피니티는 거리벡터 방식의 근본적 한계이며, RIP에서는 최대 홉 15로 제한하고 Split Horizon + Poison Reverse를 함께 적용하여 완화한다.

2) 링크스테이트 라우팅 (Link State Routing)

| 구분 | 항목 | 설명 |
|------|------|------|
| 알고리즘 | Dijkstra | 전체 LSDB를 기반으로 출발지에서 모든 목적지까지 최단 경로 계산 |
| 정보 공유 | LSA 플러딩 | 자신의 직접 연결 링크 상태만 전체 네트워크에 플러딩 |
| LSDB | 토폴로지 DB | 전체 네트워크 링크 상태 저장, 모든 라우터가 동일 LSDB 보유 |
| 업데이트 | 변화 시 즉시 | 링크 변화 시에만 LSA 생성·전파 (주기적 갱신: 30분) |
| 수렴 속도 | 빠름 | 변화 즉시 전체 네트워크에 전파, 수초 내 수렴 |
| 확장성 | Area 분할 | OSPF Area로 LSDB 범위 제한, 대규모 네트워크 지원 |
| 대표 프로토콜 | OSPF, IS-IS | 기업망·ISP 표준 IGP |

```
[링크스테이트 동작]
각 라우터가 LSA 생성:
"나는 A. 이웃: B(비용1), C(비용2)"
"나는 B. 이웃: A(비용1), D(비용1)"
"나는 C. 이웃: A(비용2), D(비용3)"

모든 LSA → LSDB(전체 토폴로지 지도) 구성
각 라우터: Dijkstra 알고리즘으로 최단 경로 독자 계산
결과: 모든 라우터가 동일한 토폴로지 뷰, 일관된 라우팅
```

- LSDB 기반 Dijkstra 계산은 링크 단절 시 카운팅 투 인피니티 없이 수초 내 정확한 대체 경로를 반영한다.

### 3. 종합 비교 및 프로토콜별 분류

| 비교 항목 | 거리벡터 | 링크스테이트 |
|----------|---------|------------|
| 알고리즘 | Bellman-Ford | Dijkstra |
| 정보 공유 | 이웃에게 라우팅 테이블 | 전체 네트워크에 링크 상태 |
| 네트워크 뷰 | 부분적 (이웃만) | 전체 토폴로지 |
| 수렴 속도 | 느림 (수분) | 빠름 (수초) |
| 메모리/CPU | 낮음 | 높음 (LSDB 유지·계산) |
| 루프 위험 | 높음 (카운팅 투 인피니티) | 낮음 |
| 확장성 | 소규모 적합 (최대 15홉) | 대규모 적합 (Area 분할) |
| 대표 프로토콜 | RIPv1/v2, EIGRP(하이브리드) | OSPF, IS-IS |

| 분류 | 프로토콜 | 알고리즘 | 특징 |
|------|---------|---------|------|
| IGP - DV | RIPv1/v2 | Bellman-Ford | 최대 15홉, UDP 520, 소규모망 |
| IGP - LS | OSPF | Dijkstra | Area 분할, 대규모 기업망 표준 |
| IGP - LS | IS-IS | Dijkstra | ISP 백본 선호, TLV 기반 확장성 |
| IGP - 하이브리드 | EIGRP (Cisco) | DUAL | DV+LS 혼합, Cisco 전용 |
| EGP | BGP | Path Vector | AS 간 라우팅, 인터넷 백본 표준 |

- 대기업 WAN 환경에서는 OSPF를 Area 단위로 분할 설계하여 LSDB 크기를 제한하고, ISP 간에는 BGP(경로 벡터 방식)로 AS_PATH 속성을 통해 루프를 방지한다.  "끝"

### 실무 제언

**대규모 기업 WAN OSPF Area 설계**
- **챌린지**: 단일 OSPF Area에 수백 대 라우터 배치로 LSDB 과대 성장, SPF 재계산 부하로 CPU 스파이크 발생
- **제언**: 코어-분배-액세스 계층 구조에 맞게 OSPF Area를 분할하고 Area 0(백본) 중심 허브앤스포크 설계로 LSDB를 Area 단위로 격리, 요약 경로(Summary Route)로 라우팅 테이블 크기 최소화

**링크 장애 시 빠른 수렴 설계**
- **챌린지**: OSPF Hello/Dead 타이머 기본값(10초/40초)으로 인해 링크 장애 감지와 수렴에 최대 40초 소요
- **제언**: BFD(Bidirectional Forwarding Detection)를 OSPF와 연동하여 밀리초 단위 링크 장애 감지를 구현하고, Hello 타이머를 1초/3초로 단축하여 SLA 요건 충족

**ISP 간 BGP 라우팅 정책 설계**
- **챌린지**: 멀티홈 환경에서 인바운드·아웃바운드 트래픽이 특정 링크에 편중되어 대역폭 낭비
- **제언**: 아웃바운드는 Local Preference로 선호 경로를 지정하고, 인바운드는 AS_PATH Prepend 또는 MED(Multi-Exit Discriminator)로 트래픽을 분산하여 링크 활용률 균형화
