## 해설

### 출제 배경 및 의도

IPv4의 32비트 주소(약 43억 개)는 이미 고갈되었다. 스마트폰·IoT 기기의 폭발적 증가로 NAT 기반의 IPv4 연장도 한계에 도달하면서, IPv6로의 전환은 선택이 아닌 필수가 되었다. 국내 주요 ISP와 LTE/5G 망은 이미 이중 스택 운용 중이며, 클라우드·CDN·모바일 환경에서 IPv6 지원 여부가 서비스 경쟁력에 직결된다.

기술사 시험에서는 IPv6의 헤더 구조 단순화 원리, 단편화 방식의 설계 변화(라우터→호스트), 전환 방안 3가지(이중 스택·터널링·NAT64)를 체계적으로 서술할 것을 요구한다. IPv4 대비 개선점을 표로 비교하고 전환 시나리오까지 서술하면 고득점이 가능하다.

### 1. IPv6(Internet Protocol version 6), 개요

정 의  • IPv6를 128비트 주소 체계로 사실상 무한한 주소 공간과 보안·QoS·자동설정을 기본 탑재한 차세대 인터넷 프로토콜
       - IPv4의 주소 고갈·헤더 복잡성·보안 한계를 근본적으로 해결하도록 설계

```
[IPv6 고정 헤더 구조 (40바이트)]

 ┌──────────┬──────────────────┬───────────────────────────┐
 │Version(4)│ Traffic Class(8) │      Flow Label(20)        │
 ├──────────────────────────────┼──────────────┬────────────┤
 │      Payload Length(16)      │Next Header(8)│Hop Limit(8)│
 ├────────────────────────────────────────────────────────────┤
 │                  Source Address (128 bits)                  │
 ├────────────────────────────────────────────────────────────┤
 │               Destination Address (128 bits)                │
 └────────────────────────────────────────────────────────────┘
 고정 40바이트 (IPv4: 가변 20~60바이트)

확장 헤더 체인 (Next Header 필드로 연결):
IPv6 헤더 → [Hop-by-Hop] → [Routing] → [Fragment] → TCP/UDP
           Next Header=0   NH=43       NH=44        NH=6/17
```

- IPv4의 체크섬(라우터 매번 재계산) 제거, 옵션 → 확장 헤더 분리로 라우터 처리 속도가 대폭 향상되었다.

### 2. IPv6 헤더 필드·단편화 방식·전환 방안

1) IPv6 헤더 필드 및 IPv4 대비 개선

| 구분 | 필드 | 크기 | 설명 |
|------|------|------|------|
| 기본 | Version | 4비트 | IP 버전 (값: 6) |
| | Traffic Class | 8비트 | QoS 트래픽 분류 (IPv4 TOS 대응) |
| | Flow Label | 20비트 | 동일 흐름 패킷 식별 (실시간 서비스 QoS) |
| | Payload Length | 16비트 | 확장 헤더 포함 페이로드 길이 |
| | Next Header | 8비트 | 다음 확장 헤더 또는 상위 프로토콜 식별 |
| | Hop Limit | 8비트 | IPv4 TTL과 동일 (라우터 통과마다 1 감소) |
| 주소 | Source/Destination | 각 128비트 | 출발지/목적지 IPv6 주소 |

- IPv4 대비 제거: 헤더 체크섬(상위 계층 위임), 단편화 관련 필드(ID·Flags·Fragment Offset), IHL(가변 헤더), Options(확장 헤더로 분리)

2) 패킷 단편화 처리 방식 비교

```
[IPv4 vs IPv6 단편화]

IPv4 (라우터 단편화 허용):
송신 [2000B] ──▶ 라우터(MTU=1500) ──▶ [1480B]+[520B] ──▶ 수신
                  (직접 분할)              (재조합)

IPv6 (라우터 단편화 불가):
송신 [2000B] ──▶ 라우터(MTU=1500) ──▶ 패킷 드롭
                                        ICMPv6 "Packet Too Big(1500)"
  ← 송신이 MTU 조절 → 송신 [1480B] ──▶ [1480B] ──▶ 수신

PMTUD (Path MTU Discovery):
송신 ─▶ [경로 최소 MTU 탐색] ─▶ 적정 크기로 처음부터 전송
```

| 구분 | 항목 | IPv4 | IPv6 |
|------|------|------|------|
| 단편화 주체 | 누가 | 라우터 또는 송신 호스트 | 송신 호스트만 |
| MTU 기본값 | 최소 | 576바이트 | 1,280바이트 |
| PMTUD | 필수 여부 | 선택적 | 필수 |
| 라우터 부하 | 처리 | 단편화 처리 부담 | 없음 (드롭+ICMPv6) |

3) IPv4 → IPv6 전환 방안 3가지

```
[전환 방안 비교]

① 이중 스택 (Dual Stack):
  호스트: IPv4 스택 + IPv6 스택 동시 운용
  목적지 DNS에 따라 IPv4/IPv6 자동 선택
  장점: 가장 자연스러운 전환
  단점: 두 프로토콜 인프라 이중 관리

② 터널링 (Tunneling):
  [IPv6 섬] ──▶ IPv6 패킷을 IPv4로 캡슐화 ──▶ [IPv4 망] ──▶ [IPv6 섬]
  종류: 6in4(수동), 6to4(자동), Teredo(NAT 통과), ISATAP
  장점: 기존 IPv4 망 그대로 활용
  단점: 오버헤드, 관리 복잡

③ 주소 변환 (NAT64 + DNS64):
  IPv6 전용 호스트 ──▶ NAT64 장비 ──▶ IPv4 서버
  DNS64: AAAA 레코드 없을 시 A 레코드로 합성 응답
  장점: IPv6 전용 환경에서 IPv4 서버 접근 가능
  단점: 애플리케이션 레벨 주소 임베딩 시 문제
```

### 3. IPv4 vs IPv6 종합 비교 및 전환 현황

| 항목 | IPv4 | IPv6 |
|------|------|------|
| 주소 길이 | 32비트 (약 43억 개) | 128비트 (약 3.4×10^38개) |
| 헤더 크기 | 가변 (20~60B) | 고정 (40B) |
| 단편화 | 라우터·호스트 | 호스트만 (PMTUD 필수) |
| 보안 | 선택적 IPSec | IPSec 기본 지원 권고 (초기 설계 의도; RFC 6434에서 필수→권고로 완화) |
| 자동 설정 | DHCP 필요 | SLAAC (Stateless Auto) |
| 멀티캐스트 | 선택적 | 기본 지원 |
| 브로드캐스트 | 있음 | 없음 (멀티캐스트로 대체) |
| ARP 대체 | ARP | NDP (ICMPv6 기반) |
| 국내 ISP | 이중 스택 운용 중 | LTE/5G IPv6 우선 정책 |

- IPv6 전환은 이중 스택 → 터널링 병행 → 완전 IPv6 단계로 진행하며, NAT64/DNS64를 레거시 IPv4 서비스와의 호환 브리지로 활용하는 것이 현실적 전략이다.  "끝"

### 실무 제언

**기업망 이중 스택 전환 단계적 적용**
- **챌린지**: 전체 네트워크를 한번에 IPv6로 전환 시 레거시 애플리케이션의 IPv6 미지원으로 서비스 중단 위험
- **제언**: 코어·배포 계층부터 이중 스택 적용 후 엣지 계층으로 확산, IPAM 도구로 IPv4·IPv6 주소 통합 관리, 애플리케이션 IPv6 대응 여부 사전 검증 필수

**PMTUD 블랙홀 문제 해결**
- **챌린지**: 방화벽이 ICMPv6 "Packet Too Big" 메시지를 차단하여 PMTUD가 동작하지 않고 패킷이 무한 드롭되는 PMTUD 블랙홀 발생
- **제언**: 방화벽 정책에서 ICMPv6 Type 2(Packet Too Big) 명시적 허용, MSS Clamping(TCP 최대 세그먼트 크기 조정)으로 단기 대응, 근본 해결은 방화벽 ICMPv6 정책 정비

**모바일 환경 IPv6 우선(Happy Eyeballs) 적용**
- **챌린지**: 서버가 IPv4만 지원 시 스마트폰 IPv6 우선 연결 시도 후 타임아웃, 수백 ms의 연결 지연 발생
- **제언**: RFC 8305(Happy Eyeballs v2) 구현으로 IPv6·IPv4 병렬 연결 시도 후 먼저 성공한 경로 사용, 서버 DNS AAAA 레코드 추가 및 NAT64 연동으로 IPv6 전용 클라이언트 지원 체계 구축
