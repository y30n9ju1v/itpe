## 해설

### 출제 배경 및 의도

패킷교환 네트워크는 본질적으로 Best-Effort 서비스로, 모든 트래픽을 동등하게 처리한다. 그러나 VoIP·영상회의 같은 실시간 트래픽은 지연과 지터에 민감하여 Best-Effort 환경에서 품질을 보장할 수 없다.

기업 내 통합 커뮤니케이션(UC)과 클라우드 협업 툴의 보편화로 음성·영상 트래픽과 일반 데이터 트래픽이 동일 네트워크를 공유하는 환경이 일반화되었다. QoS 스케줄링·트래픽 제어·서비스 모델은 이러한 환경에서 실시간 서비스 품질을 보장하는 핵심 메커니즘으로, 기술사 시험에서 가·나·다 세 항목으로 구분하여 체계적 이해를 측정하는 복합 문제로 자주 출제된다.

### 1. QoS(Quality of Service), 개요

정 의  • 네트워크에서 특정 트래픽 유형의 지연·지터·패킷손실·대역폭을 보장하기 위한 패킷 처리 우선순위 및 자원 관리 기술 체계
       - 분류(Classification) → 마킹(Marking) → 큐잉(Queuing) → 쉐이핑/폴리싱 순서로 계층 적용

```
[입력 트래픽]
     |
     +--분류(Classification)--+
     |   VoIP / 영상 / 데이터 |
     |                        v
     |              [마킹: DSCP EF/AF/BE]
     |                        |
     |           +--------+--------+--------+
     |           | LLQ    | CBWFQ  | BE큐   |
     |           | VoIP   | 영상   | 데이터  |
     |           +--------+--------+--------+
     |                        |
     +--[스케줄러]--[쉐이핑/폴리싱]-->[출력]
```

- 트래픽 분류에서 출력까지 계층적 QoS 파이프라인이 실시간 서비스 품질을 보장한다.

### 2. QoS 핵심 기술

1) 가. 패킷 스케줄링 기술

| 구분 | 알고리즘 | 설명 |
|------|----------|------|
| ① | FIFO | 선입선출, 우선순위 없음, 단순 구현 |
| ② | PQ (Priority Queuing) | 높은 우선순위 큐 우선 처리, 낮은 큐 Starvation 위험 |
| ③ | WFQ (Weighted Fair Queuing) | 가중치 비례 대역폭 할당, 공정성 보장 |
| ④ | WRR (Weighted Round Robin) | 가중치 비례 라운드로빈 서비스 |
| ⑤ | CBWFQ + LLQ | 클래스별 WFQ + VoIP 최우선 큐 결합, 실무 표준 |

```
CBWFQ + LLQ 구조:
VoIP  --> [LLQ: 절대 우선, 항상 먼저 처리]
영상  --> [CBWFQ Class1: 30% 보장 대역폭]
데이터--> [CBWFQ Class2: 나머지 대역폭]
```

- LLQ + CBWFQ 조합이 엔터프라이즈 QoS 스케줄링의 실무 표준이다.

2) 나. 버킷(Bucket) 트래픽 제어 기술

| 구분 | 항목 | 토큰 버킷 | 리키 버킷 |
|------|------|----------|----------|
| 버스트 | 허용 여부 | 허용 (토큰 축적) | 불허 |
| 출력 속도 | 패턴 | 가변 (최대 r+B) | 일정 (r) |
| 주요 용도 | 적용 | 트래픽 폴리싱 | 트래픽 쉐이핑 |
| 동작 | 메커니즘 | 토큰 소비 후 패킷 전송 | 일정 속도 출력, 초과 드롭 |

```
토큰 버킷:                    리키 버킷:
토큰생성(r/초)                입력 패킷
    |                             |
[버킷: 최대B개]               [버킷: 최대B]
    |                             |
패킷전송(토큰소비)            일정속도(r) 출력
없으면 대기                   가득 차면 드롭
```

- 토큰 버킷은 버스트 허용 폴리싱, 리키 버킷은 트래픽 평탄화(쉐이핑)에 사용된다.

### 3. IntServ vs DiffServ 모델 비교

1) IntServ (Integrated Services) 동작

```
[송신단]--RSVP PATH(QoS 요구사항)-->[라우터A]->[라우터B]->[수신단]
[수신단]<--RSVP RESV(자원 예약)----[라우터A]<-[라우터B]<-[수신단]
          (각 라우터가 2Mbps 예약)
```

2) DiffServ (Differentiated Services) 동작

```
[경계 라우터: 트래픽 분류 및 DSCP 마킹]
  VoIP  --> DSCP EF (46)  --> LLQ 처리
  영상  --> DSCP AF41(34) --> 보장 대역폭
  데이터--> DSCP BE (0)   --> Best-Effort

[코어 라우터: DSCP 값만 보고 PHB 적용, 흐름 상태 불필요]
```

3) IntServ vs DiffServ 비교

| 비교 항목 | IntServ | DiffServ |
|----------|---------|---------|
| QoS 단위 | 개별 흐름(Flow) | 트래픽 클래스 |
| 자원 예약 | RSVP로 경로상 예약 | 없음 (마킹만) |
| 보장 수준 | 엄격한 보장 | 상대적 우선순위 |
| 확장성 | 낮음 (흐름 상태 저장) | 높음 (Stateless) |
| 복잡도 | 높음 | 낮음 |
| 대규모 적용 | 어려움 | 용이 |
| 대표 기술 | RSVP | DSCP, MPLS DS-TE |

- DiffServ가 확장성·단순성 면에서 기업망·ISP의 실질적 표준이며, MPLS DS-TE와 결합한 WAN QoS가 현재 실무 표준이다.  "끝"

### 실무 제언

**엔터프라이즈 QoS 설계**
- **챌린지**: VoIP·영상회의·일반 데이터가 동일 WAN 회선을 공유할 때 실시간 서비스 품질 저하
- **제언**: DiffServ 기반 DSCP 마킹(VoIP=EF/46, 영상=AF41/34, 데이터=BE/0) + WAN 라우터 LLQ+CBWFQ 적용, SLA 측정값(지연<150ms, 지터<30ms, 손실<1%) 모니터링

**SD-WAN 환경 QoS 자동화**
- **챌린지**: 수백 개 지점 라우터에 QoS 정책을 수동 설정하는 운영 복잡성
- **제언**: SD-WAN 컨트롤러에서 DPI 기반 애플리케이션 자동 인식 후 DSCP 마킹 자동 적용, 중앙 정책 배포로 운영 효율화

**토큰 버킷 기반 SLA 강제**
- **챌린지**: ISP 회선 계약 대역폭 초과 트래픽이 네트워크 혼잡을 유발하여 SLA 위반
- **제언**: 경계 라우터에 토큰 버킷 폴리싱 적용하여 계약 대역폭 초과 트래픽 마킹(DSCP 강제 하향) 또는 드롭, 버스트 허용 범위 협상으로 유연한 SLA 구성
