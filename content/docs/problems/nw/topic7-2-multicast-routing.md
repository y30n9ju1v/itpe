## 해설

### 출제 배경 및 의도

IPTV, 화상회의, 금융 시세 데이터 배포 등 대규모 실시간 서비스가 확산되면서 멀티캐스트의 중요성이 높아지고 있다. 동일 데이터를 수천 명에게 유니캐스트로 전송하면 N배의 트래픽이 발생하지만, 멀티캐스트는 하나의 스트림으로 그룹 전체에 전달함으로써 네트워크 자원을 획기적으로 절감한다.

현업에서는 통신사 IPTV 망 설계, 증권사 시세 배포 시스템, 기업 화상회의 인프라 구축 시 PIM-SM과 IGMP가 필수 기술로 활용된다. IPv6 전환과 클라우드 환경에서의 대안 기술(오버레이 멀티캐스트, SFU)과의 비교 이해도 요구된다.

기술사 시험에서는 멀티캐스트의 개념적 이해뿐 아니라 라우팅 프로토콜 유형(Dense/Sparse Mode)과 IGMP 버전별 동작 메커니즘을 체계적으로 설명할 수 있는지를 평가한다.

### 1. 멀티캐스트(Multicast), 개요

정 의  • 멀티캐스트를 단일 스트림으로 그룹 구성원에게만 선택적으로 전달하는 1:다(多) 통신 방식
       - 유니캐스트 N배 트래픽 없이 라우터가 복사·분기하여 전달

```
[전송 방식 비교]

유니캐스트: 송신자 ──▶ 수신자1
           송신자 ──▶ 수신자2   (N명 = N개 스트림)
           송신자 ──▶ 수신자3

브로드캐스트: 송신자 ──▶ [전체 호스트] (불필요 수신자 포함)

멀티캐스트: 송신자 ──▶ [라우터] ──▶ 수신자1
                           └──────▶ 수신자2  (1개 스트림, 라우터 복사)
                           └──────▶ 수신자3
```

- 멀티캐스트 주소 D클래스(224.0.0.0/4)를 사용하며 라우터가 그룹 트리를 유지·관리한다.

### 2. 멀티캐스트 라우팅 프로토콜 및 IGMP 동작

1) 멀티캐스트 라우팅 프로토콜 유형

| 구분 | 항목 | 설명 |
|------|------|------|
| Dense Mode | 동작 방식 | 전체 플러딩 후 미수신 구간 Prune(가지치기) |
| (PIM-DM) | RP 필요 여부 | 불필요 |
| | 적합 환경 | 소규모·고밀도 수신자 |
| Sparse Mode | 동작 방식 | 명시적 Join으로 필요 경로에만 전달 |
| (PIM-SM) | RP 필요 여부 | 필요 (Rendezvous Point) |
| | 적합 환경 | 대규모·저밀도 수신자 |
| BIDIR-PIM | 동작 방식 | RP 중심 양방향 공유 트리 구성 |
| | 적합 환경 | 다수 송신자 환경 |

- PIM-SM이 인터넷 규모 멀티캐스트의 사실상 표준으로, RP를 통한 공유 트리와 소스 기반 최단경로 트리(SPT) 전환을 지원한다.

2) IGMP(Internet Group Management Protocol) 동작 및 메시지

```
[IGMP 동작 흐름]

호스트                              라우터
  │── Membership Report ──────────▶│  "224.1.1.1 그룹 참여"
  │                                │  (멀티캐스트 전달 시작)
  │◀── Membership Query ───────────│  "그룹 아직 있어요?" (125초 주기)
  │── Membership Report ──────────▶│  "네, 있어요"
  │── Leave Group ────────────────▶│  "나갈게요" (IGMPv2 이상)
  │                                │  (전달 중단)
```

| 구분 | 메시지 | IGMPv1 | IGMPv2 | IGMPv3 |
|------|------|------|------|------|
| 질의 | Membership Query | General | General + Group-Specific | Source-Specific |
| 응답 | Membership Report | 있음 | 있음 | 있음(소스 필터링) |
| 탈퇴 | Leave Group | 없음 | 있음 | 있음 |
| 특징 | - | 기본 | Fast Leave 지원 | SSM 소스 지정 |

- IGMPv3의 SSM(Source-Specific Multicast)으로 원치 않는 송신자 차단이 가능하여 보안성이 향상되었다.

### 3. 현대 멀티캐스트 기술 동향 및 IPv6 비교

| 항목 | IPv4 멀티캐스트 | IPv6 멀티캐스트 | 클라우드/오버레이 |
|------|------|------|------|
| 그룹 관리 | IGMP | MLD (ICMPv6 기반) | 애플리케이션 계층 |
| 라우팅 | PIM-DM/SM | PIM for IPv6 | SFU/CDN 복제 |
| 주소 범위 | 224.0.0.0/4 | FF00::/8 | 논리적 그룹 |
| 보안 | 취약 | CGA 기반 강화 | E2E 암호화 |
| 적합 환경 | 전용 망 | IPv6 전용 망 | 퍼블릭 클라우드 |

- 클라우드 환경에서는 네이티브 멀티캐스트 지원이 제한적이므로 WebRTC SFU(Selective Forwarding Unit)와 CDN 엣지 복제 방식이 현실적 대안으로 부상하고 있다.  "끝"

### 실무 제언

**IPTV Zapping Time 최소화**
- **챌린지**: 채널 전환 시 IGMP Join-응답 지연으로 수초의 블랙스크린 발생
- **제언**: IGMPv2 Fast Leave와 L2 스위치의 IGMP Snooping 활성화로 불필요 포트 차단 및 멀티캐스트 전달 경로 최적화

**기업망 PIM-SM 설계**
- **챌린지**: RP 단일 장애점(SPOF)으로 전체 멀티캐스트 서비스 중단 위험
- **제언**: Anycast RP 또는 PIM BSR(Bootstrap Router) 이중화로 고가용성 확보, RP 장애 시 30초 내 자동 전환 보장

**클라우드 전환 시 멀티캐스트 대체**
- **챌린지**: AWS/Azure 등 퍼블릭 클라우드에서 L2 멀티캐스트 미지원
- **제언**: WebRTC SFU 기반 선택적 전달 또는 CDN 엣지 노드 복제 방식으로 전환하여 멀티캐스트와 동등한 대역폭 효율 달성
