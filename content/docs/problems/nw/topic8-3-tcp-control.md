## 문제

TCP(Transmission Control Protocol)의 신뢰성 있는 전송을 가능케 하는 오류제어, 흐름제어, 혼잡제어에 대하여 설명하시오.

## 해설

### 출제 배경 및 의도

TCP가 "신뢰성 있는 전송"을 보장하는 것은 세 가지 제어 메커니즘의 유기적 협동 결과다. 오류제어는 데이터 무결성을, 흐름제어는 수신자 보호를, 혼잡제어는 네트워크 전체 보호를 각각 담당한다. 이 세 메커니즘은 서로 다른 보호 대상과 제어 변수를 가지며, 실제 전송 윈도우는 흐름제어와 혼잡제어의 최솟값으로 결정된다.

120회 컴시응 문제로 출제된 것은 TCP 내부 동작 원리의 깊은 이해를 요구하는 시험 출제 경향을 반영한다. 현대 TCP 구현(CUBIC, BBR)과의 비교, 클라우드 환경에서의 성능 튜닝 연계까지 서술하면 고득점이 가능하다.

### 1. TCP 신뢰성 제어 메커니즘, 개요

정 의  • TCP 신뢰성 제어를 오류·흐름·혼잡 세 축으로 데이터 무결성·수신자·네트워크를 각각 보호하는 복합 메커니즘
       - 실제 전송 윈도우 = min(rwnd, cwnd)로 흐름제어와 혼잡제어가 연동하여 최종 전송 속도 결정

```
[TCP 신뢰성 삼각형 구조]

         데이터 무결성
              ▲
    오류제어  │  (체크섬·재전송·SACK)
              │
수신자 보호 ──┼── 네트워크 보호
  (흐름제어) │ (혼잡제어)
   rwnd       │   cwnd
              │
    실제 전송 윈도우 = min(rwnd, cwnd)
```

- 세 제어는 독립적 보호 대상을 가지나 실제 전송 속도 결정에서 흐름·혼잡제어가 긴밀히 연동한다.

### 2. 오류제어·흐름제어·혼잡제어 동작

1) 오류제어 (Error Control) - 데이터 무결성 보장

```
[오류 탐지 및 복구 흐름]

정상 전송:
송신 ─▶ [Seg1][Seg2][Seg3] ─▶ 수신
수신 ─▶ [ACK1][ACK2][ACK3] ─▶ 송신

패킷 손실 시 (RTO 방식):
송신 ─▶ [Seg1][  ×  ][Seg3] ─▶ 수신
       (Seg2 손실)
송신 ← RTO 만료 → 재전송: [Seg2]

패킷 손실 시 (Fast Retransmit):
수신 ─▶ [ACK1][ACK1][ACK1] ─▶ 송신
       (Seg2 없어 Dup ACK 3회)
송신 → 즉시 재전송: [Seg2] (타임아웃 불필요)
```

| 구분 | 항목 | 설명 |
|------|------|------|
| 탐지 | 체크섬 | 16비트, 비트 오류 검출 |
| | RTO | RTT 기반 적응적 타임아웃, 만료 시 재전송 |
| | Duplicate ACK | 동일 ACK 3회 수신 시 빠른 재전송 트리거 |
| 복구 | 재전송 | Go-Back-N 또는 Selective Repeat |
| | SACK | 수신 갭 명시, 필요 세그먼트만 선택 재전송 |

- SACK(Selective Acknowledgment)은 수신 측이 갭 정보를 ACK에 포함하여 불필요한 전체 재전송을 방지하고 처리량을 향상시킨다.

2) 흐름제어 (Flow Control) - 수신자 버퍼 보호

```
[슬라이딩 윈도우 동작]

송신자                              수신자 (버퍼 5개)
  │── Seg1 ──▶ │── Seg2 ──▶ │ ─▶ 수신 버퍼: [1][2][3][_][_]
  │◀── ACK3, rwnd=2 ─────────│  (버퍼 잔여 2개)
  │── Seg4 ──▶ │── Seg5 ──▶ │
  │◀── ACK5, rwnd=0 ─────────│  (Zero Window: 버퍼 가득)
  │   [전송 중단 → Persist Timer 동작]
  │◀── ACK5, rwnd=3 ─────────│  (버퍼 비워짐, 재개)
  
윈도우 슬라이딩:
[1✓][2✓][3  ][4  ][5  ] → ACK3 수신 →
[3✓][4  ][5  ][6  ][7  ] (윈도우 오른쪽 이동)
```

- Persist Timer는 Zero Window 상태에서 수신자의 윈도우 갱신 패킷 손실 시 데드락을 방지한다.

3) 혼잡제어 (Congestion Control) - 네트워크 보호

```
[cwnd 동작 그래프]

cwnd │           /← 선형 증가(Congestion Avoidance)
(MSS)│          /
  16 │      SSthresh
   8 │       /
   4 │      /
   2 │     /← 지수 증가(Slow Start)
   1 │────/
     └──────────────────────── RTT
      SS   CA   손실(3Dup)  SS   CA

타임아웃 발생: cwnd=1, SSthresh=cwnd/2 (Slow Start 재시작)
3중복 ACK:    cwnd=SSthresh=cwnd/2 (Fast Recovery, Slow Start 생략)
```

| 구분 | 메커니즘 | 동작 | cwnd 변화 |
|------|------|------|------|
| ① Slow Start | 초기·타임아웃 후 | ACK당 1 MSS 증가 | 지수 증가 (×2/RTT) |
| ② Congestion Avoidance | SSthresh 도달 후 | RTT당 1 MSS 증가 | 선형 증가 (+1/RTT) |
| ③ Fast Retransmit | 3중복 ACK | 즉시 재전송 | - |
| ④ Fast Recovery | Fast Retransmit 후 | cwnd=SSthresh=cwnd/2 | Slow Start 생략 |

- 실제 전송 윈도우 = min(rwnd, cwnd): rwnd(수신자)가 작으면 흐름제어, cwnd(혼잡)가 작으면 네트워크 혼잡이 전송 속도를 제한한다.

### 3. 현대 TCP 혼잡제어 알고리즘 비교

| 항목 | Reno (전통) | CUBIC | BBR (Google, 2016) |
|------|------|------|------|
| 혼잡 감지 | 패킷 손실 | 패킷 손실 | 대역폭·RTT 직접 측정 |
| cwnd 증가 | 선형 | 3차 함수 | 추정 BW 기반 |
| 고속망 효율 | 낮음 | 높음 | 매우 높음 |
| 버퍼 블로트 | 취약 | 취약 | 강함 |
| Linux 기본 | 과거 | 현재 기본 | 옵션 설정 가능 |

- TCP BBR은 패킷 손실이 아닌 네트워크 상태(BW·RTT)를 직접 측정하여 버퍼 블로트 환경에서도 높은 처리량을 유지하며, 클라우드 장거리 전송에서 Reno 대비 최대 2,700배 성능 향상이 보고된 바 있다.  "끝"

### 실무 제언

**클라우드 장거리 TCP 성능 튜닝**
- **챌린지**: 서울-미국 간 RTT 150ms 환경에서 기본 소켓 버퍼(4MB)로는 BDP를 채우지 못해 처리량이 이론값의 10% 수준
- **제언**: BDP = 1Gbps × 0.15s = 18.75MB 계산 기반으로 `tcp_rmem·tcp_wmem`을 최소 20MB로 설정, TCP BBR 적용으로 버퍼 블로트 없이 파이프라인 최대 활용

**데이터센터 내 CUBIC→BBR 전환**
- **챌린지**: 고속(100Gbps) 단거리 환경에서 CUBIC의 패킷 손실 기반 혼잡 감지가 지나치게 공격적으로 cwnd를 낮춰 처리량 저하
- **제언**: 서버 커널에서 `net.ipv4.tcp_congestion_control=bbr` 설정 후 A/B 테스트로 처리량·RTT 변화 모니터링, Fq(Fair Queue) 스케줄러와 병행 적용 권장

**Zero Window 데드락 방지**
- **챌린지**: 수신 애플리케이션 처리 지연으로 rwnd=0 발생, 이후 윈도우 갱신 ACK 손실 시 송수신 모두 무한 대기
- **제언**: Persist Timer(기본 5~60초) 동작 확인 및 애플리케이션 버퍼 처리 로직 최적화, 소켓 `SO_RCVBUF` 증가로 Zero Window 발생 빈도 자체를 줄이는 근본 해결
