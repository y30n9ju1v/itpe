## 해설

### 출제 배경 및 의도

IPv4 주소 공간(약 43억 개)은 인터넷의 폭발적 성장으로 이미 고갈 상태에 이르렀다. IANA의 IPv4 주소 풀은 2011년 소진되었고, 국내 ISP 역시 신규 공인 IP 할당에 제약을 받고 있다. 이러한 상황에서 NAT는 사설 IP 주소를 공인 IP로 변환하여 주소 부족을 해소하는 가장 현실적이고 널리 보급된 기술이다.

IPv6 전환이 지속적으로 추진되고 있으나, 전 세계적으로 IPv4 인프라가 여전히 지배적이다. NAT는 그 과도기를 지탱하는 핵심 메커니즘이며, AWS NAT Gateway·GCP Cloud NAT 등 클라우드 네트워킹에서도 동일한 개념이 적용된다.

이 문제는 IP 주소 절약이라는 기술적 기능과 내부망 은폐라는 보안적 가치를 함께 논하는 능력을 평가한다.

### 1. NAT(Network Address Translation), 개요

정 의  • 사설 IP 주소와 공인 IP 주소를 상호 변환하여 IPv4 주소 부족을 해소하는 네트워크 기술
       - RFC 1918 사설 주소 공간(10.x, 172.16.x, 192.168.x)을 공인 IP에 매핑하여 인터넷 접속을 제공

```
[내부 사설망]                         [공인 인터넷]
PC1: 192.168.1.10 ─┐
PC2: 192.168.1.11 ─┤  NAT 라우터       목적지 서버
PC3: 192.168.1.12 ─┘ [공인IP: 1.2.3.4] ── 8.8.8.8

NAT 변환 테이블 (PAT 예시):
사설IP:포트           공인IP:포트
192.168.1.10:1025 → 1.2.3.4:5001
192.168.1.11:1026 → 1.2.3.4:5002
192.168.1.12:1027 → 1.2.3.4:5003
```

- 단 하나의 공인 IP로 수천 개의 내부 호스트가 동시에 인터넷에 접속할 수 있으며, 내부 IP 구조가 외부에 노출되지 않아 보안성도 향상된다.

### 2. NAT 핵심 기능 및 유형

1) NAT 핵심 기능 분류

| 구분 | 항목 | 설명 |
|------|------|------|
| 주소 변환 | Static NAT | 사설 IP ↔ 공인 IP 1:1 고정 매핑 (서버 공개용) |
| 주소 변환 | Dynamic NAT | 공인 IP 풀(Pool)에서 동적 할당 |
| 포트 다중화 | PAT/NAPT | 하나의 공인 IP + 포트 번호로 다수 호스트 구분 |
| 보안 | 내부망 은폐 | 사설 IP를 외부에 노출하지 않아 방화벽 보완 |
| 로드밸런싱 | DNAT | 복수 서버로 인바운드 요청 분산 |

- 가정용 공유기의 핵심이 PAT(NAPT)이며, 최대 65,535 포트 × 공인 IP 수만큼 동시 세션을 지원한다.

2) NAT 순방향/역방향 변환 흐름

| 구분 | 항목 | 설명 |
|------|------|------|
| 아웃바운드 | 소스 변환(SNAT) | 내부 → 외부 시 소스 IP를 공인 IP로 교체 |
| 인바운드 | 목적지 역변환(DNAT) | 외부 → 내부 시 목적지 IP를 사설 IP로 역변환 |
| 세션 추적 | Connection Tracking | NAT 테이블로 소스/목적지 IP:포트 매핑 유지 |
| 테이블 갱신 | 타임아웃 | 비활성 세션 만료 후 엔트리 삭제 |

```
[아웃바운드] 192.168.1.10:1025 → 8.8.8.8:80
             ↓ NAT 라우터 (SNAT)
             1.2.3.4:5001 → 8.8.8.8:80

[응답]       8.8.8.8:80 → 1.2.3.4:5001
             ↓ NAT 라우터 (DNAT 역변환)
             8.8.8.8:80 → 192.168.1.10:1025
```

- NAT 테이블은 세션 상태를 기억하므로 별도 설정 없이 아웃바운드 응답 패킷이 정확히 내부 호스트로 전달된다.

### 3. NAT 한계 및 발전 방향

| 구분 | 항목 | 설명 |
|------|------|------|
| 한계 | P2P 통신 제한 | 인바운드 개시 연결 불가 (NAT Traversal 필요) |
| 한계 | IPSec 충돌 | 패킷 헤더 변경으로 ESP 무결성 검증 실패 |
| 한계 | VoIP 어려움 | SIP/RTP의 IP:포트 페이로드 수정 필요 (ALG) |
| 해결 | STUN/TURN/ICE | WebRTC 등 P2P 통신을 위한 NAT 트래버설 표준 |
| 발전 | IPv6 전환 | NAT64/DNS64로 IPv4↔IPv6 혼재 환경 지원 |
| 발전 | 클라우드 NAT | AWS NAT Gateway, GCP Cloud NAT 등 관리형 서비스 |

- NAT는 IPv4 주소 고갈의 임시방편이지만, IPv6 전환이 완료될 때까지 기업·클라우드 네트워크의 필수 구성 요소로 유지될 것이다.  "끝"

### 실무 제언

**기업 네트워크 NAT 정책 문서화**
- **챌린지**: NAT 규칙이 증가하면서 어떤 포트가 외부에 열려 있는지 파악이 어려워져 보안 사고 발생
- **제언**: NAT 정책을 CMDB와 연동하여 문서화하고, DNAT 규칙은 방화벽 ACL과 반드시 쌍으로 관리

**클라우드 프라이빗 서브넷 인터넷 접근 설계**
- **챌린지**: AWS VPC 내 프라이빗 서브넷 인스턴스가 인터넷 접근 없이는 패치·업데이트 불가
- **제언**: NAT Gateway를 퍼블릭 서브넷에 배치하고 프라이빗 서브넷 라우팅 테이블에 0.0.0.0/0 → NAT GW 경로 설정

**NAT와 VPN/IPSec 혼재 환경**
- **챌린지**: IPSec 터널 패킷이 NAT 라우터를 경유하면 ESP 무결성 검증 실패로 터널 수립 불가
- **제언**: NAT-T(NAT Traversal, RFC 3948)를 적용하여 ESP 패킷을 UDP 4500 포트로 캡슐화, NAT 통과 허용
