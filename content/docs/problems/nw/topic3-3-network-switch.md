## 해설

### 출제 배경 및 의도

단순한 패킷 전달을 넘어 애플리케이션 콘텐츠를 인식하여 트래픽을 제어하는 스위치 기술은 현대 데이터센터의 핵심 인프라다. L4 스위치는 TCP/UDP 포트 기반 로드밸런싱을, L7 스위치는 HTTP URL·쿠키·SSL 세션 기반의 지능형 트래픽 제어를 제공한다.

마이크로서비스 아키텍처와 클라우드 네이티브 환경이 확산되면서 API 게이트웨이, Kubernetes Ingress Controller, AWS ALB 등 L7 스위칭 기술의 중요성이 더욱 높아졌다. 반면 고성능 저지연이 요구되는 데이터베이스 클러스터나 게임 서버는 L4 스위치를 선호한다.

이 문제는 계층별 스위치 동작 원리와 L4/L7의 기술적 차이, 실무 적용 시나리오를 이해하는 능력을 평가한다.

### 1. 네트워크 스위치(Network Switch), 개요

정 의  • 수신한 패킷을 목적지 주소를 분석하여 해당 포트로만 선택적으로 전달하는 네트워크 중계 장비
       - 허브와 달리 충돌 도메인을 포트 단위로 분리하며, 인식하는 주소 유형에 따라 L2~L7으로 구분

```
[허브] 모든 포트로 신호 복사 → 충돌 도메인 공유
A ──┬── B
    ├── C   ← 패킷이 모든 포트에 도달 (비효율, 충돌)
    └── D

[L2 스위치] MAC 테이블 기반 포트 선택 전달
A ──[스위치]── B  (A→B 통신 시 C, D 포트는 차단)
        ├── C
        └── D
MAC 테이블: {B의 MAC → Port2, C의 MAC → Port3, D의 MAC → Port4}
```

- 스위치는 MAC 주소 학습(Learning), 포워딩(Forwarding), 필터링(Filtering), 플러딩(Flooding)의 4가지 동작으로 패킷을 처리한다.

### 2. OSI 레이어별 스위치 유형

1) 계층별 스위치 분류

| 계층 | 스위치 유형 | 판단 기준 | 주요 기능 |
|------|-----------|---------|---------|
| L1 (물리) | 더미 허브 | 없음 (전기 신호 복사) | 단순 신호 중계, 충돌 도메인 공유 |
| L2 (데이터링크) | L2 스위치 | MAC 주소 | VLAN, STP, MAC 테이블 학습 |
| L3 (네트워크) | L3 스위치/라우터 | IP 주소 | 라우팅, OSPF, VLAN 간 라우팅 |
| L4 (전송) | L4 스위치 | IP + 포트 번호 | 로드밸런싱, 세션 관리, NAT |
| L7 (응용) | L7 스위치/ADC | HTTP URL, Cookie, 콘텐츠 | 콘텐츠 기반 라우팅, SSL 오프로딩 |

- L2/L3 스위치는 전통적 네트워크 인프라에 사용되고, L4/L7 스위치는 서버 팜 앞단에서 트래픽 분산 및 최적화를 담당한다.

2) L2 스위치 핵심 기술

| 구분 | 항목 | 설명 |
|------|------|------|
| 주소 학습 | MAC Learning | 수신 프레임의 소스 MAC과 포트를 테이블에 저장 |
| 루프 방지 | STP/RSTP | 스패닝 트리로 L2 루프를 차단하여 브로드캐스트 스톰 방지 |
| 망 분리 | VLAN | 물리적 포트를 논리적 네트워크로 분리, 브로드캐스트 도메인 제한 |
| 링크 집계 | LACP | 복수 포트를 논리적 하나로 묶어 대역폭 증설 및 이중화 |

- STP(802.1D)는 수렴 시간이 수십 초로 느려 현재는 RSTP(802.1w)와 MSTP(802.1s)로 대체되었다.

### 3. L4 스위치 vs L7 스위치 비교

```
[L4 스위치 판단 근거]          [L7 스위치 판단 근거]
┌────────────────────┐         ┌────────────────────┐
│ IP 헤더: 1.2.3.4  │         │ IP 헤더: 1.2.3.4  │
│ TCP 헤더: Port 80  │         │ TCP 헤더: Port 80  │
│ [페이로드 미확인]  │         │ GET /api/users     │ ← URL
└────────────────────┘         │ Cookie: vip=1      │ ← 세션
         ↓                     │ Host: m.site.com   │ ← 가상호스트
  IP+포트로 서버 선택           └────────────────────┘
                                        ↓
                               콘텐츠 기반 서버 선택
```

| 비교 항목 | L4 스위치 | L7 스위치 |
|----------|---------|---------|
| 분석 계층 | 전송계층 (IP + 포트) | 응용계층 (HTTP, URL, Cookie) |
| 처리 속도 | 빠름 (헤더만 분석) | 상대적으로 느림 (페이로드 분석) |
| 로드밸런싱 | IP/포트 기반 (Round Robin 등) | URL 경로·사용자 세션 기반 |
| SSL 처리 | 불가 (암호화 내용 미해독) | 가능 (SSL 오프로딩) |
| 콘텐츠 인식 | 불가 | URL, HTTP 헤더, 쿠키 인식 |
| 비용 | 저렴 | 고가 |
| 클라우드 대응 | AWS NLB | AWS ALB, Nginx Ingress |
| 활용 사례 | DB 서버 로드밸런싱, 게임 서버 | 마이크로서비스 API 게이트웨이 |

- L7 스위치는 SSL 오프로딩으로 백엔드 서버의 암·복호화 부하를 제거하고, URL 기반 라우팅으로 마이크로서비스의 각 서비스 엔드포인트로 트래픽을 정밀하게 분배하는 현대 클라우드 네이티브 인프라의 핵심 컴포넌트다.  "끝"

### 실무 제언

**마이크로서비스 API 게이트웨이 설계**
- **챌린지**: /api/order, /api/payment 등 경로별로 다른 서버 클러스터에 라우팅해야 하나 L4 로드밸런서로는 URL 기반 분기가 불가능
- **제언**: L7 스위치(Nginx, HAProxy, AWS ALB) 도입으로 URL 경로·HTTP 메서드 기반 라우팅을 구현하고, 인증 토큰 검증과 속도 제한(Rate Limiting)을 API 게이트웨이 레이어에서 일괄 처리

**고성능 저지연 서비스의 L4 스위치 선택**
- **챌린지**: 금융 거래·게임 서버에서 L7 스위치의 페이로드 파싱 지연이 SLA 위반 원인으로 지목
- **제언**: L4 NLB(Network Load Balancer) 사용으로 마이크로초 단위 지연을 달성하고, DSR(Direct Server Return) 모드로 응답 트래픽이 로드밸런서를 우회하도록 설계하여 처리량 극대화

**Kubernetes Ingress를 활용한 L7 트래픽 관리**
- **챌린지**: 컨테이너 환경에서 서비스가 동적으로 생성·삭제되어 정적 L4 로드밸런서 설정으로 대응 불가
- **제언**: Nginx Ingress Controller 또는 AWS ALB Ingress Controller를 Kubernetes 클러스터에 배포하여 서비스 등록/삭제를 자동으로 L7 라우팅 규칙에 반영
