## 문제

네트워크 스위치(Network Switch)와 관련하여 다음을 설명하시오.
(1) 스위치 개요
(2) OSI 참조 모델의 레이어에 따른 스위치 유형
(3) L4 스위치와 L7 스위치 비교

## 해설

> **스위치는 "더 똑똑한 허브"다 — L2는 이름표(MAC)를 보고, L4는 직업(포트), L7은 대화 내용(HTTP 경로)까지 본다**

### 1. 출제 배경 및 의도

현대 데이터센터에서 로드밸런서, 웹방화벽, ADC(Application Delivery Controller)의 핵심이 L4/L7 스위칭 기술입니다. 단순 패킷 전달에서 애플리케이션 인식 전달로 진화한 스위치의 계층별 특성을 이해하는 것이 목표입니다.

### 2. 개념의 본질과 작동 메커니즘

#### 스위치 개요

스위치는 네트워크 장비로, 수신한 패킷을 목적지에 맞는 포트로만 선택적으로 전달합니다. 허브가 모든 포트에 신호를 뿌리는 것(Flooding)과 달리, 스위치는 MAC 테이블을 학습하여 필요한 포트에만 전달해 충돌 도메인을 분리합니다.

```
[허브] 방송: 모든 포트에 동시 전송 → 충돌 발생
A ──┬── B
    ├── C   ← 모두에게 전달 (비효율)
    └── D

[스위치] 유니캐스트: 목적지 포트에만 전달
A ──[스위치]── B  (A→B 통신 시 C, D는 차단)
        ├── C
        └── D
```

#### OSI 레이어별 스위치 유형

| 계층 | 스위치 유형 | 판단 기준 | 주요 기능 |
|---|---|---|---|
| **L1 (물리)** | 더미 허브 | 없음 (전기 신호 복사) | 단순 신호 중계 |
| **L2 (데이터링크)** | L2 스위치 | MAC 주소 | VLAN, STP, MAC 테이블 |
| **L3 (네트워크)** | L3 스위치/라우터 | IP 주소 | 라우팅, OSPF, VLAN 간 라우팅 |
| **L4 (전송)** | L4 스위치 | IP + 포트 번호 | 로드밸런싱, 세션 관리 |
| **L7 (응용)** | L7 스위치/ADC | HTTP URL, Cookie, 콘텐츠 | 콘텐츠 기반 라우팅, SSL 오프로딩 |

#### L4 스위치 vs L7 스위치 비교

```
L4 스위치 판단 근거:
┌─────────────────────┐
│ IP 헤더: 1.2.3.4   │ ← IP 주소 확인
│ TCP 헤더: Port 80  │ ← 포트 번호 확인
│ [페이로드 무시]     │
└─────────────────────┘

L7 스위치 판단 근거:
┌─────────────────────┐
│ IP 헤더: 1.2.3.4   │
│ TCP 헤더: Port 80  │
│ HTTP: GET /api/... │ ← URL 경로 확인
│ Cookie: user=vip   │ ← 쿠키/세션 확인
│ Host: m.site.com   │ ← 가상호스트 확인
└─────────────────────┘
```

| 비교 항목 | L4 스위치 | L7 스위치 |
|---|---|---|
| **분석 계층** | 전송계층 (IP + 포트) | 응용계층 (HTTP, URL, Cookie) |
| **처리 속도** | 빠름 (헤더만 분석) | 상대적으로 느림 (페이로드 분석) |
| **로드밸런싱** | IP/포트 기반 | URL 경로, 사용자 세션 기반 |
| **SSL 처리** | 불가 (암호화 내용 미해독) | 가능 (SSL 오프로딩) |
| **콘텐츠 인식** | 불가 | URL, HTTP 헤더, 쿠키 인식 |
| **비용** | 저렴 | 고가 |
| **대표 제품** | F5 BIG-IP (L4 모드) | F5 BIG-IP, Nginx, HAProxy |
| **활용 사례** | DB 서버 로드밸런싱 | 마이크로서비스 API 게이트웨이 |

#### L7 스위치 동작 예시 (URL 기반 라우팅)

```
클라이언트 요청: GET /api/users  → API 서버 클러스터
                 GET /images/... → CDN/이미지 서버
                 Cookie: vip=1   → 프리미엄 서버
```

### 3. 핵심 키워드 (Must-Have)

- **MAC 테이블:** L2 스위치의 MAC 주소 학습 및 포트 매핑
- **STP (Spanning Tree Protocol):** L2 루프 방지
- **L4 로드밸런싱:** Round Robin, Least Connection, IP Hash
- **L7 콘텐츠 스위칭:** URL 기반, 쿠키 기반, 헤더 기반
- **SSL 오프로딩:** L7 스위치가 SSL 복호화를 대신 처리

### 4. 맥락과 비교

클라우드 환경에서 L7 스위치 기능은 **AWS ALB(Application Load Balancer)**, **GCP Cloud Load Balancing**, **Nginx Ingress Controller(Kubernetes)** 형태로 제공됩니다. 반면 L4 수준은 **AWS NLB(Network Load Balancer)**가 담당합니다.

### 5. 실무 제언

마이크로서비스 아키텍처에서 API 게이트웨이가 L7 스위치 역할을 수행합니다. 서비스별로 `/api/order`, `/api/payment` 경로를 다른 서버로 라우팅하고, 인증 토큰 검증, 속도 제한(Rate Limiting), SSL 종료를 일괄 처리합니다. 기술사 답안에서 L4/L7 비교표와 함께 클라우드 환경에서의 실제 적용 사례(ALB vs NLB)를 제시하면 실무 역량을 효과적으로 어필할 수 있습니다.
