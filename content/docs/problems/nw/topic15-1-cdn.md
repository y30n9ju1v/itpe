## 해설

### 출제 배경 및 의도

인터넷 콘텐츠의 폭발적 증가와 글로벌 사용자 확산으로 원본 서버 단일 운영의 한계가 명확해졌다. 미국에 위치한 서버에서 한국 사용자에게 콘텐츠를 제공하면 200ms 이상의 지연과 원본 서버 과부하가 발생하여 사용자 경험이 현저히 저하된다.

CDN(Content Delivery Network)은 전 세계에 분산된 엣지 서버(PoP)에 콘텐츠를 캐싱하여 사용자 근접 위치에서 제공함으로써 지연 최소화와 원본 서버 부하 절감을 동시에 달성한다. 넷플릭스·유튜브·네이버 등 모든 대형 서비스의 필수 인프라로, 기술사 시험에서 동작 원리·주요 기술·캐싱 방식의 통합 이해가 출제된다.

### 1. CDN(Content Delivery Network), 개요

정 의  • 전 세계에 분산된 엣지 서버(PoP)에 콘텐츠를 캐싱하여 사용자에게 지리적으로 근접한 위치에서 빠르게 제공하는 분산 콘텐츠 전송 네트워크
       - DNS 기반 최적 PoP 선택, HTTP 캐시 제어 헤더 기반 캐싱 관리

```
[CDN 없음]                        [CDN 있음]
한국 사용자                        한국 사용자
    |                                  |
    +--[인터넷, RTT: 200ms+]--+        +--[PoP(서울), RTT: 5ms]
                              |            (캐시 히트!)
                     [미국 원본서버]          |
                                    [미국 원본서버]
                                    (캐시 미스 시에만 접근)
```

- CDN 도입으로 응답 지연 10~40배 감소, 원본 서버 트래픽 90% 이상 절감이 가능하다.

### 2. CDN 동작 원리 및 주요 기술

1) DNS 기반 최적 PoP 선택 흐름

| 구분 | 단계 | 설명 |
|------|------|------|
| ① | 요청 | 사용자가 cdn.example.com/image.jpg 요청 |
| ② | DNS 질의 | CDN DNS 서버가 사용자 IP 기반 최적 PoP IP 반환 |
| ③ | 캐시 확인 | 엣지 서버에서 캐시 히트/미스 판단 |
| ④ | 히트 | 엣지에서 즉시 응답 (5ms 내외) |
| ⑤ | 미스 | 원본 서버 요청 → 캐시 저장 → 응답 |

- Anycast(동일 IP를 여러 PoP가 공유, BGP 최근접 라우팅)로 DNS 없이도 최적 경로를 자동 선택할 수 있다.

2) CDN 주요 기술

| 구분 | 기술 | 설명 |
|------|------|------|
| 복제 | Push 방식 | 원본이 능동적으로 엣지에 배포, 인기 콘텐츠 사전 준비 |
| 복제 | Pull 방식 | 첫 요청 시 원본에서 가져와 캐시, 자동 관리 |
| 부하분산 | DNS/Anycast | 지리·부하 기반 최적 PoP로 트래픽 분산 |
| 최적화 | TCP 연결 최적화 | 원본↔CDN 상시 연결 유지로 연결 오버헤드 제거 |
| 보안 | DDoS 방어 | 분산 용량으로 공격 흡수, WAF 통합 |
| 보안 | SSL/TLS 오프로딩 | CDN에서 암호화 처리, 원본 서버 부하 절감 |

3) 캐싱 방식 및 Cache-Control

| 구분 | 헤더 | 설명 |
|------|------|------|
| TTL 설정 | Cache-Control: max-age=N | N초 동안 캐시 유효 |
| CDN 허용 | Cache-Control: public | CDN 엣지 캐시 허용 |
| CDN 금지 | Cache-Control: private | 브라우저 전용, CDN 캐시 금지 |
| 재검증 | Cache-Control: no-cache | 캐시 전 원본 검증 필요 |
| 금지 | Cache-Control: no-store | 캐시 완전 금지 |
| 변경 감지 | ETag | 콘텐츠 해시로 변경 여부 판단 |

### 3. 캐시 무효화 전략 및 최신 트렌드 비교

1) 캐시 무효화(Cache Invalidation) 전략

| 구분 | 방법 | 장점 | 단점 |
|------|------|------|------|
| ① | API Purge | 즉시 무효화, 정확한 제어 | CDN API 호출 필요 |
| ② | URL Versioning (/app.v2.css) | TTL 무관 즉시 갱신 | URL 관리 복잡 |
| ③ | 단기 TTL | 자동 만료, 구현 단순 | 히트율 저하, 원본 부하 증가 |

2) CDN vs 관련 기술 비교

| 항목 | CDN | 웹 가속기 | 로드밸런서 |
|------|-----|---------|----------|
| 주 기능 | 콘텐츠 분산 캐싱 | 응답 압축·최적화 | 서버 간 부하 분산 |
| 위치 | 전 세계 엣지 | 데이터센터 | 데이터센터 |
| 캐싱 | 핵심 기능 | 보조 기능 | 없음 |
| 적용 범위 | 글로벌 | 단일 DC | 단일 DC |

- 최신 CDN(Cloudflare Workers, AWS Lambda@Edge)은 단순 캐싱을 넘어 엣지에서 코드 실행(Edge Computing)을 지원하며, CDN과 서버리스 컴퓨팅의 경계가 수렴하고 있다.  "끝"

### 실무 제언

**정적 콘텐츠 캐시 전략 최적화**
- **챌린지**: 잦은 배포 시 CDN 캐시와 최신 파일 간 불일치로 사용자에게 구버전 CSS·JS 제공
- **제언**: 정적 자산에 max-age=31536000(1년) + URL 버저닝(hash 기반 파일명) 적용, 배포 시 새 URL 자동 생성으로 캐시 무효화 문제 원천 해결

**캐시 히트율 측정 및 KPI 관리**
- **챌린지**: CDN 도입 효과를 정량적으로 증명하지 못하여 투자 타당성 의문
- **제언**: 캐시 히트율(목표 90%+), 원본 서버 트래픽 감소율, P95 응답 시간을 KPI로 설정, CDN 대시보드에서 실시간 모니터링 체계 수립

**보안 강화 통합 CDN 활용**
- **챌린지**: DDoS 공격이 원본 서버를 직접 타격하여 서비스 전체 마비 위험
- **제언**: CDN의 Anycast DDoS 흡수 기능 + WAF 통합으로 원본 서버 IP 비노출, SSL/TLS 오프로딩으로 원본 서버 암호화 부하 절감, Origin Shield 설정으로 원본 보호 레이어 추가
