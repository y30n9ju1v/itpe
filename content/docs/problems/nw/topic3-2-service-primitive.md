서비스 프리미티브(Service Primitive)에 대해 설명하시오.

## 해설

> **서비스 프리미티브는 OSI 계층 간 "공식 대화 방식"이다 — 마치 직원이 상사에게 업무를 요청하고 보고하는 절차와 같다**

OSI 모델에서 인접한 계층끼리 서비스를 주고받을 때 사용하는 표준화된 인터페이스가 서비스 프리미티브입니다.

### 1. 출제 배경 및 의도

OSI 7계층 모델은 계층 간 독립성을 보장합니다. 이 독립성의 핵심이 **서비스 프리미티브**라는 표준 인터페이스입니다. 계층 내부 구현이 달라도 동일한 프리미티브로 통신할 수 있습니다. 네트워크 프로토콜 설계와 분석의 기초 개념으로, 심화 프로토콜 이해에 필수적입니다.

### 2. 개념의 본질과 작동 메커니즘

#### 서비스 프리미티브의 4가지 유형

```
상위 계층 (N+1)
     │
     │ Request (요청): 서비스를 사용하겠다
     ▼
┌─────────────┐
│  (N) 계층   │
└─────────────┘
     │
     │ Indication (표시): 상대방으로부터 이벤트 알림
     ▼
상위 계층 (N+1)

상위 계층 (N+1)
     │
     │ Response (응답): Indication에 대한 응답
     ▼
┌─────────────┐
│  (N) 계층   │
└─────────────┘
     │
     │ Confirm (확인): Request 처리 결과 통보
     ▼
상위 계층 (N+1)
```

| 프리미티브 | 방향 | 설명 | 비유 |
|---|---|---|---|
| **Request** | 상위→하위 | 상위 계층이 하위 계층에 서비스 요청 | 직원이 상사에게 업무 요청 |
| **Indication** | 하위→상위 | 하위 계층이 상위 계층에 이벤트 통보 | 상사가 직원에게 새 업무 전달 |
| **Response** | 상위→하위 | Indication에 대한 상위 계층의 응답 | 직원이 상사에게 수락 회신 |
| **Confirm** | 하위→상위 | Request 처리 결과를 상위 계층에 통보 | 상사가 업무 완료를 직원에게 보고 |

#### 실제 TCP 연결 설정 시나리오 (3-way handshake와 대응)

```
클라이언트 상위계층    전송계층(TCP)       서버 전송계층      서버 상위계층
      │                   │                   │                  │
      │──Connect.Req ────▶│                   │                  │
      │               (SYN 전송)               │                  │
      │                   │──────────────────▶│                  │
      │                   │               (SYN 수신)              │
      │                   │                   │──Connect.Ind ───▶│
      │                   │                   │◀─Connect.Rsp ────│
      │                   │◀─────────────────(SYN+ACK)           │
      │◀──Connect.Cnf ────│                   │                  │
      │               (ACK 전송)               │                  │
```

#### 서비스 프리미티브 조합 유형

| 서비스 유형 | 사용 프리미티브 | 예시 |
|---|---|---|
| **확인형 서비스** | Req → Ind → Rsp → Cnf | TCP 연결 (신뢰성 보장) |
| **비확인형 서비스** | Req → Ind | UDP 데이터 전송 |
| **공급자 개시** | Ind 만 | 연결 끊김 알림 |

### 3. 핵심 키워드 (Must-Have)

- **4가지 프리미티브:** Request, Indication, Response, Confirm
- **SAP (Service Access Point):** 계층 간 서비스 접근점 (포트 번호와 유사)
- **확인형 vs 비확인형:** 신뢰성 보장 여부에 따른 프리미티브 조합
- **PDU:** 동일 계층 간 교환되는 데이터 단위 (프리미티브와 구분)

### 4. 맥락과 비교

**서비스 프리미티브 vs API**

| 항목 | 서비스 프리미티브 | API (Socket) |
|---|---|---|
| **정의** | OSI 계층 간 이론적 인터페이스 | 실제 구현된 프로그래밍 인터페이스 |
| **표준화** | ISO 표준 (추상적) | OS/언어별 구현 (구체적) |
| **예시** | Connect.Request | connect(), bind(), listen() |

소켓 API의 `connect()` 함수 호출이 내부적으로 Transport 계층에 **Connect.Request** 프리미티브를 발생시키는 것과 같습니다.

### 5. 실무 제언

서비스 프리미티브 개념은 프로토콜 설계 문서(RFC, 3GPP 규격)에서 상태 머신(State Machine)을 기술할 때 사용됩니다. 5G 코어 네트워크의 NF(Network Function) 간 인터페이스 설계나, SDN 컨트롤러와 데이터플레인 간 통신 규격 분석 시 이 개념이 직접 적용됩니다. 기술사 답안에서는 4가지 프리미티브의 정의와 방향, 그리고 TCP 연결 과정의 구체적 사례 대응으로 서술하면 차별화된 답안이 됩니다.
