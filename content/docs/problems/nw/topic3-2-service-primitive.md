## 해설

### 출제 배경 및 의도

OSI 7계층 모델의 핵심 가치는 계층 간 독립성이다. 각 계층은 내부 구현이 달라도 상하위 계층과의 인터페이스만 동일하면 정상 동작하는데, 이 인터페이스를 규정하는 것이 서비스 프리미티브다. 서비스 프리미티브를 이해하지 못하면 계층 간 협력 메커니즘과 프로토콜 상태 머신을 이해하기 어렵다.

실제 RFC 문서와 3GPP 5G 코어 네트워크 규격에서 NF(Network Function) 간 인터페이스를 기술할 때 서비스 프리미티브 개념이 직접 사용된다. SDN 컨트롤러와 데이터플레인 간 제어 메시지 설계에도 동일한 추상화 방식이 적용된다.

이 문제는 OSI 모델의 이론적 기반을 이해하고 TCP 연결 과정과 같은 실제 프로토콜 동작에 대응시키는 능력을 평가한다.

### 1. 서비스 프리미티브(Service Primitive), 개요

정 의  • OSI 인접 계층 간 서비스를 교환하기 위해 표준화된 4가지 유형의 추상적 인터페이스 명령
       - SAP(Service Access Point)를 통해 전달되며, 계층 내부 구현과 무관하게 동일한 방식으로 서비스를 요청·응답

```
[상위 계층 N+1]
      │  Request  (서비스 요청: 상위→하위)
      ▼
┌──────────────┐
│   (N) 계층   │  ←→  동일 계층 간 PDU 교환
└──────────────┘
      │  Indication (이벤트 통보: 하위→상위)
      ▼
[상위 계층 N+1]
      │  Response  (Indication 응답: 상위→하위)
      ▼
┌──────────────┐
│   (N) 계층   │
└──────────────┘
      │  Confirm  (Request 결과 통보: 하위→상위)
      ▼
[상위 계층 N+1]
```

- 4가지 프리미티브의 조합으로 확인형(신뢰성 보장)과 비확인형(비연결성) 서비스를 표준화된 방식으로 표현한다.

### 2. 서비스 프리미티브 4가지 유형

1) 프리미티브 유형 및 방향

| 구분 | 프리미티브 | 방향 | 설명 | 비유 |
|------|-----------|------|------|------|
| 요청 | Request | 상위→하위 | 상위 계층이 하위 계층에 서비스 요청 | 직원이 상사에게 업무 요청 |
| 통보 | Indication | 하위→상위 | 하위 계층이 상위 계층에 이벤트 알림 | 상사가 직원에게 새 업무 전달 |
| 응답 | Response | 상위→하위 | Indication에 대한 상위 계층의 응답 | 직원이 상사에게 수락 회신 |
| 확인 | Confirm | 하위→상위 | Request 처리 결과를 상위 계층에 통보 | 상사가 업무 완료를 직원에게 보고 |

- 4가지 프리미티브는 OSI 계층 간 모든 상호작용을 포괄하며, 프리미티브 조합에 따라 서비스 유형이 결정된다.

2) 서비스 유형별 프리미티브 조합

| 구분 | 서비스 유형 | 사용 프리미티브 | 예시 |
|------|-----------|----------------|------|
| 확인형 | Confirmed Service | Req → Ind → Rsp → Cnf | TCP 연결 설정 |
| 비확인형 | Unconfirmed Service | Req → Ind | UDP 데이터 전송 |
| 공급자 개시 | Provider-initiated | Ind 만 | 연결 끊김 알림 |

```
[TCP 연결 설정: 확인형 서비스]
클라이언트 상위계층    전송계층(TCP)      서버 전송계층      서버 상위계층
      │                   │                   │                  │
      │──Connect.Req ────▶│                   │                  │
      │               (SYN 전송)               │                  │
      │                   │──────────────────▶│                  │
      │                   │            (SYN 수신)                 │
      │                   │                   │──Connect.Ind ───▶│
      │                   │                   │◀─Connect.Rsp ────│
      │                   │◀────────(SYN+ACK)─│                  │
      │◀──Connect.Cnf ────│                   │                  │
      │               (ACK 전송 완료)          │                  │
```

- TCP 3-way handshake의 각 단계는 서비스 프리미티브 교환으로 추상화되며, 소켓 API의 `connect()` 호출이 내부적으로 Connect.Request 프리미티브를 발생시킨다.

### 3. 서비스 프리미티브 vs 소켓 API 비교 및 현대 적용

| 구분 | 항목 | 서비스 프리미티브 | 소켓 API |
|------|------|-----------------|---------|
| 정의 | 추상화 수준 | OSI 계층 간 이론적 인터페이스 | 실제 구현된 프로그래밍 인터페이스 |
| 표준 | 규격 | ISO 표준 (추상적) | OS/언어별 구현 (구체적) |
| 예시 | 연결 요청 | Connect.Request | connect(), bind(), listen() |
| 예시 | 데이터 전송 | Data.Request | send(), write() |
| 적용 | 현대 사례 | 5G NF 인터페이스, SDN | REST API, gRPC |

- 서비스 프리미티브 개념은 5G 코어 네트워크 SBA(Service Based Architecture)에서 NF 간 서비스 명세와 SDN 컨트롤러-데이터플레인 인터페이스 설계에 그대로 적용되고 있다.  "끝"

### 실무 제언

**프로토콜 상태 머신(State Machine) 설계**
- **챌린지**: 복잡한 프로토콜을 구현할 때 계층 간 이벤트 처리 순서를 명확히 정의하지 않아 레이스 컨디션 발생
- **제언**: 서비스 프리미티브를 기반으로 상태 전이 다이어그램을 작성하고, Request/Indication/Response/Confirm 각각의 처리 함수를 명확히 분리하여 구현

**5G 코어 네트워크 NF 인터페이스 분석**
- **챌린지**: 3GPP TS 23.502 규격에서 NF 간 서비스 호출 흐름을 이해하기 어려움
- **제언**: 서비스 프리미티브 관점으로 Nnrf_NFManagement_NFRegister Request/Response를 매핑하면 복잡한 규격서를 체계적으로 해석 가능

**SDN 컨트롤러 남향(Southbound) 인터페이스 설계**
- **챌린지**: OpenFlow 메시지 교환 순서가 불명확하여 컨트롤러-스위치 간 동기화 문제 발생
- **제언**: OpenFlow의 PacketIn(Indication), FlowMod(Request), BarrierReply(Confirm) 메시지를 서비스 프리미티브로 추상화하여 인터페이스 명세를 표준화
