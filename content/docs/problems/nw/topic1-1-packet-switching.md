## 문제

교환방식에 있어서 회선교환(Circuit Switching)방식과 패킷교환(Packet Switching) 방식에 대하여 설명하고, 음성데이터나 영상데이터와 같은 실시간 트래픽 서비스를 패킷교환방식의 네트워크에서 제공하고자 할 때 발생할 수 있는 문제점 및 해결방안에 대하여 설명하시오.

## 해설

### 출제 배경 및 의도

회선교환과 패킷교환은 현대 통신 인프라의 두 가지 핵심 철학을 대비한다. 유선 전화망(PSTN)은 회선교환 위에 세워졌고, 인터넷은 패킷교환 위에 세워졌다. 두 방식은 단순한 기술 선택이 아니라 자원 활용 효율, 신뢰성, 비용 구조의 근본적 차이를 내포한다.

120회 응용 문제로, 네트워크 교환 방식의 근본 원리와 VoIP·화상회의 등 실시간 서비스를 IP망에서 제공할 때의 기술적 도전을 묻는다. QoS, RSVP, DiffServ 등 심화 주제와 연결되며, 실무에서 IP 기반 멀티미디어 서비스 설계 능력을 평가한다.

클라우드 기반 화상회의(Zoom, Teams)와 5G 기반 미디어 스트리밍이 일상화된 현재, 패킷교환망에서의 실시간 트래픽 QoS 보장 기술은 네트워크 엔지니어의 핵심 역량이 되었다.

### 1. 교환 방식, 개요

정 의  • 네트워크에서 데이터를 목적지까지 전달하기 위해 경로와 자원을 할당하는 방식
       - 회선교환은 연결 전 전용 경로를 예약하고, 패킷교환은 데이터를 분할하여 동적으로 전달

```
[회선교환]
발신자 ─── 교환기A ─── 교환기B ─── 교환기C ─── 수신자
          [──────── 전용 회선 설정 (연결 지속) ────────]
  1단계: 회선 설정 (Call Setup)
  2단계: 데이터 전송 (독점 대역폭)
  3단계: 회선 해제 (Call Teardown)

[패킷교환]
데이터 → [패킷1][패킷2][패킷3] 으로 분할
패킷1: A → B → D (경로1)
패킷2: A → C → D (경로2)  ← 서로 다른 경로 가능
패킷3: A → B → D (경로1)
수신측: 순서 재조합 후 원본 복원
```

- 회선교환은 안정적 지연을 보장하나 자원 낭비가 크고, 패킷교환은 자원을 효율적으로 공유하나 가변 지연이 발생한다.

### 2. 교환 방식 핵심 비교

1) 회선교환 vs 패킷교환

| 구분 | 항목 | 회선교환 | 패킷교환 |
|------|------|----------|----------|
| 자원 | 대역폭 할당 | 고정 전용 (사용 안 해도 점유) | 가변 공유 (효율적) |
| 자원 | 연결 방식 | 연결 전 전용 회선 예약 | 필요 시 동적 할당 |
| 성능 | 지연 | 일정 (낮은 지연 보장) | 가변 (큐잉 지연 발생) |
| 성능 | 신뢰성 | 회선 끊기면 전체 실패 | 경로 우회 가능 |
| 비용 | 구축비 | 고가 (전용 자원) | 저렴 (자원 공유) |
| 사례 | 대표 기술 | PSTN(전화망), ISDN | 인터넷(IP), LTE |

- 패킷교환이 자원 효율과 비용 측면에서 우월하나, 실시간 트래픽에는 별도의 QoS 대책이 필요하다.

2) 패킷교환 내 가상회선 vs 데이터그램

| 구분 | 항목 | 가상회선 | 데이터그램 |
|------|------|----------|------------|
| 경로 | 설정 방식 | 연결 전 논리 경로 설정 | 패킷마다 독립 라우팅 |
| 순서 | 보장 여부 | 보장 | 미보장 |
| 오버헤드 | 설정/해제 | 있음 | 없음 |
| 사례 | 대표 기술 | ATM, Frame Relay | IP |

- 데이터그램 방식인 IP는 유연성이 높으나 순서 역전 및 가변 지연에 대한 상위 계층 대처가 필요하다.

### 3. 실시간 트래픽 문제점 및 QoS 해결방안

| 문제 | 원인 | 영향 | 해결 기법 |
|------|------|------|-----------|
| 지연(Delay) | 라우터 큐잉·전파 지연 누적 | VoIP 통화 끊김(150ms 초과) | RSVP, 우선순위 큐 |
| 지터(Jitter) | 패킷마다 다른 경로·지연 | 영상 재생 버벅임 | 지터 버퍼(Playout Buffer) |
| 패킷 손실 | 버퍼 오버플로우 시 드롭 | 음질 저하, 화면 깨짐 | FEC, RED |
| 순서 역전 | 다른 경로 도착 순서 불일치 | 재조합 오류 | 시퀀스 번호 + 버퍼링 |

```
┌─────────────────────────────────────────────────────┐
│                 QoS 보장 아키텍처                    │
├──────────────┬──────────────────┬───────────────────┤
│   IntServ    │    DiffServ      │      RSVP         │
│  (개별 흐름) │ (트래픽 분류)    │  (경로 자원 예약) │
│  per-flow    │ DSCP EF 마킹     │  Path/Resv 메시지 │
│  RFC 1633   │ RFC 2474         │  RFC 2205         │
└──────────────┴──────────────────┴───────────────────┘
```

- DSCP EF(Expedited Forwarding) 마킹으로 음성·영상 패킷을 최우선 큐에 배치하고, WAN 구간은 SD-WAN 애플리케이션 인식 라우팅으로 실시간 트래픽을 저지연 경로에 자동 우회시키는 현대적 설계가 표준이다.  "끝"

### 실무 제언

**기업 화상회의 도입 시 QoS 설계**
- **챌린지**: 기존 네트워크에 Zoom·Teams 트래픽이 혼재되어 VoIP 품질 저하 발생
- **제언**: 라우터·스위치에 DSCP EF 마킹 정책을 설정하고, 대역폭의 20~30%를 실시간 트래픽 전용 큐로 예약

**SD-WAN 기반 실시간 트래픽 최적화**
- **챌린지**: WAN 구간 혼잡으로 지터·패킷 손실 빈발, 사용자 불만 증가
- **제언**: SD-WAN의 애플리케이션 인식 라우팅으로 실시간 트래픽을 지연이 낮은 경로로 자동 스티어링하고, 다중 링크 본딩으로 손실 복구

**클라우드 UCaaS 전환 시 네트워크 아키텍처**
- **챌린지**: 전통적 MPLS 기반 WAN은 클라우드 서비스의 인터넷 직접 접근에 비효율적
- **제언**: Local Internet Breakout과 SASE(Secure Access Service Edge) 도입으로 클라우드 트래픽 최적화 경로 확보
