## Part 1. 데이터 교환의 두 가지 철학 — 회선교환과 패킷교환

### 전화와 인터넷은 왜 이렇게 다른가요?

1960년대 국제 전화를 걸려면 꽤 복잡한 절차를 거쳐야 했습니다. 수화기를 들면 교환원이 나왔고, "서울에서 뉴욕으로 연결해 주세요"라고 부탁하면 교환원이 서울 ↔ 태평양 ↔ 뉴욕을 잇는 물리적인 회선을 하나하나 손으로 연결했습니다. 일단 연결되면 그 회선은 통화가 끝날 때까지 두 사람만을 위해 독점적으로 확보됩니다. 중간에 아무 말이 없이 침묵이 흘러도, 그 회선은 여전히 두 사람을 위해 예약되어 있습니다. 이것이 **회선교환(Circuit Switching)**의 본질입니다.

회선교환은 안정적이고 예측 가능합니다. 연결이 맺어진 순간부터 두 단말 사이에는 일정한 대역폭이 보장되고, 지연이 일정합니다. 그래서 전화 통화처럼 실시간 음성을 주고받는 데 탁월합니다. 그러나 치명적인 비효율이 있습니다. "여보세요..." 하고 말하는 사이의 침묵, 말을 고르는 2초, 상대방이 이야기하는 동안 내가 아무것도 보내지 않는 시간 — 이 모든 순간에도 회선은 아무도 쓰지 않은 채 예약된 상태로 낭비됩니다. 대용량·고속 데이터 전송에는 비용이 기하급수적으로 올라갑니다.

인터넷은 완전히 다른 철학을 택했습니다. 내가 보내는 데이터를 마치 택배 상자처럼 작은 조각(패킷)으로 잘라서, 각 조각에 "받는 사람 주소"와 "몇 번째 조각인지"를 적은 라벨(헤더)을 붙입니다. 그리고 각 조각은 네트워크 안에서 독립적으로, 그때그때 가장 빠른 길을 스스로 찾아 목적지로 향합니다. 어떤 조각은 태평양을 건너고, 다른 조각은 대서양을 건너도 됩니다. 목적지에서 번호를 보고 순서대로 다시 조립하면 됩니다. 이것이 **패킷교환(Packet Switching)**입니다.

```
[회선교환 — 전용 고속도로 예약]
서울 ══════════════════════════════ 뉴욕
     (통화 내내 이 차선은 당신 전용 — 쉬어도 점유)

[패킷교환 — 택배처럼 각자 최적 경로]
서울 ──[P1]──► 라우터A ──► 라우터C ──────────► 뉴욕
     ──[P2]──► 라우터A ──► 라우터D ──► 라우터E ──► 뉴욕
     ──[P3]──► 라우터B ──► 라우터E ──────────────► 뉴욕
(패킷마다 다른 경로 — 자원을 여러 사람이 공유)
```

패킷교환의 가장 큰 장점은 **자원의 공유**입니다. 내가 보내지 않는 순간에는 그 대역폭을 다른 사람이 쓸 수 있습니다. 인터넷이 수십억 명이 동시에 쓰는데도 동작하는 비결이 바로 여기에 있습니다.

그러나 패킷교환에도 문제가 있습니다. 패킷마다 경로를 탐색해야 하므로 지연이 발생하고, 그 지연이 일정하지 않습니다(지연 변이, Jitter). 실시간 화상회의에서 누군가의 목소리가 갑자기 버벅이는 것, 온라인 게임에서 렉이 걸리는 것 — 모두 이 패킷교환 방식의 지연 특성 때문입니다.

> **이 긴장감이 QoS, CDN, 버퍼링 같은 수많은 네트워크 기술의 출발점입니다.** 패킷교환 네트워크에서 실시간 트래픽을 안정적으로 제공하려면 대역폭 보장, 패킷 우선순위 제어, 지연 최소화 등 추가 기술이 반드시 필요합니다.

| 비교 항목 | 회선교환 | 패킷교환 |
|---|---|---|
| 자원 사용 | 전용 독점 | 다수가 공유 |
| 지연 | 연결 후 일정 | 가변적 (Jitter 발생) |
| 실시간성 | 우수 | 추가 기술 필요 |
| 대역폭 효율 | 낮음 (침묵 시 낭비) | 높음 |
| 사례 | 전화망(PSTN) | 인터넷 |

## Part 2. NAT — 하나의 공인 IP로 수백 대를 인터넷에 연결하는 마법

### 공유기 안의 비밀

집에 공유기가 있다면 스마트폰, 노트북, TV, 냉장고가 모두 동시에 인터넷을 씁니다. 그런데 통신사가 우리 집에 준 공인 IP는 딱 하나뿐입니다. 어떻게 이것이 가능한 걸까요?

비유를 들어 봅시다. 작은 회사가 하나 있고 이 회사의 대표 전화번호는 하나입니다. 외부에서 전화가 오면 접수 직원이 받아서 "1번 직원한테는 내선 101번, 2번 직원한테는 내선 102번"으로 연결해 줍니다. 회사 내부에서는 각자의 내선 번호로 통화하지만, 외부 세계는 대표 번호 하나만 알고 있습니다. **NAT(Network Address Translation)**는 바로 이 접수 직원의 역할을 합니다.

공유기(NAT 장비) 안쪽에서는 `192.168.x.x` 같은 **사설 IP**를 씁니다. 이 주소는 전 세계 어느 집이나 회사에서도 같은 번호대를 쓰기 때문에 인터넷에서는 쓸 수 없습니다. 패킷이 공유기를 통해 인터넷으로 나갈 때, 공유기는 출발지 주소를 **공인 IP + 고유 포트 번호**로 바꿔치기합니다. 돌아오는 응답 패킷은 포트 번호를 보고 원래 내부 장비에게 되돌려 줍니다. 이 매핑 정보를 **NAT Binding Table**이라 합니다.

```
[내부망]                        [NAT 공유기]           [인터넷]
192.168.0.10:3000  ─┐
192.168.0.11:3001  ─┤── 203.0.113.1:60001 ──────► 서버 80
192.168.0.12:3002  ─┘   (공인 IP + 포트로 변환)

NAT Binding Table:
사설 IP:포트          ↔  공인 IP:포트
192.168.0.10:3000  ↔  203.0.113.1:60001
192.168.0.11:3001  ↔  203.0.113.1:60002
192.168.0.12:3002  ↔  203.0.113.1:60003
```

NAT의 세 가지 핵심 기능은 이렇습니다. **IP Masquerading**은 가장 일반적인 형태로, 다수의 사설 IP를 하나의 공인 IP로 변환하는 1:N 방식입니다. **Port Forwarding**은 반대 방향입니다. 외부에서 들어오는 트래픽을 내부의 특정 서버로 전달합니다. 집에서 게임 서버나 NAS를 운영할 때 공유기 설정에서 "포트 열기"를 하는 것이 바로 Port Forwarding입니다. **Load Balancing**은 하나의 공인 IP로 들어오는 트래픽을 여러 내부 서버에 분산하는 것으로, 대형 서비스에서 활용됩니다.

> **NAT는 IPv4 주소 고갈 문제의 임시 해결책으로 등장했지만, 워낙 널리 퍼져서 IPv6가 도입된 지금도 여전히 사용됩니다.** NAT 덕분에 가정과 기업이 단 하나의 공인 IP로 수백 대의 기기를 운용할 수 있게 되었고, 이는 인터넷의 폭발적 성장을 가능하게 한 핵심 기술 중 하나입니다.

## Part 3. OSI 7 계층 — 언어가 다른 나라들이 대화하는 규칙서

### 왜 삼성 스마트폰과 애플 맥북이 같은 와이파이로 통신할 수 있을까요?

1970년대만 해도 IBM 컴퓨터와 DEC 컴퓨터는 서로 통신이 불가능했습니다. 각 제조사가 독자적인 통신 방식을 사용했기 때문입니다. 이 혼돈을 정리하기 위해 ISO(국제표준화기구)는 1984년 **OSI(Open Systems Interconnection) 7 계층 참조 모델**을 발표했습니다. "어떤 제조사, 어떤 운영체제의 장비든 이 7개의 규칙을 따르면 서로 통신할 수 있다"는 약속이었습니다.

7개의 계층은 마치 편지를 보내는 과정처럼 생각할 수 있습니다. 서울의 A가 뉴욕의 B에게 편지를 보낸다고 합시다. A는 편지를 쓰고(응용), 암호화하고(표현), 대화 채널을 만들고(세션), 우체국에 맡기고(전송), 우체국은 경로를 정하고(네트워크), 각 구간 배달을 하고(데이터링크), 실제 도로 위를 달립니다(물리). B 측에서는 이 과정이 역순으로 펼쳐집니다.

```
  송신                                수신
  ┌──────────────────┐              ┌──────────────────┐
7 │ Application (응용)│ ◄── 가상 통신 ──► │ Application (응용)│
6 │ Presentation(표현)│              │ Presentation(표현)│
5 │ Session (세션)   │              │ Session (세션)   │
4 │ Transport (전송) │  ← Segment → │ Transport (전송) │
3 │ Network (네트워크)│  ← Packet  → │ Network (네트워크)│
2 │ DataLink(데이터링크│  ← Frame   → │ DataLink(데이터링크│
1 │ Physical (물리)  │  ← Bit     → │ Physical (물리)  │
  └──────────────────┘              └──────────────────┘
     캡슐화 (헤더 추가)                역캡슐화 (헤더 제거)
```

각 계층의 역할을 이해하면 네트워크 전체가 보입니다.

**물리 계층(Physical, 1계층)**은 전기 신호, 빛, 전파처럼 실제 물리 매체 위에서 0과 1을 전달합니다. "신호를 어떻게 보낼 것인가"가 관심사입니다. 케이블, 커넥터, 허브가 여기에 속합니다.

**데이터링크 계층(DataLink, 2계층)**은 같은 네트워크 안의 인접한 두 장치 사이에서 신뢰성 있게 프레임을 전달합니다. MAC 주소를 이용하며, 스위치와 브리지가 여기서 동작합니다. 이더넷, Wi-Fi, PPP 같은 프로토콜이 2계층에 해당합니다.

**네트워크 계층(Network, 3계층)**은 서로 다른 네트워크 사이를 가로질러 패킷이 목적지까지 도달하도록 경로를 결정합니다. IP 주소를 기반으로 하며, 라우터가 여기서 동작합니다. 인터넷의 핵심 계층입니다.

**전송 계층(Transport, 4계층)**은 두 호스트의 응용 프로그램 사이에서 종단 간(end-to-end) 신뢰성 있는 데이터 전달을 담당합니다. TCP와 UDP가 여기에 속하며, 포트 번호로 어떤 응용 프로그램에게 데이터를 전달할지 구분합니다.

**세션 계층(Session, 5계층)**은 통신 세션(대화)을 시작하고, 유지하고, 종료하는 역할을 합니다. 긴 파일 전송 중 연결이 끊겼을 때 처음부터 다시 보내지 않고 중간부터 재개할 수 있게 체크포인트를 관리합니다.

**표현 계층(Presentation, 6계층)**은 데이터를 서로 다른 시스템이 이해할 수 있는 형식으로 변환합니다. 암호화, 압축, 문자 인코딩(ASCII, Unicode) 변환이 여기서 일어납니다.

**응용 계층(Application, 7계층)**은 우리가 직접 다루는 소프트웨어 영역입니다. 이메일(SMTP), 파일 전송(FTP), 웹 브라우징(HTTP), 도메인 조회(DNS)가 모두 7계층 프로토콜입니다.

> **현실에서 OSI 7계층 모델은 참조 모델이고, 실제 인터넷은 4계층의 TCP/IP 모델을 씁니다.** 그러나 OSI 모델은 네트워크 문제를 진단하고 설명하는 공통 언어로 여전히 필수적입니다. "L3 문제다", "L7 스위치로 처리한다"는 말은 모두 OSI 계층 번호를 기반으로 합니다.

## Part 4. 서비스 프리미티브 — 계층끼리 주고받는 신호 체계

### 계층 간 서비스 요청은 어떻게 이루어질까요?

OSI 모델에서 각 계층은 바로 아래 계층의 서비스를 사용합니다. 그런데 이 "서비스를 요청하고 응답받는" 과정을 어떻게 표현할까요? 그것이 바로 **서비스 프리미티브(Service Primitive)**입니다.

택배 시스템으로 비유해 봅시다. 고객(상위 계층)이 택배 회사(하위 계층)에게 "이 짐을 보내주세요"라고 부탁하면, 택배 회사가 수신자 측 담당 직원에게 짐이 왔다고 알리고, 수신자가 확인하면 발신자에게 완료 통보를 합니다. 이 네 단계가 바로 네 가지 서비스 프리미티브입니다.

**Request(요청)**는 상위 계층이 하위 계층에게 서비스를 요청할 때 씁니다. 고객이 "이 짐 보내주세요"라고 말하는 순간입니다. **Indication(표시)**은 서비스 제공자가 반대편 상위 계층에 이벤트를 알릴 때입니다. 수신 측 택배 직원이 "짐 왔습니다"라고 수신자에게 알리는 것입니다. **Response(응답)**는 수신 측 상위 계층이 표시에 대해 응답하는 것입니다. 수신자가 "네, 받겠습니다"라고 서명하는 단계입니다. **Confirm(확인)**은 원래 요청한 상위 계층에게 서비스 완료를 통보합니다. 발송인에게 "배달 완료" 문자가 오는 것과 같습니다.

이 네 가지를 조합하면 `T.CONNECT.request`처럼 표현합니다. `T`는 Transport Layer, `CONNECT`는 서비스 종류, `request`는 프리미티브 종류를 뜻합니다. 계층, 서비스, 방향 — 세 가지를 한 번에 표현하는 간결한 표기법입니다.

## Part 5. 네트워크 스위치 — 우체국 직원이 봉투를 얼마나 깊이 뜯어보느냐의 차이

### 스위치는 왜 계층마다 다른가요?

우체국 직원이 편지를 배달할 때를 상상해 봅시다. 가장 간단한 방법은 겉봉투에 적힌 주소만 보고 배달하는 것입니다. 더 정확하게 배달하려면 봉투를 열어서 안에 어떤 종류의 내용인지 확인할 수도 있습니다. 하지만 봉투를 뜯으면 뜯을수록 시간이 더 걸립니다. 네트워크 스위치도 정확히 같은 트레이드오프를 가지고 있습니다.

**L2 스위치**는 데이터링크 계층(2계층)에서 동작하며 MAC 주소만 봅니다. 같은 네트워크 안의 기기들 사이에서 프레임을 전달하는 역할입니다. 전원을 켜면 MAC 주소를 학습(Learning)하고, 목적지 MAC을 알면 해당 포트로만 전달(Forwarding)하고, 모르면 모든 포트로 뿌립니다(Flooding). 빠르고 저렴하지만 다른 네트워크와 연결하거나 서비스 유형을 구분하는 기능은 없습니다.

**L3 스위치**는 IP 주소까지 봅니다. 라우터와 같은 기능을 하지만, 하드웨어로 처리해서 훨씬 빠릅니다. 서로 다른 VLAN(네트워크) 사이를 연결할 때 씁니다.

**L4 스위치**는 IP 주소와 포트 번호(TCP/UDP)까지 봅니다. 포트 번호로 서비스 종류를 구분할 수 있어서 로드밸런싱(부하 분산)에 활용됩니다. HTTP 트래픽(포트 80)은 웹 서버 그룹으로, FTP 트래픽(포트 21)은 파일 서버 그룹으로 분산하는 식입니다.

**L7 스위치**는 패킷의 내용(페이로드)까지 들여다봅니다. URL 경로, 쿠키, HTTP 헤더까지 분석해서 정교한 분산이 가능합니다. `/images` 경로의 요청은 이미지 서버로, `/api` 경로의 요청은 API 서버로 보내는 식입니다. 콘텐츠 기반 보안 필터링(DoS 방어, 악성 요청 차단)도 가능합니다. 하지만 패킷을 깊이 분석할수록 처리 부하와 지연이 커집니다.

```
L2 스위치: [MAC header | IP | TCP | Payload]
            ↑ 여기까지만 본다

L3 스위치: [MAC header | IP header | TCP | Payload]
                         ↑ 여기까지

L4 스위치: [MAC header | IP header | TCP header | Payload]
                                     ↑ 포트 번호까지

L7 스위치: [MAC header | IP header | TCP header | Payload]
                                                  ↑ 내용까지 본다
```

> **높은 계층을 볼수록 정교한 제어가 가능하지만, 성능 비용도 높아집니다.** 실무에서는 비용, 성능, 요구 기능을 고려해서 적절한 계층의 스위치를 선택합니다.

## Part 6. 오류 제어 — 데이터가 깨지면 어떻게 고치나요?

### 네트워크에서 비트 오류는 피할 수 없습니다

전기 신호는 완벽하지 않습니다. 케이블을 타고 가는 동안 전자기 간섭, 신호 감쇠, 열잡음 등으로 0이 1로 바뀌거나 1이 0으로 바뀌는 **비트 오류(Bit Error)**가 발생합니다. 무선 네트워크는 더 심합니다. 오류를 어떻게 처리할 것인가는 통신 시스템 설계의 핵심 문제입니다.

오류 제어 방식은 크게 두 가지로 나뉩니다. 오류를 스스로 수정하느냐(전진 오류 수정, FEC), 아니면 오류를 검출해서 재전송을 요청하느냐(후진 오류 수정, BEC/ARQ).

**FEC(Forward Error Correction)**는 데이터에 충분한 잉여 정보를 함께 보내서, 수신 측이 오류가 생겨도 **스스로 복원**할 수 있게 합니다. 위성 통신을 생각해 보세요. 화성에 보낸 탐사선이 데이터를 잘못 받았을 때 "다시 보내줘"라고 요청하면, 신호가 지구에 도달하는 데만 수십 분이 걸립니다. 재전송 기반 방식은 사용할 수가 없습니다. 이런 상황에서 FEC가 필수입니다. **해밍코드(Hamming Code)**나 **리드-솔로몬 코드** 등이 FEC의 대표 기술입니다.

**해밍코드**는 데이터 비트 사이에 패리티 비트를 전략적으로 삽입합니다. 패리티 비트는 2의 거듭제곱 위치(1, 2, 4, 8번 위치...)에 배치되고, 각 패리티 비트는 특정 데이터 비트들의 홀짝을 맞춥니다. 오류가 발생하면 어느 패리티 비트가 맞지 않는지를 조합해서(이것을 신드롬이라 합니다) 몇 번 위치가 틀렸는지 정확히 알 수 있습니다. 마치 "1번 패리티만 이상하다"면 1번 위치, "1번과 2번 패리티가 이상하다"면 3번 위치 식으로 위치를 콕 짚어낼 수 있습니다.

**CRC(Cyclic Redundancy Check)**는 오류를 검출하는 기법입니다. 전송할 데이터를 특정 생성 다항식으로 이진 나눗셈해서 나머지(FCS, Frame Check Sequence)를 구합니다. 이 나머지를 데이터 뒤에 붙여서 전송하면, 수신 측도 같은 나눗셈을 해서 나머지가 0이 나오면 정상, 0이 아니면 오류로 판단합니다. 이더넷 프레임마다 뒤에 붙는 4바이트 FCS가 바로 CRC입니다.

**BEC/ARQ(Backward Error Correction / Automatic Repeat reQuest)**는 오류를 검출하면 재전송을 요청합니다. 유선 인터넷처럼 재전송 비용이 낮은 환경에서는 이 방식이 더 효율적입니다. ARQ에는 세 가지 방식이 있습니다.

**Stop-and-Wait ARQ**는 패킷 하나를 보내고 ACK가 올 때까지 기다렸다가 다음 패킷을 보냅니다. 구현이 가장 단순하지만, ACK를 기다리는 동안 회선이 놀기 때문에 효율이 매우 낮습니다. 위성처럼 왕복 지연이 큰 환경에서는 거의 대부분의 시간을 기다리는 데 씁니다.

**Go-Back-N ARQ**는 ACK를 기다리지 않고 N개의 패킷을 연속으로 보냅니다. 오류가 발생한 패킷이 있으면 그 패킷부터 이후의 것들을 **모두 다시 보냅니다**. 수신 측은 순서에 맞지 않는 패킷은 버리고 순서를 기다립니다.

**Selective Repeat ARQ**는 오류가 난 패킷만 콕 집어서 재전송합니다. 수신 측이 순서가 맞지 않는 패킷도 버퍼에 보관했다가, 빠진 패킷이 도착하면 순서를 맞춰 응용 계층에 올려줍니다. 가장 효율적이지만 수신 버퍼가 필요합니다.

| 방식 | 재전송 범위 | 수신 버퍼 | 효율 |
|---|---|---|---|
| Stop-and-Wait | 1개 (즉시) | 불필요 | 낮음 |
| Go-Back-N | 오류 이후 전부 | 불필요 | 중간 |
| Selective Repeat | 오류 난 것만 | 필요 | 높음 |

## Part 7. ARP — IP 주소를 MAC 주소로 번역하는 동네 우체국

### 인터넷 주소와 물리 주소는 왜 두 가지인가요?

인터넷에서 데이터를 보내려면 IP 주소가 필요합니다. 그런데 실제로 같은 네트워크 안에서 데이터를 전달할 때는 IP 주소가 아니라 **MAC 주소**를 씁니다. 왜 두 가지 주소가 필요할까요?

IP 주소는 논리적인 주소입니다. 이사를 가거나 네트워크가 바뀌면 달라집니다. 반면 MAC 주소는 네트워크 카드 제조 시 하드웨어에 박혀 나오는 물리 주소로 (이론상) 전 세계에서 유일합니다. IP 주소는 "서울시 강남구 xxx로"처럼 어디 있는지를 나타내고, MAC 주소는 "건물 안 정확한 방 번호" 같은 것입니다. 같은 건물 안에서 움직이려면 방 번호가 필요합니다.

**ARP(Address Resolution Protocol)**는 "이 IP 주소를 가진 장비의 MAC 주소를 알려줘"라고 네트워크에 브로드캐스트로 물어보는 프로토콜입니다.

```
Host A (192.168.0.1)                    Host B (192.168.0.2)
    │                                        │
    │── ARP Request (브로드캐스트) ──────────►│
    │   "192.168.0.2 MAC 주소가 뭐야?"  ─────►│ (모두에게 질문)
    │                                        │
    │◄── ARP Reply (유니캐스트) ─────────────│
    │   "나야! MAC: AA:BB:CC:DD:EE:FF"       │
    │                                        │
    │ (이후 ARP 캐시에 저장, 일정 시간 유지)    │
```

**RARP(Reverse ARP)**는 반대 방향입니다. "내 MAC 주소는 이것인데, 내 IP 주소가 뭐죠?"라고 묻습니다. 디스크 없이 부팅하는 씬 클라이언트가 DHCP 이전에 자신의 IP 주소를 알아내기 위해 사용했습니다.

ARP는 편리하지만 보안상 심각한 취약점이 있습니다. ARP Reply를 검증하는 메커니즘이 전혀 없기 때문에, 악의적인 공격자가 **가짜 ARP Reply**를 보내서 네트워크를 속일 수 있습니다.

**ARP Spoofing**은 공격자가 게이트웨이(공유기)의 IP에 자신의 MAC 주소를 매핑한 가짜 Reply를 보냅니다. 피해자의 ARP 캐시가 오염되면, 피해자가 외부 인터넷으로 보내는 모든 패킷이 공격자를 거쳐 나가게 됩니다. 공격자는 내용을 도청하거나 변조할 수 있습니다(중간자 공격, MITM).

**ARP Redirect**는 공격자가 실제 라우터보다 더 빠르게 ARP Reply를 보내서, 네트워크 트래픽을 자신을 통해 흐르게 만드는 변형 기법입니다.

대응 방법으로는 스위치의 **Dynamic ARP Inspection(DAI)** 기능(DHCP 바인딩 테이블과 대조해서 가짜 ARP를 차단), **정적 ARP 엔트리**(중요한 서버-게이트웨이의 MAC 주소를 수동으로 고정), **VLAN 분리**(브로드캐스트 도메인을 제한해서 공격 범위 축소) 등이 있습니다.

## Part 8. IPv6 — 주소가 부족해서 더 넓은 땅을 찾아 이사

### 43억 개도 부족하다고요?

IPv4는 32비트 주소를 사용합니다. 2³² = 약 43억 개의 주소를 만들 수 있습니다. 1980년대에 인터넷을 설계하던 사람들은 43억 개면 충분하다고 생각했습니다. 그 시절엔 인터넷이 이렇게 폭발적으로 성장하리라 상상하지 못했습니다. 2011년 IANA(인터넷 주소 관리 기관)는 공식적으로 IPv4 주소가 고갈되었다고 선언했습니다.

해결책은 **IPv6**입니다. 128비트 주소를 사용해서 2¹²⁸ = 약 340언데실리언(3.4 × 10³⁸) 개의 주소를 만들 수 있습니다. 지구 표면의 모든 모래 한 알에 수십억 개씩 IP 주소를 부여해도 남을 만큼 많습니다.

IPv6는 단순히 주소가 많아진 것만이 아닙니다. 헤더 구조도 대폭 개선되었습니다. IPv4 헤더는 최소 20바이트에 옵션에 따라 최대 60바이트까지 가변 길이였습니다. 라우터는 헤더를 처리하기 전에 길이를 먼저 파악해야 하는 번거로움이 있었습니다. IPv6 헤더는 40바이트로 고정되어 라우터 처리가 단순해졌습니다. 체크섬도 없애서(상위 계층이 담당) 라우터가 더 빠르게 처리합니다.

**단편화(Fragmentation)**에서도 큰 변화가 있습니다. IPv4에서는 패킷이 중간 라우터를 지날 때 MTU(최대 전송 단위)보다 크면 라우터가 패킷을 쪼갭니다. 이 과정이 라우터에 부담을 줬습니다. IPv6에서는 단편화를 **송신 호스트에서만** 수행합니다. 라우터는 단편화를 하지 않고, 패킷이 너무 크면 "Packet Too Big" 오류 메시지로 송신 측에 알려서 다시 보내게 합니다.

보안도 강화되었습니다. IPv6는 처음 설계 시 IPsec을 내장하는 것을 목표로 했습니다. 다만 2011년 RFC 6434에서 "필수"에서 "권고(SHOULD)"로 완화되어, 실제 구현에서는 선택적으로 적용됩니다. IPv4와 비교해 IPsec 기본 지원을 전제로 설계된 점은 여전히 의미 있는 차이입니다.

그러나 지금 당장 모든 인터넷을 IPv6로 바꿀 수는 없습니다. 수십 년 동안 구축된 IPv4 기반 인프라가 너무 방대합니다. 전환 기간을 위한 세 가지 방법이 있습니다.

**듀얼스택(Dual Stack)**은 하나의 장치가 IPv4와 IPv6 주소를 모두 가집니다. IPv6를 지원하는 상대와는 IPv6로, 아직 IPv4만 지원하는 상대와는 IPv4로 통신합니다. 가장 자연스러운 전환 방법이지만 두 스택을 모두 관리해야 합니다.

**터널링(Tunneling)**은 IPv6 패킷을 IPv4 패킷 안에 캡슐화해서 IPv4 네트워크를 통과시킵니다. IPv6 섬들을 IPv4 바다 위에 띄워두는 방식입니다.

**NAT64**는 IPv6만 사용하는 호스트가 아직 IPv4를 쓰는 서버에 접근할 때, 중간에서 주소를 변환해 줍니다.

## Part 9. 서브네팅/수퍼네팅 — 아파트 동호수를 배정하듯 IP 주소를 쪼개기

### IP 주소 낭비를 막는 지혜

회사가 1,000개의 IP 주소를 받았는데 직원이 50명이라면 950개가 낭비됩니다. 반대로 직원이 2,000명인데 1,024개짜리 블록을 받으면 모자랍니다. IP 주소를 필요한 만큼 정확하게 배분하는 기술이 **서브네팅(Subnetting)**입니다.

서브넷 마스크는 IP 주소에서 '네트워크 부분'과 '호스트 부분'을 구분하는 경계선입니다. `/24`는 앞 24비트가 네트워크 주소이고 뒤 8비트(256개, 실제 사용 가능 254개)가 호스트 주소라는 뜻입니다.

**VLSM(Variable Length Subnet Mask)**은 팀마다 다른 크기의 서브넷을 할당하는 방법입니다. `192.168.0.0/22` 대역(총 1,022개 호스트 IP)을 네 개 팀에 배분하는 예를 봅시다.

```
192.168.0.0/22 (1022개 호스트 사용 가능)
       ↓ 가장 큰 팀부터 먼저 배분

개발팀 (500명 필요) → /23 필요 (2⁹-2 = 510개)
→ 192.168.0.0/23  (192.168.0.1 ~ 192.168.1.254)

영업팀 (250명 필요) → /24 필요 (2⁸-2 = 254개)
→ 192.168.2.0/24  (192.168.2.1 ~ 192.168.2.254)

인사팀 (125명 필요) → /25 필요 (2⁷-2 = 126개)
→ 192.168.3.0/25  (192.168.3.1 ~ 192.168.3.126)

보안팀 (125명 필요) → /25 필요
→ 192.168.3.128/25 (192.168.3.129 ~ 192.168.3.254)
```

**수퍼네팅(Supernetting)**은 반대로 여러 개의 소규모 네트워크를 하나의 큰 블록으로 묶어서 라우팅 테이블을 줄이는 기법입니다. 인터넷 라우터들이 수억 개의 경로를 일일이 기억하면 처리가 불가능하므로, 비슷한 주소 블록들을 묶어서 하나의 항목으로 표현합니다. 이것이 **CIDR(Classless Inter-Domain Routing)**의 핵심 아이디어입니다.

## Part 10. 라우팅 — 지도 없이 가느냐, 전체 지도를 가지고 가느냐

### 인터넷에서 최적 경로를 찾는 두 가지 방법

서울에서 부산까지 가는 방법을 생각해 봅시다. 첫 번째 방법은 길을 걸어가면서 만나는 사람마다 "부산 얼마나 멀어요?"라고 물어보고, 각자가 알려주는 가장 가까운 다음 방향으로 계속 가는 것입니다. 이것이 **거리벡터(Distance Vector) 라우팅**의 핵심입니다.

각 라우터는 이웃 라우터들과만 정보를 교환합니다. "나에게서 A 네트워크까지는 3홉이야", "나에게서 B 네트워크까지는 5홉이야"라는 식의 거리 벡터를 주고받아서 라우팅 테이블을 만듭니다. 구현이 간단하고 작은 네트워크에서 잘 동작합니다.

하지만 문제가 있습니다. 경로가 끊겼을 때 잘못된 정보가 네트워크 전체로 퍼져나가는 **Counting to Infinity** 현상입니다. A-B-C로 연결된 네트워크에서 A-B 링크가 끊기면, C가 "나는 B를 통해서 A에 갈 수 있어(거리 2)"라는 정보를 B에게 알려주고, B는 C를 통해서 A에 갈 수 있다고 착각해서 거리를 3으로 업데이트하고, C는 B가 3이라고 하니 자신은 4라고 업데이트하고... 이렇게 거리가 무한히 카운팅되며 수렴하지 않는 문제입니다. 대표 프로토콜은 **RIP(Routing Information Protocol)**이며 최대 15홉까지만 지원합니다.

두 번째 방법은 지도를 먼저 완전히 그려놓고 다익스트라 알고리즘으로 최단 경로를 계산하는 것입니다. 이것이 **링크스테이트(Link State) 라우팅**입니다.

각 라우터는 자신에게 연결된 링크의 상태(연결 여부, 비용)를 **LSA(Link State Advertisement)**로 만들어서 네트워크 전체의 모든 라우터에게 홍수처럼 전달합니다(Flooding). 모든 라우터가 같은 링크 상태 데이터베이스를 가지게 되면, 각자 다익스트라 알고리즘으로 자신을 출발점으로 하는 최단 경로 트리를 계산합니다.

수렴 속도가 빠르고 루프가 없으며, 네트워크 변화에 빠르게 적응합니다. 하지만 모든 라우터가 전체 토폴로지를 저장해야 하므로 메모리와 CPU 부담이 크고, 네트워크가 커질수록 LSA 홍수로 인한 오버헤드도 커집니다. 대표 프로토콜은 **OSPF(Open Shortest Path First)**이며 대규모 기업 및 ISP 네트워크에서 사실상의 표준입니다.

| 비교 | 거리벡터(RIP) | 링크스테이트(OSPF) |
|---|---|---|
| 정보 교환 | 이웃과만 | 전체 라우터(Flooding) |
| 알고리즘 | Bellman-Ford | Dijkstra |
| 수렴 속도 | 느림 | 빠름 |
| 루프 위험 | 있음 | 없음 |
| 메모리/CPU | 적음 | 많음 |
| 최대 홉 수 | 15 (RIP) | 제한 없음 |
| 적합 규모 | 소규모 | 대규모 |

**멀티캐스트 라우팅**은 동일한 데이터를 여러 수신자에게 동시에 효율적으로 보내는 방법입니다. IPTV, 화상회의, 주식 시세 동시 배포 같은 시나리오에서 유니캐스트로 N명에게 각각 보내면 서버와 네트워크 모두 N배의 부하를 감당해야 합니다. 멀티캐스트는 가능한 한 공통 경로를 하나만 유지하다가 수신자가 여러 곳으로 나뉘는 지점에서만 복사합니다. **IGMP(Internet Group Management Protocol)**는 호스트가 라우터에게 "나 이 멀티캐스트 그룹에 가입할게요" / "나 나갈게요"를 알리는 프로토콜입니다.

## Part 11. TCP 전송계층 — 신뢰의 우편 시스템

### 데이터가 반드시 도착하도록 보장하는 방법

인터넷은 태생적으로 신뢰할 수 없는 환경입니다. 패킷은 중간에 사라질 수 있고, 순서가 바뀔 수 있고, 복제될 수 있습니다. 그러나 우리가 파일을 다운로드하면 파일이 정확하게 원본 그대로 도착합니다. 이 마법을 만들어 내는 것이 **TCP(Transmission Control Protocol)**입니다.

**TCP는 연결을 맺고, 데이터를 주고받고, 연결을 끊는 세 단계를 거칩니다.**

연결 설정은 **3-way Handshake**로 이루어집니다. 전화를 걸 때 "여보세요" → "네, 들려요" → "저도 들려요"처럼 세 번의 신호를 주고받으며 양측이 준비되었음을 확인합니다.

```
[3-way Handshake — 연결 설정]
클라이언트                          서버
    │─── SYN (seq=100) ───────────►│  "연결 요청, 내 번호는 100"
    │◄── SYN+ACK (seq=200, ack=101)│  "OK, 내 번호는 200, 네 101번 기다림"
    │─── ACK (ack=201) ───────────►│  "OK, 네 201번 기다림"
    │          (연결 완료)           │
```

연결 해제는 **4-way Handshake**입니다. 왜 3-way가 아니라 4-way냐고요? 클라이언트가 FIN을 보냈을 때 서버는 "알겠어, 그런데 나는 아직 보낼 데이터가 있어"라고 할 수 있습니다. 그래서 ACK와 FIN이 별도로 분리됩니다.

```
[4-way Handshake — 연결 해제]
클라이언트                          서버
    │─── FIN ────────────────────►│  "나 보낼 거 다 보냈어, 끊을게"
    │◄── ACK ─────────────────────│  "알겠어" (서버는 아직 보낼 게 있을 수 있음)
    │◄── FIN ─────────────────────│  "나도 다 보냈어, 끊을게"
    │─── ACK ────────────────────►│  "OK" (TIME_WAIT 후 완전 종료)
```

TIME_WAIT 대기 시간은 RFC 793 기준 2×MSL = **4분**이지만, Linux 커널 기본값은 **60초**입니다. 초당 수천 건의 단기 연결을 처리하는 서버에서 TIME_WAIT 소켓이 쌓여 포트가 고갈되면 `tcp_tw_reuse` 옵션이나 HTTP Keep-Alive로 해결합니다.

TCP가 신뢰성을 보장하는 세 가지 핵심 제어 메커니즘이 있습니다.

**오류 제어(Error Control)**는 체크섬으로 손상된 세그먼트를 감지하고, ACK/NAK와 재전송 타이머로 손실된 세그먼트를 복구합니다. 수신 측이 특정 시간 안에 ACK를 보내지 않으면, 송신 측은 타임아웃으로 판단하고 해당 세그먼트를 재전송합니다.

**흐름 제어(Flow Control)**는 송신 측이 수신 측의 처리 속도를 초과해서 보내지 않도록 조절합니다. 수신 측은 ACK 패킷에 자신의 남은 버퍼 크기(**윈도우 크기**)를 알려줍니다. 송신 측은 이 윈도우 크기만큼만 ACK를 받지 않고도 보낼 수 있습니다. **Sliding Window** 방식으로 구현되며, 한 번에 여러 패킷을 보낼 수 있어 Stop-and-Wait보다 훨씬 효율적입니다.

**혼잡 제어(Congestion Control)**는 네트워크 자체가 혼잡해지지 않도록 조절합니다. 송신 측은 **혼잡 윈도우(cwnd)**를 유지하며 네트워크 상태에 따라 전송 속도를 조절합니다. **Slow Start** 단계에서는 처음에 cwnd를 1로 시작해서 ACK가 올 때마다 지수적으로 증가시킵니다. 임계값(ssthresh)에 도달하면 **AIMD(Additive Increase, Multiplicative Decrease)** 방식으로 전환해서 선형으로 증가하다가 패킷 손실이 감지되면 cwnd를 절반으로 급격히 줄입니다.

**UDP(User Datagram Protocol)**는 TCP와 반대입니다. 연결 설정 없이 데이터를 그냥 보냅니다. 오류 제어도, 흐름 제어도, 혼잡 제어도 없습니다. 그 대신 **속도가 빠르고 오버헤드가 적습니다**. DNS 조회처럼 질문 하나에 응답 하나인 짧은 통신, 또는 실시간 스트리밍처럼 약간의 패킷 손실을 감수하더라도 지연이 없어야 하는 경우에 씁니다.

**SCTP(Stream Control Transmission Protocol)**는 TCP와 UDP의 장점을 합친 프로토콜입니다. TCP처럼 신뢰성 있는 전달을 보장하면서, 하나의 연결 안에 **여러 개의 독립적인 스트림**을 동시에 운용합니다. TCP에서는 스트림 하나가 막히면(Head-of-Line Blocking) 뒤에 오는 모든 데이터가 기다려야 합니다. SCTP에서는 한 스트림이 막혀도 다른 스트림은 계속 흐릅니다. 또한 **멀티호밍(Multi-homing)**을 지원해서 하나의 연결이 여러 IP 주소를 사용할 수 있습니다. 하나의 인터페이스가 고장나면 자동으로 다른 IP로 전환됩니다. 통신 인프라(SS7 over IP), VoIP 등 고신뢰성이 필요한 영역에서 쓰입니다.

## Part 12. Multiple Access — 공유 채널을 사이좋게 나누는 방법

### 여러 명이 동시에 말하면 아무것도 들리지 않습니다

이더넷이 처음 등장했을 때, 같은 케이블에 연결된 모든 컴퓨터가 케이블을 공유했습니다. 누군가 데이터를 보내면 케이블에 전기 신호가 흐르고, 이 신호가 케이블 전체에 퍼졌습니다. 두 컴퓨터가 동시에 보내면 신호가 충돌해서 아무것도 전달되지 않습니다. 이 공유 채널 문제를 해결하는 것이 **Multiple Access 프로토콜**입니다.

**임의 접근(Random Access)** 방식은 아무도 순서를 강제하지 않습니다. 보내고 싶을 때 보내되, 충돌하면 적당히 기다렸다가 다시 보냅니다.

**CSMA/CD(Carrier Sense Multiple Access with Collision Detection)**는 이더넷의 방식입니다. 먼저 채널이 비어있는지 귀를 기울이고(Carrier Sense), 비어있으면 보냅니다. 보내면서도 계속 충돌이 있는지 감지합니다(Collision Detection). 충돌이 감지되면 즉시 멈추고, 모든 기기에게 충돌을 알리는 잼(Jam) 신호를 보낸 후, 랜덤한 시간을 기다렸다가 다시 시도합니다. 기다리는 시간은 충돌이 반복될수록 두 배씩 늘어납니다(지수 백오프, Exponential Backoff).

**CSMA/CA(Collision Avoidance)**는 Wi-Fi의 방식입니다. 무선 환경에서는 보내면서 동시에 충돌을 감지하기가 어렵습니다(반이중 특성). 그래서 충돌을 **회피**하는 방식을 씁니다. 채널이 비어있어도 랜덤하게 조금 더 기다린 뒤에 보냅니다.

**제어 접근(Controlled Access)** 방식은 중앙이 순서를 정해줍니다. **폴링(Polling)**은 중앙 컨트롤러가 "너 보낼 거 있어? → 없어요. 너는? → 있어요, 보내도 될까요? → 보내" 식으로 순서를 관리합니다. **토큰 링(Token Ring)**은 네트워크를 링 형태로 구성하고 '토큰'이라는 특수 패킷이 한 방향으로 계속 돌게 합니다. 토큰을 받은 기기만 데이터를 보낼 수 있습니다.

**채널화(Channelization)** 방식은 채널 자체를 물리적으로 나눕니다. **FDMA**는 주파수를 나누고, **TDMA**는 시간 슬롯을 나누고, **CDMA**는 고유한 코드로 구분합니다. 이동통신(3G, 4G LTE)의 기반 기술들이 여기에 해당합니다.

## Part 13. DHCP — IP 주소를 자동으로 임대해 주는 우체국

### 노트북을 새 카페에 들고 가도 인터넷이 되는 이유

카페에 도착해서 와이파이에 연결하면 몇 초 만에 인터넷을 쓸 수 있습니다. IP 주소를 직접 입력한 적이 없는데도요. **DHCP(Dynamic Host Configuration Protocol)**가 자동으로 해 줬기 때문입니다.

DHCP는 네트워크에 새로 접속하는 장치에게 IP 주소, 서브넷 마스크, 기본 게이트웨이 주소, DNS 서버 주소를 자동으로 배포합니다. 이 과정을 **DORA**라고 부릅니다.

```
[DORA 4단계]
클라이언트                                    DHCP 서버
    │
    │──── Discover (브로드캐스트) ───────────►│
    │     "IP 좀 빌려줄 사람 있어요?"          │
    │                                         │
    │◄─── Offer ──────────────────────────────│
    │     "192.168.0.50 어때요? 유효기간 24시간"│
    │                                         │
    │──── Request (브로드캐스트) ─────────────►│
    │     "그 주소 쓸게요!"                    │
    │                                         │
    │◄─── ACK ────────────────────────────────│
    │     "확정! 192.168.0.50 당신 겁니다"     │
```

브로드캐스트로 시작하는 이유는 클라이언트가 아직 IP 주소도, DHCP 서버의 주소도 모르기 때문입니다. 네트워크에 "누구든 대답해 줘"라고 외치는 것입니다.

임대(Lease) 개념도 중요합니다. DHCP는 IP 주소를 영구적으로 부여하지 않고 일정 시간 동안만 빌려줍니다. 임대 기간이 끝나가면 클라이언트는 갱신 요청을 보냅니다. 장치가 사라지면 임대 만료 후 IP 주소가 회수되어 다른 장치에게 줄 수 있습니다. 이렇게 IP 주소를 동적으로 관리해서 주소 풀을 효율적으로 활용합니다.

## Part 14. DNS와 DNSSEC — 인터넷 전화번호부와 그 위조 방지 인감

### www.google.com은 사실 숫자입니다

브라우저에 `www.google.com`을 입력하면 실제로는 `142.250.196.36` 같은 IP 주소로 연결됩니다. 우리가 숫자를 외울 필요 없이 이름을 쓸 수 있는 것은 **DNS(Domain Name System)**가 이름을 숫자로 변환해 주기 때문입니다.

DNS는 거대한 분산 데이터베이스입니다. 단 하나의 서버에 인터넷의 모든 도메인 정보를 저장할 수는 없습니다. 그래서 DNS는 트리 구조의 계층으로 나뉩니다.

```
.  ← 루트 (전체 트리의 꼭대기)
│
├── com  ← TLD (Top-Level Domain)
│    ├── google  ← 2차 도메인
│    │    └── www  ← 서브도메인
│    └── naver
├── kr
│    └── go
│         └── nts  (국세청)
```

`www.google.com`을 조회하는 과정은 이렇습니다. 먼저 내 컴퓨터의 로컬 캐시를 확인합니다. 없으면 내 컴퓨터의 로컬 DNS 서버(ISP가 제공하는)에 물어봅니다. 로컬 DNS 서버도 모르면 **루트 네임서버**에 물어봅니다. 루트 서버는 `.com` 관리 서버를 알려주고, `.com` 서버는 `google.com` 관리 서버를 알려주고, `google.com` 서버가 최종적으로 IP 주소를 알려줍니다. 로컬 DNS 서버는 이 결과를 캐싱해서 같은 질문이 오면 빠르게 응답합니다.

루트 네임서버는 `a.root-servers.net`부터 `m.root-servers.net`까지 **13개 주소**만 존재합니다. 그러나 실제로는 각 주소마다 **Anycast** 기술로 수백 개의 물리 서버가 전 세계에 분산 운영됩니다. 내 쿼리는 가장 가까운 물리 서버로 자동 라우팅되므로, 전 세계 어디서든 빠르게 응답받을 수 있습니다.

그런데 DNS는 심각한 보안 취약점이 있습니다. 오리지널 DNS는 응답을 검증하는 기능이 없습니다. 공격자가 가짜 DNS 응답을 먼저 보내서 DNS 캐시를 오염시키면(**DNS Cache Poisoning**), 사용자가 `www.bank.com`을 입력했을 때 피싱 사이트 IP를 받게 됩니다.

**DNSSEC(DNS Security Extensions)**는 이 문제를 해결합니다. 각 DNS 응답에 **전자서명**을 붙입니다. DNS 서버는 개인키로 응답에 서명하고, 클라이언트는 공개키로 서명을 검증합니다. 서명이 맞지 않으면 위조된 응답임을 알고 거부합니다. 서명 검증에 필요한 공개키 체인은 루트부터 아래로 신뢰 체인(Chain of Trust)을 형성합니다.

IoT 장비가 폭발적으로 늘어나면서 DNS 공격의 위험성이 급증하고 있습니다. 보안 설정이 취약한 수백만 IoT 장비가 공격자의 DNS 스푸핑에 노출되면 대규모 피해가 생깁니다. DNSSEC의 중요성이 더욱 커지는 이유입니다.

## Part 15. Ad-hoc 라우팅 — 기지국 없이 스스로 연결되는 메쉬 네트워크

### 기지국이 없어도 통신할 수 있다면?

재난이 발생해서 통신 기지국이 모두 파괴되었습니다. 구조대원들이 서로 연락할 방법이 없습니다. 이런 상황에서 **Ad-hoc 네트워크**가 빛을 발합니다. 기지국이나 중앙 서버 없이 장치들이 직접 연결되어 스스로 네트워크를 형성합니다. 각 장치가 라우터이자 호스트 역할을 동시에 수행합니다.

Ad-hoc 네트워크에서 가장 어려운 문제는 **라우팅**입니다. 고정된 인프라가 없고 노드들이 이동하며 토폴로지가 계속 변합니다.

**Proactive(선제적) 라우팅**은 경로가 필요하기 전에 미리 네트워크 전체의 라우팅 테이블을 유지합니다. 노드들이 주기적으로 라우팅 정보를 교환해서 항상 최신 경로를 알고 있습니다. 데이터 전송 시 지연이 없지만, 통신이 없어도 라우팅 정보를 계속 교환해야 하는 오버헤드가 있습니다. OLSR, DSDV가 대표 프로토콜입니다.

**Reactive(반응형) 라우팅**은 실제로 데이터를 보내야 할 때 경로를 찾습니다. 평소에는 라우팅 정보를 유지하지 않아서 오버헤드가 적습니다. 대신 처음 통신 시 경로 탐색 지연이 발생합니다. AODV, DSR이 대표 프로토콜입니다.

**Hybrid(혼합형) 라우팅**은 가까운 노드들과는 Proactive 방식으로, 멀리 있는 노드에게는 Reactive 방식으로 라우팅합니다. ZRP가 대표 프로토콜입니다.

드론 편대 통신, 군사 작전, 재난 구조, 차량 간 통신(V2V), 스마트 공장 등에서 Ad-hoc 기술이 점점 더 중요해지고 있습니다.

## Part 16. QoS — VIP 라운지와 일반석, 트래픽에도 등급이 있습니다

### 화상회의 중에 파일 다운로드가 방해해서는 안 됩니다

인터넷에서 패킷은 원칙적으로 평등합니다. 화상회의 패킷이든 파일 다운로드 패킷이든 같은 큐에 들어가서 순서대로 처리됩니다. 그런데 화상회의는 100ms 이상 지연되면 대화가 어색해지고, 패킷이 손실되면 화면이 깨집니다. 반면 파일 다운로드는 1초 정도 늦어도 아무도 신경 쓰지 않습니다. 이 차이를 인식하고 중요한 트래픽을 우선 처리하는 기술이 **QoS(Quality of Service)**입니다.

QoS를 구현하는 방법은 크게 두 가지 모델로 나뉩니다.

**IntServ(Integrated Services)**는 데이터를 보내기 전에 경로의 모든 라우터에게 자원 예약을 요청합니다. 마치 비행기를 탈 때 미리 좌석을 예약하는 것과 같습니다. **RSVP(Resource Reservation Protocol)**로 경로상의 각 라우터에 "이 흐름에 대해 대역폭 Xbps, 지연 Y ms를 보장해 주세요"라고 요청하고 모든 라우터가 수락해야 통신이 시작됩니다. 절대적인 품질 보장이 가능하지만, 모든 라우터가 각 플로우의 상태를 기억해야 하므로 확장성이 극히 떨어집니다. 수백만 개의 동시 플로우가 흐르는 인터넷 코어에서는 현실적으로 불가능합니다.

**DiffServ(Differentiated Services)**는 확장성 문제를 해결합니다. 트래픽을 수십 개의 클래스로 미리 분류하고, 패킷 헤더에 **DSCP(Differentiated Services Code Point)** 마킹을 합니다. 라우터는 이 마킹만 보고 패킷을 처리합니다. 네트워크 경계에서 한 번 마킹하면, 내부 라우터들은 마킹에 따라 큐에 넣고 처리하면 됩니다. 절대적 보장은 아니지만 확장성이 뛰어나 실제 인터넷에서 많이 씁니다.

**스케줄링 기법**은 큐에 있는 패킷을 어떤 순서로 처리할지 결정합니다. **FIFO**는 먼저 온 것을 먼저 처리하는 가장 단순한 방식입니다. **Priority Queue**는 높은 우선순위 큐가 빌 때까지 낮은 우선순위 패킷은 기다립니다. **WRR(Weighted Round Robin)**은 우선순위에 따라 가중치를 부여해서 라운드 로빈으로 처리합니다. VoIP 패킷에 가중치 3, 일반 데이터에 가중치 1을 주면 VoIP가 3배 더 자주 처리됩니다. **WFQ(Weighted Fair Queuing)**와 **DRR(Deficit Round Robin)**은 더 공정하고 정교한 방식입니다.

**버킷 기법**은 트래픽의 속도를 제어합니다. **토큰 버킷(Token Bucket)**은 버킷에 일정 속도로 토큰이 채워지고, 패킷을 보낼 때마다 토큰을 소비합니다. 버킷이 비면 토큰이 찰 때까지 기다립니다. 평균 전송률을 제한하면서도 토큰이 쌓인 만큼 순간적인 버스트(Burst)를 허용합니다. **리키 버킷(Leaky Bucket)**은 어떤 속도로 들어와도 일정한 속도로만 출력합니다. 버스트를 완전히 억제하고 일정한 출력 속도를 보장합니다.

## Part 17. CDN — 전국 곳곳에 창고를 두어 빠르게 배달하는 물류 네트워크

### 서울 본사에서 제주도 고객에게 배달하면 항상 느릴까요?

쿠팡이나 마켓컬리가 빠른 배달이 가능한 이유는 전국 곳곳에 풀필먼트 센터(물류 창고)를 두었기 때문입니다. 제주도 고객의 주문이 오면 서울 본사에서 배달하는 게 아니라 가장 가까운 창고에서 출발합니다. **CDN(Content Delivery Network)**은 인터넷 콘텐츠 배포에 같은 원리를 적용합니다.

원본 서버가 미국 서부에 있다고 가정합시다. 한국 사용자가 동영상을 요청하면 태평양을 건너 미국으로 가서 데이터를 받아 다시 태평양을 건너 돌아와야 합니다. 왕복 수백 ms의 지연이 발생합니다. CDN은 한국에 **엣지 서버(Edge Server)**를 미리 두고, 많이 요청되는 콘텐츠를 캐싱해 둡니다. 한국 사용자의 요청은 근처 엣지 서버에서 즉시 응답받습니다.

사용자가 `video.example.com`에 접속하면 먼저 DNS가 개입합니다. **GSLB(Global Server Load Balancing)**이 사용자의 위치를 감지해서 가장 가까운 엣지 서버의 IP를 DNS 응답으로 돌려줍니다. 한국 사용자에게는 한국 서버를, 미국 사용자에게는 미국 서버를 안내합니다.

엣지 서버에 해당 콘텐츠의 캐시가 있으면(**Cache HIT**) 즉시 응답합니다. 없으면(**Cache MISS**) 원본 서버에서 가져와 캐싱한 후 응답합니다. 이후 같은 요청이 오면 캐시에서 바로 줍니다.

캐시는 영원히 유지되지 않습니다. **TTL(Time-To-Live)**이 지나면 캐시를 무효화하고 원본에서 새로 가져옵니다. 콘텐츠가 자주 변하는 뉴스 기사는 TTL을 짧게, 잘 바뀌지 않는 이미지 파일은 길게 설정합니다.

Netflix, YouTube, Instagram처럼 전 세계 사용자에게 대용량 미디어 콘텐츠를 서비스하는 기업들이 CDN 없이는 운영 자체가 불가능합니다.

## Part 18. HTTP / QUIC / HTTP 3.0 — 인터넷의 국어, 그리고 그 진화

### 웹 브라우저와 서버는 어떻게 대화하나요?

웹 브라우저가 서버에게 "www.example.com의 메인 페이지 주세요"라고 요청하고, 서버가 HTML을 돌려주는 과정이 **HTTP(HyperText Transfer Protocol)**입니다. 1990년대 팀 버너스리가 인터넷의 방대한 문서들을 연결하기 위해 설계했습니다.

**HTTP/1.1**은 1997년에 등장해서 20년 넘게 웹의 기반이었습니다. 하나의 TCP 연결로 여러 요청을 순차적으로 처리합니다. 그러나 현대 웹 페이지는 수십~수백 개의 리소스(이미지, CSS, JavaScript, API 등)를 요청합니다. HTTP/1.1은 한 요청의 응답이 와야 다음 요청을 보낼 수 있어서(**HoL Blocking**, Head-of-Line Blocking) 병목이 생겼습니다. 이를 줄이기 위해 브라우저는 도메인당 6개의 TCP 연결을 병렬로 열었는데, 이 자체가 서버에 부담이었습니다.

**HTTP/2.0**은 2015년에 이 문제를 해결했습니다. **멀티플렉싱(Multiplexing)**이 핵심입니다. 하나의 TCP 연결 안에서 여러 요청과 응답이 동시에 오갈 수 있습니다. 각 메시지를 **스트림**이라는 단위로 쪼개서, 여러 스트림이 인터리빙되어 전송됩니다. **헤더 압축(HPACK)**으로 중복 헤더를 줄이고, 서버가 클라이언트의 요청 없이 미래에 필요할 리소스를 먼저 보내는 **서버 푸시(Server Push)**도 가능합니다.

그러나 HTTP/2도 한계가 있었습니다. TCP 위에서 동작하기 때문입니다. TCP는 패킷 하나가 손실되면 그 패킷이 재전송될 때까지 뒤에 오는 모든 데이터가 기다려야 합니다. 여러 스트림이 하나의 TCP 연결을 공유하므로, 패킷 하나의 손실이 모든 스트림을 멈추게 합니다. TCP 수준의 HoL Blocking이 여전히 남아있었습니다.

**HTTP/3.0**은 근본적으로 다른 접근을 택했습니다. TCP 대신 **QUIC(Quick UDP Internet Connections)** 프로토콜 위에서 동작합니다. QUIC은 UDP 기반으로 구현되어 있지만, 그 위에서 TCP의 신뢰성, TLS 1.3의 보안, HTTP/2의 멀티플렉싱을 모두 구현했습니다. 각 스트림이 완전히 독립적이어서 하나의 패킷 손실이 다른 스트림에 영향을 주지 않습니다.

QUIC의 또 다른 강점은 **0-RTT 연결 설정**입니다. TCP+TLS는 연결 설정에 최소 3번의 왕복(TCP 3-way HS + TLS HS)이 필요합니다. QUIC은 처음 연결 시 1번, 이전에 연결한 적 있는 서버라면 0번의 왕복으로 연결을 맺고 데이터를 보낼 수 있습니다.

**연결 마이그레이션**도 혁신적입니다. TCP 연결은 IP 주소와 포트로 식별됩니다. 스마트폰에서 Wi-Fi에서 LTE로 전환되면 IP가 바뀌고 TCP 연결이 끊깁니다. QUIC은 **Connection ID**로 연결을 식별해서 IP가 바뀌어도 연결이 유지됩니다. 지하철에서 핸드오버가 일어나도 동영상이 끊기지 않는 미래를 가능하게 합니다.

| 버전 | 전송 기반 | 핵심 특징 |
|---|---|---|
| HTTP/1.1 | TCP | 순차 처리, HoL Blocking |
| HTTP/2.0 | TCP | 멀티플렉싱, 헤더 압축, 서버 푸시 |
| HTTP/3.0 | QUIC(UDP) | TCP HoL 해결, 0-RTT, 연결 마이그레이션 |

## Part 19. SDN과 NFV — 소프트웨어로 네트워크 장비를 원격 제어하는 시대

### 네트워크 장비도 클라우드처럼 될 수 있을까요?

전통적인 네트워크 장비(라우터, 스위치, 방화벽)는 마치 블랙박스와 같습니다. 시스코, 주니퍼 같은 특정 벤더의 하드웨어에 해당 벤더의 운영 소프트웨어가 내장되어, 패킷을 어떻게 처리할지 결정하는 **제어 플레인**과 실제로 패킷을 전달하는 **데이터 플레인**이 하나의 박스 안에 뭉쳐 있습니다. 새로운 기능을 추가하거나 네트워크 정책을 변경하려면 장비 제조사의 업데이트를 기다리거나, 직접 장비를 방문해서 설정해야 했습니다. 유연성이 없고 변화에 느립니다.

**SDN(Software Defined Networking)**은 이 두 플레인을 분리합니다. 제어 플레인을 중앙의 **SDN 컨트롤러**로 끌어내고, 개별 스위치는 오직 데이터 플레인만 담당합니다. 컨트롤러가 전체 네트워크를 한눈에 보며 최적의 경로와 정책을 결정하고, OpenFlow 같은 프로토콜로 각 스위치에 "패킷 A를 포트 3으로 내보내라" 같은 규칙을 내려보냅니다.

```
[SDN 구조]
┌────────────────────────────────────────┐
│         SDN 컨트롤러 (소프트웨어)        │
│    "전체 네트워크 지도를 보고 결정"      │
└──────────┬────────────────┬────────────┘
           │ OpenFlow       │ OpenFlow
  ┌────────▼────┐    ┌──────▼──────┐
  │  스위치 A   │    │  스위치 B   │
  │ (데이터 플레인만) │  │ (데이터 플레인만) │
  └─────────────┘    └─────────────┘
  "시키는 대로 패킷 전달"
```

이렇게 하면 네트워크 정책을 소프트웨어로 프로그래밍할 수 있습니다. 장애가 감지되면 컨트롤러가 자동으로 경로를 우회시키고, 트래픽 패턴에 따라 실시간으로 부하를 분산합니다. 데이터센터에서 수천 대의 서버 네트워크를 자동화하는 데 핵심 기술입니다.

**NFV(Network Function Virtualization)**는 다른 각도에서 접근합니다. 방화벽, 로드밸런서, VPN 게이트웨이 같은 네트워크 기능들이 지금까지는 전용 하드웨어 장비로 구현되었습니다. NFV는 이런 기능들을 **일반 서버 위에서 소프트웨어(가상 머신 또는 컨테이너)로** 구현합니다.

기업이 방화벽이 필요하면 이제 고가의 전용 장비를 구매하는 대신, 클라우드에서 가상 방화벽 인스턴스를 수분 만에 띄울 수 있습니다. 트래픽이 늘어나면 인스턴스를 더 추가하고, 줄어들면 줄입니다. 비용이 기하급수적으로 절감되고 서비스 제공 속도가 빨라집니다.

SDN과 NFV는 서로 보완적입니다. SDN이 네트워크 제어를 소프트웨어화한다면, NFV는 네트워크 기능 자체를 소프트웨어화합니다. 5G 코어 네트워크, 클라우드 데이터센터, 엣지 컴퓨팅 인프라의 핵심 기술로 자리 잡고 있습니다.

## Part 20. Wi-Fi 7 — 무선 인터넷의 7세대

### 유선보다 빠른 무선의 시대

Wi-Fi는 1997년 IEEE 802.11 표준이 처음 등장한 이래 꾸준히 진화해 왔습니다. 2Mbps에 불과했던 속도가 이제 이론상 46Gbps에 달하는 **Wi-Fi 7(IEEE 802.11be)**까지 왔습니다.

Wi-Fi 7의 핵심 혁신은 **MLO(Multi-Link Operation)**입니다. 기존 Wi-Fi는 한 번에 하나의 주파수 대역만 사용했습니다. Wi-Fi 7은 2.4GHz, 5GHz, 6GHz 세 대역을 **동시에** 사용합니다. 마치 고속도로 세 차선을 동시에 달리는 것과 같습니다. 단순히 속도가 3배가 되는 것뿐 아니라, 한 대역에 간섭이 심하면 자동으로 다른 대역으로 전환해서 지연도 줄어듭니다.

**4K-QAM(4096-QAM)**은 신호 하나에 더 많은 데이터를 담는 기술입니다. Wi-Fi 6의 1K-QAM이 한 번의 신호 변화로 10비트를 전달했다면, 4K-QAM은 12비트를 전달합니다. 20% 더 많은 데이터를 같은 신호로 보낼 수 있습니다.

**320MHz 채널 폭**은 Wi-Fi 6의 160MHz 대비 두 배입니다. 도로를 더 넓게 만드는 것처럼 한 번에 더 많은 데이터가 지나갈 수 있습니다. 6GHz 대역이 추가되면서 이렇게 넓은 채널을 쓸 수 있게 되었습니다.

Wi-Fi 7은 특히 **저지연**에 집중합니다. AR/VR 헤드셋, 클라우드 게임, 산업용 로봇 제어처럼 1ms 이하의 지연이 필요한 응용을 무선으로 지원합니다. MLO가 가장 좋은 링크를 실시간으로 선택해서 지연을 최소화합니다.

| 기술 | Wi-Fi 6 (802.11ax) | Wi-Fi 7 (802.11be) |
|---|---|---|
| 최대 속도 | 9.6 Gbps | 46 Gbps |
| 최대 채널 폭 | 160 MHz | 320 MHz |
| 변조 방식 | 1K-QAM | 4K-QAM |
| 다중 링크 | 미지원 | MLO (2.4/5/6 GHz 동시) |
| 주요 활용 | 고밀도 환경, IoT | AR/VR, 산업 IoT, 클라우드 게임 |

---

> **네트워크를 이해하는 핵심 열쇠는 'OSI 7 계층'입니다.** 모든 네트워크 개념은 이 계층 구조와 연결됩니다. 물리 케이블(1계층)에서 시작해서 당신이 쓰는 카카오톡(7계층)까지, 데이터가 여행하는 모든 여정을 계층별로 추적할 수 있습니다. 이 계층 구조를 축으로 각 계층의 대표 프로토콜(L2: ARP/Ethernet, L3: IP/ICMP, L4: TCP/UDP, L7: HTTP/DNS)을 연결 지어 이해하고, 최신 트렌드(SDN, NFV, QUIC, Wi-Fi 7)를 그 위에 얹으면 정보처리기술사 네트워크 도메인 전체가 하나의 큰 그림으로 보입니다.
