## 해설

### 출제 배경 및 의도

클라우드 컴퓨팅의 경제성은 하나의 물리 서버에서 다수의 독립적 환경을 실행하는 가상화 기술에서 시작됩니다. 하이퍼바이저 기반 VM은 강력한 격리를 제공하나 무거운 반면, 컨테이너는 경량·고속이나 격리 수준이 낮습니다. 최근 Firecracker·gVisor 등 양자의 장점을 결합한 기술도 등장하여 가상화 패러다임이 진화하고 있습니다.

가상화는 클라우드 인프라의 핵심 기술로, 채점관은 **Type 1/2 하이퍼바이저의 구조적 차이**, **VM과 컨테이너의 격리 수준·오버헤드 비교**, 그리고 **Docker 이미지 빌드~배포의 전체 흐름**을 체계적으로 서술하는지를 평가합니다.

### 1. 가상화(Virtualization), 개요

정 의  • 물리 하드웨어 자원을 논리적으로 분리·추상화하여 다수의 독립적 실행 환경을 제공하는 기술
       - 하이퍼바이저 기반 VM과 OS 커널 공유 기반 컨테이너의 두 접근법이 공존

```
┌─────────────────────────────────────────────────────────┐
│              가상화 기술 비교 구조                       │
│                                                         │
│  VM (Hypervisor)              컨테이너 (Docker)         │
│  ┌────────┬────────┐         ┌────────┬────────┐       │
│  │App A   │App B   │         │App A   │App B   │       │
│  │Guest OS│Guest OS│         │Lib A   │Lib B   │       │
│  ├────────┴────────┤         ├────────┴────────┤       │
│  │   Hypervisor    │         │Container Runtime│       │
│  ├─────────────────┤         ├─────────────────┤       │
│  │    Host OS      │         │   Host OS(공유) │       │
│  ├─────────────────┤         ├─────────────────┤       │
│  │   물리 하드웨어  │         │   물리 하드웨어  │       │
│  └─────────────────┘         └─────────────────┘       │
│  완전 격리 (별도 커널)         프로세스 격리(커널 공유)   │
└─────────────────────────────────────────────────────────┘
```

- VM은 격리 수준이 높고 오버헤드가 크며, 컨테이너는 경량·고속이나 커널 공유로 보안 수준이 낮음

### 2. 하이퍼바이저 유형과 VM vs 컨테이너 비교

1) Type 1 vs Type 2 하이퍼바이저

| 구분 | 항목 | Type 1 (Bare-metal) | Type 2 (Hosted) |
|------|------|------|------|
| 구조 | 위치 | 하드웨어 위에 직접 설치 | 호스트 OS 위에 설치 |
| | 호스트 OS | 불필요 | 필요 |
| 성능 | 속도 | 최고 (직접 하드웨어 접근) | 중간 (OS 레이어 오버헤드) |
| 보안 | 격리 | 높음 (VM 간 완전 격리) | 중간 |
| 예시 | 제품 | VMware ESXi, KVM, Xen, Hyper-V | VMware Workstation, VirtualBox |
| 용도 | 사례 | 기업 데이터센터, AWS EC2, Azure VM | 개발/테스트, 개인 PC |

```
Type 1 구조:                      Type 2 구조:
┌──────────────────────┐          ┌──────────────────────┐
│  VM1    │  VM2       │          │  VM1    │  VM2       │
├──────────────────────┤          ├──────────────────────┤
│   Type 1 Hypervisor  │          │   Type 2 Hypervisor  │
├──────────────────────┤          ├──────────────────────┤
│    물리 하드웨어      │          │     호스트 OS         │
└──────────────────────┘          ├──────────────────────┤
                                   │    물리 하드웨어      │
                                   └──────────────────────┘
```

- KVM은 Linux 커널에 내장된 Type 1 하이퍼바이저로 AWS EC2의 기반 기술

2) Hypervisor(VM) vs Docker(컨테이너) 비교

| 비교 항목 | Hypervisor (VM) | Docker (컨테이너) |
|------|------|------|
| 격리 수준 | 완전 격리 (별도 OS·커널) | 프로세스 격리 (커널 공유) |
| 시작 시간 | 수분 (OS 부팅) | 수초 이하 |
| 오버헤드 | 높음 (Guest OS 포함) | 낮음 |
| 이미지 크기 | 수 GB (OS 포함) | 수십~수백 MB |
| 보안 | 높음 (커널 분리) | 낮음 (커널 공유) |
| 이식성 | OS 종류 제한 없음 | Linux 커널 의존 |
| 격리 기반 | 하드웨어 가상화 | Namespace + cgroup |
| 주 사용처 | 강한 격리, 다른 OS 실행 | 마이크로서비스, CI/CD |

- **Namespace**: PID·Network·Mount·UTS·IPC 커널 자원을 프로세스 그룹별 격리
- **cgroup(Control Group)**: CPU·메모리·I/O 자원 사용량 제한

### 3. Docker 컨테이너 배포 절차

```
1. Dockerfile 작성
   ↓
2. 이미지 빌드 (docker build -t myapp:v1.0 .)
   ↓
3. 레지스트리 푸시 (docker push registry/myapp:v1.0)
   ↓
4. 원격 서버 풀 (docker pull registry/myapp:v1.0)
   ↓
5. 컨테이너 실행 (docker run -p 8080:8080 myapp:v1.0)
   ↓ (프로덕션: Kubernetes)
6. K8s Deployment 적용 (kubectl apply -f deployment.yaml)
   ↓
7. 롤링 업데이트 (무중단 버전 교체)
```

| 단계 | 항목 | 설명 |
|------|------|------|
| 빌드 | Dockerfile | FROM, WORKDIR, COPY, RUN, EXPOSE, CMD 명령 |
| | 레이어 캐시 | 변경 없는 레이어 재사용으로 빌드 속도 향상 |
| 저장 | 컨테이너 레지스트리 | Docker Hub, AWS ECR, GCR 등 이미지 저장소 |
| | OCI 표준 | Open Container Initiative 이미지/런타임 표준 |
| 배포 | 쿠버네티스 | Deployment(replicas), Service(로드밸런서), Probe 설정 |
| | 롤링 업데이트 | 구버전→신버전 순차 교체, 무중단 배포 |

가상화 기술은 VM(강한 격리·무거움)과 컨테이너(경량·빠름)의 특성을 상황에 맞게 선택하거나, Firecracker·Kata Containers처럼 VM 격리+컨테이너 생태계를 결합한 방향으로 진화 중.  "끝"

### 실무 제언

**멀티테넌트 클라우드에서 격리 수준 선택**
- **챌린지**: 공용 클라우드에서 커널을 공유하는 컨테이너는 컨테이너 탈출(Container Escape) CVE 취약점으로 강한 격리를 제공하지 못함
- **제언**: AWS Lambda·Google Cloud Run은 내부적으로 Firecracker(경량 VM, ~125ms 시작)를 사용하여 컨테이너의 경량성과 VM의 완전 격리를 동시에 달성. 신뢰할 수 없는 멀티테넌트 워크로드에는 VM 또는 gVisor/Kata Containers 적용

**컨테이너 이미지 보안 관리**
- **챌린지**: 취약점이 있는 베이스 이미지를 기반으로 빌드된 컨테이너가 운영 환경에 배포되어 보안 위협 발생
- **제언**: CI/CD 파이프라인에 Trivy·Snyk 등 이미지 취약점 스캐너 통합, 최소화된 베이스 이미지(Distroless, Alpine) 사용, 정기적 이미지 갱신 자동화

**쿠버네티스 기반 무중단 배포 설계**
- **챌린지**: 롤링 업데이트 중 신버전 파드가 준비되지 않은 상태에서 구버전 파드가 제거되어 일시적 서비스 품질 저하 발생
- **제언**: Readiness Probe로 파드 준비 완료 후 트래픽 전환, PodDisruptionBudget으로 최소 가용 파드 수 보장, 카나리 배포(Argo Rollouts)로 신버전 점진적 트래픽 전환
