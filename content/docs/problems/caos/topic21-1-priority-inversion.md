## 문제

Real Time Scheduling이 갖는 문제 중 우선순위 역전(Priority Inversion)이 있다. Task 1, Task 2, Task 3 순으로 우선순위가 낮다고 할 때 우선순위 역전을 사례기반으로 설명하고, 우선순위 역전을 해결하기 위한 2가지 기법에 대하여 설명하시오.(단 P, V연산을 사용한다)

## 해설

> **높은 우선순위 태스크가 낮은 우선순위 태스크에 막히는 실시간 시스템의 역설**  
> 우선순위 상속(Priority Inheritance)과 우선순위 상한(Priority Ceiling) 기법

실시간 시스템에서 세마포어나 뮤텍스를 사용할 때 발생하는 **우선순위 역전(Priority Inversion)**은 높은 우선순위의 태스크가 낮은 우선순위의 태스크에 의해 실행이 지연되는 예상치 못한 현상입니다. 가장 유명한 사례는 1997년 화성 탐사선 **Mars Pathfinder**의 시스템 리셋 버그로, 우선순위 역전으로 인한 워치독 타이머 만료가 원인이었습니다.

### 1. 출제 배경 및 의도

126회 관리 문제로 P/V 연산을 사용하여 사례 기반으로 설명하고 해결 기법 2가지를 제시해야 합니다. 채점관은 **우선순위 역전이 발생하는 시나리오를 타임라인으로 명확히 표현**하고, **우선순위 상속과 우선순위 상한의 차이**를 정확히 설명하는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### 우선순위 역전 사례 (Task1 > Task2 > Task3 우선순위)

**Task 1(High Priority), Task 2(Medium Priority), Task 3(Low Priority)**  
세마포어 S를 Task 1과 Task 3이 공유 자원 보호에 사용

```
시간 →   t1   t2   t3   t4   t5   t6   t7   t8
Task3:   [실행P(S)] ─────────────────────[V(S)]
Task2:                   ──[선점]────────────────
Task1:         [도착] [P(S)블록]             ──[실행]──
         ─────────────────────────────────────────────▶ 시간

설명:
t1: Task3가 실행 시작, P(S)로 세마포어 획득
t2: Task1(고우선순위) 도착, Task3 선점
t3: Task1이 P(S) 호출 → 세마포어 Task3가 점유 중 → Task1 블록!
t4: Task3 실행 재개 (Task1이 기다리는 동안)
t5: Task2(중우선순위) 도착, Task3 선점!
    → Task2가 Task3를 밀어내고 실행 (Task2가 Task1보다 먼저 실행되는 역전!)
t6~t7: Task2 실행 완료
t8: Task3 재개 → V(S) → Task1 실행 재개
```

**문제의 핵심:** High-Priority Task 1이 Low-Priority Task 3를 기다리는 동안, Medium-Priority Task 2가 Task 3를 선점하여 Task 1보다 먼저 실행됩니다. 이는 Task 2의 우선순위가 실질적으로 Task 1보다 높아지는 **무제한 우선순위 역전(Unbounded Priority Inversion)**입니다.

#### 해결 기법 1: 우선순위 상속(Priority Inheritance Protocol, PIP)

**원리:** 높은 우선순위 태스크가 자원 대기 시, **해당 자원을 점유한 낮은 우선순위 태스크의 우선순위를 임시로 높은 우선순위로 상속**시켜 빠르게 자원을 해제하도록 함

```
시간 →   t1   t2   t3   t4   t5   t6   t7
Task3:   [P(S)]         [↑높은우선순위로 올림]─[V(S)][원래 우선순위로]
Task2:               [선점시도] [블록됨!]               [실행]
Task1:         [P(S)블록]                   [실행]─────────

t3: Task1이 P(S) 대기 → Task3의 우선순위가 Task1 수준으로 상속
t4: Task3가 높은 우선순위로 Task2 선점 방지
t5: Task3가 V(S) 완료 → Task3 원래 우선순위로 복귀
t6: Task1이 S 획득하여 실행 재개
t7: Task1 완료 후 Task2 실행
```

**장점:** 구현 상대적으로 단순  
**단점:** 
- **연쇄 블로킹(Chained Blocking):** 자원이 여러 개일 때 블로킹이 연쇄적으로 발생할 수 있음
- **데드락 방지 안 됨**

#### 해결 기법 2: 우선순위 상한 프로토콜(Priority Ceiling Protocol, PCP)

**원리:** 각 세마포어에 **해당 자원을 사용하는 모든 태스크 중 최고 우선순위와 동일한 우선순위 상한(Priority Ceiling)**을 미리 할당. 태스크가 세마포어를 획득하면 즉시 우선순위를 상한으로 올림

```
세마포어 S의 상한 = Task1의 우선순위 (S를 사용하는 태스크 중 최고)

시간 →   t1    t2   t3    t4   t5   t6
Task3:   [P(S)→우선순위=상한]─[V(S)→원래로]
Task2:          [선점 시도] → 블록됨 (Task3의 우선순위가 상한이므로)
Task1:                                    [P(S)] [실행]

Task3가 S를 획득한 순간 우선순위=상한(Task1 수준)
→ Task2, Task1 모두 Task3를 선점 불가
→ Task3가 빠르게 임계 구역을 완료하고 S 해제
→ 데드락 방지, 우선순위 역전 방지
```

**장점:**
- 데드락 방지 가능
- 블로킹 예측 가능 (최악 1회 블로킹 보장)  
**단점:** 불필요한 우선순위 상승이 발생할 수 있음

#### 두 기법 비교

| 구분 | 우선순위 상속(PIP) | 우선순위 상한(PCP) |
|---|---|---|
| **우선순위 상승 시점** | 높은 우선순위 태스크가 블록될 때 | 세마포어 획득 시 즉시 |
| **데드락 방지** | 안 됨 | 가능 |
| **블로킹 횟수** | 다중 블로킹 가능 | 최악 1회 블로킹 |
| **구현 복잡도** | 낮음 | 높음 (상한 값 사전 계산) |
| **사용 사례** | Linux/POSIX PTHREAD_PRIO_INHERIT | VxWorks, LynxOS RTOS |

### 3. 핵심 키워드 (Must-Have)

- **우선순위 역전(Priority Inversion):** 고우선순위 태스크가 저우선순위 태스크에 의해 지연
- **무제한 우선순위 역전:** 중간 우선순위 태스크가 끼어들어 지연이 무한히 증가
- **우선순위 상속(PIP):** 자원 점유 태스크 우선순위를 대기자 수준으로 임시 상속
- **우선순위 상한(PCP):** 세마포어 획득 시 미리 정의된 상한으로 즉시 우선순위 상승
- **P(S) / V(S) 연산:** 세마포어 대기(감소) / 해제(증가)
- **Mars Pathfinder:** 우선순위 역전 실제 발생 사례

### 4. 맥락과 비교

**Mars Pathfinder 사례 (1997):**

화성 탐사선에서 정보 버스(information bus)를 관리하는 태스크가 저우선순위 기상 데이터 태스크에 의해 지연되었고, 고우선순위 통신 태스크가 정보 버스를 기다리는 우선순위 역전이 발생했습니다. 워치독 타이머가 시스템을 리셋시켰으며, 원인 분석 후 우선순위 상속을 소프트웨어 패치로 활성화하여 해결했습니다.

### 5. 실무 제언

**Linux 실시간 커널과 우선순위 역전**

- **챌린지:** 일반 Linux 커널은 우선순위 역전 문제를 완전히 해결하지 않습니다. `PREEMPT_RT` 패치를 적용한 실시간 Linux는 거의 모든 커널 잠금을 우선순위 상속 뮤텍스로 교체하여 실시간 보장을 강화합니다.
- **제언:** 실시간 Linux(`PREEMPT_RT`) 환경에서 `pthread_mutexattr_setprotocol(attr, PTHREAD_PRIO_INHERIT)` 설정으로 우선순위 상속 뮤텍스를 명시적으로 활성화해야 합니다. 산업용 로봇, 의료 장비 등 안전 임계 리눅스 시스템에서 이 설정의 누락이 심각한 타이밍 위반을 야기합니다.
