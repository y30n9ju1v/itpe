## 문제

Real Time Scheduling이 갖는 문제 중 우선순위 역전(Priority Inversion)이 있다. Task 1, Task 2, Task 3 순으로 우선순위가 낮다고 할 때 우선순위 역전을 사례기반으로 설명하고, 우선순위 역전을 해결하기 위한 2가지 기법에 대하여 설명하시오.(단 P, V연산을 사용한다)

## 해설

### 출제 배경 및 의도

실시간 운영체제(RTOS)는 로봇·자동차·항공·의료 장비 등 안전 임계 시스템의 핵심 소프트웨어입니다. 우선순위 기반 스케줄링은 RTOS의 근간이나, 공유 자원을 사용할 때 우선순위 역전이라는 예상치 못한 현상이 발생합니다. 1997년 화성 탐사선 Mars Pathfinder가 우선순위 역전으로 인한 워치독 타이머 만료로 반복 재부팅된 사례는 이 문제의 심각성을 상징적으로 보여줍니다.

126회 관리 문제로 P/V 연산 기반 사례 제시와 해결 기법 2가지 설명이 요구됩니다. 채점관은 우선순위 역전이 발생하는 시나리오를 타임라인으로 명확히 표현하고, 우선순위 상속(PIP)과 우선순위 상한(PCP)의 원리 차이를 정확히 비교하는지를 평가합니다.

### 1. 우선순위 역전(Priority Inversion), 개요

정 의  • 고우선순위 태스크가 공유 자원 대기로 인해 저우선순위 태스크보다 늦게 실행되는 실시간 스케줄링 이상 현상
       - 세마포어 P/V 연산으로 보호된 임계 구역(Critical Section)에서 자원 충돌 시 발생

```
우선순위 역전 발생 시나리오 (Task1>Task2>Task3, 세마포어 S):

시간 →  t1    t2    t3    t4    t5    t6    t7    t8
Task3: [P(S)] ──────────────────────────────── [V(S)]
Task2:               ──────── [선점!] ──────────────
Task1:        [도착] [P(S)블록!]                [실행]
               ↑                  ↑
          Task1 블록       Task2가 Task3 선점
          (S 점유 중)      → Task1보다 먼저 실행!
```

- 중간 우선순위 Task2가 Task1보다 먼저 실행되는 무제한 우선순위 역전(Unbounded Priority Inversion) 발생

### 2. 우선순위 역전 해결 기법

1) 우선순위 상속 프로토콜(Priority Inheritance Protocol, PIP)

| 구분 | 항목 | 설명 |
|------|------|------|
| 원리 | 동작 | 고우선순위 태스크가 자원 대기 시, 해당 자원 점유 태스크의 우선순위를 대기자 수준으로 임시 상속 |
| | 복귀 | V(S) 후 원래 우선순위로 복귀 |
| 장점 | - | 구현이 상대적으로 단순 |
| 단점 | 연쇄 블로킹 | 자원이 여러 개일 때 블로킹이 연쇄 발생 가능 |
| | 데드락 | 방지 불가 |
| 적용 | OS | Linux/POSIX PTHREAD_PRIO_INHERIT |

```
PIP 적용 시 타임라인:

시간 →  t1    t2    t3    t4    t5    t6    t7
Task3: [P(S)] ── [↑상속: Task1 수준] ── [V(S)] ── [복귀]
Task2:               [선점 시도] [블록!]          [실행]
Task1:        [P(S)블록]                 [S획득][실행]

t3: Task1 블록 → Task3 우선순위가 Task1 수준으로 상속
t4: Task3가 높은 우선순위로 Task2 선점 방지
t5: Task3 V(S) → 우선순위 복귀 → Task1 실행
```

- PIP는 자원 점유 태스크가 빠르게 임계 구역을 벗어나도록 유도하여 역전 시간을 단축

2) 우선순위 상한 프로토콜(Priority Ceiling Protocol, PCP)

| 구분 | 항목 | 설명 |
|------|------|------|
| 원리 | 동작 | 세마포어마다 해당 자원 사용 태스크 중 최고 우선순위(상한)를 사전 정의, P(S) 획득 즉시 상한으로 우선순위 상승 |
| | 시점 | 세마포어 획득 순간 즉시 (블록 발생 전) |
| 장점 | 데드락 방지 | 가능 |
| | 블로킹 예측 | 최악 1회 블로킹 보장 |
| 단점 | - | 불필요한 우선순위 상승 발생 가능 |
| | 복잡도 | 상한 값 사전 계산 필요 |
| 적용 | OS | VxWorks, LynxOS RTOS |

```
PCP 적용 시 타임라인 (S의 상한 = Task1 우선순위):

시간 →  t1    t2    t3    t4    t5    t6
Task3: [P(S)→즉시 상한으로 상승] ── [V(S)→복귀]
Task2:        [선점 시도] [블록!]              [실행]
Task1:                               [P(S)] [실행]

Task3가 S 획득 즉시 우선순위=상한 → Task2/Task1 선점 불가
```

- PCP는 PIP보다 강력하나, 모든 세마포어의 상한 값을 설계 시점에 정확히 계산해야 함

### 3. 두 기법 비교 및 적용 지침

| 구분 | 우선순위 상속(PIP) | 우선순위 상한(PCP) |
|------|------|------|
| 우선순위 상승 시점 | 고우선순위 태스크가 블록될 때 | 세마포어 획득 즉시 |
| 데드락 방지 | 불가 | 가능 |
| 블로킹 횟수 | 다중 블로킹 가능 | 최악 1회 |
| 구현 복잡도 | 낮음 | 높음 (상한 사전 계산) |
| 사용 OS | Linux POSIX, FreeRTOS | VxWorks, LynxOS |
| 적합 환경 | 일반 실시간 시스템 | 안전 임계 시스템 |

Mars Pathfinder(1997)는 정보 버스 관리 태스크가 저우선순위 기상 데이터 태스크에 의해 지연되어 고우선순위 통신 태스크 대기 중 워치독 타이머가 만료된 우선순위 역전 사례로, 우선순위 상속 활성화 패치로 해결됨.  "끝"

### 실무 제언

**Linux 실시간 커널에서 우선순위 역전 방지**
- **챌린지**: 일반 Linux 커널은 우선순위 역전을 완전히 해결하지 않으며, PREEMPT_RT 패치 없이는 커널 잠금 구간에서 역전이 발생 가능
- **제언**: 실시간 Linux(PREEMPT_RT) 환경에서 `pthread_mutexattr_setprotocol(attr, PTHREAD_PRIO_INHERIT)` 설정으로 우선순위 상속 뮤텍스를 명시적으로 활성화하고, 산업용 로봇·의료 장비 등 안전 임계 시스템에서는 반드시 적용

**RTOS 설계 시 자원 관리 원칙**
- **챌린지**: 임계 구역이 길수록 우선순위 역전 지속 시간이 증가하여 실시간 마감(Deadline) 위반 위험
- **제언**: 임계 구역을 최소화(빠른 P→작업→V), 비차단(Non-blocking) 알고리즘 검토, 자원 계층(Resource Ordering)으로 순환 대기 방지

**멀티코어 환경의 우선순위 역전**
- **챌린지**: 멀티코어 RTOS에서 서로 다른 코어의 태스크가 동일 세마포어를 경합할 때 단일코어 대비 우선순위 역전 패턴이 복잡해짐
- **제언**: 멀티프로세서 PCP(MPCP) 또는 글로벌 EDF 스케줄러 적용, 코어별 자원 친화성(Affinity) 설정으로 경합 최소화
