## Part 1. 컴퓨터 기본 구조 — 모든 것의 시작

### 1-1. 컴퓨터 기본구조와 Stored-program 개념

#### "프로그램을 메모리에 넣는다"는 한 가지 아이디어가 세상을 바꿨습니다

1940년대 초, 컴퓨터를 다른 계산에 쓰려면 **전선을 손으로 다시 꽂아야 했습니다.** 하루 이틀이 아니라 여러 날이 걸리기도 했죠. 이 불편함을 근본적으로 해결한 아이디어가 바로 폰 노이만(John von Neumann)의 **"프로그램도 데이터처럼 메모리에 저장하자"**는 발상입니다. 이 단 하나의 아이디어가 현대 컴퓨터 산업과 소프트웨어 산업 전체를 탄생시켰습니다.

컴퓨터 시스템은 다섯 가지 핵심 부품으로 이루어집니다.

* **입력장치:** 키보드나 마우스처럼 외부 세계의 정보를 컴퓨터 안으로 전달하는 문입니다.
* **출력장치:** 모니터나 프린터처럼 컴퓨터가 계산한 결과를 사람이 알아볼 수 있는 형태로 내보내는 창문입니다.
* **기억장치:** 명령어와 데이터를 저장하는 보관함입니다. 전기가 있는 동안만 기억하는 주기억장치(RAM)와 전기가 꺼져도 남는 보조기억장치(SSD, HDD)로 나뉩니다.
* **연산장치(ALU):** 덧셈, 뺄셈, 논리 비교처럼 실제 계산을 처리하는 두뇌의 계산 담당 부위입니다.
* **제어장치(CU):** 명령어를 해독하고 나머지 네 장치에게 "지금 무엇을 하라"고 지휘하는 지휘관입니다.

이 중 연산장치와 제어장치를 합쳐 우리가 아는 **CPU(중앙처리장치)**라고 부릅니다.

**Stored-program(저장 프로그램) 방식**의 핵심은 이렇습니다. CPU는 메모리에서 명령어를 한 줄씩 꺼내(Fetch), 그 뜻을 해석하고(Decode), 계산을 수행합니다(Execute). 이 세 단계를 끊임없이 반복하며 프로그램 전체를 실행해 나갑니다. 덕분에 하드웨어는 그대로 두고 메모리 안의 내용(소프트웨어)만 바꾸면 컴퓨터가 완전히 다른 일을 할 수 있게 된 것입니다.

> **단점도 있습니다 — 폰 노이만 병목(Von Neumann Bottleneck)**
> 명령어와 데이터가 같은 메모리, 같은 버스를 공유하다 보니, 아무리 CPU가 빨라도 메모리에서 데이터가 오기를 기다리며 시간을 낭비하는 구조적 한계가 생깁니다. 이를 해결하기 위해 캐시메모리, 파이프라인, 하버드 아키텍처 같은 기술들이 발전했습니다.

## Part 2. 주소지정방식 — 데이터 위치를 가리키는 방법

### 2-1. CPU 주소지정방식(Addressing Mode)

#### "데이터가 어디 있어?" — 명령어가 피연산자를 찾는 여섯 가지 방법

명령어는 두 가지로 이루어집니다. "**무엇을 할 것인가(연산코드, Opcode)**"와 "**어떤 데이터로 할 것인가(피연산자, Operand)**"입니다. 주소지정방식이란 바로 이 피연산자를 어떻게 찾아낼 것인가에 대한 방법입니다.

내비게이션에 목적지를 입력하는 방식이 다양하듯("서울역"이라고 이름을 치거나, "위도 37.5, 경도 126.9"라는 좌표를 입력하거나, "현재 위치에서 북쪽으로 500m"라고 상대 위치를 입력하는 것처럼) CPU도 필요한 데이터의 위치를 여러 방식으로 지정합니다.

* **즉시 주소지정(Immediate):** 데이터가 명령어 안에 직접 들어있습니다. "5를 더해라"처럼 '5' 자체가 명령어 안에 박혀 있는 방식으로, 메모리를 전혀 읽을 필요가 없어 가장 빠릅니다.
* **직접 주소지정(Direct):** "123번지에 가서 값을 가져와라"처럼 메모리 주소를 명령어에 직접 씁니다. 메모리를 한 번만 참조하면 됩니다.
* **간접 주소지정(Indirect):** "123번지에 가면 진짜 주소가 적혀 있다"는 2단계 방식입니다. 유연하지만 메모리를 두 번 읽어야 합니다.
* **레지스터 직접지정(Register):** 데이터가 CPU 내부 레지스터에 있습니다. 메모리보다 수십 배 빠른 레지스터를 직접 쓰므로 매우 빠릅니다.
* **레지스터 간접지정(Register Indirect):** 레지스터에 주소가 들어있고, 그 주소로 메모리를 참조합니다.
* **변위 주소지정(Displacement):** 레지스터값과 명령어 안의 변위값을 더해 실제 주소를 만듭니다. 배열 원소를 순서대로 접근하거나 함수 호출 시 지역 변수를 참조할 때 매우 유용합니다.

메모리 접근 횟수가 적을수록 빠르지만 표현할 수 있는 주소 범위가 좁아집니다. 현대 CPU는 상황에 따라 가장 적합한 방식을 자동으로 선택합니다.

**RISC(ARM)와 CISC(x86)의 설계 철학 차이**

ARM 같은 RISC 프로세서는 메모리 접근을 오직 `LDR`(로드)과 `STR`(스토어) 두 명령어로만 허용합니다. `ADD` 같은 연산 명령어는 반드시 레지스터끼리만 동작합니다. 마치 "창고(메모리)에서 재료를 꺼내 작업대(레지스터)에 올려둔 다음, 작업대에서만 요리해라"는 규칙입니다.

```
ARM (RISC)                          x86 (CISC)
LDR R1, [R2, #8]  ← 메모리→레지스터   ADD [EBP+8], EAX  ← 메모리 직접 연산 가능
ADD R1, R1, R3    ← 레지스터 간 연산
STR R1, [R2, #8]  ← 레지스터→메모리
```

이 단순한 규칙 덕분에 명령어 길이가 고정되어 파이프라인이 간단해지고, 컴파일러가 레지스터 할당을 최적화하기 쉬워집니다. 컴파일러는 배열 접근에는 인덱스 방식, 포인터에는 레지스터 간접 방식, 함수 분기에는 PC 상대 방식을 우선 사용하며, 레지스터가 부족할 때만 메모리로 '스필(Spill)'합니다.

## Part 3. 가상메모리 — 메모리보다 넓은 세상 만들기

### 3-1. 가상메모리 관리기법과 단편화 해결방안

#### "4GB RAM에서 16GB 게임을 어떻게 실행하나요?"

컴퓨터의 물리 메모리(RAM)는 비싸고 용량이 제한되어 있습니다. 그런데 우리는 RAM보다 훨씬 큰 프로그램을 거뜬히 실행합니다. 어떻게 가능한 걸까요? 바로 **가상메모리(Virtual Memory)** 덕분입니다.

가상메모리의 핵심 아이디어는 이렇습니다. 프로그램 전체를 한꺼번에 메모리에 올리는 대신, **지금 당장 필요한 부분만 메모리에 올려두고** 나머지는 SSD에 잠시 내보내 두는 것입니다. 이를 **요구 페이징(Demand Paging)**이라 합니다. 필요할 때만 가져오는 "필요즉시 배달" 전략입니다.

가상메모리를 구현하는 방식은 크게 두 가지입니다.

**① 페이징(Paging) — 모든 것을 같은 크기의 상자에 담기**

메모리를 **4KB** 같은 고정 크기로 잘게 쪼갭니다. 가상 주소 공간의 조각을 **페이지(Page)**, 물리 메모리의 조각을 **프레임(Frame)**이라 부릅니다. 페이지 테이블이라는 지도를 통해 어느 페이지가 어느 프레임에 들어있는지 관리합니다.

* **장점:** 고정 크기로 나누니 남은 공간을 꽉 채우기 좋습니다. 외부 단편화가 없습니다.
* **단점:** 상자보다 내용물이 작으면 상자 안에 빈 공간이 생깁니다. 이를 **내부 단편화**라 합니다.

**② 세그멘테이션(Segmentation) — 논리적 단위로 나누기**

코드, 데이터, 스택처럼 **프로그램의 논리적 의미**에 맞게 가변 크기로 자릅니다. 사람의 사고방식과 일치하기 때문에 보호(코드 영역은 쓰기 금지 등)와 공유가 쉽습니다.

* **장점:** 의미 단위로 나뉘니 보호와 공유가 직관적입니다.
* **단점:** 크기가 제각각이라 메모리에 올리고 내리다 보면 사이사이에 자투리 빈 공간이 생깁니다. 이를 **외부 단편화**라 합니다.

**현대 x86-64의 4단계 페이지 테이블**

오늘날 64비트 운영체제(Linux, Windows)는 세그멘테이션을 사실상 쓰지 않고, 4단계 페이지 테이블만으로 48비트 가상 주소 전체를 관리합니다. 48비트는 9+9+9+9+12비트로 나뉩니다.

```
48비트 가상주소
┌────────┬────────┬────────┬────────┬──────────────┐
│ PML4   │  PDPT  │   PD   │   PT   │   Offset     │
│ 9 bit  │ 9 bit  │ 9 bit  │ 9 bit  │   12 bit     │
└────────┴────────┴────────┴────────┴──────────────┘
CR3 레지스터 → PML4 → PDPT → PD → PT → 물리 프레임
```

CR3 레지스터가 PML4 테이블의 물리 주소를 가리키며, 프로세스가 바뀔 때 CR3를 교체하는 것만으로 주소 공간이 전환됩니다. TLB가 없으면 주소 변환 한 번에 메모리를 4번이나 읽어야 하므로, 4단계 구조일수록 TLB의 중요성이 더욱 커집니다.

**단편화 해결방법:**

* **통합(Coalescing):** 인접한 빈 공간 두 개를 합쳐서 하나의 큰 빈 공간으로 만듭니다.
* **압축(Compaction):** 모든 프로세스를 한쪽으로 밀어붙여 단편화된 공간을 하나로 모읍니다. 가장 완벽하지만 시간이 오래 걸립니다.
* **버디 시스템(Buddy System):** 2의 거듭제곱 크기로만 메모리를 할당하여 분할과 병합을 빠르게 합니다.

### 3-2. 페이지 교체 알고리즘(FIFO, LRU, LFU)

#### "방이 꽉 찼는데 새 손님을 받으려면 누군가를 내보내야 합니다"

메모리가 꽉 찬 상태에서 새 페이지가 필요하면 기존의 페이지 중 하나를 내보내야 합니다. **어느 페이지를 내보낼 것인가?** 이 결정이 성능을 크게 좌우합니다. 잘못 내보냈다가 금방 다시 필요하면 또 디스크에서 읽어와야 하고, 이 비용은 매우 큽니다.

대표적인 세 가지 교체 알고리즘을 비교합니다.

* **FIFO (First In First Out) — 먼저 들어온 것 먼저 내보내기**
  가장 먼저 들어온 페이지를 내보냅니다. 구현이 단순합니다. 하지만 가장 오래된 페이지가 지금도 자주 쓰이는 중요한 페이지일 수 있는데도 무조건 내보내는 단점이 있습니다. 심지어 페이지 프레임 수가 늘어났는데 오히려 페이지 폴트가 더 많아지는 **벨라디의 이상현상(Belady's Anomaly)**이 발생할 수 있습니다.

* **LRU (Least Recently Used) — 가장 오랫동안 사용 안 된 것 내보내기**
  최근에 사용된 페이지는 앞으로도 쓰일 가능성이 높다(시간 지역성)는 원리를 활용합니다. 가장 오래 방치된 페이지를 내보냅니다. 일반적으로 FIFO보다 성능이 좋으며 실무에서 가장 많이 사용됩니다. 다만 사용 시점을 기록하는 타임스탬프나 참조 기록이 필요해 하드웨어 비용이 있습니다.

* **LFU (Least Frequently Used) — 사용 횟수가 가장 적은 것 내보내기**
  참조 빈도가 가장 낮은 페이지를 내보냅니다. 자주 쓰이는 인기 페이지는 남기고 거의 안 쓰이는 페이지를 내보내는 합리적인 발상입니다. 하지만 오래 전에 많이 쓰였지만 지금은 필요 없는 페이지를 내보내기 어렵다는 단점이 있습니다.

> **성능 비교(페이지 번호 참조 순서 4, 3, 2, 5, 2, 1, 3, 2, 6, 3, 4 / 3 프레임):**
> FIFO: 적중률 2/11, LRU: 3/11, LFU: 3/11

### 3-3. 가상주소 → 물리주소 변환 과정 및 TLB

#### "GPS 주소를 실제 도로 좌표로 바꾸는 과정"

프로그램은 가상 주소를 사용합니다. 하지만 실제로 데이터를 읽고 쓰려면 물리 주소가 필요합니다. 이 변환을 담당하는 하드웨어가 **MMU(Memory Management Unit)**이고, 변환 속도를 높이는 캐시가 **TLB(Translation Lookaside Buffer)**입니다.

**페이지 테이블만 사용하는 경우:**
가상 주소에서 페이지 번호를 뽑아 → 페이지 테이블에서 프레임 번호를 조회 → 프레임 번호 + 오프셋으로 물리 주소 완성. 메모리를 두 번(테이블 조회 + 실제 데이터) 읽어야 해서 느립니다.

**TLB를 사용하는 경우:**
TLB는 최근에 번역한 가상→물리 주소 쌍을 소수(수십~수백 개) 저장해 두는 초고속 캐시입니다.

```
가상 주소 요청
      │
      ▼
  TLB 조회
      ├── TLB Hit (약 95~99% 확률) → 즉시 물리 주소 획득 (매우 빠름!)
      └── TLB Miss → 페이지 테이블 조회 → 물리 주소 획득 + TLB에 결과 저장
```

TLB 덕분에 대부분의 주소 변환이 메모리 접근 없이 빠르게 이루어집니다. 컨텍스트 스위치가 일어나면 TLB를 비워야(Flush) 하는 비용이 있지만, 현대 CPU는 프로세스 식별자(ASID)를 TLB에 함께 저장해 이 비용을 줄입니다.

**멀티코어 환경의 TLB Shootdown**

멀티코어 시스템에서는 한 가지 까다로운 문제가 생깁니다. 코어 0이 어떤 페이지를 해제(munmap)하면, 다른 코어들의 TLB에 남아있는 구 매핑도 모두 지워야 합니다. 그렇지 않으면 코어 1이 이미 해제된 메모리를 가리키는 낡은 주소로 접근할 수 있기 때문입니다. 이 무효화 과정을 **TLB Shootdown**이라 합니다.

코어 0은 다른 모든 코어에 **IPI(Inter-Processor Interrupt)**를 보내 "이 주소 범위의 TLB를 지워라"라고 알립니다. 코어 수가 많을수록 IPI 오버헤드도 커지므로, 대형 서버에서 `fork` · `munmap`이 빈번하면 TLB Shootdown이 CPU 사용률의 수%를 차지하기도 합니다.

## Part 4. 캐시메모리 — CPU와 메모리 사이의 속달 택배

### 4-1. 캐시메모리 동작 순서와 주소 사상

#### "책상 서랍과 창고의 속도 차이를 줄이는 비서"

현대 CPU는 1초에 수십억 번 계산을 합니다. 반면 주기억장치(RAM)는 데이터를 전달하는 데 CPU보다 100배 이상 시간이 걸립니다. 빠른 일꾼(CPU)과 느린 창고(RAM) 사이에 속달 배달부가 있어야 일이 원활하게 돌아가겠죠? 그 역할을 하는 것이 **캐시메모리(Cache Memory)**입니다.

캐시가 효과적인 이유는 프로그램의 두 가지 특성 덕분입니다.

* **시간 지역성(Temporal Locality):** 방금 쓴 데이터는 곧 다시 쓸 가능성이 높습니다. 반복문(for loop) 안의 변수가 대표적인 예입니다.
* **공간 지역성(Spatial Locality):** 방금 읽은 주소 근처의 데이터도 곧 읽을 가능성이 높습니다. 배열을 순서대로 읽을 때가 이에 해당합니다.

CPU가 데이터를 요청하면 먼저 캐시를 뒤집니다. 있으면 **캐시 히트(Cache Hit)**, 없으면 **캐시 미스(Cache Miss)**입니다. 히트율이 높을수록 시스템이 빨라집니다.

**주소 사상(Address Mapping) — 데이터를 캐시의 어느 칸에 넣을까?**

* **직접 사상(Direct Mapping):** 메모리 주소를 나머지 연산으로 캐시의 특정 한 칸에 고정 배치합니다. 구현이 단순하고 빠르지만, 같은 캐시 칸을 원하는 데이터가 두 개면 계속 서로를 밀어내 성능이 뚝 떨어지는 충돌이 생깁니다.
* **완전 연관 사상(Fully Associative Mapping):** 어느 칸에든 자유롭게 넣을 수 있습니다. 충돌이 없어 히트율이 높지만, 찾을 때 모든 칸을 한꺼번에 비교해야 해서 하드웨어가 복잡하고 비쌉니다.
* **집합 연관 사상(Set Associative Mapping):** 캐시를 여러 집합(Set)으로 나누고, 집합 안에서는 자유롭게 배치합니다. 직접 사상과 완전 연관 사상의 중간 지점으로, 현대 CPU가 실제로 채택하는 방식입니다(보통 4-way 또는 8-way 집합 연관).

---

### 4-2. 캐시 쓰기정책과 캐시 일관성(Cache Coherence)

#### "여러 명이 같은 메모장을 수정하면 나중에 누구 버전이 맞을까요?"

CPU가 데이터를 수정할 때 캐시에만 쓰고 메모리에는 나중에 반영할지, 아니면 항상 동시에 두 곳 모두에 쓸지 결정해야 합니다. 이것이 **쓰기 정책(Write Policy)**입니다.

* **Write-through(즉시 쓰기):** 캐시와 주기억장치에 동시에 씁니다. 항상 메모리가 최신 상태로 유지되어 안전하지만, 쓰기 연산마다 느린 메모리 접근이 일어나 속도가 느립니다.
* **Write-back(나중에 쓰기):** 일단 캐시에만 씁니다. 나중에 그 캐시 블록이 교체될 때 메모리에 한꺼번에 반영합니다. 빠르지만 캐시와 메모리 간 내용이 일시적으로 달라집니다.

멀티코어 CPU에서는 더 복잡한 문제가 생깁니다. 코어 1과 코어 2가 각자의 캐시에 같은 메모리 데이터를 복사해 두고 서로 다르게 수정한다면 어느 값이 올바른 값일까요? 이 문제가 바로 **캐시 일관성(Cache Coherence)** 문제입니다.

대표적인 해결책이 **MESI 프로토콜**입니다. 각 캐시 블록이 Modified(내가 수정함), Exclusive(나만 가짐), Shared(다른 코어와 공유 중), Invalid(유효하지 않음) 네 가지 상태 중 하나에 있도록 추적하여, 한 코어가 데이터를 수정하면 다른 코어의 캐시에서 해당 블록을 무효(Invalid) 상태로 만들어 최신 데이터를 강제로 다시 읽게 합니다.

## Part 5. 커널과 운영체제 — 하드웨어를 지키는 보안관

### 5-1. 이중 모드(Dual Mode) 구조

#### "일반 시민과 경찰관의 권한이 달라야 하는 이유"

만약 모든 프로그램이 CPU, 메모리, 디스크에 마음대로 접근할 수 있다면 어떻게 될까요? 악성 프로그램 하나가 다른 프로그램의 데이터를 마음대로 읽거나 운영체제를 멈춰버릴 수 있습니다. 이를 막기 위해 현대 CPU는 두 가지 작동 모드를 제공합니다.

* **사용자 모드(User Mode):** 일반 응용 프로그램이 실행되는 모드입니다. 하드웨어에 직접 접근하는 명령어(특권 명령어)를 실행하려 하면 즉시 차단됩니다. 잘못된 코드가 시스템 전체를 망가뜨리지 못하도록 보호받는 울타리 안에 있는 셈입니다.
* **커널 모드(Kernel Mode) / 시스템 모드:** 운영체제의 핵심인 커널만 진입할 수 있는 모드입니다. 모든 명령어를 실행할 수 있고, 모든 하드웨어에 자유롭게 접근할 수 있습니다.

**어떻게 커널 모드로 진입할까?**

사용자 프로그램이 파일을 읽거나 네트워크 통신을 하려면 운영체제의 도움이 필요합니다. 이때 **시스템 콜(System Call)**을 통해 "커널님, 이것 좀 해주세요"라고 요청합니다. 시스템 콜이 일어나면 CPU가 자동으로 커널 모드로 전환되고, 운영체제가 요청을 처리한 뒤 다시 사용자 모드로 복귀합니다.

인터럽트가 발생했을 때도 자동으로 커널 모드로 전환됩니다. 이중 모드 덕분에 한 프로그램이 충돌해도 운영체제와 다른 프로그램들은 안전하게 계속 실행될 수 있습니다.

## Part 6. 스케줄링 알고리즘 — CPU 시간표 짜기

### 6-1. CPU 스케줄링 알고리즘(선점/비선점)

#### "한 대의 주방 오븐을 열 명의 요리사가 나눠 써야 할 때"

CPU는 한 번에 한 가지 작업(프로세스)만 처리할 수 있습니다. 수십 개의 프로그램이 동시에 실행되는 것처럼 보이는 것은 사실 CPU가 아주 빠르게 번갈아가며 조금씩 처리하는 것입니다. **스케줄러**는 이 순서와 시간을 결정하는 교통 정리 담당자입니다.

**비선점 방식 — 내가 끝날 때까지 기다려요**

* **FCFS (First Come First Served):** 먼저 온 프로세스가 끝날 때까지 CPU를 독점합니다. 구현은 단순하지만 오래 걸리는 작업이 앞에 있으면 뒤의 짧은 작업들이 오래 기다려야 합니다(호위 효과).
* **SJF (Shortest Job First):** 실행 시간이 가장 짧은 프로세스를 먼저 처리합니다. 평균 대기 시간이 가장 짧아지는 이론상 최적이지만, 다음 작업이 얼마나 걸릴지를 미리 알아야 한다는 현실적 문제가 있습니다.

**선점 방식 — 필요하면 빼앗을 수 있어요**

* **라운드 로빈(Round Robin):** 각 프로세스에 동일한 시간(타임 퀀텀, 보통 10~100ms)을 번갈아 줍니다. 시간이 끝나면 강제로 CPU를 빼앗아 다음 프로세스에 넘깁니다. 응답 시간이 균등하고 공평해 현대 운영체제의 기본 방식입니다.
* **우선순위 스케줄링(Priority Scheduling):** 각 프로세스에 우선순위를 부여하고 높은 것부터 실행합니다. 선점/비선점 모두 가능합니다. 낮은 우선순위 프로세스가 영원히 실행되지 못하는 **기아(Starvation)** 문제가 있어 시간이 지날수록 우선순위를 높여주는 **에이징(Aging)** 기법으로 해결합니다.
* **다단계 큐(Multilevel Queue):** 프로세스를 종류별(대화형/배치/시스템)로 여러 큐에 분리하고 각 큐마다 다른 스케줄링 방식을 적용합니다.

---

### 6-2. 실시간 스케줄링 RM과 EDF

#### "비행기 착륙 제어는 마감 시간을 절대 어길 수 없습니다"

일반 컴퓨터와 달리 자동차 에어백, 공장 로봇 팔, 항공 제어 시스템에서는 "응답이 빠른 것"만으로는 부족합니다. **정해진 마감 시간(Deadline) 안에 반드시 처리되어야** 합니다. 이를 **실시간 시스템(Real-Time System)**이라 하고, 이를 위한 전용 스케줄링 알고리즘이 필요합니다.

* **RM (Rate Monotonic) — 주기가 짧을수록 우선순위가 높다**
  주기(Period)가 가장 짧은 태스크에 가장 높은 우선순위를 고정 부여합니다. 예를 들어 100ms마다 실행해야 하는 태스크가 500ms마다 실행하는 태스크보다 우선순위가 높습니다. CPU 이용률이 약 69% 이하이면 모든 마감 시간을 보장할 수 있다는 수학적 증명이 있어서 예측 가능성이 높습니다.

* **EDF (Earliest Deadline First) — 마감이 가장 급박한 것 먼저**
  가장 빨리 마감이 돌아오는 태스크를 우선 처리합니다. CPU 이용률이 100%까지도 이론상 마감 보장이 가능해 RM보다 효율적이지만, 런타임에 매번 우선순위를 재계산해야 해서 구현이 복잡하고 오버로드 시 동작이 불안정합니다.

## Part 7. 교착상태(Deadlock) — 누구도 움직이지 않는 교통 마비

### 7-1. 교착상태 발생 조건과 해결방안

#### "좁은 골목길에서 두 차가 마주쳤을 때, 아무도 양보하지 않으면?"

서울 구도심 좁은 골목에서 두 대의 차가 정반대 방향으로 오다가 딱 마주칩니다. 이 골목은 한 대도 겨우 지나갈 정도로 좁습니다. 먼저 가는 차가 후진해 양보해야 하는데, 두 운전자 모두 "내가 왜 후진해야 하지?" 하며 버티면 아무도 움직이지 못합니다. 이것이 **교착상태(Deadlock)**입니다.

프로세스 A가 자원 1을 갖고 자원 2를 기다리는데, 프로세스 B가 자원 2를 갖고 자원 1을 기다리면 두 프로세스 모두 영원히 멈춥니다.

**교착상태가 생기려면 4가지 조건이 동시에 충족되어야 합니다 (코프만 조건):**

1. **상호 배제:** 한 번에 한 프로세스만 자원을 쓸 수 있습니다.
2. **점유와 대기:** 자원을 쥔 채 다른 자원을 기다립니다.
3. **비선점:** 다른 프로세스의 자원을 강제로 빼앗을 수 없습니다.
4. **원형 대기:** A가 B를 기다리고, B가 C를 기다리며, C가 A를 기다리는 원형 구조가 형성됩니다.

**해결 방법은 4가지입니다:**

* **예방(Prevention):** 위 4가지 조건 중 하나가 절대 성립하지 않도록 설계합니다. 예를 들어 원형 대기를 막으려면 모든 자원에 번호를 매기고 항상 작은 번호 자원부터 요청하도록 강제합니다.
* **회피(Avoidance):** 자원을 요청할 때마다 "지금 이 요청을 허용해도 교착상태로 가지 않을까?"를 미리 계산합니다. 이것이 **은행원 알고리즘(Banker's Algorithm)**입니다. 은행이 고객 대출 요청마다 "이 대출 후에도 다른 고객들에게 줄 돈이 남는가?"를 확인하는 것과 같습니다.
* **탐지 후 복구(Detection & Recovery):** 교착상태가 생기도록 내버려 뒀다가 주기적으로 탐지하고, 발견되면 일부 프로세스를 강제 종료하거나 자원을 빼앗아 해결합니다.
* **무시:** 교착상태가 매우 드물게 발생하는 시스템에서는 그냥 재부팅으로 해결합니다. 실제로 많은 데스크톱 OS가 이 전략을 씁니다.

## Part 8. IPC — 프로세스들이 대화하는 방법

### 8-1. IPC(Inter Process Communication) 통신방식

#### "서로 다른 사무실에서 일하는 팀원들이 업무를 주고받는 법"

운영체제는 각 프로세스를 철저하게 격리시킵니다. 내 프로세스가 다른 프로세스의 메모리를 함부로 읽거나 쓸 수 없습니다. 하지만 협력이 필요할 때는 메시지를 주고받아야 합니다. 이를 위한 메커니즘이 **IPC(Inter Process Communication)**입니다.

* **파이프(Pipe):** 물이 흐르는 파이프처럼 한쪽에서 쓰고 다른 쪽에서 읽는 단방향 통신입니다. 부모-자식 프로세스 간 통신에 주로 쓰이며, 익명 파이프는 관계없는 프로세스 간에는 쓸 수 없습니다. 이름 있는 파이프(Named Pipe)는 이 제약을 풀어줍니다.
* **공유 메모리(Shared Memory):** 여러 프로세스가 같은 메모리 영역을 직접 접근합니다. 데이터를 복사할 필요가 없어 **가장 빠른 IPC 방식**입니다. 단, 동시 접근을 제어하는 동기화 메커니즘(세마포어 등)이 별도로 필요합니다.
* **메시지 큐(Message Queue):** 메시지를 우체통에 넣어두면 상대방이 나중에 꺼내가는 방식입니다. 비동기 통신이 가능하고 메시지에 우선순위를 부여할 수 있습니다.
* **소켓(Socket):** 네트워크를 통해 다른 컴퓨터의 프로세스와도 통신할 수 있습니다. 인터넷 통신의 근간이 되는 방식입니다.
* **시그널(Signal):** 특정 이벤트를 프로세스에 알리는 가장 가벼운 방식입니다. `Ctrl+C`를 누르면 SIGINT 시그널이 프로세스에 전달되어 종료되는 것이 대표적인 예입니다.

## Part 9. 동기화 — 공유 자원을 안전하게 쓰는 법

### 9-1. 프로세스 동기화 기법(Mutex, Semaphore, Monitor)

#### "화장실이 하나인데 여러 명이 동시에 들어가려 한다면?"

공유 메모리나 파일처럼 여러 프로세스가 동시에 접근하는 자원이 있을 때, 두 프로세스가 동시에 같은 값을 읽고 각자 수정하면 결과가 뒤죽박죽이 됩니다. 예를 들어 은행 잔액이 100만원일 때 두 트랜잭션이 동시에 읽어서 각자 10만원씩 입금하면 결과가 110만원이 아니라 여전히 100만원이 되는 참사가 생깁니다. 이를 방지하는 것이 **동기화(Synchronization)**입니다.

**임계 구역(Critical Section)**은 공유 자원에 접근하는 코드 구간입니다. 한 번에 하나의 프로세스만 임계 구역에 들어갈 수 있어야 합니다.

* **뮤텍스(Mutex, Mutual Exclusion) — 열쇠가 하나뿐인 화장실**
  자물쇠처럼 작동합니다. 임계 구역에 들어가기 전에 잠금(Lock)을 걸고, 나올 때 잠금을 풉니다(Unlock). 잠금이 걸려 있으면 다른 프로세스는 문 앞에서 기다립니다. **잠근 프로세스만 풀 수 있다는 것**이 핵심입니다.

* **세마포어(Semaphore) — 주차장 남은 자리 표시판**
  정수 카운터를 이용합니다. 자원이 3개 있으면 세마포어 값이 3으로 시작합니다. 자원을 하나 점유할 때마다 카운터를 1 감소시키고(P 연산), 반납할 때 1 증가시킵니다(V 연산). 카운터가 0이 되면 다음 요청은 대기합니다. 세마포어는 잠근 프로세스가 아닌 **다른 프로세스도 풀 수 있습니다** — 이 점에서 뮤텍스와 다릅니다.

* **모니터(Monitor) — 안전한 접근을 보장하는 관리인이 있는 VIP 라운지**
  프로그래밍 언어 수준에서 지원하는 고수준 동기화 도구입니다. 공유 자원과 그것을 접근하는 함수들을 하나의 캡슐로 묶고, 한 번에 한 프로세스만 안으로 들어올 수 있도록 컴파일러가 자동으로 보장해 줍니다. Java의 `synchronized` 키워드가 대표적인 예입니다. 개발자가 직접 잠금을 관리하지 않아도 되므로 실수할 여지가 줄어듭니다.

## Part 10. 프로세스 — 실행 중인 프로그램의 모든 것

### 10-1. 프로세스 상태 다이어그램, PCB, 생성 및 종료

#### "프로그램과 프로세스의 차이 — 악보와 연주의 차이"

악보는 그냥 종이 위의 음표들입니다. 하지만 오케스트라가 그 악보를 실제로 연주하는 순간, 악보는 생동감 있는 음악이 됩니다. 마찬가지로 **프로그램**은 디스크에 저장된 코드 파일이고, **프로세스**는 그 프로그램이 메모리에 올라와 CPU에 의해 실제로 실행되고 있는 살아있는 인스턴스입니다.

**프로세스의 생애 주기:**

```
생성(New) → 준비(Ready) → 실행(Running) → 대기(Waiting) → 종료(Terminated)
                 ↑              │
                 └──── 선점 ────┘
```

* 프로세스가 생성되면 **준비(Ready)** 상태로 대기합니다.
* 스케줄러가 선택하면 **실행(Running)** 상태로 CPU를 할당받습니다.
* 입출력(I/O)을 요청하면 완료될 때까지 **대기(Waiting)** 상태로 빠집니다.
* I/O가 완료되면 다시 **준비** 상태로 돌아와 차례를 기다립니다.

**PCB(Process Control Block) — 프로세스의 신분증**

운영체제는 각 프로세스의 정보를 **PCB**라는 자료 구조에 저장합니다. PCB에는 프로세스 ID, 현재 상태, 레지스터 값, 메모리 주소, 열린 파일 목록 등이 기록됩니다. 문맥 교환이 일어날 때 CPU는 현재 작업의 상태를 PCB에 저장해두고 다른 프로세스의 PCB를 불러와 작업을 이어갑니다.

## Part 11. 인터럽트 — 긴급 신호를 처리하는 메커니즘

### 11-1. 인터럽트 동작원리와 다중 인터럽트 처리

#### "요리하는 중에 초인종이 울리면 어떻게 하나요?"

주방에서 요리에 집중하다가 초인종이 울립니다. 요리를 잠시 멈추고 문을 열어 택배를 받은 뒤, 다시 주방으로 돌아와 요리를 계속합니다. 이것이 바로 **인터럽트(Interrupt)**의 동작 원리입니다.

CPU가 현재 작업을 처리하는 중에 키보드 입력, 타이머 만료, 디스크 읽기 완료, 프로그램 오류 같은 긴급한 사건이 발생하면, CPU는 하던 일을 잠시 멈추고 그 사건에 대응하는 처리 루틴(**ISR, Interrupt Service Routine**)을 실행한 뒤 원래 작업으로 복귀합니다.

**인터럽트 발생 원인:**

* **하드웨어 인터럽트:** 키보드, 마우스, 네트워크 카드 등 외부 장치가 CPU에 신호를 보냅니다.
* **소프트웨어 인터럽트(트랩):** 프로그램이 0으로 나누기를 시도하거나, 존재하지 않는 메모리 주소에 접근하거나, 시스템 콜을 호출할 때 발생합니다.

**인터럽트 우선순위와 벡터:**

인터럽트가 동시에 여러 개 발생할 수 있습니다. 이때 더 중요한 인터럽트(예: 정전 감지)가 덜 중요한 인터럽트(예: 마우스 이동)의 처리를 끊고 먼저 처리됩니다. 각 인터럽트 유형마다 어떤 ISR을 실행해야 하는지 기록한 표가 **인터럽트 벡터 테이블**입니다.

## Part 12. 문맥교환 — 배우가 무대에서 역할을 바꾸듯

### 12-1. 문맥(Context)과 문맥교환(Context Switch)

#### "배우가 역할을 바꾸려면 의상을 갈아입어야 합니다"

연극 무대에서 배우가 왕 역할을 하다가 신하 역할로 바꾸려면 의상을 갈아입고 말투도 바꿔야 합니다. CPU가 한 프로세스에서 다른 프로세스로 전환할 때도 이와 같은 준비 작업이 필요합니다. 이것이 **문맥교환(Context Switch)**입니다.

**문맥(Context)**이란 프로세스가 언제든 실행을 재개할 수 있도록 필요한 모든 정보입니다. 레지스터 값들(PC, SP, 범용 레지스터), 메모리 맵, 열린 파일 목록 등이 포함됩니다.

**문맥교환 과정:**

1. 타이머 인터럽트나 시스템 콜 등이 발생
2. 현재 프로세스 A의 문맥(레지스터 값 등)을 PCB_A에 저장
3. 다음에 실행할 프로세스 B의 PCB_B에서 문맥을 복원
4. 프로세스 B가 CPU를 넘겨받아 실행 재개

문맥교환 자체는 **순수 오버헤드**입니다. 교환하는 동안 CPU는 유용한 계산을 전혀 못 합니다. 현대 운영체제는 타임 퀀텀을 적절히 설정하고 문맥교환 횟수를 최소화하여 이 오버헤드를 줄입니다.

---

## Part 13. 파이프라인 — 공장 컨베이어 벨트로 CPU를 빠르게

### 13-1. CPU 파이프라인과 해저드(Hazard)

#### "세탁기, 건조기, 다림질대를 동시에 돌리면 더 빠릅니다"

빨래를 혼자서 세탁하고 → 건조하고 → 다림질하는 방식(비파이프라인)보다, 첫 번째 빨래를 건조기로 넘기는 동시에 세탁기에 두 번째 빨래를 넣으면 훨씬 빠르게 처리할 수 있습니다. CPU 파이프라인도 같은 원리입니다.

명령어 처리를 여러 단계(인출 → 해독 → 실행 → 쓰기)로 쪼개어, 각 단계를 동시에 다른 명령어에 적용합니다. 4단계 파이프라인이면 이론상 처리 속도가 4배 빨라집니다.

하지만 파이프라인이 항상 완벽하게 흐르지는 않습니다. **해저드(Hazard)**라 불리는 세 가지 방해 요소가 있습니다.

* **데이터 해저드:** 앞 명령어의 계산 결과를 뒤 명령어가 바로 써야 하는데, 아직 결과가 나오지 않은 경우입니다. 해결책은 결과를 레지스터에 쓰기 전에 다음 단계로 직접 전달하는 **전방 전달(Forwarding)**이나 그냥 기다리는 **버블(Bubble) 삽입**입니다.
* **제어 해저드:** 분기 명령어(if, while)가 나왔을 때 다음에 어느 명령어를 가져와야 할지 알 수 없는 경우입니다. **분기 예측(Branch Prediction)**으로 미리 가져와 처리하다가 예측이 틀리면 파이프라인을 비우고 다시 시작합니다.
* **구조적 해저드:** 여러 명령어가 동시에 같은 하드웨어 자원(예: 메모리 버스)을 사용하려 할 때 충돌이 생깁니다. 자원을 추가하거나 스케줄링으로 해결합니다.

## Part 14. 메모리 인터리빙 — 여러 줄을 동시에 읽기

### 14-1. 메모리 인터리빙 유형과 엑세스 활용방식

#### "책을 여러 권 동시에 펼쳐두면 한 권씩 볼 때보다 빠릅니다"

메모리는 기본적으로 한 번에 하나의 접근 요청만 처리합니다. CPU가 연속된 데이터를 빠르게 읽으려 할 때 메모리가 병목이 됩니다. **메모리 인터리빙(Memory Interleaving)**은 메모리를 여러 독립적인 모듈(뱅크)로 나누어 동시에 접근하는 방법으로 이 병목을 줄입니다.

예를 들어 4개의 메모리 뱅크가 있다면, 주소 0~3번은 각각 뱅크 0, 1, 2, 3에 분산 저장합니다. 주소 4~7번도 마찬가지입니다. 연속된 주소를 읽을 때 4개의 뱅크가 동시에 동작하므로 이론상 4배 빠른 대역폭을 얻을 수 있습니다.

* **저차원 인터리빙(Low-order):** 연속 주소가 다른 뱅크에 분산됩니다. 순차 접근에 매우 효과적입니다.
* **고차원 인터리빙(High-order):** 연속 주소가 같은 뱅크에 모입니다. 특정 뱅크만 집중 접근할 때 유리하지만, 순차 접근에는 덜 효과적입니다.

현대 DDR 메모리 시스템의 듀얼 채널, 쿼드 채널 구성이 바로 인터리빙의 실제 구현입니다.

## Part 15. 아키텍처 — 폰 노이만 vs 하버드

### 15-1. 폰 노이만 아키텍처와 하버드 아키텍처

#### "명령어 도로와 데이터 도로를 따로 만들면?"

우리가 앞서 배운 폰 노이만 방식은 명령어와 데이터가 **같은 메모리, 같은 버스**를 공유합니다. 이는 명령어를 가져올 때와 데이터를 읽을 때가 서로 경쟁하는 병목을 만듭니다.

**하버드 아키텍처**는 이를 해결합니다. 명령어 메모리와 데이터 메모리를 **완전히 분리**하여 각자의 전용 버스를 통해 동시에 접근할 수 있습니다. CPU가 명령어를 가져오는 동시에 데이터를 읽을 수 있어 성능이 향상됩니다.

| 비교 | 폰 노이만 | 하버드 |
|------|---------|--------|
| 메모리 구조 | 명령어와 데이터 공유 | 명령어와 데이터 분리 |
| 버스 | 단일 공유 버스 | 독립된 두 개의 버스 |
| 장점 | 단순하고 유연함, 자가 수정 코드 가능 | 명령어/데이터 동시 접근 가능, 빠름 |
| 단점 | 폰 노이만 병목 | 복잡하고 비용 높음 |
| 현대 사용 | 대부분의 범용 CPU | DSP, 마이크로컨트롤러, CPU 내부 캐시 구조 |

현대 x86 CPU는 겉으로는 폰 노이만(같은 메모리 공간)이지만, 내부적으로 L1 캐시를 명령어 캐시(I-Cache)와 데이터 캐시(D-Cache)로 분리하는 **변형 하버드 아키텍처**를 채택하여 두 방식의 장점을 모두 취합니다.

## Part 16. GPU와 CPU — 다른 목적을 위해 태어난 두 두뇌

### 16-1. GPU와 GPGPU, GPU-CPU 차이점

#### "올라운더 선수 한 명 vs 전문 선수 천 명"

CPU는 뛰어난 올라운더입니다. 복잡한 조건 분기, 순서대로 처리해야 하는 논리 연산 등 다양한 일을 빠르고 유연하게 처리합니다. 강력한 코어 몇 개(4~32개)로 복잡한 작업을 빠르게 처리하는 데 특화되어 있습니다.

반면 GPU는 전문 선수들의 집단입니다. 화면의 수백만 픽셀 각각에 같은 계산(색상 변환, 그림자 계산)을 동시에 적용하는 작업처럼, **수천 개의 단순한 코어가 동일한 연산을 대규모로 병렬 처리**합니다.

| 비교 | CPU | GPU |
|------|-----|-----|
| 코어 수 | 수십 개 | 수천~수만 개 |
| 코어 성능 | 매우 강력 | 상대적으로 단순 |
| 메모리 | 대용량 시스템 RAM 공유 | 전용 고대역폭 메모리(GDDR, HBM) |
| 특기 | 복잡한 순차 처리, OS 제어 | 대규모 병렬 연산, 행렬 계산 |
| 주 사용처 | 일반 프로그램 실행 | 게임 렌더링, AI 학습, 과학 계산 |

**GPGPU(General-Purpose GPU)**는 GPU를 그래픽 처리 이외의 일반 계산에 활용하는 것입니다. NVIDIA의 CUDA 플랫폼 덕분에 딥러닝 모델 훈련, 분자 시뮬레이션, 암호화폐 채굴 등에 GPU가 폭넓게 쓰이고 있습니다.

**2025년 GPU 트렌드 — NVIDIA Blackwell 세대**

2025년 출시된 NVIDIA B200/GB200(Blackwell 아키텍처)은 이전 세대 H100 대비 학습 5배·추론 30배 성능을 제공합니다. FP4/FP8 Tensor Core와 192GB HBM3e 메모리, 5세대 NVLink를 탑재해 수백 개 GPU를 하나의 거대한 가속기처럼 연결합니다. LLM 파라미터가 수천억 개를 넘어서면서 단일 GPU VRAM의 한계를 넘기 위해 이런 고대역폭·대용량 메모리 설계가 핵심 경쟁력이 되었습니다.

## Part 17. DMA — CPU를 대신하는 입출력 전담 요원

### 17-1. DMA 제어기 구조와 동작 모드

#### "사장이 택배를 직접 받지 않고 전담 직원을 두는 이유"

디스크에서 대용량 파일을 읽어 메모리로 옮긴다고 생각해 봅시다. 만약 CPU가 직접 "디스크에서 1바이트 읽어, 메모리에 저장해, 디스크에서 1바이트 읽어, 메모리에 저장해..."를 반복한다면 CPU의 귀중한 시간이 허드렛일에 낭비됩니다.

**DMA(Direct Memory Access)** 컨트롤러는 이 문제를 해결합니다. CPU가 DMA에게 "디스크 섹터 42번부터 1MB를 메모리 주소 0x8000부터 복사해 놔라"라고 지시하고 본인은 다른 중요한 계산에 집중합니다. DMA가 묵묵히 데이터 이동을 완료하면 인터럽트로 "다 됐어요!"라고 보고합니다.

**DMA 동작 모드:**

* **버스트 모드(Burst Mode):** DMA가 전송 완료 시까지 버스를 독점합니다. 가장 빠르지만 그동안 CPU가 메모리에 접근하지 못합니다.
* **사이클 스틸 모드(Cycle Steal Mode):** CPU 버스 사용 주기 사이사이를 틈타 한 번에 한 단어씩 조금씩 전송합니다. CPU 성능 저하가 적지만 전체 전송 속도가 느립니다.
* **블록 모드(Block Mode):** 지정된 블록 단위로 전송하며, 버스 제어권을 블록 단위로 양보합니다.

현대 컴퓨터에서 USB 전송, 사운드 카드, 네트워크 카드 등 모든 고속 입출력 장치가 DMA를 활용합니다.

## Part 18. 저장장치 — 데이터를 안전하게 보관하는 창고

### 18-1. RAID 레벨별 비교(0, 5, 6, 1+0, 5+0)

#### "계란을 한 바구니에 담지 마세요 — 하지만 너무 많이 나눠 담으면 관리가 복잡합니다"

하드디스크 하나에 중요한 데이터를 모두 저장하면 그 디스크가 고장나는 순간 모든 데이터가 사라집니다. **RAID(Redundant Array of Independent Disks)**는 여러 디스크를 묶어 마치 하나처럼 사용하면서 성능과 안정성을 높이는 기술입니다.

| RAID 레벨 | 최소 디스크 | 공간 효율 | 고장 허용 | 특징 |
|---------|-----------|---------|---------|------|
| **RAID 0** | 2개 | 100% | 0개 | 데이터를 분산 저장(스트라이핑), 읽기/쓰기 빠름, 안전성 없음 |
| **RAID 1** | 2개 | 50% | 1개 | 모든 디스크에 동일하게 복사(미러링), 느리지만 안전 |
| **RAID 5** | 3개 | (N-1)/N | 1개 | 패리티를 분산 저장, 균형 잡힌 성능과 안전성, 실무 인기 |
| **RAID 6** | 4개 | (N-2)/N | 2개 | 패리티를 두 번 저장, 2개 동시 고장도 OK, 쓰기 느림 |
| **RAID 1+0** | 4개 | 50% | 각 미러 그룹에서 1개씩 | RAID 1 미러링 + RAID 0 스트라이핑, 고성능 고신뢰성 |
| **RAID 5+0** | 6개 | (N-2)/N | 각 그룹에서 1개씩 | RAID 5 그룹들에 RAID 0 적용, 대용량 고성능 |

### 18-2. 블록/파일/오브젝트 스토리지 유형

클라우드 시대에는 스토리지가 세 가지 형태로 분화되었습니다.

* **블록 스토리지(Block Storage):** 데이터를 고정 크기 블록으로 나누어 저장합니다. 운영체제가 직접 제어하며 가장 빠릅니다. 데이터베이스나 가상 머신 디스크에 적합합니다. AWS EBS, SAN이 여기에 해당합니다.
* **파일 스토리지(File Storage):** 우리가 흔히 쓰는 폴더와 파일 구조입니다. 네트워크 드라이브처럼 여러 서버가 공유할 수 있습니다. AWS EFS, NAS가 여기에 해당합니다.
* **오브젝트 스토리지(Object Storage):** 데이터를 파일 시스템 없이 키-값 형태로 저장합니다. 무한히 확장 가능하고, 사진, 동영상, 로그 파일 같은 비정형 대용량 데이터에 최적화되어 있습니다. AWS S3가 대표적입니다. 수정보다는 한 번 쓰고 여러 번 읽는 용도에 어울립니다.

## Part 19. FTS — 장애에도 절대 멈추지 않는 시스템

### 19-1. FTS(Fault Tolerant System)

#### "비행기 엔진이 하나 꺼져도 착륙할 수 있어야 합니다"

금융 거래 시스템, 항공 관제 시스템, 병원 생명 유지 장치는 단 1초도 멈출 수 없습니다. **FTS(Fault Tolerant System, 결함 허용 시스템)**는 일부 구성 요소가 고장나더라도 전체 서비스가 계속 정상 운영되도록 설계된 시스템입니다.

FTS가 갖춰야 할 핵심 능력들:

* **중복성(Redundancy):** 중요한 구성 요소를 두 개 이상 두어, 하나가 고장나도 다른 하나가 즉시 이어받습니다. 이중화(Active-Active) 또는 핫 스탠바이(Active-Standby) 방식이 대표적입니다.
* **오류 탐지(Fault Detection):** 구성 요소가 고장났음을 자동으로 빠르게 감지합니다.
* **자동 복구(Automatic Recovery):** 고장 감지 후 자동으로 예비 시스템으로 전환합니다(Failover).
* **무정지 업그레이드:** 서비스를 멈추지 않고 소프트웨어나 하드웨어를 교체할 수 있습니다.

신뢰성과 가용성은 두 가지 지표로 측정합니다.

* **MTBF(Mean Time Between Failures)** = 총 정상 운영 시간 ÷ 장애 횟수 — 클수록 신뢰성 높음
* **MTTR(Mean Time To Repair)** = 총 복구 시간 ÷ 장애 횟수 — 작을수록 유지보수성 높음
* **가용률** = MTBF ÷ (MTBF + MTTR) × 100%

예를 들어 MTBF = 9,000시간, MTTR = 1시간이면 가용률 ≈ 99.989%(약 4-nine)입니다. 고가용성(HA) 시스템은 연간 가동률 99.99% 이상(다운타임 연 52분 이하)을 목표로 합니다.

하드웨어 수준에서도 결함 허용 기술이 적용됩니다. **ECC(Error Correcting Code) 메모리**는 1비트 오류를 자동으로 정정하며, **이중 전원 공급 장치(Redundant PSU)**는 전원 하나가 고장나도 무중단 전환을 보장합니다. 서버급 시스템이라면 이 두 가지는 사실상 필수 요소입니다.

## Part 20. 주기억장치 — DRAM과 SRAM의 두 가지 속도

### 20-1. DRAM과 SRAM 비교

#### "가득 채워두면 흘러내리는 물컵 vs 스위치로 잠가두는 수도꼭지"

주기억장치에는 두 가지 종류가 있습니다. 각각 작동 원리가 달라 쓰임새도 다릅니다.

**DRAM(Dynamic RAM) — 컴퓨터의 주력 메모리(RAM)**

콘덴서(캐패시터)에 전하를 저장하는 방식입니다. 전하가 서서히 새기 때문에 주기적으로 재충전(Refresh)해야 합니다. 이 '동적(Dynamic)'인 유지보수가 이름의 유래입니다. 구조가 단순해 집적도가 높고(같은 크기에 더 많이 들어감) 가격이 쌉니다. 우리가 아는 4GB, 16GB RAM 슬롯이 바로 DRAM입니다.

**SRAM(Static RAM) — CPU 캐시의 재료**

플립플롭 회로를 이용해 전원이 공급되는 한 데이터를 안정적으로 유지합니다. 리프레시가 필요 없어 DRAM보다 5~10배 빠릅니다. 하지만 하나의 비트를 저장하는 데 트랜지스터가 6개 필요해(DRAM은 1개) 같은 면적에 넣을 수 있는 용량이 작고, 가격이 매우 비쌉니다.

| 비교 | DRAM | SRAM |
|------|------|------|
| 속도 | 느림 | 빠름 (5~10배) |
| 가격 | 저렴 | 고가 |
| 집적도 | 높음 | 낮음 |
| 리프레시 | 필요 | 불필요 |
| 용도 | 주기억장치(RAM) | CPU 캐시(L1, L2, L3) |

## Part 21. 우선순위 역전 — 높은 사람이 낮은 사람에게 막히는 역설

### 21-1. 우선순위 역전(Priority Inversion)과 해결 기법

#### "VIP 손님이 평범한 손님 뒤에서 기다려야 하는 황당한 상황"

실시간 시스템에서 매우 위험한 버그 중 하나가 **우선순위 역전(Priority Inversion)**입니다. 1997년 화성 탐사선 패스파인더(Pathfinder)도 이 문제 때문에 시스템이 반복적으로 재시작되는 결함이 있었습니다.

발생 시나리오를 봅시다.

```
우선순위: 높은 H > 중간 M > 낮은 L

1. L이 공유 자원(락)을 점유하며 작업 중
2. H가 실행되어 같은 공유 자원을 요청 → L이 끝날 때까지 대기 (여기까진 정상)
3. M이 실행되어 L보다 높은 우선순위로 CPU를 빼앗음
4. H는 M이 끝나길 기다리고, M은 L의 자원을 필요로 하지 않아 계속 돌아감
   → 결과: H > M > L 순서여야 하는데 H가 M에게 막혀 버림!
```

이를 해결하는 두 가지 기법:

* **우선순위 상속(Priority Inheritance):** 공유 자원을 점유한 낮은 우선순위 프로세스(L)에게 그것을 기다리는 높은 우선순위 프로세스(H)의 우선순위를 임시로 빌려줍니다. L이 H의 우선순위로 빠르게 실행되어 자원을 빨리 반납하게 합니다.
* **우선순위 상한(Priority Ceiling):** 각 공유 자원마다 "이 자원을 쓸 수 있는 가장 높은 우선순위"를 미리 설정해 두고, 자원을 점유하는 순간 프로세스 우선순위를 그 상한값으로 올립니다.

## Part 22. 클라우드 컴퓨팅 — 서버를 소프트웨어로 만들다

### 22-1. Hypervisor 유형, Docker 비교 및 배포절차

#### "하나의 건물에 여러 세입자가 독립적으로 살 수 있는 비결"

물리 서버 하나를 여러 명이 독립적으로 쓰게 하려면 **가상화(Virtualization)** 기술이 필요합니다. 가상화를 가능하게 하는 소프트웨어가 **하이퍼바이저(Hypervisor)**입니다.

* **Type 1 하이퍼바이저(Bare-metal):** 하드웨어 위에 직접 설치됩니다. 운영체제 없이 하이퍼바이저가 하드웨어를 직접 제어하므로 성능이 뛰어납니다. VMware ESXi, Microsoft Hyper-V, Xen이 여기에 해당합니다. 대형 데이터센터에서 사용합니다.
* **Type 2 하이퍼바이저(Hosted):** 기존 운영체제(Windows, macOS) 위에 프로그램처럼 설치됩니다. VirtualBox, VMware Workstation이 여기에 해당합니다. 개발자 개인 PC에서 쓰기 편하지만 성능이 약간 낮습니다.

**Docker — 더 가벼운 가상화**

가상 머신(VM)은 운영체제 전체를 통째로 가상화합니다. 반면 **Docker 컨테이너**는 운영체제의 커널을 호스트와 공유하며 프로세스와 파일 시스템만 격리합니다. 덕분에 VM보다 훨씬 가볍고(수백 MB → 수십 MB) 빠르게 시작합니다.

| 비교 | 가상 머신(VM) | Docker 컨테이너 |
|------|------------|--------------|
| 가상화 대상 | OS 전체 | 프로세스/파일 시스템 |
| 시작 시간 | 수 분 | 수 초 |
| 이미지 크기 | 수 GB | 수십~수백 MB |
| 격리 수준 | 강함 | 비교적 약함 |
| 성능 오버헤드 | 높음 | 낮음 |

### 22-2. Auto Scale Up vs Scale Out

#### "혼자 키를 키울까, 아니면 팀원을 더 뽑을까?"

서비스에 트래픽이 몰렸을 때 서버 성능을 늘리는 방법은 두 가지입니다.

* **Scale Up(수직 확장):** 현재 서버의 CPU, RAM, 디스크를 더 좋은 것으로 교체합니다. 간단하지만 한계가 있으며(물리적 한계), 비용이 기하급수적으로 증가하고, 업그레이드 중 서비스가 잠깐 중단될 수 있습니다.
* **Scale Out(수평 확장):** 서버 대수를 늘립니다. 이론상 무한히 확장 가능하고, 한 서버가 고장나도 나머지가 대신합니다. 하지만 여러 서버에 요청을 분산하는 로드 밸런서와 서버 간 세션 공유 등 아키텍처가 복잡해집니다.

클라우드 환경의 **Auto Scaling**은 트래픽에 따라 자동으로 서버 대수를 늘리거나 줄입니다.

* **스케줄 기반 Auto Scale:** "매일 오전 9시에 서버 10대 추가, 오후 6시에 다시 줄이기"처럼 예측 가능한 패턴에 따라 사전에 스케줄을 잡습니다. 쇼핑몰 점심시간 대응 등에 적합합니다.
* **부하 기반 Auto Scale:** CPU 사용률이 80%를 넘으면 자동으로 서버를 추가하는 방식입니다. 예측 불가능한 트래픽 급증에 대응합니다. 서버가 추가되는 데 수 분이 걸리므로, 대응이 약간 늦을 수 있습니다.

### 22-3. 엣지 컴퓨팅과 클라우드 컴퓨팅 차이점

#### "공장 현장에서 바로 판단 vs 본사 서버에 데이터 보내서 판단"

**클라우드 컴퓨팅**은 멀리 있는 데이터센터 서버에 데이터를 보내 처리하는 방식입니다. 인터넷이 연결되어 있어야 하고, 데이터가 왕복하는 데 시간이 걸립니다(지연 시간, Latency). 하지만 거의 무한한 컴퓨팅 자원을 쓸 수 있고 관리가 편합니다.

**엣지 컴퓨팅**은 데이터가 생성되는 현장(엣지) 바로 가까이에 작은 서버나 컴퓨팅 장치를 두어 현장에서 직접 처리합니다. 자율주행 차량이 장애물을 발견했을 때 클라우드 서버에 물어보다가는 0.5초만 늦어도 사고가 납니다. 이런 경우에는 차량 내부에서 즉시 판단해야 합니다.

| 비교 | 클라우드 컴퓨팅 | 엣지 컴퓨팅 |
|------|--------------|-----------|
| 처리 위치 | 중앙 데이터센터 | 데이터 발생 현장 근처 |
| 지연 시간 | 수십~수백 ms | 수 ms 이하 |
| 인터넷 의존 | 높음 | 낮음 (오프라인 동작 가능) |
| 컴퓨팅 자원 | 무제한(탄력적) | 제한됨 |
| 보안 | 전송 중 노출 위험 | 데이터 현장 보관 가능 |
| 적합 용도 | 빅데이터 분석, AI 모델 학습 | 자율주행, 공장 IoT, 실시간 제어 |

---

> CA/OS는 눈에 보이지 않지만 모든 소프트웨어의 발판이 되는 영역입니다. 앱 하나가 클릭될 때마다 프로세스가 생성되고, 스케줄러가 CPU를 배분하고, 가상메모리가 주소를 변환하고, 캐시가 데이터를 건네줍니다. 이 모든 것이 1초에 수십억 번 일어납니다. 이 원리들을 이해하는 순간 컴퓨터 전체가 훤히 들여다보이기 시작합니다.
