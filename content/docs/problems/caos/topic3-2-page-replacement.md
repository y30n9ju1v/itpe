## 문제

총 3개의 페이지로 구성된 페이지 기반 가상 메모리 시스템이 있다. 제시되는 페이지 교체 알고리즘의 페이지 교체 과정을 적중률(Hit Ratio)과 함께 설명하시오.
- 페이지 교체 알고리즘: FIFO, LRU, LFU
- 페이지 번호 참조 순서: 4, 3, 2, 5, 2, 1, 3, 2, 6, 3, 4

## 해설

### 출제 배경 및 의도

물리 메모리 프레임이 가득 찬 상태에서 새로운 페이지가 필요할 때 어느 기존 페이지를 교체할지 결정하는 것이 페이지 교체 알고리즘이다. 선택이 잘못되면 페이지 폴트가 연속 발생하는 스래싱(Thrashing) 현상이 나타나며, 이는 CPU 활용률을 극도로 저하시킨다.

이 문제는 알고리즘 원리를 아는 것을 넘어 직접 표를 그리며 과정을 보여야 한다. 채점관은 Hit/Miss 판별, 교체 대상 선택 과정, 최종 적중률 계산이 모두 정확한지 확인한다. 특히 LFU의 동점(tie) 처리 방식 명시, FIFO의 벨라디의 역설(Belady's Anomaly)에 대한 언급은 심화 역량 어필 포인트다.

현대 OS(Linux)는 순수 LRU 대신 Clock 알고리즘(Second Chance)을 사용하는 이유, 그리고 스래싱 방지를 위한 워킹셋 모델과 PFF(Page Fault Frequency) 제어 전략까지 연결하면 실무 역량을 효과적으로 표현할 수 있다.

### 1. 페이지 교체 알고리즘, 개요

정 의  • 물리 메모리 프레임이 모두 사용 중일 때 새로운 페이지를 적재하기 위해 교체할 기존 페이지를 선정하는 정책
       - 적중률(Hit Ratio) = Hit 횟수 / 전체 참조 횟수, 높을수록 우수

```
참조 발생
    │
    ▼
┌───────────────┐
│ 메모리에 있음? │──YES──▶ Cache Hit  → 데이터 반환
└───────────────┘
        │ NO
        ▼
   Page Fault 발생
        │
        ▼
┌──────────────────┐
│ 빈 프레임 있음?  │──YES──▶ 페이지 적재 → 완료
└──────────────────┘
        │ NO
        ▼
  교체 알고리즘 선택
  (FIFO / LRU / LFU)
        │
        ▼
  희생 페이지 교체 → 새 페이지 적재
```

- 적중률이 낮아 페이지 폴트가 빈발하면 스래싱 현상이 발생하여 CPU 활용률이 급락한다.

### 2. 알고리즘 원리 및 시뮬레이션

1) 알고리즘 특성 비교

| 알고리즘 | 교체 대상 선택 기준 | 구현 복잡도 | 특이사항 |
|----------|---------------------|-------------|----------|
| FIFO | 가장 먼저 적재된 페이지 | 낮음(큐) | 벨라디의 역설 발생 가능 |
| LRU | 가장 오래 전에 사용된 페이지 | 중간(타임스탬프/스택) | 시간 지역성 활용, 스택 알고리즘 |
| LFU | 참조 횟수가 가장 적은 페이지 | 중간(카운터) | 동점 시 처리 정책 명시 필요 |
| OPT(최적) | 미래에 가장 늦게 사용될 페이지 | 불가(미래 필요) | 이론적 상한, 비교 기준점 |

2) FIFO 시뮬레이션 (참조열: 4,3,2,5,2,1,3,2,6,3,4 / 프레임 수: 3)

| 참조 | F1 | F2 | F3 | Hit/Miss | 교체 대상 |
|------|----|----|----|----------|-----------|
| 4    | **4** | -  | -  | Miss | - |
| 3    | 4  | **3** | -  | Miss | - |
| 2    | 4  | 3  | **2** | Miss | - |
| 5    | **5** | 3  | 2  | Miss | 4(가장 오래됨) |
| 2    | 5  | 3  | 2  | **Hit** | - |
| 1    | 5  | **1** | 2  | Miss | 3 |
| 3    | 5  | 1  | **3** | Miss | 2 |
| 2    | **2** | 1  | 3  | Miss | 5 |
| 6    | 2  | **6** | 3  | Miss | 1 |
| 3    | 2  | 6  | 3  | **Hit** | - |
| 4    | 2  | **4** | 3  | Miss | 6 |

- **Hit: 2회 / Miss: 9회 / 적중률 = 2/11 ≈ 18.2%**

3) LRU 시뮬레이션 (Least Recently Used)

| 참조 | F1 | F2 | F3 | Hit/Miss | 교체 대상 |
|------|----|----|----|----------|-----------|
| 4    | **4** | -  | -  | Miss | - |
| 3    | 4  | **3** | -  | Miss | - |
| 2    | 4  | 3  | **2** | Miss | - |
| 5    | **5** | 3  | 2  | Miss | 4(최근 미사용) |
| 2    | 5  | 3  | 2  | **Hit** | - |
| 1    | 5  | **1** | 2  | Miss | 3(최근 미사용) |
| 3    | 5  | 1  | **3** | Miss | 2 |
| 2    | **2** | 1  | 3  | Miss | 5(최근 미사용) |
| 6    | 2  | **6** | 3  | Miss | 1 |
| 3    | 2  | 6  | 3  | **Hit** | - |
| 4    | **4** | 6  | 3  | Miss | 2(최근 미사용) |

- **Hit: 2회 / Miss: 9회 / 적중률 = 2/11 ≈ 18.2%**

### 3. LFU 시뮬레이션 및 종합 비교

1) LFU 시뮬레이션 (동점 시 LRU 방식 적용)

| 참조 | 메모리 상태 (페이지:횟수) | Hit/Miss |
|------|---------------------------|----------|
| 4    | [4:1] | Miss |
| 3    | [4:1, 3:1] | Miss |
| 2    | [4:1, 3:1, 2:1] | Miss |
| 5    | [3:1, 2:1, 5:1] → 4 교체(동점 중 LRU) | Miss |
| 2    | [3:1, **2:2**, 5:1] | **Hit** |
| 1    | [**2:2**, 1:1, 5:1] → 3 교체(동점 중 LRU) | Miss |
| 3    | [2:2, 1:1, **3:1**] → 5 교체 | Miss |
| 2    | [**2:3**, 1:1, 3:1] | **Hit** |
| 6    | [2:3, 3:1, **6:1**] → 1 교체 | Miss |
| 3    | [2:3, **3:2**, 6:1] | **Hit** |
| 4    | [2:3, 3:2, **4:1**] → 6 교체 | Miss |

- **Hit: 3회 / Miss: 8회 / 적중률 = 3/11 ≈ 27.3%**

2) 알고리즘별 적중률 최종 비교

| 알고리즘 | Hit 수 | Miss 수 | 적중률 | 비고 |
|----------|--------|---------|--------|------|
| FIFO | 2 | 9 | **18.2%** | 벨라디의 역설 가능 |
| LRU | 2 | 9 | **18.2%** | 스택 알고리즘(역설 없음) |
| LFU | 3 | 8 | **27.3%** | 동점 정책에 따라 결과 변동 |
| OPT(참고) | - | - | 이론적 최대 | 미래 정보 필요 |

- 이 참조열에서는 LFU가 가장 높은 적중률을 보이며, 일반적으로 LRU가 시간 지역성 활용으로 FIFO보다 우수한 성능을 낸다.  "끝"

### 실무 제언

**현대 OS에서 LRU 근사 구현(Clock 알고리즘)**
- **챌린지**: 순수 LRU는 모든 메모리 접근마다 타임스탬프를 기록해야 하므로 하드웨어 오버헤드가 크다. Linux는 Clock 알고리즘(Second Chance)이라는 LRU 근사 방법을 사용한다.
- **제언**: 페이지마다 참조 비트(Reference Bit) 하나만 유지하고, 교체 시 참조 비트가 1이면 0으로 초기화 후 다음 페이지로 이동하며, 0인 페이지를 교체한다. `vmstat`의 si/so 지표와 `sar -B`의 페이지 폴트율로 스래싱 여부를 모니터링해야 한다.

**스래싱 방지를 위한 워킹셋 모델 적용**
- **챌린지**: 다중 프로그래밍 수준이 지나치게 높으면 모든 프로세스가 스래싱 상태에 빠져 CPU 활용률이 급락한다.
- **제언**: 워킹셋(Working Set) 모델을 적용하여 각 프로세스의 최근 Δ 시간 동안 참조한 페이지 집합만큼 프레임을 보장하고, PFF(Page Fault Frequency) 임계값 기반으로 다중 프로그래밍 수준을 동적으로 조절해야 한다.

**데이터베이스 버퍼 풀 교체 정책**
- **챌린지**: Oracle, MySQL InnoDB의 버퍼 풀은 OS 페이지 교체 알고리즘과 별도로 자체 교체 정책을 운용한다. 전통적 LRU는 풀 테이블 스캔 시 버퍼 풀 전체를 오염시키는 문제가 있다.
- **제언**: InnoDB의 LRU 리스트를 Young(hot)과 Old(cold) 두 영역으로 분리하고, 처음 적재된 페이지를 Old 영역에 배치하여 단순 스캔 페이지가 Young 영역을 침범하지 못하도록 설계해야 한다.
