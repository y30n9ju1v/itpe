## 해설

### 출제 배경 및 의도

주소지정방식(Addressing Mode)은 명령어가 피연산자(데이터)의 위치를 찾아가는 규칙이다. 방식에 따라 명령어 길이, 메모리 접근 횟수, 표현 가능한 주소 범위가 달라지므로 CPU 설계와 컴파일러 최적화의 핵심 요소다.

합숙 공통 문제로 출제된 만큼 모든 수험생이 기본적으로 알아야 하는 영역이다. 채점관은 각 방식의 동작 원리와 유효 주소(EA) 계산 공식, 장단점 비교를 체계적으로 서술할 수 있는지를 본다. 단순 나열이 아니라 "왜 이 방식이 필요했는가"라는 설계 의도까지 연결하면 고득점 답안이 된다.

RISC vs CISC 설계 철학의 차이가 주소지정방식 수에 직접 반영되며, 고급 언어의 배열·포인터·분기 구조가 각각 어떤 방식으로 번역되는지까지 이해해야 한다.

### 1. 주소지정방식(Addressing Mode), 개요

정 의  • 명령어 내 피연산자(Operand) 필드를 해석하여 실제 데이터가 저장된 유효 주소(EA, Effective Address)를 결정하는 규칙 집합
       - 방식 선택에 따라 메모리 접근 횟수와 표현 가능한 주소 범위가 결정됨

```
┌──────────────────────────┬──────────────────────────────┐
│      Opcode (연산코드)    │      Operand (피연산자)       │
│   ADD, MOV, JMP 등       │  주소 또는 즉시 데이터        │
└──────────────────────────┴──────────────────────────────┘
                                     │
                            주소지정방식 적용
                                     │
                                     ▼
                            유효 주소(EA) 결정
                                     │
                                     ▼
                              실제 데이터 획득
```

- 피연산자 필드를 해석하는 방식이 곧 주소지정방식이며, CPU 설계의 유연성과 성능을 동시에 결정한다.

### 2. 주요 주소지정방식 상세

1) 비메모리 접근 방식

| 구분 | 방식 | 유효 주소(EA) | 메모리 접근 | 특징 |
|------|------|---------------|-------------|------|
| 즉시 | Immediate | 피연산자 자체가 데이터 | 0회 | 가장 빠름, 상수값 처리, 표현 범위 제한 |
| 레지스터 | Register | EA = Register 값 | 0회 | 최고속, 레지스터 수에 제한 |

2) 메모리 직접/간접 접근 방식

| 구분 | 방식 | 유효 주소(EA) | 메모리 접근 | 특징 |
|------|------|---------------|-------------|------|
| 직접 | Direct | EA = Operand | 1회 | 단순, 주소 범위 제한 |
| 간접 | Indirect | EA = M[Operand] | 2회 이상 | 넓은 주소 범위, 느림 |
| 레지스터 간접 | Register Indirect | EA = M[Register] | 1회 | 포인터 구현, C언어 `*ptr` |

3) 변위(Displacement) 방식

| 구분 | 방식 | 유효 주소(EA) | 용도 |
|------|------|---------------|------|
| 상대 | PC Relative | EA = PC + Offset | 분기 명령어, 위치 독립 코드(PIC) |
| 베이스 | Base Register | EA = Base + Offset | 세그멘테이션, 다중 프로세스 |
| 인덱스 | Indexed | EA = Index + Offset | 배열 순차 접근 |

- 변위 방식은 레지스터 값과 명령어 내 상수를 더해 유효 주소를 산출하므로 배열·스택·세그먼트 접근에 최적화된다.

### 3. 동작 원리 상세 및 RISC/CISC 비교

1) 핵심 방식 동작 예시

```
① 즉시 주소 (Immediate)
   ADD R1, #5  →  R1 = R1 + 5  (5는 명령어에 포함된 상수)

② 직접 주소 (Direct)
   LOAD R1, 1000  →  R1 = M[1000]

③ 간접 주소 (Indirect)
   LOAD R1, (1000)  →  R1 = M[M[1000]]  (2회 접근)

④ 레지스터 간접 (Register Indirect)
   LOAD R1, (R2)  →  R1 = M[R2]  (C언어 *ptr 역참조)

⑤ PC 상대 주소 (PC Relative)
   JMP +50  →  PC = PC + 50  (위치 독립 분기)

⑥ 인덱스 주소 (Indexed)
   LOAD R1, 1000(R2)  →  R1 = M[1000 + R2]  (배열 접근)
```

2) RISC vs CISC 주소지정방식 설계 철학

| 구분 | RISC (ARM, RISC-V) | CISC (x86) |
|------|--------------------|------------|
| 방식 수 | 소수(간소화) | 다양(복잡) |
| 메모리 접근 | Load/Store 방식만 허용 | 연산 명령어에서도 직접 메모리 접근 |
| 레지스터 활용 | 많은 레지스터, 레지스터 간 연산 | 적은 레지스터, 메모리-레지스터 혼용 |
| 명령어 길이 | 고정 길이 | 가변 길이 |
| 컴파일러 의존도 | 높음 | 낮음 |

3) ARM Load/Store 아키텍처와 컴파일러 관점

ARM(RISC)은 메모리 접근을 LDR/STR 명령어로만 허용하며, 연산 명령어(ADD, MUL 등)는 반드시 레지스터 간에만 수행한다.

```
ARM (RISC — Load/Store 제약)
  LDR  R1, [R2, #8]   ; R1 = M[R2 + 8]  ← 메모리→레지스터 로드
  ADD  R1, R1, R3     ; R1 = R1 + R3    ← 레지스터 간 연산만 허용
  STR  R1, [R2, #8]   ; M[R2 + 8] = R1  ← 레지스터→메모리 스토어

x86 (CISC — 메모리 직접 연산 허용)
  ADD  [EBP+8], EAX   ; M[EBP+8] += EAX ← 메모리-레지스터 혼합 연산 가능
```

| 관점 | RISC(ARM) 장점 | CISC(x86) 장점 |
|------|---------------|----------------|
| 컴파일러 | 레지스터 할당 최적화 전적으로 컴파일러 담당 | 코드 밀도 높음, 명령어 수 적음 |
| 파이프라인 | 고정 길이 명령어로 해석 단순 | 복잡한 명령어 디코더 필요 |
| 캐시 효율 | 명령어 길이 균일 → I-캐시 효율 높음 | 가변 길이로 경계 예측 복잡 |

- 컴파일러는 배열 접근에 인덱스(Base+Offset) 방식, 포인터에 레지스터 간접 방식, 분기에 PC 상대 방식을 우선 사용하며, 레지스터가 부족할 때만 메모리 직접 방식으로 스필(Spill)한다.  "끝"

### 실무 제언

**보안 관점의 주소지정방식**
- **챌린지**: 간접 주소지정방식의 포인터 체계는 버퍼 오버플로우(Buffer Overflow) 공격의 근본 원인이 된다. 공격자가 스택의 리턴 주소를 덮어쓰면 PC가 악성 코드 주소를 가리키게 된다.
- **제언**: 현대 CPU와 OS는 ASLR(Address Space Layout Randomization)로 주소 공간을 무작위화하고, 스택 카나리(Stack Canary)로 리턴 주소 변조를 탐지하며, NX/DEP로 데이터 영역 실행을 차단하는 다층 방어를 적용해야 한다.

**임베디드 시스템 전력 최적화**
- **챌린지**: 배터리 구동 IoT 디바이스에서 불필요한 메모리 접근은 전력 소모와 실행 시간을 동시에 증가시킨다.
- **제언**: 레지스터 주소지정방식 활용률을 높이면 메모리 접근 횟수를 줄일 수 있다. 컴파일러 레지스터 할당 최적화 옵션(-O2, -O3)을 적극 활용하고, 핫패스 코드는 인라인 어셈블리로 레지스터 사용을 명시적으로 제어해야 한다.

**동적 라이브러리 로딩과 PC 상대 주소**
- **챌린지**: 공유 라이브러리(.so, .dll)는 프로세스마다 다른 메모리 주소에 로드되므로 절대 주소를 사용하면 재배치(Relocation) 비용이 커진다.
- **제언**: PC 상대 주소 방식을 활용한 PIC(Position Independent Code) 컴파일(-fPIC)로 라이브러리를 빌드하면 재배치 없이 임의 주소에 로드 가능하여 ASLR 환경에서도 효율적으로 동작한다.
