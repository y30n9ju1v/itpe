IPC(Inter Process Communication) 통신방식

## 해설

> **독립된 프로세스들이 협력하는 모든 방법의 집합**  
> 공유 메모리, 파이프, 메시지 큐, 소켓, 시그널의 원리와 비교

운영체제는 각 프로세스를 독립된 메모리 공간에 격리(Isolation)합니다. 이는 보안과 안정성을 보장하지만, 동시에 협력이 필요한 프로세스들 간의 데이터 교환을 어렵게 만듭니다. **IPC(Inter Process Communication)**는 이 격리된 프로세스들이 데이터를 교환하고 동기화하는 모든 메커니즘을 총칭합니다.

### 1. 출제 배경 및 의도

IPC는 OS의 프로세스 관리와 현대 소프트웨어 아키텍처(MSA, 분산 시스템)를 연결하는 핵심 주제입니다. 채점관은 **각 IPC 방식의 동작 원리**, **동기/비동기, 단방향/양방향, 연관성(관련/비관련 프로세스) 등의 분류**, 그리고 **각 방식이 적합한 사용 사례**를 체계적으로 서술하기를 기대합니다.

### 2. 개념의 본질과 작동 메커니즘

IPC 방식은 크게 **메모리 공유 방식**과 **메시지 전달 방식**으로 분류됩니다.

```
┌─────────────────────────────────────────────────────────┐
│                    IPC 방식 분류                         │
├────────────────────────┬────────────────────────────────┤
│   메모리 공유 방식      │   메시지 전달 방식              │
│ (Shared Memory)        │ (Message Passing)               │
├────────────────────────┼────────────────────────────────┤
│ • 공유 메모리           │ • 파이프 (Pipe)                 │
│ • 메모리 맵 파일        │ • Named Pipe (FIFO)             │
│                        │ • 메시지 큐 (Message Queue)     │
│                        │ • 소켓 (Socket)                 │
│                        │ • 시그널 (Signal)               │
│                        │ • RPC (Remote Procedure Call)   │
└────────────────────────┴────────────────────────────────┘
```

#### ① 공유 메모리 (Shared Memory)

```
프로세스 A              공유 메모리              프로세스 B
┌──────────┐          ┌───────────┐          ┌──────────┐
│  가상    │          │ 공유 영역 │          │  가상    │
│  주소    │─────────▶│ (RAM에   │◀─────────│  주소    │
│  공간    │  매핑    │  실제 존재)│  매핑    │  공간    │
└──────────┘          └───────────┘          └──────────┘
```

- 두 프로세스가 동일한 물리 메모리 영역을 각자의 가상 주소 공간에 매핑
- **가장 빠른 IPC:** 커널을 거치지 않고 직접 메모리 읽기/쓰기
- **주의:** 동기화가 별도로 필요 (세마포어, 뮤텍스 사용)
- Linux: `shmget()`, `shmat()` 또는 `mmap(MAP_SHARED)`
- 사용 사례: 대용량 데이터 공유 (영상 처리, DB 버퍼 공유)

#### ② 파이프 (Pipe)

```
프로세스 A (쓰기 끝) ────[파이프 버퍼]────▶ 프로세스 B (읽기 끝)
                          (커널 관리)
```

- **단방향**, **FIFO** 방식의 데이터 스트림
- **Unnamed Pipe(익명 파이프):** 부모-자식 프로세스 간에만 사용 가능
- **Named Pipe (FIFO):** 파일 시스템에 이름으로 등록, 관련 없는 프로세스 간 사용 가능
- 쉘의 파이프라인(`ls | grep txt`)이 이 방식

```bash
# 쉘 파이프 예시
cat file.txt | grep "error" | sort | uniq -c
```

#### ③ 메시지 큐 (Message Queue)

```
프로세스 A → [메시지1] [메시지2] [메시지3] → 프로세스 B
                    커널 관리 큐
```

- 메시지 단위로 데이터 전달 (스트림이 아닌 이산 메시지)
- **비동기 통신:** 보내는 쪽이 받는 쪽의 준비를 기다리지 않음
- 메시지 타입(Type)으로 선택적 수신 가능
- Linux System V IPC: `msgget()`, `msgsnd()`, `msgrcv()`
- POSIX MQ: `mq_open()`, `mq_send()`, `mq_receive()`

#### ④ 소켓 (Socket)

```
프로세스 A          네트워크 스택           프로세스 B
┌─────────┐        ┌───────────┐        ┌─────────┐
│ Socket  │◀──────▶│  TCP/UDP  │◀──────▶│ Socket  │
│ 파일    │        │  IP 계층  │        │ 파일    │
└─────────┘        └───────────┘        └─────────┘
   같은 머신: Unix Domain Socket (loopback)
   다른 머신: TCP/UDP Socket
```

- **가장 범용적인 IPC:** 같은 머신 내(Unix Domain Socket) 및 네트워크 경유 모두 지원
- **스트림 소켓(TCP):** 신뢰성 있는 연결 지향 통신
- **데이터그램 소켓(UDP):** 비연결 지향, 빠름, 손실 허용
- 사용 사례: 웹 서버-클라이언트, MSA 서비스 간 통신

#### ⑤ 시그널 (Signal)

```
프로세스 A → kill(pid, SIGTERM) → OS → 프로세스 B가 SIGTERM 수신
```

- 비동기 알림 메커니즘 (데이터 전달 아님, 이벤트 통보)
- 주요 시그널: SIGKILL(강제 종료), SIGTERM(정상 종료 요청), SIGINT(Ctrl+C), SIGSEGV(세그폴트)
- 시그널 핸들러(Signal Handler)로 커스텀 처리 가능

#### ⑥ RPC (Remote Procedure Call)

- 원격 서버의 함수를 로컬 함수처럼 호출하는 추상화
- 내부적으로 소켓 기반 통신 사용
- gRPC, REST API, Apache Thrift 등이 이 계층의 현대적 구현

#### IPC 방식 비교

| 방식 | 통신 범위 | 속도 | 동기/비동기 | 데이터 크기 |
|---|---|---|---|---|
| **공유 메모리** | 같은 머신 | 최고 | 동기 | 대용량 |
| **파이프** | 관련 프로세스 | 빠름 | 동기 | 스트림 |
| **Named Pipe** | 같은 머신 | 빠름 | 동기/비동기 | 스트림 |
| **메시지 큐** | 같은 머신 | 중간 | 비동기 | 중간 |
| **소켓** | 네트워크 포함 | 느림 | 비동기 | 제한 없음 |
| **시그널** | 같은 머신 | 빠름 | 비동기 | 없음 |

### 3. 핵심 키워드 (Must-Have)

- **IPC(Inter Process Communication):** 프로세스 간 통신 메커니즘 총칭
- **메모리 공유 방식 vs 메시지 전달 방식**
- **공유 메모리:** 최고 속도, 동기화 별도 필요
- **파이프:** 단방향 FIFO, 부모-자식 간
- **Named Pipe(FIFO):** 비관련 프로세스 간 파이프
- **메시지 큐:** 비동기, 메시지 단위, 타입 선택 가능
- **소켓:** 범용 IPC, 네트워크 포함
- **시그널:** 비동기 이벤트 통보

### 4. 맥락과 비교

**IPC vs 스레드 간 통신 비교:**

| 구분 | 프로세스 간 (IPC) | 스레드 간 |
|---|---|---|
| **메모리 분리** | 독립적 가상 주소 공간 | 프로세스 내 공유 메모리 |
| **통신 비용** | 높음 (커널 개입 또는 매핑) | 낮음 (직접 변수 공유) |
| **안전성** | 높음 (프로세스 격리) | 낮음 (경쟁 조건 위험) |
| **동기화** | IPC 메커니즘 별도 | 뮤텍스, 세마포어 |

### 5. 실무 제언

**MSA에서 IPC 방식 선택 기준**

- **챌린지:** 마이크로서비스 아키텍처에서 서비스 간 통신 방식(REST API, gRPC, 메시지 브로커)은 시스템 성능, 결합도, 신뢰성에 직접 영향을 미칩니다. 동기 REST 호출만 남발하면 서비스 간 강한 결합과 연쇄 장애(Cascading Failure)가 발생합니다.
- **제언:** 요청-응답이 즉시 필요한 경우(사용자 인증, 결제)는 **gRPC 동기 통신**, 느슨한 결합과 장애 격리가 필요한 경우(주문→배송 알림)는 **Kafka/RabbitMQ 비동기 메시지 큐**를 선택해야 합니다. 전통적 OS IPC 원리(소켓, 메시지 큐)가 현대 MSA 통신 패턴의 이론적 기반임을 연결하여 설명하면 통찰력 있는 답안이 됩니다.
