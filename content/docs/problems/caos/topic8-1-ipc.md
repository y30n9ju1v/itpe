## 해설

### 출제 배경 및 의도

운영체제는 각 프로세스를 독립된 가상 주소 공간에 격리하여 보안과 안정성을 보장합니다. 그러나 현대 소프트웨어는 여러 프로세스가 협력하여 복잡한 기능을 구현하며, 마이크로서비스 아키텍처(MSA)에서는 수십~수백 개 서비스가 상호 통신합니다. IPC는 이 격리된 프로세스들이 데이터를 교환하고 동기화하는 메커니즘의 총칭으로, 전통적 OS 원리(소켓·메시지 큐)가 현대 MSA 통신 패턴의 이론적 기반이 됩니다.

IPC는 OS 프로세스 관리와 현대 분산 시스템을 연결하는 핵심 주제입니다. 채점관은 각 IPC 방식의 동작 원리, 메모리 공유·메시지 전달의 분류 체계, 동기/비동기·통신 범위·속도 비교, 그리고 현대 MSA와의 연관성을 체계적으로 서술하는지를 평가합니다.

### 1. IPC(Inter Process Communication), 개요

정 의  • OS가 격리된 프로세스들 간에 데이터를 교환하고 동기화하도록 제공하는 메커니즘의 총칭
       - 메모리 공유 방식(Shared Memory)과 메시지 전달 방식(Message Passing)의 두 범주로 분류

```
┌─────────────────────────────────────────────────────────┐
│                   IPC 방식 분류                          │
├────────────────────────┬────────────────────────────────┤
│   메모리 공유 방식      │   메시지 전달 방식              │
│ (Shared Memory)        │ (Message Passing)               │
├────────────────────────┼────────────────────────────────┤
│ • 공유 메모리           │ • 파이프 (Pipe/Named Pipe)      │
│ • 메모리 맵 파일        │ • 메시지 큐 (Message Queue)     │
│                        │ • 소켓 (Socket)                 │
│                        │ • 시그널 (Signal)               │
│                        │ • RPC (Remote Procedure Call)   │
└────────────────────────┴────────────────────────────────┘
```

- 메모리 공유는 커널 개입 없이 직접 접근하여 최고 속도를 제공하나 동기화 별도 필요

### 2. IPC 방식별 동작 원리와 특성

1) 메모리 공유 방식 (Shared Memory)

| 구분 | 항목 | 설명 |
|------|------|------|
| 공유 메모리 | 원리 | 두 프로세스가 동일 물리 메모리를 각자의 가상 주소 공간에 매핑 |
| | 속도 | 최고 (커널 개입 없이 직접 읽기/쓰기) |
| | 동기화 | 별도 필요 (세마포어·뮤텍스) |
| | Linux API | shmget(), shmat() 또는 mmap(MAP_SHARED) |
| | 용도 | 대용량 데이터 공유, 영상 처리, DB 버퍼 |

```
공유 메모리 구조:

프로세스 A의 가상 주소 공간          프로세스 B의 가상 주소 공간
┌──────────────────────┐             ┌──────────────────────┐
│  ...                 │             │  ...                 │
│  [공유 영역 매핑]    │◄──┐     ┌──►│  [공유 영역 매핑]    │
│  ...                 │   │     │   │  ...                 │
└──────────────────────┘   │     │   └──────────────────────┘
                            ▼     ▼
                      ┌───────────────┐
                      │ 물리 메모리   │
                      │ (공유 영역)   │
                      └───────────────┘
```

- 커널 복사 없이 직접 접근하여 IPC 방식 중 가장 빠름, 단 경쟁 조건(Race Condition) 방지를 위한 동기화 필수

2) 메시지 전달 방식 주요 기법

| 구분 | 항목 | 설명 |
|------|------|------|
| 파이프 | 방식 | 단방향 FIFO 스트림, 커널 버퍼 |
| | 범위 | Unnamed: 부모-자식 간 / Named(FIFO): 비관련 프로세스 간 |
| | 예시 | 쉘 파이프라인 (ls | grep txt) |
| 메시지 큐 | 방식 | 메시지 단위 전달, 타입별 선택적 수신 가능 |
| | 특징 | 비동기 통신, 보내는 쪽이 받는 쪽 준비를 기다리지 않음 |
| | Linux API | msgget(), msgsnd(), msgrcv() |
| 소켓 | 방식 | 양방향, 네트워크 포함 범용 IPC |
| | 종류 | TCP 스트림 소켓 (신뢰성), UDP 데이터그램 소켓 (속도) |
| | 특징 | 같은 머신(Unix Domain Socket)~네트워크 원격까지 지원 |
| 시그널 | 방식 | 비동기 이벤트 알림 (데이터 전달 아님) |
| | 주요 시그널 | SIGKILL(강제 종료), SIGTERM(정상 종료), SIGINT(Ctrl+C) |

### 3. IPC 방식 종합 비교 및 MSA 적용

| 방식 | 통신 범위 | 속도 | 동기/비동기 | 데이터 크기 |
|------|------|------|------|------|
| 공유 메모리 | 같은 머신 | 최고 | 동기 | 대용량 |
| 파이프 | 관련 프로세스 | 빠름 | 동기 | 스트림 |
| Named Pipe | 같은 머신 | 빠름 | 동기/비동기 | 스트림 |
| 메시지 큐 | 같은 머신 | 중간 | 비동기 | 중간 |
| 소켓 | 네트워크 포함 | 낮음 | 비동기 | 제한 없음 |
| 시그널 | 같은 머신 | 빠름 | 비동기 | 없음 |

```
MSA IPC 패턴과 전통 IPC 원리의 대응:

전통 OS IPC           현대 MSA 통신 패턴
─────────────────────────────────────────
소켓(TCP)        →  REST API, gRPC (동기 서비스 간 통신)
메시지 큐        →  Kafka, RabbitMQ (비동기 이벤트 스트리밍)
RPC              →  gRPC, Apache Thrift (원격 함수 호출)
```

전통적 OS IPC 원리(소켓·메시지 큐)가 현대 MSA 분산 통신 패턴의 이론적 기반임을 이해하고, 동기·비동기 적합 시나리오를 구별하여 설계하는 것이 핵심.  "끝"

### 실무 제언

**MSA 서비스 간 IPC 방식 선택 기준**
- **챌린지**: MSA에서 모든 서비스 간 통신을 동기 REST 호출로 구현하면 서비스 간 강한 결합과 연쇄 장애(Cascading Failure)가 발생하여 전체 시스템 가용성 저하
- **제언**: 즉시 응답이 필요한 경우(사용자 인증·결제)는 gRPC 동기 통신, 느슨한 결합과 장애 격리가 필요한 경우(주문→배송 알림)는 Kafka/RabbitMQ 비동기 메시지 큐 선택

**공유 메모리 동기화 설계**
- **챌린지**: 공유 메모리를 사용하는 멀티프로세스 환경에서 동기화 메커니즘 누락 시 데이터 손상이나 데드락 발생
- **제언**: 공유 메모리 접근 시 POSIX 세마포어(sem_wait/sem_post) 또는 pthread 뮤텍스(PTHREAD_PROCESS_SHARED)로 임계 구역 보호, Reader-Writer Lock으로 읽기 병렬성 극대화

**소켓 통신 성능 최적화**
- **챌린지**: 고빈도 소켓 통신에서 연결 설정 오버헤드(TCP 3-way handshake)와 컨텍스트 스위치로 인한 레이턴시 증가
- **제언**: 같은 머신 내 통신은 Unix Domain Socket(네트워크 스택 우회)으로 TCP 대비 2~3배 성능 향상, 연결 풀링(Connection Pool)으로 handshake 오버헤드 최소화
