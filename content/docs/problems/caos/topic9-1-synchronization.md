운영체제의 프로세스 동기화 기법 중 유뮤텍스(Mutex), 세마포어(Semaphore), 모니터(Monitor)에 대하여 설명하시오.

## 해설

> **공유 자원을 안전하게 사용하기 위한 협력의 규칙**  
> Mutex의 상호 배제, Semaphore의 P/V 연산, Monitor의 객체 지향적 동기화

다중 프로세스/스레드 환경에서 공유 자원(메모리, 파일, 하드웨어)에 여러 실행 주체가 동시에 접근하면 **경쟁 조건(Race Condition)**이 발생하여 예측 불가능한 결과를 낳습니다. 동기화(Synchronization)는 임계 구역(Critical Section)에 대한 접근을 질서 있게 제어하는 메커니즘입니다.

### 1. 출제 배경 및 의도

139회 관리 문제로 최근에 출제되었습니다. 채점관은 **임계 구역 문제의 3가지 해결 조건(상호 배제, 진행, 한계 대기)**, **각 동기화 기법의 동작 원리와 차이점**, 그리고 **생산자-소비자 문제 등 고전적 동기화 문제에서의 적용**을 이해하는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### 임계 구역(Critical Section) 문제

**임계 구역:** 공유 자원에 접근하는 코드 영역. 동시에 하나의 프로세스만 실행해야 함.

**3가지 해결 요건:**
1. **상호 배제(Mutual Exclusion):** 한 번에 하나의 프로세스만 임계 구역 진입
2. **진행(Progress):** 임계 구역 밖의 프로세스는 진입 결정을 방해할 수 없음
3. **한계 대기(Bounded Waiting):** 임계 구역 진입 요청 후 무한 대기하지 않음 (기아 방지)

#### ① Mutex (Mutual Exclusion Lock)

**개념:** 임계 구역에 대한 상호 배제를 보장하는 잠금(Lock) 메커니즘. 소유권 개념 포함.

```
스레드 A:           스레드 B:
lock(mutex)         lock(mutex) ← 블록 (대기)
  임계 구역          ...
unlock(mutex) ──▶  스레드 B 깨어남
                    임계 구역
                   unlock(mutex)
```

**특징:**
- **소유권(Ownership):** lock을 걸었던 스레드만 unlock 가능
- **바이너리 잠금:** 잠김(1) 또는 열림(0) 두 가지 상태만 존재
- **Busy Waiting vs Blocking:**
  - **스핀락(Spinlock):** 잠금이 풀릴 때까지 CPU를 사용하며 루프로 재시도 (CPU 낭비, 짧은 대기 시간에 유리)
  - **블로킹 뮤텍스:** 잠금 대기 시 스레드를 sleep 상태로 전환 (컨텍스트 스위치 비용, 긴 대기 시간에 유리)

#### ② Semaphore (세마포어)

**개념:** Dijkstra가 제안한 정수 카운터 기반 동기화 도구. 뮤텍스보다 일반화된 개념.

**P(wait) 연산과 V(signal) 연산:**
```
P(S): S가 0보다 크면 S--; 
      S가 0이면 현재 프로세스를 대기 큐로 이동 (블록)

V(S): S++;
      대기 큐에 프로세스가 있으면 하나 깨움 (wake up)
```

**이진 세마포어 (Binary Semaphore):**
- S 값이 0 또는 1만 가짐 → 뮤텍스와 유사하지만 소유권 개념 없음
- 임계 구역 보호에 사용

**카운팅 세마포어 (Counting Semaphore):**
- S 값이 0 이상의 정수 → 여러 단위의 자원 관리
- 예: 데이터베이스 연결 풀(최대 10개) 관리

**생산자-소비자 문제 (Producer-Consumer):**

```c
// 버퍼 크기 N
semaphore empty = N;  // 빈 슬롯 수
semaphore full = 0;   // 채워진 슬롯 수
semaphore mutex = 1;  // 버퍼 접근 상호 배제

// 생산자
P(empty);
P(mutex);
  버퍼에 아이템 삽입
V(mutex);
V(full);

// 소비자
P(full);
P(mutex);
  버퍼에서 아이템 제거
V(mutex);
V(empty);
```

**세마포어 vs 뮤텍스 차이:**

| 구분 | 뮤텍스 | 세마포어 |
|---|---|---|
| **소유권** | 있음 (lock 건 스레드만 unlock) | 없음 |
| **카운터 범위** | 0 또는 1 | 0 이상 정수 |
| **용도** | 임계 구역 보호 | 자원 카운팅, 프로세스 동기화 |
| **초기값** | 1 | 0~N |

#### ③ Monitor (모니터)

**개념:** 공유 자원과 그 자원에 대한 동기화 코드를 **하나의 객체로 캡슐화**한 고수준 동기화 추상화. Hoare가 제안.

```
┌─────────────────────────────────────────┐
│              Monitor                    │
│ ┌─────────────────────────────────────┐ │
│ │         공유 변수 (Shared Data)      │ │
│ └─────────────────────────────────────┘ │
│ ┌─────────────────────────────────────┐ │
│ │  동기화 코드 (Synchronized Methods) │ │
│ │  entry1(), entry2(), ...            │ │
│ └─────────────────────────────────────┘ │
│ ┌─────────────────────────────────────┐ │
│ │ 조건 변수 (Condition Variables)     │ │
│ │  wait(), signal()                   │ │
│ └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
    ↑ 한 번에 하나의 프로세스만 진입 가능
```

**특징:**
- 모니터 내부 메서드는 한 번에 하나의 스레드만 실행 가능 (컴파일러가 상호 배제 보장)
- **조건 변수(Condition Variable):** `wait()`(모니터에서 나가 대기), `signal()`(대기 중 스레드 깨움)
- Java의 `synchronized` 키워드가 모니터 패턴 구현

```java
// Java 모니터 예시
public class BoundedBuffer {
    synchronized void put(Object item) { // 자동으로 상호 배제
        while (buffer.isFull()) wait();  // 조건 변수 대기
        buffer.add(item);
        notifyAll();                      // 대기 스레드 깨움
    }
    synchronized Object take() {
        while (buffer.isEmpty()) wait();
        Object item = buffer.remove();
        notifyAll();
        return item;
    }
}
```

### 3. 핵심 키워드 (Must-Have)

- **임계 구역(Critical Section):** 공유 자원 접근 코드 영역
- **상호 배제, 진행, 한계 대기:** 임계 구역 문제 해결 3조건
- **뮤텍스(Mutex):** 소유권 있는 이진 잠금
- **스핀락 vs 블로킹 뮤텍스:** 대기 방식의 차이
- **세마포어 P(wait)/V(signal) 연산**
- **이진 세마포어 / 카운팅 세마포어**
- **생산자-소비자 문제:** 세마포어 3개(empty, full, mutex)로 해결
- **모니터(Monitor):** 캡슐화된 고수준 동기화, 조건 변수(wait/signal)
- **Java synchronized / Python threading.Lock**

### 4. 맥락과 비교

| 구분 | 뮤텍스 | 세마포어 | 모니터 |
|---|---|---|---|
| **추상화 수준** | 낮음 | 낮음 | 높음 |
| **소유권** | 있음 | 없음 | 없음 (메서드 단위) |
| **자원 수** | 1 | 여러 개 | 내부 정의 |
| **사용 오류 가능성** | 중간 | 높음 (순서 오류) | 낮음 (컴파일러 지원) |
| **언어 지원** | C pthread, Rust | C, 어셈블리 | Java, C#, Python |

### 5. 실무 제언

**세마포어 오용으로 인한 교착상태 — 순서가 핵심**

- **챌린지:** 세마포어의 P/V 연산 순서를 잘못 작성하면 교착상태가 발생합니다. 생산자-소비자 문제에서 P(mutex)를 P(empty) 앞에 호출하면 버퍼가 가득 찬 상태에서 뮤텍스를 들고 empty를 기다리는 교착상태가 됩니다.
- **제언:** Java의 `BlockingQueue`나 Python의 `queue.Queue` 같이 동기화 로직이 캡슐화된 고수준 추상화를 사용하는 것이 실무에서 안전합니다. 직접 세마포어를 사용할 때는 반드시 P 연산 순서를 문서화하고 코드 리뷰에서 검증해야 합니다. 모니터 패턴을 활용하면 컴파일러 수준에서 상호 배제를 강제하므로 오류 가능성이 크게 줄어듭니다.
