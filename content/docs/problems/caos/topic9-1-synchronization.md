## 해설

### 출제 배경 및 의도

다중 프로세스/스레드 환경에서 공유 자원(메모리, 파일, 하드웨어)에 여러 실행 주체가 동시에 접근하면 경쟁 조건(Race Condition)이 발생하여 예측 불가능한 결과를 낳는다. 동기화(Synchronization)는 임계 구역(Critical Section)에 대한 접근을 질서 있게 제어하는 메커니즘이다.

139회 관리 문제로 최근에 출제되었다. 채점관은 임계 구역 문제의 3가지 해결 조건(상호 배제·진행·한계 대기), 각 동기화 기법의 동작 원리와 차이점, 그리고 생산자-소비자 문제 등 고전적 동기화 문제에서의 적용을 이해하는지를 본다.

세마포어 P/V 연산 순서 오류로 인한 교착상태, Java synchronized 기반 모니터 패턴의 장점까지 연결하면 실무 역량을 효과적으로 어필할 수 있다.

### 1. 프로세스 동기화(Process Synchronization), 개요

정 의  • 임계 구역(Critical Section)에 대한 동시 접근을 제어하여 경쟁 조건(Race Condition)을 방지하고 데이터 일관성을 보장하는 운영체제 메커니즘
       - 상호 배제(Mutual Exclusion)·진행(Progress)·한계 대기(Bounded Waiting) 세 가지 요건을 모두 충족해야 함

```
스레드 A                    스레드 B
    │                           │
    ▼ 공유 변수 x = 0           │
  x 읽기  (x=0)                │
    │        ◀── 컨텍스트 스위치 ──▶ x 읽기 (x=0)
    │                           │
  x+1 계산  (1)               x+1 계산 (1)
    │                           │
  x 쓰기  (x=1)               x 쓰기  (x=1)
    │                           │
    ▼                           ▼
  결과: x=1  (기대값 x=2) ← Race Condition!
```

- 동기화 기법이 없으면 공유 변수의 최종값이 실행 순서에 따라 달라지는 비결정적(Non-deterministic) 오류가 발생한다.

### 2. 동기화 기법 상세

1) Mutex (Mutual Exclusion Lock)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 이진 잠금 | 잠김(1)/열림(0) 두 상태, 소유권(Ownership) 포함 |
| 소유권 | 있음 | lock을 걸었던 스레드만 unlock 가능 |
| 대기 방식 | 스핀락(Spinlock) | 잠금이 풀릴 때까지 CPU로 루프 재시도, 짧은 대기에 유리 |
| 대기 방식 | 블로킹 뮤텍스 | 잠금 대기 시 스레드를 sleep 전환, 긴 대기에 유리 |

```
스레드 A:           스레드 B:
lock(mutex)         lock(mutex) ← 블록(대기)
  임계 구역          ...
unlock(mutex) ──▶  스레드 B 깨어남
                    임계 구역
                   unlock(mutex)
```

2) Semaphore (세마포어, Dijkstra 제안)

| 구분 | P(wait) 연산 | V(signal) 연산 |
|------|--------------|----------------|
| 동작 | S>0이면 S-- 후 진입; S=0이면 대기 큐로 이동(블록) | S++; 대기 큐에 프로세스 있으면 하나 깨움(wake up) |

| 유형 | 값 범위 | 용도 |
|------|---------|------|
| 이진 세마포어(Binary) | 0 또는 1 | 임계 구역 보호(뮤텍스 유사, 단 소유권 없음) |
| 카운팅 세마포어(Counting) | 0 이상 정수 | 복수 자원 관리(DB 연결 풀 등) |

```
생산자-소비자 문제 (버퍼 크기 N):
  semaphore empty = N;  // 빈 슬롯 수
  semaphore full  = 0;  // 채워진 슬롯 수
  semaphore mutex = 1;  // 버퍼 접근 상호 배제

  생산자: P(empty) → P(mutex) → 버퍼 삽입 → V(mutex) → V(full)
  소비자: P(full)  → P(mutex) → 버퍼 제거 → V(mutex) → V(empty)
```

3) Monitor (모니터, Hoare 제안)

```
┌─────────────────────────────────────────────┐
│                  Monitor                    │
│  ┌───────────────────────────────────────┐  │
│  │         공유 변수 (Shared Data)        │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │  동기화 메서드 (Synchronized Methods) │  │
│  │  entry1(), entry2(), ...              │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │  조건 변수 (Condition Variables)      │  │
│  │  wait() → 모니터 반납 후 대기         │  │
│  │  signal() → 대기 스레드 하나 깨움     │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
   ↑ 한 번에 하나의 스레드만 진입 가능 (컴파일러 보장)
```

### 3. 세 기법 비교 및 고전 동기화 문제 적용

1) Mutex vs Semaphore vs Monitor 비교

| 구분 | 뮤텍스(Mutex) | 세마포어(Semaphore) | 모니터(Monitor) |
|------|---------------|---------------------|-----------------|
| 추상화 수준 | 낮음 | 낮음 | 높음 |
| 소유권 | 있음 | 없음 | 없음(메서드 단위) |
| 자원 수 | 1개 | 여러 개 | 내부 정의 |
| 사용 오류 가능성 | 중간 | 높음(순서 오류) | 낮음(컴파일러 지원) |
| 대표 언어 지원 | C pthread, Rust | C, 어셈블리 | Java synchronized, C# lock |
| 임계 구역 해결 요건 | 모두 충족 | 모두 충족 | 모두 충족 |

2) 고전 동기화 문제 해결 매핑

| 문제 | 권장 기법 | 핵심 포인트 |
|------|-----------|-------------|
| 임계 구역 보호 | Mutex | 소유권으로 unlock 오용 방지 |
| 생산자-소비자 | Semaphore(empty+full+mutex) | P 연산 순서: 자원 세마포어 → mutex 순서 엄수 |
| 독자-저자(Readers-Writers) | Semaphore + 카운터 | 복수 독자 동시 허용, 저자는 단독 |
| 식사하는 철학자 | Monitor 또는 Semaphore | 교착상태 방지: 자원 획득 순서 일관화 |

- 세마포어 P 연산 순서 오류(mutex를 자원 세마포어보다 먼저 획득)는 교착상태를 유발하므로, 모니터 패턴으로 캡슐화하여 컴파일러가 상호 배제를 강제하는 방식이 안전하다.  "끝"

### 실무 제언

**세마포어 P/V 순서 오류와 교착상태 방지**
- **챌린지**: 생산자-소비자 문제에서 P(mutex)를 P(empty) 앞에 호출하면 버퍼가 가득 찬 상태에서 mutex를 들고 empty를 기다리는 교착상태가 발생한다. 이 오류는 단위 테스트에서 발견하기 어렵고 부하가 높을 때만 나타나는 경향이 있다.
- **제언**: Java의 BlockingQueue나 Python의 queue.Queue 같이 동기화 로직이 캡슐화된 고수준 추상화를 사용하는 것이 실무에서 안전하다. 직접 세마포어를 사용할 때는 P 연산 순서를 주석으로 문서화하고 코드 리뷰에서 반드시 검증해야 한다.

**Java 가상 스레드(Virtual Threads)와 동기화 패러다임 변화**
- **챌린지**: Java 21의 가상 스레드(Project Loom)는 수백만 개의 경량 스레드를 지원하지만, synchronized 블록(모니터) 내에서 블로킹 I/O를 호출하면 캐리어 스레드(OS 스레드)가 고정(pinned)되어 가상 스레드의 장점이 상실된다.
- **제언**: 가상 스레드 환경에서는 synchronized 대신 ReentrantLock + Condition을 사용하거나, I/O 경로에서 명시적 블로킹을 최소화하도록 코드를 재구성해야 한다. JDK Flight Recorder로 pinned 이벤트를 모니터링하여 병목 지점을 파악해야 한다.

**분산 시스템에서의 동기화: 분산 락(Distributed Lock)**
- **챌린지**: 마이크로서비스 환경에서 여러 인스턴스가 공유 자원(DB 레코드, 외부 API 쿼터)에 동시 접근하면 단일 프로세스 내 뮤텍스/세마포어로는 상호 배제를 보장할 수 없다.
- **제언**: Redis의 SET NX EX 명령어(Redlock 알고리즘) 또는 ZooKeeper의 임시 노드(Ephemeral Node)를 활용한 분산 락을 적용해야 한다. 분산 락은 네트워크 파티션 시 락 해제가 자동으로 이루어지도록 TTL을 반드시 설정하고, 락 갱신(heartbeat) 메커니즘으로 긴 작업 처리 중 락 만료를 방지해야 한다.
