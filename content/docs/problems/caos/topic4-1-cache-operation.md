## 해설

### 출제 배경 및 의도

현대 CPU의 클럭 속도와 메모리(DRAM) 접근 속도의 차이는 100배 이상에 달한다. 캐시메모리는 이 속도 간격(Memory Wall)을 극복하기 위한 핵심 장치로, 프로그램의 지역성(Locality) 원리를 활용하여 자주 사용되는 데이터를 CPU 근처에 미리 배치한다.

채점관은 캐시의 동작 흐름(히트/미스 처리), 주소 사상 3가지 방식의 원리와 비교, 지역성의 종류를 정확히 서술하는지를 본다. 특히 각 사상 방식의 하드웨어 구현 방식(비교기, CAM)까지 언급하고 캐시 미스의 3C 분류(강제·용량·충돌)와 연결하면 심화 역량을 어필할 수 있다.

소프트웨어 관점에서도 캐시 친화적 코딩(루프 타일링, 데이터 구조 배치)이 성능에 미치는 영향을 이해해야 실무에서 활용할 수 있다.

### 1. 캐시메모리(Cache Memory), 개요

정 의  • 시간 지역성과 공간 지역성을 활용하여 CPU와 주기억장치 사이의 속도 격차를 완화하는 고속 소용량 메모리 계층
       - 주기억장치 블록을 캐시 라인(Cache Line) 단위로 적재하여 반복 접근 시 빠른 응답 제공

```
CPU Core
   │
   ├── L1 Cache (32~64KB, ~1ns, 4-way)  ← 명령어/데이터 분리
   │
   ├── L2 Cache (256KB~1MB, ~5ns, 8-way)
   │
   ├── L3 Cache (공유, 8~32MB, ~20ns, 16-way)
   │
   └── Main Memory (DRAM, 수~수십GB, ~100ns)
              │
         Storage (SSD/HDD, ~수ms)
```

- 계층적 캐시 구조는 용량과 속도의 트레이드오프를 단계적으로 완화하여 CPU 성능을 극대화한다.

### 2. 캐시 동작 순서 및 지역성 원리

1) 지역성(Locality) 원리

| 구분 | 정의 | 예시 |
|------|------|------|
| 시간 지역성(Temporal Locality) | 최근에 참조된 데이터는 가까운 미래에 다시 참조될 가능성이 높다 | 루프 변수, 자주 호출되는 함수 |
| 공간 지역성(Spatial Locality) | 참조된 데이터 근처의 데이터도 곧 참조될 가능성이 높다 | 배열 순차 접근, 명령어 순차 실행 |

2) 캐시 동작 순서

```
CPU가 주소 A 요청
         │
         ▼
  ┌──────────────┐
  │  캐시 탐색   │──Hit──▶ 캐시에서 데이터 반환 → CPU 전달 (수 ns)
  │  (Tag 비교)  │
  └──────────────┘
         │ Miss
         ▼
  주기억장치(RAM)에서 블록(Cache Line, 64바이트) 단위로 읽기 (수백 ns)
         │
         ▼
  캐시에 블록 적재 (교체 알고리즘 적용: LRU/LFU)
         │
         ▼
  CPU로 데이터 전달
```

- **평균 접근 시간 T = h × Tc + (1-h) × (Tc + Tm)**  
  (h: 적중률, Tc: 캐시 접근 시간, Tm: 주기억장치 접근 시간)

### 3. 주소 사상(Address Mapping) 유형 비교

1) 직접 사상(Direct Mapping)

```
메모리 주소 = [ Tag | 캐시 인덱스 | Offset ]
블록 번호 mod 캐시 슬롯 수 = 캐시 슬롯 인덱스 (1:1 고정 매핑)

메모리 블록 0  → 캐시 슬롯 0
메모리 블록 4  → 캐시 슬롯 0 (충돌 미스!)
```

| 구분 | 항목 | 내용 |
|------|------|------|
| 슬롯 결정 | 고정 | 블록 번호 mod 슬롯 수로 자동 결정 |
| 충돌 미스 | 많음 | 동일 슬롯 경쟁 블록 간 충돌 빈발 |
| 하드웨어 비용 | 낮음 | 단순 비교기 1개 |
| 검색 속도 | 빠름 | 인덱스로 즉시 위치 결정 |

2) 완전 연관 사상(Fully Associative Mapping)

```
메모리 주소 = [ Tag | Offset ]
메모리 블록 → 캐시 임의 슬롯 배치 (제한 없음)
모든 슬롯의 Tag를 병렬 비교 (CAM, Content Addressable Memory)
```

| 구분 | 항목 | 내용 |
|------|------|------|
| 슬롯 결정 | 자유 | 빈 슬롯 어디에나 배치 가능 |
| 충돌 미스 | 없음 | 임의 배치로 충돌 원천 제거 |
| 하드웨어 비용 | 높음 | 슬롯 수만큼의 비교기(CAM) 필요 |
| 실제 사용 | TLB 등 소형 | 소용량에서만 실용적 |

3) 집합 연관 사상(Set Associative Mapping, k-way)

```
메모리 주소 = [ Tag | Set 인덱스 | Offset ]
캐시를 k개 슬롯씩 묶은 '집합(Set)'으로 구성
집합 내에서는 완전 연관, 집합 간에는 직접 사상

2-way Set Associative:
  Set 0: [슬롯 A, 슬롯 B]  ← 메모리 블록 0, 4, 8 ... 배치 가능
  Set 1: [슬롯 C, 슬롯 D]  ← 메모리 블록 1, 5, 9 ... 배치 가능
```

| 구분 | 직접 사상 | 완전 연관 | 집합 연관(k-way) |
|------|-----------|-----------|-----------------|
| 슬롯 결정 | 인덱스 고정 | 임의 자유 | 집합 고정, 집합 내 자유 |
| 충돌 미스 | 많음 | 없음 | 줄어듦(k 증가할수록 감소) |
| 하드웨어 비용 | 낮음 | 높음 | 중간(비교기 k개) |
| 실제 사용 | 구형 CPU | TLB 등 소형 | 현대 CPU L1~L3 표준 |

- 집합 연관 사상은 직접 사상의 충돌 미스와 완전 연관의 하드웨어 복잡도를 동시에 완화하여 현대 CPU의 표준 방식이다.  "끝"

### 실무 제언

**캐시 친화적 코딩(Cache-Friendly Programming)**
- **챌린지**: 동일한 알고리즘이라도 캐시 친화적이냐에 따라 수십 배 성능 차이가 난다. C/C++의 2차원 배열은 행 우선(Row-major) 저장이므로, 내부 루프에서 열을 순회하면 공간 지역성이 깨져 캐시 미스가 폭발한다.
- **제언**: 고성능 연산 코드에 루프 타일링(Loop Tiling/Blocking) 기법을 적용하여 캐시에 올라온 데이터를 최대한 재사용하도록 접근 패턴을 재설계해야 한다. Intel VTune·Linux perf의 `cache-misses` 이벤트로 L3 미스율을 측정하고, 10% 이상이면 데이터 접근 패턴 재검토가 필요하다.

**캐시 미스 3C 분류별 대응 전략**
- **챌린지**: 캐시 미스는 강제(Compulsory)·용량(Capacity)·충돌(Conflict) 세 유형으로 발생하며 각각 원인과 해결책이 다르다.
- **제언**: 강제 미스는 하드웨어 프리페칭(Prefetching)과 소프트웨어 `__builtin_prefetch`로, 용량 미스는 캐시 크기 증설 또는 워킹셋 최소화로, 충돌 미스는 k-way 연관도 증가 또는 데이터 패딩으로 각각 대응해야 한다.

**NUMA 아키텍처에서 캐시 일관성**
- **챌린지**: 멀티소켓 서버(NUMA)에서 원격 소켓의 캐시 데이터 접근은 로컬 접근 대비 2~4배 지연이 발생한다. 잘못된 스레드-코어 배치는 캐시 미스와 NUMA 원격 접근을 동시에 유발한다.
- **제언**: `numactl --cpunodebind=0 --membind=0` 등으로 프로세스를 동일 NUMA 노드에 바인딩하고, `perf stat -e cache-references,cache-misses,LLC-load-misses`로 LLC(L3) 미스율을 측정하여 NUMA 친화적 메모리 배치를 검증해야 한다.
