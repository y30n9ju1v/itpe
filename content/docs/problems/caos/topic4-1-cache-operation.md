## 문제

중앙처리장치(CPU)에서 명령어 블록을 인출하기 위한 캐시기억장치(Cache Memory)의 동작 순서를 설명하고, 캐시기억장치의 주소 사상(Address Mapping) 유형별 동작 방식에 대하여 설명하시오.

## 해설

> **CPU와 메모리 사이의 속도 간격을 메우는 핵심 장치**  
> 캐시의 동작 원리와 직접 사상, 완전 연관, 집합 연관 사상의 트레이드오프

현대 CPU의 클럭 속도와 메모리(DRAM) 접근 속도의 차이는 100배 이상에 달합니다. 캐시메모리는 이 속도 간격(Memory Wall)을 극복하기 위한 핵심 장치로, 프로그램의 **지역성(Locality)** 원리를 활용하여 자주 사용되는 데이터를 CPU 근처에 미리 배치합니다.

### 1. 출제 배경 및 의도

캐시메모리는 컴퓨터구조 도메인에서 성능 최적화의 핵심입니다. 채점관은 **캐시의 동작 흐름(히트/미스 처리)**, **주소 사상 3가지 방식의 원리와 비교**, **지역성의 종류**를 정확히 서술하는지를 봅니다. 특히 각 사상 방식의 하드웨어 구현 방식(비교기, CAM)까지 언급하면 심화 역량을 어필할 수 있습니다.

### 2. 개념의 본질과 작동 메커니즘

#### 지역성의 원리 (Principle of Locality)

캐시가 효과적인 이유는 프로그램의 메모리 접근 패턴에 다음 두 가지 지역성이 있기 때문입니다:

- **시간 지역성(Temporal Locality):** 최근에 참조된 데이터는 가까운 미래에 다시 참조될 가능성이 높다 (루프, 변수 재사용)
- **공간 지역성(Spatial Locality):** 참조된 데이터 근처의 데이터도 곧 참조될 가능성이 높다 (배열 순차 접근, 명령어 순차 실행)

#### 캐시 기억장치 동작 순서

```
CPU가 주소 A 요청
         │
         ▼
  ┌──────────────┐
  │  캐시 탐색   │ → Hit? Yes → 캐시에서 데이터 반환 → CPU로 전달
  │  (Tag 비교)  │
  └──────────────┘
         │ Miss
         ▼
  주기억장치(RAM)에서 해당 블록 읽기
         │
         ▼
  캐시에 블록 적재 (교체 알고리즘 적용)
         │
         ▼
  CPU로 데이터 전달
```

**캐시 동작 세부 단계:**
1. CPU가 메모리 주소 발행
2. 캐시 컨트롤러가 해당 주소의 **태그(Tag)** 와 캐시 내 태그 비교
3. **캐시 히트(Cache Hit):** 일치하는 태그 발견 → 캐시에서 데이터 직접 반환 (수 ns)
4. **캐시 미스(Cache Miss):** 불일치 → 주기억장치 접근 (수십~수백 ns)
5. 미스 시 주기억장치에서 **블록(Block/Cache Line)** 단위로 캐시에 적재
6. 적재 시 기존 항목 교체 필요하면 LRU/LFU 등 교체 알고리즘 적용
7. 데이터를 CPU에 전달

**캐시 히트율(Cache Hit Rate):**
- 적중률(h) = 캐시 히트 횟수 / 전체 메모리 접근 횟수
- 평균 접근 시간 T = h × Tc + (1-h) × (Tc + Tm)  
  (Tc: 캐시 접근 시간, Tm: 주기억장치 접근 시간)

#### 주소 사상(Address Mapping) 유형

메인 메모리의 블록을 캐시의 어느 슬롯에 저장할지 결정하는 방식입니다.

**① 직접 사상(Direct Mapping)**

```
메인 메모리 주소 = [ Tag | 캐시 인덱스 | Offset ]

메모리 블록 → 캐시 슬롯 1:1 고정 매핑
블록 번호 mod 캐시 슬롯 수 = 캐시 슬롯 인덱스
```

```
메모리 블록 0  → 캐시 슬롯 0
메모리 블록 1  → 캐시 슬롯 1
메모리 블록 2  → 캐시 슬롯 2
메모리 블록 3  → 캐시 슬롯 3
메모리 블록 4  → 캐시 슬롯 0 (충돌!)
메모리 블록 5  → 캐시 슬롯 1 (충돌!)
```

- **장점:** 구현 단순, 빠른 조회 (인덱스로 즉시 위치 결정)
- **단점:** 같은 캐시 슬롯을 경쟁하는 블록이 많으면 충돌 미스(Conflict Miss) 빈발
- **하드웨어:** 단순한 비교기 1개

**② 완전 연관 사상(Fully Associative Mapping)**

```
메모리 블록 → 캐시 임의 슬롯 배치 (제한 없음)
주소 = [ Tag | Offset ]
```

- **장점:** 충돌 미스 없음. 공간 효율 최대
- **단점:** 모든 슬롯의 태그를 병렬 비교해야 하므로 하드웨어 복잡, 비용 대
- **하드웨어:** 슬롯 수만큼의 비교기 (CAM, Content Addressable Memory)

**③ 집합 연관 사상(Set Associative Mapping)**

```
캐시를 k개 슬롯씩 묶은 '집합(Set)'으로 구성
주소 = [ Tag | Set 인덱스 | Offset ]

집합 내에서는 완전 연관, 집합 간에는 직접 사상
```

```
2-way Set Associative (집합당 2개 슬롯):
Set 0: [슬롯 A, 슬롯 B]  ← 메모리 블록 0, 4, 8 ...이 배치 가능
Set 1: [슬롯 C, 슬롯 D]  ← 메모리 블록 1, 5, 9 ...이 배치 가능
```

- **장점:** 직접 사상의 충돌 미스 대폭 감소 + 완전 연관의 복잡도 완화
- **단점:** 직접 사상보다 하드웨어 복잡 (비교기 k개)
- **현실:** 대부분의 현대 CPU는 4-way, 8-way, 16-way 집합 연관 방식 채택

#### 주소 사상 방식 비교

| 비교 항목 | 직접 사상 | 완전 연관 | 집합 연관 (k-way) |
|---|---|---|---|
| **슬롯 결정** | 인덱스로 고정 | 임의 자유 | 집합은 고정, 집합 내 자유 |
| **충돌 미스** | 많음 | 없음 | 줄어듦 |
| **하드웨어 비용** | 낮음 | 높음 | 중간 |
| **검색 속도** | 빠름 | 느림 | 중간 |
| **실제 사용** | 구형 CPU | TLB 등 소형 | 현대 CPU L1~L3 |

### 3. 핵심 키워드 (Must-Have)

- **시간 지역성 / 공간 지역성**
- **캐시 히트(Hit) / 캐시 미스(Miss)**
- **캐시 라인(Cache Line) / 블록:** 캐시와 메모리 간 전송 단위 (64바이트 일반적)
- **직접 사상:** 블록 번호 mod 슬롯 수, 충돌 미스 빈발
- **완전 연관 사상:** 임의 배치, CAM 필요, 충돌 없음
- **집합 연관 사상(k-way):** 현대 CPU 표준, 직접+완전의 절충
- **태그(Tag):** 캐시 슬롯이 어느 메모리 블록을 담고 있는지 식별하는 비트
- **캐시 미스 유형:** 강제 미스(Cold), 용량 미스(Capacity), 충돌 미스(Conflict)

### 4. 맥락과 비교

**캐시 미스의 3C 분류:**

| 유형 | 원인 | 해결책 |
|---|---|---|
| **강제 미스 (Compulsory)** | 처음 접근하는 데이터 → 무조건 미스 | 프리페칭(Prefetching) |
| **용량 미스 (Capacity)** | 캐시 크기가 워킹셋보다 작음 | 캐시 크기 증가 |
| **충돌 미스 (Conflict)** | 같은 캐시 슬롯 경쟁 | 집합 연관도(k) 증가 |

**현대 CPU의 다단계 캐시 계층:**

```
CPU Core
  │
  L1 Cache (32~64KB, ~1ns, 4-way) ← 명령어 캐시 + 데이터 캐시 분리
  │
  L2 Cache (256KB~1MB, ~5ns, 8-way)
  │
  L3 Cache (공유, 8~32MB, ~20ns, 16-way)
  │
  Main Memory (DRAM, 수~수십GB, ~100ns)
  │
  Storage (SSD/HDD, ~수ms)
```

### 5. 실무 제언

**소프트웨어 최적화에서 캐시 친화적 코딩의 중요성**

- **챌린지:** 동일한 알고리즘이라도 캐시 친화적이냐 아니냐에 따라 수십 배 성능 차이가 납니다. 대표적인 예가 행렬 곱셈 코드에서 행(Row) 우선 접근과 열(Column) 우선 접근의 성능 차이입니다. C/C++는 행 우선(Row-major) 저장 방식이므로, 내부 루프에서 열을 순회하면 공간 지역성이 깨져 캐시 미스가 폭발합니다.
- **제언:** 고성능 연산 코드 작성 시 **루프 변환(Loop Tiling/Blocking)** 기법을 적용하여 캐시에 올라온 데이터를 최대한 재사용하도록 접근 패턴을 바꿔야 합니다. Intel VTune, Linux perf의 `cache-misses` 이벤트로 캐시 미스율을 측정하고, L3 미스율이 높으면 데이터 접근 패턴 재설계를 검토해야 합니다.
