## 문제

페이지 방식을 이용하는 가상 메모리에서 가상주소(Virtual address)를 주기억장치 주소인 물리주소(Physical Address)로 변환하는 과정을 페이지 테이블(Page Table) 및 TLB(Translation Lookaside Buffer)를 이용하여 설명하시오.

## 출제 정보

| 항목 | 내용 |
|------|------|
| 토픽(대) | 가상메모리관리 |
| 토픽(소) | 변환과정TLB |

## 해설

> **주소 변환이라는 필수 관문, 그리고 속도를 위한 TLB라는 지름길**  
> MMU와 페이지 테이블, TLB의 3자 협력으로 완성되는 가상화 원리

가상메모리에서 CPU는 항상 가상주소(Virtual Address)를 발행합니다. 이 가상주소가 실제 데이터가 있는 물리주소(Physical Address)로 변환되는 과정은 모든 메모리 접근마다 이루어지는 핵심 작업입니다. 이 변환을 하드웨어적으로 수행하는 것이 **MMU(Memory Management Unit)**이며, 변환 속도를 높이기 위한 캐시가 **TLB(Translation Lookaside Buffer)**입니다.

### 1. 출제 배경 및 의도

110회 응용 문제로 출제된 이 주제는 페이징의 내부 동작 원리를 깊이 이해하는지를 검증합니다. 채점관은 **가상주소의 구조(VPN + Offset)**, **페이지 테이블 조회 과정**, **TLB 히트/미스 시의 동작 차이**, 그리고 **유효 메모리 접근 시간(EMAT)** 계산까지 체계적으로 서술하기를 기대합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가상주소의 구조

```
가상주소 (Virtual Address)
┌─────────────────────┬────────────────────┐
│  VPN (Virtual Page  │  Offset (페이지 내  │
│  Number, 페이지 번호)│  변위)             │
└─────────────────────┴────────────────────┘
  → 페이지 테이블 인덱스    → 물리주소에 그대로
```

예: 32비트 주소 체계, 페이지 크기 4KB(2^12)일 경우
- Offset: 하위 12비트
- VPN: 상위 20비트 (최대 2^20 = 1M개 페이지)

#### 페이지 테이블(Page Table)을 통한 주소 변환

```
CPU → [가상주소: VPN=5, Offset=100]
           │
           ▼ MMU가 페이지 테이블 참조
   ┌───────────────────────┐
   │    Page Table         │
   │  VPN │ PFN │ Valid    │
   │  ...  │ ... │  ...    │
   │   5  │  12 │   1     │ ← 유효(Valid)
   │  ...  │ ... │  ...    │
   └───────────────────────┘
           │
           ▼
   [물리주소: PFN=12, Offset=100]
   = Frame 12의 시작주소 + 100
```

**변환 과정 단계:**
1. CPU가 가상주소 발행
2. MMU가 VPN을 추출하여 페이지 테이블 인덱스로 사용
3. 페이지 테이블 항목(PTE, Page Table Entry) 읽기
4. Valid 비트 확인 → 0이면 **페이지 폴트** 발생 → OS가 디스크에서 적재
5. PFN(Physical Frame Number) 획득
6. 물리주소 = PFN × 페이지 크기 + Offset

**문제점:** 페이지 테이블이 메모리에 있으므로 메모리 접근이 2번 필요 (테이블 조회 1회 + 실제 데이터 접근 1회) → 성능 저하

#### TLB(Translation Lookaside Buffer)

TLB는 최근 사용된 VPN→PFN 변환 결과를 캐싱하는 **하드웨어 연관 메모리(Associative Memory)** 입니다.

```
CPU → [가상주소: VPN=5, Offset=100]
           │
           ▼
   ┌───────────────────────┐
   │   TLB (연관 메모리)   │
   │  VPN │ PFN │ 보호비트 │
   │   3  │  7  │   R/W   │
   │   5  │  12 │   R/W   │ ← TLB Hit!
   │   8  │  2  │   R     │
   └───────────────────────┘
           │ Hit
           ▼
   [물리주소: 12 × 4096 + 100] → 메모리 직접 접근
```

**TLB Hit 경로 (빠른 경로):**
```
가상주소 → TLB 조회(병렬) → Hit → 물리주소 획득 → 메모리 접근 (총 1회 메모리 접근)
```

**TLB Miss 경로 (느린 경로):**
```
가상주소 → TLB 조회 → Miss → 페이지 테이블 조회(메모리 1회) 
→ PTE 획득 → TLB 갱신 → 물리주소 → 메모리 접근 (총 2회 메모리 접근)
```

#### 유효 메모리 접근 시간(EMAT, Effective Memory Access Time)

```
EMAT = TLB 적중률 × (TLB 접근 시간 + 메모리 접근 시간)
     + TLB 미적중률 × (TLB 접근 시간 + 2 × 메모리 접근 시간)
```

**계산 예시:**
- TLB 접근 시간: 20ns, 메모리 접근 시간: 100ns, TLB 적중률: 90%
- EMAT = 0.9 × (20 + 100) + 0.1 × (20 + 100 + 100)
- EMAT = 0.9 × 120 + 0.1 × 220 = 108 + 22 = **130ns**
- (TLB 없이는 200ns, TLB로 35% 성능 향상)

#### 다단계 페이지 테이블

64비트 시스템에서 단일 페이지 테이블은 메모리를 과도하게 차지합니다. 이를 해결하기 위해 **다단계(계층적) 페이지 테이블**을 사용합니다.

```
가상주소: [ L1 인덱스 | L2 인덱스 | L3 인덱스 | Offset ]
              │              │              │
         L1 페이지       L2 페이지       L3 페이지
          테이블 →        테이블 →        테이블 → PFN
```

- x86-64는 4단계 페이지 테이블 사용 (PGD→PUD→PMD→PTE)
- 사용되지 않는 주소 공간의 하위 테이블은 생성하지 않아 메모리 절약
- 단, 레벨만큼 메모리 접근 횟수 증가 → TLB 중요성 더욱 커짐

### 3. 핵심 키워드 (Must-Have)

- **MMU(Memory Management Unit):** 하드웨어 주소 변환 장치
- **VPN(Virtual Page Number):** 가상주소의 상위 비트
- **PFN(Physical Frame Number):** 물리 프레임 번호
- **Offset:** 페이지/프레임 내 변위 (가상→물리 동일)
- **PTE(Page Table Entry):** 페이지 테이블 항목 (PFN + Valid + Dirty + Protection 비트)
- **TLB(Translation Lookaside Buffer):** 주소 변환 캐시 (연관 메모리)
- **TLB Hit/Miss:** 히트 시 1회 메모리 접근, 미스 시 2회 이상
- **EMAT:** 유효 메모리 접근 시간 계산
- **TLB 플러시(Flush):** 컨텍스트 스위치 시 TLB 무효화
- **ASID(Address Space Identifier):** TLB 플러시 없이 프로세스 구분하는 태그

### 4. 맥락과 비교

| 구분 | TLB 없음 | TLB 있음 |
|---|---|---|
| **메모리 접근 횟수** | 2회 (단일 레벨) | 1~2회 |
| **성능** | 느림 | 빠름 (TLB 적중률 90%+ 이상이면 2배 이상 향상) |
| **컨텍스트 스위치 비용** | 없음 | TLB 플러시 필요 (ASID로 완화 가능) |

**TLB vs 캐시메모리 비교:**

| 구분 | TLB | 캐시메모리(L1, L2, L3) |
|---|---|---|
| **저장 내용** | 주소 변환 결과(VPN→PFN) | 실제 데이터/명령어 |
| **위치** | MMU 내부 | CPU 코어 근처 |
| **크기** | 수십~수백 항목 | 수KB~수MB |
| **목적** | 주소 변환 속도 향상 | 데이터 접근 속도 향상 |

### 5. 실무 제언

**컨텍스트 스위치와 TLB 플러시 성능 문제**

- **챌린지:** 운영체제가 프로세스를 교체(컨텍스트 스위치)할 때마다 이전 프로세스의 TLB 항목을 무효화(TLB 플러시)해야 합니다. TLB 플러시는 이후 수백 번의 TLB 미스를 유발하며, 이 비용이 컨텍스트 스위치 오버헤드의 상당 부분을 차지합니다.
- **제언:** ARM과 최신 x86 프로세서는 **ASID(Address Space Identifier)**를 TLB 항목에 태그하여 프로세스별로 TLB 항목을 구분합니다. 컨텍스트 스위치 시 전체 플러시 없이 다른 ASID의 항목만 무효화하여 성능을 유지합니다. 고성능 서버 환경에서 컨텍스트 스위치가 잦은 워크로드는 ASID 지원 여부를 CPU 선정 기준으로 고려해야 합니다.
