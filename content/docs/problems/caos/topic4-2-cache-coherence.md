## 해설

### 출제 배경 및 의도

단일 CPU 시절에는 캐시 하나만 존재하므로 일관성 문제가 없었다. 그러나 멀티코어 프로세서에서는 각 코어마다 독립적인 L1/L2 캐시를 가지며, 같은 메모리 주소를 여러 코어가 각자의 캐시에 복사본으로 보유한다. 한 코어가 값을 수정했을 때 다른 코어의 캐시가 옛날 값을 들고 있다면 데이터 불일치(Incoherence)가 발생한다.

공통 문제로 출제된 이 주제는 멀티코어 환경에서의 메모리 시스템 이해를 검증한다. 채점관은 쓰기 정책 2가지(Write-Through vs Write-Back)의 장단점, 캐시 일관성이 왜 문제가 되는지, 그리고 스누핑(Bus Snooping)과 디렉토리(Directory) 기반 프로토콜의 차이를 명확히 서술하기를 기대한다.

멀티스레드 프로그래밍에서 False Sharing이 성능에 미치는 영향과 alignas(64) 패딩 기법까지 연결하면 실무 역량을 효과적으로 어필할 수 있다.

### 1. 캐시 쓰기정책(Write Policy), 개요

정 의  • CPU가 데이터를 캐시에 쓸 때(Write Hit) 메인 메모리와의 동기화 시점과 방식을 결정하는 정책
       - Write-Through(즉시 쓰기)와 Write-Back(나중 쓰기)으로 구분하며 성능과 일관성이 트레이드오프 관계

```
Write-Through                       Write-Back
CPU 쓰기                             CPU 쓰기
    │                                   │
    ▼                                   ▼
캐시 업데이트                        캐시 업데이트
    │ 즉시                           (Dirty Bit = 1)
    ▼                                   │ 교체 시점에
메모리 업데이트                      메모리 업데이트

항상 동기화 → 일관성 용이            성능 우수 → 일관성 복잡
```

- 쓰기정책 선택은 시스템의 일관성 요구 수준과 성능 목표에 따라 결정된다.

### 2. 쓰기정책 상세 및 Write-Miss 처리

1) Write-Through vs Write-Back 비교

| 구분 | Write-Through | Write-Back |
|------|---------------|------------|
| 메모리 갱신 시점 | 쓰기 즉시 | 캐시 라인 교체 시 |
| 성능 | 낮음(쓰기마다 메모리 접근) | 높음(연속 쓰기 효과 극대화) |
| 일관성 유지 | 쉬움(항상 최신) | 복잡(Dirty Bit 관리) |
| 버스 대역폭 | 많이 사용 | 절약 |
| 장애 복구 | 단순 | 캐시 데이터 유실 가능 |
| 개선 기법 | 쓰기 버퍼(Write Buffer)로 큐잉 | Dirty Bit 1비트로 상태 추적 |

2) Write-Miss 처리 전략

| 전략 | 동작 | 주로 사용 |
|------|------|-----------|
| Write-Allocate | 메모리에서 블록을 캐시에 먼저 적재 후 쓰기 | Write-Back과 함께 |
| No-Write-Allocate | 캐시 적재 없이 메모리에 직접 쓰기 | Write-Through와 함께 |

- Write-Back + Write-Allocate 조합이 성능 최적화에 가장 유리하여 현대 CPU의 표준 방식으로 채택된다.

### 3. 캐시 일관성(Cache Coherence) 프로토콜 비교

1) 문제 발생 시나리오

```
메모리 주소 X = 100
  코어 0 캐시: X = 100
  코어 1 캐시: X = 100

→ 코어 0이 X = 200으로 수정 (Write-Back, 메모리 아직 100)

  코어 0 캐시: X = 200  ✓
  코어 1 캐시: X = 100  ✗  (Stale Data! 오래된 데이터)
  메인 메모리: X = 100  ✗
```

2) 스누핑(Bus Snooping) 프로토콜 - MESI

```
코어 0  코어 1  코어 2
  │       │       │
  └───────┴───────┘
          │ 공유 버스(Shared Bus)
     메인 메모리
각 코어의 캐시 컨트롤러가 버스를 감시(Snooping)하며 다른 코어의 쓰기 감지
```

| MESI 상태 | 의미 | 전이 조건 |
|-----------|------|-----------|
| M (Modified) | 수정됨, 메모리와 다름, 이 코어만 보유 | 단독 쓰기 시 |
| E (Exclusive) | 메모리와 동일, 이 코어만 보유 | 단독 읽기 시 |
| S (Shared) | 메모리와 동일, 여러 코어 공유 | 다중 코어 읽기 시 |
| I (Invalid) | 무효, 사용 불가 | 다른 코어 쓰기 감지 시 |

3) 디렉토리 기반(Directory-Based) 프로토콜

```
코어 0  코어 1  코어 N
  │       │       │
  └───────┴───────┘ 점대점 연결(NoC/Crossbar)
   ┌──────▼──────┐
   │  디렉토리   │ ← 어느 코어가 어느 블록 보유하는지 추적
   └─────────────┘
     메인 메모리
```

| 구분 | 스누핑 프로토콜 | 디렉토리 프로토콜 |
|------|----------------|------------------|
| 통신 방식 | 공유 버스 브로드캐스트 | 점대점 메시지 |
| 확장성 | 제한적(버스 대역폭 병목) | 우수(수백 코어 가능) |
| 지연 | 낮음 | 높음(디렉토리 조회) |
| 구현 복잡도 | 단순 | 복잡 |
| 적용 사례 | Intel 소켓 내 코어 간 | AMD NUMA, 서버 다소켓 |

- 소규모(~수십 코어)는 스누핑, 대규모(수십~수백 코어)는 디렉토리 기반 프로토콜이 현실적인 선택이다.  "끝"

### 실무 제언

**멀티스레드 프로그래밍에서 False Sharing 제거**
- **챌린지**: 서로 다른 스레드가 서로 다른 변수를 수정하더라도 그 변수들이 같은 캐시 라인(64바이트) 안에 위치하면 MESI 프로토콜의 불필요한 무효화가 반복 발생한다. 이를 False Sharing(거짓 공유)이라 하며 멀티스레드 코드에서 심각한 성능 저하 원인이 된다.
- **제언**: `alignas(64)` 또는 캐시 라인 크기의 패딩(Padding)을 사용하여 각 스레드의 데이터를 서로 다른 캐시 라인에 배치하면 False Sharing을 제거할 수 있다. Intel VTune "Memory Access" 분석이나 Linux perf의 `cache-references/cache-misses` 이벤트로 탐지 가능하다.

**Write-Back 정책에서의 데이터 무결성 보장**
- **챌린지**: Write-Back 정책에서 Dirty 상태의 캐시 라인이 메모리에 반영되기 전에 시스템 장애가 발생하면 데이터가 유실된다. 데이터베이스 WAL(Write-Ahead Log)과의 정합성 문제도 발생할 수 있다.
- **제언**: 영속성이 중요한 데이터 경로에는 `clflush` (Cache Line Flush) 또는 `CLWB` (Cache Line Write Back) 명령어로 특정 캐시 라인을 명시적으로 메모리에 내려보내야 한다. 영속 메모리(Persistent Memory, Intel Optane)를 사용하는 환경에서는 더욱 중요하다.

**클라우드 멀티테넌트 환경에서의 캐시 사이드 채널 공격**
- **챌린지**: 동일 물리 서버에서 실행되는 서로 다른 VM/컨테이너가 L3 캐시를 공유하면, 캐시 접근 타이밍을 분석하여 다른 테넌트의 비밀 데이터를 추론하는 Flush+Reload, Prime+Probe 류의 사이드 채널 공격이 가능하다.
- **제언**: 클라우드 환경에서 민감한 워크로드는 CAT(Cache Allocation Technology)를 통해 LLC 파티션을 격리하거나, 전용 물리 호스트(Bare Metal) 배포를 통해 캐시 공유 자체를 차단하는 것이 보안 설계의 기본 원칙이다.
