## 해설

> **멀티코어 시대의 핵심 난제: 모든 캐시가 같은 값을 보고 있는가?**  
> 쓰기정책(Write-Through/Write-Back)과 스누핑/디렉토리 기반 일관성 프로토콜

단일 CPU 시절에는 캐시 하나만 존재하므로 일관성 문제가 없었습니다. 그러나 멀티코어 프로세서에서는 각 코어마다 독립적인 L1/L2 캐시를 가지며, 같은 메모리 주소를 여러 코어가 각자의 캐시에 복사본으로 보유합니다. 한 코어가 값을 수정했을 때 다른 코어의 캐시가 옛날 값을 들고 있다면 데이터 불일치(Incoherence)가 발생합니다.

### 1. 출제 배경 및 의도

공통 문제로 출제된 이 주제는 멀티코어 환경에서의 메모리 시스템 이해를 검증합니다. 채점관은 **쓰기 정책 2가지(Write-Through vs Write-Back)**의 장단점, **캐시 일관성이 왜 문제가 되는지**, 그리고 **스누핑(Bus Snooping)과 디렉토리(Directory) 기반 프로토콜**의 차이를 명확히 서술하기를 기대합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 캐시 쓰기정책(Write Policy)

CPU가 데이터를 캐시에 쓸 때(Write Hit), 메인 메모리와의 동기화를 어떻게 처리할지 정하는 정책입니다.

**① Write-Through (즉시 쓰기)**

```
CPU 쓰기 → 캐시 동시 업데이트
           ↓ 즉시
         메인 메모리 업데이트
```

- **특징:** 쓰기가 발생할 때마다 캐시와 메모리를 **동시에** 업데이트
- **장점:** 메모리 항상 최신 상태 유지 → 일관성 보장 용이, 복구 단순
- **단점:** 쓰기마다 느린 메모리 접근 발생 → 성능 저하
- **개선:** 쓰기 버퍼(Write Buffer)를 이용해 메모리 쓰기를 큐에 모아 처리

**② Write-Back (나중 쓰기)**

```
CPU 쓰기 → 캐시만 업데이트 (Dirty Bit = 1)
           메모리는 나중에...
           ↓ 캐시 라인 교체 시점에
         메인 메모리 업데이트
```

- **특징:** 캐시에만 먼저 쓰고, 캐시 라인이 교체될 때 비로소 메모리 동기화
- **Dirty Bit:** 캐시 라인이 메모리와 다름을 표시하는 1비트 플래그
- **장점:** 메모리 쓰기 횟수 최소화 → 높은 성능 (연속 쓰기 시 효과 극대화)
- **단점:** 일관성 유지 복잡, 장애 시 캐시 데이터 유실 가능

**③ Write-Miss 처리 전략**

| 전략 | 동작 | 주로 사용 |
|---|---|---|
| **Write-Allocate** | 메모리에서 해당 블록을 캐시에 먼저 적재한 후 쓰기 | Write-Back과 함께 |
| **No-Write-Allocate** | 캐시에 적재 없이 메모리에 직접 쓰기 | Write-Through와 함께 |

#### 캐시 일관성(Cache Coherence)

**문제 발생 시나리오:**
```
메모리 주소 X = 100

코어 0 캐시: X = 100
코어 1 캐시: X = 100

→ 코어 0이 X = 200으로 수정 (Write-Back, 메모리 아직 100)

코어 0 캐시: X = 200  ✓
코어 1 캐시: X = 100  ✗ (Stale Data!)
메인 메모리: X = 100  ✗
```

이 문제를 해결하는 프로토콜이 캐시 일관성 프로토콜입니다.

**① 스누핑(Bus Snooping / Snoopy Protocol)**

```
코어 0  코어 1  코어 2
  │       │       │
  └───────┴───────┘
          │
     공유 버스 (Shared Bus)
          │
     메인 메모리

각 코어의 캐시 컨트롤러가 버스를 엿들으며(Snooping) 
다른 코어의 쓰기를 감지하고 자신의 캐시를 무효화하거나 업데이트
```

**MESI 프로토콜 (가장 대표적인 스누핑 프로토콜):**

| 상태 | 의미 |
|---|---|
| **M (Modified)** | 수정됨, 메모리와 다름, 이 코어만 보유 |
| **E (Exclusive)** | 메모리와 동일, 이 코어만 보유 |
| **S (Shared)** | 메모리와 동일, 여러 코어가 공유 |
| **I (Invalid)** | 무효, 사용 불가 |

MESI 상태 전이:
```
처음: 코어 0이 X 읽기 → E 상태 (독점)
코어 1도 X 읽기 → 코어 0은 S, 코어 1도 S (공유)
코어 0이 X 쓰기 → 코어 0은 M, 코어 1은 I (무효화)
코어 1이 X 읽기 → 코어 0이 M 데이터를 메모리/코어 1에 제공
                   → 코어 0은 S, 코어 1은 S
```

- **장점:** 간단, 저지연 (버스 직접 감시)
- **단점:** 버스 트래픽이 코어 수에 비례하여 증가 → 대규모 멀티 프로세서에 부적합

**② 디렉토리 기반(Directory-Based) 프로토콜**

```
코어 0  코어 1  코어 N
  │       │       │
  └───────┴───────┘
          │ 점대점 연결 (NoC, Crossbar)
   ┌──────▼──────┐
   │  디렉토리   │ ← 어느 코어가 어느 블록의 복사본을 가지고 있는지 추적
   └─────────────┘
          │
     메인 메모리
```

- 각 메모리 블록마다 **공유 상태 및 소유자 정보를 기록한 디렉토리** 유지
- 쓰기 발생 시 디렉토리가 해당 블록을 캐싱하는 모든 코어에 무효화 메시지 전송
- **장점:** 버스 불필요, 확장성 우수 (수십~수백 코어)
- **단점:** 디렉토리 자체의 메모리/접근 오버헤드

### 3. 핵심 키워드 (Must-Have)

- **Write-Through:** 쓰기 즉시 메모리 반영, 일관성 간단, 성능 저하
- **Write-Back:** Dirty Bit 사용, 교체 시 메모리 반영, 고성능
- **캐시 일관성(Cache Coherence):** 멀티코어에서 모든 캐시가 동일 값 보장
- **스누핑(Snooping):** 공유 버스 감시 기반, 소규모 시스템에 적합
- **MESI 프로토콜:** Modified/Exclusive/Shared/Invalid 4개 상태
- **디렉토리 프로토콜:** 중앙 디렉토리 관리, 대규모 확장 가능
- **False Sharing:** 다른 데이터지만 같은 캐시 라인을 공유하여 불필요한 무효화 발생

### 4. 맥락과 비교

| 구분 | 스누핑 프로토콜 | 디렉토리 프로토콜 |
|---|---|---|
| **통신 방식** | 공유 버스 브로드캐스트 | 점대점 메시지 |
| **확장성** | 제한적 (버스 대역폭 병목) | 우수 (수백 코어 가능) |
| **지연** | 낮음 | 높음 (디렉토리 조회) |
| **구현 복잡도** | 단순 | 복잡 |
| **적용 사례** | Intel 소켓 내 코어 간 | AMD NUMA, 서버 다소켓 |

**Write Policy 비교:**

| 구분 | Write-Through | Write-Back |
|---|---|---|
| **메모리 갱신 시점** | 쓰기 즉시 | 캐시 라인 교체 시 |
| **성능** | 낮음 | 높음 |
| **일관성 유지** | 쉬움 | 복잡 (Dirty Bit) |
| **버스 대역폭** | 많이 사용 | 절약 |

### 5. 실무 제언

**멀티스레드 프로그래밍에서 False Sharing 문제**

- **챌린지:** 서로 다른 스레드가 서로 다른 변수를 수정하더라도, 그 변수들이 **같은 캐시 라인(64바이트)** 안에 위치하면 MESI 프로토콜의 무효화가 불필요하게 발생합니다. 이를 **False Sharing(거짓 공유)**이라 하며, 멀티스레드 코드에서 심각한 성능 저하 원인이 됩니다.

  ```c
  // False Sharing 발생 예시
  struct Counter {
      int core0_count;  // 동일 캐시 라인에 존재
      int core1_count;  // 동일 캐시 라인에 존재
  } counter;
  // 코어 0이 core0_count 수정 → 코어 1의 캐시 라인 전체 무효화
  // 코어 1이 core1_count 수정 → 코어 0의 캐시 라인 전체 무효화
  // 두 코어가 계속 무효화-재적재 반복 → 성능 급락
  ```

- **제언:** `alignas(64)` 또는 패딩(Padding)을 사용하여 각 스레드의 데이터를 서로 다른 캐시 라인에 배치하면 False Sharing을 제거할 수 있습니다. Intel VTune의 "Memory Access" 분석이나 Linux perf의 `cache-references/cache-misses` 이벤트로 탐지 가능합니다.
