## 해설

### 출제 배경 및 의도

프로세스(Process)는 운영체제 이론의 출발점으로, 다중 프로그래밍 환경을 가능하게 하는 핵심 추상화 단위입니다. 1960년대 IBM OS/360에서 개념이 정립된 이후, 현대 클라우드·컨테이너 환경에 이르기까지 변함없이 핵심 개념으로 자리하고 있습니다.

채점관은 단순 정의 암기가 아닌, **상태 전이 다이어그램의 각 조건**, **PCB의 구성과 컨텍스트 스위치 역할**, **fork-exec-wait-exit 시스템 콜 흐름**을 논리적으로 연결하여 설명하는지를 평가합니다. 스레드와의 비교, 컨테이너 환경 적용까지 언급하면 심화 역량으로 인정받습니다.

### 1. 프로세스(Process), 개요

정 의  • 프로세스를 디스크의 정적 프로그램이 메모리에 적재되어 실행 중인 동적 인스턴스로 정의함
       - OS는 PCB(Process Control Block)를 통해 모든 프로세스 상태를 추적·관리함

```
프로세스 상태 전이 다이어그램

         ┌─────────────────────────────────────┐
         │           생성 (New)                │
         └──────────────────┬──────────────────┘
                            │ 승인(Admitted)
                            ▼
         ┌─────────────────────────────────────┐
         │           준비 (Ready)              │◀──────────────┐
         └──────────────────┬──────────────────┘               │
                            │ 스케줄러 선택(Dispatch)          │
              인터럽트 ┌─────▼───────────────┐ 타임아웃        │
                      │    실행 (Running)    │────────────────┘
                      └─────────────────────┘
                                │ I/O 요청/이벤트 대기
                                ▼
         ┌─────────────────────────────────────┐
         │         대기 (Waiting)              │
         └──────────────────┬──────────────────┘
                            │ I/O 완료/이벤트 발생
                            ▼ (Ready로 복귀)
         ┌─────────────────────────────────────┐
         │          종료 (Terminated)          │
         └─────────────────────────────────────┘
```

- 5가지 상태(New/Ready/Running/Waiting/Terminated)와 전이 조건을 정확히 이해하는 것이 핵심임

### 2. 프로세스 상태·PCB·생성 및 종료

1) 프로세스 상태 전이 조건

| 구분 | 전이 | 조건 |
|------|------|------|
| 진입 | New → Ready | OS가 프로세스를 Ready Queue에 수용(Admitted) |
| 활성화 | Ready → Running | 스케줄러(Dispatcher)가 CPU 할당 |
| 선점 | Running → Ready | 타임 퀀텀 만료 또는 더 높은 우선순위 프로세스 도착 |
| 블록 | Running → Waiting | I/O 요청, 세마포어 wait(), 이벤트 대기 |
| 재활성 | Waiting → Ready | I/O 완료, 이벤트 발생(인터럽트) |
| 종료 | Running → Terminated | exit() 호출 또는 비정상 종료 |

- 일부 OS는 메모리 부족 시 Suspended Ready / Suspended Waiting 상태를 추가하여 스왑 아웃을 지원함

2) 프로세스 제어 블록(PCB, Process Control Block)

| 구분 | 항목 | 설명 |
|------|------|------|
| 식별 | PID | 프로세스 고유 번호 |
| 상태 | 프로세스 상태 | New/Ready/Running/Waiting/Terminated |
| CPU | 프로그램 카운터(PC) | 다음 실행 명령어 주소 |
| CPU | 레지스터 값 | 컨텍스트 스위치 시 저장·복원 대상 |
| 스케줄링 | 우선순위, 큐 포인터 | 스케줄러 판단 기준 |
| 메모리 | 페이지/세그먼트 테이블 | 가상 주소 공간 매핑 정보 |
| 회계 | CPU 사용 시간 | 자원 사용량 추적 |
| I/O | 열린 파일 디스크립터 | 파일·장치 목록 |

- PCB가 없으면 컨텍스트 스위치가 불가능하며 다중 프로그래밍 자체가 성립하지 않음

3) 프로세스 생성 및 종료

| 구분 | 시스템 콜 | 역할 |
|------|-----------|------|
| 생성 | `fork()` | 부모를 복제하여 자식 생성(Copy-on-Write) |
| 교체 | `exec()` | 현재 프로세스 메모리를 새 프로그램으로 교체(PID 유지) |
| 대기 | `wait()` | 자식 종료 대기 및 Exit Code 수집 |
| 종료 | `exit()` | 프로세스 종료, 종료 상태를 부모에 전달 |

- 부모가 wait()를 호출하지 않으면 자식 PCB가 커널에 잔류하는 **좀비 프로세스**가 발생하고, 부모가 먼저 종료되면 자식은 init(PID=1)에 입양되는 **고아 프로세스**가 됨

### 3. 프로세스 vs 스레드 비교 및 트렌드

| 구분 | 프로세스 | 스레드 |
|------|----------|--------|
| 메모리 공간 | 독립적 가상 주소 공간 | 프로세스 내 공유(Heap, Code, Data) |
| 생성 비용 | 높음(메모리 복사) | 낮음 |
| 통신 | IPC(파이프, 소켓, 공유 메모리) | 공유 메모리(동기화 필요) |
| 독립성 | 높음(격리, 장애 격리) | 낮음(스레드 오류 → 전체 프로세스 영향) |
| 컨텍스트 스위치 | 느림(주소 공간 전환, TLB 플러시) | 빠름(레지스터만 교체) |
| 대표 사례 | Docker 컨테이너, 다중 프로세스 웹 서버 | Java 스레드 풀, Go 고루틴 |

- 컨테이너(Docker)는 Linux 네임스페이스를 이용해 PID 격리를 구현하므로 프로세스 개념의 현대적 확장으로 볼 수 있음.  "끝"

### 실무 제언

**컨테이너 환경에서의 좀비 프로세스 방지**
- **챌린지**: 도커 컨테이너에서 PID 1이 init 역할을 수행하지 않으면, 자식 프로세스가 exit()해도 wait()를 호출하는 프로세스가 없어 좀비 프로세스가 누적되고 PID 자원이 고갈됨
- **제언**: `docker run --init` 옵션이나 Dockerfile에서 `ENTRYPOINT ["/sbin/tini", "--", ...]`를 설정하여 tini/dumb-init을 PID 1로 실행하면 좀비 프로세스를 자동 수집함

**다중 프로세스 vs 다중 스레드 아키텍처 선택**
- **챌린지**: 웹 서버(Nginx, Apache) 설계 시 Worker를 프로세스로 할지 스레드로 할지 선택이 성능·안정성에 직결됨
- **제언**: 장애 격리가 중요하면 다중 프로세스(Nginx), 메모리 공유 효율이 중요하면 다중 스레드(Tomcat) 모델을 채택하되, Go/Node처럼 이벤트 루프 기반 단일 스레드도 고려함

**컨테이너 오케스트레이션에서의 프로세스 트리 관리**
- **챌린지**: Kubernetes Pod 내에서 사이드카 컨테이너가 메인 컨테이너의 종료를 감지하지 못해 고아 프로세스가 발생하는 경우가 있음
- **제언**: Kubernetes 1.28+ pidNamespace 공유 설정 또는 preStop 훅을 활용하여 프로세스 생명주기를 명시적으로 관리함
