## 해설

> **실행 중인 프로그램의 살아있는 인스턴스, 프로세스의 모든 것**  
> 상태 전이, PCB의 구성, fork/exec/exit 시스템 콜 흐름

프로그램(Program)은 디스크에 저장된 정적인 실행 파일이지만, **프로세스(Process)**는 그 프로그램이 메모리에 적재되어 실행 중인 동적인 인스턴스입니다. OS는 모든 프로세스의 상태를 추적하고 관리하여 다중 프로그래밍 환경을 지원합니다.

### 1. 출제 배경 및 의도

프로세스는 OS 도메인의 가장 기본적인 개념입니다. 채점관은 **5가지 상태와 상태 전이 조건**, **PCB의 구성 요소와 역할**, **fork-exec-wait-exit 시스템 콜 흐름**을 정확히 이해하는지를 봅니다. 스레드와의 비교를 추가하면 심화 역량을 어필할 수 있습니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 프로세스 상태 다이어그램

```
              ┌─────────────────────────────────────┐
              │           생성 (New)                │
              └──────────────────┬──────────────────┘
                                 │ 승인(Admitted)
                                 ▼
              ┌─────────────────────────────────────┐
              │           준비 (Ready)              │◀──────────────┐
              └──────────────────┬──────────────────┘               │
                                 │ 스케줄러 선택                    │
                   ┌─────────────▼──────────────┐                  │
             인터럽트│      실행 (Running)        │타임아웃          │
                   └─────────────┬──────────────┘                  │
                         │       │ 입출력 요청/이벤트 대기           │
                         │       ▼                                   │
                         │  ┌─────────────────────────────────┐     │
                         │  │         대기 (Waiting)          │──── ┘
                         │  └─────────────────────────────────┘
                         │           입출력 완료/이벤트 발생
                         ▼
              ┌─────────────────────────────────────┐
              │          종료 (Terminated)          │
              └─────────────────────────────────────┘
```

**상태 전이 조건:**

| 전이 | 조건 |
|---|---|
| New → Ready | OS가 프로세스를 Ready Queue에 수용 |
| Ready → Running | 스케줄러(Dispatcher)가 CPU 할당 |
| Running → Ready | 타임 퀀텀 만료(선점) 또는 더 높은 우선순위 프로세스 도착 |
| Running → Waiting | I/O 요청, 세마포어 대기, 이벤트 대기 |
| Waiting → Ready | I/O 완료, 이벤트 발생 |
| Running → Terminated | 프로세스 정상/비정상 종료 |

**추가 상태 - Suspended(중지):**

일부 OS는 메모리 부족 시 프로세스를 디스크로 스왑 아웃하는 **Suspended Ready** / **Suspended Wait** 상태를 추가합니다.

#### 나. 프로세스 제어 블록 (PCB, Process Control Block)

**PCB는 OS가 프로세스를 관리하기 위한 모든 정보를 담은 데이터 구조**입니다. 프로세스당 하나의 PCB가 존재하며 커널 메모리에 저장됩니다.

```
PCB 구조
┌─────────────────────────────┐
│  프로세스 식별자 (PID)       │ ← 고유 번호
├─────────────────────────────┤
│  프로세스 상태               │ ← New/Ready/Running/Waiting/Term
├─────────────────────────────┤
│  프로그램 카운터 (PC)        │ ← 다음 실행할 명령어 주소
├─────────────────────────────┤
│  CPU 레지스터 값             │ ← 컨텍스트 스위치 시 저장/복원
├─────────────────────────────┤
│  CPU 스케줄링 정보           │ ← 우선순위, 큐 포인터
├─────────────────────────────┤
│  메모리 관리 정보            │ ← 페이지 테이블, 세그먼트 테이블
├─────────────────────────────┤
│  회계 정보                  │ ← CPU 사용 시간, 시스템 콜 횟수
├─────────────────────────────┤
│  I/O 상태 정보              │ ← 열린 파일 목록, I/O 장치 목록
└─────────────────────────────┘
```

**PCB의 역할:**  
컨텍스트 스위치 시 현재 프로세스의 CPU 상태(레지스터 값, PC)를 PCB에 저장하고, 다음 프로세스의 PCB에서 상태를 복원합니다. PCB가 없으면 컨텍스트 스위치는 불가능합니다.

#### 다. 프로세스 생성 및 종료

**Unix/Linux에서의 프로세스 생성: fork-exec 모델**

```
부모 프로세스 (PID=1000)
    │
    │ fork()
    ├──────────────────────────────────────────────▶
    │                                              자식 프로세스 (PID=1001)
    │  fork() 반환값 = 1001 (자식 PID)            fork() 반환값 = 0
    │                                              │
    │                                              exec("프로그램")
    │                                              새 프로그램으로 교체
    │  wait(1001)  (자식 완료 대기)                │
    │◀─────────────────────────────────────────────
    │                                              exit(0) (종료)
    │  자식의 종료 상태 수집
    │
    (계속 실행)
```

**주요 시스템 콜:**

| 시스템 콜 | 역할 |
|---|---|
| `fork()` | 부모 프로세스를 복제하여 자식 프로세스 생성. 자식은 부모의 복사본 |
| `exec()` | 현재 프로세스의 메모리 공간을 새 프로그램으로 교체 (PID 유지) |
| `wait()` | 자식 프로세스 종료 대기 및 종료 상태(Exit Code) 수집 |
| `exit()` | 현재 프로세스 종료. 종료 상태를 부모에게 전달 |
| `getpid()` | 현재 프로세스의 PID 반환 |

**좀비 프로세스(Zombie Process):**  
자식이 exit()했지만 부모가 wait()를 호출하지 않아 PCB가 커널에 남아 있는 상태. PID 자원 낭비.

**고아 프로세스(Orphan Process):**  
부모 프로세스가 먼저 종료되어 init 프로세스(PID=1)에 입양되는 자식 프로세스.

#### 프로세스 vs 스레드 비교

| 구분 | 프로세스 | 스레드 |
|---|---|---|
| **메모리 공간** | 독립적 가상 주소 공간 | 프로세스 내 공유 (Heap, Code, Data) |
| **생성 비용** | 높음 (메모리 복사) | 낮음 |
| **통신** | IPC (복잡) | 공유 메모리 (간단, 동기화 필요) |
| **독립성** | 높음 (격리) | 낮음 (하나의 스레드 오류 → 전체 영향) |
| **컨텍스트 스위치** | 느림 (주소 공간 전환) | 빠름 |

### 3. 핵심 키워드 (Must-Have)

- **프로세스:** 실행 중인 프로그램의 인스턴스
- **5가지 상태:** New, Ready, Running, Waiting, Terminated
- **PCB(Process Control Block):** 프로세스 메타데이터 저장소
- **컨텍스트 스위치:** PCB 저장/복원으로 프로세스 교체
- **fork():** 부모 복제로 자식 생성
- **exec():** 새 프로그램으로 메모리 교체
- **wait():** 자식 종료 대기
- **exit():** 프로세스 종료
- **좀비 프로세스:** exit됐으나 wait 안 된 자식
- **고아 프로세스:** 부모 먼저 종료된 자식 → init이 입양

### 4. 맥락과 비교

**프로세스 계층 구조 (Process Tree):**

```
systemd (PID=1)
├── sshd (PID=100)
│   └── bash (PID=200)
│       └── vim (PID=300)
├── cron (PID=101)
└── dockerd (PID=102)
    └── container 프로세스 (PID=400)
```

모든 프로세스는 init/systemd를 루트로 하는 트리 구조를 형성합니다.

### 5. 실무 제언

**컨테이너 환경에서의 프로세스 격리**

- **챌린지:** 도커 컨테이너는 각 컨테이너가 독립적인 PID 네임스페이스를 가지므로, 컨테이너 내부에서 PID 1은 컨테이너의 메인 프로세스를 가리킵니다. 이 컨테이너의 PID 1이 올바른 좀비 프로세스 처리(wait 호출)를 하지 않으면, 컨테이너 내에 좀비 프로세스가 쌓이는 문제가 발생합니다.
- **제언:** 도커 공식 권장 사항은 init 시스템(tini, dumb-init)을 컨테이너의 PID 1로 실행하여 좀비 프로세스를 자동 수집하도록 하는 것입니다. `docker run --init` 옵션이나 Dockerfile의 `ENTRYPOINT ["/sbin/tini", "--", ...]` 설정으로 구현합니다. 전통적 OS 프로세스 관리 원리가 컨테이너 운영에도 직접 적용됨을 이해해야 합니다.
