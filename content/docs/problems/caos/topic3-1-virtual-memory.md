## 해설

> **물리 메모리의 한계를 뛰어넘는 소프트웨어의 마법**  
> 페이징과 세그멘테이션, 그리고 단편화라는 부작용과 그 해결책

물리 메모리(RAM)는 비싸고 유한합니다. 가상메모리는 보조기억장치(HDD/SSD)를 물리 메모리처럼 활용하여 실제보다 훨씬 큰 주소 공간을 프로세스에 제공하는 운영체제의 핵심 기법입니다. 현대 OS(Linux, Windows)는 모두 가상메모리를 기반으로 프로세스를 고립(Isolation)하고 메모리를 효율적으로 공유합니다.

### 1. 출제 배경 및 의도

가상메모리는 운영체제 도메인의 핵심 중의 핵심입니다. 채점관은 단순히 "페이징이란 고정 크기로 나누는 것이다"를 넘어, **페이징이 왜 내부 단편화를 일으키는지**, **세그멘테이션이 왜 외부 단편화를 일으키는지**, 그리고 **페이지드 세그멘테이션이 어떻게 두 방식의 장점을 결합하는지**를 논리적으로 설명할 수 있는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 가상메모리 관리기법

**가상메모리의 핵심 원리 — 요구 페이징(Demand Paging):**  
프로그램 전체를 메모리에 올리지 않고, 실제로 필요한(참조되는) 페이지만 그때그때 메모리에 적재하는 방식입니다. 처음 접근하는 페이지에서 **페이지 폴트(Page Fault)**가 발생하면 OS가 해당 페이지를 디스크에서 읽어 메모리에 올립니다.

**① 페이징(Paging) 기법**

```
가상 주소 공간              물리 메모리
┌────────────┐            ┌────────────┐
│  Page 0    │──────────▶│  Frame 3   │
├────────────┤            ├────────────┤
│  Page 1    │──────────▶│  Frame 7   │
├────────────┤            ├────────────┤
│  Page 2    │ (디스크)   │  Frame 1   │
├────────────┤            └────────────┘
│  Page 3    │──────────▶  (Page Table로 매핑)
└────────────┘
```

- 가상 주소 공간과 물리 메모리를 **동일한 고정 크기(Page/Frame)**로 분할
- **페이지 테이블(Page Table)**이 가상 페이지 번호(VPN)를 물리 프레임 번호(PFN)로 매핑
- 페이지 크기: 일반적으로 4KB (현대 시스템은 huge page로 2MB~1GB도 사용)

**② 세그멘테이션(Segmentation) 기법**

```
가상 주소 공간
┌─────────────────┐ ← 크기 가변
│   코드 세그먼트  │
├─────────────────┤
│   데이터 세그먼트│
├─────────────────┤
│   스택 세그먼트  │
└─────────────────┘
```

- 프로그램을 논리적 단위(코드, 데이터, 스택, 힙)로 **가변 크기** 분할
- **세그먼트 테이블(Segment Table)**이 세그먼트 번호를 Base 주소와 Limit으로 매핑
- 프로그래머의 논리적 관점과 일치하는 장점

**③ 페이지드 세그멘테이션(Paged Segmentation)**

세그멘테이션의 논리적 구조 + 페이징의 고정 크기 관리를 결합:
- 세그먼트를 다시 페이지 단위로 분할
- 세그먼트 테이블 → 페이지 테이블 → 물리 프레임의 2단계 변환
- Intel x86 아키텍처의 세그멘테이션+페이징 혼합 구조가 이 방식

#### 나. 가상메모리 단편화와 해결방안

**① 내부 단편화(Internal Fragmentation) — 페이징의 문제**

```
프로세스 크기: 13KB
페이지 크기: 4KB

Page 0: 4KB (완전 사용)
Page 1: 4KB (완전 사용)
Page 2: 4KB 할당 → 실제 사용 5KB → 낭비: 3KB ← 내부 단편화!
```

- 프로세스가 마지막 페이지를 완전히 채우지 못해 발생
- **해결방안:** 페이지 크기를 작게 설정 (단, 페이지 테이블 크기 증가 부작용)

**② 외부 단편화(External Fragmentation) — 세그멘테이션의 문제**

```
메모리 상태:
│ 사용 중(10KB) │ 빈공간(5KB) │ 사용 중(8KB) │ 빈공간(6KB) │
총 빈 공간 = 11KB이지만 연속된 10KB 블록 요청은 실패!
```

- 빈 공간이 불연속적으로 흩어져 있어 큰 세그먼트를 배치할 수 없는 문제
- **해결방안:**
  1. **압축(Compaction):** 모든 프로세스를 한쪽으로 몰아 연속된 빈 공간 생성 (오버헤드 큼)
  2. **페이징 도입:** 연속 메모리 불필요 → 외부 단편화 원천 제거
  3. **버디 시스템(Buddy System):** 2의 거듭제곱 단위로 할당하여 병합 용이

**③ 단편화 유형 비교 요약**

| 구분 | 내부 단편화 | 외부 단편화 |
|---|---|---|
| **발생 원인** | 고정 크기 할당 (페이징) | 가변 크기 할당 (세그멘테이션) |
| **낭비 위치** | 할당된 블록 내부 | 할당된 블록들 사이 |
| **해결책** | 페이지 크기 축소 | 압축, 페이징 혼합 |

### 3. 핵심 키워드 (Must-Have)

- **가상 주소 공간 / 물리 주소 공간**
- **요구 페이징(Demand Paging), 페이지 폴트(Page Fault)**
- **페이징:** 고정 크기(Page/Frame), 페이지 테이블, 내부 단편화
- **세그멘테이션:** 가변 크기, 세그먼트 테이블(Base+Limit), 외부 단편화
- **페이지드 세그멘테이션:** 두 기법 결합, 2단계 변환
- **내부 단편화:** 페이지 내부 낭비
- **외부 단편화:** 빈 공간이 불연속, 압축으로 해결
- **스왑 영역(Swap Space):** 디스크의 가상메모리 저장 공간

### 4. 맥락과 비교

| 비교 항목 | 페이징 | 세그멘테이션 | 페이지드 세그멘테이션 |
|---|---|---|---|
| **분할 단위** | 고정 크기 | 가변 크기 | 세그먼트→페이지 |
| **내부 단편화** | 발생 | 없음 | 소량 발생 |
| **외부 단편화** | 없음 | 발생 | 없음 |
| **보호/공유** | 페이지 단위 | 세그먼트 단위(논리적) | 세그먼트 단위 |
| **구현 복잡도** | 낮음 | 중간 | 높음 |
| **대표 사례** | Linux (x86-64) | Multics | Intel x86 (Legacy) |

### 5. 실무 제언

**클라우드 환경에서의 메모리 과다 할당(Memory Overcommitment)**

- **챌린지:** 클라우드 하이퍼바이저(KVM, VMware)는 물리 메모리보다 많은 가상 메모리를 VM에 할당하는 메모리 오버커밋(Overcommit)을 사용합니다. 이는 가상메모리의 요구 페이징 원리를 VM 레벨로 확장한 것이지만, 갑작스러운 메모리 압박 시 OOM Killer가 프로세스를 강제 종료하는 장애를 일으킵니다.
- **제언:** 프로덕션 환경에서는 메모리 사용률 모니터링과 함께 적절한 overcommit_ratio 설정(`/proc/sys/vm/overcommit_ratio`)과 메모리 리밋(cgroup memory.limit_in_bytes) 설정이 필수입니다. 가상메모리 원리를 이해해야만 이 파라미터들의 의미를 올바르게 파악할 수 있습니다.
