## 해설

### 출제 배경 및 의도

물리 메모리(RAM)는 비싸고 유한하다. 가상메모리는 보조기억장치(HDD/SSD)를 물리 메모리처럼 활용하여 실제보다 훨씬 큰 주소 공간을 프로세스에 제공하는 운영체제의 핵심 기법이다. 현대 OS(Linux, Windows)는 모두 가상메모리를 기반으로 프로세스를 고립(Isolation)하고 메모리를 효율적으로 공유한다.

가상메모리는 운영체제 도메인의 핵심 중의 핵심이다. 채점관은 단순히 "페이징이란 고정 크기로 나누는 것이다"를 넘어, 페이징이 왜 내부 단편화를 일으키는지, 세그멘테이션이 왜 외부 단편화를 일으키는지, 그리고 페이지드 세그멘테이션이 두 방식의 장점을 어떻게 결합하는지를 논리적으로 설명할 수 있는지를 본다.

클라우드 환경에서는 하이퍼바이저가 물리 메모리 이상을 VM에 할당하는 메모리 오버커밋(Overcommit)을 통해 이 원리를 VM 레벨로 확장한다. 가상메모리의 원리를 이해해야만 이 고급 주제들을 올바르게 해석할 수 있다.

### 1. 가상메모리(Virtual Memory), 개요

정 의  • 요구 페이징(Demand Paging)을 기반으로 프로세스에게 물리 메모리 이상의 주소 공간을 제공하는 운영체제의 메모리 관리 기법
       - 실제로 필요한 페이지만 메모리에 적재하고, 나머지는 디스크(Swap Space)에 유지

```
┌──────────────────┐       ┌──────────────────┐
│  가상 주소 공간   │       │   물리 메모리      │
│  (프로세스 뷰)    │       │                  │
│  ┌────────────┐  │       │  ┌────────────┐  │
│  │  Page 0    │──┼──────▶│  │  Frame 3   │  │
│  ├────────────┤  │       │  ├────────────┤  │
│  │  Page 1    │──┼──────▶│  │  Frame 7   │  │
│  ├────────────┤  │       │  ├────────────┤  │
│  │  Page 2    │  │(디스크)│  │  Frame 1   │  │
│  ├────────────┤  │       │  └────────────┘  │
│  │  Page 3    │──┼──────▶   Page Table 매핑  │
│  └────────────┘  │       └──────────────────┘
└──────────────────┘               ▲
                                   │ Page Fault 시
                              ┌────┴─────┐
                              │  디스크   │
                              │ (Swap)   │
                              └──────────┘
```

- 페이지 테이블이 가상 페이지 번호(VPN)를 물리 프레임 번호(PFN)로 매핑하여 투명한 주소 공간을 제공한다.

### 2. 가상메모리 관리기법

1) 페이징(Paging)

| 구분 | 항목 | 설명 |
|------|------|------|
| 분할 방식 | 고정 크기 | 가상 공간과 물리 메모리를 동일한 크기(Page/Frame)로 분할 |
| 매핑 구조 | 페이지 테이블 | VPN → PFN 변환 (Valid 비트, Dirty 비트, Protection 비트 포함) |
| 페이지 크기 | 일반적으로 4KB | 현대 시스템은 Huge Page(2MB~1GB)도 지원 |
| 단편화 | 내부 단편화 | 마지막 페이지를 완전히 채우지 못해 발생 |
| 특징 | 연속 메모리 불필요 | 물리 메모리 어디에나 분산 배치 가능 |

- 페이징은 외부 단편화를 원천 제거하지만 내부 단편화와 페이지 테이블 메모리 오버헤드를 수반한다.

2) 세그멘테이션(Segmentation)

| 구분 | 항목 | 설명 |
|------|------|------|
| 분할 방식 | 가변 크기 | 프로그램을 논리적 단위(코드·데이터·스택·힙)로 분할 |
| 매핑 구조 | 세그먼트 테이블 | 세그먼트 번호 → Base 주소 + Limit으로 매핑 |
| 장점 | 논리적 보호/공유 | 프로그래머의 논리적 관점과 일치 |
| 단편화 | 외부 단편화 | 빈 공간이 불연속적으로 흩어져 발생 |

- 세그멘테이션은 논리적 구조에 부합하지만 외부 단편화로 메모리 관리가 복잡해진다.

3) 페이지드 세그멘테이션(Paged Segmentation)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 두 기법 결합 | 세그먼트를 다시 페이지 단위로 분할하여 관리 |
| 변환 단계 | 2단계 변환 | 세그먼트 테이블 → 페이지 테이블 → 물리 프레임 |
| 외부 단편화 | 없음 | 페이지 단위 물리 할당으로 제거 |
| 내부 단편화 | 소량 발생 | 마지막 페이지 내부 낭비 |
| 대표 사례 | Intel x86(Legacy) | 세그멘테이션+페이징 혼합 구조 |

- Intel x86 아키텍처의 세그멘테이션+페이징 혼합 구조가 이 방식을 대표하며, 두 기법의 장점을 결합한다.

### 3. 단편화 유형, 원인, 해결방안 비교

1) 내부 단편화 vs 외부 단편화

```
내부 단편화 (페이징)
  프로세스 크기: 13KB, 페이지 크기: 4KB
  Page 0: 4KB (완전 사용)
  Page 1: 4KB (완전 사용)
  Page 2: 4KB 할당 → 실제 1KB만 사용 → 3KB 낭비 ← 내부 단편화!

외부 단편화 (세그멘테이션)
  │ 사용 중(10KB) │ 빈공간(5KB) │ 사용 중(8KB) │ 빈공간(6KB) │
  총 빈 공간 = 11KB이지만 연속된 10KB 요청은 실패! ← 외부 단편화!
```

2) 해결방안 비교

| 구분 | 내부 단편화 | 외부 단편화 |
|------|-------------|-------------|
| 발생 원인 | 고정 크기 할당(페이징) | 가변 크기 할당(세그멘테이션) |
| 낭비 위치 | 할당된 블록 내부 | 할당된 블록들 사이 |
| 해결책 ① | 페이지 크기 축소 | 압축(Compaction): 프로세스 이동으로 연속 빈 공간 생성 |
| 해결책 ② | Huge Page 선택적 적용 | 페이징 도입: 연속 메모리 불필요화 |
| 해결책 ③ | - | 버디 시스템: 2의 거듭제곱 단위 할당으로 병합 용이 |

- 페이징과 세그멘테이션을 결합한 페이지드 세그멘테이션이 두 유형의 단편화를 동시에 최소화하는 현실적 해법이다.  "끝"

### 실무 제언

**클라우드 환경 메모리 오버커밋**
- **챌린지**: 클라우드 하이퍼바이저(KVM, VMware)는 물리 메모리보다 많은 가상 메모리를 VM에 할당하는 메모리 오버커밋을 사용한다. 갑작스러운 메모리 압박 시 OOM Killer가 프로세스를 강제 종료하는 장애가 발생한다.
- **제언**: 프로덕션 환경에서는 메모리 사용률 모니터링과 함께 적절한 overcommit_ratio 설정(`/proc/sys/vm/overcommit_ratio`)과 cgroup memory.limit_in_bytes 설정이 필수이다.

**대용량 메모리 서버에서 Huge Page 적용**
- **챌린지**: 수십 GB RAM을 사용하는 데이터베이스(Oracle, PostgreSQL)에서 4KB 페이지를 사용하면 페이지 테이블 엔트리 수가 수천만 개에 달해 TLB 미스율이 급증한다.
- **제언**: Huge Page(2MB, Linux Transparent Huge Pages 또는 명시적 HugeTLBfs)를 적용하면 TLB 커버리지가 512배 향상되어 대규모 메모리 워크로드의 성능을 수십 % 개선할 수 있다.

**컨테이너 환경 메모리 격리**
- **챌린지**: Docker 컨테이너는 가상메모리 공간을 호스트 OS와 공유하므로 메모리 격리가 VM 대비 취약하다. 잘못된 컨테이너가 호스트 메모리를 고갈시킬 수 있다.
- **제언**: cgroup v2의 memory.max와 memory.swap.max를 설정하여 컨테이너별 물리 메모리와 스왑 사용량을 명시적으로 제한하고, OOM 발생 시 호스트가 아닌 해당 컨테이너만 종료되도록 설계해야 한다.
