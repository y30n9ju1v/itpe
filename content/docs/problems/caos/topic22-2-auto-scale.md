## 해설

### 출제 배경 및 의도

클라우드 컴퓨팅의 핵심 가치는 수요에 따라 자원을 자동으로 늘리고 줄이는 탄력성(Elasticity)입니다. 고정 용량 온프레미스 서버는 피크 트래픽에 맞춰 과잉 프로비저닝하거나, 부족 프로비저닝으로 서비스 장애를 감수해야 했습니다. 클라우드 Auto Scaling은 이 문제를 근본적으로 해결하여 비용 최적화와 가용성을 동시에 달성합니다.

121회 관리 문제로 Scale Up/Out 아키텍처 비교와 스케줄/부하 기반 Auto Scale 비교가 요구됩니다. 채점관은 각 확장 방식의 구조적 차이, 적합 사용 시나리오, 자동화 트리거 방식의 장단점을 체계적으로 제시하는지를 평가합니다.

### 1. Auto Scaling, 개요

정 의  • 트래픽 변화에 따라 컴퓨팅 자원을 자동으로 증감하여 탄력성(Elasticity)을 실현하는 클라우드 메커니즘
       - 수직 확장(Scale Up: 사양 향상)과 수평 확장(Scale Out: 서버 수 증가)의 두 방향으로 구분

```
Scale Up (수직 확장):          Scale Out (수평 확장):
┌──────────────┐              ┌──────────────────────────────────┐
│  서버 1개    │              │         Load Balancer            │
│  4 vCPU      │ ──▶  ┌────────────────────────────────────┐   │
│  16 GB RAM   │      │  서버1   │  서버2   │  서버3       │   │
└──────────────┘      │  4vCPU   │  4vCPU   │  4vCPU       │   │
       ↓              └────────────────────────────────────┘   │
┌──────────────────┐                                            │
│  더 큰 서버 1개  │   동일 서버를 여러 대로 수평 확장          │
│  16 vCPU         │                                            │
│  64 GB RAM       │                                            │
└──────────────────┘
```

- Scale Up은 단순하나 중단 발생·물리 한계 존재, Scale Out은 복잡하나 무중단·무한 확장 가능

### 2. Scale Up vs Scale Out 아키텍처 비교

1) Scale Up과 Scale Out 특성 비교

| 구분 | 항목 | Scale Up (수직 확장) | Scale Out (수평 확장) |
|------|------|------|------|
| 방법 | 확장 방식 | 단일 서버 CPU·메모리·스토리지 향상 | 동일 서버를 여러 대 추가 |
| | 서비스 중단 | 있음 (인스턴스 재시작) | 없음 (새 서버 준비 후 분배) |
| 성능 | 확장 한계 | 물리적 최대 사양 | 이론적 무한 |
| | 복잡도 | 낮음 | 높음 (로드밸런서, 분산 상태 관리) |
| 비용 | 효율 | 낮음 (고사양 서버 비쌈) | 높음 (표준 서버 다수) |
| | SPOF | 있음 (단일 장애점) | 없음 (개별 서버 장애 허용) |
| 적합 | 워크로드 | DB, 레거시 싱글 서버 앱 | 웹 서버, MSA, Stateless 앱 |

- Scale Out의 전제 조건: 애플리케이션이 **Stateless** 설계여야 효과적 (세션은 Redis 등 외부 저장소 사용)

2) 스케줄 기반 vs 부하 기반 Auto Scale

| 구분 | 항목 | 스케줄 기반 | 부하 기반 (Metric-based) |
|------|------|------|------|
| 트리거 | 방식 | Cron 스케줄로 사전 설정 | 실시간 지표 임계치 초과 시 |
| | 지표 | 없음 (시간 기반) | CPU·메모리·응답시간·큐 깊이 |
| 반응성 | 속도 | 낮음 (사전 설정) | 중간 (지표 수집→판단→준비 수분) |
| 예측성 | 정확도 | 높음 (규칙적 패턴) | 낮음 (급작스러운 트래픽 대응 지연) |
| 단점 | 문제점 | 예상치 못한 급증 대응 불가 | 플래핑(Flapping), Cold Start |
| 적합 | 패턴 | 규칙적 트래픽 (업무 시간대) | 불규칙 트래픽 |

```
스케줄 기반 설정 예:            부하 기반 트리거:
09:00 → 최소 10개               CPU > 70% (5분 평균) → +1 서버
18:00 → 최소  5개               메모리 > 80%         → +2 서버
23:00 → 최소  2개               응답시간 > 2초       → +3 서버
```

### 3. 예측적 Auto Scale과 클라우드 구현 비교

```
AWS Auto Scaling Group 구조:

Launch Template (인스턴스 설정)
       +
Auto Scaling Group (최소/최대/희망 인스턴스 수)
       +
Scaling Policy (스케줄/Target Tracking/Step Scaling)
       +
CloudWatch Alarm (지표 임계치 모니터링)
```

| 구분 | 스케줄 기반 | 부하 기반 | 예측적(ML) |
|------|------|------|------|
| 반응성 | 낮음 (사전 설정) | 중간 (지표 반응) | 높음 (사전 대응) |
| 비용 효율 | 중간 | 높음 | 최고 |
| 복잡도 | 낮음 | 중간 | 높음 |
| 적합 패턴 | 규칙적 트래픽 | 불규칙 트래픽 | 혼합 트래픽 |
| AWS 서비스 | Scheduled Scaling | Target Tracking | Predictive Scaling |

예측적 Auto Scale은 과거 트래픽 패턴을 ML로 분석하여 부하 발생 전에 미리 스케일 아웃하므로 Cold Start 지연 없이 최적 성능을 보장하며, 스케줄·부하 기반을 조합한 하이브리드 전략이 실무 표준.  "끝"

### 실무 제언

**쿠버네티스 HPA 고도화 설계**
- **챌린지**: HPA를 CPU 사용률 기준만으로 설정하면 큐 처리·IO 바운드 앱에서 스케일링이 부적절하게 동작하고, Cold Start로 신규 파드 준비 전 기존 파드에 과부하 발생
- **제언**: KEDA(Kubernetes Event-driven Autoscaler)로 Kafka 오프셋·RabbitMQ 큐 길이 등 비즈니스 지표 기반 스케일링 적용, VPA(Vertical Pod Autoscaler) 병행으로 파드 리소스 자동 조정

**Stateless 아키텍처 전환을 통한 Scale Out 최적화**
- **챌린지**: 세션 상태를 서버 메모리에 유지하는 Stateful 앱은 Scale Out 시 세션 불일치로 사용자 오류 발생
- **제언**: 세션을 Redis/Memcached 등 외부 분산 캐시로 이전하여 Stateless 전환, JWT 기반 무상태 인증으로 로드밸런서의 세션 스티키니스(Sticky Session) 의존도 제거

**Cooldown 기간과 플래핑 방지**
- **챌린지**: 부하 기반 Auto Scale에서 임계치 경계값 부근에서 지속적 Scale Out/In이 반복(플래핑)되어 인스턴스 비용 낭비와 불안정 운영 발생
- **제언**: Cooldown 기간(300초 권장)을 설정하여 스케일링 후 안정화 대기, Scale Out 임계치(70%)와 Scale In 임계치(40%)를 충분히 이격시켜 플래핑 방지
