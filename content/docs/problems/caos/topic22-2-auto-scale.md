## 문제

Auto Scale Up과 Auto Scale Out 아키텍처 비교, 스케줄 기반의 Auto Scale과 부하 기반의 Auto Scale 기능 비교

## 해설

> **클라우드 탄력성의 핵심: 수요에 맞춰 자동으로 규모를 조절하는 아키텍처**  
> 수직 확장(Scale Up)과 수평 확장(Scale Out)의 트레이드오프, 스케줄/부하 기반 자동화

클라우드 컴퓨팅의 핵심 가치 중 하나는 **탄력성(Elasticity)**—수요에 따라 자원을 자동으로 늘리고 줄이는 능력입니다. **Auto Scaling**은 이 탄력성을 자동으로 실현하는 클라우드 서비스입니다.

### 1. 출제 배경 및 의도

121회 관리 문제로 Scale Up/Out의 개념과 아키텍처 차이, 그리고 스케줄/부하 기반 Auto Scale을 비교해야 합니다. 채점관은 **각 확장 방식의 구조적 차이**, **적합한 사용 시나리오**, **자동화 트리거 방식의 장단점**을 체계적으로 제시하는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### Auto Scale Up (수직 확장, Vertical Scaling)

```
현재:               확장 후:
┌──────────────┐   ┌──────────────────────────┐
│  서버 1개    │   │  더 큰 서버 1개          │
│  4 vCPU      │──▶│  16 vCPU                 │
│  16 GB RAM   │   │  64 GB RAM               │
└──────────────┘   └──────────────────────────┘
              인스턴스 타입 변경 (중단 후 재시작)
```

**특징:**
- 단일 서버의 **CPU, 메모리, 스토리지 용량을 증가**
- 서버 중단 후 더 큰 인스턴스로 재시작 필요 (downtime 발생)
- 소프트웨어 변경 없이 적용 가능 (단순)
- **하드웨어 한계:** 단일 서버의 최대 사양에 제한됨

**장점:**
- 단일 서버로 일관성 있는 데이터 처리
- 분산 처리 복잡도 없음
- 레거시 애플리케이션에도 적용 가능

**단점:**
- 확장 중 서비스 중단 발생
- 물리적 한계로 무한 확장 불가
- 비용 효율 낮음 (고사양 서버 비쌈)
- 단일 장애점(SPOF) 위험

#### Auto Scale Out (수평 확장, Horizontal Scaling)

```
현재:               확장 후:
                    ┌─────────────────────────────────────────────────┐
                    │              Load Balancer                      │
                    └──────┬──────────────┬──────────────┬───────────┘
┌──────────┐               │              │              │
│  서버 1개 │  ──▶  ┌──────▼──┐  ┌───────▼─┐  ┌───────▼─┐
│  4 vCPU  │       │  서버 1  │  │  서버 2  │  │  서버 3  │
└──────────┘       │  4 vCPU  │  │  4 vCPU  │  │  4 vCPU  │
                   └──────────┘  └──────────┘  └──────────┘
                      동일한 서버를 여러 대로 늘림
```

**특징:**
- **동일한 서버를 여러 대 추가**하여 로드 밸런서로 트래픽 분산
- 무중단 확장 가능 (새 서버가 준비되면 트래픽 분배)
- **이론적으로 무한 확장** 가능 (클라우드에서)

**장점:**
- 무중단 확장
- 이론적 무한 확장성
- 개별 서버 장애에 대한 고가용성
- 비용 효율 우수 (표준 서버 사용)

**단점:**
- 분산 상태 관리 복잡 (세션, 데이터 동기화)
- 로드 밸런서 추가 필요
- 애플리케이션이 Stateless 설계여야 효과적
- 분산 트랜잭션 처리 어려움

#### Scale Up vs Scale Out 비교

| 비교 항목 | Scale Up (수직 확장) | Scale Out (수평 확장) |
|---|---|---|
| **확장 방법** | 단일 서버 사양 향상 | 서버 수 증가 |
| **서비스 중단** | 있음 (재시작 필요) | 없음 (무중단) |
| **확장 한계** | 물리적 최대 사양 | 이론적 무한 |
| **복잡도** | 낮음 | 높음 (로드밸런서, 분산) |
| **비용** | 높음 (고사양 서버) | 낮음 (표준 서버 다수) |
| **적합 워크로드** | DB, 레거시 싱글 서버 앱 | 웹 서버, MSA, Stateless 앱 |

#### 스케줄 기반 Auto Scale vs 부하 기반 Auto Scale

**① 스케줄 기반(Schedule-based) Auto Scale**

```
시간대별 서버 수:
    ┌─────────────────────────────────────────────────────┐
    │              예측 가능한 트래픽 패턴               │
10  │                  ████████████                       │
 8  │               ████████████████                     │
 6  │         ████████████████████████                   │
 4  │    ████████████████████████████████████            │
 2  │████████████████████████████████████████████        │
    └────────────────────────────────────────────────────┘
    00:00  06:00  09:00  12:00  18:00  21:00  00:00
    낮음   증가    피크   보통   피크   감소    낮음

크론 스케줄 설정:
  09:00 → 최소 10개 인스턴스
  18:00 → 최소 5개 인스턴스
  23:00 → 최소 2개 인스턴스
```

**특징:** 예측 가능한 트래픽 패턴에 대해 미리 설정된 시간표에 따라 서버 수 조절

**장점:**
- 예측 가능한 스케일링 → 준비 시간 확보
- 트래픽 도달 전 미리 확장 가능
- 간단한 구현

**단점:**
- 예상치 못한 트래픽 급증에 대응 불가
- 과소/과대 프로비저닝 위험
- 유연성 부족

**② 부하 기반(Metric-based) Auto Scale**

```
CPU 사용률이 임계치 초과 시:
    CPU  70%  80%  90%  임계치(70%)
         │    │    │
         │    │    ├─▶ 스케일 아웃 트리거
         │    ├────┘
         ├────┘
    정상  경고  경보

지표 예시:
 - CPU 사용률 > 70% (5분 평균) → +1 서버
 - 메모리 > 80% → +2 서버
 - HTTP 응답 시간 > 2초 → +3 서버
 - 큐 메시지 수 > 1000건 → +1 서버
```

**특징:** 실시간 시스템 지표를 모니터링하여 임계값 초과 시 자동 스케일링

**장점:**
- 실제 부하에 즉각 반응
- 비용 최적화 (필요할 때만 증가)
- 예상치 못한 트래픽 대응

**단점:**
- 지표 수집 → 임계치 판단 → 새 서버 준비 시간(수분) 동안 지연
- 스케일링 지표 설정의 어려움 (너무 민감하면 플래핑(Flapping) 발생)
- Cold Start 문제

**③ 예측적(Predictive) Auto Scale (AI 기반)**

과거 트래픽 패턴을 ML로 분석하여 부하 발생 전에 미리 스케일 아웃. AWS Auto Scaling의 Predictive Scaling이 대표적.

| 비교 항목 | 스케줄 기반 | 부하 기반 | 예측적 |
|---|---|---|---|
| **반응성** | 낮음 (사전 설정) | 중간 (지표 반응) | 높음 (사전 대응) |
| **비용 효율** | 중간 | 높음 | 최고 |
| **복잡도** | 낮음 | 중간 | 높음 |
| **적합 패턴** | 규칙적 트래픽 | 불규칙 트래픽 | 혼합 트래픽 |

### 3. 핵심 키워드 (Must-Have)

- **Scale Up(수직 확장):** 단일 서버 사양 향상, 중단 필요, 한계 있음
- **Scale Out(수평 확장):** 서버 수 증가, 무중단, 이론적 무한
- **로드 밸런서:** Scale Out의 트래픽 분산 핵심 장치
- **Stateless 설계:** Scale Out을 위한 애플리케이션 필수 조건
- **스케줄 기반:** Cron 스케줄로 사전 설정, 예측 가능한 패턴에 적합
- **부하 기반(Metric-based):** CPU/메모리/응답시간 임계치 기반
- **플래핑(Flapping):** 임계치 경계에서 지속적 스케일 인/아웃 반복 현상
- **Cooldown 기간:** 스케일링 후 안정화를 위한 대기 시간

### 4. 맥락과 비교

**AWS Auto Scaling Group 구성 요소:**

```
Launch Template (인스턴스 설정)
       +
Auto Scaling Group (최소/최대/희망 인스턴스 수)
       +
Scaling Policy (스케줄/Target Tracking/Step Scaling)
       +
CloudWatch Alarm (지표 임계치 모니터링)
```

### 5. 실무 제언

**쿠버네티스 HPA(Horizontal Pod Autoscaler) 설계 가이드**

- **챌린지:** 쿠버네티스 HPA를 CPU 사용률 기준으로만 설정하면 CPU 집약적이지 않은 워크로드(큐 처리, IO 바운드 앱)에서 스케일링이 적절히 동작하지 않습니다. 또한 새 파드가 준비되기 전 기존 파드에 과부하가 걸리는 Cold Start 문제도 발생합니다.
- **제언:** 
  1. **커스텀 지표 활용:** KEDA(Kubernetes Event-driven Autoscaler)로 Kafka 토픽 오프셋, RabbitMQ 큐 길이, HTTP 요청 큐 깊이 등 비즈니스 지표 기반 스케일링
  2. **VPA(Vertical Pod Autoscaler) 병행:** Scale Up 방식으로 단일 파드의 리소스 요청/제한 자동 조정
  3. **예측적 스케일링:** KEDA Cron Trigger로 스케줄 기반과 지표 기반을 조합
  스케일링 원리를 이해해야 쿠버네티스 클러스터 비용 최적화와 안정성을 동시에 달성할 수 있습니다.
