## 해설

### 출제 배경 및 의도

DMA(Direct Memory Access)는 CPU 개입 없이 I/O 장치와 메모리 간 대량 데이터를 직접 전송하는 하드웨어 메커니즘으로, 고성능 I/O의 핵심 기반 기술입니다. 네트워크 패킷 처리, NVMe 디스크 I/O, AI 학습 데이터 로딩 등 현대 컴퓨팅의 모든 대규모 데이터 이동이 DMA에 의존하며, RDMA·GPUDirect로 진화하여 AI 클러스터의 통신 병목을 해소합니다.

채점관은 **I/O 제어 방식의 진화(폴링→인터럽트→DMA)**, **DMAC 내부 구조와 버스 연결**, 그리고 **사이클 스틸·버스트·투명 모드의 차이**를 이해하는지를 평가합니다.

### 1. DMA(Direct Memory Access), 개요

정 의  • DMA를 CPU가 버스 제어권을 DMA 컨트롤러(DMAC)에 위임하여 I/O 장치와 메모리 간 데이터를 CPU 개입 없이 직접 전송하는 입출력 제어 방식으로 정의함
       - CPU는 DMA 설정과 완료 인터럽트 처리만 담당하여 연산에 집중할 수 있음

```
DMA 전체 동작 흐름도

CPU          DMAC          시스템 버스          메모리          I/O 장치
 │            │                 │                  │               │
 │─DMA 설정──▶│                 │                  │               │
 │  (주소,    │                 │                  │               │
 │   카운트,  │                 │                  │               │
 │   방향)    │                 │                  │               │
 │────────────│──BRQ(버스 요청)─▶│                  │               │
 │◀───────────│──BGA(버스 양도)──│                  │               │
 │  (내부 연산│                 │                  │               │
 │   계속)    │────────────────────────────────────────────────────│
 │            │     데이터 직접 전송(CPU 개입 없음)                 │
 │            │────────────────────────────────────────────────────│
 │◀─인터럽트──│ (전송 완료 통보) │                  │               │
 │  완료 처리 │                 │                  │               │
```

- 전송 완료 시 DMAC가 인터럽트로 CPU에 통보하므로 CPU는 DMA 진행 중 다른 연산을 수행할 수 있음

### 2. DMAC 구조·버스 연결·동작 모드

1) DMA 제어기(DMAC) 내부 구조

| 구분 | 레지스터 | 역할 |
|------|----------|------|
| 주소 | 주소 레지스터 | 메모리 전송 시작 주소 저장(전송마다 자동 증가) |
| 카운트 | 카운트 레지스터 | 남은 전송 바이트 수(전송마다 감소, 0이 되면 완료) |
| 제어 | 제어 레지스터 | 전송 방향(읽기/쓰기), 모드, 장치 ID 설정 |
| 상태 | 상태 레지스터 | 완료·오류·진행 상태 표시 |
| 중재 | 버스 인터페이스 로직 | BRQ(Bus Request)/BGA(Bus Grant Acknowledge) 처리 |

- BRQ: DMAC가 CPU에 버스 제어권 요청 신호
- BGA: CPU가 버스 제어권 양도 승인 신호

2) I/O 제어 방식 진화 비교

| 구분 | 방식 | CPU 부담 | 성능 |
|------|------|----------|------|
| ① | 폴링(Polling) | 매우 높음(반복 상태 확인) | 낮음 |
| ② | 인터럽트 방식 | 중간(I/O 완료 시 인터럽트) | 중간 |
| ③ | DMA 방식 | 낮음(설정+완료 인터럽트만) | 높음 |
| ④ | 채널 I/O | 매우 낮음(독립 I/O 프로세서) | 최고(메인프레임) |

- DMA는 폴링·인터럽트 방식의 CPU 부담을 대폭 경감하여 고속 I/O를 가능하게 하는 핵심 발전임

3) DMA 동작 모드 비교

| 구분 | 모드 | 동작 방식 | CPU 영향 | 전송 속도 | 적합 시나리오 |
|------|------|-----------|----------|-----------|---------------|
| ① | 사이클 스틸 | 버스 사이클 하나씩 빼앗아 전송 | 약간 감소 | 중간 | 소량 간헐적 전송 |
| ② | 버스트 | 버스를 독점하여 블록 전체 전송 | 완전 정지 | 빠름 | 대용량 고속 전송(디스크, 비디오) |
| ③ | 투명 | CPU 버스 유휴 사이클에만 전송 | 없음 | 느림 | CPU 성능 영향 최소화 필요 시 |

- 버스트 모드는 CPU를 완전히 정지시키므로 실시간 응답이 필요한 시스템에서는 사이클 스틸 모드를 우선 고려함

### 3. DMA 기술 진화 및 현대 적용 트렌드

| 기술 | 설명 | 적용 사례 |
|------|------|-----------|
| IOMMU | I/O 장치의 DMA 주소 변환·보안 격리 | 가상화 환경 DMA 보안 |
| DMA-BUF | Linux의 DMA 버퍼 공유 프레임워크 | GPU-카메라 제로카피 |
| RDMA | 네트워크를 통한 원격 메모리 직접 접근 | InfiniBand, RoCE |
| PCIe P2P DMA | CPU 경유 없이 GPU↔NVMe 직접 전송 | AI 데이터 로딩 최적화 |
| GPUDirect RDMA | GPU 메모리↔원격 GPU 메모리 직접 전송 | 분산 AI 학습 클러스터 |

- DMA 원리는 단일 시스템 내 I/O에서 출발하여 데이터센터 규모의 GPU 간 통신 최적화(GPUDirect)로 진화하고 있음.  "끝"

### 실무 제언

**분산 AI 학습에서 GPUDirect RDMA 활용**
- **챌린지**: 기존 방식(GPU→CPU→NIC→네트워크→원격 NIC→원격 CPU→원격 GPU)은 데이터가 CPU를 두 번 경유하여 지연이 크고, 1000GPU 클러스터에서 통신 병목이 전체 학습 시간의 30~50%를 차지함
- **제언**: NVIDIA GPUDirect RDMA를 활성화하면 GPU 메모리 데이터가 CPU를 우회하여 NIC를 통해 원격 GPU 메모리로 직접 전송되어 지연이 수십 μs에서 수 μs로 단축되고 CPU는 AI 연산에만 집중할 수 있음

**NVMe SSD 제로카피(Zero-Copy) I/O 최적화**
- **챌린지**: AI 학습 데이터 전처리 시 NVMe SSD에서 읽은 데이터가 커널 버퍼→사용자 버퍼→GPU VRAM으로 세 번 복사되어 PCIe 대역폭과 CPU 사이클을 낭비함
- **제언**: Linux io_uring + PCIe P2P DMA를 조합하여 NVMe SSD 데이터를 CPU 버퍼를 거치지 않고 GPU VRAM으로 직접 전송하는 제로카피 파이프라인을 구축하면 데이터 로딩 병목을 획기적으로 개선함

**가상화 환경에서 IOMMU를 이용한 DMA 보안 격리**
- **챌린지**: 악의적인 PCIe 디바이스(DMA 공격)가 IOMMU 없이 임의의 물리 메모리를 DMA로 읽어 다른 VM의 메모리를 탈취하는 보안 취약점이 존재함
- **제언**: IOMMU(Intel VT-d, AMD-Vi)를 활성화하여 각 I/O 장치의 DMA 접근 범위를 전용 메모리 영역으로 제한하고, 가상화 플랫폼(KVM, Hyper-V)에서 IOMMU 기반 패스스루(Passthrough) 설정으로 성능과 보안을 동시에 확보함
