## 해설

### 출제 배경 및 의도

디지털 인프라 의존도가 심화되면서 금융·의료·항공·통신 등 미션 크리티컬 시스템의 서비스 중단은 막대한 경제적 손실과 사회적 혼란을 초래합니다. AWS는 4분 장애로 수십억 원의 손실을 기록한 사례가 있으며, 항공 관제 시스템 1초 중단도 인명 사고로 이어질 수 있습니다. 이처럼 결함이 발생하더라도 서비스가 지속되는 FTS는 현대 IT 인프라의 핵심 설계 원칙입니다.

응용 등급 1번 문제로 출제되어 FTS 전반을 포괄적으로 서술해야 합니다. 채점관은 Fault-Error-Failure 관계, 이중화 기법(Active-Active/Standby/TMR), 가용성 지표(MTBF·MTTR·가용률), RTO·RPO를 체계적으로 연결하여 설명하는지를 평가합니다.

### 1. FTS(Fault Tolerant System), 개요

정 의  • 하드웨어·소프트웨어 결함(Fault) 발생 시에도 서비스가 중단 없이 지속되는 시스템
       - 이중화(Redundancy), 오류 검출·정정(Error Detection/Correction), 자동 전환(Failover)으로 구현

```
Fault(결함) ──▶ Error(오류) ──▶ Failure(장애)
  ↑                                  ↓
하드웨어 불량                    서비스 중단
소프트웨어 버그                  데이터 손실
운영 실수

FTS 목표: Fault 단계에서 오류 전파를 차단하여 Failure 방지
```

- FTS는 결함을 제거하는 것이 아니라 결함이 서비스 중단으로 이어지지 않도록 설계하는 철학

### 2. FTS 구현 기법

1) 하드웨어 이중화 (Hardware Redundancy)

| 구분 | 항목 | 설명 |
|------|------|------|
| Active-Active | 방식 | 모든 구성 요소가 동시에 부하 분담 |
| | 전환 시간 | ~0 (실시간 처리) |
| | 예시 | Dual-Active DB Cluster, 로드밸런서 |
| Active-Standby (Hot) | 방식 | 대기 장비가 실시간 동기화, 즉시 전환 |
| | 전환 시간 | 수초 이내 |
| | 예시 | HA 클러스터 (Pacemaker, Corosync) |
| Warm Standby | 방식 | 대기 장비가 주기적 동기화 |
| | 전환 시간 | 수분 내 |
| | 예시 | DR 사이트 |
| Cold Standby | 방식 | 대기 장비 꺼진 상태, 수동 전환 |
| | 전환 시간 | 수십 분 이상 |
| | 예시 | 백업 서버 |

```
구성 요소별 이중화 전략:

전원부   → UPS + 이중 전원 공급 장치
네트워크 → 이중 NIC(Bonding/LACP) + 이중 스위치
스토리지 → RAID + 이중 컨트롤러 + 다중 경로(MPIO)
서버     → HA 클러스터 (Pacemaker, Corosync)
사이트   → DR 센터 (지리적 분산)
```

- 이중화 수준이 높을수록 RTO/RPO가 작아지나 비용이 기하급수적으로 증가

2) TMR(Triple Modular Redundancy) 및 소프트웨어 기법

| 구분 | 항목 | 설명 |
|------|------|------|
| TMR | 구조 | 동일 모듈 3개 + 다수결 투표기 |
| | 허용 고장 | 모듈 1개 고장 실시간 허용 |
| | 적용 | 항공 비행 제어, 원전 제어, 우주선 |
| 소프트웨어 | 체크포인팅 | 주기적 상태 저장, 복구 지점 제공 |
| | Circuit Breaker | 연속 실패 시 자동 폴백(Fallback) |
| | Replication | Raft/Paxos 합의로 데이터 다중 복제 |
| | 롤링 업데이트 | 무중단 배포 |

```
TMR 구조:

입력 ─┬─▶ 모듈 A ─┐
      ├─▶ 모듈 B ─┼─▶ 다수결 투표기(2/3) ─▶ 출력
      └─▶ 모듈 C ─┘
```

- TMR은 하드웨어 비용이 3배이나 안전 임계 시스템에서 유일한 선택지

### 3. 가용성 지표와 FTS·HA·DR 비교

가용률(Availability): A = MTBF / (MTBF + MTTR) × 100%

```
시간 축:
백업 시점       장애 발생       복구 완료
    │◄── RPO ───►│◄─── RTO ────►│
  (최대 데이터   시점          (최대 서비스
   손실 범위)                  중단 시간)
```

| 가용률 | 연간 다운타임 | 적용 영역 |
|------|------|------|
| 99% (2-nine) | 87.6시간 | 일반 서비스 |
| 99.9% (3-nine) | 8.76시간 | 대부분 상용 서비스 |
| 99.99% (4-nine) | 52.6분 | 고가용성 서비스 |
| 99.999% (5-nine) | 5.26분 | 통신, 금융 핵심 |
| 99.9999% (6-nine) | 31.5초 | 항공, 의료 |

| 구분 | FTS | HA | DR |
|------|------|------|------|
| 목표 | 장애 중에도 서비스 지속 | 빠른 장애 전환 | 재해로부터 복구 |
| RTO | ~0 | 수초~수분 | 수분~수시간 |
| RPO | ~0 | 수초 | 수분~수시간 |
| 비용 | 매우 높음 | 높음 | 중간 |
| 예시 | 항공기 제어, 원전 | 은행 코어 시스템 | 재해 복구 사이트 |

MTBF를 높이고 MTTR을 낮추는 이중화·자동화 설계가 FTS의 핵심이며, 비용과 가용성 요건을 고려한 적정 수준 선택이 중요함.  "끝"

### 실무 제언

**클라우드 네이티브 환경의 FTS 구현**
- **챌린지**: 마이크로서비스 환경에서 수십~수백 개 서비스 간 의존성이 복잡하여 단일 서비스 장애가 연쇄 장애(Cascading Failure)로 번지는 문제가 빈발
- **제언**: Circuit Breaker(Netflix Hystrix, Resilience4j)로 연속 실패 시 자동 폴백, Kubernetes Liveness/Readiness Probe로 장애 파드 자동 재시작, 멀티 AZ 배포로 가용 영역 장애 자동 흡수

**카오스 엔지니어링으로 FTS 검증**
- **챌린지**: 실제 장애 상황에서의 FTS 동작을 사전에 검증하지 않으면 장애 시 예상치 못한 취약점 노출
- **제언**: Chaos Monkey(Netflix), Chaos Mesh(CNCF)로 의도적 장애를 주입하여 이중화 전환, 데이터 복구, 응답 시간 목표 달성 여부를 정기적으로 검증

**RPO·RTO 요건 기반 DR 전략 설계**
- **챌린지**: 모든 시스템에 동일한 DR 수준을 적용하면 비용이 과다하고, 차등 없이 낮은 수준을 적용하면 중요 시스템 복구 실패 위험
- **제언**: 업무 영향도 분석(BIA)으로 시스템별 RTO·RPO를 정의하고, 핵심 시스템은 Active-Active(RTO~0), 중요 시스템은 Hot Standby(RTO 수초), 일반 시스템은 Warm/Cold Standby로 차등 적용
