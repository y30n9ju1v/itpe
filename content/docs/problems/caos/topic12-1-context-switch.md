## 문제

운영체제에서 문맥(context)의 내용, 문맥교환(context switch)의 과정 및 문맥교환이 발생하는 조건을 설명하시오.

## 출제 정보

| 항목 | 내용 |
|------|------|
| 토픽(대) | 문맥교환 |
| 토픽(소) | 문맥교환 |
| 출제 | 110회 |
| 유형 | 응용 |
| 번호 | 4 |

## 해설

> **여러 프로세스가 하나의 CPU를 공유할 수 있는 마법의 메커니즘**  
> 문맥(Context)의 구성, 교환 과정, 발생 조건, 그리고 오버헤드

CPU는 한 순간에 하나의 프로세스만 실행할 수 있습니다. 그러나 현대 OS에서 수백 개의 프로세스가 동시에 실행되는 것처럼 보이는 것은, OS가 매우 빠른 속도로 프로세스를 교체—즉, **문맥교환(Context Switch)**—을 수행하기 때문입니다. 사람의 눈으로는 동시에 보이지만, 실제로는 빛의 속도로 번갈아가며 실행되고 있습니다.

### 1. 출제 배경 및 의도

110회 응용 문제로 문맥, 문맥교환 과정, 발생 조건을 종합적으로 물어봅니다. 채점관은 **문맥의 구성 요소(PCB 내용)**, **문맥교환의 단계별 절차**, **문맥교환 발생 조건(4가지)**을 정확히 이해하는지를 봅니다. 오버헤드와 최소화 전략까지 언급하면 완성도 높은 답안이 됩니다.

### 2. 개념의 본질과 작동 메커니즘

#### 문맥(Context)의 내용

**문맥(Context)**이란 프로세스가 실행을 중단했다가 다시 재개할 때 필요한 **모든 상태 정보**입니다. 이 정보들은 PCB(Process Control Block)에 저장됩니다.

```
프로세스 문맥(Context)의 구성

┌─────────────────────────────────────────────────┐
│              CPU 레지스터 상태                   │
│  - PC (Program Counter): 다음 실행 명령어 주소  │
│  - SP (Stack Pointer): 스택 최상단 주소         │
│  - 범용 레지스터 (R0~R15 등)                   │
│  - PSW (Program Status Word): 플래그, 모드 비트 │
├─────────────────────────────────────────────────┤
│              메모리 관리 정보                    │
│  - 페이지 테이블 주소 (CR3 레지스터, x86)       │
│  - 세그먼트 레지스터 (CS, DS, SS, ES 등)       │
├─────────────────────────────────────────────────┤
│              프로세스 상태 정보                  │
│  - 프로세스 상태 (Running/Ready/Waiting)        │
│  - PID, 우선순위                                │
├─────────────────────────────────────────────────┤
│              I/O 및 파일 정보                    │
│  - 열린 파일 디스크립터 목록                    │
│  - I/O 장치 상태                                │
└─────────────────────────────────────────────────┘
```

#### 문맥교환(Context Switch)의 과정

```
프로세스 A 실행 중
         │
         │ ← 문맥교환 트리거 발생 (인터럽트/시스템 콜)
         ▼
[1] 현재 프로세스 A의 CPU 상태를 PCB-A에 저장
    (모든 레지스터, PC, PSW → PCB-A)
         │
         ▼
[2] OS 스케줄러가 다음 실행할 프로세스 B 선택
    (Ready Queue에서 알고리즘에 따라 선택)
         │
         ▼
[3] PCB-B에서 프로세스 B의 CPU 상태 복원
    (PCB-B의 레지스터, PC, PSW → CPU)
         │
         ▼
[4] 주소 공간 전환 (CR3 레지스터 변경 → TLB 플러시)
    (프로세스 B의 페이지 테이블로 교체)
         │
         ▼
프로세스 B 실행 재개 (중단됐던 지점부터)
```

**문맥교환 시간 동안 CPU는 유용한 작업을 수행하지 못합니다** (순수 오버헤드). 이 시간은 일반적으로 수 마이크로초~수십 마이크로초입니다.

#### 문맥교환 발생 조건

| 조건 | 설명 |
|---|---|
| **타임 퀀텀 만료** | Round-Robin 등 시분할 스케줄링에서 할당 시간 초과 |
| **I/O 요청** | 프로세스가 I/O를 요청하여 Waiting 상태로 전환 |
| **인터럽트 발생** | 하드웨어 인터럽트 처리 후 다른 프로세스가 선택될 때 |
| **높은 우선순위 프로세스 등장** | 선점 스케줄링에서 더 높은 우선순위 프로세스가 Ready 상태가 될 때 |
| **프로세스 종료** | 현재 실행 프로세스가 exit() 호출 |
| **동기화 대기** | 세마포어/뮤텍스에서 wait() 호출로 블록 |

#### 프로세스 vs 스레드 문맥교환 비교

| 구분 | 프로세스 문맥교환 | 스레드 문맥교환 |
|---|---|---|
| **주소 공간 전환** | 필요 (CR3 변경, TLB 플러시) | 불필요 (같은 가상 주소 공간) |
| **비용** | 높음 (TLB 미스 폭발) | 낮음 |
| **저장 정보** | 전체 문맥 (메모리 포함) | CPU 레지스터만 |
| **캐시 효과** | 나쁨 (캐시 오염) | 좋음 (캐시 공유) |

**이 때문에 스레드를 "경량 프로세스(Lightweight Process)"라고 부릅니다.**

### 3. 핵심 키워드 (Must-Have)

- **문맥(Context):** 프로세스 재개를 위한 모든 상태 정보 (PCB에 저장)
- **PCB:** 문맥을 저장하는 데이터 구조
- **문맥교환 4단계:** 저장 → 스케줄 → 복원 → 주소 공간 전환
- **TLB 플러시:** 프로세스 전환 시 주소 변환 캐시 무효화
- **발생 조건:** 타임 퀀텀, I/O, 인터럽트, 높은 우선순위, 종료, 동기화
- **스레드 문맥교환:** 주소 공간 전환 없음 → 더 빠름
- **문맥교환 오버헤드:** 순수 낭비 시간, 최소화 필요

### 4. 맥락과 비교

**문맥교환 오버헤드 최소화 전략:**

| 전략 | 방법 |
|---|---|
| **스레드 사용** | 프로세스 대신 스레드로 전환 비용 절감 |
| **ASID 활용** | TLB 플러시 없이 프로세스별 주소 공간 구분 |
| **코루틴 / 그린 스레드** | OS 스케줄러 없이 사용자 레벨에서 협력적 스위칭 |
| **CPU 어피니티(Affinity)** | 프로세스를 특정 CPU 코어에 고정, 캐시 재사용 극대화 |
| **NUMA 최적화** | 프로세스와 메모리를 같은 NUMA 노드에 배치 |

### 5. 실무 제언

**Go 언어의 고루틴(Goroutine)이 문맥교환 비용을 혁신한 방법**

- **챌린지:** OS 스레드 기반 서버에서 초당 수만 건의 동시 요청을 처리하려면 수만 개의 OS 스레드가 필요하며, 이들 간의 문맥교환 비용이 서버 성능의 병목이 됩니다.
- **제언:** Go 언어의 고루틴은 **M:N 스레드 모델**(M개의 고루틴을 N개의 OS 스레드에 매핑)을 사용합니다. 고루틴의 스택 크기가 2KB에서 시작(OS 스레드는 1~8MB)하고, Go 런타임이 고루틴 간 협력적 문맥교환을 수행하여 OS 커널 진입 없이 전환합니다. 결과적으로 수십만 개의 고루틴을 수십 개의 OS 스레드로 처리하며, 문맥교환 비용이 OS 문맥교환의 1/10 이하입니다. 이 원리는 전통적인 OS 문맥교환 이론의 한계를 사용자 공간에서 극복한 현대적 설계 철학의 산물입니다.
