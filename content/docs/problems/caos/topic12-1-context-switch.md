## 문제

운영체제에서 문맥(context)의 내용, 문맥교환(context switch)의 과정 및 문맥교환이 발생하는 조건을 설명하시오.

## 해설

### 출제 배경 및 의도

문맥교환(Context Switch)은 단일 CPU에서 여러 프로세스가 동시에 실행되는 것처럼 보이게 하는 다중 프로그래밍의 핵심 메커니즘입니다. 1960년대 시분할 시스템(Time-Sharing System) 도입과 함께 정립된 이 개념은, 현대 클라우드·컨테이너 환경에서도 성능 최적화의 핵심 고려 사항으로 남아 있습니다.

110회 응용 문제로 문맥의 내용, 교환 과정, 발생 조건을 종합적으로 묻습니다. 채점관은 **문맥의 구성 요소(PCB 내용)**, **문맥교환의 단계별 절차**, **발생 조건 6가지**를 정확히 이해하는지를 평가합니다. TLB 플러시 오버헤드와 스레드 문맥교환과의 비교까지 언급하면 완성도 높은 답안이 됩니다.

### 1. 문맥교환(Context Switch), 개요

정 의  • 문맥교환을 CPU가 실행 중인 프로세스의 상태(문맥)를 저장하고 다른 프로세스의 저장된 상태를 복원하여 실행권을 이전하는 OS 핵심 메커니즘으로 정의함
       - 문맥교환 수행 중 CPU는 유용한 작업을 수행하지 못하므로 순수 오버헤드(Pure Overhead)임

```
문맥교환 과정 흐름도

 프로세스 A 실행                  OS 커널                  프로세스 B 실행
      │                            │                            │
      │ ←─ 트리거 발생 ────────────│                            │
      │    (인터럽트/시스템콜)      │                            │
      │                            │① PCB-A에 CPU 상태 저장     │
      │                            │  (PC, SP, 레지스터, PSW)   │
      │                            │② 스케줄러: Ready Queue에서 B 선택
      │                            │③ PCB-B에서 CPU 상태 복원   │
      │                            │④ CR3 변경 → TLB 플러시    │
      │                            │   (B의 페이지 테이블 교체) │
      │                            │─────────────────────────▶ │
      │                            │                     B 실행 재개
```

- 프로세스 문맥교환은 TLB 플러시를 수반하므로 스레드 문맥교환 대비 수 배 이상의 오버헤드가 발생함

### 2. 문맥 구성·교환 과정·발생 조건

1) 문맥(Context)의 구성 요소

| 구분 | 항목 | 설명 |
|------|------|------|
| CPU 상태 | PC(Program Counter) | 다음 실행 명령어 주소 |
| CPU 상태 | SP(Stack Pointer) | 스택 최상단 주소 |
| CPU 상태 | 범용 레지스터(R0~R15) | ALU 연산에 사용되는 레지스터 값 |
| CPU 상태 | PSW(Program Status Word) | 플래그 비트(Zero, Carry 등), 모드(User/Kernel) |
| 메모리 | CR3 레지스터(x86) | 현재 프로세스 페이지 테이블 기저 주소 |
| 메모리 | 세그먼트 레지스터 | CS, DS, SS, ES 등 |
| 프로세스 | PID, 우선순위, 상태 | 스케줄러 판단 기준 |
| I/O | 열린 파일 디스크립터 | 파일·소켓·파이프 목록 |

- 이 모든 정보가 PCB(Process Control Block)에 저장되며, PCB 없이는 문맥교환이 불가능함

2) 문맥교환 발생 조건

| 구분 | 조건 | 설명 |
|------|------|------|
| ① | 타임 퀀텀 만료 | Round-Robin 등 시분할에서 할당 시간 초과 |
| ② | I/O 요청 | read()/write() 등으로 Waiting 상태 전환 |
| ③ | 인터럽트 발생 | 하드웨어 인터럽트 처리 후 스케줄러 재실행 |
| ④ | 높은 우선순위 등장 | 선점형 스케줄링에서 더 높은 우선순위 프로세스가 Ready 상태가 될 때 |
| ⑤ | 프로세스 종료 | exit() 호출로 현재 프로세스 종료 |
| ⑥ | 동기화 대기 | 세마포어/뮤텍스 wait() 호출로 블록 |

- 조건 ①③④는 선점(Preemptive) 스케줄링, 조건 ②⑤⑥는 협력적(Cooperative) 상황에서 발생함

3) 프로세스 vs 스레드 문맥교환 비교

| 구분 | 프로세스 문맥교환 | 스레드 문맥교환 |
|------|-----------------|-----------------|
| 주소 공간 전환 | 필요(CR3 변경, TLB 플러시) | 불필요(동일 가상 주소 공간) |
| 오버헤드 | 높음(TLB 미스 폭발, 캐시 오염) | 낮음(레지스터만 교체) |
| 저장 정보 | 전체 문맥(메모리 정보 포함) | CPU 레지스터만 |
| 캐시 효과 | 나쁨(캐시 오염) | 좋음(캐시 공유) |
| 소요 시간 | 수십~수백 μs | 수~수십 μs |

- 스레드를 "경량 프로세스(Lightweight Process)"라 부르는 이유가 바로 주소 공간 전환 없는 낮은 문맥교환 비용 때문임

### 3. 문맥교환 오버헤드 최소화 전략 및 트렌드

| 전략 | 방법 | 효과 |
|------|------|------|
| 스레드 활용 | 프로세스 대신 스레드로 전환 비용 절감 | TLB 플러시 제거 |
| ASID(Address Space ID) | TLB 플러시 없이 프로세스별 주소 공간 구분 | TLB 히트율 유지 |
| 코루틴/그린 스레드 | OS 스케줄러 없이 사용자 레벨 협력적 스위칭 | 커널 진입 제거 |
| CPU 어피니티(Affinity) | 프로세스를 특정 코어에 고정 | 캐시 재사용 극대화 |
| NUMA 최적화 | 프로세스와 메모리를 동일 NUMA 노드에 배치 | 원격 메모리 접근 제거 |

- Go 고루틴(M:N 스레드 모델)처럼 사용자 공간 스케줄러를 활용하면 OS 문맥교환 비용의 1/10 이하로 줄일 수 있음.  "끝"

### 실무 제언

**Go 고루틴과 사용자 공간 문맥교환 최적화**
- **챌린지**: OS 스레드 기반 서버에서 초당 수만 건 동시 요청 처리 시 수만 개 스레드의 문맥교환 비용이 병목이 되며, 스택 메모리도 스레드당 1~8MB로 급증함
- **제언**: Go 고루틴은 스택을 2KB에서 시작하고 M:N 모델로 수십만 개 고루틴을 수십 개 OS 스레드에 매핑하여 문맥교환 비용을 OS 수준의 1/10 이하로 절감함

**쿠버네티스 컨테이너 스케줄링과 문맥교환 오버헤드**
- **챌린지**: 고밀도 컨테이너 환경에서 수백 개의 컨테이너가 CPU를 공유할 때 문맥교환 오버헤드가 누적되어 실제 처리량이 이론값 대비 30~50% 감소하는 경우가 발생함
- **제언**: CPU 어피니티 설정(cpuset.cpus)과 cgroups 기반 CPU 쿼터 제한을 조합하여 컨테이너별 문맥교환 빈도를 제어하고, NUMA 토폴로지를 고려한 파드 배치로 메모리 접근 지연을 최소화함

**실시간 시스템에서의 문맥교환 지연 시간 보장**
- **챌린지**: 로봇 제어, 자동차 ECU 등 Hard Real-Time 시스템에서 문맥교환 지연이 예측 불가능하면 마감 시간(Deadline) 위반으로 이어짐
- **제언**: PREEMPT_RT 패치가 적용된 Linux 커널이나 전용 RTOS(VxWorks, FreeRTOS)를 사용하여 문맥교환 지연을 수 μs 이내로 보장하고, WCET(최악 실행 시간) 분석 도구로 전체 경로를 정적 검증함
