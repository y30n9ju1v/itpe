## 문제

인터럽트(Interrupt)에 대하여 다음사항을 설명하시오.
가. 인터럽트 발생 요인(하드웨어 및 소프트웨어)
나. 마스커블(Maskable) 인터럽트와 넌마스커블(Non-maskable) 인터럽트
다. 벡터(Vectored) 인터럽트의 동작원리
라. 우선순위(Priority)를 보장하는 다중(Multiple) 인터럽트가 발생한 경우 이를 처리하기 위한 ISR(Interrupt Service Routine)을 프로세서 및 소프트웨어에서 제공하여야 하는 기능요소들과 상호 연관시켜 설명하시오.

## 해설

> **CPU가 세상과 소통하는 방법: 비동기 이벤트 처리 메커니즘**  
> 하드웨어/소프트웨어 인터럽트, 마스크 가능성, 벡터 테이블, 다중 인터럽트 처리

컴퓨터가 CPU에 발생하는 수많은 외부/내부 이벤트(키보드 입력, 네트워크 패킷 도착, 타이머 만료, 오류 등)를 처리하는 방법은 크게 **폴링(Polling)**과 **인터럽트(Interrupt)** 두 가지입니다. 폴링은 CPU가 능동적으로 이벤트를 확인하는 방식이고, 인터럽트는 이벤트 발생 시 하드웨어가 CPU에 비동기 신호를 보내 현재 작업을 중단하고 처리하도록 하는 방식입니다.

### 1. 출제 배경 및 의도

120회 컴시응(컴퓨터시스템응용) 문제로 가~라 항목의 상세 서술을 요구합니다. 채점관은 **인터럽트의 종류**, **마스크 가능성**, **인터럽트 벡터 테이블의 동작**, 그리고 **다중 인터럽트 처리에 필요한 하드웨어와 소프트웨어 요소**를 유기적으로 연결하여 설명하는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 인터럽트 발생 요인

**하드웨어 인터럽트 (외부 인터럽트):**
- 키보드 입력, 마우스 이동
- 네트워크 패킷 도착 (NIC 인터럽트)
- 타이머 만료 (Timer Interrupt) — OS 스케줄러의 기반
- 디스크 I/O 완료
- 하드웨어 오류 (전원 이상, 메모리 오류)

**소프트웨어 인터럽트 (내부 인터럽트, 트랩/예외):**

| 종류 | 발생 조건 | 예시 |
|---|---|---|
| **트랩 (Trap)** | 의도적 소프트웨어 발생 | 시스템 콜(INT 0x80, SYSCALL) |
| **예외 (Exception)** | 비의도적 오류 상황 | 0으로 나누기, 페이지 폴트, 세그먼트 폴트 |

#### 나. 마스커블 vs 넌마스커블 인터럽트

| 구분 | 마스커블(Maskable) 인터럽트 | 넌마스커블(Non-Maskable, NMI) |
|---|---|---|
| **특징** | CPU의 인터럽트 플래그(IF)로 차단 가능 | 어떤 상황에서도 무조건 처리 |
| **제어 명령어** | x86: CLI(비활성화), STI(활성화) | 차단 불가 |
| **발생 원인** | 대부분의 하드웨어 인터럽트 | 하드웨어 치명적 오류 (메모리 패리티 오류, 전원 이상) |
| **OS 임계 구역** | 임계 구역에서 차단 가능 | 항상 즉시 처리 |

**인터럽트 마스킹 활용:**

```c
// OS 커널의 임계 구역 보호
disable_interrupts();  // CLI 명령어 (IF=0)
  // 공유 데이터 구조 수정
  // (인터럽트로 인한 개입 없음 보장)
enable_interrupts();   // STI 명령어 (IF=1)
```

#### 다. 벡터 인터럽트 동작원리

**인터럽트 벡터 테이블(IVT / IDT, Interrupt Descriptor Table):**
각 인터럽트 번호에 대응하는 ISR(Interrupt Service Routine)의 시작 주소를 저장한 테이블.

```
인터럽트 벡터 테이블 (x86의 경우)
┌─────┬──────────────────────────────────┐
│ 번호│   ISR 주소 / 설명                │
├─────┼──────────────────────────────────┤
│  0  │  Division by Zero Handler        │
│  1  │  Debug Exception Handler         │
│  2  │  NMI Handler                     │
│  3  │  Breakpoint Handler              │
│ ... │  ...                              │
│ 14  │  Page Fault Handler              │
│ ... │  ...                              │
│ 32~ │  Hardware Interrupt (PIC/APIC)   │
│ 128 │  System Call (INT 0x80)          │
└─────┴──────────────────────────────────┘
```

**벡터 인터럽트 동작 흐름:**

```
1. 외부 장치 → 인터럽트 신호 발송 → 인터럽트 컨트롤러(PIC/APIC)
2. PIC → CPU에 인터럽트 요청(IRQ) 신호
3. CPU → 현재 명령어 완료 후 인터럽트 확인
4. CPU → 현재 CPU 상태(레지스터, PC, PSW) → 스택에 저장
5. CPU → 인터럽트 번호를 인덱스로 IVT에서 ISR 주소 조회
6. PC ← ISR 시작 주소 (ISR 실행 시작)
7. ISR 실행 → 인터럽트 처리
8. ISR 완료 → IRET 명령어 → 저장된 상태 복원
9. 중단된 프로그램 재개
```

#### 라. 다중 인터럽트 처리와 우선순위

**문제:** 인터럽트 처리 중 다른 인터럽트가 발생하면?

**해결 방안 — 우선순위 기반 선점 인터럽트:**

**프로세서가 제공해야 하는 하드웨어 기능:**
1. **인터럽트 우선순위 레지스터(IPR):** 현재 처리 중인 인터럽트 우선순위 저장
2. **인터럽트 마스크 레지스터(IMR):** 특정 우선순위 이하 인터럽트 차단 마스크
3. **APIC(Advanced PIC):** 인터럽트 벡터와 우선순위 관리 하드웨어
4. **스택(Stack):** 중첩된 컨텍스트 저장을 위한 LIFO 구조

**소프트웨어(OS)가 제공해야 하는 기능:**
1. **ISR 등록/해제:** `request_irq()` / `free_irq()` (Linux)
2. **우선순위별 ISR 분류:** Top Half(빠른 처리) / Bottom Half(지연 처리)
3. **재진입성(Reentrancy):** ISR이 같은 ISR에 의해 중단될 때의 안전한 처리
4. **인터럽트 컨텍스트 관리:** ISR 실행 중 sleep 불가 등의 제약 관리

**다중 인터럽트 처리 시나리오:**

```
시간선: ──[일반 실행]──[ISR-A 시작]──[ISR-B 선점]──[ISR-B 완료]──[ISR-A 재개]──

1. IRQ-A 발생 (우선순위 낮음) → ISR-A 시작
2. ISR-A 실행 중 IRQ-B 발생 (우선순위 높음)
3. ISR-A의 상태를 스택에 저장 → ISR-B 실행 (선점)
4. ISR-B 완료 → IRET → ISR-A 상태 복원 → ISR-A 재개
5. ISR-A 완료 → IRET → 원래 프로그램 재개

하드웨어: IPR에 현재 처리 우선순위 기록, IMR로 하위 인터럽트 자동 차단
소프트웨어: ISR 시작 시 save context, 완료 시 restore context
```

### 3. 핵심 키워드 (Must-Have)

- **하드웨어 인터럽트:** 외부 장치에서 발생하는 비동기 신호
- **소프트웨어 인터럽트(트랩/예외):** 의도적(시스템 콜) 또는 오류(페이지 폴트)
- **마스커블 인터럽트:** IF 플래그로 차단 가능
- **NMI(Non-Maskable Interrupt):** 치명적 하드웨어 오류, 차단 불가
- **인터럽트 벡터 테이블(IVT/IDT):** 인터럽트 번호 → ISR 주소 매핑
- **ISR(Interrupt Service Routine):** 인터럽트 처리 핸들러
- **IRET:** 인터럽트 반환 명령어 (컨텍스트 복원)
- **PIC/APIC:** 인터럽트 컨트롤러 하드웨어
- **Top Half / Bottom Half:** Linux의 인터럽트 처리 분리

### 4. 맥락과 비교

**폴링(Polling) vs 인터럽트:**

| 구분 | 폴링 | 인터럽트 |
|---|---|---|
| **이벤트 감지** | CPU가 주기적으로 확인 | 장치가 신호 전송 |
| **CPU 효율** | 낮음 (불필요한 체크) | 높음 (이벤트 시만 처리) |
| **응답 지연** | 폴링 주기에 따라 변동 | 즉시 처리 |
| **구현 복잡도** | 낮음 | 높음 |
| **적합 상황** | 이벤트 발생 빈도 높을 때 | 이벤트 발생 빈도 낮을 때 |

### 5. 실무 제언

**인터럽트 코얼레싱(Interrupt Coalescing)으로 네트워크 성능 향상**

- **챌린지:** 10Gbps 이상의 고속 네트워크 환경에서 NIC가 패킷마다 인터럽트를 발생시키면 초당 수백만 번의 인터럽트로 CPU 사용률이 100%에 달하는 "인터럽트 폭풍(Interrupt Storm)"이 발생합니다.
- **제언:** **인터럽트 코얼레싱(Interrupt Coalescing, NAPI in Linux)** 기법으로 일정 시간 또는 일정 패킷 수만큼 모아 한 번의 인터럽트로 처리합니다. `ethtool -C eth0 rx-usecs 50` 명령으로 50μs 동안 인터럽트를 모아 처리하도록 설정할 수 있습니다. 이는 레이턴시와 처리량 사이의 트레이드오프로, 실시간 시스템에서는 코얼레싱 시간을 줄여 낮은 레이턴시를 유지해야 합니다.
