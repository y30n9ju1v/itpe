## 해설

### 출제 배경 및 의도

인터럽트(Interrupt)는 CPU가 연속적인 프로그램 실행 중에도 외부 이벤트에 신속히 반응할 수 있게 하는 비동기 이벤트 처리 메커니즘입니다. 타이머, 네트워크, 디스크 I/O 등 현대 컴퓨터의 모든 I/O 처리가 인터럽트에 기반하며, OS 스케줄러의 선점 기반 역시 타이머 인터럽트에서 출발합니다.

120회 컴시응 문제로 가~라 4개 항목의 상세 서술을 요구합니다. 채점관은 **하드웨어·소프트웨어 인터럽트 구분**, **마스크 가능성**, **벡터 테이블의 동작 원리**, 그리고 **다중 인터럽트 처리에 필요한 하드웨어·소프트웨어 기능 요소**를 유기적으로 연결하여 설명하는지를 평가합니다.

### 1. 인터럽트(Interrupt), 개요

정 의  • 인터럽트를 CPU가 현재 실행을 중단하고 이벤트를 처리하기 위해 ISR로 분기하는 비동기 메커니즘으로 정의함
       - 이벤트 발생 시에만 CPU가 반응하므로 폴링(Polling) 대비 CPU 효율이 높음

```
인터럽트 처리 전체 흐름

외부 장치          PIC/APIC           CPU               메모리
   │                  │                │                  │
   │──IRQ 신호───────▶│                │                  │
   │                  │──인터럽트 요청─▶│                  │
   │                  │                │ 현재 명령어 완료  │
   │                  │                │ PC/레지스터 스택에 저장
   │                  │                │──IVT 조회────────▶│
   │                  │                │◀─ISR 주소 반환───│
   │                  │                │ ISR 실행          │
   │                  │                │ IRET → 상태 복원  │
   │                  │                │ 원래 프로그램 재개│
```

- 인터럽트 벡터 테이블(IVT)이 번호별 ISR 주소를 매핑하여 빠른 분기를 가능하게 함

### 2. 인터럽트 유형·벡터·다중 처리

1) 인터럽트 발생 요인

| 구분 | 종류 | 발생 조건 | 예시 |
|------|------|-----------|------|
| 하드웨어(외부) | 타이머 인터럽트 | 주기적 타이머 만료 | OS 스케줄러 선점 기반 |
| 하드웨어(외부) | I/O 인터럽트 | 키보드, NIC, 디스크 완료 | 패킷 수신, 디스크 읽기 완료 |
| 하드웨어(외부) | 하드웨어 오류 | 전원 이상, 메모리 패리티 오류 | NMI 발생 |
| 소프트웨어(내부) | 트랩(Trap) | 의도적 소프트웨어 발생 | 시스템 콜(INT 0x80, SYSCALL) |
| 소프트웨어(내부) | 예외(Exception) | 비의도적 오류 | 0 나누기, 페이지 폴트, 세그먼트 폴트 |

- 트랩은 사용자 모드 → 커널 모드 전환의 정상 경로이며, 예외는 오류 복구 또는 프로세스 종료로 이어짐

2) 마스커블 vs 넌마스커블 인터럽트

| 구분 | 마스커블(Maskable) | 넌마스커블(NMI) |
|------|-------------------|-----------------|
| 차단 가능 여부 | CPU 인터럽트 플래그(IF)로 차단 가능 | 어떤 상황에서도 무조건 처리 |
| 제어 명령어 | x86: CLI(비활성화), STI(활성화) | 차단 불가 |
| 발생 원인 | 대부분의 하드웨어 I/O 인터럽트 | 치명적 하드웨어 오류(메모리 패리티, 전원 이상) |
| OS 임계 구역 | 임계 구역 진입 시 차단하여 원자성 보장 | 항상 즉시 처리 |

- OS 커널의 임계 구역(Critical Section)은 CLI/STI 명령으로 마스커블 인터럽트를 차단하여 공유 자료구조를 보호함

3) 벡터 인터럽트 동작원리 및 다중 인터럽트 처리

| 구분 | 구성 요소 | 역할 |
|------|-----------|------|
| 하드웨어 | IVT/IDT | 인터럽트 번호 → ISR 주소 매핑 테이블 |
| 하드웨어 | PIC/APIC | 인터럽트 우선순위 중재·벡터 전달 |
| 하드웨어 | IPR(우선순위 레지스터) | 현재 처리 중인 인터럽트 우선순위 저장 |
| 하드웨어 | IMR(마스크 레지스터) | 하위 우선순위 인터럽트 자동 차단 |
| 하드웨어 | 스택(Stack) | 중첩 컨텍스트 저장을 위한 LIFO 구조 |
| 소프트웨어 | ISR 등록/해제 | request_irq() / free_irq() (Linux) |
| 소프트웨어 | Top Half / Bottom Half | 빠른 처리(Top) + 지연 처리(Bottom, softirq/tasklet) |
| 소프트웨어 | 재진입성(Reentrancy) | 동일 ISR 중첩 실행 시 안전한 처리 보장 |

- 다중 인터럽트 발생 시 APIC가 우선순위를 중재하고, 스택 기반 컨텍스트 저장으로 중첩 ISR 처리를 지원함

### 3. 폴링 vs 인터럽트 비교 및 트렌드

| 구분 | 폴링(Polling) | 인터럽트 |
|------|--------------|----------|
| 이벤트 감지 | CPU가 주기적으로 상태 레지스터 확인 | 장치가 CPU에 비동기 신호 전송 |
| CPU 효율 | 낮음(불필요한 반복 체크) | 높음(이벤트 발생 시에만 처리) |
| 응답 지연 | 폴링 주기에 따라 변동 | 즉시 처리 |
| 구현 복잡도 | 낮음 | 높음(ISR, 컨텍스트 저장/복원) |
| 적합 상황 | 이벤트 빈도 매우 높을 때(NVMe SSD) | 이벤트 빈도 낮을 때(키보드, 네트워크) |

- 10Gbps 이상 고속 네트워크에서는 패킷 폭주로 인터럽트 폭풍(Interrupt Storm)이 발생하므로, Linux NAPI처럼 인터럽트+폴링을 결합한 하이브리드 방식으로 진화하고 있음.  "끝"

### 실무 제언

**인터럽트 코얼레싱으로 고속 네트워크 성능 향상**
- **챌린지**: 10Gbps 이상 환경에서 NIC가 패킷마다 인터럽트를 발생시키면 초당 수백만 번 인터럽트로 CPU 사용률이 100%에 달하는 인터럽트 폭풍이 발생함
- **제언**: `ethtool -C eth0 rx-usecs 50` 명령으로 인터럽트 코얼레싱을 설정하여 50μs 단위로 묶어 처리하되, 실시간 시스템에서는 코얼레싱 시간을 최소화하여 레이턴시와 처리량의 균형을 맞춤

**임베디드 시스템에서의 인터럽트 지연 시간(Interrupt Latency) 관리**
- **챌린지**: Hard Real-Time 시스템(자동차 에어백, 로봇 제어)에서 인터럽트 응답 지연이 수 μs를 초과하면 안전 기능이 실패할 수 있음
- **제언**: ISR을 최소화(Top Half만 실행)하고 긴 처리는 Bottom Half(softirq, workqueue)로 위임하며, 임계 구역의 인터럽트 차단 시간을 측정·제한하여 WCET(Worst Case Execution Time)를 보장함

**보안 관점: 인터럽트 기반 사이드 채널 공격 대응**
- **챌린지**: 타이머 인터럽트를 이용한 정밀 타이밍 측정이 Spectre·Meltdown 류의 사이드 채널 공격에 활용됨
- **제언**: Linux의 timer_interrupt_freq 제한, KPTI(Kernel Page Table Isolation) 적용, 그리고 CPU 마이크로코드 업데이트를 병행하여 타이밍 기반 공격 표면을 축소함
