## 해설

> **스토리지 선택의 ABC: 블록, 파일, 오브젝트의 데이터 접근 방식 차이**  
> 저수준 블록 단위, 디렉토리 파일 시스템, 메타데이터 기반 객체 저장소

현대 IT 인프라에서 데이터를 어떻게 저장하고 접근하느냐에 따라 세 가지 스토리지 패러다임이 존재합니다. 클라우드 서비스(AWS S3, EBS, EFS)도 이 세 가지 유형에 정확히 대응됩니다.

### 1. 출제 배경 및 의도

132회 관리 문제로 출제되었습니다. 채점관은 **세 가지 스토리지의 데이터 접근 방식의 근본적 차이**, **적합한 사용 사례**, 그리고 **클라우드 서비스와의 연관성**을 체계적으로 비교하는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### 블록 스토리지 (Block Storage)

```
논리 볼륨 (예: 100GB)
┌──────────────────────────────────────┐
│ Block[0] Block[1] Block[2] ... Block[N]│  ← 고정 크기 블록 (512B ~ 4KB)
└──────────────────────────────────────┘
      ↑
   OS의 파일 시스템(ext4, NTFS)이
   블록 위에 파일 시스템 구조를 구축
```

**특징:**
- 데이터를 **고정 크기의 블록(Block)** 단위로 저장
- 접근 방식: **블록 주소(LBA, Logical Block Address)**로 직접 읽기/쓰기
- OS의 파일 시스템이 블록 위에서 파일 추상화를 제공
- **프로토콜:** SAN (iSCSI, Fibre Channel), NVMe over Fabrics
- 애플리케이션 관점에서는 로컬 디스크처럼 보임

**장점:**
- 최고의 성능 (낮은 레이턴시, 높은 IOPS)
- 애플리케이션이 I/O를 완전 제어
- 파일 시스템 선택 자유로움

**단점:**
- 파일 공유 어려움 (기본적으로 1:1 마운트)
- 관리 복잡도 높음
- 메타데이터 관리는 파일 시스템에 의존

**사용 사례:** 데이터베이스(Oracle, MySQL), OS 부팅 볼륨, 고성능 VM

**클라우드 서비스:** AWS EBS, Azure Managed Disk, GCP Persistent Disk

#### 파일 스토리지 (File Storage)

```
공유 파일 시스템
     /
    ├── home/
    │   ├── user1/
    │   │   └── document.txt
    │   └── user2/
    └── shared/
        └── report.xlsx

클라이언트 A ─┐
클라이언트 B ─┼─▶ NFS/SMB 프로토콜 ─▶ 파일 서버 (공유 스토리지)
클라이언트 C ─┘
```

**특징:**
- 데이터를 **파일과 디렉토리 계층 구조**로 저장
- 접근 방식: **파일 경로(Path)와 파일명**으로 접근
- **프로토콜:** NFS(Network File System), SMB/CIFS, AFP
- 여러 클라이언트가 동시에 같은 파일에 접근 가능 (공유)

**장점:**
- 직관적인 파일/폴더 구조
- 다중 클라이언트 동시 접근 (협업 환경)
- 기존 애플리케이션 호환성 높음 (파일 시스템 API 사용)

**단점:**
- 수백만 개 파일 시 성능 저하 (디렉토리 탐색 오버헤드)
- 대규모 수평 확장 어려움
- 블록 스토리지 대비 성능 낮음

**사용 사례:** 공유 파일 서버, 홈 디렉토리, 컨텐츠 관리, NAS

**클라우드 서비스:** AWS EFS, Azure Files, GCP Filestore

#### 오브젝트 스토리지 (Object Storage)

```
오브젝트 스토리지 (플랫 네임스페이스)
┌──────────────────────────────────────────────────────────┐
│  Bucket: my-data                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ Key: images/2024/photo.jpg                        │  │
│  │ Value: [바이너리 데이터]                           │  │
│  │ Metadata: {size:1MB, content-type:image/jpeg, ... }│  │
│  └───────────────────────────────────────────────────┘  │
│  ┌───────────────────────────────────────────────────┐  │
│  │ Key: logs/2024-06/access.log                      │  │
│  │ Value: [바이너리 데이터]                           │  │
│  │ Metadata: {size:50MB, created:2024-06-14, ...}    │  │
│  └───────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘

접근: HTTP REST API (GET/PUT/DELETE)
      URL: https://bucket.s3.amazonaws.com/images/2024/photo.jpg
```

**특징:**
- 데이터를 **오브젝트(Key + Value + Metadata)** 단위로 저장
- 접근 방식: **HTTP REST API** (GET, PUT, DELETE)
- 계층적 디렉토리 없음 → 플랫 네임스페이스 (Key에 `/`를 포함하여 유사 계층 표현)
- 각 오브젝트에 풍부한 **커스텀 메타데이터** 첨부 가능

**장점:**
- 수십억 개 오브젝트까지 **무한 수평 확장**
- HTTP API로 어디서나 접근 (인터넷 기반)
- 저비용 대용량 저장 (콜드 티어 활용)
- 버전 관리, 수명 주기 정책 내장

**단점:**
- 높은 레이턴시 (HTTP 오버헤드)
- 랜덤 I/O 불가 (오브젝트 전체를 읽기/쓰기)
- 기존 파일 시스템 API 호환 안 됨 (애플리케이션 수정 필요)
- POSIX 미지원 (파일 잠금, 부분 수정 어려움)

**사용 사례:** 이미지/동영상 저장, 백업/아카이브, 정적 웹사이트, ML 데이터셋, 로그 저장

**클라우드 서비스:** AWS S3, Azure Blob Storage, GCP Cloud Storage, MinIO(온프레미스)

#### 세 가지 스토리지 비교

| 비교 항목 | 블록 스토리지 | 파일 스토리지 | 오브젝트 스토리지 |
|---|---|---|---|
| **접근 단위** | 고정 크기 블록 | 파일/디렉토리 | 오브젝트(Key-Value) |
| **접근 방법** | 블록 주소(LBA) | 파일 경로 | HTTP REST API |
| **프로토콜** | iSCSI, FC, NVMe-oF | NFS, SMB | HTTP/HTTPS (S3 API) |
| **공유** | 제한적 | 용이 (다중 클라이언트) | 용이 (HTTP 기반) |
| **확장성** | 제한적 | 제한적 | 거의 무한 |
| **성능** | 최고 | 중간 | 낮음 (레이턴시) |
| **비용** | 높음 | 중간 | 낮음 |
| **메타데이터** | OS 파일 시스템 | OS 파일 시스템 | 풍부한 커스텀 메타데이터 |

### 3. 핵심 키워드 (Must-Have)

- **블록 스토리지:** LBA 블록 단위 접근, iSCSI/FC, 최고 성능, DB에 적합
- **파일 스토리지:** 경로 기반 파일 접근, NFS/SMB, 공유 파일 서버
- **오브젝트 스토리지:** Key-Value+메타데이터, HTTP REST, 무한 확장, S3
- **플랫 네임스페이스:** 오브젝트 스토리지의 계층 없는 구조
- **POSIX:** 파일 시스템 표준 인터페이스 (오브젝트 스토리지는 미지원)

### 4. 맥락과 비교

**클라우드 3대 스토리지 서비스 대응:**

| 스토리지 유형 | AWS | Azure | GCP |
|---|---|---|---|
| 블록 | EBS | Managed Disk | Persistent Disk |
| 파일 | EFS | Azure Files | Filestore |
| 오브젝트 | S3 | Blob Storage | Cloud Storage |

### 5. 실무 제언

**AI/ML 파이프라인에서 스토리지 선택 전략**

- **챌린지:** 머신러닝 학습 파이프라인에서 데이터 수집(오브젝트), 전처리(파일), 모델 학습(블록), 결과 저장(오브젝트)이 각각 다른 스토리지 특성을 요구합니다.
- **제언:** 표준 ML 아키텍처 패턴:
  1. **원시 데이터 수집:** S3/GCS (오브젝트) — 저비용, 무한 확장
  2. **학습 데이터 준비:** EFS/NFS (파일) — 다수 워커 동시 접근
  3. **모델 체크포인트:** EBS/NVMe (블록) — 고속 쓰기 IOPS
  4. **학습 완료 모델:** S3/GCS (오브젝트) — 모델 레지스트리

  각 단계에 적합한 스토리지를 선택하면 불필요한 비용과 성능 손실을 방지할 수 있습니다.
