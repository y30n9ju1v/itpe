## 해설

### 출제 배경 및 의도

클라우드 전환이 가속화되면서 스토리지 유형 선택은 시스템 성능·비용·확장성에 직결되는 아키텍처 결정 사항이 되었습니다. AWS S3, EBS, EFS로 대표되는 세 가지 스토리지 유형은 데이터 접근 방식에서 근본적 차이를 가지며, 잘못된 선택은 수십 배의 비용 낭비나 심각한 성능 저하를 초래합니다.

132회 관리 문제로 출제되었으며, 채점관은 세 스토리지의 **데이터 접근 단위와 프로토콜의 근본적 차이**, **확장성·성능·비용 측면의 비교**, 그리고 **클라우드 서비스와의 대응 관계 및 적합 사용 사례**를 체계적으로 서술하기를 기대합니다.

### 1. 스토리지 유형, 개요

정 의  • 데이터 저장·접근 방식에 따라 블록·파일·오브젝트로 분류되는 세 가지 스토리지 패러다임
       - 각 유형은 접근 단위, 프로토콜, 확장성, 성능, 비용이 근본적으로 상이함

```
┌───────────────────────────────────────────────────────────┐
│                  스토리지 유형 구조 비교                   │
│                                                           │
│  블록 스토리지          파일 스토리지        오브젝트 스토리지│
│  ┌──────────┐          ┌──────────┐        ┌──────────┐  │
│  │Block[0]  │          │    /     │        │Key:img/a │  │
│  │Block[1]  │          │  ├ home/ │        │Val:[데이터]│ │
│  │Block[2]  │          │  └ data/ │        │Meta:{...}│  │
│  │  ...     │          │          │        │          │  │
│  └──────────┘          └──────────┘        └──────────┘  │
│  LBA 주소 접근          파일 경로 접근       HTTP REST API  │
│  iSCSI/FC/NVMe-oF      NFS/SMB             GET/PUT/DELETE │
└───────────────────────────────────────────────────────────┘
```

- 각 유형은 데이터 접근 추상화 수준과 프로토콜이 상이하여 용도별 최적 선택이 필요함

### 2. 스토리지 유형별 접근 방식 및 특성

1) 블록 스토리지 (Block Storage)

| 구분 | 항목 | 설명 |
|------|------|------|
| 접근 방식 | 단위 | 고정 크기 블록 (512B ~ 4KB) |
| | 방법 | LBA(Logical Block Address) 직접 주소 지정 |
| | 프로토콜 | iSCSI, Fibre Channel, NVMe over Fabrics |
| 특성 | 성능 | 최고 (낮은 레이턴시, 높은 IOPS) |
| | 공유 | 제한적 (기본 1:1 마운트) |
| | 확장성 | 제한적 |
| | 비용 | 높음 |
| 용도 | 사례 | DB(Oracle, MySQL), OS 부팅 볼륨, 고성능 VM |
| | 클라우드 | AWS EBS, Azure Managed Disk, GCP Persistent Disk |

- OS의 파일 시스템(ext4, NTFS)이 블록 위에서 파일 추상화를 제공하므로 애플리케이션 I/O 완전 제어 가능

2) 파일 스토리지 (File Storage)

| 구분 | 항목 | 설명 |
|------|------|------|
| 접근 방식 | 단위 | 파일 및 디렉토리 계층 구조 |
| | 방법 | 파일 경로(Path)와 파일명으로 접근 |
| | 프로토콜 | NFS(Network File System), SMB/CIFS |
| 특성 | 성능 | 중간 |
| | 공유 | 용이 (다중 클라이언트 동시 접근) |
| | 확장성 | 제한적 (수백만 파일 시 성능 저하) |
| | 비용 | 중간 |
| 용도 | 사례 | 공유 파일 서버, 홈 디렉토리, 컨텐츠 관리, NAS |
| | 클라우드 | AWS EFS, Azure Files, GCP Filestore |

```
파일 스토리지 공유 구조:

클라이언트 A ─┐
클라이언트 B ─┼─▶ NFS/SMB ─▶ 파일 서버 (/home, /shared)
클라이언트 C ─┘
```

- POSIX 표준 인터페이스를 지원하여 기존 애플리케이션 호환성이 높음

### 3. 오브젝트 스토리지와 클라우드 대응 비교

```
오브젝트 스토리지 구조 (플랫 네임스페이스):

Bucket: my-data
┌─────────────────────────────────────────────────────┐
│ Key:   images/2024/photo.jpg                        │
│ Value: [바이너리 데이터]                            │
│ Meta:  {size:1MB, content-type:image/jpeg, ...}     │
├─────────────────────────────────────────────────────┤
│ Key:   logs/2024-06/access.log                      │
│ Value: [바이너리 데이터]                            │
│ Meta:  {size:50MB, created:2024-06-14, ...}         │
└─────────────────────────────────────────────────────┘
접근: HTTP REST API (GET/PUT/DELETE)
```

| 비교 항목 | 블록 스토리지 | 파일 스토리지 | 오브젝트 스토리지 |
|------|------|------|------|
| 접근 단위 | 고정 크기 블록 | 파일/디렉토리 | 오브젝트(Key-Value+Meta) |
| 접근 방법 | 블록 주소(LBA) | 파일 경로 | HTTP REST API |
| 프로토콜 | iSCSI, FC, NVMe-oF | NFS, SMB | HTTP/HTTPS (S3 API) |
| 공유 | 제한적 | 용이 | 용이 (HTTP 기반) |
| 확장성 | 제한적 | 제한적 | 거의 무한 |
| 성능 | 최고 | 중간 | 낮음 (HTTP 레이턴시) |
| 비용 | 높음 | 중간 | 낮음 |
| AWS | EBS | EFS | S3 |
| Azure | Managed Disk | Azure Files | Blob Storage |
| GCP | Persistent Disk | Filestore | Cloud Storage |

오브젝트 스토리지는 수십억 개 오브젝트까지 무한 수평 확장이 가능하나, POSIX 미지원으로 랜덤 I/O 및 부분 수정이 어려워 DB 용도로는 부적합함.  "끝"

### 실무 제언

**AI/ML 파이프라인에서 스토리지 선택 전략**
- **챌린지**: 머신러닝 파이프라인에서 데이터 수집·전처리·학습·결과 저장 단계마다 서로 다른 스토리지 특성이 요구되어 단일 스토리지 선택 시 성능 또는 비용 낭비 발생
- **제언**: 원시 데이터 수집은 S3/GCS(오브젝트, 저비용·무한 확장), 학습 데이터 준비는 EFS/NFS(파일, 다수 워커 동시 접근), 모델 체크포인트는 EBS/NVMe(블록, 고속 IOPS), 완성 모델은 S3/GCS(오브젝트, 모델 레지스트리)로 단계별 최적 스토리지를 조합

**멀티테넌트 환경의 스토리지 보안 설계**
- **챌린지**: 오브젝트 스토리지 버킷 ACL 오설정으로 인한 민감 데이터 공개 노출 사례가 빈발 (S3 버킷 무단 공개 접근 등)
- **제언**: 버킷 퍼블릭 접근 차단 정책 기본 활성화, IAM 최소 권한 정책 적용, 서버 측 암호화(SSE-KMS) 의무화, CloudTrail로 접근 로그 상시 모니터링

**하이브리드 스토리지 티어링 설계**
- **챌린지**: 데이터 성격(핫/웜/콜드)에 무관하게 동일 스토리지를 사용하면 불필요한 고비용 스토리지 낭비 발생
- **제언**: 핫 데이터(빈번 접근)는 블록/파일 스토리지, 웜 데이터(주기적 접근)는 오브젝트 스탠다드, 콜드 데이터(아카이브)는 S3 Glacier/Azure Archive로 수명 주기 정책 기반 자동 티어링 적용
