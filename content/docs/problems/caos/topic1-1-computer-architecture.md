## 해설

### 출제 배경 및 의도

컴퓨터구조 도메인의 가장 기초적인 뼈대로, 이후 파이프라인·캐시메모리·가상메모리 등 모든 심화 주제의 출발점이 되는 주제다. 101회 응용 문제로 출제된 이 주제는 컴퓨터구조의 원리를 근본에서부터 이해하는지 검증한다.

채점관은 단순 암기가 아닌 각 구성 요소 간의 유기적 관계와 Stored-program이 왜 혁명적인지를 논리적으로 서술할 수 있는지를 본다. 스마트폰, 데이터센터, AI 가속기 칩까지 1940년대 폰 노이만(John von Neumann)이 정립한 "명령어를 메모리에 저장하고 순서대로 꺼내 실행한다"는 원리 위에 세워진 구조이므로, 이 아이디어가 가져온 소프트웨어 산업의 탄생이라는 시대적 맥락까지 짚어 주면 고득점을 狙할 수 있다.

### 1. 컴퓨터시스템 5대 구성 요소, 개요

정 의  • 입력·출력·기억·연산·제어 다섯 장치가 버스(Bus)로 연결되어 데이터를 처리하는 디지털 시스템
       - 연산장치(ALU)와 제어장치(CU)를 합쳐 CPU(Central Processing Unit)라 부름

```
┌─────────────────────────────────────────────┐
│                  CPU                        │
│  ┌──────────────┐   ┌──────────────────┐   │
│  │  제어장치(CU) │   │  연산장치(ALU)   │   │
│  └──────┬───────┘   └────────┬─────────┘   │
└─────────┼────────────────────┼─────────────┘
          │      내부 버스      │
    ┌─────▼─────────────────────▼─────┐
    │        기억장치(Memory)          │
    │  주기억장치(RAM) + 보조기억장치   │
    └────────────┬────────────────────┘
                 │
     ┌───────────┴───────────┐
┌────▼────┐             ┌────▼────┐
│ 입력장치 │             │ 출력장치 │
└─────────┘             └─────────┘
```

- 5대 구성 요소는 유기적으로 연결되어 입력→처리→출력의 데이터 흐름을 완성한다.

### 2. 5대 구성 요소 상세

1) 각 장치 역할과 부품

| 구분 | 구성 요소 | 역할 | 부품 예시 |
|------|-----------|------|-----------|
| 입력 | 입력장치(Input Unit) | 외부 데이터를 2진수로 변환하여 내부로 전달 | 키보드, 마우스, 스캐너 |
| 출력 | 출력장치(Output Unit) | 처리 결과를 사람이 인식할 수 있는 형태로 변환 | 모니터, 프린터, 스피커 |
| 기억 | 기억장치(Memory Unit) | 명령어와 데이터를 저장 | RAM, ROM, SSD, HDD |
| 연산 | 연산장치(ALU) | 산술(+, -, ×, ÷) 및 논리(AND, OR, NOT) 연산 수행 | CPU 내부 ALU 회로 |
| 제어 | 제어장치(CU) | 명령어 해독 후 각 장치에 제어 신호 전송 | CPU 내부 CU 회로 |

- 연산장치와 제어장치를 통합한 CPU가 전체 동작을 주관한다.

2) 주요 레지스터

| 레지스터 | 역할 |
|----------|------|
| PC(Program Counter) | 다음에 실행할 명령어의 메모리 주소 보관 |
| IR(Instruction Register) | 현재 실행 중인 명령어 임시 저장 |
| MAR(Memory Address Register) | 메모리 접근 주소 저장 |
| MBR(Memory Buffer Register) | 메모리에서 읽거나 쓸 데이터 임시 저장 |

- 레지스터는 CPU 내부의 초고속 임시 기억소자로, 명령어 처리 사이클의 각 단계에서 핵심 역할을 한다.

### 3. Stored-program Computer와 폰 노이만 병목

1) Stored-program 원리 (폰 노이만, 1945)

| 구분 | ENIAC 방식 (배선 방식) | 폰 노이만 방식 (Stored-program) |
|------|------------------------|----------------------------------|
| 프로그램 변경 | 물리적 재배선(수일 소요) | 메모리 내용 교체(즉시) |
| 유연성 | 특정 계산에 고정 | 범용 연산 가능 |
| 명령어 저장 | 하드웨어에 내장 | 메모리에 데이터와 함께 2진수로 저장 |
| 소프트웨어 산업 | 불가능 | 소프트웨어 독립 산업으로 탄생 |
| 단점 | 유연성 부재 | 폰 노이만 병목(단일 버스) |

2) 명령어 처리 사이클 (Instruction Cycle)

```
┌──────────────────────────────────────────┐
│                                          │
▼                                          │
Fetch(인출) → Decode(해독) → Execute(실행) ─┘
  ↑
  PC가 다음 명령어 주소를 가리킴

① Fetch   : PC가 가리키는 주소의 명령어를 RAM에서 읽어 IR에 저장, PC 증가
② Decode  : CU가 IR의 명령어 해독, 제어 신호 생성
③ Execute : ALU 연산 수행 또는 메모리 읽기/쓰기
```

- Fetch→Decode→Execute 사이클이 반복되며 프로그램이 순차 실행된다.

3) 폰 노이만 병목과 현대적 해결

| 병목 원인 | 해결 기법 |
|-----------|-----------|
| 명령어·데이터가 단일 버스 공유 | 하버드 아키텍처(명령어·데이터 버스 분리) |
| CPU-메모리 속도 격차 | 캐시메모리(L1/L2/L3) 계층 도입 |
| 순차 실행 구조 | 파이프라인, 슈퍼스칼라, OoOE |
| 데이터 영역 코드 실행 취약 | NX 비트(No-eXecute), DEP 하드웨어 차단 |

- 현대 CPU는 Out-of-Order Execution·SIMD 등 폰 노이만 병목 최소화 기법을 겹겹이 적용한다.  "끝"

### 실무 제언

**임베디드 시스템 보안**
- **챌린지**: 순수 폰 노이만 구조에서는 데이터 영역에 악성 코드를 주입하고 실행시키는 코드 인젝션 공격이 가능하다.
- **제언**: 현대 OS는 NX 비트(No-eXecute bit) 또는 DEP(Data Execution Prevention)로 데이터 영역의 코드 실행을 하드웨어 수준에서 차단해야 한다.

**현대 서버 아키텍처 적용**
- **챌린지**: 클라우드·AI 워크로드에서 단일 코어 성능은 한계에 도달하였고, 메모리 대역폭이 새로운 병목으로 부상하고 있다.
- **제언**: HBM(High Bandwidth Memory)·CXL(Compute Express Link) 등 메모리 인터페이스 표준을 활용하여 폰 노이만 병목을 하드웨어 수준에서 완화하는 아키텍처 설계가 요구된다.

**운영체제 커널 최적화**
- **챌린지**: 명령어 캐시와 데이터 캐시를 분리하는 하버드 변형 구조를 소프트웨어에서 인지하지 못하면 불필요한 캐시 플러시가 발생한다.
- **제언**: 컴파일러 PGO(Profile-Guided Optimization)와 링커 섹션 배치 최적화를 통해 코드 지역성을 높이고 명령어 캐시 미스를 최소화해야 한다.
