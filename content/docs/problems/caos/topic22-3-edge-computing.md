엣지(Edge) 컴퓨팅과 클라우드 컴퓨팅의 차이점

## 해설

### 출제 배경 및 의도

IoT 기기 수가 2030년 500억 개를 돌파할 것으로 전망되면서, 모든 데이터를 중앙 클라우드로 전송하여 처리하는 방식은 레이턴시·대역폭·프라이버시 측면에서 명확한 한계에 봉착했습니다. 자율주행차는 200ms 레이턴시로 제동 명령을 받으면 100km/h에서 5.5m를 더 이동하여 사고가 발생하며, 스마트 팩토리의 실시간 품질 검사도 ms 단위 응답이 필수입니다. 엣지 컴퓨팅은 이러한 실시간성 요구에 대응하는 새로운 컴퓨팅 패러다임입니다.

139회 관리 1번 문제로, 채점관은 클라우드와 엣지 컴퓨팅의 **아키텍처 구조 차이**, **레이턴시·대역폭·프라이버시 비교 지표**, 그리고 **IoT·자율주행·스마트 팩토리 등의 실제 적용 사례**를 체계적으로 서술하는지를 평가합니다.

### 1. 엣지 컴퓨팅(Edge Computing), 개요

정 의  • 데이터 생성 지점(엣지)에 가까운 곳에서 컴퓨팅을 수행하여 레이턴시와 대역폭 소비를 최소화하는 분산 컴퓨팅 패러다임
       - 중앙 집중형 클라우드 처리에서 말단 디바이스 근처로 연산 위치를 이동

```
┌─────────────────────────────────────────────────────────┐
│           Cloud-Edge Continuum 아키텍처                  │
│                                                         │
│  클라우드 (중앙)                                        │
│  ┌──────────────────────────────────────┐               │
│  │ ML 모델 학습 │ 장기 분석 │ 글로벌 정책│               │
│  └──────────────────────┬───────────────┘               │
│                         │ 모델 배포, 집계 데이터         │
│  엣지 (중간 계층)       ▼                               │
│  ┌──────────────────────────────────────┐               │
│  │ 데이터 전처리 │ AI 추론 │ 실시간 집계│               │
│  └──────────────────────┬───────────────┘               │
│                         │ 원시 데이터 일부               │
│  디바이스 (말단)        ▼                               │
│  ┌──────────────────────────────────────┐               │
│  │ 센서 수집 │ 임계치 반응 │ TinyML     │               │
│  └──────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────┘
```

- 엣지 계층: ①디바이스 엣지(온디바이스 처리), ②로컬 엣지(공장 게이트웨이·기지국 MEC), ③리전 엣지(AWS Outposts)

### 2. 클라우드 vs 엣지 컴퓨팅 비교

1) 핵심 특성 비교

| 비교 항목 | 클라우드 컴퓨팅 | 엣지 컴퓨팅 |
|------|------|------|
| 처리 위치 | 원격 데이터센터 | 데이터 생성 지점 근처 |
| 레이턴시 | 높음 (수십~수백 ms) | 낮음 (수ms 이하) |
| 대역폭 소비 | 높음 (모든 데이터 전송) | 낮음 (필요 데이터만 전송) |
| 실시간 처리 | 제한적 | 우수 |
| 컴퓨팅 자원 | 거의 무한 | 제한적 |
| 비용 | 트래픽 비용 높음 | 로컬 하드웨어 비용 |
| 데이터 프라이버시 | 외부 전송 | 로컬 처리 (데이터 보유) |
| 네트워크 의존성 | 높음 | 낮음 (오프라인 가능) |
| 관리 복잡도 | 낮음 (중앙화) | 높음 (분산 관리) |
| 보안 | 중앙화 관리 용이 | 물리적 보안 취약 가능 |

2) 엣지 컴퓨팅 주요 적용 사례

| 구분 | 항목 | 설명 |
|------|------|------|
| 자율주행 | 문제 | 클라우드 200ms → 100km/h에서 5.5m 추가 이동 → 사고 위험 |
| | 해결 | 차량 내 엣지 컴퓨터 → 1ms 즉시 처리 → 실시간 제동 |
| 스마트 팩토리 | 문제 | 고속 라인에서 클라우드 왕복 시간(100ms+)으로 불량품 제거 타이밍 실패 |
| | 해결 | 엣지 AI 서버 → AI 불량 판정(< 10ms) → 즉시 제거 신호 |
| 의료 모니터링 | 문제 | 심전도 이상 감지 지연 시 응급 처치 불가 |
| | 해결 | 엣지 장치 → 이상 심박 즉시 감지 → 즉시 경보 |
| 5G MEC | 특징 | 5G 기지국 내 MEC 서버 → 1ms 이하 초저지연 AR/VR 처리 |

### 3. 클라우드-엣지 협력 표준 및 트렌드

```
스마트 팩토리 최적 아키텍처:

카메라 10대(60fps) → [엣지 AI 서버(GPU)] → AI 불량 판정(< 10ms) → 제거 신호
                          │
                    불량품 이미지+통계 (원시 데이터의 1~5%)
                          ↓
                    [클라우드] → 장기 트렌드 분석, AI 모델 재학습
```

| 기술/표준 | 설명 |
|------|------|
| AWS Outposts | AWS 인프라를 온프레미스에 배치 |
| Azure Stack Edge | MS 엣지 컴퓨팅 하드웨어+소프트웨어 |
| ETSI MEC | 5G MEC 표준화 기구 |
| KubeEdge | 쿠버네티스 기반 엣지 관리 |
| EdgeX Foundry | IoT 엣지 프레임워크 (LF Edge) |
| TinyML | 초소형 디바이스 ML 추론 (TensorFlow Lite Micro) |

5G와 AI의 결합으로 엣지 컴퓨팅 적용 범위가 자율주행·스마트 팩토리·의료·AR/VR 전 영역으로 확산되며, 클라우드와 엣지가 협력하는 Cloud-Edge Continuum이 차세대 인프라 표준으로 부상 중.  "끝"

### 실무 제언

**스마트 팩토리 클라우드-엣지 하이브리드 아키텍처 설계**
- **챌린지**: 카메라 10대가 초당 60프레임으로 불량 검사 시 초당 수 GB 영상을 클라우드로 전송하면 대역폭 비용이 연간 수억 원이며, 클라우드 왕복 100ms+로 고속 라인 타이밍 실패
- **제언**: 엣지 AI 서버(GPU 탑재)에서 즉시 불량 판정(< 10ms), 불량품 이미지+통계 데이터만 클라우드 전송(원시 영상의 1~5%)으로 네트워크 비용 90% 절감 및 실시간 대응 달성

**엣지 디바이스 보안 강화**
- **챌린지**: 엣지 디바이스는 외부 물리 환경에 노출되어 물리적 탈취·하드웨어 해킹에 취약하고, 중앙화 관리가 어려워 보안 패치 적용이 지연됨
- **제언**: TPM(Trusted Platform Module) 기반 하드웨어 보안, TLS 양방향 인증으로 엣지-클라우드 통신 암호화, OTA(Over-The-Air) 업데이트로 보안 패치 자동화

**엣지-클라우드 데이터 일관성 관리**
- **챌린지**: 네트워크 단절 시 엣지에서 수집된 데이터가 클라우드와 불일치하는 오프라인 싱크 문제 발생
- **제언**: 이벤트 소싱(Event Sourcing) 패턴으로 엣지에서 발생한 이벤트를 순서 보장 큐에 저장하고, 네트워크 복구 시 자동 동기화. 충돌 해결 정책(최신 시간 우선, 클라우드 우선 등)을 사전에 정의
