## 해설

> **서로가 서로를 기다리며 아무도 전진하지 못하는 교착 상태**  
> 코프만 4조건, 예방/회피/탐지/복구 전략, Banker's Algorithm, Wait-Die, Wound-Wait

교착상태(Deadlock)는 프로세스 집합이 각자가 다른 프로세스가 점유한 자원을 기다리면서 모두 영원히 대기하는 상태입니다. 서로가 서로를 기다리는 **원형 대기**가 형성되면 외부 개입 없이는 어떤 프로세스도 실행을 재개할 수 없습니다. 데이터베이스와 OS 모두에서 중요한 주제입니다.

### 1. 출제 배경 및 의도

교착상태는 OS의 자원 관리와 DB의 트랜잭션 관리 모두에서 핵심 이슈입니다. 채점관은 **코프만 4가지 발생 조건**, **예방/회피/탐지/복구 4가지 해결 전략**, **Banker's Algorithm의 안전 상태 개념**, 그리고 **분산 시스템의 Wait-Die와 Wound-Wait** 프로토콜까지 체계적으로 서술하기를 기대합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 교착상태 발생 4가지 조건 (코프만, Coffman 조건)

교착상태는 다음 4가지 조건이 **동시에** 성립할 때 발생합니다. 하나라도 깨면 교착상태는 발생하지 않습니다.

| 조건 | 설명 | 예시 |
|---|---|---|
| **상호 배제 (Mutual Exclusion)** | 자원은 한 번에 하나의 프로세스만 사용 가능 | 프린터는 한 번에 한 문서만 출력 |
| **점유와 대기 (Hold and Wait)** | 자원을 점유한 채 다른 자원을 기다림 | A가 리소스 1을 들고 리소스 2를 기다림 |
| **비선점 (No Preemption)** | 자원을 강제로 빼앗을 수 없음 | 프로세스가 자발적으로 반납할 때까지 기다려야 함 |
| **원형 대기 (Circular Wait)** | 프로세스들이 원형으로 서로 기다림 | A→B→C→A 형태의 대기 사이클 |

```
원형 대기 예시:
P1 → R2 (R2를 기다림)
P2 → R3 (R3를 기다림)
P3 → R1 (R1을 기다림)
R1 → P1 (P1이 점유)
R2 → P2 (P2가 점유)
R3 → P3 (P3가 점유)
```

#### 교착상태 해결 전략

**① 예방(Prevention):** 코프만 4조건 중 하나를 원천적으로 불가능하게 만듦

| 무력화할 조건 | 방법 | 부작용 |
|---|---|---|
| 상호 배제 | 자원을 공유 가능하게 → 현실적으로 불가능한 경우 많음 | 데이터 일관성 문제 |
| 점유와 대기 | 프로세스 시작 시 모든 자원을 한꺼번에 요청 | 자원 낭비, 기아 |
| 비선점 | 자원을 강제로 빼앗을 수 있게 허용 | 복잡한 상태 복원 필요 |
| 원형 대기 | 자원에 순서 번호 부여, 오름차순으로만 요청 허용 | 프로그래밍 제약 |

**② 회피(Avoidance): Banker's Algorithm**

자원 요청 시마다 "이 요청을 허용했을 때 교착상태로 빠질 위험이 없는가(안전 상태인가)"를 검사합니다.

**안전 상태(Safe State):** 모든 프로세스가 자원을 최대 요청까지 순차적으로 완료할 수 있는 순서(안전 순서, Safe Sequence)가 존재하는 상태

**Banker's Algorithm 예시:**

```
3개 프로세스, 자원 종류 A(10개), B(5개), C(7개)

         최대 요청    현재 할당    남은 필요량
P0       7 5 3       0 1 0       7 4 3
P1       3 2 2       2 0 0       1 2 2
P2       9 0 2       3 0 2       6 0 0

가용 자원 Available = [3 3 2]

안전 순서 검색:
1. P1: 필요 [1,2,2] ≤ 가용 [3,3,2] → P1 실행 가능
   P1 완료 후 Available = [2+2, 0+3, 0+2] = [5,3,2]  ← 오류수정: [3+2,3+0,2+0]=[5,3,2]

2. P0: 필요 [7,4,3] ≤ 가용 [5,3,2]? → 불가
   P2: 필요 [6,0,0] ≤ 가용 [5,3,2]? → 불가  
   → P0, P2 모두 불가... (예시 수정 필요)

실제 계산:
Available = [3,3,2]
P1 필요 [1,2,2] ≤ [3,3,2] → P1 실행 → Available = [3+2, 3+0, 2+0] = [5,3,2]
P0 필요 [7,4,3] ≤ [5,3,2] → 불가
P2 필요 [6,0,0] ≤ [5,3,2] → 불가
→ 안전하지 않음 (이 할당 거부)
```

**③ 탐지(Detection) + 복구(Recovery)**

교착상태를 허용하되, 주기적으로 자원 할당 그래프(Resource Allocation Graph)를 분석하여 사이클 탐지 후 복구:

- **복구 방법:**
  1. **프로세스 종료:** 교착상태 프로세스 중 하나(또는 모두) 강제 종료
  2. **자원 선점:** 희생 프로세스를 선택하여 자원 강제 회수 후 롤백

#### Deadlock 회피 기법: Wait-Die와 Wound-Wait (DB 분산 트랜잭션)

분산 데이터베이스에서 교착상태를 타임스탬프 기반으로 회피하는 프로토콜입니다. 더 오래된(Old, 타임스탬프 작은) 트랜잭션이 더 높은 우선순위를 가집니다.

**Wait-Die:**
- Old 트랜잭션이 Young 트랜잭션이 가진 자원을 요청 → **Wait (기다림)**
- Young 트랜잭션이 Old 트랜잭션이 가진 자원을 요청 → **Die (롤백)**

```
T1(old) 요청 → T2(young)가 점유 → T1 Wait
T2(young) 요청 → T1(old)이 점유 → T2 Die(롤백)
```

**Wound-Wait:**
- Old 트랜잭션이 Young 트랜잭션이 가진 자원을 요청 → **Wound (Young 롤백시킴)**
- Young 트랜잭션이 Old 트랜잭션이 가진 자원을 요청 → **Wait (기다림)**

```
T1(old) 요청 → T2(young)가 점유 → T2 Wound(강제 롤백) → T1 진행
T2(young) 요청 → T1(old)이 점유 → T2 Wait
```

**비교:**

| 구분 | Wait-Die | Wound-Wait |
|---|---|---|
| **Old → Young 충돌** | Old 기다림 (Wait) | Young 강제 롤백 (Wound) |
| **Young → Old 충돌** | Young 롤백 (Die) | Young 기다림 (Wait) |
| **롤백 주체** | 항상 Young | 항상 Young |
| **특성** | 비선점 (기다리다 포기) | 선점 (빼앗음) |

### 3. 핵심 키워드 (Must-Have)

- **코프만 4조건:** 상호 배제, 점유와 대기, 비선점, 원형 대기
- **예방(Prevention):** 4조건 중 하나 원천 무력화
- **회피(Avoidance):** Banker's Algorithm, 안전 상태(Safe State)
- **탐지(Detection):** 자원 할당 그래프, 사이클 탐지
- **복구(Recovery):** 프로세스 종료 또는 자원 선점
- **Wait-Die:** Old 기다림, Young 롤백 (비선점)
- **Wound-Wait:** Old가 Young을 강제 롤백, Young 기다림 (선점)
- **안전 순서(Safe Sequence):** Banker's Algorithm에서 모든 프로세스 완료 가능한 실행 순서

### 4. 맥락과 비교

| 전략 | 오버헤드 | 자원 활용률 | 적용 환경 |
|---|---|---|---|
| **예방** | 낮음 | 낮음 (과도한 제약) | 임베디드, 안전 임계 시스템 |
| **회피** | 중간 | 중간 | 자원 수 예측 가능한 시스템 |
| **탐지+복구** | 낮음 | 높음 | 범용 DB, OS |
| **무시** | 없음 | 최고 | 단순 시스템 (Ostrich Algorithm) |

### 5. 실무 제언

**마이크로서비스 분산 환경에서의 분산 교착상태**

- **챌린지:** 마이크로서비스 아키텍처에서는 여러 서비스가 서로의 API를 동기 호출(REST, gRPC)할 때 분산 교착상태가 발생할 수 있습니다. 서비스 A가 서비스 B를 기다리고, B가 A를 기다리는 원형 의존성이 형성되면 단일 노드 교착상태와 동일한 문제가 발생합니다.
- **제언:** MSA에서는 **타임아웃(Timeout) + 재시도 정책(Retry with Backoff)**을 모든 동기 호출에 적용하여 교착상태를 자동 해소합니다. 더 근본적으로는 **이벤트 드리븐 비동기 통신(Kafka, RabbitMQ)**으로 동기 의존성을 제거하는 것이 최선입니다. 기술사 답안에서 전통적 OS 이론을 현대 MSA 실무 문제와 연결하면 차별화된 답안이 됩니다.
