## 해설

### 출제 배경 및 의도

교착상태(Deadlock)는 프로세스 집합이 각자 다른 프로세스가 점유한 자원을 서로 기다리며 영원히 대기하는 상태입니다. OS의 자원 관리와 DB의 트랜잭션 관리 모두에서 핵심 이슈이며, 마이크로서비스 분산 환경에서도 서비스 간 원형 의존성으로 재현됩니다.

채점관은 **코프만 4가지 발생 조건**, **예방/회피/탐지/복구 4가지 해결 전략**, **Banker's Algorithm의 안전 상태 개념**, 그리고 **분산 시스템의 Wait-Die와 Wound-Wait 프로토콜**까지 체계적으로 서술하기를 기대합니다.

### 1. 교착상태(Deadlock), 개요

정 의  • 교착상태를 프로세스 집합이 각자 다른 프로세스가 점유한 자원을 기다리며 어떤 프로세스도 실행을 재개할 수 없는 영구 블록 상태로 정의함
       - 코프만(Coffman) 4조건이 동시에 성립할 때 발생하며, 하나라도 제거하면 교착상태는 발생하지 않음

```
교착상태 원형 대기 구조도(자원 할당 그래프)

  P1 ──점유──▶ R1 ──할당──▶ P3
  │                          │
  └──요청──▶ R2 ◀──점유─────┘
              │
  P2 ──요청──▶│
  │
  └──점유──▶ R3 ──요청──▶ P1

→ P1→R2, P2→R3, P3→R1의 원형 대기로 교착상태 발생
  외부 개입 없이 어떤 프로세스도 실행 재개 불가
```

- 원형 대기(Circular Wait)가 형성되면 자원 할당 그래프에서 사이클이 탐지되며, 이것이 교착상태의 가시적 증거임

### 2. 발생 조건·해결 전략·회피 알고리즘

1) 교착상태 발생 4가지 조건 (코프만 조건)

| 구분 | 조건 | 설명 | 무력화 방법 |
|------|------|------|-------------|
| ① | 상호 배제(Mutual Exclusion) | 자원은 한 번에 하나의 프로세스만 사용 | 자원 공유화(현실적 제약) |
| ② | 점유와 대기(Hold and Wait) | 자원 점유 중 다른 자원을 기다림 | 시작 시 모든 자원 일괄 요청 |
| ③ | 비선점(No Preemption) | 자원을 강제로 빼앗을 수 없음 | 자원 강제 선점 허용 |
| ④ | 원형 대기(Circular Wait) | 프로세스들이 원형으로 서로 기다림 | 자원 순서 번호 부여, 오름차순 요청 |

- 4조건이 동시에 성립할 때만 교착상태가 발생하므로, 예방 전략은 이 중 하나를 원천 불가능하게 만드는 것임

2) 교착상태 해결 전략 비교

| 구분 | 전략 | 동작 방식 | 오버헤드 | 자원 활용률 | 적용 환경 |
|------|------|-----------|----------|-------------|-----------|
| ① | 예방(Prevention) | 4조건 중 하나 원천 제거 | 낮음 | 낮음(과도한 제약) | 임베디드, 안전 임계 |
| ② | 회피(Avoidance) | Banker's Algorithm으로 요청 시마다 안전 상태 검사 | 중간 | 중간 | 자원 수 예측 가능한 시스템 |
| ③ | 탐지+복구(Detection+Recovery) | 자원 할당 그래프 사이클 탐지 후 프로세스 종료·자원 선점 | 낮음 | 높음 | 범용 DB, OS |
| ④ | 무시(Ignore) | 교착상태를 허용, 사용자가 수동 해결 | 없음 | 최고 | 단순 시스템(Ostrich Algorithm) |

- DB는 탐지+복구(트랜잭션 롤백), OS는 탐지+복구 또는 예방을 주로 채택함

3) Banker's Algorithm (은행원 알고리즘, 회피)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 안전 상태(Safe State) | 모든 프로세스가 최대 요청까지 순차 완료할 수 있는 안전 순서(Safe Sequence)가 존재하는 상태 |
| 동작 | 자원 요청 시 | 요청 허용 후 안전 상태인지 시뮬레이션 |
| 허용 | 안전 상태 유지 | 요청 허용 후 안전 순서가 존재 → 자원 할당 |
| 거부 | 불안전 상태 전이 | 요청 허용 후 안전 순서가 없음 → 요청 거부·대기 |
| 한계 | 사전 정보 요구 | 각 프로세스의 최대 자원 요구량을 미리 알아야 함 |

- 안전 순서 탐색: 가용 자원으로 완료 가능한 프로세스를 순차 실행하며 가용 자원을 누적하는 방식으로 안전 순서 유무를 판정함

4) Wait-Die vs Wound-Wait (DB 분산 트랜잭션 교착상태 회피)

| 구분 | Wait-Die | Wound-Wait |
|------|----------|------------|
| Old가 Young 점유 자원 요청 | Old 기다림(Wait) | Young을 강제 롤백(Wound) |
| Young이 Old 점유 자원 요청 | Young 롤백(Die) | Young 기다림(Wait) |
| 롤백 주체 | 항상 Young | 항상 Young |
| 특성 | 비선점(기다리다 포기) | 선점(빼앗음) |
| 우선순위 기준 | 타임스탬프 작을수록(오래된) 우선 | 타임스탬프 작을수록(오래된) 우선 |

- 두 프로토콜 모두 롤백 대상이 항상 Young 트랜잭션으로 일관되어 교착상태를 방지하며, 재시작 후 Old로 전환되어 결국 완료됨

### 3. 교착상태 해결 전략 총정리 및 현대 적용 트렌드

| 전략 | OS 적용 | DB 적용 | MSA 적용 |
|------|---------|---------|----------|
| 예방 | 자원 순서 번호(pthread mutex) | - | - |
| 회피 | Banker's(학문적) | - | - |
| 탐지+복구 | 주기적 그래프 탐지 | 트랜잭션 롤백 | Timeout+Retry |
| Wait-Die / Wound-Wait | - | 분산 트랜잭션 | - |
| 비동기 통신 | - | - | 이벤트 드리븐(Kafka) |

- 마이크로서비스 환경에서는 서비스 간 동기 호출의 원형 의존성이 분산 교착상태를 유발하므로, 타임아웃+재시도 또는 비동기 이벤트 드리븐 아키텍처로 근본적 해소가 트렌드임.  "끝"

### 실무 제언

**마이크로서비스의 분산 교착상태 해소**
- **챌린지**: MSA에서 서비스 A→B→A 원형 동기 호출(REST, gRPC)이 형성되면 단일 노드 교착상태와 동일하게 모든 서비스가 응답 불가 상태로 빠지며, 단일 서비스 장애가 전체 시스템으로 전파됨
- **제언**: 모든 동기 호출에 Timeout+Exponential Backoff Retry를 적용하고, 근본 해결로는 Kafka/RabbitMQ 기반 이벤트 드리븐 비동기 통신으로 원형 의존성 자체를 제거함

**관계형 DB의 교착상태 모니터링 및 자동 복구**
- **챌린지**: OLTP 환경에서 트랜잭션이 서로 다른 순서로 행 락을 획득하면 교착상태가 발생하고, 처리량이 높은 피크 타임에 교착상태 빈도가 폭증하여 응답 시간 SLA를 위반함
- **제언**: PostgreSQL의 `log_lock_waits`, MySQL의 `SHOW ENGINE INNODB STATUS`로 교착상태 패턴을 주기적으로 분석하고, 트랜잭션 내 자원 접근 순서를 자원 ID 오름차순으로 표준화하여 원형 대기 조건을 사전 제거함

**분산 DB의 Wait-Die 프로토콜 적용 사례**
- **챌린지**: Google Spanner, CockroachDB 등 분산 DB에서 여러 노드에 걸친 분산 트랜잭션이 네트워크 지연과 결합하여 교착상태 탐지가 수백 ms 지연되어 락 홀딩 시간이 길어짐
- **제언**: 타임스탬프 기반 Wait-Die 프로토콜을 채택하여 교착상태가 발생하기 전에 선제적으로 Young 트랜잭션을 롤백함으로써 탐지 지연을 우회하고, 롤백된 트랜잭션은 지수 백오프 후 재시도하여 결국 완료를 보장함
