## 해설

> **CPU라는 하나의 자원을 여러 프로세스가 공정하고 효율적으로 나누는 정책**  
> 선점/비선점 방식과 FCFS, SJF, Round-Robin, 다단계 큐 알고리즘

다중 프로그래밍 환경에서 CPU는 한 순간에 한 프로세스만 실행할 수 있습니다. 스케줄러(Scheduler)는 Ready Queue에서 다음에 실행할 프로세스를 선택하는 OS의 핵심 모듈이며, 선택 정책에 따라 시스템의 처리량, 응답 시간, 공정성이 크게 달라집니다.

### 1. 출제 배경 및 의도

스케줄링 알고리즘은 OS 도메인의 고전적이자 필수 주제입니다. 채점관은 **선점/비선점의 개념 차이**, **각 알고리즘의 원리와 장단점**, 그리고 **스케줄링 성능 지표(CPU 활용률, 처리량, 대기 시간, 반환 시간, 응답 시간)**를 정확히 이해하는지를 봅니다. 간단한 예시 계산(간트 차트)을 포함하면 이해도를 효과적으로 표현할 수 있습니다.

### 2. 개념의 본질과 작동 메커니즘

#### 스케줄링 성능 지표

| 지표 | 정의 | 목표 |
|---|---|---|
| **CPU 활용률** | CPU가 유효 작업을 수행하는 비율 | 최대화 |
| **처리량(Throughput)** | 단위 시간당 완료한 프로세스 수 | 최대화 |
| **대기 시간(Waiting Time)** | Ready Queue에서 기다린 총 시간 | 최소화 |
| **반환 시간(Turnaround Time)** | 제출~완료까지 총 소요 시간 | 최소화 |
| **응답 시간(Response Time)** | 요청 후 첫 응답까지 시간 | 최소화 |

#### 선점(Preemptive) vs 비선점(Non-preemptive) 방식

| 구분 | 비선점(Non-preemptive) | 선점(Preemptive) |
|---|---|---|
| **개념** | 프로세스가 자발적으로 CPU를 반납할 때까지 실행 | OS가 실행 중인 프로세스에서 강제로 CPU를 빼앗을 수 있음 |
| **문맥교환** | 적음 (오버헤드 낮음) | 많음 (오버헤드 높음) |
| **공정성** | 낮음 (긴 프로세스가 CPU 독점 가능) | 높음 |
| **응답성** | 낮음 | 높음 (시분할 시스템에 적합) |

#### 비선점 스케줄링 알고리즘

**① FCFS (First Come First Served)**
- 준비 큐에 도착한 순서대로 CPU 할당
- 구현 단순, 공정하지만 호위 효과(Convoy Effect) 발생 위험

```
프로세스: P1(24ms), P2(3ms), P3(3ms) 순서로 도착
간트 차트: [P1: 0-24] [P2: 24-27] [P3: 27-30]
평균 대기 시간: (0 + 24 + 27) / 3 = 17ms
```

- **호위 효과(Convoy Effect):** 실행 시간이 긴 프로세스 뒤에 짧은 프로세스들이 대기하는 현상

**② SJF (Shortest Job First) / SRTF (비선점)**
- 남은 실행 시간이 가장 짧은 프로세스 우선
- 이론적으로 평균 대기 시간 최소화 (증명됨)
- **단점:** 실행 시간 예측 어려움, 기아(Starvation) 현상 발생 가능

**③ 우선순위 스케줄링 (Priority, 비선점)**
- 각 프로세스에 우선순위 부여, 높은 우선순위 먼저 실행
- **기아(Starvation) 문제:** 낮은 우선순위 프로세스가 영원히 실행되지 못할 수 있음
- **해결책: 에이징(Aging)** — 대기 시간이 증가할수록 우선순위를 점진적으로 높임

#### 선점 스케줄링 알고리즘

**④ Round-Robin (RR)**
- 각 프로세스에 동일한 시간 할당량(Time Quantum, q)을 부여하고 순환
- 시간 할당량이 만료되면 강제로 CPU 빼앗아 큐 맨 뒤로 보냄

```
프로세스: P1(24ms), P2(3ms), P3(3ms), Time Quantum = 4ms
간트 차트: [P1:0-4][P2:4-7][P3:7-10][P1:10-14]...[P1:22-26]
```

- **Time Quantum 선택이 핵심:** 너무 크면 FCFS와 동일, 너무 작으면 컨텍스트 스위치 오버헤드 증가
- 응답 시간 보장 → 시분할(Time-sharing) 시스템의 표준

**⑤ SRTF (Shortest Remaining Time First, 선점형 SJF)**
- 실행 중 더 짧은 잔여 시간을 가진 프로세스가 도착하면 즉시 교체
- SJF보다 더 최적에 가깝지만 실행 시간 예측 필요

**⑥ 다단계 큐(Multi-level Queue)**

```
우선순위 높음
┌──────────────────────┐
│  시스템 프로세스      │ ← Round-Robin
├──────────────────────┤
│  대화형(Interactive) │ ← Round-Robin
├──────────────────────┤
│  배치(Batch) 처리    │ ← FCFS
└──────────────────────┘
우선순위 낮음
```

- 여러 Ready Queue를 계층화하여 각 큐마다 다른 알고리즘 적용
- 큐 간 이동 없음 → 기아 가능성

**⑦ 다단계 피드백 큐(Multi-level Feedback Queue, MLFQ)**
- 다단계 큐의 발전형
- 프로세스 행동에 따라 큐 간 이동 허용
- CPU 집중 프로세스는 낮은 우선순위 큐로 강등, 오래 대기하면 높은 큐로 승격
- 현대 OS(Linux CFS, Windows)의 실제 스케줄러가 이 방식 기반

### 3. 핵심 키워드 (Must-Have)

- **선점(Preemptive):** OS가 강제 CPU 회수 가능
- **비선점(Non-preemptive):** 프로세스가 자발적 반납
- **FCFS:** 도착 순서, 구현 단순, 호위 효과
- **SJF/SRTF:** 최단 작업 우선, 최적 평균 대기 시간, 기아 문제
- **Round-Robin:** 시간 할당량(Time Quantum), 공정성, 응답성
- **우선순위 스케줄링:** 기아 문제, 에이징(Aging)으로 해결
- **다단계 피드백 큐(MLFQ):** 현대 OS 스케줄러의 기반
- **반환 시간 = 완료 시간 - 도착 시간**
- **대기 시간 = 반환 시간 - CPU 사용 시간(버스트 시간)**

### 4. 맥락과 비교

| 알고리즘 | 유형 | 평균 대기 시간 | 기아 가능 | 구현 복잡도 |
|---|---|---|---|---|
| FCFS | 비선점 | 높음 | 없음 | 낮음 |
| SJF | 비선점 | 최소 (이론) | 있음 | 중간 |
| 우선순위 | 비선점/선점 | 중간 | 있음 | 중간 |
| Round-Robin | 선점 | 중간 | 없음 | 중간 |
| MLFQ | 선점 | 낮음 | 있음(에이징으로 해결) | 높음 |

### 5. 실무 제언

**Linux CFS(Completely Fair Scheduler)의 설계 철학**

- **챌린지:** 수백~수천 개의 스레드가 동시에 실행되는 현대 서버에서 전통적인 우선순위 큐 기반 스케줄러는 기아와 응답성 저하 문제를 겪습니다.
- **제언:** Linux 2.6.23부터 채택된 CFS는 **가상 실행 시간(Virtual Runtime, vruntime)**이라는 개념을 도입하여, 각 프로세스가 실제 실행된 시간을 추적하고 vruntime이 가장 작은(가장 덜 실행된) 프로세스를 다음 실행 대상으로 선택합니다. Red-Black Tree로 O(log N) 탐색을 구현하여 완전한 공정성과 O(1)에 근사한 성능을 동시에 달성합니다. 기술사 답안에서 현대 OS의 실제 구현까지 연결하면 실무 역량을 효과적으로 어필할 수 있습니다.
