## 해설

### 출제 배경 및 의도

다중 프로그래밍 환경에서 CPU는 한 순간에 한 프로세스만 실행할 수 있다. 스케줄러(Scheduler)는 Ready Queue에서 다음에 실행할 프로세스를 선택하는 OS의 핵심 모듈이며, 선택 정책에 따라 시스템의 처리량·응답 시간·공정성이 크게 달라진다.

스케줄링 알고리즘은 OS 도메인의 고전적이자 필수 주제다. 채점관은 선점/비선점의 개념 차이, 각 알고리즘의 원리와 장단점, 스케줄링 성능 지표(CPU 활용률·처리량·대기 시간·반환 시간·응답 시간)를 정확히 이해하는지를 본다. 간단한 간트 차트 계산 예시를 포함하면 이해도를 효과적으로 표현할 수 있다.

현대 OS(Linux CFS)에서 vruntime과 Red-Black Tree를 통해 완전한 공정성을 구현하는 방식까지 연결하면 실무 역량을 어필할 수 있다.

### 1. CPU 스케줄링(CPU Scheduling), 개요

정 의  • OS 스케줄러가 Ready Queue에서 다음에 CPU를 할당할 프로세스를 선택하는 정책으로, 시스템 자원의 효율과 사용자 응답성을 동시에 최적화하는 핵심 메커니즘
       - 선점(Preemptive)과 비선점(Non-preemptive) 방식으로 크게 구분

```
프로세스 생성
    │
    ▼
 New → Admitted → Ready Queue
                      │
              스케줄러 선택 (CPU 할당)
                      │
                      ▼
                  Running ──── I/O 요청 ──▶ Waiting
                      │                        │
              타임슬라이스 만료(선점)           I/O 완료
              또는 자발적 반납(비선점)           │
                      │                        ▼
                      ▼                    Ready Queue
                  Terminated
```

- Ready Queue의 어느 프로세스를 선택하느냐에 따라 평균 대기 시간, 응답 시간, CPU 활용률이 결정된다.

### 2. 선점/비선점 방식과 성능 지표

1) 선점 vs 비선점 비교

| 구분 | 비선점(Non-preemptive) | 선점(Preemptive) |
|------|------------------------|------------------|
| 개념 | 프로세스가 자발적으로 CPU 반납 시까지 실행 | OS가 실행 중인 프로세스에서 강제로 CPU 회수 |
| 문맥교환 | 적음(오버헤드 낮음) | 많음(오버헤드 높음) |
| 공정성 | 낮음(긴 프로세스 CPU 독점 가능) | 높음 |
| 응답성 | 낮음 | 높음(시분할 시스템에 적합) |

2) 스케줄링 성능 지표

| 지표 | 정의 | 목표 |
|------|------|------|
| CPU 활용률(Utilization) | CPU가 유효 작업을 수행하는 비율 | 최대화 |
| 처리량(Throughput) | 단위 시간당 완료한 프로세스 수 | 최대화 |
| 대기 시간(Waiting Time) | Ready Queue에서 기다린 총 시간 | 최소화 |
| 반환 시간(Turnaround Time) | 제출~완료까지 총 소요 시간 | 최소화 |
| 응답 시간(Response Time) | 요청 후 첫 응답까지 시간 | 최소화 |

- 반환 시간 = 완료 시간 - 도착 시간, 대기 시간 = 반환 시간 - CPU 버스트 시간

### 3. 주요 알고리즘 상세 및 현대 OS 비교

1) 비선점 알고리즘

| 알고리즘 | 원리 | 장점 | 단점 |
|----------|------|------|------|
| FCFS | 도착 순서대로 CPU 할당 | 구현 단순, 공정 | 호위 효과(Convoy Effect) |
| SJF | 가장 짧은 CPU 버스트 우선 | 평균 대기 시간 이론적 최소 | 기아(Starvation), 실행 시간 예측 어려움 |
| 우선순위 | 높은 우선순위 프로세스 우선 | 중요 작업 신속 처리 | 기아, 에이징(Aging)으로 해결 |

```
FCFS 예시: P1(24ms), P2(3ms), P3(3ms) 순서로 도착
간트 차트: [P1: 0-24] [P2: 24-27] [P3: 27-30]
평균 대기 시간: (0 + 24 + 27) / 3 = 17ms  ← 호위 효과 발생
```

2) 선점 알고리즘

| 알고리즘 | 원리 | 장점 | 단점 |
|----------|------|------|------|
| Round-Robin | 동일한 Time Quantum 순환 할당 | 공정성·응답성 보장, 기아 없음 | Quantum 크기 선택이 성능 결정 |
| SRTF | 잔여 시간 가장 짧은 프로세스 선점 | SJF보다 최적에 근접 | 실행 시간 예측 필요 |
| 다단계 큐 | 여러 Ready Queue 계층화 | 프로세스 유형별 정책 적용 | 큐 간 이동 없어 기아 가능 |
| MLFQ | 다단계 큐 + 큐 간 이동 | 동적 우선순위 조절, 기아 해소 | 구현 복잡 |

```
Round-Robin: P1(24ms), P2(3ms), P3(3ms), Time Quantum = 4ms
간트 차트: [P1:0-4][P2:4-7][P3:7-10][P1:10-14]...[P1:22-26]
Time Quantum이 너무 작으면 → 컨텍스트 스위치 오버헤드 증가
Time Quantum이 너무 크면 → FCFS와 동일 (응답성 저하)
```

3) 알고리즘 종합 비교

| 알고리즘 | 유형 | 평균 대기 시간 | 기아 가능 | 구현 복잡도 | 현대 OS |
|----------|------|----------------|-----------|-------------|---------|
| FCFS | 비선점 | 높음 | 없음 | 낮음 | - |
| SJF | 비선점 | 최소(이론) | 있음 | 중간 | - |
| 우선순위 | 비/선점 | 중간 | 있음 | 중간 | - |
| Round-Robin | 선점 | 중간 | 없음 | 중간 | 기반 |
| MLFQ | 선점 | 낮음 | 에이징 해소 | 높음 | Linux CFS, Windows |

- 현대 OS 스케줄러는 MLFQ의 발전형으로, Linux CFS는 vruntime과 Red-Black Tree를 통해 O(log N) 탐색으로 완전한 공정성을 구현한다.  "끝"

### 실무 제언

**Linux CFS(Completely Fair Scheduler) 설계 이해**
- **챌린지**: 수백~수천 개의 스레드가 동시에 실행되는 현대 서버에서 전통적인 우선순위 큐 기반 스케줄러는 기아와 응답성 저하 문제를 겪는다.
- **제언**: Linux 2.6.23부터 채택된 CFS는 vruntime(가상 실행 시간)을 추적하여 가장 덜 실행된 프로세스를 다음 실행 대상으로 선택하고, Red-Black Tree로 O(log N) 탐색을 구현한다. `schedtool`, `chrt`, `nice` 명령어로 스케줄링 정책과 우선순위를 조정할 수 있다.

**실시간 시스템에서 SCHED_FIFO/SCHED_RR 적용**
- **챌린지**: 산업용 제어 시스템, 오디오/비디오 처리 등 실시간 요구사항이 있는 워크로드는 일반 CFS 스케줄러로는 응답 시간 보장이 어렵다.
- **제언**: Linux의 SCHED_FIFO(비선점 실시간) 또는 SCHED_RR(선점 실시간) 정책을 `chrt -f -p 90 <pid>`로 적용하면 일반 프로세스보다 높은 우선순위로 실행된다. 단, 실시간 프로세스가 CPU를 독점하면 커널 자체가 응답 불능 상태가 될 수 있으므로 `ulimit -r`과 `cpu_rt_runtime` 설정으로 상한을 반드시 지정해야 한다.

**컨테이너 오케스트레이션에서 CPU 쿼터 관리**
- **챌린지**: Kubernetes 환경에서 CPU 리소스 설정 없이 다수의 컨테이너를 배포하면 일부 컨테이너가 CPU를 독점하여 전체 노드의 응답성이 저하된다.
- **제언**: Pod의 `resources.requests.cpu`와 `resources.limits.cpu`를 명시하고, cgroup v2의 cpu.weight와 cpu.max로 CFS 스케줄러 파라미터를 제어해야 한다. Requests는 스케줄링 가중치로, Limits는 CPU 쿼터 상한으로 작동하여 공정한 자원 배분을 보장한다.
