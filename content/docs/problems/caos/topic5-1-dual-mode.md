## 해설

### 출제 배경 및 의도

다중 프로그래밍 환경에서 여러 프로세스가 동시에 실행될 때, 악의적이거나 잘못된 프로세스가 하드웨어를 직접 제어하거나 다른 프로세스의 메모리를 침범하면 전체 시스템이 무너질 수 있다. 이중 모드(Dual Mode)는 이를 원천 방지하기 위해 CPU가 하드웨어 레벨에서 두 가지 실행 모드를 제공하는 구조다.

125회 응용 문제로 이중 모드 개념·구조·종류를 가·나·다 항목으로 나누어 물어봤다. 채점관은 왜 이중 모드가 필요한지(보호 목적), 모드 비트(Mode Bit)와 특권 명령어(Privileged Instruction)의 관계, 시스템 콜(System Call)을 통한 모드 전환 흐름을 논리적으로 서술할 수 있는지를 본다.

클라우드·컨테이너 환경에서 가상화 기술이 하이퍼바이저 모드(Ring -1)를 추가하는 배경, 그리고 seccomp·AppArmor 같은 현대적 보안 기법과의 연결까지 서술하면 실무 역량을 효과적으로 어필할 수 있다.

### 1. 이중 모드(Dual Mode), 개요

정 의  • CPU가 사용자 모드(User Mode)와 커널 모드(Kernel Mode) 두 가지 실행 환경을 하드웨어 수준에서 구별하여 사용자 프로세스가 특권 명령어를 직접 실행하지 못하도록 하는 보호 메커니즘
       - PSW(Program Status Word) 내 1비트 Mode Bit로 모드를 구분하며, 사용자 모드에서 특권 명령어 실행 시 트랩(Trap) 발생

```
사용자 프로그램                    운영체제 커널
┌─────────────────┐              ┌─────────────────┐
│  User Mode      │              │  Kernel Mode    │
│  Mode Bit = 1   │  시스템 콜   │  Mode Bit = 0   │
│                 │ ──────────▶ │                 │
│  일반 명령어만  │              │  특권 명령어    │
│  실행 가능      │  완료 후     │  모든 명령어    │
│  직접 하드웨어  │ ◀────────── │  하드웨어 직접  │
│  접근 불가      │              │  제어 가능      │
└─────────────────┘              └─────────────────┘
         └────────────────────────────┘
                  Hardware CPU (Mode Bit 관리)
```

- Mode Bit = 0(커널), Mode Bit = 1(사용자)로 하드웨어가 모드를 강제하여 사용자 프로그램의 특권 남용을 차단한다.

### 2. 이중 모드 구조 및 모드 전환

1) 특권 명령어(Privileged Instructions)

| 구분 | 명령어 예시 | 설명 |
|------|-------------|------|
| I/O 제어 | IN, OUT | 하드웨어 포트 직접 접근 |
| 인터럽트 관리 | STI, CLI | 인터럽트 활성화/비활성화 |
| 메모리 보호 | MMU 레지스터 설정 | 페이지 테이블 베이스 변경 |
| 타이머 설정 | 타이머 카운터 로드 | 스케줄러 타임슬라이스 제어 |
| 모드 전환 | SYSENTER, SYSEXIT | 커널↔사용자 전환 |

2) 시스템 콜(System Call) 모드 전환 흐름

```
사용자 프로그램 실행 (Mode Bit = 1)
        │
        ▼ 파일 읽기, 메모리 할당 등 OS 서비스 요청
  SYSCALL 명령어 실행 (INT 0x80 / x86-64: syscall)
        │
        ▼ 하드웨어가 Mode Bit = 0으로 전환
  커널의 시스템 콜 핸들러 실행
        │ 사용자 인자 검증, 서비스 처리
        ▼ 서비스 완료
  Mode Bit = 1로 복귀 (SYSRET)
        │
        ▼
  사용자 프로그램 실행 재개
```

- 시스템 콜은 사용자→커널→사용자 모드 전환과 컨텍스트 저장/복원이 수반되어 일반 함수 호출 대비 수십~수백 배 비용이 크다.

### 3. 이중 모드의 종류 및 현대적 확장

1) x86 보호 링(Protection Ring) 구조

| 링 레벨 | 권한 | 실제 사용 |
|---------|------|-----------|
| Ring 0 | 최대 권한, OS 커널 | Linux/Windows 커널 |
| Ring 1 | 장치 드라이버 일부 | OS/2 등 일부(현재 미사용) |
| Ring 2 | 장치 드라이버, OS 서비스 | 현재 대부분 미사용 |
| Ring 3 | 최소 권한, 사용자 애플리케이션 | 모든 사용자 프로세스 |

2) 가상화 환경의 확장 - Ring -1(하이퍼바이저)

```
┌─────────────────────────────────────────┐
│  Ring  3: 게스트 사용자 프로세스        │ ← 최소 권한
├─────────────────────────────────────────┤
│  Ring  0: 게스트 OS 커널 (VMX Non-root) │ ← 가상 머신 OS
├─────────────────────────────────────────┤
│  Ring -1: 하이퍼바이저 (VMX Root Mode)  │ ← VMware, KVM
└─────────────────────────────────────────┘
```

| 구분 | 이중 모드 | x86 보호 링 | 가상화 확장 |
|------|-----------|-------------|-------------|
| 모드 수 | 2(커널/사용자) | 4(Ring 0~3) | Ring -1 추가 |
| 구분 기준 | Mode Bit 1비트 | CPL 2비트 | VMX Root/Non-root |
| 실제 OS 활용 | 대부분의 OS | Ring 0, Ring 3만 | Intel VT-x, AMD-V |
| 격리 수준 | 프로세스↔커널 | 계층적 권한 | VM↔하이퍼바이저 |

- 현대 OS(Linux, Windows)는 주로 Ring 0과 Ring 3만 사용하는 이중 모드를 유지하며, 가상화 환경에서는 Intel VT-x·AMD-V가 Ring -1을 추가하여 하이퍼바이저를 지원한다.  "끝"

### 실무 제언

**컨테이너 보안에서의 이중 모드 중요성**
- **챌린지**: 도커(Docker) 컨테이너는 VM과 달리 게스트 OS 없이 호스트 OS 커널을 공유한다. 컨테이너 내부 프로세스는 사용자 모드에서 실행되지만, 커널 취약점(Dirty COW, Escape)이 발생하면 커널 모드 탈취가 가능하여 호스트 전체가 위험에 노출된다.
- **제언**: 컨테이너 런타임(containerd, CRI-O)은 seccomp(Secure Computing Mode)를 통해 컨테이너가 사용할 수 있는 시스템 콜 목록을 화이트리스트로 제한하고, AppArmor/SELinux로 커널 자원 접근을 추가 제한해야 한다.

**시스템 콜 성능 최적화(vDSO)**
- **챌린지**: gettimeofday, clock_gettime 같이 자주 호출되는 시스템 콜도 모드 전환 비용이 발생하면 고빈도 호출 환경에서 누적 오버헤드가 커진다.
- **제언**: Linux의 vDSO(virtual Dynamic Shared Object) 메커니즘은 일부 시스템 콜을 커널 모드 전환 없이 사용자 공간에서 처리한다. `strace -c`로 시스템 콜 빈도와 소요 시간을 측정하고, 빈도가 높은 경로에 io_uring이나 epoll 같은 배치 처리 인터페이스를 적용하여 모드 전환 횟수를 줄여야 한다.

**하이퍼바이저 취약점과 VM Escape 방어**
- **챌린지**: Ring -1 하이퍼바이저 레이어에 취약점이 있으면 게스트 VM에서 호스트 전체를 탈취하는 VM Escape가 가능하다. VENOM, Spectre-v2 등이 대표적 사례다.
- **제언**: 클라우드 인프라에서 하이퍼바이저를 정기적으로 패치하고, 민감 워크로드는 Confidential Computing(AMD SEV, Intel TDX)을 적용하여 하이퍼바이저조차 VM 내부 메모리에 접근하지 못하도록 하드웨어 수준에서 격리해야 한다.
