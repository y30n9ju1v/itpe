운영체제 다중 프로그래밍 환경에서 이중 모드(Dual mode) 구조에 대하여 다음을 설명하시오.
가. 이중모드의 개념

다. 이중모드의 종류

## 해설

> **OS 보안의 첫 번째 방어선: 사용자는 직접 하드웨어를 건드릴 수 없다**  
> 이중 모드로 커널과 사용자 공간을 분리하는 보호 메커니즘

다중 프로그래밍 환경에서 여러 프로세스가 동시에 실행될 때, 악의적이거나 잘못된 프로세스가 하드웨어를 직접 제어하거나 다른 프로세스의 메모리를 침범하면 전체 시스템이 무너질 수 있습니다. **이중 모드(Dual Mode)**는 이를 원천 방지하기 위해 CPU가 하드웨어 레벨에서 두 가지 실행 모드를 제공하는 구조입니다.

### 1. 출제 배경 및 의도

125회 응용 문제로 이중 모드 개념, 구조, 종류를 가, 나, 다 항목으로 나누어 물어봤습니다. 채점관은 **왜 이중 모드가 필요한지(보호 목적)**, **모드 비트(Mode Bit)와 특권 명령어(Privileged Instruction)의 관계**, **시스템 콜(System Call)을 통한 모드 전환 흐름**을 논리적으로 서술할 수 있는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 이중 모드의 개념

이중 모드란 CPU가 **사용자 모드(User Mode)**와 **커널 모드(Kernel Mode, Supervisor Mode)** 두 가지 실행 환경을 하드웨어 수준에서 구별하는 보호 메커니즘입니다.

**모드 비트(Mode Bit):**
- CPU의 상태 레지스터(PSW, Program Status Word)에 1비트 포함
- **0 = 커널 모드 (특권 모드):** 모든 명령어 실행 가능
- **1 = 사용자 모드 (비특권 모드):** 제한된 명령어만 실행 가능

**도입 배경:**  
단일 모드 시스템에서는 사용자 프로그램이 하드웨어 I/O 포트를 직접 접근하거나, 다른 프로세스의 메모리 영역을 읽고 쓰고, 인터럽트를 비활성화하여 스케줄러를 우회하는 등의 위험한 행위가 가능했습니다. 이중 모드는 이러한 위협을 하드웨어 수준에서 차단합니다.

#### 나. 이중 모드의 구조

```
사용자 프로그램                  운영체제 커널
┌─────────────────┐            ┌─────────────────┐
│  User Mode (1)  │            │ Kernel Mode (0)  │
│                 │            │                  │
│ 일반 명령어만   │  시스템 콜 │ 특권 명령어 포함 │
│ 실행 가능       │ ────────▶ │ 모든 명령어      │
│                 │            │ 실행 가능        │
│ 직접 하드웨어   │            │ 하드웨어 직접    │
│ 접근 불가       │            │ 제어 가능        │
└─────────────────┘            └─────────────────┘
        │                              │
        └──────── 하드웨어 CPU ─────────┘
                  (Mode Bit 관리)
```

**모드 전환(Mode Transition) 과정:**

```
사용자 프로그램 → 시스템 콜 발행 (INT 0x80 / SYSCALL 명령어)
                        │
                        ▼
              CPU Mode Bit = 0 (커널 모드로 전환)
                        │
                        ▼
              커널의 시스템 콜 핸들러 실행
                        │
                        ▼
              서비스 완료 후 Mode Bit = 1 (사용자 모드 복귀)
                        │
                        ▼
              사용자 프로그램 실행 재개
```

**특권 명령어(Privileged Instructions):**  
커널 모드에서만 실행 가능한 명령어 목록:
- I/O 명령어 (포트 직접 접근)
- 인터럽트 활성화/비활성화 (STI, CLI)
- 메모리 보호 레지스터 설정 (MMU 제어)
- 타이머 설정
- 프로세서 모드 전환 명령어 자체

사용자 모드에서 특권 명령어 실행 시도 → **트랩(Trap, 예외)** 발생 → OS가 프로세스 강제 종료 또는 오류 처리

#### 다. 이중 모드의 종류

현대 시스템은 이중 모드를 넘어 더 세분화된 보호 링(Protection Ring) 구조를 제공합니다.

**① 이중 모드 (Dual Mode) - 기본 구조**

| 모드 | Mode Bit | 권한 |
|---|---|---|
| **커널 모드 (Ring 0)** | 0 | 모든 명령어, 모든 자원 접근 |
| **사용자 모드 (Ring 3)** | 1 | 제한된 명령어, OS 서비스는 시스템 콜로만 |

**② x86 보호 링(Protection Ring) 구조**

```
┌─────────────────────────────────────────┐
│  Ring 3: 사용자 애플리케이션           │ ← 최소 권한
├─────────────────────────────────────────┤
│  Ring 2: 장치 드라이버 (일부)          │
├─────────────────────────────────────────┤
│  Ring 1: 장치 드라이버, OS 서비스 일부 │
├─────────────────────────────────────────┤
│  Ring 0: 커널 (OS 핵심)               │ ← 최대 권한
└─────────────────────────────────────────┘
```

- 실제 현대 OS(Linux, Windows)는 주로 Ring 0과 Ring 3만 사용 (이중 모드)
- Ring 1, 2는 과거 OS/2나 특수 목적에서 활용

**③ 가상화 환경의 확장: Ring -1 (하이퍼바이저)**

```
Ring -1: 하이퍼바이저 (VMX Root Mode)  ← VMware, KVM
Ring  0: 게스트 OS 커널 (VMX Non-root) ← 가상 머신의 OS
Ring  3: 게스트 사용자 프로세스
```

Intel VT-x, AMD-V 기술이 하이퍼바이저를 위한 새로운 특권 레벨을 추가합니다.

### 3. 핵심 키워드 (Must-Have)

- **이중 모드(Dual Mode):** 커널 모드 + 사용자 모드
- **모드 비트(Mode Bit):** PSW(Program Status Word) 내 1비트
- **특권 명령어(Privileged Instruction):** 커널 모드에서만 실행 가능
- **시스템 콜(System Call):** 사용자 모드 → 커널 모드 전환 메커니즘
- **트랩(Trap):** 특권 명령어 위반 또는 시스템 콜 시 발생하는 소프트웨어 인터럽트
- **보호 링(Protection Ring):** x86의 Ring 0~3 세분화 구조
- **하이퍼바이저 모드 (Ring -1):** 가상화를 위한 추가 특권 레벨

### 4. 맥락과 비교

**이중 모드 vs 보호 링 비교:**

| 구분 | 이중 모드 | x86 보호 링 |
|---|---|---|
| **모드 수** | 2 (커널/사용자) | 4 (Ring 0~3) |
| **구분 기준** | Mode Bit 1비트 | CPL(Current Privilege Level) 2비트 |
| **실제 OS 활용** | 대부분의 OS | Ring 0, Ring 3만 주로 사용 |
| **가상화 확장** | 없음 | Ring -1 (VT-x, AMD-V) |

**모드 전환 비용:**  
시스템 콜은 사용자→커널→사용자 모드 전환과 컨텍스트 정보 저장/복원이 수반되므로 일반 함수 호출보다 수십~수백 배 비용이 큽니다. 이 때문에 vDSO(virtual Dynamic Shared Object)와 같이 자주 호출되는 시스템 콜(gettimeofday 등)을 커널 모드 전환 없이 처리하는 최적화 기법이 존재합니다.

### 5. 실무 제언

**컨테이너 보안에서의 이중 모드 중요성**

- **챌린지:** 도커(Docker) 컨테이너는 VM과 달리 **게스트 OS 없이 호스트 OS 커널을 공유**합니다. 컨테이너 내부 프로세스는 사용자 모드에서 실행되지만, 커널 취약점(예: Dirty COW, Escape)이 발생하면 커널 모드 탈취가 가능하여 호스트 전체가 위험에 노출됩니다.
- **제언:** 컨테이너 런타임(containerd, CRI-O)은 **seccomp(Secure Computing Mode)**를 통해 컨테이너가 사용할 수 있는 시스템 콜 목록을 화이트리스트로 제한합니다. 또한 **AppArmor/SELinux**로 커널 자원 접근을 추가 제한합니다. 이중 모드의 원리를 이해해야 컨테이너 보안 정책을 올바르게 설계할 수 있습니다.
