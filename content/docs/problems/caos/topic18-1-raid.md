## 문제

RAID(Redundant Array of Independent/Inexpensive Disks) Level 중에 Raid 0, Raid 5, Raid 6, Raid 1+0, Raid 5+0에 대해 Disk 필요 수량, 공간 효율, 고장 허용 범위를 포함해 설명하시오.

## 해설

> **여러 디스크를 묶어 하나처럼 사용하면서 성능과 내구성을 동시에 높이는 기술**  
> RAID 0의 스트라이핑, RAID 5/6의 패리티, RAID 10/50의 중첩 구성

RAID(Redundant Array of Independent Disks)는 여러 개의 물리 디스크를 하나의 논리 드라이브로 구성하여 **성능 향상, 용량 증대, 고장 내성(Fault Tolerance)** 중 하나 이상을 달성하는 저장 기술입니다.

### 1. 출제 배경 및 의도

RAID는 서버/스토리지 인프라의 기본 구성 요소입니다. 채점관은 **각 RAID 레벨의 구성 원리(스트라이핑, 미러링, 패리티)**, **디스크 수, 공간 효율, 고장 허용 개수** 계산, 그리고 **적합한 사용 사례**를 비교 표 형태로 명확히 제시하기를 기대합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 핵심 원리

- **스트라이핑(Striping):** 데이터를 여러 디스크에 분산 저장 → 읽기/쓰기 병렬화 → 성능 향상
- **미러링(Mirroring):** 데이터를 두 개 이상의 디스크에 동일하게 복사 → 가용성 향상
- **패리티(Parity):** XOR 연산으로 생성한 오류 정정 정보 → 최소 공간으로 고장 내성

#### RAID 0 (스트라이핑, Striping)

```
데이터: [A1][A2][A3][A4]

디스크 0: [A1][A3]
디스크 1: [A2][A4]
```

| 항목 | 값 |
|---|---|
| **최소 디스크 수** | 2개 |
| **공간 효율** | 100% (패리티 없음) |
| **고장 허용** | 0개 (디스크 1개 장애 시 전체 데이터 손실) |
| **읽기 성능** | n배 향상 (n: 디스크 수) |
| **쓰기 성능** | n배 향상 |
| **용도** | 고성능 비임계 데이터 (영상 편집 작업 파일, 임시 데이터) |

#### RAID 5 (분산 패리티, Distributed Parity)

```
디스크 0: [A1][B1][C1][P(D)]
디스크 1: [A2][B2][P(C)][D1]
디스크 2: [A3][P(B)][C2][D2]
디스크 3: [P(A)][B3][C3][D3]
```

패리티(P)가 특정 디스크에 집중되지 않고 순환 배치 → 패리티 디스크 병목 제거

| 항목 | 값 |
|---|---|
| **최소 디스크 수** | 3개 |
| **공간 효율** | (n-1)/n (3개: 66.7%, 4개: 75%, 6개: 83.3%) |
| **고장 허용** | 1개 (동시 2개 장애 시 데이터 손실) |
| **읽기 성능** | 양호 (n-1 디스크에서 데이터 읽기) |
| **쓰기 성능** | 패리티 계산 오버헤드로 다소 느림 (Write Penalty) |
| **용도** | 일반 서버, NAS, 파일 서버 (중간 규모 스토리지 표준) |

#### RAID 6 (이중 분산 패리티, Dual Parity)

패리티를 2개 유지하여 동시 2개 디스크 고장 허용

| 항목 | 값 |
|---|---|
| **최소 디스크 수** | 4개 |
| **공간 효율** | (n-2)/n (4개: 50%, 6개: 66.7%, 8개: 75%) |
| **고장 허용** | 2개 (동시 2개 장애까지 안전) |
| **읽기 성능** | RAID 5와 유사 |
| **쓰기 성능** | 패리티 계산 2배로 RAID 5보다 느림 |
| **용도** | 대용량 고가용성 스토리지, 엔터프라이즈 NAS (디스크 수 많을수록 적합) |

#### RAID 1+0 (RAID 10, 미러링+스트라이핑)

RAID 0과 RAID 1을 결합: 먼저 미러링(RAID 1) 후 스트라이핑(RAID 0)

```
[미러쌍1: 디스크0+디스크1] ─ 스트라이프 ─ [미러쌍2: 디스크2+디스크3]
디스크 0: [A1][A3]    디스크 2: [A2][A4]
디스크 1: [A1][A3]    디스크 3: [A2][A4]
(미러)                  (미러)
```

| 항목 | 값 |
|---|---|
| **최소 디스크 수** | 4개 (짝수) |
| **공간 효율** | 50% |
| **고장 허용** | 각 미러쌍에서 1개씩, 최대 n/2개 (같은 미러쌍에서 2개 동시 고장 시 데이터 손실) |
| **읽기 성능** | 최고 수준 |
| **쓰기 성능** | 미러링 오버헤드만 있어 RAID 5보다 빠름 |
| **용도** | 고성능 + 고가용성이 모두 필요한 DB 서버, 트랜잭션 시스템 |

#### RAID 5+0 (RAID 50, 패리티+스트라이핑)

RAID 5 그룹 여러 개를 RAID 0으로 스트라이핑

```
[RAID 5 그룹1: 디스크0-2] ─ RAID 0 스트라이프 ─ [RAID 5 그룹2: 디스크3-5]
```

| 항목 | 값 |
|---|---|
| **최소 디스크 수** | 6개 (RAID 5 그룹 2개 × 3개) |
| **공간 효율** | 각 그룹의 (n-1)/n 효율 유지 |
| **고장 허용** | 각 RAID 5 그룹에서 1개씩 (그룹당 최대 1개) |
| **읽기 성능** | RAID 5보다 크게 향상 (두 그룹 병렬) |
| **쓰기 성능** | RAID 5보다 향상 |
| **용도** | 대용량 고성능 스토리지, 빅데이터 플랫폼 |

#### RAID 레벨 종합 비교표

| RAID | 최소 디스크 | 공간 효율 | 고장 허용 | 읽기 성능 | 쓰기 성능 | 주요 용도 |
|---|---|---|---|---|---|---|
| **RAID 0** | 2 | 100% | 0개 | ★★★★★ | ★★★★★ | 임시/비임계 고속 |
| **RAID 5** | 3 | (n-1)/n | 1개 | ★★★★ | ★★★ | 일반 파일 서버 |
| **RAID 6** | 4 | (n-2)/n | 2개 | ★★★★ | ★★ | 대용량 고가용성 |
| **RAID 1+0** | 4 | 50% | n/2개 | ★★★★★ | ★★★★ | DB/트랜잭션 |
| **RAID 5+0** | 6 | ~(n-그룹수)/n | 그룹당 1개 | ★★★★★ | ★★★★ | 대용량 고성능 |

### 3. 핵심 키워드 (Must-Have)

- **스트라이핑:** 데이터 분산, 성능 향상, 고장 내성 없음
- **미러링:** 동일 복사, 50% 공간 효율, 높은 고장 내성
- **패리티(XOR):** 최소 공간으로 고장 내성 (RAID 5: 1개, RAID 6: 2개)
- **RAID 0:** 성능 최우선, 내구성 없음
- **RAID 5:** 균형 (성능+가용성), 일반 서버 표준
- **RAID 6:** RAID 5의 이중 패리티 강화판
- **RAID 1+0:** 성능+가용성 최고, 비용 높음
- **Write Penalty:** RAID 5/6에서 패리티 계산으로 인한 쓰기 오버헤드

### 4. 맥락과 비교

**대용량 디스크 시대의 RAID 5 위험성:**

현대 16TB+ HDD는 URE(Unrecoverable Read Error) 발생 확률이 1/10^14 비트입니다. RAID 5 재구성 시 전체 어레이를 읽으므로 14TB 어레이 재구성 중 URE 발생 확률이 상당합니다. 이 때문에 대용량 어레이에서는 RAID 6 또는 ZFS의 RAIDZ2 사용이 권장됩니다.

### 5. 실무 제언

**클라우드 네이티브 환경에서 RAID의 역할 변화**

- **챌린지:** AWS EBS, Azure Disk, GCP Persistent Disk 등 클라우드 블록 스토리지는 내부적으로 RAID와 유사한 복제 메커니즘을 제공합니다. VM 레벨에서 추가 RAID를 구성하면 이중 보호가 되지만, 성능 오버헤드만 이중으로 발생합니다.
- **제언:** 클라우드 환경에서는 스토리지 서비스의 내장 복제(예: EBS의 AZ 내 자동 복제) + 스냅샷 정책으로 RAID 역할을 대체하는 것이 효율적입니다. 온프레미스 환경에서는 RAID 컨트롤러의 배터리 백업 캐시(BBWC)를 활성화하여 Write Back 모드로 RAID 5/6의 Write Penalty를 최소화해야 합니다.
