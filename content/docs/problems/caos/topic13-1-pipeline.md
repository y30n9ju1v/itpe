## 해설

> **공장 조립 라인처럼 명령어를 동시에 처리하는 CPU의 성능 혁신**  
> 파이프라인 처리 시간 공식, 구조적/데이터/제어 해저드 해결 전략

자동차 공장의 조립 라인처럼, CPU도 명령어 처리를 여러 단계로 나누어 동시에 진행합니다. 이것이 **파이프라인(Pipeline)** 기법입니다. 이를 통해 CPU는 매 클럭마다 하나의 명령어를 완료하는 이상적 처리량(throughput)을 달성합니다.

### 1. 출제 배경 및 의도

컴퓨터 아키텍처의 성능 최적화 핵심 기법입니다. 채점관은 **수식을 포함한 처리 시간 계산**, **3가지 해저드 유형의 명확한 구분**, 그리고 **각 해저드별 해결 방안**을 정확히 서술하기를 기대합니다. 현대 CPU의 Out-of-Order Execution까지 언급하면 심화 역량을 어필할 수 있습니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 파이프라인 vs 비파이프라인 처리 시간

**명령어 처리 5단계 (고전적 RISC 파이프라인):**

```
IF (Instruction Fetch) → ID (Instruction Decode) → EX (Execute) → MEM (Memory) → WB (Write Back)
```

**비파이프라인 방식:**
- 한 명령어가 모든 단계를 순차적으로 완료한 후 다음 명령어 시작
- n개 명령어, m단계, 각 단계 처리 시간 k:
  
$$T_{비파이프라인} = n \times m \times k$$

**파이프라인 방식:**

```
시간:  1    2    3    4    5    6    7    8    9
명령1: IF → ID → EX → MEM → WB
명령2:      IF → ID → EX → MEM → WB
명령3:           IF → ID → EX → MEM → WB
...
```

- 첫 명령어 완료: m × k
- 이후 매 k마다 명령어 완료
- n개 명령어 총 시간:

$$T_{파이프라인} = (m + n - 1) \times k$$

**파이프라인 속도 향상(Speedup):**

$$S = \frac{T_{비파이프라인}}{T_{파이프라인}} = \frac{n \times m \times k}{(m + n - 1) \times k} = \frac{n \times m}{m + n - 1}$$

**n → ∞ (충분히 많은 명령어)일 때:**

$$S_{이론적} = m \text{ (파이프라인 단계 수만큼 향상)}$$

**예시 계산:** m=5, k=1ns, n=100개

- 비파이프라인: 100 × 5 × 1 = **500ns**
- 파이프라인: (5 + 100 - 1) × 1 = **104ns**
- 속도 향상: 500/104 ≈ **4.8배**

#### 나. 파이프라인 해저드(Hazard) 3가지 유형과 해결방안

파이프라인이 항상 이상적으로 동작하지 않는 상황을 **해저드(Hazard)**라고 합니다. 해저드 발생 시 파이프라인을 일시 정지(Stall, Bubble 삽입)해야 합니다.

**① 구조적 해저드 (Structural Hazard)**

**원인:** 두 명령어가 같은 하드웨어 자원을 동시에 사용하려는 충돌

```
예: 단일 메모리 포트 시스템
명령1: IF 단계 → 메모리에서 명령어 읽기
명령3: MEM 단계 → 메모리에서 데이터 읽기 ← 충돌!
```

**해결방안:**
1. **하드웨어 자원 복제:** 명령어 캐시(I-Cache)와 데이터 캐시(D-Cache)를 분리 (하버드 아키텍처 방식)
2. **파이프라인 스톨:** 충돌 발생 시 한 명령어를 대기시킴 (성능 저하)

**② 데이터 해저드 (Data Hazard)**

**원인:** 이전 명령어의 결과를 다음 명령어가 사용하는 데이터 의존성

```
ADD R1, R2, R3    ← R1에 결과 쓰기 (WB 단계)
SUB R4, R1, R5   ← R1을 읽음 (ID 단계) → WB 전에 읽으면 오래된 값!

시간:   1    2    3    4    5    6    7    8    9
ADD:   IF → ID → EX → MEM → WB
SUB:        IF → ID → EX → ...
                  ↑
                  여기서 R1 읽음 (ADD의 EX 단계, 아직 결과 없음)
```

**데이터 해저드 유형:**
- **RAW (Read After Write):** 쓰기 전에 읽기 — 가장 일반적, 진짜 의존성
- **WAR (Write After Read):** 읽기 전에 쓰기 — 역의존성 (파이프라인에서 드물게 발생)
- **WAW (Write After Write):** 쓰기 후 쓰기 — 출력 의존성

**해결방안:**
1. **포워딩(Forwarding, Bypassing):** EX 단계 결과를 WB를 기다리지 않고 직접 다음 명령어의 EX 입력으로 전달 (하드웨어 추가)
   ```
   ADD: EX 결과 → [포워딩 경로] → SUB의 EX 입력
   ```
2. **파이프라인 스톨(Stall):** 의존성 해소될 때까지 NOP(버블) 삽입
3. **코드 재순서(Load-Use Hazard 등):** 컴파일러가 독립적인 명령어로 의존성 간격을 채움

**③ 제어 해저드 (Control Hazard, 분기 해저드)**

**원인:** 분기(Branch) 명령어의 목적지를 결정하기 전에 다음 명령어를 인출하는 문제

```
BEQ R1, R2, LABEL   ← 분기 여부가 EX에서 결정됨
다음 명령어1 (IF)    ← 분기 미결정 상태에서 인출
다음 명령어2 (없음)
...
LABEL:               ← 실제로 가야 할 목적지
```

**해결방안:**
1. **파이프라인 플러시(Flush):** 분기 결정 후 잘못 인출된 명령어를 버림 (분기 패널티 발생)
2. **분기 예측(Branch Prediction):**
   - **정적 예측:** 항상 분기 안 함(Not Taken) 또는 분기(Taken) 가정
   - **동적 예측:** BHT(Branch History Table)로 과거 패턴 기반 예측 (현대 CPU 채택)
   - 현대 CPU 예측 정확도: 95~99%
3. **지연 분기(Delayed Branch):** 분기 명령어 다음 슬롯에 항상 실행될 명령어를 컴파일러가 배치 (SPARC, MIPS)
4. **투기적 실행(Speculative Execution):** 예측한 경로로 미리 실행, 예측 실패 시 취소

### 3. 핵심 키워드 (Must-Have)

- **파이프라인 5단계:** IF → ID → EX → MEM → WB
- **비파이프라인 처리 시간:** n × m × k
- **파이프라인 처리 시간:** (m + n - 1) × k
- **이론적 속도 향상:** m배
- **구조적 해저드:** 하드웨어 자원 충돌, I/D 캐시 분리로 해결
- **데이터 해저드(RAW/WAR/WAW):** 데이터 의존성, 포워딩으로 해결
- **제어 해저드:** 분기 명령어, 분기 예측으로 해결
- **파이프라인 스톨(Stall, Bubble):** 해저드 발생 시 성능 저하
- **포워딩(Forwarding):** 데이터 해저드 하드웨어 해결책
- **투기적 실행(Speculative Execution):** 제어 해저드 현대적 해결책

### 4. 맥락과 비교

**파이프라인 단계 수와 클럭 속도의 트레이드오프:**

| 구분 | 얕은 파이프라인 | 깊은 파이프라인 |
|---|---|---|
| **단계 수** | 5~10단계 | 20~31단계 (Pentium 4: 31단계) |
| **클럭 속도** | 낮음 | 높음 (각 단계를 단순화) |
| **분기 패널티** | 작음 | 큼 (예측 실패 시 더 많은 스톨) |
| **해저드 위험** | 낮음 | 높음 |

### 5. 실무 제언

**투기적 실행(Speculative Execution)과 스펙터(Spectre) 취약점**

- **챌린지:** 현대 CPU의 투기적 실행과 Out-of-Order Execution 최적화는 2018년 발견된 **스펙터(Spectre)/멜트다운(Meltdown)** 보안 취약점의 근본 원인이 되었습니다. CPU가 투기적으로 실행한 코드가 남긴 캐시 측면 채널(Side Channel)을 이용해 다른 프로세스의 메모리를 읽을 수 있는 치명적 취약점이었습니다.
- **제언:** OS 패치(KPTI, Kernel Page Table Isolation)와 마이크로코드 업데이트로 일부 완화되었지만 성능 저하(5~30%)를 수반합니다. 파이프라인 최적화와 보안의 트레이드오프는 하드웨어 설계에서 근본적으로 해결이 필요한 난제로, ARM의 새로운 아키텍처나 RISC-V 구현에서 이를 설계 초기부터 고려하는 방향으로 발전하고 있습니다.
