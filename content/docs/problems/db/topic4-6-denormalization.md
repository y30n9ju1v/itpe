## 해설

### 출제 배경 및 의도

데이터베이스 설계 아키텍처에서 정규화(Normalization)와 역정규화(Denormalization)는 어느 한쪽도 버릴 수 없는 동전의 양면입니다. 정규화를 완벽히 수행하면 데이터 무결성은 극대화되지만 테이블이 잘게 쪼개져 대규모 조인 부하가 발생하고, 역정규화는 성능을 향상시키는 대신 데이터 불일치 위험을 수반합니다. 현대 MSA·클라우드 환경에서 대규모 실시간 트래픽을 처리하면서도 데이터 품질을 유지하려면 통제된 방식의 역정규화를 적용하는 데이터 아키텍트(DA)로서의 통합 조율 능력이 요구됩니다.

채점관은 두 개념의 선후 관계와 목적의 상반성을 명확히 설명하고, 역정규화의 세부 기법(테이블 병합/분할, 컬럼 중복 추가, 통계 테이블 등)과 부작용 방지 방안까지 서술하는 답안을 고득점으로 평가합니다.

### 1. 정규화와 역정규화, 개요

정 의  • 함수 종속성(FD) 분석으로 데이터 중복을 제거하고 무결성을 높이는 정규화와, 시스템 수행 속도 향상을 위해 의도적으로 중복을 허용하고 구조를 단순화하는 역정규화의 상호 보완적 엔지니어링 활동
       - "선(先)정규화 후(後)조율" 원칙: 논리 모델에서 무결성 확보 후 물리 설계에서 성능 최적화

```
[논리 설계 단계]                          [물리 및 운영 단계]
┌────────────────┐  함수종속성 분해(무결성) ┌────────────────┐
│ 원시 데이터    │ ──────────────────────> │ 정규화 모델    │ (3NF / BCNF)
└────────────────┘                         └───────┬────────┘
                                                   │
                                      트랜잭션 부하 및 조인 병목 분석
                                      CRUD 매트릭스 검증 → 성능 한계 도출
                                                   ▼
                                         ┌──────────────────┐
                                         │ 역정규화 모델    │ (고성능 스키마)
                                         └──────────────────┘
```

- 역정규화는 정규화가 완료된 이후에만 수행하며, 인덱스 조정·캐싱·파티셔닝을 우선 검토한 후 최후 수단으로 적용한다.

### 2. 정규화 단계별 전개 및 역정규화 기법 상세

1) 정규화 단계별 전개 (1NF → BCNF)

| 정규형 | 제거 대상 | 수행 내용 | 표준 적용 범위 |
|--------|-----------|-----------|----------------|
| **1NF** | 다중값 속성 | 원자값(Atomic Value)만 존재하도록 분해 | 필수 |
| **2NF** | 부분 함수 종속(PFD) | 복합키 일부에 종속된 일반 속성 분리 → 완전 FD | OLTP 표준 |
| **3NF** | 이행적 함수 종속(TFD) | 일반 속성 간 결정 관계 → 마스터/코드 테이블 분리 | OLTP 표준 |
| **BCNF** | 후보키 아닌 결정자 | 일반 속성이 PK 일부 결정 → 결정자 PK 승격 분리 | 권장 목표 |

- 일반적인 OLTP 시스템에서는 3NF 또는 BCNF까지 전개하는 것이 표준이며, 이를 기반으로 역정규화를 적용한다.

2) 역정규화 3단계 프로세스 및 기법

```
[1단계: 대상 조사] ──> [2단계: 대안 검토] ──> [3단계: 기법 적용]
 트랜잭션 빈도 분석     인덱스 최적화 선행    테이블 통합/분할
 성능 병목 쿼리 식별    애플리케이션 캐시     중복 컬럼/통계 추가
 CRUD 매트릭스 검증     파티셔닝 우선 고려    무결성 보완 대책 마련
```

| 기법 분류 | 세부 기법 | 설명 | 효과 |
|-----------|-----------|------|------|
| **테이블 역정규화** | 테이블 병합(1:1/1:M) | 항상 함께 조회되는 두 테이블 통합 | 단일 스캔으로 I/O 감소 |
| | 수직 분할(Vertical Split) | BLOB 등 대용량 컬럼 별도 테이블 분리 | 기본 컬럼 블록 집적도 향상 |
| | 수평 분할(Partitioning) | 기간/리전 단위 파티션 분할 | 디스크 탐색 범위 제한 |
| **컬럼 역정규화** | 중복 컬럼 추가 | 자식에 부모 속성 복사 저장 | 조인 제거 |
| | 파생/계산 컬럼 추가 | 합계·평균 등 집계값 사전 저장 | 실시간 연산 부하 제거 |
| **관계 역정규화** | 중복 관계 추가 | A→B→C 다단계 조인에 A→C 직접 FK 추가 | 다단계 조인 단축 |

- 역정규화는 성능 향상을 위해 무결성 위험을 매수한 행위이므로, 애플리케이션 레이어 트랜잭션 통제·DBMS 트리거·야간 배치 정합성 검증이 반드시 동반되어야 한다.

### 3. 정규화와 역정규화 비교 및 트렌드

| 비교 항목 | 정규화 (Normalization) | 역정규화 (Denormalization) |
|-----------|------------------------|---------------------------|
| **수행 단계** | 분석 및 논리 설계 (선행) | 물리 설계 및 운영/튜닝 (후행) |
| **핵심 목적** | 중복 제거, 무결성 확보, 이상현상 박멸 | 조회 성능 최적화, 조인 연산 최소화 |
| **디자인 방향** | 하나의 테이블 → 여러 개 분해 | 여러 테이블 통합 또는 의도적 중복 추가 |
| **장점** | 저장 공간 최소화, 유연한 구조 변경 | 디스크 I/O 감소, 쿼리 단순화, 응답속도 향상 |
| **위험 요인** | 다중 조인(Join) 부하 유발 | 수정 시 정합성 붕괴 위험, 저장 공간 낭비 |
| **MSA 적용** | 서비스별 논리 모델 완결성 확보 | 서비스 내 읽기 최적화, Redis 캐시 활용 |

- MSA·클라우드 네이티브 환경에서는 서비스별 논리 모델에서 정규화를 완결한 후, 읽기 중심 서비스는 CQRS 패턴과 Redis 캐시를 활용하여 역정규화 없이 성능을 확보하는 현대적 접근이 권장된다.  "끝"

### 실무 제언

**무통제 역정규화의 재앙과 선정규화 후조율 원칙**
- **챌린지**: 성능 향상만을 이유로 가이드라인 없이 무분별하게 컬럼을 복사하거나 테이블을 합치면, 데이터 수정 시 일부 데이터가 누락되어 심각한 데이터 품질 파탄을 초래한다.
- **제언**: 논리 모델 단계에서는 3NF·BCNF까지 완벽하게 도출한 후, CRUD 매트릭스 분석과 SQL 실행 계획(Execution Plan) 기반으로 성능 한계가 도출되는 지점에 한해서만 통제된 역정규화를 적용해야 한다.

**역정규화 부작용 방지를 위한 무결성 보완 대책**
- **챌린지**: 역정규화로 중복 컬럼이 생성되면 원본 수정 시 복사본이 동기화되지 않아 동일 데이터에 다른 값이 공존하는 정합성 붕괴 위험이 존재한다.
- **제언**: DAO/Repository 레이어 트랜잭션 통합 로직 명시, DBMS 트리거(Trigger)를 통한 자동 동기화, 야간 배치를 통한 정합성 상시 검증(Data Reconciliation) 3중 안전장치를 구축해야 한다.

**MSA 환경에서의 역정규화 대안 전략**
- **챌린지**: MSA에서 서비스별 DB 분리 환경에서 역정규화를 적용하면 서비스 간 데이터 동기화 복잡도가 기하급수적으로 증가한다.
- **제언**: MSA 환경에서는 CQRS(Command Query Responsibility Segregation) 패턴과 Redis 캐시 레이어를 활용하여 역정규화 없이도 읽기 성능을 확보하고, 변경 이벤트는 EDA(Event-Driven Architecture)로 전파하여 정합성을 유지해야 한다.
