## 해설

### 출제 배경 및 의도

공간 DB 쿼리의 핵심은 "이 객체가 저 객체와 어떤 공간적 관계에 있는가?"를 묻는 것입니다. GIS·자율주행·LBS 서비스에서 공간 연산자는 가장 빈번하게 사용되는 핵심 기능으로, 각 연산자의 정의와 차이점 이해가 필수입니다.

124회 기술사 시험에서 출제된 이 문제는 공간 DB의 실무 응용 능력을 평가합니다. 공간 연산자는 OGC(Open Geospatial Consortium) 표준에 정의되며, MBR 기반 1차 필터와 DE-9IM 정밀 판별 2단계 구조로 동작하여 수억 건의 공간 객체를 효율적으로 탐색합니다.

채점관은 5개 연산자를 나열하고 각각의 의미·DE-9IM 패턴·사용 시나리오를 논리적으로 서술하는 능력을 평가합니다.

### 1. 공간 연산자(Spatial Operator), 개요

정 의  • 공간 객체 간의 위상 관계(Topological Relationship)를 판별하는 함수로
       - 공간 쿼리의 WHERE 절에서 조건으로 사용되는 OGC 표준 기반 연산자

```
[공간 연산자의 2단계 처리 구조]

  쿼리 입력
      │
      ▼
  ┌─────────────────────────────────────┐
  │  1단계: MBR 필터 (Filter Phase)      │
  │  - 공간 인덱스(R-Tree) 활용          │
  │  - 각 객체의 MBR만 비교, 후보 추출   │
  │  - 빠르지만 False Positive 포함      │
  └──────────────┬──────────────────────┘
                 │ 후보 집합
                 ▼
  ┌─────────────────────────────────────┐
  │  2단계: 세밀 판별 (Refinement Phase) │
  │  - 실제 기하 객체의 정밀 계산         │
  │  - DE-9IM 위상 관계 정확 판별        │
  │  - 정확한 결과 집합 반환              │
  └─────────────────────────────────────┘
```

- 2단계 처리 구조로 대용량 공간 데이터에서 성능과 정확성을 동시에 확보함

### 2. 공간 연산자 5종 상세

1) 핵심 공간 연산자 5종 (OGC Simple Features 기준)

| 연산자 | OGC 함수 | DE-9IM 패턴 | 의미 | 사용 예시 |
|--------|---------|------------|------|---------|
| Equals | ST_Equals | TFFFTFFFT | 두 객체가 기하학적으로 동일 | 행정 경계 레이어 중복 제거 |
| Intersects | ST_Intersects | ¬Disjoint | 두 객체가 어떤 방식으로든 교차 | 도로와 행정구역 교차 여부 |
| Contains | ST_Contains | T*****FF* | A가 B를 완전히 포함 | 시(市)가 구(區)를 포함 |
| Within | ST_Within | TF*FF**** | A가 B의 내부에 완전히 포함 | 포인트가 폴리곤 내부에 위치 |
| Overlaps | ST_Overlaps | T*T***T** | 두 객체가 일부 겹치되 포함 아님 | 두 서비스 권역의 중복 면적 |

- 5종 연산자는 DE-9IM 9교차 행렬의 패턴으로 각 관계를 정형화하여 정확한 위상 판별 수행

2) 공간 연산자 시각적 비교

```
[공간 연산자 관계 시각화]  A = □(큰 사각형),  B = ●(원)

Equals:     A ≡ B          (A와 B가 정확히 같은 도형)

Intersects: A ─────┐       (겹침이 있으면 True)
                B──┘

Contains:   A ┌──────────┐  (A 안에 B 완전 포함, B 경계도 A 내부)
              │    B●    │
              └──────────┘

Within:     A●  ┌──────────┐  (A가 B 안에 완전히 포함 — Contains의 역)
                └──────────┘

Overlaps:   A ┌─────┐      (부분 교차, 서로 포함하지 않음)
              │  B──┼──┐
              └─────┘  └───┘

Disjoint:   A □          ● B    (완전히 분리)
```

- Intersects는 Disjoint의 논리적 역이며 가장 빈번하게 사용되는 공간 연산자

### 3. 연산자 비교 및 트렌드

| 연산자 | Contains | Within | Covers | CoveredBy |
|--------|---------|--------|--------|----------|
| 방향 | A가 B를 포함 | A가 B에 포함 | A가 B를 덮음 | A가 B에 덮임 |
| 경계 처리 | B가 경계에 있으면 False | A가 경계에 있으면 False | 경계에 있어도 True | 경계에 있어도 True |
| OGC 표준 | O | O | 일부 구현체 | 일부 구현체 |
| 실무 사용 | 지역 포함 관계 | 점이 구역 내 여부 | 면 객체 포함 관계 | 면 객체 포함 관계 |

- 자율주행·스마트시티 확산으로 실시간 공간 연산자 처리 성능 최적화가 핵심 기술 이슈로 부상 중.  "끝"

### 실무 제언

**공간 연산자 성능 최적화 — 인덱스 활용**
- **챌린지**: ST_Intersects 등 공간 연산자를 WHERE 절에 사용할 때, 공간 인덱스가 없으면 수백만 건의 공간 객체에서 전체 순차 탐색이 발생하여 응답 시간이 극도로 저하됩니다.
- **제언**: PostGIS 기준으로 GIST 인덱스를 반드시 생성하고, `&&` 연산자(MBR 겹침 여부)를 ST_Intersects 앞에 추가하는 2단계 쿼리 패턴으로 성능을 수십 배 향상시킬 수 있습니다.

**Contains vs Within 경계 처리 주의**
- **챌린지**: ST_Contains(A, B)와 ST_Within(B, A)는 논리적으로 역관계이나, B의 일부가 A의 경계선 위에 있는 경우 Contains는 False, Covers는 True를 반환하는 미묘한 차이로 쿼리 오류가 발생합니다.
- **제언**: 경계 포함 여부가 비즈니스 요건에서 중요한 경우 Contains 대신 ST_Covers를 사용하고, Within 대신 ST_CoveredBy를 사용하여 경계 케이스를 명확히 처리해야 합니다.

**Intersects 남용 방지 — 목적에 맞는 연산자 선택**
- **챌린지**: 범용성 때문에 ST_Intersects를 과도하게 사용하면 정확한 위상 관계가 필요한 경우에도 불필요한 False Positive가 포함되어 비즈니스 로직 오류가 발생합니다.
- **제언**: 포함 관계는 Contains/Within, 인접 관계는 Touches, 부분 중첩은 Overlaps 등 DE-9IM 패턴을 이해하고 목적에 맞는 정확한 연산자를 선택해야 합니다.
