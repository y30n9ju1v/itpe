## 문제

공간 DB 에서 사용되는 공간 연산자(Spatial Operator)를 5 개 나열하고 설명하시오.

## 해설

> **공간을 묻는 5가지 방법 — 공간 연산자의 체계**  
> OGC 기반 공간 연산자 5종과 2단계 필터링(MBR→세밀 판별)

공간 DB 쿼리의 핵심은 "이 객체가 저 객체와 어떤 공간적 관계에 있는가?"를 묻는 것입니다. 공간 연산자는 이러한 위상 관계 질의를 처리하는 함수들로, 수억 개의 공간 객체를 효율적으로 탐색하기 위해 MBR 기반 1차 필터와 정밀 판별 2차 필터의 2단계 구조로 동작합니다.

### 1. 출제 배경 및 의도

124회 기술사 시험에서 출제된 이 문제는 공간 DB의 실무 응용 능력을 평가합니다. 공간 연산자는 GIS·자율주행·LBS 서비스에서 가장 빈번하게 사용되는 기능으로, 각 연산자의 정의와 차이점을 명확히 구분하는 것이 중요합니다. 채점관은 5개 연산자를 나열하고 각각의 의미와 사용 시나리오를 논리적으로 서술하는 능력을 평가합니다.

### 2. 개념의 본질과 작동 메커니즘

공간 연산자(Spatial Operator)는 공간 객체 간의 위상 관계(Topological Relationship)를 판별하는 함수로, 공간 쿼리의 WHERE 절에서 조건으로 사용됩니다.

```text
[공간 연산자의 2단계 처리 구조]

  쿼리 입력
      │
      ▼
  ┌─────────────────────────────────────┐
  │  1단계: MBR 필터 (Filter Phase)      │
  │  - 공간 인덱스(R-Tree) 활용          │
  │  - 각 객체의 MBR(최소 경계 사각형)만  │
  │    비교하여 후보 집합 추출            │
  │  - 빠르지만 False Positive 포함       │
  └──────────────┬──────────────────────┘
                 │ 후보 집합
                 ▼
  ┌─────────────────────────────────────┐
  │  2단계: 세밀 판별 (Refinement Phase) │
  │  - 실제 기하 객체의 정밀 계산         │
  │  - DE-9IM 등 위상 관계 정확 판별      │
  │  - 정확한 결과 집합 반환              │
  └─────────────────────────────────────┘
```

### 3. 핵심 키워드 (Must-Have)

**공간 연산자 5종 (OGC Simple Features 기준)**

| 연산자 | OGC 함수 | DE-9IM 패턴 | 의미 | 사용 예시 |
|--------|---------|------------|------|---------|
| **Equals** | ST_Equals | TFFFTFFFT | 두 객체가 기하학적으로 동일 | 동일한 행정 경계 레이어 중복 제거 |
| **Intersects** | ST_Intersects | ¬Disjoint | 두 객체가 어떤 방식으로든 교차 | 도로와 행정구역 교차 여부 |
| **Contains** | ST_Contains | T*****FF* | A가 B를 완전히 포함 (B의 경계 포함) | 시(市)가 구(區)를 포함 |
| **Within** | ST_Within | TF*FF**** | A가 B의 내부에 완전히 포함 | 포인트가 폴리곤 내부에 위치 |
| **Overlaps** | ST_Overlaps | T*T***T** | 두 객체가 일부 겹치되 서로 포함하지 않음 | 두 서비스 권역의 중복 면적 |

#### 추가 주요 공간 연산자

| 연산자 | OGC 함수 | 의미 | 사용 예시 |
|--------|---------|------|---------|
| **Disjoint** | ST_Disjoint | 완전히 분리, 접촉 없음 | 관계없는 두 구역 |
| **Touches** | ST_Touches | 경계만 접촉, 내부 교차 없음 | 인접 행정구역 |
| **Crosses** | ST_Crosses | 서로 다른 차원이 교차 (점-선, 선-면) | 강이 도로를 횡단 |
| **Covers** | ST_Covers | Contains와 유사, 경계 포함 강화 | 더 엄밀한 포함 관계 판별 |
| **CoveredBy** | ST_CoveredBy | Within과 유사, Covers의 역 | 폴리곤이 다른 폴리곤에 완전히 덮임 |

#### 공간 연산자 시각적 비교

```text
[공간 연산자 관계 시각화]

A = □ (큰 사각형),  B = ● (원)

Equals:       A와 B가 정확히 같은 도형
              A ≡ B

Intersects:   A ─────┐  겹침이 있으면 True
                  B──┘

Contains:     A ┌──────────┐  A 안에 B 완전 포함
                │    B●    │  B의 경계도 A 내부
                └──────────┘

Within:       A●  ┌──────────┐  A가 B 안에 완전히 포함
                  │          │  (Contains의 역)
                  └──────────┘

Overlaps:     A ┌─────┐   부분 교차, 서로 포함하지 않음
                │  B──┼──┐
                └─────┘  │
                     └───┘

Touches:      A ┌─────┐ 경계만 접촉, 내부 교차 없음
                │     │
                └─────┼──B
                      │

Disjoint:     A □          ● B    완전히 분리
```

### 4. 맥락과 비교

| 연산자 | Contains | Within | Covers | CoveredBy |
|--------|---------|--------|--------|----------|
| **방향** | A가 B를 포함 | A가 B에 포함 | A가 B를 덮음 | A가 B에 덮임 |
| **경계 처리** | B가 경계에 있으면 False | A가 경계에 있으면 False | 경계에 있어도 True | 경계에 있어도 True |
| **OGC 표준** | O | O | 일부 구현체 | 일부 구현체 |
| **실무 사용** | 지역 포함 관계 | 점이 구역 내 여부 | 면 객체 관계 | 면 객체 관계 |

**Contains vs Within 차이점:**
- `ST_Contains(A, B)` = A가 B를 포함 = `ST_Within(B, A)`
- 경계 처리에서 미묘한 차이: B의 일부가 A의 경계선 위에 있을 경우, Contains는 False, Covers는 True

### 5. 실무 제언

**공간 연산자 성능 최적화 — 2단계 필터링의 이해와 인덱스 활용**

- **챌린지:** ST_Intersects 등 공간 연산자를 WHERE 절에 사용할 때, 공간 인덱스가 없으면 전체 테이블 순차 탐색(Sequential Scan)이 발생하여 수백만 건의 공간 객체에서 매우 느린 응답 시간이 나타납니다.
- **제언:** PostGIS 기준으로 `CREATE INDEX idx_geom ON table USING GIST(geom);` 형태의 GIST(Generalized Search Tree) 인덱스를 반드시 생성해야 합니다. 대용량 공간 데이터에서는 `&&` 연산자(MBR 겹침 여부)를 ST_Intersects 조건 앞에 추가하여 1차 MBR 필터를 명시적으로 수행하면 성능을 수십 배 향상시킬 수 있습니다.

```sql
-- 최적화된 공간 쿼리 예시
SELECT * FROM buildings b
WHERE b.geom && ST_Buffer(target_point, 1000)  -- MBR 1차 필터
  AND ST_DWithin(b.geom, target_point, 1000);  -- 정밀 2차 필터
```
