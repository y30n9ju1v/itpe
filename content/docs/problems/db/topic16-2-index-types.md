## 문제

대용량의 데이터베이스 테이블에서 데이터를 빨리 찾기 위한 인덱스에 대하여 개념을 설명하고, 인덱스의 다양한 스캔 방식과 인덱스 종류에 대하여 설명하시오.

## 해설

### 출제 배경 및 의도

대용량 데이터베이스 환경에서 인덱스의 종류와 스캔 방식을 이해하면 동일한 쿼리도 실행 방식을 최적화하여 수십~수백 배의 성능 차이를 만들 수 있다. 데이터 특성(카디널리티, 분포)과 접근 패턴(동등 검색, 범위 검색, 집계)에 따라 최적 인덱스 유형을 선택하는 역량이 핵심이다.

이 문제는 단순한 인덱스 개념을 넘어 다양한 종류의 인덱스와 복수의 스캔 방식을 상황에 따라 적용할 수 있는지를 검증한다. 특히 클러스터드 인덱스의 물리적 저장 구조, 비트맵 인덱스의 DW 활용, Index Skip Scan의 내부 동작 원리를 명확히 설명할 수 있어야 한다.

### 1. 인덱스 스캔 방식과 인덱스 종류, 개요

정 의  • 테이블의 특정 컬럼값과 레코드의 물리적 위치(ROWID)를 매핑하여 풀 스캔 없이 데이터를 빠르게 탐색하는 자료구조
       - 저장 방식(클러스터드/논클러스터드), 자료구조(B-Tree/Hash/Bitmap/FBI), 접근 컬럼 수(단일/복합)로 분류

```
[인덱스 분류 체계]

 인덱스
 ├── 저장 방식
 │    ├── 클러스터드 인덱스 (Clustered): 테이블 데이터 = 인덱스 리프
 │    └── 논클러스터드 인덱스 (Non-Clustered): 인덱스 리프 → ROWID → 테이블
 │
 ├── 자료구조
 │    ├── B-Tree: 범용, 범위 검색 지원, O(log n)
 │    ├── Hash: 동등 검색 O(1), 범위 미지원
 │    ├── Bitmap: 저 카디널리티, AND/OR 비트 연산
 │    └── FBI(Function-Based Index): 컬럼 표현식 인덱싱
 │
 └── 컬럼 수
      ├── 단일 인덱스 (Single Column)
      └── 복합 인덱스 (Composite/Concatenated)
```

- 접근 패턴과 데이터 특성에 맞는 인덱스 유형 선택이 쿼리 성능을 결정한다.

### 2. 인덱스 종류 및 스캔 방식 상세

1) 인덱스 저장 방식: 클러스터드 vs 논클러스터드

| 구분 | 항목 | 설명 |
|------|------|------|
| 클러스터드 | 리프 노드 구조 | 인덱스 리프 노드 = 실제 데이터 행 (키 순서로 물리 정렬) |
| 클러스터드 | 생성 개수 | 테이블당 1개만 가능 (물리적 정렬은 하나) |
| 클러스터드 | DBMS 특성 | MySQL InnoDB: PRIMARY KEY에 자동 생성 |
| 클러스터드 | 범위 스캔 | 연속 데이터 Sequential I/O로 범위 스캔 최강 |
| 논클러스터드 | 리프 노드 구조 | 인덱스 리프 → ROWID → 테이블 랜덤 접근 |
| 논클러스터드 | 생성 개수 | 테이블당 다수 생성 가능 |
| 논클러스터드 | I/O 특성 | 클러스터드 대비 추가 랜덤 I/O 발생 |

```
[클러스터드 vs 논클러스터드 구조]

  [클러스터드]                    [논클러스터드]
  Leaf: [키값 | 실제 데이터행]    Leaf: [키값 | ROWID] ──→ Table Block
  ← 데이터 자체가 키 순서 정렬      ← 추가 랜덤 I/O 발생
```

- 클러스터드 인덱스는 범위 스캔에 강하고, 논클러스터드는 다수 조건 컬럼에 유연하게 대응한다.

2) 인덱스 자료구조별 특성 및 스캔 방식

| 구분 | 인덱스 종류 | 등치 검색 | 범위 검색 | 카디널리티 | 주요 사용처 |
|------|---------|---------|---------|-----------|-----------|
| ① | B-Tree | O(log n) | O(log n+k) | 고/저 모두 | OLTP 범용 |
| ② | Hash | O(1) | 불가 | 고 카디널리티 | 동등 검색, 메모리 DB |
| ③ | Bitmap | O(1) (비트 연산) | 제한적 | 저 (성별, 지역) | DW, OLAP, 복합 조건 |
| ④ | FBI (함수 기반) | O(log n) | O(log n+k) | 고 | UPPER(컬럼), TO_CHAR 등 |

| 구분 | 스캔 방식 | 사용 조건 | 정렬 보장 | 멀티블록 I/O |
|------|---------|---------|---------|-----------|
| ① | Unique Scan | = 조건, Unique 인덱스 | - | X |
| ② | Range Scan | 범위 조건, 등치 조건 | O | X |
| ③ | Full Scan | ORDER BY, 커버링 인덱스 | O | X |
| ④ | Fast Full Scan | 인덱스 컬럼만 SELECT | X | O |
| ⑤ | Skip Scan | 선두 컬럼 조건 없음, NDV 낮음 | 제한 | X |

### 3. 인덱스 활용 심화 및 설계 트렌드

| 기법 | 원리 | 효과 |
|------|------|------|
| Index Skip Scan | 선두 컬럼 미사용 시 고유값(NDV) 기준 내부 분기 (성별='남'/'여' 각각 탐색) | 선두 컬럼 NDV 낮을 때 Full Scan 대체 |
| Index Fast Full Scan | 인덱스 블록만 멀티블록 I/O 스캔 (테이블 미접근) | COUNT, 집계 쿼리 고속 처리 |
| Covering Index | SELECT 컬럼을 인덱스에 모두 포함하여 테이블 접근 제거 | 읽기 집중 OLTP 최고 효율 |

클라우드 DBMS(Aurora, AlloyDB 등)에서는 LSM-Tree(Log-Structured Merge-Tree) 기반 인덱스가 쓰기 집중 워크로드에서 B-Tree 대비 높은 처리량을 제공하며, 벡터 인덱스(HNSW, IVF)는 AI 임베딩 유사도 검색을 위한 신규 인덱스 유형으로 급속히 확산되고 있다.  "끝"

### 실무 제언

**복합 인덱스의 선두 컬럼 선택 전략**
- **챌린지**: 복합 인덱스에서 선두 컬럼이 쿼리 WHERE 절에 없으면 Range Scan으로 활용할 수 없어 Full Table Scan이 발생한다.
- **제언**: 복합 인덱스 설계 시 등치(=) 조건 컬럼을 앞에, 선택률이 높은(NDV가 큰) 컬럼을 앞에 배치한다. 자주 사용되는 WHERE 조건 조합을 분석하여 여러 쿼리를 포괄하는 복합 인덱스를 설계하고, 커버링 인덱스로 테이블 접근을 최소화한다.

**Bitmap 인덱스의 OLTP 사용 금지**
- **챌린지**: Bitmap 인덱스는 비트맵 단위로 락을 잡기 때문에 한 행의 값이 변경될 때 해당 비트맵 세그먼트 전체에 락이 걸린다. OLTP 환경에서 동시 UPDATE가 많으면 심각한 락 경합과 성능 저하가 발생한다.
- **제언**: Bitmap 인덱스는 읽기 전용 또는 배치 중심의 DW/OLAP 환경에서만 사용하고, OLTP 테이블의 저 카디널리티 컬럼 검색 성능 문제는 B-Tree 복합 인덱스나 파티셔닝으로 해결한다.

**커버링 인덱스로 테이블 접근 최소화**
- **챌린지**: 인덱스를 통해 ROWID를 획득한 후 테이블 블록에 랜덤 접근하는 과정에서 대량의 랜덤 I/O가 발생하여 성능 병목이 된다.
- **제언**: 자주 실행되는 쿼리의 SELECT 컬럼을 인덱스에 포함하는 커버링 인덱스를 설계하여 테이블 접근을 완전히 제거하고, 인덱스 크기 증가와 DML 오버헤드를 고려하여 핵심 쿼리에만 선택적으로 적용한다.
