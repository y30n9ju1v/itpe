## 문제

대용량의 데이터베이스 테이블에서 데이터를 빨리 찾기 위한 인덱스에 대하여 개념을 설명하고, 인덱스의 다양한 스캔 방식과 인덱스 종류에 대하여 설명하시오.

## 해설

> **접근 패턴과 데이터 특성에 맞는 인덱스 유형 선택이 쿼리 성능을 결정한다**  
> 클러스터드/논클러스터드, B-Tree/Hash/Bitmap/함수기반 인덱스와 Index Skip Scan·Fast Full Scan 스캔 방식

인덱스의 종류와 스캔 방식을 이해하면 동일한 쿼리도 실행 방식을 최적화하여 수십~수백 배의 성능 차이를 만들 수 있습니다. 대용량 테이블에서 데이터 특성(카디널리티, 분포)과 접근 패턴(동등 검색, 범위 검색, 집계)에 따라 최적 인덱스 유형을 선택하는 역량이 핵심입니다. 카디널리티(Cardinality)란 해당 컬럼에 존재하는 고유값의 수로, 성별처럼 종류가 2개면 저 카디널리티, 주문번호처럼 종류가 많으면 고 카디널리티입니다.

### 1. 출제 배경 및 의도

이 문제는 단순한 인덱스 개념을 넘어 다양한 종류의 인덱스와 복수의 스캔 방식을 상황에 따라 적용할 수 있는지를 검증합니다. 특히 클러스터드 인덱스의 물리적 저장 구조 이해, 비트맵 인덱스의 DW 활용, Index Skip Scan의 내부 동작 원리를 명확히 설명할 수 있어야 합니다.

### 2. 개념의 본질과 작동 메커니즘

인덱스 종류는 저장 방식(클러스터드/논클러스터드), 자료구조(B-Tree/Hash/Bitmap/함수기반), 접근 컬럼 수(단일/복합)에 따라 분류됩니다. 스캔 방식은 옵티마이저가 인덱스 구조와 쿼리 조건을 분석하여 선택합니다.

```text
[인덱스 분류 체계]

 인덱스
 ├── 저장 방식
 │    ├── 클러스터드 인덱스 (Clustered): 테이블 데이터 = 인덱스 리프
 │    └── 논클러스터드 인덱스 (Non-Clustered): 인덱스 리프 → ROWID → 테이블
 │
 ├── 자료구조
 │    ├── B-Tree: 범용, 범위 검색 지원
 │    ├── Hash: 동등 검색 O(1), 범위 미지원
 │    ├── Bitmap: 저 카디널리티, AND/OR 비트 연산
 │    └── Function-Based(FBI): 컬럼 표현식 인덱싱
 │
 └── 컬럼 수
      ├── 단일 인덱스 (Single Column)
      └── 복합 인덱스 (Composite/Concatenated)
```

#### 클러스터드 인덱스 vs 논클러스터드 인덱스

```text
[클러스터드 인덱스]
  인덱스 리프 노드 = 실제 데이터 행
  ┌──────────────────────────────┐
  │ Leaf: [키값 | 실제 데이터행] │ ← 데이터 자체가 키 순서로 정렬 저장
  └──────────────────────────────┘
  → 테이블당 1개만 가능 (물리적 정렬은 하나)
  → PRIMARY KEY에 자동 생성 (MySQL InnoDB)
  → 범위 스캔 최강: 연속 데이터 Sequential I/O

[논클러스터드 인덱스]
  인덱스 리프 → ROWID → 테이블 랜덤 접근
  ┌──────────────────────┐      ┌──────────────────┐
  │ Leaf: [키값 | ROWID] │ ───→ │ Table Block      │
  └──────────────────────┘      │ (임의 위치)      │
  → 테이블당 다수 생성 가능      └──────────────────┘
  → 클러스터드 대비 추가 랜덤 I/O 발생
```

#### 인덱스 자료구조별 특성

| 인덱스 종류 | 자료구조 | 등치 검색 | 범위 검색 | 카디널리티 | 주요 사용처 |
|-----------|---------|---------|---------|-----------|-----------|
| **B-Tree** | 균형 트리 | O(log n) | O(log n + k) | 고/저 | OLTP 범용 |
| **Hash** | 해시 테이블 | O(1) | 불가 | 고 | 동등 검색, 메모리 DB |
| **Bitmap** | 비트 배열 | O(1) | 제한적 | 저(예: 성별, 지역) | DW, OLAP, 복합 조건 |
| **FBI (함수 기반)** | B-Tree | O(log n) | O(log n + k) | 고 | UPPER(컬럼), TO_CHAR 등 |

#### 다양한 스캔 방식 상세

**Index Skip Scan**
```text
복합 인덱스 (성별, 연봉) 존재 시:
WHERE 연봉 > 5000 (선두 컬럼 성별 조건 없음)

일반적으로: Full Table Scan
Skip Scan: 성별의 고유값(남/여) 기준으로 내부 분리
  → [성별='남' AND 연봉>5000] + [성별='여' AND 연봉>5000]
  → 선두 컬럼 NDV가 낮을 때 효과적 (성별은 2개뿐)
```

**Index Fast Full Scan**
```text
SELECT 이메일 FROM 회원 WHERE 이메일 LIKE '%@gmail.com'
→ 인덱스 컬럼만 SELECT, LIKE 조건이어도 인덱스 블록만 스캔
→ 멀티블록 I/O로 Full Table Scan보다 빠름
→ ORDER BY 보장 없음 (물리적 순서 스캔)
```

**Covering Index (인덱스 전용 스캔)**
```text
인덱스: (부서코드, 이름, 연봉)
SELECT 이름, 연봉 FROM 사원 WHERE 부서코드 = 'A001'
→ 테이블 접근 없이 인덱스만으로 결과 반환
→ 가장 빠른 스캔 패턴 중 하나
```

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 저장 구조 | 클러스터드(Clustered), 논클러스터드(Non-Clustered), ROWID, 물리적 정렬 |
| 자료구조 | B-Tree, Hash, Bitmap, FBI(Function-Based Index) |
| 스캔 방식 | Index Unique Scan, Index Range Scan, Index Full Scan, Index Fast Full Scan, Index Skip Scan, Covering Index Scan |
| 설계 | 카디널리티(Cardinality), NDV, 선두 컬럼(Leading Column), 복합 인덱스 |

### 4. 맥락과 비교

| 스캔 방식 | 사용 조건 | 정렬 보장 | 멀티블록 I/O | 적합 상황 |
|---------|---------|---------|-----------|---------|
| Unique Scan | = 조건, Unique 인덱스 | - | X | PK 조회 |
| Range Scan | 범위 조건, 등치 조건 | O | X | 일반 OLTP |
| Full Scan | ORDER BY, 커버링 | O | X | 소규모 인덱스 |
| Fast Full Scan | 인덱스 컬럼만 SELECT | X | O | COUNT, 집계 |
| Skip Scan | 선두 컬럼 조건 없음, NDV 낮음 | 제한 | X | 복합 인덱스 부분 활용 |

### 5. 실무 제언

**복합 인덱스의 선두 컬럼 선택 전략**

- **챌린지:** 복합 인덱스에서 선두 컬럼이 쿼리 WHERE 절에 없으면 해당 인덱스를 Range Scan으로 활용할 수 없습니다. 개발자가 인덱스 컬럼 순서를 고려하지 않고 쿼리를 작성하면 Full Table Scan이 발생합니다.
- **제언:** 복합 인덱스 설계 시 ① 등치(=) 조건 컬럼을 앞에, ② 선택률이 높은(NDV가 큰) 컬럼을 앞에 배치합니다. 자주 사용되는 WHERE 조건 조합을 분석하여 여러 쿼리를 포괄하는 복합 인덱스를 설계하고, 커버링 인덱스로 테이블 접근을 최소화합니다.

**Bitmap 인덱스의 OLTP 사용 금지**

- **챌린지:** Bitmap 인덱스는 비트맵 단위로 락을 잡기 때문에 한 행의 값이 변경될 때 해당 비트맵 세그먼트 전체에 락이 걸립니다. OLTP 환경에서 동시 UPDATE가 많으면 심각한 락 경합과 성능 저하가 발생합니다.
- **제언:** Bitmap 인덱스는 읽기 전용 또는 배치 중심의 DW/OLAP 환경에서만 사용하고, OLTP 테이블에서 저 카디널리티 컬럼 검색 성능 문제는 B-Tree 복합 인덱스나 파티셔닝으로 해결합니다.
