# ARIES 회복 기법과 로그 기반 장애 복구 프로세스 완전 해체

---

## 1. 문제를 해체하는 5대 핵심 분석 항목

### ① 출제 배경 및 의도 (Why Now?)

* **현대 DBMS 회복 알고리즘의 표준:** ARIES(Algorithms for Recovery and Isolation Exploiting Semantics)는 오늘날 거의 모든 상용 RDBMS(IBM DB2, MS-SQL, SQL Server 등)가 채택하고 있는 가장 정교한 회복 알고리즘입니다. 단순 로그 기반 기법을 넘어 '반복 기록 재실행(Repeating History)'을 통해 장애 복구 효율을 극대화합니다.
* **고가용성 아키텍처의 핵심 엔진:** 시스템이 급작스럽게 중단되었을 때, 단순히 로그를 순차 스캔하는 것이 아니라 어떻게 복구 시간(RTO)을 단축하고 일관성(Consistency)을 유지하는지 그 내부 작동 원리를 깊이 있게 이해하고 있는지 평가하기 위함입니다.

### ② 개념의 본질과 작동 메커니즘 (What & How)

* **한 문장 정의:** 트랜잭션의 기록을 로그에 남기고, 장애 발생 시 '분석(Analysis) -> 재실행(Redo) -> 취소(Undo)'의 3단계 과정을 통해 데이터베이스를 장애 직전의 일관된 상태로 돌려놓는 최적화된 회복 기법입니다.
* **작동 메커니즘 (Mechanism):**
1. **분석(Analysis):** 로그를 스캔하여 장애 시점의 dirty page와 커밋/미커밋 트랜잭션을 식별합니다.
2. **재실행(Redo):** 커밋 여부와 관계없이 과거 기록부터 장애 직전까지의 모든 변경 사항을 재현하여 역사를 반복합니다(Repeating History).
3. **취소(Undo):** 커밋되지 못한 트랜잭션들이 변경한 사항들을 역순으로 되돌려 정합성을 확보합니다.



### ③ 채점관이 눈을 켜고 찾는 'Must-Have' 키워드 (Keywords)

* **ARIES 3단계:** 분석(Analysis), 재실행(Redo), 취소(Undo).
* **핵심 기술 요소:** LSN(Log Sequence Number), Dirty Page Table(DPT), Transaction Table, CLR(Compensation Log Record), Repeating History.

### ④ 상위/하위/유사 개념과의 상호 관계 (Context & Comparison)

* **기존 기법과의 차이:** 고전적 회복 기법(지연/즉시 갱신)이 트랜잭션 단위의 REDO/UNDO를 수행한다면, ARIES는 '로그 시퀀스 번호(LSN)'를 기준으로 페이지 단위의 세밀한 복구를 수행하여 훨씬 빠르고 정확합니다.

### ⑤ 실무적 제언 및 챌린지 (So What? / Insight)

* **RTO 단축 전략:** 로그 파일이 너무 커지면 분석(Analysis) 단계에서 시간이 오래 걸립니다. 따라서 주기적인 체크포인트(Checkpoint)를 통해 DPT(Dirty Page Table)를 최신화함으로써, 복구 시 스캔 범위를 대폭 줄이는 것이 성능 튜닝의 핵심입니다.

## 2. ARIES 회복 기법의 단계별 동작 원리

ARIES는 장애 발생 후 다음과 같은 3단계를 거쳐 데이터베이스를 회복합니다.

| 단계 | 동작 방식 | 수행 내용 |
| --- | --- | --- |
| **1. 분석(Analysis)** | 로그 스캔(전방) | 마지막 체크포인트부터 장애 발생 시점까지 스캔. 트랜잭션 상태(커밋/미커밋)와 Dirty Page Table 식별. |
| **2. 재실행(Redo)** | 로그 재실행(전방) | 식별된 모든 변경 사항을 **무차별 재실행**하여 역사 반복(Repeating History). 장애 직전의 데이터 상태로 복구. |
| **3. 취소(Undo)** | 로그 역추적(후방) | 커밋하지 못한 트랜잭션들을 로그 역순으로 추적하며 CLR을 기록하고 변경 사항을 취소. |

## 3. 사례 기반 회복 과정 서술 (가상 로그 내용 적용)

사용자께서 제시하신 로그 내용이 구체적으로 명시되지 않았으나, 전형적인 ARIES 복구 시나리오를 통해 그 과정을 설명합니다.

### [가상 로그 상황]

1. `[LSN 10, T1, Start]`
2. `[LSN 20, T2, Start]`
3. `[LSN 30, T1, Write(P1, v1)]`
4. `[LSN 40, T1, Commit]`
5. `[LSN 50, T2, Write(P2, v2)]`
6. `[장애 발생]`

### [단계별 회복 과정]

**1. 분석(Analysis) 단계:**

* 체크포인트 정보를 확인합니다.
* 로그를 스캔하여 **T1은 Commit** 상태, **T2는 Active(실행 중)** 상태임을 식별합니다.

**2. 재실행(Redo) 단계:**

* `LSN 30(T1의 P1 쓰기)`과 `LSN 50(T2의 P2 쓰기)`을 모두 재실행합니다. (T2가 커밋되지 않았더라도 역사 반복 원칙에 따라 일단 다시 씁니다.)

**3. 취소(Undo) 단계:**

* Undo 리스트에 있는 **T2를 역순으로 추적**합니다.
* `LSN 50` 기록을 취소하고, 이를 보상하기 위해 CLR(Compensation Log Record)을 로그에 기록합니다.
* 최종적으로 T2의 변경 사항이 제거되어 데이터베이스는 T1만 커밋된 상태로 복구됩니다.

## 4. 기술사의 시각: ARIES 회복 기법의 차세대 활용

* **분산 트랜잭션과의 결합:** 단일 노드 DB의 ARIES 복구만으로는 분산 환경의 무결성을 보장할 수 없습니다. 분산 아키텍처에서는 ARIES의 로컬 복구 능력에 더해, 각 노드 간의 상태를 동기화하는 **Saga 패턴**이나 **2PC(Two-Phase Commit)** 프로토콜을 결합하여 전사적인 데이터 무결성을 달성해야 합니다.
* **결론:** ARIES는 고전적 회복 이론의 집대성입니다. 하지만 기술사는 여기에 안주하지 않고, 인메모리 DB의 휘발성 문제 해결을 위한 스냅샷 전략과 로그 재실행의 오버헤드를 최소화하기 위한 비휘발성 메모리(NVDIMM) 활용 등 차세대 인프라에 적합한 회복 아키텍처를 설계할 수 있어야 합니다.
