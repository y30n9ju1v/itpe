## 해설

> **분석(Analysis) → 재실행(Redo) → 취소(Undo)의 3단계로 데이터베이스를 장애 직전 상태로 복구하는 현대 RDBMS 표준 회복 알고리즘**  
> ARIES(Algorithms for Recovery and Isolation Exploiting Semantics) 회복 기법

ARIES는 오늘날 거의 모든 상용 RDBMS(IBM DB2, MS-SQL Server 등)가 채택하고 있는 가장 정교한 회복 알고리즘입니다. 단순 로그 기반 기법을 넘어 '반복 기록 재실행(Repeating History)'을 통해 장애 복구 효율을 극대화합니다. LSN(Log Sequence Number)을 기준으로 페이지 단위의 세밀한 복구를 수행하여, 트랜잭션 단위의 REDO/UNDO를 수행하는 고전적 회복 기법보다 훨씬 빠르고 정확합니다.

### 1. 출제 배경 및 의도

ARIES는 현대 DBMS 회복 알고리즘의 표준으로, 단순히 로그를 순차 스캔하는 것이 아니라 어떻게 복구 시간(RTO)을 단축하고 일관성(Consistency)을 유지하는지 그 내부 작동 원리를 깊이 있게 이해하고 있는지 평가하기 위함입니다.

채점관은 ARIES의 3단계(분석·재실행·취소)와 LSN·DPT·CLR·WAL 등의 핵심 기술 요소를 이해하고, 실제 로그 내용을 기반으로 회복 과정을 단계별로 서술할 수 있는지를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

트랜잭션의 기록을 로그에 남기고, 장애 발생 시 '분석(Analysis) → 재실행(Redo) → 취소(Undo)'의 3단계 과정을 통해 데이터베이스를 장애 직전의 일관된 상태로 돌려놓는 최적화된 회복 기법입니다.

ARIES는 **WAL(Write-Ahead Logging)** 원칙 — 데이터 페이지를 디스크에 쓰기 전에 반드시 로그 레코드를 먼저 디스크에 기록 — 을 기반으로 동작합니다. 각 로그 레코드는 고유한 **LSN(Log Sequence Number)**으로 식별되며, 페이지에는 해당 페이지를 마지막으로 수정한 로그의 LSN(PageLSN)이 기록됩니다.

### 3. 핵심 키워드 (Must-Have)

#### ARIES 3단계 동작 원리

| 단계 | 동작 방식 | 수행 내용 |
|------|----------|---------|
| **1. 분석(Analysis)** | 로그 스캔(전방) | 마지막 체크포인트부터 장애 발생 시점까지 로그 스캔. 트랜잭션 상태(커밋/미커밋)와 Dirty Page Table(DPT) 식별. REDO 시작점과 UNDO 대상 트랜잭션 목록 확정 |
| **2. 재실행(Redo)** | 로그 재실행(전방) | 식별된 모든 변경 사항을 **무차별 재실행**하여 역사 반복(Repeating History). 커밋 여부와 관계없이 장애 직전의 데이터 상태로 복구. PageLSN을 활용하여 이미 반영된 변경은 중복 재실행 방지 |
| **3. 취소(Undo)** | 로그 역추적(후방) | 커밋하지 못한 트랜잭션들을 로그 역순으로 추적하며 CLR(Compensation Log Record)을 기록하고 변경 사항을 취소. CLR 기록으로 Undo 중 재장애 발생 시에도 안전하게 복구 가능 |

#### 핵심 기술 요소

- **LSN (Log Sequence Number)**: 각 로그 레코드의 고유 단조 증가 번호. 복구 시 로그 레코드 간 순서 및 중복 처리 방지에 사용
- **Dirty Page Table (DPT)**: 버퍼 캐시에서 수정되었지만 아직 디스크에 기록되지 않은 페이지 목록. 분석 단계에서 REDO 시작 LSN 결정에 사용
- **Transaction Table**: 현재 활성 중인 트랜잭션들의 상태(커밋/미커밋) 목록. 분석 단계에서 UNDO 대상 식별에 사용
- **CLR (Compensation Log Record)**: Undo 수행 시 생성하는 특수 로그 레코드. Undo 중 재장애 발생 시 이미 수행된 Undo를 다시 반복하지 않도록 보상 기록
- **Repeating History**: Redo 단계에서 커밋 여부와 관계없이 모든 변경 사항을 무차별 재실행하는 원칙. 장애 직전 상태를 정확히 재현한 후 Undo 단계에서 미완료 트랜잭션을 제거

### 4. 맥락과 비교

기존 고전적 회복 기법과 ARIES의 차이점을 비교합니다.

| 비교 항목 | 고전적 회복 기법 (지연/즉시 갱신) | ARIES |
|-----------|--------------------------------|-------|
| **복구 단위** | 트랜잭션 단위 REDO/UNDO | LSN 기반 **페이지 단위** 세밀한 복구 |
| **Redo 범위** | 커밋된 트랜잭션만 REDO | 커밋 여부와 관계없이 **모든 변경 사항 무차별 REDO** (Repeating History) |
| **중복 방지** | 별도 메커니즘 없음 | **PageLSN**으로 이미 반영된 변경 중복 재실행 방지 |
| **Undo 안전성** | Undo 중 재장애 시 복잡한 처리 필요 | **CLR(Compensation Log Record)** 기록으로 안전한 멱등성 보장 |
| **복구 시작점** | 로그 전체 스캔 | 마지막 체크포인트부터 스캔 (DPT 활용으로 범위 최소화) |

#### 사례 기반 회복 과정 (가상 로그 적용)

```
[가상 로그 상황]
[LSN 10, T1, Start]
[LSN 20, T2, Start]
[LSN 30, T1, Write(P1, old=v0, new=v1)]
[LSN 40, T1, Commit]
[LSN 50, T2, Write(P2, old=v0, new=v2)]
[장애 발생]
```

1. **분석(Analysis) 단계**: 체크포인트 정보 확인 후 로그 스캔. T1은 Commit 상태, T2는 Active(실행 중) 상태로 식별. UNDO 리스트에 T2 등록
2. **재실행(Redo) 단계**: LSN 30(T1의 P1 쓰기)과 LSN 50(T2의 P2 쓰기)을 모두 재실행. T2가 커밋되지 않았더라도 Repeating History 원칙에 따라 일단 재실행
3. **취소(Undo) 단계**: UNDO 리스트의 T2를 역순으로 추적. LSN 50 기록을 취소하고 CLR을 로그에 기록. 최종적으로 T2의 변경 사항이 제거되어 T1만 커밋된 상태로 복구

### 5. 실무 제언

**RTO 단축 전략 — 체크포인트 주기 최적화**

- **챌린지**: 로그 파일이 너무 커지면 분석(Analysis) 단계에서 스캔 시간이 오래 걸려 복구 시간(RTO)이 늘어납니다.
- **제언**: 주기적인 체크포인트(Checkpoint)를 통해 DPT(Dirty Page Table)를 최신화함으로써, 복구 시 스캔 범위를 대폭 줄이는 것이 성능 튜닝의 핵심입니다. 오라클의 `FAST_START_MTTR_TARGET` 등 DBMS의 체크포인트 주기 제어 파라미터를 시스템 성격에 맞게 조율해야 합니다.

**분산 트랜잭션과의 결합**

- **챌린지**: 단일 노드 DB의 ARIES 복구만으로는 MSA 분산 환경의 전사적 무결성을 보장할 수 없습니다. 데이터베이스가 서비스 경계별로 파편화되어 ARIES의 로컬 복구만으로는 다른 서비스의 데이터까지 정합성을 맞출 수 없습니다.
- **제언**: 분산 아키텍처에서는 ARIES의 로컬 복구 능력에 더해, 각 노드 간의 상태를 동기화하는 Saga 패턴이나 2PC(Two-Phase Commit) 프로토콜을 결합하여 전사적인 데이터 무결성을 달성해야 합니다. 또한 인메모리 DB의 휘발성 문제 해결을 위한 스냅샷 전략과 NVDIMM(비휘발성 메모리) 활용 등 차세대 인프라에 적합한 회복 아키텍처를 설계할 수 있어야 합니다.
