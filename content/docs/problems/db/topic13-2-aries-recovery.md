## 해설

### 출제 배경 및 의도

ARIES(Algorithms for Recovery and Isolation Exploiting Semantics)는 오늘날 거의 모든 상용 RDBMS(IBM DB2, MS-SQL Server 등)가 채택하고 있는 가장 정교한 회복 알고리즘이다. 단순 로그 기반 기법을 넘어 '반복 기록 재실행(Repeating History)'을 통해 장애 복구 효율을 극대화한다. LSN(Log Sequence Number)을 기준으로 페이지 단위의 세밀한 복구를 수행하여, 트랜잭션 단위의 REDO/UNDO를 수행하는 고전적 회복 기법보다 훨씬 빠르고 정확하다.

채점관은 ARIES의 3단계(분석·재실행·취소)와 LSN·DPT·CLR·WAL 등의 핵심 기술 요소를 이해하고, 실제 로그 내용을 기반으로 회복 과정을 단계별로 서술할 수 있는지를 검증한다.

### 1. ARIES 회복 기법, 개요

정 의  • 분석(Analysis)→재실행(Redo)→취소(Undo)의 3단계로 DB를 장애 직전 상태로 복구하는 현대 RDBMS 표준 회복 알고리즘
       - WAL(Write-Ahead Logging) 원칙 기반, LSN으로 로그 레코드 순서와 중복 처리를 제어

```
[ARIES 3단계 동작 흐름도]

  장애 발생
       │
       ▼
  ┌────────────────────────────────────────────────────────┐
  │  1단계: 분석(Analysis)                                 │
  │  └─ 마지막 체크포인트 → 장애 시점 로그 전방 스캔       │
  │     • Transaction Table: 커밋/미커밋 트랜잭션 식별     │
  │     • Dirty Page Table(DPT): REDO 시작 LSN 결정        │
  │     • UNDO List 확정 (미커밋 트랜잭션 목록)            │
  └──────────────────────┬─────────────────────────────────┘
                         │
                         ▼
  ┌────────────────────────────────────────────────────────┐
  │  2단계: 재실행(Redo) — Repeating History              │
  │  └─ DPT의 최소 LSN부터 전방 스캔, 모든 변경 무차별 재실행│
  │     • PageLSN 비교: 이미 반영된 변경은 중복 재실행 방지 │
  │     • 커밋 여부 무관하게 장애 직전 상태 완전 재현       │
  └──────────────────────┬─────────────────────────────────┘
                         │
                         ▼
  ┌────────────────────────────────────────────────────────┐
  │  3단계: 취소(Undo)                                     │
  │  └─ UNDO List 트랜잭션을 로그 역순으로 추적            │
  │     • 변경 취소 시마다 CLR(Compensation Log Record) 기록│
  │     • CLR로 Undo 중 재장애 시에도 멱등성 보장          │
  └────────────────────────────────────────────────────────┘
                         │
                         ▼
       커밋된 트랜잭션만 반영된 일관성 있는 DB 상태
```

- ARIES는 Repeating History 원칙과 CLR 기록으로 고전적 기법 대비 RTO를 최소화하고 멱등적 복구를 보장한다.

### 2. ARIES 핵심 기술 요소

1) 3단계 동작 원리

| 구분 | 단계 | 동작 방향 | 주요 수행 내용 |
|------|------|---------|------------|
| ① | 분석(Analysis) | 전방 (Forward) | 체크포인트부터 장애 시점까지 스캔, Transaction Table·DPT 구성, UNDO 리스트 확정 |
| ② | 재실행(Redo) | 전방 (Forward) | Repeating History: 커밋 여부 무관 모든 변경 재실행, PageLSN으로 중복 방지 |
| ③ | 취소(Undo) | 역방향 (Backward) | UNDO 리스트 트랜잭션 역순 추적, 변경 취소, CLR 기록으로 멱등성 보장 |

- 분석 단계에서 복구 범위를 최소화하는 것이 RTO 단축의 핵심이다.

2) 핵심 기술 구성 요소

| 구분 | 항목 | 설명 |
|------|------|------|
| 로그 식별 | LSN (Log Sequence Number) | 각 로그 레코드의 고유 단조 증가 번호, 순서 및 중복 처리 방지 |
| 페이지 관리 | PageLSN | 페이지를 마지막으로 수정한 로그의 LSN, Redo 중복 재실행 방지 |
| 복구 범위 | DPT (Dirty Page Table) | 버퍼에서 수정되었으나 디스크 미기록 페이지 목록, REDO 시작 LSN 결정 |
| UNDO 추적 | Transaction Table | 활성 트랜잭션의 상태(커밋/미커밋) 목록, UNDO 대상 식별 |
| 멱등성 보장 | CLR (Compensation Log Record) | Undo 수행 시 생성되는 특수 로그, Undo 중 재장애 시 중복 Undo 방지 |
| 핵심 원칙 | Repeating History | Redo 단계에서 커밋 여부 무관 모든 변경 무차별 재실행 후 Undo에서 미완료 제거 |

### 3. ARIES vs 고전적 기법 비교 및 가상 로그 적용

| 비교 항목 | 고전적 회복 기법 | ARIES |
|---------|--------------|-------|
| 복구 단위 | 트랜잭션 단위 | LSN 기반 페이지 단위 |
| Redo 범위 | 커밋된 트랜잭션만 | Repeating History: 모든 변경 무차별 Redo |
| 중복 방지 | 별도 메커니즘 없음 | PageLSN으로 중복 재실행 방지 |
| Undo 안전성 | 재장애 시 복잡한 처리 | CLR로 멱등성 보장 |
| 복구 시작점 | 로그 전체 스캔 | 마지막 체크포인트부터 스캔 (DPT로 범위 최소화) |

```
[가상 로그 기반 ARIES 회복 과정 예시]

  [LSN 10, T1, Start]
  [LSN 20, T2, Start]
  [LSN 30, T1, Write(P1, old=v0, new=v1)]
  [LSN 40, T1, Commit]
  [LSN 50, T2, Write(P2, old=v0, new=v2)]
  [장애 발생]

  ① 분석: T1=Commit, T2=Active → UNDO List={T2}
  ② 재실행: LSN30, LSN50 모두 재실행 (T2 미커밋이어도 Repeating History)
  ③ 취소: T2의 LSN50 취소, CLR 기록 → T1만 커밋된 일관 상태 복구
```

ARIES는 페이지 단위 세밀한 복구와 CLR의 멱등성 보장으로, 장애 복구의 신뢰성과 RTO 최소화를 동시에 달성하는 현대 RDBMS의 표준으로 자리잡았다.  "끝"

### 실무 제언

**체크포인트 주기 최적화로 RTO 단축**
- **챌린지**: 로그 파일이 너무 커지면 분석 단계에서 스캔 시간이 오래 걸려 RTO가 늘어난다.
- **제언**: 주기적인 체크포인트로 DPT를 최신화하여 복구 스캔 범위를 대폭 줄인다. 오라클의 `FAST_START_MTTR_TARGET` 파라미터를 시스템 성격에 맞게 조율한다.

**분산 트랜잭션과의 결합**
- **챌린지**: 단일 노드 DB의 ARIES 복구만으로는 MSA 분산 환경의 전사적 무결성을 보장할 수 없다.
- **제언**: ARIES의 로컬 복구 능력에 Saga 패턴이나 2PC(Two-Phase Commit) 프로토콜을 결합하여 전사적 데이터 무결성을 달성한다.

**인메모리 DB의 ARIES 확장 적용**
- **챌린지**: 인메모리 DB(SAP HANA 등)에서는 전원 차단 시 데이터가 휘발하므로 전통적 ARIES 적용이 제한된다.
- **제언**: NVDIMM(비휘발성 메모리) 활용과 스냅샷 전략을 결합하여 인메모리 환경에 적합한 ARIES 기반 회복 아키텍처를 설계한다.
