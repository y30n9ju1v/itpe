## 해설

### 출제 배경 및 의도

현실 세계의 비즈니스 요구사항을 개념/논리적 구조(ERD)로 추상화한 후, 이를 관계형 데이터 모델로 변환하는 것은 데이터베이스 설계의 핵심 메커니즘입니다. 잘못된 관계 설정이나 식별자 전이는 데이터 중복, 연결 함정(Connection Trap), 참조 무결성 깨짐 등의 심각한 문제를 야기합니다.

본 문제는 ERD에서 관계형 데이터 모델로의 전환 능력을 검증하기 위해 출제되었습니다. 채점관은 개체와 속성 식별, 1:M 카디널리티 규정, 기본키(PK)와 외래키(FK)를 통한 참조 무결성 설계를 정확히 수행할 수 있는지를 평가합니다.

### 1. ERD와 관계형 데이터 모델, 개요

정 의  • ERD를 통해 현실 세계의 엔티티 간 업무 규칙을 다이어그램으로 도출하고, 이를 릴레이션·속성·키 기반의 2차원 테이블 스키마 체계로 변환하는 데이터 엔지니어링 과정
       - 개체와 속성 식별 → 카디널리티 규정 → PK의 FK 전이 순서로 진행

```
[ERD → 관계형 데이터 모델 변환 흐름]

 업무 명제(Input)                ERD(개념/논리)               관계형 모델(물리)
┌────────────────┐           ┌─────────────────┐         ┌──────────────────┐
│"학생 1명은     │  엔티티   │  [학 생]  1─────M│  PK/FK  │ 학생(학번PK,성명)│
│ 다수 수강신청  │ ─ 분석 ─> │  [수강신청서]    │ ──전이─>│ 수강신청서(신청번│
│ 가능"          │           │  비식별관계(점선)│         │ 호PK, 학번FK,…) │
└────────────────┘           └─────────────────┘         └──────────────────┘
```

- ERD는 인간이 업무 규칙을 직관적으로 가시화하는 도구이며, 관계형 데이터 모델은 DBMS가 데이터를 무결하게 저장하도록 구조화한 결과이다.

### 2. 가. ERD 작성 및 나. 관계형 데이터 모델 변환

1) 가. 개체 분석 및 ERD (IE/Crow's Foot 표기법)

| 구분 | 엔티티 | 주식별자 | 일반 속성 |
|------|--------|----------|-----------|
| 부모 | 학생(Student) | 학번(PK) | 성명, 전화번호 |
| 자식 | 수강신청서(Registration) | 신청번호(PK) | 학번(FK), 과목명, 교수명, 학점 |
| 관계 | 수강신청한다 | 카디널리티: 1:M | 비식별자 관계(점선) |

```
  [학 생]                                       [수강신청서]
┌──────────────┐                             ┌────────────────┐
│ # 학번       │                             │ # 신청번호     │
├──────────────┤   - - - - - - - - - - - <O┼ │ ────────────── │
│ ◯ 성명       │       (수강신청한다)        │ ◯ 학번 (FK)    │
│ ◯ 전화번호   │                             │ ◯ 과목명       │
└──────────────┘                             │ ◯ 교수명       │
                                             │ ◯ 학점         │
                                             └────────────────┘
  (부모측 '||': 필수 1,  자식측 'O<': 선택적 다(M))
```

- 수강신청서는 신청번호 자체로 유일성이 확보되므로 부모의 학번이 주식별자화될 필요가 없는 비식별자 관계(점선)로 표현한다.

2) 나. 관계형 데이터 모델 변환 결과

| 테이블 | 스키마 | 제약조건 |
|--------|--------|----------|
| 학생 | (학번, 성명, 전화번호) | PK: 학번 |
| 수강신청서 | (신청번호, 학번, 과목명, 교수명, 학점) | PK: 신청번호, FK: 학번 → 학생.학번 |

**데이터 무결성 제약조건**

| 무결성 유형 | 대상 | 규칙 |
|-------------|------|------|
| **개체 무결성** | 학번, 신청번호 | PK로서 고유(Unique)해야 하며, NULL 불가 |
| **참조 무결성** | 수강신청서.학번(FK) | 반드시 학생.학번(PK)에 존재하는 값이거나 NULL이어야 함 |

- 부모 테이블의 기본키(PK)가 자식 테이블의 외래키(FK) 속성으로 전이되어 물리적 연결 고리를 형성한다.

### 3. ERD와 관계형 데이터 모델 비교 및 트렌드

| 비교 항목 | 엔티티 관계 다이어그램 (ERD) | 관계형 데이터 모델 (Relational Model) |
|-----------|------------------------------|---------------------------------------|
| **모델링 단계** | 개념 및 논리 모델링 단계 | 물리 모델링 단계 |
| **핵심 목적** | 인간(사용자/분석가)의 업무 규칙 가시화 | DBMS의 무결한 데이터 저장 구조화 |
| **표현 요소** | 개체, 속성, 관계(Relationship) | 릴레이션, 속성, 키(PK/FK), 튜플 |
| **관계 매핑** | 선(Line)과 기호(까마귀발, 마름모 등) | 외래키(FK) 속성 전이를 통한 논리적 연결 |
| **MSA 환경** | 바운디드 컨텍스트(BC) 내 개념 모델 | 서비스별 물리 스키마 (DB-per-Service) |

- 현대 MSA 환경에서는 논리 모델상 FK 관계를 명확히 정의하되, 물리 DBMS에는 제약조건을 생략하고 애플리케이션 레이어에서 무결성을 검증하는 Loosely Coupled 전략이 확산되고 있다.  "끝"

### 실무 제언

**외래키(FK) 물리적 생성 여부의 딜레마**
- **챌린지**: DBMS에 FK 제약조건을 생성하면 데이터 삽입/삭제 시 성능 저하(Locking, 검증 부하)를 유발할 수 있다.
- **제언**: 현대 대용량 트랜잭션 시스템에서는 논리 모델상에서만 FK 관계를 명확히 정의하고, 물리 DBMS에는 제약조건을 생략하되 애플리케이션 레이어나 데이터 품질 진단 프로파일링 툴을 통해 무결성을 상시 검증하는 Loosely Coupled 전략이 권장된다.

**비식별자 vs 식별자 관계 설계 오류**
- **챌린지**: 수강신청서처럼 자체 식별자(신청번호)가 있는 엔티티에 부모 PK를 포함하는 식별자 관계를 잘못 적용하면 복합키 구조가 불필요하게 복잡해지고 이행적 종속성 위험이 증가한다.
- **제언**: ERD 설계 단계에서 각 엔티티의 독립적 식별 가능 여부를 판단하여 식별자/비식별자 관계를 명확히 구분해야 하며, 독립 식별 가능한 엔티티는 비식별자 관계(점선)를 적용하는 것이 원칙이다.

**카디널리티 오설계로 인한 데이터 품질 결함**
- **챌린지**: 1:M 관계를 1:1로 잘못 설계하거나, M:N 관계를 중간 테이블 없이 1:M으로 단순화하면 데이터 누락 또는 연결함정(Connection Trap)이 발생한다.
- **제언**: 비즈니스 시나리오 시뮬레이션을 통해 카디널리티를 검증하고, M:N 관계는 반드시 교차 엔티티(Intersection Entity)를 도입하여 명확한 1:M 구조로 해소해야 한다.
