# [기출 풀이] 학생과 수강신청서 간의 관계 모델링 (ERD 및 관계형 데이터 모델)

---

## 1. 문제를 해체하는 5대 핵심 분석 항목

### ① 출제 배경 및 의도 (Why Now?)

* 
**개념 전개의 기초 역량 검증:** 본 문제는 현실 세계의 비즈니스 요구사항을 '개념/논리적 구조(ERD)'로 추상화한 후, 이를 컴퓨터가 이해하고 저장할 수 있는 '물리적 구조(관계형 데이터 모델)'로 변환하는 데이터베이스 설계의 가장 핵심적인 메커니즘을 검증하기 위해 출제되었습니다.


* 
**데이터 무결성의 첫 단추:** 잘못된 관계 설정이나 식별자 전이는 데이터 중복, 연결 함정(Connection Trap), 참조 무결성 깨짐 등의 심각한 문제를 야기합니다. 이를 방지하기 위한 구조적 변환 절차를 정확히 숙지하고 있는지 평가하는 고전적이면서도 필수적인 기법 중심의 문제입니다.



### ② 개념의 본질과 작동 메커니즘 (What & How)

* 
**한 문장 정의:** 현실 세계의 엔티티 간 업무 규칙(수강신청)을 다이어그램(ERD) 형태로 도출하고, 이를 릴레이션(Relation), 속성(Attribute), 키(Key) 기반의 2차원 테이블 스키마 체계로 변환하는 데이터 엔지니어링 과정입니다.


* **작동 메커니즘 (Mechanism):**
* 
**Input:** "한 사람의 학생은 다수의 과목을 수강신청 할 수 있고, 수강신청서 1개는 특정 학생의 신청 정보이다"라는 업무 명제 규칙.


* 
**Mechanism:** 개체(Entity)와 속성(Attribute) 식별 $\rightarrow$ 1:M 관계성 규정 $\rightarrow$ 부모 엔티티의 주식별자(PK)를 자식 엔티티의 외래키(FK)로 전이.


* 
**Output:** 주식별자(PK)와 외래키(FK) 및 참조 무결성 제약조건이 확정된 릴레이션 스키마 세트.





### ③ 채점관이 눈을 켜고 찾는 'Must-Have' 키워드 (Keywords)

* 
**ERD 관점:** 엔티티(Entity), 관계(Relationship), 속성(Attribute), 식별자(Identifier), IE/Crow's Foot 표기법, Barker 표기법, 카디널리티(1:M).


* 
**관계형 데이터 모델 관점:** 릴레이션(Relation), 튜플(Tuple), 도메인(Domain), 기본키(PK), 외래키(FK), 참조 무결성 제약조건(Referential Integrity).



### ④ 상위/하위/유사 개념과의 상호 관계 (Context & Comparison)

데이터 모델링의 전개 과정에서 ERD와 관계형 데이터 모델은 각각 설계의 '개념/논리 단계'와 '물리 단계'를 대변하며 다음과 같은 명확한 상호 변환 관계를 가집니다.

| 비교 항목 | 엔티티 관계 다이어그램 (ERD) | 관계형 데이터 모델 (Relational Model) |
| --- | --- | --- |
| **모델링 단계** | 개념 및 논리 모델링 단계 

 | 물리 모델링 (또는 논리 모델의 테이블화) 단계 

 |
| **핵심 목적** | 인간(사용자/분석가)이 업무 규칙을 직관적으로 가시화 

 | 컴퓨터(DBMS)가 데이터를 무결하게 저장하도록 구조화 

 |
| **주요 표현 요소** | 개체(Entity), 속성(Attribute), 관계(Relationship) 

 | 릴레이션(Relation), 속성, 키(PK/FK), 튜플 

 |
| **관계 매핑 방식** | 선(Line)과 기호(까마귀발, 마름모 등)로 직접 연결 

 | 외래키(FK) 속성 전이를 통한 논리적 연결 

 |

### ⑤ 실무적 제언 및 챌린지 (So What? / Insight)

* **외래키(FK) 물리적 생성 여부의 Dilemma:** 실무 아키텍처 설계 시, 참조 무결성을 완벽히 통제하기 위해 DBMS 상에 하드웨어적 외래키(FK) 제약조건을 생성하면 데이터 삽입/삭제 시 성능 저하(Locking, 검증 부하)를 유발할 수 있습니다.
* 
**기술사의 제언:** 따라서 현대 대용량 트랜잭션 시스템(MSA 등) 환경에서는 논리 모델상에서만 FK 관계를 명확히 정의하여 데이터 계층 구조를 유지하고, 물리 DBMS에는 제약조건을 생략하되 애플리케이션 레이어(Application Layer)나 데이터 품질 진단 프로파일링 툴을 통해 무결성을 상시 검증하는 유연한 전략(Loosely Coupled)이 권장됩니다.

## 2. 합격자 수준의 모범 답안 가이드 (1교시형 풀이 방식)

### [가] 개체와 속성 분석 및 수강신청 비즈니스 규칙 정의

1. 개체(Entity) 및 속성(Attribute) 정의 

* 
**학생(Student) 엔티티:** 학번(기본 식별자), 성명, 전화번호 속성으로 구성.


* 
**수강신청서(Registration) 엔티티:** 신청번호(기본 식별자), 과목명, 교수명, 학점 속성으로 구성.



2. 카디널리티(Cardinality) 및 업무 규칙 파악 

* 한 명의 학생은 학기별로 여러 개의 과목을 수강신청(M)할 수 있습니다.


* 수강신청서 1장은 반드시 특정 한 명의 학생(1)에게 귀속됩니다.


* 따라서 두 엔티티 간의 관계는 학생(1) 대 수강신청서(M)의 의존적 '일대다' 관계가 성립됩니다.



---

### [나] 가. 학생과 수강신청서 간의 ERD (Entity Relationship Diagram)

ERD 표기법 중 실무 및 감리에서 가장 범용적으로 사용되는 **IE (Crow's Foot) 표기법**과 학술적 원형인 **Chen 표기법**을 병행하여 작성합니다.

1. IE (Crow's Foot / 까마귀발) 표기법 구조 

* 학생 엔티티의 식별자(학번)가 수강신청서 엔티티의 일반 속성 영역으로 내려오는 **비식별자 관계(점선)** 형태로 표현합니다. (수강신청서는 신청번호 자체로 유일성이 확보되므로 부모의 학번이 주식별자화 될 필요가 없음) .



```
  [학 생]                                       [수강신청서]
┌──────────────┐                             ┌────────────────┐
│ # 학번       │                             │ # 신청번호     │
├──────────────┤   - - - - - - - - - - - <O┼ │ ────────────── │
│ ◯ 성명       │       (수강신청한다)        │ ◯ 학번 (FK)    │
│ ◯ 전화번호   │                             │ ◯ 과목명       │
└──────────────┘                             │ ◯ 교수명       │
                                             │ ◯ 학점         │
                                             └────────────────┘

```

(참고: 부모 쪽 '작대기 2개'는 필수 1을 의미하며, 자식 쪽 '까마귀발과 원'은 0 또는 그 이상(모두 수강신청을 안 한 학생도 존재 가능)의 선택적 다(M)를 의미함).

2. Chen 표기법 (개념적 원형 표현) 

* 개체는 사각형, 관계는 마름모, 속성은 타원으로 연결하여 데이터의 구조적 본질을 도출합니다.



```
  ( 성명 )      ( 학번:PK )                      ( 신청번호:PK )   ( 과목명 )
     │             │                                   │             │
  ┌───────────────────┐        1 ┌──────────┐ M      ┌───────────────────┐
  │     학    생      ├──────────┤ 수강신청 ├───────┤    수강신청서     │
  └─────────┬─────────┘          └──────────┘        └─────────┬─────────┘
            │                                                  │
       ( 전화번호 )                                      ( 교수명 ) ── ( 학점 )

```

### [다] 나. 관계형 데이터 모델 (Relational Data Model) 변환 결과

ERD의 개념적 구조를 관계형 데이터베이스(RDBMS)의 논리·물리 체계인 릴레이션 스키마 및 테이블 인스턴스 형태로 변환합니다.

1. 릴레이션 스키마 (Relation Schema) 표현 

* 부모 테이블의 기본키(Primary Key)가 자식 테이블의 외래키(Foreign Key) 속성으로 전이되어 물리적 연결 고리를 형성합니다.


* 
**학생** (학번, 성명, 전화번호)  - *PK: 학번* 


* 
**수강신청서** (신청번호, *학번(FK)*, 과목명, 교수명, 학점) - *PK: 신청번호, FK: 학번* 



2. 데이터 무결성 제약조건 (Data Integrity Constraints) 명시 

* 
**개체 무결성(Entity Integrity):** 학생 테이블의 '학번'과 수강신청서 테이블의 '신청번호'는 기본키(PK)로서 고유해야 하며, 절대 NULL 값을 가질 수 없습니다.


* 
**참조 무결성(Referential Integrity):** 수강신청서 테이블의 '학번(FK)'에 들어오는 값은 반드시 학생 테이블의 '학번(PK)'에 실제로 존재하는 값이거나 NULL이어야 합니다.



3. 테이블 인스턴스 (Table Instance) 예시 

**[학생 릴레이션]** 

| 학번 (PK) | 성명 | 전화번호 |
| --- | --- | --- |
| 26-001 | 홍길동 | 010-1234-5678 |
| 26-002 | 이순신 | 010-9876-5432 |

**[수강신청서 릴레이션]** 

| 신청번호 (PK) | 학번 (FK) | 과목명 | 교수명 | 학점 |
| --- | --- | --- | --- | --- |
| REG-0001 | 26-001 | 데이터베이스 | 김기술 | 3 |
| REG-0002 | 26-001 | 소프트웨어공학 | 박공학 | 3 |
| REG-0003 | 26-002 | 인공지능활용 | 최지능 | 2 |

(설명: 학번 '26-001' 학생(홍길동)은 현재 2개의 과목을 성공적으로 수강신청(1:M)한 상태임을 물리 데이터 관계로 명확히 증명함).
