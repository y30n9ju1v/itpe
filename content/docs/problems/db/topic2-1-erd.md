## 해설

> **개념 세계에서 데이터 세계로의 변환**  
> ERD(개념/논리 모델)와 관계형 데이터 모델(물리 모델)의 구조적 변환

현실 세계의 비즈니스 요구사항을 개념/논리적 구조(ERD, Entity Relationship Diagram)로 추상화한 후, 이를 컴퓨터가 저장할 수 있는 물리적 구조(관계형 데이터 모델)로 변환하는 것은 데이터베이스 설계의 가장 핵심적인 메커니즘입니다. 잘못된 관계 설정이나 식별자 전이는 데이터 중복, 연결 함정(Connection Trap), 참조 무결성(Referential Integrity, 자식 테이블에 저장된 외래키 값이 반드시 부모 테이블에 존재해야 한다는 규칙) 깨짐 등의 심각한 문제를 야기합니다.

### 1. 출제 배경 및 의도

본 문제는 데이터베이스 설계의 가장 핵심적인 변환 메커니즘, 즉 ERD에서 관계형 데이터 모델로의 전환 능력을 검증하기 위해 출제되었습니다. 채점관은 개체와 속성 식별, 1:M 카디널리티 규정, 기본키(PK)와 외래키(FK)를 통한 참조 무결성 설계를 정확히 수행할 수 있는지를 평가합니다.

### 2. 개념의 본질과 작동 메커니즘

현실 세계의 엔티티 간 업무 규칙(수강신청)을 다이어그램(ERD) 형태로 도출하고, 이를 릴레이션(Relation), 속성(Attribute), 키(Key) 기반의 2차원 테이블 스키마 체계로 변환하는 데이터 엔지니어링 과정입니다.

- **Input:** "한 사람의 학생은 다수의 과목을 수강신청할 수 있고, 수강신청서 1개는 특정 학생의 신청 정보이다"라는 업무 명제
- **Mechanism:** 개체(Entity)와 속성(Attribute) 식별 → 1:M 관계성 규정 → 부모 엔티티의 주식별자(PK)를 자식 엔티티의 외래키(FK)로 전이
- **Output:** 주식별자(PK)와 외래키(FK) 및 참조 무결성 제약조건이 확정된 릴레이션 스키마 세트

#### 가. 개체 및 속성 분석

- **학생(Student) 엔티티:** 학번(기본 식별자), 성명, 전화번호 속성으로 구성
- **수강신청서(Registration) 엔티티:** 신청번호(기본 식별자), 과목명, 교수명, 학점 속성으로 구성
- **카디널리티(Cardinality, 관계에 참여하는 객체의 수적 비율):** 학생(1) 대 수강신청서(M)의 의존적 일대다 관계

#### 나. ERD (IE/Crow's Foot 표기법)

수강신청서는 신청번호 자체로 유일성이 확보되므로 부모의 학번이 주식별자화될 필요가 없는 **비식별자 관계(점선)** 형태입니다.

```
  [학 생]                                       [수강신청서]
┌──────────────┐                             ┌────────────────┐
│ # 학번       │                             │ # 신청번호     │
├──────────────┤   - - - - - - - - - - - <O┼ │ ────────────── │
│ ◯ 성명       │       (수강신청한다)        │ ◯ 학번 (FK)    │
│ ◯ 전화번호   │                             │ ◯ 과목명       │
└──────────────┘                             │ ◯ 교수명       │
                                             │ ◯ 학점         │
                                             └────────────────┘
```

(부모 쪽 '작대기 2개'는 필수 1, 자식 쪽 '까마귀발과 원'은 선택적 다(M)를 의미)

#### 다. 관계형 데이터 모델 (Relational Data Model) 변환

부모 테이블의 기본키(PK)가 자식 테이블의 외래키(FK) 속성으로 전이되어 물리적 연결 고리를 형성합니다.

- **학생** (학번, 성명, 전화번호) — *PK: 학번*
- **수강신청서** (신청번호, *학번(FK)*, 과목명, 교수명, 학점) — *PK: 신청번호, FK: 학번*

**데이터 무결성 제약조건**

- **개체 무결성(Entity Integrity):** 학생 테이블의 '학번'과 수강신청서 테이블의 '신청번호'는 PK로서 고유해야 하며, NULL 불가
- **참조 무결성(Referential Integrity):** 수강신청서 테이블의 '학번(FK)'에 들어오는 값은 반드시 학생 테이블의 '학번(PK)'에 존재하는 값이거나 NULL이어야 함

**테이블 인스턴스 예시**

| 학번 (PK) | 성명 | 전화번호 |
| --- | --- | --- |
| 26-001 | 홍길동 | 010-1234-5678 |
| 26-002 | 이순신 | 010-9876-5432 |

| 신청번호 (PK) | 학번 (FK) | 과목명 | 교수명 | 학점 |
| --- | --- | --- | --- | --- |
| REG-0001 | 26-001 | 데이터베이스 | 김기술 | 3 |
| REG-0002 | 26-001 | 소프트웨어공학 | 박공학 | 3 |
| REG-0003 | 26-002 | 인공지능활용 | 최지능 | 2 |

### 3. 핵심 키워드 (Must-Have)

- **ERD 관점:** 엔티티(Entity), 관계(Relationship), 속성(Attribute), 식별자(Identifier), IE/Crow's Foot 표기법, Barker 표기법, 카디널리티(1:M)
- **관계형 데이터 모델 관점:** 릴레이션(Relation), 튜플(Tuple), 도메인(Domain), 기본키(PK), 외래키(FK), 참조 무결성 제약조건(Referential Integrity)

### 4. 맥락과 비교

ERD와 관계형 데이터 모델은 설계의 '개념/논리 단계'와 '물리 단계'를 각각 대변하며 명확한 상호 변환 관계를 가집니다.

| 비교 항목 | 엔티티 관계 다이어그램 (ERD) | 관계형 데이터 모델 (Relational Model) |
| --- | --- | --- |
| **모델링 단계** | 개념 및 논리 모델링 단계 | 물리 모델링 (또는 논리 모델의 테이블화) 단계 |
| **핵심 목적** | 인간(사용자/분석가)이 업무 규칙을 직관적으로 가시화 | 컴퓨터(DBMS)가 데이터를 무결하게 저장하도록 구조화 |
| **주요 표현 요소** | 개체(Entity), 속성(Attribute), 관계(Relationship) | 릴레이션(Relation), 속성, 키(PK/FK), 튜플 |
| **관계 매핑 방식** | 선(Line)과 기호(까마귀발, 마름모 등)로 직접 연결 | 외래키(FK) 속성 전이를 통한 논리적 연결 |

### 5. 실무 제언

**외래키(FK) 물리적 생성 여부의 딜레마**

- **챌린지:** 참조 무결성을 완벽히 통제하기 위해 DBMS에 하드웨어적 FK 제약조건을 생성하면 데이터 삽입/삭제 시 성능 저하(Locking, 검증 부하)를 유발할 수 있습니다.
- **제언:** 현대 대용량 트랜잭션 시스템(MSA 등) 환경에서는 논리 모델상에서만 FK 관계를 명확히 정의하여 데이터 계층 구조를 유지하고, 물리 DBMS에는 제약조건을 생략하되 애플리케이션 레이어(Application Layer)나 데이터 품질 진단 프로파일링 툴을 통해 무결성을 상시 검증하는 유연한 전략(Loosely Coupled)이 권장됩니다.
