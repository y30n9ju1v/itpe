## 문제

DB 조인(Join) 방식인 Nested Loops Join, Sort Merge Join, Hash Join의 동작원리에 대하여 설명하고, 각 조인 방식을 활용하는데 있어 유리한 조건을 설명하시오.

## 해설

> **물리적 성능 최적화의 핵심 분수령**  
> 3대 물리 조인 알고리즘: Nested Loops, Sort Merge, Hash Join의 동작원리와 최적 활용 조건

SQL 표준의 JOIN이 논리적인 결합 조건을 정의한다면, Nested Loops(중첩 루프), Sort Merge(정렬 후 병합), Hash Join(해시 테이블 기반 매핑)은 이를 구현하기 위해 DBMS 옵티마이저가 수립하는 실제 물리적 실행 알고리즘입니다. 같은 쿼리도 어떤 알고리즘을 쓰느냐에 따라 수십 배 성능 차이가 납니다. 클라우드 인프라 환경에서 쿼리 최적화 실패는 곧 과도한 CPU, 메모리, 스토리지 I/O 비용 청구로 이어집니다. 시스템의 규모와 트랜잭션 성격(OLTP vs OLAP)을 파악하고 이에 부합하는 물리적 조인 메커니즘을 유도할 수 있는 역량이 고대역 아키텍트를 변별하는 기준이 됩니다.

### 1. 출제 배경 및 의도

세 가지 조인 방식의 동작원리를 각각 정확히 기술하고, 각 방식이 유리한 조건(데이터 볼륨, 인덱스 유무, 조인 조건 형태, 시스템 아키텍처)을 구체적으로 설명할 수 있는지가 핵심 채점 포인트입니다. 단순히 정의만 나열하는 수준을 넘어, OLTP와 OLAP 환경에서의 선택 기준과 힌트(Hint) 활용 등 실무적 통찰을 갖추었는지 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

두 개 이상의 테이블을 결합할 때, 데이터베이스 내부의 CPU 연산 방식, 인덱스 활용 여부, 메모리 버퍼 공간의 특성에 따라 최적의 경로로 데이터를 스캔하고 병합하는 물리적 질의 처리 알고리즘입니다.

**가. 중첩 루프 조인(Nested Loops Join)**

프로그래밍 언어의 2중 For 루프 제어 구조와 유사하게 작동합니다. 선행 테이블(Driving/Outer Table)에서 조건에 맞는 행을 한 건씩 추출한 뒤, 그 행의 조인 키 값을 가지고 후행 테이블(Driven/Inner Table)의 인덱스를 매번 반복하여 검색하며 매핑합니다.

1. 선행 테이블에서 필터 조건을 충족하는 첫 번째 튜플을 검색합니다.
2. 해당 튜플의 조인 키를 가지고 후행 테이블의 인덱스 버퍼 블록을 탐색합니다.
3. 인덱스 레코드가 가리키는 후행 테이블의 실물 데이터 블록(ROWID)을 찾아가 결합합니다.
4. 선행 테이블의 조건 만족 데이터가 소진될 때까지 1~3 과정을 반복합니다.

**나. 정렬 병합 조인(Sort Merge Join)**

두 테이블을 조인 컬럼 기준으로 각각 먼저 정렬(Sort)한 후, 정렬된 두 집합을 위에서부터 차례대로 스캔하면서 조건에 맞는 행들을 병합(Merge)해 나가는 방식입니다.

1. 선행 테이블을 조인 키 기준으로 정렬하여 임시 작업 공간(Sort Area)에 적재합니다.
2. 후행 테이블 역시 조인 키 기준으로 정렬하여 독립된 작업 공간에 적재합니다.
3. 정렬된 두 집합의 첫 번째 행부터 조인 키 값을 비교·스캔하면서 일치하는 데이터를 결합합니다 (양쪽 모두 정렬되어 있으므로 단 한 번의 Full Scan만으로 조인 완료).

**다. 해시 조인(Hash Join)**

메모리(Work Area / Hash Area) 공간을 적극적으로 활용하는 현대적 대용량 기법입니다.

1. **Build 단계:** 크기가 작은 테이블(Build Input)을 읽어 조인 키에 해시 함수를 적용하고, 메모리 내에 해시 맵(Hash Table)을 구축합니다.
2. **Probe 단계:** 대용량의 큰 테이블(Probe Input)을 순차 스캔하면서 각 행의 조인 키에 동일한 해시 함수를 적용합니다.
3. 해시 함수가 리턴한 주소 값을 통해 메모리 해시 맵을 즉시 탐색(Probe)하여 일치하는 행이 있으면 조인 결과 집합으로 즉시 반환합니다.

옵티마이저의 내부 판단 흐름:
```
[사용자 원시 SQL 유입]
          │
          ▼
〈조인 키에 인덱스가 있는가?〉 ─── (Yes: 소량 데이터) ───> [Nested Loops Join]
          │
          └─ (No / 대량 데이터 집계)
                │
                ▼
     〈조인 조건이 '=' 인가?〉 ─── (Yes) ───> [Hash Join]
                │
                └─── (No: 범위/비등가) ───> [Sort Merge Join]
```

### 3. 핵심 키워드 (Must-Have)

- **Nested Loops Join 관점:** 선행/외부 테이블(Driving/Outer), 후행/내부 테이블(Driven/Inner), Random I/O, 내부 테이블 인덱스, OLTP, 힌트(`USE_NL`)
- **Sort Merge Join 관점:** 정렬(Sort), 병합(Merge), 정렬 영역(Sort Area) 부하, 비등가 조인, 힌트(`USE_MERGE`)
- **Hash Join 관점:** 빌드 입력(Build Input), 프로브 입력(Probe Input), 해시 영역(Hash Area), 해시 함수, CPU 연산 중심, OLAP/DW, 등가 조인(`=`), 힌트(`USE_HASH`)

### 4. 맥락과 비교

물리적 조인 3대 기법은 시스템 환경과 리소스 제약에 따라 명확한 대칭 구조를 띱니다.

| 비교 항목 | 중첩 루프 조인 (Nested Loops Join) | 정렬 병합 조인 (Sort Merge Join) | 해시 조인 (Hash Join) |
| --- | --- | --- | --- |
| **인덱스(Index) 의존성** | **절대적/필수적** | 매우 낮음 (정렬 생략용) | **전혀 없음** (인덱스 불필요) |
| **적합한 데이터 볼륨** | **소량** 데이터 위주 | **대량** 및 범위 연산 데이터 | **대량** 대 대량 데이터 |
| **주요 활용 아키텍처** | OLTP, 대화형 웹 시스템 | 대용량 배치, 비등가 조인 환경 | OLAP, DW, 대용량 분석 시스템 |
| **최초 결과 반환 속도** | **가장 빠름** (First-Row 갱신 유리) | 느림 (전체 정렬 대기 필요) | 느림 (해시 맵 생성 대기 필요) |
| **조인 조건 제약** | 모든 조건 연산자 가능 | 모든 조건 연산자 가능 | **오직 등가 조인 (`=`)** 만 가능 |

**각 방식이 유리한 조건:**

- **NL 조인:** 소량 데이터(OLTP), 후행 테이블 조인 컬럼에 인덱스 완비, 빠른 초기 화면 응답(First-Row) 요구 시
- **Sort Merge 조인:** 조인 키에 인덱스가 없는 상황, 비등가 조인(`BETWEEN`, `>=`) 환경, 조인 대상 컬럼이 이미 클러스터드 인덱스로 정렬된 경우
- **Hash 조인:** 대용량 데이터 간 결합(DW/OLAP), 인덱스가 전무한 환경, 등가 조인(`=`) 조건, Build Input이 PGA Work Area 내에 100% 수용 가능한 경우

### 5. 실무 제언

**인덱스 누락 시 NL 조인 대참사 예방과 메모리 아키텍처 거버넌스**

- **챌린지:** 실무에서 가장 빈번한 안티 패턴은 후행 테이블의 조인 컬럼에 인덱스가 없는 상태에서 Nested Loops Join이 유도되는 경우입니다. 이 경우 선행 테이블의 행 수만큼 후행 테이블을 Full Table Scan하게 되어 시스템 전체가 마비됩니다. 대용량 Hash Join과 Sort Merge Join은 메모리 영역(PGA, Shared Pool)을 강하게 소모하므로, 전체 시스템의 메모리 설정(`PGA_AGGREGATE_TARGET`)이 너무 작으면 임시 Temp Tablespace 디스크 쓰기가 발생하여 성능이 수백 배 저하됩니다.
- **제언:** 기술사는 조인 키의 분포도와 카디널리티를 정량 분석하고, 랜덤 I/O를 유발하는 NL 조인의 한계를 인지하여 대량 데이터 집계 쿼리에서는 과감히 Hash Join으로의 유도를 위해 실행 계획을 제어(`HINT` 활용)해야 합니다. 현대의 분산 쿼리 엔진(Apache Spark 등)에서는 소량 테이블을 모든 노드로 복제하여 로컬에서 조인하는 브로드캐스트 조인(Broadcast Join) 패턴으로 진화하고 있으며, 이 역시 전통적인 Hash Join 원리를 네트워크 분산 인프라에 투영한 결과물임을 이해하고 시스템 구축 초기부터 정규화를 통한 데이터 독립성 보장과 운영 단계에서의 인덱스 기반 NL 조인 제어 및 인-메모리 Hash Join 최적화를 정량적 데이터 기반으로 조율해야 합니다.
