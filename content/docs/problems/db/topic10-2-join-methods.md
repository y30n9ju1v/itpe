## 문제

DB 조인(Join) 방식인 Nested Loops Join, Sort Merge Join, Hash Join의 동작원리에 대하여 설명하고, 각 조인 방식을 활용하는데 있어 유리한 조건을 설명하시오.

## 해설

### 출제 배경 및 의도

SQL 표준의 JOIN이 논리적인 결합 조건을 정의한다면, 실제 DBMS 옵티마이저는 Nested Loops, Sort Merge, Hash Join 중 하나를 선택하여 물리적으로 실행합니다. 같은 쿼리도 어떤 알고리즘을 쓰느냐에 따라 수십 배 성능 차이가 발생하며, 클라우드 인프라 환경에서는 쿼리 최적화 실패가 곧 과도한 CPU·메모리·I/O 비용으로 이어집니다.

OLTP(소량·인덱스 중심)와 OLAP(대용량·집계 중심)에서 요구되는 알고리즘이 상이하며, 현대 분산 쿼리 엔진(Spark SQL, Trino)에서도 동일한 원리가 투영됩니다. 채점관은 3대 물리 조인 알고리즘의 동작원리를 각각 정확히 기술하고, 유리한 조건(데이터 볼륨·인덱스 유무·조인 조건 형태·시스템 아키텍처)을 구체적으로 설명할 수 있는지 검증하고자 합니다.

힌트(Hint) 활용 등 실무적 통찰을 갖추었는지가 핵심 채점 포인트입니다.

### 1. 물리 조인 알고리즘, 개요

정 의  • 두 개 이상의 테이블을 결합할 때 데이터베이스 내부의 CPU 연산 방식·인덱스 활용 여부·메모리 버퍼 공간 특성에 따라 최적 경로로 데이터를 스캔·병합하는 물리적 질의 처리 알고리즘
       - 논리적 JOIN 유형(Inner, Outer 등)과 독립적으로, 옵티마이저가 비용(Cost)을 계산하여 자동 선택하거나 힌트(Hint)로 강제 제어 가능

```
[옵티마이저 물리 조인 알고리즘 선택 흐름]

[사용자 원시 SQL 유입]
          │
          ▼
〈조인 키에 인덱스가 있는가?〉 ─── (Yes: 소량 데이터) ───> [Nested Loops Join]
          │
          └─ (No / 대량 데이터 집계)
                │
                ▼
     〈조인 조건이 '=' 인가?〉 ─── (Yes) ───────────────> [Hash Join]
                │
                └─── (No: 범위/비등가) ──────────────> [Sort Merge Join]

힌트 제어: USE_NL / USE_MERGE / USE_HASH
```

- 물리 조인 알고리즘 선택은 인덱스 유무·조인 조건 형태·데이터 볼륨의 3요소로 결정된다.

### 2. 3대 물리 조인 알고리즘 동작원리 및 유리한 조건

1) 3대 알고리즘 동작원리

| 구분 | 알고리즘 | 단계 | 유리한 조건 |
|------|---------|------|-----------|
| **①** | **Nested Loops Join** | ① Driving 테이블에서 필터 만족 튜플 추출 → ② 해당 키로 후행 테이블 인덱스 탐색(Random I/O) → ③ ROWID로 데이터 블록 접근 → ④ 선행 데이터 소진까지 반복 | 소량 OLTP, 후행 인덱스 완비, First-Row 응답 요구 |
| **②** | **Sort Merge Join** | ① 양쪽 테이블을 조인 키 기준 정렬 → Sort Area 적재 → ② 정렬된 두 집합 첫 행부터 키 비교·스캔 → ③ 단 1회 Full Scan으로 병합 완료 | 인덱스 부재, 비등가 조인(`BETWEEN`, `>=`), 이미 정렬된 컬럼 |
| **③** | **Hash Join** | ① Build: 소량 테이블 → 조인 키에 해시 함수 적용 → 메모리 해시 맵 구성 → ② Probe: 대용량 테이블 순차 스캔 → 동일 해시 함수로 해시 맵 탐색 → 일치 시 반환 | 대용량 OLAP/DW, 인덱스 부재, 등가 조인(`=`), Build Input이 PGA에 수용 가능 |

- 알고리즘 선택 오류는 시스템 마비로 직결되므로, 데이터 특성 정량 분석이 선행되어야 한다.

2) 3대 알고리즘 비교

| 비교 항목 | Nested Loops Join | Sort Merge Join | Hash Join |
|----------|------------------|----------------|-----------|
| **인덱스 의존성** | 필수 (없으면 파탄) | 매우 낮음 | 전혀 없음 |
| **적합 데이터 규모** | 소량 (OLTP) | 대량, 비등가 | 대용량 (OLAP/DW) |
| **최초 결과 반환** | 가장 빠름 (First-Row) | 느림 (전체 정렬 대기) | 느림 (해시 맵 생성 대기) |
| **조인 조건 제약** | 모든 조건 | 모든 조건 | 등가 조인(`=`)만 가능 |
| **I/O 패턴** | Random I/O | Sequential I/O | Sequential I/O |
| **힌트 제어** | `USE_NL` | `USE_MERGE` | `USE_HASH` |

- 알고리즘별 유리한 조건을 명확히 파악하고, 힌트 또는 통계 정보 갱신으로 실행 계획을 제어해야 한다.

### 3. 현대 분산 환경 적용 및 성능 최적화 트렌드

| 관점 | 내용 |
|------|------|
| **NL 조인 최적화** | Driving 테이블 소량 집합 선정 + 후행 테이블 조인 컬럼 인덱스 필수 설계 |
| **Hash Join 메모리 제어** | `PGA_AGGREGATE_TARGET` 적절 설정; Build Input이 Work Area 초과 시 Temp Tablespace 디스크 쓰기로 수백 배 성능 저하 |
| **Sort Merge 활용** | 비등가 조인 또는 이미 클러스터드 인덱스로 정렬된 대용량 집합에 유리 |
| **분산 쿼리 엔진** | Spark SQL의 브로드캐스트 조인(Broadcast Join): 소량 테이블을 모든 노드로 복제하여 로컬 Hash Join — 전통 Hash Join 원리의 분산 진화 형태 |
| **실행 계획 제어** | 통계 정보 정기 갱신(`ANALYZE`/`DBMS_STATS`) + 힌트로 알고리즘 강제 유도 |

- 현대 클라우드 분산 쿼리 엔진에서도 전통 3대 조인 알고리즘 원리가 그대로 투영되며, 브로드캐스트 조인이 Hash Join의 분산 진화 형태로 정착하고 있다.  "끝"

### 실무 제언

**NL 조인 인덱스 누락 대참사 예방**
- **챌린지**: 후행 테이블의 조인 컬럼에 인덱스가 없는 상태에서 NL 조인이 유도되면, 선행 테이블 행 수만큼 Full Table Scan이 반복되어 시스템 전체가 마비된다.
- **제언**: 조인 키의 분포도·카디널리티를 정량 분석하여 인덱스를 사전 설계하고, 대량 데이터 집계 쿼리에서는 힌트(`USE_HASH`) 또는 통계 정보 갱신으로 Hash Join을 유도해야 한다.

**Hash Join 메모리 아키텍처 거버넌스**
- **챌린지**: `PGA_AGGREGATE_TARGET` 설정이 너무 작으면 Hash Join·Sort Merge Join의 Work Area가 부족하여 Temp Tablespace 디스크 쓰기가 발생하고 성능이 수백 배 저하된다.
- **제언**: 대용량 OLAP 쿼리 비중에 맞춰 PGA 메모리를 충분히 확보하고, Build Input 크기가 Work Area를 초과하는지 실행 계획의 `HASH JOIN BATCHED` 여부로 사전 검증해야 한다.

**분산 환경에서의 브로드캐스트 조인 전략**
- **챌린지**: Spark SQL 등 분산 쿼리 엔진에서 소량 테이블을 브로드캐스트하지 않으면 모든 노드 간 네트워크 셔플(Shuffle)이 발생하여 클라우드 비용이 폭증한다.
- **제언**: 소량 테이블(기본값 10MB 이하)은 `broadcast hint` 또는 `spark.sql.autoBroadcastJoinThreshold` 설정으로 브로드캐스트 조인을 강제 적용하여 셔플 비용을 원천 차단해야 한다.
