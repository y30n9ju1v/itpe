## 해설

> **릴레이션 조작 시 데이터 정합성을 유지하기 위해 연산 전후의 데이터 상태에 명세하는 의미적 제약조건 체계**  
> 릴레이션 무결성 규칙(Relation Integrity Rules)과 입체적 구현 전략

릴레이션 무결성 규칙은 데이터베이스에 잘못된 데이터가 유입되거나 조작되는 것을 물리적·논리적으로 차단하는 가장 근본적인 통제 규정입니다. 클라우드 이관·MSA 확산·데이터 수명주기 관리의 정밀화가 가속되는 현대 IT 환경에서, 데이터베이스의 기초 공학적 성질인 무결성 제약을 상황별로 정교하게 매핑할 수 있는 아키텍트의 기본기가 요구됩니다.

### 1. 출제 배경 및 의도

릴레이션 무결성 규칙은 데이터 신뢰성 확보의 최전방 방어선입니다. 채점관은 상태제약·과도제약·튜플제약·집합제약의 4가지 제약 유형을 이해하고, 각각을 선언적 DDL 제약조건, 절차적 트리거, 애플리케이션 검증 레이어로 구현하는 능력을 검증하고자 합니다.

또한 대용량 배치 환경에서 즉시제약과 지연제약을 적절히 조율하여 성능과 무결성을 동시에 확보하는 아키텍처 판단력도 평가 대상입니다.

### 2. 개념의 본질과 작동 메커니즘

릴레이션을 조작(삽입, 삭제, 갱신)하는 과정에서 데이터의 정합성과 품질을 유지하기 위해, 연산 수행 전후의 데이터 상태 및 변환 과정에 명세하는 의미적 제약조건(Constraint) 체계입니다.

무결성은 크게 속성에 정의된 값의 범위를 규정하는 **도메인 무결성**과, 관계 조작 과정의 의미적 관계를 명세하는 **릴레이션 무결성**으로 분화됩니다. 릴레이션 무결성 규정은 묵시적 무결성(모델 구조 정의 자체에 의해 자동 성립)과 명시적 무결성(사용자가 명시적으로 선언)으로 나뉩니다.

### 3. 핵심 키워드 (Must-Have)

#### 1) 상태제약 (State Constraint)

- **본질**: 데이터베이스가 항상 일관성 있는 상태를 유지하기 위해 구비해야 하는 조건. 특정 시점에 만족되어야 하는 정적제약(Static)
- **실무 사례**: `나이` 컬럼의 값은 언제 조회하더라도 항상 0보다 큰 양수여야 한다는 제약
- **구현 기법**: DDL의 `CHECK (나이 > 0)`으로 선언적 구현

#### 2) 과도제약 (Transition Constraint)

- **본질**: 데이터 상태가 변환되는 트랜잭션 과정에서 적용되는 규칙. 변경 전후 데이터를 모두 대조해야 하는 동적제약(Dynamic)
- **실무 사례**: 사원의 직급이 '대리'에서 '과장'으로 승진할 때, 연봉은 기존 금액보다 반드시 인상되어야 하며 삭감될 수 없다는 전후 비교 규칙
- **구현 기법**: BEFORE UPDATE 트리거로 `:OLD.급여`와 `:NEW.급여` 비교

#### 3) 집합제약 (Set Constraint)

- **본질**: 특정 단일 행의 연산에 그치지 않고, 릴레이션 내에 존재하는 튜플 집합 전체의 유기적 관계성과 분포도에 관련되어 적용되는 통제 규칙
- **실무 사례**: '데이터베이스' 과목의 총 수강신청 학생 수(Count)가 강의실 수용 능력을 고려하여 40명을 초과할 수 없다는 집합적 통제
- **구현 기법**: 트리거 내에서 `SELECT COUNT(*)` 집계 후 조건 검사

#### 4) 튜플제약 (Tuple Constraint)

- **본질**: 전체 집합의 상태와 무관하게, 현재 트랜잭션에 의해 처리되고 있는 단일 튜플(행) 내부의 속성 간 인과 관계에만 독립적으로 적용되는 규칙
- **실무 사례**: 계약 건을 입력할 때 `계약종료일자`는 반드시 동일 행 내의 `계약시작일자`보다 후행해야 한다는 단일 행 제약
- **구현 기법**: DDL의 `CHECK (퇴사일자 >= 입사일자)`로 선언적 구현

### 4. 맥락과 비교

릴레이션 무결성 제약은 통제의 범위와 검사 시점에 따라 수평적·수직적 대칭 구조를 형성합니다.

| 분류 기준 | 제약 유형 | 핵심 정의 및 특징 | 실무 적용 사례 |
|-----------|----------|----------------|--------------|
| **적용 시점** | **상태제약 (State)** | 특정 시점에 데이터베이스가 항상 만족해야 하는 **정적 제약** | "고객의 나이는 항상 0보다 커야 한다." |
| **적용 시점** | **과도제약 (Transition)** | 데이터 상태가 변환되는 과정에서 적용되는 **동적 제약** | "사원의 급여는 기존보다 삭감될 수 없다." |
| **검사 범위** | **튜플제약 (Tuple)** | 현재 처리되고 있는 **단일 행(Tuple)**에만 적용되는 규칙 | "주문일자는 오늘 날짜보다 클 수 없다." |
| **검사 범위** | **집합제약 (Set)** | 테이블 내의 **튜플 집합 전체**에 관련되어 적용되는 규칙 | "특정 학과의 총 학생 수는 50명을 초과할 수 없다." |

릴레이션 무결성 구현 기법은 3가지 레이어로 배치됩니다.

```
[다층 무결성 구현 아키텍처]
 ├── 1. 선언적 무결성 구현 (Declarative) ── DDL 스크립트 기반 PK/FK/CHECK 내장 제약
 ├── 2. 절차적 무결성 구현 (Procedural)  ── DBMS 커널 단 Trigger 및 Stored Procedure 가동
 └── 3. 애플리케이션 무결성 구현 (Application) ── 백엔드 Validation Layer 및 분산 Saga 패턴 매핑
```

### 5. 실무 제언

**즉시제약과 지연제약의 조율 딜레마**

- **챌린지**: 대용량 배치(Batch) 작업이나 마이그레이션 수행 시, 모든 행의 연산마다 무결성을 즉시 검사(`IMMEDIATE`)하면 시스템이 심각한 병목에 직면합니다. 결산 배치나 DW ETL 환경에서는 트랜잭션 Lock 현상과 극심한 CPU 오버헤드를 유발하여 시스템을 마비시킵니다.
- **제언**: 대량 데이터 배치 연산이나 다단계 연동 트랜잭션 구간에 한해서는 무결성 검증 시점을 `COMMIT` 시점까지 유예시키는 **지연제약(Deferred Constraint)** 구조를 스키마 설계 단에 전략적으로 반영해야 합니다.

```sql
ALTER TABLE 계약 ADD CONSTRAINT FK_고객
FOREIGN KEY(고객ID) REFERENCES 고객(고객ID)
DEFERRABLE INITIALLY DEFERRED; -- 커밋 시점에 한 번에 검사하여 대량 처리 최적화
```

데이터 아키텍트(DA)는 릴레이션 무결성 제약의 성질(상태/과도, 튜플/집합)을 명확히 분류하고, "선언적 DDL 제약조건을 통한 고성능 무결성 방어", "과도제약 대응을 위한 트리거의 제한적 활용", "대량 데이터 처리를 위한 지연제약 기술 융합"의 입체적 품질 관리 거버넌스를 지휘해야 합니다.
