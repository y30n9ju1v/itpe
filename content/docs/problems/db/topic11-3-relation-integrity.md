## 해설

### 출제 배경 및 의도

릴레이션 무결성 규칙은 데이터베이스에 잘못된 데이터가 유입되거나 조작되는 것을 물리적·논리적으로 차단하는 가장 근본적인 통제 규정입니다. 클라우드 이관·MSA 확산·데이터 수명주기 관리의 정밀화가 가속되는 현대 IT 환경에서 무결성 제약을 상황별로 정교하게 매핑할 수 있는 아키텍트의 기본기가 요구됩니다.

채점관은 상태제약·과도제약·튜플제약·집합제약의 4가지 제약 유형을 이해하고, 각각을 선언적 DDL·절차적 트리거·애플리케이션 검증 레이어로 구현하는 능력을 검증하고자 합니다. 대용량 배치 환경에서 즉시제약과 지연제약을 조율하는 아키텍처 판단력도 핵심 채점 포인트입니다.

### 1. 릴레이션 무결성 규칙, 개요

정 의  • 릴레이션을 조작(삽입·삭제·갱신)하는 과정에서 데이터 정합성과 품질을 유지하기 위해 연산 수행 전후의 데이터 상태 및 변환 과정에 명세하는 의미적 제약조건(Constraint) 체계
       - 묵시적 무결성(모델 구조 정의에 의해 자동 성립)과 명시적 무결성(사용자 명시 선언)으로 분류되며, 도메인 무결성과 릴레이션 무결성으로 분화

```
[릴레이션 무결성 4대 제약 분류 구조]

  분류 기준: 적용 시점            분류 기준: 검사 범위
  ┌────────────────────┐          ┌────────────────────┐
  │ 상태제약 (Static)  │          │ 튜플제약 (Tuple)   │
  │ - 특정 시점 정적   │          │ - 단일 행 속성 간  │
  │ - CHECK 선언       │          │ - CHECK 선언       │
  ├────────────────────┤          ├────────────────────┤
  │ 과도제약 (Dynamic) │          │ 집합제약 (Set)     │
  │ - 변환 과정 동적   │          │ - 집합 전체 분포   │
  │ - Trigger 구현     │          │ - Trigger 구현     │
  └────────────────────┘          └────────────────────┘
```

- 릴레이션 무결성 규칙은 적용 시점(정적/동적)과 검사 범위(튜플/집합)의 2차원 분류 체계로 이해해야 한다.

### 2. 4대 제약 유형 및 구현 기법

1) 4대 릴레이션 무결성 제약

| 분류 기준 | 제약 유형 | 핵심 정의 | 실무 사례 | 구현 기법 |
|----------|---------|---------|---------|---------|
| **적용 시점** | **상태제약** (State) | 특정 시점에 항상 만족해야 하는 정적 제약 | "나이 컬럼은 항상 0보다 커야 한다" | `CHECK (나이 > 0)` DDL 선언 |
| **적용 시점** | **과도제약** (Transition) | 상태 변환 과정의 전후 비교 동적 제약 | "사원 급여는 기존보다 삭감 불가" | `BEFORE UPDATE` Trigger로 `:OLD.급여 vs :NEW.급여` 비교 |
| **검사 범위** | **튜플제약** (Tuple) | 단일 행(Tuple) 내부 속성 간 인과 관계 규칙 | "계약종료일자 >= 계약시작일자 필수" | `CHECK (퇴사일자 >= 입사일자)` DDL 선언 |
| **검사 범위** | **집합제약** (Set) | 릴레이션 내 튜플 집합 전체의 분포도 통제 | "특정 과목 수강신청 학생 수 40명 초과 불가" | Trigger 내 `SELECT COUNT(*)` 집계 후 조건 검사 |

- 상태·튜플 제약은 DDL 선언적 구현이, 과도·집합 제약은 Trigger 절차적 구현이 적합하다.

2) 릴레이션 무결성 3대 구현 레이어

| 구현 레이어 | 방법 | 세부 기법 | 특징 |
|-----------|------|---------|------|
| **선언적** (Declarative) | DDL 내장 제약조건 | PK, FK, CHECK, NOT NULL, UNIQUE | 성능 우수, 직관적, 간단한 정적 규칙 적합 |
| **절차적** (Procedural) | Trigger / Stored Procedure | DML 전후 자동 구동 SQL 블록 | 동적·집합 제약 구현 가능, 디버깅 복잡 |
| **애플리케이션** (Application) | Bean Validation / 비즈니스 로직 | `@NotNull`, `@Pattern`, 도메인 서비스 예외 | DBMS 독립적, 테스트 용이, 우회 접근 시 취약 |

- 3대 구현 레이어는 상호 보완적으로 배치하여 다층 무결성 방어 아키텍처를 구성해야 한다.

### 3. 즉시제약 vs 지연제약 및 현대 환경 적용 트렌드

| 구분 | 즉시제약 (Immediate Constraint) | 지연제약 (Deferred Constraint) |
|------|-------------------------------|-------------------------------|
| **검사 시점** | 각 DML 연산 직후 즉시 검사 | COMMIT 시점에 일괄 검사 |
| **적합 환경** | 단건 OLTP 트랜잭션 | 대량 배치, 마이그레이션, 다단계 연동 |
| **DDL 예시** | `CONSTRAINT ... NOT DEFERRABLE` (기본값) | `DEFERRABLE INITIALLY DEFERRED` |
| **성능 영향** | 연산마다 Lock + 검증 오버헤드 누적 | Commit 시 일괄 처리로 병목 최소화 |

```sql
ALTER TABLE 계약 ADD CONSTRAINT FK_고객
FOREIGN KEY(고객ID) REFERENCES 고객(고객ID)
DEFERRABLE INITIALLY DEFERRED; -- 커밋 시점에 한 번에 검사하여 대량 처리 최적화
```

- 데이터 아키텍트는 "선언적 DDL로 고성능 무결성 방어 → 과도·집합 제약은 Trigger 제한적 활용 → 대량 배치는 지연제약 기술 융합"의 입체적 품질 관리 거버넌스를 지휘해야 한다.  "끝"

### 실무 제언

**즉시제약과 지연제약의 조율 딜레마**
- **챌린지**: 대용량 배치·마이그레이션 수행 시 모든 행 연산마다 즉시 무결성 검사(`IMMEDIATE`)를 수행하면 트랜잭션 Lock과 극심한 CPU 오버헤드로 시스템이 마비된다.
- **제언**: 대량 데이터 배치 연산과 다단계 연동 트랜잭션 구간에는 무결성 검증 시점을 `COMMIT`까지 유예시키는 지연제약(Deferred Constraint)을 스키마 설계 단에 전략적으로 반영하여 성능과 무결성을 동시에 확보해야 한다.

**과도제약 Trigger의 제한적 활용**
- **챌린지**: 모든 과도제약(급여 삭감 방지, 상태 전이 규칙 등)을 DB Trigger로 구현하면 디버깅 난이도가 극상으로 올라가고, 성능 모니터링이 불가능해지며 애플리케이션 레이어와의 이중 검증으로 관리 복잡도가 폭증한다.
- **제언**: 과도제약은 감사 목적의 핵심 규칙에 한해 Trigger로 구현하고, 나머지는 도메인 서비스 레이어의 비즈니스 로직으로 이관하여 테스트 가능성과 유지보수성을 우선 확보해야 한다.

**집합제약의 분산 환경 구현**
- **챌린지**: MSA 환경에서 서비스 경계별로 DB가 분리되면 집합제약(수강신청 인원 제한 등)을 위한 `COUNT(*)` 집계 Trigger를 물리적으로 설정할 수 없어 제약 단절 현상이 발생한다.
- **제언**: 분산 환경의 집합제약은 Redis 분산 카운터·Kafka 이벤트 집계·애플리케이션 레이어 동시성 제어(Optimistic/Pessimistic Lock)를 조합한 애플리케이션 사이드 집합 통제 아키텍처로 대응해야 한다.
