## 해설

### 출제 배경 및 의도

ANSI/SPARC 3단계 구조는 1975년 미국국립표준협회(ANSI) 산하 SPARC 위원회에서 정의한 DBMS의 표준 아키텍처로, 반세기가 지난 지금도 전 세계 모든 상용 DBMS(Oracle, MySQL, PostgreSQL 등)의 내부 논리적 뼈대를 지탱하고 있다. 사용자 화면(애플리케이션)을 건드리지 않고 물리적 저장 장치를 교체하거나(물리적 독립성), 비즈니스 요구사항에 따라 데이터 테이블 구조를 변경하더라도 사용자 프로그램에 영향을 주지 않는(논리적 독립성) 구조적 유연성이 아키텍처의 핵심 가치이다.

클라우드 네이티브 전환과 스토리지 가상화 환경이 보편화된 현재, 이 모델이 내포하는 데이터 독립성의 공학적 메커니즘은 더욱 중요한 가치를 가진다. 채점관은 수험생이 이 고전적 표준 모델 속에 내포된 데이터 독립성의 공학적 메커니즘을 정밀하게 이해하고 있는지 검증하고자 한다.

### 1. ANSI/SPARC 3단계 구조, 개요

정 의  • 사용자·전사·물리 저장의 3단계로 DB 구조를 격리·추상화하여 데이터 독립성을 보장하는 표준 DBMS 구조 모델
       - 각 단계 간 사상(Mapping)을 통해 상위 단계 변경이 하위 단계에 영향을 주지 않도록 설계

```
[ANSI/SPARC 3단계 아키텍처 구조도]

  사용자 A   사용자 B   사용자 C
     │          │          │
     ▼          ▼          ▼
 ┌──────────────────────────────┐
 │  외부 단계 (External Level)  │  ← 외부 스키마 (View, 서브스키마)
 │  [뷰A]   [뷰B]   [뷰C]      │     다수 공존 가능
 └──────────────┬───────────────┘
                │  외부-개념 사상 (논리적 사상)
                ▼
 ┌──────────────────────────────┐
 │ 개념 단계 (Conceptual Level) │  ← 개념 스키마 (전사 통합, 1개)
 │  [엔티티·관계·제약조건 통합] │     DA·데이터 아키텍트 담당
 └──────────────┬───────────────┘
                │  개념-내부 사상 (물리적 사상)
                ▼
 ┌──────────────────────────────┐
 │  내부 단계 (Internal Level)  │  ← 내부 스키마 (저장 스키마)
 │  [인덱스·블록·압축·레이아웃] │     DBA·시스템 엔지니어 담당
 └──────────────┬───────────────┘
                │
                ▼
          물리적 저장 장치 (HDD / SSD)
```

- 3단계 분리로 각 레벨의 변경이 다른 레벨에 영향을 미치지 않는 데이터 독립성을 실현한다.

### 2. ANSI/SPARC 3단계 구조의 핵심 내용

1) 3단계 스키마 구성

| 구분 | 스키마명 | 대상자 | 모델링 단계 | 핵심 내용 |
|------|---------|--------|------------|---------|
| 외부 단계 | 외부 스키마 (External Schema) | 최종 사용자, 응용 프로그래머 | 외부 설계 | 뷰(View), 서브스키마, 다수 공존 |
| 개념 단계 | 개념 스키마 (Conceptual Schema) | DA, 데이터 아키텍트 | 개념 모델링 | 엔티티·관계·제약조건·접근권한 통합, 조직 내 1개 존재 |
| 내부 단계 | 내부 스키마 (Internal Schema) | DBA, 시스템 엔지니어 | 물리 모델링 | 인덱스 배치, 데이터 압축, 저장 레코드 레이아웃 |

- 외부 스키마는 다수 공존 가능하나, 개념 스키마는 전사 관점에서 유일하게 1개만 존재한다.

2) 사상(Mapping)의 구조

| 구분 | 사상 유형 | 메커니즘 | 달성 독립성 |
|------|---------|---------|-----------|
| 외부-개념 사상 | 논리적 사상 | 개념 스키마 변경 시 사상 규칙만 수정, 외부 스키마 불변 | 논리적 데이터 독립성 |
| 개념-내부 사상 | 물리적 사상 | 내부 스키마 변경 시 사상 규칙만 수정, 개념 스키마 불변 | 물리적 데이터 독립성 |

- 사상 계층의 분리가 애플리케이션과 저장 구조 간 결합도를 제거하는 핵심 메커니즘이다.

### 3. 데이터 독립성과 현대적 확장

| 독립성 유형 | 정의 | 실무 사례 |
|-----------|------|---------|
| 논리적 데이터 독립성 | 개념 스키마 변경 시 외부 스키마·응용 프로그램 수정 불필요 | `회원` 테이블을 `회원기본`+`회원인증`으로 분해해도 기존 뷰 SQL 수정 불필요 |
| 물리적 데이터 독립성 | 내부 스키마 변경 시 개념 스키마·응용 프로그램 수정 불필요 | 인덱스 추가·파티셔닝 적용 후에도 SELECT 문 변경 없음 |

현대 데이터 레이크하우스(Lakehouse) 아키텍처에서는 저장 공간(AWS S3)인 물리 내부 레이어와 컴퓨팅 엔진(Spark, Trino)인 논리 개념 레이어를 철저히 분리하여, ANSI/SPARC 모델의 물리적 데이터 독립성을 클라우드 분산 시스템에 투영한다.  "끝"

### 실무 제언

**사상(Mapping) 오버헤드 최소화**
- **챌린지**: SQL 질의마다 외부→개념→내부 사상 변환이 수행되므로 연산 오버헤드가 발생한다.
- **제언**: 자주 실행되는 사상 경로와 실행 계획(Execution Plan)을 DBMS 라이브러리 캐시(Library Cache)에 고정 유지(소프트 파싱 유도)하여 오버헤드를 최소화한다.

**클라우드 분산 환경에서의 독립성 설계**
- **챌린지**: 단일 RDBMS 내부에 국한되었던 3단계 독립성 철학을 MSA·클라우드 네이티브 분산 아키텍처에 적용해야 한다.
- **제언**: 데이터 카탈로그(Data Catalog)와 데이터 사전의 품질을 상시 동기화하여 사상 정보와 스키마 명세의 불일치를 방지한다.

**다중 DBMS 환경의 스키마 통합 거버넌스**
- **챌린지**: MSA 환경에서 서비스별로 다른 DBMS를 혼용하면 개념 스키마가 서비스 경계별로 파편화되어 전사 데이터 일관성이 붕괴된다.
- **제언**: 전사 데이터 아키텍처(EA) 관점에서 논리 데이터 모델(개념 스키마)을 단일 마스터로 유지하고, 물리 구현은 DBMS별로 분리하여 관리한다.
