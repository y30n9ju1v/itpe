## 해설

> **사용자·전사·물리 저장의 3단계로 데이터베이스 구조를 격리·추상화하여 데이터 독립성을 보장하는 표준 DBMS 구조 모델**  
> ANSI/SPARC 3-Level Architecture와 데이터 독립성

ANSI/SPARC 3단계 구조는 1975년 미국국립표준협회(ANSI) 산하 SPARC 위원회에서 정의한 DBMS의 표준 아키텍처입니다. 사용자 관점·전사 관점·물리 저장 관점을 분리해 두었기 때문에, 반세기가 지난 지금도 전 세계 모든 상용 DBMS(Oracle, MySQL, PostgreSQL 등)의 내부 논리적 뼈대를 지탱하고 있습니다. 클라우드 네이티브 전환과 스토리지 가상화 환경이 보편화된 현재, 이 모델이 내포하는 데이터 독립성의 공학적 메커니즘은 더욱 중요한 가치를 가집니다.

### 1. 출제 배경 및 의도

ANSI/SPARC 3단계 구조는 DBMS 표준 아키텍처의 근간으로, 반세기가 지난 지금도 모든 상용 DBMS의 내부 뼈대를 지탱합니다. 사용자 화면(애플리케이션)을 건드리지 않고 물리적 저장 장치를 교체하거나(물리적 독립성), 비즈니스 요구사항에 따라 데이터 테이블 구조를 바꾸더라도 사용자 프로그램에 영향을 주지 않는(논리적 독립성) 구조적 유연성이 아키텍처의 핵심 가치입니다.

채점관은 수험생이 이 고전적 표준 모델 속에 내포된 데이터 독립성의 공학적 메커니즘을 정밀하게 이해하고 있는지 검증하고자 합니다.

### 2. 개념의 본질과 작동 메커니즘

사용자의 관점, 전사적 관점, 물리적 저장 장치 관점의 3단계로 데이터베이스 구조를 격리·추상화하고, 각 단계 간의 상호 사상(Mapping)을 통해 데이터 독립성을 보장하는 표준 DBMS 구조 모델입니다.

작동 메커니즘은 다음과 같습니다. 사용자 질의(External View / SQL Query)가 입력되면, 1차 논리적 사상(외부-개념 사상)을 통해 전사 공통 스키마로 변환되고, 2차 물리적 사상(개념-내부 사상)을 통해 실제 하드웨어가 이해하는 파일 블록 주소로 변환됩니다. 디스크에서 물리 데이터를 스캔하여 역순으로 추상화 버퍼를 거쳐 사용자 화면으로 최종 반환됩니다.

### 3. 핵심 키워드 (Must-Have)

#### 1) 외부 스키마 (External Schema / 서브 스키마)

- **본질**: 전체 데이터베이스 중 개별 응용 프로그래머나 현업 사용자가 관심을 가지는 일부 데이터 구조의 집합. 최종 사용자·응용 프로그래머가 대상
- **Must-Have 키워드**: 뷰(View), 서브스키마, 다수 공존, 외부 설계 단계에 대응
- **실무적 형상**: 데이터베이스의 가상 테이블인 뷰(View) 기법이나 특정 API 전용 쿼리 덤프. 하나의 데이터베이스 시스템 안에 수많은 외부 스키마가 공존할 수 있음

#### 2) 개념 스키마 (Conceptual Schema)

- **본질**: 데이터베이스 전체를 아우르는 전사적 관점의 유일한 통합 논리 구조. 데이터 관리자(DA)·데이터 아키텍트가 대상
- **Must-Have 키워드**: 개념 모델링 단계에 대응, 조직 전체에 유일하게 1개 존재, 엔티티·관계·제약조건·접근 권한 통합 기술
- **실무적 형상**: 모든 물리 테이블의 구조, 속성 데이터 타입, 도메인 제약조건, PK/FK 등이 집약된 마스터 뼈대

#### 3) 내부 스키마 (Internal Schema / 저장 스키마)

- **본질**: 물리적 저장 장치(하드디스크, SSD 등) 관점에서 데이터가 실물 스토리지 엔진에 실제로 어떻게 저장되고 배치되는가를 기술한 기술적 구조. DBA·시스템 엔지니어가 대상
- **Must-Have 키워드**: 물리 모델링 단계에 대응, 인덱스 배치, 데이터 압축 방식, 저장 레코드 레이아웃
- **실무적 형상**: 테이블 스페이스 할당 구조, 데이터 블록(Block) 크기 정의, 클러스터드/넌클러스터드 인덱스 세팅, 레코드 압축 암호화 기법

### 4. 맥락과 비교

ANSI/SPARC 3단계 구조의 각 레벨은 정보시스템 개발 생명주기 및 데이터 모델링 3단계와 수직적으로 1:1 매핑됩니다.

| ANSI/SPARC 레벨 | 매핑되는 스키마 | 주 대상자 | 데이터 모델링 단계 | 핵심 관점 |
|-----------------|---------------|----------|------------------|---------|
| **외부 단계** (External Level) | **외부 스키마** (External Schema) | 최종 사용자, 응용 프로그래머 | 외부 설계 (사용자 뷰) | 전사 데이터 중 개별 사용자에게 필요한 부분적 가상 화면 (View) |
| **개념 단계** (Conceptual Level) | **개념 스키마** (Conceptual Schema) | 데이터 관리자(DA), 데이터 아키텍트 | 개념 모델링 (전사 뷰) | 전사 관점의 통합 데이터 구조. 엔티티, 관계, 제약조건, 접근 권한 통합 기술 |
| **내부 단계** (Internal Level) | **내부 스키마** (Internal Schema) | 데이터베이스 관리자(DBA), 시스템 엔지니어 | 물리 모델링 (저장 뷰) | 실제 물리 저장 장치 관점의 구조. 인덱스 배치, 데이터 압축 방식, 저장 레코드 레이아웃 |

사상(Mapping)을 통해 달성되는 2대 데이터 독립성은 다음과 같습니다.

| 독립성 유형 | 메커니즘 | 실무 활용 사례 |
|------------|---------|--------------|
| **논리적 데이터 독립성** | 개념 스키마의 구조(테이블 컬럼 추가, 분할 등)가 바뀌더라도 **외부-개념 사상(논리적 사상)**만 변경해 주면 사용자의 외부 스키마와 응용 프로그램은 수정 불필요 | `회원` 테이블을 `회원기본`과 `회원인증`으로 분해하더라도 기존 뷰의 SQL 문 구조는 수정 비용 제로 |
| **물리적 데이터 독립성** | 내부 스키마 구조(인덱스 신설, 스토리지 교체, 파티셔닝 등)가 바뀌더라도 **개념 스키마나 응용 프로그램 코드에는 일절 영향 없음** | 대용량 매출 테이블에 파티셔닝을 적용하거나 보조 인덱스를 추가하더라도 사용자의 원시 SELECT 문은 단 한 줄도 바꿀 필요 없음 |

### 5. 실무 제언

**사상(Mapping) 오버헤드와 성능의 Trade-off**

- **챌린지**: 이론적으로 3단계 구조는 독립성을 완벽히 보장하지만, 사용자의 SQL 질의가 들어올 때마다 외부 → 개념 → 내부로 이어지는 사상(변환 계산 프로세스)을 거쳐야 하므로 하드웨어적 연산 오버헤드(지연 시간)가 발생합니다.
- **제언**: DBA는 데이터 독립성을 훼손하지 않는 선에서 자주 실행되는 사상 경로와 실행 계획(Execution Plan)이 DBMS의 공유 메모리 공간인 라이브러리 캐시(Library Cache)에 청정 상태로 고정 유지(소프트 파싱 유도)되도록 관리 거버넌스를 가동해야 합니다.

**클라우드 분산 아키텍처로의 독립성 철학 확장**

- **챌린지**: 과거의 단일 RDBMS 솔루션 내부 커널에만 갇혀 있던 ANSI/SPARC의 3단계 독립성 철학이 현대 클라우드 네이티브 분산 아키텍처 및 대형 데이터 플랫폼 설계에서 핵심 디커플링 인프라 전술로 확장 진화하고 있습니다.
- **제언**: 최신 데이터 레이크하우스(Lakehouse) 아키텍처에서는 저장 공간(AWS S3 등)인 물리 내부 레이어와 컴퓨팅 엔진(Spark, Trino 등)인 논리 개념 레이어를 철저히 분리하여 아키텍처를 구성합니다. 이는 ANSI/SPARC 모델의 물리적 데이터 독립성을 클라우드 분산 시스템에 그대로 투영한 결과입니다. 데이터 카탈로그(Data Catalog) 및 데이터 사전의 품질을 상시 동기화하여 사상 정보와 스키마 명세의 불일치를 방지해야 합니다.
