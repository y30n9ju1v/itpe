## 해설

> **다수 트랜잭션의 충돌 없는 동시 실행을 보장하는 스케줄링 엔지니어링**  
> 락 기반(2PL), 타임스탬프 기반, 낙관적(OCC), MVCC 기법으로 직렬 가능성(Serializability) 달성

데이터베이스는 수천 개의 트랜잭션이 동시에 실행되는 환경에서도 데이터 무결성을 보장해야 합니다. 동시성 제어(Concurrency Control)는 트랜잭션의 직렬 가능성(Serializability, 병렬 실행 결과가 순차 실행 결과와 동일한 성질)을 확보하여 각 트랜잭션이 마치 혼자 실행되는 것과 동일한 결과를 보장하는 핵심 메커니즘입니다. 클라우드 환경의 초고속 트랜잭션 처리와 분산 데이터베이스에서 이 기법의 설계 수준이 시스템 안정성을 결정합니다.

### 1. 출제 배경 및 의도

동시성 제어는 DBMS 내부 엔진의 핵심이며, 시험에서는 락 기반/타임스탬프 기반/낙관적 방식이라는 세 가지 패러다임의 차이를 명확히 서술할 수 있는지를 검증합니다. MSA 환경의 분산 트랜잭션, NewSQL/NoSQL의 MVCC 채택 사례가 확산되면서 MVCC의 원리까지 함께 설명할 것을 요구하는 경향이 강해졌습니다. 동시성 확보와 성능(처리량) 간의 트레이드오프를 논리적으로 설명하는 능력이 고득점 핵심입니다.

### 2. 개념의 본질과 작동 메커니즘

동시성 제어란 다수의 트랜잭션이 동시에 데이터베이스에 접근할 때 발생하는 충돌(Conflict)을 방지하고, 실행 결과가 직렬 스케줄(Serial Schedule)과 동일함을 보장하는 기법의 총칭입니다.

- **Input:** 다수의 병행 트랜잭션 및 데이터 접근 요청
- **Mechanism:** 자원 접근 제어(락/타임스탬프/버전 관리) → 충돌 감지 및 대기/롤백 결정
- **Output:** 직렬 가능한(Serializable) 실행 스케줄 및 트랜잭션 격리 보장

```text
[동시성 제어 기법 분류]

 ┌─────────────────────────────────────────────────────┐
 │              동시성 제어 (Concurrency Control)       │
 ├─────────────────┬───────────────────┬───────────────┤
 │   락 기반 (Locking)  │ 타임스탬프 기반  │  낙관적 (OCC)  │
 │                 │  (Timestamp)      │               │
 │ • 공유 락(S-Lock) │ • 트랜잭션 TS 부여 │ • 판독 단계    │
 │ • 배타 락(X-Lock) │ • TS 순서로 직렬화 │ • 검증 단계    │
 │ • 2PL 프로토콜   │ • Rollback 위주   │ • 기록 단계    │
 └─────────────────┴───────────────────┴───────────────┘
           +
 ┌─────────────────────────────────────────────────────┐
 │              MVCC (Multi-Version Concurrency Control) │
 │  • 읽기 트랜잭션은 이전 버전 스냅샷 참조              │
 │  • 쓰기 트랜잭션은 새 버전 생성 → 락 없이 읽기 허용   │
 └─────────────────────────────────────────────────────┘
```

#### 가. 락 기반 동시성 제어 (Locking Protocol)

| 락 유형 | 허용 연산 | 호환성 | 특징 |
|---------|----------|--------|------|
| **공유 락 (S-Lock, Shared Lock)** | READ | S-S 호환, S-X 불호환 | 동시 읽기 허용, 쓰기 방지 |
| **배타 락 (X-Lock, Exclusive Lock)** | READ/WRITE | X-X, X-S 모두 불호환 | 단독 점유, 타 트랜잭션 대기 |
| **의도 락 (Intent Lock)** | 계층적 잠금 | IS/IX/SIX 구분 | 상위 노드 선행 잠금 |

**2단계 잠금 프로토콜 (2PL: Two-Phase Locking)**

```text
[2PL 단계 구조]

  획득 단계(Growing Phase)       해제 단계(Shrinking Phase)
  ──────────────────────────────────────────────────────
  Lock 획득만 가능                Lock 해제만 가능
  └── Lock Point(최대 잠금 시점)을 기준으로 단계 전환

  락 수
   ↑         ┌────────────┐
   │         │            │
   │    ────→│  Lock Point│←──── 해제 시작
   │   /     └────────────┘
   │  /                   \
   │ /                     \
   └─────────────────────────→ 시간
     Growing Phase          Shrinking Phase
```

- **보수적 2PL(Conservative 2PL):** 트랜잭션 시작 전 모든 락 선점, 데드락 없음 but 병행성 저하
- **엄격 2PL(Strict 2PL):** X-Lock을 COMMIT/ROLLBACK 시점까지 유지, 연쇄 롤백 방지
- **강성 2PL(Rigorous 2PL):** 모든 락을 COMMIT/ROLLBACK 시점까지 유지

#### 나. 타임스탬프 기반 동시성 제어 (Timestamp Ordering)

```text
[타임스탬프 규칙]

 데이터 항목 Q: W-TS(Q), R-TS(Q) 유지

 트랜잭션 Ti의 READ(Q) 요청:
  TS(Ti) < W-TS(Q) → 거부, Ti 롤백 (과거 데이터 읽기 시도)
  TS(Ti) ≥ W-TS(Q) → 허용, R-TS(Q) = max(R-TS(Q), TS(Ti))

 트랜잭션 Ti의 WRITE(Q) 요청:
  TS(Ti) < R-TS(Q) → 거부, Ti 롤백 (이미 읽어간 데이터 변경 시도)
  TS(Ti) < W-TS(Q) → Thomas Write Rule: 무시(skip) 가능
  그 외 → 허용, W-TS(Q) = TS(Ti)
```

#### 다. 낙관적 동시성 제어 (OCC: Optimistic Concurrency Control, 충돌이 드물 것이라 낙관하고 미리 잠그지 않는 방식)

충돌이 드물다는 가정 하에 락 없이 실행, 커밋 시점에 충돌 검증합니다.

| 단계 | 내용 | 검증 방법 |
|------|------|----------|
| **판독(Read)** | 지역 변수에 데이터 복사, 연산 수행 | - |
| **검증(Validation)** | 다른 트랜잭션과의 충돌 여부 확인 | 판독집합/기록집합 교차 검사 |
| **기록(Write)** | 검증 통과 시 DB에 반영, 실패 시 롤백 | - |

#### 라. MVCC (Multi-Version Concurrency Control)

```text
[MVCC 작동 원리]

 쓰기 트랜잭션 T2 (UPDATE name='홍길동')
   ↓
 [버전 2] name='홍길동', TS=200 ← 최신 버전
 [버전 1] name='홍길음', TS=100 ← 스냅샷

 읽기 트랜잭션 T1 (TS=150, T2 완료 전 시작)
   → 버전 1(TS=100) 참조 → '홍길음' 반환 (락 없이!)

 PostgreSQL/MySQL InnoDB: Undo Log로 이전 버전 관리
 Oracle: Undo Tablespace
```

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 락 기반 | 공유 락(S-Lock), 배타 락(X-Lock), 2단계 잠금(2PL), 데드락(Deadlock), Lock Point |
| 타임스탬프 기반 | W-TS, R-TS, Thomas Write Rule, 롤백(Rollback), 직렬 가능성 |
| 낙관적 | 판독-검증-기록 3단계, 충돌 빈도 가정, 롤백 재시작 |
| MVCC | 스냅샷 격리(Snapshot Isolation), 버전 체인, Undo Log, 읽기-쓰기 비충돌 |
| 공통 | 직렬 가능성(Serializability), 트랜잭션 격리(Isolation), 처리량(Throughput) |

### 4. 맥락과 비교

| 비교 항목 | 락 기반(2PL) | 타임스탬프 기반 | 낙관적(OCC) | MVCC |
|----------|-------------|--------------|------------|------|
| **충돌 해결** | 대기(Wait) | 롤백(Rollback) | 롤백 후 재시작 | 버전 분리 |
| **데드락** | 발생 가능 | 없음 | 없음 | 없음 |
| **읽기 성능** | 락 경합 발생 | 보통 | 높음 | 매우 높음 |
| **쓰기 충돌** | 대기로 처리 | 롤백 처리 | 커밋 시점 검증 | 새 버전 생성 |
| **적합 환경** | OLTP 일반 | 충돌 잦은 환경 | 충돌 드문 환경 | 읽기 많은 OLTP |
| **주요 DBMS** | MySQL, Oracle | 일부 분산 DB | 일부 NoSQL | PostgreSQL, InnoDB |

### 5. 실무 제언

**데드락(Deadlock) 발생과 탐지 전략**

- **챌린지:** 락 기반 동시성 제어에서 두 트랜잭션이 서로의 락을 대기하는 순환 대기(Circular Wait) 상황이 발생하면 시스템이 교착 상태에 빠집니다. 고부하 OLTP 환경에서 데드락 발생 빈도가 높아지면 전체 처리량이 급격히 저하됩니다.
- **제언:** 데드락 탐지(Wait-for Graph 순환 감지)와 예방(Lock 획득 순서 고정, Timeout 설정) 전략을 병행 적용하고, PostgreSQL/InnoDB에서 제공하는 데드락 자동 감지 및 희생자(Victim) 선택 기능을 활용합니다. 타임아웃 기반 롤백 정책과 재시도 로직을 애플리케이션 레이어에서 구현하는 것이 실무 표준입니다.

**MVCC 도입 시 Vacuum/Purge 관리**

- **챌린지:** MVCC는 읽기 성능을 극적으로 향상시키지만, 오래된 버전 데이터(Dead Tuple)가 축적되면 스토리지 낭비와 인덱스 비대화를 초래합니다.
- **제언:** PostgreSQL의 autovacuum, MySQL InnoDB의 purge thread 설정을 업무 특성에 맞게 튜닝하고, 장시간 실행 트랜잭션(Long-Running Transaction)이 오래된 스냅샷을 붙들지 않도록 애플리케이션 트랜잭션 범위를 최소화해야 합니다.
