## 해설

### 출제 배경 및 의도

데이터베이스는 수천 개의 트랜잭션이 동시에 실행되는 환경에서도 데이터 무결성을 보장해야 한다. 동시성 제어(Concurrency Control)는 트랜잭션의 직렬 가능성(Serializability)을 확보하여 각 트랜잭션이 마치 혼자 실행되는 것과 동일한 결과를 보장하는 핵심 메커니즘이다.

시험에서는 락 기반/타임스탬프 기반/낙관적 방식이라는 세 가지 패러다임의 차이를 명확히 서술할 수 있는지를 검증한다. MSA 환경의 분산 트랜잭션, NewSQL/NoSQL의 MVCC 채택 사례가 확산되면서 MVCC의 원리까지 함께 설명할 것을 요구하는 경향이 강해졌다. 동시성 확보와 성능(처리량) 간의 트레이드오프를 논리적으로 설명하는 능력이 고득점 핵심이다.

### 1. 동시성 제어, 개요

정 의  • 다수 트랜잭션이 동시에 DB에 접근할 때 충돌을 방지하고 직렬 스케줄(Serial Schedule)과 동일한 결과를 보장하는 기법의 총칭
       - 직렬 가능성(Serializability): 병렬 실행 결과가 순차 실행 결과와 동일한 성질

```
[동시성 제어 기법 분류 구성도]

 ┌──────────────────────────────────────────────────────────┐
 │                동시성 제어 (Concurrency Control)          │
 ├─────────────────┬──────────────────┬─────────────────────┤
 │  락 기반(Locking)│ 타임스탬프 기반  │  낙관적(OCC)        │
 │                 │  (Timestamp)     │                     │
 │ • S-Lock(공유)  │ • 트랜잭션 TS 부여│ • 판독(Read)        │
 │ • X-Lock(배타)  │ • TS 순서 직렬화 │ • 검증(Validation)  │
 │ • 2PL 프로토콜  │ • Rollback 위주  │ • 기록(Write)       │
 │ • 데드락 발생   │ • 데드락 없음    │ • 충돌 드문 환경    │
 └─────────────────┴──────────────────┴─────────────────────┘
                    +
 ┌──────────────────────────────────────────────────────────┐
 │         MVCC (Multi-Version Concurrency Control)          │
 │  • 읽기: 이전 버전 스냅샷 참조 (락 없이 읽기)            │
 │  • 쓰기: 새 버전 생성 (읽기-쓰기 충돌 없음)             │
 │  • PostgreSQL/InnoDB: Undo Log로 버전 관리               │
 └──────────────────────────────────────────────────────────┘
```

- 충돌 해결 방식(대기/롤백/버전 분리)과 데드락 여부가 기법 선택의 핵심 기준이다.

### 2. 4가지 동시성 제어 기법 상세

1) 락 기반 동시성 제어 (Locking Protocol)

| 구분 | 항목 | 설명 |
|------|------|------|
| 공유 락 (S-Lock) | 허용 연산 | READ, S-S 호환, S-X 불호환 |
| 배타 락 (X-Lock) | 허용 연산 | READ/WRITE, X-X, X-S 모두 불호환 |
| 2PL 성장 단계 | Lock 획득 | Lock Point까지 Lock 획득만 가능 |
| 2PL 축소 단계 | Lock 해제 | Lock Point 이후 Lock 해제만 가능 |
| 보수적 2PL | 특징 | 트랜잭션 시작 전 모든 락 선점, 데드락 없음, 병행성 저하 |
| 엄격 2PL (Strict) | 특징 | X-Lock을 COMMIT/ROLLBACK까지 유지, 연쇄 롤백 방지 |

```
[2PL 락 획득/해제 단계]

  락 수
   ↑         ┌────────────┐
   │    ────→│ Lock Point │←── 해제 시작
   │   /     └────────────┘ \
   │  /    Growing Phase     \ Shrinking Phase
   └─────────────────────────────→ 시간
```

- 2PL은 직렬 가능성을 보장하지만 데드락(Deadlock) 발생 위험이 존재한다.

2) 타임스탬프·낙관적·MVCC 기법

| 구분 | 항목 | 설명 |
|------|------|------|
| 타임스탬프 기반 | 원리 | 데이터 항목마다 R-TS, W-TS 유지, TS 순서 위반 시 롤백 |
| 타임스탬프 기반 | Thomas Write Rule | TS(Ti) < W-TS(Q) 시 WRITE 무시(skip)로 직렬성 유지 |
| 낙관적(OCC) | 3단계 | 판독(Read) → 검증(Validation) → 기록(Write) |
| 낙관적(OCC) | 검증 방법 | 커밋 시점에 판독집합/기록집합 교차 검사, 충돌 시 롤백 |
| MVCC | 읽기 동작 | 트랜잭션 시작 시점 스냅샷 참조, 락 없이 읽기 |
| MVCC | 쓰기 동작 | 새 버전 생성, Undo Log(PostgreSQL/InnoDB)로 이전 버전 관리 |

- MVCC는 읽기-쓰기 간 충돌을 버전 분리로 해소하여 읽기 성능을 극대화한다.

### 3. 4가지 기법 비교 및 트렌드

| 비교 항목 | 락 기반(2PL) | 타임스탬프 기반 | 낙관적(OCC) | MVCC |
|---------|------------|-------------|-----------|------|
| 충돌 해결 | 대기(Wait) | 롤백(Rollback) | 롤백 후 재시작 | 버전 분리 |
| 데드락 | 발생 가능 | 없음 | 없음 | 없음 |
| 읽기 성능 | 락 경합 발생 | 보통 | 높음 | 매우 높음 |
| 쓰기 충돌 | 대기 처리 | 롤백 처리 | 커밋 시점 검증 | 새 버전 생성 |
| 적합 환경 | OLTP 일반 | 충돌 잦은 환경 | 충돌 드문 환경 | 읽기 많은 OLTP |
| 주요 DBMS | MySQL, Oracle | 일부 분산 DB | 일부 NoSQL | PostgreSQL, InnoDB |

현대 NewSQL(CockroachDB, TiDB 등)과 분산 DB는 MVCC를 기본 채택하며, 분산 트랜잭션에서는 Percolator 프로토콜 등 MVCC 기반 2PC 변형이 표준화되고 있다.  "끝"

### 실무 제언

**데드락(Deadlock) 탐지 및 예방**
- **챌린지**: 락 기반 동시성 제어에서 두 트랜잭션이 서로의 락을 대기하는 순환 대기(Circular Wait) 시 교착 상태가 발생한다. 고부하 OLTP에서 데드락 빈도가 높아지면 전체 처리량이 급격히 저하된다.
- **제언**: 데드락 탐지(Wait-for Graph 순환 감지)와 예방(Lock 획득 순서 고정, Timeout 설정)을 병행 적용하고, 타임아웃 기반 롤백 정책과 재시도 로직을 애플리케이션 레이어에서 구현한다.

**MVCC Vacuum/Purge 관리**
- **챌린지**: MVCC는 읽기 성능을 극대화하지만 오래된 버전 데이터(Dead Tuple)가 축적되면 스토리지 낭비와 인덱스 비대화를 초래한다.
- **제언**: PostgreSQL autovacuum, MySQL InnoDB purge thread를 업무 특성에 맞게 튜닝하고, 장시간 실행 트랜잭션(Long-Running Transaction)이 오래된 스냅샷을 붙들지 않도록 트랜잭션 범위를 최소화한다.

**분산 환경의 동시성 제어 확장**
- **챌린지**: 단일 노드 2PL/MVCC로는 MSA 분산 환경에서 전사적 직렬 가능성을 보장할 수 없다.
- **제언**: 분산 트랜잭션에서는 Google Spanner의 TrueTime 기반 외부 일관성(External Consistency)이나 Saga 패턴을 활용하여 전사적 데이터 무결성을 달성한다.
