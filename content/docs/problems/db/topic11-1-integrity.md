# 데이터 품질의 절대적 보루, 데이터 무결성(Data Integrity) 완전 해체

---

## 1. 문제를 해체하는 5대 핵심 분석 항목

### ① 출제 배경 및 의도 (Why Now?)

* **데이터 경제 시대의 원유(Crude Oil) 수호:** 인공지능(AI), 거대언어모델(LLM), 데이터 레이크 및 실시간 분석 인프라가 고도화될수록 인풋(Input)이 되는 데이터의 신뢰성이 시스템 전체의 성패를 가릅니다. 틀린 데이터를 넣으면 틀린 결과가 나오는 'Garbage In, Garbage Out(GIGO)' 현상을 막기 위한 최전방 방어선이 바로 데이터 무결성입니다.
* **복잡한 분정형 아키텍처에서의 정합성 붕괴:** 전통적인 단일 RDBMS 환경을 넘어 클라우드 네이티브, 마이크로서비스 아키텍처(MSA), 그리고 NoSQL 인프라가 융합되면서 데이터가 수많은 분산 노드로 파편화되고 있습니다. 이로 인해 데이터 유실, 오염, 불일치 리스크가 급증함에 따라 데이터 아키텍트가 갖추어야 할 가장 기본적이면서도 고도화된 '무결성 통제 전략'을 검증하고자 출제됩니다.

### ② 개념의 본질과 작동 메커니즘 (What & How)

* **한 문장 정의:** 데이터베이스 내에 저장된 데이터의 정확성(Accuracy), 완전성(Completeness), 유효성(Validity) 및 일관성(Consistency)을 유지하기 위해 데이터의 삽입, 삭제, 수정 연산 시 비즈니스 규칙과 공학적 제약조건을 강제하는 데이터베이스의 핵심 성질입니다.
* **작동 메커니즘 (Mechanism):**
데이터베이스 엔진(RDBMS) 내부 또는 애플리케이션 서비스 레이어에서 트랜잭션이 수행될 때, 설정된 도메인 규칙, 기본키 배타성, 외래키 참조 규칙 등의 제약조건 인터셉터가 구동됩니다(Input). 연산 과정에서 이 규칙을 위반하는 시도가 포착되면 즉시 예외(Exception)를 발생시키고 트랜잭션을 전면 철회(Rollback)함으로써(Mechanism), 오염되지 않은 청정 상태의 데이터 상태를 상시 유지(Output)합니다.

### ③ 채점관이 눈을 켜고 찾는 'Must-Have' 키워드 (Keywords)

* **4대 무결성 유형:** 개체 무결성(Entity), 참조 무결성(Referential), 도메인 무결성(Domain), 사용자 정의 무결성(User-Defined).
* **물리적 제약조건 기호:** Primary Key(PK), Foreign Key(FK), Unique, Not Null, Check Constraint.
* **참조무결성 트리거 전략:** Cascade, Set Null, Set Default, Restrict/No Action.
* **통제 아키텍처:** 데이터베이스 레이어(DBMS), 애플리케이션 레이어(ORM/Validation), 분산 트랜잭션(2PC, Saga 패턴).
* **비교 개념:** 데이터 정합성(Consistency).

### ④ 상위/하위/유사 개념과의 상호 관계 (Context & Comparison)

실무와 기술사 답안에서 가장 많이 혼용되는 '데이터 무결성'과 '데이터 정합성'은 개념적으로 명확히 분별되며 보완적인 관계를 가집니다.

| 비교 항목 | 데이터 무결성 (Data Integrity) | 데이터 정합성 (Data Consistency) |
| --- | --- | --- |
| **개념적 정의** | 데이터가 현실 세계의 비즈니스 규칙에 부합하여 **정확하고 완전하며 오염되지 않은 상태** | 논리적·물리적 위치나 시점에 상관없이 **데이터들의 값이 서로 모순 없이 일치하는 상태** |
| **핵심 포커스** | 데이터의 **'올바름(Rightness)'**과 품질 자격 | 데이터 간의 **'모순 없음(No Contradiction)'**과 일치성 |
| **상호 인과 관계** | **정합성은 무결성을 위한 하위 조건**임 (무결성이 충족되면 정합성도 달성됨) | **틀린 데이터도 일치하면 정합성은 유지**될 수 있음 (정합성이 무결성을 완벽히 보장하진 않음) |
| **실무적 위반 사례** | 고객 테이블의 나이 컬럼에 문자열 `A`가 저장된 경우 (품질 파탄) | A 서비스 DB의 홍길동 주소와 B 서비스 DB의 홍길동 주소가 서로 다른 경우 (불일치) |

## 2. 합격자 수준의 모범 답안 가이드 (2교시형 서술형)

### [1단락] 정보시스템 신뢰성의 주춧돌, 데이터 무결성(Data Integrity)의 개요

#### 가. 데이터 무결성의 정의

* 데이터베이스에 적재된 데이터가 현실 세계의 비즈니스 규칙, 도메인 제약 및 구조적 선후 관계를 완벽히 투영하여, 수명주기 전반에 걸쳐 정확성, 완전성, 일관성을 훼손당하지 않고 유지하는 품질적 특성입니다.

#### 나. 데이터 무결성 통제 메커니즘의 아키텍처 개념도

* 위 매핑 맵처럼 트랜잭션이 실물 스토리지 엔진에 반영되기 전 단계에서 DBMS의 커널 제약조건 엔진이 다각적으로 무결성을 필터링합니다.

---

### [2단락] 데이터베이스를 지탱하는 4대 데이터 무결성 유형 및 제약조건 기법

관계형 데이터 모델(RDBMS) 표준 사양에서 정의하는 4대 무결성의 상세 개념과 물리적 구현 매커니즘입니다.

#### 1. 개체 무결성 (Entity Integrity)

* **상세 개념:** 하나의 테이블(릴레이션) 내에 존재하는 행(Tuple)들은 서로 구별할 수 있는 유일한 결정자를 가져야 한다는 규칙입니다.
* **물리적 제약 기법:** **기본키(Primary Key, PK)** 설정을 통해 구현됩니다.
* **핵심 제약 조건:** 기본키로 지정된 속성은 테이블 내에서 절대 중복될 수 없으며, NULL 값(빈 값)의 유입을 강력히 거부(Not Null)해야 합니다.

#### 2. 참조 무결성 (Referential Integrity)

* **상세 개념:** 외래키 관계를 맺고 있는 두 테이블 간의 참조 관계가 상시 유효하게 유지되어야 한다는 규칙입니다. 자식 엔티티는 존재하지 않는 부모 엔티티를 가리킬 수 없습니다.
* **물리적 제약 기법:** **외래키(Foreign Key, FK)** 설정을 통해 구현됩니다.
* **핵심 제약 조건:** 자식 테이블의 외래키 컬럼에 유입되는 값은 반드시 부모 테이블의 기본키(또는 후보키) 영역에 실제로 존재하는 값이거나, 혹은 완벽히 비어있는 상태(Null)여야 합니다.

#### 3. 도메인 무결성 (Domain Integrity)

* **상세 개념:** 테이블의 특정 컬럼에 유입되는 모든 데이터 값은 사전에 정의된 데이터 타입, 길이, 허용 범위 등의 규칙을 철저히 충족해야 한다는 성질입니다.
* **물리적 제약 기법:** **Data Type, Not Null, Check Constraint, Default** 등을 활용합니다.
* **실무 예시:** 성별 컬럼의 데이터 타입을 `CHAR(1)`로 한정하고, `CHECK (성별 IN ('M', 'F'))` 제약 조건을 추가하여 엉뚱한 문자나 데이터가 침투하는 것을 차단합니다.

#### 4. 사용자 정의 무결성 (User-Defined Integrity)

* **상세 개념:** 데이터베이스 표준 제약조건(PK/FK)만으로는 투영할 수 없는, 기업 고유의 복잡한 업무 규칙(Business Rule)이나 특수한 도메인 제약을 완벽하게 구현하기 위한 무결성입니다.
* **물리적 제약 기법:** 데이터베이스 레벨의 **트리거(Trigger)**, **저장 프로시저(Stored Procedure)** 또는 애플리케이션 소스코드 비즈니스 로직 단에서 인터셉터를 강제하여 통제합니다.
* **실무 예시:** "당사 온라인 쇼핑몰 회원의 일일 총 결제 금액 한도는 등급별로 최대 5,000만 원을 초과할 수 없다"와 같은 금융성 비즈니스 규칙을 강제합니다.

### [3단락] 부모 데이터 변경 시 참조 무결성(Referential Integrity)을 수호하는 4대 트리거 전략

부모 테이블의 데이터가 삭제(`DELETE`)되거나 기본키가 수정(`UPDATE`)될 때, 자식 테이블의 외래키(FK)를 연동 제어하는 4대 공학적 전술입니다.

| 제어 전략 | 영문명 | 부모 데이터 삭제/변경 시 자식 테이블의 대응 매커니즘 | 실무 활용 가이드 |
| --- | --- | --- | --- |
| **연쇄 삭제** | **CASCADE** | 부모 테이블의 행이 삭제되면, 그 부모를 참조하고 있던 **자식 테이블의 연관 레코드들도 물리적으로 동시 연쇄 삭제** 처리 | 주문(부모) - 주문상세(자식)처럼 강력한 라이프사이클 종속 관계에 적용 |
| **널 전환** | **SET NULL** | 부모 테이블의 행이 삭제될 때, 자식 테이블의 외래키 **컬럼 값을 NULL(빈 값)로 강제 업데이트**하여 연결고리만 차단 | 부서(부모) - 사원(자식) 구조에서 부서가 사라져도 사원 학적은 보존할 때 활용 |
| **기본값 전환** | **SET DEFAULT** | 부모 테이블의 행이 삭제될 때, 자식 테이블의 외래키 **컬럼 값을 사전에 지정한 기본 디폴트 값으로 변환** | 참조 대상이 사라지면 '미지정' 또는 '시스템 기본그룹' ID로 이관할 때 유용 |
| **제한 / 거부** | RESTRICT /<br>

<br>NO ACTION | 자식 테이블에서 자신을 참조하는 레코드가 단 1건이라도 존재하면, **부모 데이터의 삭제/수정 연산 자체를 에러로 거부** | 기준 정보(예: 상품코드)가 거래 내역에 쓰이고 있을 때 원천 삭제를 차단하는 기본값 |

### [4단락] 기술사의 시각: 다층 구조(Multi-tiered) 및 분산 아키텍처 환경에서의 무결성 통제 전략

#### 가. 현대적 분산 및 MSA 아키텍처 환경에서의 무결성 훼손 위험성

* 단일 모놀리식 RDBMS 환경에서는 본 답안의 데이터베이스 제약조건(PK, FK, RI 트리거) 설정만으로 완벽한 무결성을 저비용으로 통제할 수 있었습니다.
* 그러나 마이크로서비스 아키텍처(MSA) 환경이나 클라우드 분산 데이터베이스 환경(NoSQL 등)에서는 데이터베이스가 서비스 경계별로 파편화(Database-per-Service)되며 하드웨어적 외래키(FK) 제약 조건을 맺을 수 없는 **참조 단절 현상**이 일어납니다. 이 상태에서 고전적 접근만 고집하면 시스템 전반의 무결성이 파탄에 직면하게 됩니다.

#### 나. 기술사적 실무 마스터 가이드라인 (다층 무결성 설계 아키텍처)

* 기술사는 데이터 무결성을 데이터베이스만의 책임으로 전가하지 말고, 시스템 레이어 전반에 걸쳐 3단계 방어벽을 구축하는 '다층 무결성 아키텍처(Multi-tiered Integrity Architecture)'를 주도해야 합니다.

1. **애플리케이션 검증 레이어 (1차 방어선):** Spring Boot의 `@Valid`, 자바 Bean Validation 기술 및 ORM(JPA) 엔티티 검증을 통해 도메인 무결성을 프론트엔드/백엔드 게이트웨이 단에서 1차로 걸러내어 불필요한 DB 부하를 저지합니다.
2. **분산 트랜잭션 거버넌스 레이어 (2차 방어선):** MSA 환경에서 이종 DB 간의 참조 무결성과 정합성을 유지하기 위해 하드웨어적 Lock을 거는 2PC(Two-Phase Commit) 방식 대신, 이벤트를 발행하여 보상 트랜잭션으로 최종 일관성을 달성하는 Saga 패턴(Orchestration / Choreography)을 설계 단계에 정착시켜야 합니다.
3. **데이터 사후 데이터 프로파일링 레이어 (3차 방어선):** 주기적인 데이터 품질 진단 툴을 구동하여 시스템 간 불일치 데이터나 유령 레코드를 탐색(Data Reconciliation Batch)하고 자동 보정하는 정기 배치 거버넌스를 결합해야 합니다.

* **결론:** 데이터 무결성은 양보할 수 없는 시스템의 신뢰도 지표입니다. 기술사는 논리 모델링 단계에서 암스트롱 공리와 식별 관계를 명확히 정교화하되, 물리 구현 단계에서는 하드웨어 인프라의 성능 한계와 분산 환경의 성격을 고려하여 "RDBMS 내에서의 제약조건 최적화 전략"과 "애플리케이션 레이어에서의 최종 일관성 제어"를 조화롭게 지휘하는 영민한 아키텍처 리더십을 발휘해야 합니다.
