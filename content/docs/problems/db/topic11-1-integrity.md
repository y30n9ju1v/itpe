## 해설

> **데이터베이스에 저장된 데이터의 정확성·완전성·일관성을 유지하는 핵심 성질**  
> 데이터 품질의 절대적 보루, 데이터 무결성(Data Integrity)

AI·LLM·데이터 레이크가 고도화될수록 입력 데이터의 신뢰성이 시스템 전체의 성패를 가릅니다. 'Garbage In, Garbage Out(GIGO, 잘못된 입력은 반드시 잘못된 출력을 낳는다)' 현상을 막기 위한 최전방 방어선이 데이터 무결성입니다. 클라우드 네이티브·MSA·NoSQL이 융합되면서 데이터 유실, 오염, 불일치 리스크가 급증함에 따라 데이터 아키텍트가 갖추어야 할 무결성 통제 전략이 더욱 중요해지고 있습니다.

### 1. 출제 배경 및 의도

인공지능(AI), 빅데이터 분석, 거대언어모델(LLM)이 비즈니스 핵심으로 자리 잡으면서 원천 데이터의 신뢰성이 시스템 전체의 성패를 좌우하고 있습니다. 클라우드 네이티브·MSA·NoSQL 인프라가 융합되면서 데이터가 수많은 분산 노드로 파편화되고 있습니다.

채점관은 데이터 아키텍트가 갖추어야 할 가장 기본적이면서도 고도화된 무결성 통제 전략을 종합적으로 검증하고자 본 문제를 출제합니다.

### 2. 개념의 본질과 작동 메커니즘

데이터베이스 내에 저장된 데이터의 정확성(Accuracy), 완전성(Completeness), 유효성(Validity) 및 일관성(Consistency)을 유지하기 위해 데이터의 삽입, 삭제, 수정 연산 시 비즈니스 규칙과 공학적 제약조건을 강제하는 데이터베이스의 핵심 성질입니다.

DBMS 엔진 내부 또는 애플리케이션 서비스 레이어에서 트랜잭션이 수행될 때, 설정된 도메인 규칙·기본키 배타성·외래키 참조 규칙 등의 제약조건 인터셉터가 구동됩니다. 연산 과정에서 이 규칙을 위반하는 시도가 포착되면 즉시 예외(Exception)를 발생시키고 트랜잭션을 전면 철회(Rollback)함으로써, 오염되지 않은 청정 상태의 데이터를 상시 유지합니다.

### 3. 핵심 키워드 (Must-Have)

#### 1) 개체 무결성 (Entity Integrity)

- **본질**: 하나의 테이블(릴레이션) 내에 존재하는 행(Tuple)들은 서로 구별할 수 있는 유일한 결정자를 가져야 한다는 규칙
- **물리적 제약 기법**: 기본키(Primary Key, PK) 설정
- **핵심 제약 조건**: 기본키로 지정된 속성은 절대 중복될 수 없으며, NULL 값의 유입을 강력히 거부(Not Null)해야 함

#### 2) 참조 무결성 (Referential Integrity)

- **본질**: 외래키 관계를 맺고 있는 두 테이블 간의 참조 관계가 상시 유효하게 유지되어야 한다는 규칙. 자식 엔티티는 존재하지 않는 부모 엔티티를 가리킬 수 없음
- **물리적 제약 기법**: 외래키(Foreign Key, FK, 다른 테이블의 기본키를 참조하는 컬럼) 설정
- **핵심 제약 조건**: 자식 테이블의 외래키 컬럼에 유입되는 값은 반드시 부모 테이블의 기본키(또는 후보키) 영역에 실제로 존재하는 값이거나 NULL이어야 함

#### 3) 도메인 무결성 (Domain Integrity)

- **본질**: 테이블의 특정 컬럼에 유입되는 모든 데이터 값은 사전에 정의된 데이터 타입, 길이, 허용 범위 등의 규칙을 철저히 충족해야 한다는 성질
- **물리적 제약 기법**: Data Type, Not Null, Check Constraint, Default 등을 활용
- **실무 예시**: 성별 컬럼을 `CHAR(1)`로 한정하고 `CHECK (성별 IN ('M', 'F'))` 제약 조건을 추가하여 엉뚱한 데이터 침투를 차단

#### 4) 사용자 정의 무결성 (User-Defined Integrity)

- **본질**: 표준 제약조건(PK/FK)만으로는 구현할 수 없는, 기업 고유의 복잡한 업무 규칙(Business Rule)이나 특수한 도메인 제약을 구현하기 위한 무결성
- **물리적 제약 기법**: 데이터베이스 레벨의 트리거(Trigger), 저장 프로시저(Stored Procedure) 또는 애플리케이션 비즈니스 로직으로 통제
- **실무 예시**: "온라인 쇼핑몰 회원의 일일 총 결제 금액 한도는 등급별로 최대 5,000만 원을 초과할 수 없다"는 금융성 비즈니스 규칙 강제

### 4. 맥락과 비교

실무와 기술사 답안에서 가장 많이 혼용되는 '데이터 무결성'과 '데이터 정합성'은 개념적으로 명확히 분별되며 보완적인 관계를 가집니다.

| 비교 항목 | 데이터 무결성 (Data Integrity) | 데이터 정합성 (Data Consistency) |
|-----------|------------------------------|--------------------------------|
| **개념적 정의** | 데이터가 현실 세계의 비즈니스 규칙에 부합하여 **정확하고 완전하며 오염되지 않은 상태** | 논리적·물리적 위치나 시점에 상관없이 **데이터들의 값이 서로 모순 없이 일치하는 상태** |
| **핵심 포커스** | 데이터의 **'올바름(Rightness)'**과 품질 자격 | 데이터 간의 **'모순 없음(No Contradiction)'**과 일치성 |
| **상호 인과 관계** | **정합성은 무결성을 위한 하위 조건** (무결성이 충족되면 정합성도 달성됨) | **틀린 데이터도 일치하면 정합성은 유지**될 수 있음 (정합성이 무결성을 완벽히 보장하진 않음) |
| **실무적 위반 사례** | 고객 테이블의 나이 컬럼에 문자열 `A`가 저장된 경우 (품질 파탄) | A 서비스 DB의 홍길동 주소와 B 서비스 DB의 홍길동 주소가 서로 다른 경우 (불일치) |

부모 데이터 삭제/수정 시 자식 테이블의 외래키(FK)를 연동 제어하는 4대 참조 무결성 트리거 전략도 중요합니다.

| 제어 전략 | 영문명 | 자식 테이블 대응 메커니즘 | 실무 활용 가이드 |
|-----------|--------|--------------------------|----------------|
| **연쇄 삭제** | CASCADE | 부모 행 삭제 시 자식 테이블의 연관 레코드도 연쇄 삭제 | 주문-주문상세처럼 강력한 라이프사이클 종속 관계에 적용 |
| **널 전환** | SET NULL | 부모 행 삭제 시 자식 테이블의 외래키 컬럼 값을 NULL로 업데이트 | 부서-사원 구조에서 부서가 사라져도 사원 정보는 보존할 때 활용 |
| **기본값 전환** | SET DEFAULT | 부모 행 삭제 시 자식 테이블의 외래키 컬럼 값을 사전 지정 기본값으로 변환 | 참조 대상이 사라지면 '미지정' ID로 이관할 때 유용 |
| **제한/거부** | RESTRICT / NO ACTION | 자식 테이블에 참조 레코드가 1건이라도 존재하면 부모 데이터의 삭제/수정 연산 자체를 에러로 거부 | 기준 정보(예: 상품코드)가 거래 내역에 쓰이고 있을 때 원천 삭제를 차단하는 기본값 |

### 5. 실무 제언

**다층 무결성 아키텍처(Multi-tiered Integrity Architecture)**

- **챌린지**: MSA 환경이나 클라우드 분산 데이터베이스 환경(NoSQL)에서는 데이터베이스가 서비스 경계별로 파편화(Database-per-Service)되며 외래키(FK) 제약 조건을 맺을 수 없는 참조 단절 현상이 발생합니다. 이 상태에서 고전적 접근만 고집하면 시스템 전반의 무결성이 파탄에 직면하게 됩니다.
- **제언**: 기술사는 데이터 무결성을 데이터베이스만의 책임으로 전가하지 말고, 시스템 레이어 전반에 걸쳐 3단계 방어벽을 구축하는 다층 무결성 아키텍처를 주도해야 합니다. 1차로 Spring Boot `@Valid`, Bean Validation, ORM 엔티티 검증을 통한 애플리케이션 레이어 방어, 2차로 MSA 이종 DB 간 정합성을 위한 Saga 패턴(보상 트랜잭션) 적용, 3차로 주기적인 Data Reconciliation Batch를 통한 사후 데이터 프로파일링 거버넌스를 결합해야 합니다.
