## 해설

### 출제 배경 및 의도

AI·LLM·데이터 레이크가 고도화될수록 입력 데이터의 신뢰성이 시스템 전체 성패를 좌우합니다. 'Garbage In, Garbage Out(GIGO)' 현상을 막기 위한 최전방 방어선이 데이터 무결성이며, 클라우드 네이티브·MSA·NoSQL이 융합되면서 데이터 유실·오염·불일치 리스크가 급증하고 있습니다.

채점관은 무결성의 4대 유형을 단순 암기하는 수준을 넘어, 각 유형의 물리적 구현 기법과 참조 무결성 4대 제어 전략을 구체적으로 설명하고, 분산 MSA 환경에서의 다층 무결성 아키텍처를 제시할 수 있는지 검증하고자 합니다. 데이터 무결성과 정합성의 명확한 개념 분리도 핵심 채점 포인트입니다.

### 1. 데이터 무결성, 개요

정 의  • 데이터베이스 내에 저장된 데이터의 정확성(Accuracy)·완전성(Completeness)·유효성(Validity)·일관성(Consistency)을 유지하기 위해 DML 연산 시 비즈니스 규칙과 공학적 제약조건을 강제하는 데이터베이스의 핵심 성질
       - DBMS 엔진 내부 또는 애플리케이션 레이어에서 제약조건 인터셉터가 위반 시 즉시 Exception 발생 → 트랜잭션 Rollback으로 청정 상태 유지

```
[데이터 무결성 4대 유형 및 구현 레이어]

  ┌─────────────────────────────────────────────┐
  │          데이터 무결성 (Data Integrity)       │
  ├───────────────┬───────────────┬─────────────┤
  │ 개체 무결성   │ 참조 무결성   │ 도메인 무결성│
  │ PK (Not Null │ FK (부모키    │ CHECK/Type/ │
  │  + Unique)   │  참조 유효성) │  Not Null)  │
  └───────────────┴───────────────┴─────────────┘
           │               │
           ▼               ▼
  사용자 정의 무결성 (Trigger / SP / Bean Validation)
  - 표준 제약으로 표현 불가한 복잡한 업무 규칙 강제
```

- 데이터 무결성은 GIGO 현상 방지를 위한 데이터 품질의 최전방 방어선이다.

### 2. 4대 무결성 유형 및 참조 무결성 제어 전략

1) 4대 무결성 유형

| 구분 | 유형 | 물리적 제약 기법 | 핵심 제약 내용 |
|------|------|----------------|--------------|
| **①** | **개체 무결성** (Entity) | Primary Key (PK) | 기본키는 중복 불가, NULL 불가 |
| **②** | **참조 무결성** (Referential) | Foreign Key (FK) | 외래키 값은 부모 기본키에 존재하거나 NULL이어야 함 |
| **③** | **도메인 무결성** (Domain) | Data Type, Not Null, Check, Default | 컬럼 값은 사전 정의된 타입·범위·유효값 범주를 충족해야 함 |
| **④** | **사용자 정의 무결성** (User-Defined) | Trigger, Stored Procedure, Bean Validation | 표준 제약으로 표현 불가한 복잡한 업무 규칙(일일 결제 한도 등) 강제 |

- 4대 유형은 상호 보완적으로 작동하여 데이터 품질의 다층 방어망을 구성한다.

2) 참조 무결성 4대 제어 전략

| 제어 전략 | 영문명 | 자식 테이블 대응 메커니즘 | 실무 활용 가이드 |
|-----------|--------|------------------------|----------------|
| **연쇄 삭제** | CASCADE | 부모 행 삭제 시 자식 연관 레코드도 연쇄 삭제 | 주문-주문상세 강한 종속 관계 |
| **널 전환** | SET NULL | 부모 행 삭제 시 자식 FK 컬럼 NULL 업데이트 | 부서 삭제 후 사원 정보 보존 |
| **기본값 전환** | SET DEFAULT | 부모 행 삭제 시 자식 FK를 사전 기본값으로 변환 | 참조 대상 소멸 시 '미지정' ID 이관 |
| **제한/거부** | RESTRICT / NO ACTION | 자식 참조 레코드 존재 시 부모 삭제/수정 거부 | 기준 정보(상품코드) 원천 삭제 차단 (기본값) |

- 참조 무결성 제어 전략은 비즈니스 수명주기 종속성에 따라 차별화하여 설계해야 한다.

### 3. 무결성 vs 정합성 비교 및 현대 환경 트렌드

| 비교 항목 | 데이터 무결성 (Data Integrity) | 데이터 정합성 (Data Consistency) |
|----------|------------------------------|--------------------------------|
| **개념적 정의** | 데이터가 비즈니스 규칙에 부합하여 정확하고 완전하며 오염되지 않은 상태 | 위치·시점에 상관없이 데이터 값이 서로 모순 없이 일치하는 상태 |
| **핵심 포커스** | 데이터의 '올바름(Rightness)'과 품질 자격 | 데이터 간 '모순 없음(No Contradiction)'과 일치성 |
| **인과 관계** | 정합성은 무결성의 하위 조건 (무결성 충족 → 정합성 달성) | 틀린 데이터도 일치하면 정합성 유지 가능 |
| **위반 사례** | 나이 컬럼에 문자열 'A' 저장 (품질 파탄) | A 서비스 DB vs B 서비스 DB의 동일 고객 주소 불일치 |

- 현대 MSA·분산 DB 환경에서는 물리적 FK 제약 대신 애플리케이션 검증 + Saga 패턴 + 정기 Reconciliation Batch의 다층 무결성 아키텍처로 진화하고 있다.  "끝"

### 실무 제언

**다층 무결성 아키텍처 구축**
- **챌린지**: MSA 환경에서 DB가 서비스 경계별로 파편화(Database-per-Service)되면 FK 제약 조건을 맺을 수 없는 참조 단절 현상이 발생하여 시스템 전반의 무결성이 파탄에 직면한다.
- **제언**: 1차 Spring Boot `@Valid`·Bean Validation 애플리케이션 레이어 방어, 2차 MSA 이종 DB 간 Saga 패턴(보상 트랜잭션) 적용, 3차 주기적 Data Reconciliation Batch 사후 거버넌스의 3단계 다층 무결성 아키텍처를 구축해야 한다.

**성능과 무결성의 트레이드오프 관리**
- **챌린지**: 대용량 OLTP 환경에서 모든 테이블에 FK 제약조건을 설정하면 대량 삽입·삭제 시 부모 테이블 Lock과 유효성 검증 오버헤드로 극심한 병목이 발생한다.
- **제언**: 초고속 로그 수집·이벤트 스트리밍 인프라에서는 물리적 FK를 생략하고 애플리케이션 검증 레이어를 강화하며, 사후 데이터 프로파일링 정기 배치로 품질을 보장하는 하이브리드 아키텍처를 채택해야 한다.

**사용자 정의 무결성의 트리거 과의존 방지**
- **챌린지**: 복잡한 업무 규칙을 모두 DB 트리거로 구현하면 디버깅 난이도가 극상으로 올라가고 성능 모니터링이 불가능해지며, 애플리케이션 레이어와의 이중 검증으로 관리 복잡도가 폭증한다.
- **제언**: 트리거는 단순·핵심 감사 로그(Audit Log) 목적으로 최소화하고, 복잡한 업무 규칙은 도메인 서비스 레이어의 Bean Validation과 비즈니스 로직으로 이관하여 테스트 가능성과 유지보수성을 확보해야 한다.
