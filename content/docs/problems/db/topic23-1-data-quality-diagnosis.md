## 해설

> **데이터의 신뢰성을 수치로 측정하고 체계적으로 개선하는 방법론**  
> 완전성·유일성·유효성·일관성·정확성·접근성 6대 품질 차원을 데이터 프로파일링과 비즈니스 규칙 검증으로 진단하고 개선하는 4단계 절차

AI·빅데이터 시대에 "Garbage In, Garbage Out"은 여전히 유효한 법칙입니다. AI 모델의 정확도와 비즈니스 분석의 신뢰성은 입력 데이터의 품질에 직접적으로 의존하므로, 데이터 품질진단은 데이터 기반 의사결정의 출발점입니다.

### 1. 출제 배경 및 의도

공공 데이터 품질 관리 지침(행정안전부), 데이터 품질 관리 성숙도 모델(DQM), 그리고 AI 학습 데이터 품질 인증 제도가 강화되면서 데이터 품질진단은 IT 거버넌스의 핵심 영역이 되었습니다. 채점관은 데이터 품질의 6대 차원을 정확히 알고, 진단 절차와 기법을 체계적으로 설명할 수 있는지를 검증합니다. AI/LLM 프로젝트가 증가하면서 학습 데이터 품질의 중요성이 더욱 부각되고 있습니다.

### 2. 개념의 본질과 작동 메커니즘

데이터 품질진단은 데이터의 현재 상태를 품질 기준에 따라 측정(진단)하고, 문제의 원인을 분석하여 개선 방안을 도출하는 체계적인 프로세스입니다.

```text
[데이터 품질진단 전체 흐름]

데이터 소스 (DB, 파일, API)
         │
         ▼
  1단계: 진단 준비
  ├── 진단 범위·대상 선정
  ├── 품질 기준 및 측정 지표 정의
  └── 진단 도구·환경 구성
         │
         ▼
  2단계: 데이터 진단
  ├── 데이터 프로파일링 (자동화)
  ├── 비즈니스 규칙 기반 검증
  └── 데이터 정합성 확인 (참조 무결성)
         │
         ▼
  3단계: 품질 평가
  ├── 품질 점수 산출 (차원별)
  ├── 오류 유형 분류 및 심각도 분석
  └── 근본 원인 분석 (Root Cause Analysis)
         │
         ▼
  4단계: 개선 계획 수립·실행
  ├── 단기 데이터 정제 (Data Cleansing)
  ├── 중기 프로세스·시스템 개선
  └── 장기 품질 관리 체계 수립 (DQ Governance)
```

#### 데이터 품질 6대 차원 (ISO 8000 기반)

| 차원 | 정의 | 진단 방법 | 오류 사례 |
|------|------|---------|---------|
| **완전성 (Completeness)** | 필수 속성에 값이 빠짐없이 존재하는 정도 | NULL/빈값 비율 측정 | 주민번호, 연락처 누락 |
| **유일성 (Uniqueness)** | 동일한 데이터가 중복 없이 한 번만 존재하는 정도 | 중복 키/레코드 탐지 | 고객 데이터 이중 등록 |
| **유효성 (Validity)** | 데이터가 정의된 도메인·형식·범위에 맞는 정도 | 도메인 규칙 검증, 패턴 매칭 | 날짜 형식 오류, 음수 나이 |
| **일관성 (Consistency)** | 동일 데이터가 여러 시스템/테이블에서 동일한 값을 가지는 정도 | 크로스 시스템 비교 | CRM-ERP 간 고객명 불일치 |
| **정확성 (Accuracy)** | 데이터가 실제 현실을 정확하게 반영하는 정도 | 외부 기준 데이터와 대조 | 배송 완료인데 "배송 중" 상태 |
| **접근성 (Accessibility)** | 필요한 사용자가 데이터에 적시에 접근할 수 있는 정도 | 접근 권한, 가용성 점검 | 리포팅 데이터 T+2일 지연 |

#### 데이터 프로파일링 (Data Profiling)

데이터 프로파일링은 데이터의 구조, 내용, 관계를 자동으로 분석하여 품질 현황을 파악하는 기법입니다.

```text
[데이터 프로파일링 분석 유형]

1. 컬럼 프로파일링 (Column Profiling)
   ├── 데이터 타입 (Type)
   ├── NULL 비율 (Null Rate)
   ├── 최솟값/최댓값/평균 (Min/Max/Avg)
   ├── 고유값 수 (Cardinality)
   ├── 분포 패턴 (빈도수 Top-N)
   └── 패턴 분포 (예: 전화번호 형식 다양성)

2. 테이블 프로파일링 (Table Profiling)
   ├── 전체 행 수 (Row Count)
   ├── 중복 행 비율
   └── 빈 테이블 여부

3. 관계 프로파일링 (Relationship Profiling)
   ├── 참조 무결성 위반 건수 (FK 위반)
   ├── 테이블 간 조인 가능성 분석
   └── 참조키 존재 비율 (Referential Integrity Rate)
```

#### 비즈니스 규칙 기반 검증

| 규칙 유형 | 설명 | 검증 SQL 예시 |
|---------|------|-------------|
| 형식 규칙 | 전화번호, 이메일, 날짜 등 형식 검증 | `WHERE phone NOT REGEXP '^[0-9]{2,3}-[0-9]{3,4}-[0-9]{4}$'` |
| 범위 규칙 | 값이 허용 범위 내인지 검증 | `WHERE age < 0 OR age > 150` |
| 참조 규칙 | FK 참조 무결성 검증 | `WHERE order_id NOT IN (SELECT id FROM orders)` |
| 논리 규칙 | 비즈니스 논리 일관성 검증 | `WHERE end_date < start_date` |
| 크로스 검증 | 시스템 간 동일 데이터 비교 | `CRM.customer vs ERP.customer` |

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 품질 6대 차원 | 완전성, 유일성, 유효성, 일관성, 정확성, 접근성 |
| 진단 기법 | 데이터 프로파일링, 비즈니스 규칙 검증, 크로스 시스템 비교, 참조 무결성 검증 |
| 진단 절차 | 준비 → 진단 → 평가 → 개선 (4단계) |
| 프로파일링 유형 | 컬럼 프로파일링, 테이블 프로파일링, 관계 프로파일링 |
| 품질 지표 | NULL 비율, 중복 비율, 참조 무결성 위반율, 형식 오류율 |
| 도구 | Informatica DQ, Talend DQ, AWS Deequ, Great Expectations |
| 개선 방법 | 데이터 정제(Cleansing), 표준화(Standardization), 보강(Enrichment) |

### 4. 맥락과 비교

| 비교 항목 | 데이터 품질진단 | 데이터 감사(Audit) | 데이터 마스터 관리(MDM) |
|----------|--------------|-----------------|----------------------|
| 목적 | 품질 현황 측정 및 개선 | 규정 준수·보안 점검 | 핵심 마스터 데이터 단일화 |
| 주기 | 정기(연/반기) + 상시 | 정기(연) | 상시 (실시간) |
| 대상 | 전체 데이터 | 접근·변경 이력 | 고객·제품·공급자 등 마스터 |
| 주요 기법 | 프로파일링, 규칙 검증 | 로그 분석, 접근 통제 검토 | 매칭·병합, 골든 레코드 생성 |
| 산출물 | 품질 진단 보고서 | 감사 보고서 | 마스터 데이터 허브 |

**품질 차원별 측정 지표 산출 공식**

```text
완전성 점수 = (전체 건수 - NULL/빈값 건수) / 전체 건수 × 100%

유일성 점수 = (전체 건수 - 중복 건수) / 전체 건수 × 100%

유효성 점수 = (전체 건수 - 규칙 위반 건수) / 전체 건수 × 100%

일관성 점수 = (전체 대조 건수 - 불일치 건수) / 전체 대조 건수 × 100%

종합 품질 점수 = Σ(차원별 점수 × 가중치) / Σ(가중치)
```

### 5. 실무 제언

**AI 학습 데이터 품질의 중요성**

- **챌린지:** LLM·ML 모델 구축 시 학습 데이터의 품질 문제(중복 데이터, 레이블 오류, 편향 데이터, 개인정보 포함)가 모델 정확도 저하와 편향된 출력으로 직결됩니다. 모델 학습 후 품질 문제를 발견하면 재학습 비용이 매우 높습니다.
- **제언:** 데이터 파이프라인에 자동화된 품질 검증 게이트(Quality Gate)를 내장해야 합니다. Great Expectations, AWS Deequ 같은 도구로 데이터 수집 단계부터 품질 규칙 검증을 자동화하고, 임계값 이하의 품질 점수가 나오면 파이프라인을 자동 차단하는 데이터 계약(Data Contract) 체계를 구축해야 합니다.

**데이터 품질 거버넌스 지속성 확보**

- **챌린지:** 데이터 품질진단은 일회성 이벤트로 끝나는 경우가 많아, 진단 후 개선이 이루어지더라도 시간이 지나면 품질이 다시 저하되는 "품질 저하 반복 사이클"이 발생합니다. 근본 원인인 데이터 입력 프로세스와 시스템의 구조적 문제를 해결하지 않으면 진단-개선의 무한 반복만 발생합니다.
- **제언:** 데이터 품질 관리는 3단계로 접근해야 합니다. 단기적으로는 진단된 오류 데이터를 직접 정제(Cleansing)합니다. 중기적으로는 데이터가 오염되는 입력 화면·ETL 프로세스·연계 인터페이스의 유효성 검증 로직을 강화합니다. 장기적으로는 데이터 품질 담당자(DQM, Data Quality Manager) 역할을 지정하고, 품질 KPI를 조직 성과지표에 연계하여 데이터 품질 거버넌스를 제도화해야 합니다.
