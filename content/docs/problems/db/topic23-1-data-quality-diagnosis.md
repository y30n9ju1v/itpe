## 해설

### 출제 배경 및 의도

AI·빅데이터 시대에 "Garbage In, Garbage Out" 원칙은 여전히 유효합니다. AI 모델의 정확도와 비즈니스 분석의 신뢰성은 입력 데이터 품질에 직접 의존하므로, 데이터 품질진단은 데이터 기반 의사결정의 출발점입니다.

공공 데이터 품질 관리 지침(행정안전부), 데이터 품질 관리 성숙도 모델(DQM), AI 학습 데이터 품질 인증 제도가 강화되면서 데이터 품질진단은 IT 거버넌스의 핵심 영역이 되었습니다. 채점관은 데이터 품질의 6대 차원을 정확히 알고, 진단 절차와 기법을 체계적으로 설명할 수 있는지를 검증합니다.

LLM 프로젝트 급증으로 학습 데이터 품질 중요성이 더욱 부각되며, 데이터 계약(Data Contract)·파이프라인 내 Quality Gate 구현이 현업 필수 역량으로 자리잡았습니다.

### 1. 데이터 품질진단, 개요

정 의  • 데이터의 현재 상태를 6대 품질 차원 기준으로 측정·분석하여
       - 오류 원인을 식별하고 개선 방안을 도출하는 4단계 체계적 프로세스

```
[데이터 품질진단 전체 흐름]

데이터 소스 (DB, 파일, API)
         │
         ▼
  1단계: 진단 준비          2단계: 데이터 진단
  ├── 범위·대상 선정         ├── 데이터 프로파일링 (자동화)
  ├── 품질 기준·지표 정의    ├── 비즈니스 규칙 기반 검증
  └── 진단 도구·환경 구성    └── 참조 무결성 확인
         │                          │
         ▼                          ▼
  3단계: 품질 평가          4단계: 개선 계획 수립·실행
  ├── 차원별 품질 점수 산출   ├── 단기: 데이터 정제(Cleansing)
  ├── 오류 유형·심각도 분류   ├── 중기: 프로세스·시스템 개선
  └── 근본 원인 분석(RCA)    └── 장기: DQ 거버넌스 체계 수립
```

- 준비 → 진단 → 평가 → 개선 4단계 사이클이 반복적으로 운영되어야 품질 거버넌스가 유지됨

### 2. 데이터 품질 6대 차원 및 진단 기법

1) 품질 6대 차원 (ISO 8000 기반)

| 차원 | 정의 | 진단 방법 | 오류 사례 |
|------|------|---------|---------|
| 완전성(Completeness) | 필수 속성에 값이 빠짐없이 존재하는 정도 | NULL/빈값 비율 측정 | 주민번호, 연락처 누락 |
| 유일성(Uniqueness) | 동일한 데이터가 중복 없이 한 번만 존재 | 중복 키/레코드 탐지 | 고객 데이터 이중 등록 |
| 유효성(Validity) | 정의된 도메인·형식·범위에 맞는 정도 | 도메인 규칙 검증, 패턴 매칭 | 날짜 형식 오류, 음수 나이 |
| 일관성(Consistency) | 여러 시스템/테이블에서 동일한 값을 가짐 | 크로스 시스템 비교 | CRM-ERP 간 고객명 불일치 |
| 정확성(Accuracy) | 데이터가 실제 현실을 정확하게 반영 | 외부 기준 데이터와 대조 | 배송 완료인데 "배송 중" 상태 |
| 접근성(Accessibility) | 필요한 사용자가 적시에 접근 가능 | 접근 권한, 가용성 점검 | 리포팅 데이터 T+2일 지연 |

- 6대 차원 각각에 대한 점수 산출 후 가중 평균으로 종합 품질 점수를 도출

2) 데이터 프로파일링 기법

| 유형 | 분석 항목 | 발견 이슈 |
|------|----------|---------|
| 컬럼 프로파일링 | NULL 비율, Min/Max/Avg, Cardinality, 패턴 분포 | 이상값, 과도한 NULL, 형식 혼재 |
| 테이블 프로파일링 | Row Count, 중복 행 비율 | 레코드 누락, 중복 저장 |
| 관계 프로파일링 | FK 위반 건수, 참조 무결성 비율 | 고아 레코드(Orphan Record) |

- 프로파일링은 진단 2단계의 핵심 자동화 기법으로, 대량 데이터에서 이상 징후를 신속 탐지

3) 비즈니스 규칙 기반 검증

| 규칙 유형 | 설명 | 검증 SQL 예시 |
|---------|------|-------------|
| 형식 규칙 | 전화번호, 날짜 등 형식 검증 | `WHERE phone NOT REGEXP '^[0-9]{2,3}-...'` |
| 범위 규칙 | 값이 허용 범위 내인지 검증 | `WHERE age < 0 OR age > 150` |
| 참조 규칙 | FK 참조 무결성 검증 | `WHERE order_id NOT IN (SELECT id FROM orders)` |
| 논리 규칙 | 비즈니스 논리 일관성 | `WHERE end_date < start_date` |
| 크로스 검증 | 시스템 간 동일 데이터 비교 | CRM.customer vs ERP.customer |

### 3. 품질 지표 산출 공식 및 비교

```
완전성 점수 = (전체 건수 - NULL/빈값 건수) / 전체 건수 × 100%
유일성 점수 = (전체 건수 - 중복 건수) / 전체 건수 × 100%
유효성 점수 = (전체 건수 - 규칙 위반 건수) / 전체 건수 × 100%
일관성 점수 = (전체 대조 건수 - 불일치 건수) / 전체 대조 건수 × 100%
종합 품질 점수 = Σ(차원별 점수 × 가중치) / Σ(가중치)
```

| 비교 항목 | 데이터 품질진단 | 데이터 감사(Audit) | 데이터 마스터 관리(MDM) |
|----------|--------------|-----------------|----------------------|
| 목적 | 품질 현황 측정 및 개선 | 규정 준수·보안 점검 | 핵심 마스터 데이터 단일화 |
| 주기 | 정기(연/반기) + 상시 | 정기(연) | 상시 (실시간) |
| 대상 | 전체 데이터 | 접근·변경 이력 | 고객·제품·공급자 등 마스터 |
| 주요 기법 | 프로파일링, 규칙 검증 | 로그 분석, 접근 통제 검토 | 매칭·병합, 골든 레코드 생성 |
| 산출물 | 품질 진단 보고서 | 감사 보고서 | 마스터 데이터 허브 |

- 데이터 품질진단은 일회성이 아닌 지속적 거버넌스 체계로 운영되어야 실질적 품질 개선이 가능함.  "끝"

### 실무 제언

**AI 학습 데이터 품질 게이트 내재화**
- **챌린지**: LLM·ML 모델 구축 시 중복 데이터·레이블 오류·편향·개인정보 포함이 모델 정확도 저하와 편향 출력으로 직결되며, 학습 후 품질 문제 발견 시 재학습 비용이 매우 높습니다.
- **제언**: Great Expectations, AWS Deequ 등으로 수집 단계부터 품질 규칙 검증을 자동화하고, 임계값 이하 점수 시 파이프라인을 자동 차단하는 데이터 계약(Data Contract) 체계를 구축해야 합니다.

**품질 거버넌스 지속성 확보**
- **챌린지**: 품질진단이 일회성 이벤트로 끝나면 개선 후에도 시간이 지나 품질이 재저하되는 반복 사이클이 발생하며, 입력 프로세스·시스템의 구조적 원인을 해결하지 않으면 무한 반복만 생깁니다.
- **제언**: 단기 정제(Cleansing) → 중기 ETL/입력 검증 강화 → 장기 DQM 역할 지정·KPI 연계의 3단계 접근으로 품질 거버넌스를 제도화해야 합니다.

**실시간 품질 모니터링 대시보드 운영**
- **챌린지**: 정기 진단 사이에 발생하는 품질 이슈가 장기간 방치되어 하위 시스템·분석 결과에 누적 오염을 유발합니다.
- **제언**: 6대 차원별 KPI를 실시간 대시보드로 모니터링하고, 임계값 초과 시 데이터 스튜어드에게 자동 알람이 전송되는 상시 감시 체계를 구축해야 합니다.
