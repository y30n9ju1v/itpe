## Part 1. 데이터 모델링 — 데이터의 설계도 그리기

### 1-1. 데이터 모델링의 3단계와 기본 원칙

집을 짓기 전에 건축 설계도를 그리듯, 데이터베이스를 만들기 전에 "어떤 데이터를, 어떤 구조로 저장할 것인가"를 설계하는 작업입니다.

**3단계 모델링**

| 단계 | 하는 일 | 산출물 | 비유 |
|------|--------|--------|------|
| **개념적 모델링** | 핵심 개념과 관계 도출. 기술에 독립적 | 개념 ERD | 건물의 용도와 방 구성 스케치 |
| **논리적 모델링** | 릴레이션, 속성, 키, 정규화 적용 | 논리 ERD, 테이블 정의 | 치수 포함 건축 설계도 |
| **물리적 모델링** | 특정 DBMS에 맞는 테이블, 인덱스, 파티션 | DDL 스크립트 | 실제 시공 도면 |

**3가지 기본 원칙**

- **소통(Communication):** 업무 담당자와 개발자가 함께 이해할 수 있어야 한다
- **단순화(Granularity):** 필요한 수준으로만 분해하라 (과하게 쪼개지도, 뭉뚱그리지도 말라)
- **논리적 표현(Logical Representation):** 실제 업무 규칙을 왜곡 없이 표현하라

---

### 1-2. ERD와 관계형 데이터 모델

**ERD (Entity Relationship Diagram)** 란 개체(Entity)와 관계(Relationship)를 그림으로 표현한 다이어그램입니다.

**IE/Crow's Foot 표기법 읽기**

```
고객 ||----o{ 주문      (고객 1명은 주문 0개 이상 — 1:N)
교수 }o--o{ 과목       (교수 여러명이 여러 과목 담당 — M:N)
```

**관계형 데이터 모델 핵심 개념**

| 용어 | 의미 | 비유 |
|------|------|------|
| **릴레이션(Relation)** | 테이블 — 행과 열로 이루어진 집합 | 엑셀 시트 한 장 |
| **튜플(Tuple)** | 테이블의 한 행(Row) | 엑셀의 한 줄 |
| **도메인(Domain)** | 속성이 가질 수 있는 값의 범위 | 성별 컬럼 = {남, 여} |
| **기본키(PK)** | 각 행을 유일하게 식별하는 값 | 주민등록번호 |
| **외래키(FK)** | 다른 테이블의 PK를 참조하는 키 | 주문서의 고객번호 |

---

### 1-3. 연결함정 — Fan Trap과 Chasm Trap

ERD를 잘못 설계하면 쿼리가 없는 데이터를 만들거나 있는 데이터를 잃게 됩니다.

**Fan Trap (무부착 함정) — 가짜 데이터가 생긴다**

```
[부서] -1:N-> [직원]
[부서] -1:N-> [프로젝트]

직원-프로젝트 JOIN 시 → 실제로 없는 조합이 폭발적으로 생성됨
```

**해결책:** 직원-프로젝트 배정 관계를 직접 모델링

**Chasm Trap (균열 함정) — 진짜 데이터를 잃는다**

```
[영업사원] -선택-> [고객] -필수-> [주문]

영업사원→주문 조회 시: 고객이 없는 주문은 경로 단절로 조회 안 됨
```

**해결책:** 영업사원-주문 직접 관계 추가 또는 OUTER JOIN 사용

---

### 1-4. 슈퍼타입 / 서브타입

한 개체가 여러 종류로 나뉠 때, 공통 속성은 슈퍼타입에, 고유 속성은 서브타입에 분리하는 설계 기법입니다.

```
[직원(슈퍼타입)]           공통: 이름, 사번, 입사일
  ├── [정규직(서브타입)]   고유: 연봉, 직급
  └── [계약직(서브타입)]   고유: 계약기간, 시급
```

**물리적 구현 3가지**

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **단일 테이블 (Roll-up)** | 모든 서브타입을 하나의 테이블로 합침 | 조인 없이 빠른 조회 | NULL 많음 |
| **타입별 테이블 (Roll-down)** | 슈퍼타입+서브타입 각각 테이블 분리 | 정규화됨 | 조인 필요 |
| **구체 클래스 테이블 (One-to-One)** | 서브타입마다 모든 컬럼 포함 | 조인 없음 | 공통 속성 중복 |

---

### 1-5. 식별자 관계 vs 비식별자 관계

**식별자 관계 (Identifying):** 부모 PK가 자식 PK의 일부 → ERD에서 **실선**

```
주문(주문번호 PK) ───── 주문상품(주문번호+상품번호 PK)
```

**비식별자 관계 (Non-Identifying):** 부모 PK가 자식의 일반 FK → ERD에서 **점선**

```
부서(부서번호 PK) - - - 직원(직원번호 PK, 부서번호 FK)
```

- 자식이 부모 없이 존재 불가 → 식별자 관계
- 자식이 독립적으로 존재 가능 → 비식별자 관계
- 실무에서는 유지보수 편의를 위해 비식별자 관계를 선호하는 경향

## Part 2. 정규화 — 데이터 중복을 제거하는 기술

### 2-1. 함수종속성과 암스트롱 공리

**함수종속성(FD):** 속성 X의 값이 결정되면 Y의 값도 반드시 하나로 결정 → `X → Y`

**함수종속성의 종류**

| 종류 | 설명 | 예시 |
|------|------|------|
| **완전 함수종속** | 기본키 전체에 종속 | {학번, 과목코드} → 성적 |
| **부분 함수종속** | 기본키 일부에만 종속 | {학번, 과목코드} → 학생명 (학번만으로도 결정) |
| **이행적 함수종속** | A→B, B→C일 때 A→C | 학번→학과, 학과→학과장 → 학번→학과장 |

**암스트롱 공리 3가지 (함수종속성 추론 규칙)**

| 공리 | 내용 |
|------|------|
| **반사율(Reflexivity)** | Y가 X의 부분집합이면 X→Y |
| **첨가율(Augmentation)** | X→Y이면 XZ→YZ |
| **이행율(Transitivity)** | X→Y, Y→Z이면 X→Z |

**속성 클로저(X⁺):** X에서 출발해 FD로 도달 가능한 모든 속성 집합. X⁺가 전체 속성을 포함하면 X는 후보키.

---

### 2-2. 이상현상(Anomaly)과 정규화

**이상현상이란?** 정규화가 안 된 테이블에서 발생하는 3가지 문제.

| 이상현상 | 설명 | 예시 |
|---------|------|------|
| **삽입 이상** | 원하는 데이터를 넣으려면 불필요한 데이터도 함께 넣어야 함 | 새 학과를 추가하려면 수강 학생이 있어야 함 |
| **삭제 이상** | 데이터를 지우면 의도치 않은 다른 데이터도 사라짐 | 마지막 수강생 삭제 시 학과 정보까지 삭제 |
| **수정 이상** | 하나의 사실이 여러 곳에 저장되어 일부만 수정하면 불일치 발생 | 학과장이 바뀔 때 여러 행을 모두 수정해야 함 |

**정규화:** 이상현상을 제거하기 위해 테이블을 적절히 분해하는 과정

---

### 2-3. 2NF, 3NF, BCNF — 실전 정규화

**1NF:** 모든 속성값이 원자값(Atomic)이어야 한다. 반복 그룹, 다중값 금지.

**2NF:** 1NF + **부분 함수종속 제거**

```
[위반] 수강({학번, 과목코드} → 성적, 학생명)
         학생명은 학번만으로 결정 (부분 함수종속!)
[해결] 학생(학번, 학생명) + 수강(학번, 과목코드, 성적)
```

**3NF:** 2NF + **이행적 함수종속 제거**

```
[위반] 학생(학번, 학생명, 학과코드, 학과장)
         학번→학과코드→학과장 (이행적 종속!)
[해결] 학생(학번, 학생명, 학과코드) + 학과(학과코드, 학과장)
```

**BCNF:** 3NF + **모든 결정자가 후보키**여야 한다

```
[위반] 수강(학생, 과목, 담당교수)
  담당교수 → 과목 (담당교수는 후보키 아닌데 결정자!)
[해결] 교수과목(담당교수, 과목) + 수강(학생, 담당교수)
```

**정규형 단계 요약**

```
1NF: 원자값
 ↓ 부분 FD 제거
2NF: 완전 함수종속
 ↓ 이행적 FD 제거
3NF: 비이행적 종속
 ↓ 결정자가 후보키가 아닌 경우 제거
BCNF: 모든 결정자 = 후보키
```

---

### 2-4. 4NF, 5NF — 고급 정규화

**4NF — 다치종속(MVD) 제거**

속성 A가 결정되면 B가 여러 값을 가지는 독립적 관계 = `A →→ B`

```
[위반] 과목(과목명, 담당교수, 교재)
  교수 목록과 교재 목록이 독립적 → 조합 폭발!
[해결] 과목교수(과목명, 담당교수) + 과목교재(과목명, 교재)
```

**5NF (PJ/NF) — 조인종속(JD) 제거**

3개 이상 다방향 관계에서 분해-재조인 시 원본이 복원되지 않는 경우 처리. 실무에서는 BCNF까지가 현실적 목표.

### 2-5. 역정규화(반정규화) — 성능을 위한 타협

**원칙: 선정규화 후조율** — 반드시 정규화 먼저, 성능 문제 발생 후 역정규화

| 기법 | 내용 | 예시 |
|------|------|------|
| **테이블 합병** | 자주 함께 조회되는 테이블 합치기 | 주문 + 주문상품 합병 |
| **컬럼 중복** | 조인 없이 조회하려고 FK 테이블 속성 복사 | 주문 테이블에 고객명 복사 |
| **파생 컬럼** | 계산값을 미리 저장 | 주문 테이블에 총합계 저장 |

**부작용:** 수정 이상 재발 가능 → 수정이 거의 없는 이력 데이터나 DW에 적합

## Part 3. SQL과 관계대수 — 데이터를 다루는 언어

### 3-1. 관계대수 — SQL의 이론적 기반

> 관계대수는 **절차적(HOW)** — "어떻게 가져올지" 명시  
> 관계해석은 **비절차적(WHAT)** — "무엇을 가져올지"만 명시 (SQL이 이에 해당)

**순수 관계 연산자**

| 연산자 | 기호 | SQL 대응 |
|--------|------|---------|
| **선택(Selection)** | σ | WHERE 절 |
| **프로젝션(Projection)** | π | SELECT 절 |
| **조인(Join)** | ⋈ | JOIN |
| **나누기(Division)** | ÷ | "모든 ~에 대해" 서브쿼리 |

**집합 연산자:** 합집합(∪), 교집합(∩), 차집합(-), 카티션 프로덕트(×)

---

### 3-2. JOIN의 종류와 활용

| JOIN 종류 | 설명 | 예시 결과 |
|-----------|------|---------|
| **INNER JOIN** | 두 테이블에 모두 있는 데이터만 반환 | 주문한 고객만 조회 |
| **LEFT OUTER JOIN** | 왼쪽 전체 + 오른쪽 일치 데이터 | 주문 없는 고객도 포함 |
| **FULL OUTER JOIN** | 양쪽 전체 + 일치 데이터 | 모든 고객과 모든 주문 |
| **CROSS JOIN** | 모든 행의 조합 (카티션 프로덕트) | 3명 × 3개 = 9행 |
| **SELF JOIN** | 같은 테이블끼리 JOIN | 직원-상사 관계 |
| **EQUI JOIN** | = 조건으로 JOIN | 가장 일반적 |
| **NON-EQUI JOIN** | =이 아닌 조건으로 JOIN | 급여 범위로 등급 매칭 |

---

### 3-3. JOIN 물리적 알고리즘 — NL, Sort Merge, Hash Join

**Nested Loop Join (NL Join)**

```
FOR each row in 외부테이블:
    FOR each row in 내부테이블:
        IF 조인 조건 만족: 결과에 추가
```

**Sort Merge Join:** 양쪽 테이블을 정렬 후 순차 병합. 범위 조건에 강점.

**Hash Join:** 작은 테이블로 해시 테이블 빌드 → 큰 테이블 스캔하며 매칭.

**종합 비교**

| 비교 | NL Join | Sort Merge | Hash Join |
|------|---------|-----------|-----------|
| 데이터 크기 | 소량 | 중~대 | 대용량 |
| 인덱스 의존 | 강함 | 없어도 가능 | 없어도 가능 |
| 조인 조건 | 모든 조건 | 범위 포함 | 등치(=)만 |
| 주 용도 | OLTP | 정렬 데이터 | OLAP |

---

### 3-4. 동적 SQL — 런타임에 SQL 만들기

**하드 파싱 vs 소프트 파싱**

- **하드 파싱:** SQL 첫 실행 시 파싱 → 실행 계획 생성 → 캐시 저장 (비용 큼)
- **소프트 파싱:** 캐시에 저장된 실행 계획 재사용 (빠름)

동적 SQL에서 리터럴 값 직접 삽입 → 값이 바뀔 때마다 새 SQL → 하드 파싱 반복 → 성능 저하

**해결책: 바인드 변수(Bind Variable)**

```java
// 잘못된 방법 (하드 파싱 반복)
String sql = "SELECT * FROM 고객 WHERE id = " + userId;

// 올바른 방법 (소프트 파싱 + SQL Injection 방어)
PreparedStatement stmt = conn.prepareStatement(
    "SELECT * FROM 고객 WHERE id = ?"
);
stmt.setInt(1, userId);
```

## Part 4. 무결성과 트랜잭션 — 데이터를 신뢰할 수 있게

### 4-1. 데이터 무결성의 종류와 유지 방법

**데이터 무결성 4가지**

| 무결성 종류 | 내용 | 구현 방법 |
|-----------|------|---------|
| **엔티티 무결성** | PK는 NULL 불가, 유일해야 함 | PRIMARY KEY 제약 |
| **참조 무결성** | FK는 참조하는 PK가 반드시 존재해야 함 | FOREIGN KEY 제약 |
| **도메인 무결성** | 속성값은 정의된 도메인 범위 내에 있어야 함 | CHECK 제약, NOT NULL |
| **사용자 정의 무결성** | 비즈니스 규칙에 맞는 데이터만 허용 | 트리거, 저장 프로시저 |

**참조 무결성 위반 시 처리 옵션 (ON DELETE)**

| 옵션 | 동작 |
|------|------|
| **CASCADE** | 부모 삭제 시 자식도 함께 삭제 |
| **SET NULL** | 부모 삭제 시 자식의 FK를 NULL로 변경 |
| **RESTRICT / NO ACTION** | 자식이 있으면 부모 삭제 거부 |

**릴레이션 무결성 규칙**

| 규칙 | 설명 |
|------|------|
| **상태 제약** | 특정 시점의 상태 조건 (재고 수량 ≥ 0) |
| **전이 제약** | 상태 변화 시 조건 (취소됨 → 진행중 불가) |
| **튜플 제약** | 행 내 속성 간 조건 (종료일 ≥ 시작일) |
| **집합 제약** | 여러 행 간의 조건 (한 부서 직원 수 ≤ 100) |

---

### 4-2. 트랜잭션 ACID — 믿을 수 있는 작업 단위

**ACID 특성**

| 특성 | 의미 | 보장 메커니즘 |
|------|------|------------|
| **원자성(Atomicity)** | 모두 성공 or 모두 실패 | Undo Log, Rollback |
| **일관성(Consistency)** | 트랜잭션 전후 DB는 항상 일관된 상태 | 제약조건 검사 |
| **격리성(Isolation)** | 동시 실행 트랜잭션이 서로 독립적 | Lock, MVCC |
| **지속성(Durability)** | COMMIT된 데이터는 장애 후에도 보존 | Redo Log, WAL |

**트랜잭션 상태 전이**

```
시작 → [활성(Active)] → [부분완료] → [완료(Committed)]
                ↘ 실패 → [실패(Failed)] → [철회(Aborted)]
```

**WAL (Write-Ahead Logging) 원칙**

"데이터 파일 변경 전에 반드시 로그를 먼저 기록하라"

```
쓰기 발생 → 1. Redo Log 디스크 기록 → 2. Buffer 변경 → 3. COMMIT 응답
(나중에) → 4. Buffer → Disk (Checkpoint)
```

**TCL 명령어**

| 명령어 | 기능 |
|--------|------|
| `COMMIT` | 트랜잭션 확정, 영구 저장 |
| `ROLLBACK` | 트랜잭션 시작 이후 모든 변경 취소 |
| `SAVEPOINT name` | 중간 저장점 설정 |
| `ROLLBACK TO name` | 저장점까지만 롤백 |

---

### 4-3. ANSI/SPARC 3단계 아키텍처

데이터베이스를 3개의 추상화 계층으로 분리하여 **데이터 독립성**을 확보하는 구조.

```
사용자/애플리케이션
    ↓
[외부 스키마 (External)]  ← 사용자 뷰, 부서별 뷰
    ↓ 논리적 데이터 독립성
[개념 스키마 (Conceptual)] ← 논리적 전체 구조 (DBA 수준)
    ↓ 물리적 데이터 독립성
[내부 스키마 (Internal)]  ← 물리적 저장 방식 (파일, 인덱스)
```

| 독립성 | 의미 |
|--------|------|
| **논리적 독립성** | 개념 스키마 변경해도 외부 스키마(앱) 영향 없음 |
| **물리적 독립성** | 내부 스키마 변경해도 개념 스키마 영향 없음 |

## Part 5. 동시성 제어와 회복 — 장애에도 살아남기

### 5-1. 동시성 이상현상

| 이상현상 | 설명 |
|---------|------|
| **Dirty Read** | COMMIT 안 된 데이터를 다른 트랜잭션이 읽음. T1 롤백 시 T2는 허상 값을 읽은 것 |
| **Non-Repeatable Read** | 같은 쿼리를 두 번 실행했더니 다른 값 반환. T2가 수정 COMMIT 후 T1 두 번째 읽기 시 발생 |
| **Phantom Read** | 같은 쿼리 두 번 실행 시 새로운 행이 나타남. T2가 INSERT COMMIT 후 T1 두 번째 COUNT 시 발생 |
| **Lost Update** | 두 트랜잭션이 동시에 수정하면 하나의 수정이 덮어씌워짐 |

---

### 5-2. 동시성 제어 기법 — Lock, Timestamp, OCC, MVCC

**잠금(Lock) 기반**

| 잠금 | 설명 |
|------|------|
| **S-Lock (공유 잠금)** | 읽기 잠금. 여러 트랜잭션 동시 S-Lock 가능. X-Lock과 충돌 |
| **X-Lock (배타 잠금)** | 쓰기 잠금. 다른 모든 잠금과 충돌 |

**2PL (2단계 잠금 프로토콜)**

- **성장 단계:** 잠금 획득만 가능 (해제 불가)
- **수축 단계:** 잠금 해제만 가능 (획득 불가)

| 2PL 유형 | 설명 |
|---------|------|
| **Conservative 2PL** | 트랜잭션 시작 전 모든 잠금 획득 (데드락 없음) |
| **Strict 2PL** | COMMIT까지 X-Lock 유지 (Dirty Read 방지) |
| **Rigorous 2PL** | COMMIT까지 모든 잠금 유지 (가장 엄격) |

**MVCC (Multi-Version Concurrency Control)**

데이터의 여러 버전을 유지하여, 읽기 시 잠금 없이 일관된 스냅샷을 제공합니다.

```
트랜잭션 시작 → 스냅샷 생성
읽기: 스냅샷 시점의 버전을 읽음 (잠금 없음)
쓰기: 새 버전 생성 (기존 버전은 Undo Log에 보존)
```

Oracle, PostgreSQL은 기본 MVCC 사용 → 읽기가 쓰기를 막지 않음

---

### 5-3. 격리 수준(Isolation Level)

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read | 기본 DB |
|---------|:----------:|:-------------------:|:------------:|--------|
| READ UNCOMMITTED | 발생 | 발생 | 발생 | - |
| READ COMMITTED | **방지** | 발생 | 발생 | Oracle, PostgreSQL |
| REPEATABLE READ | **방지** | **방지** | 발생 | MySQL InnoDB |
| SERIALIZABLE | **방지** | **방지** | **방지** | - |

---

### 5-4. 데이터베이스 회복 기법

**REDO (전방 회복):** COMMIT된 트랜잭션을 다시 실행하여 복원  
**UNDO (후방 회복):** 미완료 트랜잭션의 변경을 취소하여 이전 상태 복원

| 기법 | 설명 | REDO | UNDO |
|------|------|:----:|:----:|
| **지연 갱신** | COMMIT 전까지 디스크에 쓰지 않음 | O | X |
| **즉시 갱신** | 변경 즉시 디스크에 기록 | O | O |
| **체크포인트** | 주기적으로 메모리→디스크 반영 기록 | O | O |
| **섀도 페이징** | 원본과 복사본 유지 (로그 없음) | X | X |

**체크포인트 회복 동작:**

```
장애 발생 → 마지막 체크포인트부터 로그 분석
COMMIT 기록 있는 트랜잭션 → REDO
COMMIT 기록 없는 트랜잭션 → UNDO
```

---

### 5-5. ARIES 회복 알고리즘

현대 DBMS(SQL Server, DB2, PostgreSQL)의 표준 회복 알고리즘.

**핵심 개념**

| 개념 | 설명 |
|------|------|
| **LSN (Log Sequence Number)** | 로그 레코드의 고유 순서 번호 |
| **DPT (Dirty Page Table)** | 수정되었지만 디스크에 미기록된 페이지 목록 |
| **CLR (Compensation Log Record)** | UNDO 작업을 로그에 기록 (재장애 시 UNDO의 REDO 가능) |

**3단계 회복 프로세스**

```
1. 분석(Analysis): 마지막 체크포인트부터 로그 스캔 → DPT 재구성
2. REDO (역사 반복): 분석 결정 LSN부터 로그 순방향 재수행
3. UNDO: 미완료 트랜잭션을 역순으로 롤백 → CLR 생성
```

**핵심 철학: "역사를 반복하라(Repeating History)"** — 부분 복구가 아니라 장애 직전 상태를 완벽히 재현 후 미완료 트랜잭션만 취소

## Part 6. 인덱스와 성능 최적화 — 빠른 데이터베이스 만들기

### 6-1. 인덱스와 B-Tree

인덱스 없이는 전체 테이블 스캔(Full Table Scan)이 필요합니다. B-Tree 인덱스는 O(log n) 탐색을 보장합니다.

**B-Tree 구조**

```
                   [루트 노드]
                  50 | 100 | 200
                /    |          \
          [브랜치]  [브랜치]  [브랜치]
          10|20|30  60|70     110|150
               |
     [리프 노드] ↔ [리프 노드] ↔ [리프 노드]
     (이중 연결 리스트 → 범위 검색 효율적)
```

**클러스터드 vs 비클러스터드**

| 구분 | 클러스터드 | 비클러스터드 |
|------|----------|------------|
| 저장 방식 | 데이터가 인덱스 순서로 물리적 정렬 | 인덱스와 데이터 분리 |
| 테이블당 개수 | 1개만 가능 | 여러 개 가능 |
| 범위 검색 | 매우 빠름 | 느림 (랜덤 I/O) |

---

### 6-2. 인덱스 종류와 스캔 방식

**인덱스 종류**

| 종류 | 특징 | 적합한 경우 |
|------|------|-----------|
| **B-Tree** | 범용, 범위 검색 지원 | 대부분의 일반 컬럼 |
| **해시** | 등치(=) 검색만 O(1) | 등치 검색만 하는 경우 |
| **비트맵** | 낮은 카디널리티 컬럼에 효율적 | 성별, 상태코드 (OLAP) |
| **함수 기반** | 컬럼 함수 결과값에 인덱스 | `UPPER(이름)`, `YEAR(날짜)` |

**인덱스 스캔 방식**

| 스캔 방식 | 설명 |
|---------|------|
| **Unique Scan** | 단 하나의 행을 찾음 |
| **Range Scan** | 범위를 스캔 |
| **Full Scan** | 인덱스 전체 스캔 |
| **Covering Index** | 쿼리에 필요한 모든 컬럼이 인덱스에 포함. 테이블 접근 없이 반환 |

**인덱스 무력화 함정**

```sql
WHERE UPPER(이름) = 'HONG'      -- 컬럼에 함수 적용
WHERE 나이 + 1 = 30              -- 컬럼에 연산
WHERE 이름 LIKE '%동'            -- 선행 % 사용
```

### 6-3. 옵티마이저 — SQL의 두뇌

옵티마이저는 SQL을 받아서 "어떻게 실행하면 가장 빠를까?"를 결정하는 DBMS의 두뇌입니다.

```
SQL 입력 → [옵티마이저]
  1. 쿼리 변환 (뷰 머징, 서브쿼리 최적화)
  2. 비용 추정 (통계정보 활용)
  3. 실행 계획 선택 (최저 비용)
→ 실행 엔진 → 결과 반환
```

**RBO vs CBO**

| 구분 | RBO (규칙 기반) | CBO (비용 기반) |
|------|--------------|--------------|
| 기반 | 사전 정의 우선순위 규칙 | 통계정보 기반 비용 계산 |
| 현황 | Oracle 10g 이후 지원 중단 | 현대 모든 DBMS 기본 |

**힌트(HINT)** — 옵티마이저 잘못된 계획 선택 시 개발자가 직접 지시

| 힌트 | 용도 |
|------|------|
| `FULL(테이블)` | 풀 테이블 스캔 강제 |
| `INDEX(테이블 인덱스명)` | 특정 인덱스 사용 강제 |
| `LEADING(테이블)` | 조인 순서 지정 |
| `USE_NL(테이블)` | Nested Loop Join 강제 |
| `USE_HASH(테이블)` | Hash Join 강제 |

### 6-4. 데이터베이스 튜닝 방법론

**3단계 튜닝 프로세스**

```
1단계: 보색(조사) → Slow Query Log, 실행 계획 분석, 대기 이벤트 확인
2단계: 이행(적용) → 계층별 튜닝 방안 설계 및 적용
3단계: 평가       → 전후 비교, TPS/응답시간/CPU 측정
```

**4계층 튜닝 (중요도 순서)**

| 계층 | 내용 | 효과 |
|------|------|------|
| **1. 설계 튜닝** | 정규화/역정규화, 인덱스 설계, 파티셔닝 | 매우 큼 |
| **2. Application 튜닝** | SQL 재작성, 바인드 변수, 페이징, 배치처리 | 큼 |
| **3. Server 튜닝** | Buffer Pool 크기, DBMS 파라미터 최적화 | 중간 |
| **4. System 튜닝** | H/W 증설, SSD 교체, OS 파라미터 | 낮음~중간 |

> **황금 법칙:** 설계 → SQL → Server → System 순서로! 서버 증설은 최후 수단

## Part 7. 분산 데이터베이스와 확장 — 여러 서버로 나누기

### 7-1. 파티셔닝 — 대용량 테이블 쪼개기

**파티션 프루닝 (Partition Pruning)**

```
주문 테이블: 연도별 Range 파티셔닝
┌──────────┬──────────┬──────────┬──────────┐
│ 2022년   │ 2023년   │ 2024년   │ 2025년   │
└──────────┴──────────┴──────────┴──────────┘

WHERE 주문일자 BETWEEN '2024-01-01' AND '2024-12-31'
→ 2024년 파티션만 스캔! (나머지 75% I/O 제거)
```

**분할 기준 4가지**

| 방식 | 기준 | 적합 데이터 |
|------|------|-----------|
| **Range** | 값 범위 | 날짜, 연번 |
| **List** | 특정 값 목록 | 지역코드, 상태코드 |
| **Hash** | 해시 함수 결과 | 균등 분산 필요 |
| **Composite** | 두 방식 결합 | Range-Hash 조합 |

**수평 vs 수직 파티셔닝**

- **수평:** 행(Row) 단위 분할 → 동일한 열 구조, 다른 데이터 범위
- **수직:** 열(Column) 단위 분할 → 자주 쓰는 컬럼 / 드문 컬럼 분리

---

### 7-2. 샤딩, 리플리케이션, Query Offloading

**파티셔닝 vs 샤딩**

| 구분 | 파티셔닝 | 샤딩 |
|------|---------|------|
| 범위 | 단일 DBMS 내 분할 | 여러 물리 서버로 분산 |
| 확장 방식 | Scale-Up | Scale-Out |
| 조인 | 가능 | 서버 간 조인 비용 큼 |

**샤딩 키 전략**

| 전략 | 장점 | 단점 |
|------|------|------|
| Range 샤딩 | 범위 쿼리 효율 | Hot Spot 발생 |
| Hash 샤딩 | 균등 분산 | 범위 쿼리 비효율, 리샤딩 어려움 |
| Directory 샤딩 | 유연한 분배 | 룩업 테이블 병목 |

**리플리케이션 (Replication)**

```
Master-Slave:  쓰기 → [Master] → Binary Log → [Slave 1, Slave 2]
                       읽기 → [Slave들에서 분산]
Master-Master: 쓰기/읽기 ↔ [Master 1] ↔↔ [Master 2] ↔ 쓰기/읽기
```

**Query Offloading:** 읽기(SELECT)는 Replica로, 쓰기(DML)는 Master로 분기  
주의: 복제 지연(Replication Lag) 중 읽기 시 오래된 데이터를 읽을 수 있음

---

### 7-3. 분산 데이터베이스와 2PC

**분산 DB 8대 투명성**

| 투명성 | 의미 |
|--------|------|
| **위치** | 어느 서버에 있는지 몰라도 사용 가능 |
| **복제** | 복제본이 있는지 몰라도 사용 가능 |
| **단편화** | 테이블이 어떻게 분할됐는지 몰라도 사용 가능 |
| **장애** | 일부 노드 장애 시에도 서비스 계속 |
| **병행** | 동시 트랜잭션 충돌을 자동 해결 |
| **이름** | 어느 노드에서나 동일한 이름으로 접근 |
| **성능** | 최적 노드로 자동 라우팅 |
| **스케일** | 노드 추가/제거 시 앱 변경 불필요 |

**2PC (Two-Phase Commit)**

```
Phase 1: Prepare (투표)
코디네이터 → 참여자 A, B, C에게 PREPARE 전송
          ← 각 참여자 YES/NO 응답

Phase 2: Commit/Abort (실행)
모두 YES → 코디네이터 → 모든 참여자에게 COMMIT
하나라도 NO → 코디네이터 → 모든 참여자에게 ROLLBACK
```

**2PC 단점:** 블로킹(코디네이터 장애 시 무한 대기), SPOF, 성능(2번 네트워크 왕복)

**대안: Saga 패턴 (MSA 환경)**

```
각 서비스가 로컬 트랜잭션 실행 → 성공 시 이벤트 발행 → 다음 서비스 실행
실패 시 보상 트랜잭션(Compensating Transaction)으로 역순 롤백
```

### 7-4. DB 서버 용량 산정

**TPS 산정 공식**

```
피크 TPS = (일일 트랜잭션 수 × 피크 집중률) / 피크 지속 시간(초)

예시:
일일 트랜잭션: 1,000,000건 / 피크 집중률: 30% / 피크 지속 시간: 1시간(3,600초)
피크 TPS = (1,000,000 × 0.3) / 3,600 ≒ 83 TPS
여유율 30% 적용 → 설계 TPS = 108 TPS
```

**용량 산정 절차**

```
Step 1. 업무량(Workload) 분석 → 주요 트랜잭션, 피크타임 처리량, 데이터 증가율
Step 2. 성능 목표(SLA) 정의 → 최대 응답시간, 목표 가용성
Step 3. TPS 산정 (위 공식)
Step 4. 서버 자원별 산정 → CPU, 메모리(Buffer Pool), 디스크(용량+IOPS)
Step 5. 여유율 20~30% 적용
```

## Part 8. NoSQL과 현대 데이터베이스 — 관계형의 경계 넘기

### 8-1. RDBMS vs NoSQL vs NewSQL

| 항목 | RDBMS | NoSQL | NewSQL |
|------|-------|-------|--------|
| 스키마 | 고정 | 유연(Schema-less) | 유연/고정 혼용 |
| 트랜잭션 | ACID 완전 지원 | BASE (제한적) | ACID 완전 지원 |

| 쿼리 | SQL | 독자 API | SQL |
| 대표 제품 | Oracle, MySQL, PostgreSQL | MongoDB, Cassandra, Redis | CockroachDB, Spanner, TiDB |

**NoSQL 4가지 유형**

| 유형 | 구조 | 대표 제품 | 적합 사례 |
|------|------|----------|---------|
| Document DB | JSON/BSON 문서 | MongoDB, CouchDB | 카탈로그, 컨텐츠 |

| Graph DB | 노드-엣지-속성 | Neo4j, Neptune | 소셜, 추천, 사기탐지 |

**ACID vs BASE**

| 특성 | ACID | BASE |
|------|------|------|
| 일관성 | 강한 일관성 | 최종 일관성(Eventual Consistency) |
| 가용성 | 상대적 낮음 | 높음 |
| 적합 | 금융, 결제, 재고 | SNS, 로그, 캐시 |

### 8-2. CAP 이론과 PACELC 이론

**CAP 이론 (Brewer, 2000):** 분산 시스템에서 C, A, P를 동시에 완전히 만족할 수 없다.

| 특성 | 의미 |
|------|------|
| **C (Consistency)** | 모든 노드가 동시에 같은 데이터 반환 |
| **A (Availability)** | 모든 요청에 항상 정상 응답 |
| **P (Partition Tolerance)** | 네트워크 분단 시에도 동작 |

분산 환경에서 P는 포기 불가 → CP(HBase, Zookeeper) vs AP(Cassandra, DynamoDB) 선택

**PACELC 이론 (2012, Abadi):** CAP는 파티션 상황만 다룸. 정상 운영 중에도 Latency vs Consistency 트레이드오프 존재.

```
파티션 발생? YES → A(가용성) vs C(일관성)
파티션 발생? NO  → L(지연)   vs C(일관성)
```

| DB | PACELC 분류 |
|----|------------|
| Cassandra | PA/EL (가용성+낮은지연 우선) |
| DynamoDB | PA/EL |
| MongoDB | PA/EC |
| HBase | PC/EC (일관성 우선) |

---

### 8-3. NoSQL 모델링 패턴

**핵심 차이:** RDBMS는 데이터 먼저 정규화 → 이후 쿼리로 조합. NoSQL은 **쿼리 패턴 먼저** 분석 → 그 쿼리에 최적화된 구조 설계.

**3대 모델링 패턴**

**1. Embedding (내장):** 관련 데이터를 하나의 문서에 중첩 저장

```json
{
  "_id": "order_001",
  "customer_id": "cust_100",
  "items": [
    {"product": "노트북", "qty": 1, "price": 1500000}
  ]
}
```

적합: 1:N (N이 작고 고정적), 항상 함께 조회되는 데이터

**2. Referencing (참조):** 별도 컬렉션에 저장하고 ID로 참조. 데이터 중복 없음. 적합: 1:N (N이 크거나 무제한)

**3. Bucketing (버킷팅):** 시계열/이벤트 데이터를 시간 단위로 묶어 하나의 문서에 저장. IoT, 로그, 이벤트 스트림에 적합.

### 8-4. 인메모리 데이터베이스 — Redis

```
응답 시간 비교:
  HDD:  ~10 ms
  SSD:  ~0.1 ms
  DRAM: ~0.0001 ms (100ns)  ← 인메모리 DB
```

**Redis 영속성 옵션**

| 방식 | 설명 | 특징 |
|------|------|------|
| **RDB 스냅샷** | 주기적으로 전체 데이터 파일로 저장 | 빠른 재시작, 스냅샷 간 데이터 유실 가능 |
| **AOF** | 모든 쓰기 명령을 로그에 순차 기록 | 유실 최소화, 파일 크기 증가 |
| **혼합** | AOF + RDB 함께 사용 | 두 장점 결합 |

**Cache-Aside 패턴**

```
1. 앱이 Redis에서 데이터 조회
2. Cache Hit → 즉시 반환
3. Cache Miss → DB에서 조회 → Redis에 저장(TTL 설정) → 반환
```

### 8-5. Graph DB — 관계를 저장하다

노드(Node)와 엣지(Edge), 속성(Property)으로 데이터를 표현. 관계 자체를 일급 시민으로 취급합니다.

**RDBMS vs Graph DB — 깊은 관계 탐색 성능**

| 탐색 깊이 | RDBMS (JOIN) | Graph DB |
|---------|-------------|---------|
| depth=1 | 수십 ms | 수 ms |
| depth=3 | 수 초 | 수백 ms |
| depth=5 | 타임아웃 | 수십 초 |

Graph DB는 **Index-Free Adjacency** — 이웃 노드를 직접 포인터로 접근 (O(k)) vs RDBMS JOIN은 깊이에 따라 지수 증가 (O(N^k))

**Neo4j Cypher 쿼리 예시**

```cypher
-- 친구의 친구 탐색 (2단계)
MATCH (p:Person {name: "홍길동"})-[:KNOWS*1..2]->(friend)
RETURN friend.name;

-- 사기 탐지 - 순환 거래 탐지
MATCH (a)-[:TRANSFER*2..5]->(a) RETURN a;
```

**주요 활용:** 소셜 네트워크, 추천 시스템, 금융 사기 탐지, 지식 그래프

### 8-6. 벡터 데이터베이스 — AI의 두뇌 저장소

텍스트·이미지·오디오를 고차원 실수 벡터(임베딩)로 변환하여 저장하고, 의미론적으로 가장 유사한 벡터를 검색합니다.

```
텍스트 "NoSQL은 수평 확장에 유리하다"
    ↓ 임베딩 모델
[0.234, -0.891, 0.445, ..., 0.123]  (1536차원 벡터)
    ↓ 벡터 DB 저장
쿼리: "분산 DB의 장점은?" → 유사한 벡터 검색 → 관련 문서 반환
```

**ANN(Approximate Nearest Neighbor) 알고리즘**

| 알고리즘 | 특징 |
|---------|------|
| **HNSW** | 계층적 그래프 구조, 높은 정확도, 빠른 검색 |
| **IVF** | 클러스터 기반, 낮은 메모리 |
| **PQ** | 벡터 압축으로 메모리 대폭 절감 |

**RAG (Retrieval Augmented Generation)**

```
사용자 질문 → 쿼리 벡터 변환 → 벡터 DB에서 관련 문서 Top-K 검색
→ 프롬프트 = 검색된 문서 + 질문 → LLM 답변 생성 (할루시네이션 감소)
```

**대표 제품:** Pinecone (관리형), Weaviate (오픈소스), Milvus (대규모 분산), pgvector (PostgreSQL 확장)

## Part 9. 데이터 품질과 거버넌스

### 9-1. 데이터 품질 진단 절차와 기법

**데이터 품질 6대 차원**

| 차원 | 정의 | 오류 사례 |
|------|------|---------|
| **완전성(Completeness)** | 필수 속성에 빠짐없이 값이 있는가 | 주민번호 누락 |
| **유일성(Uniqueness)** | 중복 없이 한 번만 존재하는가 | 고객 이중 등록 |
| **유효성(Validity)** | 정해진 형식·범위에 맞는가 | 날짜 형식 오류, 음수 나이 |
| **일관성(Consistency)** | 여러 시스템에서 같은 값인가 | CRM과 ERP의 고객명 불일치 |
| **정확성(Accuracy)** | 실제 현실을 정확히 반영하는가 | 배송완료인데 "배송중" 표시 |
| **접근성(Accessibility)** | 필요한 시점에 접근 가능한가 | 리포팅 데이터 2일 지연 |

**품질 측정 공식**

```
완전성 = (전체 건수 - NULL 건수) / 전체 건수 × 100%
유일성 = (전체 건수 - 중복 건수) / 전체 건수 × 100%
유효성 = (전체 건수 - 규칙 위반 건수) / 전체 건수 × 100%
```

**4단계 진단 절차**

```
1단계: 진단 준비 → 범위 선정, 품질 기준·목표 수립
2단계: 데이터 진단 → 프로파일링, 비즈니스 규칙 검증
3단계: 품질 평가 → 점수 산출, 오류 분류, 근본 원인 분석
4단계: 개선 계획 수립·실행 → 정제(Cleansing), 프로세스 개선
```

### 9-2. DQ 인증과 ISO 8000

**DQ 인증:** 한국데이터산업진흥원(K-DATA)이 기업의 데이터 품질 수준을 공식 검증하는 제도. 유효기간 2년. 심사 항목: 데이터 관리 체계 + 데이터 값 품질(6대 지표).

**ISO 8000 시리즈**

| 파트 | 내용 |
|------|------|
| ISO 8000-8 | 데이터 품질 핵심 개념 |
| ISO 8000-61 | 데이터 품질관리 프로세스 참조 모델 |
| ISO 8000-110 | 마스터 데이터 교환 |
| ISO 8000-150 | 데이터 품질 성숙도 모델 |

**마스터 데이터 관리 (MDM)**

- **골든 레코드(Golden Record):** 중복 마스터 데이터를 통합한 단일 신뢰 원본(SSOT)
- **데이터 스튜어드:** 마스터 데이터 품질 책임자
- **데이터 포터빌리티:** 기관 간 데이터 전환·이동 권리

### 9-3. 데이터 마이그레이션 품질 검증

**무결성 vs 정합성**

| 항목 | 무결성(Integrity) | 정합성(Consistency) |
|------|-----------------|-------------------|
| 정의 | 규칙·제약 위반 없는 상태 | 연관 데이터 간 논리적 모순 없는 상태 |
| 위반 사례 | PK 중복, FK 불일치 | 주문금액 ≠ 단가×수량 |

**마이그레이션 검증 방법 5가지**

| 방법 | 설명 | 특징 |
|------|------|------|
| **레코드 수 검증** | 원천-목적지 ROW COUNT 비교 | 전수 검증, 누락 탐지 |
| **해시 비교** | MD5/SHA256 체크섬 비교 | 대량 데이터 고속 검증 |
| **샘플링 검증** | 통계적 샘플로 실제 값 비교 | 상세 검증, 업무 규칙 확인 |
| **집계 검증** | SUM, AVG 비교 | 수치 정합성 확인 |
| **참조 무결성 검증** | FK 위반 건수 확인 | 관계 정합성 확인 |

### 9-4. 생성형 AI 데이터 품질 관리

**AI 데이터 품질 추가 이슈**

| 이슈 | 설명 |
|------|------|
| **편향(Bias)** | 성별·인종·이념 편향 데이터 |
| **독성(Toxicity)** | 혐오·폭력·성적 콘텐츠 |
| **환각(Hallucination)** | AI가 사실과 다른 내용을 그럴듯하게 생성 |
| **Model Collapse** | AI 생성 데이터를 재학습하면서 품질이 점점 저하 |

**생성형 AI 데이터 품질 6대 지표 (가이드라인 v2.0, 2025.2)**

| 지표 | 의미 |
|------|------|
| 정확성 | 사실과 일치하는가 (환각 방지) |
| 완전성 | 필요한 정보가 빠짐없이 있는가 |
| 일관성 | 동일 개념이 일관되게 표현되는가 |
| **최신성** | 현재 시점을 반영하는가 (LLM 지식 컷오프 대응) |
| **다양성** | 다양한 관점과 출처를 포함하는가 (편향 방지) |
| **공정성** | 특정 집단에 편향이 없는가 |

### 9-6. 데이터 가치평가

**데이터 기본법(2022)** 시행으로 데이터가 공식 자산으로 인정. 객관적 가치 산정 방법론 필요.

**가치평가 3대 접근법**

| 접근법 | 질문 | 산정 방법 | 적합 상황 |
|--------|------|---------|---------|
| **비용 접근법** | "만드는 데 얼마 들었나?" | 재현/대체 비용 | 고유 내부 데이터 |
| **시장 접근법** | "유사 데이터가 얼마에 팔렸나?" | 비교 거래 가격 | 거래 시장 활성화 시 |
| **수익 접근법(DCF)** | "이 데이터로 얼마의 수익을?" | 미래 현금흐름 현재가치 | 수익화 모델 명확 |

**DCF 공식**

```
        n    FCFt
V =  Σ  ────────
       t=1  (1+r)^t

FCFt: t기 데이터 활용 순현금흐름
r   : 할인율 (리스크 반영)
n   : 평가 기간 (데이터 수명)
```

**제도 기관:** 과학기술정보통신부 + K-DATA(한국데이터산업진흥원) + KDX(한국데이터거래소)

## Part 10. 공간 데이터베이스 — 지도 위의 데이터

### 10-1. 공간 데이터베이스 개념

공간 객체와 그들의 위치·형상·위상 관계를 저장·관리·분석하는 전문 DB입니다. 자율주행 경로 탐색, 배달 앱 "근처 식당", 도시계획 시뮬레이션 등에 활용됩니다.

**공간 객체 유형 (OGC 표준)**

```
Geometry
 ├── Point          → 단일 좌표 (위치)
 ├── LineString     → 연결된 좌표 시퀀스 (도로, 강)
 ├── Polygon        → 닫힌 면 (구역, 건물)
 ├── MultiPoint     → 복수 점의 집합
 └── MultiPolygon   → 복수 면의 집합
```

**기존 RDBMS와의 차이**

| 항목 | RDBMS | 공간 DB |
|------|-------|--------|
| 데이터 유형 | 숫자·문자·날짜 | + 기하 객체 |
| 조회 연산 | =, <, > | ST_Intersects, ST_Contains, ST_Distance |
| 인덱스 | B-Tree | R-Tree |

**DE-9IM — 공간 관계 표현 모델**

두 공간 객체 간 Interior·Boundary·Exterior의 9가지 교차 관계를 행렬로 표현하는 OGC 표준.

| 관계 | 의미 |
|------|------|
| Equals | 동일한 객체 |
| Intersects | 어떤 방식으로든 교차 |
| Contains | A가 B를 완전히 포함 |
| Within | A가 B 안에 완전히 포함 |
| Overlaps | 부분적으로 겹침 |
| Touches | 경계만 접촉 |
| Disjoint | 완전히 분리 |

### 10-2. 공간 연산자

**2단계 처리 구조**

```
1단계: MBR 필터 (빠름, False Positive 포함)
   공간 인덱스(R-Tree)의 MBR(최소 경계 사각형)만 비교

2단계: 세밀 판별 (정확)
   실제 기하 객체의 정밀 계산으로 최종 결과 반환
```

**주요 공간 함수**

| 함수 | 기능 |
|------|------|
| ST_Distance | 두 객체 간 거리 계산 |
| ST_Buffer | 객체 주변 반경 영역 생성 |
| ST_Area | 폴리곤 면적 계산 |
| ST_Intersects | 교차 여부 판별 |
| ST_Contains | 포함 여부 판별 |
| ST_Transform | 좌표계 변환 |

### 10-3. 공간 인덱스 — R-Tree 계열

**R-Tree:** 공간 객체를 MBR(Minimum Bounding Rectangle)로 감싸 계층적으로 조직화하는 균형 트리.

```
복잡한 폴리곤 → ┌─────────────┐
                │  실제 객체   │  MBR (단순 사각형)
                └─────────────┘
트리: 루트 → 브랜치 → 리프 (실제 객체 포인터)
```

**R-Tree 계열 3가지 비교**

| 항목 | R-Tree (1984) | R+-Tree (1987) | R*-Tree (1990) |
|------|-------------|----------------|----------------|
| 오버랩 | 허용 | 완전 제거 | 최소화 |
| 객체 중복 저장 | 없음 | 있음 (경계 객체) | 없음 |
| 검색 성능 | 중간 | 단순 포함에 유리 | **최적** |
| 실무 채택 | 초기 시스템 | 제한적 | PostGIS, Oracle (표준) |

**R*-Tree 핵심:** Forced Reinsert — 분할 전 재삽입을 먼저 시도하여 오버랩과 면적 동시 최소화

---

### 10-4. 다차원 색인 구조

B-Tree가 1차원이라면, 다차원 색인은 2차원 이상의 공간을 효율적으로 탐색합니다.

**Grid File:** 다차원 공간을 격자(Grid)로 분할. 범위 질의에 빠름. 데이터 쏠림 시 빈 셀 낭비.

**k-d Tree:** k차원 데이터를 교대로 축을 바꿔가며 이진 분할. KNN 탐색에 적합. 불균형 분포 시 성능 저하.

**3종 종합 비교**

| 구분 | Grid File | k-d Tree | R-Tree |
|------|----------|---------|-------|
| 구조 | 격자 | 이진 트리 | 균형 트리 (MBR) |
| 데이터 유형 | 포인트 | 포인트 | 포인트·선·면 |
| KNN 지원 | 제한적 | 우수 | 가능 |
| 디스크 적합성 | 중간 | 낮음 | 높음 |
| 적합 상황 | 균일 분포 2D | 메모리 내 KNN | GIS 공간 DB |

**차원의 저주(Curse of Dimensionality)**

차원이 늘어날수록 데이터 간 거리가 비슷해져 유사도 탐색이 무의미해지는 현상. 고차원 벡터 검색에는 전통 다차원 인덱스 대신 **HNSW, IVF** 등 ANN 알고리즘을 사용합니다.
