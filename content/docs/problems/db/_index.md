## Part 1. 데이터 모델링 — 데이터의 설계도 그리기

### 1-1. 데이터 모델링의 3단계와 기본 원칙

집을 지을 때 설계도 없이 무작정 벽돌부터 쌓는 사람은 없겠죠? 설계도 없이 건물을 지었다간 화장실 문이 안 열리거나 기둥이 무너지는 대참사가 일어날 것입니다. 데이터베이스도 마찬가지입니다. 현실 세계의 복잡한 비즈니스 형태를 컴퓨터 안에 고스란히 담아내기 위해 우리는 **'데이터 모델링'**이라는 설계 작업을 합니다.

이 설계 작업은 마치 집을 짓는 과정처럼 크게 **3단계**로 나뉩니다.

* **개념적 모델링 (머릿속 스케치):** 
  가장 먼저 "이 건물에 어떤 방들이 들어서야 하지?" 하고 크게 밑그림을 그리는 단계입니다. 쇼핑몰을 예로 들면, "고객이라는 개념이 있고, 상품이라는 개념이 있으며, 고객은 상품을 주문한다"처럼 비즈니스의 뼈대가 되는 큰 틀을 잡는 것입니다. 이때 만드는 산출물을 **개념 ERD**라고 합니다.
* **논리적 모델링 (치수가 적힌 상세 설계도):** 
  스케치 위에 정확한 치수와 방문의 위치를 정하는 단계입니다. "고객 정보에는 이름, 전화번호, 주소가 들어가야 해", "회원번호를 기준으로 고객들을 구별하자(기본키 설정)"처럼 세부 속성들을 채워 넣습니다. 데이터가 중복되지 않도록 깔끔하게 정리하는 '정규화'도 이 단계의 주인공입니다.
* **물리적 모델링 (실제 시공 도면):** 
  마지막으로 "벽돌은 어떤 종류를 쓸지, 콘크리트는 얼마나 부을지" 구체적인 재료를 결정해 실전 도면을 그리는 과정입니다. 우리가 사용할 DBMS(MySQL, Oracle 등)의 종류에 맞춰 글자 수 제한을 두고, 조회를 빠르게 할 인덱스를 설계하는 등 실제 기계가 이해할 수 있는 형태의 **DDL 스크립트**를 작성하게 됩니다.

> **모델링할 때 지켜야 할 3대 원칙**
> 1. **소통(Communication):** 설계도는 건축주(현업 업무 담당자)와 시공사(개발자) 모두가 보고 쉽게 이해할 수 있어야 합니다.
> 2. **단순화(Granularity):** 정보를 필요 이상으로 복잡하게 쪼개거나 뭉뚱그리지 말고 딱 적절한 알갱이 크기로 나눕니다.
> 3. **논리적 표현(Logical Representation):** 특정 데이터베이스 프로그램의 한계에 구애받지 않고, 순수한 비즈니스의 규칙을 있는 그대로 명확하게 표현해내야 합니다.

### 1-2. ERD와 관계형 데이터 모델

설계도를 그릴 때 만국 공통의 기호가 있듯이, 데이터베이스 설계도를 그릴 때도 전 세계 개발자들이 약속한 그림 체계가 있습니다. 그것이 바로 **ERD(Entity Relationship Diagram)**입니다. 개체(Entity)와 그들 사이의 관계(Relationship)를 선으로 연결해 표현한 지도 같은 것이죠.

가장 흔히 쓰이는 **IE(Information Engineering) 표기법(일명 새발자국 표기법)**을 읽는 법은 의외로 간단합니다.

* `||----o{` : 한 명의 고객(`||`)은 여러 번 주문할 수도 있고 주문을 아예 안 할 수도 있습니다(`o{`, 0개 이상). 이것을 1:N 관계라고 합니다.
* `}o--o{` : 한 명의 교수가 여러 과목을 가르치고, 한 과목을 여러 교수가 가르칠 수도 있습니다. 이것을 M:N 관계라고 부르며, 실무에서는 가운데 연결 테이블을 두어 정리합니다.

이제 이 그림을 바탕으로 컴퓨터 메모리에 담길 **관계형 데이터 모델**의 필수 용어들을 정리해 볼까요?

* **릴레이션(Relation):** 우리가 흔히 말하는 **'테이블'**입니다. 엑셀 파일의 시트 한 장을 떠올려 보세요.
* **튜플(Tuple):** 테이블 내의 **'행(Row)'**을 뜻합니다. 엑셀 시트에서 데이터가 한 줄씩 기록된 가로줄 하나를 의미하지요.
* **도메인(Domain):** 하나의 컬럼(열)이 가질 수 있는 **'올바른 값들의 범위'**입니다. 예를 들어 '성별'이라는 컬럼의 도메인은 `[남, 여]`가 될 것입니다.
* **기본키(PK):** 주민등록번호처럼 테이블 안에서 각 행들을 **'유일하게 식별할 수 있는 핵심 열'**입니다.
* **외래키(FK):** 다른 테이블과 다리를 연결해주는 연결고리입니다. 주문서 테이블에 '고객번호'를 적어두면, 그 고객번호(FK)를 타고 고객 테이블의 정보(PK)를 찾아갈 수 있습니다.

### 1-3. 연결함정 — Fan Trap과 Chasm Trap

잘못 설계된 지도(ERD)를 따라 길을 찾다 보면, 길을 잃거나 낭떠러지에 다다르게 됩니다. 데이터베이스 설계에서도 테이블들의 관계를 엉뚱하게 꼬아놓으면 데이터가 왜곡되거나 조회 자체가 불가능해지는 현상이 발생하는데, 이를 **연결함정(Connection Trap)**이라고 합니다. 대표적으로 두 가지 함정이 있습니다.

#### 1. Fan Trap (부채꼴 함정) — 가짜 정보가 펄럭이며 생겨나요
가운데 있는 테이블을 중심으로 양옆으로 가지가 뻗어나가는 모양입니다.
```
[부서] ── 1:N ── [직원]
[부서] ── 1:N ── [프로젝트]
```
이 구조에서 "어느 직원이 어떤 프로젝트를 담당하고 있지?" 하고 조인을 해보면 어떻게 될까요? 직원이 속한 부서와 프로젝트가 속한 부서가 같다는 이유만으로, **실제로는 참여하지도 않은 프로젝트에 직원이 배정된 것처럼 가짜 데이터가 무수히 쏟아져 나오게 됩니다.** 부채(Fan)가 펴지듯 엉뚱한 정보가 늘어난 것이지요.
* **해결책:** 직원과 프로젝트 사이에 직접 배정 관계를 맺어 주거나 연결 테이블을 만들어야 합니다.

#### 2. Chasm Trap (균열 함정) — 데이터가 틈새로 뚝 떨어져 사라져요
테이블들이 일직선으로 연결되어 있는데, 중간의 연결고리가 느슨할 때 발생합니다.
```
[영업사원] ── (선택적) ── [고객] ── 1:N ── [주문]
```
영업사원이 아직 고객을 배정받지 못했거나, 고객에게 영업사원이 지정되지 않은 상태를 허용한다고 해봅시다. 이 상태에서 "영업사원별 주문 현황"을 조회하면 어떻게 될까요? **중간의 '고객' 정보가 비어 있는 바람에 경로가 뚝 끊겨서, 분명히 일어난 '주문' 데이터마저 조회 결과에서 누락되는 균열(Chasm) 현상이 일어납니다.**
* **해결책:** 영업사원과 주문 테이블 사이에 직접 관계를 만들어 주거나, 조회할 때 누락이 없도록 `OUTER JOIN`을 명시해 주어야 합니다.

### 1-4. 슈퍼타입 / 서브타입

세상에는 공통적인 특징을 공유하면서도, 세부적으로는 각기 다른 특징을 가진 개체들이 참 많습니다. 예를 들어 '직원'은 모두 이름과 사번을 갖지만, '정규직'은 연봉 정보를 갖고 '계약직'은 시급 정보를 가집니다. 

이때 공통점을 묶어둔 상위 개념을 **슈퍼타입(Supertype)**, 개별 특징을 담은 하위 개념을 **서브타입(Subtype)**이라고 합니다. 이것을 데이터베이스 테이블로 어떻게 옮길 것인가에 대해 3가지 선택지가 존재합니다.

```
                  [ 직원 (슈퍼타입) ] ─ 공통: 사번, 이름, 입사일
                         │
         ┌───────────────┴───────────────┐
  [ 정규직 (서브타입) ]           [ 계약직 (서브타입) ]
  - 고유: 연봉, 직급             - 고유: 계약기간, 시급
```

1. **단일 테이블 방식 (Roll-up):**
   * **형태:** 하나의 거대한 테이블에 사번, 이름, 입사일, 연봉, 직급, 계약기간, 시급을 싹 다 집어넣습니다.
   * **장점:** 조인을 할 필요가 없어 조회가 엄청나게 빠릅니다.
   * **단점:** 정규직 데이터를 넣을 때는 계약기간과 시급 컬럼이 텅 비게 되어(`NULL`) 공간 낭비와 관리 부담이 생깁니다.
2. **타입별 테이블 방식 (Roll-down):**
   * **형태:** `[직원]`, `[정규직]`, `[계약직]` 테이블을 각각 따로 만듭니다.
   * **장점:** 데이터가 깔끔하게 제자리에 들어가 낭비가 없고 정규화가 잘 유지됩니다.
   * **단점:** 한 직원의 전체 정보를 보려면 늘 테이블들을 조인해야 하므로 속도가 조금 느려질 수 있습니다.
3. **개별(구체) 테이블 방식 (One Table per Concrete Class):**
   * **형태:** 공통 테이블 없이 `[정규직(사번, 이름, 입사일, 연봉, 직급)]`과 `[계약직(사번, 이름, 입사일, 계약기간, 시급)]` 두 개의 테이블만 둡니다.
   * **장점:** 역시 조인 없이 빠르게 조회가 가능합니다.
   * **단점:** 만약 공통 속성(예: '이름'의 글자 수 제한 등)이 바뀌면 모든 테이블을 다 수정해 주어야 하므로 관리가 번거롭습니다.

### 1-5. 식별자 관계 vs 비식별자 관계

두 테이블을 부모와 자식 관계로 연결할 때, 아주 중요한 선택을 해야 합니다. "부모의 주민번호(기본키)를 자식이 물려받아 본인의 기본키 일부로 쓸 것인가, 아니면 그냥 단순한 참고용 열(일반 속성)로만 보관할 것인가?" 하는 질문입니다.

* **식별자 관계 (Identifying Relationship):**
  * 부모의 기본키가 자식의 기본키(PK) 영역으로 포함되는 관계입니다.
  * 예를 들어, **`주문` 테이블의 `주문번호`가 `주문상품` 테이블의 기본키(`주문번호` + `상품번호`)의 일부로 쓰이는 경우**입니다. 부모가 없으면 자식은 아예 태어날 수도, 고유하게 존재할 수도 없습니다.
  * ERD에서는 튼튼하게 연결되었다는 의미로 **실선**으로 그립니다.
* **비식별자 관계 (Non-Identifying Relationship):**
  * 부모의 기본키가 자식의 기본키가 아닌, 그냥 일반 데이터 컬럼(FK)으로만 들어가는 관계입니다.
  * 예를 들어, **`부서`와 `직원` 관계**입니다. 직원은 부서가 없어도 독립적인 존재(사번이 기본키)이며, 부서번호는 단순히 직원이 어느 팀 소속인지 가리키는 정보일 뿐입니다.
  * ERD에서는 유연한 관계라는 의미로 **점선**으로 표현합니다.

실무에서는 식별자 관계가 너무 많아지면 자식의 자식 테이블로 갈수록 기본키 개수가 눈덩이처럼 불어나 쿼리가 복잡해지기 때문에, 필요할 때를 제외하고는 **비식별자 관계를 기본으로 설계하는 경향**이 있습니다.

## Part 2. 정규화 — 데이터 중복을 제거하는 기술

### 2-1. 함수종속성과 암스트롱 공리

'정규화'라는 무시무시해 보이는 단어를 이해하려면, 먼저 **'함수종속성(Functional Dependency)'**이라는 데이터 세계의 인과관계를 이해해야 합니다. 

어려울 것 없습니다. 수학의 함수 $y = f(x)$ 처럼, "**X의 값을 알면 Y의 값도 반드시 하나로 결정된다**"는 뜻입니다. 이것을 기호로 `X → Y` 라고 씁니다. 이때 X를 결정자, Y를 종속자라고 부르지요.
> 📌 **예시:** `[학번] → [이름]`. 학번이 2026001인 학생의 이름은 단 한 명으로 정해집니다. 학번을 알면 이름을 알 수 있으므로, 이름은 학번에 종속되어 있습니다.

이러한 관계에는 몇 가지 유형이 있습니다.

* **완전 함수종속:** 기본키 전체가 있어야만 값이 결정되는 상황입니다. 예컨대 `{학번, 과목코드} → 성적`. 학번이나 과목코드 둘 중 하나만 알아서는 성적을 알 수 없습니다. 두 개의 열이 똘똘 뭉쳐야 성적을 결정할 수 있지요.
* **부분 함수종속:** 기본키가 여러 열로 뭉쳐 있는데, 그중 일부 열에만 종속되는 경우입니다. `{학번, 과목코드} → 학생이름`. 학생이름은 과목코드와는 상관없이 '학번'에만 종속됩니다. 기본키의 일부분에 얹혀가는 꼴입니다.
* **이행적 함수종속:** 3단 논법 같은 관계입니다. `A → B` 이고 `B → C` 이기 때문에 결국 `A → C` 가 성립하는 경우입니다. `[학번] → [학과코드]` 이고 `[학과코드] → [학과장 이름]` 이기 때문에, 결국 학번만 알면 학과장 이름을 알 수 있는 구조입니다.

이러한 함수종속성을 수학적으로 증명하고 찾아낼 수 있도록 1970년대에 윌리엄 암스트롱이 정리한 규칙이 바로 **'암스트롱 공리(Armstrong's Axioms)'**입니다. 3가지 기본 규칙만 기억해 볼까요?

* **반사율(Reflexivity):** 자식은 부모의 피를 이어받습니다. Y가 X에 포함되어 있다면 `X → Y`는 무조건 성립합니다. (예: `{이름, 학번} → 이름`)
* **첨가율(Augmentation):** 양변에 같은 양념을 쳐도 성립합니다. `X → Y` 이면 양쪽에 Z를 붙인 `XZ → YZ` 도 성립합니다. (예: `학번 → 이름` 이면 `{학번, 학과} → {이름, 학과}` 도 성립)
* **이행율(Transitivity):** 징검다리 규칙입니다. `X → Y` 이고 `Y → Z` 이면 `X → Z` 가 성립합니다.

그리고 어떤 속성 X로부터 출발해서 함수종속 꼬리를 물고 도달할 수 있는 모든 속성의 집합을 **속성 클로저(Attribute Closure, $X^+$)**라고 부릅니다. 이 클로저가 테이블 전체 속성을 다 찾아낼 수 있다면, X는 그 테이블의 주인(후보키)이 될 자격이 주어집니다.

### 2-2. 이상현상(Anomaly)과 정규화

테이블 하나에 너무 많은 종류의 데이터를 구겨 넣으면 어떤 문제가 생길까요? 서랍장 하나에 옷, 음식, 쓰레기봉투를 한데 섞어 넣은 것과 같습니다. 데이터베이스에서는 이를 **이상현상(Anomaly)**이라고 부르며, 크게 3가지 골칫거리가 생깁니다.

* **삽입 이상 (Insertion Anomaly):** 
  원치 않는 데이터까지 억지로 넣어야 하는 상황입니다. 예를 들어 학과 테이블과 학생 테이블이 하나로 합쳐져 있다면, "**아직 소속 학생이 단 한 명도 없는 신설 학과**"를 등록할 때 학생 이름 자리에 빈값(`NULL`이나 임시 데이터)을 채워 넣어야만 학과를 등록할 수 있게 됩니다.
* **삭제 이상 (Deletion Anomaly):** 
  데이터 하나를 지웠는데 소중한 다른 정보까지 세트로 휩쓸려 지워지는 비극입니다. 수강 신청을 취소한 학생 행을 지웠더니, **그 학생이 속해 있던 학과의 사무실 번호까지 데이터베이스에서 영원히 사라져 버리는 현상**이 이에 해당합니다.
* **수정 이상 (Update Anomaly):** 
  한 군데만 고쳐야 하는데 여러 군데를 일일이 고쳐야 하고, 만약 깜빡하면 데이터가 짝짝이가 되는 현상입니다. 어떤 학생이 이사를 갔는데, 이 학생이 수강하는 과목마다 주소 정보가 중복 기록되어 있어서 **일부 행만 수정되고 나머지는 옛날 주소로 남게 되는 불일치 문제**가 생깁니다.

이 세 가지 골칫거리를 원천 차단하기 위해 **테이블을 쪼개고 쪼개어 가장 단정하고 아름다운 구조로 만드는 과정**, 그것이 바로 **정규화(Normalization)**입니다.

### 2-3. 2NF, 3NF, BCNF — 실전 정규화

실무에서 데이터베이스의 품질을 보장하기 위해 거쳐야 하는 정규화 과정을 친근한 예시와 함께 단계별로 따라가 볼까요?

#### 제1정규형 (1NF): "한 칸에는 무조건 물건 하나만!"
* **규칙:** 테이블의 모든 칸(속성값)은 쪼갤 수 없는 단 하나의 값(원자값)만 가져야 합니다.
* **예시:** 학생 취미 컬럼에 `[독서, 축구]` 처럼 여러 개의 값이 쉼표로 묶여 들어가 있다면, 당장 행을 분리하거나 테이블을 쪼개어 칸마다 하나의 취미만 들어가도록 해야 합니다.

#### 제2정규형 (2NF): "기본키의 일부에 기생하는 녀석들을 쫓아내자!"
* **규칙:** 1NF를 만족하고, 기본키의 일부분에만 종속되는 **부분 함수종속을 제거**합니다.
* **예시:** `수강` 테이블의 기본키가 `{학번, 과목코드}`이고 일반 컬럼으로 `성적`과 `지도교수명`이 있다고 해봅시다. `성적`은 학번과 과목코드 둘 다 있어야 결정되지만, `지도교수명`은 '과목코드'만 알아도 결정됩니다. 
* **해결:** 기본키의 일부분인 과목코드에만 매달려 있는 `지도교수명`을 떼어내어 `과목(과목코드, 지도교수명)` 테이블로 독립시킵니다.

#### 제3정규형 (3NF): "이중 징검다리 관계를 끊어내자!"
* **규칙:** 2NF를 만족하고, 일반 컬럼끼리 종속되는 **이행적 함수종속을 제거**합니다. 즉, 기본키가 아닌 일반 컬럼이 다른 일반 컬럼을 결정하면 안 됩니다.
* **예시:** `학생(학번(PK), 이름, 학과코드, 학과장명)` 구조가 있습니다. 여기서 `학번 → 학과코드` 이고 `학과코드 → 학과장명` 이 성립하므로 이행적 종속이 존재합니다.
* **해결:** `학생(학번, 이름, 학과코드)`와 `학과(학과코드, 학과장명)`으로 테이블을 분리하여 이중 징검다리를 부숴버립니다.

#### 보이스-코드 정규형 (BCNF): "결정자는 무조건 왕(후보키)이어야 해!"
* **규칙:** 3NF를 만족하고, **모든 결정자가 후보키(테이블의 주인)**여야 합니다.
* **예시:** `수강(학생, 과목, 담당교수)` 테이블이 있습니다. 한 학생은 여러 과목을 들을 수 있고, 한 과목에는 여러 교수가 있으며, 교수는 오직 한 과목만 가르칩니다. 이때 기본키는 `{학생, 과목}`인데, `담당교수 → 과목` 이라는 관계가 성립합니다. 담당교수는 기본키(후보키)가 아닌데 과목을 결정하고 있으므로 BCNF 위반입니다.
* **해결:** `교수과목(담당교수(PK), 과목)` 테이블과 `수강신청(학생, 담당교수)` 테이블로 깔끔하게 쪼개어 정리합니다.

### 2-4. 4NF, 5NF — 고급 정규화

보통 실무에서는 BCNF까지만 진행해도 아주 훌륭하지만, 데이터의 구조가 극도로 복잡해지면 더 윗단계의 정규화가 필요할 때가 있습니다.

* **제4정규형 (4NF) — 다치종속(MVD) 제거**
  * **개념:** 하나의 테이블에 전혀 상관없는 두 가지 이상의 N대 관계가 얽혀 있을 때 발생합니다.
  * **예시:** 어떤 과목에 여러 개의 '추천 교재' 목록이 있고, 여러 명의 '담당 교수' 목록이 있는데 이 둘은 서로 관련이 없습니다. 그런데 테이블에 이를 한꺼번에 넣으려고 하면 `(과목명, 담당교수, 교재)` 모든 조합의 행을 억지로 다 만들어 넣어야 하는 중복 폭발이 일어납니다.
  * **해결:** `과목교수(과목명, 담당교수)` 테이블과 `과목교재(과목명, 교재)` 테이블로 쪼개어 서로 독립된 데이터를 편안하게 관리해 줍니다.
* **제5정규형 (5NF) — 조인종속(JD) 제거**
  * **개념:** 테이블을 여러 개로 쪼갰다가 다시 합쳐서(`JOIN`) 원래대로 복원하려고 할 때, 쪼개는 방식에 따라 엉뚱하게 없던 데이터가 생성되는 현상을 막는 정규형입니다. 관계가 3개 이상 얽혀 있을 때 사용하며, 실무에서는 거의 만나기 힘든 학술적인 영역에 가깝습니다.

### 2-5. 반정규화 — 성능을 위한 타협

정규화를 철저하게 해서 테이블을 수십 개로 쪼개놓으면 데이터가 참 맑고 깨끗해집니다. 하지만 큰 단점이 생깁니다. 화면에 정보 하나를 띄우려는데 테이블 10개를 조인해야 해서 로딩 속도가 굼벵이처럼 느려지는 것이죠. 이때 성능을 위해 눈물을 머금고 데이터를 일부러 중복시키거나 테이블을 다시 합치는 것을 **'반정규화'**라고 합니다.

> **대원칙: 선정규화 후조율**
> 처음부터 테이블을 뭉뚱그려 만들지 않는 것이 좋습니다. 우선 정규화를 충분히 수행한 뒤, 실제 서비스 테스트에서 성능이 병목이 되는 지점을 확인하고 나서 반정규화를 적용하는 순서가 맞습니다.

주요 기법은 다음과 같습니다.

* **테이블 병합:** 자주 같이 쓰이는 `주문` 테이블과 `주문상품` 테이블을 그냥 하나로 합쳐 조인 횟수를 줄입니다.
* **컬럼 중복:** 상품명을 알기 위해 상품 테이블로 조인해 가기 귀찮으니, `주문상품` 테이블 안에 아예 `상품명` 컬럼을 한 번 더 써둡니다.
* **파생 컬럼 추가:** 매번 모든 매출 금액을 더해서(`SUM`) 보여주면 서버가 힘들어하므로, `주문` 테이블에 아예 `총주문합계` 컬럼을 만들어 실시간으로 적어둡니다.

**부작용:** 반정규화를 적용하면 수정 이상 현상이 다시 발생할 수 있으므로, 정보의 수정이 거의 일어나지 않는 로그성 데이터나 대량 통계 시스템(DW) 위주로 활용해야 안전합니다.

## Part 3. SQL과 관계대수 — 데이터를 다루는 언어

### 3-1. 관계대수 — SQL의 이론적 기반

우리가 작성하는 `SELECT * FROM 고객 WHERE 나이 > 20` 이라는 멋진 SQL 문장은 하늘에서 뚝 떨어진 것이 아닙니다. 수학적인 수학 공식 체계인 **'관계대수(Relational Algebra)'**를 컴퓨터가 알아먹기 쉽게 포장한 것에 불과하답니다.

* **관계대수는 '절차적(HOW)'이에요:** "어떤 순서로 데이터를 끄집어내고 처리해라" 하고 행동 지침을 공식으로 명시합니다.
* **관계해석(그리고 SQL)은 '비절차적(WHAT)'이에요:** 과정은 관심 없고 "내가 원하는 데이터는 이런 모양이니까 알아서 가져와!" 하고 결과물만 선언합니다.

관계대수 공식에서 쓰이는 기본 연산자들을 SQL과 1:1로 매칭해 볼까요?

| 연산자 이름 | 수학 기호 | SQL 매칭 | 쉽게 말해서 |
| :--- | :---: | :---: | :--- |
| **선택 (Selection)** | $\sigma$ | `WHERE` | 필터링해서 조건에 맞는 **행(Row)**만 골라내기 |
| **투영 (Projection)** | $\pi$ | `SELECT` | 필요한 **열(Column)**만 쏙 빼내기 |
| **조인 (Join)** | $\bowtie$ | `JOIN` | 공통 연결고리를 기준으로 두 테이블 합치기 |
| **나누기 (Division)** | $\div$ | 서브쿼리 활용 | "특정 그룹의 모든 상품을 구매한 고객"처럼 전부 매칭되는 행 찾기 |

이 외에도 고등학교 수학 시간에 배운 합집합($\cup$, `UNION`), 교집합($\cap$, `INTERSECT`), 차집합($-$, `EXCEPT`), 그리고 두 테이블의 모든 조합을 곱해버리는 카티션 프로덕트($\times$, `CROSS JOIN`)가 있습니다.

---

### 3-2. JOIN의 종류와 활용

데이터베이스에 흩어진 테이블들을 자석처럼 찰칵 끌어당겨 하나로 만드는 핵심 마법이 바로 **JOIN(조인)**입니다. 어떤 방식으로 두 테이블을 매칭할 것인가에 따라 종류가 나뉩니다.

* **INNER JOIN (내부 조인):**
  * 서로 조건이 완벽하게 일치하는 데이터만 합쳐서 보여줍니다.
  * 예: "주문한 내역이 있는 고객 정보" (주문 안 한 고객은 탈락, 고객 정보가 없는 낙동강 오리알 주문 건도 탈락)
* **LEFT/RIGHT OUTER JOIN (외부 조인):**
  * 한쪽 테이블을 기준으로 삼고, 상대편 테이블에 일치하는 데이터가 없더라도 일단 기준 테이블 내용은 전부 보여줍니다. 비어 있는 반대편 칸은 `NULL`로 표시됩니다.
  * 예: "주문을 안 했어도 일단 우리 서비스에 등록된 모든 고객과 그들의 주문 현황 조회" (LEFT OUTER JOIN)
* **FULL OUTER JOIN (완전 외부 조인):**
  * 양쪽 테이블의 모든 데이터를 긁어모아 보여줍니다. 짝이 없는 데이터는 서로 `NULL`로 매칭됩니다.
* **CROSS JOIN (교차 조인):**
  * 수학의 곱집합입니다. 왼쪽 테이블 10개 행과 오른쪽 테이블 10개 행의 모든 조합인 100개 행을 무조건 출력합니다.
* **SELF JOIN (자체 조인):**
  * 테이블이 자기 자신과 조인하는 경우입니다. "이 직원의 사수 이름은 무엇인가?"처럼 한 테이블 안에 직원 번호와 사수 번호가 같이 묶여 있을 때 사용합니다.
* **EQUI JOIN vs NON-EQUI JOIN:**
  * 조인 조건으로 `=`을 사용하면 EQUI JOIN, `>`, `<` 이나 `BETWEEN` 처럼 범위 조건으로 묶으면 NON-EQUI JOIN 이라고 부릅니다.

---

### 3-3. JOIN 물리적 알고리즘 — NL, Sort Merge, Hash Join

우리가 똑같은 SQL JOIN 문을 날려도, 데이터베이스 내부 엔진은 데이터의 크기와 상태에 따라 서로 다른 물리적인 방식으로 계산기 판을 굴립니다. 이를 **조인 알고리즘**이라 합니다.

#### 1. Nested Loop Join (NL 조인) — 이중 반복문 돌리기
가장 직관적인 방식입니다. 바깥 테이블(Outer)에서 첫 번째 행을 꺼내어, 안쪽 테이블(Inner)을 처음부터 끝까지 스캔하며 조건이 맞는지 찾습니다. 그 다음 두 번째 행을 꺼내 반복합니다.
* **비유:** 학생 명부 10명을 들고, 출석부 책장을 한 장씩 넘기며 이름이 매치되는지 일일이 찾는 것과 같습니다.
* **특징:** 데이터양이 적고, 안쪽 테이블에 **인덱스**가 이쁘게 잘 걸려 있어서 순식간에 콕 찝어 찾을 수 있을 때 최상의 효율을 냅니다.

#### 2. Sort Merge Join — 정렬 후 지퍼 올리기
조인할 두 테이블을 정렬 기준으로 먼저 줄을 세운 다음, 머리를 맞대고 순서대로 매칭해 나가는 방식입니다.
* **비유:** 남자 줄, 여자 줄을 키 순서대로 서게 한 뒤, 양쪽 줄 앞에서부터 차례대로 비교하며 파트너를 맞춰 나가는 것입니다.
* **특징:** 데이터가 많고 조인 조건이 `=`이 아니라 범위 조건(`>`, `<`)일 때 유용하며, 이미 인덱스 덕분에 정렬이 되어 있다면 아주 빠르게 작동합니다.

#### 3. Hash Join — 해시 양동이에 담아 매칭하기
먼저 두 테이블 중 작은 테이블을 읽어 메모리에 '해시 테이블(바구니 분류함)'을 빌드합니다. 그 후 큰 테이블을 읽으며 해시 함수를 통과시켜 알맞은 바구니에 다이렉트로 던져 넣어 짝을 짓습니다.
* **비유:** 우편물을 지역별 바구니(서울, 부산, 대구 등)에 미리 분류해 두고, 편지가 올 때마다 주소를 보고 해당 바구니에 쏙쏙 집어넣는 것입니다.
* **특징:** 대용량 데이터를 처리할 때 가장 빠르며 인덱스가 없어도 무방합니다. 단, 메모리를 많이 차지하며 오직 **등치(=) 조건**에서만 사용이 가능합니다.

### 3-4. 동적 SQL — 런타임에 SQL 만들기

**정적 SQL**은 컴파일 시점에 쿼리 구조가 코드 안에 완전히 고정되어 있지만, **동적 SQL**은 사용자 입력이나 조건에 따라 런타임에 쿼리 문자열을 조립하여 실행합니다.

데이터베이스가 SQL 쿼리를 실행하려면 책을 분석하듯 문법을 확인하고 최적의 길(실행 계획)을 찾는 분석 단계를 거쳐야 합니다. 이를 **파싱(Parsing)**이라고 합니다.

* **하드 파싱(Hard Parsing):** 처음 들어온 쿼리를 위해 온갖 뇌세포를 쥐어짜며 실행 경로를 새로 짜는 힘든 과정입니다. CPU를 많이 씁니다.
* **소프트 파싱(Soft Parsing):** 이전에 한번 짰던 계획이 메모리 캐시에 남아 있어서, "아, 이거 아까 했던 거네!" 하고 계획표만 재사용하는 가벼운 과정입니다.

검색 조건이 매번 바뀌는 웹페이지에서 쿼리를 조립할 때(동적 SQL), 만약 값을 그냥 문자열 덧셈으로 쿼리에 합치면 어떻게 될까요?
```java
// 하드 파싱 지옥행 열차 ❌
String sql = "SELECT * FROM 고객 WHERE id = " + userId; 
// userId가 1일 때, 2일 때, 100일 때 쿼리 모양이 매번 달라지므로 계속 하드 파싱을 합니다.
```

이 고통을 해결해 주는 구원투수가 바로 **바인드 변수(Bind Variable)**입니다.
```java
// 소프트 파싱의 축복 ⭕
String sql = "SELECT * FROM 고객 WHERE id = ?"; 
```
물음표(`?`)라는 임시 자리를 뚫어두면 데이터베이스는 쿼리 모양 자체를 고정된 템플릿으로 인식하여 최초 1회만 하드 파싱을 하고 이후에는 값을 갈아 끼우며 **소프트 파싱**으로 초고속 실행을 합니다. 게다가 해커가 엉뚱한 SQL을 주입해 시스템을 망가뜨리는 **SQL Injection 공격**도 원천 방어해 줍니다.

## Part 4. 무결성과 트랜잭션 — 데이터를 신뢰할 수 있게

### 4-1. 데이터 무결성의 종류와 유지 방법

'무결성(Integrity)'이란 데이터에 결함이 없고 완벽하게 깨끗하며 믿을 수 있는 상태를 뜻합니다. 데이터베이스에 이상한 쓰레기 데이터가 들어오지 못하도록 막는 안전망 역할을 합니다. 크게 4가지 종류가 있습니다.

* **엔티티 무결성:** 
  테이블의 주인공인 기본키(PK)는 중복될 수 없고, 절대 빈칸(`NULL`)으로 비워둘 수 없다는 규칙입니다. (예: 학번 없는 학생은 존재할 수 없음)
* **참조 무결성:** 
  외래키(FK) 값은 반드시 부모 테이블에 존재하는 실제 PK 값만을 가리켜야 한다는 규칙입니다. (예: 존재하지 않는 '99번 부서' 소속으로 직원을 등록할 수 없음)
* **도메인 무결성:** 
  속성값이 미리 약속된 형식과 범위 내에 있어야 합니다. (예: '나이' 컬럼에 300살이나 -5살이 입력될 수 없고, '성별'에는 남/여 외에 다른 글자가 올 수 없음)
* **사용자 정의 무결성:** 
  기타 비즈니스 규칙에 맞는 데이터만 허용하는 것입니다. (예: "주문 총액은 항상 0보다 커야 한다"와 같은 고유 업무 규칙을 트리거나 제약조건으로 관리)

> **부모 데이터를 지울 때 자식 데이터를 어떻게 할까? (ON DELETE 옵션)**
> 부모인 `부서` 테이블에서 '개발팀'을 지우려고 합니다. 이때 개발팀을 참조하고 있는 자식 `직원` 테이블의 행동 방식은 3가지가 있습니다.
> - **CASCADE:** 부모 지울 때 자식도 동반 삭제합니다. (개발팀 삭제 시 소속 직원들도 전부 자동 삭제)
> - **SET NULL:** 부모가 지워지면 자식의 소속 정보를 빈값(`NULL`)으로 고칩니다. (소속 부서를 미지정 상태로 변경)
> - **RESTRICT / NO ACTION:** 자식이 달라붙어 있으면 부모 삭제 자체를 거부합니다. (소속 직원이 남아 있으면 개발팀을 삭제할 수 없음)

### 4-2. 트랜잭션 ACID — 믿을 수 있는 작업 단위

**트랜잭션(Transaction)**이란 데이터베이스에서 결코 쪼갤 수 없는 하나의 완벽한 작업 묶음입니다. 가장 대표적인 비유가 바로 **'은행 계좌 이체'**입니다.

```
[ A계좌 잔액 감소 ] ── 동시에 성공하거나, 아니면 둘 다 실패해야 함! ── [ B계좌 잔액 증가 ]
```

A계좌에서 돈은 빠져나갔는데 컴퓨터 다운으로 B계좌에 입금이 안 되었다면 난리가 나겠지요? 이런 불상사를 막기 위해 트랜잭션은 **ACID**라는 4가지 황금 약속을 반드시 지켜야 합니다.

* **원자성 (Atomicity) — "전부 적용하든지, 아니면 아예 없었던 일로!"**
  * 작업 중 하나라도 실패하면 전체 작업을 롤백(Rollback)하여 초기 상태로 되돌립니다.
  * 보장 방식: **Undo 로그**에 이전 상태를 적어두었다가 실패 시 되감기 합니다.
* **일관성 (Consistency) — "전후 사정은 언제나 올바르게!"**
  * 트랜잭션이 끝나고 나도 데이터베이스의 규칙(제약 조건)들은 변함없이 잘 지켜져야 합니다. (예: 잔액이 마이너스가 되면 안 됨)
* **격리성 (Isolation) — "네가 무슨 일을 하든 난 나만의 길을 간다"**
  * 여러 사람이 동시에 데이터를 만지더라도, 각각의 트랜잭션은 마치 나 혼자 데이터베이스를 쓰고 있는 것처럼 조용하고 독립적으로 실행되어야 합니다.
  * 보장 방식: **Lock(잠금)**을 걸거나 데이터 스냅샷 버전(**MVCC**)을 이용합니다.
* **지속성 (Durability) — "약속한 건 영원히 기억해"**
  * 일단 한 번 완료(`COMMIT`) 신호를 보낸 작업 결과는 설령 바로 다음 순간 정전이 나거나 서버가 불타더라도 디스크에 영구적으로 안전하게 저장되어 보존되어야 합니다.
  * 보장 방식: 변경사항을 실제 데이터 파일에 쓰기 전에 로그에 먼저 기록하는 **WAL(Write-Ahead Logging)** 원칙을 따릅니다.

---

### 4-3. ANSI/SPARC 3단계 아키텍처

데이터베이스를 다루는 사람들은 참 다양합니다. 홈페이지 사용자, 시스템 설계자, 하드웨어 관리자까지 바라보는 시선이 다 다르지요. 1970년대 ANSI/SPARC 위원회는 데이터베이스의 내부가 바뀌더라도 사용자가 쓰는 프로그램은 영향을 받지 않도록(데이터 독립성 확보), 데이터베이스 구조를 3층짜리 빌딩으로 깔끔하게 나누어 설계했습니다.

```
  [ 사용자 / 애플리케이션 ] ───── 각각의 화면 뷰 (외부 스키마)
             │
   논리적 데이터 독립성 (구조가 바뀌어도 화면은 그대로!)
             │
     [ 개념 스키마 ] ────────── 전사 관점의 전체 데이터 논리 구조 (설계도)
             │
   물리적 데이터 독립성 (디스크 장비를 바꿔도 논리 구조는 그대로!)
             │
     [ 내부 스키마 ] ────────── 실제 디스크 저장 방식, 파일 구조, 인덱스 등
```

* **외부 스키마 (External Schema):** 
  일반 사용자나 개발자가 바라보는 화면 뷰입니다. 예를 들어 영업팀은 고객의 매출 정보만 보이고, 인사팀은 직원의 급여 정보만 보이는 식입니다.
* **개념 스키마 (Conceptual Schema):** 
  데이터베이스 전체의 뼈대 설계도입니다. 모든 테이블의 규칙, 관계, 보안 정책이 통합되어 모여 있는 곳이며 보통 DBA가 관리합니다.
* **내부 스키마 (Internal Schema):** 
  실제 컴퓨터 디스크 안에 데이터가 어떤 물리적 파일 형식으로 배치되고 어떤 인덱스를 타는지 기록되는 물리적인 계층입니다.

이 삼중 계층 덕분에 디스크 용량이 부족해서 하드웨어를 SSD로 바꾸고 인덱스를 다시 짜더라도(내부 스키마 변경), 우리가 설계한 테이블 구조(개념 스키마)는 흔들리지 않으며(물리적 독립성), 테이블 컬럼명이 살짝 바뀌더라도 중간에서 매핑만 잘 조정해주면 사용자가 쓰는 앱 화면(외부 스키마)은 오류 없이 그대로 작동하게 됩니다(논리적 독립성).

---

## Part 5. 동시성 제어와 회복 — 장애에도 살아남기

### 5-1. 동시성 이상현상

수많은 사용자가 동시에 들어와 같은 데이터를 읽고 쓰는 현대 웹 애플리케이션에서는 교통정리가 필수입니다. 제대로 제어하지 않으면 다음과 같은 해괴한 일들이 일어납니다.

* **Dirty Read (지저분하게 읽기):**
  * 다른 사람이 아직 수정 중이고 확정(`COMMIT`) 짓지도 않은 중간 상태의 임시 데이터를 읽는 현상입니다. 상대방이 작업을 취소(`ROLLBACK`)해버리면, 나는 실존하지 않는 허깨비 데이터를 보고 의사결정을 내린 꼴이 됩니다.
* **Non-Repeatable Read (반복 불가능한 읽기):**
  * 내가 한 트랜잭션 안에서 A라는 값을 조회하고 잠시 후 다시 조회했는데, 그 짧은 새에 다른 트랜잭션이 값을 수정하고 승인해버려 다른 값이 튀어나오는 현상입니다. 같은 쿼리인데 결과가 다르게 나옵니다.
* **Phantom Read (유령 읽기):**
  * 방 안에 없던 유령이 불쑥 나타나듯, 내가 데이터를 두 번 조회하는 사이에 다른 사람이 새로운 데이터를 밀어 넣어(`INSERT`) 갑자기 한 행이 더 늘어나는 현상입니다.
* **Lost Update (수정 사항 분실):**
  * 두 명이 동시에 글을 고쳐서, 한 사람이 공들여 수정한 내용이 다른 사람이 덮어쓰는 바람에 허공으로 날아가 버리는 문제입니다. 격리성이 제대로 보장되지 않을 때 발생하며, Lock이나 MVCC로 방어합니다.

---

### 5-2. 동시성 제어 기법 — Lock, Timestamp, OCC, MVCC

교통사고를 방지하기 위해 데이터베이스 엔진이 사용하는 대표적인 4가지 무기를 소개합니다.

#### 1. 잠금(Lock) 기반 기법 — "내가 화장실 쓰고 있으니 문 잠글게"
가장 클래식한 방법입니다. 데이터를 만질 때 자물쇠를 걸어 잠급니다.
* **S-Lock (Shared / 공유 잠금):** "나 읽는 중이니까 아무도 지우거나 고치지 마!" (여러 명이 동시에 읽는 S-Lock 끼리는 허용되지만 쓰기 잠금은 대기시킵니다)
* **X-Lock (Exclusive / 배타 잠금):** "나 쓰는 중이니까 읽지도 말고 쓰지도 말고 저리 가!" (모든 접근을 차단합니다)

> **2단계 잠금 프로토콜 (2PL)**
> 락을 거는 것도 규칙이 있습니다. 아무 때나 락을 걸고 풀면 모순이 생기므로, **성장 단계**(트랜잭션 중에는 락을 얻기만 하고 해제하지 못함)와 **수축 단계**(한번 락을 풀기 시작하면 다시는 새로운 락을 얻지 못함)의 2단계로 락을 관리하는 규칙입니다.
> - **Strict 2PL:** 변경에 사용된 X-Lock은 트랜잭션이 완벽히 끝날(`COMMIT/ROLLBACK`) 때까지 꽉 쥐고 놓아주지 않는 실무적인 변형입니다. 이를 통해 Dirty Read를 완벽히 막아줍니다.

#### 2. MVCC (다중 버전 동시성 제어) — "싸우지 말고 복사본 각자 봐"
현대적인 대다수 DBMS(PostgreSQL, MySQL InnoDB, Oracle 등)가 채택한 대세 기법입니다. 
데이터를 직접 고치기보다, 데이터의 **'과거 버전 복사본'**들을 저장 공간(Undo Log)에 남겨둡니다. 덕분에 데이터를 누군가 수정하고 있어도, 읽는 사람은 락에 걸려 대기할 필요 없이 그냥 내 스냅샷 시점의 옛날 버전을 편안히 읽으면 됩니다. "**읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다**"는 놀라운 성능의 비결입니다.

#### 3. Timestamp Ordering & OCC (낙관적 분산 제어)
* **Timestamp:** 트랜잭션이 들어올 때 시간표(타임스탬프) 표를 나눠주고 그 순서대로만 작동하게 강제하는 방식입니다.
* **OCC (Optimistic Concurrency Control):** "설마 부딪히겠어?" 하고 일단 락 없이 자유롭게 작업을 실행한 뒤, 마지막에 커밋하기 직전에 "혹시 충돌난 데이터 있나요?" 검증하고 만약 문제가 생겼을 때만 롤백하는 쿨한 기법입니다. 충돌이 거의 없는 환경에서 유리합니다.

---

### 5-3. 격리 수준(Isolation Level)

앞서 배운 동시성 문제들을 얼마나 꼼꼼하게 막아줄지 레벨을 정해놓은 약속입니다. 잠금을 쎄게 걸수록 안전하지만 속도가 느려지고, 잠금을 풀수록 성능은 날아다니지만 데이터가 꼬이게 됩니다.

| 격리 수준 (Isolation Level) | Dirty Read | Non-Repeatable Read | Phantom Read | 대략적인 동작 방식과 비유 |
| :--- | :---: | :---: | :---: | :--- |
| **READ UNCOMMITTED** | 발생 가능 | 발생 가능 | 발생 가능 | 락을 거의 안 걸어요. 남이 쓰다 만 임시 데이터도 슬쩍 봅니다. |
| **READ COMMITTED** | **방지됨** | 발생 가능 | 발생 가능 | 남이 최종 승인(Commit)한 데이터만 읽습니다. (Oracle 등 기본값) |
| **REPEATABLE READ** | **방지됨** | **방지됨** | 발생 가능 | 트랜잭션 시작 때 읽은 값은 끝날 때까지 보장받습니다. (MySQL 기본값) |
| **SERIALIZABLE** | **방지됨** | **방지됨** | **방지됨** | 완벽히 차례대로만 실행시킵니다. 성능은 가장 느립니다. |

---

### 5-4. 데이터베이스 회복 기법

갑자기 전기가 나가거나 하드디스크가 뻑나더라도 데이터베이스는 오뚜기처럼 원래 정상 상태로 벌떡 일어나야 합니다. 복구를 위해 두 가지 대원칙을 씁니다.

* **REDO (다시 하기):** COMMIT까지 무사히 성공한 트랜잭션의 결과물인데 디스크에 덜 쓰여졌다면, 로그 기록을 보고 처음부터 끝까지 다시 실행해서 디스크에 확실히 심어줍니다.
* **UNDO (취소 하기):** 트랜잭션 도중에 서버가 꺼져서 미완성 상태인 녀석들은 데이터베이스를 어지럽히지 않도록 로그를 보고 흔적을 지워 깔끔하게 취소합니다.

DBMS는 이 복구를 위해 로그를 기록하는 방식에 따라 여러 회복 전략을 씁니다.
* **즉시 갱신 기법:** 트랜잭션 중간이라도 일단 디스크에 쓰고 봅니다. 장애 시 **REDO와 UNDO가 둘 다 필요**합니다.
* **지연 갱신 기법:** 커밋되기 전까진 실제 메모리 버퍼에만 얹어두고 디스크 저장은 미룹니다. 장애 시 미완성 트랜잭션은 디스크에 애초에 안 써졌으므로 **REDO만 필요**합니다(UNDO는 생략).
* **체크포인트 기법:** 5분이나 10분 간격으로 "여기까지 안전하게 디스크에 다 저장 완료함!" 하고 체크포인트 깃발을 꽂아둡니다. 사고가 나면 처음부터 다 뒤질 필요 없이 마지막 깃발 꽂힌 시점부터만 로그를 뒤져 복구 시간을 획기적으로 줄여줍니다.

### 5-5. ARIES 회복 알고리즘

현대적인 대형 데이터베이스 시스템(PostgreSQL, DB2, SQL Server 등)이 표준으로 채택하고 있는 복합적이고 아주 정교한 회복 알고리즘입니다. 

ARIES의 핵심 철학은 "**역사를 반복하라 (Repeating History)**"입니다. 
서버가 픽 쓰러졌다가 다시 켜지면, 실패했든 성공했든 상관없이 일단 장애가 났던 그 직전 0.1초 상태의 메모리와 디스크 모습을 그대로 똑같이 재현해 냅니다. 그러고 나서 느긋하게 미완성 트랜잭션만 골라내어 역순으로 지워나가는(UNDO) 방식입니다.

**3단계 회복 프로세스:**
1. **분석 단계 (Analysis):** 마지막 체크포인트 시점부터 로그를 순방향으로 읽으며, 사고 당시 수정 중이었던 더러운 페이지(Dirty Page) 목록과 미처 못 끝난 트랜잭션 목록을 파악합니다.
2. **REDO 단계 (역사 반복):** 분석 결과를 바탕으로, 옛날 로그를 전부 끄집어내어 장애 직전 시점까지 그대로 쭉 덮어쓰기 해서 메모리 상태를 부활시킵니다.
3. **UNDO 단계 (청소 시간):** 미완성 상태의 나쁜 트랜잭션들을 로그 역순으로 추적하며 값을 원래대로 되돌립니다. 이때 되돌리기 작업을 수행했다는 증거 로그(**CLR, Compensation Log Record**)를 새로 남기기 때문에, UNDO를 하는 와중에 또 서버가 꺼지는 끔찍한 연쇄 장애가 나더라도 안전하게 복구를 이어 나갈 수 있습니다.

## Part 6. 인덱스와 성능 최적화 — 빠른 데이터베이스 만들기

### 6-1. 인덱스와 B-Tree

두꺼운 전공 서적에서 "정규화"라는 단어를 찾고 싶다고 해봅시다. 책 첫 장부터 마지막 장까지 한 글자씩 다 훑는다면 하루 종일 걸릴 텐데, 이를 데이터베이스에서는 **전체 테이블 스캔(Full Table Scan)**이라고 부릅니다. 

하지만 책 맨 뒤에 있는 **'찾아보기(색인)'** 페이지를 열어서 'ㅈ' 단락의 '정규화: 154페이지'를 찾아 곧바로 154페이지로 넘어간다면 5초 안에 해결되겠지요? 이 찾아보기 페이지가 바로 **인덱스(Index)**입니다.

데이터베이스 인덱스는 대부분 **B-Tree (Balanced Tree)** 구조로 설계되어 있습니다.

```
                    [ Root Node (뿌리) ]
                         50 | 100
                       /    |    \
            [ Branch ]   [ Branch ]   [ Branch ]
            10 | 30      60 | 80      110 | 130
             /   \        /   \        /    \
            [ Leaf ] ↔ [ Leaf ] ↔ [ Leaf ] ↔ [ Leaf ]
             (실제 데이터 혹은 물리적인 디스크 주소 포인터)
```

이 구조는 데이터가 아래로 가지를 뻗을 때 좌우 밸런스를 알아서 척척 맞추기 때문에, 아무리 데이터가 수천만 건으로 불어나도 단 3~4번의 홉(Hop)만 거치면 원하는 곳에 도달하는 놀라운 효율($O(\log N)$)을 자랑합니다. 리프 노드들끼리는 양방향 연결 리스트로 손을 잡고 있어서 `WHERE 범위 BETWEEN 10 AND 50` 같은 범위 검색도 물 흐르듯 빠르게 긁어옵니다.

#### 인덱스의 갈림길: 클러스터드 vs 비클러스터드
* **클러스터드 인덱스 (Clustered) — 진짜 사전 같은 책:**
  * 책 내용 자체가 가나다순으로 정렬되어 쓰인 진짜 국어사전 같은 구조입니다. 인덱스가 곧 데이터 페이지 자체입니다.
  * 테이블당 단 **1개만** 만들 수 있고(기본키 설정 시 자동 생성), 범위 검색에서 특히 빠릅니다.
* **비클러스터드 인덱스 (Non-Clustered) — 뒤에 붙은 찾아보기 쪽지:**
  * 실제 책 페이지는 뒤죽박죽 쓰여 있고, 맨 뒷장에 정렬된 색인 쪽지만 둔 구조입니다. 색인 쪽지에는 "정규화: 102페이지로 가세요" 라는 이정표(물리 주소 포인터)가 적혀 있습니다.
  * 테이블당 **여러 개** 만들 수 있어 유연하지만, 쪽지 보고 실제 페이지 찾아가는 과정(랜덤 I/O)이 필요해 클러스터드보다는 조금 더 느립니다.

---

### 6-2. 인덱스 종류와 스캔 방식

인덱스를 걸어두기만 한다고 만사형통이 아닙니다. 어떤 인덱스를 쓸지, 그리고 데이터베이스 엔진이 그 길을 어떻게 걸어 다닐지(스캔할지) 파악하는 것이 튜닝의 핵심입니다.

#### 1. 인덱스 종류
* **B-Tree 인덱스:** 가장 대중적이고 범용적인 녀석입니다. 대소 비교나 범위 검색에 최고입니다.
* **해시 인덱스:** 오직 단 하나의 값만 정확히 찾을 때(`id = 7`) $O(1)$ 이라는 광속의 속도를 내지만, 크다/작다 범위 검색은 전혀 불가능합니다.
* **비트맵 인덱스:** 컴퓨터의 0과 1 신호(비트)로 인덱스를 만듭니다. '성별(남/여)'이나 '거래상태(대기/완료)'처럼 나올 수 있는 값의 종류가 몇 안 될 때(낮은 카디널리티) 메모리를 아주 적게 차지하면서 대용량 데이터를 처리하는 분석용 시스템(OLAP)에 찰떡궁합입니다.

#### 2. 주요 스캔 방식
* **Unique Scan:** 딱 1건만 조회합니다. (PK로 단건 조회 시)
* **Range Scan:** 인덱스의 특정 범위만 훑습니다. (가장 효율적이고 일반적)
* **Full Scan:** 인덱스 페이지 처음부터 끝까지 다 읽습니다. 테이블 전체를 읽는 것보다는 양이 적어 이득이지만 고칠 점이 있는지 의심해봐야 합니다.
* **Covering Index (커버링 인덱스):** 
  최고의 꿀팁입니다. 만약 내가 `SELECT 이름 FROM 회원 WHERE 회원번호 = 10`을 날렸는데, 인덱스가 `{회원번호, 이름}`으로 구성되어 있다면? 데이터베이스는 인덱스 페이지만 읽고 실제 디스크에 들어있는 테이블 데이터를 굳이 들여다볼 필요도 없이 즉시 결과를 반환합니다. 집 밖으로 안 나가고 현관문 앞에서 택배 상자 내용물을 전달받은 격입니다.

> **인덱스를 무력화하는 대표적인 실수들**
> 아무리 인덱스를 잘 설계해도 쿼리를 다음처럼 짜면 인덱스를 타지 못하고 풀 스캔으로 빠집니다.
> - `WHERE UPPER(이름) = 'HONG'` : 컬럼명에 함수를 씌워 가공하면 인덱스를 못 씁니다.
> - `WHERE 나이 + 1 = 30` : 컬럼에 산술 연산을 가하면 멍청해집니다. `WHERE 나이 = 29`로 바꿉시다.
> - `WHERE 이름 LIKE '%동'` : 앞글자가 뭔지 안 알려주면('어쩌구동') 사전에서 단어를 찾을 수 없겠죠? (뒤에 붙는 와일드카드 `LIKE '홍%'` 는 정상 작동합니다)

---

### 6-3. 옵티마이저 — SQL의 두뇌

우리가 짠 SQL 문장이 데이터베이스에 전달되면, 이를 맞이하는 사령관이 있습니다. 바로 **옵티마이저(Optimizer)**입니다. 어떻게 쿼리를 실행해야 가장 기름값(비용)이 적게 들고 효율적일지 머리를 쥐어짜내는 내비게이션 엔진입니다.

* **RBO (Rule-Based, 규칙 기반):** "인덱스가 있으면 무조건 타고, 없으면 풀 스캔 해라"처럼 미리 정해진 낡은 우선순위 규칙에만 기계적으로 따르는 고전적인 방식입니다. (현대 DBMS에서는 퇴출되었습니다)
* **CBO (Cost-Based, 비용 기반):** "테이블에 지금 데이터가 100만 건 있고, 그중 인덱스 타는 데이터는 10건뿐이네. 통계자료를 보아하니 이게 더 빠르겠다!" 처럼 데이터의 분포 상태(통계 정보)를 면밀히 분석해 예상 소요 비용을 똑똑하게 계산해 길을 찾는 현대적인 방식입니다.

#### 옵티마이저에게 명령 내리기: 힌트(HINT)
내비게이션이 가끔 막히는 길로 가라고 고집을 부릴 때가 있죠? 그때 운전자가 강제로 운전대를 꺾어 지름길로 가듯이, 개발자가 SQL 쿼리 안에 특수 주석 형태로 명령을 적어 실행 경로를 통제할 수 있습니다.
* `SELECT /*+ INDEX(A 회원_인덱스) */ 이름 FROM 회원 A ...` : "옵티마이저야, 딴길로 새지 말고 무조건 이 인덱스 써라!"
* `/*+ USE_HASH(B) */` : "조인할 때 꼭 해시 조인 돌려라!"

### 6-4. 데이터베이스 튜닝 방법론

느려진 데이터베이스를 고치는 튜닝 작업은 마치 의사가 환자를 진단하고 수술하는 정교한 의료 행위와 같습니다.

#### 1. 3단계 튜닝 프로세스
```
[ 1단계: 보색 (조사) ] ── 느린 쿼리 로그(Slow Query Log) 수집 및 실행 계획(Execution Plan) 분석
          │
[ 2단계: 이행 (적용) ] ── 인덱스 신설, SQL 재작성, 파티셔닝 적용 등 알맞은 조치 취하기
          │
[ 3단계: 평가 ] ───────── 개선 전후 TPS(초당 처리량)와 응답 시간, CPU 사용량 비교 평가
```

#### 2. 어디부터 손댈까? 4계층 튜닝
흔히 컴퓨터가 느려지면 메모리를 추가하거나 CPU를 업그레이드할 생각부터 합니다. 하지만 그것은 가장 돈이 많이 들고 효과가 미미한 하책입니다. 튜닝은 반드시 아래 순서대로 접근해야 합니다.

1. **설계 튜닝 (1순위 - 효과 최고):** 잘못된 데이터 모델링 수정, 과도한 조인을 없애기 위한 반정규화나 테이블 쪼개기.
2. **Application 튜닝 (2순위):** 비효율적인 SQL 쿼리 튜닝, 바인드 변수 적용, 커버링 인덱스 구성.
3. **Server 튜닝 (3순위):** DBMS가 사용하는 메모리 버퍼(Buffer Pool) 크기 튜닝, 커넥션 풀 조율.
4. **System 튜닝 (4순위 - 최후의 수단):** SSD 교체, 서버 하드웨어 증설(Scale-up).

> 좋은 SQL 한 줄이 수억 원짜리 하드웨어 증설보다 훨씬 큰 성능 향상을 가져오는 경우가 많습니다.

## Part 7. 분산 데이터베이스와 확장 — 여러 서버로 나누기

### 7-1. 파티셔닝 — 대용량 테이블 쪼개기

하나의 서랍장에 10년 치 계약서 수백만 장을 다 때려 넣었다고 생각해 보세요. 필요한 문서 한 장을 찾으려면 서랍 전체를 온종일 헤집어야 합니다. 해결책은 연도별로 서랍 칸을 분리해 두는 것이지요. 

데이터베이스에서도 한 테이블에 수억 건의 데이터가 모였을 때, 사용자 눈에는 여전히 하나의 테이블처럼 보이지만 내부 저장 공간을 여러 구역(파티션)으로 나누어 관리하는 기술을 **파티셔닝(Partitioning)**이라고 합니다. 서버는 그대로 한 대이고, 디스크를 새로 사는 것도 아닙니다. 같은 서버 안에서 파일을 연도별·지역별로 분리해 두는 것입니다.

파티셔닝의 백미는 바로 **'파티션 프루닝(Partition Pruning)'**입니다.
```
  [ 주문 테이블 ] ── 연도별로 쪼개둔 방(파티션)들
   ├── [ 2023년 방 ]
   ├── [ 2024년 방 ]
   └── [ 2025년 방 ]

  쿼리: "WHERE 주문일자 BETWEEN '2024-01-01' AND '2024-12-31'"
  동작: 2023년, 2025년 방은 열어보지도 않고, 2024년 방만 쏙 들어갑니다! (나머지 66% 파일 접근 생략)
```

어떤 기준으로 방을 쪼갤지에 따라 종류가 다양합니다.
* **Range 파티셔닝:** 날짜나 순번처럼 값의 범위(`2024-01-01 ~ 2024-12-31`)로 나눕니다. 가장 흔히 쓰입니다.
* **List 파티셔닝:** 지역 코드(`[서울, 경기]`, `[부산, 경남]`)나 부서 등 명확히 분류되는 특정 값 목록으로 나눕니다.
* **Hash 파티셔닝:** 골고루 분배하기 위해 해시 함수가 내뱉는 값에 따라 임의의 방으로 고르게 보냅니다.
* **수평 vs 수직 파티셔닝:**
  * **수평(Horizontal):** 위 방식들처럼 테이블의 행(Row)들을 쪼개어 갖는 것입니다.
  * **수직(Vertical):** 자주 쓰는 이름/전화번호 열과, 거의 안 보는 파일 첨부 바이너리 열을 별도 파일로 세로로 쪼개어 저장하는 것입니다.

---

### 7-2. 샤딩, 리플리케이션, Query Offloading

테이블을 디스크 파일 수준에서 쪼개는 것만으로 감당이 안 될 만큼 사용자가 몰려들면, 이제 서버 컴퓨터 대수 자체를 늘리는 **물리적 확장(Scale-out)**을 고려해야 합니다.

#### 1. 샤딩 (Sharding) — 컴퓨터 자체를 여러 대로 쪼개자!
파티셔닝이 한 서버 안에서 방만 나누는 것이라면, **샤딩**은 데이터베이스 서버 컴퓨터 5대를 사서 1번 서버에는 A~E 성씨 회원, 2번 서버에는 F~J 성씨 회원을 나누어 집어넣는 극단적인 분산 전략입니다.
* **샤딩 키(Shard Key) 전략:** 데이터를 어떤 규칙으로 분산 서버에 나눌지 정하는 기준(키)이 매우 중요합니다. 해시 함수를 쓸지, 범위로 나눌지 결정합니다. 샤딩을 도입하면 서버 간 조인이 어려워지므로 샤딩 키 설계를 신중하게 해야 합니다.

#### 2. 리플리케이션 (Replication) & Query Offloading (업무 분담)
똑같은 데이터를 복사하여 복제본 서버 여러 대를 두는 기술입니다.
```
             [ Master DB ] ─── (오직 저장, 수정, 삭제만 담당!)
                   │
                   ▼ 복제 로그 전송 (Replication)
         ┌─────────┴─────────┐
    [ Slave DB 1 ]     [ Slave DB 2 ]  ─── (오직 읽기/조회 요청만 분산 처리!)
```
이를 **Query Offloading(조회 분기)**이라고 부릅니다. 데이터 변경은 든든한 대장(Master) 한 곳에서 통제하고, 트래픽의 90%를 차지하는 일반 사용자들의 상품 조회 요건은 조무래기 복제 서버(Slave)들이 나누어 처리하여 서버 부하를 효과적으로 덜어내는 현대 대규모 서비스의 필수 아키텍처입니다.
* **주의점:** 대장에서 복제 서버로 복사되는 찰나의 시간 지연(Replication Lag)이 발생하여, 글을 방금 썼는데 새로고침했을 때 잠깐 글이 안 보이는 현상이 생길 수 있습니다.

---

### 7-3. 분산 데이터베이스와 2PC

전 세계 여러 도시에 흩어져 있는 지사 서버들이 하나의 통합 데이터베이스처럼 매끄럽게 연결되어 보이기 위해 분산 DB는 몇 가지 신비로운 마법(**분산 투명성**)을 제공해야 합니다. 
예를 들어, 데이터가 어느 도시 컴퓨터에 저장되어 있는지 사용자는 몰라도 쿼리가 날아가고(위치 투명성), 테이블이 조각조각 분할되어 있어도 그냥 하나의 테이블로 인식되며(단편화 투명성), 일부 서버가 꺼져도 우회 작동하는(장애 투명성) 특징들이 있습니다.

하지만 분산 시스템의 가장 큰 난제는 "**여러 컴퓨터에 걸쳐진 작업을 어떻게 동시에 원자적으로 안전하게 커밋할 것인가?**"입니다. 이때 쓰이는 클래식한 방법이 **2단계 커밋(2PC, Two-Phase Commit)** 프로토콜입니다.

```
[Phase 1: 준비 단계 (Prepare)]
 대장(코디네이터): "얘들아, 1번 서버랑 2번 서버 너희 둘 다 커밋할 준비 되었니? 투표 시작!"
 참여자 1 & 2: "네! 준비 완료!" (YES 투표)

[Phase 2: 실행 단계 (Commit/Abort)]
 대장(코디네이터): "모두 찬성했으니, 최종 COMMIT을 진행해라!"
 참여자 1 & 2: 최종 반영 완료!
```

* **단점:** 만약 투표를 마치고 대장이 커밋 지시를 내리기 전에 갑자기 정전이 되어 죽어버리면, 참여 서버들은 대장의 연락이 올 때까지 락을 잡은 채 무한 대기 상태(Blocking)에 빠져 시스템 전체가 먹통이 되는 **SPOF(Single Point of Failure)** 취약점이 있습니다.
* **대안:** 현대 마이크로서비스 아키텍처(MSA)에서는 시스템 전체를 먹통으로 만드는 2PC 대신, 각 서비스가 자기 일만 먼저 처리하고 성공 이벤트를 쏘아 보낸 뒤 실패 시 역방향으로 취소 쿼리를 순차 수행해 복구하는 **Saga 패턴(보상 트랜잭션)**을 훨씬 더 많이 씁니다.

### 7-4. DB 서버 용량 산정

새로운 프로젝트를 오픈하기 전, "서버 장비 사양을 얼마짜리로 사야 할까요?" 라는 질문에 논리적으로 대답할 수 있어야 합니다.

#### 1. 피크 TPS 산정 공식
초당 데이터베이스가 감당해야 하는 처리 건수(TPS)의 최고치를 구하는 계산법입니다.

$$\text{피크 TPS} = \frac{\text{일일 총 트랜잭션 수} \times \text{피크타임 집중률}}{\text{피크 지속 시간 (초)}}$$

* **실전 예시:**
  * 하루에 총 1,000,000건의 거래가 발생하고, 그 중 30%가 퇴근 직후 1시간(3,600초) 동안 몰린다고 가정합시다.
  * $\text{피크 TPS} = \frac{1,000,000 \times 0.3}{3,600} \approx 83 \text{ TPS}$
  * 여기에 예측 불허의 상황을 대비해 안전 여유율 30%를 곱해 줍니다.
  * 최종 설계 TPS = $83 \times 1.3 \approx 108 \text{ TPS}$

#### 2. 용량 산정 절차
1. **업무량 분석:** 하루 거래 데이터 크기, 가입자 추이, 데이터가 연간 몇 %씩 증가할지 조사합니다.
2. **성능 목표 정의:** "결제 화면은 무조건 0.5초 이내에 떠야 한다" 같은 SLA 목표치를 정합니다.
3. **TPS 산정:** 위 공식을 활용해 목표 피크 TPS를 구합니다.
4. **자원 산정:** TPS를 받아주기 위해 필요한 CPU 코어 수, 전체 데이터를 다 캐싱해둘 메모리 크기(Buffer Pool), 디스크 성능(IOPS)을 스펙표를 대조해 산출합니다.

## Part 8. NoSQL과 현대 데이터베이스 — 관계형의 경계 넘기

### 8-1. RDBMS vs NoSQL vs NewSQL

40년 동안 왕좌를 지켜온 관계형 데이터베이스(RDBMS)도 인터넷의 대폭발과 함께 큰 벽에 부딪혔습니다. 스키마가 딱딱하게 고정되어 페이스북의 뉴스피드나 인스타그램의 복잡한 피드처럼 정형화되지 않은 데이터를 빠르게 저장하고 무제한 확장하기가 벅찼기 때문입니다. 이에 따라 대안 데이터베이스들이 탄생했습니다.

* **NoSQL (Not Only SQL):** 
  * "표(Table) 모양을 포기하자!" 스키마가 없고 데이터 구조가 유연하며, 컴퓨터를 100대, 1000대로 늘려나가는 수평적 확장(Scale-out)이 아주 손쉽습니다. 대신 강한 데이터 정합성(ACID)은 조금 타협합니다.
* **NewSQL:** 
  * "RDBMS의 정합성(ACID)과 NoSQL의 수평 확장성(Scale-out)을 둘 다 훔쳐오자!" 구글 스패너(Spanner)나 콕로치DB(CockroachDB) 같은 현대식 초고가/초정밀 글로벌 분산 데이터베이스 제품들이 이에 해당합니다.

NoSQL은 데이터를 보관하는 방식에 따라 크게 4가지 장난감 상자로 나뉩니다.
* **Key-Value DB:** 사전(Dictionary) 구조처럼 키 하나에 값 하나를 쌍으로 매핑합니다. (예: Redis - 빠르고 단순함)
* **Document DB:** JSON 형태의 문서를 통째로 저장합니다. (예: MongoDB - 개발자가 다루기 가장 직관적)
* **Column-Family DB:** 열 단위로 데이터를 모아 저장해 대규모 조회 분석이나 로그성 시계열 데이터를 모을 때 유리합니다. (예: Cassandra)
* **Graph DB:** 노드(사람)와 링크(관계)를 직접 포인터 선으로 연결해 네트워크 구조를 저장합니다. (예: Neo4j)

### 8-2. CAP 이론과 PACELC 이론

분산 데이터베이스 시스템을 설계할 때 마주치는 불변의 우주 법칙 같은 이론들입니다.

#### 1. CAP 이론 — 3마리 토끼를 다 잡을 수 없다
분산 네트워크에서 다음 세 가지를 동시에 만족할 수는 없습니다.
* **C (Consistency, 일관성):** 1번 서버에 글을 쓰면, 동시에 2번 서버를 읽어도 똑같은 최신 글이 보여야 합니다.
* **A (Availability, 가용성):** 일부 서버가 터져도 내 요청에 에러 없이 무조건 응답을 해줘야 합니다.
* **P (Partition Tolerance, 분단 허용):** 서버들 사이의 통신망이 끊겨도 작동해야 합니다.

현실적인 물리 환경에서 통신 장애(P)는 언제든 일어날 수 있으므로 결국 우리는 두 갈래 길에 서게 됩니다.
* **CP 선택:** 일관성 최고! 통신 끊기면 에러를 낼지언정 짝짝이 데이터를 보여주진 않겠다. (예: HBase, MongoDB)
* **AP 선택:** 가용성 최고! 조금 옛날 데이터일지라도 서비스가 안 뜨는 것보단 무조건 대답이라도 해주는 게 낫다. (예: Cassandra)

#### 2. PACELC 이론 — 정상 상태일 때도 손해가 있어요!
CAP 이론은 안타깝게도 '통신 장애(P)'가 났을 때만 다룹니다. 시스템은 평상시(Else)가 훨씬 많은데 말이죠. 그래서 평소 성능 관점(Latency)까지 포괄해 정리한 이론입니다.
```
만약 분단이 발생하면(P) -> A(가용성)와 C(일관성) 중에 무엇을 선택할래?
그렇지 않고 정상일 때는(E) -> L(낮은 지연 속도)과 C(일관성) 중에 무엇을 희생할래?
```
예를 들어, Cassandra는 **PA/EL** 형입니다. 평상시 복제 속도를 기다리지 않고 대충 바로 응답해줘서 지연율(L)을 극한으로 낮추고 가용성(A)을 추구하는 모델입니다. 반면 MongoDB는 **PA/EC** 로 정상 상황일 때 시간이 좀 걸려도 완벽한 일관성(C)을 맞춰주려고 노력하는 모범생 스타일입니다.

### 8-3. NoSQL 모델링 패턴

RDBMS 설계를 할 때는 묻지도 따지지도 않고 데이터의 중복을 쪼개는 **'정규화'**부터 해둔 뒤, 나중에 온갖 복잡한 쿼리로 조인해서 데이터를 뽑아 썼습니다. 

하지만 NoSQL은 정반대입니다. 조인 능력이 거의 없기 때문에, "**사용자가 화면에서 이 데이터를 어떻게 읽어갈 것인가? (조회 패턴)**"을 철저히 먼저 분석한 뒤, 그 화면 조회 쿼리 딱 하나에 최적화된 형태로 테이블을 뭉뚱그려 만듭니다. 

대표적인 모델링 패턴 3가지입니다.

#### 1. Embedding (내장형)
* 관련 데이터를 다른 서랍에 두지 않고 하나의 문서(JSON) 안에 자식 배열로 쑤셔 넣어 보관합니다.
* 예: 주문 정보 안에 `items: [{상품명: '노트북', 가격: 1500000}]`을 배열로 내장합니다.
* 장점: 조인할 것 없이 문서 한 장 쓱 읽으면 화면 출력이 한 번에 끝납니다.
* 적합: 자식 데이터 개수가 많지 않고 변하지 않을 때 유용합니다.

#### 2. Referencing (참조형)
* ID 값만 적어두고 필요할 때 간접적으로 찾아갑니다. 자식의 개수가 수천 개 이상으로 불어날 수 있을 때 씁니다.

#### 3. Bucketing (버킷팅)
* 사물인터넷(IoT) 센서 데이터나 로그 데이터처럼 매초 수만 건이 들어오는 데이터의 경우, 매번 한 줄씩 문서를 새로 쓰면 서버가 과부하에 걸립니다.
* 해결: "1시간짜리 상자(Bucket)"를 하나 만들어 두고, 1시간 동안 발생하는 데이터를 그 상자 안에 차곡차곡 리스트 형태로 모아 넣습니다.

### 8-4. 인메모리 데이터베이스 — Redis

아무리 빠른 SSD 카드를 사와도, 주 기억장치인 메모리(DRAM)의 물리적인 속도는 결코 따라갈 수 없습니다.

```
 디스크(HDD) 접근 속도: ── 대략 10ms (100분의 1초)
 디스크(SSD) 접근 속도: ── 대략 0.1ms (10,000분의 1초)
 메모리(DRAM) 접근 속도: ─ 대략 0.0001ms (1,000만분의 1초) 🚀
```

**Redis**는 모든 데이터를 디스크가 아닌 메모리에 통째로 올려두고 서비스하는 초고속 인메모리 Key-Value 데이터베이스입니다. 주로 웹 로그인 세션 보관이나 자주 조회되는 공지사항 캐싱에 쓰이지요.

메모리는 컴퓨터 전원이 꺼지면 내용이 연기처럼 사라지는 휘발성 장치입니다. 이를 보완하기 위해 Redis는 디스크에 백업을 남기는 영속성 옵션을 줍니다.
* **RDB (Snapshot):** 한 시간마다 한 번씩 통째로 사진을 찍어 스냅샷 파일로 디스크에 보관합니다. 빠르지만 스냅샷 찍기 직전에 꺼지면 그 사이 데이터는 잃게 됩니다.
* **AOF (Append Only File):** 데이터가 수정되는 모든 명령어(`SET, DEL` 등)를 로그 파일에 한 줄 한 줄 계속 이어서 기록해 둡니다. 조금 느리지만 유실이 거의 없습니다.

### 8-5. Graph DB — 관계를 저장하다

"김철수와 이영희는 친구 관계이고, 이영희는 박민수와 아는 사이이다. 이영희가 좋아하는 영화를 김철수에게 추천해 주려면?" 

이런 질문을 1,000만 명의 회원 정보가 들어있는 RDBMS에서 조인으로 해결하려고 하면, `회원 ⋈ 친구관계 ⋈ 영화선호도 ⋈ 회원` 4중 조인이 들어가면서 쿼리가 멈춰버리는 현상을 보게 됩니다. 조인 연산은 데이터가 많을수록 인덱스를 탐색하는 비용이 기하급수적으로 늘어나기 때문입니다.

이 문제를 우아하게 풀어낸 것이 **Graph DB**입니다.
이곳에서는 관계 자체가 실존하는 물리적 '포인터 선(Edge)'으로 연결되어 있어서, 인덱스 검색을 할 필요도 없이 그냥 철수 노드에서 뻗어 나간 영희 실선을 손가락으로 따라 짚어가면 끝납니다. 이를 **Index-Free Adjacency** 기법이라고 부르며, 관계를 한 단계 더 깊게 타고 들어갈 때 성능 저하가 거의 제로에 가깝습니다.

**Neo4j Cypher 쿼리 예시:**
```cypher
// 철수의 친구의 친구(2단계) 관계인 사람들의 이름 가져오기
MATCH (p:Person {name: "김철수"})-[:KNOWS*1..2]->(friend)
RETURN friend.name;
```
주로 페이스북의 친구 찾기, 은행 카드 결제 망의 이상한 순환 고리 탐지(금융 사기 방지), 개인 맞춤형 추천 시스템에 필수적으로 적용됩니다.

### 8-6. 벡터 데이터베이스 — AI의 두뇌 저장소

챗GPT를 비롯한 최신 생성형 AI 열풍과 함께 가장 뜨거운 주목을 받는 차세대 데이터베이스입니다. 

인간의 복잡한 말뭉치("오늘 날씨 참 우울하네")나 사진, 오디오를 인공지능 임베딩 모델에 통과시키면 수백~수천 차원의 기나긴 숫자 배열(벡터)로 변환됩니다. 이 고차원 벡터들의 수학적 유사도(코사인 유사도 등)를 계산하여 "**말은 조금 달라도 뜻이 가장 비슷한 데이터**"를 초고속으로 검색해 주는 저장소가 바로 **벡터 데이터베이스**입니다.

* **핵심 알고리즘:** 차원이 너무 높아서 일반적인 인덱스로는 길을 찾지 못하므로, 완벽한 정답 대신 가장 가까운 녀석들을 겉보기 형태로 뭉뚱그려 고속 검색하는 **ANN(Approximate Nearest Neighbor)** 기법을 활용합니다. (대표적으로 계층적 그래프를 이용하는 **HNSW** 알고리즘이 대세입니다)
* **RAG (검색 증강 생성):** 
  챗GPT에게 사내 대외비 매뉴얼 문서를 학습시키지 않고도 질문에 정확히 답하게 만드는 비결입니다.
  ```
  [질문: "올해 휴가 정책은?"] ──► [임베딩 변환] ──► [벡터 DB에서 의미가 유사한 매뉴얼 단락 3개 쏙 뽑기] 
                                                                  │
  [답변 완성] ◄── [뽑아낸 문서 3개와 질문을 같이 프롬프트로 챗GPT에게 보내기] ◄┘
  ```
  이를 통해 챗GPT가 그럴듯하게 거짓말을 지어내는 **할루시네이션(Hallucination, 환각)** 문제를 눈에 띄게 줄일 수 있습니다.

## Part 9. 데이터 품질과 거버넌스

### 9-1. 데이터 품질 진단 절차와 기법

"쓰레기가 들어가면 쓰레기가 나온다 (Garbage In, Garbage Out)"는 데이터 분석 세계의 불변의 진리입니다. 데이터의 품질이 엉망이면 아무리 뛰어난 AI 모델이나 리포팅 툴을 돌려도 소용이 없지요. 

품질을 진단할 때는 다음과 같은 **6가지 건강 검진 차원**을 들여다봅니다.

* **완전성 (Completeness):** 꼭 입력되어야 할 필수 칸이 비어 있진 않은가? (예: 고객 정보의 전화번호 누락율)
* **유일성 (Uniqueness):** 중복된 데이터가 나돌아 다니진 않는가? (예: 똑같은 회원이 두 번 가입됨)
* **유효성 (Validity):** 올바른 포맷을 지키고 있는가? (예: 날짜 칸에 `2026-02-31`이 들어가 있음)
* **일관성 (Consistency):** 이 시스템과 저 시스템의 정보가 일치하는가? (예: 결제 시스템의 매출액과 세무 시스템의 매출액 불일치)
* **정확성 (Accuracy):** 현실 세계의 사실을 있는 그대로 반영하고 있는가? (예: 물건이 이미 배송 완료되었는데 앱에서는 여전히 배송중으로 뜸)
* **접근성 (Accessibility):** 필요할 때 바로 꺼내 쓸 수 있게 로딩이 잘 되는가?

검진을 할 때는 우선 데이터의 빈값 비율, 최대/최솟값 분포 상태를 수학적으로 측정하는 **데이터 프로파일링** 작업을 수행하고, 규칙에 어긋난 나쁜 오류 값들을 필터링해 원인을 찾아 고치는 4단계 정화 작업을 반복 수행합니다.

### 9-2. DQ 인증과 ISO 8000

기업이나 기관이 보유한 데이터가 대외적으로 얼마나 신뢰할 수 있는지 제3의 기관이 검증해 주는 등급제도입니다. 우리나라에서는 한국데이터산업진흥원(K-DATA) 주관으로 **DQ(Data Quality) 인증**을 주는데, 데이터 관리 체계와 데이터 알맹이 값의 무결성을 심사해 실버/골드/플래티넘 등급을 나누어 줍니다.

국제적으로는 국제표준화기구(ISO)에서 정한 **ISO 8000** 표준 시리즈가 데이터 품질의 헌법 역할을 하고 있습니다.

* **ISO 8000-61:** 데이터 품질 관리를 하는 표준 프로세스 프레임워크를 규정합니다.
* **마스터 데이터 관리 (MDM):** 
  회사 내에서 가장 핵심이 되는 고유 마스터 정보(고객 목록, 상품 목록 등)가 부서별로 제각각 놀지 않도록 전사 관점에서 딱 하나로 통합한 단일 신뢰 원본을 구축하는 일입니다. 이를 전문 용어로 **골든 레코드(Golden Record)**라 부르며, 이를 전담 관리하는 전문가를 **데이터 스튜어드**라고 칭합니다.

---

### 9-3. 데이터 마이그레이션 품질 검증

새로운 강남 아파트로 이사 가기 전, 이삿짐 센터 트럭에 짐을 싣고 새집에 도착해서 잃어버린 가구나 깨진 그릇이 없는지 전수 검사를 해야 합니다. 옛날 시스템(Legacy)에서 신규 시스템(Next Generation)으로 데이터를 대량으로 통째로 옮기는 것을 **데이터 마이그레이션**이라고 하며, 이때의 검증 방식은 철저해야 합니다.

> 💡 **헷갈리지 마세요! 무결성 vs 정합성**
> - **무결성(Integrity):** 데이터 자체에 흠집이 없는가? (예: 주민번호 중복이나 외래키 끊김 등의 DB 기본 제약 조건 준수)
> - **정합성(Consistency):** 앞뒤 아귀가 맞물려 돌아가는가? (예: 주문 내역에 기록된 개별 가격의 합산 결과와 영수증 테이블의 총금액이 일치하는지 비즈니스적 논리 모순 여부 확인)

**이사 후 짐 확인 방법 5가지:**
1. **레코드 수 검증:** 이사 전 박스가 1,000개였는데, 이사 후에도 딱 1,000개인지 전체 행 개수(`COUNT`)를 맞춰 봅니다. (누락 확인)
2. **해시 비교:** 대량의 텍스트 데이터를 문자열 해시 함수(SHA-256 등)에 넣어 결과 해시코드 값을 대조하여, 데이터 속의 콤마 하나 글자 하나도 안 틀리고 똑같이 이사 갔는지 검증합니다.
3. **샘플링 검증:** 전체를 다 뒤지기 힘들 때 무작위로 일부 박스를 뜯어 물건 상태를 봅니다.
4. **집계 검증:** 원본 금액 합계와 타겟 금액 합계를 계산해 1원도 안 맞는지 맞춰 봅니다.
5. **참조 무결성 검증:** 이사 간 자식 데이터 중에 짝 잃은 낙동강 오리알 데이터가 없는지 외래키 연결 상태를 최종 스캔합니다.

---

### 9-4. 생성형 AI 데이터 품질 관리

기존의 정형 데이터와 달리, 거대 언어 모델(LLM) 학습에 들어가는 텍스트나 말뭉치는 또 다른 관점의 품질 기준이 개입됩니다. 나쁜 품질의 학습 데이터를 먹고 자란 AI는 심각한 사회적 물의를 빚을 수 있기 때문입니다.

#### 생성형 AI의 4대 데이터 품질 문제
* **편향 (Bias):** 데이터 편향으로 인해 인종, 성별, 종교적 차별 발언을 쏟아내는 현상.
* **독성 (Toxicity):** 욕설, 비하, 폭력적 언어가 학습 자료에 포함되어 인공지능이 거친 표현을 배출하는 현상.
* **환각 (Hallucination):** 팩트가 아닌 가짜 상식을 그럴듯한 말투로 그 자리에서 창조해 내는 현상.
* **Model Collapse (모델 붕괴):** AI가 지어낸 거짓 데이터를 바탕으로 다음 세대 AI를 재학습시키는 일을 반복하다가, 점차 AI의 지능과 성능이 바보처럼 쇠퇴하고 무너지는 현상.

이에 따라 정부 가이드라인(2025.2 개정 v2.0)에서는 AI용 말뭉치 데이터에 대해 **최신성**(옛날 상식 검증), **다양성**(편향 제거를 위한 골고루 섞기), **공정성** 지표 등을 포함한 6대 품질 차원을 제정해 관리하고 있습니다.

---

### 9-5. 공공데이터 품질 관리

국가 기관이나 지자체가 개방하는 공공데이터는 국민들의 창업과 서비스 개발의 마중물입니다. 따라서 행정안전부의 **'공공데이터 품질관리 매뉴얼 V2.0'** 표준 지침에 따라 공무원들과 담당 공학자들이 엄격한 라이프사이클 관리를 적용하고 있습니다.

```
[ 생산 단계 ] ── 데이터 표준 규칙 수립, 메타데이터 정보망 등록
     │
[ 관리 단계 ] ── 매월 정기 오류 진단, 쓰레기 값 정제 및 최신 업데이트 갱신
     │
[ 활용 단계 ] ── 포털 개방 전 최종 품질 검증 및 오픈 후 대국민 피드백 수집 및 모니터링
```

오류 데이터가 발견되면 공공데이터 포털의 지침에 따라 오류 원인을 분석하고 데이터 정제 가이드에 맞추어 보정 조치 보고서를 발행해야 합니다.

### 9-6. 데이터 가치평가

"데이터가 석유다!" 라는 말을 많이 들으셨을 겁니다. 2022년 시행된 **데이터 기본법** 덕분에 이제 데이터는 특허나 부동산처럼 기업의 합법적인 공식 재산권 자산으로 당당히 대접받게 되었습니다. 그렇다면 이 무형의 데이터 뭉치가 돈으로 치면 얼마짜리일까요? 가치를 계산하는 3가지 접근법이 있습니다.

* **비용 접근법 (Cost Approach):** 
  "이 데이터를 오늘 날짜에 똑같이 다시 수집하고 서버 인프라 깔아 구축하려면 원가가 얼마 들까?" 하고 개발 비용 관점에서 산정합니다.
* **시장 접근법 (Market Approach):** 
  "비슷한 형태의 고객 행동 분석 데이터가 데이터 거래소(KDX)에서 최근에 5,000만 원에 거래되었네!" 처럼 유사 거래 사례를 비교해 가격표를 매깁니다. (다만 데이터 거래 시장이 아직 초기라 비교군 찾기가 어렵다는 단점이 있습니다)
* **수익 접근법 (Income Approach - DCF 공식):** 
  가장 프로페셔널한 방식입니다. "**이 고객 예측 데이터를 마케팅에 활용함으로써 향후 3년 동안 추가로 벌어들일 순이익의 현재 가치가 총 얼마인가?**"를 복리 이자율을 거꾸로 계산해 평가하는 방법론입니다.

$$V = \sum_{t=1}^{n} \frac{FCF_t}{(1+r)^t}$$

* $FCF_t$: $t$년 차에 데이터로 벌어들일 순 현금 흐름
* $r$: 할인율 (위험 부담 리스크를 반영한 금리)
* $n$: 데이터의 기대 수명 (트렌드 변화 주기)

## Part 10. 공간 데이터베이스 — 지도 위의 데이터

### 10-1. 공간 데이터베이스 개념

우리가 사용하는 배달 앱의 "내 주변 500m 이내 치킨집 찾기", 카카오맵의 "목적지까지 최단 경로 안내", 자율주행 차량의 장애물 거리 파악 등의 뒤편에는 위도/경도 좌표와 면적 정보를 처리하는 특수한 **공간 데이터베이스(Spatial DB)**가 숨어 일하고 있습니다.

* **공간 객체 유형 (국제 OGC 표준):**
  * **Point:** 하나의 점 좌표 (예: 치킨집 위/경도 위치)
  * **LineString:** 점들을 이은 선 (예: 도로 노선, 한강 물줄기)
  * **Polygon:** 닫힌 면 영역 (예: 행정구역 경계, 아파트 단지 면적)
  * **MultiPoint / MultiPolygon:** 여러 점이나 면들의 묶음 (예: 대한민국 영토 - 울릉도, 독도, 제주도 섬들을 다 묶은 면적)
* **DE-9IM 모델:**
  공간 데이터베이스는 두 도형의 안쪽(Interior), 경계선(Boundary), 바깥쪽(Exterior)이 서로 어떻게 교차하는지를 9칸짜리 수학 매트릭스로 계산해 내어 공간적 관계를 밝혀냅니다.
  이를 통해 `ST_Contains(A, B)` (A가 B를 완벽히 품고 있는지), `ST_Touches(A, B)` (두 땅이 경계선만 맞닿아 있는지), `ST_Disjoint(A, B)` (전혀 상관없이 서로 떨어져 있는지) 등을 정확하게 판별해 낼 수 있습니다.

---

### 10-2. 공간 연산자

공간 데이터는 연산량이 깡패 수준으로 많습니다. 복잡하게 굽이치는 서울 강남구 행정구역 폴리곤(수천 개의 꼭짓점) 안에 내 배달 위치 위경도 좌표 점이 들어가 있는지 컴퓨터가 정밀 수학 기하 계산을 매번 하려면 CPU가 타버릴 것입니다. 

이 고통을 피하고자 공간 DB는 영리하게 **2단계 필터링 구조**를 씁니다.

```
[ 1단계: MBR 필터링 (가볍고 고속) ]
  복잡한 강남구 영토를 감싸는 가장 단순한 직사각형(MBR, Minimum Bounding Rectangle)을 그립니다.
  내 위치 좌표가 일단 이 사각형 안에 들어왔는지 단순 비교식으로 걸러냅니다. (90% 쓰레기 데이터 광속 탈락)
         │
[ 2단계: 정밀 필터링 (무겁고 정밀) ]
  1단계를 통과한 유력 후보 데이터만 모아서, 실제 굴곡진 다각형 선 내부와 점의 충돌 여부를 
  기하학 알고리즘으로 엄밀하게 계산해 정답을 확정 짓습니다.
```

주요 기하학 공간 함수들입니다.
* `ST_Distance(점A, 점B)`: 두 좌표 사이의 실제 거리를 구합니다.
* `ST_Buffer(도형, 반경)`: 특정 역 주변 500m 영역에 해당하는 둥근 폴리곤 빵을 만들어 냅니다.
* `ST_Transform(도형, 3857)`: 구글 지도 좌표계에서 네이버 지도 좌표계로 데이터를 변환해 줍니다.

---

### 10-3. 공간 인덱스 — R-Tree 계열

공간 데이터는 1차원 숫자가 아니어서 일반 B-Tree 인덱스에 넣고 줄을 세울 수가 없습니다. 위도와 경도라는 2차원 이상의 면적을 가지고 있기 때문이지요. 

대안으로 탄생한 공간용 인덱스가 바로 **R-Tree (Rectangle Tree)**입니다.
도형들을 사각형 상자(MBR)로 감싼 다음, 그 상자들을 더 큰 상자로 감싸고, 또 더 큰 상자로 감싸 올려 상자들의 나무(Tree) 계층 구조를 만듭니다.

```
 ┌──────────────────────────────────────────────┐
 │ [루트 상자 R1]                               │
 │   ┌──────────────────┐    ┌──────────────┐   │
 │   │ [자식 상자 R2]    │    │ [자식 R3]     │   │
 │   │   ┌───┐   ┌───┐  │    │   ┌───┐      │   │
 │   │   │도형│   │도형│  │    │   │도형│      │   │
 │   │   └───┘   └───┘  │    │   └───┘      │   │
 │   └──────────────────┘    └──────────────┘   │
 └──────────────────────────────────────────────┘
```

R-Tree 도 발전하면서 계열사들이 나뉩니다.
* **R-Tree (1984):** 초기 모델입니다. 상자들끼리 겹치는 부분(Overlap)이 많아서 검색 시 여러 상자를 중복 스캔해야 해 성능이 다소 떨어졌습니다.
* **R+-Tree:** 상자가 겹치는 영역을 강제로 찢어버려 겹침을 완전 제거했습니다. 대신 경계에 걸친 한 도형이 양쪽 상자에 중복 저장되어 저장 공간을 더 씁니다.
* **R*-Tree:** 현대 표준입니다. 데이터를 새로 넣을 때, 그냥 자리가 남는 상자에 쑤셔 넣지 않고 상자들의 면적 변화와 겹치는 면적이 동시 최소화되도록 강제로 기존 상자를 허물고 재배치하는 **Forced Reinsert(강제 재삽입)** 알고리즘을 사용해 가장 똑똑한 인덱스 밸런스를 유지합니다.

---

### 10-4. 다차원 색인 구조

B-Tree가 1차원(선), R-Tree가 2차원(면)을 주로 다룬다면, 그 이상의 공간이나 3차원 입체를 더 빠르게 찾기 위한 기법들도 존재합니다.

* **Grid File (격자 파일):** 
  도화지에 격자 모눈종이 선을 그어 나누듯 공간을 칼같이 나눈 뒤 주소를 찾는 방식입니다. 속도가 매우 균일하지만 특정 동네에 데이터가 쏠리면 빈 모눈종이 방들이 메모리를 낭비한다는 흠이 있습니다.
* **k-d Tree (K차원 트리):** 
  데이터가 들어올 때마다 X축으로 반 쪼개고, 그 자식은 Y축으로 반 쪼개고, 그 자식은 다시 Z축으로 쪼개며 차례대로 번갈아가며 차원을 이진 분할하는 구조입니다. 머신러닝의 다차원 거리 기반 추천이나 KNN 탐색에 우수합니다.

#### 주의: 차원의 저주 (Curse of Dimensionality)
데이터의 차원(특징 수)이 100차원, 1000차원으로 불어나면 전통적인 기하 공간 인덱스(R-Tree, k-d Tree 등)는 무력화됩니다. 차원이 커질수록 공간 속의 모든 데이터 간 거리가 거의 동일해지면서, 사각형 상자로 묶는 연산 자체가 무의미해지기 때문입니다. 

그래서 인공지능 분야의 고차원 벡터 데이터베이스들은 전통 트리 인덱스 대신 앞서 언급한 **HNSW(계층형 그래프 탐색)**나 **IVF(군집 분류형 탐색)** 같은 지름길 찾기(Approximate) 전용 알고리즘을 절대적으로 사용한답니다.

## 최종 핵심 키워드 한 줄 요약

데이터베이스 시험이나 기술 면접을 앞두고 있다면 아래 키워드를 중심으로 정리해두면 좋습니다.

* **데이터 모델링:** 건축 과정에 빗대어 `개념` -> `논리` -> `물리` 3단계를 외우세요.
* **정규화:** 이상현상(삽입/삭제/수정 오류)을 없애려 테이블을 쪼개는 마법입니다.
* **함수종속성:** 원인과 결과입니다. 결정자($X$)가 종속자($Y$)를 찍어 결정합니다.
* **트랜잭션:** 은행 이체처럼 나눌 수 없는 일의 단위입니다. 약속은 **ACID** 입니다.
* **동시성 제어:** 여러 명이 쓸 때 데이터가 꼬이지 않게 문을 거는 `Lock`과 복사본을 보여주는 `MVCC`가 대표적입니다.
* **회복:** 사고 시 되감는 `UNDO`와 다시 고쳐 쓰는 `REDO`를 로그 기반으로 수행합니다.
* **인덱스:** 책 맨 뒤의 찾아보기 카드입니다. 핵심 구조는 균형 잡힌 **B-Tree** 입니다.
* **옵티마이저:** 비용을 따져서 가장 빠른 길로 안내하는 내비게이션 두뇌입니다.
* **파티셔닝 & 샤딩:** 데이터가 커질 때 서랍을 나누거나(파티셔닝) 아예 컴퓨터를 쪼갭니다(샤딩).
* **NoSQL:** 관계형을 포기한 대신 대량의 유연한 데이터와 무제한 수평 확장을 얻었습니다.
* **공간 & 벡터 DB:** 지도상의 위치를 찾는 R-Tree 기반 기술과 AI를 돕는 의미 유사도 탐색 기술입니다.
