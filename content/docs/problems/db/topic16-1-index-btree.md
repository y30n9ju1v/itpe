## 해설

> **대용량 데이터를 O(log n) 탐색으로 처리하는 트리 기반 자료구조**  
> B-Tree 인덱스의 루트→브랜치→리프 탐색 구조, 스캔 방식 분류, 인덱스 튜닝 원칙

인덱스는 데이터베이스 성능 튜닝에서 가장 즉효성이 높은 수단입니다. 책의 색인처럼 원하는 데이터의 위치를 미리 기록해 두는 구조로, 수억 건 이상의 대용량 테이블에서도 B-Tree 인덱스(Balanced Tree Index, 루트→브랜치→리프 노드로 이어지는 균형 트리 구조)는 3~4회의 I/O로 원하는 레코드를 찾아냅니다. 반면, 잘못 설계된 인덱스는 오히려 쓰기 성능을 저하시키고 옵티마이저를 혼란에 빠뜨립니다.

### 1. 출제 배경 및 의도

인덱스는 DB 성능의 핵심 요소로서, 실무에서 가장 자주 튜닝 대상이 되는 항목입니다. 출제 의도는 B-Tree의 자료구조적 원리를 이해하고, 단순 인덱스 사용을 넘어 스캔 방식 선택과 설계 원칙까지 종합적으로 설명할 수 있는지를 검증하는 것입니다. 가/나/다 세 항목 구조에 맞춰 각각 개념, 스캔 방식, 튜닝 방안을 균형 있게 서술해야 합니다.

### 2. 개념의 본질과 작동 메커니즘

인덱스란 테이블의 특정 컬럼값과 해당 레코드의 물리적 위치(ROWID)를 매핑하여 저장한 별도의 자료구조로, 풀 테이블 스캔 없이 원하는 데이터를 빠르게 접근할 수 있게 합니다.

- **Input:** SQL 쿼리의 WHERE 절 조건 컬럼
- **Mechanism:** B-Tree 루트→브랜치→리프 노드 순차 탐색 → ROWID로 테이블 랜덤 접근
- **Output:** 조건에 맞는 레코드의 빠른 반환 (O(log n) 복잡도)

#### 가. B-Tree 인덱스 구조 및 탐색 과정

```text
[B-Tree 인덱스 구조]

                   ┌─────────────┐
                   │  Root Node  │  ← 항상 메모리에 캐시
                   │  [50 | 100] │
                   └──┬──────┬──┘
                      │      │
            ┌─────────┘      └─────────┐
            ▼                          ▼
     ┌─────────────┐           ┌─────────────┐
     │ Branch Node │           │ Branch Node │
     │ [20 | 35]   │           │ [70 | 85]   │
     └──┬───────┬──┘           └──┬───────┬──┘
        │       │                 │       │
        ▼       ▼                 ▼       ▼
   ┌────────┐ ┌────────┐   ┌────────┐ ┌────────┐
   │ Leaf   │ │ Leaf   │   │ Leaf   │ │ Leaf   │
   │10→RID1 │ │20→RID2 │   │70→RID8 │ │85→RID9 │
   │12→RID3 │→│25→RID4 │→  │72→RID10│→│90→RID11│
   └────────┘ └────────┘   └────────┘ └────────┘
     ↑ 리프 노드는 이중 연결 리스트로 연결(Range Scan 지원)

  탐색 예: WHERE 컬럼 = 25
  1. Root [50|100]: 25 < 50 → 왼쪽 Branch
  2. Branch [20|35]: 20 ≤ 25 < 35 → 두 번째 리프
  3. Leaf: 25 → RID4 (ROWID) → 테이블 블록 랜덤 접근
  I/O 횟수: 트리 높이(보통 3~4회) + 테이블 접근 1회
```

**B-Tree의 핵심 특성**
- 모든 리프 노드는 동일 깊이(균형 트리, Balanced Tree)
- 리프 노드는 정렬된 이중 연결 리스트 → 범위 스캔 효율
- 브랜치/루트 노드는 분기 키(Separator Key)만 저장
- 데이터 삽입/삭제 시 자동 재균형(Split/Merge)

#### 나. 인덱스 스캔 방식

| 스캔 방식 | 동작 원리 | 적합 조건 | 효율 |
|----------|----------|----------|------|
| **Index Unique Scan** | 루트→리프 1건 탐색 후 즉시 종료 | Unique 조건, `=` 연산 | 최고 |
| **Index Range Scan** | 시작 리프 탐색 후 연결 리스트 순회 | `BETWEEN`, `<`, `>`, `LIKE 'ABC%'` | 높음 |
| **Index Full Scan** | 리프 노드 전체 순회 | ORDER BY, 인덱스 컬럼만 SELECT | 중간 |
| **Index Fast Full Scan** | 인덱스 블록 멀티블록 I/O | COUNT(*), 인덱스 컬럼 집계 | 중간 |
| **Index Skip Scan** | 복합 인덱스 선두 컬럼 미사용 시 내부 분기 | 선두 컬럼 NDV 낮을 때 | 낮음~중간 |
| **Full Table Scan** | 테이블 전체 블록 스캔 | 대부분의 행 접근, 인덱스 없음 | 조건부 |

```text
[Index Range Scan vs Full Table Scan 선택 기준]

  선택률(Selectivity) = 결과 건수 / 전체 건수

  0%         5~10%         15~20%        100%
  ┣━━━━━━━━━━━┫━━━━━━━━━━━━━┫━━━━━━━━━━━━━┫
  Index Range Scan      ◀▶     Full Table Scan
  (Random I/O 적음)   전환점   (Sequential I/O 효율)

  → 선택률이 낮을수록(소수 건 조회) Index Range Scan 유리
  → 선택률이 높을수록(다수 건 조회) Full Table Scan이 오히려 빠름
```

#### 다. 인덱스 튜닝 방안

| 튜닝 영역 | 문제 상황 | 해결 방안 |
|---------|---------|---------|
| **인덱스 설계** | 자주 검색하는 컬럼에 인덱스 없음 | WHERE/JOIN/ORDER BY 컬럼에 인덱스 생성 |
| **복합 인덱스 컬럼 순서** | 선택률 낮은 컬럼이 후순위 | 선택률 높은 컬럼을 앞에 배치 |
| **인덱스 손상 방지** | 인덱스 컬럼에 함수/형변환 적용 | 함수 기반 인덱스(FBI) 또는 쿼리 변환 |
| **카디널리티** | 성별 같은 NDV 낮은 컬럼 단독 인덱스 | Bitmap Index 또는 복합 인덱스 구성 |
| **인덱스 과다** | 과도한 인덱스로 DML 성능 저하 | 불필요 인덱스 제거, 사용 빈도 모니터링 |
| **통계 정보 갱신** | 오래된 통계로 옵티마이저 오판 | ANALYZE/DBMS_STATS 주기적 수행 |

### 3. 핵심 키워드 (Must-Have)

- **구조 키워드:** Root Node, Branch Node, Leaf Node, ROWID, 이중 연결 리스트, Balanced Tree
- **탐색 키워드:** Index Range Scan, Index Unique Scan, Full Table Scan, 선택률(Selectivity)
- **설계 키워드:** 카디널리티(Cardinality), NDV(Number of Distinct Values), 복합 인덱스, 선두 컬럼
- **튜닝 키워드:** 함수 기반 인덱스(FBI), 커버링 인덱스(Covering Index), Index Merge, 통계 정보

### 4. 맥락과 비교

| 비교 항목 | B-Tree 인덱스 | Hash 인덱스 | Bitmap 인덱스 |
|---------|------------|-----------|------------|
| **탐색 복잡도** | O(log n) | O(1) | O(1) (비트 연산) |
| **범위 스캔** | 지원 | 미지원 | 제한적 |
| **카디널리티** | 고/저 모두 | 고 카디널리티 | 저 카디널리티 |
| **DML 성능** | 중간 | 높음 | 낮음(락 문제) |
| **사용 환경** | OLTP 범용 | In-Memory, Hash Join | DW, 분석계 |

### 5. 실무 제언

**인덱스 설계의 3대 원칙 준수**

- **챌린지:** 개발 단계에서 성능 테스트 없이 "필요할 것 같은" 컬럼에 무분별하게 인덱스를 생성하면, INSERT/UPDATE/DELETE 시마다 모든 인덱스를 유지해야 하므로 쓰기 성능이 급격히 저하됩니다. 테이블당 인덱스 20개 이상은 쓰기 성능을 심각하게 위협합니다.
- **제언:** ① 조회 빈도와 선택률을 기반으로 필요한 인덱스만 생성, ② 복합 인덱스로 여러 단일 인덱스를 통합, ③ 분기별 사용 통계 기반 미사용 인덱스 정리(Oracle: V$INDEX_USAGE, MySQL: sys.schema_unused_indexes)를 실시합니다.
