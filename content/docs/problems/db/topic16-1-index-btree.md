## 해설

### 출제 배경 및 의도

인덱스는 데이터베이스 성능 튜닝에서 가장 즉효성이 높은 수단이다. 수억 건 이상의 대용량 테이블에서도 B-Tree 인덱스는 3~4회의 I/O로 원하는 레코드를 찾아낸다. 반면, 잘못 설계된 인덱스는 쓰기 성능을 저하시키고 옵티마이저를 혼란에 빠뜨린다.

출제 의도는 B-Tree의 자료구조적 원리를 이해하고, 단순 인덱스 사용을 넘어 스캔 방식 선택과 설계 원칙까지 종합적으로 설명할 수 있는지를 검증하는 것이다. 가/나/다 세 항목 구조에 맞춰 각각 개념, 스캔 방식, 튜닝 방안을 균형 있게 서술해야 한다.

### 1. 데이터베이스 인덱스와 B-Tree 탐색, 개요

정 의  • 테이블의 특정 컬럼값과 레코드의 물리적 위치(ROWID)를 매핑하여 저장한 별도의 자료구조
       - B-Tree(Balanced Tree) 구조로 루트→브랜치→리프 노드를 탐색하여 O(log n) 복잡도 달성

```
[B-Tree 인덱스 구조 및 탐색 흐름]

                   ┌─────────────┐
                   │  Root Node  │  ← 항상 메모리에 캐시
                   │  [50 | 100] │
                   └──┬──────┬──┘
                      │      │
            ┌─────────┘      └─────────┐
            ▼                          ▼
     ┌─────────────┐           ┌─────────────┐
     │ Branch Node │           │ Branch Node │
     │ [20 | 35]   │           │ [70 | 85]   │
     └──┬───────┬──┘           └──┬───────┬──┘
        │       │                 │       │
        ▼       ▼                 ▼       ▼
   ┌────────┐ ┌────────┐   ┌────────┐ ┌────────┐
   │ Leaf   │ │ Leaf   │   │ Leaf   │ │ Leaf   │
   │10→RID1 │ │20→RID2 │   │70→RID8 │ │85→RID9 │
   │12→RID3 │→│25→RID4 │→  │72→RID10│→│90→RID11│
   └────────┘ └────────┘   └────────┘ └────────┘
     ↑ 리프 노드는 이중 연결 리스트로 연결 (Range Scan 지원)

  탐색 예: WHERE 컬럼 = 25
  1. Root [50|100]: 25 < 50 → 왼쪽 Branch
  2. Branch [20|35]: 20 ≤ 25 < 35 → 두 번째 리프
  3. Leaf: 25 → RID4 (ROWID) → 테이블 블록 랜덤 접근
  I/O 횟수: 트리 높이(보통 3~4회) + 테이블 접근 1회
```

- 모든 리프 노드는 동일 깊이(균형 트리)를 유지하며, 데이터 삽입/삭제 시 자동 재균형(Split/Merge)한다.

### 2. 가. B-Tree 인덱스 특성, 나. 스캔 방식, 다. 튜닝 방안

1) 가. B-Tree 인덱스 구성 요소

| 구분 | 항목 | 설명 |
|------|------|------|
| 루트 노드 | Root Node | 트리의 최상위, 항상 메모리(버퍼 캐시)에 캐시 유지 |
| 브랜치 노드 | Branch Node | 분기 키(Separator Key)만 저장, 리프 노드로 탐색 경로 제공 |
| 리프 노드 | Leaf Node | 키값+ROWID 저장, 정렬된 이중 연결 리스트로 연결 |
| 균형 유지 | Split/Merge | 데이터 삽입/삭제 시 자동 재균형, 항상 동일 깊이 보장 |
| 범위 스캔 | 연결 리스트 | 리프 노드 간 이중 연결 리스트로 Range Scan 고효율 |

- B-Tree 인덱스는 OLTP 환경의 범용 인덱스로 등치·범위 검색 모두에 효율적이다.

2) 나. 인덱스 스캔 방식 분류

| 구분 | 스캔 방식 | 동작 원리 | 적합 조건 | 효율 |
|------|---------|---------|---------|------|
| ① | Index Unique Scan | 루트→리프 1건 탐색 후 즉시 종료 | Unique 조건, = 연산 | 최고 |
| ② | Index Range Scan | 시작 리프 탐색 후 연결 리스트 순회 | BETWEEN, <, >, LIKE 'ABC%' | 높음 |
| ③ | Index Full Scan | 리프 노드 전체 순회(정렬 보장) | ORDER BY, 인덱스 컬럼만 SELECT | 중간 |
| ④ | Index Fast Full Scan | 인덱스 블록 멀티블록 I/O(정렬 미보장) | COUNT(*), 인덱스 컬럼 집계 | 중간 |
| ⑤ | Index Skip Scan | 선두 컬럼 미사용 시 고유값 기준 내부 분기 | 선두 컬럼 NDV 낮을 때 | 낮음~중간 |
| ⑥ | Full Table Scan | 테이블 전체 블록 순차 스캔 | 대부분의 행 접근, 인덱스 없음 | 조건부 |

```
[스캔 방식 선택 기준: 선택률(Selectivity)]

  선택률 = 결과 건수 / 전체 건수

  0%         5~10%         15~20%        100%
  ┣━━━━━━━━━━━┫━━━━━━━━━━━━━┫━━━━━━━━━━━━━┫
  Index Range Scan      ◀▶     Full Table Scan
  (Random I/O 적음)   전환점   (Sequential I/O 효율)
```

- 선택률이 낮을수록 Index Range Scan, 높을수록 Full Table Scan이 유리하다.

### 3. 다. 인덱스 튜닝 방안 및 인덱스 유형 비교

| 튜닝 영역 | 문제 상황 | 해결 방안 |
|---------|---------|---------|
| 인덱스 설계 | 자주 검색하는 컬럼에 인덱스 없음 | WHERE/JOIN/ORDER BY 컬럼에 인덱스 생성 |
| 복합 인덱스 컬럼 순서 | 선택률 낮은 컬럼이 후순위 | 선택률 높은(NDV 큰) 컬럼을 앞에 배치 |
| 인덱스 손상 방지 | 인덱스 컬럼에 함수/형변환 적용 | 함수 기반 인덱스(FBI) 또는 쿼리 변환 |
| 카디널리티 | 성별 같은 NDV 낮은 컬럼 단독 인덱스 | Bitmap Index 또는 복합 인덱스 구성 |
| 인덱스 과다 | 과도한 인덱스로 DML 성능 저하 | 불필요 인덱스 제거, 사용 빈도 모니터링 |
| 통계 정보 갱신 | 오래된 통계로 옵티마이저 오판 | ANALYZE/DBMS_STATS 주기적 수행 |

| 비교 항목 | B-Tree 인덱스 | Hash 인덱스 | Bitmap 인덱스 |
|---------|------------|-----------|------------|
| 탐색 복잡도 | O(log n) | O(1) | O(1) (비트 연산) |
| 범위 스캔 | 지원 | 미지원 | 제한적 |
| 카디널리티 | 고/저 모두 | 고 카디널리티 | 저 카디널리티 |
| DML 성능 | 중간 | 높음 | 낮음 (락 문제) |
| 사용 환경 | OLTP 범용 | In-Memory, Hash Join | DW, 분석계 |

커버링 인덱스(Covering Index)는 SELECT 컬럼을 인덱스 내에 모두 포함시켜 테이블 접근을 완전히 제거하며, 읽기 집중 OLTP에서 가장 효과적인 튜닝 기법이다.  "끝"

### 실무 제언

**인덱스 설계의 3대 원칙 준수**
- **챌린지**: 개발 단계에서 성능 테스트 없이 무분별하게 인덱스를 생성하면 INSERT/UPDATE/DELETE 시마다 모든 인덱스를 유지해야 하므로 쓰기 성능이 급격히 저하된다.
- **제언**: 조회 빈도와 선택률을 기반으로 필요한 인덱스만 생성하고, 복합 인덱스로 여러 단일 인덱스를 통합한다. 분기별 사용 통계 기반 미사용 인덱스를 정리한다(Oracle: V$INDEX_USAGE, MySQL: sys.schema_unused_indexes).

**함수 적용으로 인한 인덱스 무력화 방지**
- **챌린지**: 인덱스 컬럼에 함수나 형변환을 적용하면 인덱스 스캔이 비활성화되어 Full Table Scan이 발생한다.
- **제언**: 함수 기반 인덱스(FBI)를 생성하거나, 쿼리에서 컬럼을 변환 없이 직접 조건에 사용하도록 개발 표준을 수립한다.

**복합 인덱스 선두 컬럼 전략**
- **챌린지**: 복합 인덱스에서 선두 컬럼이 쿼리 WHERE 절에 없으면 해당 인덱스를 Range Scan으로 활용할 수 없어 Full Table Scan이 발생한다.
- **제언**: 복합 인덱스 설계 시 등치(=) 조건 컬럼을 앞에, 선택률이 높은(NDV가 큰) 컬럼을 앞에 배치하고, 자주 사용되는 WHERE 조건 조합을 분석하여 여러 쿼리를 포괄하는 복합 인덱스를 설계한다.
