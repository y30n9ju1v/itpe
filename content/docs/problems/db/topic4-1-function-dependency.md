## 해설

### 출제 배경 및 의도

함수 종속성(FD)은 데이터베이스 설계의 성패를 가르는 정규화(Normalization)의 시작이자 끝입니다. 속성 간의 비즈니스적 인과 관계를 파악하지 못하면 잘못된 스키마 설계로 이어져 중복 데이터와 이상 현상(Anomaly)이 발생합니다. 빅데이터와 클라우드 네이티브 환경에서 데이터 마이그레이션 오류나 정합성 붕괴를 원천 차단하려면 속성 간의 종속 관계를 명확히 이해하고 통제해야 합니다.

채점관은 함수 종속성의 개념 자체보다, 각 종속성 유형이 어떤 정규화 단계와 1:1로 대응하는지, 어떤 이상 현상을 유발하는지를 연결하여 설명할 수 있는지를 검증합니다. 완전 함수 종속, 부분 함수 종속, 이행적 함수 종속, BCNF 대상인 결정자적 함수 종속의 구분이 핵심입니다.

### 1. 함수 종속성 (Functional Dependency), 개요

정 의  • 릴레이션에서 속성 X의 값이 속성 Y의 값을 유일하게 결정하는 속성 간의 논리적 제약조건이자 대수적 관계
       - 기호 X → Y로 표기, X를 결정자(Determinant), Y를 종속자(Dependent)라 함

```
[함수 종속성 → 정규화 연계 흐름]

[원시 테이블]
     │
     ├─(1NF: 도메인 원자값 확보)──────────────> [제1정규형]
     │                                               │
     ├─(2NF: 부분 함수 종속 제거)─────────────> [제2정규형]  ← 완전 함수 종속화
     │                                               │
     ├─(3NF: 이행적 함수 종속 제거)──────────> [제3정규형]  ← 속성 간 독립성 확보
     │                                               │
     └─(BCNF: 후보키 아닌 결정자 제거)──────> [보이스-코드 정규형]
```

- 수학의 함수 y=f(x)처럼, 입력값 X가 주어지면 대응하는 출력값 Y는 단 하나만 존재해야 하는 1:1 대응 메커니즘을 따른다.

### 2. 함수 종속성 4대 유형 상세

1) 완전·부분 함수 종속 (2NF 관련)

| 구분 | 항목 | 설명 |
|------|------|------|
| **완전 함수 종속** (FFD) | 정의 | 일반 속성이 복합키 전체에만 종속되며, 기본키 일부에는 종속되지 않는 청정 상태 |
| | 사례 | (학번, 과목번호) → 성적 (학번만으로는 성적 결정 불가) |
| | 정규형 | 제2정규형(2NF) 달성 상태 |
| **부분 함수 종속** (PFD) | 정의 | 복합키 전체가 아닌 일부(진부분집합)에 일반 속성이 종속되는 구조적 결함 |
| | 사례 | (학번, 과목번호) → 지도교수명 (실제: 학번 → 지도교수명) |
| | 정규형 | 2NF 위반 원인 → 2NF 적용 대상 |

- 부분 함수 종속은 복합키 구조에서만 발생하며, 제2정규화를 통해 결정자 단위로 분해하여 해소한다.

2) 이행적·결정자적 함수 종속 (3NF/BCNF 관련)

| 구분 | 항목 | 설명 |
|------|------|------|
| **이행적 함수 종속** (TFD) | 정의 | X→Y이고 Y→Z일 때, 일반 속성 간의 인과 관계로 X→Z가 유도되는 현상 |
| | 사례 | 학번→학과코드, 학과코드→학과명 ∴ 학번→학과명 (이행 종속) |
| | 정규형 | 3NF 위반 원인 → 3NF 적용 대상 |
| **결정자적 함수 종속** | 정의 | 결정자가 후보키(Candidate Key)가 아닌 일반 속성인 변칙적 종속 상태 |
| | 사례 | (학번, 과목명)이 PK인데, 담당교수→과목명 (일반 속성이 PK 구성요소를 결정) |
| | 정규형 | BCNF 위반 원인 → BCNF 적용 대상 |

- 이행적 종속은 일반 속성이 마스터/코드 테이블 역할을 하는 구조이며, BCNF는 모든 결정자가 후보키가 되도록 강제하여 이를 해소한다.

### 3. 함수 종속성과 정규화 매핑 및 트렌드

| 함수 종속성 종류 | 핵심 정의 | 이상현상 유형 | 매핑 정규형 |
|-----------------|-----------|---------------|-------------|
| **완전 함수 종속** (FFD) | 복합키 전체에만 종속 | 없음 (정상 상태) | **2NF** 달성 |
| **부분 함수 종속** (PFD) | 복합키 일부에 종속 | 삽입/삭제/갱신 이상 | **2NF** 적용 |
| **이행적 함수 종속** (TFD) | 일반 속성이 다른 일반 속성 결정 | 갱신 이상, 중복 | **3NF** 적용 |
| **결정자적 함수 종속** | 후보키 아닌 결정자 존재 | 삽입 이상, 정보 손실 위험 | **BCNF** 적용 |
| **다치 종속** (MVD) | 한 결정자에 독립적 다치 속성 공존 | 조합 중복 폭발 | **4NF** 적용 |
| **조인 종속** (JD) | 3원 이상 순환 관계 | 가짜 튜플 생성 | **5NF** 적용 |

- MSA 환경에서 서비스 경계를 넘는 이행적 종속성이나 참조 무결성은 이벤트 기반 아키텍처(EDA)와 Saga 패턴을 적용하여 최종 일관성(Eventual Consistency)으로 달성해야 한다.  "끝"

### 실무 제언

**성능(Performance)과 무결성(Integrity)의 격돌**
- **챌린지**: 함수 종속성을 철저히 분해하여 완전 함수 종속 상태만 남기면 무결성은 극대화되지만, 테이블 수가 늘어나 대규모 시스템에서 다중 조인에 의한 CPU/IO 병목이 발생할 수 있다.
- **제언**: 논리 모델 단계에서는 함수 종속성 다이어그램(FDD)을 그려 완벽한 무결성을 확보하되, 물리 설계 단계에서는 트랜잭션 빈도를 분석하여 역정규화, 파티셔닝, Redis 캐시 레이어 도입을 종합 조율하는 능력이 필요하다.

**MSA 환경에서의 함수 종속성 경계 확장**
- **챌린지**: MSA 환경에서 서비스별 DB 파편화(Database-per-Service)로 시스템 전반의 함수 종속성이 서비스 경계(Bounded Context)를 넘어설 때 데이터 무결성 결함 위험이 급증한다.
- **제언**: 서비스 간 이행적 종속성이나 참조 무결성은 EDA와 Saga 패턴을 적용하여 애플리케이션 레이어에서 최종 일관성(Eventual Consistency)을 달성해야 한다.

**함수 종속성 다이어그램(FDD) 기반 설계 검증**
- **챌린지**: 수백 개 속성을 가진 복잡한 릴레이션에서 종속성 유형을 수작업으로 판별하면 누락이나 오판이 발생할 수 있다.
- **제언**: FDD(Functional Dependency Diagram)를 공식 설계 산출물로 의무화하고, CASE 도구의 폐포 자동 계산 기능을 활용하여 설계 초기 단계부터 종속성을 정형화된 방식으로 문서화·검증해야 한다.
