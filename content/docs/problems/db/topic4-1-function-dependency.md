## 해설

> **정규화의 시작이자 끝, 속성 간의 인과 관계**  
> 함수 종속성(FD): 결정자 X가 종속자 Y를 유일하게 결정하는 논리적 제약조건

함수 종속성(FD)은 데이터베이스 설계의 성패를 가르는 정규화(Normalization)의 시작이자 끝입니다. 속성 간의 비즈니스적 인과 관계를 파악하지 못하면 잘못된 스키마 설계로 이어져 중복 데이터와 이상 현상(Anomaly, 잘못 설계된 테이블에서 데이터를 삽입·삭제·수정할 때 원치 않는 부작용이 발생하는 현상)이 발생합니다. 빅데이터와 클라우드 네이티브 환경에서 데이터 마이그레이션 오류나 정합성 붕괴를 원천 차단하려면 속성 간의 종속 관계를 명확히 이해하고 통제해야 합니다.

### 1. 출제 배경 및 의도

채점관은 함수 종속성의 개념 자체보다, 각 종속성 유형이 어떤 정규화 단계와 1:1로 대응하는지, 그리고 어떤 이상 현상을 유발하는지를 연결하여 설명할 수 있는지를 검증합니다. 특히 완전 함수 종속, 부분 함수 종속, 이행적 함수 종속, 그리고 BCNF 대상인 결정자적 함수 종속의 구분이 핵심입니다.

### 2. 개념의 본질과 작동 메커니즘

**정의:** 어떤 릴레이션에서 속성 X의 값이 속성 Y의 값을 유일하게 결정하는 속성 간의 논리적인 제약조건이자 대수적 관계입니다. 기호로는 X → Y로 표기하며, X를 **결정자(Determinant)**, Y를 **종속자(Dependent)**라고 부릅니다.

수학의 함수 y = f(x) 구조처럼, 입력값 X(학번)가 주어지면 그에 대응하는 출력값 Y(성명, 학과)는 단 하나만 존재해야 하는 1:1 대응 메커니즘을 따릅니다.

- **수학적 명제:** 릴레이션 R에 속한 임의의 두 튜플 t1, t2에 대하여, t1[X] = t2[X]일 때 언제나 t1[Y] = t2[Y]가 성립하면 릴레이션 R은 함수 종속성 X→Y를 만족합니다.

#### 함수 종속성 4대 유형

##### 1. 완전 함수 종속 (Full Functional Dependency)

- **개념:** 일반 속성이 기본키(특히 복합키) 전체에 종속되며, 기본키를 구성하는 하위 속성 집합에는 함수 종속되지 않는 청정 상태
- **사례:** 기본키가 `(학번, 과목번호)`일 때, `성적`은 학번과 과목번호가 동시에 결합되어야만 확정됨
- **수식:** (학번, 과목번호) → 성적

##### 2. 부분 함수 종속 (Partial Functional Dependency)

- **개념:** 복합 기본키(두 개 이상의 컬럼으로 구성된 기본키) 전체가 아니라, 일부분(진부분집합)에 일반 속성이 종속되는 구조적 결함 상태
- **사례:** 기본키가 `(학번, 과목번호)`인데, `지도교수명`은 `학번`에 의해서만 결정되는 경우
- **수식:** (학번, 과목번호) → 지도교수명 (실제로는 학번 → 지도교수명이 성립)

##### 3. 이행적 함수 종속 (Transitive Functional Dependency)

- **개념:** X→Y이고 Y→Z일 때, 일반 속성 간의 인과 관계로 인해 X→Z가 유도되는 현상
- **사례:** 학번 → 학과코드, 학과코드 → 학과명 이면, 일반 속성인 학과코드가 학과명을 결정하는 구조
- **수식:** 학번 → 학과코드 → 학과명

##### 4. 결정자적 함수 종속 (BCNF 대상)

- **개념:** 결정자가 후보키(Candidate Key)가 아닌 일반 속성인 변칙적인 종속 상태 (BCNF는 모든 결정자가 후보키여야 함)
- **사례:** `(학번, 과목명)`이 기본키인데, 일반 속성인 `담당교수`가 `과목명`을 결정하는 구조적 모순
- **수식:** 담당교수 → 과목명

#### 정규화와의 연계 흐름

```
[원시 테이블] ──( 1NF: 도메인 원자값 확보 )──> [제1정규형]
     │
     └──( 2NF: 부분 함수 종속성 제거 )────> [제2정규형 (완전 함수 종속화)]
           │
           └──( 3NF: 이행적 함수 종속성 제거 )──> [제3정규형]
                 │
                 └──( BCNF: 후보키 아닌 결정자 제거 )─> [보이스-코드 정규형]
```

### 3. 핵심 키워드 (Must-Have)

- **구성 요소:** 결정자(Determinant), 종속자(Dependent), 튜플(Tuple), 만족(Satisfy)
- **종속성 유형:** 완전 함수 종속(FFD), 부분 함수 종속(PFD), 이행적 함수 종속(TFD)
- **기타 특수 유형:** 다치 종속(MVD), 조인 종속(JD), 결정자적 함수 종속(BCNF 대상)
- **연계 개념:** 이상 현상(Anomaly), 암스트롱 공리(Armstrong's Axioms), 정규형(1NF~BCNF)

### 4. 맥락과 비교

함수 종속성의 유형들은 정규화 단계(1NF → 2NF → 3NF → BCNF)와 1:1로 매핑되는 상속 및 진화 구조를 가집니다.

| 함수 종속성 종류 | 영문명 | 핵심 정의 및 특징 | 매핑되는 정규형 |
| --- | --- | --- | --- |
| **완전 함수 종속** | Full FD (FFD) | 종속자가 복합 기본키 전체에만 종속되며, 기본키의 일부 속성에는 종속되지 않는 상태 | **제2정규형(2NF)** (부분 함수 종속 제거) |
| **부분 함수 종속** | Partial FD (PFD) | 복합 기본키를 구성하는 속성 중 일부(진부분집합)에 다른 일반 속성이 종속되는 현상 | **제2정규형(2NF) 위반 원인** (2NF로의 정규화 대상) |
| **이행적 함수 종속** | Transitivity FD (TFD) | X→Y이고 Y→Z일 때, 논리적으로 X→Z가 성립하는 간접 종속 현상 | **제3정규형(3NF)** (이행 함수 종속 제거) |
| **결정자적 함수 종속** | Determinant FD | 후보키(Candidate Key)가 아닌 일반 속성이 다른 속성을 결정하는 구조적 결함 상태 | **BCNF** (일반 결정자 분리) |

### 5. 실무 제언

**성능(Performance)과 무결성(Integrity)의 격돌**

- **챌린지:** 함수 종속성을 철저히 분해하여 완전 함수 종속(FFD) 상태만 남기면 무결성은 극대화되지만, 테이블 수가 늘어나 대규모 시스템에서 다중 조인(Join)에 의한 CPU/IO 병목이 발생할 수 있습니다.
- **제언:** 기술사는 기계적인 정규화만을 고집해서는 안 됩니다. 논리 모델 단계에서는 함수 종속성 다이어그램(FDD)을 그려 완벽한 무결성을 확보하되, 물리 설계 단계에서는 자주 조회되는 종속 속성에 한해 트랜잭션 빈도를 분석하여 역정규화(반정규화), 파티셔닝, 레디스(Redis) 같은 캐시 레이어 도입을 종합 조율하는 능력이 필요합니다.

**MSA 환경에서의 함수 종속성 경계 확장**

- MSA 환경에서는 서비스별로 DB가 파편화(Database-per-Service)되므로, 시스템 전반의 함수 종속성이 서비스 경계(Bounded Context)를 넘어설 때 데이터 무결성 결함 위험이 급증합니다. 서로 다른 마이크로서비스 간에 발생하는 이행적 종속성이나 참조 무결성은 이벤트 기반 아키텍처(EDA)와 Saga 패턴을 적용하여 애플리케이션 레이어에서 최종 일관성(Eventual Consistency)을 달성해야 합니다.
