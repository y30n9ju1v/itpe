# 데이터베이스 함수 종속성(Functional Dependency) 완전 해체

---

## 1. 문제를 해체하는 5대 핵심 분석 항목

### ① 출제 배경 및 의도 (Why Now?)

* **데이터 정합성과 무결성의 본질:** 함수 종속성(FD)은 데이터베이스 설계의 성패를 가르는 '정규화(Normalization)'의 시작이자 끝입니다. 속성 간의 비즈니스적 인과 관계를 파악하지 못하면 잘못된 스키마 설계로 이어져 중복 데이터와 '이상 현상(Anomaly)'이 발생합니다.
* **데이터 거버넌스와 비용 최적화:** 빅데이터와 클라우드 네이티브 환경이 확산되면서 데이터 레이크 및 분산 DB 설계가 빈번해졌습니다. 시스템 변경 시 데이터 마이그레이션 오류나 정합성 붕괴를 원천 차단하기 위해서는 가장 밑바닥에 흐르는 '속성 간의 종속 관계'를 명확히 이해하고 통제할 수 있는지 검증하기 위함입니다.

### ② 개념의 본질과 작동 메커니즘 (What & How)

* **한 문장 정의:** 어떤 릴레이션에서 속성 $X$의 값이 속성 $Y$의 값을 유일하게 결정(결정자 $X$, 종속자 $Y$)하는 속성 간의 논리적인 제약조건이자 대수적 관계입니다.
* **작동 메커니즘 (Mechanism):**
수학의 함수 $y = f(x)$ 구조처럼, 입력값 $X$(학번)가 주어지면 그에 대응하는 출력값 $Y$(성명, 학과)는 단 하나만 존재해야 하는 1:1 대응 메커니즘을 따릅니다.
* **Input:** 특정 릴레이션 내 임의의 두 튜플 $t_1, t_2$.
* **Mechanism:** 두 튜플의 $X$ 값이 같으면 ($t_1[X] = t_2[X]$), 반드시 $Y$ 값도 같아야 함 ($t_1[Y] = t_2[Y]$).
* **Output:** $X \rightarrow Y$ (X는 Y를 결정한다 / Y는 X에 함수적으로 종속된다) 성립.



### ③ 채점관이 눈을 켜고 찾는 'Must-Have' 키워드 (Keywords)

* **구성 요소:** 결정자(Determinant), 종속자(Dependent), 튜플(Tuple), 만족(Satisfy).
* **종속성 유형:** 완전 함수 종속(FFD), 부분 함수 종속(PFD), 이행적 함수 종속(TFD).
* **기타 특수 유형:** 다치 종속(MVD), 조인 종속(JD), 결정자적 함수 종속(BCNF 대상).
* **연계 개념:** 이상 현상(Anomaly), 암스트롱 공리(Armstrong's Axioms), 정규형(1NF~BCNF).

### ④ 상위/하위/유사 개념과의 상호 관계 (Context & Comparison)

함수 종속성의 유형들은 정규화 단계(1NF $\rightarrow$ 2NF $\rightarrow$ 3NF $\rightarrow$ BCNF)와 1:1로 매핑되는 상속 및 진화 구조를 가집니다.

| 함수 종속성 종류 | 영문명 | 핵심 정의 및 특징 | 매핑되는 정규형 |
| --- | --- | --- | --- |
| **완전 함수 종속** | Full FD (FFD) | 종속자가 복합 기본키 전체에만 종속되며, 기본키의 일부 속성에는 종속되지 않는 상태 | **제2정규형(2NF)**<br>

<br>(부분 함수 종속 제거) |
| **부분 함수 종속** | Partial FD (PFD) | 복합 기본키를 구성하는 속성 중 일부(진부분집합)에 다른 일반 속성이 종속되는 현상 | **제1정규형(1NF)**의 문제점<br>

<br>(이상현상의 원인) |
| **이행적 함수 종속** | Transitivity FD (TFD) | $X \rightarrow Y$ 이고 $Y \rightarrow Z$ 일 때, 논리적으로 $X \rightarrow Z$ 가 성립하는 간접 종속 현상 | **제3정규형(3NF)**<br>

<br>(이행 함수 종속 제거) |
| **결정자적 함수 종속** | Determinant FD | 후보키(Candidate Key)가 아닌 일반 속성이 다른 속성을 결정하는 구조적 결함 상태 | **BCNF**<br>

<br>(일반 결정자 분리) |

### ⑤ 실무적 제언 및 챌린지 (So What? / Insight)

* **성능(Performance)과 무결성(Integrity)의 격돌:** 함수 종속성을 철저히 분해하여 완전 함수 종속(FFD) 상태만 남기면 무결성은 극대화되지만, 테이블 수가 늘어나 대규모 시스템에서 다중 조인(Join)에 의한 CPU/IO 병목이 발생할 수 있습니다.
* **기술사의 제언:** 기술사는 기계적인 정규화만을 고집해서는 안 됩니다. 논리 모델 단계에서는 함수 종속성 다이어그램(FDD)을 그려 완벽한 무결성을 확보하되, 물리 설계 단계에서는 '자주 조회되는 종속 속성'에 한해 트랜잭션 빈도를 분석하여 역정규화(반정규화), 파티셔닝, 레디스(Redis) 같은 캐시 레이어 도입을 종합 조율하는 능력이 필요합니다.

## 2. 합격자 수준의 모범 답안 가이드 (2교시형 서술형)

### [1단락] 데이터 정규화의 핵심 이정표, 함수 종속성(Functional Dependency)의 개념

#### 가. 함수 종속성의 정의

* 데이터베이스의 릴레이션(Relation) 인스턴스 내에 존재하는 속성(Attribute)들 간의 논리적 제약조건으로, 어떤 속성 $X$의 값이 속성 $Y$의 값을 유일하게 결정하는 관계를 의미합니다.
* 기호로는 $X \rightarrow Y$ 로 표기하며, 이때 $X$를 **결정자(Determinant)**, $Y$를 종속자(Dependent)라고 부릅니다.

#### 나. 함수 종속성의 수학적 명제 및 다이어그램 가시화

* 릴레이션 $R$에 속한 임의의 두 튜플 $t_1, t_2$에 대하여, $t_1[X] = t_2[X]$ 일 때 언제나 $t_1[Y] = t_2[Y]$ 가 성립하면 릴레이션 $R$은 함수 종속성 $X \rightarrow Y$를 만족합니다.
* 위 가시화 도구처럼 함수 종속성 다이어그램(FDD)을 활용하면 복잡한 테이블 구조 속에서 정규화 대상을 직관적으로 식별할 수 있습니다.

---

### [2단락] 데이터베이스 이상 현상을 유발하는 함수 종속성의 주요 유형 분석

릴레이션 설계의 건전성을 평가하고 정규화를 진행하기 위해 함수 종속성은 크게 4가지 기본 유형으로 분류됩니다.

#### 가. 함수 종속성의 4대 유형 세부 비교

##### 1. 완전 함수 종속 (Full Functional Dependency)

* **상세 개념:** 일반 속성이 기본키(특히 복합키) 전체에 종속되며, 기본키를 구성하는 하위 속성 집합에는 함수 종속되지 않는 청정 상태입니다.
* **사례식 표현:** 기본키가 `(학번, 과목번호)`일 때, `성적`은 학번과 과목번호가 동시에 결합되어야만 확정되므로 완전 함수 종속입니다.
* **수식:** $(학번, 과목번호) \rightarrow 성적$

##### 2. 부분 함수 종속 (Partial Functional Dependency)

* **상세 개념:** 복합키로 구성된 기본키의 전체가 아니라, 일부분(진부분집합)에 일반 속성이 종속되는 구조적 결함 상태입니다.
* **사례식 표현:** 기본키가 `(학번, 과목번호)`인데, 일반 속성인 `지도교수명`은 `학번`에 의해서만 결정되므로 복합키의 일부에 종속하는 부분 함수 종속입니다.
* **수식:** $(학번, 과목번호) \rightarrow 지도교수명$ (실제로는 $학번 \rightarrow 지도교수명$ 이 성립)

##### 3. 이행적 함수 종속 (Transitive Functional Dependency)

* **상세 개념:** 속성 $X, Y, Z$에 대해 $X \rightarrow Y$ 가 성립하고 $Y \rightarrow Z$ 가 성립하여, 궁극적으로 일반 속성 간의 인과 관계로 인해 $X \rightarrow Z$ 가 유도되는 현상입니다.
* **사례식 표현:** `학번`이 결정되면 `학과코드`가 결정되고, `학과코드`가 결정되면 `학과명`이 결정되는 구조입니다. 일반 속성인 학과코드가 학과명을 결정하고 있어 이상 현상을 유발합니다.
* **수식:** $학번 \rightarrow 학과코드 \rightarrow 학과명$

##### 4. 결정자적 함수 종속 (Boyce-Codd FD)

* **상세 개념:** 후보키(Candidate Key)가 아닌 일반 속성이 기본키의 일부 속성을 결정자로 지배해 버리는 변칙적인 종속 상태입니다.
* **사례식 표현:** `(학번, 과목명)`이 기본키인데, 일반 속성인 `담당교수`가 `과목명`을 결정해 버리는 구조적 모순 상황입니다.
* **수식:** $담당교수 \rightarrow 과목명$

### [3단락] 함수 종속성 기반의 데이터베이스 정규화(Normalization) 전개 절차

함수 종속성을 식별하여 순차적으로 구조를 분해(Decomposition)하는 과정이 곧 무손실 분해 정규화 알고리즘입니다.

```
[원시 테이블] ──( 1NF: 도메인 원자값 확보 )──> [제1정규형]
     │
     └──( 2NF: 부분 함수 종속성 제거 )────> [제2정규형 (완전 함수 종속화)]
           │
           └──( 3NF: 이행적 함수 종속성 제거 )──> [제3정규형 (일반반복 제거)]
                 │
                 └──( BCNF: 후보키 아닌 결정자 제거 )─> [보이스-코드 정규형]

```

#### 가. 정규화 단계별 함수 종속성 해결 매커니즘

1. **1NF $\rightarrow$ 2NF (부분 함수 종속 제거):** 복합키의 일부에 종속되던 일반 속성들을 찾아내어 별도의 부모-자식 테이블로 프로젝션(Projection) 분리합니다.
2. **2NF $\rightarrow$ 3NF (이행 함수 종속 제거):** 주식별자를 거치지 않고 일반 속성끼리 종속 관계를 맺고 있던 속성들을 잘라내어 독립적인 코드/마스터 테이블로 분리합니다.
3. **3NF $\rightarrow$ BCNF (강한 결정자 분리):** 후보키가 아니면서 다른 속성을 결정하던 주체(예: 담당교수)를 새로운 테이블의 PK로 승격시키며 분리합니다.

### [4단락] 기술사의 시각: 현대적 아키텍처(MSA/NoSQL) 환경에서의 함수 종속성 제언

#### 가. 마이크로서비스 아키텍처(MSA)에서의 함수 종속성 경계(Boundary) 확장

* 전통적 RDBMS 환경에서는 단일 데이터베이스 인스턴스 안에서 모든 함수 종속성이 엄격하게 통제되었습니다.
* 하지만 MSA 환경에서는 서비스별로 DB가 파편화(Database-per-Service)되므로, 시스템 전반의 함수 종속성이 서비스 경계(Bounded Context)를 넘어설 때 **데이터 무결성 결함 위험**이 급증합니다.

#### 나. 기술사적 실무 마스터 가이드

* **Saga 패턴 및 EDA 도입:** 서로 다른 마이크로서비스 간에 발생하는 이행적 종속성이나 참조 무결성은 하드웨어적 제약조건(FK)으로 해결할 수 없습니다. 따라서 이벤트 기반 아키텍처(EDA)와 Saga 패턴을 적용하여 애플리케이션 레이어에서 '최종 일관성(Eventual Consistency)'을 달성해야 합니다.
* **결론:** 함수 종속성은 관계형 모델의 기본 원리이지만, 클라우드 대용량 환경에서는 무조건적인 분해(정규화)보다 비즈니스 도메인의 범위(Domain Boundary)를 정밀하게 획정하는 도구로 재해석되어야 합니다. 기술사는 논리적 완벽성과 물리적 성능 간의 균형을 유지하는 아키텍처 설계를 주도해야 합니다.
