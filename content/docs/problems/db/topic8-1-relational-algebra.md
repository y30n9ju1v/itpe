## 해설

> **SQL의 수학적 뿌리**  
> 관계대수(Relational Algebra): 릴레이션을 입력받아 새로운 릴레이션을 만들어내는 절차적 질의 언어

우리가 매일 사용하는 SQL은 하늘에서 갑자기 떨어진 언어가 아닙니다. 관계대수와 관계해석이라는 탄탄한 수학적·대수적 기초 위에 구축된 파생 언어입니다. 관계대수는 하나 이상의 릴레이션에 대하여 행을 필터링하거나(Select) 열을 잘라내고(Project), 테이블을 결합하는(Join) 연산자들을 연쇄적으로 적용하여 목적에 맞는 정제된 결과 집합을 도출합니다. 현대 대용량 분산 쿼리 엔진(Spark SQL, Trino 등)의 내부 옵티마이저도 이 관계 대수적 변환 원리를 기반으로 실행 계획을 수립합니다.

### 1. 출제 배경 및 의도

채점관은 수험생이 단순히 SQL 문법을 외워서 쓰는 수준을 넘어, 데이터베이스 내부 옵티마이저가 쿼리를 분석하고 실행 계획(Execution Plan)을 수립할 때 사용하는 관계 대수적 변환 원리를 깊이 있게 이해하고 있는지 검증하려 합니다. 특히 관계대수(절차적)와 관계해석(비절차적)의 패러다임 차이를 명확히 구분하고, 각 연산자의 기호와 특성을 정확히 기술할 수 있는지가 핵심 채점 포인트입니다.

### 2. 개념의 본질과 작동 메커니즘

관계형 데이터베이스에서 원하는 정보를 얻기 위해 릴레이션을 입력받아 수학적 연산 규칙에 따라 새로운 릴레이션을 만들어내는 절차적(Procedural) 언어입니다. 릴레이션을 연산의 대상(Input)으로 삼아 연산 결과물로 또 다른 새로운 릴레이션을 생성(Output)하는 폐쇄 성질(Closure Property)을 가집니다.

**가. 4대 순수관계연산** — 관계형 데이터베이스의 독자적인 구조적 특성을 다루기 위해 고안된 연산자 집합입니다.

- **셀렉션(Selection, σ):** 릴레이션에서 지정된 조건(Predicate)을 만족하는 특정 행(Tuple)들의 부분집합을 구하는 수평적 연산입니다. 결과 릴레이션의 차수(Degree)는 원본과 동일하고 카디널리티(Cardinality)는 감소합니다. 예: $σ_{학과='컴퓨터공학'}(학생)$
- **프로젝션(Projection, π):** 릴레이션에서 특정 열(Column)들만 추출하는 수직적 연산입니다. 결과 릴레이션의 차수가 변경되며, 중복 튜플은 집합의 개념에 따라 자동으로 제거됩니다. 예: $π_{학번, 성명}(학생)$
- **조인(Join, ⋈):** 두 개의 릴레이션을 공통된 속성을 매개체로 결합하여 하나의 새로운 릴레이션을 생성하는 연산입니다. 교차곱(×) 연산 후 셀렉션(σ)을 적용하는 것과 수학적으로 동일합니다. 예: $학생 ⋈_{학생.학번=수강.학번} 수강$
- **디비전(Division, ÷):** 릴레이션 R(X, Y)와 S(Y)가 있을 때, S의 모든 튜플과 연관을 맺고 있는 R의 X 속성 값들을 추출하는 연산입니다. "모든(ALL)" 조건을 처리할 때 사용됩니다. 예: $대리점제품 ÷ 대리점S$

**나. 4대 일반집합연산** — 수학의 전통적 집합 이론을 관계형 데이터베이스의 테이블 구조에 대입한 연산자 그룹입니다. 합집합·교집합·차집합은 두 테이블의 열 구조가 일치하는 합집합 호환성(Union Compatibility)을 반드시 충족해야 합니다.

- **합집합(Union, ∪):** 두 릴레이션의 모든 튜플의 총합을 구하며, 중복 튜플은 자동 제거됩니다. SQL의 `UNION`과 매핑됩니다.
- **교집합(Intersection, ∩):** 두 릴레이션에 공통으로 존재하는 튜플들의 집합을 구합니다. SQL의 `INTERSECT`와 매핑됩니다.
- **차집합(Difference, -):** 첫 번째 릴레이션에는 있지만 두 번째 릴레이션에는 없는 튜플들을 추출하는 비가환성 연산입니다. SQL의 `EXCEPT` 또는 `MINUS`와 매핑됩니다.
- **교차곱(Cartesian Product, ×):** 두 릴레이션의 모든 튜플들을 1:1로 결합하는 모든 조합을 생성합니다. 합집합 호환성이 필요 없으며, 결과 릴레이션의 차수는 두 테이블 차수의 합, 카디널리티는 두 테이블의 곱으로 폭발합니다.

### 3. 핵심 키워드 (Must-Have)

- **순수관계연산자:** 셀렉션(Selection, σ), 프로젝션(Projection, π), 조인(Join, ⋈), 디비전(Division, ÷)
- **일반집합연산자:** 합집합(Union, ∪), 교집합(Intersection, ∩), 차집합(Difference, -), 교차곱(Cartesian Product, ×)
- **연계 및 비교 개념:** 절차적 언어(HOW), 비절차적 언어(WHAT), 프레디키트 해석(Predicate Calculus), 에드가 커드(E. F. Codd), 관계적 완전성(Relational Completeness)

### 4. 맥락과 비교

관계대수와 관계해석은 관계형 데이터 모델을 지탱하는 양대 질의어 모델로서, 표현할 수 있는 질의의 범위와 기능적 능력이 완벽하게 동등한 관계적 완전성(Relational Completeness)을 가집니다.

| 비교 항목 | 관계대수 (Relational Algebra) | 관계해석 (Relational Calculus) |
| --- | --- | --- |
| **개념적 정의** | 릴레이션을 처리하는 연산자와 연산 규칙을 기술 | 원하는 정보가 무엇(What)인지만 정의하는 식을 기술 |
| **제어 패러다임** | **절차적 언어 (Procedural)** | **비절차적 언어 (Non-Procedural)** |
| **핵심 질문** | **HOW** (어떻게 유도하고 계산할 것인가?) | **WHAT** (어떤 데이터를 찾고자 하는가?) |
| **수학적 기반** | 집합론(Set Theory) 및 대수학 구조 | 수리 논리학의 술어 해석 (Predicate Calculus) |
| **구성 종류** | 순수관계연산자, 일반집합연산자 | 튜플 관계해석, 도메인 관계해석 |
| **실무 아키텍처 매핑** | DBMS 옵티마이저의 내부 **실행계획 트리** | 사용자가 작성하는 **원시 SQL 질의문** |

### 5. 실무 제언

**RDBMS 옵티마이저 튜닝과의 직결성 — Push-down Selection 전략**

- **챌린지:** 실무에서 작성하는 SQL은 비절차적(관계해석적)이지만, 데이터베이스 내부의 비용기반 옵티마이저(CBO)는 이를 실행하기 위해 트리 형태의 절차적(관계대수적) 실행 계획으로 변환합니다. 개발자가 이 변환 과정을 이해하지 못하면 비효율적인 쿼리를 작성하게 됩니다.
- **제언:** 기술사는 관계대수 법칙을 활용하여 조인(⋈) 연산을 수행하기 전에 셀렉션(σ)을 먼저 실행하여 데이터의 카디널리티(행의 수)를 최소화하는 Push-down Selection 전략을 쿼리 튜닝과 인덱스 설계에 반영해야 합니다. 조인 연산의 대수적 비용은 두 테이블 카디널리티의 곱에 비례하므로, 조인 전 선(先)필터링을 통해 연산 대상 행 수를 최소화하는 것이 고성능 데이터베이스 솔루션 설계의 핵심 공학적 안목입니다.
