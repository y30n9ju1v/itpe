## 해설

### 출제 배경 및 의도

우리가 매일 사용하는 SQL은 관계대수와 관계해석이라는 수학적 기초 위에 구축된 파생 언어입니다. 관계형 데이터베이스(RDBMS)의 모든 질의 처리는 릴레이션을 입력받아 새로운 릴레이션을 생성하는 대수적 변환 원리에 기반합니다.

현대 대용량 분산 쿼리 엔진(Spark SQL, Trino 등)의 내부 옵티마이저도 동일한 관계 대수적 변환 원리로 실행 계획을 수립합니다. 채점관은 수험생이 단순 SQL 문법 암기를 넘어, 옵티마이저가 실행 계획을 수립할 때 사용하는 관계 대수적 변환 원리를 깊이 있게 이해하는지 검증하고자 합니다.

특히 관계대수(절차적, HOW)와 관계해석(비절차적, WHAT)의 패러다임 차이와 각 연산자의 기호 및 특성을 정확히 기술할 수 있는지가 핵심 채점 포인트입니다.

### 1. 관계대수, 개요

정 의  • 릴레이션을 입력받아 수학적 연산 규칙에 따라 새로운 릴레이션을 만들어내는 절차적(Procedural) 질의 언어
       - 폐쇄 성질(Closure Property): 연산의 입력과 출력이 모두 릴레이션이므로 연산자를 연쇄 적용 가능

```
[관계대수 연산 구조도]

입력 릴레이션(들)
       │
       ▼
┌─────────────────────────────────────┐
│         관계대수 연산자              │
│                                     │
│  순수관계연산        일반집합연산    │
│  ┌──────────┐      ┌──────────┐    │
│  │ σ 셀렉션 │      │ ∪ 합집합 │    │
│  │ π 프로젝션│      │ ∩ 교집합 │    │
│  │ ⋈ 조인  │      │ - 차집합 │    │
│  │ ÷ 디비전 │      │ × 교차곱 │    │
│  └──────────┘      └──────────┘    │
└─────────────────────────────────────┘
       │
       ▼
결과 릴레이션 (새로운 릴레이션)
```

- 관계대수는 SQL 옵티마이저의 내부 실행 계획 트리 생성의 수학적 기반이다.

### 2. 순수관계연산과 일반집합연산

1) 4대 순수관계연산

| 연산자 | 기호 | 정의 | 결과 특성 | SQL 매핑 |
|--------|------|------|----------|---------|
| **셀렉션** | σ | 조건(Predicate)을 만족하는 행(Tuple) 추출 — 수평적 연산 | 차수(Degree) 동일, 카디널리티 감소 | WHERE 절 |
| **프로젝션** | π | 특정 열(Column)만 추출 — 수직적 연산 | 차수 변경, 중복 튜플 자동 제거 | SELECT 절 |
| **조인** | ⋈ | 공통 속성으로 두 릴레이션 결합 (교차곱+셀렉션과 동일) | 차수 = 두 테이블 차수의 합 | JOIN 절 |
| **디비전** | ÷ | R(X,Y) ÷ S(Y): S의 모든 튜플과 연관된 R의 X값 추출 | "모든(ALL)" 조건 처리 | EXISTS / ALL 서브쿼리 |

- 순수관계연산은 관계형 DB 고유의 구조적 특성을 다루기 위한 연산자 집합이다.

2) 4대 일반집합연산

| 연산자 | 기호 | 정의 | 합집합 호환성 | SQL 매핑 |
|--------|------|------|------------|---------|
| **합집합** | ∪ | 두 릴레이션의 모든 튜플 합산, 중복 제거 | 필요 | UNION |
| **교집합** | ∩ | 두 릴레이션에 공통으로 존재하는 튜플 | 필요 | INTERSECT |
| **차집합** | - | 첫 번째에는 있고 두 번째에는 없는 튜플 (비가환성) | 필요 | EXCEPT / MINUS |
| **교차곱** | × | 모든 튜플 조합 생성, 카디널리티 M×N으로 폭발 | 불필요 | CROSS JOIN |

- 합집합·교집합·차집합은 합집합 호환성(Union Compatibility: 속성 수와 각 도메인이 일치)이 전제 조건이다.

### 3. 관계대수와 관계해석 비교 및 현대 적용

| 비교 항목 | 관계대수 (Relational Algebra) | 관계해석 (Relational Calculus) |
|----------|------------------------------|-------------------------------|
| **패러다임** | 절차적 언어 (Procedural) | 비절차적 언어 (Non-Procedural) |
| **핵심 질문** | HOW (어떻게 유도·계산할 것인가?) | WHAT (어떤 데이터를 찾는가?) |
| **수학적 기반** | 집합론(Set Theory) 및 대수학 | 수리 논리학의 술어 해석(Predicate Calculus) |
| **구성 종류** | 순수관계연산자, 일반집합연산자 | 튜플 관계해석, 도메인 관계해석 |
| **실무 매핑** | DBMS 옵티마이저의 내부 실행계획 트리 | 사용자가 작성하는 원시 SQL 질의문 |
| **표현 능력** | 관계적 완전성(Relational Completeness) 동등 | 동일 |

- 관계대수와 관계해석은 표현 가능한 질의 범위가 동등한 관계적 완전성(Relational Completeness)을 가지며, SQL은 양측의 개념을 융합한 실용적 표준 언어이다.  "끝"

### 실무 제언

**Push-down Selection으로 옵티마이저 성능 극대화**
- **챌린지**: 실무에서 작성하는 SQL은 비절차적(관계해석적)이나, DBMS 내부의 비용기반 옵티마이저(CBO)는 이를 절차적(관계대수적) 실행 계획 트리로 변환한다. 개발자가 이 변환 과정을 이해하지 못하면 조인 전 필터링 없이 대용량 교차곱이 먼저 수행되는 비효율 쿼리를 작성하게 된다.
- **제언**: 관계대수 법칙을 활용하여 조인(⋈) 전에 셀렉션(σ)을 먼저 실행하여 카디널리티를 최소화하는 Push-down Selection 전략을 쿼리 튜닝과 인덱스 설계에 반영해야 한다. 조인 비용은 두 테이블 카디널리티의 곱에 비례하므로 조인 전 선(先)필터링이 성능의 핵심이다.

**디비전(÷) 연산의 실무 활용**
- **챌린지**: 디비전 연산은 "모든 조건을 만족하는" 쿼리(예: 모든 과목을 수강한 학생 조회)를 표현하는 데 사용되나, 직접 지원하는 DBMS가 드물어 구현이 어렵다.
- **제언**: 디비전 연산은 이중 NOT EXISTS 패턴이나 GROUP BY + HAVING COUNT(*)를 활용하여 SQL로 변환할 수 있으며, 이 원리를 이해하면 복잡한 "전부 충족" 조건 쿼리 작성 시 논리적 오류를 방지할 수 있다.

**분산 쿼리 엔진에서의 관계대수 활용**
- **챌린지**: Spark SQL, Trino 등 현대 분산 쿼리 엔진에서 비효율적인 조인 순서는 네트워크 셔플(Shuffle) 비용 폭발로 이어져 클라우드 비용을 수십 배 증가시킨다.
- **제언**: 관계대수의 교환·결합 법칙을 기반으로 작은 테이블을 먼저 조인하고, 셀렉션을 가능한 이른 단계로 밀어내는(Push-down) Logical Plan 최적화 원칙을 분산 환경에서도 일관되게 적용해야 한다.
