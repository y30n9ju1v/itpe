## 해설

> **객체지향 상속 구조와 관계형 모델의 가교**  
> 슈퍼/서브타입: 비즈니스 다형성을 논리 모델로 표현하고 물리 테이블로 최적화하는 기법

현대 비즈니스는 매우 다형적인(Polymorphic) 특성을 가집니다. '고객'이라는 하나의 엔티티는 '개인 고객', '법인 고객', '외국인 고객' 등으로 나뉘며 각각 관리해야 하는 속성과 규칙이 다릅니다. 이를 하나의 테이블에 무조건 욱여넣거나 무분별하게 쪼개면 무결성과 성능이 모두 무너집니다. 슈퍼/서브타입 모델링은 공통 속성을 상위 엔티티(슈퍼타입)로 통합하고 개별 속성을 하위 엔티티(서브타입)로 분할하여 이 간극을 해결하는 핵심 기법입니다.

### 1. 출제 배경 및 의도

개발 패러다임은 상속(Inheritance)과 다형성을 지원하는 객체지향(OOP) 중심인 반면, 저장소는 관계형 데이터베이스(RDBMS)를 주로 사용합니다. 이 두 패러다임 간의 간극(Impedance Mismatch)을 논리 모델링 단계에서 유연하게 해결하는 방법이 슈퍼/서브타입이며, 채점관은 수험생이 논리 모델의 제약 조건과 물리 변환의 3대 전략 및 트레이드오프를 명확히 이해하는지 검증하고자 합니다.

### 2. 개념의 본질과 작동 메커니즘

업무를 구성하는 데이터 개체들의 공통 속성을 상위 엔티티(슈퍼타입)로 통합하고, 개별 개체만이 가지는 고유한 속성을 하위 엔티티(서브타입)로 분할하여 상속 구조를 형성하는 고도화된 논리 데이터 모델링 기법입니다.

- **슈퍼타입(Super-type):** 하위 엔티티들이 공통으로 가지는 식별자와 일반 속성들을 보유한 모태 엔티티
- **서브타입(Sub-type):** 슈퍼타입의 식별자와 속성을 상속받으며 자신만의 독자적인 속성 및 관계를 가지는 하위 엔티티
- **변별자(Discriminator):** 슈퍼타입 인스턴스가 어떤 서브타입에 귀속되는지 판별하는 기준 컬럼 (예: 고객구분코드)

하향식 접근(Top-down)을 통해 단일 엔티티를 세분화(Specialization)하거나, 상향식 접근(Bottom-up)을 통해 유사 엔티티들의 공통 분모를 추출하여 일반화(Generalization)합니다. 이후 논리 모델을 물리 모델로 변환할 때 성능·확장성·트랜잭션 집중도를 고려하여 세 가지 변환 전략(Single Table, Table per Type, Table per Concrete Class) 중 최적의 대안을 선택합니다.

### 3. 핵심 키워드 (Must-Have)

- **논리 모델 기법:** 일반화(Generalization), 세분화(Specialization), 변별자(Discriminator), 속성 상속
- **제약 조건:** 배타적 관계(Exclusive), 다중/포함 관계(Inclusive), 완전성 제약(Total/Partial)
- **물리 변환 전략:** Single Table(Roll-up, 싱글 타입), Table per Type(Roll-down, 플러스 타입), Table per Concrete Class(One-to-One, 개별 타입)

#### 2대 핵심 제약 조건

**분할 제약 조건(Disjointness Constraints):** 슈퍼타입 인스턴스가 서브타입들에 동시에 속할 수 있는지 여부를 정의합니다.

| 구분 | 배타적 제약 조건 (Exclusive) | 포함적 제약 조건 (Inclusive) |
| --- | --- | --- |
| **개념** | 하나의 인스턴스는 **오직 하나의 서브타입**에만 매핑 가능 | 하나의 인스턴스가 **여러 서브타입에 동시** 매핑 가능 |
| **예시** | 고객은 '개인'이거나 '법인' 중 하나임 (동시 만족 불가) | 사원은 '사내강사'이면서 동시에 '평가위원'일 수 있음 |

**완전성 제약 조건(Completeness Constraints):** 슈퍼타입에 등록된 데이터가 반드시 서브타입 중 하나에는 등록되어야 하는지 여부를 결정합니다.

| 구분 | 완전 제약 조건 (Total) | 부분 제약 조건 (Partial) |
| --- | --- | --- |
| **개념** | 슈퍼타입의 모든 데이터는 **반드시 하나 이상의 서브타입에 존재**해야 함 | 서브타입에 속하지 않고 **슈퍼타입에만 머무르는 데이터가 존재**할 수 있음 |
| **예시** | 모든 '사원'은 반드시 '정규직' 아니면 '계약직'이어야 함 | '고객' 중에는 개인, 법인 외에 아직 분류되지 않은 비회원이 존재함 |

### 4. 맥락과 비교

슈퍼/서브타입 모델을 물리 데이터베이스 테이블로 전환하는 세 가지 핵심 전략은 트레이드오프(Trade-off) 관계를 가집니다.

| 변환 유형 | Single Table (싱글 타입) | Table per Type (플러스 타입) | Table per Concrete Class (개별 타입) |
| --- | --- | --- | --- |
| **개념적 형상** | 슈퍼+서브를 **1개 테이블**로 통합 | 슈퍼타입 테이블 + 각 서브타입 **별도 테이블**로 분리 (JOIN으로 결합) | 슈퍼타입 속성을 각 서브타입 테이블에 **복사 포함** (슈퍼타입 테이블 없음) |
| **조인(Join) 성능** | **우수** (조인 불필요) | **불리** (슈퍼-서브 간 잦은 조인 발생) | **우수** (서브만 조회 시 조인 없음) |
| **I/O 성능 및 저장 공간** | **불리** (Null 값 양산, Full Scan 부담) | **우수** (각 테이블 크기 최소화, Null 없음) | **보통** (공통 속성 중복 저장으로 공간 낭비) |
| **유지보수 및 확장성** | **불리** (새 서브타입 추가 시 컬럼 추가) | **우수** (독립적 구조로 변경에 유연) | **보통** (공통 속성 변경 시 모든 서브타입 테이블 수정 필요) |
| **선택 기준** | 트랜잭션이 전체를 통합 조회할 때 | 전체조회/서브조회 빈도가 유사하고 Null을 거부할 때 | 트랜잭션이 서브타입 단위로 분리 처리되고 조인을 기피할 때 |

### 5. 실무 제언

**논리적 유연성과 물리적 성능의 충돌 — 트랜잭션 프로파일링 기반 의사결정**

- **챌린지:** 설계 단계에서 슈퍼/서브타입 구조는 가독성이 높아 모델러들이 선호합니다. 그러나 물리 변환 단계에서 트랜잭션의 성격을 정밀하게 분석하지 않고 무조건 Table per Type(플러스 타입)으로 쪼개 놓으면, 화면 하나를 그릴 때마다 슈퍼타입과 서브타입 간 조인이 유발되어 성능 저하의 주범이 됩니다.
- **제언:** 기술사는 논리 모델의 수려함에 매몰되지 말고, CRUD 매트릭스 및 트랜잭션 집중도(Access Frequency)를 정량적으로 분석해야 합니다. 업무 트랜잭션의 데이터 접근 형태를 분석하여 항상 슈퍼+서브를 묶어 통계·조회 중심으로 가동한다면 Single Table(Roll-up) 전략을, Null을 최소화하고 서브타입 단위 독립 처리가 주라면 Table per Type(Roll-down) 전략을, 서브타입 단위로만 조회하고 공통 속성 변경이 드물다면 Table per Concrete Class(One-to-One) 전략을 선택해야 합니다. 데이터가 수천만 건에 달하고 조인 부담이 크다면 과감히 Single Table 전략을 채택하되, 애플리케이션 레이어에서 제약 조건을 보완하는 실무적 유연성을 발휘해야 합니다.
