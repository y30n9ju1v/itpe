## 해설

### 출제 배경 및 의도

현대 비즈니스는 매우 다형적인(Polymorphic) 특성을 가집니다. '고객'이라는 하나의 엔티티는 '개인 고객', '법인 고객', '외국인 고객' 등으로 나뉘며 각각 관리해야 하는 속성과 규칙이 다릅니다. 이를 하나의 테이블에 욱여넣거나 무분별하게 쪼개면 무결성과 성능이 모두 무너집니다.

개발 패러다임은 상속(Inheritance)과 다형성을 지원하는 객체지향(OOP) 중심인 반면, 저장소는 관계형 데이터베이스(RDBMS)를 주로 사용합니다. 이 두 패러다임 간의 간극(Impedance Mismatch)을 논리 모델링 단계에서 유연하게 해결하는 방법이 슈퍼/서브타입입니다.

채점관은 수험생이 논리 모델의 제약 조건과 물리 변환의 3대 전략 및 트레이드오프를 명확히 이해하는지 검증하고자 합니다.

### 1. 슈퍼/서브타입, 개요

정 의  • 공통 속성을 슈퍼타입으로 통합하고 개별 속성을 서브타입으로 분할하여 상속 구조를 형성하는 논리 데이터 모델링 기법
       - 하향식(Top-down) 세분화 또는 상향식(Bottom-up) 일반화를 통해 도출하며, 물리 변환 시 3대 전략 중 최적 대안을 선택

```
[비즈니스 다형성 → 논리 모델 → 물리 변환]

         슈퍼타입 (고객)
        ┌──────────────┐
        │ 고객ID (PK)  │
        │ 고객명       │  ← 공통 속성
        │ 고객구분코드  │  ← 변별자(Discriminator)
        └──────┬───────┘
               │ 상속
       ┌───────┼───────┐
       ▼       ▼       ▼
   [개인고객] [법인고객] [외국인고객]
    주민번호   사업자번호  국적코드
              ← 고유 속성 →

물리 변환 3대 전략
  ┌─────────────┬──────────────┬──────────────────────┐
  │ Single Table │ Table per Type│ Table per Concrete   │
  │ (Roll-up)   │ (Roll-down)  │ Class (One-to-One)   │
  └─────────────┴──────────────┴──────────────────────┘
```

- 슈퍼/서브타입은 논리 모델의 다형성을 표현하고, 물리 변환 전략 선택이 성능과 유지보수성을 결정한다.

### 2. 핵심 제약 조건 및 물리 변환 전략

1) 2대 핵심 제약 조건

| 구분 | 제약 유형 | 내용 | 예시 |
|------|----------|------|------|
| **분할 제약** | 배타적 (Exclusive) | 인스턴스는 오직 하나의 서브타입에만 매핑 가능 | 고객은 '개인'이거나 '법인' 중 하나 |
| **분할 제약** | 포함적 (Inclusive) | 인스턴스가 여러 서브타입에 동시 매핑 가능 | 사원은 '사내강사'이면서 '평가위원' 겸임 가능 |
| **완전성 제약** | 완전 (Total) | 슈퍼타입의 모든 데이터는 반드시 서브타입에 존재해야 함 | 모든 사원은 정규직 또는 계약직이어야 함 |
| **완전성 제약** | 부분 (Partial) | 서브타입에 속하지 않는 데이터가 슈퍼타입에 존재 가능 | 아직 분류되지 않은 비회원 고객 존재 허용 |

- 제약 조건은 논리 모델 설계 시 변별자(Discriminator) 컬럼과 함께 ERD에 명시해야 한다.

2) 물리 변환 전략 비교

| 변환 유형 | Single Table (싱글 타입) | Table per Type (플러스 타입) | Table per Concrete Class (개별 타입) |
|----------|------------------------|---------------------------|-------------------------------------|
| **개념** | 슈퍼+서브를 1개 테이블로 통합 | 슈퍼타입 + 서브타입 별도 테이블 (JOIN 결합) | 슈퍼속성을 서브타입에 복사 포함 (슈퍼타입 테이블 없음) |
| **조인 성능** | 우수 (조인 불필요) | 불리 (슈퍼-서브 간 잦은 조인) | 우수 (서브만 조회 시 조인 없음) |
| **저장 공간** | 불리 (Null 양산, Full Scan 부담) | 우수 (테이블 크기 최소화, Null 없음) | 보통 (공통 속성 중복 저장으로 공간 낭비) |
| **확장성** | 불리 (새 서브타입 추가 시 컬럼 추가) | 우수 (독립적 구조로 변경 유연) | 보통 (공통 속성 변경 시 모든 서브타입 수정 필요) |
| **선택 기준** | 통합 조회 중심 트랜잭션 | 전체/서브 조회 빈도 유사, Null 불가 요건 | 서브타입 단위 독립 처리, 조인 기피 |

- 물리 변환 전략은 CRUD 매트릭스와 트랜잭션 집중도를 정량 분석하여 선택해야 한다.

### 3. 설계 원칙 및 현대 적용 트렌드

| 관점 | 내용 |
|------|------|
| **논리 모델 원칙** | 일반화(Generalization)·세분화(Specialization)·변별자(Discriminator) 명시 |
| **물리 변환 원칙** | 트랜잭션 프로파일링 기반 3대 전략 선택, 복합키 인플레이션 방지 |
| **ORM 연동** | JPA @Inheritance 전략(SINGLE_TABLE/JOINED/TABLE_PER_CLASS)과 1:1 매핑 |
| **NoSQL 적용** | Document DB에서 슈퍼타입을 최상위 도큐먼트, 서브타입을 내장(Embedded) 구조로 구현 |
| **MSA 환경** | 서브타입별 독립 마이크로서비스로 분리 시 Table per Concrete Class 전략 적합 |

- 슈퍼/서브타입 설계 원칙은 RDBMS를 넘어 ORM·NoSQL·MSA 환경까지 일관되게 적용되는 보편 아키텍처 원칙으로 진화하고 있다.  "끝"

### 실무 제언

**논리적 유연성과 물리적 성능의 충돌**
- **챌린지**: 설계 단계에서 슈퍼/서브타입 구조는 가독성이 높아 선호되나, 물리 변환 시 트랜잭션 성격을 분석하지 않고 무조건 Table per Type으로 쪼개면 화면 하나를 그릴 때마다 슈퍼-서브 간 조인이 유발되어 성능 저하의 주범이 된다.
- **제언**: CRUD 매트릭스 및 트랜잭션 집중도(Access Frequency)를 정량 분석하여, 통합 조회 중심이면 Single Table, Null 최소화·독립 처리 우선이면 Table per Type, 서브타입 단위 조회가 지배적이면 Table per Concrete Class 전략을 선택해야 한다.

**복합키 인플레이션 방지**
- **챌린지**: 식별자 관계 기반 슈퍼/서브타입 설계 시 하위 계층으로 내려갈수록 복합키가 누적되어 자식 엔티티의 PK 컬럼이 5~6개 이상으로 폭증한다.
- **제언**: 복합키 컬럼이 3개 이상 누적되는 지점에서는 인위적 대리키(Surrogate Key)를 주식별자로 승격하고 상위 PK는 비식별자 관계(FK)로 강등시켜 복합키 인플레이션을 물리 레벨에서 차단해야 한다.

**ORM 환경에서의 전략 선택**
- **챌린지**: JPA/Hibernate 환경에서 JOINED 전략(Table per Type)을 채택하면 단순 조회에도 슈퍼타입과 서브타입 간 JOIN이 자동 발생하여 N+1 문제와 성능 저하를 유발한다.
- **제언**: OLTP 고빈도 조회 엔티티에는 SINGLE_TABLE 전략을 채택하고, 서브타입별 독립 처리가 필요한 도메인에는 TABLE_PER_CLASS를 적용하는 도메인별 차별화 전략으로 ORM 성능과 모델 표현력을 동시에 확보해야 한다.
