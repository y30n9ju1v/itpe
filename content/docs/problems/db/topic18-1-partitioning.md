## 문제

대용량의 데이터 처리가 산업전반에 걸쳐 상용화되고 있다. 대용량 데이터의 처리 및 검색 성능을 고려하여 데이터베이스를 파티션을 통해 분산 및 저장하는 것을 검토하고 있다. 다음에 대하여 설명하시오.

1) 파티셔닝을 추진하는 목적  
2) 파티셔닝 종류  
3) 분할 기준

## 해설

> **대용량 테이블을 논리적으로 분할하여 성능과 관리성을 동시에 향상시키는 기법**  
> Range/List/Hash/Composite 파티셔닝, 수평/수직 분할, 파티션 프루닝(Pruning)으로 I/O 최소화

테이블 파티셔닝은 수십억 건 이상의 대용량 데이터를 효율적으로 관리하기 위한 핵심 기법입니다. 파티션된 테이블은 하나의 논리적 테이블처럼 보이지만 내부적으로 별도의 물리 세그먼트로 분리되어 쿼리 I/O를 최소화하고 관리 작업(백업, 삭제, 이관)을 단순화합니다.

### 1. 출제 배경 및 의도

119회 기출 문제로, 대용량 데이터 처리가 산업 전반에 확산되면서 파티셔닝의 실무 적용이 필수가 되었습니다. 목적-종류-분할기준의 3가지 항목을 구조적으로 서술하고, 각 파티셔닝 방식의 적합 데이터 특성을 비교하여 설명하는 것이 핵심입니다. 파티션 프루닝(Pruning)의 원리를 함께 설명하면 깊이 있는 답안이 됩니다.

### 2. 개념의 본질과 작동 메커니즘

파티셔닝이란 하나의 대용량 테이블(또는 인덱스)을 정의된 기준에 따라 여러 개의 물리적 저장 단위(파티션, Partition)로 분할하여 저장하는 기법입니다.

- **Input:** 대용량 테이블, 파티션 키(Partition Key) 컬럼, 분할 기준 정의
- **Mechanism:** 파티션 키 값 기반으로 레코드를 특정 파티션에 저장 → 쿼리 시 관련 파티션만 스캔(Pruning)
- **Output:** 쿼리 I/O 감소, 병렬 처리 향상, 파티션 단위 관리(DROP PARTITION, EXCHANGE PARTITION)

```text
[파티셔닝 효과: 파티션 프루닝]

  테이블: 주문 (파티션 키: 주문일자, Range 파티셔닝)
  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │ 2023년 Q1    │ 2023년 Q2    │ 2023년 Q3    │ 2023년 Q4    │
  │ (파티션 1)   │ (파티션 2)   │ (파티션 3)   │ (파티션 4)   │
  └──────────────┴──────────────┴──────────────┴──────────────┘

  WHERE 주문일자 BETWEEN '2023-07-01' AND '2023-09-30'
  → 파티션 3만 스캔 (Partition Pruning)
  → 나머지 75% 데이터 블록 I/O 완전 제거
```

#### 1) 파티셔닝 추진 목적

| 목적 | 내용 |
|------|------|
| **성능 향상** | 파티션 프루닝으로 스캔 범위 축소, 병렬 쿼리 처리 가능 |
| **가용성 향상** | 파티션 단위 백업/복구, 특정 파티션 장애 시 타 파티션 정상 서비스 |
| **관리성 향상** | 오래된 데이터 DROP PARTITION으로 즉시 삭제(DELETE보다 빠름) |
| **데이터 이관** | EXCHANGE PARTITION으로 논리적 이동(물리 복사 없음) |
| **I/O 부하 분산** | 파티션별 다른 테이블스페이스/디스크에 배치 |

#### 2) 파티셔닝 종류 (수평/수직)

```text
[수평 파티셔닝 (Horizontal Partitioning)]
  행(Row) 기준으로 분할 - 동일한 컬럼 구조 유지
  
  원본 테이블                파티션 1          파티션 2
  ┌──────────┬──────────┐   ┌──────────────┐  ┌──────────────┐
  │ 주문번호 │ 주문일자 │   │ 1월~6월 행들 │  │ 7월~12월 행들│
  │ 001      │ 2023-01  │ → │              │  │              │
  │ 002      │ 2023-07  │   └──────────────┘  └──────────────┘
  └──────────┴──────────┘

[수직 파티셔닝 (Vertical Partitioning)]
  컬럼(Column) 기준으로 분할 - PK는 공유
  
  원본 테이블                기본 파티션       대형 컬럼 파티션
  ┌────┬──────┬────────┐   ┌────┬──────┐    ┌────┬────────┐
  │ ID │ 이름 │ 이력서 │ → │ ID │ 이름 │    │ ID │ 이력서 │
  └────┴──────┴────────┘   └────┴──────┘    └────┴────────┘
  (자주 조회: ID, 이름)    (드물게 조회: 이력서, 이미지)
```

#### 3) 분할 기준 (Range/List/Hash/Composite)

| 분할 방식 | 기준 | 적합 데이터 | 예시 |
|---------|------|-----------|------|
| **Range 파티셔닝** | 연속적 값 범위 | 날짜, 연번, 금액 | 월별/분기별/연도별 주문 |
| **List 파티셔닝** | 지정된 값 목록 | 지역코드, 상태코드 | 국가='KR','US','JP' |
| **Hash 파티셔닝** | 해시 함수 결과 | 균등 분산 필요 시 | 고객ID mod 8 |
| **Composite 파티셔닝** | 두 가지 방식 결합 | 복합 조건 최적화 | Range-Hash, Range-List |

```text
[Range-Hash Composite 파티셔닝 예]

  주파티션: 연도별 Range
  서브파티션: 고객ID Hash 4개

  2023년 파티션
  ├── SubPartition_0 (HashID % 4 = 0)
  ├── SubPartition_1 (HashID % 4 = 1)
  ├── SubPartition_2 (HashID % 4 = 2)
  └── SubPartition_3 (HashID % 4 = 3)

  → WHERE 연도=2023 AND 고객ID=특정값
  → 1개 서브파티션만 스캔 (전체의 1/4 × 연도 1/n)
```

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 목적 | 성능, 가용성, 관리성, 파티션 프루닝(Partition Pruning) |
| 종류 | 수평/수직 파티셔닝 |
| 분할 기준 | Range, List, Hash, Composite(Range-Hash, Range-List) |
| 구현 요소 | 파티션 키(Partition Key), DROP PARTITION, EXCHANGE PARTITION, 로컬/글로벌 인덱스 |

### 4. 맥락과 비교

| 분할 방식 | 데이터 분포 | 프루닝 효과 | DML 성능 | 관리 용이성 |
|---------|-----------|-----------|---------|-----------|
| **Range** | 순차적 | 높음(날짜 조건) | 최신 파티션 집중 | 높음(기간 단위 삭제) |
| **List** | 범주형 | 높음(코드 조건) | 균등 분포 필요 | 중간 |
| **Hash** | 균등 분산 | 낮음(특정 파티션 지정 불가) | 균등 | 낮음 |
| **Composite** | 복합 조건 | 매우 높음 | 균등 + 시간성 | 중간 |

### 5. 실무 제언

**파티션 프루닝이 작동하지 않는 함정**

- **챌린지:** 파티션 키 컬럼에 함수나 형변환을 적용하면 파티션 프루닝이 비활성화되어 전체 파티션을 스캔합니다. `WHERE TO_CHAR(주문일자, 'YYYYMM') = '202307'`과 같은 쿼리가 대표적입니다.
- **제언:** 파티션 키 컬럼은 반드시 변환 없이 직접 조건에 사용해야 합니다(`WHERE 주문일자 >= DATE'2023-07-01' AND 주문일자 < DATE'2023-10-01'`). 개발 표준에 파티션 키 사용 지침을 포함하고, 실행 계획에서 `PARTITION RANGE ALL` 여부를 정기적으로 모니터링합니다.

**인덱스 전략: 로컬 vs 글로벌 인덱스**

- **챌린지:** 파티션 테이블에서 글로벌 인덱스는 강력한 성능을 제공하지만 파티션 추가/삭제(DROP PARTITION) 시 글로벌 인덱스가 UNUSABLE 상태가 되어 인덱스 재빌드가 필요합니다. 대용량 테이블에서 인덱스 재빌드는 서비스 중단을 초래할 수 있습니다.
- **제언:** 파티션 관리(정기적 DROP PARTITION)가 필요한 이력성 테이블은 로컬 파티션 인덱스를 기본으로 채택하고, 글로벌 인덱스는 파티션 구조가 안정적인 경우에만 선택적으로 활용합니다.
