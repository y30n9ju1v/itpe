## 해설

### 출제 배경 및 의도

테이블 파티셔닝(Partitioning)은 수십억 건 이상의 대용량 데이터를 효율적으로 관리하기 위한 핵심 기법이다. 파티션된 테이블은 하나의 논리적 테이블처럼 보이지만 내부적으로 별도의 물리 세그먼트로 분리되어 쿼리 I/O를 최소화하고 관리 작업(백업, 삭제, 이관)을 단순화한다.

119회 기출 문제로, 대용량 데이터 처리가 산업 전반에 확산되면서 파티셔닝의 실무 적용이 필수가 되었다. 목적-종류-분할기준의 3가지 항목을 구조적으로 서술하고, 각 파티셔닝 방식의 적합 데이터 특성을 비교하여 설명하는 것이 핵심이다. 파티션 프루닝(Pruning)의 원리를 함께 설명하면 깊이 있는 답안이 된다.

### 1. DB 파티셔닝, 개요

정 의  • 하나의 대용량 테이블(또는 인덱스)을 정의된 기준에 따라 여러 개의 물리적 저장 단위(파티션)로 분할하여 저장하는 기법
       - 파티션 키(Partition Key) 값 기반으로 레코드를 특정 파티션에 저장, 쿼리 시 관련 파티션만 스캔(Pruning)

```
[파티셔닝 효과: 파티션 프루닝]

  테이블: 주문 (파티션 키: 주문일자, Range 파티셔닝)
  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │ 2023년 Q1    │ 2023년 Q2    │ 2023년 Q3    │ 2023년 Q4    │
  │ (파티션 1)   │ (파티션 2)   │ (파티션 3)   │ (파티션 4)   │
  └──────────────┴──────────────┴──────────────┴──────────────┘

  WHERE 주문일자 BETWEEN '2023-07-01' AND '2023-09-30'
  → 파티션 3만 스캔 (Partition Pruning)
  → 나머지 75% 데이터 블록 I/O 완전 제거
```

- 파티셔닝은 성능(프루닝), 가용성(단위 복구), 관리성(단위 삭제/이관)을 동시에 달성하는 기법이다.

### 2. 파티셔닝 목적, 종류, 분할 기준

1) 1) 파티셔닝 추진 목적

| 구분 | 목적 | 내용 |
|------|------|------|
| 성능 | 파티션 프루닝 | 쿼리 시 관련 파티션만 스캔하여 I/O 대폭 감소 |
| 성능 | 병렬 처리 | 파티션별 병렬 쿼리로 처리량 향상 |
| 가용성 | 단위 백업/복구 | 파티션 단위 백업/복구로 특정 파티션 장애 시 타 파티션 정상 서비스 |
| 관리성 | DROP PARTITION | 오래된 데이터를 DELETE보다 빠르게 즉시 삭제 |
| 관리성 | EXCHANGE PARTITION | 논리적 데이터 이관 (물리 복사 없음) |
| 관리성 | I/O 부하 분산 | 파티션별 다른 테이블스페이스/디스크에 배치 |

- 이력성 대용량 테이블에서 DROP PARTITION은 DELETE 대비 수십~수백 배 빠른 관리 작업을 가능하게 한다.

2) 2) 파티셔닝 종류 및 3) 분할 기준

| 구분 | 분할 방식 | 기준 | 적합 데이터 | 프루닝 효과 |
|------|---------|------|-----------|-----------|
| 수평 분할 | Range 파티셔닝 | 연속적 값 범위 | 날짜, 연번, 금액 | 높음 (날짜 조건) |
| 수평 분할 | List 파티셔닝 | 지정된 값 목록 | 지역코드, 상태코드 | 높음 (코드 조건) |
| 수평 분할 | Hash 파티셔닝 | 해시 함수 결과 | 균등 분산 필요 시 | 낮음 (특정 파티션 지정 불가) |
| 수평 분할 | Composite 파티셔닝 | 두 가지 방식 결합 | 복합 조건 최적화 | 매우 높음 |
| 수직 분할 | 수직 파티셔닝 | 컬럼(Column) 기준, PK 공유 | 자주/드물게 조회 컬럼 혼재 | I/O 블록 효율 향상 |

```
[수평 파티셔닝 vs 수직 파티셔닝]

  [수평 파티셔닝]               [수직 파티셔닝]
  행(Row) 기준 분할              컬럼(Column) 기준 분할
  ┌────────────┬──────────────┐  원본: [ID | 이름 | 이력서(BLOB)]
  │ 파티션 1   │ 파티션 2     │  → 기본 파티션: [ID | 이름]
  │ 1월~6월    │ 7월~12월     │  → 대형 파티션: [ID | 이력서]
  └────────────┴──────────────┘  (자주/드물게 조회 컬럼 분리)
```

### 3. Range-Hash Composite 파티셔닝 및 트렌드

```
[Range-Hash Composite 파티셔닝 예]

  주파티션: 연도별 Range
  서브파티션: 고객ID Hash 4개

  2023년 파티션
  ├── SubPartition_0 (HashID % 4 = 0)
  ├── SubPartition_1 (HashID % 4 = 1)
  ├── SubPartition_2 (HashID % 4 = 2)
  └── SubPartition_3 (HashID % 4 = 3)

  → WHERE 연도=2023 AND 고객ID=특정값
  → 1개 서브파티션만 스캔 (전체의 1/4 × 연도 1/n)
```

| 분할 방식 | 데이터 분포 | 프루닝 효과 | DML 성능 | 관리 용이성 |
|---------|-----------|-----------|---------|-----------|
| Range | 순차적 | 높음 (날짜 조건) | 최신 파티션 집중 | 높음 (기간 단위 삭제) |
| List | 범주형 | 높음 (코드 조건) | 균등 분포 필요 | 중간 |
| Hash | 균등 분산 | 낮음 | 균등 | 낮음 |
| Composite | 복합 조건 | 매우 높음 | 균등 + 시간성 | 중간 |

클라우드 네이티브 환경에서는 Snowflake의 자동 클러스터링, BigQuery의 파티션+클러스터링 조합이 OLAP 워크로드에서 파티셔닝 효과를 자동화한다. 파티션 키 컬럼에 함수를 적용하면 프루닝이 비활성화되어 전체 파티션을 스캔하므로, 파티션 키는 변환 없이 직접 조건에 사용해야 한다.  "끝"

### 실무 제언

**파티션 프루닝이 작동하지 않는 함정**
- **챌린지**: 파티션 키 컬럼에 함수나 형변환을 적용하면 파티션 프루닝이 비활성화되어 전체 파티션을 스캔한다. `WHERE TO_CHAR(주문일자, 'YYYYMM') = '202307'`과 같은 쿼리가 대표적이다.
- **제언**: 파티션 키 컬럼은 반드시 변환 없이 직접 조건에 사용해야 한다(`WHERE 주문일자 >= DATE'2023-07-01' AND 주문일자 < DATE'2023-10-01'`). 개발 표준에 파티션 키 사용 지침을 포함하고, 실행 계획에서 `PARTITION RANGE ALL` 여부를 정기적으로 모니터링한다.

**인덱스 전략: 로컬 vs 글로벌 인덱스**
- **챌린지**: 파티션 테이블에서 글로벌 인덱스는 강력한 성능을 제공하지만 파티션 추가/삭제(DROP PARTITION) 시 글로벌 인덱스가 UNUSABLE 상태가 되어 인덱스 재빌드가 필요하다. 대용량 테이블에서 인덱스 재빌드는 서비스 중단을 초래할 수 있다.
- **제언**: 파티션 관리(정기적 DROP PARTITION)가 필요한 이력성 테이블은 로컬 파티션 인덱스를 기본으로 채택하고, 글로벌 인덱스는 파티션 구조가 안정적인 경우에만 선택적으로 활용한다.

**Hash 파티셔닝의 데이터 편중 방지**
- **챌린지**: Hash 파티셔닝은 균등 분산을 목표로 하지만 파티션 수가 2의 거듭제곱이 아닌 경우 특정 파티션에 데이터가 편중되어 Hot Spot이 발생할 수 있다.
- **제언**: Hash 파티션 수는 2의 거듭제곱(4, 8, 16 등)으로 설정하고, 주기적으로 파티션별 레코드 수를 모니터링하여 편중 발생 시 파티션 재구성을 검토한다.
