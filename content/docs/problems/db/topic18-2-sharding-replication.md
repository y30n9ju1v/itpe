## 문제

데이터베이스에서 성능 향상을 위한 샤딩(Sharding), 파티셔닝(Partitioning) 기법과 가용성을 위한 리플리케이션(Replication)과 쿼리 오프로딩(Query offloading)기법에 대해 설명하시오.

## 해설

### 출제 배경 및 의도

MSA(마이크로서비스 아키텍처) 전환과 글로벌 클라우드 네이티브 서비스 확산으로 단일 DBMS가 처리할 수 없는 초대용량 트래픽이 일반화되었습니다. 일 수십억 건의 트랜잭션을 처리하는 커머스·소셜·금융 서비스에서 데이터 분산 전략은 아키텍처 설계의 최우선 과제입니다.

샤딩과 파티셔닝의 개념적 차이, 리플리케이션 유형별 트레이드오프, Query Offloading을 통한 읽기/쓰기 분리 아키텍처를 종합적으로 이해하는 것이 시험 채점의 핵심입니다. 특히 CAP 정리와의 연계, 복제 지연(Replication Lag) 문제, 리샤딩의 운영 비용까지 논술하면 고득점 답안이 됩니다.

### 1. 분산 DB 아키텍처, 개요

정 의  • 샤딩·파티셔닝·리플리케이션을 결합하여 성능과 가용성을 동시에 확보하는 분산 데이터베이스 설계 기법
       - 샤딩은 수평 분산으로 쓰기 성능을, 리플리케이션은 복제로 읽기 성능 및 가용성을 향상

```
  ┌─────────────────────────────────────────────────────┐
  │                  Application Layer                   │
  │          읽기 요청 ◄─────────┬──────────► 쓰기 요청  │
  └──────────────────────────────┼─────────────────────┘
                                 │
              ┌──────────────────┴──────────────────┐
              │       DB Proxy / Router Layer        │
              │  (읽기→Replica, 쓰기→Master 라우팅)  │
              └──────────┬───────────────────────────┘
                         │
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
   ┌─────────┐     ┌─────────┐     ┌─────────┐
   │ Shard 1 │     │ Shard 2 │     │ Shard 3 │  ← 샤딩
   │(Master) │     │(Master) │     │(Master) │
   └────┬────┘     └────┬────┘     └────┬────┘
        │               │               │        ← 리플리케이션
   ┌────┴────┐     ┌────┴────┐     ┌────┴────┐
   │Replica1 │     │Replica2 │     │Replica3 │
   └─────────┘     └─────────┘     └─────────┘
```

- 샤딩과 리플리케이션을 결합하면 Scale-Out(쓰기)과 고가용성(읽기)을 동시에 달성할 수 있음

### 2. 핵심 분산 기법

1) 샤딩(Sharding)과 파티셔닝(Partitioning)

| 구분 | 항목 | 설명 |
|------|------|------|
| 파티셔닝 | 분산 단위 | 단일 DBMS 내 논리/물리 분할 (투명성 보장) |
| 파티셔닝 | 확장성 | 수직 확장(Scale-Up) 연계, 애플리케이션 변경 불필요 |
| 파티셔닝 | 조인 | 파티션 간 조인 가능 |
| 샤딩 | 분산 단위 | 다수의 독립 DBMS 서버로 수평 분할 |
| 샤딩 | 확장성 | 수평 확장(Scale-Out), 라우팅 로직 필요 |
| 샤딩 | 조인 | 샤드 간 조인 비용 매우 큼 |

샤딩 키(Shard Key) 전략 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| Range 샤딩 | 방법 | 키 범위로 샤드 결정 |
| Range 샤딩 | 장단점 | 범위 쿼리 효율 / 핫스팟(Hot Spot) 발생 |
| Hash 샤딩 | 방법 | Hash(키) mod N으로 균등 분산 |
| Hash 샤딩 | 장단점 | 균등 분산 / 범위 쿼리 비효율, 리샤딩 어려움 |
| Directory 샤딩 | 방법 | 룩업 테이블로 라우팅 |
| Directory 샤딩 | 장단점 | 유연한 분배 / 룩업 테이블 병목 |
| Geo 샤딩 | 방법 | 지역 기반 분배 |
| Geo 샤딩 | 장단점 | 레이턴시 최소화 / 지역별 데이터 불균형 |

- Hash 샤딩은 균등 분산에 유리하나, 리샤딩 시 데이터 재배치 비용이 매우 큼

2) 리플리케이션(Replication)

| 구분 | 항목 | 설명 |
|------|------|------|
| Master-Slave | 구조 | Master 1개 쓰기, Slave 다수 읽기 전용 |
| Master-Slave | 복제 방식 | Binary Log → Relay Log 순방향 복제 |
| Master-Slave | 장단점 | 구성 단순, 읽기 확장 / Master SPOF, 복제 지연 |
| Master-Master | 구조 | 양방향 동기화, 모든 노드 읽기/쓰기 가능 |
| Master-Master | 복제 방식 | 양방향 실시간 동기화 |
| Master-Master | 장단점 | 고가용성, 지역 분산 쓰기 / 충돌 해결(Conflict Resolution) 복잡 |

```
[Master-Slave 아키텍처]
  Write   ┌──────────────┐
  ──────→ │   Master DB  │ ─→ Binary Log ─→ Relay Log
          └──────────────┘                      │
                                    ┌───────────┴───────────┐
                                    ▼                       ▼
                           ┌──────────────┐       ┌──────────────┐
                  Read ──→ │   Slave DB 1 │       │   Slave DB 2 │ ← Read
                           └──────────────┘       └──────────────┘
```

- Master-Slave는 읽기 확장에 유리하나 Master 단일 장애점(SPOF) 해소를 위한 HA 구성 필수

### 3. Query Offloading과 CAP 트레이드오프

1) Query Offloading(쿼리 오프로딩)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 읽기 쿼리를 Replica로 분기하여 Master 부하 감소 |
| 구현 | DB Proxy | ProxySQL, MaxScale, AWS RDS Proxy로 SQL 파싱 후 R/W 자동 분기 |
| 효과 | 부하 감소 | Master CPU/I/O 부하 70~80% 감소 |
| 주의 | 복제 지연 | Replication Lag 중 읽기 시 Stale Read 발생 가능 |

2) CAP 이론 관점 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| Master-Slave (동기) | Consistency | 강함 (복제 완료 후 응답) |
| Master-Slave (동기) | Availability | 낮음 (복제 대기) |
| Master-Slave (비동기) | Consistency | 약함 (복제 지연 허용) |
| Master-Slave (비동기) | Availability | 높음 |
| Master-Master | Consistency | 중간 (충돌 해결 비용) |
| Master-Master | Availability | 높음 (멀티 노드 쓰기) |
| 샤딩 | Partition Tolerance | 높음 (교차 샤드 트랜잭션 어려움) |

- 실제 대규모 시스템에서는 비동기 복제의 Replication Lag을 비즈니스 중요도에 따라 계층적으로 관리해야 하며, CAP의 이분법보다 PACELC 관점의 세밀한 트레이드오프 설계가 필요함.  "끝"

### 실무 제언

**리샤딩(Re-sharding) 선제 대비**
- **챌린지**: Hash 샤딩은 샤드 수 변경 시 대부분의 데이터를 재배치해야 하는 극단적 운영 부하가 발생함
- **제언**: 초기 설계 시 Consistent Hashing 또는 가상 샤드(Virtual Shard) 방식을 채택하고, MongoDB Atlas·Vitess 등 샤딩 미들웨어로 리샤딩 복잡도를 최소화

**복제 지연(Replication Lag) 관리**
- **챌린지**: Master에 쓴 직후 Slave에서 읽으면 구 버전 데이터를 반환하는 "Read-Your-Writes 불일치" 발생
- **제언**: 강한 일관성이 필요한 요청(결제, 주문 확인)은 반드시 Master에서 읽도록 라우팅하고, DB Proxy 레벨에서 Session Consistency 또는 모노토닉 읽기 보장 정책 구현

**Query Offloading 적용 범위 설계**
- **챌린지**: Replica로 분기된 읽기 쿼리에서 Stale Read 발생 시 비즈니스 오류로 이어질 수 있음
- **제언**: 읽기 쿼리를 "지연 허용 읽기"와 "즉시 일관성 읽기"로 분류하고, 지연 허용 읽기만 Replica로 오프로딩하는 계층적 라우팅 정책을 수립
