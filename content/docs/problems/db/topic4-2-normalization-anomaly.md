## 해설

> **데이터 품질 저하의 원천, 이상 현상과 정규화**  
> 부분·이행·결정자 함수 종속을 단계적으로 제거하여 무결성을 달성하는 프로세스

부적절하게 설계된 테이블은 데이터 중복을 유발하고, 삽입·삭제·수정 시 정보가 왜곡되는 이상 현상(Anomaly)이 발생합니다. 함수적 종속성 다이어그램(FDD)은 복잡하게 얽힌 속성 간의 인과관계를 가시화하는 강력한 도구이며, 2NF → 3NF → BCNF로 이행하는 수학적·공학적 분해 과정을 직관적으로 이해하는 것이 본 문제의 핵심입니다.

### 1. 출제 배경 및 의도

채점관은 각 정규형이 어떤 함수 종속성 유형을 제거하는지, 그리고 함수적 종속성 다이어그램(FDD)을 통해 그 구조적 결함을 시각적으로 표현하고 무손실 분해를 수행할 수 있는지를 평가합니다. 이론적 정의와 함께 구체적인 사례를 제시하는 답안이 고득점을 받습니다.

### 2. 개념의 본질과 작동 메커니즘

함수적 종속성 분석을 기반으로 속성 간의 종속 관계를 분해(Projection)함으로써 중복성을 제거하고, 이상 현상을 원천적으로 방지하는 점진적 스키마 최적화 프로세스입니다.

정규화되지 않은 릴레이션(Input)에 존재하는 복합키의 일부 종속성(부분 FD), 일반 속성 간의 종속성(이행 FD), 후보키가 아닌 결정자(BCNF 대상)를 순차적으로 식별하고, 무손실 분해(Lossless Decomposition) 규칙에 따라 독립된 릴레이션(Output)들로 쪼개어 무결성을 달성합니다.

#### 가. 이상 현상(Anomaly)의 3대 유형

**정의:** 함수적 종속성 분석을 간과하여 테이블에 중복 데이터가 집중될 때, 릴레이션 조작(DML) 시 데이터 정합성이 깨지고 모순이 발생하는 불일치 현상

**1. 삽입 이상 (Insertion Anomaly)**

- **개념:** 데이터를 추가할 때, 불필요하거나 원하지 않는 정보까지 함께 입력해야 하거나 PK 제약(Null 불가)으로 인해 삽입이 불가능해지는 현상
- **사례:** 수강 테이블에 `(학번, 과목번호)`가 복합 PK로 지정될 때, 아직 수강신청을 하지 않은 신입생 고유 정보(학번, 이름)를 등록하려면 `과목번호` 자리에 가짜 값을 채워 넣어야만 삽입이 가능

**2. 삭제 이상 (Deletion Anomaly)**

- **개념:** 특정 튜플을 삭제할 때, 업무적으로 유지되어야 하는 유용한 정보까지 연쇄적으로 함께 소실되는 현상
- **사례:** 특정 학생이 수강을 취소하여 해당 수강 튜플을 삭제했더니, 그 테이블에 유일하게 존재하던 해당 과목의 `담당교수`나 `강의실 위치` 정보까지 통째로 사라지는 현상

**3. 갱신 이상 (Update Anomaly)**

- **개념:** 특정 속성 값을 수정할 때, 중복 저장된 데이터 중 일부만 변경되어 데이터의 일관성이 무너지고 모순이 발생하는 현상
- **사례:** 학번 '26-001' 학생이 여러 과목을 신청하여 이름과 전화번호가 행마다 중복 저장된 상태에서, 전화번호가 변경될 때 일부 행만 수정하고 나머지를 누락하면 동일 인물의 연락처가 행마다 다르게 표현되는 모순 발생

#### 나. 정규화 단계별 FDD 분석

**1. 제2정규형 (2NF) — 부분 함수 종속 제거**

적용 대상: 1NF 충족 상태에서 복합 기본키의 일부분에 일반 속성이 종속되는 부분 함수 종속성이 존재하는 경우

```text
[분해 전 FDD: 부분 함수 종속 존재]
┌───────────────┐
│  학번  , 과목번호 │ ──┐
└───────────────┘   │ (완전 함수 종속)
    │               ▼
    │            ┌──────┐
    │            │ 성적 │
    │            └──────┘
    ▼ (부분 함수 종속)
┌──────────┐
│ 지도교수 │
└──────────┘

       ▼ 무손실 분해 (Projection) 적용

[분해 후 2NF 충족 상태]
- 릴레이션 1: 학생지도 ( 학번, 지도교수 )  → 학번이 결정자
- 릴레이션 2: 수강성적 ( 학번, 과목번호, 성적 ) → (학번, 과목번호)가 결정자
```

**2. 제3정규형 (3NF) — 이행적 함수 종속 제거**

적용 대상: 2NF 충족 상태에서 일반 속성이 또 다른 일반 속성을 결정하는 이행적 함수 종속성이 존재하는 경우

```text
[분해 전 FDD: 이행적 함수 종속 존재]
┌──────┐            ┌──────────┐            ┌────────┐
│ 학번 │ ─────────> │ 학과코드 │ ─────────> │ 학과명 │
└──────┘            └──────────┘            └────────┘
   │                                           ▲
   └───────────────────────────────────────────┘
                 (이행적 함수 종속 발생)

       ▼ 무손실 분해 (Projection) 적용

[분해 후 3NF 충족 상태]
- 릴레이션 1: 학생학과 ( 학번, 학과코드 ) → 학번이 결정자
- 릴레이션 2: 학과정보 ( 학과코드, 학과명 ) → 학과코드가 결정자 (일반 속성의 분리)
```

**3. 보이스-코드 정규형 (BCNF) — 후보키 아닌 결정자 제거**

적용 대상: 3NF 충족 상태에서 복합키 테이블에 후보키가 아닌 일반 속성이 기본키의 일부 속성을 지배하는 결정자적 함수 종속성이 존재하는 경우

```text
[분해 전 FDD: 후보키가 아닌 결정자 존재]
┌───────────────┐
│  학번  , 과목명   │ ──┐
└───────────────┘   │
    ▲               ▼
    │           ┌──────────┐
    │           │ 담당교수 │
    │           └──────────┘
    └─────────────────┘
       (담당교수가 과목명을 결정함: 결정자 역전)

       ▼ 무손실 분해 (Projection) 적용

[분해 후 BCNF 충족 상태]
- 릴레이션 1: 교수과목 ( 담당교수, 과목명 ) → 일반 속성이던 담당교수가 PK로 승격
- 릴레이션 2: 학생교수 ( 학번, 담당교수 )   → 학번과 교수 매핑으로 관계 유지
```

### 3. 핵심 키워드 (Must-Have)

- **이상 현상 관점:** 삽입 이상(Insertion), 삭제 이상(Deletion), 갱신/수정 이상(Update), 데이터 중복, 정보 손실
- **정규형 및 다이어그램 관점:** 결정자(Determinant), 종속자(Dependent), 부분 함수 종속(Partial FD), 완전 함수 종속(Full FD), 이행적 함수 종속(Transitive FD), 후보키(Candidate Key), 무손실 분해

### 4. 맥락과 비교

정규화 단계는 하위 정규형의 제약조건을 만족해야만 다음 상위 정규형으로 진급할 수 있는 누적적(Cumulative) 포함 관계를 가집니다.

- **1NF:** 모든 속성의 도메인이 원자값(Atomic Value)을 만족하는 단계
- **2NF:** 기본키의 진부분집합이 일반 속성을 결정하는 부분 함수 종속을 제거하여 완전 함수 종속화하는 단계
- **3NF:** 주식별자를 거쳐 간접적으로 결정되는 이행적 함수 종속을 제거하여 속성 간 독립성을 확보하는 단계
- **BCNF:** 모든 결정자가 후보키(Candidate Key)가 되도록 강제하여 변칙적 종속성을 해결하는 단계

### 5. 실무 제언

**과도한 분해에 따른 성능 저하**

- **챌린지:** 이론에만 치우쳐 BCNF 이상으로 정규화를 적용하면 테이블 개수가 급증합니다. 이는 조회(Select) 요청 시 과도한 조인(Join) 연산과 디스크 I/O 병목을 유발합니다.
- **제언:** 설계 초기 단계에는 암스트롱 공리와 함수적 종속성 다이어그램에 입각하여 완벽한 논리 모델(BCNF)을 확보해야 합니다. 이후 애플리케이션의 트랜잭션 집중도와 쿼리 패턴을 분석하여 의도된 반정규화(Denormalization)나 테이블 파티셔닝(Partitioning), 인덱스 튜닝을 조화롭게 배치하는 아키텍처 리더십이 요구됩니다.
