## 해설

### 출제 배경 및 의도

부적절하게 설계된 테이블은 데이터 중복을 유발하고, 삽입·삭제·수정 시 정보가 왜곡되는 이상 현상(Anomaly)이 발생합니다. 함수적 종속성 다이어그램(FDD)은 복잡하게 얽힌 속성 간의 인과관계를 가시화하는 강력한 도구이며, 2NF → 3NF → BCNF로 이행하는 수학적·공학적 분해 과정을 직관적으로 이해하는 것이 본 문제의 핵심입니다.

채점관은 각 정규형이 어떤 함수 종속성 유형을 제거하는지, 그리고 FDD를 통해 구조적 결함을 시각적으로 표현하고 무손실 분해를 수행할 수 있는지를 평가합니다. 이론적 정의와 함께 구체적인 사례를 제시하는 답안이 고득점을 받습니다.

### 1. 이상현상과 정규화, 개요

정 의  • 함수적 종속성 분석을 기반으로 속성 간의 종속 관계를 분해(Projection)함으로써 중복성을 제거하고, 이상 현상을 원천적으로 방지하는 점진적 스키마 최적화 프로세스
       - 부분 FD → 이행 FD → 후보키 아닌 결정자 순서로 순차적 제거

```
[정규화 단계별 이상현상 제거 흐름]

 비정규 릴레이션
      │ 1NF: 원자값 확보
      ▼
 [제1정규형] ─── 부분 FD 잔존 → 삽입/삭제/갱신 이상 발생
      │ 2NF: 부분 FD 제거 (완전 함수 종속화)
      ▼
 [제2정규형] ─── 이행 FD 잔존 → 갱신 이상, 중복 발생
      │ 3NF: 이행 FD 제거 (속성 간 독립성)
      ▼
 [제3정규형] ─── 후보키 아닌 결정자 잔존 → 삽입/삭제 이상
      │ BCNF: 모든 결정자 = 후보키
      ▼
 [보이스-코드 정규형] ── 이상현상 완전 제거
```

- 정규화 단계는 하위 정규형의 제약조건을 만족해야만 상위 정규형으로 진급하는 누적적(Cumulative) 포함 관계를 가진다.

### 2. 가. 이상현상 3대 유형 및 나. 단계별 FDD

1) 가. 이상현상(Anomaly) 3대 유형

| 유형 | 정의 | 사례 | 발생 원인 |
|------|------|------|-----------|
| **삽입 이상** (Insertion) | 데이터 추가 시 불필요한 정보까지 입력하거나 PK 제약으로 삽입 불가 | 수강 미신청 신입생의 학적 등록 시 과목번호 가짜값 필요 | 부분 함수 종속으로 학생+과목+성적이 혼재 |
| **삭제 이상** (Deletion) | 특정 튜플 삭제 시 유지되어야 할 유용한 정보까지 연쇄 소실 | 수강 취소 시 해당 과목의 담당교수 정보까지 삭제 | 하나의 테이블에 서로 다른 업무 주체 혼재 |
| **갱신 이상** (Update) | 속성값 수정 시 중복 행 중 일부만 변경되어 데이터 모순 발생 | 학생 전화번호 변경 시 일부 행만 수정 → 동일 학번에 다른 전화번호 공존 | 중복 데이터 집중으로 일관성 훼손 |

- 3대 이상현상의 근본 원인은 하나의 릴레이션에 서로 다른 업무 주체(학생, 과목, 성적)가 혼재하여 함수 종속성이 뒤엉킨 것이다.

2) 나. 정규화 단계별 FDD 분석

```
[2NF: 부분 함수 종속 제거]          [3NF: 이행적 함수 종속 제거]

학번                                학번 ────────> 학과코드 ────────> 학과명
  │  \(부분FD)                         │                              ▲
  │   ───────> 지도교수                 └──────────────────────────────┘
  │                                              (이행적 종속 발생)
(학번,과목번호) ──(완전FD)──> 성적
                                      ▼ 분해
      ▼ 분해
학생지도 (학번, 지도교수)             학생학과 (학번, 학과코드)
수강성적 (학번, 과목번호, 성적)       학과정보 (학과코드, 학과명)


[BCNF: 후보키 아닌 결정자 제거]

(학번, 과목명) ──> 성적
      ▲
담당교수 ──────> 과목명  ← 담당교수가 후보키 아닌데 과목명 결정 (역전!)

      ▼ 분해
교수과목 (담당교수, 과목명)   ← 담당교수가 PK로 승격
학생교수 (학번, 담당교수)    ← 학번-교수 매핑
```

- 무손실 분해(Lossless Decomposition): 분해된 릴레이션들을 다시 자연 조인(Natural Join)하면 원본과 동일한 데이터를 손실 없이 복원할 수 있어야 한다.

### 3. 정규화 단계 비교 및 트렌드

| 정규형 | 제거 대상 | 적용 조건 | 분해 결과 | 이상현상 해소 |
|--------|-----------|-----------|-----------|---------------|
| **1NF** | 비원자값 도메인 | 다중값 속성 존재 | 속성 분리 | 구조적 정형화 |
| **2NF** | 부분 함수 종속 | 복합키 + 부분종속 | 결정자 단위 분리 | 삽입/삭제 이상 |
| **3NF** | 이행적 함수 종속 | 일반속성→일반속성 | 코드/마스터 분리 | 갱신 이상, 중복 |
| **BCNF** | 후보키 아닌 결정자 | 일반속성이 PK일부 결정 | 결정자 → PK 승격 분리 | 삽입/삭제 이상 |

- OLTP 환경에서는 3NF 또는 BCNF가 표준 정규화 목표이며, 이후 트랜잭션 패턴 분석 기반의 통제된 반정규화를 통해 성능을 최적화하는 '선정규화 후조율' 원칙을 준수해야 한다.  "끝"

### 실무 제언

**과도한 분해에 따른 성능 저하**
- **챌린지**: BCNF 이상으로 정규화를 적용하면 테이블 개수가 급증하여 조회 시 과도한 조인(Join) 연산과 디스크 I/O 병목을 유발한다.
- **제언**: 설계 초기 단계에는 FDD에 입각하여 완벽한 논리 모델(BCNF)을 확보하고, 이후 트랜잭션 집중도와 쿼리 패턴을 분석하여 의도된 반정규화·파티셔닝·인덱스 튜닝을 조화롭게 배치하는 아키텍처 리더십이 요구된다.

**FDD 산출물 부재로 인한 설계 위험**
- **챌린지**: FDD를 공식 설계 산출물로 관리하지 않으면 정규화 근거가 불명확하여 유지보수 시 이상현상 재발 위험이 증가한다.
- **제언**: FDD를 논리 설계의 필수 산출물로 의무화하고, ERwin·DA#과 같은 CASE 도구의 종속성 자동 탐지 기능을 활용하여 정규화 준수 여부를 정기적으로 검증하는 거버넌스를 구축해야 한다.

**BCNF와 3NF의 선택적 적용**
- **챌린지**: BCNF는 3NF보다 강한 제약이지만, 특정 케이스에서는 BCNF 분해 후 함수 종속성 보존(Dependency Preservation)이 깨지는 경우가 발생한다.
- **제언**: 함수 종속성 보존이 중요한 업무 영역에서는 BCNF 대신 3NF를 목표로 하고, BCNF 적용 시에는 분해 후 원래 FD가 모두 보존되는지 반드시 검증해야 한다.
