# 데이터베이스 정규화와 함수적 종속성 완전 해체

---

## 1. 문제를 해체하는 5대 핵심 분석 항목

### ① 출제 배경 및 의도 (Why Now?)

* **데이터품질 저하의 원천 차단:** 실무 환경에서 부적절하게 설계된 테이블은 데이터의 중복을 유발하고, 이로 인해 삽입·삭제·수정 시 정보가 왜곡되는 '이상 현상(Anomaly)'이 발생합니다.
* **단계적 아키텍처 역량 검증:** 함수적 종속성 다이어그램(FDD)은 복잡하게 얽힌 속성 간의 인과관계를 가시화하는 강력한 도구입니다. 제2정규형, 제3정규형, BCNF로 이행하는 수학적·공학적 분해 과정을 직관적인 다이어그램을 통해 명확히 이해하고 설계에 적용할 수 있는지 평가하는 의도입니다.

### ② 개념의 본질과 작동 메커니즘 (What & How)

* **한 문장 정의:** 함수적 종속성 분석을 기반으로 속성 간의 종속 관계를 분해(Projection)함으로써 중복성을 제거하고, 이상 현상을 원천적으로 방지하는 점진적 스키마 최적화 프로세스입니다.
* **작동 메커니즘 (Mechanism):**
정규화되지 않은 릴레이션(Input)에 존재하는 복합키의 일부 종속성(부분 FD), 일반 속성 간의 종속성(이행 FD), 후보키가 아닌 결정자(BCNF 대상)를 순차적으로 식별해 냅니다. 이후 무손실 분해(Lossless Decomposition) 규칙에 따라 독립된 릴레이션(Output)들로 쪼개어 무결성을 달성합니다.

### ③ 채점관이 눈을 켜고 찾는 'Must-Have' 키워드 (Keywords)

* **이상 현상 관점:** 삽입 이상(Insertion), 삭제 이상(Deletion), 갱신/수정 이상(Update), 데이터 중복, 정보 손실.
* **정규형 및 다이어그램 관점:** 결정자(Determinant), 종속자(Dependent), 부분 함수 종속(Partial FD), 완전 함수 종속(Full FD), 이행적 함수 종속(Transitive FD), 후보키(Candidate Key), 무손실 분해.

### ④ 상위/하위/유사 개념과의 상호 관계 (Context & Comparison)

정규화 단계는 하위 정규형의 제약조건을 만족해야만 다음 상위 정규형으로 진급할 수 있는 누적적(Cumulative) 포함 관계를 가집니다.

* **1NF:** 모든 속성의 도메인이 원자값(Atomic Value)을 만족하는 단계.
* **2NF:** 기본키의 진부분집합이 일반 속성을 결정하는 '부분 함수 종속'을 제거하여 완전 함수 종속화하는 단계.
* **3NF:** 주식별자를 거쳐 간접적으로 결정되는 '이행적 함수 종속'을 제거하여 속성 간 독립성을 확보하는 단계.
* **BCNF:** 모든 결정자가 후보키(Candidate Key)가 되도록 강제하여 변칙적 종속성을 해결하는 단계.

### ⑤ 실무적 제언 및 챌린지 (So What? / Insight)

* **과도한 분해에 따른 성능 저하:** 이론에만 치우쳐 BCNF 이상으로 정규화를 적용하면 테이블 개수가 급증하게 됩니다. 이는 조회(Select) 요청 시 과도한 조인(Join) 연산과 디스크 I/O 병목을 유발합니다.
* **기술사의 제언:** 실무 환경이나 데이터 웨어하우스(DW) 설계 시에는 정규화 모델을 먼저 완벽하게 구축한 뒤, 애플리케이션의 트랜잭션 집중도와 쿼리 패턴을 분석하여 '의도된 반정규화(Denormalization)'나 테이블 파티셔닝(Partitioning), 인덱스 튜닝을 조화롭게 배치하는 아키텍처 리더십이 요구됩니다.

## 2. 합격자 수준의 모범 답안 가이드 (2교시형 서술형)

### [가] 가. 이상 현상(Anomaly)의 개념 및 사례 기반 유형 분석

#### 1. 이상 현상(Anomaly)의 정의

* 데이터베이스 설계 시 선행되어야 할 함수적 종속성 분석을 간과하여 테이블에 중복 데이터가 집중될 때, 릴레이션 조작(DML) 시 데이터 정합성이 깨지고 모순이 발생하는 불일치 현상입니다.

#### 2. 구체적 사례 중심의 이상 현상 3대 유형

##### ① 삽입 이상 (Insertion Anomaly)

* **개념:** 데이터를 테이블에 추가할 때, 불필요하거나 원하지 않는 정보까지 함께 입력해야 하거나 주식별자(PK) 제약(Null 불가)으로 인해 삽입이 불가능해지는 현상입니다.
* **사례:** 수강 테이블에 `(학번, 과목번호)`가 복합 기본키로 지정되어 있을 때, 아직 어떤 과목도 신청하지 않은 신입생 고유 정보(학번, 이름)를 등록하려면 `과목번호` 자리에 가짜 값이나 가상 데이터를 채워 넣어야만 삽입이 가능해집니다.

##### ② 삭제 이상 (Deletion Anomaly)

* **개념:** 테이블의 특정 튜플을 삭제할 때, 업무적으로 유지되어야 하는 유용한 정보까지 연쇄적으로 함께 소실되는 정보의 유실 현상입니다.
* **사례:** 특정 학생이 수강을 취소하여 해당 수강 튜플을 삭제했더니, 그 테이블에 유일하게 존재하던 해당 과목의 `담당교수`나 `강의실 위치` 정보까지 시스템에서 통째로 사라져 버리는 현상입니다.

##### ③ 갱신 이상 (Update Anomaly)

* **개념:** 튜플의 특정 속성 값을 수정할 때, 중복 저장된 수많은 데이터 중 일부만 변경되어 데이터의 일관성이 무너지고 모순(정보 왜곡)이 발생하는 현상입니다.
* **사례:** 학번 '26-001' 학생이 여러 과목을 신청하여 이름과 전화번호가 복합키 개수만큼 중복 저장되어 있을 때, 해당 학생의 `전화번호`가 변경되어 1개의 행만 수정하고 나머지를 누락하면 동일 인물의 연락처가 행마다 다르게 표현되는 모순이 발생합니다.

### [나] 나. 제2정규형, 제3정규형, BC 정규형의 함수적 종속성 다이어그램(FDD) 해설

각 정규형의 대상이 되는 테이블 구조적 결함 사례와 이를 함수적 종속성 다이어그램(FDD)을 통해 해결하는 무손실 분해 프로세스입니다.

#### 1. 제2정규형 (2nd Normal Form)

* **적용 대상:** 제1정규형을 충족하면서, 복합 기본키의 일부분에 일반 속성이 종속되는 **부분 함수 종속성**이 존재하는 상태입니다.
* **사례 명제:** 기본키가 `(학번, 과목번호)`이며, `성적`은 복합키 전체에 종속되나 `지도교수`는 `학번`에 의해서만 결정되는 구조입니다.

```text
[분해 전 FDD: 부분 함수 종속 존재]
┌───────────────┐
│  학번  , 과목번호 │ ──┐
└───────────────┘   │ (완전 함수 종속)
    │               ▼
    │            ┌──────┐
    │            │ 성적 │
    │            └──────┘
    ▼ (부분 함수 종속)
┌──────────┐
│ 지도교수 │
└──────────┘

       ▼ 무손실 분해 (Projection) 적용

[분해 후 2NF 충족 상태]
- 릴레이션 1: 학생지도 ( 학번, 지도교수 )  -> 학번이 결정자
- 릴레이션 2: 수강성적 ( 학번, 과목번호, 성적 ) -> (학번, 과목번호)가 결정자

```

#### 2. 제3정규형 (3rd Normal Form)

* **적용 대상:** 제2정규형을 충족하면서, 일반 속성이 또 다른 일반 속성을 결정하는 주객전도 형태의 **이행적 함수 종속성**이 존재하는 상태입니다.
* **사례 명제:** 기본키인 `학번`이 `학과코드`를 결정하고, 일반 속성인 `학과코드`가 다시 `학과명`을 결정하여 건너뛰는 형태의 종속 구조입니다.

```text
[분해 전 FDD: 이행적 함수 종속 존재]
┌──────┐            ┌──────────┐            ┌────────┐
│ 학번 │ ─────────> │ 학과코드 │ ─────────> │ 학과명 │
└──────┘            └──────────┘            └────────┘
   │                                           ▲
   └───────────────────────────────────────────┘
                 (이행적 함수 종속 발생)

       ▼ 무손실 분해 (Projection) 적용

[분해 후 3NF 충족 상태]
- 릴레이션 1: 학생학과 ( 학번, 학과코드 ) -> 학번이 결정자
- 릴레이션 2: 학과정보 ( 학과코드, 학과명 ) -> 학과코드가 결정자 (일반 속성의 분리)

```

#### 3. 보이스-코드 정규형 (Boyce-Codd Normal Form, BCNF)

* **적용 대상:** 제3정규형을 충족하면서, 복합키로 구성된 테이블에서 후보키가 아닌 일반 속성이 기본키의 일부 속성을 지배하는 **결정자적 함수 종속성**이 존재하는 상태입니다.
* **사례 명제:** 학생이 과목별로 한 명의 교수에게만 튜터링을 받으므로 기본키가 `(학번, 과목명)`인데, 일반 속성인 `담당교수`가 거꾸로 `과목명`을 결정해 버리는 변칙 구조입니다.

```text
[분해 전 FDD: 후보키가 아닌 결정자 존재]
┌───────────────┐
│  학번  , 과목명   │ ──┐
└───────────────┘   │
    ▲               ▼
    │           ┌──────────┐
    │           │ 담당교수 │
    │           └──────────┘
    └─────────────────┘ 
       (담당교수가 과목명을 결정함: 결정자 역전)

       ▼ 무손실 분해 (Projection) 적용

[분해 후 BCNF 충족 상태]
- 릴레이션 1: 교수과목 ( 담당교수, 과목명 ) -> 일반 속성이던 담당교수가 당당히 PK로 승격
- 릴레이션 2: 학생교수 ( 학번, 담당교수 )   -> 학번과 교수 매핑을 통해 관계 유지

```

### [다] 기술사의 시각: 실무 데이터 아키텍처 관점에서의 정규화 가이드

#### 1. 정규화와 시스템 성능 간의 Trade-off 통제

* 정규화 단계를 BCNF 수준까지 엄격하게 전개하면 데이터의 이상 현상과 중복 저장 비용은 제로(Zero)에 수렴하게 됩니다.
* 반면, 대규모 트랜잭션 중심의 OLTP 시스템이나 대량 조회가 핵심인 OLAP 환경에서는 수많은 테이블 분해로 인해 고비용의 쿼리 조인 연산이 발생하여 CPU 및 메모리 병목 현상이 일어납니다.

#### 2. 아키텍처적 극복 전략 및 제언

* **선(先)정규화, 후(後)조율 원칙:** 설계 초기 단계에는 암스트롱 공리와 함수적 종속성 다이어그램에 입각하여 완벽한 '논리 모델(BCNF)'을 확보해야 합니다. 처음부터 성능을 핑계로 정규화를 생략하면 데이터 모델이 왜곡됩니다.
* **통제된 반정규화(Denormalization) 프로세스:** 성능 한계가 도출되는 쿼리에 한해, 중복 컬럼 추가나 테이블 통합을 진행하되, 데이터 사전(Data Dictionary)에 반정규화 이력을 명시하고 애플리케이션 트리거나 배치 프로시저를 통해 정합성 검증 로직을 보완하는 통제된 거버넌스를 정착시키는 것이 최고 수준의 차별화 덤프 전략입니다.
