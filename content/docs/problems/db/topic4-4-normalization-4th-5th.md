## 해설

> **BCNF를 넘어선 고차 정규화와 연결함정**  
> 다치 종속(MVD)→4NF, 조인 종속(JD)→5NF, 관계 설계 오류→연결함정

본 문제는 데이터 모델링 설계 중 발생할 수 있는 치명적인 논리적 오류인 연결함정(Connection Trap)과, 전통적인 BCNF를 넘어 다치 종속(MVD, Multi-Valued Dependency) 및 조인 종속(JD, Join Dependency)을 다루는 고차 정규화(4~5차)의 메커니즘을 융합하여 묻고 있습니다. 정보시스템 감리나 전사 데이터 거버넌스 정비 시, 테이블 간의 관계 설정 오류로 인해 엉뚱한 데이터가 유도되는 현상을 잡아내야 합니다.

### 1. 출제 배경 및 의도

채점관은 연결함정의 두 가지 유형(무부착 함정, 균열 함정)과 고차 정규화(4NF, 5NF)의 대상 종속성 및 분해 메커니즘을 명확히 구분하여 설명할 수 있는지를 평가합니다. 특히 4NF와 5NF의 분해 테이블 수(각각 2개, 3개 이상)의 차이가 핵심 키워드입니다.

### 2. 개념의 본질과 작동 메커니즘

관계형 모델에서 테이블 간 조인(Join) 관계를 부적절하게 설계하여 가짜 정보가 유도되는 연결함정을 식별하고, 1:M 구조를 넘어 다대다(M:N) 관계망 속 숨겨진 종속성을 분해하는 고차 정규화(4NF, 5NF) 프로세스입니다.

삼차원 이상의 관계성(예: 대리점-제품-공급업체)이 얽혀있을 때, 단순히 선형적인 1:M 관계로만 연결하면 정보의 왜곡(연결함정)이나 불필요한 행의 중복이 나타납니다. 이를 **다치 종속(MVD)** 및 **조인 종속(JD)** 관점에서 다원 분해하여 독립된 테이블로 분리합니다.

#### 연결함정(Connection Trap)의 2대 유형

**1. 무부착 함정 (Fan Trap)**

- **개념:** 한 엔티티가 다른 두 엔티티와 각각 1:M 관계를 맺고 있어, 부모를 공유하는 두 자식 엔티티 간의 관계가 모호해지는 현상 (구조: `자식A ← 1:M → 부모 ← 1:M → 자식B`)
- **사례:** 하나의 `교수`가 여러 `학생`을 지도하고, 동일한 `교수`가 여러 `학과`에 소속되어 있을 때, 특정 학생이 어떤 학과 소속인지 조인을 통해 명확히 추론할 수 없는 경로 단절이 발생
- **해결책:** `학과 (1) → 교수 (M) → 학생 (M)` 형태로 관계의 순서를 구조적으로 재배치

**2. 균열 함정 (Chasm Trap)**

- **개념:** 엔티티 간에 관계는 정의되어 있으나 일부 튜플 사이에 관계 선로가 끊어져(선택적 관계, Optional), 필요한 경로를 찾아가지 못하는 현상
- **사례:** `학과 (1) → 교수 (M) → 학생 (M)` 구조에서 특정 교수가 아직 어떤 학과에도 배정되지 않은 임시 교수(선택적 관계)일 경우, 그 교수 밑에 있는 학생이 어느 학과 소속인지 추론할 수 없는 균열 발생
- **해결책:** `학과`와 `학생` 간에 직접적인 관계(`학과 → 학생`)를 추가로 개설하여 미아 경로를 차단

#### 제4정규화(4NF) — 다치 종속(MVD) 제거

**다치 종속(Multi-Valued Dependency, MVD):** 하나의 결정자 값에 대해 종속자의 값이 두 개 이상(다치) 결정되는 제약조건. 기호로는 X ↠ Y로 표기. 테이블 내에 아무 관계가 없는 독립적인 다치 속성들이 공존할 때 중복이 폭발적으로 증가함.

**사례:** 한 과목은 여러 교재를 사용하고, 여러 취미를 가진 학생이 들을 수 있을 때, 교재와 취미는 아무런 상관관계가 없음.

```text
[원시 릴레이션 (BCNF 상태이나 중복 발생)]
| 과목명 (PK) | 교재명 (PK) | 취미 (PK) |
| :--- | :--- | :--- |
| 데이터베이스 | DB개론 | 등산 |
| 데이터베이스 | DB개론 | 독서 | (교재는 같으나 취미 조합 때문에 중복 행 생성)
| 데이터베이스 | SQL완성 | 등산 |
| 데이터베이스 | SQL완성 | 독서 |
```

**4NF 해결방안:** 다치 종속 관계를 맺고 있는 두 독립적 속성을 별도의 테이블로 프로젝션(Projection)하여 제4정규형(4NF)을 달성합니다.

```text
[4NF 충족 테이블 재구성]
- 릴레이션 1 <과목교재>: ( 과목명, 교재명 )
- 릴레이션 2 <과목취미>: ( 과목명, 취미 )
```

#### 제5정규화(5NF) — 조인 종속(JD) 제거

**조인 종속(Join Dependency, JD):** 하나의 릴레이션을 여러 개의 테이블로 분해했다가 다시 내추럴 조인(Natural Join)했을 때, 원래의 릴레이션과 똑같은 결과가 나와야 하는 무손실 분해 성질을 확장한 개념. 4NF를 만족하면서, 릴레이션이 자신의 프로젝션(분해 테이블)들을 모두 조인한 결과와 언제나 같아야 성립함.

**사례:** 공급자-제품-소비자의 삼항 관계에서, 만약 이를 2개의 테이블로만 쪼개면 조인 시 존재하지 않던 가짜 행(Spurious Tuple)이 유도되는 투플 생성 이상이 발생합니다.

**5NF 해결방안:** 조인 종속성을 해결하기 위해 **3개의 대칭적 릴레이션으로 n-분해**하여 제5정규형(5NF)을 달성합니다.

```text
[5NF 충족 3원 분해 결과]
- 릴레이션 1 <공급제품>: ( 공급자, 제품 )
- 릴레이션 2 <제품소비>: ( 제품, 소비자 )
- 릴레이션 3 <공급소비>: ( 공급자, 소비자 )
```

### 3. 핵심 키워드 (Must-Have)

- **연결함정 관점:** 무부착 함정(Fan Trap), 균열 함정(Chasm Trap), 미아 경로, 추론 오류, 가짜 튜플(Spurious Tuple)
- **4차 정규화 관점:** BCNF 만족, 다치 종속(Multi-Valued Dependency, MVD), 독립적인 다대다 관계, X ↠ Y
- **5차 정규화 관점:** 4NF 만족, 조인 종속(Join Dependency, JD), 투플 생성 이상, 무손실 분해, n-분해(n-Decomposition)

### 4. 맥락과 비교

고차 정규화는 일반적인 기본키/후보키 기반의 함수 종속성(FD) 단계를 넘어, 속성 군(Group) 간의 다차원적 결합 구조를 다룹니다.

| 구분 | 대상 종속성 | 분해 방식 및 특징 | 달성 목적 |
| --- | --- | --- | --- |
| **BCNF** | 함수 종속성 (FD) | 모든 결정자가 후보키가 되도록 2개로 분해 | 기본키 패러다임의 완성 |
| **4차 정규화 (4NF)** | 다치 종속 (MVD) | 한 테이블 내 공존하는 독립적 다치 속성을 2개로 이원 분해 | 무의미한 조합 중복 제거 |
| **5차 정규화 (5NF)** | 조인 종속 (JD) | 2개로 분해 시 정보가 왜곡되므로, 3개 이상의 테이블로 n-분해 | 순환 관계 구조의 무결성 확보 |

### 5. 실무 제언

**이론적 엄격성과 실무적 효용성의 괴리**

- **챌린지:** 실무 프로젝트에서 4차, 5차 정규화까지 도달하는 경우는 매우 드뭅니다. 과도하게 테이블이 쪼개지면 SQL 작성 난이도가 극상으로 올라가고 복잡한 다중 조인으로 인해 심각한 성능 저하가 발생하기 때문입니다.
- **제언:** 기술사는 ERD 설계 단계에서 연결함정을 조기에 식별하여 비즈니스 관계의 왜곡을 막는 데 주력해야 합니다. 4~5차 정규화는 기계적으로 적용하기보다 공통 매핑 테이블(Intersection Table) 설계 패턴을 바르게 정착시키고 데이터 표준 가이드라인을 제시하는 도구로 가치 있게 활용해야 합니다. 공학적으로 4NF와 5NF는 중복을 제로화하는 이상적인 구조이지만, 현실 세계의 대규모 MSA나 OLTP 환경에서는 수많은 조인 연산을 유발하므로, 다치 속성들의 데이터 변경 빈도를 모니터링하여 변경이 거의 없고 조회만 발생하는 영역이라면 JSON 데이터 타입(반정형) 구조로 통합 저장하거나 인덱스 아키텍처 다각화를 통해 해결하는 유연한 성능 중심의 물리 모델링 최적화 능력을 발휘해야 합니다.
