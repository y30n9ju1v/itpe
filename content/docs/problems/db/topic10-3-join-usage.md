## 해설

> **정규화로 분리된 테이블을 비즈니스 요건에 맞게 결합하는 관계 대수 연산**  
> SQL JOIN의 논리적 유형과 실무 활용

관계형 데이터베이스의 핵심 설계 원칙은 정규화입니다. 이상현상(데이터 중복·삭제·갱신 오류)을 방지하기 위해 분리된 엔티티들을 비즈니스 화면이나 API 요건에 맞게 결합하는 표준 SQL 구문이 JOIN입니다. 데이터 아키텍트는 각 JOIN 유형의 특성과 성능적 영향을 이해하고 적재적소에 활용할 수 있어야 합니다.

### 1. 출제 배경 및 의도

관계형 데이터베이스 아키텍처의 근간은 정규화입니다. 쪼개진 엔티티들을 비즈니스 서비스 요건에 맞게 결합하는 구문이 JOIN이며, 클라우드·MSA 환경에서도 데이터 정합성을 유지하면서 고성능 조회를 달성하려면 표준 SQL JOIN 메커니즘을 완벽히 다룰 줄 알아야 합니다.

채점관은 단순 쿼리 작성자를 넘어, 각 JOIN 유형의 특징과 성능적 영향을 파악하여 적재적소에 배치하는 실무형 데이터 아키텍트(DA)를 선별하고자 본 문제를 출제합니다.

### 2. 개념의 본질과 작동 메커니즘

관계형 데이터베이스에서 두 개 이상의 릴레이션(테이블)을 공통 속성(연관 관계 키)을 매개체로 결합하여 하나의 통합된 결과 레코드 세트를 생성하는 논리적 대수 연산 및 질의 기법입니다.

작동 메커니즘은 다음과 같습니다. 조인 조건(`ON`/`WHERE`)으로 연결된 둘 이상의 원시 테이블을 입력받아, 연산 형태에 따라 교차곱을 형성한 뒤 등가 조건, 범위 조건, 또는 외부 참조 NULL 매핑 조건을 필터링 처리합니다. 결과로 비즈니스 요구사항에 최적화된 새로운 가상 릴레이션(Result Set)을 반환합니다.

ANSI SQL 표준에서는 `ON` 절로 조인 조건을 명시하거나, 동명 컬럼이 있는 경우 `USING` 절을 사용하거나, `NATURAL JOIN`으로 자동 매핑하는 방법을 제공합니다.

### 3. 핵심 키워드 (Must-Have)

#### 1) 내부 조인 (Inner Join)

- **본질**: 조인 조건(`ON` 또는 `WHERE`)에 명시된 공통 결합키 값이 양쪽 테이블에 모두 존재하는 행들만 매치시켜 교집합 형태로 반환
- **Must-Have 키워드**: 교집합, 조인 순서 자유, 옵티마이저 최적화 유리
- **활용 사례**: 주문과 결제가 모두 존재하는 완성된 거래 건 추출

```sql
SELECT o.주문번호, o.주문상품명, p.결제금액, p.결제일자
FROM 주문 o INNER JOIN 결제 p ON o.주문번호 = p.주문번호;
```

#### 2) 외부 조인 (Outer Join)

- **본질**: 조인 조건에 불일치하는 행도 기준 방향의 테이블 데이터를 보존하고, 상대 테이블 값은 NULL 처리하여 반환
- **Must-Have 키워드**: LEFT/RIGHT/FULL OUTER JOIN, NULL 매핑, 조인 순서 고정
- **활용 사례**: 전체 고객 명단에서 이번 달 구매 이력이 없는 고객도 포함한 매출 현황 분석

```sql
SELECT c.고객ID, c.고객명, NVL(s.구매금액, 0) AS 이번달매출
FROM 고객 c LEFT OUTER JOIN 월별매출 s
ON c.고객ID = s.고객ID AND s.매출년월 = '202606';
```

#### 3) 교차 조인 (Cross Join / Cartesian Product)

- **본질**: 조인 조건 없이 양쪽 테이블의 모든 행을 1:1로 전수 조합하여 M×N 카디널리티 생성
- **Must-Have 키워드**: 카디널리티 폭발, 대용량 시스템 기피 대상

#### 4) 셀프 조인 (Self Join)

- **본질**: 단일 테이블에 별칭(Alias)을 부여하여 자기 자신과 조인하는 특수 기법
- **Must-Have 키워드**: 순환 관계 모델, 계층 구조 추적, Alias

```sql
SELECT e.사원명 AS 직원, m.사원명 AS 직속상사
FROM 사원 e LEFT OUTER JOIN 사원 m ON e.관리자사번 = m.사번;
```

#### 5) 등가/비등가 조인 (Equi/Non-Equi Join)

- **본질**: 조인 조건의 연산자 종류에 따른 분류 — `=` 기호 기반의 등가 조인, `>·<·BETWEEN` 연산자를 활용하는 비등가 조인
- **Must-Have 키워드**: 구간 매핑, 범위 조건, BETWEEN

```sql
SELECT c.고객명, c.누적포인트, g.등급명
FROM 고객 c INNER JOIN 등급기준표 g
ON c.누적포인트 BETWEEN g.시작포인트 AND g.종료포인트;
```

### 4. 맥락과 비교

SQL 표준 JOIN 유형들은 데이터 결합 범위와 매핑 규칙에 따라 벤 다이어그램(Venn Diagram) 형태로 상호 관계성을 가집니다.

| 비교 항목 | 내부 조인 (Inner Join) | 외부 조인 (Outer Join) | 교차 조인 (Cross Join) |
|-----------|----------------------|----------------------|----------------------|
| **결합 데이터 범위** | 양쪽 테이블 모두에 **일치하는 행만** 추출 | 조건 불일치 행도 **한쪽 테이블 기준 보존** | 양쪽 테이블의 **모든 행을 1:1 전수 조합** |
| **데이터 왜곡/누락 위험** | 일치하지 않는 실물 데이터 누락 위험 | 불필요한 NULL 값 유입 및 중복 데이터 유도 | 카디널리티의 폭발적 증가 (M×N) |
| **성능적 특징 (옵티마이저)** | 조인 순서 변경이 자유로워 최적화 용이 | 조인 순서가 고정되어 비효율 유발 가능 | 실행 비용 극대화로 시스템 병목 주범 |
| **매핑 벤다이어그램** | 두 집합의 **교집합** | **여집합을 포함한 단방향/양방향 합집합** | 두 집합의 **공간적 평면 결합** |

### 5. 실무 제언

**Outer Join 오용에 따른 성능 대참사**

- **챌린지**: 실무 개발 현장에서 가장 빈번한 안티 패턴 중 하나는 데이터 유실을 두려워하여 모든 쿼리에 `LEFT OUTER JOIN`을 배치하는 것입니다. Outer Join은 DBMS 옵티마이저가 수립할 수 있는 조인 순서(Join Ordering)의 선택지를 박탈하고 강제로 선후 관계를 묶어 버리므로 최적의 인덱스 검색 실행 계획을 방해합니다.
- **제언**: 데이터 모델링 단계에서 필수적 관계(Mandatory)를 유도하여 최대한 `INNER JOIN` 구조로 설계하되, 비즈니스상 불가피한 외부 조인 영역은 필터 조건을 하부 인라인 뷰로 전진 배치(Push-down Selection)하여 성능 리스크를 능동적으로 통제해야 합니다.

**MSA 환경에서의 조인 거버넌스**

- **챌린지**: 마이크로서비스 아키텍처(MSA) 환경에서는 데이터베이스가 서비스 경계별로 파편화되어 물리적 네트워크 단절로 인해 표준 SQL 조인이 원천적으로 불가능해지는 한계에 직면합니다.
- **제언**: OLTP 화면 조회 시에는 각 도메인 마이크로서비스로부터 API로 데이터를 수신한 뒤 BFF(Backend For Frontend) 레이어의 메모리 단에서 조인을 처리하는 디커플링 전략이 필요합니다. 대용량 분석 통계성 트랜잭션은 CQRS 아키텍처를 도입하여 Elasticsearch나 NoSQL 도큐먼트 스토어에 역정규화된 Read-Model을 사전 구축해 두는 패러다임 시프트가 필요합니다.
