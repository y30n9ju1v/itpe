## 해설

### 출제 배경 및 의도

관계형 데이터베이스의 핵심 설계 원칙인 정규화로 인해 엔티티가 분리되며, 비즈니스 화면·API 요건에 맞게 이를 결합하는 표준 SQL 구문이 JOIN입니다. 데이터 규모가 TB·PB 수준으로 증가하는 현대 환경에서 전체 쿼리 성능의 상당 부분은 JOIN 유형 선택과 인덱스 설계에 의해 결정됩니다.

채점관은 단순 쿼리 작성자를 넘어, 각 JOIN 유형의 특징과 성능적 영향을 파악하여 적재적소에 배치하는 실무형 데이터 아키텍트(DA) 역량을 선별하고자 출제합니다. Outer Join 오용 패턴, MSA 환경에서의 조인 거버넌스, Push-down Selection 등 실무 최적화 사례 제시 가능 여부가 핵심 채점 포인트입니다.

### 1. SQL JOIN, 개요

정 의  • 두 개 이상의 릴레이션(테이블)을 공통 속성(연관 관계 키)을 매개체로 결합하여 하나의 통합된 결과 레코드 세트를 생성하는 논리적 대수 연산 및 질의 기법
       - 조인 조건(`ON`/`WHERE`)으로 연결된 원시 테이블을 입력받아, 교차곱 후 등가·범위·외부 참조 NULL 매핑 조건으로 필터링하여 새로운 가상 릴레이션(Result Set) 반환

```
[SQL JOIN 논리 유형 구조도]

  테이블 A (고객)    테이블 B (주문)
  ┌──────────┐      ┌──────────┐
  │  A only  │──────│  A∩B     │──────│  B only  │
  └──────────┘      └──────────┘      └──────────┘

  INNER JOIN  : A ∩ B (양쪽 일치 교집합만)
  LEFT OUTER  : A 전체 + B 매칭 (B 없으면 NULL)
  RIGHT OUTER : B 전체 + A 매칭 (A 없으면 NULL)
  FULL OUTER  : A ∪ B (전체 합집합, 불일치 시 NULL)
  CROSS JOIN  : A × B (모든 행 전수 조합, M×N 카디널리티)
  SELF JOIN   : 단일 테이블 Alias로 자기 자신과 조인 (계층 구조)
```

- JOIN 유형 선택은 비즈니스 데이터 결합 범위와 NULL 허용 여부로 결정된다.

### 2. SQL 표준 JOIN 유형 및 활용 사례

1) 5대 JOIN 유형 비교

| JOIN 유형 | 결합 범위 | 활용 사례 | 성능 특징 |
|----------|---------|---------|---------|
| **Inner Join** | 양쪽 일치 행만 (교집합) | 주문과 결제가 모두 존재하는 완성 거래 조회 | 조인 순서 자유, 옵티마이저 최적화 용이 |
| **Left Outer Join** | 왼쪽 전체 + 오른쪽 매칭 (없으면 NULL) | 구매 이력 없는 고객도 포함한 매출 현황 | 조인 순서 고정, 최적화 제한 |
| **Full Outer Join** | 양쪽 전체 보존 (불일치 시 NULL) | 두 집합의 완전한 합집합 조회 | 비용 극대 |
| **Cross Join** | 조건 없이 모든 행 조합 (M×N) | 달력·좌석 배치 등 전수 조합 생성 | 카디널리티 폭발, 성능 주의 |
| **Self Join** | 단일 테이블 자기 자신과 조인 | 조직도·계층 구조·순환 관계 모델 | Alias 필수 |

- Outer Join 남용은 옵티마이저 최적화를 제한하므로 Inner Join 우선 설계가 원칙이다.

2) 등가/비등가 조인 및 SQL 활용 사례

| 구분 | 조인 조건 | 활용 사례 | SQL 예시 |
|------|---------|---------|---------|
| **Equi Join** | `=` 등가 조건 | 주문번호로 주문-결제 결합 | `ON o.주문번호 = p.주문번호` |
| **Non-Equi Join** | `BETWEEN`, `>`, `<` 범위 조건 | 누적 포인트로 등급 구간 매핑 | `ON c.포인트 BETWEEN g.시작 AND g.종료` |
| **Self Join** | 동일 테이블 Alias 부여 | 사원-상사 계층 구조 추적 | `FROM 사원 e LEFT JOIN 사원 m ON e.관리자번호 = m.사번` |

- 비등가 조인은 Sort Merge Join이 유리하며, 구간 매핑 비즈니스 로직에 활용된다.

### 3. JOIN 유형 비교 및 현대 환경 트렌드

| 비교 항목 | Inner Join | Outer Join | Cross Join |
|----------|-----------|-----------|-----------|
| **결합 데이터 범위** | 양쪽 일치 행만 | 기준 테이블 전체 보존 | 모든 행 전수 조합 |
| **NULL 유입** | 없음 | 불일치 시 NULL 매핑 | 없음 |
| **옵티마이저 자유도** | 조인 순서 변경 자유 | 조인 순서 고정 | 실행 비용 극대 |
| **MSA 대응** | BFF 레이어 애플리케이션 조인으로 대체 가능 | 동일 | 사용 자제 |

- 현대 MSA 환경에서는 물리 DB 분리로 SQL JOIN이 불가능해져, BFF 레이어 애플리케이션 사이드 조인과 CQRS 역정규화 Read-Model 구축으로 패러다임이 전환되고 있다.  "끝"

### 실무 제언

**Outer Join 오용 방지와 Inner Join 우선 설계**
- **챌린지**: 데이터 유실을 우려하여 모든 쿼리에 `LEFT OUTER JOIN`을 배치하면 옵티마이저가 조인 순서를 자유롭게 변경할 수 없어 최적 실행 계획 수립이 불가능해지고 성능이 저하된다.
- **제언**: 데이터 모델링 단계에서 필수적 관계(Mandatory)를 유도하여 최대한 `INNER JOIN` 구조로 설계하고, 불가피한 Outer Join은 필터 조건을 하부 인라인 뷰로 전진 배치(Push-down Selection)하여 성능 리스크를 능동적으로 통제해야 한다.

**MSA 환경에서의 조인 거버넌스**
- **챌린지**: MSA 환경에서 물리 DB가 서비스 경계별로 분리되면 표준 SQL JOIN이 원천적으로 불가능해지며, DB 레이어 JOIN에 의존하는 설계는 모놀리식으로의 회귀를 강요한다.
- **제언**: OLTP 화면 조회 시 BFF(Backend For Frontend) 레이어에서 애플리케이션 사이드 조인을 처리하고, 분석성 대용량 쿼리는 CQRS 아키텍처로 Kafka·CDC를 통해 역정규화된 Read-Model을 사전 구축하는 패러다임 시프트가 필요하다.

**비등가 조인 구간 매핑 설계**
- **챌린지**: 포인트 등급·요금 구간 등 비등가 조인이 필요한 도메인에서 BETWEEN 조건 사용 시 인덱스 Range Scan 범위 설정이 잘못되면 Full Table Scan이 발생한다.
- **제언**: 비등가 조인 대상 컬럼에는 Sort Merge Join이 유리하며, 구간 기준 테이블을 소량으로 유지하고 Sort Area 설정을 최적화하거나 구간 코드 정규화로 등가 조인으로 전환하는 설계를 우선 검토해야 한다.
