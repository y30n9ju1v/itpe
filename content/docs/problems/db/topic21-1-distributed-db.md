## 해설

> **지리적으로 분산된 데이터베이스를 하나의 DB처럼 사용하게 하는 투명성 기술**  
> 분산 투명성 8가지로 복잡성을 은닉하고, 2PC 프로토콜로 분산 트랜잭션의 원자성을 보장

분산 데이터베이스는 물리적으로 여러 노드에 분산된 데이터를 논리적으로 하나의 데이터베이스처럼 관리하는 시스템입니다. 핵심 가치는 "분산의 복잡성을 사용자로부터 숨기는 투명성"이며, 이를 위해 8가지 분산 투명성과 2PC 프로토콜 같은 분산 트랜잭션 메커니즘이 필요합니다.

### 1. 출제 배경 및 의도

클라우드·글로벌 서비스 시대에 멀티 리전 분산 DB는 선택이 아닌 필수가 되었습니다. Google Spanner, CockroachDB, AWS Aurora Global Database 등이 주목받으면서 분산 DB의 원리를 이해하는 것이 중요해졌습니다. 채점관은 분산 투명성의 종류와 의미, 분산 DB의 트레이드오프, 그리고 2PC의 동작 메커니즘과 한계를 명확히 설명할 수 있는지를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 분산 투명성 (Distribution Transparency)

분산 투명성은 분산 환경의 복잡성을 응용 프로그램과 사용자로부터 완전히 숨겨, 마치 단일 중앙집중식 DB를 사용하는 것처럼 느끼게 해주는 추상화 능력입니다.

```text
[분산 투명성의 은닉 계층]

사용자/애플리케이션
       │  (투명성 레이어: 분산의 복잡성 은닉)
       ▼
┌───────────────────────────────────────────────┐
│  분산 DBMS 미들웨어                            │
│  ┌─────────┐ ┌──────────┐ ┌────────────────┐  │
│  │위치투명성│ │복제투명성│ │ 단편화투명성   │  │
│  │장애투명성│ │병행투명성│ │ 이름투명성     │  │
│  │성능투명성│ │스케일투명성│               │  │
│  └─────────┘ └──────────┘ └────────────────┘  │
└───────────────────────────────────────────────┘
       │
  ┌────┼─────┐
  ▼    ▼     ▼
노드A  노드B  노드C   (물리적 분산 저장)
(서울) (도쿄) (LA)
```

**분산 투명성 8가지**

| 투명성 종류 | 영문 | 정의 | 예시 |
|------------|------|------|------|
| **위치 투명성** | Location Transparency | 데이터의 물리적 위치를 모르고 접근 가능 | 서울/도쿄 어느 서버에 있든 동일한 SQL 사용 |
| **복제 투명성** | Replication Transparency | 데이터 복제 여부를 모르고 접근 가능 | 복제본이 3개 있어도 단일 원본처럼 사용 |
| **단편화 투명성** | Fragmentation Transparency | 데이터 분할(수평/수직) 여부를 모르고 접근 가능 | 테이블이 여러 노드에 분산되어 있어도 전체 조회 가능 |
| **장애 투명성** | Failure Transparency | 일부 노드 장애 시에도 서비스 계속 가능 | 노드B 장애 시 노드A·C에서 자동 처리 |
| **병행 투명성** | Concurrency Transparency | 다수 트랜잭션 동시 수행 시 데이터 정합성 자동 유지 | 분산 잠금·타임스탬프로 충돌 자동 해결 |
| **이름 투명성** | Naming Transparency | 어느 노드에서나 동일한 이름으로 객체 접근 가능 | 테이블 이름이 노드마다 달라도 통합 이름 사용 |
| **성능 투명성** | Performance Transparency | 시스템 부하에 따라 자동으로 성능 최적화 | 쿼리를 최적 노드로 자동 라우팅 |
| **스케일 투명성** | Scaling Transparency | 노드 추가·제거 시 애플리케이션 변경 불필요 | 노드 증설해도 클라이언트 코드 수정 없음 |

#### 나. 분산 데이터베이스 장단점

| 구분 | 내용 |
|------|------|
| **장점** | |
| 신뢰성·가용성 향상 | 일부 노드 장애 시 다른 노드에서 서비스 계속 |
| 성능 향상 | 지역 데이터 접근으로 네트워크 지연 최소화 |
| 수평 확장 | 노드 추가로 용량·처리량 증가 |
| 자율성 | 각 지역 사이트가 로컬 데이터 자율 관리 |
| **단점** | |
| 복잡성 증가 | 분산 트랜잭션·장애 처리·일관성 유지 복잡 |
| 보안 취약점 | 다수 노드·네트워크 구간의 보안 관리 어려움 |
| 데이터 일관성 유지 비용 | 2PC 등 분산 합의 프로토콜의 오버헤드 |
| 설계·운영 비용 | 전문 인력 및 인프라 비용 증가 |

#### 다. 2PC (Two-Phase Commit)

2PC는 분산 트랜잭션에서 **모든 참여 노드가 동일하게 커밋 또는 롤백**되도록 보장하는 원자성 프로토콜입니다. 코디네이터(Coordinator)와 참여자(Participant)로 구성됩니다.

```text
[2PC 프로토콜 흐름]

코디네이터              참여자A           참여자B
    │                     │                  │
    │──── PREPARE ────────→│                  │
    │──── PREPARE ─────────────────────────→ │
    │                     │                  │
    │←── VOTE: YES ────── │  (준비 완료)      │
    │←── VOTE: YES ────────────────────────  │  (준비 완료)
    │                     │                  │
  모든 YES?              │                  │
  (YES면 Phase 2 커밋)   │                  │
    │                     │                  │
    │──── COMMIT ─────────→│                  │
    │──── COMMIT ──────────────────────────→ │
    │                     │                  │
    │←── ACK ─────────── │                  │
    │←── ACK ──────────────────────────────  │
    │                     │                  │
  트랜잭션 완료           │                  │

※ 어느 하나라도 VOTE: NO 또는 장애 시 → ROLLBACK 전송
```

**Phase 1: Prepare (투표 단계)**
- 코디네이터가 모든 참여자에게 PREPARE 메시지 전송
- 각 참여자는 트랜잭션 실행 가능 여부 확인 후 YES/NO 응답
- YES 응답 시 참여자는 임시 커밋 가능 상태(Redo Log 기록)로 대기

**Phase 2: Commit/Abort (결정 단계)**
- 모든 참여자가 YES → COMMIT 메시지 전송
- 하나라도 NO 또는 타임아웃 → ABORT 메시지 전송

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 분산 투명성 | 위치/복제/단편화/장애/병행/이름/성능/스케일 투명성 |
| 2PC 구성 | 코디네이터(Coordinator), 참여자(Participant) |
| 2PC 단계 | Phase 1 (Prepare/투표), Phase 2 (Commit/Abort) |
| 2PC 한계 | 블로킹 프로토콜, 코디네이터 SPOF, 네트워크 지연 |
| 분산 DB 특성 | 자율성, 수평 확장, 지역성, 복잡성 |
| 대안 프로토콜 | 3PC (Three-Phase Commit), Paxos, Raft, Saga 패턴 |

### 4. 맥락과 비교

| 비교 항목 | 2PC | 3PC | Saga 패턴 |
|----------|-----|-----|----------|
| 블로킹 여부 | 블로킹 (코디네이터 장애 시 참여자 대기) | 비블로킹 | 비블로킹 |
| 복잡도 | 중간 | 높음 | 높음 |
| 일관성 | 강한 일관성 (ACID) | 강한 일관성 | 최종적 일관성 |
| 성능 | 중간 (2번 네트워크 왕복) | 낮음 (3번 왕복) | 높음 (비동기) |
| 장애 내성 | 코디네이터 SPOF | 개선됨 | 분산 보상 트랜잭션 |
| 적합 환경 | 소규모 분산 트랜잭션 | 이론적 개선 | MSA 분산 트랜잭션 |

**단편화(Fragmentation) 유형**

| 유형 | 분할 방식 | 예시 |
|------|----------|------|
| 수평 단편화 (Horizontal) | 행(Row) 단위로 분할 (샤딩) | 고객 테이블을 지역별로 분할 |
| 수직 단편화 (Vertical) | 열(Column) 단위로 분할 | 자주 사용하는 컬럼과 드문 컬럼 분리 |
| 혼합 단편화 (Mixed) | 수평 + 수직 결합 | 컬럼 분리 후 각 파티션을 행 단위로 분산 |

### 5. 실무 제언

**2PC의 코디네이터 단일 장애점(SPOF) 해결**

- **챌린지:** 2PC에서 코디네이터가 Phase 1 이후 Phase 2 메시지를 보내기 전에 장애가 발생하면, 참여자들은 커밋/롤백 결정을 알 수 없어 트랜잭션을 보유 중인 상태로 무한 대기하는 블로킹(Blocking) 문제가 발생합니다. 이는 시스템 전체를 마비시키는 심각한 SPOF입니다.
- **제언:** MSA 환경에서는 2PC 대신 Saga 패턴(Choreography/Orchestration)을 적극 검토해야 합니다. Saga는 각 서비스가 로컬 트랜잭션을 실행하고, 실패 시 보상 트랜잭션(Compensating Transaction)을 역순으로 실행하여 일관성을 회복하는 방식으로 블로킹 없이 분산 트랜잭션을 처리합니다. 단, 최종적 일관성(Eventual Consistency) 모델임을 설계 단계부터 인지해야 합니다.

**분산 투명성과 성능 트레이드오프**

- **챌린지:** 분산 투명성이 높을수록 사용자 편의성은 높아지지만, 투명성을 구현하기 위한 미들웨어 오버헤드와 네트워크 왕복 비용이 증가합니다. 특히 복제 투명성과 일관성 투명성 동시 구현은 분산 합의 프로토콜 비용으로 인해 응답 시간이 크게 증가할 수 있습니다.
- **제언:** 투명성 수준을 비즈니스 요구사항에 맞게 조절해야 합니다. 읽기 비율이 높은 워크로드는 복제 투명성 + 최종적 일관성(읽기는 가까운 복제본에서)으로 성능을 최우선시하고, 쓰기 트랜잭션은 코디네이터를 통한 강한 일관성을 선택적으로 적용하는 계층적 투명성 전략을 수립해야 합니다.
