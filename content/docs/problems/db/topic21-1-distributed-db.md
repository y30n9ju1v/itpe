## 해설

### 출제 배경 및 의도

클라우드·글로벌 서비스 시대에 멀티 리전 분산 DB는 선택이 아닌 필수가 되었습니다. Google Spanner, CockroachDB, AWS Aurora Global Database 등이 주목받으면서 분산 DB의 원리를 이해하는 것이 중요해졌습니다. 분산 데이터베이스는 물리적으로 여러 노드에 분산된 데이터를 논리적으로 하나의 데이터베이스처럼 관리하는 시스템으로, 핵심 가치는 "분산의 복잡성을 사용자로부터 숨기는 투명성"에 있습니다.

채점관은 분산 투명성 8가지의 종류와 의미, 분산 DB의 장단점 트레이드오프, 그리고 2PC의 동작 메커니즘과 블로킹 한계, 이를 극복하는 Saga 패턴까지 체계적으로 설명할 수 있는지를 검증합니다.

### 1. 분산 데이터베이스, 개요

정 의  • 물리적으로 여러 노드에 분산된 데이터를 논리적으로 단일 데이터베이스처럼 관리하며, 분산 투명성으로 복잡성을 은닉하고 2PC 프로토콜로 분산 트랜잭션의 원자성을 보장하는 시스템
       - 수평 확장(Scale-Out)과 고가용성(HA)을 동시에 달성하는 현대 대규모 서비스의 핵심 인프라

```
[분산 투명성의 은닉 계층]

사용자/애플리케이션
       │  (투명성 레이어: 분산의 복잡성 은닉)
       ▼
┌──────────────────────────────────────────┐
│  분산 DBMS 미들웨어                       │
│  위치투명성  복제투명성  단편화투명성      │
│  장애투명성  병행투명성  이름투명성        │
│  성능투명성  스케일투명성                 │
└──────────────────────────────────────────┘
       │
  ┌────┼─────┐
  ▼    ▼     ▼
노드A  노드B  노드C   (물리적 분산 저장)
(서울) (도쿄) (LA)
```

- 분산 투명성은 8가지 유형으로 분산 환경의 복잡성을 응용 프로그램으로부터 완전히 은닉하여 단일 DBMS와 동일한 개발 경험을 제공

### 2. 가. 분산 투명성과 나. 분산 DB 장단점

1) 분산 투명성 8가지

| 구분 | 항목 | 설명 |
|------|------|------|
| 위치 투명성 | 정의/예시 | 데이터의 물리적 위치를 모르고 접근 / 서울·도쿄 어느 서버든 동일한 SQL 사용 |
| 복제 투명성 | 정의/예시 | 데이터 복제 여부를 모르고 접근 / 복제본 3개 있어도 단일 원본처럼 사용 |
| 단편화 투명성 | 정의/예시 | 데이터 분할(수평/수직) 여부를 모르고 접근 / 테이블이 여러 노드에 분산되어도 전체 조회 가능 |
| 장애 투명성 | 정의/예시 | 일부 노드 장애 시에도 서비스 계속 / 노드B 장애 시 노드A·C에서 자동 처리 |
| 병행 투명성 | 정의/예시 | 다수 트랜잭션 동시 수행 시 데이터 정합성 자동 유지 / 분산 잠금·타임스탬프로 충돌 자동 해결 |
| 이름 투명성 | 정의/예시 | 어느 노드에서나 동일한 이름으로 객체 접근 / 테이블 이름이 노드마다 달라도 통합 이름 사용 |
| 성능 투명성 | 정의/예시 | 시스템 부하에 따라 자동으로 성능 최적화 / 쿼리를 최적 노드로 자동 라우팅 |
| 스케일 투명성 | 정의/예시 | 노드 추가·제거 시 애플리케이션 변경 불필요 / 노드 증설해도 클라이언트 코드 수정 없음 |

2) 분산 DB 장단점

| 구분 | 항목 | 설명 |
|------|------|------|
| 장점 | 신뢰성·가용성 | 일부 노드 장애 시 다른 노드에서 서비스 계속 |
| 장점 | 성능 향상 | 지역 데이터 접근으로 네트워크 지연 최소화 |
| 장점 | 수평 확장 | 노드 추가로 용량·처리량 증가 |
| 장점 | 자율성 | 각 지역 사이트가 로컬 데이터 자율 관리 |
| 단점 | 복잡성 증가 | 분산 트랜잭션·장애 처리·일관성 유지 복잡 |
| 단점 | 보안 취약점 | 다수 노드·네트워크 구간의 보안 관리 어려움 |
| 단점 | 일관성 유지 비용 | 2PC 등 분산 합의 프로토콜의 오버헤드 |
| 단점 | 설계·운영 비용 | 전문 인력 및 인프라 비용 증가 |

### 3. 다. 2PC와 분산 트랜잭션 대안 비교

1) 2PC(Two-Phase Commit) 동작 메커니즘

| 구분 | 항목 | 설명 |
|------|------|------|
| Phase 1 (Prepare) | 코디네이터 역할 | 모든 참여자에게 PREPARE 메시지 전송 |
| Phase 1 (Prepare) | 참여자 역할 | 트랜잭션 실행 가능 여부 확인 후 YES/NO 응답, YES 시 Redo Log 기록 후 대기 |
| Phase 2 (Commit) | 커밋 조건 | 모든 참여자가 YES → COMMIT 메시지 전송 |
| Phase 2 (Abort) | 롤백 조건 | 하나라도 NO 또는 타임아웃 → ABORT 메시지 전송 |
| 한계 | 블로킹 문제 | 코디네이터 Phase 2 직전 장애 시 참여자 무한 대기 |
| 한계 | SPOF | 코디네이터 단일 장애점 |

```
[2PC 프로토콜 흐름]

코디네이터        참여자A          참여자B
    │──PREPARE──→│                 │
    │──PREPARE─────────────────→  │
    │←─YES────── │                 │
    │←─YES──────────────────────  │
    │──COMMIT───→│ (모든 YES 시)   │
    │──COMMIT──────────────────→  │
    │←─ACK────── │                 │
    │←─ACK──────────────────────  │
※ 하나라도 NO/장애 → ROLLBACK 전송
```

2) 분산 트랜잭션 프로토콜 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| 2PC | 블로킹/일관성/성능 | 블로킹(코디네이터 SPOF) / 강한 일관성(ACID) / 중간(2번 왕복) |
| 2PC | 적합 환경 | 소규모 분산 트랜잭션 |
| 3PC | 블로킹/일관성/성능 | 비블로킹(PreCommit 단계 추가) / 강한 일관성 / 낮음(3번 왕복) |
| 3PC | 적합 환경 | 이론적 개선, 실무 적용 드묾 |
| Saga | 블로킹/일관성/성능 | 비블로킹 / 최종적 일관성(Eventual) / 높음(비동기) |
| Saga | 적합 환경 | MSA 분산 트랜잭션 (보상 트랜잭션으로 복구) |

3) Raft vs 2PC — 현대 분산 DB의 합의 알고리즘

2PC는 코디네이터 기반 합의이고, **Raft**는 리더 선출 기반 합의 알고리즘으로 서로 다른 문제를 해결한다.

| 구분 | 2PC | Raft |
|------|-----|------|
| 목적 | 분산 트랜잭션 원자성 | 복제 로그의 일관된 합의 |
| 리더 장애 시 | 블로킹(무한 대기) | 자동 리더 재선출(Failover) |
| 적용 제품 | 전통 분산 RDBMS | etcd, CockroachDB, TiKV, Kafka |
| SPOF | 코디네이터 | 없음(과반수 노드 생존 시 운영) |

- etcd(Kubernetes 상태 저장소), CockroachDB, TiDB는 Raft로 리더 선출과 로그 복제를 처리한 뒤, 그 위에서 분산 트랜잭션(Percolator 프로토콜)을 수행하는 방식으로 2PC의 블로킹 문제를 극복함.
- MSA 환경에서 2PC의 블로킹·SPOF 문제를 극복하기 위해 Saga 패턴이 표준 분산 트랜잭션 대안으로 부상하고 있으나, 최종적 일관성 모델임을 설계 단계부터 명시해야 함.  "끝"

### 실무 제언

**2PC의 코디네이터 SPOF 해결**
- **챌린지**: 2PC에서 코디네이터가 Phase 1 이후 장애 시 참여자들이 커밋/롤백 결정을 알 수 없어 무한 대기하는 블로킹 문제가 발생하여 시스템 전체를 마비시킬 수 있음
- **제언**: MSA 환경에서는 Saga 패턴(각 서비스가 로컬 트랜잭션 실행 후 실패 시 보상 트랜잭션을 역순으로 실행)을 적극 검토. 단, 최종적 일관성 모델임을 인지하고 강한 일관성이 필수인 결제·정산 로직은 단일 서비스/DB 내 배치 설계 우선

**분산 투명성과 성능 트레이드오프**
- **챌린지**: 분산 투명성이 높을수록 사용자 편의성은 높아지지만 미들웨어 오버헤드와 네트워크 왕복 비용이 증가하고, 복제·일관성 투명성 동시 구현 시 분산 합의 프로토콜 비용으로 응답 시간이 크게 증가할 수 있음
- **제언**: 읽기 비율이 높은 워크로드는 복제 투명성 + 최종적 일관성(가까운 복제본 읽기)으로 성능을 최우선시하고, 쓰기 트랜잭션만 코디네이터를 통한 강한 일관성을 적용하는 계층적 투명성 전략 수립

**단편화 설계와 운영 복잡도 관리**
- **챌린지**: 수평·수직 단편화를 조합한 혼합 단편화는 질의 최적화 효과가 크지만, 단편화 위치와 재결합 로직을 미들웨어가 모두 추적해야 하므로 운영 복잡도가 급증함
- **제언**: 단편화 설계 시 주요 조회 패턴에 맞는 수평 단편화를 기본으로 하고, 컬럼 접근 빈도 분석을 통해 자주 사용하는 컬럼과 드문 컬럼을 수직 분리하는 최소한의 혼합 단편화만 적용하며, 단편화 메타데이터를 카탈로그에 명시적으로 관리
