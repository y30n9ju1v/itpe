## 해설

> **정규화 모델의 필연적 역설이자 성능의 결정자**  
> SQL JOIN: 논리적 유형 5가지와 물리적 알고리즘(NL/Hash/Sort Merge)의 최적 조합

관계형 데이터베이스(RDBMS) 설계에서 정규화를 수행하면 테이블이 분해됩니다. 분해된 데이터를 다시 비즈니스 로직에 맞춰 통합 조회하려면 필수적으로 JOIN 연산이 동반됩니다. 데이터의 규모가 테라바이트(TB)나 페타바이트(PB) 수준으로 증가하는 현대 클라우드 환경에서, 전체 쿼리 성능의 90% 이상은 어떤 JOIN 메커니즘을 선택하고 어떻게 인덱스를 설계했는가에 의해 결정됩니다.

### 1. 출제 배경 및 의도

채점관은 수험생이 SQL 표준의 논리적 JOIN 유형을 정확히 숙지하고 있는지, 그리고 이를 DBMS 내부의 물리적 알고리즘(NL, Hash, Sort Merge) 관점에서 성능 최적화를 이끌어낼 수 있는 실무형 데이터베이스 엔지니어링 역량이 있는지 검증하고자 합니다. Outer Join 오용 패턴, Driving 테이블 선정, 인덱스 배치 등 실무 최적화 사례를 구체적으로 제시할 수 있는지가 핵심입니다.

### 2. 개념의 본질과 작동 메커니즘

두 개 이상의 릴레이션(테이블)을 공통된 속성(키)을 매개체로 결합하여 하나의 통합된 결과 집합을 도출하는 논리적 질의 기법이자, 이를 데이터베이스 엔진 내에서 디스크 I/O와 메모리를 활용해 수립하는 물리적 실행 프로세스입니다.

**가. SQL 표준이 정의하는 5대 논리적 JOIN 유형:**

- **내부 조인(Inner Join):** 조인 조건에 명시된 양쪽 테이블의 결합키 값이 완벽하게 일치하는 행들만 추출하는 가장 일반적인 방식입니다. `SELECT * FROM A INNER JOIN B ON A.키 = B.키`
- **외부 조인(Outer Join):** 조인 조건에 일치하지 않는 행이라도 기준 테이블 데이터를 유실 없이 보존하고, 매칭되는 상대 테이블 값이 없으면 NULL로 채워 반환합니다. Left Outer Join(왼쪽 테이블 전체 보존), Right Outer Join(오른쪽 테이블 전체 보존), Full Outer Join(양쪽 테이블 전체 보존)으로 세분됩니다.
- **교차 조인(Cross Join / Cartesian Product):** 조인 조건이 없거나 누락 시 발생하며, 테이블 A의 모든 행과 테이블 B의 모든 행을 1:1로 결합하여 카디널리티가 M×N으로 폭발합니다.
- **셀프 조인(Self Join):** 단일 테이블 자체를 자기 자신과 조인하는 방식으로, 카테고리 계층 구조·조직도·순환 관계 모델을 풀어낼 때 사용됩니다.
- **등가/비등가 조인(Equi/Non-Equi Join):** 조인 조건이 `=` 연산자이면 등가 조인, 범위 조건(`BETWEEN`, `>`, `<=`)이나 불일치(`!=`) 조건이면 비등가 조인입니다.

**나. JOIN 성능을 높이기 위한 3대 물리적 최적화 기법:**

**기법 1 — 중첩 루프(NL) 조인의 Driving 테이블 선정 및 인덱스 배치**

NL 조인은 선행(Driving) 테이블의 조건 만족 행 수만큼 후행(Driven) 테이블을 반복 탐색합니다. 선행 테이블의 행 수를 최소화하고, 후행 테이블의 조인 컬럼에 인덱스를 생성해야 Random I/O 부하를 차단할 수 있습니다.

```sql
-- 개선 전: 주문(1,000만 건) 풀스캔 후 고객 테이블 반복 Full Scan
SELECT c.고객명, o.주문금액
FROM 고객 c, 주문 o
WHERE c.고객ID = o.고객ID AND o.주문일자 = '20260601';

-- 개선 후: 하루치 주문(수천 건)을 Driving으로 강제, 고객ID 인덱스 신설
-- CREATE INDEX idx_고객_01 ON 고객(고객ID);
SELECT /*+ LEADING(o) USE_NL(c) */ c.고객명, o.주문금액
FROM 주문 o, 고객 c
WHERE o.고객ID = c.고객ID AND o.주문일자 = '20260601';
```

**기법 2 — 대용량 데이터 간 해시 조인(Hash Join) 전환 및 Build Input 제어**

수백만 건 이상 대형 테이블 간 조인 시 NL 조인의 Random I/O 감당이 불가능하면 Hash Join으로 전환합니다. 크기가 작은 테이블을 Build Input으로 삼아 메모리에 해시 맵을 생성하고, 큰 테이블(Probe Input)을 순차 스캔하여 매핑합니다.

```sql
-- 매출상세(5억 건) × 상품마스터(50만 건) 결합: 소량 테이블을 Build Input으로 지정
SELECT /*+ LEADING(p) USE_HASH(d) SWAP_JOIN_INPUTS(p) */
       p.상품명, SUM(d.판매수량)
FROM 상품마스터 p, 매출상세 d
WHERE p.상품ID = d.상품ID
GROUP BY p.상품명;
```

**기법 3 — 조인 조건의 선행 필터링(Push-down Selection)**

조인 연산 전 인라인 뷰(Inline View) 내에서 선행 필터링을 수행하여 카디널리티를 절감시킨 뒤 조인 레벨로 승격합니다.

```sql
-- 개선 후: 게시글을 먼저 필터링(수천 건)한 후 댓글(2억 건)과 조인
SELECT b_tech.제목, r.댓글내용
FROM (
    SELECT 게시글ID, 제목
    FROM 게시글
    WHERE 카테고리코드 = 'TECH' AND 등록일자 > '20260101'
) b_tech, 댓글 r
WHERE b_tech.게시글ID = r.게시글ID;
```

### 3. 핵심 키워드 (Must-Have)

- **논리적 유형 관점:** 내부 조인(Inner Join), 외부 조인(Left/Right/Full Outer Join), 교차 조인(Cross Join), 셀프 조인(Self Join), 등가/비등가 조인
- **물리적 알고리즘 관점:** 중첩 루프 조인(Nested Loop Join), 해시 조인(Hash Join), 정렬 병합 조인(Sort Merge Join)
- **최적화 기법 관점:** Driving 테이블 선정, 인덱스(Index) 배치, Push-down Selection, Build Input/Probe Input, 힌트(Hint) 활용

### 4. 맥락과 비교

DBMS 내부에서 JOIN을 처리하는 3대 물리적 결합 알고리즘은 트랜잭션의 성격과 데이터의 규모에 따라 명확한 트레이드오프(Trade-off) 관계를 가집니다.

| 물리 조인 유형 | 중첩 루프 조인 (Nested Loop Join) | 해시 조인 (Hash Join) | 정렬 병합 조인 (Sort Merge Join) |
| --- | --- | --- | --- |
| **작동 메커니즘** | 2중 For 루프 형태로 선행 테이블의 한 행마다 후행 테이블 인덱스 탐색 | 선행 테이블로 메모리에 Hash Table을 생성 후 후행 테이블을 스캔하며 매핑 | 두 테이블을 조인 키 기준으로 각각 정렬한 후 스캔하며 병합 |
| **인덱스 의존성** | **필수적** (후행 테이블 인덱스 없으면 파탄) | **없음** (인덱스가 없어도 무방) | **낮음** (인덱스가 있으면 정렬 단계 생략 가능) |
| **적합한 데이터 규모** | 소량 데이터 처리, OLTP 환경 | 대용량 데이터 처리, DW/OLAP 환경 | 대용량 데이터 처리, 비등가 조인 등 |
| **주요 장점** | 먼저 만족하는 행을 즉시 반환 (Online 쿼리에 유리) | 정렬 부하가 없고 대량의 Random I/O를 순차 Scan으로 대체 | 조인 조건이 `=`이 아닌 범위 조건(`>`, `<`)에서도 강력함 |
| **주요 단점** | 무분별한 Random I/O 발생, 대량 데이터 처리 시 성능 급감 | 메모리(Hash Area) 사용량 큼, 최초 결과 반환까지 대기 시간 소요 | 두 테이블 모두 정렬해야 하므로 메모리(Sort Area) 및 CPU 부하 극심 |

### 5. 실무 제언

**Outer Join 오용과 MSA 환경에서의 JOIN 패러다임 시프트**

- **챌린지:** 실무에서 개발자들이 데이터 유실을 막기 위해 무분별하게 `LEFT OUTER JOIN`을 남용하면 DBMS 옵티마이저가 수립할 수 있는 조인 순서(Join Ordering)의 선택지를 강제로 고정시켜 버리므로 최적의 실행 계획 수립을 방해하고 성능 저하를 유발합니다. 마이크로서비스 아키텍처(MSA) 환경처럼 물리 DB가 분리된 환경에서는 RDBMS 표준 SQL을 통한 전통적 JOIN 연산 자체가 원천적으로 불가능해집니다.
- **제언:** 기술사는 대용량 조회 성능을 높이기 위해 무조건적인 DB 레이어 조인에 의존하기보다 인덱스 튜닝을 선행해야 합니다. MSA 환경에서 파편화된 서비스 데이터를 결합해야 할 경우 BFF(Backend For Frontend) 레이어에서 애플리케이션 사이드 조인(Application Side Join)을 처리하는 전략을 구사해야 합니다. 수억 건에 달하는 분석 통계성 트랜잭션은 CQRS 아키텍처를 도입하여 CDC 툴이나 이벤트 아키텍처(Kafka)를 통해 미리 역정규화된 통합 Read-Model(Denormalized Materialized View)을 사전에 빌드해 두는 패러다임 시프트가 필요합니다.
