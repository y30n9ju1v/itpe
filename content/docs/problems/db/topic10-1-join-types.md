## 해설

### 출제 배경 및 의도

관계형 데이터베이스(RDBMS) 설계에서 정규화를 수행하면 테이블이 분해됩니다. 분해된 데이터를 비즈니스 로직에 맞춰 통합 조회하려면 JOIN 연산이 필수적입니다. 데이터 규모가 TB·PB 수준으로 증가하는 현대 클라우드 환경에서 전체 쿼리 성능의 90% 이상은 JOIN 메커니즘 선택과 인덱스 설계에 의해 결정됩니다.

채점관은 SQL 표준의 논리적 JOIN 유형 정확한 숙지 여부와, DBMS 내부 물리적 알고리즘(NL, Hash, Sort Merge) 관점의 성능 최적화 역량을 검증하고자 합니다. Outer Join 오용 패턴, Driving 테이블 선정, 인덱스 배치 등 실무 최적화 사례 제시 가능 여부가 핵심 채점 포인트입니다.

### 1. SQL JOIN, 개요

정 의  • 두 개 이상의 릴레이션(테이블)을 공통된 속성(키)을 매개체로 결합하여 하나의 통합된 결과 집합을 도출하는 논리적 질의 기법이자 물리적 실행 프로세스
       - 정규화로 분해된 엔티티를 비즈니스 요건에 맞게 재결합하는 RDBMS의 핵심 연산

```
[JOIN 논리 유형 구조도]

  테이블 A      테이블 B
  ┌───────┐    ┌───────┐
  │  A∩B  │    │  A∩B  │
  │  only │────│  only │
  └───────┘    └───────┘

  INNER JOIN  : A ∩ B (교집합만)
  LEFT OUTER  : A 전체 + B 매칭 (B 없으면 NULL)
  RIGHT OUTER : B 전체 + A 매칭 (A 없으면 NULL)
  FULL OUTER  : A ∪ B (전체 합집합)
  CROSS JOIN  : A × B (카디널리티 M×N 폭발)
  SELF JOIN   : 단일 테이블 자기 자신과 조인 (계층 구조)
```

- JOIN 유형 선택은 비즈니스 데이터 결합 범위와 NULL 허용 여부로 결정된다.

### 2. 5대 논리적 JOIN 유형 및 성능 최적화 기법

1) SQL 표준 5대 논리적 JOIN 유형

| JOIN 유형 | 결합 범위 | 활용 사례 | 성능 특징 |
|----------|---------|---------|---------|
| **Inner Join** | 양쪽 일치 행만 추출 (교집합) | 주문과 결제가 모두 존재하는 완성 거래 조회 | 조인 순서 자유, 옵티마이저 최적화 용이 |
| **Left Outer Join** | 왼쪽 테이블 전체 + 오른쪽 매칭 (없으면 NULL) | 구매 이력 없는 고객도 포함한 매출 현황 | 조인 순서 고정, 최적화 제한 |
| **Right Outer Join** | 오른쪽 테이블 전체 + 왼쪽 매칭 | Left Outer Join의 방향 전환 | 동일 |
| **Full Outer Join** | 양쪽 테이블 전체 보존 (불일치 시 NULL) | 양쪽 집합의 완전한 합집합 조회 | 비용 극대 |
| **Cross Join** | 조건 없이 모든 행 조합 (M×N) | 달력·좌석 배치 등 전수 조합 생성 | 카디널리티 폭발, 성능 주의 |
| **Self Join** | 단일 테이블 자기 자신과 조인 | 조직도·계층 구조·순환 관계 모델 | Alias 필수 |
| **Equi/Non-Equi Join** | `=` 조건: 등가, `BETWEEN`·`>` 등: 비등가 | 포인트 등급 구간 매핑 | 비등가 시 Sort Merge Join 유리 |

- JOIN 유형은 비즈니스 데이터 결합 요건에 맞게 선택하되, Outer Join 남용은 옵티마이저 최적화를 제한한다.

2) JOIN 성능 향상 3대 물리적 최적화 기법

| 기법 | 내용 | 적용 조건 |
|------|------|---------|
| **① NL 조인 Driving 테이블 선정** | 소량 집합을 Driving으로, 후행 테이블 조인 컬럼에 인덱스 생성 | OLTP, 소량 데이터, 인덱스 완비 환경 |
| **② Hash Join 전환 및 Build Input 제어** | 소량 테이블을 Build Input으로 메모리 해시 맵 구성, 대용량 테이블을 Probe로 순차 스캔 | 대용량 데이터, 인덱스 부재, 등가 조인 |
| **③ Push-down Selection (선행 필터링)** | 조인 전 인라인 뷰에서 카디널리티를 최소화하여 조인 입력 행 수 절감 | 조인 대상 테이블에 대량 데이터가 존재할 때 |

- 조인 성능의 핵심은 조인 전 카디널리티 최소화(Push-down)와 물리 알고리즘 선택의 정합성이다.

### 3. 물리 조인 알고리즘 비교 및 현대 트렌드

| 물리 조인 유형 | Nested Loop Join | Hash Join | Sort Merge Join |
|-------------|-----------------|-----------|----------------|
| **인덱스 의존성** | 필수 (없으면 파탄) | 없음 | 낮음 |
| **적합 데이터 규모** | 소량 (OLTP) | 대용량 (OLAP/DW) | 대용량, 비등가 조인 |
| **최초 결과 반환** | 가장 빠름 (First-Row 유리) | 느림 (해시 맵 생성 대기) | 느림 (전체 정렬 대기) |
| **조인 조건 제약** | 모든 조건 | 등가 조인(`=`)만 가능 | 모든 조건 |
| **힌트 제어** | USE_NL | USE_HASH | USE_MERGE |

- 현대 분산 쿼리 엔진(Spark SQL)에서는 소량 테이블을 모든 노드로 복제하는 브로드캐스트 조인(Broadcast Join)이 전통 Hash Join 원리를 네트워크 분산 환경에 투영한 진화된 형태이다.  "끝"

### 실무 제언

**Outer Join 오용 방지와 Inner Join 우선 설계**
- **챌린지**: 개발자들이 데이터 유실을 우려하여 모든 쿼리에 `LEFT OUTER JOIN`을 배치하면, 옵티마이저가 조인 순서를 자유롭게 변경할 수 없어 최적 실행 계획 수립이 불가능해지고 성능이 저하된다.
- **제언**: 데이터 모델링 단계에서 필수적 관계(Mandatory)를 유도하여 최대한 INNER JOIN 구조로 설계하고, 불가피한 Outer Join은 필터 조건을 하부 인라인 뷰로 전진 배치(Push-down)하여 성능 리스크를 능동적으로 통제해야 한다.

**MSA 환경에서의 JOIN 패러다임 시프트**
- **챌린지**: MSA 환경에서 물리 DB가 서비스 경계별로 분리되면 표준 SQL JOIN이 원천적으로 불가능해지며, DB 레이어의 JOIN에 의존하는 설계는 모놀리식으로의 회귀를 강요한다.
- **제언**: OLTP 화면 조회 시 BFF(Backend For Frontend) 레이어에서 애플리케이션 사이드 조인을 처리하고, 분석성 대용량 쿼리는 CQRS 아키텍처로 Kafka·CDC를 통해 역정규화된 Read-Model을 사전 구축하는 패러다임 시프트가 필요하다.

**NL 조인 인덱스 누락 대참사 예방**
- **챌린지**: 후행 테이블의 조인 컬럼에 인덱스가 없는 상태에서 NL 조인이 유도되면, 선행 테이블 행 수만큼 Full Table Scan이 반복되어 시스템 전체가 마비된다.
- **제언**: 조인 키의 분포도와 카디널리티를 정량 분석하여 인덱스를 사전 설계하고, 대량 데이터 집계 쿼리에서는 Hash Join으로 유도하기 위해 힌트(USE_HASH)나 통계 정보 갱신으로 실행 계획을 제어해야 한다.
