## 해설

> **정규화 이론의 실질적 최상위 단계**  
> 제5정규형(5NF, PJ/NF): 조인 종속성(JD)을 3개 이상의 테이블로 n-분해하여 무결성을 달성

제5정규형(5NF)은 관계형 데이터베이스 정규화 이론의 실질적인 최상위 단계입니다. 함수 종속성(FD)이나 다치 종속(MVD)처럼 직관적으로 식별하기 쉬운 제약조건과 달리, 결합성과 대칭성에 의존하는 조인 종속성(JD)을 다루기 때문에 기술사 시험에서 최고 난이도의 아키텍처 역량을 검증할 때 출제됩니다. 특히 2개로 분해 시 가짜 튜플이 발생하고, 반드시 3개 이상으로 분해해야 무손실 복원이 가능하다는 구조적 특성이 핵심입니다.

### 1. 출제 배경 및 의도

ERP, 금융권, 제조 SCM 등에서는 3개 이상의 엔티티가 순환 형태의 다대다(M:N) 관계로 얽히는 경우가 많습니다. 이때 테이블을 잘못 쪼개면 데이터 조회 시 가짜 데이터가 유도되는 투플 생성 이상이 발생하는데, 이를 공학적으로 해결할 수 있는 기준을 아는지 평가하기 위함입니다. 채점관은 5NF의 정의와 함께 조인 종속(JD) 개념, 그리고 n-분해 메커니즘을 사례와 함께 서술하는 답안을 고득점으로 평가합니다.

### 2. 개념의 본질과 작동 메커니즘

**정의:** 제4정규형(4NF)을 만족하면서, 릴레이션에 존재하는 모든 조인 종속성(JD)의 결정자가 해당 릴레이션의 후보키(Candidate Key)가 되도록 강제하여 불필요한 데이터 중복과 조인 이상을 제거하는 프로세스입니다. 분해 기법 측면에서 테이블을 쪼갰다가 다시 결합(Join)하는 대칭적 성질을 지니므로 **PJ/NF(Projection-Join Normal Form)**라고도 명명합니다.

**조인 종속성(JD) 개념:** 릴레이션 R을 복수 개의 릴레이션 R1, R2, ..., Rn으로 프로젝션 분해했다가, 이들을 다시 내추럴 조인(Natural Join)한 결과가 원래의 릴레이션 R과 한 치의 오차도 없이 일치할 때 "릴레이션 R은 조인 종속성을 만족한다"고 정의합니다.

#### 5NF 사례: 공급망 삼항 관계

**비즈니스 제약조건:**
1. 공급업자는 특정 부품을 생산하여 공급할 수 있습니다.
2. 기업(소비자)은 프로젝트 수행을 위해 특정 부품을 조달받아 사용합니다.
3. 기업은 특정 공급업자와 계약을 맺고 부품을 제공받습니다.
4. **핵심 룰:** 어떤 공급업자가 부품 P를 생산하고, 기업 C가 부품 P를 사용하며, 기업 C가 해당 공급업자와 거래 중이라면, 그 공급업자는 반드시 기업 C에게 부품 P를 공급해야만 하는 대칭적 순환 규칙이 존재합니다.

```text
[원시 릴레이션 <공급망(SPC)> — 모든 속성이 복합키]
| 공급업자 (PK) | 부품명 (PK) | 기업명 (PK) |
| :--- | :--- | :--- |
| 김기술 | 반도체 | S전자 |
| 박공학 | 디스플레이 | S전자 |
| 김기술 | 디스플레이 | L디스플레이 |
| 김기술 | 디스플레이 | S전자 | (제약조건 4에 의해 반드시 존재해야 하는 데이터)
```

만약 이 구조를 단순히 2개의 테이블(예: <공급-부품>, <부품-기업>)로 쪼개면 조인 시 원래 존재하지 않던 가짜 튜플(Spurious Tuple)이 생성되는 투플 생성 이상을 겪게 됩니다.

**5NF 해결방안: 3원 무손실 분해**

```text
[검증: 이 분해된 3개의 테이블을 동시에 내추럴 조인(SP ⋈ PC ⋈ SC)하면,
 가짜 데이터의 침투 없이 원래의 청정 데이터 4건으로 완벽하게 무손실 복원됨.]
```

- **릴레이션 1 <공급부품(SP)>** (공급업자, 부품명)
- **릴레이션 2 <부품기업(PC)>** (부품명, 기업명)
- **릴레이션 3 <공급기업(SC)>** (공급업자, 기업명)

### 3. 핵심 키워드 (Must-Have)

- **선행 조건:** 제4정규형(4NF) 만족, 다치 종속(MVD) 제거 상태
- **핵심 종속성:** 조인 종속성(Join Dependency, JD), n-분해(n-Decomposition), 삼항 관계(Ternary Relationship)
- **부작용 및 현상:** 투플 생성 이상, 가짜 튜플(Spurious Tuple), 무손실 분해(Lossless Decomposition)
- **동의어:** PJ/NF (Projection-Join Normal Form)

### 4. 맥락과 비교

제5정규형은 이전 단계의 정규형들이 해결하지 못하는 복합적인 다중 종속 관계를 최종적으로 해결합니다.

| 구분 | 주된 제거 대상 | 분해 한계 및 특징 | 달성 상태 |
| --- | --- | --- | --- |
| **제3정규형 / BCNF** | 함수 종속성 (FD) | 결정자가 후보키가 되도록 2개의 테이블로 분해 | 기본키 기반 무결성 확보 |
| **제4정규형 (4NF)** | 다치 종속성 (MVD) | 결정자 하나에 독립적인 다치 속성이 얽힐 때 2개로 분해 | 무의미한 조합 중복 제거 |
| **제5정규형 (5NF)** | **조인 종속성 (JD)** | **2개로 분해 불가.** 반드시 3개 이상의 테이블로 대칭 분해해야 함 | 순환적 관계 구조의 완전화 |

| 비교 항목 | 제4정규형 (4NF) | 제5정규형 (5NF) |
| --- | --- | --- |
| **핵심 타깃 종속성** | 다치 종속성 (Multi-Valued Dependency, MVD) | 조인 종속성 (Join Dependency, JD) |
| **관계의 구조적 특징** | 한 개의 결정자에 2개의 독립적인 다치 속성이 나열됨 | 3개 이상의 속성이 꼬리를 물고 순환하는 구조 |
| **최소 분해 테이블 수** | **2개**의 테이블로 프로젝션 분해 가능 | 반드시 **3개 이상**의 테이블로 대칭 분해 필수 |
| **실무적 식별 난이도** | 비교적 용이 (속성 간 독립성 확인) | 매우 어려움 (삼항 이상의 순환 룰 검증 필요) |

### 5. 실무 제언

**실무 적용의 극악한 난이도와 트레이드오프**

- **챌린지:** 실무 데이터 모델링 현장에서 제5정규형을 완벽히 적용하면 데이터 적재 시 무결성은 완벽히 방어됩니다. 그러나 이를 다시 조회하려면 3개 이상의 테이블을 항상 동시에 조인해야 하므로 SQL 작성이 매우 복잡해지고 인덱스 스캔 부하 및 CPU 병목을 유발합니다. 특히 대규모 트랜잭션이 발생하는 클라우드 네이티브 OLTP 환경에서 5NF 스키마를 채택하면, 분산 노드 간의 데이터 네트워크 전송(Shuffle)과 디스크 I/O 병목이 발생하여 시스템 전체의 응답 속도(Latency)가 기하급수적으로 저하됩니다.
- **제언:** 프로젝트 분석 단계에서는 비즈니스 규칙이 왜곡되거나 연결함정에 빠지는 것을 방지하기 위해 반드시 5NF 수준의 논리 모델을 설계해야 합니다. 5NF 구조로 판명된 영역의 데이터 변경 주기(DML 빈도)를 분석하여, 실시간 변경이 적고 복합적인 조회가 주로 일어나는 영역이라면 물리 단계에서 테이블을 3개로 쪼개기보다 PostgreSQL의 JSONB 타입이나 MongoDB 같은 Document Model을 활용하여 삼항 관계 데이터를 하나의 문서(Document) 구조 안에 임베딩(Embedding) 처리함으로써, 무결성과 고성능 조회를 동시에 달성하는 융합형 아키텍처 전략을 리드해야 합니다.
