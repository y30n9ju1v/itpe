효율적인 정보화 투자 및 안정적인 정보시스템 운영 등을 위해 정보시스템 도입 시 적정한 하드웨어 규모를 산정하는 것이 매우 중요하다. 하드웨어 용량산정의 개념과 절차에 대해서 설명하고, DB서버를 위한 용량 산정을 위한 기초자료에 대해서 상세히 설명하시오.

## 해설

### 출제 배경 및 의도

공공 정보화 사업(행정안전부 지침, 정보시스템 감리)에서 하드웨어 용량 산정은 법적 의무 산출물입니다. 정보시스템을 도입할 때 하드웨어를 과소 산정하면 운영 중 성능 저하·서비스 장애가 발생하고, 과대 산정하면 불필요한 투자 낭비가 됩니다. DB 서버는 전체 시스템의 성능 병목이 가장 자주 발생하는 핵심 컴포넌트로, 과학적이고 체계적인 용량 산정 방법론이 필요합니다.

채점관은 단순한 개념 나열을 넘어 실제 DB 서버 용량 산정에 필요한 기초자료의 종류, 각 자료가 왜 필요한지, 그리고 이를 활용한 TPS·CPU·메모리·디스크 산정 공식까지 설명할 수 있는지를 검증합니다.

### 1. 하드웨어 용량 산정, 개요

정 의  • 예상 업무량(Workload)과 데이터 증가를 분석하여 CPU·Memory·Disk·Network 자원이 SLA(서비스 수준 목표)를 만족하는 최적 하드웨어 규모를 사전에 결정하는 공학적 프로세스
       - DB 서버는 테이블별 건수·증가율·보존기간을 기초자료로 TPS→CPU→Memory→Disk 순서로 체계적 산출

```
[DB 서버 용량 산정 흐름도]

기초자료 수집 (테이블 정의서, 업무 분석서)
        │
        ▼
데이터 용량 산정
= Σ(테이블별 건수 × 행크기 × 보존기간 증가)
+ 인덱스 용량(테이블의 약 20~30%) + 시스템 여유(20%)
        │
        ▼
TPS 산정
= 일일 트랜잭션 × 피크 집중률 / 피크 지속 시간
        │
        ▼
CPU 산정 = TPS × SQL당 처리시간 / 목표 CPU 활용률
        │
        ▼
Memory 산정 = Buffer Pool + PGA/SGA + 연결 수 × 세션 크기
        │
        ▼
최종 사양 = 산정치 × (1 + 여유율 20~30%)
```

- 용량 산정은 반드시 기초자료 수집→TPS→CPU→Memory→Disk 순서로 진행해야 하며, 서버 증설 여유율은 클라우드 환경에서는 탄력적으로 조정 가능

### 2. 하드웨어 용량 산정 절차

1) 산정 5단계 절차

| 구분 | 항목 | 설명 |
|------|------|------|
| Step 1 | 업무량(Workload) 분석 | 주요 트랜잭션 목록화, 일평균/피크타임 처리량 추정, 데이터 증가율·보존기간 확인 |
| Step 2 | 성능 목표(SLA) 정의 | 최대 응답시간(예: 3초 이내), 목표 가용성(예: 99.9%), 최대 동시 사용자 수 |
| Step 3 | 서버별 용량 산정 | WEB/WAS: TPS 기준 CPU 코어 수 / DB: TPS·데이터 용량 기준 CPU/Memory/Disk 산정 |
| Step 4 | 여유율·피크 계수 적용 | 최종 권고 사양 = 산정치 × (1 + 여유율) × 피크 계수 |
| Step 5 | 결과 검토 및 확정 | 벤더 BMT(Benchmark Test) 검토, 유사 사업 사례 비교 검증 |

2) DB 서버 기초자료 3종

| 구분 | 항목 | 설명 |
|------|------|------|
| 테이블 정보 | 초기 건수(Row Count) | 테이블별 시스템 오픈 초기 데이터 건수 |
| 테이블 정보 | 증가율(Growth Rate) | 연간/월간 데이터 증가율 (%) |
| 테이블 정보 | 보존 기간(Retention) | 데이터 법적·업무적 보존 기간 |
| 테이블 정보 | 행 크기(Row Size) | 테이블 1행의 평균 바이트 수 |
| 트랜잭션 정보 | 일평균 트랜잭션 건수 | 하루 총 처리 트랜잭션 수 |
| 트랜잭션 정보 | 피크타임 배율(Peak Factor) | 피크 시간대 트랜잭션 집중률 (%) |
| 트랜잭션 정보 | R/W 비율 | 읽기/쓰기 트랜잭션 비율 (CPU/I/O 산정 영향) |
| 동시 사용자 | 일평균 동시 접속자 수 | 평균 DB 연결 세션 수 |
| 동시 사용자 | 최대 동시 접속자(Peak) | 피크타임 최대 DB 연결 세션 수 |

### 3. DB 서버 자원별 산정 공식

1) TPS 및 데이터 용량 산정

| 구분 | 항목 | 설명 |
|------|------|------|
| TPS 산정식 | 공식 | 피크 TPS = (일일 트랜잭션 × 피크 집중률) / 피크 지속 시간(초) |
| TPS 산정 예시 | 계산 | 1,000,000건 × 30% / 3,600초 ≒ 83 TPS → 여유율 30% 적용 → 설계 TPS 108 |
| 데이터 용량 | 공식 | 총 용량 = Σ(테이블 건수 × 행 크기 × 보존기간 증가) + 인덱스(20~30%) + 여유(20%) |

2) CPU·Memory·Disk(IOPS) 산정 공식

| 구분 | 항목 | 설명 |
|------|------|------|
| CPU 산정 | 공식 | CPU 코어 수 = (설계 TPS × SQL당 CPU 사용시간) / (CPU 활용률 목표 × 코어당 처리능력) |
| Memory 산정 | 공식 | 메모리 = SGA/PGA + Buffer Pool + 연결 수 × 세션 메모리 |
| Memory 산정 | 세부 항목 | Oracle SGA/PGA, MySQL InnoDB Buffer Pool이 주요 구성 요소 |
| Disk 용량 | 공식 | 총 용량 = 데이터 + 인덱스 + 임시 공간 + 아카이브 로그 + 시스템 여유 |
| Disk 성능(IOPS) | 공식 | IOPS = 쓰기 TPS × SQL당 I/O 횟수 + 읽기 TPS × 버퍼미스율 × SQL당 I/O |

- WEB/WAS 서버는 TPS·동시 접속자 기준 CPU/메모리 산정, DB 서버는 TPS·데이터 용량·IOPS까지 추가 산정이 필요하며 디스크 I/O가 핵심 병목 요인임.  "끝"

### 실무 제언

**초기 산정과 실제 운영 간 갭 관리**
- **챌린지**: 용량 산정은 사전 추정치이므로 실제 운영 시 예상보다 트랜잭션 급증, 데이터 증가율 초과, 특정 SQL CPU 과다 사용 등이 빈번하게 발생하여 초기 산정 오류가 운영 중 장애로 이어지는 사례가 많음
- **제언**: 시스템 오픈 후 3~6개월 내에 실제 TPS·디스크 증가율·CPU/메모리 사용률 추이를 모니터링하여 초기 산정과 비교 검증하는 "운영 검증 리뷰"를 수행하고, 클라우드 환경에서는 Auto Scaling 정책을 수립하여 실제 운영 데이터 기반으로 지속 조정

**클라우드 환경의 용량 산정 패러다임 변화**
- **챌린지**: 온프레미스 환경에서는 H/W 변경이 어려워 여유율 30~50%로 과대 산정하는 경향이 있었으나, 클라우드 환경에서는 이런 안전 마진이 불필요한 비용 낭비가 됨
- **제언**: 클라우드 DB(AWS RDS, Azure SQL, Cloud Spanner)는 최소 사양으로 시작하고 실제 사용 메트릭(CPU·I/O·스토리지) 기반 Auto Scaling 또는 온디맨드 증설을 적용하는 "Rightsize-and-Scale" 전략 채택. 예약 인스턴스는 6개월 운영 후 확정된 베이스라인 용량에만 적용하여 비용 최적화

**피크 타임 분석의 정확도 확보**
- **챌린지**: 피크타임 집중률과 지속 시간을 과소 추정하면 실제 피크 시 TPS를 크게 초과하여 응답 시간 SLA를 위반하고, 과대 추정하면 불필요한 H/W 투자가 발생함
- **제언**: 유사 사업의 실제 트랜잭션 로그를 분석하여 시간대별 분포를 도출하고, 피크 배율(피크 TPS / 평균 TPS)을 실측 데이터 기반으로 산정. 신규 사업은 업종별 표준 피크 배율(일반 업무: 2~3배, 이커머스 이벤트: 5~10배)을 참조하여 보수적으로 설계
