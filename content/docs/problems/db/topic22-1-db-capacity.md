## 문제

효율적인 정보화 투자 및 안정적인 정보시스템 운영 등을 위해 정보시스템 도입 시 적정한 하드웨어 규모를 산정하는 것이 매우 중요하다. 하드웨어 용량산정의 개념과 절차에 대해서 설명하고, DB서버를 위한 용량 산정을 위한 기초자료에 대해서 상세히 설명하시오.

## 해설

> **정보시스템 도입의 첫 번째 엔지니어링 결정 - 하드웨어 규모 산정**  
> DB 서버 용량 산정은 테이블별 건수·증가율·보존기간을 기초자료로 TPS·CPU·메모리·디스크를 체계적으로 산출하는 공학적 프로세스

정보시스템을 도입할 때 하드웨어를 과소 산정하면 운영 중 성능 저하·서비스 장애가 발생하고, 과대 산정하면 불필요한 투자 낭비가 됩니다. 옷을 살 때 너무 작으면 불편하고 너무 크면 예산 낭비인 것처럼, DB 서버 규모 산정도 딱 맞는 균형을 찾는 과정입니다. DB 서버는 전체 시스템의 성능 병목이 가장 자주 발생하는 핵심 컴포넌트로, 과학적이고 체계적인 용량 산정 방법론이 필요합니다.

### 1. 출제 배경 및 의도

공공 정보화 사업(행정안전부 지침, 정보시스템 감리)에서 하드웨어 용량 산정은 법적 의무 산출물입니다. 채점관은 단순한 개념 나열을 넘어 실제 DB 서버 용량 산정에 필요한 기초자료의 종류, 각 자료가 왜 필요한지, 그리고 이를 활용한 산정 공식까지 설명할 수 있는지를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 하드웨어 용량 산정 개념 및 절차

용량 산정(Capacity Planning)은 예상되는 업무량(Workload)과 데이터 증가를 분석하여 시스템 구성 요소(CPU, Memory, Disk, Network)가 서비스 수준 목표(SLA, Service Level Agreement)를 만족하는지 사전에 검증하고, 필요한 하드웨어 규모를 결정하는 프로세스입니다.

```text
[하드웨어 용량 산정 절차]

Step 1. 업무량(Workload) 분석
  ├── 주요 트랜잭션 목록화
  ├── 일평균/피크타임 처리량 추정
  └── 데이터 증가율·보존기간 확인

Step 2. 성능 목표(SLA) 정의
  ├── 최대 응답시간 (예: 3초 이내)
  ├── 목표 가용성 (예: 99.9% = 연 8.76시간 다운 허용)
  └── 최대 동시 사용자 수

Step 3. 서버별 용량 산정
  ├── WEB/WAS 서버: TPS 기준 CPU 코어 수 산정
  ├── DB 서버: TPS·데이터 용량 기준 CPU/Memory/Disk 산정
  └── 네트워크: 대역폭 산정

Step 4. 여유율(Growth Factor) 및 피크 계수 적용
  └── 최종 권고 사양 = 산정치 × (1 + 여유율) × 피크 계수

Step 5. 산정 결과 검토 및 최종 확정
  ├── 벤더 BMT(Benchmark Test) 검토
  └── 유사 사업 사례 비교 검증
```

#### DB 서버 용량 산정 기초자료 (핵심)

```text
[DB 서버 용량 산정 기초자료 구성]

┌──────────────────────────────────────────────┐
│           DB 용량 산정 기초자료               │
│                                              │
│  1. 테이블 정보                               │
│     ├── 테이블별 초기 건수 (Row Count)        │
│     ├── 연간/월간 증가율 (Growth Rate)        │
│     ├── 데이터 보존 기간 (Retention Period)   │
│     └── 행 크기 (Row Size, Bytes)            │
│                                              │
│  2. 트랜잭션 정보                             │
│     ├── 일평균 트랜잭션 건수 (Daily TRX)      │
│     ├── 피크타임 배율 (Peak Factor)           │
│     └── 트랜잭션별 SQL 유형 (R/W 비율)        │
│                                              │
│  3. 동시 사용자 정보                          │
│     ├── 일평균 동시 접속자 수                 │
│     └── 최대 동시 접속자 수 (Peak)            │
└──────────────────────────────────────────────┘
```

**기초자료 1: 테이블별 데이터 용량 산정표**

| 테이블명 | 초기 건수 | 월 증가율 | 보존 기간 | 행 크기 (Bytes) | 보존 기간 말 예상 건수 | 총 용량 (GB) |
|---------|---------|---------|---------|---------------|------------------|------------|
| 주문 | 1,000,000 | 5% | 3년 | 500 | ~2,160,000 | ~1.1 GB |
| 주문상품 | 3,000,000 | 5% | 3년 | 200 | ~6,480,000 | ~1.3 GB |
| 사용자 | 500,000 | 2% | 영구 | 1,000 | ~860,000 | ~0.9 GB |
| ... | ... | ... | ... | ... | ... | ... |

**기초자료 2: TPS(Transaction Per Second) 산정**

```text
[TPS 산정 공식]

피크타임 TPS = (일일 총 트랜잭션 수 × 피크타임 집중률) / 피크타임 지속 시간(초)

예시:
- 일일 총 트랜잭션: 1,000,000건
- 피크타임 집중률: 30% (피크타임에 전체의 30% 집중)
- 피크타임 지속 시간: 1시간 = 3,600초

피크타임 TPS = (1,000,000 × 0.30) / 3,600 ≒ 83 TPS

여유율 30% 적용: 83 × 1.3 ≒ 108 TPS (설계 TPS)
```

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 기초자료 | 테이블별 건수, 증가율(Growth Rate), 보존기간(Retention), 행 크기(Row Size) |
| TPS 산정 | 일일 트랜잭션, 피크타임 집중률, Peak Factor, 설계 TPS |
| 서버 산정 | CPU 코어 수, 메모리 용량, 디스크 용량, I/O 처리량(IOPS) |
| 여유율 | 성장 여유율(20~30%), 피크 계수, 안전 계수 |
| 공식 | TPS 산정식, CPU 산정식, 디스크 용량 산정식 |
| 산정 절차 | 업무량 분석, SLA 정의, 자원별 산정, 여유율 적용, 검증 |

**CPU/Memory/Disk 산정 공식 요약**

| 자원 | 산정 공식 | 설명 |
|------|---------|------|
| **CPU** | CPU 코어 수 = (설계 TPS × SQL당 CPU 사용시간) / (CPU 활용률 목표 × 코어당 처리능력) | DB 쿼리 처리 비용 기반 산정 |
| **Memory** | 메모리 = SGA/PGA 크기 + 버퍼 풀 + 연결 수 × 세션 메모리 | Oracle SGA/PGA, MySQL InnoDB Buffer Pool 등 |
| **Disk (용량)** | 총 용량 = Σ(테이블 용량) + 인덱스 용량 + 임시 공간 + 아카이브 로그 + 시스템 여유 | 보존기간 말 기준 산정 |
| **Disk (성능)** | IOPS = 쓰기 TPS × SQL당 I/O 횟수 + 읽기 TPS × 버퍼미스율 × SQL당 I/O | 디스크 처리 성능(IOPS, 초당 입출력 횟수) 기준 |

```text
[DB 서버 용량 산정 흐름도]

기초자료 수집
(테이블 정의서, 업무 분석서)
        │
        ▼
데이터 용량 산정
= Σ (테이블별 건수 × 행크기 × 보존기간 증가)
+ 인덱스 용량 (테이블의 약 20~30%)
+ 시스템 여유 (20%)
        │
        ▼
TPS 산정
= 일일 트랜잭션 × 피크 집중률 / 피크 지속 시간
        │
        ▼
CPU 산정
= TPS × SQL당 처리시간 / 목표 CPU 활용률
        │
        ▼
Memory 산정
= Buffer Pool + PGA/SGA + 연결 수 × 세션 크기
        │
        ▼
최종 사양 = 산정치 × (1 + 여유율 20~30%)
```

### 5. 실무 제언

**초기 산정과 실제 운영 간 갭 관리**

- **챌린지:** 용량 산정은 사전 추정치이므로 실제 운영 시 예상보다 트랜잭션이 급증하거나, 데이터 증가율이 훨씬 높거나, 특정 SQL의 CPU 사용이 예상치를 크게 초과하는 경우가 빈번합니다. 초기 산정 오류가 운영 중 장애로 이어지는 사례가 많습니다.
- **제언:** 시스템 오픈 후 3~6개월 내에 실제 TPS·디스크 증가율·CPU/메모리 사용률 추이를 모니터링하여 초기 산정과 비교 검증하는 "운영 검증 리뷰"를 수행해야 합니다. 클라우드 환경이라면 Auto Scaling 정책을 사전에 수립하고 실제 운영 데이터를 기반으로 지속 조정하는 방식이 효과적입니다.

**클라우드 환경에서의 용량 산정 패러다임 변화**

- **챌린지:** 전통적 온프레미스 환경에서는 하드웨어를 구매하면 용량을 변경하기 어렵기 때문에 여유율을 크게(30~50%) 잡아 과대 산정하는 경향이 있었습니다. 클라우드 환경에서는 이런 안전 마진이 오히려 불필요한 비용 낭비가 됩니다.
- **제언:** 클라우드 DB(AWS RDS, Azure SQL, Cloud Spanner)는 초기에는 최소 사양으로 시작하고, 실제 사용 메트릭(CPU, I/O, 스토리지)을 기반으로 Auto Scaling 또는 온디맨드 증설을 적용하는 "Rightsize-and-Scale" 전략을 채택해야 합니다. 예약 인스턴스(Reserved Instance)는 6개월 운영 후 확정된 베이스라인 용량에만 적용하여 비용을 최적화합니다.
