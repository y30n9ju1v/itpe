## 문제

인메모리 데이터베이스(In-memory Database)

## 해설

> **디스크 I/O를 제거하여 마이크로초 단위 응답을 실현하는 데이터베이스**  
> DRAM을 주 저장소로 사용하며, 영속성 보장을 위한 AOF·RDB 스냅샷과 함께 캐싱·세션·실시간 분석에 활용

전통적인 RDBMS는 데이터를 디스크에 저장하므로 I/O 레이턴시(수 ms~수십 ms)가 병목이 됩니다. 인메모리 데이터베이스(IMDB)는 주 저장소를 DRAM으로 변경하여 I/O 지연을 나노초~마이크로초 수준으로 낮춰 초당 수백만 건의 읽기·쓰기를 처리할 수 있습니다.

### 1. 출제 배경 및 의도

실시간 게임, 금융 HFT(고빈도 거래), 실시간 추천 시스템 등 마이크로초 단위의 응답이 요구되는 분야가 급증하면서, 인메모리 DB는 현대 고성능 아키텍처의 필수 구성 요소가 되었습니다. 117회 기술사 시험에서는 IMDB의 개념과 구조, 영속성 해결 방안, 그리고 실무 활용 사례를 체계적으로 설명할 수 있는지를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

인메모리 데이터베이스는 데이터를 디스크가 아닌 주 메모리(RAM)에 상주시켜 데이터 접근 속도를 극대화한 데이터베이스 시스템입니다.

```text
[디스크 기반 DB vs 인메모리 DB 아키텍처 비교]

디스크 기반 DB (RDBMS):
  쿼리 → 버퍼 풀 확인 → [Miss] → 디스크 I/O (5~15ms)
                       → [Hit]  → 메모리에서 직접 반환 (0.1ms)
  ※ 버퍼 풀 히트율이 성능을 결정

인메모리 DB (IMDB):
  쿼리 → DRAM 직접 접근 (100ns~1μs) → 결과 반환
  ※ 영속성을 위해 디스크에 비동기 백업(AOF/Snapshot)

[응답 시간 비교]
  DRAM 접근:   ~100 nanoseconds
  SSD 접근:    ~100 microseconds (약 1,000배 느림)
  HDD 접근:    ~10 milliseconds  (약 100,000배 느림)
```

#### 영속성 보장 메커니즘

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **RDB (Redis Database) 스냅샷** | 일정 시간/변경 횟수 조건 시 전체 데이터를 디스크에 바이너리 스냅샷으로 저장 | 빠른 재시작, 작은 파일 크기 | 스냅샷 간 데이터 유실 가능 |
| **AOF (Append Only File)** | 모든 쓰기 명령을 로그 파일에 순차 추가 | 데이터 유실 최소화 (fsync 정책 설정 가능) | 파일 크기 증가, 재시작 시간 증가 |
| **RDB + AOF 혼합** | AOF로 일관성 확보, RDB로 빠른 복구 | 두 방식의 장점 결합 | 디스크 I/O 이중 부담 |

### 4. 맥락과 비교

| 비교 항목 | RDBMS | 인메모리 DB | 디스크 기반 NoSQL |
|----------|-------|------------|----------------|
| 주 저장소 | 디스크 (SSD/HDD) | DRAM | 디스크 (SSD) |
| 읽기 지연 | 수 ms | 수십~수백 μs | 수 ms |
| 쓰기 지연 | 수 ms | 수십 μs | 수 ms |
| 데이터 영속성 | 완전 보장 | 부분적 (설정 필요) | 완전 보장 |
| 용량 한계 | 테라바이트급 | 메모리 크기 (수십~수백 GB) | 페타바이트급 |
| 비용 | 중간 | 높음 (DRAM 가격) | 낮음 |

**인메모리 DB 활용 아키텍처 패턴**

```text
[캐시 어사이드(Cache-Aside) 패턴]

애플리케이션
     │
     ├─1. 데이터 요청
     ▼
  Redis (캐시)
     │
     ├─ Cache Hit → 즉시 반환 (수십 μs)
     │
     └─ Cache Miss
          │
          ▼
        RDBMS
          │
          └─ 결과 반환 + Redis에 캐시 저장 (TTL 설정)
```

### 5. 실무 제언

**Cache Stampede(캐시 쏠림) 방지**

- **챌린지:** 인기 데이터의 캐시 TTL이 만료되는 순간, 다수의 요청이 동시에 캐시 미스가 발생하여 모두 DB로 몰리는 Cache Stampede(또는 Thundering Herd) 현상이 발생할 수 있습니다. 이는 순간적으로 DB에 과부하를 주어 장애로 이어질 수 있습니다.
- **제언:** TTL에 소폭의 무작위 지터(Jitter)를 추가하여 동시 만료를 분산시키고, 분산 락(Distributed Lock, Redis SETNX)으로 캐시 재생성을 직렬화하는 방법을 적용해야 합니다. 또한 중요 데이터는 TTL 만료 전 백그라운드에서 미리 갱신하는 Proactive Refresh 전략도 효과적입니다.

**메모리 용량 계획 및 데이터 증발 정책**

- **챌린지:** DRAM 용량은 디스크에 비해 한정적이며 비용이 높습니다. 캐시가 가득 찰 경우 어떤 데이터를 제거할지 결정하는 Eviction 정책을 잘못 설정하면, 자주 접근하는 Hot 데이터가 삭제되어 캐시 효율이 급감할 수 있습니다.
- **제언:** Redis의 Eviction 정책은 워크로드에 맞게 설정해야 합니다. 캐시 용도라면 `allkeys-lru`(가장 오래 사용되지 않은 키 제거)를 기본으로 설정하고, 세션처럼 TTL이 있는 데이터만 관리한다면 `volatile-lru`를 적용합니다. 메모리 사용률 모니터링을 통해 70% 이상이면 용량 증설을 계획해야 합니다.
