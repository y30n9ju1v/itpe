인메모리 데이터베이스(In-memory Database)

## 해설

### 출제 배경 및 의도

실시간 게임, 금융 HFT(고빈도 거래), 실시간 추천 시스템 등 마이크로초 단위의 응답이 요구되는 분야가 급증하면서, 인메모리 DB는 현대 고성능 아키텍처의 필수 구성 요소가 되었습니다. 전통적인 RDBMS는 데이터를 디스크에 저장하므로 I/O 레이턴시(수 ms~수십 ms)가 병목이 되는 반면, 인메모리 DB는 DRAM을 주 저장소로 사용하여 이를 나노초~마이크로초 수준으로 낮춥니다.

117회 기술사 시험에서는 IMDB의 개념과 구조, 영속성 해결 방안, 그리고 실무 활용 사례를 체계적으로 설명할 수 있는지를 검증합니다. 단순히 "빠른 DB"로 설명하는 수준을 넘어 RDB 스냅샷·AOF 영속성 메커니즘, Redis vs Memcached 비교, Cache-Aside 아키텍처 패턴까지 논술해야 고득점이 가능합니다.

### 1. 인메모리 데이터베이스, 개요

정 의  • 데이터를 디스크가 아닌 주 메모리(DRAM)에 상주시켜 I/O 병목을 제거하고 마이크로초 단위 응답을 실현하는 데이터베이스 시스템
       - 영속성 보장을 위해 AOF(Append Only File)·RDB 스냅샷을 비동기로 디스크에 기록

```
[디스크 기반 DB vs 인메모리 DB 응답 시간 비교]

RDBMS:
  쿼리 → 버퍼 풀 확인 → [Miss] → 디스크 I/O (5~15ms)
                       → [Hit]  → 메모리에서 반환 (0.1ms)

인메모리 DB:
  쿼리 → DRAM 직접 접근 (100ns~1μs) → 결과 반환

저장 매체 비교:
  DRAM 접근: ~100 nanoseconds
  SSD 접근:  ~100 microseconds (약 1,000배 느림)
  HDD 접근:  ~10 milliseconds  (약 100,000배 느림)
```

- DRAM은 SSD 대비 약 1,000배, HDD 대비 100,000배 빠른 접근 속도를 제공하여 초당 수백만 건의 읽기·쓰기를 처리 가능

### 2. 핵심 구조 및 영속성 메커니즘

1) 인메모리 DB 아키텍처 (Redis 기준)

| 구분 | 항목 | 설명 |
|------|------|------|
| 데이터 구조 | String | 단순 키-값, 카운터, 세션 토큰 |
| 데이터 구조 | Hash | 객체 속성 맵 (사용자 프로필) |
| 데이터 구조 | List | 순서 있는 큐/스택 (작업 큐) |
| 데이터 구조 | Sorted Set | 점수 기반 정렬 집합 (실시간 리더보드) |
| 데이터 구조 | Stream | 이벤트 로그, 메시지 브로커 |
| 영속성 레이어 | RDB 스냅샷 | 주기적 전체 데이터 바이너리 스냅샷 (빠른 재시작) |
| 영속성 레이어 | AOF | 모든 쓰기 명령을 로그 파일에 순차 추가 (데이터 유실 최소화) |
| 영속성 레이어 | RDB + AOF 혼합 | AOF로 일관성 확보, RDB로 빠른 복구 (권장 설정) |

```
[Redis 아키텍처 구조]

클라이언트들
     │
     ▼
┌─────────────────────────────────┐
│           Redis 서버             │
│  ┌──────────────────────────┐   │
│  │  인메모리 데이터 (DRAM)   │   │
│  │  String/Hash/List/Set/   │   │
│  │  Sorted Set/Stream       │   │
│  └──────────┬───────────────┘   │
│             │ (비동기)           │
│  ┌──────────┴───────────────┐   │
│  │  영속성 레이어 (Disk)     │   │
│  │  ┌────────┐ ┌──────────┐ │   │
│  │  │RDB 스냅│ │AOF 로그  │ │   │
│  │  │(주기적)│ │(매 명령) │ │   │
│  │  └────────┘ └──────────┘ │   │
│  └──────────────────────────┘   │
└─────────────────────────────────┘
```

- RDB 스냅샷은 빠른 재시작에, AOF는 데이터 유실 최소화에 유리하며, 운영 환경에서는 두 방식을 혼합하는 것이 권장됨

2) Redis vs Memcached 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| Redis | 데이터 구조 | String·Hash·List·Set·Sorted Set·Stream 다양 |
| Redis | 영속성 | RDB 스냅샷/AOF 지원 |
| Redis | 복제 | Master-Slave/Cluster 지원 |
| Redis | 트랜잭션 | MULTI/EXEC 지원 |
| Redis | 용도 | 다목적 캐시·세션·실시간 분석·메시지 브로커 |
| Memcached | 데이터 구조 | Key-Value(String)만 |
| Memcached | 영속성 | 미지원 (순수 캐시) |
| Memcached | 복제 | 미지원 |
| Memcached | 용도 | 단순 고속 캐시, 멀티스레드 구조로 CPU 효율 높음 |

- Redis는 기능 풍부·영속성 지원, Memcached는 단순 고속 캐시에 특화. 세션·리더보드·메시지 브로커가 필요하면 Redis, 순수 캐시 성능만 필요하면 Memcached 선택

### 3. 활용 아키텍처 패턴과 DB 유형 비교

1) Cache-Aside(캐시 어사이드) 패턴

| 구분 | 항목 | 설명 |
|------|------|------|
| Cache Hit | 흐름 | 애플리케이션 → Redis 조회 → 즉시 반환 (수십 μs) |
| Cache Miss | 흐름 | 애플리케이션 → Redis 미스 → RDBMS 조회 → Redis 캐시 저장(TTL 설정) → 반환 |
| 장점 | 성능 | RDBMS 부하 70~80% 감소 |
| 주의 | Cache Stampede | TTL 동시 만료 시 다수 요청이 RDBMS로 집중되는 Thundering Herd 현상 |

2) RDBMS vs 인메모리 DB vs 디스크 기반 NoSQL 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| RDBMS | 주 저장소/읽기 지연 | 디스크(SSD/HDD) / 수 ms |
| 인메모리 DB | 주 저장소/읽기 지연 | DRAM / 수십~수백 μs |
| 디스크 기반 NoSQL | 주 저장소/읽기 지연 | 디스크(SSD) / 수 ms |
| RDBMS | 영속성/용량 | 완전 보장 / 테라바이트급 |
| 인메모리 DB | 영속성/용량 | 부분적(설정 필요) / 수십~수백 GB |
| 디스크 기반 NoSQL | 영속성/용량 | 완전 보장 / 페타바이트급 |

- 인메모리 DB는 압도적인 성능을 제공하지만 메모리 용량 제한과 비용(DRAM 가격)이 높으므로, 캐시·세션·실시간 분석 등 Hot 데이터 계층에 선택적으로 적용하는 계층화 아키텍처가 권장됨.  "끝"

### 실무 제언

**Cache Stampede(캐시 쏠림) 방지**
- **챌린지**: 인기 데이터의 캐시 TTL이 만료되는 순간 다수의 요청이 동시에 캐시 미스 발생하여 모두 RDBMS로 몰리는 Thundering Herd 현상이 발생하여 DB 과부하로 장애가 이어질 수 있음
- **제언**: TTL에 소폭의 무작위 지터(Jitter)를 추가하여 동시 만료를 분산시키고, 분산 락(Redis SETNX)으로 캐시 재생성을 직렬화하며, 중요 데이터는 TTL 만료 전 백그라운드에서 미리 갱신하는 Proactive Refresh 전략 적용

**메모리 용량 계획 및 Eviction 정책**
- **챌린지**: DRAM 용량은 한정적이며 캐시가 가득 찰 경우 Eviction 정책을 잘못 설정하면 자주 접근하는 Hot 데이터가 삭제되어 캐시 효율이 급감함
- **제언**: 캐시 용도는 `allkeys-lru`(가장 오래 사용되지 않은 키 제거), TTL 있는 데이터만 관리하면 `volatile-lru`를 적용하고, 메모리 사용률 70% 이상이면 용량 증설 계획 수립

**인메모리 DB 계층화 아키텍처**
- **챌린지**: 모든 데이터를 인메모리 DB에 저장하려 하면 용량 부족과 높은 비용 문제가 발생하고, DRAM 장애 시 데이터 유실 위험이 있음
- **제언**: 자주 읽히는 Hot 데이터(세션, 캐시, 리더보드)는 인메모리 DB, 영구 저장이 필요한 Cold 데이터는 RDBMS/NoSQL에 저장하는 계층화 아키텍처를 적용하고, AOF fsync 정책으로 영속성 수준을 비즈니스 요구에 맞게 조정
