## 해설

### 출제 배경 및 의도

본 문제는 기술사 시험에서 가장 선호하는 유형인 '구체적 사례(수강테이블)를 제시하고 실무 해결 능력을 검증'하는 고전적인 문제입니다. 딕셔너리식 정의를 암기한 수험생과 실제 데이터 구조를 다룰 줄 아는 기술사를 변별하기 위해 출제됩니다.

클라우드 네이티브 전환이나 레거시 시스템 고도화 사업 시, 축적된 데이터의 품질을 진단(Data Profiling)하고 정제하기 위한 첫 단추가 바로 이상현상 분석과 무손실 분해입니다. 채점관은 이상현상의 개념적 정의에 더해, 수강테이블에서 어떤 구체적인 상황에서 이상현상이 발생하는지를 명확히 서술하는 답안을 고득점으로 평가합니다.

### 1. 수강테이블 이상현상과 정규화, 개요

정 의  • 수강테이블 내 잘못된 함수 종속성(부분 함수 종속)으로 발생하는 3대 이상현상을 식별하고, 관계 대수의 프로젝션(Projection) 연산으로 테이블을 무손실 분해하는 정규화 프로세스
       - 대상 스키마: 수강테이블 (학번, 과목번호, 학생이름, 학과, 과목명, 성적), PK: (학번, 과목번호)

```
[수강테이블 부분 함수 종속 구조]

       학번 ─────────────────────> 학생이름, 학과   (부분 FD: 학번에만 종속)
         │
(학번, 과목번호) ─────────────────> 성적             (완전 FD: 복합키 전체에 종속)
         │
     과목번호 ────────────────────> 과목명            (부분 FD: 과목번호에만 종속)

※ 부분 FD 존재 → 삽입/삭제/갱신 이상현상 발생
```

- 하나의 테이블에 서로 다른 업무 주체(학생, 과목, 성적)가 혼재되어 데이터 중복을 야기하는 것이 이상현상의 근본 원인이다.

### 2. 가. 이상현상 3가지 및 나. 해결방안

1) 가. 이상현상(Anomaly) 3가지와 발생 이유

| 유형 | 구체적 사례 | 발생 조건 |
|------|-------------|-----------|
| **삽입 이상** | 신입생(학번:26005, 강감찬, 전기공학)이 수강신청 전에는 PK(과목번호) NULL 불가 제약으로 학적 정보 등록 불가 | 복합키(학번+과목번호)에서 과목번호 없이 삽입 불가 |
| **삭제 이상** | 학번 26002 이순신의 DB-100 수강 취소(행 삭제) 시, 유일하게 존재하던 '이순신=컴퓨터공학부' 학적 정보까지 연쇄 소실 | 하나의 튜플에 수강 정보 + 학생 정보가 혼재 |
| **갱신 이상** | 학번 26001 홍길동이 3과목 수강 → 3행 중복 저장. 학과 변경 시 일부 행만 수정하면 동일 학번에 다른 학과가 공존 | 중복 저장된 속성의 부분 수정으로 불일치 발생 |

- 발생 이유: 복합키(학번, 과목번호) 중 `학생이름, 학과`는 학번에만, `과목명`은 과목번호에만 종속되는 부분 함수 종속성(Partial FD)이 존재하기 때문이다.

2) 나. 해결방안: 제2정규형(2NF) 수행

기본키의 진부분집합에 일반 속성이 종속되는 구조적 결함을 치유하기 위해 제2정규형(2NF)을 적용합니다. 결정자(X)를 기준으로 테이블을 프로젝션(Projection) 분할하여 완전 함수 종속(Full FD) 상태로 전환하며, 부모 테이블의 PK를 자식 테이블에 FK로 전이하여 무결성을 유지합니다.

### 3. 다. 테이블 재구성 결과

**릴레이션 스키마:**

- **학생 테이블** (학번, 학생이름, 학과) → *PK: 학번*
- **과목 테이블** (과목번호, 과목명) → *PK: 과목번호*
- **성적 테이블** (학번, 과목번호, 성적) → *PK: (학번, 과목번호), FK: 학번, 과목번호*

| 학번 (PK) | 학생이름 | 학과 |
|-----------|----------|------|
| 26001 | 홍길동 | 컴퓨터공학 |
| 26002 | 이순신 | 컴퓨터공학 |
| 26005 | 강감찬 | 전기공학 ← 수강신청 없어도 등록 가능 (삽입이상 해결) |

| 과목번호 (PK) | 과목명 |
|--------------|--------|
| DB-100 | 데이터베이스 |
| SE-200 | 소프트웨어공학 |

| 학번 (PK,FK) | 과목번호 (PK,FK) | 성적 |
|-------------|----------------|------|
| 26001 | DB-100 | A+ |
| 26001 | SE-200 | A0 |
| 26002 | SE-200 | B+ ← DB-100 삭제해도 이순신 학적 안전 (삭제이상 해결) |

- 3개 테이블 분해 후 자연 조인(Natural Join)하면 원본 수강테이블과 동일한 데이터 복원 가능(무손실 분해 검증).  "끝"

### 실무 제언

**무조건적인 분해의 맹점**
- **챌린지**: 정규화로 테이블이 3개로 분리되면 무결성은 완벽해지지만, 대량 수강 현황 조회 시 3-way JOIN이 필요하여 시스템 성능(I/O 부하)이 저하될 수 있다.
- **제언**: 논리 설계 단계에서는 정규화를 엄격히 수행하여 이상현상을 원천 차단하고, 물리 구현 단계에서는 인덱스 튜닝·캐싱(Redis)을 우선 적용한 후 최후 수단으로 통제된 반정규화를 검토해야 한다.

**참조 무결성 제약의 물리적 구현 전략**
- **챌린지**: 성적 테이블에 존재하지 않는 학번이나 과목번호가 입력되면 데이터 정합성이 깨지며, 대용량 트랜잭션 환경에서 FK 제약의 검증 부하가 문제가 된다.
- **제언**: DBMS 레벨 FK 제약조건을 통해 참조 무결성을 강제하되, 대용량 환경에서는 FK 제약을 논리 모델에만 정의하고 애플리케이션 레이어에서 입력값 검증 후 데이터 품질 프로파일링 도구로 상시 검증하는 전략을 선택적으로 적용해야 한다.

**레거시 시스템 데이터 품질 정비 시 활용**
- **챌린지**: 레거시 시스템의 비정규화 테이블에서 이상현상이 반복 발생하면 데이터 신뢰성이 훼손되어 AI/BI 분석의 정확도가 저하된다.
- **제언**: 데이터 프로파일링(Data Profiling) 도구를 활용하여 부분 함수 종속성이 존재하는 테이블을 자동 탐지하고, 본 문제의 수강테이블 사례처럼 결정자 단위 무손실 분해를 통해 데이터 품질을 체계적으로 정비해야 한다.
