## 문제

데이터베이스에서 정규화는 이상현상(Anomaly)이 있는 릴레이션(Relation)을 해결하기 위한 방법이다. 다음의 `<수강테이블>`을 활용하여 설명하시오.

가. 이상현상 3 가지와 발생 이유  
나. 해결방안  
다. 테이블 재구성

## 해설

> **구체적 사례로 검증하는 정규화 실전 능력**  
> 수강테이블의 부분 함수 종속성을 식별하고 2NF 무손실 분해로 이상현상을 해결

본 문제는 기술사 시험에서 가장 선호하는 유형인 '구체적 사례(수강테이블)를 제시하고 실무 해결 능력을 검증'하는 고전적인 문제입니다. 딕셔너리식 정의를 암기한 수험생과 실제 데이터 구조를 다룰 줄 아는 기술사를 변별하기 위해 출제됩니다. 대상 원시 스키마는 **수강테이블 (학번, 과목번호, 학생이름, 학과, 과목명, 성적)** 이며, 기본키(PK, Primary Key)는 `(학번, 과목번호)` 복합키입니다.

### 1. 출제 배경 및 의도

클라우드 네이티브 전환이나 레거시 시스템 고도화 사업 시, 축적된 데이터의 품질을 진단(Data Profiling)하고 정제하기 위한 첫 단추가 바로 이상현상 분석과 무손실 분해입니다. 채점관은 이상현상의 개념적 정의에 더해, 수강테이블에서 어떤 구체적인 상황에서 이상현상이 발생하는지를 명확히 서술하는 답안을 고득점으로 평가합니다.

### 2. 개념의 본질과 작동 메커니즘

수강테이블 내에 존재하는 속성 간의 잘못된 함수 종속성(부분 함수 종속)으로 인해 발생하는 3대 이상현상을 식별하고, 관계 대수의 프로젝션(Projection) 연산을 통해 테이블을 무손실 분해하는 정규화 프로세스입니다.

- **Input:** 복합키 `(학번, 과목번호)` 하위에 일반 속성인 `학생이름`, `학과`, `과목명`, `성적`이 혼재되어 중복을 양산하는 원시 수강테이블
- **Mechanism:** 복합키의 일부에 일반 속성이 종속되는 부분 함수 종속성을 식별 → 테이블을 결정자 단위로 쪼개어 종속성 제거
- **Output:** 이상현상이 완전히 박멸되고 데이터 무결성이 보장된 3개의 분리된 릴레이션 스키마

#### 가. 이상현상(Anomaly) 3가지와 발생 이유

**1. 삽입 이상 (Insertion Anomaly)**

- **사례:** 신입생(`학번: 26005, 이름: 강감찬, 학과: 전기공학`)이 아직 수강신청을 하지 않아 `과목번호`가 없는 상태에서 학적 정보를 등록하려면, 기본키 제약(PK는 NULL 불가)에 의해 `과목번호`에 가짜 값을 넣어야만 삽입이 가능합니다.

**2. 삭제 이상 (Deletion Anomaly)**

- **사례:** 학번 `26002`인 이순신 학생이 `DB-100(데이터베이스)` 과목을 전면 취소하여 해당 튜플(행)을 삭제하면, 수강 정보를 지웠을 뿐인데 이 테이블에 유일하게 남아있던 '이순신 학생이 컴퓨터공학부 소속이다'라는 학적 정보까지 연쇄적으로 유실됩니다.

**3. 갱신 이상 (Update Anomaly)**

- **사례:** 학번 `26001`인 홍길동 학생이 다수의 과목을 신청하여 테이블 내에 데이터가 3줄 이상 중복 저장된 상태에서, 학과가 `컴퓨터공학`에서 `AI소프트웨어학과`로 변경될 때, 일부 행만 수정하고 누락할 경우 동일한 학번의 학생이 서로 다른 학과에 동시에 소속되어 있는 데이터 불일치 현상이 나타납니다.

**발생 이유:** 수강테이블의 기본키는 복합키인 `(학번, 과목번호)`이지만, 일반 속성 중 `학생이름, 학과`는 오직 `학번`에만 종속되고, `과목명`은 오직 `과목번호`에만 종속되는 부분 함수 종속성(Partial Functional Dependency)이 존재하기 때문입니다. 하나의 테이블에 서로 다른 업무 주체(학생, 과목, 성적)가 혼재되어 데이터 중복을 야기하는 것이 근본 원인입니다.

#### 나. 해결방안: 제2정규형(2NF) 수행

기본키의 진부분집합에 일반 속성이 종속되는 구조적 결함을 치유하기 위해 제2정규형(2NF)을 적용합니다. 관계 대수의 프로젝션(Projection) 연산을 통해 결정자(X)를 기준으로 테이블을 분할하여, 모든 일반 속성이 기본키에만 매핑되는 **완전 함수 종속(Full FD)** 상태로 전환합니다. 분해된 테이블들을 다시 조인했을 때 원래의 데이터 구조로 정보 손실 없이 복원될 수 있도록, 부모 테이블의 기본키(PK)를 자식 테이블에 외래키(FK) 형태로 전이시켜 연결 고리를 유지합니다.

#### 다. 테이블 재구성 결과

**릴레이션 스키마:**

- **학생 테이블** (학번, 학생이름, 학과) → *PK: 학번*
- **과목 테이블** (과목번호, 과목명) → *PK: 과목번호*
- **성적 테이블** (학번, 과목번호, 성적) → *PK: (학번, 과목번호), FK: 학번, 과목번호*

**[학생 테이블] — 삽입/삭제 이상 해결**

| 학번 (PK) | 학생이름 | 학과 |
| --- | --- | --- |
| 26001 | 홍길동 | 컴퓨터공학 |
| 26002 | 이순신 | 컴퓨터공학 |
| 26005 | 강감찬 | 전기공학 *(수강신청을 안 해도 학적 등록 가능: 삽입이상 해결)* |

**[과목 테이블] — 과목 도메인의 독립성 보장**

| 과목번호 (PK) | 과목명 |
| --- | --- |
| DB-100 | 데이터베이스 |
| SE-200 | 소프트웨어공학 |

**[성적 테이블] — 수강신청 및 성적 트랜잭션 전용 매핑 테이블**

| 학번 (PK, FK) | 과목번호 (PK, FK) | 성적 |
| --- | --- | --- |
| 26001 | DB-100 | A+ |
| 26001 | SE-200 | A0 |
| 26002 | SE-200 | B+ *(DB-100 과목을 취소하여 이 행을 지워도 학생 테이블의 이순신 학적은 안전함: 삭제이상 해결)* |

### 3. 핵심 키워드 (Must-Have)

- **가. 이상현상 관점:** 삽입 이상, 삭제 이상, 갱신(수정) 이상, 불필요한 데이터 중복, 주식별자 제약(Null 불가), 정보의 연쇄 소실
- **나. 해결방안 관점:** 함수 종속성 분석(FDA), 복합 기본키 `(학번, 과목번호)`, 부분 함수 종속(Partial FD), 제2정규형(2NF), 무손실 분해(Lossless Decomposition)
- **다. 테이블 재구성 관점:** 프로젝션(Projection), 기본키(PK)/외래키(FK) 전이, 무결성 제약조건

### 4. 맥락과 비교

이상현상을 유발하는 근본 원인은 릴레이션에 함수 종속성이 엉켜있기 때문이며, 이를 해결하는 정규화 단계는 아래와 같은 선후 관계를 갖습니다.

| 정규화 단계 | 제거 대상 종속성 | 수강테이블 사례 적용 및 결과 |
| --- | --- | --- |
| **제1정규형 (1NF)** | 원자값이 아닌 도메인 | 모든 속성이 단일 값을 갖도록 구성 (수강테이블의 기본 전제) |
| **제2정규형 (2NF)** | **부분 함수 종속** (Partial FD) | **본 문제의 핵심.** 복합키 일부에 종속된 학생 정보와 과목 정보를 별도 테이블로 분리 |
| **제3정규형 (3NF)** | **이행적 함수 종속** (Transitive FD) | 일반 속성 간의 인과관계(예: 학과 → 학과사무실 위치 등)가 추가 존재할 경우 이를 다시 분리 |

### 5. 실무 제언

**무조건적인 분해의 맹점**

- **챌린지:** 정규화를 거쳐 테이블이 3개로 쪼개지면 무결성이 완벽해지지만, 대량의 수강 현황 조회를 수행할 때 3개 테이블을 조인(Join)해야 하므로 시스템 성능(I/O 부하)이 저하될 수 있습니다.
- **제언:** 논리 설계 단계에서는 본 답안처럼 정규화를 엄격히 수행하여 이상현상을 원천 차단하고, 최종 물리 구현 단계에서 트랜잭션 빈도를 분석하여 성능 병목이 예상되는 곳에 한해 인덱스 튜닝이나 애플리케이션 캐싱(Redis 등)을 우선 적용한 후, 최후의 수단으로 통제된 반정규화를 검토해야 합니다. 참조 무결성 제약(RI Constraint)은 성적 테이블에 데이터를 입력할 때 반드시 학생 테이블에 존재하는 학번과 과목 테이블에 존재하는 과목번호만 입력되도록 DBMS 레벨에서 강제해야 합니다.
