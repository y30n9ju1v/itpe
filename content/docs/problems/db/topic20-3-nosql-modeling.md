## 해설

### 출제 배경 및 의도

빅데이터·실시간 처리 환경에서 MongoDB, Cassandra 등 NoSQL DB를 활용하는 프로젝트가 증가하면서, 데이터 아키텍트가 NoSQL 모델링 패턴과 절차를 체계적으로 이해해야 하는 필요성이 높아졌습니다. 124회 기술사 시험에서 출제된 이 문제는 NoSQL의 단순한 개념 설명을 넘어 실제 설계 방법론을 묻는 심화 문제입니다.

RDBMS 모델링은 "데이터를 먼저 정규화하고, 이후 쿼리로 조합한다"는 철학인 반면, NoSQL 모델링은 "어떤 쿼리가 들어올지를 먼저 분석하고, 그 쿼리에 최적화된 구조로 데이터를 비정규화한다"는 Query-Driven 접근입니다. 채점관은 이 근본적 철학 차이와 Embedding·Referencing·Bucketing 패턴을 상황에 맞게 선택·적용할 수 있는지를 검증합니다.

### 1. NoSQL 모델링, 개요

정 의  • 쿼리 패턴(Query Pattern)을 먼저 분석하여 그에 최적화된 비정규화 구조로 데이터를 설계하는 Query-Driven 데이터 모델링 방법론
       - RDBMS의 정규화 중심 접근과 달리, 읽기 성능 최적화를 위해 의도적 비정규화를 적용

```
RDBMS 모델링:
요구사항 → 엔티티 도출 → 정규화 → ERD → 테이블 생성 → SQL 쿼리 작성

NoSQL 모델링 (Query-Driven):
요구사항 → 쿼리 패턴 분석 → 액세스 패턴 정의 → 비정규화/어그리게이션 → 컬렉션/테이블 설계
```

- NoSQL을 RDBMS처럼 정규화 기반으로 설계하면 다중 조회와 애플리케이션 레벨 JOIN이 증가하여 성능을 저하시키는 안티패턴에 빠짐

### 2. NoSQL 모델링 3대 패턴

1) Embedding(임베딩) 패턴

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 관련 데이터를 하나의 문서 내에 중첩(Nested) 구조로 내장 |
| 장점 | 읽기 성능 | 단일 읽기로 모든 관련 데이터 조회, 네트워크 왕복 최소화 |
| 장점 | 원자성 | 단일 문서 내 원자적 업데이트 보장 |
| 단점 | 문서 크기 | MongoDB 16MB 제한, 무한 성장 배열 시 I/O 급증 |
| 적용 기준 | 관계 유형 | 1:1 또는 1:N(N이 작고 고정적), 항상 함께 읽히는 데이터 |

```
[Embedding 패턴 - 주문과 주문상품 통합]
{
  "_id": "order_001",
  "customer_id": "cust_100",
  "items": [                   ← 내장(Embed)
    {"product": "노트북", "qty": 1, "price": 1500000},
    {"product": "마우스",  "qty": 2, "price": 30000}
  ],
  "total": 1560000
}
```

- Embedding은 읽기 집중 워크로드에 최적이나, N이 수백 건을 초과할 가능성이 있으면 Referencing으로 전환 필요

2) Referencing(참조) 패턴

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 별도 컬렉션에 저장하고 ID로 참조 (RDBMS 외래 키와 유사) |
| 장점 | 데이터 중복 | 중복 최소화, 독립적 업데이트 용이, 문서 크기 관리 |
| 단점 | 읽기 성능 | 관련 데이터 조회 시 다중 쿼리 또는 $lookup 집계 필요 |
| 적용 기준 | 관계 유형 | 1:N(N이 크거나 무제한), 독립적으로 조회·업데이트되는 데이터 |

- Referencing은 쓰기 집중·독립 엔티티에 적합하나, 조회 시 Application-side Join 비용 발생을 고려해야 함

3) Bucketing(버킷팅) 패턴

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 시계열·이벤트 데이터를 시간/크기 단위로 묶어 하나의 문서(버킷)에 저장 |
| 장점 | 문서 수 | 문서 수 대폭 감소, 집계(Aggregation) 연산 최적화, 인덱스 효율 향상 |
| 장점 | 사전 집계 | 버킷 내 min/max/avg/sum 사전 집계 필드로 리포팅 성능 극대화 |
| 단점 | 설계 복잡도 | 버킷 크기 설계 필요, 데이터 추가 시 업데이트 충돌 가능 |
| 적용 기준 | 데이터 유형 | IoT 시계열, 로그, 이벤트 스트림 데이터 |

- Bucketing은 수억 건의 시계열 데이터를 집계할 때 컬렉션 스캔을 버킷 단위 스캔으로 대체하여 수십~수백 배의 성능 향상을 달성함

### 3. NoSQL 모델링 절차와 패턴 비교

1) Query-Driven 모델링 절차

| 구분 | 항목 | 설명 |
|------|------|------|
| Step 1 | 요구사항 및 쿼리 패턴 분석 | 주요 기능 목록화, 읽기/쓰기 비율, 조회 조건·정렬·집계 식별 |
| Step 2 | 엔티티 및 관계 도출 | 핵심 엔티티 정의, 관계 유형 파악 (1:1/1:N/M:N) |
| Step 3 | 액세스 패턴별 문서 구조 설계 | Embedding vs Referencing vs Bucketing 패턴 결정 |
| Step 4 | 인덱스 전략 수립 | 단일/복합/TTL 인덱스 설계 |
| Step 5 | 샤딩 키 선정 | 카디널리티 높은 필드, Hot Spot 방지, 범위 쿼리 균형 고려 |
| Step 6 | 프로토타입 및 성능 검증 | EXPLAIN 실행 계획 검토, 부하 테스트 |

2) 3대 패턴 종합 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| Embedding | 읽기 성능 | 매우 빠름 (단일 조회) |
| Embedding | 쓰기 성능 | 느림 (대형 문서 재기록) |
| Embedding | 적합 데이터 | 부모-자식 관계, 함께 읽히는 데이터 |
| Referencing | 읽기 성능 | 느림 (다중 조회) |
| Referencing | 쓰기 성능 | 빠름 (독립 문서) |
| Referencing | 적합 데이터 | 독립 엔티티, 무제한 성장 데이터 |
| Bucketing | 읽기 성능 | 빠름 (집계 최적화) |
| Bucketing | 쓰기 성능 | 중간 |
| Bucketing | 적합 데이터 | IoT 시계열, 로그, 이벤트 스트림 |

- NoSQL 모델링은 단일 패턴을 고수하기보다 도메인·쿼리 특성에 따라 Embedding·Referencing·Bucketing을 혼합 적용하는 하이브리드 전략이 실무에서 가장 효과적임.  "끝"

### 실무 제언

**임베딩의 함정 - 무한 성장 배열**
- **챌린지**: 처음에는 1:N 관계를 Embedding으로 설계했으나 N이 예상보다 빠르게 증가하여 MongoDB 16MB 제한을 초과하거나 전체 문서 재기록으로 I/O가 급증하는 경우가 빈번함
- **제언**: N이 수백 건을 초과할 가능성이 있으면 Referencing으로 전환하고, 최근 N건만 임베딩하고 나머지는 별도 컬렉션에 보관하는 Extended Reference 패턴을 활용

**Bucketing 패턴과 사전 집계(Pre-Aggregation)**
- **챌린지**: IoT·로그 데이터를 문서 단위로 저장하면 하루 수억 건의 문서가 생성되어 집계 쿼리 시 전체 컬렉션 스캔으로 응답 시간이 수십 초~수 분에 달할 수 있음
- **제언**: 버킷팅 패턴을 적용하면서 버킷 문서 내에 min/max/avg/sum 등의 사전 집계 필드를 함께 저장하면 대시보드·리포팅 쿼리가 버킷 레벨의 집계 값만 읽어 수십~수백 배의 성능 향상을 달성

**샤딩 키 선정의 중요성**
- **챌린지**: 카디널리티가 낮은 필드(예: 성별, 상태코드)를 샤딩 키로 선택하면 특정 샤드에 데이터가 집중되는 Hot Spot이 발생하여 수평 확장 효과가 사라짐
- **제언**: 샤딩 키는 카디널리티가 높고 쓰기가 균등하게 분산되는 필드(예: 사용자 ID, UUID)를 선택하고, 범위 쿼리 빈도가 높으면 Hash 샤딩 대신 Range 샤딩과 Compound 인덱스를 조합하여 설계
