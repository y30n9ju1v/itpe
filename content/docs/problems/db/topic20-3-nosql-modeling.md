## 문제

NoSQL 모델링 패턴 3가지 및 NoSQL 모델링 절차를 설명하시오.

## 출제 정보

|------|------|
| 토픽(대) | NOSQL |
| 토픽(소) | NOSQL |

## 해설

> **쿼리 패턴이 스키마를 결정하는 역발상 설계**  
> NoSQL 모델링은 RDBMS의 정규화 중심이 아닌 Query-Driven 접근으로, Embedding·Referencing·Bucketing 패턴을 상황에 맞게 선택해야 합니다.

RDBMS 모델링은 "데이터를 먼저 정규화하고, 이후 쿼리로 조합한다"는 철학인 반면, NoSQL 모델링은 "어떤 쿼리가 들어올지를 먼저 분석하고, 그 쿼리에 최적화된 구조로 데이터를 비정규화한다"는 Query-Driven 접근입니다. 이 근본적 철학 차이를 이해하지 못하면 NoSQL을 RDBMS처럼 설계하는 안티패턴에 빠집니다.

### 1. 출제 배경 및 의도

124회 기술사 시험에서 출제된 이 문제는 NoSQL의 단순한 개념 설명을 넘어 실제 설계 방법론을 묻는 심화 문제입니다. 빅데이터·실시간 처리 환경에서 MongoDB, Cassandra 등 NoSQL DB를 활용하는 프로젝트가 증가하면서, 데이터 아키텍트가 NoSQL 모델링 패턴과 절차를 체계적으로 알고 있는지를 검증하려는 의도입니다.

### 2. 개념의 본질과 작동 메커니즘

#### NoSQL 모델링의 기본 원칙

```text
[RDBMS vs NoSQL 모델링 접근 방식 비교]

RDBMS 모델링:
요구사항 → 엔티티 도출 → 정규화 → ERD → 테이블 생성 → SQL 쿼리 작성

NoSQL 모델링 (Query-Driven):
요구사항 → 쿼리 패턴 분석 → 액세스 패턴 정의 → 비정규화/어그리게이션 → 컬렉션/테이블 설계
```

**NoSQL 모델링 3대 패턴**

#### 패턴 1: Embedding (내장/임베딩) 패턴

관련 데이터를 하나의 문서 내에 중첩(Nested) 구조로 내장하는 패턴입니다.

```text
[Embedding 패턴 예시 - 주문과 주문상품]

// RDBMS 방식: 2개 테이블 JOIN 필요
Orders 테이블: order_id, customer_id, order_date
OrderItems 테이블: item_id, order_id, product_name, qty, price

// Embedding 방식: 단일 문서로 통합
{
  "_id": "order_001",
  "customer_id": "cust_100",
  "order_date": "2024-01-15",
  "items": [                          ← 내장(Embed)
    {"product_name": "노트북", "qty": 1, "price": 1500000},
    {"product_name": "마우스",  "qty": 2, "price": 30000}
  ],
  "total": 1560000
}
```

| 구분 | 내용 |
|------|------|
| **장점** | 단일 읽기로 모든 관련 데이터 조회, 네트워크 왕복 최소화, 원자적 업데이트 |
| **단점** | 문서 크기 증가(MongoDB 16MB 제한), 내장 데이터 독립 조회 어려움 |
| **적용 기준** | 1:1 또는 1:N(N이 작고 고정적), 항상 함께 읽히는 데이터 |

#### 패턴 2: Referencing (참조) 패턴

별도 컬렉션에 저장하고 ID로 참조하는 패턴으로 RDBMS의 외래 키와 유사합니다.

```text
[Referencing 패턴 예시 - 사용자와 게시글]

// Users 컬렉션
{ "_id": "user_001", "name": "홍길동", "email": "hong@example.com" }

// Posts 컬렉션 (user_id로 참조)
{
  "_id": "post_001",
  "user_id": "user_001",    ← 참조(Reference)
  "title": "NoSQL 모델링",
  "content": "..."
}
// Application 레벨에서 2번 조회 또는 $lookup 집계 사용
```

| 구분 | 내용 |
|------|------|
| **장점** | 데이터 중복 최소화, 독립적 업데이트 용이, 문서 크기 관리 |
| **단점** | 관련 데이터 조회 시 다중 쿼리 필요, JOIN 비용 발생 |
| **적용 기준** | 1:N(N이 크거나 무제한), 독립적으로 조회·업데이트되는 데이터 |

#### 패턴 3: Bucketing (버킷팅) 패턴

시계열·이벤트 데이터를 시간 단위나 크기 단위로 묶어 하나의 문서(버킷)에 저장하는 패턴입니다.

```text
[Bucketing 패턴 예시 - IoT 센서 데이터]

// 안티패턴: 매 측정값마다 1개 문서 (수억 건 → 성능 저하)
{ "_id": "...", "sensor_id": "s001", "ts": "10:00:01", "temp": 22.5 }
{ "_id": "...", "sensor_id": "s001", "ts": "10:00:02", "temp": 22.6 }
...

// Bucketing 패턴: 1시간 단위로 묶어 1개 문서에 저장
{
  "_id": "s001_2024011510",          ← 센서ID + 시간 버킷 키
  "sensor_id": "s001",
  "hour": "2024-01-15T10:00",
  "count": 3600,
  "measurements": [22.5, 22.6, 22.7, ...],  ← 1시간치 데이터
  "avg_temp": 22.6,
  "min_temp": 22.1,
  "max_temp": 23.0
}
```

| 구분 | 내용 |
|------|------|
| **장점** | 문서 수 대폭 감소, 집계(Aggregation) 연산 최적화, 인덱스 효율 향상 |
| **단점** | 버킷 크기 설계 필요, 데이터 추가 시 업데이트 충돌 가능 |
| **적용 기준** | IoT 시계열, 로그, 이벤트 스트림 데이터 |

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 모델링 철학 | Query-Driven Design, 비정규화(Denormalization), 어그리게이션(Aggregation) |
| 3대 패턴 | Embedding, Referencing, Bucketing |
| 임베딩 관련 | Nested Document, 원자적 업데이트, 16MB 제한 |
| 참조 관련 | $lookup, Application-side Join, 데이터 중복 최소화 |
| 버킷팅 관련 | 시계열, 버킷 단위(시간/크기), 사전 집계 |
| 모델링 절차 | 요구사항 분석, 액세스 패턴, 엔티티 설계, 쿼리 패턴 검증 |

#### NoSQL 모델링 절차

```text
[NoSQL 모델링 절차 (Query-Driven Approach)]

Step 1. 요구사항 및 쿼리 패턴 분석
  ├── 주요 기능 목록화
  ├── 예상 읽기/쓰기 비율 분석
  └── 필요한 조회 조건·정렬·집계 식별

Step 2. 엔티티 및 관계 도출
  ├── 핵심 엔티티 정의 (RDBMS ERD 참고)
  └── 엔티티 간 관계 파악 (1:1 / 1:N / M:N)

Step 3. 액세스 패턴별 문서 구조 설계
  ├── Embedding vs Referencing 패턴 결정
  ├── Bucketing 패턴 적용 여부 결정
  └── 문서 크기 및 성장 예측

Step 4. 인덱스 전략 수립
  ├── 단일 필드 인덱스
  ├── 복합 인덱스 (쿼리 조건 + 정렬 필드)
  └── TTL 인덱스 (시계열·로그 데이터 자동 삭제)

Step 5. 샤딩 키 선정
  ├── 카디널리티 높은 필드 선택
  ├── 쓰기 집중(Hot Spot) 방지
  └── 범위 쿼리와 균등 분산 균형 고려

Step 6. 프로토타입 및 성능 검증
  ├── 대표 쿼리 실행 계획(EXPLAIN) 검토
  └── 예상 트래픽 기준 부하 테스트
```

### 4. 맥락과 비교

| 비교 항목 | Embedding | Referencing | Bucketing |
|----------|-----------|-------------|-----------|
| 데이터 중복 | 높음 | 낮음 | 중간 |
| 읽기 성능 | 매우 빠름 (단일 조회) | 느림 (다중 조회) | 빠름 (집계 최적화) |
| 쓰기 성능 | 느림 (대형 문서) | 빠름 | 중간 |
| 일관성 | 원자적 업데이트 용이 | 다중 문서 업데이트 복잡 | 중간 |
| 적합 데이터 | 부모-자식 관계 | 독립적 엔티티 | 시계열·이벤트 |
| RDBMS 유사 개념 | 비정규화 테이블 | 정규화 + 외래 키 | 파티셔닝 |

### 5. 실무 제언

**임베딩의 함정 - 무한 성장 배열**

- **챌린지:** 처음에는 1:N 관계를 Embedding으로 설계했으나, N이 예상보다 빠르게 증가하여 MongoDB 문서 16MB 제한을 초과하거나 문서 업데이트 시 전체 문서 재기록으로 I/O가 급증하는 경우가 빈번히 발생합니다. SNS 댓글이나 상품 리뷰처럼 무제한 증가 가능한 데이터를 임베딩으로 설계하면 심각한 문제가 생깁니다.
- **제언:** N이 수백 건을 초과할 가능성이 있으면 Referencing으로 전환하고, 최근 N건만 임베딩하고 나머지는 별도 컬렉션에 보관하는 하이브리드 Extended Reference 패턴을 활용해야 합니다.

**Bucketing 패턴과 사전 집계(Pre-Aggregation)**

- **챌린지:** IoT·로그 데이터를 문서 단위로 저장하면 하루 수억 건의 문서가 생성되어, 집계 쿼리 시 전체 컬렉션 스캔으로 응답 시간이 수십 초~수 분에 달할 수 있습니다.
- **제언:** 버킷팅 패턴을 적용하면서 버킷 문서 내에 min/max/avg/sum 등의 사전 집계 필드를 함께 저장하면, 대시보드·리포팅 쿼리가 전체 데이터를 재집계하지 않고 버킷 레벨의 집계 값만 읽으면 되어 수십~수백 배의 성능 향상을 얻을 수 있습니다.
