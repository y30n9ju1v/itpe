## 해설

> **4대 무결성 유형을 정의하고, 선언적·절차적·애플리케이션·분산 트랜잭션 기법으로 유지하는 데이터 품질 보장 체계**  
> 데이터베이스 무결성(Data Integrity)과 입체적 품질 유지 전략

AI·빅데이터 분석이 비즈니스 핵심이 된 현대에 원천 데이터의 신뢰성은 시스템 전체의 성패를 좌우합니다. MSA·클라우드 기반 데이터 레이크 환경이 확산되면서 데이터 오염과 참조 단절 현상이 급증하고 있으므로, 데이터 아키텍트(DA)의 무결성 관리 체계가 더욱 중요해지고 있습니다.

### 1. 출제 배경 및 의도

데이터베이스에 저장된 원천 데이터의 신뢰성이 시스템 전체의 성패를 좌우하고 있습니다. 틀린 데이터를 넣으면 틀린 결과가 나오는 'GIGO(Garbage In, Garbage Out)' 현상을 막기 위한 최전방 방어선이 데이터 무결성입니다.

채점관은 무결성의 4대 유형을 단순 암기하는 수준을 넘어, 각 유형을 구현하는 선언적·절차적·분산 트랜잭션 기법을 다각적으로 설명할 수 있는 데이터 아키텍트의 실무 역량을 검증하고자 본 문제를 출제합니다.

### 2. 개념의 본질과 작동 메커니즘

데이터베이스 내에 저장된 데이터의 정확성(Accuracy), 완전성(Completeness), 유효성(Validity) 및 일관성(Consistency)을 유지하기 위해 데이터의 생성, 수정, 삭제 전 과정에서 제약조건을 강제하는 데이터베이스의 고유 성질입니다.

DBMS 커널 내부 또는 외부 애플리케이션 레이어에서 트랜잭션(DML) 연산이 수행될 때, 기선언된 도메인 제약·기본키 배타성·외래키 참조 규칙 등의 인터셉터가 작동합니다. 연산 도중 이 규칙을 위반하는 결함 데이터가 발견되면 즉시 예외(Exception)를 발생시키고 트랜잭션을 전면 철회(Rollback)함으로써 오염되지 않은 청정 상태의 데이터베이스 스키마를 유지합니다.

### 3. 핵심 키워드 (Must-Have)

#### 4대 무결성 유형

| 무결성 유형 | 물리적 구현 기법 | 핵심 제약 내용 |
|-------------|----------------|---------------|
| **개체 무결성** (Entity Integrity) | Primary Key (PK) | 기본키는 중복 불가, NULL 불가 |
| **참조 무결성** (Referential Integrity) | Foreign Key (FK) | 외래키 값은 부모 기본키에 존재하거나 NULL이어야 함 |
| **도메인 무결성** (Domain Integrity) | Data Type, Not Null, Check, Default | 컬럼 값은 사전 정의된 타입·범위·유효값 범주를 만족해야 함 |
| **사용자 정의 무결성** (User-Defined Integrity) | Trigger, Stored Procedure, 애플리케이션 Validation | 표준 제약조건으로 표현할 수 없는 복잡한 업무 규칙 강제 |

#### 무결성 유지 방법

무결성은 구현 레이어의 위치와 통제 패러다임에 따라 선언적, 절차적, 애플리케이션, 분산 트랜잭션 기법으로 다각화됩니다.

| 대분류 | 구체적 유지 방법 | 세부 구현 메커니즘 | 장점 및 단점 |
|--------|----------------|-------------------|------------|
| **선언적 무결성** (Declarative) | DBMS 내장 제약조건 (Constraints) | DDL 시 `PRIMARY KEY`, `FOREIGN KEY`, `CHECK`, `NOT NULL`을 명시적으로 선언 | 장점: 성능 우수, 아키텍처 직관적 / 단점: 복잡한 연동 비즈니스 룰 반영 불가 |
| **절차적 무결성** (Procedural) | 트리거 및 프로시저 (Trigger / SP) | DML 연산 전후에 사전 정의된 SQL 블록이 자동 구동되어 정합성 검증 | 장점: 복잡한 연산 비즈니스 통제 가능 / 단점: 성능 모니터링 어렵고 디버깅 난이도 극상 |
| **애플리케이션 무결성** (Application) | 검증 프레임워크 (Validation Layer) | Spring Boot, JPA 엔티티에 Bean Validation 어노테이션(`@NotNull`, `@Size`, `@Pattern`) 명시, 비즈니스 로직 단 예외 처리 | 장점: DBMS 리소스 소모 없어 유연하고 확장성 우수 / 단점: 우회 접근(DB 직접 접속) 시 오염 위험 |
| **분산 정합성** (Distributed) | 분산 트랜잭션 합의 (2PC / Saga) | MSA 이종 DB 간 정합성을 위해 2-Phase Commit 적용, 비동기 환경에서는 이벤트 발행과 보상 트랜잭션을 통한 Saga 패턴 가동 | 장점: 분산 파편화 환경에서 무결성 파탄 방어 / 단점: 시스템 복잡도 급증, 지연 시간(Latency) 상승 |

### 4. 맥락과 비교

데이터 품질 관리에서 '데이터 무결성'과 '데이터 정합성'은 명확한 대칭적·보완적 차이점을 가집니다.

| 비교 항목 | 데이터 무결성 (Data Integrity) | 데이터 정합성 (Data Consistency) |
|-----------|------------------------------|--------------------------------|
| **개념적 본질** | 데이터가 현실 세계의 비즈니스 규칙에 부합하여 **정확하고 완전하며 오염되지 않은 상태** | 논리적·물리적 위치나 시점에 상관없이 **데이터들의 값이 서로 모순 없이 일치하는 상태** |
| **핵심 포커스** | 데이터의 **'올바름(Rightness)'**과 원천 품질 기준 충족 | 데이터 간의 **'모순 없음(No Contradiction)'**과 동일성 |
| **인과적 관계** | **정합성은 무결성을 만족하기 위한 하위 조건** | **틀린 데이터도 양쪽이 일치하면 정합성은 만족**할 수 있음 |
| **실무적 위반 사례** | 고객 테이블의 나이 컬럼에 문자열 `A`가 저장된 경우 | A 서비스 DB의 홍길동 전화번호와 B 서비스 DB의 홍길동 전화번호가 서로 다른 경우 |

### 5. 실무 제언

**성능(Performance)과 무결성(Integrity)의 트레이드오프**

- **챌린지**: 대용량 트랜잭션 환경(OLTP)에서 참조 무결성을 보장하기 위해 모든 테이블에 외래키(FK) 제약조건을 걸어두면, 데이터가 대량 삽입·삭제될 때마다 상위 부모 테이블에 상시적인 Lock이 발생하고 유효성 검증 오버헤드가 누적되어 극심한 병목 현상이 발생합니다.
- **제언**: MSA나 초고속 로그 수집 인프라 환경에서는 데이터베이스 레이어의 물리적 제약조건(FK)을 과감히 생략하되, 애플리케이션 검증 레이어 강제 및 사후 데이터 프로파일링 정기 배치 거버넌스를 도입하여 결합도를 낮추고 성능과 품질을 동시에 수호하는 하이브리드 아키텍처 리더십을 발휘해야 합니다. 시스템 전반에 걸쳐 백엔드 API 게이트웨이 단의 1차 검증, 도메인 엔티티 단의 2차 검증을 거친 청정 트랜잭션만 DB에 유입되도록 아키텍처를 혁신해야 합니다.
