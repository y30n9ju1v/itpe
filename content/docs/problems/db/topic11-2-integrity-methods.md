## 해설

### 출제 배경 및 의도

AI·빅데이터 분석이 비즈니스 핵심이 된 현대에 원천 데이터 신뢰성은 시스템 전체 성패를 좌우합니다. GIGO(Garbage In, Garbage Out) 현상을 막기 위한 최전방 방어선이 데이터 무결성이며, MSA·클라우드 기반 데이터 레이크 확산으로 데이터 오염과 참조 단절 현상이 급증하고 있습니다.

채점관은 무결성의 4대 유형을 단순 암기하는 수준을 넘어, 각 유형을 구현하는 선언적·절차적·애플리케이션·분산 트랜잭션 기법을 다각적으로 설명하는 데이터 아키텍트(DA) 실무 역량을 검증하고자 출제합니다. 무결성과 정합성의 개념 구분, 그리고 성능과 무결성의 트레이드오프 설계 판단력이 핵심 채점 포인트입니다.

### 1. 데이터베이스 무결성, 개요

정 의  • 데이터베이스 내에 저장된 데이터의 정확성(Accuracy)·완전성(Completeness)·유효성(Validity)·일관성(Consistency)을 유지하기 위해 DML 연산 전 과정에서 제약조건을 강제하는 데이터베이스의 고유 성질
       - DBMS 커널 내부 또는 애플리케이션 레이어에서 인터셉터가 작동하여 위반 시 Exception 발생 → 트랜잭션 전면 Rollback으로 청정 DB 스키마 유지

```
[무결성 유지 다층 방어 아키텍처]

  ┌──────────────────────────────────────────────────┐
  │  Layer 3: 분산 정합성 (Saga / 2PC)              │
  │  MSA 이종 DB 간 보상 트랜잭션 및 합의            │
  ├──────────────────────────────────────────────────┤
  │  Layer 2: 애플리케이션 무결성 (Bean Validation)  │
  │  Spring Boot @Valid, @NotNull, @Pattern          │
  ├──────────────────────────────────────────────────┤
  │  Layer 1: 선언적/절차적 무결성 (DBMS 내장)       │
  │  PK / FK / CHECK / Trigger / Stored Procedure   │
  └──────────────────────────────────────────────────┘
```

- 현대 분산 환경에서 무결성은 DBMS 단일 레이어를 넘어 다층 방어 아키텍처로 구현해야 한다.

### 2. 4대 무결성 유형 및 유지 방법

1) 4대 무결성 유형

| 구분 | 유형 | 물리적 구현 기법 | 핵심 제약 내용 |
|------|------|----------------|--------------|
| **①** | **개체 무결성** (Entity) | Primary Key (PK) | 기본키는 중복 불가, NULL 불가 |
| **②** | **참조 무결성** (Referential) | Foreign Key (FK) | 외래키 값은 부모 기본키에 존재하거나 NULL이어야 함 |
| **③** | **도메인 무결성** (Domain) | Data Type, Not Null, Check, Default | 컬럼 값은 사전 정의된 타입·범위·유효값 범주를 만족해야 함 |
| **④** | **사용자 정의 무결성** (User-Defined) | Trigger, Stored Procedure, 애플리케이션 Validation | 표준 제약으로 표현 불가한 복잡한 업무 규칙 강제 |

- 4대 유형은 상호 보완적으로 작동하여 전방위 데이터 품질 방어망을 구성한다.

2) 4대 무결성 유지 방법

| 대분류 | 구체적 방법 | 세부 구현 메커니즘 | 장단점 |
|--------|-----------|-----------------|--------|
| **선언적 무결성** (Declarative) | DBMS 내장 제약조건 | DDL 시 `PRIMARY KEY`, `FOREIGN KEY`, `CHECK`, `NOT NULL` 명시 | 장점: 성능 우수, 직관적 / 단점: 복잡한 연동 비즈니스 룰 반영 불가 |
| **절차적 무결성** (Procedural) | Trigger / Stored Procedure | DML 전후 SQL 블록 자동 구동으로 정합성 검증 | 장점: 복잡한 연산 통제 가능 / 단점: 성능 모니터링 어렵고 디버깅 난이도 극상 |
| **애플리케이션 무결성** (Application) | Bean Validation Layer | Spring Boot `@NotNull`, `@Size`, `@Pattern` 어노테이션, 비즈니스 로직 예외 처리 | 장점: DBMS 리소스 소모 없어 유연·확장성 우수 / 단점: DB 직접 접속 시 우회 위험 |
| **분산 정합성** (Distributed) | 분산 트랜잭션 합의 (2PC / Saga) | MSA 이종 DB 간 2-Phase Commit; 비동기 환경에서는 이벤트 발행 + 보상 트랜잭션(Saga) 가동 | 장점: 분산 파편화 환경 무결성 방어 / 단점: 시스템 복잡도 급증, Latency 상승 |

- 무결성 유지 방법은 시스템 레이어와 환경에 따라 다각화하여 다층 방어 체계를 구축해야 한다.

### 3. 무결성 vs 정합성 비교 및 현대 환경 트렌드

| 비교 항목 | 데이터 무결성 (Data Integrity) | 데이터 정합성 (Data Consistency) |
|----------|------------------------------|--------------------------------|
| **개념적 본질** | 데이터가 비즈니스 규칙에 부합하여 정확하고 완전하며 오염되지 않은 상태 | 위치·시점에 상관없이 데이터 값이 서로 모순 없이 일치하는 상태 |
| **핵심 포커스** | 데이터의 '올바름(Rightness)'과 원천 품질 기준 충족 | 데이터 간 '모순 없음(No Contradiction)'과 동일성 |
| **인과 관계** | 정합성은 무결성의 하위 조건 | 틀린 데이터도 양쪽 일치하면 정합성 만족 가능 |
| **위반 사례** | 나이 컬럼에 문자열 'A' 저장 (품질 파탄) | A 서비스 DB vs B 서비스 DB의 동일 고객 전화번호 불일치 |

- 현대 MSA 환경에서는 물리적 FK 제약의 한계를 보완하여 애플리케이션 검증 + Saga 패턴 + 정기 Reconciliation Batch의 하이브리드 무결성 아키텍처가 표준으로 정착하고 있다.  "끝"

### 실무 제언

**성능과 무결성의 트레이드오프 관리**
- **챌린지**: 대용량 OLTP 환경에서 모든 테이블에 FK 제약조건을 설정하면 대량 삽입·삭제 시 부모 테이블 Lock과 유효성 검증 오버헤드가 누적되어 극심한 병목이 발생한다.
- **제언**: MSA·초고속 로그 수집 환경에서는 물리적 FK를 생략하되, API 게이트웨이 단 1차 검증, 도메인 엔티티 단 2차 검증, 정기 Data Profiling Batch 사후 거버넌스로 결합도와 성능·품질을 동시에 확보하는 하이브리드 아키텍처를 채택해야 한다.

**절차적 무결성(Trigger)의 최소화 원칙**
- **챌린지**: 복잡한 업무 규칙을 모두 DB 트리거로 구현하면 디버깅 난이도가 극상으로 올라가고, 성능 모니터링이 불가능해지며 애플리케이션 레이어와의 이중 검증으로 관리 복잡도가 폭증한다.
- **제언**: 트리거는 핵심 감사 로그(Audit Log) 목적으로 최소화하고, 복잡한 업무 규칙은 도메인 서비스 레이어의 Bean Validation과 비즈니스 로직으로 이관하여 테스트 가능성과 유지보수성을 확보해야 한다.

**분산 환경의 Saga 패턴 설계**
- **챌린지**: MSA 환경에서 이종 DB 간 분산 트랜잭션을 2PC(2-Phase Commit)로 구현하면 코디네이터 장애 시 전체 시스템이 Block 상태에 빠지는 SPOF(Single Point of Failure)가 발생한다.
- **제언**: 비동기 Saga 패턴(Choreography 또는 Orchestration)을 도입하여 각 서비스가 이벤트를 발행하고 보상 트랜잭션으로 실패를 복구하는 Eventually Consistent 아키텍처를 구현하고, 주기적 Reconciliation Batch로 데이터 품질을 검증해야 한다.
