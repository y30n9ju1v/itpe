## 해설

> **유연성과 성능의 변증법**  
> 동적 SQL(Dynamic SQL): 런타임에 SQL을 조립하는 강력함과 그 대가

동적 SQL은 프로그램 컴파일 시점에 쿼리 구문이 완전히 고정되지 않고, 애플리케이션 실행(Runtime) 중에 사용자 입력이나 비즈니스 로직에 따라 SQL 문자열을 동적으로 조립하여 실행하는 SQL 처리 기법입니다. 예를 들어 검색 화면에서 사용자가 조건을 선택한 경우에만 WHERE 절에 해당 조건을 추가하는 방식이 대표적입니다. MyBatis의 `<if>`, `<choose>` 태그나 JPA의 Criteria, Querydsl과 같은 최신 데이터 액세스 기술들은 내부적으로 모두 이 메커니즘을 기반으로 작동합니다. 그러나 코드를 잘못 결합할 경우 SQL 인젝션(SQL Injection, 사용자 입력값에 악의적인 SQL 구문을 끼워 넣어 DB를 조작하는 공격) 보안 취약점의 주범이 되며, 하드 파싱(Hard Parsing, DBMS가 쿼리를 처음 받아 구문 분석·최적화 과정을 전부 새로 수행하는 것) 부하를 유발한다는 점에서 공학적 통제가 반드시 필요합니다.

### 1. 출제 배경 및 의도

현대 웹 아키텍처에서는 사용자가 검색 조건(필터, 정렬 기준, 선택 컬럼 등)을 실시간으로 다각화할 수 있는 다차원 검색 인터페이스가 보편화되었습니다. 채점관은 수험생이 동적 SQL의 유연성(장점)과 리스크(단점)를 공학적으로 명확히 계량하고, 바인드 변수(Bind Variable)와 PreparedStatement 등의 기법으로 통제할 수 있는 설계 역량을 가졌는지 검증하고자 합니다.

### 2. 개념의 본질과 작동 메커니즘

정적 SQL이 컴파일 단계에서 파싱과 최적화를 마치고 실행 메모리(Library Cache)에 대기하는 것과 달리, 동적 SQL은 런타임에 변수 조건에 따라 문자열 결합이 일어납니다. 이후 DBMS에 유입되면 매번 새로운 쿼리로 인식되어 파싱 및 최적화 엔진(Optimizer)을 거쳐 쿼리 실행 계획을 수립한 뒤 데이터 블록을 스캔합니다.

**동적 SQL 핵심 처리 기법:**
- **EXECUTE IMMEDIATE:** PL/SQL에서 문자열로 조립된 SQL을 즉시 실행하는 구문
- **OPEN/FETCH/CLOSE 커서:** 동적 SQL에서 다건 결과를 처리하기 위한 커서 제어
- **바인드 변수(Bind Variable, `#{param}`):** 값이 변하더라도 SQL의 뼈대 문장은 변하지 않도록 하여 DBMS가 기존 실행계획을 재이용(소프트 파싱)하게 하는 핵심 기법

### 3. 핵심 키워드 (Must-Have)

- **개념 관점:** 런타임(Runtime) 조립, 문자열 바인딩, 컴파일 시점 미확정
- **장점 관점:** 유연성(Flexibility) 극대화, 다중 조건 검색, 코드 감소, 선택적 로직 처리
- **단점 관점:** 하드 파싱(Hard Parsing) 오버헤드, 라이브러리 캐시(Library Cache) 부하, SQL 인젝션(SQL Injection) 취약점, 컴파일 시점 오류 검증 불가능
- **해결 및 대안 기술:** 바인드 변수(Bind Variable), 소프트 파싱(Soft Parsing), Querydsl, PreparedStatement

### 4. 맥락과 비교

동적 SQL은 고정형 구조를 가진 정적 SQL(Static SQL)과 성능 및 개발 방식 측면에서 명확한 트레이드오프(Trade-off)를 형성합니다.

| 비교 항목 | 정적 SQL (Static SQL / Embedded) | 동적 SQL (Dynamic SQL) |
| --- | --- | --- |
| **구문 확정 시점** | 애플리케이션 **컴파일 시점**에 100% 확정 | 애플리케이션 **실행 시점 (Runtime)**에 조립 및 확정 |
| **변수 처리 방식** | 호스트 변수, 바인드 변수 활용 고정 | 쿼리 문자열 자체를 결합하거나 조건문 분기 처리 |
| **DBMS 파싱 형태** | 최초 1회 파싱 후 공유 메모리 재사용 (**소프트 파싱**) | 구문 변경 시 매번 최적화를 재수행 (**하드 파싱** 우세) |
| **실행 성능** | **신속하고 안정적** (실행 계획이 고정 캐싱됨) | **비교적 저하 가능성** (최적화 및 파싱 오버헤드 존재) |
| **비즈니스 유연성** | **낮음** (조건에 따라 SQL을 개별 작성해야 함) | **매우 높음** (복잡한 다중 조건 및 선택적 검색 수용) |
| **보안적 위험도** | **안전함** (구문 구조가 변하지 않아 인젝션 방어) | **취약할 수 있음** (문자열 직접 결합 시 취약점 노출) |

### 5. 실무 제언

**DBMS 공유 메모리 파탄 방지 — 바인드 변수와 PreparedStatement 의무화**

- **챌린지:** 문자열을 직접 더하는(`WHERE ID = '` + userId + `'`) 방식의 동적 SQL을 남용하면, 사용자 수만큼 수만 개의 서로 다른 쿼리가 생성되어 데이터베이스의 공유 메모리 영역(Shared Pool / Library Cache)이 고갈되는 재앙이 발생합니다. 또한 사용자 입력값을 검증하지 않고 문자열로 직접 엮으면 악의적인 SQL 구문(`' OR '1'='1`)이 유입되어 전사 테이블이 유실·탈취되는 초대형 보안 사고로 직결됩니다.
- **제언:** 동적 SQL을 작성하더라도 값의 조건이 들어가는 플레이스홀더 영역은 반드시 바인드 변수(MyBatis 기준 `#{param}`)를 사용해야 합니다. 이를 통해 쿼리의 골격(형태)은 동적으로 제어하면서도 DBMS 레벨에서는 동일한 쿼리로 인식시켜 소프트 파싱을 유도합니다. DBMS 드라이버 레벨에서 `PreparedStatement` 객체를 활성화하여 사용자 입력값을 데이터 리터럴로 강제 인코딩하면 SQL 인젝션을 완벽히 봉쇄할 수 있습니다.

**정적 타입 체킹 도구로의 패러다임 시프트**

- **챌린지:** 소스코드가 빌드되는 시점에는 SQL의 문법적 오류, 오타, 존재하지 않는 컬럼 참조 등을 잡아낼 수 없습니다. 오직 런타임에 해당 비즈니스 로직이 실행되어야만 에러가 검출되므로 테스트 커버리지 관리와 품질 보증이 매우 까다롭습니다.
- **제언:** 소스코드 내부에서 문자열(String) 기반으로 동적 쿼리를 조립하는 구형 패턴을 지양하고, 자바 언어 구조 내에서 컴파일러가 직접 타입과 문법을 검증해 주는 Querydsl 아키텍처를 도입하여 잘못된 컬럼명이나 문법적 오류를 애플리케이션 컴파일 시점에 즉시 검출하는 품질 통제력을 확보해야 합니다.
