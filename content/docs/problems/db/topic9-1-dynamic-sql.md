## 해설

### 출제 배경 및 의도

현대 웹 아키텍처에서는 사용자가 검색 조건(필터, 정렬 기준, 선택 컬럼 등)을 실시간으로 다각화할 수 있는 다차원 검색 인터페이스가 보편화되었습니다. 정적 SQL만으로는 조건 조합이 폭발적으로 늘어나는 요건을 수용할 수 없어 동적 SQL이 필수가 되었습니다.

동적 SQL은 런타임에 SQL을 조립하는 유연성을 제공하지만, 하드 파싱(Hard Parsing) 부하와 SQL 인젝션(SQL Injection) 보안 취약점이라는 대가를 동반합니다. MyBatis의 `<if>`, `<choose>` 태그나 JPA Criteria, Querydsl 등 현대 데이터 액세스 기술들은 모두 이 메커니즘을 기반으로 합니다.

채점관은 동적 SQL의 유연성(장점)과 리스크(단점)를 공학적으로 명확히 계량하고, 바인드 변수와 PreparedStatement로 통제할 수 있는 설계 역량을 가졌는지 검증하고자 합니다.

### 1. 동적 SQL, 개요

정 의  • 애플리케이션 실행(Runtime) 중에 사용자 입력이나 비즈니스 로직에 따라 SQL 문자열을 동적으로 조립하여 실행하는 SQL 처리 기법
       - 컴파일 시점에 구문이 고정되는 정적 SQL과 달리, 런타임에 WHERE 절·테이블명·컬럼명 등을 조건에 따라 동적 결합

```
[정적 SQL vs 동적 SQL 처리 흐름 비교]

정적 SQL                          동적 SQL
────────────────────────          ────────────────────────────────────
컴파일 시점:                      실행 시점(Runtime):
  SQL 구문 확정                      조건 분기 → SQL 문자열 조립
  파싱·최적화 완료                         │
  Library Cache 적재                       ▼
        │                          DBMS 수신 → 새 쿼리 인식?
        ▼                                  │
실행 시점:                          Y: 하드 파싱 → 실행계획 재수립
  소프트 파싱(캐시 재사용)           N: 소프트 파싱(바인드 변수 사용 시)
  데이터 블록 스캔                         │
        │                                  ▼
        └──────────── 결과 반환 ───────────┘

핵심 처리 기법:
  - EXECUTE IMMEDIATE  : PL/SQL 즉시 실행
  - OPEN/FETCH/CLOSE   : 다건 결과 커서 처리
  - 바인드 변수(#{param}): 소프트 파싱 유도, SQL 인젝션 차단
```

- 동적 SQL의 유연성과 성능·보안 리스크는 바인드 변수 사용 여부로 결정된다.

### 2. 장단점 및 정적 SQL과의 비교

1) 동적 SQL 장점

| 구분 | 내용 | 실무 예시 |
|------|------|---------|
| **유연성 극대화** | 사용자 입력 조건에 따라 WHERE·ORDER BY·컬럼 동적 결합 | 다중 검색 필터 화면 |
| **코드 감소** | 조건별 개별 SQL 작성 불필요, 단일 로직으로 처리 | 200가지 조건 조합을 1개 쿼리로 커버 |
| **선택적 로직 처리** | 입력된 조건만 반영하여 불필요한 풀스캔 방지 | MyBatis `<if>` 태그 |
| **런타임 확장성** | 테이블명·컬럼명까지 동적 지정 가능 | 멀티테넌트 DB 파티션 조회 |

- 동적 SQL은 다차원 검색 인터페이스와 복잡한 비즈니스 로직 처리에 필수적이다.

2) 동적 SQL 단점 및 대응 방안

| 단점 | 발생 원인 | 대응 방안 |
|------|----------|---------|
| **하드 파싱 오버헤드** | 문자열 직접 결합 시 매번 새 쿼리로 인식 → 실행계획 재수립 | 바인드 변수(`#{param}`) 사용으로 소프트 파싱 유도 |
| **Library Cache 고갈** | 사용자 수만큼 상이한 쿼리 생성 → Shared Pool 고갈 | PreparedStatement 강제 적용 |
| **SQL 인젝션 취약점** | 문자열 직접 결합 시 악의적 SQL 구문 유입 (`' OR '1'='1`) | 바인드 변수·입력값 화이트리스트 검증 |
| **컴파일 시점 오류 검증 불가** | 런타임에만 SQL 오류 검출 가능 | Querydsl 도입으로 컴파일 타임 타입 체킹 |

- 동적 SQL의 단점은 바인드 변수 의무화와 타입 안전 쿼리 빌더(Querydsl) 도입으로 대부분 해소 가능하다.

### 3. 정적 SQL vs 동적 SQL 비교 및 현대 기술 트렌드

| 비교 항목 | 정적 SQL (Static SQL) | 동적 SQL (Dynamic SQL) |
|----------|----------------------|----------------------|
| **구문 확정 시점** | 컴파일 시점 100% 확정 | 실행 시점(Runtime) 조립·확정 |
| **DBMS 파싱 형태** | 소프트 파싱 (캐시 재사용) | 하드 파싱 우세 (바인드 변수 없을 시) |
| **실행 성능** | 신속·안정적 (실행계획 고정 캐싱) | 파싱 오버헤드 존재 가능 |
| **비즈니스 유연성** | 낮음 (조건별 개별 SQL 필요) | 매우 높음 (다중 조건·선택적 검색 수용) |
| **보안 위험도** | 안전 (구문 구조 불변) | 취약 가능 (문자열 직접 결합 시) |
| **현대 구현 기술** | JDBC PreparedStatement (고정 쿼리) | MyBatis `<if>`, JPA Criteria, **Querydsl** |

- 현대 엔터프라이즈 환경에서는 타입 안전 쿼리 빌더(Querydsl, JOOQ, Spring Data JPA Specifications 등)로 컴파일 타임 타입 체킹을 확보하면서 동적 쿼리 유연성을 달성하는 것이 권장된다. 단, Querydsl은 2024년 기준 메인테이너 활동이 저조하여 일부 프로젝트에서 JOOQ로의 전환이 진행 중임을 실무 선택 시 고려해야 한다.  "끝"

### 실무 제언

**바인드 변수와 PreparedStatement 의무화**
- **챌린지**: 문자열 직접 결합(`WHERE ID = '` + userId + `'`) 방식의 동적 SQL 남용 시, 사용자 수만큼 상이한 쿼리가 생성되어 DBMS의 Shared Pool/Library Cache가 고갈되는 재앙이 발생하고, SQL 인젝션 공격으로 전사 데이터가 유실·탈취된다.
- **제언**: 동적 SQL에서도 값 플레이스홀더에는 반드시 바인드 변수(MyBatis 기준 `#{param}`)를 사용하여 쿼리 골격을 고정하고, DBMS 드라이버 레벨에서 `PreparedStatement`를 활성화하여 사용자 입력을 데이터 리터럴로 강제 인코딩함으로써 소프트 파싱과 인젝션 방어를 동시에 달성해야 한다.

**Querydsl로 컴파일 타임 품질 확보**
- **챌린지**: 문자열 기반 동적 SQL은 컴파일 시점에 SQL 문법 오류·오타·존재하지 않는 컬럼 참조를 검출할 수 없어, 런타임에 해당 로직이 실행되어야만 오류가 드러난다. 테스트 커버리지 관리와 품질 보증이 극도로 어렵다.
- **제언**: 문자열 기반 동적 쿼리 조립 패턴을 지양하고, Java 언어 구조 내에서 컴파일러가 타입과 문법을 직접 검증하는 Querydsl 아키텍처를 도입하여 잘못된 컬럼명이나 문법 오류를 애플리케이션 컴파일 시점에 즉시 검출하는 품질 통제력을 확보해야 한다.

**대용량 검색 시스템에서의 동적 SQL 설계 패턴**
- **챌린지**: 수억 건 이상의 데이터를 대상으로 하는 검색 화면에서 동적 SQL의 조건 분기가 잘못 설계되면, 조건이 없을 때 Full Table Scan이 발생하여 시스템 전체가 마비된다.
- **제언**: 필수 조건(인덱스 컬럼)이 반드시 포함되도록 입력 검증을 선행하고, Elasticsearch나 역정규화된 검색 전용 테이블을 도입하여 OLTP DB와 분리하는 CQRS 아키텍처로 동적 검색 부하를 분산해야 한다.
