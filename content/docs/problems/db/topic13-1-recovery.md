# 데이터베이스 회복 기법(Recovery) 완전 해체

---

## 1. 문제를 해체하는 5대 핵심 분석 항목

### ① 출제 배경 및 의도 (Why Now?)

* **시스템 가용성과 비즈니스 연속성(BCP) 보장:** 현대 데이터베이스 환경은 클라우드 네이티브 아키텍처와 마이크로서비스 아키텍처(MSA) 하에서 초고속 트랜잭션을 실시간으로 처리합니다. 인프라의 장애, 전원 차단, 하드웨어 결함 등으로 시스템이 급작스럽게 다운되었을 때 데이터 손실 없이 청정 상태로 원상 복구하는 회복 기법은 가용성 관리의 핵심 지표입니다.
* **장애 복구 메커니즘의 공학적 역량 검증:** 트랜잭션의 원자성(Atomicity)과 지속성(Durability)을 완벽히 수호하기 위해 DBMS 커널 내부에서 트랜잭션 로그와 체크포인트를 어떻게 유기적으로 활용하는지 검증하고자 합니다. 채점관은 고전적인 로그 기반 기법부터 미디어 회복까지 데이터 아키텍트가 갖추어야 할 최전방 백업·복구 통제 전략을 계량 평가하기 위해 본 문제를 출제합니다.

### ② 개념의 본질과 작동 메커니즘 (What & How)

* **한 문장 정의:** 데이터베이스 운영 중 시스템 장애, 비정상 종료, 미디어 결함 등이 발생했을 때, 장애 직전의 일관성 있고 무결한 상태(Consistency)로 데이터를 복구해 내는 DBMS 커널의 핵심 기능입니다.
* **작동 메커니즘 (Mechanism):**
장애가 발생하면 복구 엔진이 기기록된 트랜잭션 로그(Log File)나 그림자 페이지 테이블을 추적(Input)합니다. 커밋된 트랜잭션은 디스크에 재반영하고(**REDO**), 커밋되지 못한 채 중단된 트랜잭션은 연산을 취소(**UNDO**)하는 메커니즘을 구동(Mechanism)하여 오염이나 누락이 없는 최종 일관성 상태의 데이터셋을 실물 스토리지에 복원(Output)합니다.

### ③ 채점관이 눈을 켜고 찾는 'Must-Have' 키워드 (Keywords)

* **1) REDO/UNDO:** 원자성(Atomicity), 지속성(Durability), Commit 상태, Active/Abort 상태, 전방 회복, 후방 회복.
* **2) 로그기반:** 지연 갱신(Deferred Update), 즉시 갱신(Immediate Update), 로그 레코드(`[Start, Commit, Rollback, Write]`), 하드 파싱 방지.
* **3) 체크포인트:** 검사점(Checkpoint), 검사점 로그, 복구 시간 단축(RTO), 무차별 스캔 축소, REDO/UNDO 리스트 분리.
* **4) 그림자페이지:** Shadow Page Table, Current Page Table, 디스크 블록 복사, 로그 오버헤드 소멸.
* **5) 미디어기반:** 덤프(Dump), 아카이브 로그(Archive Log), 디스크 크래시, 미디어 회복 툴웨어.

### ④ 상위/하위/유사 개념과의 상호 관계 (Context & Comparison)

데이터베이스 회복 기법들은 복구 속도(RTO), 데이터 저장 방식, 로그 오버헤드의 존재 여부에 따라 명확한 공학적 대칭 구조를 형성합니다.

| 분류 항목 | 지연 갱신 기법 (Deferred) | 즉시 갱신 기법 (Immediate) | 그림자 페이지 기법 (Shadow) |
| --- | --- | --- | --- |
| **디스크 반영 시점** | 트랜잭션이 **Commit되기 전까지 유예** | 트랜잭션 **수행 중 실시간 반영** | 변경 데이터를 **새 페이지 블록에 기록** |
| **필수 복구 연산** | **오직 REDO만 수행** (UNDO 불필요) | **REDO와 UNDO 모두 수행** | **둘 다 불필요** (테이블 포인터 전환) |
| **로그(Log) 오버헤드** | 존재함 (Write-Ahead Logging) | 존재함 (가장 큼) | **전혀 없음** (로그 파일 비생성) |
| **장애 복구 속도(RTO)** | **보통** (로그 분석 및 재실행 필요) | **느림** (대량의 변경 취소 부하 가능) | **가장 빠름** (포인터 복원 즉시 완결) |
| **실무적 활용 가치** | 소량 트랜잭션 버퍼링 환경 | 대부분의 현대 메이저 RDBMS 커널 | 특수 목적 임베디드, 파일 파일 엔진 |

### ⑤ 실무적 제언 및 챌린지 (So What? / Insight)

* **복구 시간(RTO)과 성능 오버헤드의 Dilemma:** 로그를 실시간으로 디스크에 완벽히 쓰면 무결성은 올라가지만 디스크 I/O 병목이 발생하고, 체크포인트(Checkpoint)를 너무 자주 가동하면 운영 성능이 저하됩니다. 반대로 체크포인트를 너무 뜸하게 잡으면 장애 발생 시 로그 스캔 범위가 비대해져 복구 시간(RTO)이 기하급수적으로 늘어납니다.
* **기술사의 제언:** 기술사는 단순 이론 암기를 넘어 인프라 파라미터를 조율해야 합니다. 오라클의 `FAST_START_MTTR_TARGET` 등 DBMS의 체크포인트 주기 제어 아키텍처를 시스템 성격에 맞게 튜닝하고, 현대 분산 MSA 환경에서는 서비스 간 정합성 붕괴를 막기 위해 **Saga 패턴 기반의 '보상 트랜잭션(Compensating Transaction)'** 기술을 결합하는 거시적 거버넌스 안목을 발휘해야 합니다.

## 2. 합격자 수준의 모범 답안 가이드 (2교시형 서술형)

### [1단락] 트랜잭션 ACID 수호의 핵심 인프라, 데이터베이스 회복 기법의 개요

#### 가. 데이터베이스 회복(Recovery)의 정의

* 트랜잭션 수행 중 시스템 다운, 하드웨어 장애, 스토리지 결함 등으로 인해 데이터베이스 내부가 모순되거나 오염되었을 때, 장애 발생 이전의 가장 최근 청정 일관성 상태(Consistent State)로 복구해 내는 DBMS 커널의 핵심 제어 기능입니다.

#### 나. 회복 기법의 근간, ACID 제약조건과의 인과 관계

* **원자성(Atomicity):** 장애 시 실행 중이던 트랜잭션의 잔재를 완벽히 취소(**UNDO**)하여 'All or Nothing'을 보장합니다.
* **지속성(Durability):** 이미 성공적으로 완료된 트랜잭션의 결과물은 장애 여파와 무관하게 완전히 재반영(**REDO**)하여 유실을 차단합니다.

---

### [2단락] 1) 전방 및 후방 회복의 핵심 연산, REDO와 UNDO 분석

DBMS 복구 엔진이 로그 레코드를 기반으로 판단하고 실행하는 가장 기초적인 대수적 복구 연산 메커니즘입니다.

#### 가. REDO (전방 회복 / 재실행)

* **상세 개념:** 장애 발생 전, 트랜잭션이 성공적으로 완료(`COMMIT`)되었으나 버퍼 캐시 플러시 지연 등으로 디스크 실물 블록에 최종 반영되지 못했을 가능성이 있는 데이터를 로그를 참조하여 "다시 똑같이 실행"하는 기법입니다.
* **구동 조건:** 로그 파일 내에 `[Start]` 레코드와 `[Commit]` 레코드가 **동시에 정상적으로 존재**하는 트랜잭션을 대상으로 추출합니다.
* **동작 특징:** 데이터베이스의 버퍼 캐시 내용을 로그 파일의 최신 값(New Value)으로 덮어써서 지속성을 수호합니다.

#### 나. UNDO (후방 회복 / 취소)

* **상세 개념:** 장애 발생 시점에 트랜잭션이 아직 완료되지 못하고 실행 중(`Active` 또는 `Abort`)인 상태였다면, 데이터 불완전 상태를 방지하기 위해 해당 트랜잭션이 수행한 모든 변경 연산을 **"장애 이전의 과거 상태로 완전히 되돌리는(취소)"** 기법입니다.
* **구동 조건:** 로그 파일 내에 `[Start]` 레코드는 존재하나, `[Commit]` 또는 `[Rollback]` 레코드가 **누락되어 유령 상태인 트랜잭션**을 대상으로 추출합니다.
* **동작 특징:** 로그에 기록된 비포 이미지(Old Value)를 디스크에 역순으로 덮어써서 원자성을 수호합니다.

### [3단락] 2) 동기화 시점에 따른 분화, 로그 기반 회복 기법 (지연/즉시 갱신)

트랜잭션이 유입되어 데이터를 변경할 때, 실시간으로 디스크에 쓰느냐 커밋 시점에 모아서 쓰느냐에 따라 두 가지 기법으로 양분됩니다. 단, 두 기법 모두 로그를 데이터보다 디스크에 먼저 기록하는 **WAL(Write-Ahead Logging) 원칙**을 반드시 사전에 준수해야 합니다.

#### 가. 지연 갱신 회복 기법 (Deferred Update)

* **동작 메커니즘:** 트랜잭션이 실행되는 동안 모든 변경 내용을 디스크에 즉시 반영하지 않고, 전용 **로그 버퍼(Log Buffer)에만 기록하며 유예**합니다. 이후 최종 `COMMIT` 명령이 떨어지는 순간 로그 파트를 디스크에 영구 적재하고 데이터베이스 반영을 시작하는 구조입니다.
* **장애 시 회복 전술:** 트랜잭션 미완료 상태에서 장애가 나면 디스크 원본 데이터는 오염되지 않은 상태이므로 **UNDO 연산 자체가 불필요**합니다. 오직 완료된 건에 한해서만 **REDO 연산만 순차 적용**하면 복구가 완결됩니다.

#### 나. 즉시 갱신 회복 기법 (Immediate Update)

* **동작 메커니즘:** 트랜잭션이 완료되기 전(`Active` 상태)이라도 버퍼 캐시 링에 공간이 부족하거나 스케줄러 가동 시 **변경 데이터를 디스크 실물 테이블에 실시간으로 즉시 갱신** 처리하는 적극적 구조입니다.
* **장애 시 회복 전술:** 미완료된 데이터가 이미 디스크를 오염시켰을 확률이 100%이므로, 장애 복구 시 로그의 Old Value를 활용한 **강력한 UNDO 연산이 필수적으로 수반**됩니다. 물론 성공 완료 건에 대한 **REDO 연산도 대칭적으로 병행 실행**해야 합니다.

### [4단락] 3) 복구 타임라인의 혁신, 체크포인트 회복 기법 (Check-Point)

#### 가. 체크포인트 회복 기법의 등장 배경 및 개념

* **기존 로그 기법의 한계:** 시스템을 수개월간 운영하면 로그 파일의 크기가 수십 기가바이트(GB)로 비대해집니다. 이 상태에서 장애 발생 시 첫 번째 로그 레코드부터 전수 조사해야 하므로 복구 시간(RTO)이 무한정 늘어나는 재앙이 발생합니다.
* **해결 방안:** DBMS 복구 엔진이 주기적으로 메모리의 변경 버퍼를 디스크로 완전 플러시(Flush)하고, 해당 시점의 트랜잭션 명세를 로그 파일에 `[Checkpoint]` 마크로 박아두는 기술입니다. 장애 발생 시 **체크포인트 이전의 로그 스캔을 과감히 생략**하므로 복구 가동력을 극대화합니다.

#### 나. 체크포인트 기준의 REDO / UNDO 트랜잭션 판별 매커니즘사례

* **타임라인 분석 및 복구 리스트 분류 예시:**
* **$T_1$ (Checkpoint 전 완료):** 이미 디스크 쓰기가 완결되었으므로 복구 대상에서 제외 (No Action)합니다.
* **$T_2$ ($T_1$ 이후 시작, 장애 전 완료) / $T_3$ (Checkpoint 전 시작, 장애 전 완료):** 장애 시점에 정상 커밋 상태이므로 **REDO List**에 등록하여 재실행합니다.
* **$T_4$ (Checkpoint 전 시작, 장애 시 실행 중) / $T_5$ (Checkpoint 이후 시작, 장애 시 실행 중):** 커밋 마크가 없는 미완결 유령 상태이므로 **UNDO List**에 등록하여 전면 철회 역연산을 수행합니다.

### [5단락] 4) 로그 부하가 없는 대체 구조, 그림자 페이지 회복 기법 (Shadow Page)

#### 가. 그림자 페이지 회복 기법의 개념 및 동작 원리

* 트랜잭션 수행 시 로그 파일을 별도로 생성하여 디스크 쓰기 오버헤드를 유발하는 기존 패러다임을 탈피한 기법입니다.
* 데이터베이스 테이블을 페이지(Page) 블록 단위로 관리하며, 트랜잭션 시작 시점에 현재 페이지 테이블과 똑같은 복사본인 그림자 페이지 테이블(Shadow Page Table)을 메모리에 이중으로 복제해 관리하는 독특한 메커니즘입니다.

#### 나. 정상 운영 및 장애 복구 시의 포인터 변환 메커니즘

* **정상 운영 시:** 변경 연산이 발생하면 기존 블록을 덮어쓰지 않고, 디스크의 완전히 새로운 빈 블록을 할당받아 수정한 뒤 현재 페이지 테이블(Current Page Table)의 포인터 주소만 새 블록으로 갱신합니다.
* **장애 시 복구 메커니즘:** 복잡한 로그 분석이나 REDO/UNDO 연산이 원천적으로 필요 없습니다. 메모리에 백업 보존되어 있던 **그림자 페이지 테이블의 포인터 구조를 현재 페이지 테이블 포인터로 1:1 '스왑(Swap)' 전환**하는 것만으로 단 1밀리초 만에 장애 이전 청정 상태로 원상 복구됩니다.

### [6단락] 5) 대규모 디스크 파탄의 최종 병기, 미디어 회복 기법 (Media Recovery)

#### 가. 미디어 회복 기법의 개념 및 타깃 장애

* 앞서 언급된 1~4번 회복 기법은 모두 디스크 헤드나 물리 섹터가 살아있고 메모리/소프트웨어적 오류가 발생한 '시스템 장애(System Crash)'를 복구하는 기술입니다.
* 반면, 화재, 천재지변, 물리적 디스크 긁힘(Scratch) 등 스토리지 장치 자체가 통째로 파괴되는 '미디어 장애(Media Crash)'가 발생하면 내부 로그 파일조차 함께 증발하므로 고전적 기법은 무력화됩니다. 이를 해결하기 위해 외부 인프라 백업 자원을 결합하는 기술이 미디어 회복 기법입니다.

#### 나. 덤프(Dump)와 아카이브 로그(Archive Log)를 활용한 복구 메커니즘

* **복구 자원 매핑:** 물리적으로 분리된 안전한 3차 저장소(오브젝트 스토리지, 테이프 장치 등)에 주기적으로 백업해 둔 데이터베이스 전체 복사본인 **덤프(Dump)** 데이터와, 매 가동 시점마다 가득 찬 온라인 로그 파일을 안전한 보관소로 복제 이관해 둔 **아카이브 로그(Archive/In-active Log)** 세트를 동원합니다.

```
[미디어 장애 복구 타임라인 매커니즘]
  장애 디스크 하드웨어 전면 교체 (인프라 리빌딩)
         │
         ▼
  가장 최근의 정기 덤프(Dump) 리스토어 ────> [덤프 시점 상태 복원 완료]
         │
         ▼
  덤프 시점 이후부터 장애 직전까지 누적된 아카이브 로그들을 
  순차적으로 전방 재실행(Roll-Forward REDO) 적용 ────> [장애 직전 100% 무손실 복구 성공]

```

### [7단락] 기술사의 시각: 클라우드 분산(MSA) 및 인메모리(IMDB) 환경에서의 회복 확장 전략

#### 가. 현대 초고속 인메모리 데이터베이스(IMDB)에서의 회복 챌린지

* 최근 처리 속도 향상을 위해 DRAM 상에 데이터를 적재·운영하는 인메모리 데이터베이스(Redis, SAP HANA 등)의 채택이 대폭 늘어났습니다. 인메모리 구조는 전원 차단(장애) 시 데이터가 통째로 휘발하는 치명적인 약점이 존재하므로 회복 메커니즘 통제가 더욱 중요합니다.
* **실무적 대응 전술:** 인메모리 환경이라 할지라도 비휘발성 저장소(SSD)에 로그를 남기는 **AOF(Append Only File)** 기술이나 정기적 스냅샷(**RDB 기법**, 체크포인트의 현대적 확장)을 반드시 하이브리드로 결합하여 유기적인 가용성 아키텍처를 가동해야 합니다.

#### 나. 마이크로서비스 아키텍처(MSA) 환경에서의 분산 회복 거버넌스 제언

* **최종 일관성(Eventual Consistency)으로의 패러다임 시프트:** 분산 환경에서는 단일 DBMS의 회복 엔진이 시스템 전체의 무결성을 지켜줄 수 없습니다. 데이터베이스가 서비스 경계별로 철저히 파편화되어 있기 때문입니다.
* 따라서 시스템 설계 단계부터 Saga 패턴 기반의 오케스트레이터(Orchestrator)를 배치해야 합니다. 분산 노드 중 한 곳에서 장애가 포착되면, 이미 완료된 앞선 마이크로서비스들의 데이터를 역으로 롤백시키는 **'보상 트랜잭션(Compensating Transaction)' 이벤트를 애플리케이션 레이어에서 강제 발행**해야 합니다. 이를 통해 데이터베이스 레이어의 고전적 회복 대수 규칙(REDO/UNDO)을 거시적인 클라우드 레이어의 분산 회복 거버넌스로 승격·융합시키는 입체적인 아키텍처 리더십을 발휘해야 합니다.
