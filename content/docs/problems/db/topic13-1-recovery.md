## 해설

### 출제 배경 및 의도

클라우드 네이티브 아키텍처·MSA 하에서 초고속 트랜잭션을 처리하는 현대 데이터베이스 환경에서, 장애 발생 시 데이터 손실 없이 청정 상태로 원상 복구하는 회복 기법은 가용성 관리의 핵심 지표이다. 트랜잭션의 원자성(Atomicity)과 지속성(Durability)을 완벽히 수호하기 위해 DBMS 커널 내부에서 트랜잭션 로그와 체크포인트를 유기적으로 활용하는 메커니즘을 이해해야 한다.

채점관은 고전적인 로그 기반 기법부터 미디어 회복까지 데이터 아키텍트가 갖추어야 할 최전방 백업·복구 통제 전략을 계량 평가하기 위해 본 문제를 출제한다. 단순 기법 암기를 넘어, REDO/UNDO가 ACID의 어떤 속성을 수호하는지, 체크포인트가 RTO를 어떻게 단축하는지, 그림자 페이지의 로그 오버헤드 제거 원리를 설명할 수 있는지를 검증한다.

### 1. 데이터베이스 회복 기법, 개요

정 의  • 장애 발생 시 REDO(재실행)·UNDO(취소) 연산을 기반으로 DB를 일관성 있는 상태로 복구하는 DBMS 핵심 기능
       - WAL(Write-Ahead Logging) 원칙 기반: 로그를 데이터보다 디스크에 먼저 기록

```
[회복 기법 분류 구성도]

  장애 발생 (System Crash / Media Failure)
            │
            ▼
  ┌─────────────────────────────────────────────┐
  │          회복 엔진 (Recovery Manager)        │
  │                                             │
  │  ┌──────────────┐   ┌──────────────────┐   │
  │  │ 로그 기반    │   │ 그림자 페이지    │   │
  │  │ 회복 기법    │   │ 회복 기법        │   │
  │  │ (Log-Based)  │   │ (Shadow Page)    │   │
  │  │              │   │                  │   │
  │  │ ① 지연 갱신  │   │ 포인터 전환으로  │   │
  │  │   → REDO만  │   │ 즉시 복구        │   │
  │  │ ② 즉시 갱신  │   │ (로그 오버헤드   │   │
  │  │   → REDO+   │   │  없음)           │   │
  │  │   UNDO      │   └──────────────────┘   │
  │  └──────────────┘                          │
  │                                             │
  │  ③ 체크포인트 기법 → RTO 단축              │
  │  ④ 미디어 기반 기법 → 덤프+아카이브 로그   │
  └─────────────────────────────────────────────┘
```

- REDO는 Durability를 수호하고, UNDO는 Atomicity를 수호한다.

### 2. 5가지 회복 기법의 상세 내용

1) REDO와 UNDO

| 구분 | 항목 | 설명 |
|------|------|------|
| REDO (전방 회복) | 대상 트랜잭션 | 로그 내 [Start]+[Commit] 레코드가 모두 존재하는 트랜잭션 |
| REDO (전방 회복) | 동작 | COMMIT되었으나 버퍼 플러시 지연으로 디스크 미반영 시 재실행 |
| REDO (전방 회복) | 수호 속성 | 지속성(Durability) |
| UNDO (후방 회복) | 대상 트랜잭션 | 로그 내 [Start]만 존재, [Commit]/[Rollback] 누락 트랜잭션 |
| UNDO (후방 회복) | 동작 | 장애 시 실행 중(Active)이던 트랜잭션의 모든 변경 연산 취소 |
| UNDO (후방 회복) | 수호 속성 | 원자성(Atomicity) |

- WAL 원칙(Write-Ahead Logging)을 반드시 준수해야 로그 기반 회복이 보장된다.

2) 로그 기반 회복 기법 — 지연/즉시 갱신 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| 지연 갱신 | 디스크 반영 시점 | COMMIT 이후에만 DB 반영(COMMIT 전까지 로그 버퍼에만 기록) |
| 지연 갱신 | 필요 연산 | REDO만 수행 (UNDO 불필요) |
| 지연 갱신 | 특징 | 장애 시 DB 미반영 상태이므로 UNDO 대상 없음 |
| 즉시 갱신 | 디스크 반영 시점 | 트랜잭션 Active 상태에서도 즉시 DB 반영 |
| 즉시 갱신 | 필요 연산 | REDO + UNDO 모두 수행 |
| 즉시 갱신 | 특징 | 대부분의 현대 상용 RDBMS 커널이 채택 |

- 즉시 갱신 기법이 더 높은 신뢰성을 제공하나, UNDO 처리 부하가 추가 발생한다.

### 3. 체크포인트·그림자 페이지·미디어 기법 비교 및 트렌드

| 기법 | 메커니즘 | REDO/UNDO | 복구 속도(RTO) | 로그 오버헤드 | 적합 환경 |
|------|---------|----------|--------------|------------|---------|
| 체크포인트 (Checkpoint) | 주기적 버퍼 플러시 + [Checkpoint] 마크 기록, CP 이전 로그 스캔 생략 | 둘 다 가능 | 빠름 (스캔 범위 축소) | 있음 | 대용량 RDBMS |
| 그림자 페이지 (Shadow Page) | 트랜잭션 시작 시 페이지 테이블 복사, 변경 시 새 블록 기록 후 포인터 전환 | 불필요 | 가장 빠름 (포인터 전환만) | 없음 | 임베디드, 특수 목적 |
| 미디어 기반 (Media Recovery) | 덤프(Dump) + 아카이브 로그로 디스크 물리 파괴 복구 | REDO (Roll-Forward) | 느림 (복구 범위 최대) | 있음 | 물리 장애, 재해 복구 |

체크포인트 기법은 오라클의 `FAST_START_MTTR_TARGET` 파라미터로 주기를 제어하며, 인메모리 DB(Redis)에서는 AOF(Append Only File)와 스냅샷의 하이브리드 방식으로 확장 적용된다. MSA 환경에서는 Saga 패턴의 보상 트랜잭션(Compensating Transaction)이 분산 환경의 UNDO 역할을 담당한다.  "끝"

### 실무 제언

**RTO와 성능 오버헤드의 균형**
- **챌린지**: 체크포인트를 너무 자주 가동하면 운영 성능이 저하되고, 너무 뜸하게 잡으면 장애 시 RTO가 기하급수적으로 늘어난다.
- **제언**: DBMS 체크포인트 주기 제어 파라미터를 시스템 성격에 맞게 튜닝하고, 인메모리 DB는 AOF + 스냅샷을 하이브리드로 결합한다.

**미디어 장애 대비 3중화 전략**
- **챌린지**: 화재·천재지변 등 물리적 디스크 전체 파괴 시 내부 로그 파일까지 함께 손실되어 로그 기반 회복이 무력화된다.
- **제언**: 덤프(Dump)와 아카이브 로그를 물리적으로 분리된 원격지 3차 저장소에 보관하고, RTO/RPO 목표에 맞는 정기 백업 스케줄을 수립한다.

**MSA 분산 환경의 회복 전략**
- **챌린지**: 단일 노드 DB의 ARIES/로그 기반 회복만으로는 MSA 분산 환경의 전사적 무결성을 보장할 수 없다.
- **제언**: 각 서비스 노드의 로컬 회복 능력에 더해 Saga 패턴이나 2PC(Two-Phase Commit) 프로토콜을 결합하여 전사적 데이터 무결성을 달성한다.
