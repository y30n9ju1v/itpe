## 해설

> **장애 발생 시 REDO(재실행)·UNDO(취소) 연산을 기반으로 데이터베이스를 일관성 있는 상태로 복구하는 DBMS 핵심 기능**  
> 데이터베이스 회복 기법(Recovery) — 5가지 기법 완전 해체

클라우드 네이티브 아키텍처·MSA 하에서 초고속 트랜잭션을 처리하는 현대 데이터베이스 환경에서, 장애 발생 시 데이터 손실 없이 청정 상태로 원상 복구하는 회복 기법은 가용성 관리의 핵심 지표입니다. 트랜잭션의 원자성(Atomicity)과 지속성(Durability)을 완벽히 수호하기 위해 DBMS 커널 내부에서 트랜잭션 로그와 체크포인트를 유기적으로 활용하는 메커니즘을 이해해야 합니다.

### 1. 출제 배경 및 의도

채점관은 고전적인 로그 기반 기법부터 미디어 회복까지 데이터 아키텍트가 갖추어야 할 최전방 백업·복구 통제 전략을 계량 평가하기 위해 본 문제를 출제합니다. 단순 기법 암기를 넘어, REDO/UNDO가 ACID의 어떤 속성을 수호하는지, 체크포인트가 RTO를 어떻게 단축하는지, 그림자 페이지의 로그 오버헤드 제거 원리를 설명할 수 있는지를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

데이터베이스 운영 중 시스템 장애, 비정상 종료, 미디어 결함 등이 발생했을 때, 장애 직전의 일관성 있고 무결한 상태(Consistency)로 데이터를 복구해 내는 DBMS 커널의 핵심 기능입니다.

장애가 발생하면 복구 엔진이 기기록된 트랜잭션 로그(Log File)나 그림자 페이지 테이블을 추적합니다. 커밋된 트랜잭션은 디스크에 재반영하고(**REDO — 전방 회복**), 커밋되지 못한 채 중단된 트랜잭션은 연산을 취소(**UNDO — 후방 회복**)하는 메커니즘을 구동하여 최종 일관성 상태의 데이터셋을 실물 스토리지에 복원합니다.

- **원자성(Atomicity) 수호**: 장애 시 실행 중이던 트랜잭션의 잔재를 완벽히 취소(**UNDO**)하여 'All or Nothing' 보장
- **지속성(Durability) 수호**: 이미 성공적으로 완료된 트랜잭션의 결과물은 장애 여파와 무관하게 완전히 재반영(**REDO**)하여 유실 차단

### 3. 핵심 키워드 (Must-Have)

#### 1) REDO와 UNDO

- **REDO (전방 회복/재실행)**: 트랜잭션이 성공적으로 `COMMIT`되었으나 버퍼 캐시 플러시 지연으로 디스크에 최종 반영되지 못한 데이터를 로그를 참조하여 "다시 똑같이 실행"하는 기법. 로그 파일 내에 `[Start]`와 `[Commit]` 레코드가 동시에 존재하는 트랜잭션이 대상
- **UNDO (후방 회복/취소)**: 장애 발생 시점에 트랜잭션이 아직 완료되지 못하고 실행 중(`Active` 또는 `Abort`)인 상태였다면, 해당 트랜잭션이 수행한 모든 변경 연산을 장애 이전의 과거 상태로 되돌리는 기법. 로그 파일 내에 `[Start]` 레코드는 존재하나 `[Commit]` 또는 `[Rollback]` 레코드가 누락된 트랜잭션이 대상

#### 2) 로그 기반 회복 기법 (지연/즉시 갱신)

두 기법 모두 **WAL(Write-Ahead Logging)** 원칙 — 로그를 데이터보다 디스크에 먼저 기록 — 을 반드시 준수해야 합니다.

- **지연 갱신 회복 기법 (Deferred Update)**: 트랜잭션 실행 중 모든 변경 내용을 로그 버퍼(Log Buffer)에만 기록하며 유예. `COMMIT` 명령 시 로그를 디스크에 영구 적재하고 DB 반영을 시작. 장애 시 오직 **REDO만 수행**(UNDO 불필요)
- **즉시 갱신 회복 기법 (Immediate Update)**: 트랜잭션이 완료되기 전(`Active` 상태)이라도 변경 데이터를 디스크 실물 테이블에 실시간으로 즉시 갱신. 장애 시 **REDO와 UNDO 모두 수행** 필요

#### 3) 체크포인트 회복 기법 (Check-Point)

- **등장 배경**: 로그 파일이 수십 GB로 비대해지면 첫 번째 로그 레코드부터 전수 조사해야 하므로 복구 시간(RTO)이 무한정 늘어나는 문제 발생
- **메커니즘**: DBMS 복구 엔진이 주기적으로 메모리의 변경 버퍼를 디스크로 완전 플러시(Flush)하고, 해당 시점의 트랜잭션 명세를 로그 파일에 `[Checkpoint]` 마크로 박아두는 기술. 장애 발생 시 체크포인트 이전의 로그 스캔을 생략하여 복구 가동력 극대화
- **REDO/UNDO 트랜잭션 판별**: Checkpoint 전 완료된 T1은 복구 대상 제외(No Action). Checkpoint 이후 시작하여 장애 전 완료된 트랜잭션은 REDO List 등록. Checkpoint 전후에 시작하여 장애 시 실행 중인 트랜잭션은 UNDO List 등록

#### 4) 그림자 페이지 회복 기법 (Shadow Page)

- **메커니즘**: 트랜잭션 시작 시점에 현재 페이지 테이블과 똑같은 복사본인 그림자 페이지 테이블(Shadow Page Table)을 메모리에 이중으로 복제. 변경 연산 발생 시 기존 블록을 덮어쓰지 않고 디스크의 새로운 빈 블록에 수정한 뒤 현재 페이지 테이블(Current Page Table)의 포인터 주소만 새 블록으로 갱신
- **장애 시 복구**: 복잡한 로그 분석이나 REDO/UNDO 연산이 원천적으로 불필요. 그림자 페이지 테이블의 포인터 구조를 현재 페이지 테이블 포인터로 1:1 스왑(Swap) 전환하는 것만으로 즉시 복구
- **특징**: 로그 파일을 별도로 생성하지 않으므로 **로그 오버헤드 전혀 없음**. 복구 속도 가장 빠름

#### 5) 미디어 기반 회복 기법 (Media Recovery)

- **대상 장애**: 화재, 천재지변, 물리적 디스크 파괴 등 스토리지 장치 자체가 통째로 파괴되는 미디어 장애(Media Crash). 내부 로그 파일조차 함께 증발하므로 고전적 기법이 무력화됨
- **복구 자원**: 물리적으로 분리된 안전한 3차 저장소에 주기적으로 백업해 둔 데이터베이스 전체 복사본인 **덤프(Dump)**와, 가득 찬 온라인 로그 파일을 안전한 보관소로 복제 이관해 둔 **아카이브 로그(Archive Log)** 세트
- **복구 절차**: 장애 디스크 교체 → 가장 최근 덤프(Dump) 리스토어 → 덤프 시점 이후부터 장애 직전까지 누적된 아카이브 로그들을 순차적으로 전방 재실행(Roll-Forward REDO) 적용 → 장애 직전 100% 무손실 복구 완료

### 4. 맥락과 비교

회복 기법들은 복구 속도(RTO), 데이터 저장 방식, 로그 오버헤드 여부에 따라 명확한 공학적 대칭 구조를 형성합니다.

| 분류 항목 | 지연 갱신 기법 (Deferred) | 즉시 갱신 기법 (Immediate) | 그림자 페이지 기법 (Shadow) |
|-----------|--------------------------|--------------------------|--------------------------|
| **디스크 반영 시점** | 트랜잭션이 **Commit되기 전까지 유예** | 트랜잭션 **수행 중 실시간 반영** | 변경 데이터를 **새 페이지 블록에 기록** |
| **필수 복구 연산** | **오직 REDO만 수행** (UNDO 불필요) | **REDO와 UNDO 모두 수행** | **둘 다 불필요** (테이블 포인터 전환) |
| **로그(Log) 오버헤드** | 존재함 (WAL 원칙 적용) | 존재함 (가장 큼) | **전혀 없음** (로그 파일 비생성) |
| **장애 복구 속도(RTO)** | **보통** (로그 분석 및 재실행 필요) | **느림** (대량의 변경 취소 부하 가능) | **가장 빠름** (포인터 복원 즉시 완결) |
| **실무적 활용 가치** | 소량 트랜잭션 버퍼링 환경 | 대부분의 현대 메이저 RDBMS 커널 | 특수 목적 임베디드, 파일 엔진 |

### 5. 실무 제언

**복구 시간(RTO)과 성능 오버헤드의 딜레마**

- **챌린지**: 로그를 실시간으로 디스크에 완벽히 쓰면 무결성은 올라가지만 디스크 I/O 병목이 발생하고, 체크포인트를 너무 자주 가동하면 운영 성능이 저하됩니다. 반대로 체크포인트를 너무 뜸하게 잡으면 장애 발생 시 로그 스캔 범위가 비대해져 RTO가 기하급수적으로 늘어납니다.
- **제언**: 오라클의 `FAST_START_MTTR_TARGET` 등 DBMS의 체크포인트 주기 제어 아키텍처를 시스템 성격에 맞게 튜닝해야 합니다. 인메모리 데이터베이스(Redis, SAP HANA 등) 환경에서는 전원 차단 시 데이터가 휘발하므로, 비휘발성 저장소(SSD)에 로그를 남기는 **AOF(Append Only File)** 기술이나 정기적 스냅샷(체크포인트의 현대적 확장)을 반드시 하이브리드로 결합해야 합니다. MSA 환경에서는 Saga 패턴 기반의 보상 트랜잭션(Compensating Transaction) 기술을 결합하여 전사적 데이터 무결성을 달성해야 합니다.
