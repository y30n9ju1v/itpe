## 해설

> **정규화 이론의 수학적 주춧돌**  
> 함수 종속성의 폐포(Closure)를 도출하는 건전하고 완전한 추론 규칙 집합

암스트롱 공리는 관계형 데이터베이스(RDBMS) 설계의 핵심인 정규화(Normalization)와 함수 종속성(Functional Dependency, 한 속성의 값이 다른 속성의 값을 유일하게 결정하는 관계)을 판별하는 가장 근본적인 수학적 규칙(추론 규칙)입니다. 최근 클라우드 이관 및 레거시 DB 정비 사업이 늘어나면서 수천 개의 테이블 속 함수 종속성을 알고리즘으로 자동 검증하는 이론적 뼈대로서 그 중요성이 더욱 강조되고 있습니다.

### 1. 출제 배경 및 의도

채점관이 암스트롱 공리를 출제하는 의도는 정규화의 수학적 정당성을 이해하는지 검증하기 위함입니다. 함수 종속성의 폐포(Closure)를 계산하고 정규형 충족 여부를 자동 검증하는 이론적 기반이 되기 때문에, 단순히 규칙을 암기하는 수준이 아니라 각 규칙이 어떤 정규화 단계와 연결되는지까지 설명할 수 있어야 합니다.

### 2. 개념의 본질과 작동 메커니즘

**정의:** 릴레이션 스키마 R과 R에 고정된 함수 종속성 집합 F가 주어졌을 때, 논리적으로 유도 가능한 모든 함수 종속성을 찾아내기 위해 사용하는 수학적 추론 규칙(Inference Rules) 집합입니다. 1974년 윌리엄 암스트롱(William W. Armstrong)이 제안하였으며, **건전성(Soundness)**과 **완전성(Completeness)**을 동시에 만족합니다.

- **건전성:** 암스트롱 공리로 유도된 임의의 함수 종속성 X→Y는 F의 가정을 만족하는 모든 릴레이션에서 반드시 참입니다. 즉, 잘못된 종속성은 절대 유도되지 않습니다.
- **완전성:** F에 의해 논리적으로 함축된(참인) 모든 함수 종속성 X→Y는 오직 암스트롱 공리만을 반복 적용하여 반드시 유도해 낼 수 있습니다.

주어진 초기 함수 종속성 집합 F가 입력되면, 3대 기본 규칙(반사율·첨가율·이행율) 및 부가 규칙들을 논리적 추론 엔진처럼 반복 적용하여, 해당 릴레이션이 가질 수 있는 모든 함수 종속성의 총집합인 폐포(Closure, F⁺, 주어진 종속성 집합으로부터 유도 가능한 모든 종속성을 모아 놓은 집합)를 도출합니다.

#### 3대 기본 공리

| 규칙명 | 수학적 정의 | 직관적 의미 및 예시 |
| --- | --- | --- |
| **1. 반사율** *(Reflexivity)* | Y ⊆ X 이면, X → Y | 자신이 포함하는 하위 속성은 스스로를 결정할 수 있다. 예: (학번, 주민번호) → 학번 |
| **2. 첨가율** *(Augmentation)* | X → Y 이면, XZ → YZ | 참인 종속성의 양변에 동일한 속성을 추가해도 여전히 참이다. 예: 학번 → 성명 이면, (학번, 학과) → (성명, 학과) |
| **3. 이행율** *(Transitivity)* | X → Y 이고 Y → Z 이면, X → Z | 징검다리를 건너 뛰어 궁극적인 종속성을 유도한다. 예: 학번 → 학과코드, 학과코드 → 학과명 이면, 학번 → 학과명 |

※ 기본 공리 명칭: 반사율(Reflexivity), 첨가율(Augmentation, 증가율이라고도 함), 이행율(Transitivity)

#### 3대 부가 규칙 (기본 공리로부터 증명 가능)

| 규칙명 | 수학적 정의 | 증명 메커니즘 |
| --- | --- | --- |
| **1. 합집합 규칙** *(Union)* | X→Y이고 X→Z이면, X→YZ | ① X→Y에 Z 첨가율 적용 → XZ→YZ, ② X→Z에 Y 첨가율 적용 → XY→YZ, ③ X→Y이고 XY→YZ이므로 이행율 적용 → X→YZ |
| **2. 분해 규칙** *(Decomposition)* | X→YZ이면, X→Y이고 X→Z | ① 반사율에 의해 YZ→Y 성립, ② X→YZ이고 YZ→Y이므로 이행율 적용 → X→Y |
| **3. 의사이행 규칙** *(Pseudo-transitivity)* | X→Y이고 WY→Z이면, WX→Z | ① X→Y에 첨가율 적용 → WX→WY, ② WX→WY이고 WY→Z이므로 이행율 적용 → WX→Z |

#### 애트리뷰트 폐포(X⁺) 도출 알고리즘

```text
Input: 애트리뷰트의 집합 X, 함수 종속성 집합 F
Output: X의 폐포인 X+

Algorithm:
  1. 초기화: X+ := X;
  2. 반복수행: 
     - F에 있는 각각의 함수 종속성 Y -> Z 에 대하여
     - 만약 Y ⊆ X+ 이면, 
     - X+ := X+ ∪ Z;
  3. 판단조건: X+에 더 이상 애트리뷰트가 추가되지 않을 때까지 2번을 반복함.
```

**폐포 도출 사례:** 릴레이션 R(A, B, C, D, E), F = {A→B, B→C, CD→E} 에서 (AD)⁺를 구하면:

```
Step 1. 초기화 ──────────────> (AD)+ = {A, D}
Step 2. A → B 적용 ─────────> A가 {A, D}의 부분집합이므로 B 추가 → (AD)+ = {A, D, B}
Step 3. B → C 적용 ─────────> B가 {A, D, B}의 부분집합이므로 C 추가 → (AD)+ = {A, D, B, C}
Step 4. CD → E 적용 ────────> CD가 {A, D, B, C}의 부분집합이므로 E 추가 → (AD)+ = {A, D, B, C, E}

결과: (AD)+ = {A, B, C, D, E} → AD는 해당 릴레이션의 '슈퍼키'임을 판별 가능
```

### 3. 핵심 키워드 (Must-Have)

- **성질 관점:** 건전성(Soundness, 거짓 유도 안 됨), 완전성(Completeness, 모든 참인 FD 유도 가능)
- **3대 기본 공리:** 반사율(Reflexivity), 첨가율(Augmentation), 이행율(Transitivity)
- **3대 부가 규칙:** 합집합 규칙(Union), 분해 규칙(Decomposition), 의사이행 규칙(Pseudo-transitivity)
- **활용 관점:** 함수 종속성 폐포(F⁺), 애트리뷰트 폐포(X⁺), 최소 보존 집합(Minimal Cover)

### 4. 맥락과 비교

암스트롱 공리는 단순히 수학 공식에 머무는 것이 아니라, 데이터베이스 정규화 이론의 하부 토대로서 상위 개념들과 긴밀한 인과 관계를 맺고 있습니다.

```
[데이터베이스 수학적 기초] 암스트롱 공리 (기본/부가 규칙)
         │
         ▼ (도출 및 증명)
[논리 구조 분석 도구] 함수 종속성(FD) 및 폐포(Closure) 계산
         │
         ▼ (설계 가이드라인 적용)
[이상현상 제거 프로세스] 데이터베이스 정규화 (1NF ~ BCNF, 4NF/5NF)
```

애트리뷰트 폐포 계산은 설계 도구(CASE Tool)에서 슈퍼키(Super Key)와 후보키(Candidate Key)를 자동으로 식별하는 알고리즘의 근간이 됩니다. 유도된 후보키를 바탕으로 2NF(부분함수종속 제거), 3NF(이행함수종속 제거)의 대상을 명확히 판별해 냄으로써 스키마 설계의 정밀도를 보장합니다.

### 5. 실무 제언

**수학적 이론과 실무 튜닝의 균형**

- **챌린지:** 암스트롱 공리를 엄격하게 적용하여 3NF나 BCNF까지 도달하면 데이터의 이상 현상(삽입, 삭제, 수정)은 완벽히 차단됩니다. 하지만 실무에서는 테이블 파편화로 인해 과도한 조인(Join) 연산이 발생하여 시스템 성능이 저하되는 트레이드오프(Trade-off)가 발생합니다.
- **제언:** 기술사는 암스트롱 공리를 바탕으로 완벽한 논리 스키마를 도출한 후, 실제 대용량 트랜잭션 환경의 부하를 고려하여 의도적이고 통제된 반정규화(Denormalization)나 인덱스 전략을 설계 단계에서 역으로 제시할 수 있는 안목을 갖추어야 합니다.

**최소 보존 집합(Minimal Cover)을 통한 기술 부채 최소화**

- 공리를 통해 중복되거나 무의미한 함수 종속성을 제거하고, 릴레이션의 모든 종속성을 표현할 수 있는 가장 단순화된 형태인 최소 보존 집합을 도출할 수 있습니다. 데이터 아키텍트(DA)는 이 공리를 기반으로 설계 검증을 자동화하고 정형화된 거버넌스를 구축함으로써, 시스템 고도화 시 발생할 수 있는 데이터 구조적 리스크와 기술 부채를 설계 초기 단계부터 완벽하게 통제해야 합니다.
