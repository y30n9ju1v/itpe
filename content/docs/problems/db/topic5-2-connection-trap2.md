## 해설

> **데이터 모델 품질의 핵심 검증 지표**  
> 연결함정(Connection Trap): 조인 시 가짜 정보를 유발하거나 경로를 단절시키는 논리적 설계 오류

데이터 모델링 단계에서 개체 간의 관계성을 명확히 파악하지 못하거나 생략하면, 릴레이션 조인 시 실제 비즈니스 사실과 다른 가짜 정보가 유도되거나 관계 경로가 끊어지는 구조적 모순 현상이 발생합니다. 레거시 시스템을 클라우드로 이관하거나 전사 데이터 거버넌스를 재정비할 때 관계가 모호하게 설계되면 SQL 조인 시 원하지 않는 가짜 데이터가 추출되는 심각한 품질 결함으로 이어집니다. 쿼리가 정상 수행되어 결과를 반환하기 때문에 개발 단계에서 인지하기 어렵고, 오픈 후 통계·정산·결산 시스템에서 심각한 데이터 불일치를 일으킵니다.

### 1. 출제 배경 및 의도

연결함정은 엔티티(Entity) 간의 관계(Relationship)를 명확히 정의하지 못했을 때 발생하는 가장 대표적인 논리적 설계 오류로, 데이터 모델의 정확성과 정보 보존 능력을 판별하기 위해 단골로 출제되는 핵심 기술 검증 토픽입니다. 채점관은 수험생이 관계의 카디널리티와 선택성을 검증하는 비즈니스 시나리오 테스트를 정기적으로 수행할 수 있는 역량을 갖추었는지 확인하고자 합니다.

### 2. 개념의 본질과 작동 메커니즘

주로 3개 이상의 엔티티가 얽히는 삼항 관계 이상의 업무 도메인을 단순한 1:M 구조로 잘못 연결할 때 발생합니다. 부모를 공유하는 자식 간의 관계 왜곡(무부착 함정)이나, 선택적 관계선 배치 오류에 따른 경로 미아 현상(균열 함정)으로 구체화됩니다.

**무부착 함정(Fan Trap):** 하나의 부모 엔티티가 두 개의 자식 엔티티와 각각 1:M 관계를 가질 때, 두 자식 엔티티 간의 직접적인 연결 고리가 소실되어 올바른 비즈니스 유추가 불가능해지는 현상입니다.

- **구조적 형상:** `대리점 (M) <- (1) 회사 (1) -> (M) 제품`
- **모순 사례:** 회사 X가 대리점 A, B를 운영하고, 회사 X가 제품 1, 2를 생산할 때, 대리점 A가 오직 제품 1만 판매하는 비즈니스 규칙이 있더라도 조인 시 대리점 A가 제품 1과 2를 모두 판매하는 것처럼 가짜 튜플(Spurious Tuple)이 유도됩니다.
- **해결 방안:** 비즈니스 인과 관계를 재분석하여 연결 순서를 수직적 사슬 구조로 재배치합니다. `회사 (1) -> (M) 대리점 (1) -> (M) 제품`

**균열 함정(Chasm Trap):** 엔티티 간의 관계를 이어주는 중간 경로 상에 선택적(Optional, 0 또는 1) 관계 제약 조건이 존재하여, 특정 데이터 간의 연결 선로가 완전히 끊어지는 현상입니다.

- **구조적 형상:** `대리점 (1) -> (M) 사원 (0..1) -> (M) 고객`
- **모순 사례:** 대리점과 사원은 필수 관계이나 사원과 고객은 선택적 관계일 때, 담당 사원이 아직 정해지지 않은 신규 고객이 추가되면 그 고객이 어느 대리점 관리 체계에 속하는지 경로를 타고 올라갈 수 없는 관계 균열(Lost Path)이 일어납니다.
- **해결 방안:** 단절된 경로를 직접 메우기 위해 최상위 엔티티와 최하위 엔티티 간에 직접 관계(Direct Relationship)를 신설합니다.

### 3. 핵심 키워드 (Must-Have)

- **유형 관점:** 무부착 함정(Fan Trap), 균열 함정(Chasm Trap)
- **현상 관점:** 가짜 튜플(Spurious Tuple), 미아 경로(Lost Path), 추론 오류, 관계 단절
- **해결 기술:** 관계 구조 재배치(Re-ordering), 직접 관계 추가(Direct Relationship), 필수(Mandatory) 제약 조건, 교차 엔티티(Intersection Entity)

### 4. 맥락과 비교

연결함정은 릴레이션 간의 구조적 결합도를 대변하며, 그 형태에 따라 다음과 같이 대칭적인 특성을 지닙니다.

| 비교 항목 | 무부착 함정 (Fan Trap) | 균열 함정 (Chasm Trap) |
| --- | --- | --- |
| **발생 원인** | 하나의 부모 엔티티에 2개의 자식 엔티티가 1:M 관계로 결합됨 | 엔티티 간 경로 상에 선택적(Optional) 관계선이 존재함 |
| **치명적 현상** | 조인 시 존재하지 않는 **가짜 데이터(Spurious Tuple)** 양산 | 특정 데이터 간의 **연결 경로가 유실(Lost Path)**됨 |
| **구조적 형상** | `자식 A (M) <- (1) 부모 (1) -> (M) 자식 B` | `엔티티 A (1) -> (M) 엔티티 B (0..1) -> (M) 엔티티 C` |
| **해결 전략** | 비즈니스 선후 규칙에 따라 관계 순서를 수직 재배치 | 단절을 유발하는 상-하위 엔티티 간 직접 관계선 신설 |

### 5. 실무 제언

**연결함정 예방을 위한 진단 체크리스트**

- **챌린지:** ERD 상에서 점선(Optional) 관계가 연속으로 나열되어 미아 경로를 유발하는지, 1:M 관계가 한 엔티티를 중심으로 사방으로 뻗어나가는 부채꼴(Fan) 모양을 띠는지를 개발 단계에서 발견하기 어렵습니다.
- **제언:** 외래키(FK)의 Null 허용 구간을 집중 추적·프로파일링하고, 샘플 데이터를 통한 무손실 조인 및 가짜 튜플 유도 여부 테스트를 정기적으로 수행해야 합니다.

**MSA 환경으로 전이된 연결함정의 리스크 확대**

- **챌린지:** 현대의 마이크로서비스 아키텍처(MSA) 환경에서는 서비스별로 DB가 분리(Database-per-Service)되므로 관련 테이블들이 서로 다른 네트워크 노드에 파편화됩니다. 이 상태에서 연결함정이 내포되면 서비스 간 API 다중 호출(Chaining) 시 감당할 수 없는 데이터 정합성 파탄과 대규모 트래픽 병목을 유발합니다.
- **제언:** 프로젝트 초기 스프린트에서 도메인 주도 설계(DDD)를 적용할 때 애그리거트(Aggregate) 경계를 명확히 하고, 삼항 이상의 복잡한 순환 관계는 교차 엔티티(Intersection Entity) 패턴을 의무 적용하여 관계를 명확한 1:M 구조로 단순화해야 합니다.
