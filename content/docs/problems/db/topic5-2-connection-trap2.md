## 해설

### 출제 배경 및 의도

데이터 모델링 단계에서 개체 간의 관계성을 명확히 파악하지 못하거나 생략하면, 릴레이션 조인 시 실제 비즈니스 사실과 다른 가짜 정보가 유도되거나 관계 경로가 끊어지는 구조적 모순 현상이 발생합니다. 쿼리가 오류 없이 정상 수행되기 때문에 "쿼리는 돌아가는데 결과가 이상하다"는 증상으로만 나타나 원인 파악이 까다롭습니다.

레거시 시스템을 클라우드로 이관하거나 전사 데이터 거버넌스를 재정비할 때 관계가 모호하게 설계되면 SQL 조인 시 원하지 않는 가짜 데이터가 추출되는 심각한 품질 결함으로 이어집니다. 채점관은 카디널리티와 선택성을 검증하는 비즈니스 시나리오 테스트를 정기적으로 수행할 수 있는 역량을 갖추었는지 확인합니다.

### 1. 연결함정(Connection Trap), 개요

정 의  • 3개 이상의 엔티티가 얽히는 삼항 관계를 단순한 1:M 구조로 잘못 연결할 때 발생하는 논리적 설계 오류로, 조인 시 가짜 튜플 유도 또는 연결 경로 단절을 일으키는 데이터 모델 품질 결함
       - SQL 구문 오류 없이 정상 실행되어 통계·정산 시스템에서 치명적 데이터 왜곡으로 발현되는 잠복형 결함

```
[연결함정 발생 맥락]

 3개 이상 엔티티의 복잡한 비즈니스 관계
            │
            ├──[단순 1:M 선형 구조로 잘못 연결]──────> Fan Trap (가짜 튜플 양산)
            │     부모 공유 자식 간 직접 연결 소실
            │
            └──[선택적 관계 배치 오류]──────────────> Chasm Trap (경로 단절)
                  중간 경로 Optional 관계로 단절
```

- 연결함정은 ERD 상에 명시적 오류가 없고 SQL도 정상 동작하므로, 비즈니스 시나리오 기반 데이터 무결성 검증 없이는 발견이 사실상 불가능하다.

### 2. 무부착 함정(Fan Trap)과 균열 함정(Chasm Trap) 상세

1) 무부착 함정(Fan Trap) — 비즈니스 사례 중심

| 구분 | 항목 | 설명 |
|------|------|------|
| **구조적 형상** | ERD 패턴 | 대리점(M) ← (1) 회사(1) → (M) 제품 |
| **모순 사례** | 비즈니스 규칙 | 회사 X가 대리점 A·B 운영, 제품 1·2 생산 |
| | 왜곡 현상 | 대리점 A가 제품 1만 판매함에도 조인 시 제품 2 판매도 포함되는 가짜 튜플 생성 |
| **해결 방안** | 재배치 | 회사(1) → (M) 대리점(1) → (M) 제품 — 인과 관계 수직 사슬 구조로 재배치 |

```
[Fan Trap 발생]                        [Fan Trap 해결]

 ┌────┐ 1   M ┌────┐ M   1 ┌────┐      ┌────┐ 1  M ┌────┐ 1  M ┌────┐
 │제 품├───────┤회 사├───────┤대리│      │회 사├──────┤대리├──────┤제 품│
 └────┘       └────┘       │ 점 │      └────┘      │ 점 │      └────┘
                            └────┘                  └────┘
 (제품과 대리점 직접 관계 없음 → 카티션)  (수직 사슬: 회사→대리점→제품)
```

- Fan Trap의 근본 원인은 비즈니스 인과 관계(선후 관계)를 무시하고 단순히 공유 부모를 통해 자식들을 연결한 것이므로, 인과 관계 재분석이 핵심 해결책이다.

2) 균열 함정(Chasm Trap) — 비즈니스 사례 중심

| 구분 | 항목 | 설명 |
|------|------|------|
| **구조적 형상** | ERD 패턴 | 대리점(1) → (M) 사원(0..1) → (M) 고객 |
| **모순 사례** | 비즈니스 규칙 | 대리점↔사원 필수 관계, 사원↔고객 선택 관계 |
| | 왜곡 현상 | 담당 사원 미배정 신규 고객의 소속 대리점 추론 불가(Lost Path) |
| **해결 방안** | 직접 관계 신설 | 대리점과 고객 간 직접 관계(Direct Relationship) 추가 |

```
[Chasm Trap 발생]                      [Chasm Trap 해결]

                                        ┌─────────────────────────┐
                                        │    (직접 관계 추가)      │
                                        ▼ 1                     M ▲
 ┌────┐ 1  M ┌────┐ 0..1  M ┌────┐    ┌────┐ 1 M ┌────┐ 0..1 M ┌────┐
 │대리├──────┤사 원├── ── ──┤고 객│    │대리├────┤사 원├──────┤고 객│
 │ 점 │      └────┘ (선택적) └────┘    │ 점 │    └────┘         └────┘
 └────┘                                └────┘
 (사원 미배정 고객 → 대리점 역추적 불가)   (대리점→고객 직접 관계로 균열 차단)
```

- 균열 함정은 선택적(Optional) 관계가 경로 중간에 위치할 때 발생하므로, ERD 설계 시 필수(Mandatory) 관계만으로 구성된 완전한 연결 경로가 존재하는지 반드시 검증해야 한다.

### 3. 연결함정 2대 유형 비교 및 트렌드

| 비교 항목 | 무부착 함정(Fan Trap) | 균열 함정(Chasm Trap) |
|-----------|----------------------|----------------------|
| **발생 원인** | 부모 공유 자식 2개의 직접 연결 소실 | 경로 중간 선택적(Optional) 관계 |
| **치명적 현상** | **가짜 튜플(Spurious Tuple)** 양산 | **연결 경로 유실(Lost Path)** |
| **SQL 증상** | 조회 결과 건수가 예상보다 폭발적 증가 | 특정 데이터 간 조인 결과 NULL 또는 누락 |
| **해결 전략** | 비즈니스 인과 관계 기반 수직 재배치 | 상·하위 엔티티 간 직접 관계선 신설 |
| **설계 도구** | 교차 엔티티(Intersection Entity) 도입 | FK Null 허용 구간 제거 또는 직접 FK 추가 |

- 현대 MSA 환경에서 DB-per-Service 분리 구조에서 연결함정이 내포되면 서비스 간 API 다중 호출 시 데이터 정합성 파탄이 발생하므로, DDD 기반 바운디드 컨텍스트 경계 설계와 애그리거트 패턴의 중요성이 더욱 증가하고 있다.  "끝"

### 실무 제언

**연결함정 예방을 위한 진단 체크리스트**
- **챌린지**: ERD 상에서 점선(Optional) 관계가 연속으로 나열되어 미아 경로를 유발하는지, 1:M 관계가 한 엔티티를 중심으로 사방으로 뻗어나가는 Fan 모양인지를 개발 단계에서 발견하기 어렵다.
- **제언**: FK의 Null 허용 구간을 집중 추적·프로파일링하고, 샘플 데이터를 통한 무손실 조인 및 가짜 튜플 유도 여부 테스트를 ERD 설계 리뷰 단계에서 정기적으로 수행해야 한다.

**MSA 환경으로 전이된 연결함정의 리스크 확대**
- **챌린지**: MSA 환경에서 DB-per-Service 분리로 관련 테이블들이 파편화되면, 연결함정이 내포된 상태에서 서비스 간 API 다중 호출 시 감당할 수 없는 데이터 정합성 파탄과 트래픽 병목을 유발한다.
- **제언**: 프로젝트 초기 스프린트에서 DDD를 적용할 때 애그리거트(Aggregate) 경계를 명확히 하고, 삼항 이상의 복잡한 순환 관계는 교차 엔티티(Intersection Entity) 패턴을 의무 적용하여 명확한 1:M 구조로 단순화해야 한다.

**레거시 이관 시 연결함정 사전 진단**
- **챌린지**: 레거시 시스템 클라우드 이관 시 수천 개 테이블의 관계를 수작업으로 검증하면 연결함정 식별이 누락되어 이관 후 데이터 품질 결함이 재발한다.
- **제언**: CASE 도구(ERwin, DA# 등)와 FK Null 프로파일링 스크립트를 활용하여 Fan Trap 패턴(부모 1개에 자식 2개 이상 Fan 구조)과 Chasm Trap 패턴(연속 Optional 관계)을 자동 탐지·분류하는 데이터 모델 품질 진단 자동화 체계를 구축해야 한다.
