## 문제

데이터 통합 및 마이그레이션 프로젝트에서 데이터 무결성 목표를 달성하기 위해서는 데이터들의 정합성을 확보하고 신뢰도를 높이는 일이 매우 중요하다. 다음의 내용을 설명하시오.

가. 데이터 무결성(Integrity)과 정합성(Consistency)의 차이  
나. 데이터 값(Value) 진단 프로파일링(Data Profiling)의 중점 분석 관점  
다. 데이터 마이그레이션 검증 테스트 방법

## 해설

### 출제 배경 및 의도

클라우드 전환, 시스템 통합, 레거시 현대화 프로젝트에서 데이터 마이그레이션은 가장 리스크가 높은 단계입니다. 마이그레이션 후 데이터 오류는 운영 중단·업무 오류로 직결되며, 사후 복구 비용은 사전 검증 비용의 10배 이상에 달합니다.

128회 정보처리기술사 시험에서 출제된 이 문제는 데이터 품질 관리와 마이그레이션 엔지니어링이 결합된 실무 중심 토픽입니다. Oracle → Aurora, MS-SQL → PostgreSQL 등 클라우드 DB 마이그레이션이 활발히 진행되면서 마이그레이션 품질 검증 역량이 핵심 기술사 역량으로 부각되었습니다.

채점관은 무결성과 정합성의 개념 차이를 명확히 구분하고, 프로파일링 관점과 검증 방법을 구체적으로 서술하는 능력을 평가합니다.

### 1. 데이터 마이그레이션 품질 확보, 개요

정 의  • 원천 시스템에서 목적지 시스템으로 데이터를 이전할 때
       - 무결성·정합성을 보장하기 위해 프로파일링·ETL·검증 테스트를 수행하는 품질 확보 체계

```
[데이터 마이그레이션 품질 확보 흐름]

원천 시스템                 마이그레이션                목적지 시스템
    │                          │                            │
    ├─ 데이터 프로파일링 ───────→│                            │
    │  (값 진단, 이상 탐지)       │                            │
    │  ←── 품질 규칙 정의 ─────── │                            │
    │                          │                            │
    ├─ 추출(Extract) ───────────→│                            │
    │                          ├─ 변환(Transform) ───────────│
    │                          │  (매핑, 정제, 변환)          │
    │                          ├─ 적재(Load) ────────────────→│
    │                          │                            │
    │                          │           ├─ 검증 테스트 ───│
    │                          │           │  (Count/Hash/  │
    │                          │           │   Sampling)    │
    │  ←───────────── 결과 비교 ──────────────────────────────│
```

- 원천 프로파일링 단계에서 품질 이슈를 사전 탐지해야 마이그레이션 후 재작업 비용을 최소화

### 2. 핵심 내용 — 무결성, 프로파일링, 검증 테스트

1) 가. 데이터 무결성(Integrity)과 정합성(Consistency)의 차이

| 항목 | 데이터 무결성(Integrity) | 데이터 정합성(Consistency) |
|------|------------------------|--------------------------|
| 정의 | 데이터가 정해진 규칙·제약 조건을 위반하지 않는 상태 | 연관된 데이터들이 서로 논리적으로 모순되지 않는 상태 |
| 관점 | 개별 데이터 항목의 규칙 준수 여부 | 복수 데이터 간의 관계·일관성 |
| 위반 사례 | PK 중복, NOT NULL 위반, FK 불일치, 도메인 위반 | 주문 금액 ≠ 단가 × 수량, 주소 코드 ≠ 실제 주소 |
| DB 관점 | 엔티티·참조·도메인 무결성 | ACID의 Consistency, 교차 테이블 일관성 |
| 마이그레이션 시 | 원천 DB 제약 위반 데이터 탐지 | 원천-목적지 간 집계 값·연관 데이터 일치 여부 |

```
[무결성 vs 정합성 개념 비교]

무결성(Integrity)              정합성(Consistency)
    ├─ 엔티티 무결성                ├─ 수평 정합성
    │  (PK 유일성·NOT NULL)        │  (동일 테이블 내 논리 규칙)
    ├─ 참조 무결성                  ├─ 수직 정합성
    │  (FK → PK 참조 보장)         │  (상세-요약 테이블 간 합계 일치)
    └─ 도메인 무결성                └─ 시스템 간 정합성
       (값 범위·형식 준수)             (원천 ↔ 목적지 데이터 일치)
```

2) 나. 데이터 값 진단 프로파일링 — 중점 분석 관점

| 분석 관점 | 세부 내용 | 발견 이슈 유형 |
|---------|----------|--------------|
| 컬럼 통계 분석 | Min/Max/Avg·분포·NULL 비율 측정 | 이상값(Outlier), 과도한 NULL, 편향 분포 |
| 패턴 분석 | 전화번호·이메일·날짜 형식 패턴 검사 | 형식 불일치, 다중 형식 혼재 |
| 중복 분석 | 동일·유사 레코드 탐지(퍼지 매칭 포함) | 중복 고객, 중복 제품 코드 |
| 참조 무결성 분석 | FK 컬럼이 PK 테이블에 존재하는지 검사 | 고아 레코드(Orphan Record) |
| 도메인 분석 | 허용 가능한 값 목록(코드 도메인) 준수 여부 | 미정의 코드값, 폐기된 코드 사용 |
| 관계 분석 | 테이블 간 업무 규칙 준수 여부 | 논리적 정합성 위반 |

3) 다. 데이터 마이그레이션 검증 테스트 방법

| 검증 방법 | 설명 | 장점 | 단점 |
|---------|------|------|------|
| 레코드 수 검증(Count Test) | 원천·목적지 테이블별 ROW COUNT 비교 | 전수 검증, 누락 탐지 | 값 오류 탐지 불가 |
| 해시 비교(Hash Comparison) | 행·컬럼 단위 MD5/SHA256 체크섬 비교 | 대량 데이터 고속 검증 | 데이터 변환 시 허용 로직 필요 |
| 샘플링 검증(Sampling Test) | 통계적 샘플링으로 실제 값 비교 | 상세 값 검증, 업무 규칙 확인 | 오류 누락 가능성 |
| 집계 검증(Aggregation Test) | SUM·AVG·MAX·MIN 비교 | 수치 정합성 검증에 효과적 | 집계 외 필드 검증 불가 |
| 참조 무결성 검증 | FK 제약 위반 건수 확인 | 관계 정합성 확인 | 목적지 DB 제약 설정 필요 |

### 3. 마이그레이션 vs 유사 작업 비교

| 비교 항목 | 데이터 마이그레이션 | 데이터 통합(Integration) | 데이터 동기화(Synchronization) |
|---------|----------------|------------------------|-------------------------------|
| 시점 | 일회성 이전 | 지속적 또는 일회성 | 지속적·실시간 |
| 방향 | 단방향 | 단방향/양방향 | 양방향 |
| 주요 관심사 | 완전성·정확성 보장 | 변환 규칙·통합 로직 | 지연시간·충돌 해결 |
| 검증 방법 | Count/Hash/Sampling | 데이터 품질 모니터링 | CDC 기반 변경 추적 |

- 마이그레이션 성공의 핵심은 사전 프로파일링으로 품질 이슈를 발굴하고, 다층 검증 테스트로 완전성·정확성을 입증하는 것임.  "끝"

### 실무 제언

**마이그레이션 전후 정제 전략 이원화**
- **챌린지**: 원천 데이터에 품질 이슈가 많을 경우, 전 정제는 일정이 지연되고 후 정제는 오염 데이터가 신규 시스템에 유입되는 딜레마가 발생합니다.
- **제언**: 비즈니스 크리티컬 데이터(고객 식별자, 계약 정보)는 마이그레이션 전 필수 정제하고, 비크리티컬 데이터는 마이그레이션 후 점진적 정제 로드맵을 수립하는 이원화 전략이 효과적입니다.

**검증 자동화 및 Quality Gate 운영**
- **챌린지**: 수동 검증은 대규모 마이그레이션에서 인력·시간 한계로 검증 범위가 제한되어 오류 누락 위험이 높습니다.
- **제언**: Count/Hash/Aggregation 검증 스크립트를 CI/CD 파이프라인에 통합하여 전체 마이그레이션 단계마다 Quality Gate를 자동화 운영해야 합니다.

**고아 레코드 처리 전략**
- **챌린지**: 원천 시스템의 참조 무결성이 깨진 고아 레코드(Orphan Record)를 마이그레이션하면 목적지 시스템에서 FK 제약 오류가 대량 발생합니다.
- **제언**: 프로파일링 단계에서 고아 레코드를 사전 식별하고, 더미 부모 레코드 생성·고아 레코드 격리·삭제 중 비즈니스 정책에 맞는 처리 방법을 사전 결정해야 합니다.
