## 해설

### 출제 배경 및 의도

데이터 모델링 단계에서 엔티티 간의 관계를 잘못 파악하거나 생략하면, 조인(Join) 시 원래 존재하지 않는 가짜 정보(Spurious Tuple)가 나타나거나 관계 경로가 단절되는 구조적 모순이 발생합니다. 연결함정(Connection Trap)은 SQL 구문 오류 없이 쿼리가 정상 실행되므로 개발 단계에서 발견하기 매우 어렵고, 통계·정산 시스템에서 치명적인 데이터 왜곡을 일으키는 가장 위험한 설계 오류 중 하나입니다.

전사 데이터 표준화 사업이나 차세대 시스템 고도화 시 관계의 모호성으로 인한 가짜 정보 추출 현상이 빈번하게 발생합니다. 채점관은 단순히 ERD 가독성만 보지 않고, 카디널리티와 선택성(Optionality)을 공학적으로 검증하여 데이터 모델의 정합성과 정보 보존 능력을 진단하는 역량을 갖추었는지 확인합니다.

### 1. 연결함정(Connection Trap), 개요

정 의  • 3개 이상의 엔티티가 다차원 비즈니스 규칙을 형성할 때, 관계의 카디널리티나 선택성 오설계로 인해 조인 시 가짜 튜플(Spurious Tuple)이 유도되거나 연결 경로가 단절되는 논리적 설계 결함
       - SQL 구문 오류 없이 정상 실행되어 개발 단계 발견이 극히 어려운 잠복형 데이터 품질 결함

```
[연결함정 2대 유형 발생 구조]

Fan Trap (무부착 함정)             Chasm Trap (균열 함정)
자식A (M) ← (1) 부모 (1) → (M) 자식B    A(1)→(M) B(0..1)→(M) C
         ↑                                         ↑
두 자식 간 직접 연결 소실              선택적 관계로 경로 단절
→ 조인 시 가짜 튜플 양산              → Lost Path: C의 A 추론 불가
```

- 연결함정은 "쿼리는 돌아가는데 결과가 이상하다"는 증상으로만 나타나 원인 파악이 까다롭고, 오픈 후 정산 오류로 발견되는 경우가 대부분이다.

### 2. 무부착 함정(Fan Trap)과 균열 함정(Chasm Trap) 상세

1) 무부착 함정(Fan Trap) — 가짜 튜플 양산

| 구분 | 항목 | 설명 |
|------|------|------|
| **발생 원인** | 구조 | 한 부모 엔티티에 2개 자식 엔티티가 각각 1:M 관계로 결합 |
| | 핵심 현상 | 두 자식 간 직접 연결 소실 → 조인 시 가짜 데이터 대량 양산 |
| | 사례 | 교수가 여러 학생 지도 + 여러 학과 소속 → 특정 학생의 학과 추론 불가 |
| **해결 방안** | 전략 | 인과 관계에 따라 관계 순서를 수직 재배치 |

```
[Fan Trap 발생]                      [Fan Trap 해결: 수직 재배치]

 ┌─────┐ 1   M ┌─────┐ M   1 ┌─────┐    ┌─────┐ 1  M ┌─────┐ 1  M ┌─────┐
 │소 속├───────┤교 수├───────┤학 생│    │소 속├──────┤교 수├──────┤학 생│
 └─────┘       └─────┘       └─────┘    └─────┘      └─────┘      └─────┘
 (소속이 부모 → 교수와 학생 직접 관계 없음)  (인과 관계 수직 사슬 구조)
```

- 부모 공유 구조에서 두 자식 엔티티 간의 독립적인 관계가 숨어 있는 경우, 반드시 교차 엔티티(Intersection Entity)를 도입하여 명확한 1:M 구조로 해소해야 한다.

2) 균열 함정(Chasm Trap) — 경로 단절

| 구분 | 항목 | 설명 |
|------|------|------|
| **발생 원인** | 구조 | 엔티티 간 경로상에 선택적(Optional, 0..1) 관계 존재 |
| | 핵심 현상 | 특정 데이터 간 연결 경로 유실(Lost Path) → 미아 경로 발생 |
| | 사례 | 학과→교수(필수), 교수→학생(선택) 구조 시 미배정 교수의 학생 학과 추론 불가 |
| **해결 방안** | 전략 | 단절된 최상위-최하위 엔티티 간 직접 관계(Direct Relationship) 신설 |

```
[Chasm Trap 발생]                    [Chasm Trap 해결: 직접 관계 추가]

                                      ┌──────────────────────────┐
                                      │      (직접 관계 추가)    │
                                      ▼ 1                      M ▲
 ┌─────┐ 1  M ┌─────┐ 0..1  M ┌─────┐  ┌─────┐ 1 M ┌─────┐ 0..1 M ┌─────┐
 │학 과├──────┤교 수├── ── ──┤학 생│  │학 과├────┤교 수├──────┤학 생│
 └─────┘      └─────┘ (선택적) └─────┘  └─────┘     └─────┘         └─────┘
 (교수 미배정 시 학생의 학과 추론 경로 단절)  (학과→학생 직접 관계로 균열 차단)
```

- 선택적(Optional) 관계가 연속으로 나열되는 ERD 구조를 CRUD 매트릭스와 비즈니스 시나리오로 검증하여 균열 함정을 조기 탐지해야 한다.

### 3. 연결함정 2대 유형 비교 및 트렌드

| 비교 항목 | 무부착 함정 (Fan Trap) | 균열 함정 (Chasm Trap) |
|-----------|------------------------|------------------------|
| **발생 원인** | 부모 1개에 자식 2개가 1:M 결합 | 경로상 선택적(Optional) 관계 존재 |
| **핵심 현상** | 조인 시 **가짜 튜플(Spurious Tuple)** 양산 | 연결 경로 유실 **(Lost Path)** |
| **구조 형상** | 자식A(M)←(1)부모(1)→(M)자식B | A(1)→(M)B(0..1)→(M)C |
| **해결 전략** | 인과 관계 기반 관계 수직 재배치 | 상·하위 엔티티 간 직접 관계선 신설 |

- 현대 MSA 환경에서 DB-per-Service 분리로 관련 테이블이 파편화되면 연결함정이 서비스 간 API 다중 호출 시 데이터 정합성 파탄과 트래픽 병목을 유발하므로, DDD 기반 애그리거트 경계 설계가 더욱 중요해지고 있다.  "끝"

### 실무 제언

**눈에 보이지 않는 위험성 — 연결함정 진단 체계 확립**
- **챌린지**: 연결함정은 컴파일 에러나 SQL 구문 오류를 발생시키지 않으므로 개발자가 초기에 인지하기 어렵고, 오픈 후 통계·정산 시스템에서 치명적인 데이터 왜곡으로 발현된다.
- **제언**: ERD 가독성만 볼 것이 아니라 CRUD 매트릭스와 비즈니스 시나리오 시뮬레이션을 정기적으로 가동해야 하며, M:N 관계를 단순 1:M으로 치환할 때 연결함정이 숨어들기 쉬우므로 개념 설계 단계부터 교차 엔티티를 명확히 설계하는 거버넌스가 정착되어야 한다.

**MSA 환경에서의 연결함정 리스크 확대**
- **챌린지**: MSA 환경에서 DB-per-Service 분리로 관련 테이블들이 서로 다른 네트워크 노드에 파편화되면, 연결함정이 내포된 상태에서 서비스 간 API 다중 호출(Chaining) 시 데이터 정합성 파탄과 대규모 트래픽 병목을 유발한다.
- **제언**: DDD(Domain-Driven Design) 적용 시 애그리거트(Aggregate) 경계를 명확히 하고, 삼항 이상의 복잡한 순환 관계는 교차 엔티티(Intersection Entity) 패턴을 의무 적용하여 관계를 명확한 1:M 구조로 단순화해야 한다.

**FK Null 허용 구간 집중 프로파일링**
- **챌린지**: ERD 상에서 점선(Optional) 관계가 연속으로 나열되어 미아 경로를 유발하는지, 1:M 관계가 한 엔티티 중심으로 사방으로 뻗어나가는 Fan 모양인지를 개발 단계에서 발견하기 어렵다.
- **제언**: 외래키(FK)의 Null 허용 구간을 집중 추적·프로파일링하고, 샘플 데이터를 통한 무손실 조인 및 가짜 튜플 유도 여부 테스트를 설계 리뷰 단계에서 정기적으로 수행해야 한다.
