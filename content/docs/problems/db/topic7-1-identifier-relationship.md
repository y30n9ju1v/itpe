## 해설

> **데이터 상속 구조의 방향타**  
> 식별자·비식별자 관계: 부모 PK의 전이 방식이 자식의 무결성·유연성·성능을 결정한다

식별자 관계(Identifying Relationship)는 부모 엔티티의 기본키(PK)가 자식 엔티티의 주식별자(PK)의 일부로 참여하는 강한 종속 관계이며, 비식별자 관계(Non-Identifying Relationship)는 부모의 PK가 자식의 일반 속성(FK) 영역으로만 전이되는 약한 연관 관계입니다. 쉽게 말해, 식별자 관계는 "부모 없으면 자식도 없다"는 강한 연결이고, 비식별자 관계는 "참조만 하되 독립적으로도 살 수 있다"는 느슨한 연결입니다. 실무 프로젝트에서는 설계자의 성향이나 ORM 도구의 편의성에 치우쳐 어느 한쪽 관계만을 맹목적으로 남용하는 설계 안티 패턴이 자주 발생하므로, 두 관계의 특성과 극단적 사용 시 문제점을 명확히 이해하는 것이 중요합니다.

### 1. 출제 배경 및 의도

부모 엔티티의 기본키(PK)가 자식 엔티티로 전이될 때의 방식을 결정하는 것은 데이터 아키텍처의 핵심 이정표입니다. 과도한 식별자 관계는 물리 테이블의 복합키 결합도를 높여 유연성을 떨어뜨리고, 과도한 비식별자 관계는 무분별한 조인(Join) 부하를 유발합니다. 채점관은 이러한 극단적 설계의 리스크를 인지하고 상황별 최적의 관계 모델링 전략을 제시할 수 있는지 검증하고자 합니다.

### 2. 개념의 본질과 작동 메커니즘

부모 엔티티의 주식별자가 자식 엔티티의 주식별자(PK) 컴포넌트로 상속되는가, 혹은 일반 속성(FK) 영역으로 전이되는가에 따라 데이터의 종속성과 수명주기(Lifecycle)를 제어하는 기법입니다.

**식별자 관계(Identifying Relationship):** 강한 종속성을 가진 실체 관계형 도메인에서는 부모의 PK가 자식의 PK에 결합되어 자식 인스턴스의 고유성을 함께 증명합니다. 자식은 자동으로 복합키(Composite PK) 구조를 띠며, 부모 인스턴스 삭제 시 자식도 연쇄 삭제(ON DELETE CASCADE)됩니다. ERD에서 **실선(Solid Line)**으로 표기합니다.

**비식별자 관계(Non-Identifying Relationship):** 약한 연관 관계성 도메인에서는 부모의 PK가 자식의 일반 속성(FK)으로 내려가 참조 모델링만 형성합니다. 자식은 자기 고유의 단일키(예: 시퀀스, UUID)를 유지하며, 부모가 삭제되어도 외래키(FK) 컬럼을 Null로 채우거나 다른 부모 ID로 변경하여 독립적으로 살아남을 수 있습니다. ERD에서 **점선(Dashed Line)**으로 표기합니다.

### 3. 핵심 키워드 (Must-Have)

- **식별자 관계 관점:** 강한 종속 관계(Strong Dependency), 주식별자(PK) 전이, 실선 표현, 복합키(Composite Key) 형성, 부모-자식 수명주기 동기화
- **비식별자 관계 관점:** 약한 연관 관계(Weak Association), 일반 속성(FK) 전이, 점선 표현, 단일키 유지, 독립적 수명주기, Null 허용 가능
- **문제점 관점:** 주식별자 속성 증가, SQL 복잡도 상승, 인덱스 효율성 저하, 불필요한 조인 유발, 데이터 정합성 유지 비용 증가, 개발 가독성 파탄

### 4. 맥락과 비교

두 관계 설정 기법은 설계의 물리적 형상과 데이터 운영 모델 측면에서 완벽하게 대칭적인 특성을 지닙니다.

| 비교 항목 | 식별자 관계 (Identifying Relationship) | 비식별자 관계 (Non-Identifying) |
| --- | --- | --- |
| **개념 정의** | 부모 엔티티의 주식별자가 자식 엔티티의 **주식별자(PK)**로 상속 | 부모 엔티티의 주식별자가 자식 엔티티의 **일반 속성(FK)**으로 상속 |
| **결합도 및 종속성** | **강한 종속 관계** (부모 없는 자식 존재 불가) | **약한 연관 관계** (독립적 개체 간 연결) |
| **ERD 표기법 (IE 기준)** | **실선 (Solid Line)** | **점선 (Dashed Line)** |
| **자식 식별자 구조** | 기본 복합키(Composite PK) 구성원 증가 | 단일 주식별자 유지 및 일반 컬럼 추가 |
| **자식 데이터 수명** | 부모 인스턴스 삭제 시 자식도 연쇄 삭제(Cascade) | 부모가 삭제되어도 탈착하여 독자 생존 가능 |
| **외래키(FK) Null 여부** | **Null 불가 (Not Null)** | **Null 허용 가능 (Optional)** |

#### 단일 기법 극단 사용 시 발생하는 문제점

**식별자 관계만으로 설정할 때의 문제점 (Over-Identifying Anomaly):**
- 부모에서 자식, 자식에서 손자로 깊어질수록 상위 계층의 PK들이 계속 누적 전이되어 최하위 엔티티에 복합 주식별자 인플레이션이 발생합니다 (컬럼 5~6개 이상).
- WHERE 절에 수많은 복합키 조건을 명시해야 해 SQL문이 비대해지고 개발 생산성이 저하됩니다.
- PK 기반 클러스터드 인덱스(Clustered Index)의 집적도 저하로 디스크 I/O 횟수가 증가하고 인덱스 페이지 분할(Page Split) 부하가 심화됩니다.

**비식별자 관계만으로 설정할 때의 문제점 (Over-Non-Identifying Anomaly):**
- 자식 테이블들이 부모의 식별자를 직접 가지지 못하므로, 최하위 자식 데이터를 최상위 부모 조건으로 필터링하려면 중간 계층의 모든 테이블을 반드시 조인으로 엮어야 합니다.
- 외래키(FK) 경로 우회로 인해 데이터 모델의 직관성이 떨어지고, 쿼리 작성 시 징검다리 테이블을 누락하여 데이터 미아나 정보 유실 현상이 발생하기 쉽습니다.
- FK 컬럼마다 독립적인 보조 인덱스(Secondary Index)를 중복 생성해야 하므로 저장소 비용과 인덱스 유지 비용이 불필요하게 가중됩니다.

### 5. 실무 제언

**패러다임 충돌(RDBMS vs ORM)과 하이브리드 배치 전략**

- **챌린지:** 전통적 RDBMS 중심 설계에서는 데이터의 참조 무결성을 위해 식별자 관계를 선호했습니다. 반면 JPA나 Hibernate 같은 ORM 환경에서는 복합키 매핑 클래스를 별도로 구현해야 하는 번거로움 때문에 모든 관계를 비식별자(대리키 중심)로 구성하려는 성향이 짙습니다.
- **제언:** 기술사는 특정 진영의 편의성에 매몰되어서는 안 됩니다. 데이터 마스터 성격의 기준 정보(예: 부서-사원)는 비식별자 관계로 설계하여 유연성을 확보하되, 트랜잭션 성격의 이력 정보나 종속성이 명확한 하위 엔티티(예: 주문-주문상세)는 식별자 관계로 묶어 조인 없이 자식 테이블 스캔만으로 부모 식별자를 즉시 추적할 수 있도록 도메인 비즈니스 수명주기에 따른 하이브리드 배치 전략을 리드해야 합니다. 식별자 관계로 인해 자식의 복합키가 3개 이상으로 늘어나는 지점에서는 인위적인 대리키(Surrogate Key)를 주식별자로 승격하고 기존 부모의 PK는 비식별자 관계(FK)로 강등시켜 복합키 인플레이션을 물리 레벨에서 차단해야 합니다.
