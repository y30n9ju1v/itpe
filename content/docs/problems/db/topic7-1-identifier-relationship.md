## 해설

### 출제 배경 및 의도

부모 엔티티의 기본키(PK)가 자식 엔티티로 전이될 때의 방식을 결정하는 것은 데이터 아키텍처의 핵심 이정표입니다. 식별자 관계는 "부모 없으면 자식도 없다"는 강한 연결이고, 비식별자 관계는 "참조만 하되 독립적으로도 살 수 있다"는 느슨한 연결입니다.

과도한 식별자 관계는 물리 테이블의 복합키 결합도를 높여 유연성을 떨어뜨리고, 과도한 비식별자 관계는 무분별한 조인(Join) 부하를 유발합니다. 실무 프로젝트에서는 ORM 도구의 편의성에 치우쳐 어느 한쪽만 맹목적으로 남용하는 설계 안티 패턴이 자주 발생합니다.

채점관은 이러한 극단적 설계의 리스크를 인지하고 상황별 최적의 관계 모델링 전략을 제시할 수 있는지 검증하고자 합니다.

### 1. 식별자/비식별자 관계, 개요

정 의  • 부모 엔티티의 주식별자(PK)가 자식 엔티티의 PK 컴포넌트로 상속되는가(식별자), 일반 속성(FK)으로 전이되는가(비식별자)에 따라 종속성과 수명주기를 제어하는 데이터 모델링 기법
       - 데이터의 비즈니스 수명주기와 종속성 강도에 따라 선택하는 관계 설계의 방향타

```
[식별자 관계 vs 비식별자 관계 구조도]

  식별자 관계 (Identifying)           비식별자 관계 (Non-Identifying)
  ─────────────────────────           ────────────────────────────────
  부모 ══════════ 자식                부모 - - - - - - 자식
  (실선: 강한 종속)                   (점선: 약한 연관)

  부모: [PK: 주문ID]                  부모: [PK: 부서코드]
         │ PK 전이                           │ FK만 전이
  자식: [PK: 주문ID + 상품ID]         자식: [PK: 사원번호(독립)]
         ↑ 복합키 구성                        [FK: 부서코드]

  - 부모 삭제 → 자식 연쇄 삭제        - 부모 삭제 → 자식 FK = NULL 가능
  - FK: Not Null                     - FK: Null 허용
```

- 식별자 관계는 강한 종속 도메인에, 비식별자 관계는 독립적 수명주기 도메인에 적용해야 한다.

### 2. 개념 비교 및 단일 기법 극단 사용 시 문제점

1) 식별자/비식별자 관계 비교

| 비교 항목 | 식별자 관계 (Identifying) | 비식별자 관계 (Non-Identifying) |
|----------|------------------------|-------------------------------|
| **PK 전이 방식** | 부모 PK → 자식 PK(복합키 구성원) | 부모 PK → 자식 일반 속성(FK) |
| **종속성** | 강한 종속 (부모 없이 자식 존재 불가) | 약한 연관 (독립적 개체 간 연결) |
| **ERD 표기 (IE)** | 실선 (Solid Line) | 점선 (Dashed Line) |
| **자식 식별자** | 복합키(Composite PK) 구조 | 단일 주식별자 유지 |
| **FK Null 여부** | Not Null (필수) | Null 허용 가능 |
| **삭제 연쇄** | ON DELETE CASCADE 적용 | SET NULL / RESTRICT 선택 가능 |

- 관계 유형 선택은 비즈니스 수명주기 종속성과 트랜잭션 접근 패턴에 근거해야 한다.

2) 단일 기법 극단 사용 시 문제점

| 구분 | 문제 유형 | 발생 원인 | 영향 |
|------|----------|----------|------|
| **식별자 관계만 사용** | 복합키 인플레이션 (Over-Identifying) | 부모→자식→손자로 PK 누적 전이 | 컬럼 5~6개 이상 복합키, SQL 비대화 |
| **식별자 관계만 사용** | 인덱스 효율 저하 | 복합 클러스터드 인덱스 집적도 저하 | 디스크 I/O 증가, Page Split 부하 심화 |
| **비식별자 관계만 사용** | 징검다리 조인 폭발 (Over-Non-Identifying) | 최상위 조건으로 최하위 조회 시 모든 중간 테이블 JOIN 필요 | 쿼리 복잡도 급증, 데이터 미아 발생 |
| **비식별자 관계만 사용** | 인덱스 중복 비용 증가 | FK 컬럼마다 보조 인덱스 중복 생성 | 저장소 비용 및 인덱스 유지 비용 가중 |

- 어느 한쪽만 극단적으로 사용하면 성능·유지보수·무결성이 모두 저하된다.

### 3. 하이브리드 설계 전략 및 현대 적용 트렌드

| 도메인 성격 | 권장 관계 유형 | 근거 |
|-----------|--------------|------|
| 기준 정보 (부서-사원) | 비식별자 | 독립적 수명주기, 유연성 우선 |
| 이력/종속 정보 (주문-주문상세) | 식별자 | 강한 종속, 조인 없이 부모 추적 가능 |
| 복합키 3개 이상 누적 시점 | 비식별자로 전환 | 대리키(Surrogate Key) 승격으로 복합키 인플레이션 차단 |
| ORM(JPA/Hibernate) 환경 | 비식별자 우선 | @IdClass 복잡도 회피, 단순 @ManyToOne FK 매핑 |
| MSA 분산 DB 환경 | 비식별자 | 서비스 경계 넘는 FK 제약 불가, 이벤트 기반 참조 |

- 하이브리드 배치 전략으로 도메인 비즈니스 수명주기에 따른 관계 유형을 차별화하는 것이 현대 데이터 아키텍처의 핵심이다.  "끝"

### 실무 제언

**RDBMS vs ORM 패러다임 충돌**
- **챌린지**: JPA/Hibernate 환경에서는 복합키 매핑 클래스(@IdClass, @EmbeddedId)를 별도 구현해야 하는 번거로움 때문에 모든 관계를 비식별자(대리키 중심)로 구성하려는 경향이 강하다.
- **제언**: 기준 정보성 엔티티(부서-사원)는 비식별자 관계로 유연성을 확보하고, 이력·종속 엔티티(주문-주문상세)는 식별자 관계로 설계하여 조인 없이 부모 식별자를 즉시 추적할 수 있는 하이브리드 배치 전략을 적용해야 한다.

**복합키 인플레이션 방지**
- **챌린지**: 식별자 관계가 3계층 이상 중첩되면 최하위 엔티티의 복합키 컬럼이 5~6개로 폭증하여 클러스터드 인덱스 효율이 급격히 저하된다.
- **제언**: 복합키 컬럼이 3개 이상 누적되는 지점에서는 대리키(Surrogate Key)를 주식별자로 승격하고 부모 PK는 비식별자 관계(FK)로 강등시켜 복합키 인플레이션을 물리 레벨에서 차단해야 한다.

**MSA 분산 환경 대응**
- **챌린지**: 서비스 경계별로 데이터베이스가 분리된 MSA 환경에서는 서비스 간 FK 제약 조건을 물리적으로 설정할 수 없어 참조 무결성이 보장되지 않는다.
- **제언**: MSA 환경에서는 비식별자 관계를 기본으로 채택하고, 서비스 간 참조 무결성은 이벤트 기반 Saga 패턴과 주기적 데이터 Reconciliation 배치로 보완하는 아키텍처를 구성해야 한다.
