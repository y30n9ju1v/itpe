## 해설

> **AI 임베딩 벡터의 의미론적 유사도 검색에 최적화된 데이터베이스**  
> 고차원 벡터를 저장하고 ANN(근사 최근접 이웃) 알고리즘으로 밀리초 내 유사 벡터를 검색하며, RAG 기반 LLM 시스템의 핵심 인프라

ChatGPT, Claude 같은 LLM(대형 언어 모델)의 실용화로 텍스트·이미지·오디오를 수백~수천 차원의 실수 벡터(임베딩)로 변환하여 저장·검색하는 요구가 폭발적으로 증가했습니다. 전통적인 RDBMS나 키-값 DB로는 수백만 개의 고차원 벡터에서 "의미론적으로 가장 유사한 것"을 찾는 작업을 효율적으로 수행할 수 없어, 전용 벡터 데이터베이스가 AI 인프라의 핵심 컴포넌트로 부상했습니다.

### 1. 출제 배경 및 의도

2024년 함속 기술사 스터디 문제로 출제된 이 주제는 GenAI(생성형 AI) 붐과 직결되는 가장 최신 DB 토픽입니다. LLM 기반 서비스 구축에서 RAG(Retrieval Augmented Generation) 아키텍처가 필수화되면서, 벡터 DB는 AI 엔지니어링의 핵심 인프라가 되었습니다. 채점관은 임베딩 개념, ANN 인덱스 알고리즘, 대표 제품, RAG 연계 아키텍처를 체계적으로 설명할 수 있는지를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

벡터 데이터베이스는 **고차원 실수 벡터(임베딩, Embedding)**를 저장하고, 쿼리 벡터와 저장된 벡터들 사이의 **유사도(Similarity)**를 기준으로 가장 가까운 벡터를 빠르게 검색하는 데이터베이스입니다.

```text
[임베딩 벡터 생성 및 저장 흐름]

원본 데이터 (텍스트/이미지/오디오)
         │
         ▼
  임베딩 모델 (Embedding Model)
  (text-embedding-ada-002, BGE-M3 등)
         │
         ▼
  고차원 실수 벡터 변환
  "NoSQL은 수평 확장에 유리하다"
  → [0.234, -0.891, 0.445, ..., 0.123]  (1536차원)
         │
         ▼
  벡터 DB 저장
  ┌──────────────────────────────────────────┐
  │  Vector ID │ 메타데이터    │ 벡터값       │
  │  doc_001   │ {source:...} │ [0.234, ...] │
  │  doc_002   │ {source:...} │ [0.891, ...] │
  └──────────────────────────────────────────┘
         │
  ▼ 쿼리 시
  쿼리 텍스트 → 임베딩 모델 → 쿼리 벡터
  → ANN 검색 → 유사도 상위 K개 반환
```

#### 유사도 측정 방법

| 방법 | 공식 개념 | 특징 |
|------|----------|------|
| **코사인 유사도 (Cosine Similarity)** | 벡터 간 각도 | 방향성 기반, 크기 무관, 텍스트 임베딩에 주로 사용 |
| **유클리드 거리 (L2 Distance)** | 벡터 간 직선 거리 | 크기와 방향 모두 반영 |
| **내적 (Inner Product/Dot Product)** | 벡터 성분 곱의 합 | 정규화된 벡터에서 코사인과 동일 |

#### ANN(Approximate Nearest Neighbor, 정확한 최근접 이웃 대신 근사값으로 빠르게 찾는 방법) 인덱스 알고리즘

| 알고리즘 | 구조 | 특징 | 대표 적용 |
|---------|------|------|---------|
| **HNSW (Hierarchical Navigable Small World)** | 계층적 그래프 | 높은 검색 정확도, 빠른 검색, 메모리 소비 큼 | Weaviate, Qdrant, Milvus |
| **IVF (Inverted File Index)** | 클러스터 기반 역인덱스 | 낮은 메모리, 빠른 검색, 정확도 낮을 수 있음 | Faiss, Milvus |
| **PQ (Product Quantization)** | 벡터 압축 | 메모리 대폭 절감, 속도 빠름, 정확도 일부 손실 | Faiss, ScaNN |
| **ScaNN (Scalable Nearest Neighbor)** | 구글 개발, 이방성 양자화 | 대규모에서 높은 정확도·속도 | Google 내부, Vertex AI |

```text
[HNSW 인덱스 구조]

레이어 2 (희소):  노드A ──────────────── 노드B
                   │
레이어 1 (중간):  노드A ── 노드C ──── 노드B
                   │         │
레이어 0 (밀집):  노드A─노드D─노드C─노드E─노드B─노드F...

쿼리 벡터 탐색: 최상위 레이어에서 시작 → 하위 레이어로 세밀하게 탐색
→ 로그 시간 복잡도 O(log N) 검색 가능
```

#### RAG (Retrieval Augmented Generation) 아키텍처

```text
[RAG 파이프라인]

[사용자 질문]
      │
      ▼
  쿼리 임베딩 변환
      │
      ▼
  ┌──────────────┐    유사도 검색    ┌──────────────────┐
  │ 벡터 DB 검색 │ ←────────────── │  사전 구축된      │
  │ (ANN 탐색)   │                  │  지식 벡터 인덱스  │
  └──────────────┘                  └──────────────────┘
      │
      ▼
  관련 문서 Top-K 검색
      │
      ▼
  [프롬프트 컨텍스트 조립]
  "다음 문서를 참고하여 답하시오: [검색된 문서들]
   질문: [사용자 질문]"
      │
      ▼
  ┌──────────────┐
  │   LLM 응답   │ (GPT-4, Claude 등)
  └──────────────┘
      │
      ▼
  [최종 답변] - 할루시네이션 감소, 최신 정보 반영
```

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 핵심 개념 | 임베딩(Embedding), 고차원 벡터, 유사도 검색, 시맨틱 검색 |
| ANN 알고리즘 | HNSW(계층적 탐색 그래프 기반 ANN), IVF(클러스터 기반 역인덱스), PQ(벡터 압축 방식), ScaNN |
| 유사도 지표 | 코사인 유사도, 유클리드 거리(L2), 내적(Inner Product) |
| 대표 제품 | Pinecone, Weaviate, Milvus, Qdrant, Chroma, pgvector |
| AI 연계 | RAG (Retrieval Augmented Generation), LLM, 임베딩 모델 |
| 성능 지표 | Recall@K, QPS (Queries Per Second), 인덱스 빌드 시간 |

**벡터 DB 대표 제품 비교**

| 제품 | 유형 | 주요 특징 | 적합 사례 |
|------|------|----------|----------|
| Pinecone | 완전 관리형 SaaS | 간편한 API, 자동 확장 | 스타트업, RAG 빠른 구축 |
| Weaviate | 오픈소스/클라우드 | 다중 모달, 내장 ML 모듈 | 엔터프라이즈 RAG |
| Milvus | 오픈소스 | 대규모 분산, 고성능 | 수십억 벡터 대규모 서비스 |
| Qdrant | 오픈소스 | Rust 기반, 필터링 강력 | 정밀 메타데이터 필터링 필요 시 |
| pgvector | PostgreSQL 확장 | 기존 PG와 통합 | 소규모, RDBMS 병행 사용 |
| Chroma | 오픈소스 | LangChain 친화적 | 프로토타이핑, 개발 환경 |

### 4. 맥락과 비교

| 비교 항목 | RDBMS | 전통 검색 엔진 (Elasticsearch) | 벡터 DB |
|----------|-------|-------------------------------|---------|
| 검색 방식 | 정확한 키워드·조건 매칭 | 역인덱스 기반 키워드 검색 (TF-IDF/BM25) | 의미론적 유사도 검색 |
| 쿼리 예시 | WHERE name = '노트북' | "노트북" 키워드 포함 문서 | "휴대용 컴퓨팅 기기" 의미 유사 문서 |
| 다국어/동의어 | 미지원 | 형태소 분석 필요 | 자동 처리 (임베딩이 의미 포착) |
| AI 연계 | 어려움 | 제한적 | 네이티브 지원 (RAG, 추천) |
| 확장성 | 수직 확장 중심 | 수평 확장 지원 | 수평 확장 (분산 ANN) |

### 5. 실무 제언

**청킹(Chunking) 전략이 RAG 품질을 결정**

- **챌린지:** 긴 문서를 벡터화할 때 문서 전체를 하나의 벡터로 만들면 세부 내용이 평균화되어 정밀 검색이 어렵습니다. 반대로 너무 작은 단위(문장 1개)로 청킹하면 맥락이 없어 LLM이 충분한 컨텍스트를 얻지 못합니다.
- **제언:** 문서의 성격에 따라 청킹 단위를 조절해야 합니다. 일반 문서는 256~512 토큰 단위로 오버랩(50~100토큰 중복)을 두어 맥락 연속성을 확보하고, 코드나 표처럼 구조화된 데이터는 의미 단위(함수·섹션)로 청킹하는 Semantic Chunking을 적용해야 합니다. 또한 부모-자식 청킹(Parent-Child Retrieval)을 통해 작은 단위로 검색하고 큰 단위를 LLM에 전달하는 방법도 효과적입니다.

**하이브리드 검색 (Hybrid Search)으로 정확도 향상**

- **챌린지:** 벡터 검색만으로는 정확한 제품 코드, 고유명사, 날짜 등의 키워드 검색이 약하고, 키워드 검색만으로는 의미론적 유사 문서를 찾지 못합니다.
- **제언:** 벡터 검색(Dense Retrieval)과 키워드 검색(Sparse Retrieval, BM25)을 결합하는 하이브리드 검색을 적용하고, RRF(Reciprocal Rank Fusion)나 가중 결합으로 결과를 재순위화(Re-ranking)하면 검색 정확도를 크게 높일 수 있습니다. Weaviate, Qdrant, Elasticsearch(kNN)는 이 하이브리드 검색을 기본 지원합니다.
