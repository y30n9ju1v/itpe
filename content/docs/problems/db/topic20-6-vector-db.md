## 해설

### 출제 배경 및 의도

ChatGPT, Claude 같은 LLM(대형 언어 모델)의 실용화로 텍스트·이미지·오디오를 수백~수천 차원의 실수 벡터(임베딩)로 변환하여 저장·검색하는 요구가 폭발적으로 증가했습니다. 전통적인 RDBMS나 키-값 DB로는 수백만 개의 고차원 벡터에서 "의미론적으로 가장 유사한 것"을 찾는 작업을 효율적으로 수행할 수 없어, 전용 벡터 데이터베이스가 AI 인프라의 핵심 컴포넌트로 부상했습니다.

2024년 함속 기술사 스터디 문제로 출제된 이 주제는 GenAI(생성형 AI) 붐과 직결되는 가장 최신 DB 토픽입니다. LLM 기반 서비스 구축에서 RAG(Retrieval Augmented Generation) 아키텍처가 필수화되면서, 채점관은 임베딩 개념, ANN 인덱스 알고리즘, 대표 제품, RAG 연계 아키텍처를 체계적으로 설명할 수 있는지를 검증합니다.

### 1. 벡터 데이터베이스, 개요

정 의  • 고차원 실수 벡터(임베딩)를 저장하고 ANN(Approximate Nearest Neighbor, 근사 최근접 이웃) 알고리즘으로 밀리초 내에 의미론적으로 유사한 벡터를 검색하는 AI 특화 데이터베이스
       - RAG(Retrieval Augmented Generation) 기반 LLM 시스템의 지식 저장소 역할을 담당

```
[임베딩 벡터 생성 및 검색 흐름]

원본 데이터 (텍스트/이미지/오디오)
         │
         ▼ 임베딩 모델 (text-embedding-ada-002, BGE-M3 등)
         │
         ▼
  "NoSQL은 수평 확장에 유리하다"
  → [0.234, -0.891, 0.445, ..., 0.123]  (1536차원)
         │
         ▼ 벡터 DB 저장 (ID + 메타데이터 + 벡터값)
         │
  ▼ 쿼리 시
  쿼리 텍스트 → 임베딩 모델 → 쿼리 벡터
  → ANN 검색 → 유사도 상위 K개 반환
```

- 전통 DB가 "정확한 매칭"을 수행하는 반면, 벡터 DB는 "의미론적 유사도"를 기준으로 검색하여 동의어·다국어·맥락 기반 검색을 자동 처리

### 2. ANN 인덱스 알고리즘과 유사도 측정

1) ANN 인덱스 알고리즘 비교

| 구분 | 항목 | 설명 |
|------|------|------|
| HNSW | 구조/특징 | 계층적 그래프 / 높은 정확도·빠른 검색, 메모리 소비 큼 |
| HNSW | 대표 적용 | Weaviate, Qdrant, Milvus (엔터프라이즈 RAG) |
| IVF | 구조/특징 | 클러스터 기반 역인덱스 / 낮은 메모리, 빠른 검색 |
| IVF | 대표 적용 | Faiss, Milvus (대용량 배치 검색) |
| PQ | 구조/특징 | 벡터 압축 / 메모리 대폭 절감, 정확도 일부 손실 |
| PQ | 대표 적용 | Faiss, ScaNN (수십억 벡터 규모) |
| ScaNN | 구조/특징 | 구글 이방성 양자화 / 대규모 높은 정확도·속도 |
| ScaNN | 대표 적용 | Google Vertex AI |

```
[HNSW 인덱스 구조]

레이어 2 (희소):  노드A ─────────────── 노드B
레이어 1 (중간):  노드A ── 노드C ──── 노드B
레이어 0 (밀집):  노드A─노드D─노드C─노드E─노드B─...

탐색: 최상위 레이어에서 시작 → 하위 레이어로 세밀하게 탐색
→ O(log N) 검색 복잡도
```

2) 유사도 측정 방법

| 구분 | 항목 | 설명 |
|------|------|------|
| 코사인 유사도 | 공식 개념 | 벡터 간 각도 기반 |
| 코사인 유사도 | 특징/적합 사례 | 크기 무관, 방향성 기반 / 텍스트 임베딩에 주로 사용 |
| 유클리드 거리(L2) | 공식 개념 | 벡터 간 직선 거리 |
| 유클리드 거리(L2) | 특징/적합 사례 | 크기·방향 모두 반영 / 이미지·오디오 임베딩 |
| 내적(Dot Product) | 공식 개념 | 벡터 성분 곱의 합 |
| 내적(Dot Product) | 특징/적합 사례 | 정규화된 벡터에서 코사인과 동일 / 추천 시스템 |

### 3. RAG 아키텍처와 주요 제품 비교

1) RAG(Retrieval Augmented Generation) 파이프라인

| 구분 | 항목 | 설명 |
|------|------|------|
| 사전 구축 | 문서 청킹 | 256~512 토큰 단위로 분할, 50~100 토큰 오버랩으로 맥락 연속성 확보 |
| 사전 구축 | 임베딩 변환 | 청크별 임베딩 모델 적용 → 벡터 DB 저장 |
| 쿼리 단계 | 쿼리 임베딩 | 사용자 질문을 동일 임베딩 모델로 벡터 변환 |
| 쿼리 단계 | ANN 검색 | 벡터 DB에서 유사도 상위 Top-K 문서 검색 |
| 응답 단계 | 프롬프트 조립 | 검색된 문서를 컨텍스트로 포함하여 LLM에 전달 |
| 응답 단계 | LLM 생성 | 검색된 지식 기반으로 할루시네이션 감소, 최신 정보 반영 |

- 벡터 DB는 GenAI 시대의 핵심 인프라로, 단독 사용보다 RDBMS(정형 데이터)·Elasticsearch(키워드 검색)와 결합한 하이브리드 아키텍처가 검색 정확도와 운영 효율을 극대화함.  "끝"

### 실무 제언

**청킹(Chunking) 전략이 RAG 품질을 결정**
- **챌린지**: 긴 문서를 하나의 벡터로 만들면 세부 내용이 평균화되어 정밀 검색이 어렵고, 너무 작은 단위(문장 1개)로 청킹하면 맥락이 없어 LLM이 충분한 컨텍스트를 얻지 못함
- **제언**: 일반 문서는 256~512 토큰 단위로 오버랩(50~100 토큰 중복)을 두어 맥락 연속성을 확보하고, 구조화된 데이터(코드·표)는 의미 단위(함수·섹션)로 청킹하는 Semantic Chunking 적용. 부모-자식 청킹(Parent-Child Retrieval)으로 작은 단위 검색 + 큰 단위 LLM 전달 전략도 효과적

**하이브리드 검색(Hybrid Search)으로 정확도 향상**
- **챌린지**: 벡터 검색만으로는 정확한 제품 코드·고유명사·날짜 등 키워드 검색이 약하고, 키워드 검색만으로는 의미론적 유사 문서를 찾지 못함
- **제언**: 벡터 검색(Dense Retrieval)과 키워드 검색(Sparse Retrieval, BM25)을 결합하는 하이브리드 검색을 적용하고, RRF(Reciprocal Rank Fusion)나 가중 결합으로 결과를 재순위화(Re-ranking)하여 검색 정확도를 크게 향상

**벡터 DB 인덱스 파라미터 최적화**
- **챌린지**: HNSW의 ef_construction(인덱스 빌드 품질)·M(이웃 노드 수) 파라미터를 잘못 설정하면 검색 정확도(Recall@K)와 QPS(Queries Per Second) 간 트레이드오프가 비최적화 상태가 됨
- **제언**: Recall@K vs QPS 벤치마크를 수행하여 서비스 SLA(예: Recall@10 > 0.95, QPS > 1000)를 만족하는 최소 파라미터를 선택하고, 인덱스 빌드 후 ann-benchmarks 툴로 실제 워크로드 기반 성능 검증 수행
