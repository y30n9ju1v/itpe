## 해설

### 출제 배경 및 의도

데이터 모델링 3단계와 기본원칙은 데이터베이스 도메인의 가장 고전적이고 기초적인 토픽이지만, MSA 확산으로 데이터가 서비스별로 파편화되는 현대 환경에서 오히려 그 가치가 강화되고 있습니다. 빅데이터와 클라우드, AI 환경에서도 고품질 데이터의 출발점은 초기 모델링의 엄밀함에 달려 있으며, 건물의 설계 도면처럼 잘못된 데이터 모델은 시공 중의 대규모 재작업(Rework)과 직결됩니다.

전사 관점의 고품질 데이터를 유지하려면 개념 모델링 단계에서의 철저한 데이터 분류와 원칙 준수가 필수적입니다. AI/LLM 활용을 위한 데이터 신뢰성도 초기 모델링 원칙 준수에서 시작되며, 채점관은 엔지니어링 프로세스의 첫 단추인 모델링 원칙을 명확히 알고 있는지 검증하려는 의도를 가집니다.

### 1. 데이터 모델링, 개요

정 의  • 현실 세계의 복잡한 비즈니스 프로세스와 요구사항을 업무 규칙화하여 데이터베이스 구조로 추상화하는 체계적인 설계 엔지니어링 활동
       - 사용자의 정성적/정량적 요구사항을 받아 점진적 추상화 단계를 거쳐 최종 물리 스키마(Output)로 변환되는 일련의 흐름

```
[데이터 모델링 흐름도]

 현실 세계             개념 세계              논리 세계             물리 세계
┌──────────┐        ┌──────────┐           ┌──────────┐         ┌──────────┐
│ 업무요건 │ ──①──> │ 개념모델 │ ───②────> │ 논리모델 │ ──③───> │ 물리모델 │
│(인터뷰)  │        │ (Core ERD│           │(상세 ERD │         │ (DDL/    │
│          │        │  주제영역│           │  정규화) │         │  스키마) │
└──────────┘        └──────────┘           └──────────┘         └──────────┘
   Input           추상화 수준: 高          추상화 수준: 中        추상화 수준: 低
                  (비즈니스 중심)          (시스템 독립)          (DBMS 종속)
```

- 데이터 모델링은 개념→논리→물리로 갈수록 구체성이 증가하는 수직적 상속 구조를 가진다.

### 2. 데이터 모델링 3단계와 3대 기본원칙

1) 3단계 상세 프로세스

| 단계 | 세부 절차 | 수행 내용 | 산출물 |
|------|-----------|-----------|--------|
| **개념 모델링** | ① 주제 영역 정의 ② 후보 Entity 선정 ③ 핵심 Entity 정의 ④ 관계 정의 | 전사적 관점 최상위 데이터 집합 분류, 문서/인터뷰 기반 엔티티 도출 | 개념 ERD, 주제 영역 설명서 |
| **논리 모델링** | ① 속성 정의 ② Entity 상세화 ③ 이력 관리 정의 | 최소 데이터 단위로 속성 상세화, 식별자 확정 및 상세 정규화(1차~BCNF) 수행 | 상세 ERD, 데이터 사전, 엔티티 정의서 |
| **물리 모델링** | ① 물리 환경 조사 ② 논리 모델 변환 ③ 반정규화 | 하드웨어 자원 및 DBMS 조사, 엔티티→테이블/속성→컬럼 변환, 성능 확보 | 물리 스키마, 테이블 정의서, DDL 스크립트 |

- 논리 모델은 특정 DBMS 제약에 구애받지 않는 순수 비즈니스 구조를 표현하며, 물리 모델은 DBMS 특성에 최적화된 실제 저장구조를 설계한다.

2) 3대 기본 원칙

| 원칙 | 영문명 | 핵심 의미 | 위반 시 결과 |
|------|--------|-----------|--------------|
| **커뮤니케이션 원칙** | Communication Principle | 모든 이해관계자가 동일하게 이해하는 모델 제시 | 설계 오해로 인한 Rework 발생 |
| **모델링 상세화 원칙** | Granularity Principle | 정보 구조의 최소 원자(Atomic) 수준까지 분할 | 원자성 미확보로 정규화 불가 |
| **논리적 표현 원칙** | Logical Representation | 물리적 DBMS 제약 완전 배제한 순수 논리 구조 | 논리/물리 모델 혼재 → 기술 부채 |

- 3대 원칙은 고품질 데이터 모델의 기준점이자 데이터 거버넌스(Data Governance)의 근간을 형성한다.

### 3. 단계별 비교 및 현대적 트렌드

| 비교 항목 | 개념 모델링 | 논리 모델링 | 물리 모델링 |
|-----------|-------------|-------------|-------------|
| **추상화 수준** | 高 (비즈니스 중심) | 中 (시스템 독립적) | 低 (기술/하드웨어 종속) |
| **주요 대상** | CEO, 현업 책임자, DA | 시스템 분석가, DA | DBA, DB 엔지니어 |
| **핵심 활동** | 주제 영역 정의, 핵심 엔티티 도출 | 속성 정의, 식별자 확정, 정규화 | 테이블 변환, 인덱스 설계, 반정규화 |
| **MSA 적용** | 바운디드 컨텍스트(BC) 정의 | 서비스별 도메인 모델 설계 | Database-per-Service 물리화 |

- MSA·클라우드 환경에서 서비스별로 파편화된 데이터를 통합 관리하려면 개념 모델링 단계부터 전사 표준 데이터 사전과 연계한 거버넌스 체계 수립이 필수적이다.  "끝"

### 실무 제언

**논리 모델과 물리 모델의 혼재 리스크**
- **챌린지**: 일정 압박으로 논리 모델링 단계를 건너뛰거나, 처음부터 특정 DBMS 제약을 의식해 물리 모델 위주로 설계하는 안티 패턴이 자주 발생하여 시스템 확장 시 거대한 기술 부채(Technical Debt)로 돌아온다.
- **제언**: 데이터 아키텍트(DA)와 현업 도메인 전문가의 긴밀한 협업 거버넌스를 구축하고, 공학적으로 무결한 논리 모델 확보 후 '선(先)정규화 후(後)조율' 체계를 정착시켜야 한다.

**반정규화(Denormalization)의 거버넌스**
- **챌린지**: 초기 정규화가 완료되지 않은 상태에서 성능 향상을 목적으로 무분별하게 반정규화를 적용하면 심각한 데이터 품질 파탄을 초래한다.
- **제언**: 반드시 인덱스 조정, 애플리케이션 튜닝, 파티셔닝을 우선 검토한 후, 최소화 원칙에 따라 통제된 반정규화만 적용해야 한다.

**MSA 환경에서의 데이터 모델링 확장**
- **챌린지**: 마이크로서비스 아키텍처에서 서비스별 DB 분리(Database-per-Service)로 인해 논리적으로 연결된 엔티티 간 관계가 서비스 경계를 넘어서는 경우가 발생한다.
- **제언**: DDD(Domain-Driven Design)의 애그리거트(Aggregate)와 바운디드 컨텍스트(Bounded Context) 개념을 데이터 모델링에 통합 적용하여, 서비스 간 데이터 정합성은 이벤트 기반 아키텍처(EDA)로 달성해야 한다.
