## 해설

> **정보시스템의 뼈대를 세우는 설계 엔지니어링**  
> 데이터 모델링 3단계(개념→논리→물리)와 3대 기본원칙

빅데이터와 클라우드, AI 환경에서도 고품질 데이터의 출발점은 초기 모델링의 엄밀함에 달려 있습니다. 건물로 치면 설계 도면이 잘못되면 시공 중에 뜯어고치는 것이 가장 비쌉니다. 데이터 모델의 결함은 프로젝트 후반부의 대규모 재작업(Rework)과 직결되며, AI/LLM 활용을 위한 데이터 신뢰성도 초기 모델링 원칙 준수에서 시작됩니다.

### 1. 출제 배경 및 의도

데이터 모델링 3단계와 기본원칙은 데이터베이스 도메인의 가장 고전적이고 기초적인 토픽이지만, MSA 확산으로 데이터가 서비스별로 파편화되는 현대 환경에서 오히려 그 가치가 강화되고 있습니다. 전사 관점의 고품질 데이터를 유지하려면 개념 모델링 단계에서의 철저한 데이터 분류와 원칙 준수가 필수적입니다. 채점관은 엔지니어링 프로세스의 첫 단추인 모델링 원칙을 명확히 알고 있는지 검증하려는 의도를 가집니다.

### 2. 개념의 본질과 작동 메커니즘

데이터 모델링은 현실 세계의 복잡한 비즈니스 프로세스와 요구사항을 업무 규칙화하여 데이터베이스 구조로 추상화하는 체계적인 설계 엔지니어링 활동입니다. 사용자의 정성적/정량적 요구사항(Input)을 받아 점진적 추상화 단계를 거쳐 최종 물리 스키마(Output)로 변환되는 일련의 흐름을 가집니다.

- **Input:** 현업 사용자 인터페이스 및 업무 요구사항 명제
- **Mechanism:** 현실 세계의 개체(Entity) 도출 → 비즈니스 규칙에 따른 관계(Relationship) 정의 및 정규화(Normalization, 데이터 중복과 이상현상을 없애기 위해 테이블을 올바른 구조로 분해하는 작업)를 통한 데이터 구조화 → DBMS 성능과 인덱스를 고려한 변환 및 반정규화
- **Output:** DDL(Data Definition Language) 스크립트 및 저장 데이터베이스 스키마

#### 데이터 모델링 3단계 상세 프로세스

| 단계 | 세부 절차 | 수행 내용 | 필수 고려사항 |
| --- | --- | --- | --- |
| **개념 모델링** *(Conceptual)* | ① 주제 영역 정의 ② 후보 Entity 선정 ③ 핵심 Entity 정의 ④ 관계 정의 | 전사적 관점 최상위 데이터 집합 분류, 문서/인터뷰 기반 엔티티 도출, 본질 식별자 정의 및 1:M 연관성 매핑 | 함부로 제외 금지, 상세화 지양, 확장성 보장 |
| **논리 모델링** *(Logical)* | ① 속성 정의 ② Entity 상세화 ③ 이력 관리 정의 | 최소 데이터 단위로 속성 상세화, 식별자 확정 및 상세 정규화(1차~BCNF) 수행, 시점/선분 이력 추적성 확보 | 최소 분할 필수, 업무 요구 근거 원자성 확보, 참조무결성 규칙 |
| **물리 모델링** *(Physical)* | ① 물리 환경 조사 ② 논리 모델 변환 ③ 반정규화 | 하드웨어 자원 및 DBMS 조사, 엔티티→테이블/속성→컬럼 변환, 성능 확보를 위한 분할 및 중복 컬럼 배치 | 반정규화 최소화, 인덱스/SQL 튜닝 선행 검토 |

#### 데이터 모델링 3대 기본 원칙

```
[데이터 모델링 3대 원칙]
 ├── 1. 커뮤니케이션 원칙 (Communication Principle) ── 사용자 중심 가시화
 ├── 2. 모델링 상세화 원칙 (Granularity Principle)   ── 정보 구조의 최소 원자화
 └── 3. 논리적 표현 원칙 (Logical Representation)    ── 물리 제약의 완전 배제
```

- **커뮤니케이션 원칙:** 모든 이해관계자(최종 사용자, 시스템 분석가, DBA 등)가 명확하고 동일하게 이해할 수 있는 모델 제시. 기술적 용어 지양, 업무 지향적 용어로 표기
- **모델링 상세화 원칙:** 조직이 관리하는 정보 구조의 최소 공통분모(원자성) 수준까지 분할. 불필요한 중복은 논리 모델 단계에서 선행 제거
- **논리적 표현 원칙:** 특정 하드웨어나 DBMS의 제약조건에 구애받지 않고 순수한 비즈니스 규칙과 구조를 논리적으로 반영

### 3. 핵심 키워드 (Must-Have)

- **모델링 3단계 관점:** 주제 영역(Subject Area), 핵심 엔티티(Core Entity), ERD, 속성 정의, 식별자 확정, 정규화(1차~BCNF), 반정규화, 물리 환경 조사
- **모델링 기본원칙 관점:** 커뮤니케이션(Communication), 상세화(Granularity), 논리적 표현(Logical Representation)
- **핵심 메커니즘 관점:** 추상화(Abstraction), 가시화(Visualization), 데이터 독립성(Data Independence), 업무 규칙(Business Rule)

### 4. 맥락과 비교

데이터 모델링 3단계는 서로 단절된 활동이 아니라 ISMP 및 개발 생명주기와 긴밀히 연계되며, 개념 → 논리 → 물리로 갈수록 구체성이 증가하는 수직적 상속 구조를 가집니다.

| 비교 항목 | 개념 모델링 (Conceptual) | 논리 모델링 (Logical) | 물리 모델링 (Physical) |
| --- | --- | --- | --- |
| **수행 목적** | 전사 관점의 핵심 데이터 골격 정의 | 비즈니스 요구사항의 완벽한 구조화 | 특정 DBMS 환경에 최적화된 저장구조 설계 |
| **추상화 수준** | 상 (High - 비즈니스 중심) | 중 (Medium - 시스템 독립적) | 하 (Low - 기술/하드웨어 종속적) |
| **주요 활동** | 주제 영역 정의, 핵심 엔티티 도출, 관계 정의 | 속성 정의, 식별자 확정, 상세 정규화 수행 | 테이블 변환, 인덱스/뷰 설계, 반정규화 고려 |
| **산출물** | 개념 ERD, 주제 영역 설명서 | 상세 ERD, 데이터 사전, 엔티티 정의서 | 물리 스키마, 테이블 정의서, DDL 스크립트 |
| **주요 대상** | CEO, 현업 부서 책임자, DA | 시스템 분석가, 데이터 아키텍트(DA) | DB 관리자(DBA), 데이터베이스 엔지니어 |

### 5. 실무 제언

**논리 모델과 물리 모델의 혼재 리스크**

- **챌린지:** 실무 프로젝트 현장에서는 일정 압박으로 인해 논리 모델링 단계를 건너뛰거나, 처음부터 특정 DBMS의 제약조건을 의식하여 물리 모델 위주로 설계하는 심각한 안티 패턴이 자주 발생합니다. 이는 향후 데이터 이관이나 시스템 확장 시 거대한 기술 부채(Technical Debt)로 돌아옵니다.
- **제언:** 프로젝트 초기 단계부터 데이터 아키텍트(DA)와 비즈니스 도메인 전문가(현업 사용자)의 긴밀한 협업 거버넌스를 구축해야 합니다. 공학적으로 무결한 논리 모델을 확보한 후(원칙 준수), 인프라 아키텍처와 트랜잭션 비용 관점에서 물리 모델 및 반정규화 튜닝을 단계별로 통제하는 '선(先)정규화 후(後)조율' 체계를 정착시켜야 합니다.

**반정규화(Denormalization)의 거버넌스**

- **챌린지:** 초기 정규화가 완료되지 않은 상태에서 성능 향상을 목적으로 데이터 무결성을 희생시켜 무분별한 반정규화를 적용하면 심각한 데이터 품질 파탄을 초래합니다.
- **제언:** 반드시 인덱스 조정, 애플리케이션 튜닝, 파티셔닝을 우선 검토한 후, 최소화 원칙에 따라 통제된 반정규화만 적용해야 합니다.
