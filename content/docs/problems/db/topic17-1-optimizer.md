## 해설

### 출제 배경 및 의도

옵티마이저(Optimizer)는 개발자가 작성한 선언형 SQL(무엇을 가져와라)을 DBMS 내부의 절차형 실행 계획(어떻게 가져와라)으로 변환하는 핵심 엔진이다. 같은 SELECT 문도 옵티마이저가 어떤 경로를 선택하느냐에 따라 수천 배의 성능 차이가 난다. 이를 이해하는 것이 SQL 튜닝의 출발점이다.

옵티마이저는 DBA와 데이터 아키텍트가 반드시 이해해야 할 DBMS 핵심 컴포넌트이다. RBO와 CBO의 차이, CBO의 통계정보 의존성, 힌트를 통한 개발자 개입 방법을 체계적으로 설명할 수 있는지를 검증한다. 최근 머신러닝 기반 학습형 옵티마이저(Oracle ML-based Cardinality Estimation 등)까지 연계하면 고득점이 가능하다.

### 1. 옵티마이저(Optimizer), 개요

정 의  • SQL 파싱 후 가능한 실행 방법(후보 플랜) 중 최소 비용(I/O, CPU)의 실행 계획을 선택하는 DBMS 내부 소프트웨어 모듈
       - SQL 파싱 → 논리적 최적화(쿼리 변환) → 물리적 최적화(비용 계산 및 플랜 선택) → 실행 계획(Execution Plan) 출력

```
[옵티마이저 처리 흐름]

  SQL 문장 입력
       │
       ▼
  ┌──────────────┐
  │  SQL 파서    │  ← 문법 검사, 파스 트리 생성
  └──────┬───────┘
         │
         ▼
  ┌──────────────────────────────────────┐
  │         옵티마이저 (Optimizer)        │
  │  ┌──────────────────────────────┐   │
  │  │ 1. 쿼리 변환 (Query Transform)│   │  ← 뷰 머징, 서브쿼리 최적화
  │  │ 2. 비용 추정 (Cost Estimation) │   │  ← 통계정보(행 수, NDV, 히스토그램)
  │  │ 3. 플랜 선택 (Plan Selection) │   │  ← 최저 비용 플랜 선택
  │  └──────────────────────────────┘   │
  └──────────────────────────────────────┘
         │
         ▼
  실행 계획 (Execution Plan) → Row Source Generation → 실행 엔진
```

- 옵티마이저의 핵심은 카디널리티(결과 행 수) 추정이며, 이것이 조인 순서와 접근 방법 선택을 결정한다.

### 2. 가. 옵티마이저 역할, 나. 유형(RBO/CBO), 다. 힌트 상세

1) 가. 옵티마이저의 역할

| 구분 | 역할 | 설명 |
|------|------|------|
| 쿼리 변환 | Query Transformation | 논리적으로 동일한 쿼리를 더 효율적인 형태로 변환 (서브쿼리→조인, 뷰 머징) |
| 비용 추정 | Cost Estimation | 각 실행 방법의 I/O, CPU, 메모리 비용을 통계정보 기반으로 계산 |
| 플랜 선택 | Plan Selection | 최소 비용의 실행 계획 선택 (인덱스 스캔 vs 풀 스캔, 조인 방법 등) |
| 카디널리티 추정 | Cardinality Estimation | 각 연산의 결과 행 수 추정 → 조인 순서 결정의 핵심 |

- 카디널리티 추정의 정확도가 옵티마이저 성능의 핵심이다.

2) 나. 옵티마이저 유형: RBO vs CBO

| 비교 항목 | RBO (Rule-Based Optimizer) | CBO (Cost-Based Optimizer) |
|---------|--------------------------|--------------------------|
| 기반 | 사전 정의 규칙 (우선순위 1~15) | 비용 추정 (통계정보) |
| 통계정보 | 불필요 | 필수 (정확도 의존) |
| 데이터 특성 반영 | 불가 (규칙만 적용) | 가능 (히스토그램, NDV 활용) |
| 예측 가능성 | 높음 (규칙 고정) | 통계 정확도에 의존 |
| 현재 상태 | 대부분 DBMS 지원 중단 | 현대 모든 DBMS 기본 채택 |
| 대표 DBMS | 구 Oracle 9i 이하 | Oracle 10g+, PostgreSQL, MySQL 8.0+ |

```
[RBO 규칙 우선순위 예시]
 1. Single Row by ROWID (최우선)
 2. Single Row by Unique Index
 ...
15. Full Table Scan (최후순위)
→ 데이터 분포 무관, 항상 동일 경로 선택
```

- CBO는 통계정보 정확도에 성능이 의존하므로, 통계 관리가 필수이다.

### 3. 다. 힌트(HINT)와 옵티마이저 오판 제어

| 힌트 분류 | 예시 | 용도 |
|---------|------|------|
| 접근 방법 | FULL, INDEX, INDEX_FFS | 테이블/인덱스 접근 방식 강제 지정 |
| 조인 순서 | LEADING, ORDERED | 조인 구동 테이블(Driving Table) 결정 |
| 조인 방법 | USE_NL, USE_HASH, USE_MERGE | Nested Loop/Hash/Sort Merge Join 강제 |
| 병렬 처리 | PARALLEL, NO_PARALLEL | 병렬 쿼리 활성화/비활성화 |

| 오판 상황 | 원인 | 힌트 해결 |
|---------|------|---------|
| 작은 테이블에 Index Scan | 오래된 통계 (실제 행 수 많음) | FULL 힌트 또는 통계 갱신 |
| 풀 스캔이 더 빠른데 Index 사용 | 잘못된 카디널리티 추정 | FULL 힌트 |
| 나쁜 조인 순서 | 선행 테이블 잘못 선택 | LEADING 힌트 |
| Hash Join 대신 NL Join 선택 | 대용량 집합 조인 | USE_HASH 힌트 |

머신러닝 기반 학습형 옵티마이저(Oracle Adaptive Query Optimization, PostgreSQL AI Planner 연구 등)가 등장하여 카디널리티 추정 정확도를 극적으로 향상시키는 방향으로 발전 중이다. 힌트는 통계 갱신, 인덱스 재설계가 불가능한 긴급 상황에서의 최후 수단으로 사용하고, SQL Plan Management(SPM)나 Query Store로 실행 계획을 안정화하는 것이 바람직하다.  "끝"

### 실무 제언

**통계정보 관리의 중요성**
- **챌린지**: CBO는 통계정보가 부정확하면 심각하게 잘못된 실행 계획을 생성한다. 대량 INSERT/DELETE/UPDATE 후 통계 미갱신, 데이터 분포가 편향된 컬럼의 히스토그램 미수집이 대표적 문제이며, 운영 중 갑자기 쿼리 성능이 저하되는 원인의 상당수가 여기에 해당한다.
- **제언**: Oracle `DBMS_STATS.GATHER_TABLE_STATS`, MySQL `ANALYZE TABLE`, PostgreSQL `VACUUM ANALYZE`를 주기적으로 실행하고, 야간 배치 작업 직후 자동 통계 수집 스케줄을 구성한다. 히스토그램은 데이터 분포가 편향된 컬럼에 반드시 수집해야 한다.

**힌트 남용 방지**
- **챌린지**: 힌트는 강력하지만 데이터 특성이 변화할 때 힌트 고정 실행 계획이 오히려 성능을 더 저하시킬 수 있다. 힌트에 의존한 코드가 많으면 데이터 변화에 취약한 시스템이 된다.
- **제언**: 힌트는 통계 갱신, 인덱스 재설계, 쿼리 리팩터링이 불가능한 긴급 상황에서의 최후 수단으로 사용하고, SQL Plan Management(SPM, Oracle) 또는 Query Store(SQL Server) 등 실행 계획 안정화 기능을 활용한다.

**카디널리티 추정 오류 진단**
- **챌린지**: 옵티마이저가 잘못된 조인 순서나 접근 방법을 선택하면 수천 배의 성능 저하가 발생하지만, 원인이 카디널리티 추정 오류인지 바로 판단하기 어렵다.
- **제언**: 실행 계획(EXPLAIN PLAN)에서 Estimated Rows와 Actual Rows의 차이를 확인하여 카디널리티 추정 오류를 조기 진단하고, 히스토그램 수집과 통계 갱신으로 근본 원인을 해결한다.
