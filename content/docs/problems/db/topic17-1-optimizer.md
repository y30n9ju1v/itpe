## 해설

> **SQL을 가장 효율적인 실행 계획으로 변환하는 DBMS의 두뇌**  
> RBO(규칙 기반)와 CBO(비용 기반) 옵티마이저, 실행계획(Execution Plan), 통계정보, 힌트(HINT)를 통한 실행 제어

옵티마이저(Optimizer)는 개발자가 작성한 선언형 SQL(무엇을 가져와라)을 DBMS 내부의 절차형 실행 계획(어떻게 가져와라)으로 변환하는 핵심 엔진입니다. 같은 SELECT 문도 옵티마이저가 어떤 경로를 선택하느냐에 따라 수천 배의 성능 차이가 납니다. 옵티마이저의 선택이 잘못되면 동일한 SQL이 수천 배의 성능 차이를 낼 수 있으므로, 그 원리와 제어 방법을 이해하는 것이 SQL 튜닝의 출발점입니다.

### 1. 출제 배경 및 의도

옵티마이저는 DBA와 데이터 아키텍트가 반드시 이해해야 할 DBMS 핵심 컴포넌트입니다. RBO와 CBO의 차이, CBO의 통계정보 의존성, 힌트를 통한 개발자 개입 방법을 체계적으로 설명할 수 있는지를 검증합니다. 최근에는 머신러닝 기반 학습형 옵티마이저(Oracle ML-based Cardinality Estimation 등)도 등장하여, 전통적 원리를 바탕으로 신기술 트렌드까지 연계하면 고득점이 가능합니다.

### 2. 개념의 본질과 작동 메커니즘

옵티마이저란 SQL 파싱 후 가능한 실행 방법(후보 플랜) 중에서 최소 비용(응답 시간, I/O, CPU)의 실행 계획을 선택하는 DBMS 내부 소프트웨어 모듈입니다.

- **Input:** SQL 문장, 테이블/인덱스 통계정보, 시스템 자원 정보
- **Mechanism:** SQL 파싱 → 논리적 최적화(쿼리 변환) → 물리적 최적화(비용 계산 및 플랜 선택)
- **Output:** 실행 계획(Execution Plan) - 테이블 접근 방법, 조인 순서/방법, 정렬 방법 등

```text
[옵티마이저 처리 흐름]

  SQL 문장 입력
       │
       ▼
  ┌──────────────┐
  │  SQL 파서    │ ← 문법 검사, 파스 트리 생성
  └──────┬───────┘
         │
         ▼
  ┌──────────────────────────────────────┐
  │         옵티마이저 (Optimizer)         │
  │  ┌────────────────────────────────┐  │
  │  │ 1. 쿼리 변환 (Query Transform) │  │ ← 뷰 머징, 서브쿼리 최적화
  │  │ 2. 비용 추정 (Cost Estimation)  │  │ ← 통계정보 활용
  │  │ 3. 플랜 선택 (Plan Selection)  │  │ ← 최저 비용 플랜 선택
  │  └────────────────────────────────┘  │
  └──────────────────────────────────────┘
         │
         ▼
  실행 계획 (Execution Plan)
  → Row Source Generation → 실행 엔진
```

#### 가. 옵티마이저의 개념 및 역할

**주요 역할**
- **쿼리 변환:** 논리적으로 동일한 쿼리를 더 효율적인 형태로 변환 (서브쿼리 → 조인, 뷰 머징 등)
- **비용 추정:** 각 실행 방법의 I/O, CPU, 메모리 비용을 통계정보 기반으로 계산
- **플랜 선택:** 최소 비용의 실행 계획 선택 (인덱스 스캔 vs 풀 스캔, 조인 방법 등)
- **카디널리티 추정:** 각 연산의 결과 행 수 추정 → 조인 순서 결정에 핵심

#### 나. 옵티마이저의 유형

```text
[RBO vs CBO 비교]

  RBO (Rule-Based Optimizer)          CBO (Cost-Based Optimizer)
  ─────────────────────────          ──────────────────────────
  규칙 테이블 (우선순위 1~15)         통계정보 기반 비용 계산
  
  우선순위 예:                        비용 = f(Cardinality, NDV,
  1. Single Row by ROWID              데이터 분포, 블록 수, ...)
  2. Single Row by Unique Index
  ...                                 통계 정보 필요:
  15. Full Table Scan                 - 행 수(Row Count)
                                      - 블록 수(Block Count)
  특징:                               - 컬럼 NDV
  - 통계 불필요, 예측 가능            - 히스토그램(데이터 분포)
  - 데이터 특성 무시                  - 클러스터링 팩터
  - Oracle 10g 이후 지원 중단
```

| 비교 항목 | RBO | CBO |
|---------|-----|-----|
| **기반** | 사전 정의 규칙(우선순위) | 비용 추정(통계정보) |
| **통계정보** | 불필요 | 필수 (정확도 의존) |
| **데이터 특성 반영** | 불가 | 가능 |
| **예측 가능성** | 높음 | 통계 정확도에 의존 |
| **현재 상태** | 대부분 DBMS 지원 중단 | 현대 모든 DBMS 기본 |
| **대표 DBMS** | 구 Oracle 9i 이하 (10g부터 지원 중단) | Oracle 10g+, PostgreSQL, MySQL 8.0+ |

#### 다. 힌트 (HINT)

힌트(HINT)란 SQL에 주석 형태(`/*+ ... */`)로 삽입하여 옵티마이저의 실행 계획 선택을 개발자가 직접 제어하는 지시자입니다.

```sql
-- Oracle 힌트 예시
SELECT /*+ INDEX(emp emp_dept_idx) LEADING(dept) USE_NL(emp) */
       emp.name, dept.name
FROM   emp, dept
WHERE  emp.dept_id = dept.id
AND    emp.salary > 5000;

-- 힌트 종류 예시
/*+ FULL(테이블명) */          -- 풀 테이블 스캔 강제
/*+ INDEX(테이블 인덱스명) */  -- 특정 인덱스 사용 강제
/*+ NO_INDEX(테이블 인덱스) */ -- 특정 인덱스 사용 금지
/*+ LEADING(테이블1 테이블2)*/ -- 조인 순서 지정
/*+ USE_NL(테이블) */          -- Nested Loop Join 강제
/*+ USE_HASH(테이블) */        -- Hash Join 강제
/*+ PARALLEL(테이블 4) */      -- 4개 병렬 처리
```

| 힌트 분류 | 예시 | 용도 |
|---------|------|------|
| **접근 방법** | FULL, INDEX, INDEX_FFS | 테이블/인덱스 접근 방식 지정 |
| **조인 순서** | LEADING, ORDERED | 조인 구동 테이블 결정 |
| **조인 방법** | USE_NL, USE_HASH, USE_MERGE | Nested Loop/Hash/Sort Merge |
| **병렬처리** | PARALLEL, NO_PARALLEL | 병렬 쿼리 활성화 |
| **통계 무시** | OPT_PARAM | 파라미터 일시 변경 |

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 개념 | 실행 계획(Execution Plan), 쿼리 변환(Query Transformation), 비용 추정(Cost Estimation) |
| 유형 | RBO(규칙 기반), CBO(비용 기반), 통계정보(Statistics), 히스토그램(Histogram) |
| 비용 요소 | 카디널리티(Cardinality), NDV, 클러스터링 팩터(Clustering Factor) |

### 4. 맥락과 비교

**옵티마이저 오판 시나리오와 힌트 개입**

| 오판 상황 | 원인 | 힌트 해결 |
|---------|------|---------|
| 작은 테이블에 Index Scan | 오래된 통계 (실제 행 수 많음) | FULL 힌트 또는 통계 갱신 |
| 풀 스캔이 더 빠른데 Index 사용 | 잘못된 카디널리티 추정 | FULL 힌트 |
| 나쁜 조인 순서 | 선행 테이블 잘못 선택 | LEADING 힌트 |
| Hash Join 대신 NL Join 선택 | 대용량 집합 조인 | USE_HASH 힌트 |

### 5. 실무 제언

**통계정보 관리의 중요성**

- **챌린지:** CBO는 통계정보가 부정확하면 심각하게 잘못된 실행 계획을 생성합니다. 대량 INSERT/DELETE/UPDATE 후 통계 미갱신, 데이터 분포가 편향된 컬럼의 히스토그램 미수집이 대표적인 문제입니다. 운영 중 갑자기 쿼리 성능이 저하되는 원인의 상당수가 통계정보 문제입니다.
- **제언:** Oracle `DBMS_STATS.GATHER_TABLE_STATS`, MySQL `ANALYZE TABLE`, PostgreSQL `VACUUM ANALYZE`를 주기적으로 실행하고, 데이터 분포가 크게 변하는 야간 배치 작업 직후 자동 통계 수집 스케줄을 구성합니다. 히스토그램은 데이터 분포가 편향된 컬럼에 반드시 수집해야 합니다.

**힌트 남용 방지**

- **챌린지:** 힌트는 강력하지만 데이터 특성이 변화할 때 힌트 고정 실행 계획이 오히려 성능을 더 저하시킬 수 있습니다. 힌트는 통계정보 기반 CBO를 무력화하므로, 힌트에 의존한 코드가 많으면 데이터 변화에 취약한 시스템이 됩니다.
- **제언:** 힌트는 통계 갱신, 인덱스 재설계, 쿼리 리팩터링이 불가능한 긴급 상황에서의 최후 수단으로 사용하고, 근본 원인(통계 부정확, 인덱스 누락)을 별도 해결합니다. SQL Plan Management(SPM, Oracle) 또는 Query Store(SQL Server) 등 실행 계획 안정화 기능을 활용하는 것이 더 바람직합니다.
