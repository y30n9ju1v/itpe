## 해설

> **DB 성능 문제를 체계적으로 진단하고 개선하는 3단계 엔지니어링 프로세스**  
> 조사(보색)→설계(이행)→평가 3단계와 설계/Application/Server/System 4계층 튜닝 절차

데이터베이스 튜닝은 단순한 SQL 개선을 넘어, 성능 문제의 근본 원인을 계층적으로 파악하고 체계적으로 해결하는 엔지니어링 프로세스입니다. 문제를 먼저 정확히 진단(보색)하고, 최적 해결책을 설계(이행)한 후, 효과를 검증(평가)하는 3단계 접근이 실패 없는 튜닝의 기본입니다. 병목(Bottleneck)이란 시스템 전체 성능을 가장 크게 제한하는 지점을 뜻하며, 튜닝은 이를 찾아 제거하는 작업입니다.

### 1. 출제 배경 및 의도

DB 성능 문제는 모든 운영 시스템에서 반드시 직면하는 현실적 과제입니다. 이 문제는 튜닝의 방법론적 절차(3단계)와 실제 튜닝 대상 계층(4단계)을 체계적으로 서술할 수 있는지를 검증합니다. "보색, 이행, 평가"라는 한국어 용어와 영문 대응을 명확히 하고, 설계→Application→Server→System 순서의 논리적 근거를 설명하는 것이 핵심입니다.

### 2. 개념의 본질과 작동 메커니즘

DB 튜닝이란 데이터베이스 시스템의 성능 병목(Bottleneck)을 식별하고, 최소 비용으로 최대 성능 향상을 달성하기 위한 체계적인 최적화 활동입니다.

- **Input:** 성능 저하 현상(느린 쿼리, 높은 CPU/I/O 사용률), SLA 위반 지표
- **Mechanism:** 원인 진단 → 계층별 최적화(설계→App→Server→System) → 효과 검증 반복
- **Output:** 쿼리 응답 시간 단축, 처리량(Throughput) 향상, 자원 사용률 정상화

```text
[DB 튜닝 전체 프로세스]

  성능 문제 감지
       │
       ▼
  ┌─────────────────────────────────────────────────┐
  │            튜닝 3단계                            │
  │  1단계: 보색(조사)   → 문제 식별 및 원인 분석    │
  │  2단계: 이행(설계)   → 튜닝 방안 설계 및 적용    │
  │  3단계: 평가         → 효과 측정 및 검증         │
  └──────────────────────────┬──────────────────────┘
                             │ (반복)
                             ▼
  ┌─────────────────────────────────────────────────┐
  │            튜닝 4계층 (우선순위 순)              │
  │  1. 설계 튜닝 ─── 가장 효과 큼, 재작업 비용 큼   │
  │  2. Application 튜닝 ── SQL 최적화 중심          │
  │  3. Server 튜닝 ─── DBMS 파라미터 조정           │
  │  4. System 튜닝 ─── H/W, OS, 네트워크            │
  └─────────────────────────────────────────────────┘
```

#### 가. 튜닝 3단계

**1단계: 보색 (조사, Identification)**

성능 문제를 정확히 측정하고 병목의 근본 원인을 파악합니다.

| 조사 항목 | 도구 및 방법 | 파악 내용 |
|---------|-----------|---------|
| 슬로우 쿼리 분석 | Slow Query Log, AWR, ASH | 실행 시간 상위 쿼리 |
| 대기 이벤트 분석 | Oracle: V$SESSION_WAIT | CPU Wait, I/O Wait, 락 대기 |
| 실행 계획 확인 | EXPLAIN PLAN, EXPLAIN ANALYZE | Full Scan, 나쁜 조인 순서 |
| 자원 사용률 | OS 모니터링, DB 메모리 뷰 | CPU, I/O, 메모리, 네트워크 |
| 인덱스 사용 현황 | V$INDEX_USAGE, pg_stat_user_indexes | 미사용/중복 인덱스 |

**2단계: 이행 (설계 및 적용, Implementation)**

진단 결과를 바탕으로 계층별 튜닝 방안을 설계하고 적용합니다.

**3단계: 평가 (Evaluation)**

튜닝 적용 전후의 지표를 비교하여 효과를 검증하고, 목표 미달 시 1단계부터 반복합니다.

| 평가 지표 | 설명 |
|---------|------|
| **응답 시간 (Response Time)** | 개별 쿼리/트랜잭션 처리 시간 |
| **처리량 (Throughput)** | 단위 시간당 처리 트랜잭션 수(TPS) |
| **자원 사용률** | CPU, I/O, 메모리 사용률 정상화 |
| **SLA 충족 여부** | 서비스 레벨 목표 달성 확인 |

#### 나. 4계층 튜닝 절차

**계층 1: 설계 튜닝 (Design Tuning) - 가장 우선**

```text
[설계 튜닝 항목]
 ├── 정규화/반정규화 조정
 │    ├── 과도한 정규화로 조인 증가 → 반정규화(컬럼 중복, 테이블 합병)
 │    └── 반정규화 과다 → 데이터 불일치 → 재정규화
 ├── 파티셔닝
 │    └── 대용량 테이블 → Range/Hash 파티셔닝으로 I/O 범위 축소
 ├── 인덱스 설계
 │    ├── 누락 인덱스 추가
 │    ├── 불필요 인덱스 제거
 │    └── 복합 인덱스 컬럼 순서 최적화
 └── 클러스터링(Cluster) / 클러스터드 인덱스
      └── 자주 함께 조회되는 테이블을 물리적으로 가까이 저장
```

**계층 2: Application 튜닝 (SQL/App 레벨)**

| 튜닝 항목 | 문제 | 해결 방법 |
|---------|------|---------|
| **SQL 재작성** | 비효율적 SQL, 서브쿼리 과다 | EXISTS→JOIN 변환, 인라인 뷰 활용 |
| **인덱스 활용** | 인덱스 컬럼 함수 적용 | 함수 기반 인덱스, 쿼리 변환 |
| **페이징 처리** | 대량 데이터 전체 SELECT | ROWNUM/LIMIT 기반 페이지 단위 처리 |
| **배치 처리** | 실시간 대량 DML | 야간 배치, 부분 커밋(Array Processing) |
| **커넥션 풀링** | 커넥션 과다 생성/소멸 | Connection Pool 설정 최적화 |
| **바인드 변수** | 동적 SQL로 Hard Parse 반복 | PreparedStatement, 바인드 변수 사용 |

**계층 3: Server 튜닝 (DBMS 파라미터)**

```text
[Server 튜닝 주요 파라미터]
 ├── 메모리 튜닝
 │    ├── Buffer Pool/Cache 크기 (InnoDB: innodb_buffer_pool_size)
 │    ├── Shared Pool (Oracle: 실행 계획 캐시)
 │    └── Sort Area, Hash Area 크기
 ├── I/O 튜닝
 │    ├── 체크포인트 빈도 조정
 │    └── 비동기 I/O 활성화
 └── 병렬 처리
      └── 병렬 쿼리 최대 세션 수
```

**계층 4: System 튜닝 (H/W, OS, 네트워크)**

- 디스크: HDD → SSD/NVMe 교체, RAID 구성 최적화
- 메모리: 물리 메모리 증설, NUMA 아키텍처 고려
- OS: Kernel 파라미터(vm.swappiness, I/O 스케줄러), Huge Pages 설정
- 네트워크: DB 서버와 AP 서버 간 네트워크 대역폭, 레이턴시 감소

### 3. 핵심 키워드 (Must-Have)

| 분류 | 키워드 |
|------|--------|
| 3단계 | 보색(조사/Identification), 이행(설계/Implementation), 평가(Evaluation) |
| 4계층 | 설계 튜닝, Application 튜닝, Server 튜닝, System 튜닝 |
| SQL 튜닝 | 실행 계획, Full Scan 제거, 인덱스 활용, 바인드 변수, Hard/Soft Parse |
| 측정 지표 | 응답 시간(Response Time), 처리량(TPS), 대기 이벤트, 슬로우 쿼리 |

### 4. 맥락과 비교

| 계층 | 효과 크기 | 적용 비용 | 재작업 비용 | 우선 적용 시기 |
|------|---------|---------|-----------|------------|
| 설계 튜닝 | 매우 큼 | 큼 | 매우 큼 | 설계/개발 초기 |
| Application 튜닝 | 큼 | 중간 | 중간 | 개발/운영 |
| Server 튜닝 | 중간 | 낮음 | 낮음 | 운영 |
| System 튜닝 | 낮음~중간 | 매우 큼 | 낮음 | 최후 수단 |

### 5. 실무 제언

**튜닝 우선순위 원칙: 설계→SQL→Server→System**

- **챌린지:** 성능 문제가 발생하면 즉각적인 처방으로 서버 증설(Scale-Up)이나 DBMS 파라미터 조정을 시도하는 경우가 많습니다. 그러나 근본 원인이 잘못된 인덱스 설계나 비효율적 SQL에 있다면, 서버 증설은 비용만 증가하고 문제는 반복됩니다.
- **제언:** 튜닝은 반드시 설계→Application→Server→System 순서로 접근합니다. Slow Query Log와 실행 계획 분석으로 Top 5 문제 쿼리를 먼저 해결하면 대부분의 성능 문제가 해소됩니다. 서버 증설은 설계/SQL 튜닝이 충분히 완료된 후의 용량 확장 수단으로만 활용합니다.

**성능 기준선(Baseline) 수립의 중요성**

- **챌린지:** 튜닝 전후 비교가 없으면 "얼마나 좋아졌는가"를 증명할 수 없고, 오히려 튜닝이 다른 부분의 성능을 저하시켰을 때 인지하지 못하는 리스크가 있습니다.
- **제언:** 튜닝 착수 전 반드시 현재 TPS, 평균 응답 시간, 최대 응답 시간, 자원 사용률 등의 기준선(Baseline)을 측정하고 문서화합니다. 튜닝 후 동일 조건에서 지표를 재측정하여 정량적 개선 효과를 증명하는 것이 프로세스의 완결입니다.
