## 해설

### 출제 배경 및 의도

DB 성능 문제는 모든 운영 시스템에서 반드시 직면하는 현실적 과제입니다. 서비스 장애의 상당수가 잘못된 SQL 또는 부적절한 인덱스 설계에서 기인하며, 운영 중 성능 저하를 체계적으로 해결하는 방법론은 DBA·아키텍트의 핵심 역량입니다.

이 문제는 튜닝의 방법론적 절차(3단계: 보색·이행·평가)와 실제 튜닝 대상 계층(4계층: 설계→Application→Server→System)을 체계적으로 서술할 수 있는지를 검증합니다. "보색, 이행, 평가"라는 한국어 용어와 영문 대응을 명확히 하고, 계층별 튜닝의 논리적 우선순위 근거까지 설명하면 고득점이 가능합니다.

### 1. DB 튜닝, 개요

정 의  • DB 성능 병목(Bottleneck)을 체계적으로 진단·제거하여 최소 비용으로 최대 성능을 달성하는 최적화 활동
       - 보색(문제 조사)→이행(방안 적용)→평가(효과 검증)의 반복적 사이클로 수행

```
  성능 문제 감지 (SLA 위반, 슬로우 쿼리)
       │
       ▼
  ┌─────────────────────────────────────────────┐
  │              튜닝 3단계                      │
  │  ① 보색(조사)   → 병목 식별 및 원인 분석    │
  │  ② 이행(설계)   → 튜닝 방안 설계 및 적용    │
  │  ③ 평가         → 효과 측정 및 검증(반복)   │
  └──────────────────────┬──────────────────────┘
                         │
                         ▼
  ┌─────────────────────────────────────────────┐
  │        튜닝 4계층 (우선순위 순)              │
  │  1. 설계 튜닝    ── 효과 최대, 재작업 비용 큼│
  │  2. Application  ── SQL·인덱스 최적화        │
  │  3. Server 튜닝  ── DBMS 파라미터 조정       │
  │  4. System 튜닝  ── H/W·OS·네트워크          │
  └─────────────────────────────────────────────┘
```

- 튜닝은 반드시 설계→Application→Server→System 순으로 접근해야 하며, 서버 증설은 모든 소프트웨어 튜닝 완료 후 최후 수단으로 활용

### 2. 가. 튜닝 3단계

1) 1단계: 보색(조사, Identification)

| 구분 | 항목 | 설명 |
|------|------|------|
| 슬로우 쿼리 분석 | 도구 | Slow Query Log, AWR, ASH |
| 슬로우 쿼리 분석 | 파악 내용 | 실행 시간 상위 쿼리 목록 |
| 대기 이벤트 분석 | 도구 | Oracle V$SESSION_WAIT, pg_stat_activity |
| 대기 이벤트 분석 | 파악 내용 | CPU Wait, I/O Wait, Lock 대기 |
| 실행 계획 확인 | 도구 | EXPLAIN PLAN, EXPLAIN ANALYZE |
| 실행 계획 확인 | 파악 내용 | Full Table Scan, 비효율 조인 순서 |
| 자원 사용률 | 도구 | OS 모니터링, DB 메모리 뷰 |
| 자원 사용률 | 파악 내용 | CPU·I/O·메모리·네트워크 병목 |

- 보색 단계에서 병목의 근본 원인을 정확히 파악해야 이행 단계의 방향이 결정됨

2) 2단계: 이행(설계 및 적용, Implementation)

| 구분 | 항목 | 설명 |
|------|------|------|
| 설계 튜닝 | 정규화/반정규화 | 조인 과다 시 반정규화, 데이터 불일치 시 재정규화 |
| 설계 튜닝 | 인덱스 재설계 | 누락 인덱스 추가, 불필요 인덱스 제거, 복합 인덱스 순서 최적화 |
| Application 튜닝 | SQL 재작성 | EXISTS→JOIN 변환, 서브쿼리 제거, 바인드 변수 사용 |
| Application 튜닝 | 페이징·배치 | ROWNUM/LIMIT 페이징, 야간 배치, Array Processing |
| Server 튜닝 | 메모리 파라미터 | innodb_buffer_pool_size, SGA/PGA 크기 조정 |
| Server 튜닝 | I/O 파라미터 | 체크포인트 빈도, 비동기 I/O 활성화 |
| System 튜닝 | H/W 개선 | HDD→SSD/NVMe 교체, 물리 메모리 증설 |
| System 튜닝 | OS 최적화 | vm.swappiness, I/O 스케줄러, Huge Pages |

- 이행은 진단 결과에 기반하여 가장 효과가 큰 계층부터 순차 적용

3) 3단계: 평가(Evaluation)

| 구분 | 항목 | 설명 |
|------|------|------|
| 응답 시간 | Response Time | 개별 쿼리/트랜잭션 처리 시간 단축 여부 |
| 처리량 | TPS(Throughput) | 단위 시간당 트랜잭션 수 향상 여부 |
| 자원 사용률 | CPU·I/O·메모리 | 사용률 정상 범위(70% 이하) 복귀 여부 |
| SLA 충족 | 서비스 레벨 목표 | 목표 미달 시 1단계부터 반복 |

- 평가는 튜닝 전 기준선(Baseline)과의 정량 비교로 수행해야 하며, 목표 미달 시 보색→이행→평가 사이클 반복

### 3. 나. 4계층 튜닝 절차와 우선순위

| 구분 | 항목 | 설명 |
|------|------|------|
| 설계 튜닝 | 효과 크기 | 매우 큼 (구조적 변화) |
| 설계 튜닝 | 적용 시기 | 설계/개발 초기 우선 |
| 설계 튜닝 | 재작업 비용 | 매우 큼 (스키마 변경) |
| Application 튜닝 | 효과 크기 | 큼 (SQL·인덱스) |
| Application 튜닝 | 적용 시기 | 개발/운영 단계 |
| Application 튜닝 | 재작업 비용 | 중간 |
| Server 튜닝 | 효과 크기 | 중간 (파라미터 조정) |
| Server 튜닝 | 적용 시기 | 운영 단계 |
| Server 튜닝 | 재작업 비용 | 낮음 |
| System 튜닝 | 효과 크기 | 낮음~중간 (H/W 교체) |
| System 튜닝 | 적용 시기 | 최후 수단 |
| System 튜닝 | 재작업 비용 | 매우 큼 (H/W 투자) |

- 튜닝 계층은 효과 대비 비용이 높은 순서로 적용하며, System 튜닝(H/W 증설)은 소프트웨어 튜닝이 충분히 완료된 후 용량 확장 수단으로만 활용해야 함.  "끝"

### 실무 제언

**튜닝 우선순위 원칙: 설계→SQL→Server→System**
- **챌린지**: 성능 문제 발생 시 서버 증설(Scale-Up)이나 DBMS 파라미터 조정을 먼저 시도하는 경향이 있으나, 근본 원인이 잘못된 설계·SQL에 있으면 비용만 증가하고 문제는 반복됨
- **제언**: Slow Query Log와 실행 계획 분석으로 Top 5 문제 쿼리를 먼저 해결하면 대부분의 성능 문제가 해소되며, 서버 증설은 설계/SQL 튜닝 완료 후 최후 수단으로만 활용

**성능 기준선(Baseline) 수립 필수**
- **챌린지**: 튜닝 전후 비교 지표가 없으면 개선 효과를 정량적으로 증명할 수 없고, 부작용 발생 시 인지하지 못하는 리스크 존재
- **제언**: 튜닝 착수 전 TPS, 평균/최대 응답 시간, CPU/I/O/메모리 사용률을 기준선(Baseline)으로 측정·문서화하고, 튜닝 후 동일 조건에서 재측정하여 정량적 개선 효과 증명

**바인드 변수 미사용 안티패턴 제거**
- **챌린지**: 동적 SQL 생성 시 리터럴 값이 쿼리에 직접 포함되면 Hard Parse가 반복 발생하여 Shared Pool 경합과 CPU 과부하를 유발함
- **제언**: PreparedStatement·바인드 변수 사용을 코딩 표준으로 강제하고, Slow Query Log 분석 시 Hard Parse 비율을 주요 지표로 모니터링
