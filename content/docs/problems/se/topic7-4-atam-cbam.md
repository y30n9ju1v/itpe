## 해설

### 출제 배경 및 의도

ATAM(Architecture Tradeoff Analysis Method)과 CBAM(Cost Benefit Analysis Method)은 CMU SEI가 개발한 아키텍처 평가 방법론으로, 대규모 미션크리티컬 시스템(국방·항공·공공 인프라)의 아키텍처를 개발 초기에 검증하기 위해 만들어졌다.

기술사 시험에서 반복 출제되는 이유는, 아키텍처 결정이 시스템 생애주기 전반의 품질과 비용에 지배적인 영향을 미치기 때문이다. 채점관은 ATAM의 9단계 프로세스와 유틸리티 트리, 4가지 분석 결과(감도점·트레이드오프점·위험점·비위험점), 그리고 CBAM이 ATAM의 경제적 확장이라는 관계를 논리적으로 설명하는 능력을 검증한다.

### 1. SA 평가 모델 개요

정 의  • 소프트웨어 아키텍처가 품질속성 요구사항을 충족하는지 체계적으로 검증하는
       방법론 집합 (CMU SEI)
       - 시나리오 기반 평가가 핵심: 이해관계자의 품질 요구를 시나리오로 표현

```
[SA 평가 모델 분류 체계]

  소프트웨어 아키텍처(SA) 평가 방법
               │
  ┌────────────┴──────────────────┐
  │                               │
  질문지 기반                  시나리오 기반
  (Questionnaire)              (Scenario-Based)
                                   │
                      ┌────────────┴──────────────┐
                      │                           │
                    SAAM                      ATAM / CBAM
            (단일 품질속성,              (다중 품질속성 트레이드오프,
             변경용이성 중심)             CBAM: 경제적 ROI 추가)
```

- SA 평가는 설계 완료 후 구현 전에 수행하여 변경 비용을 최소화하는 것이 핵심이다.

### 2. ATAM 및 CBAM 상세

1) ATAM (Architecture Tradeoff Analysis Method)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 아키텍처가 복수의 품질속성 요구사항을 충족하는지 평가하고 트레이드오프를 식별하는 방법론 |
| 주체 | 구성 | 외부 평가팀(3~5명) + 아키텍트 + 이해관계자 |
| 산출물 | 4종 | 위험점(Risks), 비위험점(Non-Risks), 감도점(Sensitivity Points), 트레이드오프점(Tradeoff Points) |
| 규모 | 소요 | 2~3일 공식 워크샵, 대규모 미션크리티컬 시스템 |

```
[ATAM 9단계 프로세스]

  1단계: ATAM 발표         ← 평가 방법을 이해관계자에게 설명
  2단계: 비즈니스 동인 발표  ← 시스템 핵심 목적·제약 공유
  3단계: 아키텍처 발표      ← 아키텍트가 현재 아키텍처 설명
  4단계: 아키텍처 접근법 식별← 사용된 패턴/전술 식별
               │
               ▼
  5단계: 유틸리티 트리 생성  ← 품질속성을 계층적으로 구조화
               │
               ▼
  6단계: 아키텍처 접근법 분석← QAS 만족/위반 분석
                              → 감도점·트레이드오프점·위험점·비위험점 식별
               │
               ▼
  7단계: 시나리오 브레인스토밍← 이해관계자 추가 시나리오 제안
  8단계: 시나리오 우선순위화 ← 투표로 중요 시나리오 선정
  9단계: 아키텍처 접근법 재분석← 6단계를 신규 시나리오에 반복
               │
               ▼
  결과물: 감도점·트레이드오프점·위험점 목록, 위험 테마
```

| 분류 | 정의 | 예시 |
|------|------|------|
| 감도점(Sensitivity Point) | 특정 결정이 하나의 품질속성에 강하게 영향 | 캐시 크기 → 성능에만 영향 |
| 트레이드오프점(Tradeoff Point) | 하나의 결정이 2개 이상 품질속성에 상반 영향 | 암호화 강도↑ → 보안↑ but 성능↓ |
| 위험점(Risk) | 아직 해결되지 않은 잠재적 아키텍처 문제 | 단일 DB 서버 → 가용성 위험 |
| 비위험점(Non-Risk) | 분석 후 위험이 없다고 확인된 결정 | 읽기 전용 캐시 → 성능·일관성 양호 |

- ATAM은 유틸리티 트리로 QAS 우선순위를 명확히 하고, 트레이드오프점 식별로 아키텍처 의사결정을 객관화한다.

2) CBAM (Cost Benefit Analysis Method)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | ATAM을 경제적 관점으로 확장하여 아키텍처 전략의 비용 대비 효익(ROI)을 정량화 |
| 공식 | ROI | ROI = 기대 편익의 총 가중 효용 / 비용 |
| 핵심 | 도구 | 효용 함수(Utility Function): 품질속성 수준별 이해관계자 만족도를 수치화 |
| 결과 | 산출 | ROI 기준 아키텍처 전략 우선순위, 최적 전략 선택 근거 |

```
[CBAM 프로세스]

  ATAM 결과 (시나리오 + 아키텍처 접근법)
               │
  1단계: 시나리오 우선순위 재확인
               │
  2단계: 효용 함수 적용 (현재/이상/최악 수준 비교)
               │
  3단계: 아키텍처 전략별 품질속성 응답값 산정
         전략 A: 성능 +30%, 비용 $50K
         전략 B: 성능 +10%, 가용성 +20%, 비용 $30K
               │
  4단계: 전략별 비용 추정
               │
  5단계: ROI 계산 (기대 편익 가중 효용 / 비용)
               │
  6단계: ROI 기준 최적 전략 선택 및 문서화
```

- CBAM의 효용 함수는 비선형적으로, 2초→1초 응답 개선이 5초→4초보다 훨씬 높은 가치를 가진다.

### 3. SA 평가 모델 비교 및 트렌드

1) SAAM / ATAM / CBAM 종합 비교

| 구분 | SAAM | ATAM | CBAM |
|------|------|------|------|
| 개발 | CMU SEI | CMU SEI | CMU SEI |
| 초점 | 변경용이성 단일 속성 | 다중 QA 트레이드오프 | ATAM + 경제적 ROI |
| 방법 | 시나리오 기반 | 시나리오 + 유틸리티 트리 | 시나리오 + 효용 함수 + 비용 |
| 결과물 | 수정/직접 시나리오 분류 | 위험점·트레이드오프점·감도점 | ROI 순위, 최적 전략 |
| 규모 | 소규모 | 대규모 (2~3일) | ATAM 후속 경제 분석 |
| 정량화 | 어려움 | 상대적 우선순위 | ROI 완전 정량화 |

- SAAM → ATAM → CBAM 순으로 평가의 정교함과 비용이 증가하며, 프로젝트 규모와 중요도에 따라 적합한 방법론을 선택해야 한다.  "끝"

### 실무 제언

**ATAM의 현실적 적용: 경량화 필요성**
- **챌린지**: 완전한 ATAM 수행에는 외부 평가팀, 2~3일 워크샵, 수십 개 시나리오 분석 등 상당한 비용이 필요하다. 애자일 환경에서 스프린트 단위로 빠르게 진행되는 아키텍처 결정에 전통적 ATAM을 그대로 적용하는 것은 비현실적이다.
- **제언**: 경량 ATAM(Lightweight ATAM)을 적용하여 유틸리티 트리로 QAS 우선순위를 명확히 하고, 주요 결정에 대해 트레이드오프점 3~5개만 집중 식별한다. ADR(Architecture Decision Record)에 트레이드오프를 기록하는 것이 현실적인 ATAM 정신의 계승이다.

**CBAM의 재무적 효익 측정**
- **챌린지**: CBAM의 핵심인 효익(Benefit) 정량화는 실무에서 매우 어렵다. "가용성이 99.9% → 99.99%로 개선되면 비즈니스 가치는 얼마인가?"를 정확히 산정하는 방법이 없는 경우가 대부분이다.
- **제언**: SLA 위반 패널티, 다운타임으로 인한 매출 손실, 보안 사고 예상 손해액 등 리스크 기반 비용을 효익의 대리 지표로 활용한다. "연간 예상 다운타임 비용 = 다운타임 시간 × 시간당 매출" 공식으로 가용성 개선의 경제적 가치를 산정하면 CBAM 의사결정의 객관성을 높일 수 있다.
