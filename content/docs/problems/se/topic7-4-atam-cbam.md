## 해설

> **아키텍처를 '고르는' 것이 아니라 '검증하는' 방법론**  
> ATAM은 트레이드오프를, CBAM은 ROI를 기준으로 아키텍처를 평가한다

좋은 아키텍처를 설계했다고 주장하는 것은 쉽습니다. 하지만 "이 아키텍처가 성능 요구를 충족하면서도 변경용이성을 희생하지 않는가?", "아키텍처 전환에 드는 비용이 얻는 품질 개선 효과보다 큰가?"라는 질문에 체계적으로 답하는 것이 진짜 아키텍트의 역할입니다. ATAM과 CBAM은 이 질문에 답하는 공식적인 도구입니다.

### 1. 출제 배경 및 의도

ATAM(Architecture Tradeoff Analysis Method)과 CBAM(Cost Benefit Analysis Method)은 CMU(카네기멜론 대학) SEI가 개발한 아키텍처 평가 방법론으로, 대규모 미션크리티컬 시스템(국방, 항공, 공공 인프라)의 아키텍처를 개발 초기에 검증하기 위해 만들어졌습니다.

기술사 시험에서는 두 방법론을 단독으로 또는 비교하여 설명하는 문제가 반복 출제됩니다. 채점관은 **ATAM의 9단계 프로세스**, **유틸리티 트리(Utility Tree)**, **감도점·트레이드오프점·위험점·비위험점**의 4가지 분류, 그리고 **CBAM이 ATAM의 경제적 확장**이라는 관계를 명확히 설명하는 능력을 검증합니다.

### 3. 핵심 키워드 (Must-Have)

#### 가. ATAM (Architecture Tradeoff Analysis Method)

**ATAM이란** 아키텍처가 품질속성 요구사항을 충족하는지를 평가하고, 여러 품질속성 간의 트레이드오프를 식별하는 방법론입니다.

**ATAM 9단계 프로세스**:

```text
[ATAM 9단계]

  1단계: ATAM 발표       ─ 평가 방법 자체를 이해관계자에게 설명
  2단계: 비즈니스 동인 발표 ─ 시스템의 핵심 목적·제약 공유
  3단계: 아키텍처 발표    ─ 아키텍트가 현재 아키텍처 설명
  4단계: 아키텍처 접근법  ─ 사용된 아키텍처 패턴/전술 식별
         (Approach) 식별
              │
              ▼
  5단계: 유틸리티 트리 생성 ─ 품질속성을 계층적으로 구조화
         (Utility Tree)
              │
              ▼
  6단계: 아키텍처 접근법  ─ 각 접근법이 QAS를 어떻게 만족/위반하는지 분석
         분석             → 감도점, 트레이드오프점, 위험점, 비위험점 식별
              │
              ▼
  7단계: 시나리오 브레인스토밍 ─ 이해관계자가 시나리오 추가 제안
  8단계: 시나리오 우선순위화 ─ 투표로 중요 시나리오 선정
  9단계: 아키텍처 접근법  ─ 6단계를 신규 시나리오에 반복 적용
         재분석
              │
              ▼
  결과물: 감도점·트레이드오프점·위험점 목록, 위험 테마
```

**유틸리티 트리 (Utility Tree)**:

```text
[유틸리티 트리 구조]

  유틸리티(Utility)
       │
  ┌────┼────────────────────────┐
  │    │                        │
성능  보안                  변경용이성
(Performance)(Security)    (Modifiability)
  │         │                   │
  ├─ 응답시간 ├─ 데이터 기밀성   ├─ 기능 변경
  │  (H, M)  │  (H, H)          │  (M, H)
  │          │                   │
  └─ 처리량   └─ 권한 제어       └─ 플랫폼 이식
     (M, L)     (M, M)              (L, M)

     (우선순위, 구현난이도) = (H:높음, M:중간, L:낮음)
```

**4가지 분석 결과 분류**:

| 분류 | 정의 | 예시 |
|------|------|------|
| **감도점(Sensitivity Point)** | 특정 아키텍처 결정이 하나의 품질속성에 강하게 영향 | 캐시 크기 → 성능에만 영향 |
| **트레이드오프점(Tradeoff Point)** | 하나의 아키텍처 결정이 2개 이상의 품질속성에 상반된 영향 | 암호화 강도 ↑ → 보안↑ but 성능↓ |
| **위험점(Risk)** | 아직 해결되지 않은 잠재적 아키텍처 문제 | "단일 DB 서버 → 가용성 위험" |
| **비위험점(Non-Risk)** | 분석 후 위험이 없다고 확인된 결정 | "읽기 전용 캐시 사용 → 성능·일관성 모두 양호" |

#### 나. CBAM (Cost Benefit Analysis Method)

**CBAM이란** ATAM을 경제적 관점으로 확장한 방법론으로, 아키텍처 전략 채택의 비용(Cost) 대비 효익(Benefit)을 ROI로 정량화하여 최적 아키텍처를 선택하는 기법입니다.

```text
[CBAM 프로세스]

  ATAM 결과 (시나리오 + 아키텍처 접근법)
          │
          ▼
  1단계: 시나리오 우선순위 재확인
          │
          ▼
  2단계: 효용 함수(Utility Function) 적용
         - 각 QAS 시나리오에 대한 효용 측정
         - 현재 수준 vs 이상적 수준 vs 최악 수준 비교
          │
          ▼
  3단계: 아키텍처 전략별 품질속성 응답값 산정
         - 전략 A: 성능 +30%, 비용 $50K
         - 전략 B: 성능 +10%, 가용성 +20%, 비용 $30K
          │
          ▼
  4단계: 전략별 비용 추정
          │
          ▼
  5단계: ROI(기대 편익/비용) 계산
         ROI = (기대 편익의 총 가중 효용) / 비용
          │
          ▼
  6단계: ROI 기준 아키텍처 전략 선택 및 문서화
```

**CBAM 효용 함수(Utility Function)**:

```text
[효용 함수 예시 - 응답시간]

  효용값
   1.0 │──────────────────────────╮
   0.9 │                          │  ← 이상적(Ideal): 1초 이하
   0.5 │────────────╮             │
   0.1 │            ╰─────────────╯
   0.0 │──────────────────────────╯
       0   1   2   3   4   5초 (응답시간)
                    ↑최악(Worst): 5초 초과 시 효용 0

  효용값이 높을수록 이해관계자 만족도 높음
  비선형 함수: 2초 → 1초 개선이 5초 → 4초 개선보다 가치 큼
```

### 4. 맥락과 비교

| 구분 | SAAM | ATAM | CBAM |
|------|------|------|------|
| 개발 | CMU SEI | CMU SEI | CMU SEI |
| 초점 | 변경용이성 중심 단일 속성 | 다중 품질속성 트레이드오프 | ATAM + 경제적 ROI 분석 |
| 방법 | 시나리오 기반 | 시나리오 + 유틸리티 트리 | 시나리오 + 효용 함수 + 비용 |
| 결과물 | 수정/직접 시나리오 분류 | 위험점·트레이드오프점·감도점 | ROI 순위, 최적 아키텍처 전략 |
| 규모 | 소규모 평가 | 대규모 공식 평가 (2~3일) | ATAM 후 추가 경제 분석 |
| 참여자 | 아키텍트, 이해관계자 | 외부 평가팀 + 이해관계자 | ATAM 팀 + 재무 담당자 |
| 정량화 | 어려움 | 상대적 우선순위 | ROI로 완전 정량화 |

### 5. 실무 제언

**ATAM의 현실적 적용: 경량화 필요성**

- **챌린지**: 완전한 ATAM 수행에는 외부 평가팀 구성, 2~3일간의 이해관계자 워크샵, 수십 개의 시나리오 분석 등 상당한 비용과 시간이 필요합니다. 애자일 방법론이 주류인 현재 환경에서 스프린트 단위로 빠르게 진행되는 아키텍처 결정에 전통적 ATAM을 그대로 적용하는 것은 비현실적입니다.
- **제언**: 경량화된 ATAM(Lightweight ATAM)을 적용해야 합니다. 핵심은 **유틸리티 트리로 QAS 우선순위를 명확히 하고, 주요 아키텍처 결정에 대해 트레이드오프점을 3~5개만 집중 식별**하는 것입니다. ADR(Architecture Decision Record)에 "이 결정으로 X 품질이 Y% 향상되나 Z 품질이 W% 저하된다"는 트레이드오프 기록을 남기는 것이 현실적인 ATAM의 정신을 계승하는 방법입니다.

**CBAM의 재무적 효익 측정**

- **챌린지**: CBAM의 핵심인 효익(Benefit)의 정량화는 실무에서 매우 어렵습니다. "가용성이 99.9%에서 99.99%로 개선되면 비즈니스 가치는 얼마인가?"를 정확히 산정하는 방법이 없는 경우가 대부분입니다.
- **제언**: 비용 모델 대신 **SLA(Service Level Agreement) 위반 패널티, 다운타임으로 인한 매출 손실, 보안 사고 발생 시 예상 손해액** 등의 리스크 기반 비용을 효익의 대리 지표로 활용하는 접근법이 현실적입니다. 예를 들어, "연간 예상 다운타임 비용 = 다운타임 시간 × 시간당 매출" 공식으로 가용성 개선의 경제적 가치를 산정하고 CBAM에 입력하면 의사결정의 객관성을 크게 높일 수 있습니다.
