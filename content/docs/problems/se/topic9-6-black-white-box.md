## 해설

> **겉만 보느냐, 속까지 보느냐**  
> 블랙박스는 입출력 관점에서, 화이트박스는 코드 내부 구조 관점에서 결함을 탐지한다.

비행기의 블랙박스는 내부 구조를 모르는 사람도 기록된 데이터(입출력)를 분석할 수 있습니다. 반면 화이트박스(투명한 박스)는 내부의 회로가 다 보입니다. 소프트웨어 테스팅에서 이 두 은유는 테스터가 소프트웨어의 내부 구현을 알고 테스트하느냐(화이트박스 테스트), 모르고 테스트하느냐(블랙박스 테스트)를 구분하는 핵심 기준입니다.

### 1. 출제 배경 및 의도

이 문제는 두 축의 분류 체계를 동시에 묻고 있습니다.

- **첫 번째 축**: 블랙박스 vs. 화이트박스 (테스터의 시스템 내부 지식 수준 기준)
- **두 번째 축**: 명세기반 vs. 구조기반 vs. 경험기반 (테스트 케이스 설계 근거 기준)

채점관은 이 두 분류 체계가 **어떤 관계인지** 명확히 이해하고 있는지를 봅니다. 블랙박스 테스트 = 명세기반 테스트, 화이트박스 테스트 = 구조기반 테스트라는 대응 관계가 핵심입니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 블랙박스(Black-Box) vs. 화이트박스(White-Box)

```text
[블랙박스 테스트]
           ┌──────────────┐
입력(Input)│  내부 구조를  │ 출력(Output)
──────────→│  모름/무시    │──────────→
           └──────────────┘
  테스터는 요구사항/명세서를 기반으로 테스트 케이스 설계
  내부 코드를 알 필요 없음
  주체: 사용자, 별도 테스트 팀


[화이트박스 테스트]
           ┌──────────────┐
입력(Input)│  if(a>0)     │ 출력(Output)
──────────→│  { ... }     │──────────→
           │  else { ... }│
           └──────────────┘
  테스터는 소스코드를 분석하여 테스트 케이스 설계
  코드의 모든 경로/분기/조건을 실행시키는 것이 목표
  주체: 개발자
```

| 구분 | 블랙박스 테스트 | 화이트박스 테스트 |
|------|-------------|----------------|
| 관점 | 외부(입력→출력) | 내부(코드 구조) |
| 기반 정보 | 요구사항 명세서 | 소스 코드 |
| 목적 | 기능적 동작 검증 | 코드 경로/구조 검증 |
| 주체 | 독립 테스터, QA | 개발자 |
| 발견 결함 | 요구사항 불일치, 누락 기능 | 논리 오류, 미실행 코드 |
| 대표 기법 | 동등 분할, 경계값 분석 | 구문/결정/조건 커버리지 |

#### 나. 블랙박스 테스트의 주요 기법

**① 동등 분할(Equivalence Partitioning)**
- 입력 도메인을 동일한 방식으로 처리되는 클래스(파티션)로 분할
- 각 클래스에서 대표값 하나만 테스트해도 동일한 효과 기대
- 유효 파티션(Valid Partition) + 무효 파티션(Invalid Partition)

**② 경계값 분석(Boundary Value Analysis, BVA)**
- 파티션의 경계(Boundary)에서 결함이 가장 많이 발생한다는 경험칙 기반
- 경계값, 경계-1, 경계+1을 테스트 (3-value BVA)
- 상세 내용: topic9-7-equivalence-boundary 참조

**③ 결정 테이블(Decision Table)**
- 여러 입력 조건의 조합과 그에 따른 행동을 표로 나타냄
- 복잡한 비즈니스 규칙(예: 보험 가입 조건)을 테스트할 때 유용

```text
[결정 테이블 예시: 할인 적용 규칙]
조건                   │ TC1 │ TC2 │ TC3 │ TC4
회원인가?              │  Y  │  Y  │  N  │  N
구매금액 5만원 이상?   │  Y  │  N  │  Y  │  N
─────────────────────────────────────────────
10% 할인 적용         │  ●  │     │     │
5% 할인 적용          │     │  ●  │  ●  │
할인 없음              │     │     │     │  ●
```

**④ 상태 전이 테스트(State Transition Testing)**
- 시스템이 다양한 상태를 가지고 이벤트에 따라 상태가 변화하는 경우
- 상태, 전이, 이벤트, 액션을 정의하고 모든 유효한/무효한 전이를 테스트

```text
[ATM 상태 전이 예시]
  [유휴] ─카드삽입→ [카드인식] ─PIN입력→ [인증됨] ─출금선택→ [처리중]
    ↑                                                              │
    └──────────────────────────────────────────────────────────────┘
                           거래완료
```

#### 다. 화이트박스 테스트의 주요 기법: 구조 기반 커버리지

화이트박스 테스트의 핵심은 코드의 어느 부분을 얼마나 실행했는지를 측정하는 **커버리지(Coverage)**입니다.

```text
[커버리지 강도 순서: 약함 → 강함]

구문 커버리지(Statement Coverage)
    ↓  모든 실행 가능한 문장을 최소 1회 실행
결정 커버리지(Decision/Branch Coverage)
    ↓  모든 결정(if/switch)의 TRUE/FALSE를 최소 1회씩 실행
조건 커버리지(Condition Coverage)
    ↓  각 개별 조건을 TRUE/FALSE로 최소 1회씩 실행
MC/DC (Modified Condition/Decision Coverage)
    ↓  각 조건이 독립적으로 결정에 영향을 미치는 테스트 케이스
경로 커버리지(Path Coverage)
       모든 가능한 실행 경로를 실행 (이론적으로 완전하지만 비현실적)
```

| 커버리지 | 기준 | 최소 테스트 수 | 도달 가능 여부 |
|---------|------|------------|-------------|
| 구문 (SC) | 모든 statement 1회 이상 실행 | 적음 | 100% 가능 |
| 결정 (DC) | 모든 branch TRUE/FALSE | SC보다 많음 | 100% 가능 |
| 조건 (CC) | 각 조건 T/F | DC와 유사 | 100% 가능 |
| MC/DC | 각 조건이 독립적 영향 증명 | n+1개 (n=조건 수) | 100% 가능 |
| 경로 (PC) | 모든 실행 경로 | 조합 폭발 | 루프 있으면 불가 |

### 3. 핵심 키워드 (Must-Have)

**명세기반 = 블랙박스, 구조기반 = 화이트박스, 경험기반 = 탐색적**

| 분류 | 설명 | 대표 기법 |
|------|------|---------|
| 명세기반 (Specification-Based) | 요구사항/명세 기반 테스트 케이스 설계 | 동등 분할, 경계값 분석, 결정 테이블, 상태 전이 |
| 구조기반 (Structure-Based) | 소스코드 구조 기반 테스트 케이스 설계 | 구문/결정/조건/MC/DC 커버리지 |
| 경험기반 (Experience-Based) | 테스터의 경험과 직관 기반 | 탐색적 테스팅(Exploratory), 오류 추측(Error Guessing), 체크리스트 |

### 4. 맥락과 비교

**그레이박스(Gray-Box) 테스트**

완전한 블랙박스와 화이트박스의 중간 형태입니다. 내부 구조를 부분적으로 알고 있는 상태(예: DB 스키마는 알지만 내부 알고리즘은 모름)에서 수행하는 테스트로, 통합 테스트에서 자주 활용됩니다.

**탐색적 테스팅(Exploratory Testing)**

경험기반 테스팅의 대표 기법으로, 스크립트 없이 테스터가 시스템을 자유롭게 탐색하면서 동시에 학습, 설계, 실행을 수행합니다. 애자일 개발 환경에서 빠르게 변화하는 시스템에 매우 효과적입니다.

### 5. 실무 제언

**실제 프로젝트에서는 두 방식의 균형이 핵심**

블랙박스 테스트만으로는 코드의 미실행 경로(Dead Code)에 숨은 결함을 발견하기 어렵습니다. 반대로 화이트박스 테스트만으로는 요구사항 누락(기능 자체가 없는 경우)을 발견하기 어렵습니다.

프로젝트 현장에서의 권장 조합:
- 단위 테스트: 화이트박스 위주 (개발자가 수행, 높은 커버리지 목표)
- 통합 테스트: 그레이박스 (인터페이스 명세 + 내부 연동 지식 활용)
- 시스템/인수 테스트: 블랙박스 위주 (독립 QA 또는 사용자가 수행)

**커버리지 100%의 함정**

구문 커버리지 100%를 달성했다고 해서 모든 결함을 발견했다는 의미가 아닙니다. "테스트가 실행되었다"는 것과 "테스트가 결함을 검증했다"는 전혀 다른 개념입니다. 어설션(Assertion)이 없는 테스트 코드는 커버리지를 채울 뿐 실제 검증 가치가 없습니다. 커버리지 지표는 테스트의 최소 기준일 뿐 품질 보장의 충분조건이 아님을 항상 인식해야 합니다.
