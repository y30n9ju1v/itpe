## 해설

### 출제 배경 및 의도

소프트웨어 테스트 기법은 ISTQB 실라버스와 ISO/IEC/IEEE 29119 표준에서 두 가지 분류 축으로 체계화되어 있다. 첫 번째 축은 테스터의 내부 구현 지식 수준에 따른 블랙박스/화이트박스 구분이며, 두 번째 축은 테스트 케이스 설계 근거에 따른 명세기반/구조기반/경험기반 구분이다.

기술사 시험에서 이 주제가 반복 출제되는 이유는 테스트 기법 선택이 결함 발견 효율성과 직결되기 때문이다. 기능 요구사항 누락은 블랙박스로, 논리적 코드 오류는 화이트박스로, 자동화 케이스가 발견하지 못하는 경계 결함은 경험기반 탐색적 테스팅으로 각각 보완하는 전략이 현업의 표준이다.

채점관은 두 분류 체계의 대응 관계(블랙박스=명세기반, 화이트박스=구조기반)를 명확히 이해하고, 각 기법의 적용 시나리오와 트레이드오프를 논리적으로 설명하는 능력을 평가한다.

### 1. 테스트 기법 분류 체계, 개요

정 의  • 소프트웨어 테스트 기법을 테스터의 내부 지식 수준(블랙박스/화이트박스)과
       설계 근거(명세기반/구조기반/경험기반) 두 축으로 체계화한 분류 체계
       - ISTQB 및 ISO/IEC/IEEE 29119 표준 기반

```
[SW 테스트 기법 분류 체계]

  테스트 기법
       │
  ┌────┴──────────────────────────────┐
  │                                   │
  내부 지식 기준                    설계 근거 기준
  ├── 블랙박스 (외부 관점)           ├── 명세기반 (Specification-Based)
  │     요구사항/명세 기반            │     동등분할, 경계값, 결정테이블
  │     주체: 독립 QA 테스터          │     상태전이 테스팅
  │                                   │
  ├── 화이트박스 (내부 관점)          ├── 구조기반 (Structure-Based)
  │     소스코드 구조 기반            │     구문/결정/조건/MC/DC 커버리지
  │     주체: 개발자                  │
  │                                   ├── 경험기반 (Experience-Based)
  └── 그레이박스 (부분 지식)              탐색적 테스팅, 오류 추측
        DB 스키마 등 부분 지식 활용        체크리스트 기반
        주로 통합 테스트에서 활용

  [대응 관계]
  블랙박스 ↔ 명세기반
  화이트박스 ↔ 구조기반
  경험기반 ↔ 탐색적 (분류 독립적)
```

- 블랙박스와 화이트박스는 상호 보완적이며, 결함 유형별로 적합한 기법을 조합하는 것이 핵심이다.

### 2. 블랙박스·화이트박스 테스트 상세

1) 블랙박스(Black-Box) vs. 화이트박스(White-Box) 비교

| 구분 | 블랙박스 테스트 | 화이트박스 테스트 |
|------|-------------|----------------|
| 관점 | 외부(입력→출력) | 내부(코드 구조) |
| 기반 정보 | 요구사항 명세서 | 소스 코드 |
| 목적 | 기능적 동작 검증 | 코드 경로/구조 검증 |
| 주체 | 독립 테스터, QA | 개발자 |
| 발견 결함 | 요구사항 불일치, 누락 기능 | 논리 오류, 미실행 코드(Dead Code) |
| 대표 기법 | 동등분할, 경계값 분석, 결정 테이블 | 구문/결정/조건/MC/DC 커버리지 |
| 표준 적용 | 시스템·인수 테스트 | 단위 테스트, 안전핵심 시스템 |

```
[블랙박스 vs 화이트박스 시각화]

[블랙박스 테스트]
           ┌──────────────┐
입력(Input)│  내부 구조를  │ 출력(Output)
──────────→│  모름/무시    │──────────→
           └──────────────┘
  요구사항/명세서 기반 테스트 케이스 설계
  주체: 독립 QA 테스터

[화이트박스 테스트]
           ┌──────────────┐
입력(Input)│  if(a>0)     │ 출력(Output)
──────────→│  { ... }     │──────────→
           │  else { ... }│
           └──────────────┘
  소스코드 분석 → 모든 경로/분기/조건 실행 목표
  주체: 개발자
```

- 블랙박스만으로는 Dead Code 결함을, 화이트박스만으로는 요구사항 누락을 발견하기 어려우므로 두 기법을 병행해야 한다.

2) 명세기반·구조기반·경험기반 설계 기법

| 분류 | 설계 근거 | 대표 기법 | 발견 결함 유형 |
|------|---------|---------|-------------|
| 명세기반 (Specification-Based) | 요구사항·명세서 | 동등분할, 경계값 분석, 결정 테이블, 상태 전이 | 요구사항 불일치, 경계 오류 |
| 구조기반 (Structure-Based) | 소스코드 구조 | 구문/결정/조건/MC/DC/경로 커버리지 | 논리 오류, 미실행 경로 |
| 경험기반 (Experience-Based) | 테스터 경험·직관 | 탐색적 테스팅, 오류 추측, 체크리스트 | 예상치 못한 사용 패턴, 살충제 패러독스 대응 |

```
[블랙박스 주요 기법: 결정 테이블 예시 - 할인 적용 규칙]

조건                   │ TC1 │ TC2 │ TC3 │ TC4
회원인가?              │  Y  │  Y  │  N  │  N
구매금액 5만원 이상?   │  Y  │  N  │  Y  │  N
─────────────────────────────────────────────
10% 할인 적용          │  ●  │     │     │
5% 할인 적용           │     │  ●  │  ●  │
할인 없음              │     │     │     │  ●
```

```
[화이트박스 커버리지 강도 계층]

구문 커버리지(SC)   → 모든 statement 최소 1회 실행
       ↓
결정 커버리지(DC)   → 모든 if/switch의 TRUE/FALSE 최소 1회
       ↓
조건 커버리지(CC)   → 각 개별 조건을 T/F로 최소 1회
       ↓
MC/DC              → 각 조건이 독립적으로 결정에 영향함을 증명
       ↓
경로 커버리지(PC)   → 모든 실행 경로 (루프 포함 시 비현실적)
```

| 커버리지 | 기준 | 최소 테스트 수 | 적용 도메인 |
|---------|------|------------|-----------|
| 구문 (SC) | 모든 statement 1회 이상 실행 | 적음 | 일반 상용 SW |
| 결정 (DC) | 모든 branch TRUE/FALSE | SC보다 많음 | 일반 상용 SW |
| MC/DC | 각 조건이 독립적 영향 증명 | n+1개 (n=조건 수) | 항공·의료·원전 |
| 경로 (PC) | 모든 실행 경로 | 조합 폭발 | 루프 있으면 불가 |

- 안전핵심(Safety-Critical) 시스템(항공 DO-178C)은 MC/DC를 의무 적용하며, 일반 상용 SW는 결정 커버리지 80% 이상을 표준 목표로 설정한다.

### 3. 테스트 기법 조합 전략 및 트렌드

| 테스트 레벨 | 권장 기법 조합 | 이유 |
|-----------|------------|------|
| 단위 테스트 | 화이트박스(구조기반) 위주, TDD | 개발자 수행, 높은 커버리지 목표 |
| 통합 테스트 | 그레이박스 (명세 + 내부 연동 지식) | 인터페이스 명세와 실제 연동 동시 검증 |
| 시스템 테스트 | 블랙박스(명세기반) 위주 | 독립 QA 수행, 요구사항 추적성 확보 |
| 인수 테스트 | 블랙박스 + 경험기반(탐색적) | 사용자 관점 검증, 예상치 못한 패턴 발견 |
| 회귀 테스트 | 자동화(명세기반) + 탐색적 병행 | 살충제 패러독스 대응 |

- 단일 기법만 사용하는 프로젝트는 테스트 사각지대를 가지므로, 테스트 레벨별 기법 조합 전략을 사전에 수립해야 한다.  "끝"

### 실무 제언

**블랙박스·화이트박스 균형 설계**
- **챌린지**: 블랙박스 테스트만으로는 코드의 미실행 경로(Dead Code)에 숨은 결함을 발견하기 어렵고, 화이트박스만으로는 요구사항 누락(기능 자체가 없는 경우)을 발견할 수 없다.
- **제언**: 단위 테스트는 화이트박스(JUnit + JaCoCo 결정 커버리지 80% 이상), 시스템·인수 테스트는 블랙박스(동등분할·경계값·결정 테이블) 위주로 구성하고, 탐색적 테스팅을 매 스프린트마다 병행하여 자동화 테스트의 사각지대를 보완해야 한다.

**커버리지 100%의 함정**
- **챌린지**: 구문 커버리지 100%를 달성했음에도 릴리스 후 주요 버그가 발생하는 사례가 빈번하다. "테스트가 실행되었다"와 "테스트가 결함을 검증했다"는 전혀 다른 개념이다.
- **제언**: 어설션(Assertion)이 없는 테스트 코드는 커버리지를 채울 뿐 검증 가치가 없다. 커버리지 지표를 최소 기준으로 활용하되, Mutation Testing(PIT)으로 테스트 케이스의 실제 결함 감지력을 주기적으로 측정하여 테스트 품질을 관리해야 한다.

**경험기반 테스팅의 체계적 활용**
- **챌린지**: 탐색적 테스팅은 테스터의 역량에 크게 의존하여 결과의 재현성과 추적성이 낮다는 비판을 받는다.
- **제언**: Session-Based Exploratory Testing(SBET) 방식을 도입하여 탐색 목표(Charter), 시간(Timebox), 결과(Notes)를 구조화한다. 발견된 결함 패턴은 자동화 테스트 스위트에 추가하여 경험기반 발견이 명세기반 케이스로 체계화되는 선순환을 만들어야 한다.
