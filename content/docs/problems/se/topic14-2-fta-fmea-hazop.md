## 해설

> **세 가지 시각으로 위험을 분석하는 상호 보완적 기법들**  
> FTA(하향식), FMEA(상향식), HAZOP(편차 중심) — 각기 다른 관점으로 위험을 찾는다

안전 필수 시스템에서 위험 분석은 단 하나의 기법으로는 불충분합니다. FTA는 "왜 이 사고가 발생했는가"를 역추적하고, FMEA는 "이 부품이 고장나면 무슨 일이 일어나는가"를 전진 추적하며, HAZOP는 "설계 의도에서 벗어나는 모든 경우를 체계적으로 상상"합니다. 세 기법을 이해하고 상황에 맞게 적용하는 능력이 기술사에게 요구됩니다.

### 1. 출제 배경 및 의도

이 문제는 단순히 세 기법의 정의를 묻는 것이 아닙니다. **세 기법의 공통점과 차이점**, 그리고 **언제 어떤 기법을 사용해야 하는지** 판단 능력을 평가합니다.

공통 유형(관리+응용)으로 출제된 만큼, IT 관리 직군도 응용 직군도 모두 답할 수 있어야 합니다. 각 기법을 소프트웨어 시스템에 적용하는 구체적인 예시를 들 수 있으면 고득점입니다.

### 2. 개념의 본질과 작동 메커니즘

#### 세 기법의 위치

```text
[안전성 분석 기법의 분류]

접근 방향
    │
    ├── 하향식(Top-Down, 연역적)
    │       └── FTA (Fault Tree Analysis)
    │           "사고에서 시작해 원인으로 내려간다"
    │
    ├── 상향식(Bottom-Up, 귀납적)
    │       └── FMEA (Failure Mode and Effect Analysis)
    │           "부품 고장에서 시작해 영향을 올려간다"
    │
    └── 편차 분석(Deviation-Based)
            └── HAZOP (Hazard and Operability Study)
                "설계 의도에서 벗어나는 경우를 체계적으로 열거"
```

### 3. 핵심 키워드 (Must-Have)

#### 1) FTA (Fault Tree Analysis, 결함수 분석)

**개요**: 시스템이나 소프트웨어에서 바람직하지 않은 특정 사고 사건(Top Event)에서 출발하여, 그 원인이 되는 결함들을 논리 게이트로 연결하며 하향식으로 분석하는 정성적·정량적 기법

**핵심 특성**
- 방향: 하향식(Top-Down)
- 성격: 연역적(Deductive)
- 관심: "이 사고가 왜 발생하는가?"
- 주요 표현 도구: 결함 트리(Fault Tree), AND/OR 논리 게이트

**주요 기호**

| 기호 | 명칭 | 의미 |
|------|------|------|
| 원(Event) | 기본 사건 | 더 이상 분해 불필요한 기본 결함 |
| 사각형(Rectangle) | 중간 사건 | 추가 분석이 필요한 결함 사건 |
| 다이아몬드 | 미전개 사건 | 정보 부족으로 분석 중단 |
| AND Gate | AND 게이트 | 모든 입력 사건이 동시에 발생해야 출력 |
| OR Gate | OR 게이트 | 입력 사건 중 하나라도 발생하면 출력 |

**FTA 수행 절차**

```text
① Top Event 정의
   └→ "시스템 전체 장애 발생" 또는 "사용자 데이터 손실" 등
   
② 직접 원인 열거
   └→ Top Event를 유발하는 직접적 원인 사건들 열거
   └→ AND/OR 관계로 연결

③ 반복적 원인 분석
   └→ 각 원인에 대해 다시 원인을 분석
   └→ 기본 사건(더 이상 분해 불필요) 까지 반복

④ 최소 컷셋(MCS: Minimal Cut Set) 도출
   └→ Top Event를 발생시킬 수 있는 최소 기본 사건 조합
   └→ MCS가 단일 사건이면 단일 점 장애(Single Point of Failure)

⑤ 정량 분석 (선택)
   └→ 각 기본 사건의 고장률을 알면 Top Event 발생 확률 계산 가능
```

**소프트웨어 적용 예시**

```text
[Top Event: 결제 서비스 완전 장애]
           │
        [OR Gate]
    ┌──────┴──────┐
[DB 접속 불가]  [API 서버 전체 다운]
                        │
                     [AND Gate]
              ┌─────────┴──────────┐
    [애플리케이션 서버1 장애]   [애플리케이션 서버2 장애]
```

AND Gate에 연결된 사건들은 모두 동시에 발생해야 Top Event 발생 → 이중화(Redundancy)로 안전 달성

**장점 / 단점**

| 장점 | 단점 |
|------|------|
| 사고 원인을 명확히 시각화 | Top Event가 명확해야 분석 가능 |
| 정량적 확률 계산 가능 | 동시 발생 고장 분석이 복잡 |
| 단일 점 장애(SPF) 식별에 탁월 | 초기 설계에 적용하기 어려움 |

#### 2) FMEA (Failure Mode and Effect Analysis, 고장 모드 영향 분석)

**개요**: 시스템을 구성하는 개별 컴포넌트의 잠재적 고장 모드(Failure Mode)를 체계적으로 식별하고, 각 고장이 상위 시스템에 미치는 영향(Effect)을 상향식으로 분석하는 기법

**핵심 특성**
- 방향: 상향식(Bottom-Up)
- 성격: 귀납적(Inductive)
- 관심: "이 고장의 영향은 무엇인가?"
- 핵심 지표: RPN(Risk Priority Number)

**FMEA 수행 절차 및 워크시트**

```text
[FMEA 워크시트 구조]

컴포넌트 | 고장 모드 | 영향(Effect) | S | 원인(Cause) | O | 현재 관리 | D | RPN | 권장 조치
---------|---------|-------------|---|------------|---|----------|---|-----|--------
결제 API | 응답 없음 | 결제 불가   | 9 | 메모리 누수 | 4 | 모니터링 | 3 | 108 | 자동 재시작
결제 API | 오답     | 이중 결제  | 10| 트랜잭션 버그| 2 | 코드 리뷰 | 5 | 100 | 트랜잭션 테스트 강화
```

**RPN 계산과 의미**

```text
RPN = S(Severity, 심각도) × O(Occurrence, 발생도) × D(Detection, 검출도)

각 항목: 1~10 척도

S(심각도)
  10: 안전 사고 유발 가능 (경고 없이)
   9: 안전 사고 유발 가능 (경고 있음)
   7~8: 주요 기능 장애
   4~6: 성능 저하
   1~3: 경미한 영향

O(발생도)
  10: 거의 확실히 발생
   7~9: 반복적 발생
   4~6: 간헐적 발생
   1~3: 거의 발생하지 않음

D(검출도)
  10: 고장을 감지할 수 없음
   7~9: 감지 어려움
   4~6: 어느 정도 감지 가능
   1~3: 거의 확실히 사전 감지
```

RPN 100 이상이면 즉시 대응 조치가 필요합니다.

**장점 / 단점**

| 장점 | 단점 |
|------|------|
| 모든 컴포넌트를 체계적으로 검토 | 컴포넌트가 많으면 작업량 방대 |
| 수치적 우선순위 설정 가능(RPN) | 복합 고장 분석이 어려움 |
| 설계 초기부터 적용 가능 | 고장 모드 누락 위험 |

#### 3) HAZOP (Hazard and Operability Study, 위험과 운용성 검토)

**개요**: 설계 의도(Design Intent)에서 벗어나는 모든 가능한 편차(Deviation)를 가이드워드(Guide Words)를 사용하여 체계적으로 열거하고, 각 편차의 위험성과 운용성 문제를 분석하는 기법

원래 화학 공정 설계 검토를 위해 ICI(Imperial Chemical Industries)에서 개발되었으나, 소프트웨어 시스템에도 폭넓게 적용됩니다.

**핵심 특성**
- 방향: 편차(Deviation) 중심
- 성격: 귀납적, 탐색적
- 관심: "의도한 대로 동작하지 않으면 무슨 일이 생기는가?"
- 핵심 도구: 가이드워드(Guide Words)

**HAZOP 가이드워드**

| 가이드워드 | 의미 | 소프트웨어 적용 예시 |
|----------|------|------------------|
| NO / NOT | 설계 의도가 완전히 없음 | 인증 토큰이 전혀 발급되지 않는다면? |
| MORE | 설계 의도보다 양이 많음 | 요청이 초당 1000배 더 많이 들어온다면? |
| LESS | 설계 의도보다 양이 적음 | 데이터 일부만 저장된다면? |
| AS WELL AS | 의도한 것에 추가 발생 | 인증 외에 의도하지 않은 권한도 부여된다면? |
| PART OF | 설계 의도가 일부만 실행 | 트랜잭션 일부만 커밋된다면? |
| REVERSE | 설계 의도와 반대 방향 | 잠금이 해제되어야 할 때 잠긴다면? |
| OTHER THAN | 완전히 다른 것이 발생 | 다른 사용자의 데이터가 반환된다면? |
| EARLY / LATE | 시간적으로 빠르거나 늦음 | 타임아웃이 너무 빨리 발생한다면? |
| BEFORE / AFTER | 순서가 바뀜 | 인증 전에 데이터가 반환된다면? |

**HAZOP 수행 절차**

```text
① 분석 노드(Node) 정의
   └→ 분석 대상을 의미 있는 단위(프로세스 단계, API 등)로 나눔

② 설계 의도 명시
   └→ 각 노드가 정상적으로 해야 하는 동작 명세

③ 가이드워드 × 매개변수 조합
   └→ 모든 가이드워드를 각 매개변수에 적용하여 편차 열거

④ 원인 및 결과 분석
   └→ 각 편차의 원인 파악 및 시스템에 미치는 결과 평가

⑤ 권장 조치 도출
   └→ 안전 조치, 설계 변경, 운영 절차 등 제안
```

**장점 / 단점**

| 장점 | 단점 |
|------|------|
| 놓치기 쉬운 편차를 체계적으로 발견 | 시간이 매우 많이 소요 |
| 설계 단계에서 운용 문제 사전 예방 | 경험 있는 팀이 필요 |
| 복합 상황(운용+안전) 동시 분석 | 상호작용 고장 분석 한계 |

### 4. 맥락과 비교

**FTA / FMEA / HAZOP 종합 비교표**

| 구분 | FTA | FMEA | HAZOP |
|------|-----|------|-------|
| 분석 방향 | 하향식 | 상향식 | 편차 중심 |
| 논리 | 연역적 | 귀납적 | 귀납적/탐색적 |
| 시작점 | 사고 사건(Top Event) | 컴포넌트 고장 모드 | 설계 의도 |
| 분석 도구 | 논리 게이트(AND/OR) | RPN 점수(S×O×D) | 가이드워드 |
| 결과물 | 결함 트리, MCS | FMEA 워크시트 | HAZOP 레코드 |
| 정량적 분석 | 가능 | 가능(RPN) | 일반적으로 불가 |
| 최적 적용 단계 | 설계 검증, 사고 조사 | 상세 설계 | 상세 설계 검토 |
| 강점 | SPF 식별, 확률 계산 | 전수 검토, 우선순위 | 편차/운용성 발견 |
| 표준 근거 | IEC 61025 | IEC 60812 | IEC 61882 |

**상호 보완적 활용 전략**

```text
[프로젝트에서 세 기법의 순차 적용]

1단계: PHA(예비 위험 분석) → 전체 위험원 개략 파악
         │
2단계: HAZOP → 상세 설계 단계에서 운용 편차 탐색
         │
3단계: FMEA → 개별 컴포넌트 고장 모드 전수 분석, RPN 산출
         │
4단계: FTA → 고위험(High RPN) 사건을 Top Event로 심층 원인 분석
         │
5단계: 안전 요구사항 통합 → 설계 반영 → 재분석
```

### 5. 실무 제언

**소프트웨어 시스템에서 세 기법 활용 가이드**

소프트웨어 시스템은 물리 시스템과 다른 특성(복잡한 상호작용, 비결정론적 동작, 소프트웨어 고장의 체계적 특성)을 가지므로, 세 기법 적용 시 다음을 고려해야 합니다.

- **FTA 적용 시**: 소프트웨어는 하드웨어와 달리 마모 고장이 없습니다. 소프트웨어 고장은 설계 결함에 기인하므로, FTA의 기본 사건은 "특정 입력 조건 + 소프트웨어 버그의 조합"으로 설정해야 합니다.
- **FMEA 적용 시**: 소프트웨어 FMEA(Software FMEA)는 하드웨어 FMEA와 달리 "기능 고장 모드" 중심으로 분석합니다. "잘못된 출력", "응답 없음", "기능 손실" 등의 고장 모드를 사전 정의합니다.
- **HAZOP 적용 시**: 소프트웨어 HAZOP은 데이터 흐름, API 인터페이스, 상태 전이를 노드로 정의하고 각각에 가이드워드를 적용합니다.

**AI 기반 시스템의 새로운 과제**

AI/ML 시스템은 비결정론적 동작으로 인해 전통적인 FTA/FMEA 적용이 어렵습니다. 학습 데이터의 편향(Bias), 분포 이탈(Out-of-Distribution) 입력, 적대적 공격(Adversarial Attack) 등 새로운 유형의 위험에 대응하기 위해 STPA(System-Theoretic Process Analysis)와 같은 현대적 안전 분석 기법이 주목받고 있습니다.
