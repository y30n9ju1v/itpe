## 해설

### 출제 배경 및 의도

안전 필수 시스템 개발에서 위험 분석은 단일 기법으로 충분하지 않다. 자동차·항공·원전 등 고신뢰성 도메인에서 IEC 61508, ISO 26262 등 기능안전 표준이 확산되며, 복수 기법의 보완적 활용이 산업 현장의 필수 역량이 되었다. 공공 정보시스템에서도 장애·보안사고 원인 분석에 FTA·FMEA가 광범위하게 쓰이며, 기술사 시험에서도 세 기법의 차이와 조합 전략을 묻는 문제가 반복 출제된다.

공통 유형으로 출제된 만큼, 단순 정의 열거가 아니라 "세 기법이 서로 어떻게 다른 시각으로 위험을 보는가"와 "언제 어떤 기법을 선택하는가"의 판단 논리를 서술해야 고득점이 가능하다. 각 기법을 SW 시스템에 적용하는 구체 예시를 들면 더욱 차별화된다.

### 1. FTA·FMEA·HAZOP 개요

정 의  • **FTA·FMEA·HAZOP**을 접근 방향·분석 논리별로 분류한 위험 분석 기법군
       - FTA(하향식·연역): 사고 사건(Top Event)에서 출발해 원인으로 역추적
       - FMEA(상향식·귀납): 컴포넌트 고장 모드에서 출발해 영향을 상위로 추적
       - HAZOP(편차 중심·탐색): 설계 의도 이탈 편차를 가이드워드로 체계적 열거

```
[안전성 분석 기법 분류]

접근 방향
    │
    ├── 하향식(Top-Down, 연역적)
    │       └── FTA (Fault Tree Analysis)
    │           사고에서 원인으로 역추적
    │           표준: IEC 61025
    │
    ├── 상향식(Bottom-Up, 귀납적)
    │       └── FMEA (Failure Mode and Effect Analysis)
    │           부품 고장 → 상위 영향 추적
    │           표준: IEC 60812
    │
    └── 편차 분석(Deviation-Based, 탐색적)
            └── HAZOP (Hazard and Operability Study)
                가이드워드로 설계 의도 이탈 열거
                표준: IEC 61882
```

- 세 기법은 상호 보완적이며 단계별 순차 적용이 안전 요구사항 도출의 모범 사례이다.

### 2. 핵심 기법별 상세 내용

1) FTA (Fault Tree Analysis, 결함수 분석)

| 구분 | 항목 | 설명 |
|------|------|------|
| 방향 | 하향식(Top-Down) | 사고 사건에서 원인으로 연역 |
| 표현 | AND/OR 논리 게이트 | 모든 입력 동시 발생(AND), 하나라도 발생(OR) |
| 결과물 | 결함 트리 + 최소 컷셋(MCS) | MCS가 단일 사건이면 단일 점 장애(SPF) |
| 정량 분석 | 가능 | 기본 사건 고장률 → Top Event 발생 확률 계산 |
| 강점 | SPF 식별·확률 계산 탁월 | 이중화 설계 근거 제공 |
| 표준 | IEC 61025 | - |

```
[FTA 수행 절차]

① Top Event 정의 → "결제 서비스 완전 장애"
② 직접 원인 열거 → AND/OR 게이트로 연결
③ 반복 원인 분석 → 기본 사건까지 분해
④ 최소 컷셋(MCS) 도출 → SPF 식별
⑤ 정량 분석(선택) → Top Event 발생 확률 계산

[소프트웨어 FTA 예시]

        [Top Event: 결제 서비스 완전 장애]
                       │
                   [OR Gate]
           ┌───────────┴────────────┐
      [DB 접속 불가]       [API 서버 전체 다운]
                                   │
                               [AND Gate]
                      ┌────────────┴────────────┐
              [AP 서버1 장애]          [AP 서버2 장애]
```

- AND Gate 하위 사건들은 동시 발생 시에만 Top Event 발생 → 이중화(Redundancy)로 위험 저감

2) FMEA (Failure Mode and Effect Analysis, 고장 모드 영향 분석)

| 구분 | 항목 | 설명 |
|------|------|------|
| 방향 | 상향식(Bottom-Up) | 컴포넌트 고장에서 상위 영향으로 귀납 |
| 핵심 지표 | RPN = S × O × D | 심각도×발생도×검출도, 각 1~10 척도 |
| RPN 임계값 | 100 이상 즉시 대응 | 우선순위별 개선 조치 수행 |
| 강점 | 전수 검토·수치 우선순위 | 설계 초기부터 적용 가능 |
| 표준 | IEC 60812 | - |

```
[FMEA 워크시트 구조]

컴포넌트 | 고장 모드  | 영향(Effect) | S  | 원인         | O | 현재 관리 | D | RPN | 권장 조치
---------|-----------|-------------|----|-----------  |---|---------|---|-----|----------
결제 API | 응답 없음 | 결제 불가    |  9 | 메모리 누수   | 4 | 모니터링  | 3 | 108 | 자동 재시작
결제 API | 오답 반환 | 이중 결제    | 10 | 트랜잭션 버그 | 2 | 코드 리뷰 | 5 | 100 | 트랜잭션 테스트 강화
```

- RPN 100 이상 항목에 대해 즉각 개선 조치를 실행하고 재산정하여 위험을 허용 수준으로 낮춘다.

3) HAZOP (Hazard and Operability Study, 위험과 운용성 검토)

| 구분 | 항목 | 설명 |
|------|------|------|
| 방향 | 편차(Deviation) 중심 | 설계 의도 이탈을 가이드워드로 탐색 |
| 핵심 도구 | 7개 가이드워드 (IEC 61882 표준) | NO·MORE·LESS·AS WELL AS·PART OF·REVERSE·OTHER THAN (EARLY/LATE·BEFORE/AFTER는 시간적 HAZOP 비표준 확장) |
| 출발점 | 화학 공정 ICI → SW 확장 | 데이터 흐름·API·상태 전이에 적용 |
| 강점 | 놓치기 쉬운 편차 체계적 발견 | 운용성 문제 사전 예방 |
| 표준 | IEC 61882 | - |

```
[HAZOP 가이드워드 × SW 적용 예시]

가이드워드   | SW 적용 예시
------------|-------------------------------------------
NO / NOT    | 인증 토큰이 전혀 발급되지 않는다면?
MORE        | 요청이 초당 1,000배 더 많이 들어온다면?
LESS        | 데이터 일부만 저장된다면?
AS WELL AS  | 의도하지 않은 권한도 함께 부여된다면?
PART OF     | 트랜잭션 일부만 커밋된다면?
REVERSE     | 잠금이 해제되어야 할 때 잠긴다면?
OTHER THAN  | 다른 사용자의 데이터가 반환된다면?
EARLY/LATE  | 타임아웃이 너무 빨리 발생한다면?
BEFORE/AFTER| 인증 전에 데이터가 반환된다면?
```

- HAZOP 레코드는 각 편차의 원인·결과·권장 조치를 포함하며, 안전 요구사항 도출의 핵심 근거가 된다.

### 3. 종합 비교 및 상호 보완 활용 전략

1) FTA / FMEA / HAZOP 종합 비교

| 구분 | FTA | FMEA | HAZOP |
|------|-----|------|-------|
| 분석 방향 | 하향식·연역 | 상향식·귀납 | 편차 중심·탐색 |
| 시작점 | 사고 사건(Top Event) | 컴포넌트 고장 모드 | 설계 의도 |
| 분석 도구 | AND/OR 논리 게이트 | RPN(S×O×D) | 7개 가이드워드 (IEC 61882) |
| 결과물 | 결함 트리, MCS | FMEA 워크시트 | HAZOP 레코드 |
| 정량 분석 | 가능 | 가능(RPN) | 일반적 불가 |
| 최적 적용 | 설계 검증·사고 조사 | 상세 설계 단계 | 상세 설계 검토 |
| 표준 | IEC 61025 | IEC 60812 | IEC 61882 |

2) 순차 적용 전략

```
[프로젝트 단계별 위험 분석 기법 적용]

1단계: PHA(예비 위험 분석)
       → 전체 위험원 개략 파악
                 │
2단계: HAZOP
       → 상세 설계 단계 운용 편차 탐색
                 │
3단계: FMEA
       → 개별 컴포넌트 고장 모드 전수 분석, RPN 산출
                 │
4단계: FTA
       → 고위험(High RPN) 사건을 Top Event로 심층 원인 분석
                 │
5단계: 안전 요구사항 통합 → 설계 반영 → 재분석
```

- 세 기법을 순차·복합 적용함으로써 안전 필수 시스템의 위험을 허용 수준으로 낮출 수 있다.  "끝"

### 실무 제언

**SW 시스템에서의 FTA 적용 한계**
- **챌린지**: SW는 HW와 달리 마모 고장이 없어 고장률 산정이 어렵고, FTA의 기본 사건 정의가 모호해지는 경향이 있다.
- **제언**: SW FTA에서는 기본 사건을 "특정 입력 조건 + 설계 결함 조합"으로 설정하고, 정량 분석보다는 MCS 기반 정성 분석(SPF 식별·이중화 설계)에 집중하는 것이 현실적이다.

**FMEA 측정자 간 편차 통제**
- **챌린지**: RPN 산정에서 S·O·D 각 값은 주관적 판단에 의존하므로 측정자마다 결과가 달라질 수 있다.
- **제언**: 사전에 S·O·D 등급 정의 기준표를 팀 공통으로 수립하고, 복수 심사자 합의(Consensus) 방식으로 RPN을 결정하면 편차를 30% 이하로 줄일 수 있다.

**AI 기반 시스템의 새로운 위험 분석 과제**
- **챌린지**: AI/ML 시스템은 비결정론적 동작 특성으로 인해 전통적 FTA/FMEA 적용에 구조적 한계가 있다. 적대적 공격(Adversarial Attack), 분포 이탈(Out-of-Distribution) 입력 등 새로운 위험 유형이 등장했다.
- **제언**: STPA(System-Theoretic Process Analysis) 등 현대적 안전 분석 기법을 병행 적용하고, AI 안전성 표준(ISO/IEC 42001) 동향을 지속 모니터링해야 한다.
