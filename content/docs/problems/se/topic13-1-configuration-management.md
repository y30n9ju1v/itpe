## 해설

> **소프트웨어의 변경을 통제하는 제도적 장치**  
> 형상관리(SCM) — 무엇이 언제 왜 바뀌었는지 모든 것을 추적한다

"어제까지 잘 되던 것이 오늘 갑자기 안 된다." 소프트웨어 개발 현장에서 가장 자주 듣는 말입니다. 형상관리(SCM: Software Configuration Management)는 이 혼돈을 질서로 바꾸는 핵심 관리 활동입니다. 단순히 Git을 쓰는 것이 형상관리가 아닙니다. 무엇을 관리할지 식별하고(식별), 변경을 통제하며(통제), 제대로 되었는지 확인하고(감사), 모든 것을 기록(기록)하는 체계 전체가 형상관리입니다.

### 1. 출제 배경 및 의도

형상관리는 SDLC(소프트웨어 개발 생명주기) 전반에 걸쳐 적용되는 핵심 프로세스로, 기술사 시험에서 꾸준히 출제됩니다. 이 문제는 세 가지를 묻고 있습니다.

- **가. 표준 프로세스**: 형상관리를 수행하는 절차와 단계
- **나. 수행 활동**: 형상관리를 구성하는 4대 핵심 활동
- **다. SDLC 단계별 기준선(Baseline)**: 각 개발 단계에서 어떤 산출물이 기준선이 되는가

채점관은 형상관리를 단순히 버전 관리 도구 사용법으로 이해하는 수준과, 조직의 품질 관리 체계 안에서 SCM을 전략적으로 운용할 수 있는 수준을 구분합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 형상(Configuration)이란 무엇인가?

**형상(Configuration)**이란 소프트웨어를 구성하는 모든 산출물의 특성(기능적, 물리적 특성)과 그 관계를 말합니다. 쉽게 말하면 "소프트웨어를 만드는 데 필요한 모든 것"입니다.

**형상 항목(CI: Configuration Item)**은 형상관리의 대상이 되는 개별 단위입니다.

```text
[형상 항목(CI)의 범위]

소스코드
  └→ 모든 .java, .py, .js 파일
  └→ 설정 파일 (application.yml, .env)
  └→ 빌드 스크립트 (pom.xml, Gradle)

산출물 문서
  └→ 요구사항 명세서 (SRS)
  └→ 설계 문서 (HLD, LLD)
  └→ 테스트 계획서, 테스트 케이스

실행 파일
  └→ 컴파일된 바이너리
  └→ 배포 패키지 (.jar, .war, Docker Image)

관련 데이터
  └→ DB 스키마 정의서
  └→ 테스트 데이터
```

#### 기준선(Baseline)의 개념

**기준선(Baseline)**은 특정 시점에 검토·승인된 형상 항목의 공식적인 스냅샷입니다. 기준선이 설정되면 이후 변경은 반드시 공식적인 변경 통제 절차를 거쳐야 합니다.

비유하자면, 기준선은 "이 상태가 우리가 합의한 공식 버전입니다"라는 서명입니다. 기준선 없이는 무엇이 '공식 버전'인지 알 수 없습니다.

### 3. 핵심 키워드 (Must-Have)

#### 가. 형상관리 표준 프로세스

IEEE 828(소프트웨어 형상관리 계획 표준)과 ISO/IEC 12207의 SCM 프로세스를 기반으로 하는 표준 흐름입니다.

```text
[형상관리 표준 프로세스 흐름]

① SCM 계획 수립
   └→ 관리 대상 CI 식별 기준 정의
   └→ 베이스라인 정의
   └→ CCB 구성 (Change Control Board: 변경통제위원회)
   └→ 형상관리 도구 선정 (Git, SVN, ClearCase 등)
   └→ 산출물: SCM 계획서

② 형상 식별 (Configuration Identification)
   └→ 각 CI에 고유 식별자 부여
   └→ 버전 번호 체계 정의 (예: v1.2.3 — Major.Minor.Patch)

③ 형상 통제 (Configuration Control)
   └→ 변경 요청서(CR: Change Request) 접수
   └→ CCB 검토 및 승인/거부 결정
   └→ 승인된 변경만 적용

④ 형상 감사 (Configuration Audit)
   └→ 물리적 감사: 실제 코드가 설계 문서와 일치하는가?
   └→ 기능적 감사: 소프트웨어가 기능 요구사항을 충족하는가?

⑤ 형상 상태 보고 (Configuration Status Accounting)
   └→ 변경 이력 기록 및 보고
   └→ "무엇이, 언제, 왜, 누가 바꿨는가" 추적
```

#### 나. 형상관리 4대 수행 활동

**1) 식별 (Identification)**

형상관리의 시작점. 무엇을 관리할지 결정합니다.

- 모든 CI에 유일한 식별자와 버전 번호 부여
- CI 간의 관계(의존성) 정의
- 버전 명명 규칙: `[Major].[Minor].[Patch]` (시맨틱 버저닝)
  - Major: 하위 호환 불가능한 변경
  - Minor: 하위 호환 가능한 기능 추가
  - Patch: 버그 수정

**2) 통제 (Control)**

승인되지 않은 변경을 차단하는 게이트키퍼 역할.

```text
[변경 통제 프로세스 (CCB)]

변경 필요 발생
      │
      ▼
변경 요청서(CR) 작성
  - 변경 이유
  - 영향 범위 (Impact Analysis)
  - 작업 추정
      │
      ▼
CCB (Change Control Board) 검토
  - 프로젝트 관리자, 품질 관리자, 기술 리드 등으로 구성
  - 비용/일정/품질 영향 평가
      │
      ▼
승인 → 변경 수행 → 테스트 → 기준선 갱신
거부 → 요청자에게 통보 및 사유 설명
```

**3) 감사 (Audit)**

형상관리가 제대로 수행되고 있는지, 산출물이 기준선과 일치하는지 독립적으로 검증.

| 감사 유형 | 목적 | 시점 |
|---------|------|------|
| 기능 형상 감사(FCA) | 소프트웨어가 기능 요구사항을 충족하는가 | 릴리즈 전 |
| 물리 형상 감사(PCA) | 소프트웨어 제품이 관련 문서와 일치하는가 | 릴리즈 전 |
| 프로세스 감사 | SCM 절차가 준수되고 있는가 | 개발 중 주기적 |

**4) 기록 (Status Accounting)**

모든 형상 변경 이력을 추적 가능하게 기록.

- 각 CI의 현재 상태와 변경 이력
- 승인된/거부된 변경 요청 목록
- 결함과 변경 요청의 연계 추적
- **감사 추적(Audit Trail)**: 누가, 언제, 무엇을, 왜 변경했는지 기록

#### 다. SDLC 단계별 기준선과 관리 항목

기준선은 SDLC의 핵심 마일스톤마다 설정됩니다.

| SDLC 단계 | 기준선 명칭 | 기준선 구성 항목 |
|---------|------------|----------------|
| 요구사항 분석 | 기능 기준선 (Functional Baseline) | 요구사항 명세서(SRS), 시스템 아키텍처 문서 |
| 설계 | 할당 기준선 (Allocated Baseline) | 기능 설계서(HLD), 인터페이스 명세서 |
| 구현/통합 | 제품 기준선 (Product Baseline) | 소스코드, 상세 설계서(LLD), 테스트 결과 |
| 테스트/배포 | 운영 기준선 (Operational Baseline) | 실행 파일, 운영 매뉴얼, 릴리즈 노트 |

```text
[SDLC와 기준선 관계도]

요구사항 → [기능 기준선] → 설계 → [할당 기준선] → 구현 → [제품 기준선] → 배포 → [운영 기준선]
              ↑                      ↑                    ↑                      ↑
          (SRS 승인)           (HLD 승인)            (코드 동결)            (릴리즈 승인)
```

각 기준선은 CCB의 공식 승인을 통해 설정되며, 이후 변경은 반드시 변경 통제 절차를 통해야 합니다.

### 4. 맥락과 비교

**Git을 활용한 형상관리 실천 — 브랜치 전략**

현대 소프트웨어 개발에서 Git은 형상관리의 핵심 도구입니다. 브랜치 전략은 기준선 개념을 코드 레벨에서 구현합니다.

```text
[Git Flow 브랜치 전략]

main (운영 기준선)
  └→ 항상 배포 가능한 상태 유지
  └→ 직접 커밋 금지, Pull Request만 허용

develop (통합 기준선)
  └→ 다음 릴리즈를 위한 통합 브랜치
  └→ feature 브랜치들이 이곳으로 병합

feature/* (기능 개발)
  └→ 각 기능별 독립적 작업 공간
  └→ 완료 후 develop에 PR → 코드 리뷰 → 병합

hotfix/* (긴급 수정)
  └→ main에서 분기, 수정 후 main+develop 모두 병합
  └→ 운영 기준선의 긴급 변경 통제
```

이 구조에서 `main` 브랜치의 각 태그(v1.0.0, v1.1.0)가 공식 기준선에 해당합니다.

**형상관리 vs 버전 관리 비교**

| 구분 | 버전 관리(VCS) | 형상관리(SCM) |
|------|--------------|-------------|
| 범위 | 소스코드 파일의 변경 이력 관리 | 모든 CI(소스코드+문서+설정+바이너리) 관리 |
| 통제 | 기술적 도구 중심 | 프로세스와 거버넌스 포함 |
| 감사 | 없음 | CCB, 형상 감사 포함 |
| 표준 | 없음 | IEEE 828, ISO/IEC 12207 |

버전 관리(Git)는 형상관리의 '식별'과 '기록' 활동을 지원하는 도구이며, 형상관리는 이를 포함하는 더 넓은 관리 체계입니다.

### 5. 실무 제언

**CI/CD 파이프라인과 형상관리의 통합**

현대 DevOps 환경에서 형상관리는 CI/CD 파이프라인과 긴밀히 연동됩니다.

- **Infrastructure as Code(IaC)**: 서버 인프라 설정(Terraform, Ansible)도 형상 항목으로 관리. 인프라 변경도 코드 변경과 동일하게 PR → 리뷰 → 병합 프로세스 적용
- **GitOps**: Git 저장소가 운영 환경의 단일 진실 원천(Single Source of Truth). Git 변경이 자동으로 운영 배포를 트리거
- **불변 인프라(Immutable Infrastructure)**: 배포된 서버를 수정하지 않고, 새 이미지로 교체. 이 자체가 기준선 관리의 현대적 구현

**규모별 SCM 전략**

소규모 팀(5인 이하)에서 엄격한 CCB 프로세스를 강제하는 것은 오버헤드가 됩니다. 조직 규모와 위험도에 맞는 SCM 전략이 필요합니다.

- **소규모**: GitHub Flow (main + feature 브랜치), PR 리뷰로 가벼운 통제
- **중규모**: Git Flow, 릴리즈 매니저 지정
- **대규모/고위험**: 엄격한 CCB, 공식 감사, 기준선 동결(Code Freeze) 기간 운영
