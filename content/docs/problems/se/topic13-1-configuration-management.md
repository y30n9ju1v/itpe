## 해설

### 출제 배경 및 의도

"어제까지 잘 되던 것이 오늘 갑자기 안 된다." 소프트웨어 개발 현장에서 반복되는 이 혼돈을 질서로 바꾸는 것이 형상관리(SCM: Software Configuration Management)이다. 단순히 Git을 쓰는 것이 형상관리가 아니라, 무엇을 관리할지 식별(식별)하고, 변경을 통제(통제)하며, 제대로 되었는지 확인(감사)하고, 모든 것을 기록(기록)하는 체계 전체가 형상관리이다.

이 문제는 표준 프로세스(가)·4대 수행 활동(나)·SDLC 단계별 기준선(다)의 세 관점을 종합적으로 묻는다. 채점관은 형상관리를 버전 관리 도구 사용법으로 이해하는 수준과, CCB·기준선·감사 추적을 포함하는 거버넌스 체계로 이해하는 수준을 명확히 구분한다.

### 1. 형상관리(SCM) 개요

정 의  • 형상관리(SCM)를 소프트웨어를 구성하는 모든 형상 항목(CI)을 식별·통제·감사·기록하여, 무엇이 언제 왜 바뀌었는지 추적하고 변경을 통제하는 체계적 관리 활동이라 한다(IEEE 828, ISO/IEC 12207).
       - 버전 관리(VCS)가 소스코드 파일의 변경 이력만 관리한다면, SCM은 문서·설정·바이너리까지 포함하는 더 넓은 관리 체계이다.

```
[SCM 표준 프로세스 흐름도]

① SCM 계획 수립
   └─ CI 식별 기준 정의 / CCB 구성 / 도구 선정(Git, SVN)
           │
           ▼
② 형상 식별(CI 식별 + 버전 번호 체계)
   └─ Major.Minor.Patch 시맨틱 버저닝
           │
           ▼
③ 형상 통제(CCB 기반 변경 통제)
   └─ CR 접수 → CCB 검토/승인 → 변경 수행 → 기준선 갱신
           │
           ▼
④ 형상 감사(FCA + PCA)
   └─ 기능 형상 감사(FCA): 기능 요구사항 충족 여부
   └─ 물리 형상 감사(PCA): 제품과 문서 일치 여부
           │
           ▼
⑤ 형상 상태 보고(Audit Trail)
   └─ "누가, 언제, 무엇을, 왜 변경했는가" 추적
```

- 기준선(Baseline)은 CCB 공식 승인을 통해 설정된 CI의 스냅샷이며, 이후 변경은 반드시 변경 통제 절차를 거쳐야 한다.

### 2. 형상관리 4대 수행 활동 및 SDLC 기준선

1) 나. 형상관리 4대 수행 활동

| 활동 | 목적 | 핵심 내용 |
|------|------|---------|
| ① 식별(Identification) | 무엇을 관리할지 결정 | CI 고유 식별자 부여, 시맨틱 버저닝(Major.Minor.Patch) |
| ② 통제(Control) | 승인되지 않은 변경 차단 | CR 접수 → CCB 검토 → 승인/거부 → 기준선 갱신 |
| ③ 감사(Audit) | 형상관리 준수 여부 독립 검증 | FCA(기능), PCA(물리), 프로세스 감사 |
| ④ 기록(Status Accounting) | 모든 변경 이력 추적 | Audit Trail: 누가·언제·무엇을·왜 변경 |

- CCB(Change Control Board): 프로젝트 관리자·품질 관리자·기술 리드로 구성된 변경 심의 위원회로, 변경의 비용/일정/품질 영향을 평가하여 승인 여부를 결정한다.

2) 다. SDLC 단계별 기준선

| SDLC 단계 | 기준선 명칭 | 구성 항목 | 설정 트리거 |
|---------|----------|---------|-----------|
| 요구사항 분석 | 기능 기준선(Functional Baseline) | SRS, 시스템 아키텍처 문서 | SRS 승인 |
| 설계 | 할당 기준선(Allocated Baseline) | HLD(기능 설계서), 인터페이스 명세서 | HLD 승인 |
| 구현/통합 | 제품 기준선(Product Baseline) | 소스코드, LLD, 테스트 결과 | 코드 동결 |
| 테스트/배포 | 운영 기준선(Operational Baseline) | 실행 파일, 운영 매뉴얼, 릴리즈 노트 | 릴리즈 승인 |

```
[SDLC와 기준선 관계도]
요구사항 → [기능 기준선] → 설계 → [할당 기준선] → 구현 → [제품 기준선] → 배포 → [운영 기준선]
              ↑(SRS 승인)       ↑(HLD 승인)          ↑(코드 동결)         ↑(릴리즈 승인)
```

### 3. 형상관리 vs 버전 관리 비교 및 DevOps 연계

1) SCM vs VCS 비교

| 구분 | 버전 관리(VCS) | 형상관리(SCM) |
|------|-------------|------------|
| 범위 | 소스코드 변경 이력 | 소스코드+문서+설정+바이너리 모든 CI |
| 통제 | 기술적 도구 중심 | 프로세스와 거버넌스 포함 |
| 감사 | 없음 | CCB, FCA/PCA 포함 |
| 표준 | 없음 | IEEE 828, ISO/IEC 12207 |

2) 현대 DevOps에서의 SCM 구현

```
Git Flow 기준선 구현:
main (운영 기준선) ← PR+코드리뷰만 허용, 직접 커밋 금지
develop (통합 기준선) ← feature/* 브랜치 병합 대상
hotfix/* (긴급 변경) ← main에서 분기, main+develop 모두 병합
각 릴리즈 태그(v1.0.0) = 공식 기준선
```

- IaC(Infrastructure as Code): Terraform·Ansible도 형상 항목으로 관리하여 인프라 변경도 PR→리뷰→병합 프로세스 적용
- GitOps: Git 저장소가 운영 환경의 단일 진실 원천(Single Source of Truth)으로, Git 변경이 배포를 자동 트리거한다.  "끝"

### 실무 제언

**규모별 SCM 강도 조절**
- **챌린지**: 소규모 팀(5인 이하)에서 엄격한 CCB 프로세스는 오버헤드가 된다.
- **제언**: 소규모는 GitHub Flow(PR 리뷰), 중규모는 Git Flow(릴리즈 매니저), 대규모·고위험은 공식 CCB·Code Freeze 기간 운영으로 조직 규모와 위험도에 맞는 SCM 전략을 수립한다.

**문서와 코드의 형상 불일치 문제**
- **챌린지**: 소스코드는 최신이지만 설계 문서·테스트 케이스가 갱신되지 않아 PCA에서 불일치가 발견된다.
- **제언**: '문서 as Code' 접근으로 설계 문서를 Git에 함께 관리하고, CI 파이프라인에서 문서 버전과 코드 버전을 자동 검증하면 불일치를 조기에 탐지할 수 있다.

**GitOps와 불변 인프라 적용**
- **챌린지**: 운영 서버를 수동으로 설정 변경하면 형상 불일치가 발생하고 추적이 불가능하다.
- **제언**: 불변 인프라(Immutable Infrastructure) 전략으로 서버 수정 대신 새 이미지로 교체하고, Terraform 상태를 Git으로 관리하면 인프라 변경도 완전한 Audit Trail이 확보된다.
