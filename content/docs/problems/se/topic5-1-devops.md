## 해설

### 출제 배경 및 의도

클라우드 네이티브 환경과 MSA 도입이 일반화된 현재, 시스템이 수십~수백 개의 서비스로 쪼개지면서 매일 수십 번의 배포를 안전하게 수행해야 하는 요구가 발생했다. 과거 한 달에 한 번 새벽 수동 배포 방식은 원천적으로 불가능해졌으며, DevOps는 선택이 아닌 생존을 위한 필수 인프라가 되었다.

채점관이 DevOps 개념·저해 요인·방법론을 묻는 의도는 Jenkins·Docker 도구를 다루는 '엔지니어' 수준을 넘어, 조직의 문화적 결함과 부서 이기주의를 진단하고 시스템적으로 치유하는 변화 관리자(Change Agent)로서의 CTO급 관점을 가졌는지 검증하는 것이다.

### 1. 데브옵스(DevOps) 개요

정 의  • 소프트웨어 개발(Development)과 운영(Operations)의 합성어로, 개발·운영 담당자가 상호 협력·융합하여 빌드~배포~모니터링 전 수명주기를 자동화하고 가시성을 확보함으로써 고품질 서비스를 지속적으로 빠르게 제공하는 조직 문화이자 방법론
       - 개발과 운영이 뫼비우스의 띠처럼 연결되는 피드백 루프 아키텍처

```
[DevOps 무한 피드백 루프]

Dev (개발 영역)
Plan → Code → Build → Test  ──────▶  지속적 통합(CI)

Ops (운영 영역)
Release → Deploy → Operate → Monitor  ──▶  지속적 배포(CD)
     ▲                                           │
     └──────────── 피드백 루프(Feedback) ◀────────┘
```

- DevOps는 도구 체계가 아닌 조직 문화와 프로세스 혁신이 본질이다

### 2. 가) 데브옵스 성공적 도입 저해 요인

1) 부서 간 사일로(Silo) 현상 및 이해 충돌

| 구분 | 항목 | 설명 |
|------|------|------|
| 개발 부서 KPI | 기능 배포 속도 | 신기능을 최대한 빠르게 배포하는 것이 목표 |
| 운영 부서 KPI | 시스템 안정성 | 중단 없이 보수적으로 시스템 유지하는 것이 목표 |
| 리스크 | Silo Effect | 코드 수락 거부·검증 지연으로 협업 문화 시작부터 좌초 |

- 목표 KPI의 사상적 충돌이 DataOps 도입 전 선결해야 할 조직 거버넌스 과제다

2) 도구 중심 맹신 및 규제·리더십 저해 요인

| 구분 | 저해 요인 | 리스크 |
|------|----------|--------|
| 도구 과신 | 문화 변화 없이 Kubernetes·ArgoCD·Terraform만 도입 | 숙련도 부재 시 기술 부채·비용 오버헤드 증가 |
| 규제 충돌 | 금융·공공 도메인 직무 분리(Segregation of Duties) 강제 | 수동 결재 단계 개입으로 자동화 흐름 단절 |
| 단기 성과주의 | 경영진의 월별 기능 출시 건수 압박 | 테스트 코드 생략, 파이프라인 우회 만연 |

- DevOps 실패의 본질은 기술 문제가 아닌 조직 구조와 인간 관성의 문제다

### 3. 나) 데브옵스 방법론 — Three Ways & CALMS

1) Gene Kim의 Three Ways(3대 핵심 원칙)

| 구분 | 원칙 | 설명 |
|------|------|------|
| 제1 원칙 | 흐름(Flow) | 요구→개발→운영 속도 극대화, Kanban·Small Batch 실천 |
| 제2 원칙 | 피드백(Feedback) | 운영 장애·모니터링 로그가 개발로 초고속 동기화, 즉각 코드 수정 |
| 제3 원칙 | 지속적 학습(Learning) | Blameless Post-Mortem 문화, 카오스 엔지니어링으로 복원력 향상 |

- Three Ways는 DevOps의 철학적 기반이며, 도구 도입 전 조직이 먼저 내면화해야 한다

2) CALMS 성숙도 프레임워크

| 구분 | 요소 | 설명 |
|------|------|------|
| C | Culture(문화) | 개발·운영이 '하나의 제품'을 공동 책임지는 상호 신뢰 문화 |
| A | Automation(자동화) | 수동 공정 제거, CI/CD 파이프라인·IaC(인프라 코드화) 의무화 |
| L | Lean(린) | 가치 흐름 지도(Value Stream Mapping)로 낭비 요인 제거 |
| M | Measurement(측정) | 배포 빈도·변경 실패율·MTTR 등 DORA 메트릭 상시 계측 |
| S | Sharing(공유) | 성공·실패 교훈 및 툴체인 노하우 조직 전체 투명 공유 |

- CALMS는 DevOps 성숙도를 정량 평가하고 도입 로드맵을 수립하는 실천 프레임워크다.  "끝"

### 실무 제언

**DevSecOps로의 진화 — 보안을 파이프라인 내로 내장**
- **챌린지**: 배포 속도 가속화로 인해 사후 수동 보안성 심의 방식이 파이프라인의 병목 구간이 된다
- **제언**: 개발자 코드 Push 즉시 SAST 도구(SonarQube)·오픈소스 취약점 스캐너(Trivy)가 자동 트리거되는 DevSecOps 파이프라인을 구축하여 속도와 보안성을 동시에 달성한다

**SRE 에러 버젯 연계로 운영 안정성 담보**
- **챌린지**: 개발팀에 배포 전권을 부여하면 운영팀은 가동성 저하 불안감에 시달리며, 단순 협업 구호만으로는 간극을 메울 수 없다
- **제언**: SRE의 에러 버젯(Error Budget = 100% - SLO) 메커니즘을 거버넌스에 도입하여, 버젯 소진 시 신규 기능 배포를 자동 차단하고 안정성 강화에 강제 투입하는 과학적 완충 규칙을 경영진 합의 문서에 명문화한다
