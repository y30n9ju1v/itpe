## 해설

> **단 1초의 다운타임도 허용하지 않는 서비스 진화**  
> 무중단 배포 기법(Zero-Downtime Deployment)의 모든 것

과거의 소프트웨어 배포는 주로 이용자가 적은 새벽 시간에 서버를 완전히 내린 뒤 진행하는 것이 상식이었습니다. 마치 공항을 보수하려면 비행기 운항을 모두 멈춰야 했던 것처럼요. 그러나 글로벌 서비스가 보편화되고 MSA 및 DevOps 환경이 정착된 현재, 시스템 점검을 위해 서비스를 중단하는 것은 심각한 매출 손실과 비즈니스 신뢰도 저하를 야기합니다. 사용자가 서비스를 실시간으로 이용하는 도중에도 시스템 아키텍처를 안전하게 업그레이드할 수 있는 기술이 바로 무중단 배포 기법(Zero-Downtime Deployment)입니다.

### 1. 출제 배경 및 의도

현대 클라우드 네이티브 환경에서는 하루에도 수십 번씩 비즈니스 기능이 배포 및 업데이트됩니다. 채점관이 무중단 배포 기법을 묻는 의도는 수험생이 단순히 "서버를 앙상블로 묶어 무중단으로 배포한다"는 추상적인 개념에 머물러 있는지, 아니면 각 배포 기법이 가진 트래픽 제어 메커니즘, 리소스 비용, 리스크 관리(Rollback) 측면의 장단점을 명확히 비교하고 적절한 아키텍처적 솔루션을 처방할 수 있는지 검증하기 위함입니다.

### 2. 개념의 본질과 작동 메커니즘

무중단 배포 기법의 본질은 "로드 밸런서(Load Balancer, 들어오는 트래픽을 여러 서버에 분산시켜 주는 장치) 또는 리버스 프록시(Reverse Proxy, 클라이언트 요청을 받아 내부 서버로 전달하는 중간 서버) 레이어에서 트래픽 라우팅을 동적으로 제어하여, 구버전(V1)에서 신버전(V2)으로 사용자의 요청을 단절 없이 스위칭하는 인프라 제어 기술"입니다.

이 메커니즘을 성공적으로 구현하기 위해서는 가용 서버 그룹의 논리적 격리, 헬스 체크(Health Check)를 통한 인스턴스 상태 모니터링, 그리고 예기치 못한 장애 발생 시 1초 만에 배포 전 상태로 복구할 수 있는 고속 롤백(Rollback) 체계가 반드시 연동되어야 합니다.

### 3. 무중단 배포 기법의 3대 종류 및 메커니즘

#### 가. 블루-그린 배포 (Blue-Green Deployment)

- **개념**: 운영 중인 구버전 환경과 똑같은 전 가용량(Full Capacity)의 신버전 인프라 환경을 별도로 구축하여, 로드 밸런서의 포인터를 한순간에 전수 전환하는 방식

```text
[현재]  LB ──▶ [Blue: V1 × 4대] (운영 트래픽)
                [Green: V2 × 4대] (배포/테스트 중)

[전환]  LB ──▶ [Green: V2 × 4대] (운영 트래픽)
                [Blue: V1 × 4대] (롤백 대기)
```

- **장점**: 구버전과 신버전이 섞이지 않아 데이터 정합성 관리가 비교적 단순하며, 문제가 생겼을 때 포인터를 다시 블루로 돌리기만 하면 되므로 롤백(Rollback) 속도가 극도로 빠릅니다.
- **단점**: 시스템 전체 자원을 동일하게 두 세트 구성해야 하므로 인프라 비용이 정확히 2배로 소모됩니다.

#### 나. 롤링 배포 (Rolling Deployment)

- **개념**: 기존 운영 서버 그룹의 인스턴스를 하나씩 혹은 일정 비율로 나누어 점진적으로 구버전을 신버전으로 교체해 나가는 방식

```text
[1단계] LB ──▶ [V1] [V1] [V1] [V1]     (4대 운영)
[2단계] LB ──▶ [V2] [V1] [V1] [V1]     (1번 서버 교체 완료)
[3단계] LB ──▶ [V2] [V2] [V1] [V1]     (2번 서버 교체 완료)
[4단계] LB ──▶ [V2] [V2] [V2] [V2]     (전수 교체 완료)
```

- **장점**: 추가적인 여유 인프라를 많이 필요로 하지 않으므로 자원 효율성과 비용 절감 측면에서 매우 우수합니다.
- **단점**: 배포가 진행되는 동안 전체 가용 인스턴스 수가 일시적으로 줄어들어 나머지 서버에 트래픽 부하가 가중될 수 있으며, 구버전과 신버전이 동시에 유저의 요청을 처리하는 공존 기간(Dual Version State)이 발생하므로 하위 호환성 관리가 필수적입니다.

#### 다. 카나리 배포 (Canary Deployment)

- **개념**: 신버전 소프트웨어를 전체 트래픽의 소수 비율(예: 5%~10%)에만 먼저 노출시켜 모니터링한 후, 안전성이 검증되면 점진적으로 노출 범위를 확대하는 위험 통제형 배포 기법 (과거 광부들이 유독가스를 감지하기 위해 카나리아 새를 탄광에 들여보냈던 역사적 사실에서 유래)

```text
[1단계] LB ──▶ V1: 95% / V2: 5%   (소수 유저 노출, SLI 모니터링)
[2단계] LB ──▶ V1: 80% / V2: 20%  (이상 없으면 비율 상향)
[3단계] LB ──▶ V1: 50% / V2: 50%  (지속 모니터링)
[4단계] LB ──▶ V1:  0% / V2: 100% (전수 전환 완료)
```

- **장점**: 신버전 코드의 잠재적 버그나 아키텍처적 결함이 유발하는 피해 범위(Blast Radius)를 최소한으로 격리 및 통제할 수 있습니다.
- **단점**: 사용자별로 구버전과 신버전 화면을 번갈아 보며 혼란을 겪지 않도록 세션 고정(Sticky Session)이나 라우팅 정책 제어가 정교하게 설계되어야 하므로 인프라 복잡도가 매우 높습니다.

### 4. 맥락과 비교

| 비교 항목 | 블루-그린 배포 (Blue-Green) | 롤링 배포 (Rolling) | 카나리 배포 (Canary) |
|----------|--------------------------|-------------------|-------------------|
| 인프라 비용 | 매우 높음 (자원 2배 요구) | 낮음 (여유 자원 최소화 가능) | 낮음~중간 |
| 롤백 난이도 | 극히 낮음 (라우터 포인터 즉시 전환) | 높음 (업데이트된 서버를 역순 재배포) | 낮음 (가중치 라우팅 0% 조정) |
| 버전 공존 여부 | 없음 (순간 완전 전환) | 있음 (배포 기간 내내 상존) | 있음 (의도적 장기 공존 및 모니터링) |
| 위험 통제 수준 | 중간 (결함 시 전체 유저 일시 영향) | 중간 (순차적 결함 인지 가능) | 최고 우수 (피해 범위 소수 격리 가능) |
| 핵심 제어 대상 | 로드 밸런서 스위칭 구조 | 인스턴스 그룹별 순차적 라이프사이클 | API 게이트웨이 가중치 라우팅 엔진 |

### 5. 실무 제언

**무중단 배포의 가장 거대한 장벽, 데이터베이스 스키마 호환성**

- **챌린지**: 대다수의 아키텍트들이 애플리케이션 서버(응용 레이어)의 무중단 배포 기법만 신경 쓰다가 정작 데이터베이스 스키마가 변경(예: 특정 테이블 컬럼 삭제 또는 타입 변경)될 때 배포 프로세스를 깨뜨리는 실책을 범합니다. 롤링이나 카나리 배포 중 구버전 소스코드가 바뀐 DB 구조를 읽으려 하거나, 반대로 신버전 코드가 예전 DB 스키마에 데이터를 밀어 넣으며 SQL Exception 장애가 대량 유발됩니다.
- **제언**: 데이터베이스의 무중단을 달성하기 위해 'Expand and Contract(확장 및 축소)' 패러다임을 의무화해야 합니다. 테이블 구조를 바꿀 때 1단계: 신규 컬럼 추가(Expand) → 2단계: 신버전 애플리케이션 배포(V1, V2 모두 새 컬럼 인식) → 3단계: 기존 데이터를 새 컬럼으로 마이그레이션 → 4단계: 기존 구버전 컬럼 삭제(Contract) 순으로 최소 3~4회의 단계별 릴리즈로 쪼개어 배포하는 스키마 거버넌스 가이드라인을 정립해야만 진정한 완전 무중단 시스템이 완성됩니다.

**관측 가능성(Observability) 기반의 자동 롤백(Automated Rollback) 아키텍처 연계**

- **챌린지**: 카나리나 롤링 배포를 적용하더라도 엔지니어가 대시보드를 수동으로 모니터링하다가 롤백 버튼을 늦게 누르면 대형 서비스 장애를 막을 수 없습니다. 인지 부하가 걸린 인간의 판단력은 실시간 트래픽 환경에서 병목이 됩니다.
- **제언**: Prometheus 및 프로그레시브 딜리버리 도구인 Argo Rollouts를 결합하여 무중단 배포 파이프라인을 자동화해야 합니다. 카나리 배포 진행 중, Argo Rollouts이 수집되는 실시간 SLI 지표(예: HTTP 5XX 에러율 > 1%)를 모니터링하여 메트릭 분석이 무너지는 즉시 인간의 개입 없이 시스템이 알아서 배포를 중단하고 0.1초 만에 자동으로 롤백(Automated Rollback Circuit Breaker)을 수행하는 실전적 통제 시스템을 구축해야 합니다.
