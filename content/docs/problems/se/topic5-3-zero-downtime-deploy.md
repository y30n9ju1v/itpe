## 해설

### 출제 배경 및 의도

글로벌 서비스가 보편화되고 MSA·DevOps 환경이 정착된 현재, 새벽 서버 중단 배포 방식은 심각한 매출 손실과 비즈니스 신뢰도 저하를 야기한다. 하루 수십 번 배포가 요구되는 클라우드 네이티브 환경에서 무중단 배포(Zero-Downtime Deployment)는 선택이 아닌 필수 인프라 전략이다.

채점관의 출제 의도는 "서버를 앙상블로 묶어 무중단 배포"라는 추상적 개념에 머무는지, 각 배포 기법의 트래픽 제어 메커니즘·리소스 비용·롤백 측면의 장단점을 비교하고 상황에 맞는 아키텍처 솔루션을 처방할 수 있는지 검증하는 것이다.

### 1. 무중단 배포 개요

정 의  • 로드 밸런서(LB) 또는 리버스 프록시 레이어에서 트래픽 라우팅을 동적으로 제어하여, 구버전(V1)에서 신버전(V2)으로 사용자 요청을 단절 없이 스위칭하는 인프라 제어 기술
       - 헬스 체크(Health Check)·고속 롤백(Rollback) 체계와 반드시 연동 필요

```
[무중단 배포 핵심 원리]

  사용자 요청
      │
      ▼
  Load Balancer / Reverse Proxy
  (트래픽 라우팅 동적 제어)
      │
      ├──▶ [V1 서버 그룹] ← 구버전 (단계적 제거)
      │
      └──▶ [V2 서버 그룹] ← 신버전 (단계적 투입)

  ① Health Check 지속 모니터링
  ② 이상 감지 시 Rollback 자동 수행
```

- 무중단 배포의 본질은 LB/프록시 계층의 트래픽 라우팅 정밀 제어에 있다

### 2. 무중단 배포 기법의 3대 종류

1) 블루-그린 배포 (Blue-Green Deployment)

```
[현재]  LB ──▶ [Blue: V1 × 4대] (운영 트래픽)
                [Green: V2 × 4대] (배포/테스트 중)

[전환]  LB ──▶ [Green: V2 × 4대] (운영 트래픽)
                [Blue: V1 × 4대] (롤백 대기)
```

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 전체 교체 | 동일 규모의 신버전 환경을 별도 구축 후 LB 포인터 순간 전환 |
| 장점 | 롤백 최속 | 포인터를 Blue로 되돌리기만 하면 즉시 복구 |
| 단점 | 비용 2배 | 전체 자원을 동일하게 두 세트 유지 필요 |

- 데이터 정합성 관리가 단순하고 롤백이 즉각적이나 인프라 비용 부담이 크다

2) 롤링 배포 (Rolling Deployment)

```
[1단계] LB ──▶ [V1] [V1] [V1] [V1]   (4대 운영)
[2단계] LB ──▶ [V2] [V1] [V1] [V1]   (1번 교체)
[3단계] LB ──▶ [V2] [V2] [V1] [V1]   (2번 교체)
[4단계] LB ──▶ [V2] [V2] [V2] [V2]   (완료)
```

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 순차 교체 | 인스턴스를 하나씩(또는 일정 비율) 점진적으로 V2로 교체 |
| 장점 | 비용 절감 | 추가 인프라 최소화, 자원 효율성 우수 |
| 단점 | 버전 공존 | 배포 중 V1·V2 동시 운영 → 하위 호환성 관리 필수 |

- 비용 효율적이나 배포 중 용량 일시 감소 및 구/신버전 공존 구간 발생에 유의한다

3) 카나리 배포 (Canary Deployment)

```
[1단계] LB ──▶ V1: 95% / V2:  5%  (소수 유저 노출, SLI 모니터링)
[2단계] LB ──▶ V1: 80% / V2: 20%  (이상 없으면 비율 상향)
[3단계] LB ──▶ V1: 50% / V2: 50%  (지속 모니터링)
[4단계] LB ──▶ V1:  0% / V2: 100% (전수 전환 완료)
```

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 가중치 제어 | 소수 트래픽(5~10%)에 먼저 노출 후 점진적 비율 확대 |
| 장점 | 위험 최소화 | 피해 범위(Blast Radius)를 소수 유저로 격리·통제 |
| 단점 | 복잡도 높음 | Sticky Session·가중치 라우팅 정교한 설계 필요 |

- 신버전 결함의 영향 범위를 최소화할 수 있는 가장 안전한 위험 통제형 배포 기법이다.  "끝"

### 실무 제언

**데이터베이스 스키마 무중단 — Expand and Contract 패턴**
- **챌린지**: 애플리케이션 레이어 무중단에 집중하다가 DB 스키마 변경(컬럼 삭제·타입 변경) 시 롤링·카나리 배포 중 SQL Exception 장애가 대량 발생한다
- **제언**: Expand and Contract 패턴 의무화 — ①신규 컬럼 추가(Expand) → ②신버전 배포(V1·V2 모두 새 컬럼 인식) → ③기존 데이터 마이그레이션 → ④구버전 컬럼 삭제(Contract) 순으로 3~4회 단계적 릴리즈

**Observability 기반 자동 롤백(Automated Rollback) 아키텍처**
- **챌린지**: 수동 모니터링으로는 실시간 트래픽 환경에서 롤백 타이밍을 놓쳐 대형 장애로 확대된다
- **제언**: Argo Rollouts + Prometheus를 결합하여, 카나리 배포 중 HTTP 5XX 에러율 > 1% 감지 시 인간 개입 없이 0.1초 내 자동 롤백(Automated Rollback Circuit Breaker)이 실행되는 완전 자동화 통제 체계를 구축한다
