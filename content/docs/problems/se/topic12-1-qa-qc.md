## 문제

**품질보증(QA : Quality Assurance), 품질통제(QC : Quality Control)**에 대하여 설명하시오.

## 해설

> **예방하는 자와 발견하는 자**  
> QA(품질보증)는 프로세스를 관리하고, QC(품질통제)는 결과물을 검사한다

품질은 소프트웨어 개발의 최종 목적지입니다. 그런데 "어떻게 하면 좋은 품질을 얻을 수 있는가?"에 대한 두 가지 철학적 접근이 있습니다. QA(Quality Assurance, 품질보증)는 "제대로 된 프로세스를 밟으면 좋은 결과가 나온다"는 예방 철학이고, QC(Quality Control, 품질통제)는 "결과물을 직접 검사하여 불량을 걸러낸다"는 검사 철학입니다. 현대 소프트웨어 공학은 이 두 접근의 균형을 요구합니다.

### 1. 출제 배경 및 의도

114회 기술사 시험에서 QA와 QC를 묻는 문제는 이 개념을 단순히 정의만 외우는 것을 넘어, **실제 프로젝트에서 어떻게 적용하는지** 그리고 **두 개념의 차이를 정확히 구분**할 수 있는지를 검증합니다.

많은 사람이 QA와 QC를 혼용하거나, QA를 단순히 "테스팅"으로 잘못 이해합니다. 채점관은 이 혼동을 정확히 짚고 넘어갑니다.

- QA ≠ 테스팅 (QA는 테스팅을 포함하지만, 테스팅은 QC의 영역)
- QA = 프로세스 심사, 표준 준수, 예방
- QC = 결과물 검사, 결함 발견, 수정

### 2. 개념의 본질과 작동 메커니즘

#### QA vs QC 본질 비교

```text
[QA와 QC의 위치]

        [입력]          [프로세스]         [출력]
요구사항  →  ┌──────────────────────────┐  →  소프트웨어
            │   개발 / 테스트 / 빌드    │
            └──────────────────────────┘
                 ▲                ▲
                 │                │
              [QA]              [QC]
         프로세스를 감시      결과물을 검사
         "올바른 방법으로    "올바른 것을
          개발하고 있는가?"   만들었는가?"
         예방(Prevention)   발견/수정(Detection/Correction)
```

**비유: 식품 제조 공장**

- **QA**: 공장의 위생 규정, 레시피 표준화, 직원 교육, HACCP 프로세스 — 불량이 생기지 않도록 프로세스를 관리
- **QC**: 출하 전 제품 샘플링 검사, 이물질 검출기 — 불량품이 고객에게 나가지 않도록 검사

#### 품질 관리 프레임워크 전체 그림

```text
[품질 관리 프레임워크]

품질 계획 (Quality Planning)
  └→ "무엇이 품질인가?" 정의
  └→ 품질 목표, 기준, 측정 방법 설정
  └→ 산출물: 품질 관리 계획서(QMP)
         │
         ▼
품질 보증 (QA: Quality Assurance)
  └→ 프로세스가 계획대로 수행되고 있는가 감시
  └→ 프로세스 감사(Process Audit), 표준 준수 확인
  └→ 산출물: 감사 보고서, 개선 권고안
         │
         ▼
품질 통제 (QC: Quality Control)
  └→ 개별 산출물이 기준을 충족하는가 검사
  └→ 테스트, 인스펙션, 리뷰, 검수
  └→ 산출물: 결함 보고서, 수정 완료 확인서
         │
         ▼
품질 개선 (Quality Improvement)
  └→ QA/QC 결과를 분석하여 프로세스 개선
  └→ PDCA 사이클(Plan-Do-Check-Act)
  └→ CMM/CMMI 성숙도 레벨 향상
```

### 3. 핵심 키워드 (Must-Have)

#### 1) QA (Quality Assurance, 품질보증)

**정의**: 소프트웨어가 의도한 품질 목표를 달성할 수 있도록 개발 프로세스를 계획하고, 수행하며, 감시하는 체계적인 활동

**핵심 특성**
- **예방 중심**: 결함이 발생하기 전에 프로세스를 통제
- **프로세스 지향**: 특정 산출물이 아닌 개발 방법 자체를 검토
- **전 생명주기**: 요구사항 정의부터 유지보수까지 전 단계 관여
- **독립성**: 개발팀과 독립적인 QA 조직이 감사 수행

**SQA (Software Quality Assurance) 주요 활동**

| 활동 | 내용 | 시점 |
|------|------|------|
| 프로세스 감사 | 표준/절차 준수 여부 검토 | 개발 중 |
| 요구사항 검토 | 요구사항의 완전성, 일관성, 추적성 확인 | 분석 단계 |
| 설계 검토 | 아키텍처 기준 준수, 결함 조기 발견 | 설계 단계 |
| 테스트 계획 검토 | 테스트 전략의 적정성 확인 | 테스트 계획 시 |
| 형상 감사 | CI(형상 항목) 변경 관리 적절성 확인 | 전 단계 |

**ISO 9001과 QA**

ISO 9001은 품질경영시스템(QMS) 국제 표준으로, QA의 핵심 프레임워크입니다. 소프트웨어 분야에서는 ISO 9001을 기반으로 한 ISO/IEC 90003이 적용됩니다. 핵심은 "말한 대로 하고, 한 대로 기록한다(Say what you do, Do what you say, Record what you did)"입니다.

#### 2) QC (Quality Control, 품질통제)

**정의**: 소프트웨어 산출물이 정해진 품질 기준을 충족하는지 검사하고, 기준에 미달하는 결함을 발견하여 수정하는 활동

**핵심 특성**
- **발견/수정 중심**: 이미 만들어진 결과물에서 결함을 찾아냄
- **제품 지향**: 특정 산출물(코드, 문서, 테스트 결과)을 검사
- **측정 가능**: 결함 밀도(Defect Density), 결함 제거율(DRE) 등 수치로 측정

**소프트웨어 QC의 대표 기법**

| 기법 | 설명 | 특징 |
|------|------|------|
| 인스펙션 (Inspection) | 공식적인 동료 검토 프로세스 (Fagan Inspection) | 가장 체계적, 결함 발견율 높음 |
| 워크스루 (Walkthrough) | 작성자가 팀에게 설명하며 비공식 검토 | 교육 효과, 비교적 가벼움 |
| 테스팅 (Testing) | 코드 실행하여 예상 결과와 비교 | 동적 검증 |
| 정적 분석 (Static Analysis) | 코드 실행 없이 도구로 분석 | 자동화 가능 |

#### 3) QA와 QC의 명확한 비교

| 구분 | QA (품질보증) | QC (품질통제) |
|------|--------------|--------------|
| 목적 | 결함 예방 | 결함 발견/제거 |
| 대상 | 프로세스 | 제품/산출물 |
| 시점 | 개발 전 과정 | 산출물 완성 후 |
| 성격 | 능동적(Proactive) | 수동적(Reactive) |
| 주체 | QA 팀(독립적) | 테스터, 검수자 |
| 기준 | 표준, 절차, 방법론 | 사양서, 테스트 케이스 |
| ISO 표준 | ISO 9001 | 결함 측정 지표 |
| 비용 효과 | 조기 발견 → 저비용 | 후기 발견 → 고비용 |

### 4. 맥락과 비교

**비용 관점: 예방이 발견보다 저렴하다**

소프트웨어 공학의 황금 법칙 중 하나는 **결함 발견 비용의 기하급수적 증가** 법칙입니다.

```text
[결함 수정 비용 (Barry Boehm의 연구)]

요구사항 단계에서 발견: 비용 × 1
설계 단계에서 발견:     비용 × 5
구현 단계에서 발견:     비용 × 10
테스트 단계에서 발견:   비용 × 25
릴리즈 후 발견:         비용 × 100
```

이것이 QA(예방)에 투자해야 하는 이유입니다. QC(테스팅)만으로는 이미 늦습니다.

**TQM (Total Quality Management)과의 연계**

TQM은 조직 전체가 품질에 참여하는 경영 철학으로, 데밍(Deming)의 PDCA 사이클을 기반으로 합니다. TQM에서 QA는 전략적 프로세스 관리이고, QC는 운영적 실행 도구입니다.

### 5. 실무 제언

**애자일 환경에서의 QA/QC 통합**

전통적인 폭포수(Waterfall) 모델에서는 QA와 QC가 단계별로 분리되어 있었습니다. 그러나 2주 단위 스프린트로 진행되는 애자일 환경에서는 다음과 같이 통합됩니다.

- **Shift-Left Testing**: QC 활동(테스트)을 개발 초기로 당깁니다. TDD(테스트 주도 개발)가 대표적 사례
- **지속적 품질 보증**: CI/CD 파이프라인 자체가 QA 프로세스의 자동화입니다. SonarQube로 코드 품질을 측정하고, 게이트를 통과하지 못하면 배포를 막는 것이 자동화된 QC입니다.
- **DevSecOps**: QA의 범위가 보안(Security)까지 확장되어 보안 테스트가 파이프라인에 통합됩니다.

**기술사 시험 응용 포인트**

QA/QC 문제는 단독으로 출제되기도 하지만, CMM/CMMI(성숙도 모델), SPICE(ISO 15504), ISO/IEC 25010(품질 모델) 등과 연계하여 출제될 때 고득점이 가능합니다. 특히 "조직의 소프트웨어 품질 수준을 어떻게 평가하고 개선할 것인가"라는 맥락에서 QA는 CMMI의 핵심 프로세스 영역(KPA)과 연결됩니다.
