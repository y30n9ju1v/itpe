## 해설

### 출제 배경 및 의도

122회 기술사 시험의 출제 배경은 공공 IT 사업의 고질적 품질 문제다. 스탠디시 그룹 CHAOS Report에 따르면 IT 프로젝트 절반 이상이 예산 초과·일정 지연을 경험하며, 그 뿌리에는 항상 불완전한 요구사항이 있다. 행안부·과기정통부가 반복적으로 '요구사항 명확화'를 품질 개선 핵심 과제로 지목하면서 요구공학 전반의 실무 이해가 기술사 자격 검증의 핵심 항목이 되었다.

채점관이 기대하는 것은 단순 용어 정의가 아닌, 기능/비기능/인터페이스 요구사항을 실제 프로젝트에 적용하여 분류하고 IEEE 830(SRS)·PMBOK 범위 관리와 연결하는 통합적 사고다.

### 1. IT 프로젝트 관리 및 요구사항 개요

정 의  • IT 프로젝트 관리: 프로젝트 목표 달성을 위해 지식·스킬·도구·기법을 활동에 적용하는 것(PMBOK 6판), 10대 지식 영역·5대 프로세스 그룹으로 구성
       - 요구사항은 범위관리(Scope Management)의 출발점이자 전체 관리의 기준선(Baseline)

```
[요구사항 계층 구조]

  비즈니스 요구사항 — "왜 이 시스템을 만드는가?" (목적·목표)
          │
          ▼
  이해관계자 요구사항 — "사용자가 무엇을 원하는가?" (사용자 관점)
          │
          ▼
  시스템 요구사항
     ┌────┴──────────────────────────┐
     │                               │
  기능 요구사항                  비기능 요구사항
  "무엇을 해야 하는가?"          "어떻게 잘 해야 하는가?"
     │                               │
     └──────────────┬────────────────┘
                    ▼
           인터페이스 / 제약사항
```

- 요구사항은 발주자와 개발자 간 '계약서'이자 '공통 언어'로, 프로젝트 전체 범위를 확정하는 근거다

### 2. 가. IT 프로젝트 관리 정의 및 관리 영역

1) PMBOK 기반 관리 체계

| 구분 | 항목 | 설명 |
|------|------|------|
| 5대 프로세스 | 착수→계획→실행→감시통제→종료 | 프로젝트 전 생애주기 관리 단계 |
| 10대 지식 영역 | 통합·범위·일정·원가·품질 등 | 범위관리가 요구사항의 핵심 출발점 |
| 핵심 산출물 | 프로젝트 헌장·WBS·기준선 | 요구사항이 범위 기준선(Baseline)을 구성 |

### 3. 나·다. IT개발 요구사항 정의·특징 및 유형

1) 요구사항 5대 특징

| 구분 | 특징 | 위반 시 문제 |
|------|------|------------|
| ① | 완전성(Completeness) | 누락된 기능으로 재작업 발생 |
| ② | 일관성(Consistency) | 설계 단계 충돌 발생 |
| ③ | 추적성(Traceability) | 변경 영향도 분석 불가 |
| ④ | 검증가능성(Verifiability) | 인수테스트 기준 불명확 |
| ⑤ | 우선순위(Priority) | Scope Creep 범위 폭증 |

2) 요구사항 4대 유형

```
[비기능 요구사항 - ISO/IEC 25010 분류]

  ┌─────────────────────────────────────────┐
  │ 성능(Performance)   │ 응답시간 ≤ 3초    │
  │ 가용성(Availability)│ 99.9% Uptime     │
  │ 보안성(Security)    │ OWASP Top 10     │
  │ 유지보수성          │ MTTR 기준         │
  │ 신뢰성(Reliability) │ MTBF ≥ 720h      │
  └─────────────────────────────────────────┘
```

| 구분 | 유형 | 질문 | 사례 |
|------|------|------|------|
| 기능 | Functional | 무엇을 하는가? | "회원은 ID/PW로 로그인할 수 있어야 한다" |
| 비기능 | Non-Functional | 얼마나 잘? | "응답시간 ≤ 3초 (동시 500명 기준)" |
| 인터페이스 | Interface | 어떻게 연결? | "PG사와 REST API(JSON) 연동" |
| 제약사항 | Constraints | 어떤 조건? | "Java 17 이상, AES-256 암호화 필수" |

### 4. 라. IT개발 요구사항 명세 조건 및 우선순위화

1) IEEE 830 기준 SRS 8대 명세 조건

| 구분 | 명세 조건 | 설명 |
|------|---------|------|
| ① | 정확성(Correct) | 이해관계자 실제 필요를 정확히 반영 |
| ② | 명확성(Unambiguous) | 하나의 의미로만 해석 가능 |
| ③ | 완전성(Complete) | 모든 기능·제약 누락 없이 기술 |
| ④ | 일관성(Consistent) | 요구사항 간 충돌·모순 없음 |
| ⑤ | 우선순위화(Prioritized) | MoSCoW 기법으로 중요도 분류 |
| ⑥ | 검증가능성(Verifiable) | 테스트로 충족 여부 측정 가능 |
| ⑦ | 수정가능성(Modifiable) | 변경 시 영향 범위 파악 용이 |
| ⑧ | 추적가능성(Traceable) | 출처와 구현 산출물 간 매핑(RTM) |

- MoSCoW(Must/Should/Could/Won't Have)는 요구사항 우선순위화의 표준 기법으로 Scope Creep을 예방한다.  "끝"

### 실무 제언

**Scope Creep 통제 — 요구사항 ID + CCB 프로세스 제도화**
- **챌린지**: 수치화되지 않은 비기능 요구사항과 무분별한 변경 수용이 인수 단계 분쟁과 범위 폭증의 원인이 된다
- **제언**: 착수 시 모든 요구사항에 유일 식별자(Requirement ID)를 부여하고, CR(Change Request)은 CIA(변경 영향도 분석)를 거쳐 CCB(변경통제위원회) 승인을 받는 프로세스를 제도화한다

**비기능 요구사항 수치화 — QAS + ATAM 연계**
- **챌린지**: "빠른 응답", "안정적 운영" 같은 정성적 비기능 명세는 인수테스트 단계에서 개발자-발주자 간 심각한 갈등으로 이어진다
- **제언**: 비기능 요건을 측정 가능한 수치 목표로 기술하고(예: 동시 500명 기준 응답 3초 이내, 95 퍼센타일), ATAM(Architecture Tradeoff Analysis Method)과 연계하여 아키텍처 설계 단계에서 QAS로 구체화한다
