## 해설

> **IT 시스템을 만드는 설계도의 출발점**  
> 요구사항은 프로젝트 성패를 좌우하는 가장 중요한 산출물이다

소프트웨어 프로젝트의 실패 원인 1위는 변함없이 "요구사항 불명확"입니다. 미국 스탠디시 그룹의 카오스 보고서(CHAOS Report)에 따르면 IT 프로젝트의 절반 이상이 예산 초과나 일정 지연을 경험하며, 그 뿌리에는 항상 불완전한 요구사항이 있습니다. 좋은 요구사항은 발주자와 개발자 사이의 '계약서'이자 '공통 언어'로, 같은 단어를 두고 서로 다른 그림을 그리는 일이 없도록 해주는 기반입니다.

### 1. 출제 배경 및 의도

122회 기술사 시험에서 이 문제가 출제된 배경은 공공 IT 사업의 고질적인 품질 문제입니다. 행정안전부와 과학기술정보통신부가 반복적으로 '요구사항 명확화'를 소프트웨어 품질 개선의 핵심 과제로 지목하면서, 요구공학(Requirements Engineering) 전반에 대한 실무적 이해가 기술사 자격 검증의 핵심 항목이 되었습니다.

채점관이 기대하는 답안은 단순한 용어 정의가 아니라, 기능/비기능/인터페이스 요구사항을 실제 프로젝트 사례에 적용하여 분류할 수 있는 역량, 그리고 IEEE 830(SRS)이나 PMBOK의 범위 관리와 연결하는 통합적 사고입니다.

### 2. 개념의 본질과 작동 메커니즘

**IT 프로젝트 관리(IT Project Management)란** 명확하게 정의된 목표를 달성하기 위해 인적·물적 자원을 계획하고 조직하며 통제하는 일련의 관리 활동입니다. PMBOK(Project Management Body of Knowledge) 기준으로 10개 지식 영역과 5개 프로세스 그룹으로 구성됩니다.

```text
[IT 프로젝트 관리 10대 지식 영역]

  통합관리 ──── 프로젝트 헌장 → 범위·일정·원가·품질·자원·의사소통·
  (Integration)       리스크·조달·이해관계자 관리 통합

  ┌─────────────────────────────────────────────────────┐
  │  범위관리 ─→ 요구사항 수집 ─→ WBS 생성 ─→ 범위 통제 │
  │  일정관리 ─→ 활동 정의 ─→ 네트워크 다이어그램 ─→ CPM │
  │  원가관리 ─→ 비용 산정 ─→ 예산 책정 ─→ EVM 통제    │
  │  품질관리 ─→ 품질 계획 ─→ QA ─→ QC               │
  └─────────────────────────────────────────────────────┘
  ※ 요구사항은 '범위관리'의 출발점이자 전체 관리의 기준선(Baseline)
```

**요구사항(Requirements)이란** 이해관계자(Stakeholder)가 특정 목적을 달성하기 위해 시스템 또는 시스템 구성 요소가 반드시 갖추어야 하는 조건이나 역량의 집합입니다.

```text
[요구사항 계층 구조]

  비즈니스 요구사항 (Business Requirements)
          │  "왜 이 시스템을 만드는가?" (목적, 목표)
          ▼
  이해관계자 요구사항 (Stakeholder Requirements)
          │  "사용자가 무엇을 원하는가?" (사용자 관점)
          ▼
  시스템 요구사항 (System Requirements)
     ┌────┴──────────────────┐
     │                       │
  기능 요구사항           비기능 요구사항
  (Functional)            (Non-Functional)
     │                       │
  "무엇을 해야 하는가?"   "어떻게 잘 해야 하는가?"
     │                       │ (품질속성: 성능/보안/가용성 등)
     └────────────┬──────────┘
                  ▼
         인터페이스/제약사항
         (Interface / Constraints)
```

### 3. 핵심 키워드 (Must-Have)

#### 가. IT 프로젝트 관리의 정의 및 관리 영역

- **정의**: 프로젝트의 목표를 달성하기 위해 지식·스킬·도구·기법을 프로젝트 활동에 적용하는 것 (PMBOK 6판)
- **5대 프로세스 그룹**: 착수(Initiating) → 계획(Planning) → 실행(Executing) → 감시/통제(Monitoring & Controlling) → 종료(Closing)
- **10대 지식 영역**: 통합·범위·일정·원가·품질·자원·의사소통·리스크·조달·이해관계자 관리
- **Must-Have 키워드**: 프로젝트 헌장(Project Charter), 기준선(Baseline), 형상관리(Configuration Management), WBS(Work Breakdown Structure)

#### 나. IT개발 요구사항의 정의 및 특징

- **정의**: 시스템이 이해관계자의 필요를 충족시키기 위해 반드시 보유해야 하는 능력(Capability) 또는 충족해야 하는 조건(Condition)
- **5대 특징 (SMART 변형)**:

| 특징 | 설명 | 위반 시 문제 |
|------|------|------------|
| 완전성(Completeness) | 모든 요구사항이 빠짐없이 기술 | 누락된 기능으로 재작업 발생 |
| 일관성(Consistency) | 요구사항 간 상호 모순 없음 | 설계 단계에서 충돌 발생 |
| 추적성(Traceability) | 요구사항의 출처와 구현을 추적 가능 | 변경 영향도 분석 불가 |
| 검증가능성(Verifiability) | 테스트로 충족 여부 확인 가능 | 인수테스트 기준 불명확 |
| 우선순위(Priority) | 중요도와 긴급도에 따라 분류 | 범위 폭증(Scope Creep) 발생 |

- **Must-Have 키워드**: 요구공학(Requirements Engineering), IEEE 830(SRS 표준), 요구사항 기준선(Requirements Baseline)

#### 다. IT개발 요구사항의 유형

**기능 요구사항 (Functional Requirements)**

- 시스템이 "무엇을 해야 하는가"를 정의하는 행위/기능 명세
- 예시: "회원은 아이디와 비밀번호로 로그인할 수 있어야 한다"
- 특징: 입력-처리-출력으로 표현 가능, 동사형 기술, 유스케이스(Use Case, 사용자와 시스템 간 상호작용을 시나리오 형태로 기술한 것)로 상세화

**비기능 요구사항 (Non-Functional Requirements)**

- 시스템이 "어떻게 동작해야 하는가"를 정의하는 품질 속성(Quality Attributes)
- ISO/IEC 25010(SQuaRE) 기준 8대 품질 특성:

```text
[비기능 요구사항 분류 - ISO/IEC 25010]

  ┌─────────────────────────────────────┐
  │         비기능 요구사항              │
  ├─────────────┬───────────────────────┤
  │ 성능(Performance) │ 응답시간 ≤ 3초  │
  │ 가용성(Availability)│ 99.9% Uptime │
  │ 보안성(Security)  │ OWASP Top 10  │
  │ 유지보수성(Maintainability)│ MTTR  │
  │ 이식성(Portability)│ 멀티 OS 지원  │
  │ 신뢰성(Reliability)│ MTBF ≥ 720h  │
  │ 사용성(Usability) │ SUS Score ≥ 68│
  │ 기능적합성(Functional Suitability)  │
  └─────────────────────────────────────┘
```

**인터페이스 요구사항 (Interface Requirements)**

- 시스템이 외부 시스템·사용자·하드웨어와 상호작용하는 방식
- 분류: 사용자 인터페이스(UI), 하드웨어 인터페이스(HW I/F), 소프트웨어 인터페이스(SW I/F), 통신 인터페이스
- 예시: "결제 시스템은 KG이니시스 PG 시스템과 REST API(JSON)로 연동해야 한다"

**제약사항 (Constraints)**

- 설계 및 구현에 대한 제한 조건 (기술적·법적·환경적)
- 예시: "본 시스템은 Java 17 이상, Spring Boot 3.x 기반으로 개발해야 한다"
- 예시: "개인정보는 개인정보보호법에 따라 AES-256으로 암호화해야 한다"

#### 라. IT개발 요구사항의 명세 조건 (SRS: Software Requirements Specification)

IEEE 830 기준 SRS(소프트웨어 요구사항 명세서)의 구성과 조건:

| 명세 조건 | 설명 | 작성 기법 |
|---------|------|---------|
| 정확성(Correct) | 요구사항이 이해관계자의 실제 필요를 정확히 반영 | 인터뷰, 프로토타이핑 |
| 명확성(Unambiguous) | 하나의 의미로만 해석 가능, 모호한 표현 배제 | 형식적 명세, Use Case |
| 완전성(Complete) | 시스템의 모든 기능과 제약을 빠짐없이 기술 | 체크리스트, 리뷰 |
| 일관성(Consistent) | 요구사항 간 충돌이나 모순 없음 | 요구사항 관리 도구 |
| 우선순위화(Prioritized) | MoSCoW 등 방법으로 중요도 분류 | MoSCoW, 카노 모델 |
| 검증가능성(Verifiable) | 테스트나 검사로 충족 여부 측정 가능 | 인수기준(AC) 명시 |
| 수정가능성(Modifiable) | 변경 시 영향 범위 파악이 용이한 구조 | 요구사항 ID 체계화 |
| 추적가능성(Traceable) | 요구사항의 출처와 구현 산출물 간 매핑 | RTM(요구사항 추적 매트릭스) |

**MoSCoW 우선순위화 기법**:
- **Must Have**: 반드시 구현 (핵심 기능, 법적 요건)
- **Should Have**: 가능하면 구현 (중요하나 대안 존재)
- **Could Have**: 시간이 허락하면 구현 (부가 기능)
- **Won't Have (this time)**: 이번 릴리스에서는 제외

### 4. 맥락과 비교

| 구분 | 기능 요구사항 | 비기능 요구사항 | 인터페이스 요구사항 | 제약사항 |
|------|------------|--------------|----------------|--------|
| 질문 | 무엇을 하는가? | 얼마나 잘 하는가? | 어떻게 연결하는가? | 어떤 조건 하에? |
| 표현 방식 | 동사형 ("~해야 한다") | 수치형 ("≤ 3초") | 프로토콜/포맷 명시 | 규정/기술 스택 명시 |
| 검증 방법 | 기능 테스트 | 성능/보안 테스트 | 통합 테스트 | 컴플라이언스 감사 |
| 산출물 | 유스케이스, 기능 목록 | QAS(Quality Attribute Scenario, 품질속성 시나리오) | I/F 정의서 | 기술 스택 문서 |
| 우선순위 | 높음 (Must Have 多) | 중~높음 | 중간 | 초기 고정 |

### 5. 실무 제언

**요구사항 크리프(Scope Creep) 통제**

- **챌린지**: IT 프로젝트에서 가장 빈번한 실패 패턴은 초기에는 명확했던 범위가 프로젝트 진행 중 이해관계자의 요청으로 지속적으로 확장되는 '범위 폭증(Scope Creep)'입니다. 특히 비기능 요구사항("빠르게", "안전하게")처럼 수치화되지 않은 요구사항은 인수 단계에서 분쟁의 씨앗이 됩니다.
- **제언**: 프로젝트 착수 시 모든 요구사항에 **유일 식별자(Requirement ID)**를 부여하고, 변경 요청(CR: Change Request)은 반드시 **변경영향도 분석(CIA: Change Impact Analysis)**을 거친 후 CCB(변경통제위원회)의 승인을 받는 프로세스를 제도화해야 합니다.

**비기능 요구사항의 수치화**

- **챌린지**: "빠른 응답", "안정적인 운영" 같은 정성적 비기능 요구사항은 개발자와 발주자가 서로 다른 기준을 갖고 있어 인수테스트 단계에서 심각한 갈등으로 이어집니다.
- **제언**: 비기능 요구사항은 반드시 **측정 가능한 수치 목표(Measurable Target)**로 기술해야 합니다. 예) "응답 시간: 동시 사용자 500명 기준, 주요 화면 로딩 3초 이내(95 퍼센타일)", "가용성: 연간 다운타임 8.76시간 이내(99.9% SLA)". ATAM(Architecture Tradeoff Analysis Method)과 연계하여 아키텍처 설계 단계에서 이를 품질속성 시나리오(QAS)로 구체화하는 것이 고득점 답안의 핵심입니다.
