## 해설

### 출제 배경 및 의도

소프트웨어 테스트는 전체 개발 비용의 30~50%를 차지하는 핵심 활동임에도, 많은 조직에서 "코드 완성 후 남은 시간에 하는 것"으로 오해한다. ISTQB가 정립한 7원칙은 테스팅에 대한 잘못된 믿음을 깨뜨리고, 제한된 자원으로 가장 효과적인 테스트 전략을 수립하는 지침이다.

채점관이 기대하는 것은 각 원칙의 단순 나열이 아니라, 각 원칙이 실제 테스트 케이스 설계에 어떻게 반영되어야 하는지를 구체적으로 설명하는 능력이다. 특히 Boehm의 결함 비용 증가 법칙(원칙 3), 파레토 법칙(원칙 4), 살충제 패러독스(원칙 5)를 실무 사례와 연결하면 고득점이 가능하다.

### 1. 소프트웨어 테스트 개요

정 의  • 소프트웨어가 명시된 요구사항을 충족하는지 검증(Verification)하고, 사용자의
       실제 요구를 만족하는지 확인(Validation)하며, 결함을 발견하기 위해
       소프트웨어를 실행하는 과정

```
[소프트웨어 테스트 V-모델]

  요구사항 분석 ─────────────────── 인수 테스트(AT)
       │                                  │
  시스템 설계 ──────────────── 시스템 테스트(ST)
       │                            │
  아키텍처 설계 ──────── 통합 테스트(IT)
       │                      │
  상세 설계 ──────── 단위 테스트(UT)
       │                  │
       └────── 구현 ───────┘

  검증(Verification): 올바르게 만들고 있는가? (내부 기준)
  확인(Validation):   올바른 것을 만들고 있는가? (외부 기준)
```

- 테스트는 Verification과 Validation을 모두 포함하며, 어느 한쪽만으로는 품질을 보장할 수 없다.

### 2. ISTQB 소프트웨어 테스트 7원칙

1) 원칙 1~4: 결함 발견 철학

| 원칙 | 핵심 메시지 | 테스트 케이스 반영 |
|------|-----------|---------|
| 1. 결함 존재 증명 | 테스트 = 결함 발견, 결함 부재 증명 불가 | 경계값·오류·예외 케이스 의도적 포함 |
| 2. 완벽 테스트 불가 | 모든 입력 조합·경로 테스트는 수학적 불가 | 위험 기반 테스팅(RBT)으로 우선순위화 |
| 3. 조기 테스팅 | 결함 발견 지연 → 수정 비용 기하급수적 증가 | Shift Left, TDD, BDD 도입 |
| 4. 결함 군집 | 20% 모듈에 80% 결함 집중 (파레토 법칙) | CC 높은 모듈·과거 버그 빈발 모듈 집중 |

```
[원칙 3: Boehm의 결함 수정 비용 증가]

  요구사항 단계: $1
  설계 단계:    $6
  구현 단계:    $10
  테스트 단계:  $40
  운영 단계:    $100 ~ $1,000

[원칙 1 적용 테스트 케이스 예시 - 나이 입력(0~150세)]

  TC-001: 입력=30   → 정상 통과     (정상값)
  TC-002: 입력=0    → 정상 통과     (경계값)
  TC-003: 입력=150  → 정상 통과     (경계값)
  TC-004: 입력=-1   → 오류 메시지   (결함 탐지)
  TC-005: 입력=151  → 오류 메시지   (결함 탐지)
  TC-006: 입력="abc"→ 오류 메시지   (결함 탐지)
```

- 원칙 1~4는 테스트가 결함 발견을 위한 의도적·전략적 활동임을 강조한다.

2) 원칙 5~7: 테스트 전략 고도화

| 원칙 | 핵심 메시지 | 실무 대응 |
|------|-----------|---------|
| 5. 살충제 패러독스 | 반복 테스트 → 새 결함 미발견 (해충의 살충제 내성) | 테스트 케이스 정기 갱신, 탐색적 테스팅 병행 |
| 6. 정황 의존성 | 시스템 유형별 다른 테스트 전략 필요 | 안전핵심 시스템: 형식 검증, 상용 애자일: TDD·CI |
| 7. 오류 부재 궤변 | 결함 없음 ≠ 사용자 만족 (잘못된 요구의 완벽한 구현) | UAT·V&V 강화, 요구사항 검증 병행 |

- 원칙 5~7은 테스트 전략의 지속적 개선과 비즈니스 가치 중심 관점을 강조한다.

### 3. 테스트 레벨 비교 및 트렌드

1) 테스트 레벨별 비교

| 레벨 | 대상 | 목적 | 기법 | 수행자 |
|------|------|------|------|-------|
| 단위 테스트(UT) | 함수/클래스 | 모듈 정확성 | 화이트박스, TDD | 개발자 |
| 통합 테스트(IT) | 모듈 간 인터페이스 | 인터페이스 오류 | 상향식/하향식 | 개발자/테스터 |
| 시스템 테스트(ST) | 전체 시스템 | 기능·비기능 검증 | 블랙박스, 성능/보안 | 테스터 |
| 인수 테스트(AT) | 사용자 관점 | 사용자 요구 확인 | UAT, 알파/베타 | 사용자/발주자 |

- 테스트 피라미드: 단위 테스트(넓은 기반) → 통합 테스트 → E2E 테스트(좁은 정점) 구조로 자동화 비용과 속도를 균형화한다.  "끝"

### 실무 제언

**테스트 자동화와 살충제 패러독스의 딜레마**
- **챌린지**: 동일한 자동화 테스트 스위트가 수개월 반복 실행되면 원칙 5(살충제 패러독스)에 따라 새로운 결함을 발견하지 못한다. "테스트 커버리지 80%"를 달성했음에도 릴리스 후 주요 버그가 발생하는 이유다.
- **제언**: 자동화 회귀 테스트(안전망)와 탐색적 테스팅(새로운 결함 발견)을 병행해야 한다. 매 스프린트마다 Session-Based Exploratory Testing을 수행하고, 발견된 결함 유형을 자동화 스위트에 추가하는 선순환 구조를 만들어야 한다.

**Shift Left와 BDD의 결합**
- **챌린지**: 원칙 3(조기 테스팅)을 실천하려면 요구사항 단계부터 테스트를 고려해야 하지만, 대부분의 조직에서 QA 팀은 개발 완료 후에야 참여한다.
- **제언**: BDD(Behavior-Driven Development)를 도입한다. Given-When-Then 형식으로 요구사항을 작성하면 명세와 인수 테스트 케이스가 동시에 완성된다. Cucumber, SpecFlow 활용 시 비즈니스 분석가의 시나리오가 자동화 코드로 직접 연결되어 원칙 3과 원칙 7을 동시에 실현할 수 있다.

**정황 의존성(원칙 6)의 실무 적용**
- **챌린지**: 팀마다 동일한 테스트 프로세스를 강요하면 금융·의료·게임 등 도메인별 특성이 무시되어 테스트 효율이 저하된다.
- **제언**: 도메인 리스크 프로파일에 따라 테스트 전략을 차별화한다. 안전핵심(Safety-Critical) 시스템은 MC/DC 커버리지·형식 검증, 빠른 상용 서비스는 TDD·BDD·탐색적 테스팅을 중심으로 구성하여 원칙 6의 정황 의존성을 전략적으로 활용해야 한다.
