## 문제

테스트 7원칙의 주요 원칙을 쓰고 **주요 원칙에 따른 테스트 케이스**를 작성하시오.

## 해설

> **테스트는 결함을 제거하는 것이 아니라 결함의 존재를 증명하는 활동이다**  
> 소프트웨어 테스트 7원칙은 올바른 테스트 철학의 기반이자 효율적 테스트 전략의 출발점

"테스트를 충분히 많이 하면 버그가 없다고 확신할 수 있다"는 것은 사실일까요? 아닙니다. 이것이 테스트 7원칙이 존재하는 이유입니다. ISTQB(International Software Testing Qualifications Board)가 정립한 7원칙은 테스팅에 대한 잘못된 믿음을 깨뜨리고, 제한된 시간과 자원으로 가장 효과적인 테스트 전략을 수립하는 지침입니다.

### 1. 출제 배경 및 의도

소프트웨어 테스트는 전체 개발 비용의 30~50%를 차지하는 핵심 활동임에도, 많은 조직에서 "코드 완성 후 남은 시간에 하는 것"으로 오해합니다. 기술사 시험에서 테스트 7원칙을 출제하는 의도는 **올바른 테스트 철학 위에서 체계적인 테스트 계획을 수립하고 케이스를 설계하는 역량**을 검증하기 위함입니다.

이 문제에서 채점관이 특히 중요하게 보는 것은 각 원칙의 단순 나열이 아니라, **각 원칙이 실제 테스트 케이스 설계에 어떻게 반영되어야 하는지**를 구체적으로 설명하는 능력입니다.

### 2. 개념의 본질과 작동 메커니즘

**소프트웨어 테스트란** 소프트웨어가 명시된 요구사항을 충족하는지 검증하고(Verification), 사용자의 실제 요구를 만족하는지 확인(Validation)하며, 결함을 발견하기 위해 소프트웨어를 실행하는 과정입니다.

```text
[소프트웨어 테스트 레벨 (V-모델)]

  요구사항 분석 ────────────────── 인수 테스트(AT)
       │                               │
  시스템 설계 ──────────────── 시스템 테스트(ST)
       │                         │
  아키텍처 설계 ───────── 통합 테스트(IT)
       │                   │
  상세 설계 ──────── 단위 테스트(UT)
       │               │
       └──── 구현 ─────┘

  검증(Verification):   ← 올바르게 만들고 있는가? (내부 기준)
  확인(Validation):     ← 올바른 것을 만들고 있는가? (외부 기준)
```

### 3. 핵심 키워드 (Must-Have)

#### ISTQB 소프트웨어 테스트 7원칙

**원칙 1: 테스팅은 결함의 존재를 보여주는 것이다 (Testing shows presence of defects)**

- **의미**: 테스트는 결함이 있다는 것을 증명할 수 있지만, 결함이 없다는 것은 절대 증명할 수 없다
- **핵심**: "테스트를 통과했다 = 버그가 없다"는 틀린 명제
- **테스트 케이스 반영**: 테스트 케이스는 결함을 발견하기 위해 의도적으로 경계값, 예외 상황, 오류 케이스를 포함해야 함

```text
[원칙 1 적용 테스트 케이스 예시 - 나이 입력 유효성 검사]

  입력 범위: 0~150세 허용

  TC-001 (정상 케이스): 입력=30, 기대=통과
  TC-002 (경계값):     입력=0,  기대=통과
  TC-003 (경계값):     입력=150,기대=통과
  TC-004 (결함 탐지):  입력=-1, 기대=오류 메시지
  TC-005 (결함 탐지):  입력=151,기대=오류 메시지
  TC-006 (결함 탐지):  입력="abc",기대=오류 메시지
```

**원칙 2: 완벽한 테스팅은 불가능하다 (Exhaustive testing is impossible)**

- **의미**: 모든 입력 조합, 모든 경로를 테스트하는 것은 수학적으로 불가능
- **대안**: 위험도 분석(Risk Analysis)과 우선순위화를 통한 **위험 기반 테스팅(Risk-Based Testing)**
- **테스트 케이스 반영**: MoSCoW 기반으로 중요 기능과 고위험 모듈 중심으로 테스트 커버리지 집중

**원칙 3: 조기 테스팅이 시간과 비용을 절감한다 (Early testing saves time and money)**

- **의미**: 결함은 발견이 늦을수록 수정 비용이 기하급수적으로 증가 (Boehm의 법칙)
- **구현**: Shift Left Testing — 요구사항 단계부터 테스트 설계 시작 (TDD, BDD)

```text
[Boehm의 결함 수정 비용 증가]

  요구사항 단계: $1
  설계 단계:    $6
  구현 단계:    $10
  테스트 단계:  $40
  운영 단계:    $100 ~ $1000
```

**원칙 4: 결함은 군집되어 있다 (Defect clustering)**

- **의미**: 소수의 모듈(약 20%)에서 대부분의 결함(약 80%)이 발견됨 (파레토 법칙)
- **적용**: 과거 결함 이력, 복잡도 메트릭으로 고위험 모듈 식별 → 집중 테스트
- **테스트 케이스 반영**: 복잡도(Cyclomatic Complexity)가 높은 모듈, 과거 버그 빈발 모듈에 테스트 케이스 집중

**원칙 5: 살충제 패러독스 (Pesticide paradox)**

- **의미**: 동일한 테스트를 반복하면 새로운 결함을 더 이상 발견하지 못함 (살충제에 내성이 생긴 해충처럼)
- **대안**: 테스트 케이스의 정기적 검토·갱신, 다양한 테스트 기법 혼용, 탐색적 테스팅(Exploratory Testing)
- **테스트 케이스 반영**: 회귀 테스트(Regression Test) 스위트도 주기적으로 새 케이스 추가 필요

**원칙 6: 테스팅은 정황에 의존한다 (Testing is context dependent)**

- **의미**: 의료 기기 소프트웨어, 전자상거래 사이트, 모바일 게임은 각각 다른 테스트 전략이 필요
- **적용**: 안전 핵심 시스템(Safety-Critical)은 형식 검증·철저한 V&V, 애자일 상용 SW는 TDD·CI
- **테스트 케이스 반영**: 금융 시스템이라면 IFRS·내부통제 관련 케이스 추가, IoT라면 환경 조건 케이스 추가

**원칙 7: 오류 부재의 궤변 (Absence-of-errors fallacy)**

- **의미**: 결함이 없다고 해서 사용자가 만족하는 소프트웨어는 아님. 잘못된 요구사항을 완벽하게 구현해도 실패
- **적용**: 검증(Verification)만큼 확인(Validation)이 중요. 사용자 수용 테스트(UAT)의 중요성
- **테스트 케이스 반영**: 기능 명세 충족(Verification) 케이스뿐 아니라 실제 사용 시나리오(UAT) 케이스도 설계

#### 테스트 7원칙 요약표

| 원칙 | 핵심 메시지 | 실무 대응 |
|------|-----------|---------|
| 1. 결함 존재 증명 | 테스트 = 결함 발견, 결함 부재 증명 불가 | 경계값·오류 케이스 중심 테스트 |
| 2. 완벽 테스트 불가 | 모든 케이스 테스트는 불가능 | 위험 기반 테스팅, 우선순위화 |
| 3. 조기 테스팅 | 결함 발견 지연 → 비용 급증 | Shift Left, TDD, BDD |
| 4. 결함 군집 | 20% 모듈에 80% 결함 | 고위험 모듈 집중 테스트 |
| 5. 살충제 패러독스 | 반복 테스트 → 새 결함 미발견 | 테스트 케이스 정기 갱신 |
| 6. 정황 의존성 | 시스템 유형별 다른 전략 | 도메인별 맞춤 테스트 접근법 |
| 7. 오류 부재 궤변 | 결함 없음 ≠ 사용자 만족 | UAT, 요구사항 검증 강화 |

### 4. 맥락과 비교

**테스트 레벨 비교**:

| 레벨 | 대상 | 목적 | 기법 | 수행자 |
|------|------|------|------|-------|
| 단위 테스트(UT) | 함수/클래스 | 모듈 정확성 | 화이트박스, TDD | 개발자 |
| 통합 테스트(IT) | 모듈 간 인터페이스 | 인터페이스 오류 | 상향식/하향식 통합 | 개발자/테스터 |
| 시스템 테스트(ST) | 전체 시스템 | 기능·비기능 요구 검증 | 블랙박스, 성능/보안 | 테스터 |
| 인수 테스트(AT) | 사용자 관점 | 사용자 요구 충족 확인 | UAT, 알파/베타 | 사용자/발주자 |

**정적 테스팅 vs 동적 테스팅**:
- **정적 테스팅**: 코드를 실행하지 않고 검사 (코드 리뷰, 인스펙션, 정적 분석 도구)
- **동적 테스팅**: 코드를 실행하여 검사 (단위·통합·시스템·인수 테스트)

### 5. 실무 제언

**테스트 자동화와 살충제 패러독스의 딜레마**

- **챌린지**: 회귀 테스트를 자동화하면 CI/CD에서 빠른 피드백을 받을 수 있지만, 동일한 자동화 테스트 스위트가 수개월 반복 실행되면 원칙 5(살충제 패러독스)에 따라 새로운 결함을 발견하지 못하게 됩니다. 많은 팀이 "테스트 커버리지 80%"를 달성했음에도 릴리스 후 주요 버그가 발생하는 이유입니다.
- **제언**: 자동화 회귀 테스트(안전망 역할)와 탐색적 테스팅(Exploratory Testing, 새로운 결함 발견)을 병행해야 합니다. 매 스프린트마다 탐색적 테스팅을 세션 기반으로 수행(Session-Based Exploratory Testing)하고, 발견된 새로운 결함 유형을 자동화 테스트 스위트에 추가하는 선순환 구조를 만들어야 합니다.

**Shift Left와 BDD의 결합**

- **챌린지**: 원칙 3(조기 테스팅)을 실천하려면 요구사항 단계부터 테스트를 고려해야 하지만, 대부분의 조직에서 QA 팀은 개발이 완료된 후에야 참여합니다.
- **제언**: BDD(Behavior-Driven Development) 접근법을 도입해야 합니다. Given-When-Then 형식으로 요구사항을 작성하면 요구사항 명세와 인수 테스트 케이스가 동시에 완성됩니다. Cucumber, SpecFlow 같은 BDD 도구를 활용하면 비즈니스 분석가가 작성한 시나리오가 자동화 테스트 코드로 직접 연결되어 원칙 3과 원칙 7을 동시에 실현할 수 있습니다.
