## 해설

### 출제 배경 및 의도

소프트웨어 아키텍처 패턴은 1996년 Buschmann 등이 저술한 "Pattern-Oriented Software Architecture(POSA)" 시리즈에서 체계화되었다. 기술사 시험에서 이 주제가 반복 출제되는 이유는 패턴 선택이 시스템의 품질 속성(성능·변경용이성·이식성)에 직접적인 영향을 미치며, 이를 실무 맥락에서 설명할 수 있는 능력이 아키텍트의 핵심 역량이기 때문이다.

채점관이 기대하는 것은 단순한 패턴 정의가 아니라 각 패턴이 어떤 문제를 해결하는지, 그리고 어떤 상황에서 적합하지 않은지를 구체적 사례와 함께 설명하는 것이다. 아키텍처 패턴은 GoF 디자인 패턴보다 상위 추상화 수준으로 시스템 전체 구조를 결정하며, 이후 개발·운영의 품질 기반을 형성한다.

### 1. 소프트웨어 아키텍처 패턴 개요

정 의  • 특정 설계 상황에서 반복적으로 발생하는 문제에 대해 검증된 설계 구조를
       일반화한 템플릿
       - GoF 디자인 패턴이 클래스/객체 수준의 해법이라면, 아키텍처 패턴은
         시스템 전체 구조 수준의 해법

```
[아키텍처 패턴 계층 구조]

  비즈니스 아키텍처 패턴
  ┌───────────────────────────────────────┐
  │   MSA  │  SOA  │  Event-Driven │ CQRS │
  └───────────────────────────────────────┘
                   ↑ 조합
  구조적 아키텍처 패턴
  ┌───────────────────────────────────────┐
  │  Layered │ Broker │ MVC │ P&F │ Repo  │
  └───────────────────────────────────────┘
                   ↑ 구현
  설계 패턴 (GoF 23개)
  ┌───────────────────────────────────────┐
  │  Singleton │ Observer │ Facade │ ...  │
  └───────────────────────────────────────┘
```

- 아키텍처 패턴은 품질 속성 요구사항(QAS)을 기준으로 선택하며, 트레이드오프 분석이 필수적이다.

### 2. 핵심 패턴 상세: Layer / Broker / MVC

1) Layer 패턴 (계층형 아키텍처)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 시스템을 기능적으로 유사한 계층으로 분리하고, 각 계층은 바로 아래 계층의 서비스만 사용하는 구조 |
| 구조 | 4계층 | Presentation → Application → Domain → Infrastructure (단방향 의존) |
| 장점 | SoC | 관심사 분리로 변경 영향 최소화, 계층별 독립 테스트 용이 |
| 장점 | 교체성 | 계층 독립 교체 가능 (예: DB 변경 시 Infrastructure만 수정) |
| 단점 | 성능 | 모든 요청이 전 계층 통과 → 오버헤드 발생 |
| 단점 | 생산성 | 단순 CRUD도 4계층 통과, 계층 간 DTO 변환 Boilerplate 증가 |
| 적용 | 사례 | 엔터프라이즈 애플리케이션, 공공 정보 시스템 (Spring Boot, Django) |

```
[Layer 패턴 의존성 방향]

  ┌──────────────────────────┐
  │  Presentation (UI)       │  ← 사용자 인터페이스
  ├──────────────────────────┤
  │  Application (Use Case)  │  ← 워크플로우 조정
  ├──────────────────────────┤
  │  Domain (Business Logic) │  ← 핵심 비즈니스, 엔티티
  ├──────────────────────────┤
  │  Infrastructure (DB/API) │  ← 외부 시스템 연동
  └──────────────────────────┘
     의존성 방향: 위 → 아래 (단방향)
```

- Layer 패턴은 변경용이성·이식성에 최적화되나, 모놀리식 구조로 독립 확장이 어렵다.

2) Broker 패턴

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 분산 환경에서 Broker가 중개자 역할을 수행하여 위치 투명성(Location Transparency)을 제공 |
| 구성 | 요소 | Client Proxy, Broker, Server Proxy, Registry (서버 등록부) |
| 장점 | 투명성 | 클라이언트가 서버 위치 몰라도 서비스 호출 가능 |
| 장점 | 확장성 | 서버 추가·제거 시 클라이언트 변경 불필요 |
| 단점 | SPOF | Broker가 단일 장애점(Single Point of Failure)으로 고가용성 설계 필요 |
| 단점 | 성능 | Broker 통신 오버헤드 발생 |
| 구현 | 현대 | API Gateway(Kong, AWS API GW), ESB, gRPC, K8s Service |

```
[Broker 패턴 구조]

  Client Proxy ──요청──▶ ┌────────┐ ──전달──▶ Server A
                         │ BROKER │
  Client Proxy ──요청──▶ │(중개자)│ ──전달──▶ Server B
                         └────────┘
                              │
                         ┌─────────┐
                         │Registry │  ← 서버 등록/탐색
                         └─────────┘
```

- Broker 패턴은 분산 투명성·확장성을 확보하나, SPOF와 성능 오버헤드를 관리해야 한다.

3) MVC 패턴 (Model-View-Controller)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 정의 | 시스템을 Model(데이터/비즈니스), View(표현), Controller(제어 흐름)로 분리하는 패턴 |
| 목적 | 분리 | UI와 비즈니스 로직의 독립적 변경 가능 |
| 장점 | 다변화 | View 다변화 용이 (웹/앱/콘솔이 동일 Model 공유) |
| 장점 | 테스트 | Model 독립 단위 테스트 가능, 다수 개발자 병렬 작업 |
| 단점 | 비대화 | Controller 비대화(Massive View Controller) 현상 |
| 단점 | 복잡성 | 단순 앱에 과도한 구조(Overengineering) |
| 파생 | 패턴 | MVP(Model-View-Presenter), MVVM(Model-View-ViewModel) |

```
[MVC 패턴 흐름]

  사용자 입력
       │
       ▼
  ┌──────────┐  이벤트 처리  ┌──────────┐
  │Controller│─────────────▶│  Model   │
  │ (제어자) │◀─ 상태 변경 ──│(데이터/  │
  └──────────┘               │ 비즈니스)│
       │                     └──────────┘
       ▼                          │ 변경 알림
  ┌──────────┐◀────────────────────┘
  │   View   │  데이터 조회 후 화면 렌더링
  │  (화면)  │
  └──────────┘
```

- MVC는 분리성·테스트 용이성에 최적화되나, Controller 비대화를 방지하는 설계 규율이 필요하다.

### 3. 패턴 비교 및 트렌드

1) 아키텍처 패턴 종합 비교

| 패턴 | 핵심 구조 | 최적 품질속성 | 취약점 | 현대적 구현 |
|------|---------|------------|-------|-----------|
| Layered | 계층 스택 | 변경용이성, 이식성 | 성능, 모놀리식 | Spring Boot, Django |
| Broker | 중개자 + Registry | 분산 투명성, 확장성 | SPOF, 오버헤드 | API Gateway, K8s |
| MVC | Model-View-Controller | 분리성, 테스트성 | Controller 비대화 | Spring MVC, React+Redux |
| MSA | 독립 서비스 메시 | 독립 배포, 확장성 | 분산 복잡성 | Docker, Kubernetes |
| EDA | 이벤트 버스 | 느슨한 결합, 반응성 | 디버깅, 순서 보장 | Kafka, RabbitMQ |
| Pipe-and-Filter | 파이프 + 필터 체인 | 재사용성, 병렬성 | 상태 공유 | Apache Beam, Unix pipe |

- 패턴 선택은 품질 속성 요구사항(QAS) 기반 트레이드오프 분석으로 결정하며, 트렌드 추종을 지양해야 한다.  "끝"

### 실무 제언

**아키텍처 패턴 선택의 함정: 트렌드 추종**
- **챌린지**: MSA는 사실상 디폴트 선택처럼 여겨지지만, Amazon Prime Video가 MSA를 모놀리식으로 전환하여 비용을 90% 절감한 사례처럼, 무조건적 MSA 도입이 오히려 더 큰 운영 비용과 분산 복잡성을 초래하는 경우가 빈번하다.
- **제언**: 패턴 선택은 "독립 배포가 반드시 필요한가?", "팀 규모가 독립 운영을 지원하는가?", "분산 트랜잭션을 감당할 수 있는가?"라는 질문에서 출발해야 한다. Strangler Fig Pattern처럼 모놀리식에서 점진적으로 MSA로 전환하는 단계적 접근이 현실적 대안이다.

**MVC의 진화: 프론트엔드 분리 아키텍처**
- **챌린지**: 전통적 서버사이드 MVC는 React·Vue·Angular 등 SPA 프레임워크 등장으로 그 경계가 흐려지고 있다. 프론트엔드가 자체 MVVM을 구현하고 백엔드는 REST API 서버로만 동작하는 구조가 일반화되었다.
- **제언**: 현대 웹 아키텍처에서는 백엔드: 클린 아키텍처(Layered + DDD) + 프론트엔드: MVVM(React/Vue) 조합을 표준으로 제안해야 한다. API 계약(OpenAPI Spec)을 기준으로 BFF(Backend for Frontend) 패턴을 적용하면 고득점 답안을 완성할 수 있다.

**Pipe-and-Filter 및 Repository 패턴 적용**
- **챌린지**: Pipe-and-Filter는 데이터 스트림 처리에 최적이나 필터 간 상태 공유가 어렵고, Repository 패턴은 데이터 일관성에 유리하나 저장소 병목 위험이 있다.
- **제언**: 컴파일러·ETL 파이프라인은 Pipe-and-Filter, DBMS 중심 시스템이나 IDE(여러 도구가 AST 공유)는 Repository 패턴을 적용하되, 병목 구간에 캐싱 계층을 추가하여 성능 트레이드오프를 완화해야 한다.
