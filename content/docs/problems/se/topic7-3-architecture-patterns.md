## 해설

> **소프트웨어 세계의 도시 설계 도면**  
> 아키텍처 패턴은 반복적으로 등장하는 설계 문제에 대한 검증된 해결책의 템플릿이다

도시를 설계할 때 주거지역, 상업지역, 공업지역을 분리하듯, 소프트웨어도 구조를 어떻게 분리하고 연결하느냐에 따라 유지보수성과 성능이 극적으로 달라집니다. Layer 패턴은 책임의 분리를, Broker 패턴은 분산 서비스의 투명한 연결을, MVC 패턴은 UI와 비즈니스 로직의 독립을 보장합니다.

### 1. 출제 배경 및 의도

소프트웨어 아키텍처 패턴은 1996년 Buschmann 등이 저술한 "Pattern-Oriented Software Architecture (POSA)" 시리즈에서 체계화되었습니다. 기술사 시험에서 이 문제가 반복 출제되는 이유는 패턴 선택이 시스템의 품질 속성(성능/변경용이성/이식성)에 직접적인 영향을 미치며, 이를 실무 맥락에서 트레이드오프를 설명할 수 있는 역량이 아키텍트의 핵심 역량이기 때문입니다.

채점관이 기대하는 것은 단순한 패턴 정의가 아니라 **각 패턴이 어떤 문제를 해결하는지, 그리고 어떤 상황에서는 적합하지 않은지**를 구체적 사례와 함께 설명하는 것입니다.

### 2. 개념의 본질과 작동 메커니즘

**아키텍처 패턴(Architecture Pattern)이란** 특정 설계 상황에서 반복적으로 발생하는 문제에 대해 검증된 설계 구조를 일반화한 템플릿입니다. 디자인 패턴(GoF)이 클래스/객체 수준의 해법이라면, 아키텍처 패턴은 시스템 전체 구조 수준의 해법입니다.

```text
[아키텍처 패턴 계층]

  비즈니스 아키텍처 패턴
  ┌──────────────────────────────────────────┐
  │  MSA   │  SOA   │  Event-Driven  │  CQRS │
  └──────────────────────────────────────────┘
              ↑ 조합
  구조적 아키텍처 패턴
  ┌──────────────────────────────────────────┐
  │  Layered │  Broker  │  MVC  │  P&F  │ Repo│
  └──────────────────────────────────────────┘
              ↑ 구현
  설계 패턴 (GoF 23개)
  ┌──────────────────────────────────────────┐
  │  Singleton │ Observer │ Facade │ Strategy │
  └──────────────────────────────────────────┘
```

### 3. 핵심 키워드 (Must-Have)

#### 1) Layer 패턴 (계층형 아키텍처)

**개념**: 시스템을 기능적으로 유사한 계층(Layer)으로 분리하고, 각 계층은 바로 아래 계층의 서비스만 사용하는 구조입니다.

```text
[전형적인 4계층 아키텍처]

  ┌─────────────────────────────────┐
  │   프레젠테이션 계층(Presentation) │  ← 사용자 인터페이스 (Web, App)
  ├─────────────────────────────────┤
  │   애플리케이션 계층(Application)  │  ← 유스케이스, 워크플로우 조정
  ├─────────────────────────────────┤
  │   도메인 계층(Domain)            │  ← 핵심 비즈니스 로직, 엔티티
  ├─────────────────────────────────┤
  │   인프라스트럭처 계층(Infrastructure)│← DB, 외부 API, 메시징
  └─────────────────────────────────┘
  ※ 의존성 방향: 위 계층 → 아래 계층 (단방향)
```

| 구분 | 내용 |
|------|------|
| **장점** | - 관심사 분리(SoC)로 변경 영향 최소화<br>- 계층별 독립적 테스트 용이<br>- 계층 교체 가능 (예: DB 변경)<br>- 표준화된 구조로 신규 개발자 온보딩 쉬움 |
| **단점** | - 모든 요청이 모든 계층을 통과 → 성능 오버헤드<br>- 단순 CRUD도 4계층 통과 → 개발 생산성 저하<br>- 계층 간 DTO 변환 코드 증가 (Boilerplate)<br>- 모놀리식 구조로 부분 확장(Scale-Out) 어려움 |
| **적용 대상** | 엔터프라이즈 애플리케이션, 공공 정보 시스템 |

#### 2) Broker 패턴

**개념**: 분산 환경에서 서버-클라이언트 간 직접 통신 대신 브로커(Broker)가 중개자 역할을 수행하여 위치 투명성(Location Transparency)을 제공하는 패턴입니다.

```text
[Broker 패턴 구조]

  클라이언트                               서버
  ┌──────────┐                       ┌──────────┐
  │  Client  │──요청 전송──┐   ┌──전달──│ Server A │
  │  Proxy   │             │   │       └──────────┘
  └──────────┘             ▼   ▼       ┌──────────┐
                        ┌────────┐ ──→ │ Server B │
  ┌──────────┐           │BROKER  │     └──────────┘
  │  Client  │──요청 전송──│(중개자)│     ┌──────────┐
  │  Proxy   │             └────────┘ ──→ │ Server C │
  └──────────┘                 │          └──────────┘
                               │  서버 등록/탐색
                          ┌────────┐
                          │  등록부  │
                          │(Registry)│
                          └────────┘
```

| 구분 | 내용 |
|------|------|
| **장점** | - 서비스 위치 투명성 (클라이언트가 서버 위치 몰라도 됨)<br>- 서버 추가/제거 시 클라이언트 변경 불필요<br>- 로드밸런싱, 장애 복구를 브로커에서 중앙 처리 가능 |
| **단점** | - 브로커가 단일 장애점(SPOF: Single Point of Failure)<br>- 브로커 통신 오버헤드로 성능 저하<br>- 브로커 자체의 고가용성 설계 필요 |
| **현대적 구현** | API Gateway(Kong, AWS API GW), ESB, CORBA, gRPC |
| **적용 대상** | 마이크로서비스 API Gateway, 분산 객체 시스템 |

#### 3) MVC 패턴 (Model-View-Controller)

**개념**: 사용자 인터페이스와 비즈니스 로직을 분리하기 위해 시스템을 Model(데이터/비즈니스), View(표현), Controller(제어 흐름)의 세 컴포넌트로 분리하는 패턴입니다.

```text
[MVC 패턴 흐름]

  사용자 입력
       │
       ▼
  ┌────────────┐   이벤트 처리   ┌───────────┐
  │  Controller│────────────→  │   Model   │
  │  (제어자)  │                │  (데이터) │
  └────────────┘ ←── 상태 변경 ──│  비즈니스 │
       │                        │  로직     │
       │                        └───────────┘
       │                             │
       ▼                             │ 변경 알림
  ┌────────────┐  ←──────────────────┘
  │    View    │  데이터 조회
  │  (화면)    │────────────────────→ 화면 렌더링
  └────────────┘
```

| 구분 | 내용 |
|------|------|
| **장점** | - UI와 비즈니스 로직의 완전한 분리<br>- View 다변화 용이 (웹/앱/콘솔 동일 Model 공유)<br>- 단위 테스트 용이 (Model 독립 테스트)<br>- 다수 개발자 병렬 작업 가능 |
| **단점** | - Controller 비대화 현상 (Massive View Controller)<br>- 단순 앱에 과도한 구조 (Overengineering)<br>- View-Model 간 의존성이 완전히 제거되지 않는 경우 발생 |
| **파생 패턴** | MVP(Model-View-Presenter), MVVM(Model-View-ViewModel) |
| **적용 대상** | 웹 프레임워크(Spring MVC, Django, Rails), GUI 애플리케이션 |

#### 추가 주요 패턴 (심화)

**Pipe-and-Filter 패턴**:

```text
Input ──→ [Filter1: 파싱] ──→ [Filter2: 변환] ──→ [Filter3: 출력] ──→ Output
              (Pipe)               (Pipe)               (Pipe)
```
- 데이터 스트림 처리에 최적 (컴파일러, ETL 파이프라인, Unix 쉘)
- 필터 재조합으로 처리 로직 유연 변경 가능

**Repository(공유 저장소) 패턴**:
- 모든 컴포넌트가 중앙 데이터 저장소를 공유
- 데이터 일관성 보장에 유리, 저장소가 병목 위험
- DBMS 중심 시스템, IDE(여러 도구가 AST 공유)

### 4. 맥락과 비교

| 패턴 | 핵심 구조 | 최적 품질속성 | 취약점 | 현대적 구현 |
|------|---------|------------|-------|-----------|
| Layered | 계층 스택 | 변경용이성, 이식성 | 성능, 모놀리식 확장 | Spring Boot, Django |
| Broker | 중개자 + Registry | 분산 투명성, 확장성 | SPOF, 성능 오버헤드 | API Gateway, K8s Service |
| MVC | Model-View-Controller 삼각형 | 분리성, 테스트 용이성 | Controller 비대화 | Spring MVC, React+Redux |
| MSA | 독립 서비스 메시 | 독립 배포, 확장성 | 분산 복잡성, 운영 비용 | Docker, Kubernetes |
| EDA | 이벤트 버스 | 느슨한 결합, 반응성 | 디버깅, 순서 보장 | Kafka, RabbitMQ |
| Pipe-and-Filter | 파이프 + 필터 체인 | 재사용성, 병렬성 | 상태 공유, 에러 전파 | Apache Beam, Unix pipe |

### 5. 실무 제언

**아키텍처 패턴 선택의 함정: 트렌드 추종**

- **챌린지**: 2024년 현재 MSA(마이크로서비스 아키텍처)는 사실상 디폴트 선택처럼 여겨지지만, Amazon이 Prime Video의 MSA를 모놀리식으로 전환하여 비용을 90% 절감한 사례처럼, 무조건적인 MSA 도입이 오히려 더 큰 운영 비용과 분산 시스템 복잡성을 초래하는 경우가 빈번합니다.
- **제언**: 패턴 선택은 **시스템의 품질속성 요구사항(QAS)**에서 출발해야 합니다. "독립 배포가 반드시 필요한가?", "팀이 독립적으로 개발·운영할 수 있는 규모인가?", "분산 트랜잭션을 감당할 수 있는가?"라는 질문에 먼저 답해야 합니다. Strangler Fig Pattern처럼 모놀리식에서 점진적으로 MSA로 전환하는 단계적 접근이 현실적인 대안입니다.

**MVC의 진화: 프론트엔드 분리 아키텍처**

- **챌린지**: 전통적인 서버사이드 MVC(Spring MVC)는 React, Vue, Angular 등 SPA(Single Page Application) 프레임워크의 등장으로 그 경계가 흐려지고 있습니다. 프론트엔드에서 자체적으로 MVVM 패턴을 구현하고, 백엔드는 REST API 또는 GraphQL 서버로만 동작하는 구조가 일반화되었습니다.
- **제언**: 현대 웹 아키텍처에서는 **백엔드: 클린 아키텍처(Layered + DDD) + 프론트엔드: MVVM(React/Vue)** 조합을 표준으로 제안해야 합니다. API 계약(OpenAPI Spec)을 기준으로 프론트엔드와 백엔드를 완전히 분리 개발하는 BFF(Backend for Frontend) 패턴도 함께 언급하면 고득점 답안을 완성할 수 있습니다.
