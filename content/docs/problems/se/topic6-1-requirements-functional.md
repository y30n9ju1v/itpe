## 해설

### 출제 배경 및 의도

클라우드·MSA·AI 환경에서 보안·성능·확장성·가용성 등 비기능적 요건이 비즈니스의 생사를 가르는 시대가 되었다. 과거 업무 자동화 위주 시스템에서는 기능적 요건 구현만으로 품질을 인정받았으나, 현대 아키텍처에서는 비기능 요건의 누락이 시스템 전체를 침몰시킨다.

채점관의 출제 의도는 요구사항을 수동적으로 받아 적는 '서기' 수준을 넘어, 비기능적 숨은 요건까지 능동적으로 발굴(Elicitation)하고 고객과의 소통 장벽을 조율하는 요구공학 전문가·수석 아키텍트 자질을 검증하는 것이다.

### 1. 요구사항 분류 개요

정 의  • 소프트웨어 시스템 구축을 위해 이해관계자가 필요로 하는 조건과 역량의 집합으로, 기능적(Functional)과 비기능적(Non-Functional) 요구사항으로 대분류
       - 기능은 "무엇을(What)", 비기능은 "어떻게 잘(How Well)" 동작해야 하는지 정의

```
[요구사항 분류 체계]

  요구사항 (Requirements)
       │
       ├──▶ 기능적 요구사항 (Functional)
       │         │
       │         ├── 자료 처리 로직
       │         ├── 사용자 인터페이스 기능
       │         └── 외부 시스템 연계 기능
       │
       └──▶ 비기능적 요구사항 (Non-Functional)
                 │         [ISO/IEC 25010 기반]
                 ├── 성능 효율성 (Performance)
                 ├── 보안성 (Security)
                 ├── 신뢰성 (Reliability)
                 ├── 유지보수성 (Maintainability)
                 └── 사용성 (Usability) 등
```

- 요구사항의 정밀한 분류가 프로젝트 범위 확정과 품질 목표 설정의 출발점이다

### 2. 가. 기능적/비기능적 요구사항 분류 및 사례

1) 기능적 요구사항 (Functional Requirements)

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 행위 명세 | 시스템이 무엇(What)을 해야 하는지 입력·출력·프로세스 기술 |
| 주요 분류 | 처리·연계·출력 | 자료 처리 로직, UI 기능, 외부 시스템 연계, 조회/출력 제어 |
| 표현 방식 | 사용자 스토리·유스케이스 | 동사형 행위 기술 ("~해야 한다") |

**금융 커머스 도메인 실무 사례**
- "고객이 결제 버튼을 누르면 잔액 조회 후 결제 처리하고, 알림톡을 발송해야 한다"
- "관리자는 미승인 회원의 로그인 권한을 정지할 수 있어야 한다"

- 기능적 요구사항 누락 시 해당 기능을 사용하지 못하는 국소적 영향에 그친다

2) 비기능적 요구사항 (Non-Functional Requirements)

| 구분 | 항목 | 실무 사례 (금융 커머스) |
|------|------|----------------------|
| 성능 | 응답 시간 | 동시 접속자 10,000명 기준 결제 API 평균 응답 2초 이하 |
| 보안성 | 암호화 | 개인정보(주민번호·계좌번호) DB 저장 시 AES-256 필수 |
| 가용성 | 가동률 | 연간 총 가동률 99.99%(Four Nines) 이상 유지 |
| 유지보수성 | MTTR | 장애 발생 후 평균 복구 시간 30분 이내 |

- 비기능 요구사항 누락 시 시스템 전체 침몰 또는 법적 제재로 이어지는 치명적 리스크가 발생한다

### 3. 나. 요구사항 추출의 현실적 어려움

1) 4대 추출 난제

| 구분 | 난제 | 설명 |
|------|------|------|
| ① | 암묵적 요구 (Implicit Needs) | 고객은 업무 전문가이나 IT 언어로 번역 불가 — "이쁘고 빠르게" 수준의 모호한 표현 |
| ② | 변동성 (Volatility) | 시장 상황·법률 개정·경영진 교체로 요구사항 수시 변동, Scope Creep 위험 |
| ③ | 이해관계자 상충 (Conflict) | 마케팅(간편 로그인) vs 보안(3차 인증) 등 부서 간 요구 충돌 |
| ④ | 비기능 과소평가 | 성능·보안 요건을 "당연히 된다"고 가정, 오픈 직전 부하 테스트에서 서버 다운 참사 |

- 추출 단계의 실패는 프로젝트 전 생애주기의 결함으로 기하급수적 비용을 발생시킨다.  "끝"

### 실무 제언

**비기능 요구사항 — 품질 속성 시나리오(QAS) 정량 명세화**
- **챌린지**: "안전하고 빠르게 구축"이라는 추상적 표현은 감리·검수 단계에서 발주처-수행사 간 분쟁의 씨앗이 된다
- **제언**: 모든 비기능 요건을 QAS(Quality Attribute Scenario) 템플릿 `[자극원]-[자극]-[환경]-[대상]-[응답]-[측정지표]` 형식으로 정량화한다 (예: "해커가 초당 10,000건 DoS 공격 시 방화벽이 1초 내 IP 격리, 정상 유저 응답 3초 이내 유지")

**진화형 프로토타이핑으로 추출 병목 해소**
- **챌린지**: 문서·말만으로 소통하면 고객은 완성 후에야 "이게 아니다"라며 밥상을 엎는다
- **제언**: Figma 등을 활용한 진화형 프로토타이핑(Evolutionary Prototyping)으로 주 단위 더미 화면을 고객에게 릴리즈하여 피드백 루프를 단축하고, RTM(요구사항 추적 매트릭스)을 상시 동기화하여 불법적 과업 변경 압박으로부터 범위를 과학적으로 방어한다
