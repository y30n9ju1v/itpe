## 문제

- 가. 변경유형에 따른 S/W 유지보수의 분류

## 해설

> **소프트웨어의 80%는 만든 후에 시작된다**  
> ISO/IEC 14764는 소프트웨어 유지보수를 개발의 끝이 아닌, 생명주기의 핵심 단계로 정의한다.

소프트웨어 개발 비용 중 60~80%가 유지보수에 쓰인다는 것은 업계의 잘 알려진 통계입니다. 건물을 짓는 것보다 유지 관리하는 데 더 많은 비용이 드는 것과 같습니다. 그러나 많은 조직이 유지보수를 체계 없이 임기응변식으로 수행합니다. ISO/IEC 14764는 ISO/IEC 12207(소프트웨어 생명주기 프로세스, SW 개발부터 폐기까지의 전체 과정을 다루는 표준)의 유지보수 프로세스를 구체화한 국제 표준으로, 변경 유형의 분류부터 체계적인 유지보수 프로세스까지 정의합니다.

### 1. 출제 배경 및 의도

이 문제는 세 가지 관점을 동시에 묻는 종합 문제입니다.

1. **분류(가)**: 4가지 유지보수 유형을 정확히 구분하고, 각각의 예시를 들 수 있는가
2. **기법(나)**: 유지보수를 수행하는 구체적인 기술(리팩토링, 리엔지니어링 등)을 알고 있는가
3. **프로세스(다)**: 유지보수 요청부터 완료까지의 체계적 절차를 설명할 수 있는가

채점관은 특히 분류에서 학생들이 가장 많이 혼동하는 **완전화(Perfective) vs. 예방적(Preventive)** 유지보수의 차이를 명확히 이해하는지 봅니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 변경 유형에 따른 S/W 유지보수 분류 (4유형)

ISO/IEC 14764는 소프트웨어 유지보수를 변경의 목적과 성격에 따라 4가지로 분류합니다.

```text
[유지보수 4유형 분류]

발생 원인 ────────────────────────────────────→
    결함(Bug)                   환경/요구 변화

    수정적                  완전화
  (Corrective)           (Perfective)
  결함 수정               기능/성능 개선
    ↕                          ↕
  긴급 수정              예방적(Preventive)
  (Emergency Fix)         리팩토링/재구조화
         │
         └── 적응적(Adaptive)
              환경 변화 대응
```

**① 수정적 유지보수 (Corrective Maintenance)**
- 목적: 발견된 **결함(Fault/Bug)** 수정
- 원인: 개발 단계에서 발견되지 못한 오류가 운영 중 발견
- 예시: 결제 금액 계산 오류 수정, 로그인 실패 버그 수정, 데이터 손상 버그 패치
- 특징: 사용자에게 가장 직접적으로 인식되는 유지보수

**② 적응적 유지보수 (Adaptive Maintenance)**
- 목적: 변화된 **운영 환경**에 소프트웨어를 맞추는 작업
- 원인: OS 업그레이드, 하드웨어 교체, 법규 변경, 외부 시스템 인터페이스 변경
- 예시: Windows 7에서 Windows 11로 OS 변경에 따른 소프트웨어 수정, GDPR(개인정보보호법) 준수를 위한 시스템 변경
- 특징: 소프트웨어 자체의 기능은 변경되지 않지만, 새로운 환경에서 동작하게 함

**③ 완전화 유지보수 (Perfective Maintenance)**
- 목적: 사용자의 **신규 요구사항** 또는 성능 개선 요청 반영
- 원인: 비즈니스 확장, 사용자 피드백, 경쟁 환경 변화
- 예시: 모바일 앱에 다크모드 추가, 검색 기능 성능 개선, 새로운 보고서 양식 추가
- 특징: 가장 많은 비율(전체 유지보수의 약 50~65%)을 차지
- **수정적과의 차이**: 수정적은 '존재하는 기능의 오류 수정', 완전화는 '새로운 기능 추가 또는 기존 기능 개선'

**④ 예방적 유지보수 (Preventive Maintenance)**
- 목적: 미래의 결함을 **예방**하기 위해 소프트웨어 내부 구조 개선
- 원인: 복잡도 증가 경고, 향후 변경 용이성 확보
- 예시: 레거시 코드 리팩토링, 모듈 재설계, 문서화 개선, 기술 부채 해소
- **완전화와의 차이**: 완전화는 사용자가 요청한 기능 개선, 예방적은 내부 품질 개선(사용자에게 눈에 보이지 않음)

| 유형 | 목적 | 원인 | 예시 | 비율 |
|------|------|------|------|------|
| 수정적 | 결함 수정 | 운영 중 오류 발견 | 버그 패치 | 20% |
| 적응적 | 환경 적응 | OS/법규/인터페이스 변경 | OS 업그레이드 대응 | 25% |
| 완전화 | 기능/성능 개선 | 사용자 요구사항 추가 | 신기능 추가 | 50% |
| 예방적 | 미래 결함 예방 | 복잡도 증가 경고 | 리팩토링 | 5% |

#### 나. S/W 유지보수의 기법 (3가지 이상)

**① 리팩토링(Refactoring)**
- 소프트웨어의 외부 동작을 변경하지 않으면서 내부 코드 구조를 개선
- 예방적 유지보수의 핵심 기법
- 대표적 리팩토링 기법: 메서드 추출(Extract Method), 클래스 분리(Extract Class), 중복 코드 제거(Remove Duplication)

**② 리엔지니어링(Re-Engineering, 재공학)**
- 레거시 소프트웨어를 분석하여 더 나은 형태로 재구성하는 과정
- 역공학(Reverse Engineering, 코드에서 설계를 역으로 복원) + 재구조화(Restructuring) + 순공학(Forward Engineering, 설계를 다시 구현으로 변환)의 조합

```text
[리엔지니어링 프로세스]

레거시 시스템
    ↓ 역공학(Reverse Engineering)
추상화된 설계/명세 추출
    ↓ 재구조화(Restructuring)
개선된 설계
    ↓ 순공학(Forward Engineering)
새로운 시스템
```

**③ 역공학(Reverse Engineering)**
- 기존 소스코드를 분석하여 설계 문서, 명세, 데이터 흐름도 등을 복원
- 문서가 없는 레거시 시스템을 이해하기 위해 사용
- 도구: SonarQube(정적 분석), Understand(의존성 분석)

**④ 형상 관리(Configuration Management)**
- 유지보수 과정의 모든 변경 사항을 추적하고 통제
- 유지보수 기법이라기보다 유지보수를 지원하는 필수 인프라
- 도구: Git, SVN, Jira(변경 추적)

**⑤ 회귀 테스트(Regression Test)**
- 유지보수 변경 후 기존 기능이 손상되지 않았음을 검증
- 자동화 필수: JUnit, Selenium 등

#### 다. S/W 유지보수 프로세스 (ISO/IEC 14764)

```text
[ISO/IEC 14764 유지보수 프로세스]

  1. 유지보수 준비(Implementation)
     └─ 유지보수 계획 수립, 지원 조직 구성, SLA 정의

  2. 문제 및 수정 분석(Problem and Modification Analysis)
     └─ 변경 요청(MR: Modification Request) 접수
     └─ 영향도 분석, 우선순위 결정, 타당성 검토

  3. 수정 구현(Modification Implementation)
     └─ 설계 변경
     └─ 코딩 및 단위 테스트

  4. 유지보수 검토/인수(Maintenance Review/Acceptance)
     └─ 변경 검토, 통합/회귀 테스트
     └─ 고객 인수 테스트(UAT)

  5. 이행(Migration)
     └─ 새 환경으로의 이전이 필요한 경우 (적응적 유지보수 시)
     └─ 이행 계획, 데이터 변환, 사용자 교육

  6. 폐기(Retirement)
     └─ 소프트웨어 서비스 종료
     └─ 데이터 아카이빙, 대체 시스템으로 마이그레이션
```

### 3. 핵심 키워드 (Must-Have)

| 키워드 | 설명 |
|--------|------|
| 수정적(Corrective) | 결함 수정 |
| 적응적(Adaptive) | 환경 변화 대응 |
| 완전화(Perfective) | 기능/성능 개선 (가장 높은 비율) |
| 예방적(Preventive) | 미래 결함 예방, 내부 품질 개선 |
| 변경 요청(MR: Modification Request) | 유지보수 프로세스의 시작점 |
| 리팩토링 | 동작 유지, 내부 구조 개선 |
| 리엔지니어링 | 레거시 시스템의 근본적 재구성 |
| SLA (Service Level Agreement) | 유지보수 서비스 수준 협약 |

### 4. 맥락과 비교

**ISO/IEC 14764 vs. ITSM (IT Service Management)**

IT 서비스 관리 관점에서 소프트웨어 유지보수는 ITSM의 변경 관리(Change Management), 문제 관리(Problem Management)와 깊이 연관됩니다.

| 관점 | ISO/IEC 14764 | ITSM (ITIL) |
|------|-------------|-----------|
| 수정적 유지보수 | 결함 수정 프로세스 | 인시던트 관리 + 문제 관리 |
| 완전화 유지보수 | 기능 개선 프로세스 | RFC(Request for Change) + 변경 관리 |
| 적응적 유지보수 | 환경 변화 대응 | 변경 관리 |
| 공통 기반 | 형상 관리 | CMDB(Configuration Management DB) |

```text
[ITSM 서비스 데스크와 유지보수 유형의 연계]

사용자 연락
    ↓
서비스 데스크 접수
    ↓
인시던트(Incident)인가? ─── YES → 수정적 유지보수
    │ NO
    ↓
서비스 요청(Service Request)인가? ─── YES → 완전화 유지보수
    │ NO
    ↓
변경 요청(RFC)인가? ─── YES → 적응적/예방적 유지보수
```

### 5. 실무 제언

**유지보수 비용의 레버리지 포인트: 예방적 유지보수의 투자**

전체 유지보수 중 예방적 유지보수 비율은 약 5%에 불과합니다. 이는 대부분의 조직이 예방적 유지보수에 충분히 투자하지 않는다는 의미입니다. 그러나 Lehman의 법칙 2(복잡도 증가)와 7(품질 저하)이 시사하듯, 예방적 유지보수에 투자하지 않으면 수정적 유지보수와 완전화 유지보수 비용이 기하급수적으로 증가합니다. '리팩토링 스프린트' 또는 'Tech Debt Day'를 제도화하는 것이 장기적 유지보수 비용 절감의 핵심 레버리지입니다.

**유지보수 SLA와 MTTR(Mean Time To Repair) 관리**

유지보수 계약에서 가장 중요한 지표는 장애 심각도별 MTTR(평균 복구 시간)입니다. 예를 들어:
- 심각도 1 (Critical): MTTR 4시간 이내
- 심각도 2 (High): MTTR 24시간 이내
- 심각도 3 (Medium): MTTR 72시간 이내

이 SLA를 실현하기 위해서는 자동화된 모니터링(APM), 온콜(On-Call) 체계, CI/CD 파이프라인을 통한 빠른 핫픽스 배포 능력이 갖춰져야 합니다. ISO/IEC 14764의 유지보수 프로세스와 이 운영 인프라를 연계하는 것이 현대적 소프트웨어 유지보수의 핵심입니다.
