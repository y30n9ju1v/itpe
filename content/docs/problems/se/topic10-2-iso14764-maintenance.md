## 문제

ISO/IEC 14764는 ISO/IEC 12207의 유지보수 부분을 상세화 한, S/W 유지보수의 표준 프로세스이다. ISO/IEC 14764 측면에서 아래를 설명하시오.

- 가. 변경유형에 따른 S/W 유지보수의 분류
- 나. S/W 유지보수의 기법 3가지 이상

## 해설

### 출제 배경 및 의도

소프트웨어 개발 비용 중 60~80%가 유지보수에 소요된다는 사실은 유지보수가 개발의 끝이 아닌 생명주기의 핵심임을 말해준다. 그러나 많은 조직이 체계 없이 임기응변식으로 유지보수를 수행하여 비용이 통제 불능 상태가 된다.

ISO/IEC 14764는 ISO/IEC 12207의 유지보수 프로세스를 구체화한 국제 표준이다. 이 문제는 변경 유형 분류(가), 유지보수 기법(나), 표준 프로세스(다)의 세 관점을 종합적으로 묻는다. 채점관은 특히 완전화(Perfective)와 예방적(Preventive) 유지보수의 차이, 그리고 리팩토링·리엔지니어링·역공학의 관계를 명확히 구분하는지를 집중 평가한다.

### 1. ISO/IEC 14764 S/W 유지보수 개요

정 의  • ISO/IEC 14764를 ISO/IEC 12207의 유지보수 프로세스를 구체화하여 변경 유형 분류, 유지보수 기법, 표준 프로세스를 정의한 소프트웨어 유지보수 국제 표준이라 한다.
       - 전체 소프트웨어 생명주기 비용의 60~80%를 차지하는 유지보수의 체계적 관리 기반을 제공한다.

```
[ISO/IEC 14764 유지보수 4유형 분류 구조]

         발생 원인
  결함(Bug) ◄─────────────────────► 환경/요구 변화

  ┌──────────────┐              ┌──────────────────┐
  │ ① 수정적     │              │ ③ 완전화         │
  │ (Corrective) │              │ (Perfective)     │
  │ 결함 수정    │              │ 기능/성능 개선   │
  │ 비율: 20%   │              │ 비율: 50%        │
  └──────────────┘              └──────────────────┘
  ┌──────────────┐              ┌──────────────────┐
  │ ② 적응적     │              │ ④ 예방적         │
  │ (Adaptive)  │              │ (Preventive)     │
  │ 환경 변화   │              │ 미래 결함 예방   │
  │ 비율: 25%   │              │ 비율: 5%         │
  └──────────────┘              └──────────────────┘
```

- 완전화(사용자 요구 기능 개선)와 예방적(내부 품질 개선, 사용자에게 비가시적) 유지보수를 명확히 구분하는 것이 핵심이다.

### 2. 유지보수 분류, 기법, 프로세스

1) 가. 변경 유형에 따른 분류 (4유형)

| 유형 | 목적 | 발생 원인 | 예시 | 비율 |
|------|------|---------|------|------|
| ① 수정적(Corrective) | 결함 수정 | 운영 중 오류 발견 | 결제 계산 오류 패치, 로그인 버그 수정 | 20% |
| ② 적응적(Adaptive) | 환경 적응 | OS/법규/인터페이스 변경 | Win7→Win11 대응, GDPR 준수 변경 | 25% |
| ③ 완전화(Perfective) | 기능/성능 개선 | 사용자 신규 요구사항 | 다크모드 추가, 검색 성능 개선 | 50% |
| ④ 예방적(Preventive) | 미래 결함 예방 | 복잡도 증가 경고 | 레거시 코드 리팩토링, 기술 부채 해소 | 5% |

- 수정적: '존재 기능 오류 수정' vs 완전화: '신기능 추가·기존 기능 개선'으로 명확히 구분한다.

2) 나. S/W 유지보수 기법 (3가지 이상)

| 기법 | 정의 | 대표 활동 |
|------|------|---------|
| ① 리팩토링(Refactoring) | 외부 동작 유지, 내부 구조 개선 | Extract Method, Extract Class, Remove Duplication |
| ② 리엔지니어링(Re-Engineering) | 역공학+재구조화+순공학의 결합 | 레거시 COBOL 시스템 현대화 |
| ③ 역공학(Reverse Engineering) | 코드→설계→요구사항 역방향 복원 | SonarQube 정적 분석, UML 역생성 |
| ④ 형상 관리(CM) | 모든 변경 사항 추적·통제 | Git, SVN, Jira |
| ⑤ 회귀 테스트(Regression Test) | 변경 후 기존 기능 손상 여부 검증 | JUnit, Selenium 자동화 |

- 리엔지니어링 = 역공학(레거시 이해) + 재구조화(설계 개선) + 순공학(새 구현)의 3단계 프로세스이다.

### 3. ISO/IEC 14764 유지보수 표준 프로세스 및 ITSM 연계

1) 다. 표준 유지보수 프로세스

```
[ISO/IEC 14764 유지보수 프로세스 흐름]

① 유지보수 준비(Implementation)
   └─ 유지보수 계획, 조직 구성, SLA 정의

② 문제 및 수정 분석(Problem/Modification Analysis)
   └─ MR(Modification Request) 접수 → 영향도 분석 → 우선순위 결정

③ 수정 구현(Modification Implementation)
   └─ 설계 변경 → 코딩 → 단위 테스트

④ 유지보수 검토/인수(Review/Acceptance)
   └─ 통합 테스트 → 회귀 테스트 → UAT

⑤ 이행(Migration) ← 적응적 유지보수 시
   └─ 이행 계획 → 데이터 변환 → 사용자 교육

⑥ 폐기(Retirement)
   └─ 데이터 아카이빙 → 대체 시스템 마이그레이션
```

2) ISO/IEC 14764 vs ITSM 연계

| 유지보수 유형 | ISO/IEC 14764 | ITSM(ITIL) |
|------------|--------------|-----------|
| 수정적 | 결함 수정 프로세스 | 인시던트 관리 + 문제 관리 |
| 완전화 | 기능 개선 프로세스 | RFC + 변경 관리 |
| 적응적 | 환경 변화 대응 | 변경 관리 |
| 공통 기반 | 형상 관리 | CMDB |

- 예방적 유지보수 비율은 전체의 5%에 불과하지만 투자 없으면 수정적·완전화 비용이 기하급수적으로 증가하므로 'Tech Debt Day' 제도화가 핵심 레버리지이다.  "끝"

### 실무 제언

**예방적 유지보수 투자 부족 문제**
- **챌린지**: 전체 유지보수 중 예방적 비율은 5%에 불과하여 기술 부채가 쌓인다.
- **제언**: Lehman 법칙 2·7에 근거하여 예방적 유지보수 비용을 경영진에 수치로 제시하고, 분기별 '리팩토링 스프린트'를 정례화하여 기술 부채 해소를 제도화한다.

**유지보수 SLA와 MTTR 관리**
- **챌린지**: 장애 복구 시간(MTTR)을 SLA로 약속했으나 레거시 코드로 핫픽스가 어렵다.
- **제언**: CI/CD 파이프라인과 자동화 테스트(회귀 테스트)를 구축하여 Critical MTTR 4시간 이내를 달성하고, APM(Application Performance Monitoring)으로 장애를 사전 탐지한다.

**레거시 시스템 리엔지니어링 실패 방지**
- **챌린지**: 문서 없는 레거시 COBOL 시스템을 신규 개발로 대체하다 사업 논리 누락으로 프로젝트가 실패한다.
- **제언**: 역공학으로 숨겨진 비즈니스 로직을 먼저 문서화하고, 스트랭글러 피그(Strangler Fig) 패턴으로 점진적 전환하여 위험을 최소화한다.
