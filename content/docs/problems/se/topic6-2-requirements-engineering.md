## 해설

### 출제 배경 및 의도

디지털 플랫폼 정부·금융권 차세대 시스템 등 대규모 정보화 사업의 복잡도가 극에 달한 현재, 프로젝트 실패 원인의 70% 이상이 잘못된 요구사항 분석에서 비롯된다는 것은 업계의 정설이다. 요구공학은 고객의 모호한 바람을 구현 가능한 논리적 명세로 변환하고 개발 전 기간 일관성을 유지하는 고도의 엔지니어링 절차다.

채점관의 출제 핵심은 요구사항을 어떻게 개발(Develop)하고 통제(Manage)할 것인지 전체 프로젝트 기간의 거버넌스 능력을 갖추었는지 평가하는 것이다.

### 1. 요구공학(Requirements Engineering) 개요

정 의  • 이해관계자의 모호한 요구를 시스템이 구현 가능한 논리적 명세로 변환하고, 개발이 끝날 때까지 일관성을 유지하는 체계적인 엔지니어링 절차
       - 요구사항 개발(Development)과 요구사항 관리(Management)의 두 축으로 구성

```
[요구공학 전체 프로세스]

  요구사항 개발 (Development)
  ┌──────────────────────────────────────────────────┐
  │ 도출 ──▶ 분석 ──▶ 명세 ──▶ 확인                  │
  │(Elicitation)(Analysis)(Specification)(Validation)│
  └──────────────────────────────────────────────────┘
                          │
                          ▼
  요구사항 관리 (Management)
  ┌────────────────────────────────────┐
  │ 변경 관리(CR) ──▶ 추적성 관리(RTM) │
  └────────────────────────────────────┘
```

- 요구공학은 프로젝트 범위의 기준선(Baseline)을 확정하고 변경을 통제하는 핵심 거버넌스다

### 2. 가. 요구사항의 분류

1) 기능적·비기능적 요구사항

| 구분 | 기능적 요구사항 | 비기능적 요구사항 |
|------|--------------|----------------|
| 개념 | 시스템이 제공해야 할 서비스·입출력·프로세스 행위 명세 | 품질 속성·성능·보안 등 시스템이 갖추어야 할 품질 수준 |
| 의문문 | What? (무엇을 하는가?) | How Well? (얼마나 잘 하는가?) |
| 사례 | "사용자는 휴대폰 인증을 수행해야 한다" | "평균 응답 시간 1초 이내(성능)" |

- 기능 요건은 국소적 영향, 비기능 요건 누락은 시스템 전체 침몰의 치명적 리스크를 초래한다

2) 사용자 요구사항 vs 시스템 요구사항

| 구분 | 사용자 요구사항 | 시스템 요구사항 |
|------|--------------|--------------|
| 수준 | 상위 (Why — 목적·목표) | 상세 (How — 기술적 구현 명세) |
| 주체 | 비즈니스 담당자·현업 사용자 | 개발자·시스템 아키텍트 |

### 3. 나. 요구공학 프로세스 및 주요기법

1) 요구사항 개발 4단계

| 구분 | 단계 | 주요 기법 | 산출물 |
|------|------|---------|--------|
| ① | 도출 (Elicitation) | 인터뷰·워크숍·관찰·프로토타이핑 | 요구사항 목록(초안) |
| ② | 분석 (Analysis) | 우선순위화·실현 가능성(Feasibility)·충돌 분석 | 분석 보고서 |
| ③ | 명세 (Specification) | SRS(소프트웨어 요구사항 명세서) 작성, 정형/비정형 언어 | SRS 문서 |
| ④ | 확인 (Validation) | 이해관계자 검토·승인, 프로토타입 검증 | 승인된 SRS |

2) 요구사항 관리 단계

| 구분 | 항목 | 설명 |
|------|------|------|
| 변경 관리 | CR(Change Request) | 변경 요청 영향도 분석 후 CCB 승인/기각 결정 |
| 추적성 관리 | RTM | 요구→설계→구현→테스트 전 과정 매핑으로 일관성 유지 |

### 4. 다. 요구사항 명세서(SRS) 6대 평가기준

| 구분 | 평가기준 | 설명 |
|------|---------|------|
| ① | 명확성 (Unambiguous) | 하나의 표현이 하나의 의미만 가져야 함 |
| ② | 완전성 (Complete) | 모든 기능·비기능 요건 누락 없이 기술 |
| ③ | 검증 가능성 (Verifiable) | 정량적 표현 필수 ("빠르게" → "1초 이내") |
| ④ | 일관성 (Consistent) | 요구사항 간 충돌·모순 없음 |
| ⑤ | 수정 가능성 (Modifiable) | 변경 시 영향도 파악 용이한 구조 |
| ⑥ | 추적 가능성 (Traceable) | 요구사항 출처와 구현 위치 추적 가능 |

- IEEE 830 기준 6대 평가기준 준수 여부가 SRS 품질의 최종 판단 기준이다.  "끝"

### 실무 제언

**RTM 기반 요구사항 변경 통제 — 범위(Scope) 방어**
- **챌린지**: 오픈 직전까지 요구사항이 바뀌는 대규모 프로젝트에서 무분별한 수용은 반드시 실패로 귀결된다
- **제언**: RTM과 Jira 등 형상관리 시스템을 연동하여 모든 변경 요구에 변경 영향도 분석(Impact Analysis) 보고서를 강제한다 — "기능 하나 추가 = 예산 5% 증액 + 오픈 2주 지연"의 수치적 근거로만 범위를 과학적으로 방어한다

**비기능 요구사항 QAS 정량화**
- **챌린지**: "안전하게", "빠르게" 수준의 추상적 비기능 명세는 감리 단계에서 주관적 논쟁만 야기한다
- **제언**: 모든 비기능 요건을 QAS(Quality Attribute Scenario) 템플릿 `[자극원]-[자극]-[대상]-[반응]` 형태로 정량화 — 기술사가 작성한 명세서는 논쟁의 여지 없이 시스템 품질의 표준이 되어야 한다
