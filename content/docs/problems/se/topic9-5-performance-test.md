## 해설

> **시스템이 얼마나 빠르고, 얼마나 오래 버티는가**  
> 성능 테스트는 부하 상황에서 시스템의 응답성, 처리량, 안정성을 정량적으로 측정한다.

"기능은 완벽한데 배포하자마자 서버가 죽었습니다"라는 문장은 소프트웨어 팀의 악몽입니다. 기능 테스트를 통과한 시스템도 실제 사용자 부하 아래서는 전혀 다르게 동작할 수 있습니다. 도로가 교통 법규를 다 지켜도 차가 너무 많으면 막히는 것처럼요. 성능 테스트는 시스템의 **속도(Speed)**, **확장성(Scalability)**, **안정성(Stability)**을 비즈니스 요구사항 기준으로 검증하는 비기능 테스트의 핵심입니다.

### 1. 출제 배경 및 의도

117회 기출에서 리틀의 법칙(Little's Law)을 성능 테스트와 연계하여 출제한 것은 매우 중요한 포인트입니다. 채점관은 단순히 부하 테스트, 스트레스 테스트의 정의를 나열하는 것이 아니라, **리틀의 법칙이라는 수학적 근거**로 성능 목표를 설정하고 분석할 수 있는 능력을 봅니다.

핵심 채점 포인트: 리틀의 법칙 공식 → 성능 테스트 종류 4가지 → 구성 요소(TPS, 응답시간, 처리량) → 도구(JMeter, Gatling)의 논리적 연결

### 2. 개념의 본질과 작동 메커니즘

#### 가. 리틀의 법칙(Little's Law)을 통한 성능 테스트 목적

**리틀의 법칙(Little's Law)**: 대기 이론(Queueing Theory)에서 유래한 공식으로, 안정적인 시스템에서 시스템 내 평균 요청 수(L), 처리 속도(λ), 평균 처리 시간(W) 사이에는 다음 관계가 성립합니다.

```text
L = λ × W

L (Little): 시스템 내에 머무는 평균 요청(사용자) 수
λ (Lambda): 단위 시간당 시스템으로 유입되는 요청 수 (처리율, Throughput)
W (Wait):   하나의 요청이 시스템 내에 머무는 평균 시간 (응답 시간, Response Time)
```

**실제 적용 예시**:
- 온라인 쇼핑몰의 목표: "동시 접속자 1,000명, 평균 응답 시간 2초 이내"
- L = 1,000명, W = 2초 라면, λ = L/W = 500 TPS(초당 처리 요청 수)
- 이 500 TPS를 시스템이 처리할 수 있는지 검증하는 것이 성능 테스트의 목적

리틀의 법칙은 **성능 목표의 수학적 근거**를 제공하며, 테스트 시나리오 설계와 병목 분석의 이론적 토대가 됩니다.

#### 나. 성능 테스트의 종류

```text
[성능 테스트 종류와 부하 패턴]

부하
 ↑
 │         ┌─────────────────┐
 │         │  스트레스 테스트  │  ← 한계점(Breaking Point) 탐색
 │         │                 │
 │  ┌──────┤  부하 테스트    │  ← 예상 최대 부하로 유지
 │  │      │                 │
 │──┤      ├─────────────────┤──────────────── 시간
 │  │                              ↑
 │  └──────────────────────────────┘
 │  (내구성/Soak 테스트: 정상 부하를 장시간 유지)
 │
 │     ↑↓↑↓ (스파이크 테스트: 급격한 부하 증감)
```

**① 부하 테스트(Load Test)**
- 목적: 예상되는 최대 사용자 부하 조건에서 시스템이 성능 목표를 충족하는지 검증
- 방법: 목표 TPS나 동시 사용자 수를 점진적으로 증가시키며 응답 시간, 오류율 측정
- 예: "명절 연휴에 최대 5,000명이 동시 접속할 것으로 예상 → 5,000명 부하를 30분간 유지"

**② 스트레스 테스트(Stress Test)**
- 목적: 시스템의 한계점(Breaking Point)과 최대 처리 용량을 파악. 한계 초과 시 우아한 성능 저하(Graceful Degradation) 여부 확인
- 방법: 부하를 계속 증가시켜 시스템이 실패하는 지점을 관찰
- 핵심: 시스템이 과부하 상황에서 완전히 죽는가, 아니면 일부 기능을 유지하면서 천천히 성능이 저하되는가

**③ 내구성 테스트(Endurance Test / Soak Test)**
- 목적: 장시간 운영 시 메모리 누수(Memory Leak), 자원 고갈 등 시간에 따른 성능 저하 감지
- 방법: 정상 부하를 12~48시간 이상 지속적으로 가함
- 발견 가능 결함: 메모리 누수, DB 커넥션 풀 소진, 로그 파일 무한 증가, GC(Garbage Collection) 압박

**④ 스파이크 테스트(Spike Test)**
- 목적: 급격한 부하 증가(스파이크)에 대한 시스템의 반응성과 회복성 검증
- 방법: 평상시 부하에서 순간적으로 수배의 부하를 가했다가 정상으로 복귀
- 예: 유명 유튜버가 특정 서비스를 소개하여 순간적으로 접속자가 폭증하는 상황

| 테스트 종류 | 목적 | 부하 패턴 | 시간 |
|-----------|------|---------|------|
| 부하 (Load) | 목표 부하에서 성능 목표 충족 여부 | 점진적 증가 후 유지 | 수십 분 ~ 수 시간 |
| 스트레스 (Stress) | 한계점 및 최대 처리 용량 파악 | 한계까지 계속 증가 | 수 시간 |
| 내구성 (Endurance/Soak) | 장시간 자원 누수, 메모리 누수 탐지 | 정상 부하 장기 유지 | 12~48시간 이상 |
| 스파이크 (Spike) | 급격한 부하 증감에 대한 회복성 | 급등 후 급강하 반복 | 수십 분 |

### 3. 핵심 키워드 (Must-Have)

| 키워드 | 설명 |
|--------|------|
| 리틀의 법칙 (Little's Law) | L = λ × W. 시스템 용량 설계와 성능 목표 설정의 수학적 근거 |
| TPS (Transactions Per Second) | 초당 처리 트랜잭션 수. 시스템 처리 능력의 핵심 지표 |
| 응답 시간 (Response Time) | 요청 후 응답 수신까지의 시간. 사용자 체감 성능 |
| 처리량 (Throughput) | 단위 시간당 처리한 작업량. 시스템의 생산성 지표 |
| 메모리 누수 (Memory Leak) | 내구성 테스트로 발견. 장시간 운영 시 메모리를 계속 점유 |
| 한계점 (Breaking Point) | 스트레스 테스트로 확인. 시스템이 실패하기 시작하는 부하 지점 |
| JMeter / Gatling | 대표적 성능 테스트 오픈소스 도구 |

### 4. 맥락과 비교

#### 성능 테스트 도구 비교: JMeter vs. Gatling

| 구분 | Apache JMeter | Gatling |
|------|-------------|--------|
| 언어 | Java | Scala |
| 스크립트 방식 | GUI 기반 (XML 설정) | 코드 기반 (Scala DSL) |
| 리포트 | 기본 리포트 (HTML 리포트 플러그인 필요) | 내장 HTML 리포트 (매우 시각적) |
| 러닝 커브 | 낮음 | 중간 (Scala 이해 필요) |
| 동시성 처리 | 스레드 기반 | 비동기 이벤트 기반 (Akka) → 더 효율적 |
| CI/CD 통합 | 용이 (Maven/Gradle 플러그인) | 용이 (Maven/Gradle 플러그인) |
| 주 사용 용도 | 전통적 웹 앱, 다양한 프로토콜 | 대규모 부하 시뮬레이션, 현대 웹 앱 |

#### 성능 테스트와 APM(Application Performance Monitoring)

성능 테스트는 배포 전의 사전 검증이고, APM은 운영 중의 실시간 모니터링입니다. 두 활동은 상호 보완적입니다.

- 성능 테스트 결과: "피크 타임에 응답 시간 95퍼센타일이 3.2초"
- APM 모니터링(DataDog, New Relic, Pinpoint): 실제 운영 환경에서 응답 시간, 오류율, DB 쿼리 지연 등 실시간 추적

### 5. 실무 제언

**성능 목표는 비즈니스와 합의해야 한다**

"응답 시간 1초 이내"라는 성능 목표는 어떻게 도출해야 할까요? 기술팀이 임의로 설정하면 비즈니스 요구와 괴리가 생깁니다. Jakob Nielsen의 웹 가용성 연구에 따르면 응답 시간 0.1초(즉각적), 1초(사용자가 흐름을 잃지 않는 한계), 10초(집중력 유지 한계)가 기준점입니다. 비즈니스 도메인(금융, 게임, 전자상거래)에 따라 허용 가능한 응답 시간이 다르므로, 성능 목표는 반드시 이해관계자와 합의하여 문서화해야 합니다.

**병목 지점 분석의 체계화**

성능 테스트에서 목표를 충족하지 못할 때 가장 중요한 것은 **병목 지점(Bottleneck)의 빠른 식별**입니다. 일반적인 병목 위치와 분석 방법:

- CPU 바운드: CPU 사용률 100% 근접 → 알고리즘 최적화 또는 스케일 아웃
- 메모리 바운드: GC 빈번 발생, 힙 메모리 포화 → 메모리 누수 수정 또는 JVM 튜닝
- DB 바운드: 슬로우 쿼리 다수 발생 → 인덱스 추가, 쿼리 최적화, 커넥션 풀 조정
- 네트워크 바운드: 대역폭 포화 → CDN 도입, 응답 페이로드 최소화
