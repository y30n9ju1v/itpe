## 해설

### 출제 배경 및 의도

117회 기출에서 리틀의 법칙(Little's Law)을 성능 테스트와 연계하여 출제한 것은 매우 중요한 포인트다. 기능 테스트를 통과한 시스템도 실제 사용자 부하 아래서는 전혀 다르게 동작할 수 있으며, 성능 문제는 운영 배포 후 발견되면 비즈니스 손실이 막대하다.

채점관은 리틀의 법칙 공식 → 성능 테스트 종류 4가지 → 구성 요소(TPS·응답시간·처리량) → 도구(JMeter, Gatling)의 논리적 연결을 평가한다. 수학적 근거인 리틀의 법칙으로 성능 목표를 설정하고, 병목 분석 전략까지 서술하면 고득점이 가능하다.

### 1. 성능 테스트 및 리틀의 법칙 개요

정 의  • 성능 테스트: 부하 상황에서 시스템의 속도(Speed)·확장성(Scalability)·
       안정성(Stability)을 비즈니스 요구사항 기준으로 검증하는 비기능 테스트
       - 리틀의 법칙: 안정적 시스템에서 L = λ × W 관계가 성립

```
[리틀의 법칙(Little's Law) - 대기 이론 기반]

  L = λ × W

  L (Little): 시스템 내에 머무는 평균 요청(사용자) 수
  λ (Lambda): 단위 시간당 유입되는 요청 수 (처리율, TPS)
  W (Wait):   하나의 요청이 시스템에 머무는 평균 시간 (응답 시간)

  [실제 적용 예시]
  온라인 쇼핑몰 목표: 동시 접속자 L=1,000명, 응답시간 W=2초
  → 필요 처리율: λ = L/W = 1,000/2 = 500 TPS
  → 시스템이 500 TPS를 처리할 수 있는지 성능 테스트로 검증
```

- 리틀의 법칙은 성능 목표의 수학적 근거를 제공하며, 테스트 시나리오 설계와 병목 분석의 이론적 토대다.

### 2. 성능 테스트 종류 및 구성 요소

1) 성능 테스트 4가지 종류

| 테스트 종류 | 목적 | 부하 패턴 | 소요 시간 |
|-----------|------|---------|---------|
| 부하 테스트(Load) | 예상 최대 부하에서 성능 목표 충족 여부 | 점진적 증가 후 유지 | 수십 분~수 시간 |
| 스트레스 테스트(Stress) | 한계점(Breaking Point)·최대 처리 용량 파악 | 한계까지 계속 증가 | 수 시간 |
| 내구성 테스트(Endurance/Soak) | 장시간 메모리 누수·자원 고갈 탐지 | 정상 부하 장기 유지 | 12~48시간 이상 |
| 스파이크 테스트(Spike) | 급격한 부하 증감에 대한 회복성 검증 | 급등 후 급강하 반복 | 수십 분 |

```
[성능 테스트 종류와 부하 패턴]

부하
 ↑
 │        ┌──────────────────┐
 │        │   스트레스 테스트  │ ← 한계점 탐색
 │  ┌─────┤   부하 테스트    │ ← 최대 부하 유지
 │──┤     └──────────────────┘────────────── 시간
 │  │         내구성 테스트: 정상 부하 장기 유지
 │
 │  ↑↓↑↓  스파이크 테스트: 급격한 부하 증감
```

- 스트레스 테스트는 시스템이 한계 초과 시 완전히 죽는지, 우아한 성능 저하(Graceful Degradation)를 하는지 확인하는 것이 핵심이다.

2) 성능 테스트 핵심 구성 요소

| 구성 요소 | 설명 | 중요성 |
|---------|------|------|
| TPS(Transactions Per Second) | 초당 처리 트랜잭션 수. 시스템 처리 능력의 핵심 지표 | 시스템 용량 설계 기준 |
| 응답 시간(Response Time) | 요청 후 응답 수신까지의 시간. 사용자 체감 성능 | SLA 기준 지표 |
| 처리량(Throughput) | 단위 시간당 처리한 작업량. 시스템 생산성 지표 | 확장성 평가 기준 |
| 한계점(Breaking Point) | 시스템이 실패하기 시작하는 부하 지점 | 스트레스 테스트 목표 |
| 메모리 누수(Memory Leak) | 장시간 메모리를 계속 점유하는 결함 | 내구성 테스트 발견 대상 |

- 성능 테스트는 95퍼센타일(P95) 응답 시간을 기준으로 SLA를 설정하는 것이 실무 표준이다.

### 3. 성능 테스트 도구 및 트렌드

1) JMeter vs. Gatling 비교

| 구분 | Apache JMeter | Gatling |
|------|-------------|--------|
| 언어 | Java | Scala / Java / Kotlin |
| 스크립트 방식 | GUI 기반 (XML 설정) | 코드 기반 (DSL) |
| 동시성 처리 | 스레드 기반 | 비동기 이벤트 기반 (자체 I/O 엔진, v3.7+에서 Akka 제거) |
| 리포트 | 플러그인 필요 | 내장 HTML 리포트 |
| 주 용도 | 전통적 웹 앱, 다양한 프로토콜 | 대규모 부하 시뮬레이션 |

- 성능 테스트는 배포 전 사전 검증이고, APM(DataDog·New Relic·Pinpoint)은 운영 중 실시간 모니터링으로 상호 보완적으로 활용해야 한다.  "끝"

### 실무 제언

**성능 목표는 비즈니스와 합의해야 한다**
- **챌린지**: 기술팀이 임의로 성능 목표를 설정하면 비즈니스 요구와 괴리가 생긴다. "응답 시간 1초 이내"가 모든 도메인에 동일하게 적용되지 않는다.
- **제언**: Jakob Nielsen의 연구(0.1초=즉각적, 1초=흐름 유지 한계, 10초=집중력 한계)와 도메인 특성을 기반으로, 이해관계자와 합의하여 SLA로 문서화해야 한다. 리틀의 법칙으로 TPS 목표를 수학적으로 도출하면 의사결정의 객관성이 높아진다.

**병목 지점 분석의 체계화**
- **챌린지**: 성능 목표 미달 시 병목 원인을 찾지 못하면 잘못된 최적화로 시간을 낭비한다.
- **제언**: CPU 바운드(알고리즘 최적화·스케일 아웃), 메모리 바운드(GC 튜닝·메모리 누수 수정), DB 바운드(인덱스·쿼리 최적화·커넥션 풀), 네트워크 바운드(CDN·페이로드 최소화)로 원인 유형을 분류하고, APM 도구로 병목 위치를 먼저 확정한 후 최적화를 수행하는 순서를 반드시 지켜야 한다.
