## 문제

A사는 공공 소프트웨어 개발 프로젝트 규모를 산정하기 위해 기능점수(Function Point)를 사용하고자 한다. 다음에 대하여 설명하시오. (단, "SW사업 대가산정 가이드 2018년 개정판" 기준)

- 가. 기능점수 방식의 정의 및 특징
- 나. 기능점수 산정방식의 종류 및 소프트웨어 개발비의 구성요소
- 다. 기능점수 방식에 의한 소프트웨어 개발비 산정 절차, 개발원가 보정의 필요성, 5가지 보정평가 계수

## 해설

> **코드 줄 수가 아니라 기능의 가치로 소프트웨어 크기를 측정**  
> 사용자 관점에서 소프트웨어의 기능적 규모를 정량화하는 국제 표준 측정 방법

"이 소프트웨어 개발에 얼마의 비용이 필요한가?"라는 질문에 답하려면 먼저 소프트웨어의 크기(규모)를 측정해야 합니다. 기능점수(Function Point, FP)는 개발 언어나 기술과 무관하게 "사용자에게 제공되는 기능의 수"를 기준으로 소프트웨어 규모를 측정하는 방법입니다. 국내 공공 SW 사업에서는 "SW사업 대가산정 가이드"에 따라 FP 방식이 사실상 표준으로 사용됩니다.

### 1. 출제 배경 및 의도

공공 SW 조달에서 계약 금액 산정의 법적 근거가 되는 방법이므로, 기술사 시험에서 매우 자주 출제됩니다. 채점관은 FP 5가지 요소(EI, EO, EQ, ILF, EIF)의 정확한 정의, UFP와 AFP 산정 공식, 그리고 보정 계수 적용의 필요성을 실무적 맥락에서 설명할 수 있는지를 평가합니다.

### 2. 개념의 본질과 작동 메커니즘

기능점수 방식의 핵심 철학은 **"기술 독립성(Technology Independence)"**입니다. C로 짜든 Java로 짜든, 온프레미스든 클라우드든, 동일한 기능을 제공한다면 동일한 기능점수를 부여합니다.

```text
[기능점수 산정 전체 흐름]

① 측정 범위 및 경계 설정
   → 어떤 기능들을 측정할 것인지 경계 정의

② 5가지 기능 요소 분류 및 복잡도 결정
   → 데이터 기능: ILF, EIF
   → 트랜잭션 기능: EI, EO, EQ

③ UFP (미조정 기능점수) 산정
   → 각 요소별 개수 × 복잡도 가중치 합산

④ VAF (기술복잡도 조정인수) 산정 [구버전 IFPUG]
   → 14개 일반 시스템 특성(GSC) 평가

⑤ AFP (조정된 기능점수) 산정
   → AFP = UFP × VAF (또는 국내 가이드 상 보정 계수 적용)

⑥ 개발비 산정
   → 개발비 = AFP × 기능점수당 단가(원/FP)
```

### 3. 핵심 키워드 (Must-Have)

#### 가. 기능점수 방식의 정의 및 특징

**정의**: 사용자가 요청하고 인식하는 기능적 요구사항의 양을 측정하여 소프트웨어의 규모를 기능점수(FP) 단위로 산출하는 방법 (IFPUG 제정)

**특징**:
- 사용자 관점(외부 관점) 측정 — 개발자의 내부 구현 방식 불문
- 언어·기술 독립적 — Java, Python, C# 동일 측정
- 요구사항 단계에서 측정 가능 — 설계/구현 전에도 산정 가능
- 국제 표준(ISO/IEC 20926 IFPUG) 기반

#### 나-1. 기능점수 5대 요소

**데이터 기능 (Data Function)**

| 요소 | 영문 | 정의 | 예시 |
|------|------|------|------|
| **ILF** | Internal Logical File | 시스템 내부에서 유지·관리되는 논리적 데이터 그룹 | 고객 마스터 테이블, 주문 테이블 |
| **EIF** | External Interface File | 참조만 하고 자신이 유지하지 않는 외부 데이터 | 타 시스템의 공통 코드 테이블 |

**트랜잭션 기능 (Transaction Function)**

| 요소 | 영문 | 정의 | 예시 |
|------|------|------|------|
| **EI** | External Input | 외부에서 데이터를 입력받아 내부 데이터를 변경 | 회원가입, 주문 등록, 데이터 수정 |
| **EO** | External Output | 내부 데이터를 처리·계산하여 외부로 출력 | 통계 보고서, 합계 조회, 계산 결과 화면 |
| **EQ** | External inQuiry | 단순 조회(계산 없이) 데이터를 외부로 출력 | 목록 조회, 상세 조회 화면 |

```text
[5대 기능 요소 구분 기준]

데이터가 외부에서 들어오는가? → YES → EI (External Input)
데이터가 외부로 나가는가?
  → 단순 조회인가? → YES → EQ (External inQuiry)
  → 계산/처리가 있는가? → YES → EO (External Output)
시스템 내부에 데이터를 유지하는가?
  → 자체 유지 → ILF (Internal Logical File)
  → 외부 참조 → EIF (External Interface File)
```

#### 나-2. UFP 산정: 복잡도별 가중치

각 기능 요소는 Low/Average/High 3단계 복잡도로 분류되며, 각각 다른 가중치를 가집니다.

| 기능 요소 | Low | Average | High |
|-----------|-----|---------|------|
| EI | 3 | 4 | 6 |
| EO | 4 | 5 | 7 |
| EQ | 3 | 4 | 6 |
| ILF | 7 | 10 | 15 |
| EIF | 5 | 7 | 10 |

```text
UFP = Σ(각 요소 개수 × 복잡도 가중치)
    = (EI 합계) + (EO 합계) + (EQ 합계) + (ILF 합계) + (EIF 합계)
```

#### 나-3. FP 산정 방식 종류

| 방식 | 특징 | 용도 |
|------|------|------|
| **IFPUG** | 원조 표준, 가장 정교 | 상세 분석·설계 이후 |
| **NESMA** | 간이 산정 지원 (표시/추정/상세) | 요구사항 초기 단계 |
| **COSMIC** | 소프트웨어 움직임(Movement) 기반 | 실시간·임베디드 SW |

#### 다. SW 개발비 산정 절차 및 보정 계수 (국내 가이드 기준)

**개발비 산정 절차**

```text
1단계: 기능점수 산정 (UFP → AFP)
   ↓
2단계: 개발원가 계산
   개발원가 = AFP × 기능점수당 단가(원/FP)
   ↓
3단계: 개발원가 보정
   보정된 개발원가 = 개발원가 × 보정계수(들의 곱)
   ↓
4단계: 직접경비 추가
   (소프트웨어 개발비 = 보정된 개발원가 + 직접경비)
```

**5가지 보정평가 계수 (개발원가 보정의 필요성)**

동일한 기능점수를 가진 프로젝트라도 환경에 따라 실제 비용은 크게 달라집니다. 이를 반영하기 위해 보정 계수를 적용합니다.

| 보정 계수 | 의미 | 보정 방향 |
|-----------|------|---------|
| **규모 보정** | 대규모 프로젝트는 규모의 경제로 단가 하락 | FP 많으면 단가 하락 |
| **언어 보정** | 개발 언어별 생산성 차이 반영 | C언어는 Java보다 단가 높음 |
| **연도 보정** | 물가 상승 반영 | 연도별 단가 테이블 적용 |
| **품질·특성 보정** | 보안, 성능 등 비기능 요구사항 복잡도 | 요구 수준 높을수록 비용 증가 |
| **애플리케이션 유형 보정** | 업무유형별(일반/통신/임베디드) 생산성 차이 | 임베디드가 일반 업무보다 단가 높음 |

#### SW 개발비의 구성요소

```text
SW 개발비 = 개발원가 + 직접경비 + 이윤

개발원가 = 인건비(노임단가 × 공수) + 경비
직접경비 = 여비, 교육훈련비, 장비 임차료 등
이윤 = 개발원가의 최대 25%
```

### 4. 맥락과 비교

#### FP 방식 vs LOC(Line of Code) 방식 비교

| 구분 | FP 방식 | LOC 방식 |
|------|---------|---------|
| **측정 기준** | 기능(사용자 관점) | 코드 줄 수(개발자 관점) |
| **언어 독립성** | 독립적 (Java든 C든 동일) | 언어 의존적 (Java 1줄 ≠ C 1줄) |
| **측정 시점** | 요구사항 단계부터 가능 | 구현 완료 후 측정 |
| **재사용 코드** | 기능 관점이므로 자연스럽게 처리 | 재사용 코드 계산 논란 |
| **단점** | 측정 기준 학습 필요, 시간 소요 | 프로그래밍 기술에 따라 편차 큼 |

### 5. 실무 제언

**기능점수 측정의 주관성 통제**

- **챌린지**: FP 측정은 "이 기능이 EI인가 EQ인가?", "복잡도가 Average인가 High인가?" 등의 판단에서 측정자 간 편차가 발생합니다. 같은 시스템을 두 사람이 측정하면 20~30% 차이가 나기도 합니다.
- **제언**: IFPUG 인증 측정사(CFPS: Certified Function Point Specialist) 자격을 보유한 전문가가 측정해야 하며, 발주자와 수급자 간 측정 결과 검증 프로세스(독립 검증)를 계약에 포함해야 합니다.

**기술사 시험 답안 작성 팁**

- EI, EO, EQ, ILF, EIF 다섯 가지 요소를 표 형식으로 정의하고, 각각의 예시(예: EI=주문등록, EQ=목록조회)를 들면 이해도를 어필할 수 있습니다.
- UFP 산정 공식과 복잡도 가중치 표를 포함하면 수치 근거가 있는 답안이 됩니다.
- "사용자 관점, 언어 독립적"이라는 FP의 핵심 특성을 LOC와 대비하여 설명하면 차별화됩니다.
