## 문제

A사는 공공 소프트웨어 개발 프로젝트 규모를 산정하기 위해 기능점수(Function Point)를 사용하고자 한다. 다음에 대하여 설명하시오. (단, "SW사업 대가산정 가이드 2018년 개정판" 기준)

- 다. 기능점수 방식에 의한 소프트웨어 개발비 산정 절차, 개발원가 보정의 필요성, 5가지 보정평가 계수

## 해설

### 출제 배경 및 의도

공공 SW 조달에서 계약 금액 산정의 법적 근거가 되는 방법이 기능점수(FP)이다. "SW사업 대가산정 가이드"(행정안전부)에 따라 FP 방식이 사실상 표준으로 사용되며, 기술사 시험에서도 매우 자주 출제된다. FP의 핵심 철학은 "기술 독립성(Technology Independence)"으로, 개발 언어나 플랫폼과 무관하게 사용자에게 제공되는 기능의 수로 SW 규모를 측정한다.

채점관은 FP 5대 요소(EI·EO·EQ·ILF·EIF)의 정확한 정의·복잡도 가중치, UFP→AFP 산정 공식, 5가지 보정 계수의 필요성과 내용을 실무 맥락에서 설명할 수 있는지 평가한다.

### 1. 기능점수(Function Point) 개요

정 의  • **기능점수(FP)**를 사용자가 요청하고 인식하는 기능적 요구사항의 양을 측정하여 SW 규모를 산출하는 사용자 관점·언어 독립적 측정 방법
       - 표준: ISO/IEC 20926 IFPUG(국제 기능점수 사용자 그룹)
       - 요구사항 단계부터 측정 가능 → 공공 SW 사업 대가 산정 표준 방법

```
[기능점수 산정 전체 흐름]

① 측정 범위·경계 설정
② 5대 기능 요소 분류 및 복잡도 결정
   → 데이터 기능: ILF, EIF
   → 트랜잭션 기능: EI, EO, EQ
③ UFP(미조정 기능점수) 산정
   → UFP = Σ(각 요소 개수 × 복잡도 가중치)
④ AFP(조정 기능점수) 산정
   → AFP = UFP × VAF(기술복잡도 조정인수) 또는 보정 계수 적용
⑤ 개발비 산정
   → 개발원가 = AFP × 기능점수당 단가(원/FP)

[5대 기능 요소 구분 기준]

외부 입력?  → YES → EI (External Input)
외부 출력?
  단순 조회?       → YES → EQ (External inQuiry)
  계산·처리 있음?  → YES → EO (External Output)
내부 데이터 유지?
  자체 유지 → ILF (Internal Logical File)
  외부 참조 → EIF (External Interface File)
```

- FP 방식은 사용자 관점·언어 독립·요구사항 단계 측정이라는 세 가지 특성으로 LOC 방식의 한계를 극복한다.

### 2. 핵심 내용 상세

1) 가. 기능점수 방식의 정의 및 특징

| 구분 | 항목 | 설명 |
|------|------|------|
| 정의 | 사용자 관점 기능 규모 측정 | IFPUG 제정, ISO/IEC 20926 표준 |
| 특징① | 기술 독립성 | Java든 C든 동일 기능이면 동일 FP |
| 특징② | 요구사항 단계 측정 가능 | 설계·구현 전 조기 산정 가능 |
| 특징③ | 외부(사용자) 관점 | 내부 구현 방식 불문 |
| 국내 적용 | SW사업 대가산정 가이드(행안부) | 공공 SW 계약 금액의 법적 근거 |

2) 나. 기능점수 5대 요소 및 복잡도 가중치

| 요소 | 영문 | 정의 | 예시 |
|------|------|------|------|
| EI | External Input | 외부 데이터 입력 → 내부 데이터 변경 | 회원가입·주문 등록·데이터 수정 |
| EO | External Output | 내부 데이터 처리·계산 → 외부 출력 | 통계 보고서·합계 계산 화면 |
| EQ | External inQuiry | 단순 조회(계산 없이) → 외부 출력 | 목록 조회·상세 조회 화면 |
| ILF | Internal Logical File | 시스템 내부 유지·관리 논리적 데이터 | 고객 마스터·주문 테이블 |
| EIF | External Interface File | 참조만 하고 유지하지 않는 외부 데이터 | 타 시스템 공통 코드 테이블 |

UFP 복잡도별 가중치:

| 기능 요소 | Low | Average | High |
|-----------|-----|---------|------|
| EI | 3 | 4 | 6 |
| EO | 4 | 5 | 7 |
| EQ | 3 | 4 | 6 |
| ILF | 7 | 10 | 15 |
| EIF | 5 | 7 | 10 |

- `UFP = Σ(각 요소 개수 × 복잡도 가중치)` 로 산정하며, ILF/EIF 가중치가 트랜잭션 기능보다 높음에 유의한다.

### 3. SW 개발비 산정 절차 및 보정 계수

1) SW 개발비 산정 절차

```
[SW 개발비 산정 절차]

1단계: FP 산정 (UFP → AFP)
       ↓
2단계: 개발원가 계산
       개발원가 = AFP × 기능점수당 단가(원/FP)
       ↓
3단계: 개발원가 보정 (5가지 보정 계수 적용)
       보정된 개발원가 = 개발원가 × 보정계수들의 곱
       ↓
4단계: SW 개발비 확정
       SW 개발비 = 보정된 개발원가 + 직접경비 + 이윤(최대 25%)
```

2) 5가지 보정 계수 및 개발원가 보정의 필요성

동일한 기능점수라도 개발 환경에 따라 실제 비용은 크게 달라진다. 이를 반영하기 위해 보정 계수를 적용한다.

| 보정 계수 | 의미 | 보정 방향 |
|-----------|------|---------|
| 규모 보정 | 대규모 프로젝트는 규모의 경제로 단가 하락 | FP 증가 → 단가 하락 |
| 언어 보정 | 개발 언어별 생산성 차이 반영 | C언어 > Java 단가 |
| 연도 보정 | 물가 상승 반영 | 연도별 단가 테이블 적용 |
| 품질·특성 보정 | 보안·성능 등 비기능 요구 복잡도 | 요구 수준 높을수록 비용 증가 |
| 애플리케이션 유형 보정 | 일반/통신/임베디드 생산성 차이 | 임베디드 > 일반 업무 단가 |

- FP 산정 방식 종류: IFPUG(정교, 상세 설계 이후)·NESMA(간이 산정, 초기 단계)·COSMIC(실시간·임베디드 SW)

| FP 방식 vs LOC 방식 | FP | LOC |
|---------------------|----|-----|
| 측정 기준 | 기능(사용자 관점) | 코드 줄 수(개발자 관점) |
| 언어 독립성 | 독립적 | 언어 의존적 |
| 측정 시점 | 요구사항 단계 가능 | 구현 완료 후 측정 |

- FP 방식은 LOC 대비 언어 독립성·조기 측정 가능성이 우수하여 공공 SW 대가 산정의 표준으로 자리잡고 있다.  "끝"

### 실무 제언

**측정자 간 편차 통제**
- **챌린지**: "이 기능이 EI인가 EQ인가?", "복잡도가 Average인가 High인가?" 등 판단에서 측정자 간 20~30% 편차가 발생하는 경우가 많다.
- **제언**: CFPS(Certified Function Point Specialist, IFPUG 인증 측정사) 자격 보유 전문가가 측정해야 하며, 발주자·수급자 간 독립 검증 프로세스를 계약에 포함하여 측정 결과의 신뢰성을 확보해야 한다.

**AI·클라우드 서비스의 FP 측정 한계**
- **챌린지**: AI API 호출·마이크로서비스 아키텍처·이벤트 기반 서비스는 전통적 EI/EO/EQ/ILF/EIF 분류 체계로 측정하기 어려운 기능 경계 문제가 발생한다.
- **제언**: COSMIC FP(ISO/IEC 19761)의 데이터 움직임(Entry·Exit·Read·Write) 기반 측정 방식을 마이크로서비스·이벤트 드리븐 아키텍처에 적용하고, AI 기능은 추론 요청(EQ 유사)·모델 학습(EI 유사)으로 매핑하는 방법론 가이드를 사전 확립해야 한다.

**FP 기반 대가 산정과 애자일의 충돌**
- **챌린지**: 애자일 개발에서는 요구사항이 스프린트마다 변경되어 초기 FP 산정값이 지속적으로 달라지므로, 계약 금액 고정 방식과 충돌이 발생한다.
- **제언**: 초기 FP 산정은 에픽(Epic) 수준으로 수행하고, 스프린트별 완성 FP 누적 방식의 변동형 계약(Time & Material 또는 FP 단가 × 실측 FP) 적용을 발주자와 협의하는 것이 현실적이다.
