## 해설

> **소프트웨어가 안전 기능을 제대로 수행하는지 보증하는 체계**  
> 기능 안전(Functional Safety) — E/E 시스템이 안전 기능을 올바르게 실행하여 위험을 수용 가능 수준으로 낮춘다

자동차 에어백, 원전 비상 냉각 시스템, 엘리베이터 비상 정지 — 이 모든 것은 특정 위험 상황에서 반드시 동작해야 하는 **안전 기능(Safety Function)**입니다. 기능 안전(Functional Safety)은 이러한 안전 기능을 담당하는 전기/전자(E/E) 시스템이 의도한 대로 동작하지 않을 때 발생하는 위험을 관리하는 공학 분야입니다. IEC 61508이 이 분야의 기본 국제 표준입니다.

### 1. 출제 배경 및 의도

이 문제는 세 가지를 묻고 있습니다.

- **가. 소프트웨어 안전 vs 보안 차이점**: 두 개념의 명확한 구분
- **나. IEC 61508의 안전 기능 요구사항 도출 과정**: 표준 기반 프로세스 이해
- **다. IEC 61508과 도메인별 파생 표준 비교**: 의료(IEC 62304), 항공(DO-178C), 자동차(ISO 26262) 비교

채점관은 기능 안전이 단순히 버그 없는 소프트웨어를 만드는 것이 아니라, **위험 기반 접근(Risk-Based Approach)**으로 안전성을 체계적으로 보증하는 엔지니어링 프로세스임을 이해하는지 확인합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 기능 안전의 핵심 개념 정의

**기능 안전(Functional Safety)**: "전기/전자/프로그래머블 전자(E/E/PE) 시스템의 위험한 오작동으로 인해 발생하는 위험을 수용 가능한 수준으로 낮추기 위한 안전 기능의 올바른 실행"

쉽게 말하면: "시스템이 오작동할 때도 사람이 다치지 않도록 하는 안전 기능이 제대로 동작함을 보증"

```text
[기능 안전의 위치]

전체 안전성
(Overall Safety)
    │
    ├── 기능 안전 (Functional Safety)
    │       └→ E/E/PE 시스템의 안전 기능이 올바로 수행되는가?
    │       └→ IEC 61508 적용 범위
    │
    ├── 기계적 안전
    │       └→ 구조적 강도, 압력 용기 등
    │
    └── 공정 안전
            └→ 위험 화학물질 취급 절차
```

#### 가. 소프트웨어 안전(Safety) vs 보안(Security)

| 구분 | Safety (안전) | Security (보안) |
|------|--------------|----------------|
| 위협 원천 | 우발적 오작동, 결함 | 의도적 공격자 |
| 보호 대상 | 사람, 환경, 재산 | 데이터, 시스템, 서비스 |
| 핵심 질문 | "시스템이 사람을 다치게 하는가?" | "공격자가 시스템을 침해할 수 있는가?" |
| 주요 위험 | 기능 고장, 소프트웨어 결함 | 해킹, 인증 우회, 데이터 탈취 |
| 표준 | IEC 61508, ISO 26262 | ISO 27001, OWASP(웹 보안 취약점 가이드) |
| 관계 | 안전하면 보안이 보장되지 않음 | 보안 침해가 안전 사고로 이어질 수 있음 |

**교차 영역의 중요성**: 자동차 분야에서 Safety와 Security는 점점 수렴합니다. 해커가 자율주행차의 제어 시스템에 접근하면 Security 침해가 Safety 사고로 직결됩니다. 이를 "Safety & Security Co-Engineering"이라 하며, ISO 21434(자동차 사이버보안)가 이 영역을 다룹니다.

#### 나. IEC 61508의 안전 기능 요구사항 도출 과정

IEC 61508은 위험을 기반으로 안전 요구사항을 도출하는 체계적 프로세스를 제시합니다.

```text
[IEC 61508 안전 기능 요구사항 도출 과정]

Step 1: 위험원(Hazard) 식별
  └→ 시스템이 유발할 수 있는 모든 위험 상황 열거
  └→ 기법: PHA, HAZOP, FMEA 활용

Step 2: 위험 이벤트(Hazardous Event) 정의
  └→ 위험원 + 인과 사건 조합
  └→ 예: "브레이크 고장 + 고속 주행 + 보행자 근접 = 충돌 사고"

Step 3: 위험 평가 (Risk Assessment)
  ┌─────────────────────────────────────┐
  │ 위험도 = 심각도(Severity) × 빈도(Frequency) × 노출도(Exposure) │
  └─────────────────────────────────────┘
  └→ 허용 가능한 위험(Tolerable Risk) 수준 결정

Step 4: SIL (Safety Integrity Level) 결정
  └→ 위험도를 허용 수준으로 낮추기 위해 요구되는 안전 기능의 수준
  └→ SIL 1~4 중 해당 등급 결정

Step 5: 안전 기능 요구사항 도출
  └→ 각 SIL에서 요구되는 안전 기능 명세
  └→ 예: SIL 2 → "브레이크 고장 감지 후 2초 내 경고, 자동 감속"

Step 6: 안전 요구사항 할당
  └→ 하드웨어 / 소프트웨어 / 운영 절차 중 어느 레이어에 구현할지 결정
```

**SIL 결정 매트릭스 예시**

```text
              심각도(Severity)
              경미  중간  심각  치명
빈도   높음 │SIL1 SIL2 SIL3 SIL4│
(Freq)중간  │ -   SIL1 SIL2 SIL3│
       낮음 │ -    -   SIL1 SIL2│
       매우 │ -    -    -   SIL1│
       낮음 └─────────────────────┘
```

### 3. 핵심 키워드 (Must-Have)

#### IEC 61508 안전 생명주기 (Safety Lifecycle)

IEC 61508은 안전 시스템의 개념 설계부터 폐기까지 전 단계를 커버하는 안전 생명주기를 정의합니다.

```text
[IEC 61508 안전 생명주기 (개요)]

전체 안전 생명주기 (E/E/PE Safety Related System):
  ① 범위 정의
  ② 위험 및 위험원 분석
  ③ 전체 안전 요구사항 명세
  ④ 안전 요구사항 할당 (H/W, S/W, 운용 절차)
        │
        ▼
소프트웨어 안전 생명주기:
  ⑤ S/W 안전 요구사항 명세
  ⑥ S/W 설계 및 개발
  ⑦ S/W 통합 테스트
  ⑧ S/W 안전 검증
        │
        ▼
  ⑨ 전체 시스템 통합
  ⑩ 안전 검증 (Safety Validation)
  ⑪ 운영 및 유지보수
  ⑫ 폐기
```

#### 안전 케이스 (Safety Case)

**안전 케이스**는 "이 시스템은 충분히 안전하다"는 주장을 근거(Evidence)를 통해 체계적으로 논증하는 문서 패키지입니다. GSN(Goal Structuring Notation)으로 표현하는 것이 일반적입니다.

```text
[안전 케이스 구조 — GSN]

주장(Claim/Goal): 브레이크 시스템은 SIL 2 요구사항을 만족한다
         │
    [전략(Strategy)]
         │
    ├── 근거 1: FMEA 분석 결과 — RPN 100 이하
    ├── 근거 2: FTA 분석 — 고장 확률 10⁻³ 이하 확인
    ├── 근거 3: 소프트웨어 테스트 커버리지 100% (MC/DC)
    └── 근거 4: 독립적 안전 검증 리뷰 완료
```

#### 다. IEC 61508과 도메인별 파생 표준 비교

IEC 61508은 산업 전반에 적용되는 기본 표준이며, 각 도메인은 이를 기반으로 도메인 특화 표준을 만들었습니다.

| 항목 | IEC 61508 | ISO 26262 (자동차) | IEC 62304 (의료기기) | DO-178C (항공) |
|------|---------|-----------------|-----------------|--------------|
| 적용 도메인 | E/E/PE 시스템 전반 | 자동차 전기/전자 | 의료기기 소프트웨어 | 항공기 기내 소프트웨어 |
| 위험 분류 | SIL 1~4 | ASIL A~D | 클래스 A/B/C | DAL A~E |
| 최고 위험 등급 | SIL 4 | ASIL D | 클래스 C | DAL A |
| 제정 기관 | IEC | ISO | IEC | RTCA/EUROCAE |

### 4. 맥락과 비교

**SIL과 도메인별 위험 등급 매핑**

```text
위험 수준   IEC 61508   ISO 26262   IEC 62304   DO-178C
가장 높음 │  SIL 4   │  ASIL D  │  Class C  │  DAL A  │
          │  SIL 3   │  ASIL C  │  Class C  │  DAL B  │
          │  SIL 2   │  ASIL B  │  Class B  │  DAL C  │
          │  SIL 1   │  ASIL A  │  Class A  │  DAL D  │
가장 낮음 │    -     │  QM(품질)│  Class A  │  DAL E  │
```

**각 등급별 요구되는 소프트웨어 개발 엄밀도 비교**

SIL/ASIL 등급이 높아질수록:
- 더 엄격한 코딩 표준 (MISRA-C 등)
- 더 높은 테스트 커버리지 (SIL 4/ASIL D: MC/DC 100% 필수)
- 더 많은 독립 검증 활동 (독립적 V&V 팀)
- 더 엄격한 형상 관리

### 5. 실무 제언

**기능 안전 개발의 실무적 난제**

기능 안전 표준을 준수하는 개발은 일반 소프트웨어 개발에 비해 2~5배의 비용이 듭니다. 따라서 다음 원칙이 중요합니다.

1. **위험 기반 우선순위 (Risk-Based Prioritization)**: 모든 소프트웨어 컴포넌트에 최고 SIL을 적용하는 것이 아니라, 실제 위험도에 따라 SIL을 차등 적용합니다. 안전 기능 외의 컴포넌트는 일반 품질 관리(QM) 수준으로 개발합니다.

2. **혼합 SIL (Mixed SIL) 아키텍처**: 고 SIL 컴포넌트와 저 SIL 컴포넌트가 동일 시스템에 공존할 경우, 저 SIL 컴포넌트가 고 SIL 컴포넌트에 영향을 미치지 않도록 명확한 격리(Partitioning)가 필요합니다.

3. **도구 적격성 확인(Tool Qualification)**: SIL 2 이상에서는 개발에 사용한 도구(컴파일러, 테스트 도구)도 적격성 확인이 필요합니다. 도구 버그가 안전 기능 결함을 유발할 수 있기 때문입니다.

**자율주행 시대의 기능 안전 진화**

ISO 26262:2018 2판에서는 소프트웨어 업데이트(SOTIF), AI/ML 컴포넌트, 사이버보안(ISO 21434와 연계)에 대한 내용이 강화되었습니다. 특히 머신러닝 기반 ADAS(첨단 운전자 지원 시스템)는 결정론적이지 않아 전통적인 SIL 개념 적용에 한계가 있으며, 이를 다루는 ISO 21448(SOTIF)이 기능 안전의 새 축으로 자리잡고 있습니다.
