## 해설

### 출제 배경 및 의도

자동차 에어백·원전 비상 냉각·엘리베이터 비상 정지 등 E/E 시스템의 안전 기능이 오작동할 경우 인명 피해로 직결된다. IEC 61508은 이러한 위험을 수용 가능 수준으로 낮추기 위한 범용 기능안전 표준으로, 자동차(ISO 26262)·의료(IEC 62304)·항공(DO-178C) 등 도메인별 파생 표준의 모체가 된다.

관리 유형 출제로, 채점관은 Safety와 Security 개념 구분, SIL 기반 안전 요구사항 도출 프로세스, 그리고 도메인별 표준 비교를 체계적으로 설명할 수 있는지 평가한다. "버그 없는 SW 제작"이 아닌 "위험 기반 접근(Risk-Based Approach)으로 안전성을 체계 보증하는 엔지니어링 프로세스"라는 본질을 답안 서두에 제시하면 고득점 포인트가 된다.

### 1. 소프트웨어 기능안전 개요

정 의  • **소프트웨어 기능안전**을 E/E/PE 시스템의 위험한 오작동으로 발생하는 위험을 수용 가능 수준으로 낮추기 위해 안전 기능의 올바른 실행을 보증하는 엔지니어링 체계
       - IEC 61508: 산업 전반 E/E/PE 안전 관련 시스템의 기본 국제 표준

```
[기능 안전의 위치]

전체 안전성 (Overall Safety)
    │
    ├── 기능 안전 (Functional Safety) ← IEC 61508 적용 범위
    │       └── E/E/PE 시스템 안전 기능이 올바로 수행되는가?
    │
    ├── 기계적 안전
    │       └── 구조적 강도, 압력 용기 등
    │
    └── 공정 안전
            └── 위험 화학물질 취급 절차

[IEC 61508 기반 도메인별 파생 표준]

IEC 61508 (범용 E/E/PE)
    ├── ISO 26262 (자동차) → ASIL A~D
    ├── IEC 62304 (의료기기) → 클래스 A/B/C
    └── DO-178C (항공) → DAL A~E
```

- 기능 안전은 위험 기반 접근(Risk-Based Approach)으로 안전성을 체계 보증하는 엔지니어링 프로세스이다.

### 2. 핵심 내용 상세

1) 가. SW 안전(Safety) vs SW 보안(Security)

| 구분 | Safety (안전) | Security (보안) |
|------|--------------|----------------|
| 위협 원천 | 우발적 오작동·결함 | 의도적 공격자 |
| 보호 대상 | 사람·환경·재산 | 데이터·시스템·서비스 |
| 핵심 질문 | 시스템이 사람을 다치게 하는가? | 공격자가 시스템을 침해할 수 있는가? |
| 주요 위험 | 기능 고장, SW 결함 | 해킹·인증 우회·데이터 탈취 |
| 표준 | IEC 61508, ISO 26262 | ISO 27001, OWASP |
| 수렴 영역 | Safety & Security Co-Engineering | ISO 21434 자동차 사이버보안 |

- 자율주행차에서 Security 침해(해킹)가 Safety 사고(충돌)로 직결되어 두 영역의 통합 설계가 필수화되고 있다.

2) 나. IEC 61508 안전 기능 요구사항 도출 과정

| 구분 | 단계 | 설명 |
|------|------|------|
| ① | 위험원(Hazard) 식별 | PHA·HAZOP·FMEA 활용, 모든 위험 상황 열거 |
| ② | 위험 이벤트 정의 | 위험원 + 인과 사건 조합 → 예: 브레이크 고장 + 고속 주행 = 충돌 |
| ③ | 위험 평가 | 위험도 = 심각도 × 빈도 × 노출도 → 허용 가능 위험 수준 결정 |
| ④ | SIL 결정 | 허용 위험 대비 실제 위험 감소량으로 SIL 1~4 결정 |
| ⑤ | 안전 기능 요구사항 도출 | SIL별 안전 기능 명세 → 예: SIL 2 → 고장 감지 후 2초 내 경고·자동 감속 |
| ⑥ | 안전 요구사항 할당 | H/W·S/W·운영 절차 중 구현 레이어 결정 |

```
[SIL 결정 매트릭스]

              심각도(Severity)
              경미   중간   심각   치명
빈도   높음 │ SIL1  SIL2  SIL3  SIL4 │
(Freq)중간  │  -    SIL1  SIL2  SIL3 │
       낮음 │  -     -    SIL1  SIL2 │
    매우낮음 │  -     -     -    SIL1 │
             └─────────────────────────┘

SIL별 위험 감소량:
  SIL 1:     10 ~    100배 감소
  SIL 2:    100 ~  1,000배 감소
  SIL 3:  1,000 ~ 10,000배 감소
  SIL 4: 10,000 ~100,000배 감소
```

- SIL이 높아질수록 MC/DC 100% 테스트 커버리지·MISRA-C 코딩 표준·독립 V&V 팀 등 개발 엄밀도가 기하급수적으로 증가한다.

### 3. IEC 61508과 도메인별 파생 표준 비교

1) 표준 비교

| 항목 | IEC 61508 | ISO 26262 (자동차) | IEC 62304 (의료기기) | DO-178C (항공) |
|------|-----------|------------------|---------------------|---------------|
| 적용 도메인 | E/E/PE 전반 | 자동차 전기/전자 | 의료기기 SW | 항공기 기내 SW |
| 위험 등급 | SIL 1~4 | ASIL A~D + QM | 클래스 A/B/C | DAL A~E |
| 최고 등급 | SIL 4 | ASIL D | 클래스 C | DAL A |
| 제정 기관 | IEC | ISO | IEC | RTCA/EUROCAE |
| 특화 내용 | 범용 기준 | ASIL 분해·SEooC·V-모델 | 유지보수 이력 관리 | 구조적 커버리지 DO-330 |

2) SIL과 도메인별 위험 등급 매핑

```
위험 수준    IEC 61508   ISO 26262   IEC 62304   DO-178C
가장 높음  │  SIL 4   │  ASIL D  │  Class C  │  DAL A │
          │  SIL 3   │  ASIL C  │  Class C  │  DAL B │
          │  SIL 2   │  ASIL B  │  Class B  │  DAL C │
          │  SIL 1   │  ASIL A  │  Class A  │  DAL D │
가장 낮음  │    -     │  QM(품질)│  Class A  │  DAL E │
```

- 각 도메인 표준은 IEC 61508의 원칙을 계승하되 도메인 특수성(자동차 V-모델, 의료기기 유지보수, 항공 구조적 커버리지)을 반영하여 특화되었다.  "끝"

### 실무 제언

**위험 기반 SIL 차등 적용**
- **챌린지**: 기능안전 표준 준수 개발은 일반 개발 대비 2~5배 비용이 소요되어, 전 컴포넌트에 최고 SIL을 적용하면 현실적으로 불가능하다.
- **제언**: 안전 기능 컴포넌트만 SIL을 차등 할당하고, 비안전 컴포넌트는 QM 수준으로 개발하는 위험 기반 우선순위 전략이 핵심이다. 혼합 SIL 아키텍처에서는 저 SIL 컴포넌트가 고 SIL 컴포넌트에 영향을 미치지 않도록 명확한 격리(Partitioning)가 필수이다.

**도구 적격성 확인(Tool Qualification)**
- **챌린지**: SIL 2 이상에서는 개발에 사용한 컴파일러·테스트 도구도 적격성 확인이 필요하나, 상용 도구에 대한 적격성 확인은 시간과 비용이 매우 크다.
- **제언**: TÜV·BSI 등 인증 기관이 적격성 확인을 완료한 도구(예: LDRA, VectorCAST)를 사전 선정하거나, 도구 제조사의 TQL(Tool Qualification Level) 문서를 활용하여 중복 검증 비용을 절감해야 한다.

**자율주행 AI 안전성 새 패러다임**
- **챌린지**: ISO 26262 2판(2018)에서도 ML 기반 ADAS의 비결정론적 동작에 대한 SIL 증명 방법이 완전히 정립되지 않았다.
- **제언**: ISO 21448(SOTIF)과 UL 4600(자율주행 안전성 표준)을 병행 적용하고, AI 안전성을 위한 ISO/IEC 42001 동향을 지속 모니터링하며 조직의 기능안전 프로세스를 지속 갱신해야 한다.
