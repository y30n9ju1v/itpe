## 해설

> **소프트웨어 설계의 영원한 북극성**  
> 추상화, 정보은닉, 단계적 분해, 모듈화

소프트웨어 공학의 역사는 '복잡성(Complexity)과의 사투'라고 해도 과언이 아닙니다. 기술의 트렌드는 매년 화려하게 바뀌지만, 그 화려한 아키텍처들을 지탱하는 지하실에는 수십 년 전 정립된 4대 설계 원리가 단단한 주춧돌로 자리 잡고 있습니다. 마치 건물의 외관은 시대에 따라 달라져도 기초 공사의 원리는 변하지 않는 것처럼요.

### 1. 출제 배경 및 의도

현대 소프트웨어 패러다임은 클라우드 네이티브(Cloud Native), 마이크로서비스 아키텍처(MSA), 그리고 AI 기반 대규모 가속 개발 시대로 빠르게 전환되고 있습니다. 대형 시스템이 고도화될수록 현업 아키텍트들이 마주하는 가장 큰 적은 거대해진 시스템의 복잡성과 이로 인한 기술 부채(Technical Debt)입니다.

채점관이 수십 년 전 정립된 고전적인 '소프트웨어 설계 원리'를 지금 이 시점에 다시 묻는 이유는 명확합니다. 최신 아키텍처 스타일일수록 본질적인 설계 원리가 흔들리면 시스템 전체가 모래성처럼 무너지기 때문입니다. 예컨대, MSA(마이크로서비스 아키텍처)의 서비스 경계 획정(Bounded Context)이나 데이터 격리(Database-per-Service) 기술은 결국 '모듈화'와 '정보은닉'의 현대적 재해석에 불과합니다.

### 2. 개념의 본질과 작동 메커니즘

소프트웨어 설계 원리란 "시스템 개발 조건이나 운용될 환경 조건의 제약 안에서 효율성과 단순성의 특성을 고려한 최적의 설계안을 발견하는 모든 작업"으로 정의할 수 있습니다.

4대 원리는 다음과 같은 유기적 인과관계를 가지고 작동합니다.

```text
[현실의 복잡한 문제]
       │
       ▼
 1. 추 상 화 (Abstraction)          ──▶ 복잡한 현실에서 핵심 본질만 추출
       │
       ▼
 2. 정 보 은 닉 (Information Hiding) ──▶ 추출된 본질 외 세부 구현은 외부 차단
       │
       ▼
 3. 단계적 분해 (Stepwise Refinement)──▶ 상위 추상 수준에서 하위 수준으로 점진적 분할
       │
       ▼
 4. 모 듈 화 (Modularization)       ──▶ 최종적으로 독립된 소프트웨어 단위로 형상화
```

- **추상화**: 현실 세계의 복잡한 개체들을 프로그램으로 변환하기 위해 불필요한 특성은 제거하고 공통의 본질만 파악하여 개념을 정립하는 작업
- **정보은닉**: 모듈의 자세한 내부 처리 내용이나 전역 데이터 등을 시스템의 다른 부분으로부터 감추고, 오직 인터페이스만으로 소통하게 하여 독립성을 보장
- **단계적 분해**: 하향식(Top-Down) 설계 전략에 따라 문제를 상위 수준의 추상적 개념에서 출발하여 점진적으로 구체적인 하위 수준의 기능과 제어로 분할해 나가는 기법
- **모듈화**: 앞선 추상화와 분해 과정을 거쳐 소프트웨어를 나누어 놓은 최적의 프로그램 단위(Module)를 구성하는 일련의 행위

### 3. 핵심 키워드 (Must-Have)

#### 추상화 (Abstraction)의 3대 유형

- **기능 추상화 (Functional)**: 입력 자료를 출력 자료로 변환하는 과정 자체를 추상화하여 부프로그램의 목적과 기능에만 집중 (예: `obj.getName()`)
- **자료 추상화 (Data)**: 데이터 객체를 구성하는 세부 속성 대신 객체 클래스 자체를 데이터 타입으로 정의 (예: `String`, `User` 타입)
- **제어 추상화 (Control)**: 실행 제어 흐름의 세부 명세를 하위 수준으로 미루고 명령이나 행위를 개념화 (예: `if-then-else`, `loop` 구조)

#### 정보은닉 (Information Hiding)의 핵심 기법

- **캡슐화 (Encapsulation)**: 모듈의 데이터(속성)와 기능(메서드)을 하나로 묶어 외부 접근을 제한
- **인터페이스 정의 (Interface)**: 모듈 내부가 수정되더라도 다른 모듈에 구애받지 않도록 외부 노출용 명세 최소화
- **전역변수 지양 (Anti-Global Variable)**: 사이드 이펙트(Side Effect)를 방지하기 위해 데이터 오염 경로 차단

#### 단계적 분해 (Stepwise Refinement)의 통제 방식

- **하향식 접근 (Top-Down)**: 닉클라우스 비르트(Niklaus Wirth)가 제안한 패러다임으로 거시 구조에서 미시 구조로 진입
- **상세 내역 지연 (Deferred Details)**: 세부 알고리즘이나 구체적 자료구조 결정은 가능한 한 구현 직전 단계 뒤로 미룸

#### 모듈화 (Modularization)의 양대 측정 척도

**응집도 (Cohesion)**: 모듈 내부의 요소들이 단일 목적을 위해 얼마나 집중되어 있는가의 척도 (강할수록 우수)

> 암기 키워드 (강함→약함): **기순통절시논우** — 기능적 ◀ 순차적 ◀ 통신적 ◀ 절차적 ◀ 시간적 ◀ 논리적 ◀ 우연적

**결합도 (Coupling)**: 모듈과 모듈 사이의 상호 연관도 및 의존성을 측정하는 척도 (약할수록 우수)

> 암기 키워드 (약함→강함): **자스제외공내** — 자료 ◀ 스탬프 ◀ 제어 ◀ 외부 ◀ 공통 ◀ 내용

**Fan-In / Fan-Out**: 상위 제어 모듈(Fan-In)은 극대화하고 하위 제어 모듈(Fan-Out)은 적정 수준으로 관리하여 시스템 복잡도 최적화

### 4. 맥락과 비교

본 설계 원리들은 고립된 구세대 이론이 아닙니다. 이후 등장한 객체지향(OO), 컴포넌트(CBD), 마이크로서비스(MSA)의 거대한 강줄기를 형성한 원천입니다.

| 구분 | 고전적 설계 원리 | 객체지향 SOLID | CBD | MSA |
|------|----------------|----------------|-----|-----|
| 중심 대상 | 프로시저, 서브루틴 및 모듈 | 클래스(Class)와 객체 | 배포·재사용 단위 컴포넌트 | 독립 운영·배포 가능한 서비스 단위 |
| 추상화 방식 | 기능/자료/제어 추상화 | 인터페이스와 추상 클래스 | 인터페이스 명세서 (IDL 등) | Open API 및 스키마 명세 (REST/gRPC) |
| 은닉 및 격리 | 내정 변수 은닉 및 입출력 격리 | 캡슐화 및 접근제어자 (`private`) | 바이너리 형태 내부 로직 은닉 | 서비스별 별도 Database를 통한 완벽 격리 |
| 품질 평가지표 | 높은 응집도, 낮은 결합도 | 단일책임(SRP), 의존성역전(DIP) 등 | 컴포넌트 표준 인터페이스 적합성 | 느슨한 결합, 장애 전파 차단(서킷브레이커) |

### 5. 실무 제언

**과도한 추상화의 늪 (Over-Engineering)**

- **챌린지**: 미래의 확장성만을 고려해 수많은 인터페이스와 추상 계층을 겹겹이 쌓아두면 코드의 가독성이 급격히 저하됩니다. 추상화 레이어가 지나치게 깊어질 경우 호출 스택 증가로 인해 런타임 오버헤드가 발생해 시스템 성능이 저하됩니다.
- **제언**: 실무에서는 'YAGNI(You Aren't Gonna Need It)' 원칙을 병행해야 합니다. 비즈니스의 변경 주기가 빠른 영역과 고정적인 영역을 구분하고, 변화가 확실시되는 임계 영역에만 선택적으로 고차원 추상화를 도입해야 합니다.

**모듈화 세분도와 비용의 트레이드오프 (Trade-off)**

- **챌린지**: 소프트웨어를 지나치게 잘게 쪼개면 개별 모듈의 개발 복잡도는 낮아지지만, 모듈 통합 비용(Integration Cost)과 시스템 복잡도는 기하급수적으로 증가합니다.
- **제언**: 아키텍트는 프로젝트의 예산, 인력의 숙련도, 인프라 비용을 복합적으로 고려하여 '전체 소프트웨어 생명주기 비용'이 최저점이 되는 최적의 모듈 개수를 산정하는 모듈 거버넌스를 확립해야 합니다.

**AI 프롬프트 엔지니어링 시대의 설계 원리**

- **챌린지**: LLM 기반 가속 개발 도구(GitHub Copilot 등)가 확산되고 있습니다. 그러나 AI에게 요구사항을 전달할 때 시스템 구조와 상호작용이 명확히 정제되어 있지 않으면, AI 역시 유지보수가 불가능한 스파게티 코드를 대량 양산해 냅니다.
- **제언**: 코드를 구현하는 하위 태스크는 AI가 맡더라도, 요구사항을 해체하는 '단계적 분해'와 '모듈 경계 정의'는 오직 인간 아키텍트의 고유 역량입니다. AI 시대일수록 본질적 원리를 깊이 이해하는 아키텍트가 AI를 제어하여 진정한 고품질 소프트웨어를 만들어낼 수 있습니다.
