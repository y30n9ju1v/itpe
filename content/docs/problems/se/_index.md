# 1부: 소프트웨어 설계와 개발 방법론

## 1장: SW 설계 원칙과 개발 방법론

### 소프트웨어 설계, 왜 원칙이 필요한가?

집을 지을 때 건축 원칙 없이 그냥 벽돌을 쌓으면 무너집니다. 소프트웨어도 마찬가지입니다. 처음에는 빠르게 만들었어도, 나중에 요구사항이 바뀌거나 기능을 추가할 때 무너지지 않으려면 **설계 원칙**이 필요합니다.

#### 좋은 소프트웨어 설계의 핵심 개념

**결합도(Coupling)와 응집도(Cohesion)**는 소프트웨어 설계의 양대 축입니다.

- **결합도(Coupling)**: 모듈 간의 의존성 정도. **낮을수록 좋습니다.** A를 고쳤을 때 B가 영향을 받지 않아야 이상적입니다.
- **응집도(Cohesion)**: 모듈 내부 요소들이 얼마나 관련된 일을 하는지. **높을수록 좋습니다.** 하나의 모듈이 하나의 잘 정의된 역할만 해야 합니다.

**비유**: 좋은 팀은 팀원 간 불필요한 간섭(낮은 결합도)이 없고, 팀 내에서는 모두 같은 목표(높은 응집도)를 위해 일합니다.

#### 소프트웨어 개발 방법론의 3대 계보

| 방법론 | 핵심 철학 | 적합한 상황 |
|--------|-----------|-------------|
| **구조적 방법론** | 기능 중심 분해, 절차적 사고 | 단순한 배치성 업무 |
| **정보공학 방법론** | 데이터 중심 설계, 전사 통합 | 대형 ERP, 금융 시스템 |
| **객체지향 방법론** | 현실 세계를 객체로 모델링 | 대부분의 현대 시스템 |

현대 소프트웨어는 **객체지향 + 컴포넌트 기반**으로 수렴하는 추세입니다.

#### 테일러링(Tailoring): 방법론을 현장에 맞게 조정하기

어떤 방법론도 모든 프로젝트에 100% 적합하지는 않습니다. **테일러링**은 표준 방법론을 특정 프로젝트의 특성(규모, 복잡도, 위험도)에 맞게 커스터마이징하는 활동입니다.

테일러링 시 고려 사항:
- 프로젝트 규모 — 소규모 프로젝트에 무거운 CMMI Level 5 프로세스는 과잉
- 도메인 특성 — 금융/의료는 엄격한 문서화, 스타트업 MVP는 경량 애자일
- 팀 역량 — 팀이 경험 없는 방법론 강요는 역효과

## 2장: 정보시스템 마스터플랜 (ISMP/ISP)

### "IT 전략 없이 시스템 만들지 마라"

아파트 단지를 지을 때 각 동이 제각각 건축되면 상수도·전기·도로가 연결되지 않습니다. IT 시스템도 마찬가지로, **전략적 계획(ISP/ISMP) 없이 각 부서가 독자적으로 시스템을 만들면** 데이터가 불일치하고 통합이 불가능해집니다.

#### ISP vs ISMP: 무엇이 다른가?

| 구분 | ISP (Information Strategy Planning) | ISMP (Information System Master Plan) |
|------|--------------------------------------|----------------------------------------|
| 의미 | 정보화 전략 계획 | 정보시스템 마스터플랜 (법적 용어) |
| 법적 근거 | 없음 (일반 용어) | 전자정부법 (공공기관 의무) |
| 주요 산출물 | IT 전략, AS-IS/TO-BE 분석 | 구축 우선순위, 예산 계획, 이행 로드맵 |
| 적용 | 민간기업 | 주로 공공기관 |

**핵심**: ISMP는 ISP를 포함하는 더 상위 개념이며, 공공 정보화 사업에서 **사업 발주 전에 반드시 수행**해야 하는 선행 절차입니다.

#### ISMP 방법론: EA(Enterprise Architecture)와의 연계

EA는 조직 전체의 IT 구조를 **비즈니스·데이터·애플리케이션·기술** 네 계층으로 체계화하는 틀입니다. ISMP는 EA 관점에서 현재 상태(AS-IS)를 진단하고 미래 목표 상태(TO-BE)를 설계한 뒤, 그 간극을 메우는 이행 계획을 수립합니다.

## 3장: 객체지향과 SOLID 원칙

### 객체지향이란? 현실을 코드로 옮기는 방법

"**현실 세계의 사물을 객체(Object)로 표현하고, 그 객체들이 서로 메시지를 주고받으며 프로그램이 동작하는 패러다임**"입니다.

사람(Person)이라는 객체는 이름·나이라는 **속성(Attribute)**을 가지고, 말하기·걷기라는 **행동(Method)**을 수행합니다.

#### 객체지향의 4대 특성

| 특성 | 설명 | 비유 |
|------|------|------|
| **캡슐화(Encapsulation)** | 데이터와 메서드를 묶고, 내부 구현을 숨김 | 자판기: 내부 동작 몰라도 버튼만 누르면 됨 |
| **상속(Inheritance)** | 부모 클래스의 특성을 자식이 물려받음 | 동물→강아지: 공통 특성 재사용 |
| **다형성(Polymorphism)** | 같은 메시지에 각 객체가 다르게 반응 | 그리기() → 원은 원을, 사각형은 사각형을 그림 |
| **추상화(Abstraction)** | 복잡한 현실을 핵심 속성만 추려 모델링 | 자동차의 "운전" = 가속·조향만 알면 됨 |

#### SOLID: 좋은 객체지향 설계의 5가지 원칙

SOLID는 로버트 마틴(Uncle Bob)이 정리한 객체지향 설계의 다섯 가지 황금 원칙입니다.

**S — Single Responsibility Principle (단일 책임 원칙)**
> "클래스는 단 하나의 변경 이유만 가져야 한다."

나쁜 예: 직원(Employee) 클래스가 급여 계산도 하고, 근태 관리도 하고, 보고서 출력도 한다면? → 급여 계산 로직 변경 시 보고서 출력 코드가 망가질 위험이 있습니다.

**O — Open/Closed Principle (개방-폐쇄 원칙)**
> "확장에는 열려 있고, 수정에는 닫혀 있어야 한다."

새로운 기능을 추가할 때 기존 코드를 수정하지 않고도(폐쇄) 새 코드를 추가(개방)할 수 있어야 합니다. 인터페이스와 추상화가 핵심 도구입니다.

**L — Liskov Substitution Principle (리스코프 치환 원칙)**
> "자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다."

Bird → FlyingBird, Penguin처럼, 날지 못하는 펭귄에게 fly() 메서드를 강제하면 LSP 위반입니다.

**I — Interface Segregation Principle (인터페이스 분리 원칙)**
> "클라이언트가 사용하지 않는 인터페이스에 의존하면 안 된다."

뚱뚱한 인터페이스 하나보다 작은 인터페이스 여럿이 낫습니다.

**D — Dependency Inversion Principle (의존성 역전 원칙)**
> "고수준 모듈은 저수준 모듈에 의존하면 안 된다. 둘 다 추상화에 의존해야 한다."

```
나쁜 예: 주문서비스(고수준) → MySQL데이터베이스(저수준)
좋은 예: 주문서비스 → 데이터저장소 인터페이스 ← MySQL데이터베이스
```
의존성 역전 덕분에 MySQL을 PostgreSQL로 바꿔도 주문서비스 코드는 그대로입니다.

#### 의존성 주입(DI)과 다형성

**의존성 주입(Dependency Injection, DI)**은 DIP를 구현하는 대표적인 패턴입니다. 객체가 스스로 의존성을 생성하는 대신, 외부에서 주입받습니다.

**다형성(Polymorphism)**: 같은 이름의 메서드가 호출 대상 객체에 따라 다르게 동작하는 성질입니다.

```java
// 다형성 예시
Shape shape1 = new Circle();
Shape shape2 = new Rectangle();
shape1.draw();  // 원을 그림
shape2.draw();  // 사각형을 그림
```

## 4장: 애자일, Scrum, XP, TDD

### 애자일이란? "변화에 빠르게 적응하는 방법"

전통적인 폭포수(Waterfall) 방식은 요구사항 → 설계 → 개발 → 테스트 → 배포를 순차적으로 진행합니다. 문제는 마지막 단계에 가서야 "이게 아니잖아요"라는 피드백이 나온다는 것입니다.

**애자일**은 짧은 주기(스프린트)로 작동하는 소프트웨어를 반복해서 전달하며, 그때그때 피드백을 반영하는 방법론입니다.

#### CBD(컴포넌트 기반 개발) vs 애자일

| 구분 | CBD | 애자일 |
|------|-----|--------|
| 핵심 가치 | 재사용 가능한 컴포넌트 조립 | 변화에 빠른 적응, 고객과의 협력 |
| 산출물 | 독립 컴포넌트 라이브러리 | 작동하는 소프트웨어 |
| 계획 방식 | 선행 설계 중심 | 점진적·반복적 계획 |
| 현대적 진화 | MSA(마이크로서비스) | 스크럼·XP 등 |

#### 대규모 애자일: 혼자가 아닌 여럿이 함께

스크럼은 소규모 팀에 적합하지만, 수백 명이 협업할 때는 **SAFe, LeSS, Nexus** 같은 대규모 애자일 프레임워크가 필요합니다.

**SAFe(Scaled Agile Framework)**: 팀 → 프로그램 → 포트폴리오 3단계로 확장하는 가장 대중적인 프레임워크.

#### 스크럼(Scrum): 가장 많이 쓰이는 애자일 프레임워크

**스크럼의 3대 역할**:
- **Product Owner(PO)**: "무엇을 만들어야 하는가?" — 요구사항 우선순위 결정
- **Scrum Master**: "어떻게 잘 만들 수 있는가?" — 팀의 걸림돌 제거
- **개발팀(Dev Team)**: "직접 만드는 사람들" — 자율적으로 작업 선택

**스크럼의 5대 이벤트**:
1. **스프린트 계획(Sprint Planning)**: 이번 스프린트에 무엇을 할지 결정
2. **데일리 스크럼(Daily Scrum)**: 매일 15분, 어제/오늘/장애물 공유
3. **스프린트 리뷰(Sprint Review)**: 완성된 결과물을 고객에게 시연
4. **스프린트 회고(Retrospective)**: 팀이 어떻게 일했는지 반성·개선
5. **스프린트(Sprint)**: 1~4주 단위 실제 개발 사이클

#### XP(eXtreme Programming): 기술 실천법 중심

XP는 코딩 실천법에 집중합니다. 핵심 실천법은:
- **짝 프로그래밍(Pair Programming)**: 두 사람이 한 컴퓨터로 코딩
- **TDD(테스트 주도 개발)**: 테스트 먼저, 코드 나중
- **지속적 통합(CI)**: 매일 여러 번 코드를 통합하고 빌드
- **리팩토링**: 기능을 바꾸지 않고 코드 구조를 개선

#### TDD(Test-Driven Development): 테스트가 설계를 이끈다

TDD는 **Red → Green → Refactor** 사이클을 반복합니다.

1. **Red**: 실패하는 테스트를 먼저 작성
2. **Green**: 테스트를 통과하는 최소한의 코드 작성
3. **Refactor**: 동작은 유지하면서 코드 품질을 개선

TDD의 핵심 가치: "테스트 코드가 설계 문서이자 안전망입니다."

## 5장: DevOps, SRE, 무중단 배포

### DevOps: 개발팀과 운영팀의 벽을 허물다

전통적으로 개발팀(Dev)은 "빨리 배포하자", 운영팀(Ops)은 "안정적으로 유지하자"라는 상반된 목표로 충돌했습니다. **DevOps**는 이 두 팀이 협력하여 빠른 배포와 안정적 운영을 동시에 달성하는 문화·방법론·도구의 조합입니다.

#### DevOps의 성공 저해 요인

기술보다 조직과 문화가 더 큰 장벽입니다:
1. **Silo 현상**: 개발팀과 운영팀이 서로 담쌓고 이기주의
2. **도구 맹신**: Kubernetes만 도입하면 DevOps가 될 것이라는 착각
3. **규제와의 충돌**: 직무 분리 규정이 자동화 흐름을 차단
4. **단기 성과주의**: 경영진의 빠른 결과 요구로 자동화 투자 포기

#### DevOps의 실천: CALMS 프레임워크

| 요소 | 의미 |
|------|------|
| **C**ulture | 공동 책임 문화 |
| **A**utomation | CI/CD, IaC 자동화 |
| **L**ean | 낭비 제거, 가치 흐름 최적화 |
| **M**easurement | 배포 빈도, MTTR 등 측정 |
| **S**haring | 성공과 실패 경험 공유 |

#### SRE(Site Reliability Engineering): DevOps의 구글식 구현

DevOps가 "철학"이라면, SRE는 "구현 방법"입니다. 구글이 만든 개념으로, 소프트웨어 엔지니어가 운영 업무를 맡아 자동화로 해결합니다.

**SRE의 핵심: SLI → SLO → SLA**

```
SLI(지표): "API 응답 시간이 200ms 이하인 비율"
SLO(목표): "위 비율이 99.9% 이상이어야 한다"
SLA(계약): "SLO 미달 시 10% 환불"
```

**에러 버젯(Error Budget)**: `에러 버젯 = 100% - SLO`

SLO가 99.9%이면 에러 버젯은 0.1%입니다. 이 버젯이 남아있는 동안은 과감히 배포하고, 다 써버리면 모든 신규 배포를 자동 차단합니다. 개발 속도와 안정성을 균형 있게 제어하는 수학적 장치입니다.

#### 무중단 배포(Zero-Downtime Deployment) 3가지 방법

새벽 점검 시간에 서버를 내리고 배포하는 시대는 끝났습니다. 글로벌 서비스는 언제나 가동 중이어야 합니다.

**1. 블루-그린 배포(Blue-Green)**

```
현재: 로드밸런서 → [블루: V1 × 4대] (운영)
                  [그린: V2 × 4대] (대기)
전환: 로드밸런서 → [그린: V2 × 4대] (운영)
                  [블루: V1 × 4대] (롤백 대기)
```

- 장점: 즉각 롤백 가능
- 단점: 인프라 비용 2배

**2. 롤링 배포(Rolling)**

서버를 하나씩 순차적으로 교체합니다. 추가 비용이 적지만, 배포 중 구버전·신버전이 공존합니다.

**3. 카나리 배포(Canary)**

```
1단계: V1 95% / V2 5% (소수에만 노출, 모니터링)
2단계: V1 80% / V2 20% (이상 없으면 확대)
4단계: V1 0% / V2 100% (완전 전환)
```

- 장점: 문제 발생 시 영향 범위 최소화
- 단점: 세션 고정 등 인프라 복잡도 높음

| 방식 | 비용 | 롤백 난이도 | 위험 통제 |
|------|------|------------|----------|
| 블루-그린 | 높음 | 매우 쉬움 | 중간 |
| 롤링 | 낮음 | 어려움 | 중간 |
| 카나리 | 낮음~중간 | 쉬움 | 최고 |

# 2부: 요구사항과 소프트웨어 아키텍처

## 6장: 요구사항 공학

### 요구사항이 왜 중요한가?

소프트웨어 프로젝트 실패의 가장 큰 원인은 **잘못 이해된 요구사항**입니다. 아무리 훌륭한 코드도 고객이 원하는 게 아니라면 실패입니다. 요구사항 공학(Requirements Engineering)은 "고객이 진짜 원하는 게 무엇인지"를 체계적으로 발견하고 명세화하는 분야입니다.

#### 기능 요구사항 vs 비기능 요구사항

| 구분 | 설명 | 예시 |
|------|------|------|
| **기능 요구사항** | 시스템이 무엇을 해야 하는가 | "로그인 기능", "결제 기능" |
| **비기능 요구사항** | 시스템이 어떻게 해야 하는가 | "응답시간 1초 이내", "99.9% 가용성" |

비기능 요구사항은 종종 무시되다가 운영 단계에서 심각한 문제를 일으킵니다.

#### IT 요구사항의 분류 (Wiegers 모델)

요구사항은 여러 계층으로 구성됩니다:

```
비즈니스 요구사항 (왜 만드는가?)
    ↓
이해관계자 요구사항 (누가 무엇을 원하는가?)
    ↓
시스템 요구사항 (시스템이 무엇을 해야 하는가?)
    ↓
소프트웨어 요구사항 (소프트웨어가 어떻게 구현해야 하는가?)
```

#### 요구사항 명세의 핵심 품질 속성

좋은 요구사항 명세는 다음을 만족해야 합니다:
- **완전성(Completeness)**: 빠진 요구사항이 없을 것
- **일관성(Consistency)**: 서로 모순되지 않을 것
- **추적성(Traceability)**: 각 요구사항이 어디서 왔는지 추적 가능할 것
- **검증 가능성(Verifiability)**: 요구사항 충족 여부를 테스트로 확인 가능할 것

## 7장: 소프트웨어 아키텍처

### 아키텍처란? "크고 중요한 설계 결정들"

건물의 뼈대처럼, 소프트웨어 아키텍처는 시스템의 핵심 구조를 결정합니다. 나중에 바꾸기 어렵기 때문에 초기에 신중하게 결정해야 합니다.

#### ISO 42010: 아키텍처를 보는 여러 관점

아키텍처는 단 하나의 다이어그램으로 표현할 수 없습니다. ISO 42010은 **이해관계자마다 관심사(Concern)가 다르므로, 다양한 관점(Viewpoint)으로 아키텍처를 기술**해야 한다고 규정합니다.

예를 들어:
- **개발자 관점**: 코드 모듈 구조
- **운영자 관점**: 배포·인프라 구조
- **보안 담당자 관점**: 접근 통제·데이터 흐름

#### 아키텍처 품질 속성(Quality Attribute Scenarios, QAS)

아키텍처는 기능만 담당하지 않습니다. 품질 속성도 아키텍처가 결정합니다:

| 품질 속성 | 설명 | 관련 전술 |
|-----------|------|----------|
| **가용성(Availability)** | 장애 없이 서비스 가능 | 이중화, 헬스체크 |
| **성능(Performance)** | 빠른 응답 | 캐싱, 비동기 처리 |
| **보안성(Security)** | 비인가 접근 방지 | 인증·인가, 암호화 |
| **유지보수성(Maintainability)** | 쉽게 변경 가능 | 모듈화, 인터페이스 기반 설계 |
| **확장성(Scalability)** | 부하에 따라 확장 | 스케일 아웃, MSA |

#### 주요 아키텍처 패턴

**레이어드 아키텍처(Layered Architecture)**
- Presentation → Business → Persistence → Database
- 장점: 단순하고 이해하기 쉬움
- 단점: 계층 간 결합으로 유연성 제한

**MSA(Microservices Architecture)**
- 하나의 큰 서비스를 작은 독립 서비스들로 분해
- 장점: 독립 배포, 기술 이질성 허용
- 단점: 분산 시스템 복잡성 증가

**이벤트 드리븐 아키텍처(Event-Driven)**
- 서비스 간 이벤트(메시지)로 통신
- 장점: 느슨한 결합
- 단점: 디버깅과 추적이 어려움

#### ATAM(Architecture Tradeoff Analysis Method): 아키텍처 평가

ATAM은 아키텍처가 품질 요구사항을 얼마나 잘 충족하는지 평가하는 방법론입니다.

핵심 단계:
1. 아키텍처 발표
2. 이해관계자별 품질 속성 시나리오 도출
3. 각 시나리오에 대한 아키텍처 접근법 분석
4. **트레이드오프(Trade-off) 식별**: "성능을 높이면 유지보수성이 낮아진다"
5. **위험 식별** 및 **민감 지점(Sensitivity Point)** 발굴

# 3부: 소프트웨어 테스트

## 8장: 테스트 원칙과 계획

### "테스트는 버그가 없음을 증명하는 활동이 아니다"

많은 사람이 테스트를 "버그가 없다는 것을 확인하는 과정"으로 오해합니다. 하지만 **테스트는 결함이 존재함을 증명하는 활동**입니다. 결함이 없다는 것은 증명 불가능합니다.

#### ISTQB 소프트웨어 테스트 7원칙

**원칙 1: 테스팅은 결함의 존재를 보여준다**
- 테스트 통과 ≠ 버그 없음
- 테스트 케이스는 결함을 찾도록 설계해야 함

**원칙 2: 완벽한 테스팅은 불가능하다**
- 모든 입력 조합 테스트는 수학적으로 불가능
- **대안**: 리스크 기반 테스팅으로 우선순위 설정

**원칙 3: 조기 테스팅이 비용을 절감한다**
- Barry Boehm의 법칙: 요구사항 단계에서 발견한 결함 수정 비용 = $1, 운영 단계 = $100~$1000
- **Shift Left Testing**: 테스트를 개발 초기로 당기자

**원칙 4: 결함은 군집되어 있다 (파레토 법칙)**
- 전체 결함의 80%는 20%의 모듈에서 발생
- 복잡도 높은 모듈, 과거 버그 빈발 모듈 집중 테스트

**원칙 5: 살충제 패러독스**
- 같은 테스트를 반복하면 새로운 결함을 찾지 못함
- 테스트 케이스를 주기적으로 갱신·다양화해야 함

**원칙 6: 테스팅은 정황에 의존한다**
- 의료 기기 SW ≠ 모바일 게임 — 서로 다른 테스트 전략 필요

**원칙 7: 오류 부재의 궤변**
- 결함이 없어도 고객이 원하는 게 아니면 실패
- **확인(Validation)**: "올바른 것을 만들었는가?"

#### IEEE 829 테스트 문서 표준

IEEE 829는 테스트 관련 문서의 표준 형식을 정의합니다. 핵심 산출물:
- **테스트 계획서(Test Plan)**: 전략, 범위, 일정, 자원
- **테스트 케이스(Test Case)**: 입력, 실행 조건, 예상 결과
- **테스트 결과 보고서**: 발견된 결함, 통과/실패 현황

## 9장: 테스트 기법

### 9.1 통합 테스트: 모듈을 합치면 새로운 문제가 생긴다

개별 모듈이 완벽해도 **합쳤을 때 인터페이스에서 문제**가 생깁니다. 통합 테스트는 이 문제를 발견합니다.

**빅뱅 통합**: 모든 모듈을 한꺼번에 합쳐 테스트. 문제 발생 시 원인 찾기 어려움.

**점진적 통합**: 하나씩 추가하며 통합.
- **하향식(Top-down)**: 상위 모듈부터 통합, 하위 모듈은 스텁(Stub)으로 대체
- **상향식(Bottom-up)**: 하위 모듈부터 통합, 상위 모듈은 드라이버(Driver)로 대체

### 9.2 회귀 테스트: "고쳤더니 다른 게 망가졌어요"

변경 후 기존 기능이 여전히 정상 동작하는지 확인하는 테스트입니다.

- **자동화 필수**: JUnit, Selenium 등으로 CI/CD 파이프라인에 통합
- **살충제 패러독스 주의**: 회귀 테스트 스위트도 주기적으로 새 케이스 추가 필요

### 9.3 리스크 기반 테스팅(RBT)

**"모든 것을 테스트할 수 없다면, 가장 위험한 것부터 테스트하라"**

```
리스크 = 발생 확률 × 영향도
```

**4단계 프로세스**:
1. 리스크 식별 — 브레인스토밍, 체크리스트
2. 리스크 평가 — 확률 × 영향도 산정
3. 우선순위화 — 리스크 매트릭스
4. 테스트 배분 — High Risk는 집중 테스트, Low Risk는 최소 테스트

### 9.4 성능 테스트: 빠르게, 안정적으로, 오래

**리틀의 법칙(Little's Law)**:
```
L = λ × W

L: 시스템 내 평균 요청 수
λ: 초당 유입 요청 수 (처리율)
W: 평균 처리 시간 (응답 시간)
```

예) 동시 접속 1,000명, 평균 응답 2초 → λ = 500 TPS 필요

**성능 테스트 4가지 유형**:

| 종류 | 목적 | 부하 패턴 |
|------|------|----------|
| **부하 테스트** | 목표 부하에서 성능 목표 충족 여부 | 점진적 증가 후 유지 |
| **스트레스 테스트** | 한계점(Breaking Point) 탐색 | 계속 증가 |
| **내구성 테스트(Soak)** | 메모리 누수, 장시간 자원 고갈 | 정상 부하 12~48시간 유지 |
| **스파이크 테스트** | 급격한 부하 변동에 대한 반응 | 급등 후 급강하 |

### 9.5 블랙박스 vs 화이트박스 테스트

| 구분 | 블랙박스 | 화이트박스 |
|------|---------|----------|
| 관점 | 외부(입출력) | 내부(소스 코드) |
| 기반 | 요구사항 명세서 | 소스 코드 |
| 주체 | 독립 테스터 | 개발자 |
| 대표 기법 | 동등분할, 경계값 분석 | 커버리지 기반 |

**블랙박스 = 명세기반, 화이트박스 = 구조기반, 경험기반 = 탐색적 테스팅**

### 9.6 동등분할과 경계값 분석

**동등분할(Equivalence Partitioning)**: 입력 도메인을 같은 방식으로 처리되는 그룹으로 나누고, 각 그룹에서 대표값 하나만 테스트합니다.

예) 공기업 채용: 19세 미만 / 19~20세(인턴) / 21~60세(정규직) / 61세 이상 — 4개 파티션

**경계값 분석(BVA)**: 경계에서 결함이 가장 많이 발생합니다.
- 2-value BVA: 경계-1 + 경계 (예: 18, 19)
- 3-value BVA: 경계-1 + 경계 + 경계+1 (예: 18, 19, 20)

**Off-by-One 오류**: `if (age >= 19)` 대신 `if (age > 19)`처럼 경계를 1만큼 잘못 지정하는 가장 흔한 버그.

### 9.7 조건/결정 커버리지와 MC/DC

**커버리지 강도 순서 (약함 → 강함)**:
```
구문 커버리지(SC) → 결정 커버리지(DC) → 조건 커버리지(CC) → MC/DC → 경로 커버리지
```

**MC/DC(Modified Condition/Decision Coverage)**: 각 조건이 독립적으로 결정 결과에 영향을 미침을 증명해야 합니다. n개 조건 → 최소 n+1개 테스트 케이스로 달성 가능합니다.

왜 중요한가? 항공 안전 표준 DO-178C Level A(생명 손실 가능 수준)에서 MC/DC가 **의무**입니다.

# 4부: 유지보수와 품질 관리

## 10장: SW 유지보수

### 소프트웨어는 만드는 것보다 유지하는 데 더 많은 비용이 든다

소프트웨어 개발 비용의 60~80%가 **유지보수**에 쓰입니다.

#### Lehman의 SW 변화관리 8대 법칙

Manny Lehman이 실증 연구로 발견한 소프트웨어 진화의 보편 법칙들. **E-type 소프트웨어**(급여 시스템, ERP처럼 실세계 환경과 상호작용하는 소프트웨어)에만 적용됩니다.

| 법칙 | 핵심 메시지 |
|------|-----------|
| **1. 지속적 변화** | 업데이트 안 하면 도태됨 |
| **2. 복잡도 증가** | 관리 안 하면 엔트로피가 쌓임 |
| **3. 자기 조절** | 릴리즈 패턴은 통계적으로 안정됨 |
| **4. 조직적 안정** | 인력 늘린다고 속도가 무한히 빨라지지 않음 |
| **5. 변화 보존** | 릴리즈당 변경량은 일정함 |
| **6. 기능 지속 증가** | 경쟁에서 살아남으려면 계속 기능을 추가해야 함 |
| **7. 품질 저하** | 적극적 관리 없이는 품질은 자동으로 나빠짐 |
| **8. 피드백 시스템** | 위 법칙들은 서로 영향을 주고받는 복잡 시스템 |

**현장 대응**: 법칙 2와 7에 맞서려면 지속적 리팩토링과 기술 부채 가시화 관리가 필수입니다.

#### ISO/IEC 14764: 유지보수 4가지 유형

| 유형 | 목적 | 원인 | 비율 |
|------|------|------|------|
| **수정적(Corrective)** | 결함 수정 | 운영 중 버그 발견 | 20% |
| **적응적(Adaptive)** | 환경 적응 | OS·법규 변경 | 25% |
| **완전화(Perfective)** | 기능/성능 개선 | 사용자 신규 요구 | 50% |
| **예방적(Preventive)** | 미래 결함 예방 | 복잡도 증가 경고 | 5% |

**가장 많이 혼동하는 부분**: 완전화(사용자가 요청한 기능 추가)와 예방적(내부 품질 개선, 사용자에게 보이지 않음)의 차이.

**유지보수 프로세스 (ISO 14764 6단계)**:
변경 요청(MR) 접수 → 영향도 분석 → 수정 구현 → 검토/인수 → 이행 → (폐기)

## 11장: 3R (재사용·역공학·재공학)

### 소프트웨어 위기를 극복하는 세 가지 열쇠

**소프트웨어 위기(Software Crisis)**: 1968년 NATO 컨퍼런스에서 등장한 개념. 복잡성 증가·생산성 정체·레거시 누적이 원인입니다.

3R은 기존 자산을 살리면서 품질을 높이는 전략입니다:
- **Reuse(재사용)**: 검증된 컴포넌트를 다시 쓴다
- **Reverse Engineering(역공학)**: 코드에서 설계를 복원한다
- **Restructuring(재구조화)**: 기능은 그대로, 구조를 개선한다

#### 리팩토링: 코드를 새로 짜지 않고 더 나은 코드로

**리팩토링(Refactoring)**: "외부에서 관찰 가능한 행동을 변경하지 않으면서 내부 구조를 개선하는 것" (Martin Fowler)

**핵심 조건**: ① 기능(입출력)은 동일해야 함, ② 내부 구조는 개선되어야 함. 테스트 코드 없이 하는 리팩토링은 '모험'입니다.

**코드 스멜(Code Smell)**: 리팩토링이 필요하다는 신호들:

| 코드 스멜 | 설명 | 대응 기법 |
|----------|------|---------|
| 긴 메서드 | 한 함수가 너무 많은 일을 함 | Extract Method |
| 중복 코드 | 같은 코드 반복 | Extract Method, Pull Up Method |
| 산탄총 수술 | 하나 바꾸면 여러 곳을 고쳐야 함 | Move Method |
| 죽은 코드 | 실행되지 않는 코드 | Remove Dead Code |

## 12장: QA/QC와 품질 도구

### QA와 QC: 예방하는 자와 발견하는 자

| 구분 | QA (품질보증) | QC (품질통제) |
|------|--------------|--------------|
| 목적 | 결함 **예방** | 결함 **발견/제거** |
| 대상 | 프로세스 | 제품/산출물 |
| 성격 | 능동적(Proactive) | 수동적(Reactive) |
| 비용 효과 | 조기 발견 → 저비용 | 후기 발견 → 고비용 |

**QA ≠ 테스팅**: QA는 테스팅을 포함하지만, 테스팅 자체는 QC의 영역입니다.

**Boehm의 결함 수정 비용**: 요구사항 단계 = $1, 릴리즈 후 = $100. QA에 투자해야 하는 이유입니다.

#### QC 7도구: 데이터로 품질 문제를 진단하다

**구 QC 7도구 (수치 데이터 분석)**:

1. **파레토도**: 80-20 법칙으로 가장 중요한 원인 20%를 찾는 막대+꺾은선 차트
2. **특성요인도(이시카와/어골도)**: 문제의 근본 원인을 생선뼈 모양으로 시각화
3. **히스토그램**: 데이터 분포 형태 파악
4. **산점도**: 두 변수 간 상관관계 파악
5. **체크시트**: 결함 유형별 발생 빈도 기록
6. **층별**: 데이터를 조건별 그룹으로 분류
7. **관리도**: UCL/CL/LCL으로 프로세스 이상 모니터링

**신 QC 7도구 (언어 데이터 분석)**: 친화도, 연관도, 계통도, 매트릭스도, PDPC, 애로우다이어그램, 매트릭스 데이터 해석

## 13장: 형상관리 (SCM)

### "어제까지 잘 됐는데 오늘 갑자기 안 돼요"를 방지하는 관리 체계

형상관리(SCM: Software Configuration Management)는 소프트웨어의 모든 변경을 통제하고 추적하는 체계입니다. Git만 쓰는 것이 형상관리가 아닙니다. Git은 형상관리의 '식별'과 '기록' 도구일 뿐입니다.

#### 형상 항목(CI): 관리 대상

소스 코드뿐 아니라 **요구사항 명세서, 설계 문서, 테스트 케이스, 빌드 스크립트, 배포 패키지**까지 모두 형상 항목입니다.

#### 기준선(Baseline): "이 상태가 공식 버전"

기준선이 설정되면 이후 변경은 반드시 공식 변경 통제 절차(CCB 승인)를 거쳐야 합니다.

**SDLC 단계별 기준선**:

| 단계 | 기준선 이름 | 주요 산출물 |
|------|-----------|-----------|
| 요구사항 분석 | 기능 기준선 | SRS(요구사항 명세서) |
| 설계 | 할당 기준선 | HLD(기능 설계서) |
| 구현/통합 | 제품 기준선 | 소스코드, LLD |
| 배포 | 운영 기준선 | 실행 파일, 릴리즈 노트 |

#### 형상관리 4대 수행 활동

1. **식별**: 모든 CI에 유일한 식별자 부여, 버전 번호 체계 정의
2. **통제**: CCB(변경 통제 위원회) 통해 승인된 변경만 적용
3. **감사**: 실제 코드가 문서와 일치하는지 독립 검증
4. **기록**: 누가, 언제, 무엇을, 왜 변경했는지 감사 추적(Audit Trail)

**버전 명명 규칙 (시맨틱 버저닝)**: `[Major].[Minor].[Patch]`
- Major: 하위 호환 불가 변경
- Minor: 하위 호환 기능 추가
- Patch: 버그 수정

# 5부: 기능안전과 품질인증

## 14장: 소프트웨어 기능안전

### "소프트웨어가 사람을 다치게 하지 않도록"

자동차 브레이크, 항공기 제어, 의료기기처럼 오작동 시 인명 피해가 발생할 수 있는 시스템을 **안전 필수(Safety-Critical) 시스템**이라고 합니다.

#### Safety vs Reliability vs Security: 헷갈리지 말자

| 개념 | 정의 | 핵심 질문 |
|------|------|---------|
| **Safety(안전)** | 시스템이 사람/환경에 해를 끼치지 않을 것 | "사람이 다칠 수 있는가?" |
| **Reliability(신뢰성)** | 명세된 기능을 정해진 시간 동안 수행 | "기대한 대로 동작하는가?" |
| **Security(보안)** | 의도적 공격으로부터 보호 | "해킹될 수 있는가?" |

신뢰성이 높아도 안전하지 않을 수 있습니다! (완벽하게 명세대로 동작해도, 명세 자체가 위험한 상황을 고려 안 했다면)

#### 안전성 분석 4가지 기법

**FTA (Fault Tree Analysis, 결함수 분석)**:
- 방향: **하향식(Top-Down, 연역적)**
- 출발점: 최상위 사고(Top Event)에서 원인으로 내려감
- 도구: AND/OR 논리 게이트
- 강점: 단일 점 장애(Single Point of Failure) 식별

**FMEA (Failure Mode and Effect Analysis)**:
- 방향: **상향식(Bottom-Up, 귀납적)**
- 출발점: 개별 컴포넌트 고장 → 시스템 영향 분석
- 지표: RPN = 심각도(S) × 발생도(O) × 검출도(D)
- 강점: 전수 검토, 수치적 우선순위

**HAZOP (Hazard and Operability Study)**:
- 방향: **편차 중심**
- 출발점: 설계 의도에서 벗어나는 모든 경우 탐색
- 도구: 가이드워드(NO/MORE/LESS/REVERSE 등)
- 강점: 운용성·안전성 동시 분석

**PHA (Preliminary Hazard Analysis)**:
- 프로젝트 초기에 전체 위험원을 빠르게 파악
- FTA/FMEA/HAZOP 수행 전 선행 작업

| 기법 | 방향 | 강점 | 약점 |
|------|------|------|------|
| FTA | 하향식 | SPF 식별, 확률 계산 | Top Event 사전 정의 필요 |
| FMEA | 상향식 | 전수 검토, RPN | 복합 고장 분석 어려움 |
| HAZOP | 편차 중심 | 운용 편차 발견 | 시간 소요 |

#### IEC 61508: 기능안전의 범용 표준

**SIL (Safety Integrity Level)**: 안전 기능의 성능 등급 (1~4)
- SIL 4: 생명 위협 수준 (원전 핵심 안전 시스템)
- SIL 1: 일반 산업 기계 안전 기능

**IEC 61508 파생 표준**:

| 도메인 | 표준 | 위험 등급 |
|--------|------|---------|
| E/E/PE 시스템 전반 | IEC 61508 | SIL 1~4 |
| **자동차** | ISO 26262 | **ASIL A~D** |
| 의료기기 SW | IEC 62304 | Class A/B/C |
| **항공** | DO-178C | **DAL A~E** |

#### ISO 26262: 자동차 기능안전

**ASIL (Automotive Safety Integrity Level)**: IEC 61508의 SIL을 자동차에 특화.

ASIL 결정 파라미터:
- **심각도(Severity, S)**: S0~S3 (부상 없음~생명 위협)
- **노출도(Exposure, E)**: E0~E4 (해당 상황 노출 빈도)
- **제어 가능성(Controllability, C)**: C0~C3 (운전자가 회피 가능한가?)

DO-178C(항공)에서 Level A는 MC/DC 커버리지 100%가 **의무**입니다.

## 15장: SW 품질인증 제도

### GS 인증: "이 소프트웨어는 합격입니다"

**GS(Good Software) 인증**: TTA(한국정보통신기술협회)가 ISO/IEC 25010 기준으로 소프트웨어 품질을 제3자 검증하는 국내 인증 제도.

#### ISO/IEC 25010: 8대 품질 특성

| 품질 특성 | 설명 |
|-----------|------|
| 기능 적합성 | 명시된 기능을 올바르게 수행하는가 |
| 성능 효율성 | 자원(시간, 메모리) 사용이 효율적인가 |
| 호환성 | 다른 환경·시스템과 잘 작동하는가 |
| 사용성 | 쉽고 효율적으로 사용할 수 있는가 |
| 신뢰성 | 장애 없이 안정적으로 동작하는가 |
| 보안성 | 비인가 접근으로부터 보호되는가 |
| 유지보수성 | 수정·개선이 쉬운가 |
| 이식성 | 다른 환경으로 쉽게 이전되는가 |

**GS 인증의 실용적 가치**: 공공기관 조달 우선 구매, 유효기간 **2년**.

#### SW 프로세스 품질인증 (SP 인증/CMMI)

| 구분 | GS 인증 (제품) | SP/CMMI 인증 (프로세스) |
|------|--------------|----------------------|
| 평가 대상 | 소프트웨어 제품 | 개발 프로세스 성숙도 |
| 비유 | 음식 맛 직접 평가 | 주방 조리 시스템 평가 |

**CMMI 5단계 성숙도**:

| 레벨 | 이름 | 핵심 |
|------|------|------|
| 1 | 초기 (Initial) | 혼돈, 개인 능력에 의존 |
| 2 | 관리 (Managed) | 프로젝트 단위 계획·추적 |
| 3 | 정의 (Defined) | 조직 표준 프로세스 정의 |
| 4 | 정량적 관리 (Quantitatively Managed) | 통계적 통제 |
| 5 | 최적화 (Optimizing) | 데이터 기반 지속 개선 |

**SPICE(ISO/IEC 33000)**: 0~5의 **6단계** (CMMI는 1~5의 5단계 — 이 차이 중요!)

**국내 SP 인증**: SP 1등급(CMMI Level 3 수준)은 공공 입찰 시 최대 6점 가점.

# 6부: 비용산정과 발주

## 16장: H/W 용량산정

### "얼마나 큰 서버가 필요한가?"

너무 작으면 서비스 오픈 후 성능 장애, 너무 크면 예산 낭비. 적절한 H/W 용량산정은 IT 사업의 출발점입니다.

#### 핵심 산정 흐름

```
일 최대 거래건수 ÷ 유효시간(초) × 피크 배율 = TPS(초당 처리량)
     ↓
TPS × 트랜잭션당 CPU 처리시간 ÷ 목표 CPU 활용률 = 필요 CPU 용량
```

**주요 결정점**:
- **피크 배율**: 최대 트래픽이 평균의 몇 배인가? (보통 2~3배)
- **목표 CPU 활용률**: 통상 60~70% (여유율 확보)
- **동시 사용자 수**: 전체 사용자의 5~10%

**성능 기준 지표**:
- **tpmC**: 분당 신규 주문 트랜잭션 처리 수 (CPU 성능 기준)
- **IOPS**: 초당 I/O 작업 수 (스토리지 성능 기준)

## 17장: SW 비용산정

### 기능점수(Function Point): 사용자 관점의 소프트웨어 크기 측정

**기능점수(FP)**: 코드 줄 수(LOC)가 아니라 **사용자가 인식하는 기능의 수**로 소프트웨어 규모를 측정. 언어·기술 독립적입니다.

#### FP의 5대 요소

**데이터 기능**:
- **ILF(Internal Logical File)**: 시스템이 직접 유지·관리하는 내부 데이터 (예: 고객 테이블)
- **EIF(External Interface File)**: 외부에서 참조만 하는 데이터 (예: 타 시스템 공통 코드)

**트랜잭션 기능**:
- **EI(External Input)**: 외부에서 입력받아 내부 데이터 변경 (예: 회원가입)
- **EO(External Output)**: 내부 데이터 처리·계산 후 출력 (예: 통계 보고서)
- **EQ(External inQuiry)**: 단순 조회 (예: 목록 조회 화면)

```
UFP(미조정 FP) = Σ(각 요소 개수 × 복잡도 가중치)
AFP(조정 FP) = UFP × 보정 계수
개발비 = AFP × 기능점수당 단가
```

**5가지 보정 계수**: 규모 보정 / 언어 보정 / 연도 보정 / 품질·특성 보정 / 애플리케이션 유형 보정

#### SW 비용산정 방법 비교

| 방법 | 특징 | 장점 | 단점 | 적합 시기 |
|------|------|------|------|----------|
| **전문가 판단** | 경험 기반 직관적 산정 | 빠름 | 주관적 | 프로젝트 초기 |
| **델파이 기법** | 다수 전문가 합의 | 객관성 향상 | 시간 소요 | 초기~중기 |
| **상향식 WBS** | 작업 분해 후 합산 | 정확도 높음 | 요구사항 명확해야 함 | 중기~후기 |
| **COCOMO II** | 수학적 알고리즘 | 재현 가능 | 파라미터 보정 어려움 | 중기~후기 |

**COCOMO II 3단계**: Application Composition(초기) → Early Design(설계) → Post-Architecture(구현, 가장 정확)

**불확실성 콘(Cone of Uncertainty)**: 프로젝트 초기 = 오차 ±4배, 상세 설계 후 = ±1.5배. 따라서 초기에는 범위 산정, 후기에는 WBS 상향식 산정을 씁니다.

## 18장: 발주 프로세스

### 단계별 발주: "요구사항도 모르고 전체 계약하지 마라"

전통적인 일괄 발주: 요구사항이 불명확한 상태에서 전체 개발 비용을 계약 → 나중에 변경 분쟁.

**단계별 발주**: 분석(1단계)과 설계/구현(2단계)을 분리 계약. 앞 단계 결과물을 발주자가 검수한 뒤 다음 단계를 계약합니다.

| 구분 | 일반 발주 | 단계별 발주 |
|------|----------|-----------|
| 계약 구조 | 전체 1건 | 단계별 복수 계약 |
| 위험 부담 | 수급자 집중 | 발주자·수급자 분산 |
| 품질 | 낮을 위험 | 높음 |

**단계별 발주의 한계**: 계약 공백(1단계 완료 후 2단계 입찰 기간), 발주자 IT 전문성 요구.

#### 상용 SW 직접구매 제도

기존: SW 제조사 → SI/대리점 → 수요기관 (중간 마진, 가격 불투명)
직접구매: SW 제조사 → 나라장터·디지털서비스몰 → 수요기관 (투명한 가격)

**라이선스 유형**:
- **영구 라이선스**: 한 번 구매, CAPEX 방식
- **구독 라이선스**: 연간 납부, OPEX 방식, 항상 최신 버전
- **SaaS**: 클라우드 기반, 종량제 — 클라우드 퍼스트 정책과 부합

**디지털서비스몰**: 클라우드·SaaS·AI 서비스 특화 공공조달 채널. 기존 나라장터(HW 중심)의 보완.

## 마무리: 소프트웨어 공학을 배우는 이유

소프트웨어 공학의 모든 개념들은 결국 하나의 문제를 해결하려 합니다: **"어떻게 하면 좋은 소프트웨어를 예측 가능하게, 반복적으로 만들 수 있는가?"**

- **설계 원칙**(SOLID, 결합도/응집도): 변경하기 쉬운 코드를 만들기 위해
- **방법론**(애자일, DevOps): 변화에 빠르게 적응하기 위해
- **테스트**: 결함을 조기에 발견해 비용을 줄이기 위해
- **품질 관리**(QA/QC, CMMI): 좋은 소프트웨어를 지속적으로 만드는 능력을 쌓기 위해
- **비용산정·발주**: 프로젝트를 예측 가능하게 관리하기 위해

이 모든 것이 **"좋은 소프트웨어를 만드는 훈련"**입니다. 핵심을 이해하면 개별 용어는 자연스럽게 따라옵니다.
