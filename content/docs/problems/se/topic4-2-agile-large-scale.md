## 문제

수십 명이 참여하는 대규모 IT프로젝트에 애자일(Agile)을 적용하기 위해서는 효과적인 방안수립이 필요하다. 아래사항에 대하여 설명하시오.

- 가. 대규모 IT 프로젝트에서 발생하는 주요 문제점
- 나. 대규모 IT 프로젝트의 애자일 적용 전략
- 다. 대규모 IT 프로젝트의 애자일 적용 절차(로드맵)

## 해설

### 출제 배경 및 의도

오늘날 엔터프라이즈 소프트웨어 개발의 본질적인 딜레마는 '거대함(Scale)과 기민함(Agility)의 공존'입니다. 클라우드 인프라와 MSA의 확산으로 시스템은 잘개 쪼개졌지만, 역설적으로 그 쪼개진 수십 개의 모듈을 통합하고 관리해야 하는 복잡성은 기하급수적으로 증가했습니다.

채점관이 이 문제를 출제하는 의도는 수험생이 단순히 5~8명이 모여 데일리 스크럼을 하는 '소팀 이론가'인지, 아니면 거대한 예산과 수십 명의 이해관계자가 얽힌 거시적 복잡계에서 조직론, 위험 관리, 거버넌스 프레임워크를 조율해낼 수 있는 '엔터프라이즈 아키텍트' 역량을 갖추었는지 검증하기 위함입니다.

소규모 팀 단위의 애자일을 대규모 복잡계 프로젝트에 무작정 투입하면 통제 불능의 카오스 상태를 초래하므로, 확장형 애자일 프레임워크와 조직 거버넌스 설계 능력이 핵심입니다.

### 1. 대규모 IT 프로젝트 애자일 적용 개요

정 의  • 수십 명 이상이 참여하는 대규모 IT 프로젝트에서
       - 폭포수의 통제 안정성과 애자일의 기민성을 결합한
       - 엔터프라이즈 수준의 확장형 애자일(Scaled Agile)을 적용하는 방법론

```
[기존 대규모 프로젝트]              [엔터프라이즈 애자일 적용]
폭포수 순차 개발                    다중 스크럼팀 + 릴리즈 트레인
  ↓                                   ↓
요구사항 빅뱅 통합                 PI(Program Increment) 기반 동기화
  ↓                                   ↓
후반부 통합 블랙박스 장애           CI/CD 기반 점진적 빌드·검증
```

- 대규모 프로젝트에서는 PMBOK 관리 영역 전반에 걸친 병목과 리스크를 선제적으로 진단해야 함

### 2. 대규모 IT 프로젝트의 주요 문제점

1) PMBOK 관리 영역별 문제점 분석

| 관리 영역 | 주요 문제점 | 근본 원인 | 애자일 해법 지향점 |
|----------|-----------|----------|-----------------|
| 의사소통 | 커뮤니케이션 오버헤드 폭발 | 채널 수 N(N-1)/2로 증가, 사일로(Silo) 발생 | 교차 기능팀(Cross-Functional) 전환 |
| 일정관리 | 납기·일정 지연 고질화 | 파킨슨의 법칙, 학생 증후군에 따른 버퍼 잠식 | 점진적 Sprint를 통한 상시 가치 인도 |
| 범위관리 | 무분별한 요구사항 확장 | 스코프 크립(Scope Creep), 골드 플래팅 | 제품 백로그의 엄격한 우선순위 통제 |
| 비용관리 | 예산 초과·원가 통제 상실 | 실시간 실제 원가 측정 모호성 | 릴리즈 단위 EVM 분석 연계 |
| 품질관리 | 통합 시점 블랙박스 장애 | 요구사항 변경 지연 반영, 개발-테스트 격리 | CI 및 TDD 의무화 |
| 위험관리 | 리스크 조기 감지 실패 | 후반부 마일스톤 집중의 단발성 위험 검토 | 스프린트 데모·상시 피드백 통한 조기 제거 |

- 대규모 프로젝트의 문제점은 개별 관리 영역 단위로 진단해야 하며, 각 문제점에 대응하는 애자일 해법이 유기적으로 결합되어야 근본적 해소가 가능함

### 3. 애자일 적용 전략과 로드맵

1) 애자일 적용 전략 수립 5단계 프로세스

| 구분 | 단계 | 수행 내용 | 분석 도구 |
|------|------|----------|---------|
| 1단계 | 환경·구조 파악 | 기술적 위험 요인, 인프라 제약, 품질 요소 식별 | RFP/요구사항 정의서 |
| 2단계 | 목표 설정 | 애자일 전환이 비즈니스에 기여하는 정량적 지표 정의 | Strategy Map / BSC |
| 3단계 | 내부역량 분석 | 조직의 전략·시스템·구조·팀역량·스킬·스타일·공유가치 분석 | McKinsey 7S |
| 4단계 | 적용 전략 수립 | 폭포수 강점과 애자일 기회요인을 매핑한 하이브리드 전략 도출 | SWOT / 4C |
| 5단계 | 실행 및 평가 | 이터레이션 단위 재무적·품질적 평가 및 피드백 루프 완성 | IT-BSC / IT-ROI |

- 5단계 전략 수립 과정은 McKinsey 7S·SWOT 등 다차원 분석 도구를 연계하여 조직 내부 역량과 외부 환경을 종합적으로 고려한 하이브리드 전략으로 완성해야 함

2) 애자일 적용 절차 (로드맵)

| 구분 | 단계 | 주요 활동 | 핵심 산출물 |
|------|------|----------|-----------|
| ① | 반복 계획 수립 | 비즈니스 목표 수립, PO 백로그 설명, 공수 산정, DoD 정의 | 스프린트 계획서, 작업 백로그 |
| ② | 정밀 설계 | UI/UX 정의, 컴포넌트 인터페이스·ERD 설계, 테스트케이스 매핑 | 화면·프로그램 설계서, ERD |
| ③ | 증분 개발 | TDD 기반 구현, Git PR 검토, CI 점진적 모듈 통합 | 소스코드, 단위 테스트 결과서 |
| ④ | 검증·테스트 | PO·QA 주도 자동화 인수 테스트 가동 | 인수 테스트 결과서, 결함 추적 매트릭스 |
| ⑤ | 검토·회고 | 스프린트 Demo 시연·평가, Lessons Learned 도출, 백로그 업데이트 | 회고 결과서, 차기 개선 과제 정의서 |

- 전체 SDLC 뼈대(분석-설계-개발-테스트-전개) 위에 하위 이터레이션 루프를 유기적으로 동기화하는 것이 핵심.  "끝"

### 실무 제언

**대형 애자일 확장 프레임워크(Scaled Agile)의 전략적 채택**
- **챌린지**: 수십 개 팀이 동시에 기민하게 움직이기 위해서는 개별 팀 마일스톤을 전사 포트폴리오 수준으로 동기화하는 확장 아키텍처 모델이 요구됨
- **제언**: 조직 규모·통제 성향에 맞춰 SAFe(전사 포트폴리오~릴리즈 트레인의 엄격한 거버넌스), LeSS(계층 최소화·린 사상 강조), DaD(상황별 하이브리드 툴킷) 중 적합한 모델을 테일러링 기준으로 선제 선택해야 함

**'Water-Scrum-Fall' 모순 탈피와 서번트 리더십 전환**
- **챌린지**: 공공·금융 대형 SI 사업은 폭포수식 고정 예산·납기 계약 체계를 유지하면서 수행팀에게만 애자일을 강요하는 Water-Scrum-Fall 구조 모순이 이중 산출물 부담을 가중함
- **제언**: CTO·아키텍트는 Command & Control 관리에서 벗어나 팀원의 자율 해결과 변화 대응을 지원하는 서번트 리더십(Servant Leadership)으로 전환하고, 인프라 테스트·데이터 이관 등 병목 영역은 플랫폼 엔지니어링 팀을 전담 편성하여 스크럼 팀이 구현에만 집중할 수 있는 환경을 구축해야 함

**대규모 애자일 전환 시 조직 변화관리(OCM) 병행 추진**
- **챌린지**: 기술적 프레임워크 도입에만 집중하고 조직 문화·구성원 인식 변화를 소홀히 하면, 형식적 스크럼 세레모니만 남고 실질적 자기조직화 팀 운영이 실패하는 'Fake Agile' 현상이 발생함
- **제언**: 애자일 전환 프로젝트에는 반드시 조직 변화관리(OCM: Organizational Change Management) 계획을 병행하여, 리더십 인식 개선 워크숍, 애자일 코치 배치, 성공 사례 공유 체계를 구조적으로 운영해야 함
