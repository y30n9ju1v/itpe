## 문제

테스트 설계를 위한 기초 자료가 되는 테스트 케이스의 도출은 테스트에서 중요한 활동 중의 하나이다. **소프트웨어 테스트 용이성(Testability)**을 설명하고, **IEEE829 기반의 테스트케이스(Test Case) 구성요소**의 테스트케이스 설계 기법의 유형에 대하여 설명하시오.

## 해설

> **테스트 케이스는 테스트의 '원자'이자 품질의 '계약서'다**  
> IEEE 829는 테스트 문서화의 국제 표준을, 테스트 용이성은 테스트 가능한 소프트웨어를 만드는 설계 원칙을 정의한다

"테스트는 개발 완료 후 하는 것"이라는 생각이 틀린 이유가 있습니다. 소프트웨어를 테스트하기 어렵게 설계하면, 아무리 훌륭한 테스터도 결함을 찾지 못합니다. 마치 뚜껑이 용접된 기계를 고장 진단하려는 것과 같습니다. 테스트 용이성(Testability, 소프트웨어가 결함을 발견하기 쉽도록 설계된 정도)은 소프트웨어의 설계 품질 속성 중 하나로, 개발 초기부터 고려해야 합니다. IEEE 829는 이 테스트 활동을 문서로 표준화하여 체계적이고 추적 가능한 테스트를 가능하게 합니다.

### 1. 출제 배경 및 의도

이 문제는 **"테스트 케이스가 어떻게 도출되고 구성되어야 하는가"**에 대한 표준과 설계 원칙을 묻는 심화 문제입니다. 기술사 시험에서 반복 출제되는 이유는, 많은 SW 프로젝트에서 테스트가 체계 없이 임기응변으로 수행되어 테스트 결과의 재현성과 추적성이 보장되지 않는 현실을 반영합니다.

채점관이 기대하는 핵심은 세 가지입니다: (1) **테스트 용이성(Testability) 특성** 설명, (2) **IEEE 829 기반 테스트케이스 구성 요소**, (3) **화이트박스/블랙박스 테스트 설계 기법 비교**. 이 세 가지를 유기적으로 연결하여 "테스트 가능한 소프트웨어를 어떻게 설계하고 테스트하는가"라는 큰 그림을 그려야 합니다.

```text
  └──────────────────┴───────────────────┘
```

### 3. 핵심 키워드 (Must-Have)

#### 가. 소프트웨어 테스트 용이성 (Testability)

**테스트 용이성이란** 소프트웨어가 테스트 케이스를 통해 결함을 발견할 수 있도록 지원하는 정도를 나타내는 품질 속성입니다. ISO/IEC 25010(SQuaRE)의 유지보수성(Maintainability) 하위 특성 중 하나입니다.

**테스트 용이성의 핵심 특성 (RUCC + OC)**:

| 특성 | 정의 | 낮을 때 문제 | 개선 방법 |
|------|------|-----------|---------|
| **관찰가능성 (Observability)** | 내부 상태·결과를 외부에서 확인 가능한 정도 | 결함이 발생해도 증상이 보이지 않음 | 로깅, 모니터링 포인트, getter 메서드 |
| **제어가능성 (Controllability)** | 시스템 상태와 입력을 테스트 목적으로 제어 가능한 정도 | 특정 상태를 만들기 위해 복잡한 사전 조건 필요 | 의존성 주입(DI), Mock 객체 |
| **분리가능성 (Separability)** | 컴포넌트를 독립적으로 테스트할 수 있는 정도 | 하나를 테스트하려면 전체 시스템 필요 | SRP 준수, 인터페이스 분리 |
| **이해가능성 (Understandability)** | 테스트 대상 컴포넌트가 문서화되어 이해하기 쉬운 정도 | 테스트 케이스 설계 자체가 어려움 | API 문서, 코드 주석, 명확한 명명 |
| **자동화가능성 (Automatability)** | 자동화 도구로 테스트를 수행할 수 있는 정도 | 수동 테스트만 가능 → CI/CD 불가 | 표준 인터페이스, REST API 설계 |
| **헤타로지니티 방지 (Heterogeneity)** | 다양한 기술·플랫폼에서의 일관된 테스트 가능성 | 환경마다 다른 테스트 결과 | Docker 컨테이너화, 환경 표준화 |

**테스트 용이성을 높이는 설계 원칙**:

```text
[테스트 용이성 높은 설계 vs 낮은 설계]

  낮은 테스트 용이성:
  public class OrderService {
    public void createOrder() {
      // DB에 직접 연결 (제어 불가)
      Connection conn = DriverManager.getConnection("jdbc:...");
      // 현재 시간 직접 사용 (제어 불가)
      Date now = new Date();
      // 외부 결제 API 직접 호출 (관찰 불가)
      PaymentAPI.charge(now, ...);
    }
  }

  높은 테스트 용이성:
  public class OrderService {
    // 의존성 주입 (DI) → 제어가능성 확보
    public OrderService(Repository repo, Clock clock, PaymentGateway pg)
    
    public void createOrder() {
      // Mock 주입 가능 → 독립 테스트 가능
      // Clock.now() 제어 가능 → 시간 의존 테스트 가능
    }
  }
```

#### 나. IEEE 829 기반 테스트케이스 구성 요소

**IEEE 829란** 소프트웨어 테스트 문서의 구성 요소와 형식을 정의하는 국제 표준(IEEE Standard for Software and System Test Documentation)입니다.

**IEEE 829 테스트 문서 체계**:

```text
[IEEE 829 문서 계층]

  테스트 계획서 (Test Plan)
  ├── 테스트 범위, 접근 방법, 자원, 일정, 위험 정의
  │
  테스트 설계 명세서 (Test Design Specification)
  ├── 테스트 조건, 테스트 케이스 식별, 합격 기준 정의
  │
  테스트 케이스 명세서 (Test Case Specification)
  ├── 개별 테스트 케이스의 상세 입력/출력/절차
  │
  테스트 절차 명세서 (Test Procedure Specification)
  ├── 테스트 케이스 실행 순서와 환경 설정 절차
  │
  테스트 항목 전달 보고서 (Test Item Transmittal Report)
  ├── 테스트 대상 소프트웨어 버전 정보
  │
  테스트 로그 (Test Log)
  ├── 실제 테스트 수행 이력 기록
  │
  테스트 인시던트 보고서 (Test Incident Report)
  ├── 예상과 다른 결과 발생 시 기록 (버그 리포트)
  │
  테스트 요약 보고서 (Test Summary Report)
  └── 테스트 활동 전체 요약, 품질 평가
```

**테스트 케이스 구성 요소 (IEEE 829)**:

| 구성 요소 | 설명 | 예시 |
|---------|------|------|
| **테스트 케이스 ID** | 유일 식별자 | TC-LOGIN-001 |
| **테스트 항목** | 테스트 대상 | 로그인 기능, LoginService.authenticate() |
| **입력 명세** | 테스트 입력 데이터 | 아이디: "user01", 비밀번호: "Pass123!" |
| **출력 명세** | 기대 결과 | HTTP 200, JWT 토큰 반환, 세션 생성 |
| **환경 요구사항** | 테스트 수행 환경 | JDK 17, MySQL 8.0, 테스트 서버 |
| **특수 절차 요구사항** | 사전 조건, 설정 | 테스트 사용자 DB 사전 등록 필요 |
| **의존성** | 선행 테스트 케이스 | TC-USER-REGISTER-001 수행 후 실행 |

#### 다. 테스트 케이스 설계 기법 유형

**화이트박스 테스트 (구조 기반 / White-box Testing)**:

```text
[화이트박스 테스트 커버리지 단계]

  구문 커버리지 (Statement Coverage)
  - 모든 실행 문장을 최소 1회 실행
  - 가장 기본적인 커버리지 (권장 최소: 100%)

  분기 커버리지 (Branch Coverage)
  - 모든 분기(if/else, switch)의 참/거짓 모두 커버
  - 구문 커버리지 포함

  조건 커버리지 (Condition Coverage)
  - 각 조건식의 참/거짓 모두 커버
  - if (A && B): A=T/F, B=T/F 각각 커버

  MC/DC (Modified Condition/Decision Coverage)
  - 항공 소프트웨어 표준(DO-178C)에서 요구
  - 각 조건이 독립적으로 결정에 영향을 줌을 증명
```

**블랙박스 테스트 (명세 기반 / Black-box Testing)**:

| 기법 | 설명 | 예시 |
|------|------|------|
| **동등 분할 (Equivalence Partitioning)** | 유사한 입력을 그룹으로 묶어 대표값 1개만 테스트 | 나이 0~150: [유효 그룹], [-∞~-1]: [무효], [151~∞]: [무효] |
| **경계값 분석 (Boundary Value Analysis)** | 등가 분할 경계의 안팎 값 집중 테스트 | 0, 1, 149, 150, -1, 151 |
| **결정 테이블 (Decision Table)** | 조건 조합에 따른 행동 정의 | 할인: [회원여부 × 주문금액] 조합표 |
| **상태 전이 테스트 (State Transition)** | 상태 변화에 따른 동작 검증 | 주문 상태: 결제대기→결제완료→배송중→배송완료 |
| **유스케이스 테스트 (Use Case Testing)** | 유스케이스의 시나리오별 테스트 | UC-ORDER-001: 정상 주문 시나리오 |

**경험 기반 테스트**:
- **오류 추측 (Error Guessing)**: 경험에 기반하여 발생 가능한 결함 예측
- **탐색적 테스팅 (Exploratory Testing)**: 동시에 테스트를 설계하고 실행하는 비형식적 기법

### 4. 맥락과 비교

| 구분 | 화이트박스 테스트 | 블랙박스 테스트 | 그레이박스 테스트 |
|------|--------------|-------------|--------------|
| 지식 수준 | 내부 구조(코드) 알아야 함 | 내부 구조 몰라도 됨 | 부분적 내부 지식 활용 |
| 테스터 | 주로 개발자 | 테스터, 최종 사용자 | 개발자 + 테스터 협업 |
| 커버리지 기준 | 코드 커버리지 (구문/분기) | 요구사항 커버리지 | 혼합 |
| 발견 결함 유형 | 논리 오류, 경로 오류 | 기능 오류, 인터페이스 오류 | 통합 오류 |
| 자동화 | 단위 테스트 (JUnit, NUnit) | API 테스트 (Postman, RestAssured) | 혼합 |
| 적용 레벨 | 단위·통합 테스트 | 시스템·인수 테스트 | 통합·시스템 테스트 |

### 5. 실무 제언

**테스트 용이성과 마이크로서비스 아키텍처**

- **챌린지**: 마이크로서비스(MSA) 환경에서는 각 서비스를 독립적으로 테스트(단위 테스트)하기는 쉽지만, 서비스 간 통합 테스트는 매우 어렵습니다. 실제 다른 서비스를 모두 띄워야 하는 E2E 테스트는 환경 구성만 수 시간이 걸리는 경우도 있습니다.
- **제언**: Consumer-Driven Contract Testing(CDCT) 기법을 활용해야 합니다. Pact 라이브러리를 이용하면 소비자(Consumer) 서비스가 공급자(Provider) 서비스와의 계약(Contract)을 명세하고, 공급자는 이 계약을 단위 테스트 수준에서 검증할 수 있습니다. 이를 통해 무거운 통합 환경 없이도 서비스 간 인터페이스 계약을 지속적으로 검증하는 테스트 용이성 높은 MSA 테스트 전략을 구현할 수 있습니다.

**IEEE 829의 현대화: 경량 문서화**

- **챌린지**: IEEE 829를 완전히 준수하면 테스트 계획서, 설계서, 케이스 명세서, 절차서, 보고서 등 방대한 문서가 필요합니다. 애자일 환경에서는 이러한 문서를 매 스프린트마다 갱신하는 것이 현실적으로 불가능하여 테스트 문서와 실제 테스트 케이스가 동기화되지 않는 문제가 발생합니다.
- **제언**: Cucumber/JUnit과 같은 도구로 **실행 가능한 명세(Executable Specification)**를 작성하는 것이 IEEE 829의 정신을 현대적으로 구현하는 방법입니다. 코드로 작성된 테스트 케이스가 곧 테스트 문서가 되어 항상 최신 상태를 유지하고, CI/CD에서 자동 실행됩니다. Allure, Extent Reports 같은 보고서 도구를 연계하면 IEEE 829의 테스트 요약 보고서도 자동 생성됩니다.
