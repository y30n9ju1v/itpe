## 문제

테스트 설계를 위한 기초 자료가 되는 테스트 케이스의 도출은 테스트에서 중요한 활동 중의 하나이다. **소프트웨어 테스트 용이성(Testability)**을 설명하고, **IEEE829 기반의 테스트케이스(Test Case) 구성요소**의 테스트케이스 설계 기법의 유형에 대하여 설명하시오.

## 해설

### 출제 배경 및 의도

이 문제는 "테스트 케이스가 어떻게 도출되고 구성되어야 하는가"에 대한 표준과 설계 원칙을 묻는 심화 문제다. 많은 SW 프로젝트에서 테스트가 임기응변으로 수행되어 결과의 재현성과 추적성이 보장되지 않는 현실을 반영한다.

채점관이 기대하는 핵심은 세 가지다: ①테스트 용이성(Testability) 특성 설명, ②IEEE 829 기반 테스트케이스 구성 요소, ③화이트박스/블랙박스 테스트 설계 기법 비교. 이 세 가지를 유기적으로 연결하여 "테스트 가능한 소프트웨어를 어떻게 설계하고 테스트하는가"라는 큰 그림을 그려야 한다.

### 1. 테스트 용이성·IEEE 829·설계 기법 연결 구조

정 의  • 테스트 용이성(Testability): 소프트웨어가 테스트 케이스를 통해 결함을
       발견할 수 있도록 지원하는 정도를 나타내는 품질 속성
       - ISO/IEC 25010(SQuaRE) 유지보수성의 하위 특성

```
[테스트 용이성 → IEEE 829 → 설계 기법의 연결]

  소프트웨어 설계 단계
  ┌──────────────────────────────────────┐
  │  테스트 용이성(Testability) 고려      │
  │  - 관찰가능성: 내부 상태 외부 확인    │
  │  - 제어가능성: 입력 자유 주입 가능    │
  └──────────────────────────────────────┘
               │
               ▼
  테스트 계획·설계 단계 (IEEE 829 문서 체계)
  ┌──────────────────────────────────────┐
  │  계획서 → 설계 명세 → 케이스 명세    │
  │  → 절차 명세 → 결과 보고서           │
  └──────────────────────────────────────┘
               │
               ▼
  테스트 케이스 설계 기법 선택
  ┌─────────────────┬───────────────────┐
  │   화이트박스     │    블랙박스        │
  │  구문/분기/MC/DC │  동등분할/경계값  │
  └─────────────────┴───────────────────┘
```

- 테스트 용이성은 소프트웨어 설계 단계에서 결정되며, IEEE 829는 테스트 활동의 표준 문서 체계를 제공한다.

### 2. 테스트 용이성 및 IEEE 829 상세

1) 소프트웨어 테스트 용이성 (Testability)

| 특성 | 정의 | 낮을 때 문제 | 개선 방법 |
|------|------|-----------|---------|
| 관찰가능성(Observability) | 내부 상태·결과를 외부에서 확인 가능한 정도 | 결함 발생해도 증상 미확인 | 로깅, 모니터링 포인트, getter |
| 제어가능성(Controllability) | 상태와 입력을 테스트 목적으로 제어 가능한 정도 | 복잡한 사전 조건 필요 | 의존성 주입(DI), Mock 객체 |
| 분리가능성(Separability) | 컴포넌트를 독립적으로 테스트할 수 있는 정도 | 전체 시스템 필요 | SRP 준수, 인터페이스 분리 |
| 이해가능성(Understandability) | 테스트 대상이 문서화되어 이해하기 쉬운 정도 | 케이스 설계 자체가 어려움 | API 문서, 명확한 명명 |
| 자동화가능성(Automatability) | 자동화 도구로 테스트를 수행할 수 있는 정도 | 수동 테스트만 가능 | 표준 인터페이스, REST API |

- 테스트 용이성은 의존성 주입(DI)과 단일 책임 원칙(SRP) 준수만으로도 크게 향상된다.

2) IEEE 829 테스트 문서 체계 및 테스트케이스 구성 요소

```
[IEEE 829 문서 계층]

  테스트 계획서(Test Plan)          ← 범위·접근방법·자원·일정·위험
  테스트 설계 명세서                ← 테스트 조건·케이스 식별·합격 기준
  테스트 케이스 명세서              ← 상세 입력/출력/절차
  테스트 절차 명세서                ← 실행 순서·환경 설정 절차
  테스트 항목 전달 보고서           ← 대상 SW 버전 정보
  테스트 로그                      ← 실제 수행 이력
  테스트 인시던트 보고서            ← 예상과 다른 결과(버그 리포트)
  테스트 요약 보고서                ← 전체 활동 요약·품질 평가
```

| 구성 요소 | 설명 | 예시 |
|---------|------|------|
| 테스트 케이스 ID | 유일 식별자 | TC-LOGIN-001 |
| 테스트 항목 | 테스트 대상 | 로그인 기능, LoginService.authenticate() |
| 입력 명세 | 테스트 입력 데이터 | 아이디: "user01", 비밀번호: "Pass123!" |
| 출력 명세 | 기대 결과 | HTTP 200, JWT 토큰 반환, 세션 생성 |
| 환경 요구사항 | 테스트 수행 환경 | JDK 17, MySQL 8.0, 테스트 서버 |
| 특수 절차 요구사항 | 사전 조건·설정 | 테스트 사용자 DB 사전 등록 필요 |
| 의존성 | 선행 테스트 케이스 | TC-USER-REGISTER-001 수행 후 실행 |

- IEEE 829는 테스트 활동의 추적성(Traceability)을 보장하는 국제 표준이다.

### 3. 테스트 케이스 설계 기법 비교 및 트렌드

1) 화이트박스 vs 블랙박스 vs 그레이박스 비교

| 구분 | 화이트박스 | 블랙박스 | 그레이박스 |
|------|---------|---------|---------|
| 지식 수준 | 내부 코드 알아야 함 | 내부 구조 몰라도 됨 | 부분적 내부 지식 |
| 테스터 | 개발자 | QA 테스터 | 개발자+테스터 협업 |
| 커버리지 기준 | 코드 커버리지 (구문/분기) | 요구사항 커버리지 | 혼합 |
| 발견 결함 | 논리 오류, 경로 오류 | 기능 오류, 인터페이스 오류 | 통합 오류 |
| 대표 기법 | 구문/분기/조건/MC/DC | 동등분할/경계값/결정테이블 | 혼합 |

2) 블랙박스 주요 기법

| 기법 | 설명 | 예시 |
|------|------|------|
| 동등 분할(Equivalence Partitioning) | 유사 입력을 그룹으로 묶어 대표값 1개 테스트 | 나이 0~150: 유효/무효 파티션 분리 |
| 경계값 분석(BVA) | 경계의 안팎 값 집중 테스트 | 0, 1, 149, 150, -1, 151 |
| 결정 테이블(Decision Table) | 조건 조합별 행동 정의 | [회원여부×주문금액] 할인 조합표 |
| 상태 전이 테스트 | 상태 변화에 따른 동작 검증 | 주문: 결제대기→결제완료→배송중→완료 |

- 기법 선택은 시스템 특성(상태 기반·규칙 기반·데이터 집약)에 따라 결정하며, 복수 기법을 조합하면 결함 발견율이 높아진다.  "끝"

### 실무 제언

**테스트 용이성과 마이크로서비스 아키텍처**
- **챌린지**: MSA 환경에서 각 서비스를 독립 단위 테스트하기는 쉽지만, 서비스 간 통합 테스트는 실제 서비스를 모두 띄워야 하는 E2E 테스트로 환경 구성만 수 시간이 걸린다.
- **제언**: Consumer-Driven Contract Testing(CDCT)을 활용한다. Pact 라이브러리를 이용하면 소비자 서비스가 공급자 서비스와의 계약(Contract)을 명세하고, 공급자는 이를 단위 테스트 수준에서 검증할 수 있어 무거운 통합 환경 없이도 인터페이스 계약을 지속적으로 검증할 수 있다.

**IEEE 829의 현대화: 경량 문서화**
- **챌린지**: IEEE 829 완전 준수 시 방대한 문서가 필요하며, 애자일 환경에서 매 스프린트마다 갱신하는 것이 현실적으로 불가능하다.
- **제언**: Cucumber/JUnit으로 실행 가능한 명세(Executable Specification)를 작성하면 테스트 코드가 곧 최신 문서가 된다. Allure, Extent Reports 연계 시 IEEE 829의 테스트 요약 보고서도 자동 생성되어 IEEE 829의 정신을 현대적으로 구현할 수 있다.
