## 해설

> **"아키텍처를 기술하는 아키텍처" — 소프트웨어 아키텍처 표준**  
> ISO/IEC/IEEE 42010은 소프트웨어 아키텍처를 문서화하고 표현하는 국제 표준 프레임워크다

거대한 소프트웨어 시스템에서 개발자는 코드를, 운영자는 배포 환경을, 보안 담당자는 취약점을 각각 다른 시각으로 바라봅니다. 이들 모두가 "같은 시스템을 올바르게 이해"하려면 공통의 기술(記述) 언어가 필요합니다. ISO/IEC/IEEE 42010은 바로 그 공통 언어의 문법을 정의하는 표준입니다.

### 1. 출제 배경 및 의도

소프트웨어 아키텍처(SA)는 시스템의 품질 속성을 결정하는 핵심 설계 결정의 집합입니다. 그런데 아키텍처를 어떻게 기술할 것인지에 대한 표준이 없다면, 조직마다 제각각의 방식으로 문서를 작성하여 소통이 불가능해집니다. ISO/IEC/IEEE 42010(구: IEEE 1471)은 이 문제를 해결하기 위한 국제 표준으로, 아키텍처 기술(Architecture Description)의 핵심 개념과 관계를 정의합니다.

기술사 시험에서는 이 표준의 핵심 개념들—특히 **관점(Viewpoint), 뷰(View), 이해관계자(Stakeholder), 관심사(Concern)**—을 정확히 구분하고, 이들의 상호 관계를 다이어그램으로 표현하는 능력을 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

**ISO/IEC/IEEE 42010이란** 소프트웨어 집약적 시스템(Software-Intensive System)의 아키텍처를 표현(Expression), 기록(Recording), 전달(Communication)하기 위한 개념, 용어, 실천 방법을 정의한 국제 표준입니다.

```text
[ISO/IEC/IEEE 42010 핵심 개념 모델]

  이해관계자          관심사(Concern)
  (Stakeholder)  ────────────────────────┐
       │              "성능은 충분한가?"  │
       │              "보안 취약점은?"    │
       │              "확장은 쉬운가?"    │
       │                                │
       ▼                                ▼
  아키텍처 기술 (Architecture Description: AD)
       │
  ┌────┴────────────────────────────────────┐
  │                                         │
  ▼                                         ▼
관점(Viewpoint)                           뷰(View)
"어떤 시각으로 볼 것인가?"              "그 시각으로 본 결과물"
 - 논리 관점(Logical VP)                - 논리 뷰(Logical View)
 - 프로세스 관점(Process VP)            - 프로세스 뷰(Process View)
 - 배포 관점(Deployment VP)             - 배포 뷰(Deployment View)
 - 개발 관점(Development VP)            - 개발 뷰(Dev View)
       │                                    │
       └─────────── 관점을 적용하여 ─────────┘
                     뷰를 생성

  ▼
아키텍처 모델 (Architecture Model)
  - 각 뷰를 구성하는 다이어그램, 표, 텍스트
  - UML, SysML, ArchiMate 등 표기법 사용
```

### 3. 핵심 키워드 (Must-Have)

#### 1) 핵심 구성 요소 정의

| 개념 | 정의 | 예시 |
|------|------|------|
| **이해관계자(Stakeholder)** | 시스템에 이해관계를 갖는 개인 또는 조직 | 발주자, 개발자, 운영자, 보안 감사자, 최종 사용자 |
| **관심사(Concern)** | 이해관계자가 시스템에 대해 가지는 관심·우려·기대 | "응답시간은 3초 이내여야 한다", "재해복구 계획이 있는가?" |
| **아키텍처 기술(AD)** | 이해관계자의 관심사를 다루기 위해 아키텍처를 표현한 작업 산출물 | 아키텍처 설계서, 아키텍처 기술 문서 |
| **관점(Viewpoint)** | 특정 관심사를 다루기 위한 뷰 생성의 규칙·관례·언어 집합 (템플릿) | "논리 관점 = UML 클래스/시퀀스 다이어그램으로 표현" |
| **뷰(View)** | 특정 관점을 시스템에 적용하여 생성된 실제 표현물 | 클래스 다이어그램, 배포 다이어그램 |
| **아키텍처 모델(Model)** | 뷰를 구성하는 개별 표현 요소 | 특정 컴포넌트 다이어그램 하나 |

**관점(Viewpoint) vs 뷰(View) 구분 — 시험 핵심**

```text
관점(Viewpoint) = 렌즈(Lens), 필터, 청사진의 양식
  → "논리 관점으로 보면: UML 컴포넌트 다이어그램을 사용하고,
     시스템의 기능적 구조와 컴포넌트 인터페이스를 표현한다"

뷰(View) = 렌즈를 통해 실제로 촬영된 사진
  → "주문 시스템의 논리 뷰: [OrderService] ─ [PaymentGateway]
     [InventoryService] 컴포넌트 다이어그램"

비유: 건축에서
  관점 = 평면도 그리는 방법(표기 규칙, 척도)
  뷰   = 실제로 그린 이 건물의 평면도
```

#### 2) 4+1 뷰 모델 (Kruchten, 42010 적용 사례)

Philippe Kruchten이 제안한 4+1 뷰 모델은 ISO 42010 개념의 대표적 실천 사례입니다:

```text
[4+1 뷰 모델]

       [논리 뷰]           [개발 뷰]
  기능적 구조 표현         소스코드 구조
  UML 클래스/객체         패키지/모듈 계층
         │                      │
         └──────┐  ┌────────────┘
                ▼  ▼
           [유스케이스 뷰] ← 중심(+1)
           핵심 요구사항·시나리오
           아키텍처 검증의 기준
                ▲  ▲
         ┌──────┘  └────────────┐
         │                      │
  [프로세스 뷰]          [물리 뷰 (배포 뷰)]
  런타임 동작·프로세스    하드웨어·네트워크 배포
  동시성·스레드 관리      서버·클라우드 구성
```

| 뷰 | 관심사 | 이해관계자 | 주요 표기법 |
|----|-------|----------|-----------|
| 논리 뷰(Logical View) | 기능 분해, 컴포넌트 구조 | 개발자, 설계자 | UML 클래스, 컴포넌트 |
| 프로세스 뷰(Process View) | 동시성, 통신, 성능 | 성능 엔지니어, 시스템 통합자 | UML 활동, 시퀀스 |
| 개발 뷰(Development View) | 소프트웨어 계층, 모듈 | 개발 관리자, 프로그래머 | UML 패키지 |
| 물리 뷰(Physical View) | 배포 토폴로지, 인프라 | 운영자, 시스템 관리자 | UML 배포 다이어그램 |
| 유스케이스 뷰(+1) | 핵심 시나리오, 검증 기준 | 모든 이해관계자 | UML 유스케이스 |

#### 3) 아키텍처 결정(Architectural Decision)

ISO 42010은 아키텍처를 단순한 구조도가 아닌 **설계 결정의 집합**으로 정의합니다:

- **아키텍처 결정**: "왜 이 구조를 선택했는가?"에 대한 근거(Rationale)와 트레이드오프를 포함한 결정
- **ADR (Architecture Decision Record)**: 아키텍처 결정을 기록하는 경량 문서 형식
  - 상태(Status), 문맥(Context), 결정(Decision), 결과(Consequences)

### 4. 맥락과 비교

| 구분 | IEEE 1471 (구) | ISO/IEC/IEEE 42010:2011 | TOGAF | Zachman Framework |
|------|---------------|------------------------|-------|-------------------|
| 초점 | SW 집약 시스템 아키텍처 기술 | 동일 + 모델·결정 개념 추가 | 엔터프라이즈 아키텍처 프레임워크 | EA 분류 체계 |
| 산출물 | 뷰포인트, 뷰 | 뷰포인트, 뷰, AD, 아키텍처 결정 | ADM 단계별 산출물 | 6×6 셀 매트릭스 |
| 대상 | 소프트웨어 시스템 | 소프트웨어+시스템 전반 | 기업 전체 IT | 기업 전체 아키텍처 |
| 적용 도구 | UML | UML, SysML, ArchiMate | ArchiMate, Sparx EA | Sparx EA, Avolution |

### 5. 실무 제언

**아키텍처 기술 문서의 형식적 완성도 vs 활용도 딜레마**

- **챌린지**: ISO 42010을 엄격히 준수한 아키텍처 기술 문서는 수십~수백 페이지에 달하여 실무에서 지속적으로 갱신하고 활용하기 어렵습니다. 아키텍처 문서는 종종 초기 설계 시에만 작성되고 시스템이 발전하면 금세 구식 문서가 되어버립니다(Documentation Drift).
- **제언**: ADR(Architecture Decision Record) 형식을 Git 저장소에 텍스트 파일로 관리하는 **경량 아키텍처 거버넌스**를 도입해야 합니다. 또한 C4 모델(Context-Container-Component-Code)처럼 수준별로 계층화된 다이어그램을 PlantUML, Mermaid 등의 코드 기반 다이어그램 도구로 자동 생성하여 코드와 함께 버전 관리하는 것이 현대적 실무 접근법입니다.

**이해관계자별 맞춤형 뷰 제공**

- **챌린지**: 동일한 시스템을 CTO, 개발자, 운영자, 감사자에게 동일한 형식의 문서로 설명하면 효과가 없습니다. 기술적 세부사항에 묻힌 CTO는 핵심 의사결정을 내리기 어렵고, 추상적인 다이어그램만 본 개발자는 구현 방향을 잡지 못합니다.
- **제언**: ISO 42010의 관점(Viewpoint) 개념을 적극 활용하여 이해관계자별로 **맞춤형 뷰(Tailored View)**를 제공해야 합니다. C-Level에게는 컨텍스트 다이어그램(Context Diagram)으로 시스템 경계와 외부 연계만 보여주고, 개발팀에게는 컴포넌트-커넥터 다이어그램으로 내부 구조를 제공하는 계층적 커뮤니케이션 전략이 효과적입니다.
