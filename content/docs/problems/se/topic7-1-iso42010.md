## 해설

### 출제 배경 및 의도

소프트웨어 아키텍처(SA)는 시스템의 품질 속성을 결정하는 핵심 설계 결정의 집합이다. 같은 건물을 두고 시공자·전기 기사·인테리어 디자이너가 각각 다른 도면을 보듯, 개발자·운영자·보안 담당자가 시스템을 각기 다른 시각으로 바라본다. 아키텍처 기술(記述) 표준이 없으면 조직마다 제각각의 문서를 작성하여 소통이 불가능해진다.

기술사 시험에서는 표준의 핵심 개념인 관점(Viewpoint)·뷰(View)·이해관계자(Stakeholder)·관심사(Concern)를 정확히 구분하고, 상호 관계를 구조도로 표현하는 능력을 검증한다.

### 1. ISO/IEC/IEEE 42010 개요

정 의  • 소프트웨어 집약적 시스템(Software-Intensive System)의 아키텍처를 표현·기록·전달하기 위한 개념·용어·실천 방법을 정의한 국제 표준 (구 IEEE 1471 대체)
       - "아키텍처를 기술하는 아키텍처" — 이해관계자별 공통 기술 언어의 문법 제공

```
[ISO/IEC/IEEE 42010 핵심 개념 모델]

  이해관계자(Stakeholder) ──▶ 관심사(Concern)
                                "성능은 충분한가?"
                                "보안 취약점은?"
                                    │
                                    ▼
            아키텍처 기술(Architecture Description: AD)
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
  관점(Viewpoint)           뷰(View)
  "어떤 시각으로 볼까?"     "그 시각으로 본 결과물"
  (렌즈 = 템플릿)           (렌즈로 찍은 사진)
          │                   │
          └────── 적용 ────────┘
                    │
                    ▼
           아키텍처 모델(Model)
           UML/SysML/ArchiMate 등 표기
```

- 관점(Viewpoint)은 '평면도 그리는 규칙', 뷰(View)는 '실제로 그린 이 건물의 평면도'로 구분한다

### 2. 핵심 구성 요소 및 4+1 뷰 모델

1) 6대 핵심 구성 요소

| 구분 | 개념 | 정의 | 예시 |
|------|------|------|------|
| ① | 이해관계자(Stakeholder) | 시스템에 이해관계를 갖는 개인·조직 | 발주자·개발자·운영자·보안 감사자 |
| ② | 관심사(Concern) | 이해관계자가 가지는 관심·우려·기대 | "응답시간 3초 이내", "재해복구 계획?" |
| ③ | 아키텍처 기술(AD) | 관심사를 다루기 위해 아키텍처를 표현한 산출물 | 아키텍처 설계서 |
| ④ | 관점(Viewpoint) | 뷰 생성의 규칙·관례·언어 집합 (템플릿) | "논리 관점 = UML 클래스 다이어그램" |
| ⑤ | 뷰(View) | 관점을 시스템에 적용하여 생성된 실제 표현물 | 배포 다이어그램, 클래스 다이어그램 |
| ⑥ | 아키텍처 모델(Model) | 뷰를 구성하는 개별 표현 요소 | 특정 컴포넌트 다이어그램 하나 |

2) 4+1 뷰 모델 (Kruchten — ISO 42010 대표 실천 사례)

```
[4+1 뷰 모델]

     [논리 뷰]              [개발 뷰]
  기능적 구조·UML 클래스   소스코드·패키지 계층
         │                      │
         └──────┐  ┌────────────┘
                ▼  ▼
           [유스케이스 뷰] ← 중심(+1)
           핵심 요구사항·시나리오·검증 기준
                ▲  ▲
         ┌──────┘  └────────────┐
         │                      │
  [프로세스 뷰]          [물리 뷰(배포 뷰)]
  런타임·동시성·성능      서버·클라우드 배포 토폴로지
```

| 구분 | 뷰 | 관심사 | 이해관계자 | 표기법 |
|------|---|------|----------|-------|
| 논리 | Logical View | 기능 분해·컴포넌트 구조 | 개발자·설계자 | UML 클래스·컴포넌트 |
| 프로세스 | Process View | 동시성·통신·성능 | 성능 엔지니어 | UML 활동·시퀀스 |
| 개발 | Development View | SW 계층·모듈 | 개발 관리자 | UML 패키지 |
| 물리 | Physical View | 배포 토폴로지·인프라 | 운영자 | UML 배포 다이어그램 |
| +1 | UseCase View | 핵심 시나리오·검증 기준 | 모든 이해관계자 | UML 유스케이스 |

### 3. 아키텍처 결정 및 관련 표준 비교

1) ADR(Architecture Decision Record) — 아키텍처 결정 기록

| 구분 | 항목 | 설명 |
|------|------|------|
| 개념 | 설계 결정의 집합 | ISO 42010은 아키텍처를 구조도가 아닌 설계 결정의 집합으로 정의 |
| ADR 구성 | Status·Context·Decision·Consequences | 왜 이 구조를 선택했는지 근거(Rationale)·트레이드오프 기록 |
| 관리 방식 | Git 텍스트 파일 | 코드와 함께 버전 관리 — Documentation Drift 방지 |

2) 관련 표준 비교

| 구분 | IEEE 1471 | ISO/IEC/IEEE 42010 | TOGAF | Zachman |
|------|-----------|-------------------|-------|---------|
| 초점 | SW 아키텍처 기술 | 동일 + 모델·결정 추가 | EA 프레임워크 | EA 분류 체계 |
| 대상 | 소프트웨어 시스템 | SW + 시스템 전반 | 기업 전체 IT | 기업 전체 아키텍처 |

- ISO 42010은 뷰포인트·뷰 외에 아키텍처 결정(Architectural Decision) 개념을 추가하여 설계 근거 추적성을 강화한다.  "끝"

### 실무 제언

**경량 아키텍처 거버넌스 — ADR + C4 모델**
- **챌린지**: ISO 42010 엄격 준수 시 수백 페이지 문서가 되어 갱신·활용이 어렵고 시스템 발전 후 구식 문서(Documentation Drift)로 전락한다
- **제언**: ADR을 Git 저장소 텍스트 파일로 관리하는 경량 거버넌스를 도입하고, C4 모델(Context-Container-Component-Code)을 PlantUML·Mermaid 코드 기반으로 자동 생성하여 코드와 함께 버전 관리한다

**이해관계자별 맞춤형 뷰 제공**
- **챌린지**: 동일 문서를 CTO·개발자·운영자·감사자에게 제공하면 기술 세부사항에 묻힌 CTO는 의사결정 불가, 추상 다이어그램만 본 개발자는 구현 방향 미확정 문제가 발생한다
- **제언**: ISO 42010의 관점(Viewpoint) 개념을 활용하여 C-Level에게는 컨텍스트 다이어그램(시스템 경계·외부 연계)만, 개발팀에게는 컴포넌트-커넥터 다이어그램(내부 구조)을 제공하는 계층적 커뮤니케이션 전략을 수립한다
