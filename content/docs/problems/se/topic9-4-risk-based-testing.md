## 해설

> **모든 것을 테스트할 수 없다면, 가장 위험한 것부터 테스트하라**  
> 리스크 기반 테스팅은 테스트 자원의 전략적 배분을 위한 의사결정 프레임워크다.

무한한 시간과 인력이 있다면 모든 기능을 완벽하게 테스트할 수 있겠지만, 현실 프로젝트에서 테스트 자원은 항상 부족합니다. 리스크 기반 테스팅(RBT, Risk-Based Testing)은 "어디에 테스트 자원을 집중해야 하는가"라는 질문에 대한 체계적인 답을 제시합니다. 결함이 발생했을 때 비즈니스에 가장 큰 피해를 줄 수 있는 영역을 먼저, 더 집중적으로 테스트하는 전략입니다.

### 1. 출제 배경 및 의도

리스크 기반 테스팅은 ISO/IEC/IEEE 29119 소프트웨어 테스팅 표준과 ISTQB(국제 소프트웨어 테스팅 자격 위원회) 실라버스에 공식적으로 포함된 주류 테스팅 접근법입니다. 기술사 채점관은 단순히 RBT의 정의를 외우는 것이 아니라, 리스크 식별 → 리스크 평가 → 우선순위 결정 → 테스트 배분이라는 **4단계 프로세스**를 실제 프로젝트에 적용하는 능력을 봅니다.

함속(함께 공부하는 속도전) 시험에서 출제될 정도로 기술사 시험에서 자주 등장하는 주제입니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 리스크(Risk)의 정의와 분류

테스팅에서의 **리스크(Risk)**는 "결함이 존재할 가능성(확률, Likelihood)"과 "결함이 발견되었을 때의 비즈니스 영향(Impact)"의 조합입니다.

```text
리스크 = 발생 확률(Likelihood) × 영향도(Impact)
```

**리스크의 분류**

| 분류 | 설명 | 예시 |
|------|------|------|
| 기능적 리스크 (Product Risk) | 소프트웨어 기능이 요구사항을 충족하지 못할 리스크 | 결제 금액 오류, 로그인 실패 |
| 비기능적 리스크 (Product Risk) | 성능, 보안, 가용성 등 품질 속성 관련 리스크 | 피크 타임 서버 다운, 개인정보 유출 |
| 프로젝트 리스크 (Project Risk) | 일정, 예산, 조직 관련 리스크 (테스트 활동 자체에 영향) | 개발 지연으로 인한 테스트 기간 단축 |

리스크 기반 테스팅은 주로 **제품 리스크(Product Risk)**를 다룹니다.

#### 나. RBT 수행 프로세스 4단계

```text
[리스크 기반 테스팅 프로세스]

  1단계: 리스크 식별 (Risk Identification)
     ↓
  브레인스토밍, 체크리스트, 전문가 인터뷰
  기능적 리스크 + 비기능적 리스크 열거

  2단계: 리스크 평가 (Risk Assessment)
     ↓
  각 리스크에 대해 확률(P)과 영향도(I) 평가
  리스크 레벨 = P × I (1~5 척도 또는 상/중/하)

  3단계: 리스크 우선순위화 (Risk Prioritization)
     ↓
  리스크 매트릭스로 High/Medium/Low 구분
  우선순위 높은 항목에 테스트 자원 집중

  4단계: 테스트 설계 및 실행 (Test Design & Execution)
     ↓
  High Risk → 상세 테스트 설계, 충분한 시간 배정
  Medium Risk → 표준 테스트
  Low Risk → 최소한의 테스트 또는 생략
```

#### 다. 리스크 매트릭스(Risk Matrix)

```text
[리스크 매트릭스]

영향도(Impact)
 높음 │  중간   │  높음   │  매우높음 │
      ├─────────┼─────────┼──────────┤
 중간 │  낮음   │  중간   │  높음    │
      ├─────────┼─────────┼──────────┤
 낮음 │  매우낮음│  낮음   │  중간    │
      └─────────┴─────────┴──────────┘
           낮음      중간      높음     발생 확률(Probability)
```

- **매우 높음(Critical)**: 가장 먼저, 가장 집중적으로 테스트. 결함 하나가 시스템 전체나 비즈니스에 치명적 영향
- **높음(High)**: 충분한 테스트 설계와 실행 필요
- **중간(Medium)**: 표준적 테스트
- **낮음(Low)**: 시간이 허용될 경우에만 테스트

### 3. 핵심 키워드 (Must-Have)

| 키워드 | 설명 |
|--------|------|
| 제품 리스크 (Product Risk) | 소프트웨어 자체의 결함에 관한 리스크 (기능적 + 비기능적) |
| 프로젝트 리스크 (Project Risk) | 테스트 활동에 영향을 미치는 관리적 리스크 |
| 리스크 레벨 = 확률 × 영향도 | 리스크 정량화의 핵심 공식 |
| 리스크 매트릭스 | 확률과 영향도 2축으로 리스크를 시각화하는 도구 |
| 잔존 리스크 (Residual Risk) | 테스트 수행 후에도 남아있는 수용된 리스크 |
| 리스크 모니터링 | 프로젝트 진행 중 리스크 수준의 변화를 지속 추적 |

### 4. 맥락과 비교

**RBT vs. 요구사항 기반 테스트**

| 구분 | 요구사항 기반 테스트 | 리스크 기반 테스팅 |
|------|--------------------|-----------------|
| 테스트 대상 선정 기준 | 요구사항 목록 (모든 요구사항 커버) | 리스크 수준 (높은 리스크 우선) |
| 자원 배분 | 균등 배분 | 리스크 비례 배분 |
| 장점 | 명확한 추적성 | 효율적 자원 사용 |
| 단점 | 위험한 기능에 충분한 집중 어려움 | 리스크 평가의 주관성 |
| 적용 상황 | 요구사항이 명확하고 충분한 시간이 있는 경우 | 테스트 시간/자원이 제한된 경우 |

**ISTQB 시험과의 연계**

RBT는 ISTQB FL(Foundation Level) 및 CTAL-TM(Test Manager) 자격 시험의 핵심 주제입니다. 정보처리기술사와 ISTQB 자격증을 함께 준비하는 경우 시너지 효과가 큽니다.

### 5. 실무 제언

**리스크 평가의 객관성 확보**

리스크 기반 테스팅의 가장 큰 도전은 리스크 평가의 주관성입니다. 테스트 매니저 한 사람의 판단에 의존하면 편향이 발생합니다. 이를 극복하기 위해:

- **다양한 이해관계자 참여**: 개발자, 비즈니스 분석가, 운영팀, 보안팀이 함께 참여하는 리스크 워크숍 진행
- **과거 결함 데이터 활용**: 이전 버전의 결함 이력을 분석하여 결함 발생 확률을 데이터 기반으로 추정
- **FMEA(Failure Mode and Effect Analysis)**: 각 기능의 잠재적 실패 모드와 영향을 체계적으로 분석하는 기법을 병행

**리스크는 동적으로 변화한다**

프로젝트 초기에 설정한 리스크 우선순위는 프로젝트 진행 중 변화합니다. 개발 완료 상황, 추가 요구사항, 외부 환경 변화(예: 보안 취약점 발견)에 따라 리스크를 재평가하고 테스트 우선순위를 동적으로 조정하는 **지속적 리스크 모니터링** 체계를 갖추어야 합니다.
