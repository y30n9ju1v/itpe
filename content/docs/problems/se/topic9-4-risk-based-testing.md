## 해설

### 출제 배경 및 의도

리스크 기반 테스팅(RBT)은 ISO/IEC/IEEE 29119 소프트웨어 테스팅 표준과 ISTQB 실라버스에 공식 포함된 주류 테스팅 접근법이다. 현실 프로젝트에서 테스트 자원은 항상 부족하므로, "어디에 자원을 집중할 것인가"라는 의사결정이 테스트 품질을 결정한다.

채점관은 RBT의 단순 정의가 아니라, 리스크 식별 → 평가 → 우선순위화 → 테스트 배분이라는 4단계 프로세스와 리스크 매트릭스를 실제 프로젝트에 적용하는 능력을 평가한다. 잔존 리스크(Residual Risk)와 동적 리스크 모니터링 개념을 추가하면 고득점이 가능하다.

### 1. 리스크 기반 테스팅(RBT) 개요

정 의  • 결함이 발생할 확률(Likelihood)과 비즈니스 영향도(Impact)를 기반으로
       테스트 자원을 전략적으로 배분하는 테스팅 접근법
       - 리스크 레벨 = 발생 확률(P) × 영향도(I)

```
[리스크 기반 테스팅 프로세스]

  1단계: 리스크 식별(Risk Identification)
         브레인스토밍·체크리스트·전문가 인터뷰
         기능적 리스크 + 비기능적 리스크 열거
               │
  2단계: 리스크 평가(Risk Assessment)
         각 리스크에 확률(P)·영향도(I) 평가
         리스크 레벨 = P × I (1~5 척도 또는 상/중/하)
               │
  3단계: 리스크 우선순위화(Risk Prioritization)
         리스크 매트릭스로 High/Medium/Low 구분
         우선순위 높은 항목에 테스트 자원 집중
               │
  4단계: 테스트 설계 및 실행(Test Design & Execution)
         High Risk  → 상세 테스트 설계, 충분한 시간 배정
         Medium Risk → 표준 테스트
         Low Risk   → 최소한의 테스트 또는 생략
```

- RBT는 완벽한 테스팅이 불가능한 현실에서 제한된 자원으로 최대의 결함 발견 효과를 달성하는 전략이다.

### 2. 리스크 분류 및 매트릭스

1) 리스크 분류

| 분류 | 설명 | 예시 |
|------|------|------|
| 기능적 리스크(Product Risk) | SW 기능이 요구사항 미충족 리스크 | 결제 금액 오류, 로그인 실패 |
| 비기능적 리스크(Product Risk) | 성능·보안·가용성 등 품질속성 리스크 | 피크타임 서버 다운, 개인정보 유출 |
| 프로젝트 리스크(Project Risk) | 일정·예산·조직 관련 리스크 | 개발 지연으로 인한 테스트 기간 단축 |

- RBT는 주로 제품 리스크(Product Risk)를 다루며, 프로젝트 리스크는 테스트 계획 수립 시 별도 관리한다.

2) 리스크 매트릭스(Risk Matrix)

```
[리스크 매트릭스]

영향도(Impact)
 높음 │  중간   │  높음   │  매우높음 │
      ├─────────┼─────────┼──────────┤
 중간 │  낮음   │  중간   │  높음    │
      ├─────────┼─────────┼──────────┤
 낮음 │매우낮음 │  낮음   │  중간    │
      └─────────┴─────────┴──────────┘
          낮음      중간      높음    발생 확률(Probability)
```

| 리스크 수준 | 테스트 전략 |
|-----------|-----------|
| 매우 높음(Critical) | 가장 먼저·가장 집중적으로 테스트, 전문 테스터 배정 |
| 높음(High) | 상세 테스트 설계, 충분한 시간 배정 |
| 중간(Medium) | 표준적 테스트 수행 |
| 낮음(Low) | 시간 허용 시에만 테스트 또는 생략 |

- 리스크 매트릭스는 이해관계자와 테스트 전략을 공유하는 커뮤니케이션 도구이기도 하다.

### 3. RBT vs 요구사항 기반 테스트 비교 및 트렌드

1) RBT vs 요구사항 기반 테스트

| 구분 | 요구사항 기반 테스트 | 리스크 기반 테스팅(RBT) |
|------|------------------|---------------------|
| 대상 선정 기준 | 요구사항 목록 (모든 요구사항 커버) | 리스크 수준 (높은 리스크 우선) |
| 자원 배분 | 균등 배분 | 리스크 비례 배분 |
| 장점 | 명확한 추적성 | 효율적 자원 사용, 비즈니스 리스크 관리 |
| 단점 | 위험 기능에 집중 어려움 | 리스크 평가의 주관성 |
| 적합 상황 | 요구사항 명확·시간 충분 | 자원 제한·일정 압박 |

- RBT와 요구사항 기반 테스트는 상호 보완적으로 활용하며, 리스크가 높은 영역에 RBT를 우선 적용하는 것이 현실적이다.  "끝"

### 실무 제언

**리스크 평가의 객관성 확보**
- **챌린지**: 테스트 매니저 한 사람의 판단에 의존하면 리스크 평가에 편향이 발생한다.
- **제언**: 개발자·비즈니스 분석가·운영팀·보안팀이 함께 참여하는 리스크 워크숍을 진행하고, 이전 버전 결함 이력 데이터로 발생 확률을 객관적으로 산정한다. FMEA(Failure Mode and Effect Analysis) 기법을 병행하면 잠재적 실패 모드를 체계적으로 분석할 수 있다.

**리스크는 동적으로 변화한다**
- **챌린지**: 프로젝트 초기에 설정한 리스크 우선순위는 개발 진행, 추가 요구사항, 외부 환경 변화(보안 취약점 발견 등)에 따라 변화한다.
- **제언**: 스프린트마다 리스크 재평가 세션을 운영하고, 잔존 리스크(Residual Risk)를 이해관계자와 공유하는 지속적 리스크 모니터링 체계를 구축해야 한다. 리스크 레지스터(Risk Register)를 CI/CD 파이프라인과 연동하여 고리스크 영역의 테스트 커버리지를 자동으로 강화하는 방향이 궁극적 목표다.
