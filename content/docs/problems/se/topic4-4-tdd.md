## 문제

대규모 프로젝트에 애자일 개발 방법론을 적용하려는 시도가 늘어나고 있다. 애자일 개발방법론 적용에 대해 다음을 설명하시오.

- 가. 테스트 중심의 애자일 개발 방법론 TDD 설명
- 나. 성공적인 프로젝트를 위한 테스트 전략

## 해설

### 출제 배경 및 의도

생성형 AI(Copilot, LLM) 도입으로 코드 생산량이 폭발적으로 증가하면서 역설적으로 코드 신뢰성은 더욱 위협받고 있다. 대규모 프로젝트에서 결함 수정 비용은 발견 시점이 늦을수록 기하급수적으로 증가하며, 하나의 버그 수정이 다른 기능을 깨뜨리는 회귀 결함(Regression Failure)이 반복된다.

채점관이 TDD와 테스트 전략을 묻는 의도는 사후 결함 수정에 머무는 '수동적 테스터' 수준을 넘어, 개발 초기 단계부터 결함을 원천 차단하는 시프트 레프트(Shift-Left) 전략과 전체 테스트 파이프라인 자동화를 설계할 수 있는 품질 거버넌스 아키텍트 역량을 검증하는 것이다.

### 1. TDD(Test Driven Development) 개요

정 의  • 실제 코드 작성 전 테스트 케이스를 먼저 작성하고, 이를 통과하는 최소 코드 구현 후 지속적 리팩토링으로 고품질 소스코드를 완성하는 미시적 반복(Micro-Iteration) 기반의 소프트웨어 디자인 패러다임
       - 테스트 코드 자체가 최신의 살아있는 요구사항 명세서 역할 수행

```
[TDD Red-Green-Refactor 주기]

  ┌─────────────────────────────────────────────────────┐
  │                                                     │
  │   ① Red                ② Green            ③ Refactor│
  │  (실패 테스트 작성)    (최소 코드 구현)    (코드 정제) │
  │                                                     │
  │   테스트 코드 작성  →  구현 코드 작성  →  코드 개선  │
  │   (아직 구현 없음)     (테스트 통과)      (기능 유지) │
  │        │                    │                │      │
  │      실패(Red)           성공(Green)       반복     │
  │        └────────────────────┘                │      │
  │                   루프                        │      │
  │        ◀──────────────────────────────────────┘      │
  └─────────────────────────────────────────────────────┘
```

- TDD는 요구사항 모호성 제거, 인터페이스 규격 명확화, 기술 부채 조기 해소를 동시에 달성하는 설계 패러다임이다

### 2. 가. TDD 핵심 메커니즘 및 단계별 효과

1) Red 단계 — 실패하는 테스트 작성

| 구분 | 항목 | 설명 |
|------|------|------|
| 수행 활동 | 테스트 코드 선작성 | 구현 기능의 비즈니스 요구사항을 테스트 케이스로 먼저 작성 |
| 실행 결과 | 강제 실패(Red) | 실제 코드 미존재로 테스트 반드시 실패 — 의도된 상태 |
| 효과 | 인터페이스 명확화 | 요구사항 모호성 제거 및 구현 인터페이스 규격 확정 |

- 요구사항을 검증 가능한 코드 형태로 전환함으로써 발주자-개발자 간 해석 충돌을 사전 차단한다

2) Green 단계 — 테스트 통과 코드 작성

| 구분 | 항목 | 설명 |
|------|------|------|
| 수행 활동 | 최소 코드 구현 | 테스트 통과를 위한 최소한의 실제 소스코드 빠르게 작성 |
| 코드 품질 | 임시 허용 | 중복·지저분함 허용 — 기능 동작 우선 |
| 효과 | 즉각적 피드백 | 작동 가능 상태(Green) 확보로 개발자 심리적 안정감 제공 |

- 빠른 Green 달성으로 개발 사이클 모멘텀 유지 및 조기 동작 검증이 가능하다

3) Refactor 단계 — 코드 정제 및 품질 개선

| 구분 | 항목 | 설명 |
|------|------|------|
| 수행 활동 | 구조적 개선 | Green 유지 상태에서 코드 스멜 제거, 가독성 향상 |
| 안전 장치 | 자동화 테스트 | 기존 기능 보장 하에 기술 부채 과감히 해소 |
| 효과 | 회귀 결함 차단 | Regression Failure 없이 지속적 아키텍처 개선 가능 |

- 테스트 세트가 안전망이 되어 리팩토링 비용과 공포를 동시에 제거한다

### 3. 나. 성공적인 대규모 프로젝트 테스트 전략

1) 테스트 피라미드(Testing Pyramid) 전략

```
[Mike Cohn 테스트 피라미드]

         ▲
        /E2E\        [10%] UI/E2E Test — 핵심 비즈니스 흐름 최소 적용
       /─────\
      /Service\      [20%] 서비스/통합 테스트 — API, DB 연계 무결성 검증
     /─────────\
    /  Unit Test \   [70%] 단위 테스트 — TDD 기반, 0.1초 미만 고속 실행
   /─────────────\
```

| 구분 | 유형 | 비중 | 특성 |
|------|------|------|------|
| 하부 | 단위 테스트 | 70% | TDD 기반, 격리된 메서드·클래스 단위, 고속 실행 |
| 중부 | 서비스/통합 테스트 | 20% | 모듈 간 상호작용, API·DB 연계 무결성 검증 |
| 상부 | UI/E2E 테스트 | 10% | 핵심 사용자 시나리오만 적용, 비용 최소화 |

- 수동 UI 테스트에 의존하는 아이스크림 콘(Ice Cream Cone) 안티패턴을 철저히 배제해야 한다

2) 시프트 레프트(Shift-Left) + CI/CD 품질 게이트 전략

| 구분 | 항목 | 설명 |
|------|------|------|
| 시프트 레프트 | 품질 활동 조기화 | 요구사항 명세 검토 단계부터 기획서 모순 발굴 |
| CI 자동화 | 전수 테스트 | Git Push 시 CI 서버 자동 트리거, 전수 테스트 실행 |
| 품질 게이트 | 머지 차단 | 커버리지 미달(80% 기준) 또는 테스트 실패 시 브랜치 병합 원천 차단 |
| 정적 분석 | SonarQube | 코드 커버리지, 코드 스멜, 보안 취약점 자동 스캔 |

- CI/CD 파이프라인 내 품질 게이트는 대규모 프로젝트에서 결함 비용을 최소화하는 핵심 통제 장치다.  "끝"

### 실무 제언

**코드 커버리지 만능주의 경계**
- **챌린지**: '커버리지 80% 이상 무조건 달성' 강제 규칙이 getter/setter 줄 채우기식 테스트 양산을 초래한다
- **제언**: 양적(커버리지 수치) 지표와 함께 돌연변이 테스팅(Mutation Testing) 등 질적 지표를 병행하여 비즈니스 규칙의 제어 흐름(Decision/Condition) 검증을 우선한다

**플레이키 테스트(Flaky Test) 통제**
- **챌린지**: 동일 코드에서 성공/실패가 반복되는 플레이키 테스트는 외부 DB 상태·네트워크 지연·전역 상태 오염에서 비롯되며, 파이프라인 전체를 무용지물로 만든다
- **제언**: Mock 객체·인메모리 DB(H2)·Testcontainers를 CI/CD 파이프라인에 내장하여 매 테스트마다 독립적 샌드박스 환경을 자동 빌드·삭제하는 인프라 거버넌스를 확립한다

**DevOps 파이프라인과 TDD 품질 통제 통합**
- **챌린지**: TDD 도입 후에도 파이프라인 통제 없이 개발자 자율에 맡기면 테스트 코드 작성이 형식화된다
- **제언**: SonarQube Quality Gate를 PR 병합 필수 조건으로 설정하고, 아키텍처 적합성 테스트(ArchUnit)를 파이프라인에 추가하여 설계 원칙 위반을 자동 차단하는 품질 거버넌스를 제도화한다
