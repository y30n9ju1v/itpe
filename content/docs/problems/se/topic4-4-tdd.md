## 문제

대규모 프로젝트에 애자일 개발 방법론을 적용하려는 시도가 늘어나고 있다. 애자일 개발방법론 적용에 대해 다음을 설명하시오.

- 가. 테스트 중심의 애자일 개발 방법론 TDD 설명
- 나. 성공적인 프로젝트를 위한 테스트 전략

## 해설

> **결함 제로를 향한 아키텍처적 선언**  
> TDD(Test Driven Development)와 대규모 프로젝트 테스트 전략

대규모 복잡계 시스템 개발에서 가장 막대한 비용이 소모되는 구간은 코드를 작성하는 시간이 아닙니다. 바로 개발 후반부 시스템 통합 단계에서 터져 나오는 원인 불명의 결함들을 추적하고 수정하는 시간입니다. 시스템의 규모가 커질수록 결함 수정의 비용은 기하급수적으로 증가하며, 하나의 버그를 고치기 위해 다른 기능을 망가뜨리는 회귀 결함(Regression Failure, 수정한 코드가 멀쩡히 동작하던 기존 기능을 깨뜨리는 현상)의 공포에 시달리게 됩니다.

### 1. 출제 배경 및 의도

생성형 AI(Copilot, LLM)의 도입으로 소스코드의 생산량은 폭발적으로 증가했습니다. 그러나 역설적으로 코드의 신뢰성과 무결성은 그 어느 때보다 위협받고 있습니다. AI가 순식간에 짜놓은 수천 줄의 코드가 기존 레거시 시스템과 결합할 때 발생하는 사이드 이펙트(Side Effect)를 인간의 눈으로 검증하는 것은 불가능에 가깝습니다.

채점관이 TDD와 테스트 전략을 묻는 의도는 수험생이 사후에 결함을 찾아내는 '수동적 테스터'의 마인드에 머물러 있는지, 아니면 개발 초기 단계부터 결함을 원천 차단하는 '시프트 레프트(Shift-Left, 테스트와 품질 활동을 개발 주기의 더 이른 단계로 당기는 전략) 기법'을 실천하고 전체 테스트 파이프라인을 자동화할 수 있는 '품질 거버넌스 아키텍트' 역량이 있는지 검증하려는 것입니다.

### 2. 가. TDD(Test Driven Development) 설명

#### 1) TDD의 정의 및 개념

TDD는 "실제 코드를 작성하기 전에 소스코드의 요구사항을 검증하는 테스트 케이스를 먼저 작성하고, 이를 통과하기 위한 최소한의 실제 코드를 구현한 후, 지속적인 리팩토링을 통해 고품질의 소스코드를 완성해 나가는 미시적 반복(Micro-Iteration) 기반의 소프트웨어 디자인 패러다임"입니다.

#### 2) TDD의 핵심 작동 메커니즘: Red-Green-Refactor 주기

TDD는 '질문(Test) → 응답(Code) → 정제(Refactor)'라는 명확한 3단계 루프를 돌며 작동합니다.

**1단계: Red (실패하는 테스트 작성)**
- **수행 활동**: 구현하고자 하는 기능의 비즈니스 요구사항을 만족하는 테스트 코드를 먼저 작성합니다. 실제 코드가 아직 존재하지 않으므로 이 테스트는 반드시 실패(Red)해야 합니다.
- **효과**: 요구사항의 모호성이 제거되고 구현해야 할 인터페이스의 규격이 명확해집니다.

**2단계: Green (테스트를 통과하는 코드 작성)**
- **수행 활동**: 앞서 작성한 테스트 케이스를 성공(Pass)시키기 위한 최소한의 실제 소스코드를 빠르게 작성합니다. 코드의 중복이나 지저분함은 이 단계에서 무시합니다.
- **효과**: 빠르게 작동 가능한 상태(Green)를 만들어 개발자에게 즉각적인 심리적 안정감과 피드백을 제공합니다.

**3단계: Refactor (코드 정제 및 품질 개선)**
- **수행 활동**: 테스트의 초록불(Green)을 유지한 상태에서 코드 스멜(중복 코드, 하드코딩, 설계 결함)을 제거하고 가독성을 높이는 구조적 개선을 수행합니다.
- **효과**: 기존 기능이 깨지지 않는다는 강력한 안전장치(테스트 자동화)가 확보된 상태이므로, 개발자는 기술 부채를 과감하게 해소할 수 있습니다.

### 3. 나. 성공적인 프로젝트를 위한 입체적 테스트 전략

수십 명이 참여하는 대규모 프로젝트에서 테스트 전략이 부실하면 시스템 통합 단계에서 장애가 도미노처럼 터지는 '소프트웨어 위기'를 맞이하게 됩니다. 대형 프로젝트를 성공으로 이끄는 품질 아키텍처 전략의 3대 핵심 축은 다음과 같습니다.

#### 1) 애자일 테스트 피라미드(Testing Pyramid) 전략 준수

대규모 프로젝트에서 흔히 저지르는 실수는 수동 UI 테스트에 전적으로 의존하는 '아이스크림 콘(Ice Cream Cone)' 안티 패턴입니다. 성공적인 프로젝트는 마이크 코언(Mike Cohn)의 테스트 피라미드 구조를 엄격히 준수해야 합니다.

```text
         ▲
        /E2E\        [10%] UI/E2E Test — 핵심 비즈니스 흐름만 최소 적용
       /─────\
      /Service\      [20%] 서비스/통합 테스트 — API, DB 연계 무결성 검증
     /─────────\
    /  Unit Test \   [70%] 단위 테스트 — TDD 기반, 0.1초 미만 고속 실행
   /─────────────\
```

- **하부 구조 (단위 테스트, 70%)**: TDD를 기반으로 생성된 개별 클래스, 메서드 단위의 격리된 테스트. 실행 속도가 극도로 빠르고 비용이 저렴하므로 전체 테스트의 절대다수를 차지해야 합니다.
- **중부 구조 (서비스/통합 테스트, 20%)**: 모듈 간의 상호작용, API 인터페이스, 데이터베이스 연계 무결성을 검증합니다. MSA 환경에서 서비스 간 결합 리스크를 차단하는 핵심 구간입니다.
- **상부 구조 (UI/E2E 테스트, 10%)**: 실제 사용자 시나리오대로 시스템 전체 아키텍처를 검증합니다. 비용이 가장 비싸고 변경에 취약하므로, 반드시 필요한 핵심 비즈니스 흐름(핵심 결제, 로그인 등)에만 최소한으로 적용합니다.

#### 2) 시프트 레프트(Shift-Left Testing) 전략

전통적인 폭포수 모델에서는 테스트가 소프트웨어 수명주기(SDLC)의 가장 우측(후반부)에 배치되었습니다. 성공적인 애자일 전략은 품질 활동의 시점을 대대적으로 왼쪽(초기 단계)으로 이동시키는 '시프트 레프트'를 선언해야 합니다. 요구사항 명세 검토 단계에서 기획서의 모순을 잡아내고, 개발 착수와 동시에 TDD를 수행하여 소스코드 레벨의 버그를 조기에 리젝(Reject)함으로써 결함 예방 효율성을 극대화합니다.

#### 3) CI/CD 파이프라인 기반 자동화 테스트 게이트(Test Gate) 구축

수십 명의 개발자가 소스코드를 Git에 올릴 때마다, CI 서버가 자동으로 트리거되어 전수 테스트를 수행해야 합니다. SonarQube 등의 정적 분석을 통해 코드 커버리지 지표가 목표치(예: 80%)에 미달하거나, 단위 테스트가 단 하나라도 실패할 경우 본 배포 브랜치로의 병합(Merge)을 원천 차단하는 강력한 품질 게이트 제어를 수행합니다.

### 4. 맥락과 비교

| 비교 항목 | 전통적 개발 방식 (Code-Driven) | TDD 기반 개발 방식 (Test-Driven) |
|----------|-----------------------------|---------------------------------|
| 작성 순서 | 실제 소스코드 → 테스트 케이스 | 테스트 케이스(Red) → 실제 코드(Green) |
| 중심 초점 | 요구사항의 빠른 기능 구현 자체에 집중 | 인터페이스 설계의 정밀성 및 구조적 유연성 |
| 결함 발견 시점 | 후반부 통합 테스트 단계 (사후 대처) | 개발 초기 단계 (실시간 사전 예방) |
| 회귀 테스트 비용 | 수동 테스트 증가로 인해 기하급수적으로 폭증 | 자동화된 테스트 세트 보유로 비용 최저화 |
| 문서화 역할 | 별도의 아키텍처/설계 산출물 문서 요구 | 테스트 코드 자체가 최신의 살아있는 명세서 역할 |
| 초기 개발 속도 | 매우 빠름 (설계 오버헤드 없음) | 상대적으로 느림 (테스트 코드 작성 비용 발생) |
| 전체 생명주기 비용 | 결함 수정 비용 누적으로 인해 폭발적 증가 | 지속적인 리팩토링으로 전체 비용 대폭 절감 |

### 5. 실무 제언

**코드 커버리지 100%라는 만능주의의 늪 경계**

- **챌린지**: 프로젝트 관리자들이 흔히 저지르는 실책은 정량적 지표인 '코드 커버리지 80% 이상 무조건 달성'을 강제하는 규칙입니다. 개발자들이 비즈니스 가치를 검증하는 정밀한 테스트를 짜는 것이 아니라, 수치를 채우기 위해 의미 없는 getter/setter나 단순 조회 쿼리에 줄 채우기식 테스트 코드를 양산하게 됩니다.
- **제언**: 아키텍트는 양적(Quantity) 지표인 커버리지 수치 준수와 더불어 질적(Quality) 지표인 '돌연변이 테스팅(Mutation Testing)' 기법이나 핵심 도메인 영역에 대한 가중치 필터링을 병행해야 합니다. 의미 없는 소스코드 라인 커버리지보다는 비즈니스 규칙의 제어 흐름(Decision/Condition)이 제대로 검증되었는지 파악하는 품질 거버넌스가 우선되어야 합니다.

**플레이키 테스트(Flaky Test) 통제와 테스트 인프라 격리**

- **챌린지**: 대규모 시스템에서 자동화 테스트 파이프라인을 운영할 때 가장 큰 골칫거리는 동일한 코드임에도 어떨 때는 성공하고 어떨 때는 실패하는 '플레이키 테스트(Flaky Test)'의 발생입니다. 이는 주로 외부 데이터베이스 상태, 네트워크 지연, 테스트 간 전역 상태 오염 때문에 일어납니다. 불신을 주는 테스트 환경은 결국 개발팀이 테스트 결과를 무시하게 만들어 파이프라인 전체를 무용지물로 만듭니다.
- **제언**: 모든 단위 테스트는 철저히 격리(Isolation)되어야 합니다. 실제 외부 인프라에 의존하지 않도록 DIP를 준수하고, Mock 객체나 인메모리 데이터베이스(H2, SQLite 등) 및 도커 컨테이너 기술인 Testcontainers를 CI/CD 파이프라인 내에 내장해야 합니다. 매 테스트 수행마다 독립적이고 순수한 샌드박스 환경을 자동 빌드·삭제하는 인프라 거버넌스를 확립해야만 대규모 프로젝트의 품질 혁신을 성공적으로 유지할 수 있습니다.
