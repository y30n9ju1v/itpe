## 해설

### 출제 배경 및 의도

자율주행차 사고는 "소프트웨어 기능안전"이라는 개념을 사회적 의제로 부상시켰다. IEC 61508은 E/E/PE 시스템 전반의 기능안전 범용 표준이며, ISO 26262는 이를 자동차 도메인에 특화한 표준이다. 자동차 OEM 및 부품 공급사(Tier-1/2)에서 ECU 소프트웨어 개발 시 ISO 26262 준수가 사실상 의무화되면서, 두 표준의 관계·차이·실무 적용 방법이 기술사 시험에서 반복 출제된다.

응용 유형 출제로, 단순 표준 나열이 아닌 두 표준의 구조적 차이(SIL vs ASIL 결정 파라미터), 자동차 특화 개념(ASIL 분해·SEooC·HARA), 그리고 자율주행 맥락에서의 한계와 진화 방향을 논리적으로 서술해야 한다.

### 1. IEC 61508과 ISO 26262 개요

정 의  • **IEC 61508**을 E/E/PE 안전 관련 시스템의 기능안전을 위한 범용 국제 표준(IEC, 1998/2010)
       - **ISO 26262**: IEC 61508을 자동차 도메인에 특화한 파생 표준(ISO, 2011/2018 2판)
       - 관계: ISO 26262는 IEC 61508의 원칙을 계승하되 ASIL·V-모델·SEooC 등 자동차 특수 요구사항을 추가

```
[IEC 61508과 ISO 26262의 관계]

IEC 61508 (1998/2010) — E/E/PE 범용 표준
┌─────────────────────────────────────────────────────┐
│  적용: 산업 자동화·원전·의료·철도·자동차 등          │
│  위험 등급: SIL 1~4                                  │
└────────────────────┬────────────────────────────────┘
                     │ 특화(Tailoring)
         ┌───────────┼───────────┐
         ▼           ▼           ▼
   ISO 26262     IEC 62304    DO-178C
   (자동차)      (의료기기)    (항공)
   ASIL A~D     Class A/B/C  DAL A~E

[ISO 26262 V-모델 개발 프로세스]

차량 수준: 아이템 정의 + HARA  ─────────────→ 차량 수준 검증
           ↓                                         ↑
기능 안전 개념: 안전 목표·ASIL 할당 ──────→ 차량 통합 테스트
           ↓                                         ↑
시스템 수준: 기술적 안전 개념  ──────────→ 시스템 통합 테스트
           ↓                                         ↑
HW/SW 수준: 상세 설계·구현  ────────────→ HW/SW 테스트
```

- ISO 26262는 IEC 61508의 SIL 체계를 ASIL(A~D)로 재정의하고, 자동차 개발 V-모델과 SEooC 프레임워크를 추가하여 부품 공급망 전체에 적용 가능하게 특화하였다.

### 2. 핵심 내용 상세

1) 가. IEC 61508 상세 설명

| 구분 | 항목 | 설명 |
|------|------|------|
| 정식 명칭 | IEC 61508:2010 | Functional Safety of E/E/PE Safety-Related Systems |
| 구성 | Part 1~7 (7개 파트) | Part 3(SW 요구사항)이 핵심 |
| 핵심 개념 | SIL 1~4 | 위험 감소량 기반 4단계 안전 무결성 등급 |
| SW 커버리지 | SIL 4: MC/DC 100% 필수 | SIL 등급 높을수록 테스트·코딩 엄밀도 증가 |
| 안전 생명주기 | 개념 설계 → 폐기 16단계 | V&V 활동이 각 단계마다 수행 |

```
[SIL 결정 프로세스]

위험원(Hazard) 식별
      │
위험 평가: 심각도 × 노출 빈도 × 회피 가능성 → 위험도
      │
허용 가능 위험 대비 실제 위험 = 요구 위험 감소량
      │
    10 ~   100배 감소 → SIL 1
   100 ~ 1,000배 감소 → SIL 2
 1,000 ~ 10,000배 감소 → SIL 3
10,000 ~100,000배 감소 → SIL 4
```

2) 나. ISO 26262 상세 설명

| 구분 | 항목 | 설명 |
|------|------|------|
| 정식 명칭 | ISO 26262:2018 | Road Vehicles — Functional Safety (12개 파트) |
| 핵심 개념 | ASIL A~D + QM | Automotive Safety Integrity Level |
| ASIL 결정 | S × E × C | 심각도(S0~S3) × 노출도(E0~E4) × 제어가능성(C0~C3) |
| ASIL 분해 | ASIL D = B(D)+B(D) | 독립 채널 분리로 고 ASIL 요구사항 충족 |
| 특화 개념 | SEooC, HARA, MISRA-C | 부품 공급망 대응·위험 분석·코딩 표준 |

```
[ASIL 결정 매트릭스 — S3(치명) 심각도]

        C1(쉽게 제어)  C2(보통 제어)  C3(제어 곤란)
E1(드문 노출) │   QM      │    QM     │   ASIL A  │
E2(드물게)    │   QM      │  ASIL A   │   ASIL B  │
E3(종종)      │  ASIL A   │  ASIL B   │   ASIL C  │
E4(자주 노출) │  ASIL B   │  ASIL C   │   ASIL D  │

ASIL 분해 예시:
  ASIL D = ASIL B(D) + ASIL B(D)  (독립 채널 분리)
  조건: 두 채널 간 공통 원인 고장(CCF) 방지 필수
```

### 3. IEC 61508 vs ISO 26262 종합 비교 및 자율주행 함의

1) 종합 비교

| 비교 항목 | IEC 61508 | ISO 26262 |
|----------|-----------|-----------|
| 적용 범위 | E/E/PE 시스템 전반 | 자동차 전기/전자 시스템 |
| 초판 제정 | 1998년 | 2011년 (2판: 2018년) |
| 위험 등급 | SIL 1~4 | ASIL A~D + QM |
| 위험 결정 파라미터 | 심각도 × 빈도 | S(심각도) × E(노출도) × C(제어가능성) |
| 부품 재사용 | 일반 지침 | SEooC(Safety Element out of Context) |
| 위험 분석 방법 | PHA·FMEA·FTA 등 | HARA(Hazard Analysis and Risk Assessment) |
| 코딩 표준 | 도메인 선택 | MISRA-C 강력 권장 |
| 독립성 요구 | 독립 안전 평가자(FISA) | 기능 안전 매니저(FSM) + 독립 심사 |

2) 자율주행 시대의 기능안전 진화 과제

```
[기능안전 표준 발전 방향]

IEC 61508 / ISO 26262
     │ 한계: AI/ML 비결정론적 동작 증명 미완
     ▼
ISO 21448 (SOTIF)
  → 기능 부족·의도적 오용으로 인한 위험 관리
ISO 21434 (자동차 사이버보안)
  → Security 침해 → Safety 사고 연계 대응
UL 4600 (자율주행 안전성)
  → SAE Level 3~5 안전 케이스 작성 표준
```

- 자율주행 수준이 높아질수록 제어 가능성(C)이 낮아져 ASIL D가 광범위하게 요구되며, AI/ML 기반 ADAS의 안전성 증명을 위한 새로운 표준 체계가 필요하다.  "끝"

### 실무 제언

**조기 ASIL 결정과 아키텍처 전략**
- **챌린지**: ASIL이 높아질수록 개발 비용이 기하급수적으로 증가하여 ASIL D 컴포넌트는 QM 대비 10배 이상 비용이 소요된다.
- **제언**: 아키텍처 설계 단계에서 ASIL 분해(ASIL D → B+B)를 통해 고 ASIL 컴포넌트의 복잡도를 낮추는 전략이 핵심이다. 고 ASIL과 저 ASIL 컴포넌트 간 명확한 격리(Partitioning)로 저 SIL 컴포넌트의 간섭을 차단해야 한다.

**OTA(Over-The-Air) 업데이트와 재검증**
- **챌린지**: 차량 SW를 무선으로 업데이트하는 OTA는 ISO 26262의 "수정 후 재검증" 요구사항과 충돌하여 매번 전체 인증 사이클을 반복해야 하는 현실적 부담이 있다.
- **제언**: ISO 26262 2판과 UN-R156(OTA 규정) 기반의 증분 안전 케이스(Incremental Safety Case) 접근을 채택하고, 변경 영향 분석(CIA: Change Impact Analysis) 프로세스를 사전 구축하여 재검증 범위를 최소화해야 한다.

**기능안전 문화와 FSM 역할**
- **챌린지**: ISO 26262는 기술 표준인 동시에 조직 문화 변혁을 요구한다. FSM(기능안전 매니저)이 프로젝트 후반에 투입되면 재설계 비용이 급증한다.
- **제언**: FSM을 프로젝트 킥오프 시점부터 참여시키고, 전 개발팀에 기능안전 교육을 의무화하는 조직 내재화 전략이 ISO 26262 준수의 실질적 기반이 된다.
