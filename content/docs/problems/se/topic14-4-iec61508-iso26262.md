## 문제

최근 미국에서 자율자동차의 사고로 인하여 인명 피해가 발생하였다. 이로 인해 자율 시스템에 포함된 소프트웨어의 기능 안전 문제가 크게 대두되고 있다. 이러한 기능 안전성 보증과 관련하여 다음에 대해 답하시오.

- 가. IEC 61508
- 나. ISO 26262
- 다. IEC-61508과 ISO 26262를 비교.

## 해설

> **기능 안전의 범용 표준과 자동차 특화 표준**  
> IEC 61508 — 모든 산업의 기준; ISO 26262 — 자동차를 위한 심화 표준

자율주행차 사고는 "소프트웨어 기능 안전"이라는 개념을 대중에게 각인시켰습니다. IEC 61508은 전기/전자 시스템 전반의 기능 안전을 다루는 범용 표준이고, ISO 26262는 자동차 산업의 특수성을 반영하여 IEC 61508을 특화한 표준입니다. 두 표준의 관계와 차이를 이해하는 것은 안전 필수 소프트웨어 개발의 핵심 역량입니다.

### 1. 출제 배경 및 의도

이 문제는 자율주행차 사고라는 구체적인 현실 맥락에서 출제되었습니다. 응용 유형으로 출제된 만큼, 단순 표준 나열에 그치지 않고 **실제 자동차 시스템 개발에 어떻게 적용되는지**, **두 표준의 관계와 선택 기준은 무엇인지** 논리적으로 서술해야 합니다.

채점관은 다음을 확인합니다.
- IEC 61508의 핵심 개념(SIL, 안전 생명주기)을 정확히 설명하는가
- ISO 26262의 차별점(ASIL, 자동차 특수 요구사항)을 구체적으로 서술하는가
- 두 표준의 비교를 체계적 표로 정리하는가
- 자율주행 맥락에서의 시사점을 제시하는가

### 2. 개념의 본질과 작동 메커니즘

#### 두 표준의 관계

```text
[IEC 61508과 ISO 26262의 관계]

IEC 61508 (1998/2010)
┌─────────────────────────────────────────────────────┐
│  E/E/PE Safety-Related Systems를 위한 범용 표준      │
│  적용: 산업 자동화, 원전, 의료, 철도, 자동차 등       │
│  위험 등급: SIL 1~4                                  │
└──────────────────────────┬──────────────────────────┘
                           │ 기반 (basis)
        ┌──────────────────┼───────────────────┐
        ▼                  ▼                   ▼
  ISO 26262          IEC 62304           DO-178C
  (자동차)           (의료기기)           (항공)
  ASIL A~D           Class A/B/C         DAL A~E
```

ISO 26262는 IEC 61508을 자동차 도메인에 맞게 **특화(Tailoring)**한 표준입니다. IEC 61508의 원칙과 프로세스를 계승하되, 자동차 개발 프로세스(V-모델), 자동차 특유의 위험 분류(ASIL), 차량 개발 특성을 반영했습니다.

### 3. 핵심 키워드 (Must-Have)

#### 가. IEC 61508 상세 설명

**정식 명칭**: IEC 61508:2010 "전기/전자/프로그래머블 전자 안전 관련 시스템의 기능 안전 (Functional Safety of E/E/PE Safety-Related Systems)"

**구성**: Part 1~7 (7개 파트)

| Part | 내용 |
|------|------|
| Part 1 | 일반 요구사항 (General Requirements) |
| Part 2 | E/E/PE 시스템 요구사항 |
| Part 3 | 소프트웨어 요구사항 (핵심) |
| Part 4 | 용어 정의 |
| Part 5 | SIL 결정 방법 |
| Part 6 | Part 2, 3 적용 가이드라인 |
| Part 7 | 기술 및 방법 개요 |

**핵심 개념: SIL (Safety Integrity Level)**

SIL은 안전 기능이 요구되는 성능을 달성할 확률을 나타내는 4단계 등급입니다.

```text
[SIL 결정 프로세스]

위험원(Hazard) 식별
         │
위험 평가: 심각도 × 노출 빈도 × 회피 가능성 → 위험도
         │
허용 가능 위험 대비 실제 위험 = 요구 위험 감소량
         │
요구 위험 감소량 → SIL 등급 결정
    10 ~ 100배 감소   → SIL 1
   100 ~ 1,000배 감소 → SIL 2
 1,000 ~ 10,000배 감소 → SIL 3
10,000 ~ 100,000배 감소 → SIL 4
```

**SIL별 소프트웨어 개발 요구사항 (Part 3)**

| SIL | 권장 테스트 커버리지 | 공식 검증 | 코딩 표준 |
|-----|------------------|---------|---------|
| SIL 1 | 구문 커버리지 (Statement) | 권장 | 권장 |
| SIL 2 | 분기 커버리지 (Branch) | 권장 | 권장 |
| SIL 3 | 변경 조건/결정 커버리지 (MC/DC) | 강력 권장 | 강력 권장 |
| SIL 4 | MC/DC 100% | 필수 | 필수 |

**안전 생명주기 모델**

IEC 61508은 시스템의 개념 설계부터 폐기까지 16단계의 안전 생명주기를 정의합니다. 핵심은 V&V(검증과 확인, Verification & Validation) 활동이 각 단계마다 수행되어야 한다는 것입니다.

#### 나. ISO 26262 상세 설명

**정식 명칭**: ISO 26262:2018 "Road Vehicles — Functional Safety"

**구성**: Part 1~12 (12개 파트)

| Part | 내용 |
|------|------|
| Part 1 | 용어 |
| Part 2 | 기능 안전 관리 |
| Part 3 | 개념 단계 |
| Part 4 | 시스템 레벨 개발 |
| Part 5 | 하드웨어 레벨 개발 |
| Part 6 | 소프트웨어 레벨 개발 (핵심) |
| Part 7 | 생산 및 운영 |
| Part 8 | 지원 프로세스 |
| Part 9 | ASIL 지향적 분석 |
| Part 10 | 가이드라인 |
| Part 11 | 반도체 적용 |
| Part 12 | 오토바이 적용 |

**핵심 개념: ASIL (Automotive Safety Integrity Level)**

ASIL은 IEC 61508의 SIL을 자동차 도메인에 맞게 재정의한 위험 등급입니다.

```text
[ASIL 결정 파라미터]

ASIL = f(심각도, 노출도, 제어 가능성)

심각도 (Severity, S): S0~S3
  S0: 부상 없음
  S1: 가벼운 부상
  S2: 중등도~심각한 부상 (생존 가능)
  S3: 생명을 위협하는 부상/사망

노출도 (Exposure, E): E0~E4
  E0: 사고 상황이 발생하지 않음
  E1: 매우 드물게 노출 (운전 시간의 1% 미만)
  E2: 드물게 노출
  E3: 종종 노출
  E4: 매우 자주 노출 (운전 시간의 10% 이상)

제어 가능성 (Controllability, C): C0~C3
  C0: 일반적으로 제어 가능
  C1: 간단히 제어 가능
  C2: 일반적으로 제어 가능
  C3: 거의 제어 불가능 또는 불가능
```

**ASIL 결정 매트릭스**

```text
[S3 심각도에서 ASIL 결정]

        C1      C2      C3
E1 │   QM    │  QM   │ ASIL A │
E2 │   QM    │ ASIL A│ ASIL B │
E3 │  ASIL A │ ASIL B│ ASIL C │
E4 │  ASIL B │ ASIL C│ ASIL D │
```

**ASIL 분해 (ASIL Decomposition)**

높은 ASIL 요구사항을 충족하기 어려울 때, 하나의 요구사항을 두 개의 독립적인 하위 ASIL 요구사항으로 분해하여 구현할 수 있습니다.

```text
ASIL D = ASIL B(D) + ASIL B(D)  또는  ASIL C(D) + ASIL A(D)
         (독립 채널 1)  (독립 채널 2)

조건: 두 채널은 반드시 독립적이어야 함
     (공통 원인 고장 방지 — Common Cause Failure 예방)
```

**자동차 V-모델 개발 프로세스**

ISO 26262는 자동차 도메인에 특화된 V-모델을 요구합니다.

```text
[ISO 26262 V-모델]

차량 수준(Vehicle Level)                    차량 수준 검증
    └→ 아이템 정의, 위험 분석 (HARA)  →  ┐      ↑
           │                              │      │
기능 안전 개념(FSC)                         │    차량 통합 테스트
    └→ 안전 목표, ASIL 할당        →  ┐  │      ↑
           │                          │  │      │
시스템 수준 개발                            시스템 통합 테스트
    └→ 기술적 안전 개념          →  ┐  │  │      ↑
           │                      │  │  │      │
HW/SW 수준 개발                   HW/SW 테스트──┘
    └→ 상세 설계, 구현      →  ─────┘
```

### 4. 맥락과 비교

**IEC 61508 vs ISO 26262 종합 비교**

| 비교 항목 | IEC 61508 | ISO 26262 |
|---------|---------|---------|
| 적용 범위 | E/E/PE 시스템 전반 | 자동차 전기/전자 시스템 |
| 초판 제정 | 1998년 | 2011년 (2판: 2018년) |
| 위험 등급 체계 | SIL 1~4 | ASIL A~D + QM |
| ASIL 결정 파라미터 | 심각도 × 빈도 | 심각도(S) × 노출도(E) × 제어가능성(C) |
| 소프트웨어 개발 모델 | 일반적 V-모델 | 자동차 특화 V-모델 |
| 독립성 요구 | 독립 안전 평가자(FISA) | 기능 안전 매니저(FSM), 독립 심사 |
| 부품 재사용 | 일반 지침 | SEooC(Safety Element out of Context) |
| 알려진 사용 사례 | 철도, 원전, 산업 자동화 | 자동차 전자제어장치(ECU) |
| 코딩 표준 권장 | 없음 (도메인 선택) | MISRA-C 강력 권장 |

**핵심 차이점 심층 분석**

1. **ASIL vs SIL**: SIL이 단순히 위험도 지표라면, ASIL은 자동차의 운전 상황에서 운전자가 위험을 제어할 수 있는지(Controllability)를 추가로 고려합니다. 같은 심각도의 사고라도 운전자가 회피할 수 있으면 낮은 ASIL이 결정됩니다.

2. **SEooC (Safety Element out of Context)**: ISO 26262의 핵심 개념으로, 완성차 개발에서 분리된 부품 공급자(Tier-1, Tier-2)가 최종 시스템 컨텍스트를 모르는 상태에서도 ASIL 준수 부품을 개발할 수 있는 프레임워크입니다.

3. **HARA (Hazard Analysis and Risk Assessment)**: ISO 26262 Part 3에서 요구하는 위험 분석 방법으로, IEC 61508보다 자동차 도메인에 특화된 체계적 위험 분석 방법입니다.

### 5. 실무 제언

**자동차 소프트웨어 개발에서 ISO 26262 적용의 현실**

ISO 26262 준수는 비용과 시간 면에서 매우 부담스럽습니다. 실무에서 고려할 점들입니다.

1. **조기 ASIL 결정의 중요성**: ASIL이 높아질수록 개발 비용이 기하급수적으로 증가합니다. 아키텍처 설계 단계에서 ASIL 분해를 통해 고 ASIL 컴포넌트의 복잡도를 낮추는 것이 핵심 전략입니다.

2. **기능 안전 문화**: ISO 26262는 도구가 아니라 문화입니다. 개발 조직 전체가 기능 안전의 중요성을 이해하고, 기능 안전 매니저(FSM)가 프로젝트 초기부터 참여해야 합니다.

3. **자율주행(SAE Level 3~5)의 도전**: 높은 자율주행 수준에서는 "운전자가 제어 가능성(C)"이 낮아 ASIL D가 광범위하게 요구됩니다. AI/ML 기반 인식 시스템의 기능 안전 증명은 ISO 26262 2판에서도 아직 완전히 다루지 못하며, ISO 21448(SOTIF)과 UL 4600(자율주행 안전성 표준)이 이 간극을 채우려 하고 있습니다.

4. **OTA(Over-The-Air) 업데이트**: 차량 소프트웨어를 무선으로 업데이트하는 OTA는 ISO 26262의 "수정 후 재검증" 요구사항과 충돌합니다. 이에 대한 규정 명확화가 2024년 이후 업계의 핵심 과제입니다.
