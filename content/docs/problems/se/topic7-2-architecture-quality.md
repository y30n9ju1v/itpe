## 해설

### 출제 배경 및 의도

소프트웨어 아키텍처의 핵심 역할은 기능(What)이 아니라 품질(How Well)을 보장하는 것이다. 클라우드 네이티브·MSA 환경에서 시스템 복잡도가 폭발적으로 증가하면서, "왜 이 아키텍처를 선택했는가?"에 대한 근거(Rationale) 없이 트렌드만 쫓는 설계가 운영 단계의 대형 장애로 이어지는 사례가 반복되고 있다.

카네기멜론 대학(CMU) SEI의 Bass, Clements, Kazman이 정립한 품질속성 프레임워크는 ATAM, CBAM 등 아키텍처 평가 기법의 이론적 기반으로, 기술사 시험의 아키텍처 관련 문제에서 반드시 등장하는 핵심 개념이다. 채점관은 QAS(품질속성 시나리오)-전술(Tactics)-스타일(Style)의 3층 체계 개념과 상호 관계를 논리적으로 설명하는 능력을 평가한다.

### 1. 소프트웨어 아키텍처 품질속성 체계 개요

정 의  • 이해관계자 요구에서 출발하여 품질속성 시나리오(QAS)로 정량화하고, 설계 전술(Tactics)로 구현하며, 아키텍처 스타일(Style)로 패키징하는 3층 설계 의사결정 프레임워크
       - "좋은 아키텍처 = 품질 속성 달성을 위한 의식적 설계 결정의 집합"

```
[QAS → 전술 → 스타일의 3층 관계]

  이해관계자 요구
        │
        ▼
  품질속성 시나리오(QAS)
  "500명 동시접속 시 응답 2초 이내"
  "해킹 시도 발생 시 30분 내 탐지"
        │ "이 품질을 어떻게 달성할 것인가?"
        ▼
  아키텍처 설계 전술(Tactics)  ← 단일 품질속성 원자적 결정
  - 성능: 캐싱, 로드밸런싱, 비동기 처리
  - 보안: 인증/권한부여, 침입탐지, 암호화
        │ "전술들을 패키징하면?"
        ▼
  아키텍처 스타일(Style)  ← 여러 전술을 묶은 상위 구조
  - 계층형(Layered): 변경용이성, 이식성 최적화
  - 마이크로서비스(MSA): 독립 배포, 확장성 최적화
  - 이벤트 드리븐(EDA): 느슨한 결합, 비동기 처리 최적화
```

- QAS-전술-스타일은 요구 → 설계 결정 → 구조의 일관된 추적성을 제공하는 아키텍처 거버넌스의 뼈대다

### 2. 가·나. 품질속성 시나리오 및 설계 전술

1) 품질속성 시나리오(QAS: Quality Attribute Scenario) 6대 구성 요소

```
[QAS 구성 요소 흐름도]

  자극원(Stimulus Source) ─→ 자극(Stimulus)
  "악의적 외부 사용자"          "로그인 시도 1000회/분"
                                      │
                                      ▼
                              환경(Environment)
                              "정상 운영 중"
                                      │
                                      ▼
  응답(Response)          ←── 대상체(Artifact)
  "30분간 IP 차단,               "인증 서비스"
   보안팀 알림 발송"
                                      │
                                      ▼
                           응답 측정값(Response Measure)
                           "탐지 시간 < 30초, 오탐률 < 1%"
```

| 품질 속성 | 자극 | 응답 | 측정값 |
|---------|-----|------|-------|
| 가용성(Availability) | 서버 장애 발생 | 페일오버 자동 전환 | 복구 시간 < 30초 |
| 성능(Performance) | 1000명 동시 요청 | 요청 처리 및 응답 | 응답시간 ≤ 2초 (95 퍼센타일) |
| 보안(Security) | SQL Injection 시도 | 입력 검증 및 차단 | 탐지율 > 99.9% |
| 변경용이성(Modifiability) | 결제 모듈 교체 요구 | 해당 모듈만 수정 | 변경 영향 모듈 ≤ 3개, 4시간 이내 |
| 테스트용이성(Testability) | 신규 기능 출시 | 자동화 테스트 실행 | 코드 커버리지 ≥ 80%, 실행 < 10분 |
| 사용성(Usability) | 신규 사용자 접속 | 도움말 없이 핵심 기능 수행 | 10분 내 첫 주문 완료율 > 80% |

- QAS는 "성능이 좋아야 한다"는 모호한 요구를 정량 검증 가능한 명세로 변환하는 아키텍처 계약서다

2) 아키텍처 설계 전술(Architectural Tactics) — 품질속성별 주요 전술

```
[가용성(Availability) 전술]

  결함 탐지             결함 복구                  결함 예방
  ───────────          ─────────────────          ──────────────
  핑/에코(Ping/Echo)    투표(Voting)               트랜잭션(Tx)
  하트비트(Heartbeat)   능동 이중화(Active          예측 모델
  예외 탐지              Redundancy)               프로세스 모니터
                        패시브 이중화(Standby)      증분 업데이트
```

| 구분 | 품질속성 | 주요 전술 | 적용 예 |
|------|---------|---------|--------|
| 성능 | Performance | 캐싱·동시성·비동기 | Redis 캐싱, 이벤트 큐, 멀티스레딩 |
| 보안 | Security | 인증·권한부여·암호화·IDS | OAuth2, JWT, AES-256, WAF |
| 변경용이성 | Modifiability | 인터페이스 추상화·캡슐화·런타임 바인딩 | DIP, 플러그인 아키텍처 |
| 가용성 | Availability | 하트비트·이중화·페일오버 | Kubernetes HPA, Active-Standby |

- 전술은 단일 품질속성에 초점을 맞춘 원자적(Atomic) 설계 결정으로, 스타일의 구성 요소가 된다

### 3. 다. 아키텍처 스타일 비교 및 진화 트렌드

1) 주요 아키텍처 스타일 비교

| 스타일 | 핵심 구조 | 최적화 품질속성 | 취약 품질속성 |
|-------|---------|--------------|------------|
| 계층형(Layered) | 계층 간 단방향 호출 | 변경용이성, 이식성 | 성능(레이어 통과 오버헤드) |
| 마이크로서비스(MSA) | 독립 서비스 + API Gateway | 확장성, 독립 배포 | 운영 복잡성, 분산 트랜잭션 |
| 이벤트 드리븐(EDA) | 이벤트 버스 + 핸들러 | 확장성, 느슨한 결합 | 디버깅 어려움, 이벤트 순서 보장 |
| Pipe-and-Filter | 파이프 + 필터 체인 | 재사용성, 병렬처리 | 상태 공유, 에러 처리 복잡 |
| Repository | 공유 데이터 저장소 | 데이터 일관성 | 병목(저장소 성능), 결합도 |
| 서비스 지향(SOA) | 서비스 + ESB | 상호운용성, 재사용 | 성능, ESB 단일 장애점 |

2) 전술 vs 스타일 vs 패턴 구분

| 구분 | 범위 | 초점 | 예시 |
|------|------|-----|------|
| 전술(Tactic) | 단일 품질속성 | 원자적 설계 결정 | 캐싱, 하트비트, 암호화 |
| 패턴(Pattern) | 반복되는 문제-해법 | 구체적 설계 문제 | Singleton, Observer, Facade |
| 스타일(Style) | 시스템 전체 구조 | 컴포넌트 구성 방식 | Layered, MSA, EDA |

- QAS로 품질 목표를 정량화하고, 전술로 달성 방법을 결정하며, 스타일로 전술을 아키텍처 구조로 패키징하는 3층 체계가 설계 결정의 추적성과 근거를 보장한다.  "끝"

### 실무 제언

**품질속성 간 트레이드오프 관리 — ATAM 활용**
- **챌린지**: 보안 강화는 성능을 저하시키고(암호화 오버헤드), 가용성 이중화는 비용을 급증시키며, 변경용이성 극대화는 아키텍처 복잡성과 운영 난이도를 높이는 상충이 반복된다
- **제언**: ATAM(Architecture Tradeoff Analysis Method)으로 감도점(Sensitivity Point)과 트레이드오프점(Tradeoff Point)을 식별하고, 이해관계자 합의로 우선순위를 결정한 뒤 ADR(Architecture Decision Record)에 명문화한다 — "보안 강화로 응답시간 50ms 증가 허용" 수준의 명시적 기록이 성숙한 아키텍처 거버넌스다

**QAS 기반 아키텍처 검증 자동화 — 진화적 아키텍처**
- **챌린지**: QAS는 설계 시 작성되지만 시스템 변경 과정에서 검증이 분리되어 아키텍처 드리프트(Architectural Drift)가 발생한다
- **제언**: 각 QAS를 Fitness Function으로 자동화한다 — "변경 모듈 수 ≤ 3" 제약은 ArchUnit, "응답시간 ≤ 2초"는 Gatling/k6, "커버리지 ≥ 80%"는 JaCoCo로 CI/CD 파이프라인에 내장하여 Neal Ford의 진화적 아키텍처(Evolutionary Architecture) 원칙을 실천한다

**아키텍처 스타일 선택의 맥락화 — 품질속성 우선순위 기반 선택**
- **챌린지**: "요즘 MSA가 트렌드라서" 수준의 선택은 불필요한 분산 복잡성과 운영 비용을 초래하며, 소규모 팀에게는 오히려 독이 된다
- **제언**: 도메인 규모·팀 성숙도·핵심 품질속성을 QAS로 정량화한 뒤, 그 달성에 최적화된 스타일을 선택한다 — 독립 배포·확장성이 핵심이면 MSA, 단순 변경용이성·이식성이면 Layered, 비동기 처리·느슨한 결합이 핵심이면 EDA가 적합하다
