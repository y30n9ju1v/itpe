## 문제

소프트웨어 아키텍처란 소프트웨어 시스템의 구성을 위한 중대한 결정사항들이라 할 수 있다. 소프트웨어 아키텍처에 대한 다음 문제에 답하시오.

- 가. 품질속성 시나리오
- 나. 아키텍처 설계 전술
- 다. 아키텍처 스타일

## 해설

> **아키텍처 설계는 품질 속성에 대한 결정의 집합이다**  
> 품질속성 시나리오(QAS), 설계 전술(Tactics), 아키텍처 스타일(Style)은 고품질 아키텍처를 만드는 3층 체계

"왜 이 아키텍처를 선택했는가?"라는 질문에 "그냥 MSA가 요즘 트렌드라서"라고 답한다면 기술사로서 실격입니다. 좋은 아키텍처란 이해관계자가 요구하는 **품질 속성(Quality Attributes)**을 달성하기 위해 의식적으로 설계된 구조여야 합니다. 이를 위한 체계가 바로 QAS-전술-스타일의 3층 프레임워크입니다.

### 1. 출제 배경 및 의도

소프트웨어 아키텍처의 핵심 역할은 기능(What)이 아니라 품질(How Well)을 보장하는 것입니다. 카네기멜론 대학(CMU) SEI(Software Engineering Institute)의 Bass, Clements, Kazman이 정립한 품질속성 프레임워크는 ATAM, CBAM 등 아키텍처 평가 기법의 이론적 기반으로, 기술사 시험의 아키텍처 관련 문제에서 반드시 등장하는 핵심 개념입니다.

채점관은 3가지 소문항—**품질속성 시나리오(QAS), 설계 전술(Tactics), 아키텍처 스타일(Style)**—의 개념과 상호관계를 체계적으로 설명하는 능력을 평가합니다.

### 2. 개념의 본질과 작동 메커니즘

```text
[QAS → 전술 → 스타일의 관계]

  이해관계자 요구
        │
        ▼
  품질속성 시나리오(QAS)
  "500명 동시접속 시 응답 2초 이내"
  "해킹 시도 발생 시 30분 내 탐지"
        │
        │ "이 품질을 어떻게 달성할 것인가?"
        ▼
  아키텍처 설계 전술(Tactics)
  - 성능: 캐싱, 로드밸런싱, 비동기 처리
  - 보안: 인증/권한부여, 침입탐지, 암호화
        │
        │ "전술들을 패키징하면?"
        ▼
  아키텍처 스타일(Style)
  - 계층형(Layered): 변경용이성, 이식성 최적화
  - 마이크로서비스(MSA): 독립 배포, 확장성 최적화
  - 이벤트 드리븐(EDA): 느슨한 결합, 비동기 처리 최적화
```

### 3. 핵심 키워드 (Must-Have)

#### 가. 품질속성 시나리오 (QAS: Quality Attribute Scenario)

**QAS란** 품질 속성 요구사항을 테스트 가능한 형태로 구체화한 명세입니다. Bass et al.은 QAS를 6가지 요소로 구성합니다:

```text
[QAS 6가지 구성 요소]

  자극원(Stimulus Source)  ─→  자극(Stimulus)
  "악의적 외부 사용자"           "로그인 시도 1000회/분"
                                       │
                                       ▼
                               환경(Environment)
                               "정상 운영 중"
                                       │
                                       ▼
  응답(Response)          ←──  대상체(Artifact)
  "30분간 IP 차단,                "인증 서비스"
   보안팀 알림 발송"
                                       │
                                       ▼
                            응답 측정값(Response Measure)
                            "탐지 시간 < 30초, 오탐률 < 1%"
```

**주요 품질속성 시나리오 예시**:

| 품질 속성 | 자극 | 응답 | 측정값 |
|---------|-----|------|-------|
| 가용성(Availability) | 서버 장애 발생 | 페일오버 자동 전환 | 복구 시간 < 30초 |
| 성능(Performance) | 1000명 동시 요청 | 요청 처리 및 응답 | 응답시간 ≤ 2초 (95 퍼센타일) |
| 보안(Security) | SQL Injection 시도 | 입력 검증 및 차단 | 탐지율 > 99.9% |
| 변경용이성(Modifiability) | 결제 모듈 교체 요구 | 해당 모듈만 수정 | 변경 영향 모듈 ≤ 3개, 4시간 이내 |
| 테스트용이성(Testability) | 신규 기능 출시 | 자동화 테스트 실행 | 코드 커버리지 ≥ 80%, 실행 < 10분 |
| 사용성(Usability) | 신규 사용자 접속 | 도움말 없이 핵심 기능 수행 | 10분 내 첫 주문 완료율 > 80% |

#### 나. 아키텍처 설계 전술 (Architectural Tactics)

**전술(Tactics)이란** 특정 품질 속성 응답값을 제어하기 위한 설계 결정입니다. 전술은 단일 품질 속성에 초점을 맞춘 원자적(Atomic) 설계 기법입니다.

**가용성 전술**:
```text
[가용성(Availability) 전술]

  결함 탐지         결함 복구            결함 예방
  ─────────        ─────────────        ──────────────
  핑/에코(Ping/Echo)  투표(Voting)        트랜잭션(Tx)
  하트비트          능동 이중화          예측 모델
  (Heartbeat)      (Active Redundancy)  프로세스 모니터
  예외 탐지         패시브 이중화        예외 방지
                   (Standby)           증분 업데이트
```

**성능 전술**:
- 자원 수요 제어: 이벤트 발생률 감소, 샘플링, 실행 시간 제한, 우선순위 큐
- 자원 관리: 동시성 증가(멀티스레딩), 자원 복제(캐싱), 자원 스케줄링

**보안 전술**:
- 공격 저항: 인증(Authentication), 권한부여(Authorization), 데이터 기밀성(암호화), 무결성(해시)
- 공격 탐지: 침입 탐지 시스템(IDS), 감사 추적(Audit Trail)
- 공격 복구: 권한 취소, 서비스 잠금, 증거 보존

**변경용이성 전술**:
```text
[변경용이성(Modifiability) 전술]

  모듈 결합도 감소            변경의 국소화
  ──────────────────         ─────────────────
  인터페이스 추상화           의미적 일관성 유지
  (Abstract Interface)       공통성 일반화
  구성 요소 캡슐화            여분 중간자 제거
  (Encapsulation)            (Intermediary)
  컴포넌트 대체 가능          실행 시점 바인딩
  (Substitutability)         (Runtime Binding)
```

#### 다. 아키텍처 스타일 (Architecture Style)

**스타일(Style)이란** 컴포넌트·커넥터·데이터 타입의 어휘와 이들을 결합하는 제약 조건의 집합입니다. 스타일은 여러 전술을 묶은 상위 수준의 패턴입니다.

| 스타일 | 핵심 구조 | 최적화 품질 속성 | 취약 품질 속성 |
|-------|---------|---------------|-------------|
| 계층형(Layered) | 계층 간 단방향 호출 | 변경용이성, 이식성 | 성능 (레이어 통과 오버헤드) |
| 마이크로서비스(MSA) | 독립 서비스 + API Gateway | 확장성, 독립 배포 | 운영 복잡성, 분산 트랜잭션 |
| 이벤트 드리븐(EDA) | 이벤트 버스 + 핸들러 | 확장성, 느슨한 결합 | 디버깅 어려움, 이벤트 순서 보장 |
| Pipe-and-Filter | 파이프 + 필터 체인 | 재사용성, 병렬처리 | 상태 공유, 에러 처리 복잡 |
| Repository | 공유 데이터 저장소 | 데이터 일관성 | 병목(저장소 성능), 결합도 |
| 서비스 지향(SOA) | 서비스 + ESB | 상호운용성, 재사용 | 성능, ESB 단일 장애점 |

### 4. 맥락과 비교

**전술 vs 스타일 vs 패턴 구분**:

| 구분 | 범위 | 초점 | 예시 |
|------|------|-----|------|
| 전술(Tactic) | 단일 품질속성 | 원자적 설계 결정 | 캐싱, 하트비트, 암호화 |
| 패턴(Pattern) | 반복되는 문제-해법 | 구체적 설계 문제 | Singleton, Observer, Facade |
| 스타일(Style) | 시스템 전체 구조 | 컴포넌트 구성 방식 | Layered, MSA, EDA |

### 5. 실무 제언

**품질속성 간 트레이드오프 관리**

- **챌린지**: 품질 속성들은 상호 충돌(Conflict)하는 경우가 많습니다. 보안을 강화할수록 성능이 저하되고(암호화 오버헤드), 가용성을 높이려면 비용이 급증하며(이중화 구성), 변경용이성을 극대화하면 아키텍처가 복잡해져 성능과 운영용이성이 저하됩니다.
- **제언**: ATAM(Architecture Tradeoff Analysis Method) 프로세스를 통해 **감도점(Sensitivity Point)**과 **트레이드오프점(Tradeoff Point)**을 식별하고, 이해관계자가 참여하여 우선순위를 합의해야 합니다. 예를 들어 "보안 강화로 응답시간이 50ms 증가하는 것을 허용할 것인가?"를 명시적으로 결정하고 기록(ADR)하는 것이 성숙한 아키텍처 거버넌스입니다.

**QAS를 기반으로 한 아키텍처 검증 자동화**

- **챌린지**: QAS는 설계 시에 작성되지만 실제로 시스템이 이를 충족하는지 검증하는 활동이 분리되어 있어 아키텍처 드리프트(Architectural Drift)가 발생합니다.
- **제언**: 각 QAS를 기반으로 **아키텍처 적합성 테스트(Fitness Functions)**를 자동화해야 합니다. 예를 들어 "변경 모듈 수 ≤ 3" 제약은 ArchUnit 테스트로, "응답시간 ≤ 2초" 제약은 Gatling/k6 성능 테스트로, "코드 커버리지 ≥ 80%" 제약은 JaCoCo로 CI/CD 파이프라인에서 자동 검증합니다. 이 접근법은 Neal Ford가 제시한 **진화적 아키텍처(Evolutionary Architecture)**의 핵심 실천 기법입니다.
