## 문제

소프트웨어 제품에 대한 가시성을 제공하는 애자일(Agile) 현장을 기술하시고 애자일 개발방법론인 스크럼(SCRUM)과 XP 대하여 아래 항목을 설명하시오.

- 가. 스크럼(SCRUM)의 개념 및 프로세스
- 나. XP(eXtreme Programming) 프로세스 및 12가지 실천사항(7개이상)
- 다. 사용자스토리(User Story), 스토리포인트(Story Point)

## 해설

### 출제 배경 및 의도

소프트웨어 아키텍처가 모놀리식에서 MSA로 분할되고 DevSecOps 시대가 도래하면서, 상위 기획과 하위 구현 간의 '개발 가시성(Visibility)' 확보는 프로젝트 성패의 결정적 요인이 되었습니다.

채점관이 애자일 헌장과 함께 스크럼 프로세스, XP 실천사항, 측정 도구들을 패키지로 묶어 출제하는 의도는 수험생이 애자일의 사상적 배경(헌장)을 이해하고, 관리 관점(Scrum)과 엔지니어링 관점(XP)으로 분리하여 실무에 정교하게 적용할 수 있는 '실천적 PM이자 아키텍트'의 자질을 갖추었는지 종합 검증하기 위함입니다.

스크럼과 XP는 독립된 방법론이 아닌 상호 보완 관계로, 스크럼의 관리 프레임워크 안에 XP의 엔지니어링 실천법을 바인딩해야만 진정한 품질 혁신이 일어납니다.

### 1. 애자일 헌장(Manifesto)과 가시성 확보

정 의  • 2001년 17명의 소프트웨어 개발 대가들이 정립한 선언문
       - 전통적 관료주의적 방법론의 폐해를 극복하기 위한 4대 핵심 가치

```
[애자일 4대 가치 — 오른쪽에 더 높은 가치를 둠]
프로세스·도구          < 개인과 상호작용
광범위한 문서          < 작동하는 소프트웨어
계약 협상             < 고객과의 협력
계획을 따르기          < 변화에 대응하기
      ↓
작동하는 SW를 짧은 주기마다 Demo로 노출 → 화이트박스형 가시성 확보
(vs 폭포수의 후반부 통합 전까지 블랙박스 유지)
```

- 애자일 헌장은 프로젝트 가시성(Visibility)을 매 Sprint 종료 시점 Demo로 상시 제공하는 사상적 기반

### 2. 스크럼(SCRUM) 개념과 프로세스

1) 스크럼 개요

정 의  • 팀원 간 유기적 소통·협업을 중심으로, 정해진 Sprint 동안 목표를 달성하기 위해
       - 스스로 조직화된(Self-Organizing) 팀 기반의 복잡계 프로젝트 관리 프레임워크
       - 3대 역할자: 제품소유자(PO), 스크럼마스터(SM), 개발팀

```
[제품 백로그] ──▶ [스프린트 계획] ──▶ [스프린트 백로그] ──▶ [데일리 스크럼]
                                                                    │
[스프린트 회고] ◀── [스프린트 리뷰] ◀── [제품 증분] ◀────────────┘
(개선점 도출)        (Demo 시연)       (Increment)
```

2) 스크럼 프로세스 7단계

| 구분 | 단계 | 주요 활동 |
|------|------|----------|
| ① | 제품 백로그 작성 | PO가 비즈니스 가치에 따라 정렬한 시스템 요구사항 목록 수립 |
| ② | 스프린트 계획 회의 | 개발팀이 Sprint(1~4주 주기) 수행 가능 범위 논의·할당 |
| ③ | 스프린트 백로그 도출 | Sprint 내 처리할 구체적 Task 목록을 칸반에 배치 |
| ④ | 데일리 스크럼 | 매일 15분, 어제 한 일·오늘 할 일·장애 요소 공유, 위험 조기 통제 |
| ⑤ | 제품 증분 생성 | Sprint 종료 시 실행 가능한 수준의 Increment 빌드 |
| ⑥ | 스프린트 리뷰 | 고객·이해관계자 앞 Demo 시연, 피드백 접수 |
| ⑦ | 스프린트 회고 | SM 주도로 프로세스 잘된 점·개선점 도출, 차기 Sprint 반영 |

- 스크럼의 7단계는 자기조직화 팀이 Sprint 내에서 목표를 달성하고 지속적으로 개선하는 경험적(Empirical) 프로세스 제어 루프를 구성함

### 3. XP 실천사항과 사용자 스토리

1) XP(eXtreme Programming) 12가지 핵심 실천사항

정 의  • 요구사항 변경이 심한 소규모 프로젝트에서 SW 품질과 생산성을 극대화하기 위해
       - 검증된 최우수 엔지니어링 기법들을 극단적(eXtreme)으로 실천하는 개발자 중심 방법론

| 구분 | 실천사항 | 핵심 설명 |
|------|---------|---------|
| ① | 짝 프로그래밍 | 내비게이터+드라이버 역할 분담, 실시간 코드 리뷰로 결함 원천 차단 |
| ② | TDD | 실패 테스트 케이스 먼저 작성, 결함 예방 및 정밀 설계 유도 |
| ③ | 리팩토링 | 외부 동작 유지·내부 구조 지속 개선, 기술 부채 상시 제거 |
| ④ | 지속적 통합(CI) | 하루 수차례 자동 빌드·통합 테스트, 형상 충돌 방지 |
| ⑤ | 계획 게임 | 비즈니스 가치 vs. 기술 구현 비용 저울질, 릴리즈 계획 결정 |
| ⑥ | 소규모 릴리즈 | 작은 단위를 자주 배포하여 시장 피드백 선점 |
| ⑦ | 공동 소유권 | 팀원 전체가 모든 코드를 수정·책임질 수 있도록 개방 |
| ⑧ | 현장 고객 | 발주처 전문가 개발 공간 상주, 의문 사항 실시간 협의 |
| ⑨ | 단순한 설계 | 현재 요구사항만 만족하는 최단결선 코드, 오버 엔지니어링 지양 |
| ⑩ | 시스템 은유 | 프로젝트 전체 구성을 공통 비유적 용어 체계로 통일 |
| ⑪ | 코딩 표준 | 팀 전체 일관된 규칙으로 가독성·유지보수성 극대화 |
| ⑫ | 지속 가능한 속도 | 주 40시간 근무 준수, 개발자 최상의 컨디션 보존 |

- XP의 12가지 실천사항은 짝 프로그래밍·TDD·CI 등 엔지니어링 품질을 직접 강화하는 도구로, 스크럼의 관리 프레임워크와 결합할 때 진정한 품질 혁신이 실현됨

2) 사용자 스토리와 스토리포인트

| 구분 | 개념 | 핵심 내용 |
|------|------|---------|
| 사용자 스토리 | 사용자 관점 요구사항 기술 | 형식: "[누구]로서, [무엇]을 원한다. 왜냐하면 [가치] 때문이다." |
| INVEST 원칙 | 좋은 스토리 기준 | Independent·Negotiable·Valuable·Estimable·Small·Testable |
| 스토리포인트 | 상대적 규모 추정 단위 | 노력·복잡성·리스크를 종합 고려한 상대 크기 산정 |
| 플래닝 포커 | 피보나치 기반 합의 도출 | 1,2,3,5,8,13... 단위로 팀 합의, 개인 숙련도 갭 해소 |

- 사용자 스토리와 스토리포인트는 고객 관점의 요구사항을 팀이 공동으로 추정하는 도구로, INVEST 원칙을 준수한 스토리만이 Sprint 계획과 Velocity 측정에 유효하게 활용될 수 있음

3) 스크럼 vs. XP 비교

| 비교 항목 | 스크럼(SCRUM) | XP(eXtreme Programming) |
|----------|-------------|------------------------|
| 핵심 포커스 | 프로젝트 관리·팀 운영·협업 거버넌스 | 실전 엔지니어링·소스코드 품질 향상 |
| 요구사항 통제 | Sprint 도중 원칙적으로 변경 불가 | 이터레이션 도중 동일 비용 스토리 교체 가능 |
| 반복 주기 | 2~4주 Sprint | 1~2주 극단적으로 짧은 릴리즈 |
| 핵심 역할자 | PO, SM, 개발팀 | 고객, 개발자, 코치 |

- 스토리포인트 기반의 팀 속도(Velocity)로 번다운 차트를 가시화해야만 Scope Creep을 과학적으로 통제하는 애자일 거버넌스가 완성됨.  "끝"

### 실무 제언

**스크럼+XP 통합 운영 — 관리의 뼈대에 엔지니어링 살을 붙여야 함**
- **챌린지**: 스크럼을 통해 Sprint 주기와 칸반 보드를 관리하더라도, 개발자들의 코드 베이스가 TDD·리팩토링·CI라는 XP 주춧돌 위에 올라타지 않으면 매 Sprint 종료 시점마다 깨진 빌드와 결함 가득한 Increment만 남게 됨
- **제언**: 스크럼 DoD(Definition of Done)에 TDD 커버리지 기준·리팩토링 체크·CI 빌드 성공을 필수 완료 조건으로 명시하여, XP 실천사항이 스크럼 프로세스와 구조적으로 결합되도록 거버넌스를 설계해야 함

**팀 속도(Velocity) 기반 과학적 스코프 크립 통제**
- **챌린지**: 스토리포인트를 도입했더라도 Velocity 측정과 번다운 차트 가시화 없이는 Sprint마다 범위가 묵시적으로 확장되는 Scope Creep을 통제하기 어려움
- **제언**: 최소 3~4 Sprint의 평균 Velocity를 기준으로 릴리즈 계획을 수립하고, 번다운 차트의 추세선이 목표 완료일과 괴리될 경우 즉시 PO와 함께 백로그 우선순위를 재조정하는 거버넌스 루프를 운영해야 함

**데일리 스크럼의 형식화 방지와 실질적 장애 제거**
- **챌린지**: 시간이 지남에 따라 데일리 스크럼이 단순 업무 보고 시간으로 전락하여 장애 요소(Impediment)가 식별되어도 SM이 적극적으로 제거하지 않는 '형식적 세레모니화' 현상이 발생함
- **제언**: SM은 데일리 스크럼에서 도출된 모든 장애 요소를 Impediment Backlog에 등록하고 24시간 내 제거 계획을 수립하는 SLA를 정립하여, 팀의 자기조직화 역량이 실질적으로 발현될 수 있는 환경을 보장해야 함
