**Software 위기 극복을 위한 3R 기법**에 대하여 서술하시오.

## 해설

> **소프트웨어 위기를 극복하는 세 가지 열쇠**  
> Reuse(재사용), Reverse Engineering(역공학), Restructuring(재구조화) — 3R

수십 년간 쌓인 레거시 시스템은 새로 짜기도, 버리기도 어렵습니다. 소프트웨어 위기(Software Crisis)의 본질은 개발 비용과 유지보수 비용이 통제를 벗어난다는 점인데, 3R은 바로 이 문제를 정면 돌파하는 공학적 처방입니다. 기존 자산을 최대한 살리면서 품질을 올리는 전략이기 때문에 기술사 시험에서 꾸준히 출제됩니다.

### 1. 출제 배경 및 의도

소프트웨어 위기(Software Crisis)라는 용어는 1968년 NATO 소프트웨어 공학 컨퍼런스에서 공식적으로 등장했습니다. 당시의 문제는 지금도 반복됩니다.

- **일정·비용 초과**: 대형 시스템 프로젝트의 70% 이상이 납기를 지키지 못함
- **품질 저하**: 출시 후 수정이 폭증하고 신뢰성 확보 불가
- **유지보수 불가**: 수십 년 된 COBOL 레거시 코드를 이해하는 인력이 사라짐
- **생산성 정체**: 하드웨어 성능은 무어의 법칙으로 기하급수적으로 향상되는데 소프트웨어 생산성은 정체

채점관이 이 문제를 출제한 의도는 단순히 3R 용어 정의 암기를 확인하는 것이 아닙니다. **소프트웨어 위기의 구조적 원인을 진단**하고, 각 R이 그 원인의 어느 부분을 해결하는지 연결 고리를 논리적으로 설명할 수 있느냐를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 소프트웨어 위기(Software Crisis)의 원인

소프트웨어 위기는 단순히 '코드가 복잡해서'가 아닙니다. 구조적 원인이 있습니다.

```text
[소프트웨어 위기 원인 구조]

복잡성(Complexity)
  └→ 소프트웨어 규모가 커질수록 복잡도는 비선형 증가
  └→ 브룩스(Brooks)의 법칙: "늦어진 프로젝트에 인력을 추가하면 더 늦어진다"

가시성 부재(Invisibility)
  └→ 하드웨어와 달리 소프트웨어는 눈에 보이지 않음
  └→ 진행 상황, 품질 파악이 어려움

변경성(Changeability)
  └→ 소프트웨어는 쉽게 변경할 수 있다는 인식 → 요구사항 무한 변경
  └→ 아키텍처 부패(Architecture Decay) 가속화

레거시 자산 누적
  └→ 수십 년 된 코드, 문서 없음, 개발자 없음
  └→ 재개발 비용도, 유지보수 비용도 감당 불가
```

#### 3R의 전체 구조와 상호 관계

```text
[3R 개념 맵]

기존 소프트웨어
       │
       ▼
┌─────────────────────────────────────┐
│  Reverse Engineering (역공학)        │
│  코드 → 설계 → 요구사항 복원          │
│  "이 코드가 무엇을 하는지 이해한다"   │
└──────────────┬──────────────────────┘
               │ 복원된 설계/명세
               ▼
┌─────────────────────────────────────┐
│  Restructuring (재구조화)            │
│  기능은 그대로, 구조만 개선           │
│  "더 이해하기 쉽고 유지보수하기 쉽게" │
└──────────────┬──────────────────────┘
               │ 개선된 컴포넌트
               ▼
┌─────────────────────────────────────┐
│  Reuse (재사용)                      │
│  검증된 컴포넌트를 다시 쓴다          │
│  "만들지 않고 조립한다"              │
└─────────────────────────────────────┘
```

### 3. 핵심 키워드 (Must-Have)

#### 1) Reuse (재사용) — 만들지 않고 조립한다

재사용(Reuse)은 이미 개발되고 검증된 소프트웨어 자산을 새로운 시스템 개발에 활용하는 것입니다.

**재사용의 수준 (레벨별 이해)**

```text
수준 1: 코드 복사 (낮은 재사용)
  └→ 함수나 클래스를 복사-붙여넣기
  └→ 나중에 수정이 두 곳에서 일어나야 해서 오히려 문제

수준 2: 라이브러리 재사용
  └→ 공통 함수 묶음을 import하여 사용
  └→ 예: Java의 Apache Commons, Python의 NumPy

수준 3: 컴포넌트 재사용
  └→ 독립적으로 배포 가능한 기능 단위
  └→ 예: React 컴포넌트, Spring Bean

수준 4: 프레임워크 재사용 (높은 재사용)
  └→ 개발 뼈대 자체를 재사용
  └→ 제어의 역전(IoC): 내 코드가 프레임워크를 호출하는 것이 아니라
     프레임워크가 내 코드를 호출
  └→ 예: Spring Framework, Django

수준 5: 아키텍처 패턴 재사용 (최고 수준)
  └→ 설계 구조 자체를 재활용
  └→ 예: MVC, MSA(마이크로서비스 아키텍처)
```

**재사용의 핵심 조건**

| 조건 | 설명 |
|------|------|
| 독립성 | 다른 컴포넌트에 의존하지 않을수록 재사용성 높음 |
| 일반성 | 특정 도메인에만 국한되지 않을수록 재사용성 높음 |
| 문서화 | 인터페이스와 사용법이 명확히 문서화되어 있어야 함 |
| 신뢰성 | 충분한 테스트로 검증된 컴포넌트여야 함 |

#### 2) Reverse Engineering (역공학) — 코드에서 설계를 복원한다

역공학(Reverse Engineering)은 기존 소프트웨어의 소스코드나 바이너리를 분석하여 설계(Design), 나아가 요구사항(Requirements)을 복원하는 과정입니다.

비유하자면, 이미 만들어진 요리를 먹어보고 레시피를 역추적하는 것과 같습니다. 문서가 없는 레거시 시스템을 이해하거나 유지보수하기 위해 반드시 필요합니다.

```text
[역공학 단계]

소스코드 (구현 레벨)
       │
       │ 코드 분석 (Code Analysis)
       │ 복잡도 측정, 제어 흐름 파악
       ▼
설계 문서 (설계 레벨)
  - 클래스 다이어그램 복원
  - 시퀀스 다이어그램 도출
  - 컴포넌트 의존 관계 파악
       │
       │ 추상화 (Abstraction)
       ▼
요구사항 / 비즈니스 규칙 (요구사항 레벨)
  - "이 코드가 왜 존재하는가?"
  - 비즈니스 로직 명세화
```

**역공학 도구 예시**

- **정적 분석 도구**: SonarQube, Understand (코드 구조 시각화)
- **UML 역생성**: IntelliJ IDEA, Enterprise Architect (코드→다이어그램)
- **의존성 분석**: JDepend, Structure101

역공학만으로는 코드가 개선되지 않습니다. 역공학은 이해를 위한 수단이며, 실제 코드 개선은 재구조화(Restructuring)로 이어집니다. 역공학 + 재구조화를 합쳐 **재공학(Re-Engineering)**이라고도 합니다.

#### 3) Restructuring (재구조화) — 기능은 그대로, 구조를 개선한다

재구조화(Restructuring)는 소프트웨어의 외부 동작(기능)은 변경하지 않으면서 내부 구조를 개선하여 유지보수성과 가독성을 높이는 활동입니다.

리팩토링(Refactoring)이 소스코드 레벨의 재구조화라면, 재구조화는 더 넓은 범위(아키텍처, 모듈 구조)까지 포함하는 개념입니다.

```text
[재구조화 대상과 목표]

대상 | 문제 상황 | 재구조화 후
-----|----------|----------
스파게티 코드 | goto문, 복잡한 제어 흐름 | 구조적 프로그래밍 적용
데이터 중복 | 동일 데이터가 여러 곳에 흩어짐 | 정규화/데이터 모델 개선
모듈 결합도 | 하나 고치면 다른 곳이 터짐 | 인터페이스 기반 결합 완화
문서 부재 | 코드만 있고 설명 없음 | 주석, 문서 보강
```

**재구조화 vs 리팩토링 비교**

| 구분 | 재구조화(Restructuring) | 리팩토링(Refactoring) |
|------|------------------------|----------------------|
| 범위 | 아키텍처/모듈 레벨 | 소스코드(클래스/함수) 레벨 |
| 트리거 | 레거시 시스템 현대화 | 일상적 개발 활동 |
| 도구 | 아키텍처 분석 도구 | IDE 자동 리팩토링 |
| 기간 | 수 주 ~ 수 개월 | 수 분 ~ 수 시간 |

### 4. 맥락과 비교

**재공학(Re-Engineering)과 3R의 관계**

재공학은 역공학(Reverse Engineering)과 재구조화(Restructuring)를 결합한 개념으로, 기존 시스템을 분석하여 더 나은 형태로 재구성하는 전체 프로세스를 말합니다.

```text
재공학(Re-Engineering) = 역공학 + 재구조화
    └→ 기존 시스템 이해(역공학) 후
       새로운 설계로 변환(재구조화)
```

**3R과 신규 개발의 비용 비교**

| 전략 | 초기 비용 | 위험도 | 결과물 품질 | 권장 상황 |
|------|----------|--------|------------|----------|
| 신규 개발 | 높음 | 높음 | 높음 | 레거시 구제 불능, 기술 완전 교체 시 |
| 재공학(Reverse+Restructuring) | 중간 | 중간 | 중간~높음 | 레거시 가치 있음, 점진적 개선 시 |
| 재사용 강화 | 낮음 | 낮음 | 중간 | 유사 시스템 반복 개발 시 |

### 5. 실무 제언

**레거시 현대화 전략으로서의 3R 적용**

공공기관이나 금융권에는 수십 년 된 COBOL/메인프레임 시스템이 여전히 운영 중입니다. 이를 무작정 신규 개발로 대체하려다 실패한 사례가 수없이 많습니다. 

현실적인 접근은 다음 순서를 따릅니다.

1. **역공학 우선**: 기존 시스템을 이해하지 못한 채 신규 개발을 시작하면 비즈니스 규칙을 놓치게 됩니다. 먼저 역공학으로 숨겨진 비즈니스 로직을 문서화합니다.
2. **재구조화로 안정화**: 전면 교체 전에 재구조화로 구조를 정리하면 마이그레이션 리스크가 줄어듭니다.
3. **재사용 자산화**: 검증된 비즈니스 로직을 API나 컴포넌트로 포장하여 재사용 자산으로 등록합니다.
4. **스트랭글러 피그(Strangler Fig) 패턴**: 레거시를 한 번에 교체하는 대신, 새로운 시스템이 레거시를 조금씩 대체하며 공존하는 전략으로 위험을 최소화합니다.

**생성형 AI 시대의 재사용 전략 변화**

ChatGPT, GitHub Copilot과 같은 LLM 기반 코드 생성 도구가 보편화되면서 재사용의 개념이 확장되고 있습니다. 기존의 코드 라이브러리 재사용에서 '프롬프트 재사용', '파인튜닝된 도메인 모델 재사용'으로 패러다임이 이동하고 있습니다. 그러나 생성된 코드의 품질을 보장하기 위해서는 여전히 역공학적 코드 리뷰와 재구조화 역량이 필수적입니다.
