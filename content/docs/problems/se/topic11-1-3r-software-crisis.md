## 해설

### 출제 배경 및 의도

소프트웨어 위기(Software Crisis)는 1968년 NATO 컨퍼런스에서 공식 등장한 개념으로, 일정·비용 초과, 품질 저하, 유지보수 불가, 생산성 정체라는 구조적 문제를 지칭한다. 수십 년간 쌓인 레거시 시스템은 새로 짜기도, 버리기도 어렵다.

3R(Reuse·Reverse Engineering·Restructuring)은 기존 소프트웨어 자산을 최대한 살리면서 품질을 높이는 공학적 처방으로, 기술사 시험에서 꾸준히 출제된다. 채점관은 소프트웨어 위기의 구조적 원인을 진단하고 각 R이 그 원인의 어느 부분을 해결하는지 논리적 연결 고리를 설명할 수 있는지를 검증한다.

### 1. 소프트웨어 위기 극복을 위한 3R 개요

정 의  • 3R을 레거시 소프트웨어의 자산을 역공학(Reverse Engineering)으로 이해하고, 재구조화(Restructuring)로 개선하여, 재사용(Reuse)으로 새로운 시스템에 활용하는 소프트웨어 위기 극복 기법 체계라 한다.
       - 신규 개발 대비 비용·위험을 최소화하면서 기존 자산의 가치를 보전하는 것이 핵심 목표이다.

```
[3R 상호 관계도]

기존 레거시 소프트웨어
         │
         ▼
┌────────────────────────────────┐
│  Reverse Engineering (역공학)   │
│  코드→설계→요구사항 역방향 복원  │
│  "이 코드가 무엇을 하는지 이해"  │
└──────────────┬─────────────────┘
               │ 복원된 설계/명세
               ▼
┌────────────────────────────────┐
│  Restructuring (재구조화)       │
│  기능 유지 + 내부 구조 개선      │
│  "더 이해하기 쉽고 유지보수 용이" │
└──────────────┬─────────────────┘
               │ 개선된 컴포넌트
               ▼
┌────────────────────────────────┐
│  Reuse (재사용)                 │
│  검증된 컴포넌트 재활용          │
│  "만들지 않고 조립한다"         │
└────────────────────────────────┘
  역공학 + 재구조화 = 재공학(Re-Engineering)
```

- 3R은 순차적으로 연결되며, 역공학 없이 재구조화는 맹목적이고, 재구조화 없이 재사용은 위험하다.

### 2. 3R 핵심 구성요소

1) Reverse Engineering (역공학)

| 구분 | 항목 | 설명 |
|------|------|------|
| 정의 | 역방향 복원 | 소스코드→설계→요구사항을 역방향으로 복원 |
| 단계 | ① 코드 분석 | 복잡도 측정, 제어 흐름 파악 |
| | ② 설계 복원 | 클래스 다이어그램, 컴포넌트 의존 관계 |
| | ③ 요구사항 추출 | 비즈니스 로직 명세화 |
| 도구 | 정적 분석 | SonarQube, Understand |
| | UML 역생성 | IntelliJ IDEA, Enterprise Architect |
| 한계 | 이해 수단 | 코드 개선은 재구조화로 이어져야 함 |

- 역공학은 문서 없는 레거시 이해의 유일한 수단이며, 재구조화의 전제 조건이다.

2) Restructuring (재구조화)

| 구분 | 재구조화(Restructuring) | 리팩토링(Refactoring) |
|------|----------------------|---------------------|
| 범위 | 아키텍처/모듈 레벨 | 소스코드(클래스/함수) 레벨 |
| 트리거 | 레거시 시스템 현대화 | 일상적 개발 활동 |
| 기간 | 수 주 ~ 수 개월 | 수 분 ~ 수 시간 |
| 공통점 | 기능(외부 동작) 불변, 내부 구조 개선 ||

- 재구조화는 스파게티 코드→구조적 프로그래밍, 높은 결합도→인터페이스 기반 결합으로 유지보수성을 회복한다.

3) Reuse (재사용) — 재사용 수준별 특성

| 수준 | 유형 | 예시 | 재사용성 |
|------|------|------|---------|
| 1 | 코드 복사 | Copy-Paste | 낮음(수정 중복) |
| 2 | 라이브러리 | Apache Commons, NumPy | 중간 |
| 3 | 컴포넌트 | React Component, Spring Bean | 높음 |
| 4 | 프레임워크(IoC) | Spring Framework, Django | 매우 높음 |
| 5 | 아키텍처 패턴 | MVC, MSA | 최고 수준 |

- 재사용의 핵심 조건: 독립성(낮은 결합도), 일반성(도메인 비종속), 문서화, 신뢰성(충분한 테스트)

### 3. 3R 전략 비교 및 레거시 현대화 로드맵

1) 전략별 비교

| 전략 | 초기 비용 | 위험도 | 결과물 품질 | 권장 상황 |
|------|---------|--------|----------|---------|
| 신규 개발 | 높음 | 높음 | 높음 | 레거시 구제 불능, 기술 완전 교체 |
| 재공학(역공학+재구조화) | 중간 | 중간 | 중간~높음 | 레거시 가치 있음, 점진적 개선 |
| 재사용 강화 | 낮음 | 낮음 | 중간 | 유사 시스템 반복 개발 |

2) 레거시 현대화 권장 순서

```
① 역공학 우선: 비즈니스 로직 문서화 (신규 개발 전 필수)
② 재구조화로 안정화: 마이그레이션 리스크 최소화
③ 재사용 자산화: 검증된 로직을 API/컴포넌트로 등록
④ 스트랭글러 피그 패턴: 레거시를 조금씩 대체하며 공존
```

- 역공학 없이 신규 개발을 시작하면 숨겨진 비즈니스 규칙을 놓쳐 프로젝트가 실패한다.  "끝"

### 실무 제언

**레거시 COBOL 시스템 현대화 실패 패턴**
- **챌린지**: 금융·공공기관의 COBOL 레거시를 무작정 신규 개발로 대체하다 수천억 원 손실이 발생한다.
- **제언**: 역공학으로 숨겨진 비즈니스 로직을 먼저 문서화하고, 스트랭글러 피그 패턴으로 점진적 전환하면 위험을 최소화할 수 있다.

**재사용 자산 관리 부재**
- **챌린지**: 조직마다 같은 기능을 별도로 개발하여 생산성이 정체된다.
- **제언**: 검증된 비즈니스 로직을 API나 내부 라이브러리로 포장하고 내부 패키지 저장소(Nexus, Artifactory)에 등록하여 재사용 자산을 체계적으로 관리한다.

**생성형 AI 시대의 재사용 패러다임 변화**
- **챌린지**: GitHub Copilot 등 LLM 기반 코드 생성 도구가 보편화되면서 생성 코드의 품질 보장이 어렵다.
- **제언**: 생성 코드에도 역공학적 코드 리뷰와 재구조화 역량을 적용하며, 프롬프트 재사용과 도메인 파인튜닝 모델을 조직의 재사용 자산으로 관리한다.
