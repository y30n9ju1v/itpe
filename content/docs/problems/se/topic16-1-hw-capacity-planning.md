## 해설

> **"얼마나 큰 서버가 필요한가?"에 과학적으로 답하는 방법**  
> 예상 업무량(TPS)과 자원 소비량을 기반으로 CPU·Memory·Disk 용량을 산정하는 체계

정보시스템 구축 사업의 초반에 서버 규모를 너무 작게 잡으면 서비스 오픈 후 성능 장애가 발생하고, 너무 크게 잡으면 예산을 낭비하게 됩니다. H/W 용량산정은 이 딜레마를 해결하기 위해 비즈니스 요구사항(동시 사용자 수, 거래량)을 정량적 수식으로 변환하여 필요한 H/W 규모를 도출하는 공학적 접근법입니다. ITA(정보기술아키텍처) 기반 사업에서는 이 산정 결과가 사업 예산 편성의 근거가 됩니다.

### 1. 출제 배경 및 의도

클라우드 시대에도 공공 및 금융 분야의 핵심 시스템은 온프레미스(On-Premises) 서버를 사용하는 경우가 많고, 클라우드 환경에서도 인스턴스 유형(CPU 코어 수, 메모리 크기) 선정에 동일한 용량산정 논리가 적용됩니다. 채점관은 각 자원(CPU, Memory, Disk)별 산정 공식을 알고, 그 공식에 등장하는 요소(TPS, 피크 배율, 목표 활용률 등)의 의미를 설명할 수 있는지를 봅니다.

### 2. 개념의 본질과 작동 메커니즘

H/W 용량산정의 핵심 흐름은 다음과 같습니다.

```text
[H/W 용량산정 전체 흐름]

① 업무 요구사항 분석
   → 일 최대 트랜잭션 수, 동시 사용자 수, 데이터 증가율 등

② TPS(초당 트랜잭션) 산정
   → TPS = 일 최대 거래건수 ÷ 유효시간(초) × 피크 배율

③ 자원별 용량 계산
   → CPU: TPS × 트랜잭션당 CPU시간 ÷ 목표 CPU 활용률
   → Memory: 동시 사용자 수 × 세션당 메모리 + OS/미들웨어
   → Disk: (데이터량 × 증가율 × 보존기간) × 여유율

④ 여유율·이중화 반영
   → 피크 처리 능력, N+1 이중화, DR(재해복구) 구성 고려

⑤ 최종 H/W 사양 결정
   → 서버 대수, CPU 코어 수, 메모리 용량, 스토리지 용량
```

### 3. 핵심 키워드 (Must-Have)

#### 1) H/W 규모산정의 개념 및 결정점

**개념**: 목표 시스템의 업무 처리 능력 요구사항(성능 목표)을 만족하는 하드웨어 자원(CPU, Memory, Storage, Network)의 최소 필요 규모를 논리적·정량적으로 도출하는 프로세스.

**결정점 (Decision Point)**: H/W 규모산정 결과에 영향을 미치는 핵심 변수들

| 결정점 | 설명 | 비고 |
|--------|------|------|
| **피크 배율** | 최대 트래픽이 평균 대비 몇 배인가 | 보통 2~3배 적용 |
| **목표 CPU 활용률** | CPU를 어느 수준까지 사용할 것인가 | 보통 60~70% (여유율 확보) |
| **동시 사용자 수** | 동시에 시스템을 사용하는 사용자 수 | 전체 사용자의 5~10% 적용 |
| **트랜잭션당 자원 소비량** | 거래 1건 처리에 소비되는 CPU·메모리 | 벤치마크 또는 유사 시스템 참조 |
| **데이터 증가율** | 연간 데이터 증가량 | 보존 기간과 함께 스토리지 산정에 사용 |

#### 2) 규모산정 대상

```text
[H/W 규모산정 대상 영역]

┌─────────────────────────────────────────┐
│              정보시스템 H/W              │
│                                         │
│  ① 서버 (Server)                         │
│     - Web 서버                           │
│     - WAS (Web Application Server)      │
│     - DB 서버                            │
│     - 배치/연계 서버                      │
│                                         │
│  ② 스토리지 (Storage)                    │
│     - DB 데이터 영역                      │
│     - 백업 영역                           │
│     - 아카이브 영역                       │
│                                         │
│  ③ 네트워크 장비                          │
│     - L4 스위치 (로드밸런서)               │
│     - 방화벽, IDS/IPS                    │
└─────────────────────────────────────────┘
```

#### 3) CPU 용량 산정

**핵심 공식**:

```text
필요 CPU 용량(tpmC 또는 코어 수)
    = TPS × 트랜잭션당 CPU 처리시간(초)
      ÷ 목표 CPU 활용률(%)

여기서:
  TPS = Peak 거래 건수 ÷ 유효시간(초) × 피크 배율
  목표 CPU 활용률: 통상 0.6~0.7 (60~70%)
  피크 배율: 통상 2~3배
```

**예시 계산**:
- 일 최대 거래: 100만 건
- 유효시간: 8시간(28,800초)
- 피크 배율: 2배
- 트랜잭션당 CPU시간: 0.01초
- 목표 CPU 활용률: 60%

```text
TPS = 1,000,000 ÷ 28,800 × 2 ≈ 69.4 TPS
필요 CPU 용량 = 69.4 × 0.01 ÷ 0.6 ≈ 1.16 CPU
→ 최소 2코어 이상 서버 필요 (이중화 고려 시 4코어)
```

#### 4) Memory 용량 산정

**핵심 공식**:

```text
필요 메모리 = (동시 사용자 수 × 세션당 메모리)
              + OS 메모리
              + 미들웨어(WAS) 메모리
              + DB 버퍼 캐시 메모리
              + 여유 메모리(20~30%)
```

- **동시 사용자 수**: 전체 등록 사용자의 5~10% 또는 최대 동시 접속자 분석값 적용
- **세션당 메모리**: 업무 유형에 따라 1~10MB 범위

#### 5) Disk(스토리지) 용량 산정

**핵심 공식**:

```text
필요 스토리지 = 초기 데이터량
                + (연간 데이터 증가량 × 보존 기간)
                + 인덱스 영역 (데이터의 20~30%)
                + 임시/작업 영역
                + RAID 오버헤드 (RAID-5: 25%, RAID-10: 50%)

최종 용량 = 상기 합계 × 여유율(1.2~1.5)
```

**성능 기준지 (CPU 및 스토리지)**

| 지표 | 설명 | 단위 |
|------|------|------|
| **tpmC** | 분당 신규 주문 트랜잭션 처리 수 (CPU 성능 기준) | 트랜잭션/분 |
| **IOPS** | 초당 I/O 작업 수 (스토리지 성능 기준) | I/O/초 |
| **처리량(Throughput)** | 초당 전송 데이터량 | MB/s |
| **응답시간(Response Time)** | 요청 후 응답까지 소요 시간 | ms |

### 4. 맥락과 비교

#### 전통적 On-Premise vs 클라우드 용량산정 비교

| 구분 | 전통적 서버 (On-Premise) | 클라우드 (IaaS/PaaS) |
|------|------------------------|---------------------|
| **산정 단위** | 서버 대수, 코어 수, GB | 인스턴스 유형 (t3.xlarge 등) |
| **확장 방식** | 수직 확장(Scale-up) 중심 | 수평 확장(Scale-out) 용이 |
| **피크 대응** | 피크 기준으로 과잉 프로비저닝 필요 | 오토스케일링으로 탄력적 대응 |
| **비용 구조** | 초기 CAPEX 집중 | 사용량 기반 OPEX |
| **용량산정 방식** | 상세 수식 계산 필요 | 부하 테스트 후 인스턴스 선택도 가능 |

#### 용량산정의 3대 원칙

1. **보수적 산정**: 불확실성이 높으면 여유율을 높여 과잉 프로비저닝
2. **단계별 확장 계획**: 3년 후 증가분까지 초기 설계에 반영
3. **피크 기준 설계**: 평균이 아닌 최대 부하 기준으로 설계

### 5. 실무 제언

**클라우드 환경에서의 용량산정 변화**

- **챌린지**: 퍼블릭 클라우드(AWS, Azure, GCP)에서는 오토스케일링(Auto Scaling)이 가능하기 때문에 "정확한 초기 용량산정"의 중요성이 과거보다 낮아진 것처럼 보입니다. 그러나 무분별한 오토스케일링은 예산을 통제 불가능한 수준으로 증가시킬 수 있습니다.
- **제언**: 클라우드에서는 **기준 인스턴스 유형(Baseline) + 오토스케일링 정책(Upper bound)**의 이중 접근이 필요합니다. 기준 인스턴스 선정을 위한 초기 용량산정 역량은 여전히 필수적입니다.

**기술사 시험 답안 작성 팁**

- CPU 용량 산정 공식 `TPS × CPU시간 ÷ 목표활용률`을 수식으로 명시하고, 피크 배율의 개념(평균 대비 최대 부하 배수)을 설명하면 고득점입니다.
- tpmC(CPU), IOPS(스토리지)라는 성능 기준 단위를 반드시 언급하십시오.
- 텍스트 다이어그램으로 "업무량 → TPS → CPU/Memory/Disk 산정"의 흐름을 시각화하면 채점관에게 구조적 이해력을 어필할 수 있습니다.
