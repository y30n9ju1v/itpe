## 해설

### 출제 배경 및 의도

정보시스템 구축 사업 초반에 서버 규모를 과소 산정하면 서비스 오픈 후 성능 장애가 발생하고, 과대 산정하면 예산을 낭비하게 된다. ITA(정보기술아키텍처) 기반 공공 사업에서는 H/W 용량산정 결과가 예산 편성의 공식 근거가 되므로 정확한 산정 역량이 필수이다. 클라우드 환경에서도 기준 인스턴스 유형 선정에 동일한 용량산정 논리가 적용된다.

채점관은 CPU·Memory·Disk 각 자원별 산정 공식과 결정점(TPS·피크 배율·목표 활용률), 성능 기준 단위(tpmC·IOPS), 그리고 클라우드 환경과의 비교를 체계적으로 서술할 수 있는지 평가한다.

### 1. H/W 용량산정 개요

정 의  • **H/W 용량산정**을 목표 시스템의 업무 처리 능력 요구사항을 만족하는 CPU·Memory·Storage·Network의 최소 필요 규모를 정량적으로 도출하는 프로세스
       - ITA 기반 사업에서 H/W 규모산정 결과는 예산 편성의 공식 근거

```
[H/W 용량산정 전체 흐름]

① 업무 요구사항 분석
   → 일 최대 트랜잭션 수·동시 사용자 수·데이터 증가율

② TPS(초당 트랜잭션) 산정
   → TPS = 일 최대 거래건수 ÷ 유효시간(초) × 피크 배율

③ 자원별 용량 계산
   → CPU: TPS × 트랜잭션당 CPU시간 ÷ 목표 CPU 활용률
   → Memory: 동시 사용자 수 × 세션당 메모리 + OS/WAS/버퍼
   → Disk: (초기 데이터 + 증가량 × 보존기간) × 여유율

④ 여유율·이중화 반영
   → N+1 이중화·DR(재해복구) 구성 고려

⑤ 최종 H/W 사양 결정
   → 서버 대수·CPU 코어 수·메모리 용량·스토리지 용량
```

- 보수적 산정(여유율 확보)·피크 기준 설계·3년 증가분 반영이 용량산정 3대 원칙이다.

### 2. 핵심 내용 상세

1) H/W 규모산정 개념 및 결정점

| 구분 | 결정점 | 설명 | 통상 기준 |
|------|--------|------|---------|
| ① | 피크 배율 | 최대 트래픽이 평균 대비 몇 배인가 | 2~3배 |
| ② | 목표 CPU 활용률 | CPU를 어느 수준까지 사용할 것인가 | 60~70% |
| ③ | 동시 사용자 수 | 동시에 시스템 사용 사용자 수 | 전체의 5~10% |
| ④ | 트랜잭션당 자원 소비량 | 거래 1건 처리 CPU·메모리 소비 | 벤치마크·유사 시스템 참조 |
| ⑤ | 데이터 증가율 | 연간 데이터 증가량 | 보존 기간과 함께 스토리지 산정 |

2) 규모산정 대상 및 자원별 산정 공식

```
[H/W 규모산정 대상]

┌──────────────────────────────────────┐
│           정보시스템 H/W              │
│  ① 서버: Web·WAS·DB·배치/연계         │
│  ② 스토리지: DB 데이터·백업·아카이브   │
│  ③ 네트워크: L4 스위치·방화벽·IDS/IPS │
└──────────────────────────────────────┘

[CPU 용량 산정]

필요 CPU = TPS × 트랜잭션당 CPU 처리시간(초) ÷ 목표 CPU 활용률

  TPS = 1,000,000건 ÷ 28,800초 × 2(피크) ≈ 69.4 TPS
  필요 CPU = 69.4 × 0.01초 ÷ 0.6 ≈ 1.16 → 최소 2코어(이중화 시 4코어)

[Memory 용량 산정]

필요 메모리 = (동시 사용자 수 × 세션당 메모리)
              + OS + WAS(미들웨어) + DB 버퍼 캐시 + 여유(20~30%)

[Disk(스토리지) 용량 산정]

필요 스토리지 = 초기 데이터량
                + (연간 증가량 × 보존 기간)
                + 인덱스 영역(데이터의 20~30%)
                + RAID 오버헤드(RAID-5: 25%, RAID-10: 50%)
                × 여유율(1.2~1.5)
```

- tpmC(분당 신규 주문 트랜잭션 처리 수, TPC-C 벤치마크 기준 CPU 성능 지표)와 IOPS(초당 I/O 작업 수, 스토리지 성능 기준)가 산정 결과를 검증하는 핵심 성능 기준 단위이다.

### 3. 성능 기준지 및 환경별 용량산정 비교

1) CPU 및 스토리지 성능 기준지

| 지표 | 설명 | 단위 | 적용 |
|------|------|------|------|
| tpmC | 분당 신규 주문 트랜잭션 처리 수 (TPC-C 기준) | 트랜잭션/분 | CPU 성능 비교 |
| IOPS | 초당 I/O 작업 수 | I/O/초 | 스토리지 성능 기준 |
| 처리량(Throughput) | 초당 전송 데이터량 | MB/s | 네트워크·디스크 대역폭 |
| 응답시간(Response Time) | 요청 후 응답까지 소요 시간 | ms | 사용자 체감 성능 |

2) 전통적 On-Premise vs 클라우드 용량산정 비교

| 구분 | On-Premise | 클라우드 (IaaS/PaaS) |
|------|-----------|---------------------|
| 산정 단위 | 서버 대수·코어 수·GB | 인스턴스 유형 (t3.xlarge 등) |
| 확장 방식 | 수직 확장(Scale-up) 중심 | 수평 확장(Scale-out) 용이 |
| 피크 대응 | 피크 기준 과잉 프로비저닝 | 오토스케일링 탄력 대응 |
| 비용 구조 | 초기 CAPEX 집중 | 사용량 기반 OPEX |

- 클라우드 환경에서도 기준 인스턴스 유형(Baseline) 선정을 위한 초기 용량산정 역량은 여전히 필수이며, 오토스케일링 Upper bound(비용 상한) 설정에 직접 활용된다.  "끝"

### 실무 제언

**클라우드 오토스케일링과 비용 통제**
- **챌린지**: 클라우드 오토스케일링이 가능하다는 이유로 초기 용량산정을 생략하면, 무분별한 인스턴스 증가로 예산이 통제 불가능한 수준으로 증가할 수 있다.
- **제언**: 기준 인스턴스 유형(Baseline) + 오토스케일링 정책(Upper bound)의 이중 접근이 필요하다. 기준 인스턴스 선정을 위한 초기 용량산정은 클라우드 환경에서도 필수 역량이다.

**피크 배율 산정의 정확도 향상**
- **챌린지**: 신규 시스템은 유사 시스템 참조 데이터가 없어 피크 배율 산정이 주관적으로 결정되는 경우가 많다.
- **제언**: 유사 도메인의 트래픽 패턴 분석 데이터(공공 사업은 한국지능정보사회진흥원 데이터 활용)를 참조하고, 부하 테스트(Load Test) 결과로 초기 산정값을 검증하는 2단계 접근법을 적용해야 한다.

**3년 증가분 반영과 비용 최적화**
- **챌린지**: 3년 후 증가분까지 초기에 모두 프로비저닝하면 초기 구축 비용이 과다해지고, 초기 자원 활용률이 낮아 비효율이 발생한다.
- **제언**: 초기(Year 1) + 확장(Year 2~3) 분리 구성을 사전 설계하고, 클라우드는 오토스케일링으로, On-Premise는 모듈형 서버 증설(블레이드 서버)로 단계적 확장 계획을 구체화하는 것이 최적이다.
