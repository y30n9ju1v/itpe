## 해설

### 출제 배경 및 의도

초거대 클라우드 인프라와 MSA 환경에서 시스템 복잡성은 직관적 제어의 임계치를 넘어섰다. 서비스 장애가 즉각 수십억 원 손실과 브랜드 신뢰도 추락으로 이어지는 초연결 사회에서, "장애 없는 완벽한 시스템"이라는 환상을 버리고 "모든 시스템은 반드시 실패한다(Design for Failure)"는 대전제를 수용해야 한다.

채점관의 출제 의도는 안정성(Reliability)과 배포 속도(Velocity) 간 트레이드오프를 에러 버젯(Error Budget)이라는 수학적 기법으로 정량화·통제하는 거버넌스 역량을 갖추었는지 검증하는 것이다.

### 1. SRE(Site Reliability Engineering) 개요

정 의  • DevOps가 "개발-운영 협력"이라는 철학적 선언(interface)이라면, SRE는 구글이 소프트웨어 공학 방법론으로 정교하게 구현한 실전 구현체(class SRE implements DevOps)
       - 운영(Operations)의 전 과정을 자동화하고, 정량화된 지표 기반 제어로 복잡계 시스템 가동성을 유지하는 엔지니어링 체계

```
[SRE 핵심 지표 구동 체계]

  측정          ──▶  SLI (Service Level Indicator)
  (무엇을 잴까?)       지연시간·에러율·처리량 정량 지표

  목표 설정     ──▶  SLO (Service Level Objective)
  (얼마나 잘?)         "API 응답 200ms 이하 비율 99.9% 이상"

  계약          ──▶  SLA (Service Level Agreement)
  (법적 의무)          SLO 미달 시 환불·배상 등 외부 계약

                      ▼
             Error Budget = 100% - SLO
             (허용 가능한 불완전함의 크기)

  버젯 잔여  ──▶  개발팀: 공격적 신규 기능 배포 허용
  버젯 소진  ──▶  신규 배포 자동 차단, 안정화 강제 투입
```

- SLO는 내부 목표, SLA는 외부 계약 — SLO를 SLA보다 엄격하게 설정해야 완충 지대가 확보된다

### 2. SRE 핵심 구성 요소

1) 에러 버젯(Error Budget) 메커니즘

| 구분 | 항목 | 설명 |
|------|------|------|
| 공식 | Error Budget = 100% - SLO | SLO 99.9% 시 에러 버젯 0.1% |
| 버젯 잔여 | 배포 허용 | 개발팀 공격적 신규 기능 릴리즈·아키텍처 변경 가능 |
| 버젯 소진 | 배포 자동 차단 | 안정화 코드·버그 수정·SRE 자동화 고도화 강제 투입 |
| 거버넌스 | 경영진 명문화 | 에러 버젯 소진 시 Deployment Lock 경영진 합의 필수 |

- 에러 버젯은 개발 속도와 운영 안정성 간 갈등을 감정이 아닌 수치로 중재하는 과학적 제어 장치다

2) Toil 통제 및 관측 가능성(Observability)

| 구분 | 항목 | 설명 |
|------|------|------|
| Toil 통제 | 반복 업무 50% 상한 | 수동·반복적 운영 업무(Toil)에 50% 이상 투입 금지 |
| Toil 전환 | 엔지니어링 투자 | 나머지 50%는 IaC·아키텍처 개선 등 순수 엔지니어링 |
| Observability | 3대 기둥 | Metrics(지표)·Logs(로그)·Traces(분산 추적) 통합 관리 |
| Blameless | 자책 없는 회고 | 장애 시 시스템 결함 분석·기록·공유, 인원 비난 금지 |

- Observability는 단순 모니터링을 넘어 복잡계 시스템 내부 상태를 완벽히 추적하는 능동적 체계다

### 3. DevOps vs SRE 비교 및 확장 트렌드

1) DevOps와 SRE 관계

| 구분 | DevOps | SRE |
|------|--------|-----|
| 개념적 위상 | 추상적 사상·문화·철학적 선언 | 사상을 실체화한 구체적 엔지니어링 실천법 |
| 관계 매핑 | interface DevOps (추상 인터페이스) | class SRE implements DevOps (구현 클래스) |
| 중심 대상 | 조직 구조 변화, Silo 해소 | 소스코드 기반 자동화, 수학적 지표 통제 |
| 갈등 해결 | 상호 소통, 공동 책임 의식 고취 | 에러 버젯 정책 기반 강제 제어 |
| 품질 측정 | CI/CD 파이프라인 성숙도 | SLI/SLO 달성 비율, Toil 감소율 |

- SRE는 DevOps라는 레시피를 실제 주방에서 구현하는 방법론으로, 둘은 상호 보완적 관계다.  "끝"

### 실무 제언

**에러 버젯 거버넌스 경영진 합의 선행**
- **챌린지**: 에러 버젯 소진에도 경영진이 "비즈니스 우선"으로 규칙을 깨뜨리는 순간 SRE는 24시간 당직 밤샘 운영(SM) 구조로 퇴화한다
- **제언**: SRE 도입 초기에 "에러 버젯 소진 시 배포 자동 잠금(Deployment Lock)" 거버넌스 규칙을 경영진 공표 문서에 명문화하고 전사 합의를 선행한다 — 이것이 없는 SRE는 허울 좋은 구호에 불과하다

**플랫폼 엔지니어링(Platform Engineering)과 정렬**
- **챌린지**: 엔터프라이즈 규모 확장 시 스크럼 팀마다 SRE 엔지니어 배치는 인력 병목과 중복 투자를 초래하며 개발자 인지 부하가 가중된다
- **제언**: SRE 팀이 개별 장애 대응자 역할에서 벗어나 개발자가 자율적으로 인프라 프로비저닝·SLO 설정·텔레메트리 대시보드를 생성할 수 있는 내부 개발자 플랫폼(IDP)의 황금 경로(Golden Path)를 빌드업하는 가이드 공급자로 진화해야 한다
