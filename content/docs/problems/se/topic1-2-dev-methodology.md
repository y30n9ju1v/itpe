## 해설

### 출제 배경 및 의도

현대 비즈니스는 '속도와 유연성'이 생명입니다. 클라우드, MSA, 생성형 AI의 등장으로 시스템의 변화 주기가 극단적으로 짧아지면서, 과거의 무겁고 엄격한 통제 중심의 방법론은 현업에서 외면받고 있습니다.

채점관들이 구조적 방법론 같은 고전 패러다임부터 현대의 애자일까지의 발전과정을 묻는 이유는 현업에서 발생하는 "무늬만 애자일(Fake Agile)" 현상에 대한 경고이자, 비즈니스 특성에 맞춰 방법론을 스스로 조합하고 재정의할 수 있는 방법론 테일러링(Tailoring) 능력이 있는지 검증하기 위함입니다.

과거 패러다임의 한계를 명확히 알아야 현재 마주한 프로젝트의 문제점을 진단하고 치료할 수 있기 때문에, 발전과정 전체를 통시적으로 파악하는 역량이 핵심입니다.

### 1. 소프트웨어 개발방법론 개요

정 의  • 소프트웨어 생명주기(SDLC) 전 과정에서 생산성과 품질을 높이기 위해
       - 수행 절차, 방법, 산출물, 기법, 도구들을 체계적으로 정리한 패러다임

```
[1970년대: 구조적]   ──▶ 프로세스 중심, 하향식 기능 분할 (DFD)
       │
[1980년대: 정보공학] ──▶ 데이터 중심, 전사적 자원 모델링 (ISP, ERD)
       │
[1990년대: 객체지향] ──▶ 데이터+프로세스 융합, 현실 세계 모델링 (UML)
       │
[2000년대: CBD]      ──▶ 재사용성 극대화, 컴포넌트 조립 (Component Assembly)
       │
[2000년대~: 애자일]  ──▶ 지속적 소통, 불확실성 수용, 피드백 (Scrum, XP)
```

- 방법론의 발전 축은 '개발의 중심축'과 '불확실성 대응 방식'의 두 차원을 따라 진화함

### 2. 방법론별 발전과정과 특징

1) 구조적 방법론 (Structured, 1970년대)

| 구분 | 항목 | 설명 |
|------|------|------|
| 핵심 사상 | 하향식 분할 | 기능별 Top-Down Division, 프로세스 중심주의 |
| 필수 도구 | DFD/DD | 자료흐름도(DFD), 자료사전(DD), 소단위명세서 |
| 개발 모델 | 폭포수 | 요구→설계→구현→테스트 순차 진행 |
| 핵심 한계 | 재작업 | 변경 시 전 단계 재수행 필요, 유연성 부재 |

- 명확한 요구사항이 확정된 소규모 단순 시스템 구축에 최적화된 고전 패러다임

2) 정보공학 방법론 (Information Engineering, 1980년대)

| 구분 | 항목 | 설명 |
|------|------|------|
| 핵심 사상 | 데이터 중심 | 전사적 데이터 자원 모델링, ISP 기반 계획 |
| 필수 도구 | ERD/CRUD | 개체관계도(ERD), CRUD 매트릭스, Data-Driven |
| 개발 모델 | 순차 생명주기 | 전략계획→분석→설계→구축의 4단계 |
| 핵심 한계 | 대응 지연 | 환경 변화에 따른 계획 수정 비용 과다 |

- 대형 엔터프라이즈 시스템의 전사적 데이터 표준화 및 통합에 강점

3) 객체지향 방법론 (OOP, 1990년대)

| 구분 | 항목 | 설명 |
|------|------|------|
| 핵심 사상 | 객체 중심 | 현실 세계를 데이터+행위가 결합된 객체로 모델링 |
| 필수 도구 | UML | Unified Modeling Language, 4+1 View |
| 설계 원칙 | SOLID | 단일책임·개방폐쇄·리스코프치환·인터페이스분리·의존역전 |
| 핵심 한계 | 난이도 | 추상화 설계의 높은 학습 곡선 |

- 유지보수성과 확장성이 높아 MSA, 클린 아키텍처의 기반으로 현재도 주류

4) CBD 방법론 (Component Based Development, 2000년대)

| 구분 | 항목 | 설명 |
|------|------|------|
| 핵심 사상 | 부품화 조립 | 독립 컴포넌트를 저장소에 축적하여 재조합 |
| 이원 공정 | 도메인/응용 | 도메인 공학(컴포넌트 개발) + 응용 공학(컴포넌트 조립) |
| 핵심 산출물 | IDL 명세 | 인터페이스 정의서, 바이너리 패키지 모듈 |
| 핵심 한계 | 시장 한계 | 컴포넌트 유통 생태계 미성숙 |

- 재사용성 극대화로 생산성 향상, 현대 MSA와 PBC 아키텍처의 사상적 기원

### 3. 애자일 방법론과 현대 트렌드

1) 애자일 방법론 (Agile, 2001년 선언 / 2000년대~)

| 구분 | 항목 | 설명 |
|------|------|------|
| 핵심 사상 | 사람·소통 | 변화 수용, 고객 협력, 작동하는 소프트웨어 우선 |
| 실천 프레임 | Scrum/XP | 스프린트 반복, TDD, 리팩토링, CI/CD |
| 측정 도구 | 번다운차트 | 사용자 스토리, 스토리포인트, 팀 속도(Velocity) |
| 핵심 한계 | 통제 난항 | 대규모 프로젝트 거버넌스 및 문서화 부족 |

- 애자일은 불확실성이 높은 디지털 비즈니스 환경에서 가장 강력한 방법론이나, SAFe·LeSS 등 확장형 프레임워크 없이 대규모 프로젝트에 무작정 적용하면 통제 불능의 카오스로 귀결됨

2) 방법론 전체 비교

| 비교 항목 | 구조적 | 정보공학 | 객체지향 | CBD | 애자일 |
|----------|--------|---------|---------|-----|--------|
| 중심 축 | 프로세스 | 데이터 | 객체 | 컴포넌트 | 사람·고객 |
| 개발 모델 | 폭포수 | 순차적 | 반복점진 | 아키텍처 중심 | 이터레이티브 |
| 재사용 단위 | 서브루틴 | 논리 데이터모델 | 클래스 | 바이너리 컴포넌트 | 프레임워크·패턴 |
| 모델링 도구 | DFD, DD | ERD | UML | IDL 명세 | 사용자 스토리 |
| 핵심 한계 | 변경 재작업 | 대응 지연 | 추상화 난이도 | 유통 한계 | 대규모 통제 난항 |

- AI 기반 자동 코드 생성 시대에는 '지속적 명세화(Continuous Specification)' 기반의 AI-Augmented SDLC로 방법론이 진화 중임.  "끝"

### 실무 제언

**'Water-Scrum-Fall'의 모순 극복**
- **챌린지**: 기획/계약 단계는 폭포수 방식으로 예산·범위를 고정하고, 구현 단계에서만 스크럼을 강요하는 구조적 모순이 현업에서 빈번하게 발생함
- **제언**: 통제가 필요한 계약·거버넌스 영역은 예측형(Predictive) 방식을, 가변적 비즈니스 로직 및 UI 영역은 적응형(Adaptive) 방식을 취하는 하이브리드 방법론을 테일러링 기준으로 명확히 수립해야 함

**AI-Augmented SDLC 전환 대응**
- **챌린지**: 생성형 AI가 코드와 명세서를 자동 작성하면서 전통적 문서화 방법론의 가치가 급감하는 패러다임 전환이 진행 중
- **제언**: 방법론적 절차를 슬림화(Lean)하고, 인간은 AI 산출물의 오케스트레이션 및 위험 통제에 집중하는 프롬프트 기반 아키텍처 설계 체계로 재정립해야 함

**하이브리드 방법론 테일러링 거버넌스**
- **챌린지**: 무조건적인 애자일 맹신 또는 방법론 도그마에 빠져 프로젝트 특성을 무시한 획일적 적용이 부실 프로젝트의 근본 원인이 됨
- **제언**: 프로젝트 착수 시점에 테일러링 매트릭스를 작성하고, CMMI OPD 기준에 맞춰 QA 승인을 통한 공식적 방법론 커스터마이징 거버넌스를 수립해야 함
