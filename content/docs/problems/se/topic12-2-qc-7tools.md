## 해설

> **데이터로 품질 문제를 진단하는 통계적 도구 세트**  
> QC 7도구 — 문제의 원인을 찾고 관리하는 시각화 기법들

품질 개선은 감(感)이 아니라 데이터에 기반해야 합니다. QC(품질통제) 7도구는 현장의 품질 데이터를 수집·분석·시각화하여 문제의 원인을 찾고 개선 방향을 결정하는 7가지 통계적 기법의 집합입니다. 1940~50년대 일본 품질 혁명의 핵심 무기였으며, 소프트웨어 품질 관리에서도 결함 분석, 성능 모니터링, 테스트 결과 분석 등에 폭넓게 활용됩니다.

### 1. 출제 배경 및 의도

QC 7도구는 제조업 출신이지만 소프트웨어 공학에 적용하면 다음 상황에서 강력합니다.

- **결함 분석**: 어느 모듈에서 결함이 집중 발생하는가? (파레토도)
- **원인 추적**: 이 결함은 왜 발생했는가? (특성요인도)
- **분포 파악**: 응답 시간은 어떤 분포를 보이는가? (히스토그램)
- **통계적 관리**: 테스트 통과율이 정상 범위 안에 있는가? (관리도)

채점관은 각 도구의 **이름, 형태, 사용 목적, 소프트웨어 적용 사례**를 모두 서술할 수 있는지를 봅니다. 또한 구 QC 7도구와 신 QC 7도구의 차이를 설명하면 고득점입니다.

### 2. 개념의 본질과 작동 메커니즘

QC 7도구는 **데이터 수집 → 원인 분석 → 우선순위 결정 → 관리/예방**의 흐름으로 품질 개선 사이클을 지원합니다.

```text
[QC 도구 활용 흐름]

① 데이터 수집
   └→ 체크시트(Check Sheet): 결함 발생 유형과 빈도를 기록

② 데이터 시각화 및 분석
   └→ 히스토그램: 결함 빈도 분포 파악
   └→ 산점도: 두 변수 간 상관관계 파악
   └→ 층별(Stratification): 데이터를 조건별로 분류

③ 우선순위 결정
   └→ 파레토도: 중요 원인 20%가 문제 80% 유발

④ 원인 추적
   └→ 특성요인도: 원인-결과 관계를 뼈대 구조로 시각화

⑤ 지속적 관리
   └→ 관리도(Control Chart): 시간에 따른 품질 추이 모니터링
```

### 3. 핵심 키워드 (Must-Have)

#### 구 QC 7도구 (Old QC 7 Tools) — 수치 데이터 분석에 강함

##### 1) 파레토도 (Pareto Diagram)

**목적**: 문제의 원인 중 어느 것이 가장 중요한지 우선순위를 결정

**형태**: 막대그래프(빈도) + 꺾은선(누적 비율)의 복합 차트

**원리**: 파레토 법칙 (80-20 법칙) — 전체 결함의 80%는 20%의 원인에서 발생

```text
[파레토도 예시 — 소프트웨어 결함 원인]

결함 수
 50 │▓▓▓
    │▓▓▓
 30 │▓▓▓  ▓▓▓
    │▓▓▓  ▓▓▓  ▓▓▓
 10 │▓▓▓  ▓▓▓  ▓▓▓  ▓▓▓  ▓▓▓
    └────────────────────────── 원인
      요구사항  코딩  설계  테스트  기타
       불명확  오류  결함  부족

누적 비율: 요구사항+코딩 = 80%의 결함 → 이 둘을 먼저 잡아라
```

##### 2) 특성요인도 (Cause and Effect Diagram, 이시카와 다이어그램)

**목적**: 결함(결과)의 근본 원인(원인)을 체계적으로 파악

**형태**: 생선 뼈 모양(Fishbone Diagram이라고도 함)

**구조**: 등뼈(문제)에 대가지(주 원인 범주), 소가지(세부 원인)가 붙음

```text
[특성요인도 — 소프트웨어 배포 지연 원인 분석]

         인력(Man)           방법(Method)
           │                    │
           │                    │
경험 부족 ─┤                    ├─ 배포 프로세스 미흡
의사소통 ─┤                    ├─ 테스트 자동화 부재
           │                    │
           └────────────────────┘
                     │
                배포 지연 ◄──── 문제(결과)
                     │
           ┌────────────────────┐
           │                    │
환경 불일치─┤                    ├─ 요구사항 변경 잦음
인프라 미비─┤                    ├─ 우선순위 불명확
           │                    │
         기계(Machine)        재료(Material/요구사항)
```

주 원인 범주는 제조업의 4M(Man, Machine, Method, Material)을 소프트웨어에 맞게 변형하여 사용합니다.

##### 3) 히스토그램 (Histogram)

**목적**: 데이터의 분포(Distribution) 형태를 파악

**형태**: 연속 데이터를 구간별로 나눈 막대그래프

**소프트웨어 적용**: API 응답 시간 분포, 코드 복잡도 분포, 모듈별 결함 수 분포

```text
[히스토그램 독해 패턴]

정상 분포 (종 모양): 프로세스가 안정적
편향 분포 (한쪽 치우침): 상한/하한 문제 의심
이봉 분포 (두 개의 봉): 두 그룹이 섞여 있음 → 층별 필요
절단 분포 (한쪽이 잘림): 검사에서 불합격 데이터 제거됨
```

##### 4) 산점도 (Scatter Diagram)

**목적**: 두 변수 사이의 상관관계(양의 상관, 음의 상관, 무상관) 파악

**소프트웨어 적용**: 
- 코드 복잡도 vs 결함 수 (양의 상관 → 복잡도 낮추면 결함 감소 예상)
- 테스트 커버리지 vs 결함 발견율

##### 5) 체크시트 (Check Sheet)

**목적**: 결함 발생 빈도와 유형을 체계적으로 기록하는 데이터 수집 도구

**형태**: 미리 설계된 양식에 체크 표시로 데이터 수집

**소프트웨어 적용**: 코드 리뷰 시 결함 유형별 발생 횟수 기록, 테스트 실패 유형 기록

##### 6) 층별 (Stratification)

**목적**: 전체 데이터를 의미 있는 그룹(층, Stratum)으로 나누어 분석

**소프트웨어 적용**: 결함을 '팀별', '모듈별', '개발자별', '릴리즈별'로 분류하여 패턴 발견

파레토도, 히스토그램, 산점도 등 다른 도구들과 함께 사용하면 숨겨진 원인을 찾아낼 수 있습니다.

##### 7) 관리도 (Control Chart)

**목적**: 프로세스가 통계적으로 안정된 상태(관리 상태)에 있는지 시간적 추이로 모니터링

**형태**: 시계열 데이터에 관리 상한선(UCL), 관리 하한선(LCL), 중심선(CL)을 표시

```text
[관리도 구조]

      UCL (관리 상한선) ─ ─ ─ ─ ─ ─ ─ ─ ─
           ●  ●     ●
      CL   ─────●───────●─────●──────── (중심선, 평균)
              ●      ●
      LCL (관리 하한선) ─ ─ ─ ─ ─ ─ ─ ─ ─
      
      관리 이탈 신호:
      - UCL/LCL 초과
      - 연속 7개 이상 같은 방향으로 치우침
      - 기타 이상 패턴
```

**소프트웨어 적용**: 스프린트별 결함 발견율 추이, CI/CD 빌드 성공률 모니터링

#### 신 QC 7도구 (New QC 7 Tools) — 언어 데이터 분석에 강함

구 7도구가 수치 데이터를 다룬다면, 신 7도구는 언어적·정성적 데이터(아이디어, 의견, 계획)를 정리하고 구조화하는 데 강합니다.

| 도구 | 목적 | 특징 |
|------|------|------|
| 친화도(Affinity Diagram, KJ법) | 아이디어/의견 그룹화 | 브레인스토밍 결과 정리, 창의적 문제 해결 |
| 연관도(Relations Diagram) | 원인-결과 연쇄 관계 파악 | 복잡한 상호 관계를 화살표로 표현 |
| 계통도(Tree Diagram) | 목표 달성 수단을 계층적 분해 | 수단-목적 관계 트리 구조 |
| 매트릭스도(Matrix Diagram) | 두 요소 간 관계 강도 표현 | 요구사항 vs 기능 매핑(QFD에서 활용) |
| PDPC(과정 결정 프로그램 차트) | 계획 진행 중 예상 문제와 대응책 사전 설계 | 리스크 대응 계획 |
| 애로우다이어그램(Arrow Diagram) | 작업 순서와 일정 관리 | PERT/CPM과 유사 |
| 매트릭스 데이터 해석(Matrix Data Analysis) | 다변량 데이터의 주성분 분석 | 통계적 기법, 신 7도구 중 유일한 수치 분석 도구 |

### 4. 맥락과 비교

**구 7도구 vs 신 7도구 비교**

| 구분 | 구 QC 7도구 | 신 QC 7도구 |
|------|------------|------------|
| 등장 배경 | 1940~50년대 제조업 품질 혁명 | 1970년대 기획/관리 문제 해결 |
| 데이터 유형 | 수치(정량) 데이터 | 언어(정성) 데이터 |
| 주 활용 | 현장 품질 관리, 결함 분석 | 품질 기획, 전략적 문제 해결 |
| 소프트웨어 적용 | 결함 분석, 성능 모니터링 | 요구사항 분석, 품질 계획 |

**소프트웨어 품질 관리에서의 실제 활용 시나리오**

```text
[스프린트 회고 미팅에서의 QC 도구 활용 예시]

1단계 (데이터 수집): 체크시트로 스프린트 중 발생한 결함 유형 기록
                    └→ "로직 오류 7건, UI 버그 3건, 성능 이슈 2건..."

2단계 (우선순위): 파레토도로 "로직 오류 + UI 버그 = 전체의 83%"
                  └→ "이 두 가지를 집중적으로 예방하자"

3단계 (원인 분석): 특성요인도로 로직 오류의 근본 원인 추적
                   └→ "요구사항 불명확, 단위테스트 부재, 코드 리뷰 생략"

4단계 (개선 모니터링): 관리도로 다음 스프린트부터 결함 발생 추이 추적
                       └→ 개선 효과를 데이터로 확인
```

### 5. 실무 제언

**소프트웨어 QC 도구의 현대적 확장**

전통적인 QC 7도구는 수동으로 데이터를 수집하고 그리는 방식이었지만, 현대 소프트웨어 개발 환경에서는 도구가 자동화를 지원합니다.

- **파레토도**: Jira, GitHub Issues의 태그별 결함 통계 자동 생성
- **관리도**: Grafana, Datadog 등 모니터링 도구에서 이상 탐지 알림으로 구현
- **특성요인도**: 장애 원인 분석(RCA, Root Cause Analysis) 툴에서 템플릿 제공

**기술사 시험 전략**

QC 7도구 문제에서 고득점을 받으려면 각 도구의 이름만 나열하는 것에서 그치지 말고, **소프트웨어 품질 관리에 어떻게 적용하는지 구체적 예시**를 들어야 합니다. 또한 구 7도구와 신 7도구를 비교하고, "언제 어떤 도구를 선택해야 하는가"에 대한 선택 기준까지 서술하면 완성도 높은 답안이 됩니다.
