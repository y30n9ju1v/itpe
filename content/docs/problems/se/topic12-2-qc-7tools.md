## 해설

### 출제 배경 및 의도

품질 개선은 감(感)이 아니라 데이터에 기반해야 한다. QC(품질통제) 7도구는 현장의 품질 데이터를 수집·분석·시각화하여 문제의 원인을 찾고 개선 방향을 결정하는 7가지 통계적 기법의 집합이다. 1940~50년대 일본 품질 혁명의 핵심 무기였으며, 소프트웨어 품질 관리에서도 결함 분석·성능 모니터링·테스트 결과 분석에 폭넓게 활용된다.

채점관은 각 도구의 이름·형태·사용 목적·소프트웨어 적용 사례를 모두 서술할 수 있는지를 확인하고, 구 QC 7도구와 신 QC 7도구의 데이터 유형 차이(수치 vs 언어)까지 설명하면 고득점이 가능하다.

### 1. QC 7도구 개요

정 의  • QC 7도구를 품질 데이터를 수집·시각화·분석하여 품질 문제의 원인을 파악하고 개선하는 7가지 통계적 기법이라 하며, 구 7도구(수치 데이터)와 신 7도구(언어 데이터)로 구분한다.
       - 데이터 수집 → 원인 분석 → 우선순위 결정 → 관리/예방의 품질 개선 사이클을 지원한다.

```
[QC 도구 활용 흐름도]

① 데이터 수집
   └→ 체크시트: 결함 발생 유형과 빈도 기록

② 데이터 시각화 및 분포 파악
   └→ 히스토그램: 결함 빈도 분포
   └→ 산점도: 두 변수 간 상관관계
   └→ 층별: 팀별·모듈별 그룹 분리

③ 우선순위 결정 (80-20 법칙)
   └→ 파레토도: 중요 원인 20% → 문제 80% 유발

④ 근본 원인 추적
   └→ 특성요인도(피시본): 원인-결과 뼈대 구조

⑤ 지속적 프로세스 관리
   └→ 관리도(UCL/CL/LCL): 통계적 이상 탐지
```

- 파레토도로 집중할 문제를 선정하고, 특성요인도로 근본 원인을 도출하며, 관리도로 개선 효과를 지속 모니터링한다.

### 2. 구 QC 7도구 상세 (수치 데이터)

1) 7가지 도구 요약

| 도구 | 형태 | 목적 | 소프트웨어 적용 |
|------|------|------|-------------|
| ① 파레토도 | 막대+꺾은선 복합 | 80-20 법칙으로 우선순위 결정 | 결함 원인 상위 20% 식별 |
| ② 특성요인도 | 피시본 구조 | 근본 원인 체계적 파악 | 배포 지연·장애 원인 분석 |
| ③ 히스토그램 | 구간별 막대 | 데이터 분포 형태 파악 | API 응답 시간 분포 |
| ④ 산점도 | 점 분포 | 두 변수 상관관계 | 코드 복잡도 vs 결함 수 |
| ⑤ 체크시트 | 집계 양식 | 결함 빈도·유형 체계적 기록 | 코드 리뷰 결함 유형 집계 |
| ⑥ 층별 | 그룹 분리 | 의미 있는 그룹으로 재분류 | 팀별·모듈별·릴리즈별 분류 |
| ⑦ 관리도 | UCL/CL/LCL 시계열 | 프로세스 통계적 관리 상태 모니터링 | 스프린트별 결함 발생 추이 |

2) 핵심 도구 구조

```
[파레토도 — 소프트웨어 결함 원인 우선순위]
결함 수         누적비율
 50 │▓▓▓        80%─ ─ ─ ─ ●
    │▓▓▓  ▓▓▓        ─ ─●
 20 │▓▓▓  ▓▓▓  ▓▓▓  ▓▓▓  ▓▓▓
    └────────────────────────
      요구사항  코딩  설계  테스트  기타
      불명확   오류  결함  부족
→ 요구사항 불명확 + 코딩 오류 = 80% → 이 둘에 집중

[관리도 구조 — 이상 탐지]
UCL ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ (관리 상한선)
     ●  ●     ●
CL  ──────●───────●───────── (중심선, 평균)
         ●      ●
LCL ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ (관리 하한선)
이탈 신호: UCL/LCL 초과, 연속 7개 이상 한쪽으로 치우침
```

### 3. 신 QC 7도구 및 활용 시나리오

1) 신 QC 7도구 (언어 데이터)

| 도구 | 목적 | 소프트웨어 적용 |
|------|------|-------------|
| 친화도(KJ법) | 아이디어/의견 그룹화 | 스프린트 회고 문제점 분류 |
| 연관도 | 원인-결과 연쇄 관계 | 복잡한 시스템 장애 상호 영향 |
| 계통도 | 목표 달성 수단 계층적 분해 | 품질 목표 달성 수단 도출 |
| 매트릭스도 | 두 요소 간 관계 강도 | 요구사항 vs 기능 매핑(QFD) |
| PDPC | 예상 문제와 대응책 사전 설계 | 배포 시나리오 리스크 대응 |
| 애로우다이어그램 | 작업 순서·일정 관리 | 테스트 일정 네트워크(PERT/CPM) |
| 매트릭스 데이터 해석 | 다변량 데이터 주성분 분석 | 신 7도구 중 유일한 수치 분석 |

2) 스프린트 회고에서의 QC 도구 활용 시나리오

```
1단계 (수집): 체크시트 → "로직 오류 7건, UI 버그 3건, 성능 2건..."
2단계 (우선순위): 파레토도 → "로직 오류+UI 버그 = 83%"
3단계 (원인 분석): 특성요인도 → "요구사항 불명확, 단위테스트 부재"
4단계 (모니터링): 관리도 → 다음 스프린트부터 결함 추이 추적
```

- 구 7도구(수치·현장 분석)와 신 7도구(언어·기획 분석)를 조합하면 결함 발생부터 프로세스 개선까지 전 주기를 커버한다.  "끝"

### 실무 제언

**수동 데이터 수집의 한계**
- **챌린지**: 전통적 체크시트·파레토도 작성은 수동 작업으로 시간이 많이 소요된다.
- **제언**: Jira·GitHub Issues의 태그별 결함 통계로 파레토도를 자동 생성하고, Grafana·Datadog으로 관리도를 실시간 구현하면 QC 도구를 DevOps 파이프라인에 통합할 수 있다.

**도구 이름만 나열하는 답안의 한계**
- **챌린지**: 7개 도구를 단순 열거하면 채점관에게 실무 이해 부족으로 평가된다.
- **제언**: 각 도구의 소프트웨어 적용 사례(파레토도: 결함 원인 우선순위, 관리도: 스프린트 결함 추이)를 구체적으로 서술하고, 구 7도구와 신 7도구의 데이터 유형 차이(수치 vs 언어)를 명확히 비교하면 고득점이 가능하다.

**장애 원인 분석(RCA)과의 연계**
- **챌린지**: 반복되는 장애의 근본 원인을 찾지 못해 동일 문제가 재발한다.
- **제언**: 특성요인도를 RCA(Root Cause Analysis) 템플릿으로 활용하고, 5-Why 기법과 결합하면 표면적 원인을 넘어 구조적 원인까지 파악하여 재발을 방지할 수 있다.
