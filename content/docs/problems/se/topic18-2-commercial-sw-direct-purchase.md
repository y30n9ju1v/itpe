## 해설

### 출제 배경 및 의도

국내 SW 생태계의 고질적 문제인 "SI 종속", "불공정 하도급 구조"를 해결하기 위한 정책 제도가 상용 SW 직접구매이다. 기존에는 공공기관이 Microsoft Office나 Oracle DB 같은 상용 SW를 구매할 때도 SI 사업자가 납품하는 구조여서 중간 마진이 발생하고 납품 가격이 불투명하였다. 직접구매 제도는 수요기관이 나라장터·디지털서비스몰을 통해 SW 제조사와 직접 계약하는 방식으로, 가격 투명성을 높이고 SW 기업의 수익성을 개선하는 공공조달 혁신 제도이다.

채점관은 직접구매 제도의 개념, 기존 구매 방식과의 차이, 조달 절차, 라이선스 유형, 그리고 디지털서비스몰과 클라우드 퍼스트 정책과의 연계를 종합적으로 설명할 수 있는지 평가한다. 관련 법령(소프트웨어산업 진흥법, 조달청 고시)과 함께 제도의 도입 배경과 한계를 균형 있게 서술하는 것이 고득점 핵심이다.

### 1. 상용 SW 직접구매 제도 개요

정 의  • **상용 SW 직접구매 제도**를 수요기관이 상용 소프트웨어를 HW·SI 사업에 포함하지 않고 분리하여 나라장터·디지털서비스몰을 통해 SW사와 직접 계약하는 공공조달 방식
       - 법적 근거: 「소프트웨어산업 진흥법」, 「국가를 당사자로 하는 계약에 관한 법률」, 조달청 고시
       - 주관기관: 과학기술정보통신부, 조달청

```
[기존 구매 방식 vs 직접구매 방식]

기존 방식 (간접 구매):
  SW 제조사 → SI/대리점 → 수요기관
                   ↑
              중간 마진 발생
              가격 불투명
              SW 기업 수익 감소

직접구매 방식:
  SW 제조사 ──────────────→ 수요기관
  (또는 공식 유통사)          ↑
                         나라장터(G2B)/
                         디지털서비스몰 통해
                         직접 계약

효과:
  → 중간 마진 제거 → 가격 인하
  → 가격 공개 → 투명성 확보
  → SW 기업 직접 수익 확보 → 생태계 건전화
```

- 상용 SW 직접구매의 본질은 유통 단계 단축을 통한 가격 투명성 확보이며, 수요기관의 제품 선택권 강화가 핵심 효과이다.

### 2. 핵심 내용 상세

1) 직접구매 vs 일반(간접) 구매 비교

| 구분 | 일반 구매 (간접) | 직접구매 |
|------|--------------|---------|
| 거래 구조 | 수요기관 → SI/대리점 → SW사 | 수요기관 → SW사(또는 공식 유통사) |
| 가격 결정 | SI가 협상 (불투명) | 등록 가격 기준 (투명) |
| 계약 유형 | SW가 HW·SI에 포함 | SW 단독 계약 |
| 수요기관 선택권 | 제한적 (SI 종속) | 확대 (제품 직접 선택) |
| SW 기업 수익 | SI 마진 제외 후 수취 | 직접 수취 |
| AS/지원 | SI가 중간 처리 | SW사 직접 지원 |

2) 직접구매 표준 조달 절차 및 라이선스 유형

```
[직접구매 표준 조달 절차]

① 수요기관 구매 계획 수립
   → 필요한 SW 제품, 수량, 예산 확정

② 제품 검색 및 선정
   → 나라장터(G2B) 또는 디지털서비스몰에서 등록 제품 검색
   → GS 인증 여부, 가격, 사양 비교

③ 구매 요청 → 수요기관 내부 결재 (예산 집행 결의)

④ 계약 체결
   → 조달청 계약 또는 기관 자체 계약
   → SW사(또는 공식 대리점)와 직접 계약

⑤ 납품·검수
   → 라이선스 키 발급 또는 설치 매체 납품
   → 수요기관 검수 (정품 확인, 라이선스 수량 확인)

⑥ 사후 관리 → 유지보수 계약 (별도), 라이선스 갱신 관리
```

라이선스 유형별 특성:

| 라이선스 유형 | 특징 | 공공 조달 관점 |
|-------------|------|-------------|
| 영구 라이선스 (Perpetual) | 한 번 구매 → 영구 사용권 | CAPEX 방식, 예산 1회 집행 |
| 구독 라이선스 (Subscription) | 연간/월간 구독료 납부 | OPEX 방식, 연도별 예산 편성 필요 |
| SaaS (서비스형 소프트웨어) | 클라우드 기반, 인터넷으로 사용 | 디지털서비스몰 통해 구매, 탄력적 확장 |
| 동시 사용자 (Concurrent User) | 동시에 접속하는 사용자 수 기준 | 사용 패턴 분석 후 적정 수량 산정 필요 |
| Named User | 지정된 개인 사용자 수 기준 | 사용자 목록 관리 필요 |

- 라이선스 유형 선택은 총소유비용(TCO) 관점에서 장기 사용 예정 시 영구 라이선스, 최신 기능이 중요하거나 단기 프로젝트는 구독/SaaS가 유리하다.

### 3. 조달 채널 비교 및 공공 SW 구매 제도 연계

1) 나라장터(G2B) vs 디지털서비스몰 비교

| 채널 | 나라장터(G2B) | 디지털서비스몰 |
|------|------------|--------------|
| 대상 | 패키지 SW, HW | 클라우드, SaaS, AI 서비스 |
| 계약 방식 | 물품 구매 계약 | 서비스 이용 계약 |
| 가격 방식 | 단가 고정 | 종량제/구독 지원 |
| 기술 검증 | GS 인증 | 보안 인증, 클라우드 인증 |

2) 공공 SW 구매 제도 전체 연계 구조

```
[공공 SW 구매 제도 상호 연계]

상용 SW 직접구매 ─── 가격 투명성, 유통 단순화
       ↓
SW 분리발주 ────────── 기능별 최적 사업자 선택
       ↓
단계별(분석/구현) 발주 ─ 요구사항 명확화
       ↓
PMO(사업관리) 제도 ──── 발주자 역량 보완
       ↓
GS 인증 우선 구매 ──── 품질 검증된 제품 구매

→ 4개 제도가 상호 보완적으로 공공 SW 발주 생태계 개선
```

- 정부의 클라우드 퍼스트(Cloud-First) 정책과 디지털 전환 정책에 따라 디지털서비스몰을 통한 SaaS·AI 서비스 직접구매 비중이 확대되고 있다.  "끝"

### 실무 제언

**구독형 SaaS 도입 시 예산 구조 변화 대응**
- **챌린지**: 전통적인 공공 예산은 연간 단위 CAPEX에 익숙하나, SaaS 구독 모델은 OPEX 방식이어서 매년 예산 편성이 필요하고, 중간에 예산이 삭감되면 서비스 중단 위기가 발생한다.
- **제언**: SaaS 도입 시 3~5년 장기 계약으로 단가를 낮추고, 예산 지속성을 법적으로 보장하는 다년도 계약(Multi-year Contract) 활용을 검토해야 한다. 행정안전부의 클라우드 이용 지침에도 다년도 계약 근거가 마련되어 있다.

**라이선스 수량 관리와 컴플라이언스**
- **챌린지**: 직접구매 후 라이선스 수량 관리를 소홀히 하면 SW 저작권 위반(Over-deployment) 또는 과잉 구매(Under-deployment)가 발생하여 감사 위험과 예산 낭비가 동시에 초래된다.
- **제언**: SAM(Software Asset Management) 도구를 도입하여 설치 수량·사용자 수를 실시간 모니터링하고, 연간 라이선스 감사(License Audit)를 수행하여 컴플라이언스를 유지하는 체계적 자산 관리가 필수이다.

**클라우드 전환 과정에서의 벤더 종속 위험**
- **챌린지**: 디지털서비스몰을 통해 특정 클라우드 사업자의 SaaS를 도입하면 데이터 이식성 부족과 서비스 중단 시 대체 수단 부재로 인한 벤더 종속(Vendor Lock-in) 위험이 증가한다.
- **제언**: 클라우드 서비스 계약 시 데이터 이식성(Data Portability) 조항과 서비스 종료 시 데이터 반환 보장(Exit Plan)을 계약서에 명시하고, 공공 클라우드 보안 인증(CSAP) 취득 사업자를 우선 선정하는 위험 관리 전략이 필요하다.
