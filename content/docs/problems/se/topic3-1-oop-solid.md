## 해설

### 출제 배경 및 의도

오늘날 소프트웨어 공학의 가장 큰 화두 중 하나는 코파일럿(Copilot)과 같은 생성형 AI에 의한 대규모 코드 양산 환경에서의 '코드 스멜(Code Smell) 통제'입니다. AI가 작성한 코드는 겉보기에는 빠르게 동작하지만, 객체지향의 설계 원칙을 무시하면 순식간에 유지보수가 불가능한 레거시 스파게티 코드로 변모합니다.

채점관이 SOLID 원칙을 출제하는 의도는 각 원칙이 소프트웨어의 변경 비용(Change Cost)을 어떻게 낮추는지, 그리고 각 원칙이 깨졌을 때 발생하는 부작용을 실무적인 관점에서 진단하고 처방할 수 있는지를 검증하기 위함입니다.

클라우드 네이티브와 MSA가 주류가 된 현재, 시스템의 최소 단위인 클래스·인터페이스가 부실하면 거대한 시스템 전체가 기술 부채의 늪에 빠지게 되어 SOLID 원칙은 여전히 핵심 주제입니다.

### 1. OOP SOLID 원칙 개요

정 의  • 로버트 C. 마틴이 정립한 OOP 5대 설계 원칙의 두문자어
       - 높은 응집도(High Cohesion)와 낮은 결합도(Low Coupling)를 달성하여
       - 변경의 파도 속에서도 부러지지 않는 유연한 코드를 만드는 구조적 지침

```
[SRP: 단일 책임] ──▶ 클래스 분리 (응집도 향상)
        │
[OCP: 개방-폐쇄] ──▶ 추상화 기반 확장 (변경 없이 기능 추가)
        │
[LSP: 리스코프]  ──▶ 올바른 상속 계약 (다형성 신뢰성 보장)
        │
[ISP: 인터페이스] ──▶ 날씬한 인터페이스 (불필요 의존 차단)
        │
[DIP: 의존역전]  ──▶ 추상화에 의존 (결합도 완화, DI 실현)
```

- SRP로 클래스를 쪼개야 OCP의 추상화 토대가 마련되고, DIP로 모듈 간 결합도가 최종 완화되는 유기적 연쇄 구조

### 2. SOLID 5대 원칙 상세

1) 원칙별 본질과 실무 의미

| 원칙 | 이름 | 본질 | 핵심 키워드 |
|------|------|------|-----------|
| SRP | 단일 책임 | 클래스는 단 하나의 변경 이유만 가져야 함 | Single reason to change, 액터(Actor) 분리, 높은 응집도 |
| OCP | 개방-폐쇄 | 확장에는 열려있고, 변경에는 닫혀있어야 함 | 추상화(Interface/Abstract), 다형성, 코드 수정 없는 기능 추가 |
| LSP | 리스코프 치환 | 서브타입은 언제나 기반 타입을 대체할 수 있어야 함 | 다형성 계약 준수, 직사각형-정사각형 반례, 올바른 상속 |
| ISP | 인터페이스 분리 | 클라이언트는 사용하지 않는 메서드에 의존하면 안 됨 | 날씬한 인터페이스(Lean Interface), 역할 중심 분리, 다중 상속 |
| DIP | 의존성 역전 | 고수준/저수준 모듈 모두 추상화에 의존해야 함 | 추상화에 의존, 결합도 완화, 의존성 주입(DI) |

2) 원칙 위반 시 발생 현상과 MSA 확장 맥락

| 원칙 | 깨졌을 때 현상 | MSA 확장 맥락 |
|------|-------------|-------------|
| SRP | 하나의 기능 수정 시 사방에서 에러 발생 (경직성) | 마이크로서비스 경계(Bounded Context) 정의 기준 |
| OCP | 기능 추가 시 기존 코드에 if-else 중첩 발생 | 플러그인 아키텍처·무중단 배포 환경의 근간 |
| LSP | 하위 클래스 사용 시 형변환 오류·예외 처리 남발 | 계약(Contract) 기반 컴포넌트 대체 가능성 보장 |
| ISP | 미사용 메서드 변경으로 무관한 모듈까지 재배포 | 서비스 간 최소 API 인터페이스 노출 |
| DIP | 하위 모듈(DB 등) 교체 시 상위 모듈(비즈니스 로직) 전체 재작성 | Hexagonal/Clean 아키텍처의 포트와 어댑터 구조로 발현 |

- SOLID 원칙은 MSA 마이크로서비스 경계 설정, 플러그인 아키텍처 설계, Hexagonal 아키텍처 포트·어댑터 구현의 공통 이론적 기반으로 작동함

### 3. 현대 아키텍처와의 연계

1) SOLID와 Clean/Hexagonal 아키텍처 맵핑

| 구분 | SOLID 원칙 | 아키텍처 적용 |
|------|-----------|------------|
| 도메인 계층 | SRP + OCP | 비즈니스 로직과 인프라 관심사 분리 |
| 포트 & 어댑터 | DIP + ISP | 인터페이스(Port) 기반 외부 기술 격리 |
| 상속 설계 | LSP | 전략 패턴(Strategy Pattern)으로 대체 권장 |

- AI 자동 코드 생성 환경에서도 SOLID 원칙 기반의 정적 아키텍처 검증 도구(SonarQube, ArchUnit)를 CI/CD 파이프라인에 연계해야 코드 스멜을 자동 차단할 수 있음.  "끝"

### 실무 제언

**인터페이스 폭발(Interface Explosion)과 과도한 추상화 경계**
- **챌린지**: OCP·DIP를 과도하게 의식한 나머지 변경 가능성이 없는 유틸리티·CRUD 로직에까지 인터페이스를 구현하는 오버 엔지니어링이 코드 가독성을 저해함
- **제언**: 비즈니스 규칙이 자주 바뀌는 핵심 도메인(Core Domain)에는 SOLID를 엄격 적용하되, 변경 가능성이 낮은 인프라성 영역은 YAGNI 원칙과 균형을 잡아야 함

**정적 코드 분석 도구를 통한 품질 거버넌스 자동화**
- **챌린지**: 코드 리뷰 단계에서 수만 줄의 소스코드에 대한 SOLID 원칙 준수를 인간의 눈으로 검증하는 것은 현실적으로 불가능하고 주관적 평가에 치우침
- **제언**: SonarQube, ArchUnit 등을 CI/CD 파이프라인과 연계하여 "고수준 모듈이 하위 구현체 패키지를 직접 import 불가"와 같은 DIP 규정을 코드로 자동화하여 SOLID 위반을 빌드 단계에서 원천 차단해야 함

**생성형 AI 코드 양산 환경에서의 SOLID 거버넌스 강화**
- **챌린지**: GitHub Copilot 등 AI 도구가 분기문(if-else)을 덕지덕지 붙인 비구조적 코드를 대량 생성하여, 순식간에 유지보수 불가능한 레거시 스파게티 코드가 쌓임
- **제언**: AI 코드 생성 정책에 SOLID 원칙 준수 프롬프트 가이드라인을 내재화하고, PR(Pull Request) 단계에서 아키텍처 검증 게이트를 의무화하는 코드 거버넌스 체계를 수립해야 함
