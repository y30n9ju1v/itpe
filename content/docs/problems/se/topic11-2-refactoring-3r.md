## 문제

정보시스템 운영 및 유지보수 과정에서 소프트웨어 공학의 리팩토링(Refactoring)기법을 활용한 **3R(Reverse-Engineering, Re-Engineering, Re-use)**에 대하여 설명하시오.

## 해설

### 출제 배경 및 의도

정보시스템 운영·유지보수 현장에서 운영 중인 시스템을 새로 만들기는 어렵다. 기존 자산을 활용하면서 품질을 높이는 3R 접근법(Reverse Engineering·Re-Engineering·Reuse) 안에서 리팩토링을 올바르게 위치시키는 것이 기술사의 핵심 역량이다.

119회 관리 유형 문제로서 채점관은 리팩토링의 본질(기능 불변+구조 개선), 코드 스멜이라는 리팩토링 트리거, 3R과의 연계 관계, 그리고 테스트 없는 리팩토링의 위험성 인식을 모두 확인한다. 리팩토링을 버그 수정이나 기능 추가와 혼동하지 않는 것이 고득점의 첫 번째 조건이다.

### 1. 리팩토링 개요

정 의  • 리팩토링(Refactoring)을 외부에서 관찰 가능한 행동(Observable Behavior)을 변경하지 않으면서 소프트웨어의 내부 구조를 개선하는 규율이라 한다(Martin Fowler).
       - 기능 불변(테스트 통과 유지) + 내부 구조 개선(가독성·복잡도·결합도)의 두 조건이 반드시 동시에 충족되어야 한다.

```
[리팩토링과 3R의 관계 구조도]

  Reverse Engineering (역공학)
    └→ 코드 구조 파악 → 리팩토링 대상 식별
            │
            ▼
  Re-Engineering (재공학) = 역공학 + 리팩토링(재구조화)
    └→ 기존 시스템 이해 후 더 나은 구조로 변환
            │
            ▼
  Reuse (재사용)
    └→ 리팩토링으로 정리된 컴포넌트 → Extract Class → 독립 컴포넌트화

  [TDD + 리팩토링 사이클]
  [RED: 실패 테스트] → [GREEN: 최소 구현] → [REFACTOR: 구조 개선] → 반복
```

- 테스트가 없으면 리팩토링 후 기능 손상 여부를 확인할 수 없어 '모험'이 된다.

### 2. 코드 스멜과 주요 리팩토링 기법

1) 코드 스멜(Code Smell) — 리팩토링 트리거

| 코드 스멜 | 설명 | 대응 리팩토링 기법 |
|----------|------|----------------|
| 긴 메서드(Long Method) | 한 함수가 너무 많은 일을 함 | Extract Method |
| 거대한 클래스(Large Class) | 필드/메서드 과다, 응집도 낮음 | Extract Class |
| 중복 코드(Duplicated Code) | 같은 코드가 여러 곳에 반복 | Extract Method, Pull Up Method |
| 긴 매개변수 목록 | 파라미터 5개 이상 | Introduce Parameter Object |
| 산탄총 수술(Shotgun Surgery) | 하나 바꾸면 여러 곳을 고쳐야 함 | Move Method, Inline Class |
| 기능 편애(Feature Envy) | A 클래스가 B 클래스 데이터를 과다 사용 | Move Method |
| 죽은 코드(Dead Code) | 실행되지 않는 코드 | Remove Dead Code |

- 코드 스멜은 결함은 아니지만 유지보수성을 저하시키는 구조적 징후이며, SonarQube로 자동 탐지 가능하다.

2) 리팩토링 기법별 특성

| 구분 | 기법 | 목적 | 적용 시 효과 |
|------|------|------|------------|
| 메서드 | Extract Method | 긴 메서드 분리 | 응집도 향상, 재사용성 확보 |
| 이름 | Rename | 의도 명확화 | 주석 불필요, 자기 문서화 |
| 이동 | Move Method | 책임 재배치 | 결합도 감소 |
| 매개변수 | Introduce Parameter Object | 파라미터 그룹화 | 인터페이스 단순화 |
| 클래스 | Extract Class | 클래스 분리 | 단일 책임 원칙 준수 |

### 3. 리팩토링 vs 3R 비교 및 적용 전략

1) 범위별 비교

| 구분 | 리팩토링 | 재구조화(Restructuring) | 재공학(Re-Engineering) |
|------|---------|----------------------|---------------------|
| 범위 | 코드 레벨 | 아키텍처/모듈 레벨 | 시스템 전체 |
| 기능 변경 | 없음 | 없음 | 있을 수 있음 |
| 주체 | 개발자(일상적) | 아키텍트 | 프로젝트팀 |
| 기간 | 분 ~ 시간 | 주 ~ 월 | 월 ~ 년 |
| 트리거 | 코드 스멜 | 아키텍처 부패 | 레거시 한계 |

2) 리팩토링 적용 시점

```
리팩토링 해야 하는 때:
  O 새 기능 추가 전 (기반 정리)
  O 버그 수정 후 (재발 방지)
  O 코드 리뷰 중 코드 스멜 발견 시

리팩토링 하지 말아야 할 때:
  X 릴리즈 직전 (위험)
  X 테스트가 없는 레거시 코드 (먼저 특성 테스트 작성)
  X 처음부터 다시 짜는 것이 더 빠를 때
```

- 보이스카우트 원칙("캠핑지를 처음보다 더 깨끗하게 두고 떠나라")으로 코드 건드릴 때마다 조금씩 개선하면 기술 부채를 점진적으로 해소할 수 있다.  "끝"

### 실무 제언

**테스트 없는 레거시에서의 리팩토링**
- **챌린지**: 레거시 코드에 테스트가 없어 리팩토링 후 기능 손상 여부를 확인할 수 없다.
- **제언**: 마이클 페더스의 특성 테스트(Characterization Test)를 먼저 작성하여 현재 동작을 기록한 후 리팩토링하면, 특성 테스트가 안전망 역할을 한다.

**CI/CD와 리팩토링 통합**
- **챌린지**: 리팩토링이 기능을 망가뜨렸는지 수작업으로 확인하는 데 시간이 많이 소요된다.
- **제언**: CI 파이프라인에 자동화 테스트와 SonarQube를 연결하면 push 시마다 코드 스멜이 탐지되고 리팩토링 후 기능 손상이 즉시 확인된다.

**3R 연계 전략 수립**
- **챌린지**: 리팩토링만으로는 레거시 시스템의 아키텍처 부패를 해결할 수 없다.
- **제언**: 코드 레벨 문제는 리팩토링, 모듈/아키텍처 레벨 문제는 재구조화, 시스템 전체 문제는 재공학으로 단계적으로 접근하고, 역공학으로 비즈니스 로직을 먼저 이해한 후 적용한다.
