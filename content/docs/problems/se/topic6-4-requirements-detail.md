## 문제

## 해설

### 출제 배경 및 의도

2020년 행정안전부 "요구사항 상세화 실무 가이드라인" 발간은 공공 IT 사업에서 반복되는 비극, 즉 수십억 원 규모 시스템이 납품 후 요구사항 해석 차이로 소송까지 이어지는 참사를 막기 위한 정책 대응이다. "관리자가 사용자를 관리할 수 있어야 한다"는 한 문장이 누군가에게는 조회만, 다른 누군가에게는 삭제까지를 의미할 수 있다.

채점관이 검증하는 것은 기법 목록 나열이 아닌, 각 기법을 언제·왜·어떻게 적용하는지, 그리고 요구사항 상세화 여부를 결정하는 판단 기준(Decision Criteria)을 실무적으로 설명할 수 있는가다.

### 1. 요구사항 상세화 개요

정 의  • 추상적·고수준의 요구사항을 개발자가 설계·구현할 수 있을 만큼 구체적이고 명확하게 세분화하는 활동(Requirements Elaboration/Detailing)
       - 계약적 리스크 사전 차단 + 개발팀의 올바른 방향 나침반 역할

```
[요구사항 상세화 전/후 비교]

Before: "관리자가 사용자를 관리할 수 있어야 한다"
        │
        ▼ [상세화 프로세스: 인터뷰 → Use Case → 사용자 스토리 → 프로토타이핑]
        │
After:
  REQ-USER-001: 관리자는 전체 사용자 목록 조회 (50건/페이지, 필터링)
  REQ-USER-002: 관리자는 사용자 계정 활성/비활성화 (사유 입력·이메일 알림)
  REQ-USER-003: 비활성화 후 로그인 즉시 차단 (응답 1초 이내, 세션 강제 종료)
```

- 상세화는 발주자의 막연한 기대를 개발자가 구현 가능한 검증 가능 명세로 변환하는 과정이다

### 2. 가. 요구사항 상세화 5대 방안

1) Use Case·사용자 스토리·프로토타이핑

```
[유스케이스 다이어그램]

  ┌──────────────────────────────────────┐
  │           <<시스템 경계>>              │
  │  (사용자 로그인) ─────────── (회원)    │
  │  (상품 주문) ─extend─▷ (결제 처리)    │
  │  (주문 조회) ─include─▷ (인증 확인)   │
  │  (회원 관리) ─────────── (관리자)      │
  └──────────────────────────────────────┘
```

| 구분 | 기법 | 적합 환경 | 산출물 | 장/단점 |
|------|------|---------|--------|--------|
| ① | Use Case | 폭포수·공공사업 | 유스케이스 명세서 | 체계적 / 시간 소요 |
| ② | 사용자 스토리 | 애자일·스타트업 | 스토리 카드·백로그 | 빠름·유연 / 대규모 전체 파악 어려움 |
| ③ | 프로토타이핑 | UI 중심·요구 불명확 | 목업·프로토타입 | 검증 빠름 / 완성품 오해 위험 |

2) JAD 워크샵·인터뷰

| 구분 | 기법 | 적합 환경 | 특징 |
|------|------|---------|------|
| ④ | JAD 워크샵 | 다수 이해관계자 참여 필요 | 단시간 광범위 수집·갈등 조기 해소 |
| ⑤ | 인터뷰·설문 | 핵심 이해관계자 의견 수집 | 심층 정보 수집 / 시간 소요 |

- 상세화 기법 선택은 프로젝트 규모·개발 방법론·이해관계자 특성에 따라 복합 적용한다

### 3. 나. 요구사항 상세화 여부 결정 항목 및 RTM

1) 7대 결정 항목

| 구분 | 결정 항목 | 판단 기준 | 상세화 필요 조건 |
|------|---------|---------|--------------|
| ① | 명확성(Clarity) | 해석이 2가지 이상 가능? | "빠른 응답" → 구체적 수치 필요 |
| ② | 완전성(Completeness) | 구현 필요 정보 충분? | 입력/출력/오류 처리 미기술 시 |
| ③ | 검증가능성(Verifiability) | 테스트 케이스 도출 가능? | 정성적 표현만 있으면 필요 |
| ④ | 일관성(Consistency) | 다른 요구사항과 모순? | 충돌 발견 시 우선순위 결정 |
| ⑤ | 구현 가능성(Feasibility) | 기술적·예산적 달성 가능? | 실현 불가 요구는 조정 필요 |
| ⑥ | 필요성(Necessity) | 목적 달성에 필수인가? | Nice-to-have는 우선순위 하향 |
| ⑦ | 추적가능성(Traceability) | 비즈니스 요구와 연결? | 출처 불명확 시 확인 필요 |

2) RTM(요구사항 추적성 매트릭스) 구조

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│요구사항ID │ 요구사항 │ 설계문서  │ 구현모듈  │ 테스트ID │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ REQ-001  │ 로그인   │ DS-UC-01 │ AuthSvc  │ TC-001   │
│ REQ-002  │ 주문생성 │ DS-UC-03 │ OrderSvc │ TC-002   │
│ REQ-003  │ 결제처리 │ DS-UC-04 │ PaySvc   │ TC-003   │
└──────────┴──────────┴──────────┴──────────┴──────────┘
  ※ 비즈니스 요구 → 설계 → 구현 → 테스트 양방향 추적
```

- RTM은 요구사항 변경 시 영향 받는 설계·코드·테스트를 식별하는 변경 관리의 핵심 도구다.  "끝"

### 실무 제언

**상세화 수준 딜레마 — MoSCoW Must Have 우선 단계적 접근**
- **챌린지**: 과도한 상세화는 수백 페이지 사문서(死文書), 과소 상세화는 구현 단계 해석 충돌 — 공공사업에서 발주자의 참여 거부도 큰 장벽이다
- **제언**: NIA 가이드라인에 따라 요구사항 ID 체계를 구축하고, 각 요구사항에 인수기준(Acceptance Criteria)을 명기하는 수준으로 범위를 제한하며, MoSCoW Must Have 항목을 우선 상세화하는 단계적 접근을 채택한다

**ALM 도구 기반 RTM 자동화**
- **챌린지**: 요구사항은 살아있는 문서(Living Document)로 프로젝트 전반에 걸쳐 변경되나, 수동 추적은 누락을 초래한다
- **제언**: Jira·Confluence·Polarion 등 ALM(Application Lifecycle Management) 도구로 RTM을 자동화하여, 요구사항 변경 시 연결된 테스트 케이스에 '재검토 필요' 플래그가 자동 설정되는 환경을 구축한다
