## 문제

최근 정부에서 소프트웨어사업 기획 시 불명확한 요구사항으로 인해 사업부실과 품질저하 등의 문제가 지속적으로 제기되는 현실을 고려하여 "요구사항 상세화 실무 가이드라인"을 발간하였다. 요구사항 명확화 수준 제고를 위한 **요구사항 상세화 방안**과 **요구사항 상세히 여부 결정 항목**에 대해서 설명하시오.

## 해설

> **불명확한 요구사항을 명확한 설계 기준으로 전환하는 기술**  
> 요구사항 상세화는 발주자의 막연한 기대를 개발자가 구현 가능한 명세로 변환하는 과정이다

정부의 "요구사항 상세화 실무 가이드라인"이 발간된 이유는 공공 IT 사업에서 반복되는 비극 때문입니다. 수십억 원 규모의 시스템이 납품 후 발주자와 수주자 간의 요구사항 해석 차이로 소송까지 이어지는 사례가 비일비재합니다. 요구사항 상세화는 이러한 '계약적 리스크'를 사전에 차단하는 동시에 개발 팀이 올바른 방향으로 질주할 수 있도록 나침반을 제공합니다.

### 1. 출제 배경 및 의도

이 문제는 공공소프트웨어 사업의 고질적 품질 문제, 즉 **불명확한 요구사항으로 인한 사업 부실**을 직접적인 배경으로 합니다. 2020년 행정안전부 실무 가이드라인 발간은 기술사 수험생들이 반드시 숙지해야 할 정책 맥락입니다.

채점관은 단순히 "요구사항 상세화 기법 목록"을 원하는 것이 아닙니다. 각 기법을 **언제, 왜, 어떻게 적용**하는지, 그리고 **요구사항 상세화 여부를 결정하는 판단 기준(Decision Criteria)**을 실무적으로 설명할 수 있는지를 검증합니다.

### 2. 개념의 본질과 작동 메커니즘

**요구사항 상세화(Requirements Elaboration/Detailing)란** 추상적·고수준의 요구사항을 개발자가 설계·구현할 수 있을 만큼 구체적이고 명확하게 세분화하는 활동입니다.

```text
[요구사항 상세화 전/후 비교]

Before (모호한 요구사항):
  "관리자가 사용자를 관리할 수 있어야 한다"
         │
         ▼  [상세화 프로세스]
         │
         ├─ 인터뷰: "어떤 관리자가? 어떤 사용자를? 어떤 관리 행위를?"
         ├─ Use Case: 관리자 로그인 → 사용자 목록 조회 → 선택 → CRUD
         ├─ 사용자 스토리: "관리자로서, 특정 사용자를 비활성화하여
         │                   부정 사용을 방지하고 싶다"
         └─ 프로토타이핑: 화면 목업으로 인터페이스 확인
         │
         ▼
After (상세화된 요구사항):
  REQ-USER-001: 시스템 관리자는 전체 사용자 목록을 조회할 수 있다
                (조건: 50건/페이지 페이징, 이름/이메일/상태 필터링)
  REQ-USER-002: 시스템 관리자는 특정 사용자 계정을 활성/비활성화할 수 있다
                (조건: 비활성화 시 사유 입력 필수, 이메일 알림 발송)
  REQ-USER-003: 사용자 비활성화 후 해당 계정의 로그인은 즉시 차단된다
                (응답시간: 1초 이내, 세션 강제 종료 포함)
```

### 3. 핵심 키워드 (Must-Have)

#### 가. 요구사항 상세화 방안

**1) Use Case (유스케이스)**

- **개념**: 시스템과 외부 행위자(Actor) 사이의 상호작용 시나리오를 기술하는 기법
- **구성 요소**: 행위자(Actor), 유스케이스 이름, 사전조건(Precondition), 사후조건(Postcondition), 기본 흐름(Basic Flow), 대안 흐름(Alternative Flow), 예외 흐름(Exception Flow)
- **적용 시점**: 기능 요구사항의 상세화 단계, 시스템 범위 확정 시
- **장점**: 이해관계자와 개발자가 공통 언어로 소통 가능, 테스트 케이스 도출의 기준

```text
[유스케이스 다이어그램 텍스트 표현]

  ┌──────────────────────────────────────┐
  │           <<시스템 경계>>              │
  │                                      │
  │  (사용자 로그인)                       │
  │       ◎ ─────────────────── (회원)    │
  │  (상품 주문) ──extend──▷ (결제 처리)   │
  │       ◎ ─────────────────── (비회원)  │
  │  (주문 조회) ──include──▷ (인증 확인) │
  │                                      │
  │  (회원 관리)                           │
  │       ◎ ─────────────────── (관리자)  │
  └──────────────────────────────────────┘
```

**2) 사용자 스토리 (User Story)**

- **개념**: 애자일(Agile) 환경에서 사용하는 가벼운 요구사항 기술 형식
- **형식**: "**[역할]**로서, **[목표/기능]**을 원한다. 왜냐하면 **[이유/가치]**이기 때문이다."
- **인수 기준(Acceptance Criteria)**: 구체적인 완료 조건을 BDD(Behavior-Driven Development) 형식으로 기술
  - Given(전제 조건) - When(행위) - Then(결과)
- **장점**: 비즈니스 가치 중심, 짧고 이해하기 쉬움, 스프린트 계획에 즉시 활용 가능

**3) 프로토타이핑 (Prototyping)**

- **개념**: 시스템의 일부 기능을 조기에 시험 버전으로 구현하여 요구사항을 검증하는 기법
- **유형**:
  - 폐기형 프로토타입(Throwaway): 검증 후 버리는 목업, 페이퍼 프로토타입
  - 진화형 프로토타입(Evolutionary): 점진적으로 발전시켜 최종 제품화
- **적용 시점**: 사용자 인터페이스(UI) 요구사항, 복잡한 비즈니스 로직 검증

**4) 요구사항 워크샵 (Requirements Workshop / JAD)**

- **개념**: JAD(Joint Application Development) 세션으로 이해관계자, 개발자, 분석가가 집중 토론
- **특징**: 단시간에 광범위한 요구사항을 수집·합의, 갈등 사전 조정

**5) 인터뷰 및 설문 (Interview & Survey)**

- 구조적 인터뷰: 사전 질문지 기반의 체계적 요구사항 수집
- 맥락적 조사(Contextual Inquiry): 사용자의 실제 업무 환경을 직접 관찰

#### 나. 요구사항 상세화 여부 결정 항목

다음 기준으로 각 요구사항의 상세화 필요성을 판단합니다:

| 결정 항목 | 판단 기준 | 상세화 필요 |
|---------|---------|-----------|
| 명확성(Clarity) | 해석이 2가지 이상 가능한가? | "빠른 응답" → 구체적 수치 필요 |
| 완전성(Completeness) | 구현에 필요한 정보가 충분한가? | 입력/출력/오류 처리 미기술 시 필요 |
| 검증가능성(Verifiability) | 테스트 케이스 도출이 가능한가? | 정성적 표현만 있으면 필요 |
| 일관성(Consistency) | 다른 요구사항과 모순이 없는가? | 충돌 발견 시 우선순위 결정 필요 |
| 구현 가능성(Feasibility) | 기술적·예산적으로 달성 가능한가? | 실현 불가능한 요구는 조정 필요 |
| 필요성(Necessity) | 시스템 목적 달성에 꼭 필요한가? | Nice-to-have는 우선순위 하향 |
| 추적가능성(Traceability) | 비즈니스 요구와 연결되는가? | 출처 불명확 시 요구 출처 확인 필요 |

**요구사항 추적성 매트릭스 (RTM: Requirements Traceability Matrix)**

```text
[RTM 구조 예시]

┌──────────┬──────────┬──────────┬──────────┬──────────┐
│ 요구사항ID│ 요구사항  │ 설계문서  │ 구현모듈  │ 테스트ID │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ REQ-001  │ 로그인   │ DS-UC-01 │ AuthSvc  │ TC-001   │
│ REQ-002  │ 주문생성  │ DS-UC-03 │ OrderSvc │ TC-002   │
│ REQ-003  │ 결제처리  │ DS-UC-04 │ PaySvc   │ TC-003   │
└──────────┴──────────┴──────────┴──────────┴──────────┘
  ※ 비즈니스 요구 → 시스템 요구 → 설계 → 구현 → 테스트 양방향 추적
```

### 4. 맥락과 비교

| 상세화 기법 | 적합한 환경 | 산출물 | 장점 | 단점 |
|-----------|-----------|-------|------|------|
| Use Case | 폭포수, 규모 큰 공공사업 | 유스케이스 명세서 | 체계적, 이해관계자 동의 쉬움 | 시간 많이 소요 |
| 사용자 스토리 | 애자일(Scrum), 스타트업 | 스토리 카드, 백로그 | 빠름, 변경에 유연 | 대규모 시스템 전체 파악 어려움 |
| 프로토타이핑 | UI 중심 시스템, 요구 불명확 | 목업, 프로토타입 | 눈에 보임, 검증 빠름 | 완성품 오해 위험, 비용 발생 |
| JAD 워크샵 | 여러 이해관계자 참여 필요 시 | 회의록, 합의 문서 | 빠른 합의, 갈등 조기 해소 | 참석자 스케줄 조율 어려움 |
| 인터뷰 | 핵심 이해관계자 의견 수집 | 인터뷰 보고서 | 심층 정보 수집 | 시간 소요, 편향 위험 |

### 5. 실무 제언

**요구사항 상세화 수준의 딜레마**

- **챌린지**: 너무 세밀하게 상세화하면 요구사항 문서가 수백 페이지가 되어 아무도 읽지 않는 사문서(死文書)가 됩니다. 반대로 너무 추상적이면 구현 단계에서 해석 충돌이 발생합니다. 공공사업 특성상 발주자가 요구사항 상세화에 참여하지 않으려는 경향도 큰 장벽입니다.
- **제언**: 한국정보화진흥원(NIA)의 가이드라인에 따라 **요구사항 ID 체계**를 구축하고, 각 요구사항에 **인수기준(Acceptance Criteria)**을 명기하는 수준으로 상세화 범위를 제한해야 합니다. 전체 요구사항의 상세화보다는 MoSCoW Must Have 항목을 우선 상세화하는 단계적 접근이 현실적입니다.

**RTM 기반의 변경 영향도 자동 추적**

- **챌린지**: 요구사항은 살아있는 문서(Living Document)로, 프로젝트 진행 중 지속적으로 변경됩니다. 변경 시 어떤 설계·코드·테스트가 영향을 받는지 수동으로 추적하면 누락이 생깁니다.
- **제언**: Jira, Confluence, Polarion 등의 ALM(Application Lifecycle Management) 도구를 도입하여 RTM을 자동화해야 합니다. 요구사항 변경 시 연결된 테스트 케이스에 자동으로 '재검토 필요' 플래그가 설정되는 환경을 구축하면 변경 관리의 완전성을 크게 높일 수 있습니다.
