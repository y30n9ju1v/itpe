## 해설

### 출제 배경 및 의도

회귀 테스트는 단순히 "전에 했던 테스트를 다시 한다"는 의미가 아니다. CI/CD가 표준이 된 오늘날, 회귀 테스트의 자동화 수준은 소프트웨어 품질의 가장 직접적인 척도가 되었다.

채점관은 세 가지 핵심 역량을 평가한다: ①변경 영향 분석(Change Impact Analysis), ②재실행 전략(전체/선택적/우선순위 기반), ③CI/CD와의 연계. 이 세 가지를 논리적으로 연결하면 고득점이 가능하다.

### 1. 회귀 테스트 개요

정 의  • 소프트웨어 변경(버그 수정·기능 추가·리팩토링) 후 기존에 정상 동작하던
       기능이 의도치 않게 손상(Regression)되지 않았음을 체계적으로 검증하는 활동
       - 소프트웨어 회귀: 변경으로 인해 기존 기능이 이전의 나쁜 상태로 되돌아가는 현상

```
[회귀 결함의 발생 메커니즘]

  변경 전: 기능A ✓  기능B ✓  기능C ✓
               코드 변경 (기능D 추가)
                       ↓
  변경 후: 기능A ✓  기능B ✗  기능C ✓  기능D ✓
                          ↑
             기능D 추가로 인한 회귀 결함(Regression Defect) 발생
```

- 회귀 결함은 변경 코드 자체가 아닌 그 파급 영향에서 발생하므로 변경 영향 분석이 핵심이다.

### 2. 회귀 테스트 핵심 구성 요소

1) 변경 영향 분석 (Change Impact Analysis)

| 분석 방법 | 설명 | 도구 |
|---------|------|------|
| 의존성 그래프(Dependency Graph) | 모듈 간 호출 관계 그래프화, 변경 모듈 기점 영향 범위 추적 | JDepend, ArchUnit |
| 코드 커버리지 맵 | 변경된 코드 라인을 실행하는 테스트 케이스를 역추적 | JaCoCo, Istanbul |
| 정적 분석 도구 | 의존성 자동 분석으로 영향 모듈 식별 | SonarQube, Understand |

- 변경 영향 분석은 재실행 대상 테스트를 선별하는 기초이며, 정확도가 회귀 테스트의 신뢰성을 결정한다.

2) 재실행 대상 선정 전략 3가지

| 전략 | 방법 | 장점 | 단점 |
|------|------|------|------|
| 전체 재실행(Retest-All) | 모든 테스트 케이스 실행 | 완전한 안전 보장 | 시간/비용 과다, 대규모에서 비현실적 |
| 선택적 재실행(Selective) | 변경 영향 모듈의 테스트만 선별 실행 | 효율적, 빠른 피드백 | 분석 오류 시 결함 누락 위험 |
| 우선순위 기반(Prioritization) | 리스크·중요도·결함 이력 기준 우선순위 | 제한 시간 내 최대 효과 | 우선순위 선정 기준의 주관성 |

```
[CI/CD 파이프라인에서의 회귀 테스트]

  git push
      │
  CI 서버 트리거 (Jenkins, GitHub Actions)
      │
  ┌── 빌드(Build)          ── 실패 시 즉시 알림
  ├── 단위 테스트(UT)       ── 실패 시 즉시 알림
  ├── 통합 테스트(IT)       ── 실패 시 즉시 알림
  └── 회귀 테스트 스위트    ── 핵심 안전망
          │
  결과 리포트 + Slack 알림
          │
  통과 → 스테이징 배포 / 실패 → 배포 차단
```

- CI/CD 파이프라인에서 회귀 테스트는 코드 변경의 영향을 즉시 감지하는 안전망 역할을 한다.

### 3. 재테스트 vs 회귀 테스트 비교 및 스위트 관리

1) 재테스트 vs 회귀 테스트

| 구분 | 재테스트(Re-test) | 회귀 테스트(Regression Test) |
|------|----------------|--------------------------|
| 목적 | 특정 결함이 수정되었는지 확인 | 변경으로 인한 새로운 결함 발생 여부 확인 |
| 대상 | 결함이 있었던 그 기능 | 변경의 영향을 받을 수 있는 다른 기능들 |
| 시점 | 버그 수정 직후 | 코드 변경(수정·추가·리팩토링) 이후 |

- 재테스트와 회귀 테스트는 목적과 대상이 다르며, 두 활동 모두 코드 변경 후 수행해야 한다.  "끝"

### 실무 제언

**테스트 실행 시간과 피드백 속도의 균형**
- **챌린지**: 회귀 테스트 스위트가 수천 개 케이스로 구성되면 CI 파이프라인에서 30분 이상 소요되어 개발자 생산성을 심각하게 저해한다.
- **제언**: 병렬 실행(Selenium Grid, pytest-xdist), 테스트 피라미드 적용(단위 테스트 중심), Test Impact Analysis(변경 코드와 연관된 테스트만 자동 선별)를 조합하여 피드백 속도를 10분 이내로 유지해야 한다.

**회귀 테스트 스위트 부채(Test Suite Debt) 관리**
- **챌린지**: 테스트 코드에도 기술 부채가 쌓인다. 중복·무의미한 테스트 케이스가 축적되면 실행 시간이 증가하고, 개발팀이 테스트를 기피하게 되는 역효과가 발생한다.
- **제언**: 새로운 버그 발견 시 버그 고정 테스트(Bug Fix Test)를 스위트에 추가하고, 분기별 '테스트 케이스 정원 가꾸기(Test Gardening)' 활동으로 중복·오래된 케이스를 정리하는 선순환 구조를 CI/CD 운영 계획에 명시적으로 포함시켜야 한다.
