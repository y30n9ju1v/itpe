## 해설

> **변경의 그림자를 추적하는 테스트**  
> 코드를 고쳤을 때, 이미 잘 동작하던 기능이 망가지지 않았음을 증명하는 것이 회귀 테스트다.

소프트웨어를 수정하는 것은 마치 정교한 건물의 벽을 허물고 새 창문을 다는 것과 같습니다. 새 창문은 잘 달렸더라도, 기존 기둥에 균열이 생겼을 수 있습니다. 회귀 테스트(Regression Test)는 코드 변경 이후 **의도치 않게 기존 기능이 손상(regression)되지 않았음**을 체계적으로 검증하는 활동입니다. CI/CD가 표준이 된 오늘날, 회귀 테스트의 자동화 수준은 소프트웨어 품질의 가장 직접적인 척도입니다.

### 1. 출제 배경 및 의도

회귀 테스트는 단순히 "전에 했던 테스트를 다시 한다"는 의미가 아닙니다. 채점관은 다음 세 가지를 핵심 역량으로 평가합니다.

1. **변경 영향 분석(Change Impact Analysis)**: 어떤 코드가 변경되었을 때 어떤 기능이 영향을 받는지 추적하는 능력
2. **재실행 전략**: 전체 재실행, 선택적 재실행, 우선순위 기반 재실행 중 상황에 맞는 선택
3. **CI/CD와의 연계**: 자동화된 회귀 테스트가 파이프라인에서 어떻게 동작하는지

"회귀 테스트에 대해 설명하시오"라는 단순한 문제처럼 보이지만, 위 세 가지를 논리적으로 연결하여 서술하면 고득점 답안이 됩니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 회귀(Regression)의 의미

'회귀'란 소프트웨어가 이전의 나쁜 상태로 되돌아간다는 의미입니다. 버그를 수정하거나 기능을 추가했는데, 그 과정에서 이전에 정상 작동하던 다른 기능이 고장나는 현상을 **소프트웨어 회귀(Software Regression)**라고 합니다.

```text
[회귀 결함의 발생 메커니즘]

   변경 전 상태
   ┌──────────────────────────┐
   │ 기능 A ✓  기능 B ✓  기능 C ✓ │
   └──────────────────────────┘
           코드 변경 (기능 D 추가)
                   ↓
   변경 후 상태
   ┌──────────────────────────────┐
   │ 기능 A ✓  기능 B ✗  기능 C ✓  기능 D ✓ │
   └──────────────────────────────┘
                         ↑
               기능 D 추가로 인해 기능 B에
               회귀 결함(Regression Defect) 발생!
```

#### 나. 변경 영향 분석(Change Impact Analysis)

모든 변경이 동일한 영향을 미치지는 않습니다. 핵심은 **어떤 변경이 어떤 모듈에 영향을 미치는지**를 사전에 분석하는 것입니다.

분석 방법:
- **의존성 그래프(Dependency Graph)**: 모듈 간 호출 관계를 그래프로 표현. 변경된 모듈을 시작점으로 하는 모든 경로상의 모듈이 잠재적 영향 범위
- **코드 커버리지 맵**: 변경된 코드 라인을 실행하는 테스트 케이스를 역추적
- **정적 분석 도구**: SonarQube, Understand 등이 자동으로 의존성 분석 제공

#### 다. 재실행 대상 선정 전략 (3가지)

회귀 테스트에서 가장 중요한 결정은 "어떤 테스트 케이스를 얼마나 실행할 것인가"입니다.

```text
[재실행 전략 비교]

전체 재실행 ─────────────────────────────────────────
 모든 테스트 케이스 실행
 장점: 완전한 안전 보장
 단점: 시간/비용 과다 → 대규모 시스템에서 비현실적

선택적 재실행 ──────────────────────────────────────
 변경 영향 분석 결과, 영향받는 모듈의 테스트만 선별 실행
 장점: 효율적, 빠른 피드백
 단점: 분석 오류 시 결함 누락 위험

우선순위 기반 재실행 ────────────────────────────────
 리스크, 비즈니스 중요도, 최근 결함 이력 기준으로 우선순위 결정
 장점: 제한된 시간 내 최대 효과
 단점: 우선순위 선정 기준의 주관성
```

#### 라. 회귀 테스트 스위트(Regression Test Suite) 관리

회귀 테스트 스위트는 장기간 유지·관리해야 하는 살아있는 자산입니다.

- **테스트 케이스 추가**: 새로운 버그가 발견될 때마다 해당 버그를 재현하는 테스트 케이스를 스위트에 추가 (버그 고정 테스트, Bug Fix Test)
- **테스트 케이스 정리(Test Suite Pruning)**: 오래되거나 중복된 테스트 케이스를 제거. 스위트가 비대해지면 실행 시간이 증가하여 CI/CD 파이프라인을 저해
- **테스트 케이스 업데이트**: 요구사항 변경 시 기존 테스트 케이스도 함께 업데이트

### 3. 핵심 키워드 (Must-Have)

| 키워드 | 설명 |
|--------|------|
| 소프트웨어 회귀 (Software Regression) | 변경으로 인해 기존 기능이 손상되는 현상 |
| 변경 영향 분석 (Change Impact Analysis) | 변경 범위와 영향받는 모듈을 사전에 파악 |
| 전체 재실행 (Retest-All) | 모든 테스트 케이스를 재실행하는 완전한 방식 |
| 선택적 재실행 (Regression Test Selection) | 영향받는 테스트만 선별 실행 |
| 우선순위 기반 (Test Case Prioritization) | 리스크/중요도 기준으로 실행 순서 결정 |
| 회귀 테스트 스위트 | 회귀 테스트용으로 관리되는 테스트 케이스 집합 |
| 버그 고정 테스트 (Bug Fix Test) | 수정된 버그의 재발을 방지하는 테스트 |

### 4. 맥락과 비교

#### CI/CD 파이프라인에서의 회귀 테스트 자동화

현대 DevOps 환경에서 회귀 테스트는 수동으로 수행하지 않습니다. Git 저장소에 코드가 병합(Merge)되는 순간 자동으로 실행됩니다.

```text
[CI/CD 파이프라인에서의 회귀 테스트]

  git push
      ↓
  CI 서버 트리거 (Jenkins, GitHub Actions 등)
      ↓
  ┌─── 빌드 (Build) ────────────── 실패 시 즉시 알림
  ├─── 단위 테스트 (Unit Test) ─── 실패 시 즉시 알림
  ├─── 통합 테스트 ────────────── 실패 시 즉시 알림
  └─── 회귀 테스트 스위트 실행 ─── ← 핵심
            ↓
    결과 리포트 생성 + Slack/이메일 알림
            ↓
     통과 시 → 스테이징 환경 배포
     실패 시 → 배포 차단 + 원인 분석
```

**회귀 테스트 vs. 재테스트(Re-test)의 차이**

| 구분 | 재테스트 (Re-test) | 회귀 테스트 (Regression Test) |
|------|-----------------|-----------------------------|
| 목적 | 특정 결함이 수정되었는지 확인 | 변경으로 인한 새로운 결함 발생 여부 확인 |
| 대상 | 결함이 있었던 그 기능 | 변경의 영향을 받을 수 있는 다른 기능들 |
| 시점 | 버그 수정 직후 | 코드 변경(버그 수정, 기능 추가) 이후 |

### 5. 실무 제언

**테스트 실행 시간과 피드백 속도의 균형**

회귀 테스트 스위트가 수천 개의 테스트 케이스로 구성되면 실행에 수십 분이 소요됩니다. CI 파이프라인에서 30분이 넘는 피드백 지연은 개발자 생산성을 심각하게 저해합니다. 이를 해결하기 위해 다음 전략을 권장합니다.

- **병렬 실행(Parallel Execution)**: 테스트를 여러 에이전트로 분산 실행 (Selenium Grid, pytest-xdist)
- **테스트 계층화(Test Pyramid 적용)**: 빠른 단위 테스트 위주로 구성하고, 느린 E2E 테스트 비중 최소화
- **스마트 선택(Test Impact Analysis)**: Microsoft Research의 TIA(Test Impact Analysis)처럼 변경된 코드와 연관된 테스트만 자동으로 선별하여 실행

**회귀 테스트 스위트 부채(Test Suite Debt) 관리**

테스트 코드도 기술 부채가 쌓입니다. 중복되거나 더 이상 의미없는 테스트 케이스를 정기적으로 리팩토링하지 않으면, 오히려 개발팀이 테스트를 기피하게 되는 역효과가 발생합니다. 분기별 '테스트 케이스 정원 가꾸기(Test Gardening)' 활동을 CI/CD 운영 계획에 명시적으로 포함시켜야 합니다.
