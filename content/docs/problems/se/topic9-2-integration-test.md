## 문제

통합 테스트(Integration Test)에 대하여 다음을 설명하시오.

- 가. 비점진적 통합 방식과 점진적 통합 방식
- 나. 하향식(Top Down) 통합 테스트와 상향식(Bottom Up) 통합 테스트
- 다. 테스트 드라이버(Test Driver)와 테스트 스텁(Test Stub)

## 해설

> **모듈이 만나는 순간의 진실**  
> 단위 테스트를 통과한 모듈들도 함께 연결하면 새로운 결함이 발생한다. 통합 테스트는 그 연결 지점을 검증하는 과정이다.

단위 테스트(Unit Test)에서 완벽하게 동작하던 모듈들이 막상 하나로 합쳐지면 예상치 못한 오류가 터지는 경험은 모든 개발자에게 익숙합니다. 인터페이스 규격의 미세한 불일치, 공유 자원 경쟁, 오류 처리 방식의 차이가 그 원인입니다. 통합 테스트는 이러한 모듈 간 상호작용(Interface)에서 발생하는 결함을 체계적으로 발견하는 테스트 단계입니다.

### 1. 출제 배경 및 의도

기술사 시험에서 통합 테스트가 자주 출제되는 이유는 현실 프로젝트에서 가장 많은 결함이 발생하는 지점이 바로 '모듈 간 인터페이스'이기 때문입니다. 채점관은 단순히 방식의 이름을 나열하는 것이 아니라, 각 통합 방식이 **어떤 프로젝트 환경에 적합한지**, **스텁과 드라이버라는 테스트 더블(Test Double)이 왜 필요한지**를 실무적 맥락에서 설명할 수 있는지를 봅니다.

131회 기출 문제의 핵심은 세 가지 관점을 동시에 연결하는 것입니다. 비점진적 vs 점진적, 하향식 vs 상향식이라는 방식론의 차이를 이해하고, 그 차이에서 파생되는 스텁/드라이버의 역할까지 논리적으로 서술해야 고득점이 가능합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 비점진적 통합 방식 vs. 점진적 통합 방식

```text
[비점진적 통합 - 빅뱅(Big Bang)]

  모듈 A  모듈 B  모듈 C  모듈 D
     |       |       |       |
     +-------+-------+-------+
             한꺼번에 통합
                   |
             통합 시스템 테스트
                   |
           결함 발생 → 원인 추적 어려움
```

**비점진적 통합(빅뱅, Big Bang Integration)**은 모든 모듈을 한꺼번에 결합하여 테스트하는 방식입니다. 개발 완료 후 한 번에 통합하므로 일정이 빠른 것처럼 보이지만, 결함 발생 시 어느 모듈 인터페이스가 원인인지 찾기 매우 어렵습니다.

**점진적 통합(Incremental Integration)**은 모듈을 하나씩 또는 소그룹 단위로 단계적으로 결합하며 테스트하는 방식입니다. 결함 발생 시 가장 최근에 추가된 모듈이 원인일 가능성이 높아 디버깅이 훨씬 용이합니다.

| 구분 | 빅뱅(Big Bang) | 점진적(Incremental) |
|------|-------------|-------------------|
| 통합 시점 | 모든 모듈 완성 후 일괄 | 모듈 완성 즉시 단계적 |
| 결함 발견 | 통합 후 한꺼번에 노출 | 통합 단계마다 즉시 발견 |
| 결함 원인 추적 | 매우 어려움 | 용이함 |
| 스텁/드라이버 필요 여부 | 불필요 | 필요 |
| 적합한 상황 | 소규모, 단순 시스템 | 대규모, 복잡 시스템 |

#### 나. 하향식(Top-Down) vs. 상향식(Bottom-Up) 통합

```text
[하향식 통합 - 스텁(Stub) 사용]

        [메인 모듈]  ← 먼저 테스트
        /         \
  [모듈 A]      [스텁 B]  ← 미완성 B를 대체
    /    \
[스텁 C] [스텁 D]

→ 위에서 아래로 통합, 핵심 제어 흐름 조기 검증


[상향식 통합 - 드라이버(Driver) 사용]

[드라이버] ← 상위 모듈 역할 대행
    |
[모듈 C]  [모듈 D]  ← 먼저 완성/테스트
    \         /
     [모듈 B]
         |
    [메인 모듈]  ← 마지막에 통합

→ 아래서 위로 통합, 핵심 기능 모듈 조기 검증
```

**하향식 통합(Top-Down Integration)**
- 최상위 제어 모듈부터 시작하여 하위 모듈로 점진적으로 통합
- 아직 완성되지 않은 **하위 모듈 대신 스텁(Stub)**을 사용
- 스텁(Stub): 실제 모듈을 대신하는 껍데기 모듈. 호출을 받으면 하드코딩된 단순 값을 반환
- 장점: 초기에 시스템의 전체 구조와 제어 흐름 검증 가능. 사용자 인터페이스(UI)를 일찍 확인 가능
- 단점: 스텁 개발 비용 발생. 핵심 하위 기능의 검증이 늦어짐

**상향식 통합(Bottom-Up Integration)**
- 최하위 모듈부터 시작하여 상위 모듈로 점진적으로 통합
- 아직 완성되지 않은 **상위 모듈 대신 드라이버(Driver)**를 사용
- 드라이버(Driver): 하위 모듈을 호출하고 결과를 확인하는 임시 상위 모듈
- 장점: 핵심 기능(알고리즘, DB 처리) 모듈을 조기에 검증 가능
- 단점: 드라이버 개발 비용 발생. 시스템 전체 구조 검증이 늦어짐

#### 샌드위치(Sandwich/혼합) 통합

실제 프로젝트에서는 순수한 하향식이나 상향식만 사용하기 어렵습니다. 샌드위치 통합은 두 방식을 동시에 적용하는 **중간 만남(Meeting in the Middle)** 전략입니다.

```text
[샌드위치 통합]

    [최상위 모듈] → 하향식으로 내려옴
         ↓
    [중간 레이어] ← 만나는 지점
         ↑
[하위 핵심 모듈들] → 상향식으로 올라옴
```

### 3. 핵심 키워드 (Must-Have)

| 키워드 | 정의 | 연관 방식 |
|--------|------|----------|
| 스텁 (Stub) | 아직 개발되지 않은 하위 모듈을 대체하는 더미 모듈 | 하향식 통합 |
| 드라이버 (Driver) | 아직 개발되지 않은 상위 모듈 역할을 대행하는 임시 모듈 | 상향식 통합 |
| 빅뱅 (Big Bang) | 모든 모듈을 한꺼번에 통합하는 비점진적 방식 | 비점진적 |
| 샌드위치 | 하향식+상향식 혼합, 중간에서 만나는 방식 | 점진적 |
| 회귀 테스트 | 새 모듈 추가 후 기존 기능 이상 없음을 재확인 | 통합 전반 |

### 4. 맥락과 비교

통합 테스트는 V-모델(V-Model)에서 단위 테스트와 시스템 테스트 사이에 위치하며, ISO/IEC 29119 테스트 프로세스 표준에서도 명시적으로 구분되는 테스트 레벨입니다.

```text
[V-모델에서의 통합 테스트 위치]

요구사항 정의 ─────────────────── 인수 테스트
  시스템 설계 ─────────────── 시스템 테스트
    아키텍처 설계 ─────── 통합 테스트  ← 여기
      모듈 설계 ─── 단위 테스트
          구현
```

**CI/CD 파이프라인에서의 현대적 적용**

현대 DevOps 환경에서는 Git push 이벤트마다 자동화된 통합 테스트가 수행됩니다.

- 컨테이너 기반 테스트 환경(Docker Compose)으로 실제 모듈 간 통합 검증
- 마이크로서비스 아키텍처(MSA)에서는 서비스 간 계약(Contract)을 검증하는 **Consumer-Driven Contract Testing**(Pact 프레임워크)이 통합 테스트의 현대적 대안으로 부상
- 스텁과 드라이버의 역할은 오늘날 **Mock 서버, WireMock, 서비스 가상화(Service Virtualization)**로 발전

### 5. 실무 제언

**테스트 더블의 과도한 사용 경계**

스텁과 드라이버는 아직 준비되지 않은 모듈을 대체하기 위한 임시 수단입니다. 하지만 스텁이 너무 단순하게 구현될 경우, 스텁을 대상으로 한 테스트가 통과되어도 실제 모듈이 연결될 때 새로운 결함이 발생할 수 있습니다. 스텁이 실제 모듈의 동작 명세를 정확히 반영하는지 지속적으로 검토하는 **계약 기반 명세(Contract Specification)**가 병행되어야 합니다.

**통합 순서는 리스크 기반으로 결정**

하향식이냐 상향식이냐의 선택보다 더 중요한 것은, 프로젝트의 **가장 높은 리스크를 가진 인터페이스를 먼저 통합**하는 전략입니다. 결제 시스템이나 인증 모듈처럼 결함 발생 시 전체 시스템에 영향을 미치는 핵심 인터페이스를 우선적으로 통합하여 검증해야 합니다. 이는 리스크 기반 테스팅(Risk-Based Testing)과 통합 테스트 전략이 연계되는 지점입니다.
