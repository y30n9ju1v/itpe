## 문제

통합 테스트(Integration Test)에 대하여 다음을 설명하시오.

- 가. 비점진적 통합 방식과 점진적 통합 방식
- 나. 하향식(Top Down) 통합 테스트와 상향식(Bottom Up) 통합 테스트
- 다. 테스트 드라이버(Test Driver)와 테스트 스텁(Test Stub)

## 해설

### 출제 배경 및 의도

기술사 시험에서 통합 테스트가 자주 출제되는 이유는 현실 프로젝트에서 가장 많은 결함이 발생하는 지점이 바로 '모듈 간 인터페이스'이기 때문이다. 단위 테스트를 통과한 모듈들도 함께 연결하면 인터페이스 규격의 미세한 불일치, 공유 자원 경쟁, 오류 처리 방식의 차이로 새로운 결함이 발생한다.

채점관은 비점진적 vs 점진적, 하향식 vs 상향식이라는 방식론의 차이를 이해하고, 그 차이에서 파생되는 스텁/드라이버의 역할까지 논리적으로 연결하여 서술하는 능력을 평가한다. 현대 DevOps 환경에서 Consumer-Driven Contract Testing 등 통합 테스트의 현대적 발전 방향을 언급하면 고득점이 가능하다.

### 1. 통합 테스트 개요

정 의  • 단위 테스트를 통과한 모듈들을 결합하여 모듈 간 상호작용(Interface)에서
       발생하는 결함을 체계적으로 발견하는 테스트 단계
       - V-모델에서 단위 테스트와 시스템 테스트 사이에 위치

```
[V-모델에서의 통합 테스트 위치]

  요구사항 정의 ─────────────── 인수 테스트
    시스템 설계 ──────────── 시스템 테스트
      아키텍처 설계 ───── 통합 테스트  ← 핵심
        모듈 설계 ──── 단위 테스트
              구현
```

- 통합 테스트는 인터페이스 오류, 공유 자원 경쟁, 오류 전파 등 모듈 조합에서 발생하는 결함을 발견하는 것이 목적이다.

### 2. 핵심 내용: 통합 방식 및 스텁/드라이버

1) 비점진적 vs 점진적 통합 방식 비교

| 구분 | 빅뱅(Big Bang) | 점진적(Incremental) |
|------|-------------|-------------------|
| 통합 시점 | 모든 모듈 완성 후 일괄 | 모듈 완성 즉시 단계적 |
| 결함 발견 | 통합 후 한꺼번에 노출 | 통합 단계마다 즉시 발견 |
| 결함 원인 추적 | 매우 어려움 | 용이함 (최근 추가 모듈이 원인) |
| 스텁/드라이버 필요 | 불필요 | 필요 |
| 적합한 상황 | 소규모·단순 시스템 | 대규모·복잡 시스템 |

```
[비점진적 통합 - 빅뱅(Big Bang)]

  모듈A  모듈B  모듈C  모듈D
    │      │      │      │
    └──────┴──────┴──────┘
              한꺼번에 통합
                  │
         결함 발생 → 원인 추적 어려움
```

- 실제 프로젝트에서는 점진적 통합이 표준이며, 빅뱅은 단순 시스템 또는 프로토타입에만 적용한다.

2) 하향식·상향식·샌드위치 통합 비교

| 구분 | 하향식(Top-Down) | 상향식(Bottom-Up) | 샌드위치 |
|------|--------------|--------------|-------|
| 통합 방향 | 최상위 → 최하위 | 최하위 → 최상위 | 양방향 동시 |
| 대체 더블 | 스텁(Stub) | 드라이버(Driver) | 스텁 + 드라이버 |
| 조기 검증 | 제어 흐름·UI | 핵심 기능·알고리즘 | 양쪽 동시 |
| 단점 | 스텁 개발 비용, 하위 기능 검증 지연 | 드라이버 비용, 전체 구조 검증 지연 | 복잡도 증가 |

```
[하향식 통합 - 스텁(Stub) 사용]

  [메인 모듈]  ← 먼저 테스트
   /        \
[모듈A]   [스텁B]  ← 미완성 B를 대체
  / \
[스텁C][스텁D]

[상향식 통합 - 드라이버(Driver) 사용]

[드라이버]  ← 상위 모듈 역할 대행
    │
[모듈C] [모듈D]  ← 먼저 완성·테스트
    \      /
    [모듈B]
       │
  [메인 모듈]  ← 마지막 통합
```

- 스텁은 하위 모듈을 대체하는 껍데기로 하드코딩된 단순 값을 반환하고, 드라이버는 하위 모듈을 호출·결과 확인하는 임시 상위 모듈이다.

### 3. 테스트 더블 및 현대적 통합 테스트 트렌드

1) 테스트 더블 유형 및 역할

| 키워드 | 정의 | 연관 방식 |
|--------|------|----------|
| 스텁(Stub) | 하위 모듈을 대체하는 더미, 하드코딩 값 반환 | 하향식 통합 |
| 드라이버(Driver) | 상위 모듈 역할 대행, 하위 모듈 호출 | 상향식 통합 |
| Mock 객체 | 스텁+드라이버 기능 포함, 검증 기능 추가 | 단위·통합 테스트 |
| 서비스 가상화 | MSA 환경에서 서비스 전체를 가상화 | 현대 통합 테스트 |

- 스텁과 드라이버의 역할은 오늘날 Mock 서버, WireMock, 서비스 가상화(Service Virtualization)로 발전했다.  "끝"

### 실무 제언

**테스트 더블의 과도한 사용 경계**
- **챌린지**: 스텁이 너무 단순하게 구현되면, 스텁 대상 테스트 통과 후 실제 모듈 연결 시 새로운 결함이 발생한다. 스텁이 실제 모듈의 동작 명세를 정확히 반영하지 않으면 테스트 신뢰성이 저하된다.
- **제언**: 스텁은 계약 기반 명세(Contract Specification)를 기반으로 구현해야 한다. Consumer-Driven Contract Testing(Pact 프레임워크)을 적용하면 스텁이 실제 공급자 서비스 계약을 자동으로 검증하여 테스트 신뢰성을 높일 수 있다.

**통합 순서는 리스크 기반으로 결정**
- **챌린지**: 하향식이냐 상향식이냐의 선택보다 중요한 것은, 어떤 인터페이스를 먼저 통합할 것인가의 전략이다. 모든 인터페이스를 동일한 우선순위로 통합하면 핵심 리스크 검증이 늦어진다.
- **제언**: 결제·인증 모듈처럼 결함 발생 시 전체 시스템에 치명적 영향을 미치는 핵심 인터페이스를 먼저 통합하는 리스크 기반 통합 전략을 적용해야 한다. 이는 리스크 기반 테스팅(RBT)과 통합 테스트 전략이 연계되는 지점이다.
