구조기반 기법의 **조건 커버리지(CC), 결정 커버리지(DC), 변경조건/결정 커버리지(MC/DC)**를 설명하시오.

## 해설

> **조건 하나하나가 결과에 진짜 영향을 미치는가**  
> MC/DC는 각 개별 조건이 독립적으로 최종 결정에 영향을 줄 수 있음을 증명하는 가장 엄격한 커버리지 기준이다.

항공기 소프트웨어에서 `if (altitude_low AND engine_fail OR fuel_empty)`라는 조건이 잘못 처리되면 수백 명의 목숨이 위험합니다. 각 조건(`altitude_low`, `engine_fail`, `fuel_empty`)이 서로 독립적으로 최종 경보 발생 여부에 영향을 미치는지 철저히 검증해야 합니다. 이것이 MC/DC(Modified Condition/Decision Coverage, 변경조건/결정 커버리지)가 항공 안전 표준의 요구사항이 된 이유입니다.

### 1. 출제 배경 및 의도

MC/DC는 화이트박스 테스트 커버리지 중 가장 엄격하고 실용적인 수준으로 평가받습니다. 채점관은 세 가지 커버리지의 단순 정의 나열이 아니라 다음을 요구합니다.

1. CC(조건 커버리지)와 DC(결정 커버리지)의 **각각의 약점**이 무엇인지
2. MC/DC가 두 약점을 **어떻게 동시에 극복**하는지
3. 실제 논리식에 대해 **MC/DC 테스트 케이스를 직접 설계**하는 능력

특히 "응용" 유형 문제이므로 개념 설명에 그치지 않고 예제를 통한 케이스 설계까지 서술해야 고득점이 가능합니다.

### 2. 개념의 본질과 작동 메커니즘

#### 가. 세 가지 커버리지 비교

예제 코드: `if (A AND B)` — A, B 두 조건이 있는 결정

```text
예제: D = A AND B

[구문 커버리지(SC): 모든 statement 실행]
  TC1: A=T, B=T → D=T  (단 1개면 충족)

[결정 커버리지(DC): 결정이 TRUE/FALSE 모두 되도록]
  TC1: A=T, B=T → D=T
  TC2: A=F, B=T → D=F  (또는 A=T, B=F)
  → 총 2개 필요

[조건 커버리지(CC): 각 개별 조건이 T/F 모두 되도록]
  TC1: A=T, B=F → D=F
  TC2: A=F, B=T → D=F
  → A는 T,F 모두 경험 ✓, B는 T,F 모두 경험 ✓
  → 그러나! 결정(D)은 두 경우 모두 F → DC는 미충족!!

[MC/DC: 각 조건이 독립적으로 결정에 영향을 줌을 증명]
  A의 독립적 영향 증명: A만 바뀌었을 때 D가 바뀌어야 함
    → A=T,B=T → D=T  /  A=F,B=T → D=F  (A만 변경, D 변경됨)
  B의 독립적 영향 증명: B만 바뀌었을 때 D가 바뀌어야 함
    → A=T,B=T → D=T  /  A=T,B=F → D=F  (B만 변경, D 변경됨)
  최소 케이스: TC1(T,T), TC2(F,T), TC3(T,F) → 3개 = n+1 = 2+1
```

#### 나. MC/DC의 정의

**MC/DC(Modified Condition/Decision Coverage)**: 다음 세 가지 조건을 동시에 만족하는 테스트 커버리지

1. **결정 커버리지(DC)**: 모든 결정(Decision)이 TRUE/FALSE 값을 모두 취해야 함
2. **조건 커버리지(CC)**: 모든 개별 조건(Condition)이 TRUE/FALSE 값을 모두 취해야 함
3. **독립 영향(Independence)**: 각 개별 조건이 **다른 조건의 값을 고정한 채** 혼자서 결정의 결과를 변경할 수 있음을 증명해야 함

세 번째 조건이 CC와 다른 핵심입니다. 단순히 각 조건이 T/F를 경험하는 것이 아니라, 각 조건이 **독립적으로** 결과에 영향을 미침을 증명해야 합니다.

#### 다. n+1 테스트 케이스 최소화

MC/DC의 강력한 특징은 n개의 조건이 있는 결정에 대해 **최소 n+1개의 테스트 케이스**로 커버리지를 달성할 수 있다는 것입니다.

이론적으로 n개 조건의 모든 조합은 2^n개이지만, MC/DC는 n+1개로 줄여줍니다. 예: 5개 조건 → 32개 조합 vs. MC/DC는 최소 6개

#### 라. 3조건 예제 완전 분석

예제: `D = (A AND B) OR C`

```text
조건: A, B, C (n=3) → 최소 4개(n+1) 테스트 케이스

전체 진리표:
TC  │ A │ B │ C │ D=(A∧B)∨C │
────┼───┼───┼───┼───────────┤
T1  │ T │ T │ T │     T      │
T2  │ T │ T │ F │     T      │  ← A,B 독립 영향 증명에 사용
T3  │ T │ F │ T │     T      │  ← C 독립 영향 증명에 사용 (T3,T4: C만 변경, D 변경)
T4  │ T │ F │ F │     F      │  ← B 독립 영향 증명에 사용 (T2,T4: B만 변경, D 변경)
T5  │ F │ T │ T │     T      │
T6  │ F │ T │ F │     F      │  ← A 독립 영향 증명에 사용 (T2,T6: A만 변경, D 변경)
T7  │ F │ F │ T │     T      │
T8  │ F │ F │ F │     F      │

MC/DC 최소 케이스 선택:
  A의 영향: T2(T,T,F→T) vs T6(F,T,F→F): A만 다름, D 다름 ✓
  B의 영향: T2(T,T,F→T) vs T4(T,F,F→F): B만 다름, D 다름 ✓
  C의 영향: T3(T,F,T→T) vs T4(T,F,F→F): C만 다름, D 다름 ✓

최종 선택: {T2, T3, T4, T6} → 4개 = n+1 = 3+1 ✓
```

### 3. 핵심 키워드 (Must-Have)

| 키워드 | 설명 |
|--------|------|
| 조건 커버리지(CC) | 각 개별 조건이 T/F 모두 취하는 것. DC를 보장하지 않는 약점 |
| 결정 커버리지(DC) | 결정 전체가 T/F 모두 취하는 것. 개별 조건의 영향 미확인 약점 |
| MC/DC | CC + DC + 각 조건의 독립적 영향 증명. 세 가지 동시 만족 |
| 독립 쌍 (Independence Pair) | 한 조건만 다르고 나머지는 같으면서 결정 결과가 달라지는 두 TC 쌍 |
| n+1 최소성 | n개 조건 → 최소 n+1개 TC로 MC/DC 달성 가능 |
| DO-178B/C | 항공 소프트웨어 안전 표준. Level A(최고 수준)에서 MC/DC 의무화 |

### 4. 맥락과 비교

**항공 안전 표준 DO-178B/C에서의 MC/DC**

미국 FAA와 EASA가 요구하는 항공 소프트웨어 안전 표준 DO-178B(현재 DO-178C)는 소프트웨어 레벨에 따라 요구하는 커버리지가 다릅니다.

| DO-178C 레벨 | 위험 수준 | 요구 커버리지 |
|------------|---------|------------|
| Level A (Catastrophic) | 생명 손실 가능 | MC/DC |
| Level B (Hazardous) | 심각한 부상 가능 | DC (결정 커버리지) |
| Level C (Major) | 큰 불편 | SC (구문 커버리지) |
| Level D (Minor) | 작은 불편 | 별도 커버리지 불필요 |

Level A 소프트웨어는 항공기 자동조종장치, 비행 제어 소프트웨어 등이 해당됩니다.

**다른 안전 표준에서의 MC/DC**

| 표준 | 도메인 | MC/DC 관련 |
|------|--------|---------|
| DO-178C | 항공 | Level A 의무 |
| IEC 61508 | 산업 기능 안전 | SIL 4에서 권고 |
| ISO 26262 | 자동차 기능 안전 | ASIL D에서 고려 |

### 5. 실무 제언

**MC/DC 자동화 도구 활용**

MC/DC 테스트 케이스를 수작업으로 설계하는 것은 조건 수가 늘어날수록 매우 복잡합니다. 실무에서는 다음 도구들을 활용합니다.

- **LDRA TBrun**: DO-178C 인증 프로젝트에서 널리 사용. MC/DC 자동 분석 및 케이스 생성
- **VectorCAST**: 임베디드 시스템 테스트 자동화. MC/DC 커버리지 자동 측정
- **Cantata**: C/C++ 단위 테스트 프레임워크. MC/DC 리포트 지원

**MC/DC의 현실적 적용 범위**

DO-178C Level A 인증이 필요한 항공 소프트웨어 외에도, 생명안전(Life-Critical), 임무중요(Mission-Critical) 소프트웨어(자동차 ABS, 의료기기, 원자력 제어)에 적용이 권고됩니다. 일반적인 비즈니스 소프트웨어에 MC/DC를 전면 적용하는 것은 비용 대비 효과가 낮을 수 있으므로, 리스크 기반 접근으로 가장 위험한 로직에만 선택적으로 적용하는 것이 현실적입니다.
