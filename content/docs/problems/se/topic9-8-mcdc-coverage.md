## 해설

### 출제 배경 및 의도

항공기 비행 제어 소프트웨어, 자동차 ABS, 원전 제어 시스템 등 안전 필수(Safety-Critical) 소프트웨어에서는 논리 조건의 오류가 인명 피해로 직결된다. 이에 따라 각 개별 조건이 최종 결정에 독립적으로 영향을 미치는지 검증하는 MC/DC가 DO-178C Level A 등 국제 안전 표준에서 의무화되었다.

기술사 시험에서 이 주제는 세 가지 커버리지(CC/DC/MC/DC)의 단순 정의 나열이 아니라, CC와 DC 각각의 약점을 정확히 이해하고 MC/DC가 어떻게 두 약점을 동시에 극복하는지를 논증하며, 실제 논리식에 대해 독립 쌍(Independence Pair) 기반의 테스트 케이스를 직접 설계하는 능력을 검증한다.

### 1. 커버리지 기법 개요

정 의  • MC/DC(Modified Condition/Decision Coverage)를 각 개별 조건이 다른 조건을 고정한 채 독립적으로 최종 결정 결과를 변경할 수 있음을 증명하는 화이트박스 커버리지 기준이라 한다.
       - n개 조건에 대해 이론상 2^n개 조합이 필요하나 MC/DC는 최소 n+1개 케이스로 달성 가능하다.

```
[CC / DC / MC/DC 비교 구조도 — D = A AND B]

커버리지  테스트 케이스       결과    CC충족  DC충족  독립영향
─────────────────────────────────────────────────────────
CC만      TC1:(T,F)→F         둘다 F    O       X(F만)   X
          TC2:(F,T)→F
─────────────────────────────────────────────────────────
DC        TC1:(T,T)→T         T+F       △       O        X
          TC2:(F,T)→F
─────────────────────────────────────────────────────────
MC/DC     TC1:(T,T)→T  ┐A쌍   T+F       O       O        O
          TC2:(F,T)→F  ┘
          TC3:(T,F)→F  ─B쌍
          ※ n+1=3개로 달성
```

- CC만 충족해도 결정(D)이 항상 False이면 DC를 보장하지 못하는 치명적 약점이 있다.

### 2. MC/DC 핵심 구성 요소

1) 세 가지 커버리지 비교

| 구분 | 조건 커버리지(CC) | 결정 커버리지(DC) | MC/DC |
|------|----------------|----------------|-------|
| 검증 대상 | 개별 조건 T/F | 결정 전체 T/F | CC + DC + 독립 영향 |
| 약점 | DC 미보장 | 개별 조건 영향 미검증 | 없음 (세 가지 동시 충족) |
| 테스트 케이스 | 2개 (n=2) | 2개 | n+1개 |
| 안전 표준 적용 | 미채택 | DO-178C Level B | DO-178C Level A |

- MC/DC는 CC와 DC의 합집합이 아닌 독립 영향 조건을 추가로 요구한다.

2) 독립 쌍(Independence Pair) 개념

| 항목 | 설명 |
|------|------|
| 독립 쌍 정의 | 한 조건만 다르고 나머지는 동일하면서 결정 결과가 달라지는 두 TC 쌍 |
| A의 독립 쌍 | (T,T)→T vs (F,T)→F: A만 변경, D 변경됨 |
| B의 독립 쌍 | (T,T)→T vs (T,F)→F: B만 변경, D 변경됨 |
| 선택 기준 | 한 조건당 1개 독립 쌍 필요 → n개 조건 → n+1개 최소 TC |

- 독립 쌍을 식별하여 중복 테스트 케이스를 공유하면 n+1개 최소성이 달성된다.

### 3. 3조건 예제 완전 분석 및 안전 표준 적용

1) D = (A AND B) OR C 에 대한 MC/DC 설계

```
TC  │ A │ B │ C │ D=(A∧B)∨C │ 역할
────┼───┼───┼───┼───────────┼──────────────────────────
T2  │ T │ T │ F │     T      │ A독립쌍(T2,T6), B독립쌍(T2,T4)
T3  │ T │ F │ T │     T      │ C독립쌍(T3,T4)
T4  │ T │ F │ F │     F      │ B독립쌍(T2,T4), C독립쌍(T3,T4)
T6  │ F │ T │ F │     F      │ A독립쌍(T2,T6)

최종: {T2,T3,T4,T6} = 4개 = n+1 = 3+1 ✓
```

- 각 조건의 독립 쌍이 모두 확인되어 MC/DC 커버리지 달성; 안전 표준 증거 자료로 활용된다.

2) DO-178C 안전 레벨별 커버리지 요구

| DO-178C 레벨 | 위험 수준 | 요구 커버리지 | 적용 예시 |
|------------|---------|------------|---------|
| Level A (Catastrophic) | 생명 손실 | MC/DC | 자동조종장치, 비행 제어 |
| Level B (Hazardous) | 심각한 부상 | 결정 커버리지(DC) | 경보 시스템 |
| Level C (Major) | 큰 불편 | 구문 커버리지(SC) | 지상 지원 시스템 |
| Level D (Minor) | 작은 불편 | 불필요 | 관리 유틸리티 |

- IEC 61508 SIL 4, ISO 26262 ASIL D도 MC/DC를 권고하며 안전 필수 소프트웨어의 국제 표준 커버리지로 정착하였다.  "끝"

### 실무 제언

**조건 수 증가에 따른 복잡도 문제**
- **챌린지**: 조건이 5개 이상이면 독립 쌍 식별과 최소 케이스 선택이 수작업으로는 비현실적이다.
- **제언**: LDRA TBrun, VectorCAST, Cantata 등 DO-178C 인증 도구를 활용하면 MC/DC 케이스 자동 생성 및 커버리지 측정이 가능하고 감사 근거 문서도 자동 생성된다.

**일반 비즈니스 소프트웨어에의 과잉 적용 문제**
- **챌린지**: 모든 로직에 MC/DC를 적용하면 테스트 비용이 과도해진다.
- **제언**: 리스크 기반 접근으로 안전 필수 컴포넌트(결제 로직, 인증 로직 등)에만 선택적으로 적용하고, 나머지는 결정 커버리지(DC)로 충분하다.

**AI/ML 기반 시스템의 MC/DC 한계**
- **챌린지**: 비결정론적(Non-deterministic) AI 시스템에는 전통적 MC/DC 적용이 어렵다.
- **제언**: ISO 21448(SOTIF)과 같은 AI 전용 안전 표준을 병행 적용하고, 기존 MC/DC는 AI를 둘러싼 결정론적 안전 로직에 집중 적용한다.
