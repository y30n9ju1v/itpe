## 해설

> **소프트웨어 개발에 드는 비용을 과학적으로 예측하는 방법론의 총체**  
> 하향식·상향식·알고리즘 방식의 장단점을 비교하고 실무 적용 맥락을 이해

"이 프로젝트에 얼마가 들 것인가?"라는 질문은 SW 개발에서 가장 어렵고 중요한 질문 중 하나입니다. 잘못된 비용 산정은 프로젝트 실패, 계약 분쟁, 품질 저하로 직결됩니다. SW 비용 산정 방법론은 크게 하향식(Top-down), 상향식(Bottom-up), 알고리즘 방식(Algorithmic)으로 나뉘며, 각각의 강점이 다르기 때문에 실무에서는 복합적으로 활용합니다.

### 1. 출제 배경 및 의도

공공 SW 사업의 예산 편성, 입찰 원가 산정, 계약 후 변경 관리 모든 단계에서 비용 산정 능력이 필요합니다. 채점관은 각 방법의 메커니즘과 함께 "언제, 왜 어떤 방법을 선택하는지"의 판단력을 평가합니다. 단순 암기보다 장단점 비교와 복합 활용 전략을 서술하는 것이 고득점의 핵심입니다.

### 2. 개념의 본질과 작동 메커니즘

SW 비용 산정의 본질은 **불확실성 관리**입니다. 프로젝트 초기에는 요구사항이 불명확하므로 불확실성이 크고 산정 오차도 큽니다. 프로젝트가 진행될수록 불확실성이 줄어 산정 정확도가 높아집니다. 이를 "불확실성 콘(Cone of Uncertainty)"이라 합니다.

```text
[불확실성 콘과 산정 방법 적합성]

프로젝트 초기                          프로젝트 후기
(불확실성 높음)                      (불확실성 낮음)

←───────────────────────────────────────────→
│   개념화   │  요구사항  │  설계  │  구현  │
│            │            │        │        │
│ 하향식     │ 하향식/FP  │  FP    │ 상향식 │
│ 전문가     │ COCOMO II  │ 상세   │ WBS기반│
│ 판단·델파이│ 초기모델   │ FP산정 │        │
│            │            │        │        │
│ 오차 ±4x  │ 오차 ±2x  │ ±1.5x │ ±1.25x│
```

### 3. 핵심 키워드 (Must-Have)

#### 분류 체계: SW 비용 산정 방법 전체 지도

```text
SW 비용 산정 방법
├── 하향식 (Top-down)
│   ├── 전문가 판단 (Expert Judgment)
│   └── 델파이 기법 (Delphi Technique)
├── 상향식 (Bottom-up)
│   └── WBS 기반 세부 산정
└── 알고리즘 방식 (Algorithmic/Parametric)
    ├── LOC 기반: COCOMO, COCOMO II
    └── FP 기반: 기능점수 × 단가
```

#### 1) 하향식 방법 (Top-down)

**가. 전문가 판단 (Expert Judgment)**

- **방법**: 경험 많은 전문가(PM, 아키텍트)가 유사 프로젝트 경험을 바탕으로 총 비용을 직관적으로 산정
- **장점**: 빠른 산정, 전문가의 암묵지(Tacit Knowledge) 활용
- **단점**: 주관적, 검증 어려움, 전문가 편향(낙관적 편향) 발생 가능
- **적합**: 초기 개념 단계, 예비 타당성 조사

**나. 델파이 기법 (Delphi Technique)**

- **방법**: 여러 전문가의 독립적 의견을 반복적으로 수렴하여 합의(Consensus)를 도출
- **절차**: 전문가 선정 → 익명 설문(1라운드) → 결과 공유 → 재설문(N라운드) → 수렴

```text
[델파이 기법 절차]

전문가 A ──→ ┐
전문가 B ──→ ├→ 익명 응답 → 조정자 취합 → 결과 피드백
전문가 C ──→ ┘         ↑                        │
                        └────────────────────────┘
                              (수렴할 때까지 반복)
```

- **장점**: 여러 전문가 의견 수렴으로 객관성 향상, 집단사고(Groupthink) 방지
- **단점**: 시간 소요, 전문가 확보 어려움
- **적합**: 요구사항 불명확 단계, 대형 프로젝트 초기 타당성 검토

#### 2) 상향식 방법 (Bottom-up)

- **방법**: WBS(Work Breakdown Structure)로 작업을 최소 단위까지 분해한 후, 각 작업의 공수를 개별 산정하여 합산
- **절차**: WBS 작성 → 작업 패키지별 일정·공수 산정 → 총합 계산

```text
[상향식 산정 구조]

프로젝트 전체
├── 요구분석 (10 MM)
│   ├── 인터뷰 (3 MM)
│   ├── 문서화 (4 MM)
│   └── 검토 (3 MM)
├── 설계 (30 MM)
│   ├── 아키텍처 설계 (10 MM)
│   └── 상세 설계 (20 MM)
└── 구현 (60 MM)
    ├── 모듈 A 개발 (20 MM)
    └── 모듈 B 개발 (40 MM)

총 공수 = 100 MM
```

- **장점**: 높은 정확도, 추적 가능(작업별 원가 통제 가능)
- **단점**: 시간 소요, 요구사항이 명확해야 적용 가능, WBS 작성 역량 필요
- **적합**: 요구사항 분석 완료 후, 상세 설계 단계

#### 3) 알고리즘 방식 — COCOMO II

COCOMO(Constructive Cost Model)는 Boehm이 제안한 수학적 비용 산정 모델. COCOMO II는 1990년대 이후 현대 개발 환경(객체지향, 재사용, 나선형 개발)에 맞게 개정된 버전.

**COCOMO II의 3가지 세부 모델**

| 모델 | 적용 단계 | 산정 입력값 | 정확도 |
|------|----------|-----------|--------|
| **Application Composition** (응용 구성) | 요구사항 초기 | Object Points (화면 수 등) | 낮음 (±4배) |
| **Early Design** (초기 설계) | 개념/아키텍처 설계 | UFP (미조정 FP) | 중간 (±2배) |
| **Post-Architecture** (아키텍처 이후) | 상세 설계 이후 | LOC, FP | 높음 (±1.5배) |

**COCOMO II 기본 공식 (Post-Architecture)**

```text
공수(MM) = A × Size^E × Π(EMi)

여기서:
  A: 상수 (보통 2.94)
  Size: KSLOC (천 줄의 소스 코드) 또는 FP
  E: 규모 지수 (1.01~1.26, 프로젝트 규모 요인)
  EMi: 비용 동인(Cost Driver) 보정 계수 17가지
       (신뢰성 요구, 데이터베이스 크기, 개발자 역량 등)
```

**COCOMO 구버전 vs COCOMO II 비교**

| 구분 | COCOMO (1981) | COCOMO II (1995~) |
|------|--------------|------------------|
| 기반 | LOC 중심 | FP + LOC 모두 지원 |
| 개발 방법론 | 폭포수 모델 가정 | 나선형, 프로토타입 지원 |
| 재사용 처리 | 없음 | 재사용 코드 보정 계수 포함 |
| 소프트웨어 유형 | Organic/Semi-detached/Embedded | 프로젝트 특성에 따라 동적 적용 |

#### COCOMO 구버전의 3가지 유형 (참고)

| 유형 | 특징 | 예시 |
|------|------|------|
| **Organic (유기적)** | 소규모, 단순, 익숙한 팀 | 내부 업무 자동화 툴 |
| **Semi-detached (반분리)** | 중규모, 혼합 팀 | 중소형 정보시스템 |
| **Embedded (내장)** | 대규모, 복잡한 제약 조건 | 항공기 제어 시스템 |

### 4. 맥락과 비교

#### 3가지 방법 종합 비교

| 구분 | 하향식 (전문가) | 하향식 (델파이) | 상향식 (WBS) | 알고리즘 (COCOMO/FP) |
|------|--------------|--------------|------------|---------------------|
| **적용 시기** | 초기 | 초기~중기 | 중기~후기 | 중기~후기 |
| **정확도** | 낮음 | 중간 | 높음 | 중간~높음 |
| **소요 시간** | 짧음 | 중간 | 길음 | 중간 |
| **객관성** | 낮음 | 중간 | 높음 | 높음 |
| **요구사항** | 불필요 | 개략적 필요 | 상세 필요 | 규모 산정 필요 |
| **주요 단점** | 주관적 편향 | 전문가 확보 어려움 | WBS 상세화 필요 | 모델 파라미터 보정 어려움 |

### 5. 실무 제언

**복합 산정 접근법의 필요성**

- **챌린지**: 단일 방법으로는 신뢰성 있는 산정이 어렵습니다. 상향식 WBS 산정만 믿었다가 누락된 작업이 나중에 발견되는 "누락 오류"가 빈번합니다.
- **제언**: 실무에서는 **3점 산정(Three-Point Estimation)** — 낙관치(O), 최빈치(M), 비관치(P) — 을 결합한 PERT 방식 `공수 = (O + 4M + P) / 6`을 상향식 WBS와 함께 사용하면 불확실성을 정량화할 수 있습니다. 또한 알고리즘 방식(FP기반)으로 교차 검증하여 두 결과의 차이가 15% 이내면 신뢰, 이상이면 재검토하는 프로세스가 효과적입니다.

**기술사 시험 답안 작성 팁**

- 하향식/상향식/알고리즘 3분류 체계를 명확히 제시하고 각각 대표 기법을 설명하십시오.
- COCOMO II의 3단계 모델(Application Composition / Early Design / Post-Architecture)을 언급하면 깊이 있는 답안이 됩니다.
- "불확실성 콘(Cone of Uncertainty)"의 개념으로 각 방법의 적용 시점을 연결하면 차별화됩니다.
