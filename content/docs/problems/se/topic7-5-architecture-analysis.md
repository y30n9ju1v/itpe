## 문제

소프트웨어 구조의 개량 및 개선을 위한 소프트웨어 아키텍처 분석방법에 대하여 다음 내용을 설명하시오.

1. 소프트웨어 아키텍처 분석의 필요성
2. 소프트웨어 아키텍처 정방향 분석과 역방향 분석 개념
3. 소프트웨어 아키텍처 평가 기법 중 ATAM(Architecture Trade-off Analysis Method)

## 해설

### 출제 배경 및 의도

126회 기술사 시험에서 이 문제가 출제된 것은 레거시 시스템의 현대화(Modernization)가 공공 및 민간 IT 모두에서 핵심 과제로 부상했기 때문이다. 기존 시스템을 이해하기 위한 역방향 분석(Reverse Engineering)과 새로운 아키텍처를 검증하기 위한 정방향 분석(Forward Analysis)을 구분하고, 이를 ATAM으로 통합하는 관점이 이 문제의 핵심이다.

채점관은 단순한 ATAM 설명을 넘어, 아키텍처 분석의 필요성에서 시작하여 정방향/역방향 분석의 차이, ATAM의 단계와 결과물을 논리적으로 연결하는 서술 능력을 평가한다. Boehm의 결함 비용 증가 법칙과 아키텍처 부식(Architecture Erosion) 개념을 연계하면 고득점이 가능하다.

### 1. 소프트웨어 아키텍처 분석 개요

정 의  • 소프트웨어 아키텍처가 품질속성 요구사항을 충족하고 의도된 구조를
       유지하는지 체계적으로 검증·평가하는 활동
       - 정방향: 설계 → 구현 방향의 사전 검증
       - 역방향: 구현 → 설계 방향의 구조 복원

```
[소프트웨어 아키텍처 분석 전체 지형]

  요구사항·품질속성
         │
         ▼  ← 정방향 분석(Forward Analysis)
  아키텍처 설계
  ├── 설계 검토(Design Review)
  ├── 시뮬레이션/프로토타입
  └── ATAM/CBAM 평가
         │
         ▼
  구현·배포된 시스템
         │
         ▼  ← 역방향 분석(Reverse Analysis)
  아키텍처 복원(Architecture Recovery)
  ├── 코드 분석 (ArchUnit, SonarQube)
  ├── 의존성 추출 (Dependency Graph)
  └── 아키텍처 재문서화
```

- 아키텍처 분석은 설계 단계에서 수행할수록 결함 수정 비용이 최소화된다(Boehm 법칙).

### 2. 핵심 내용 상세

1) 소프트웨어 아키텍처 분석의 필요성

| 필요성 | 설명 | 미수행 시 결과 |
|-------|------|-------------|
| 조기 결함 발견 | 구현 전 설계 단계에서 아키텍처 결함 제거 | 운영 단계 발견 시 수정 비용 10~100배 |
| 품질속성 충족 검증 | 성능·가용성·보안 등 비기능 요구 사전 확인 | 인수 단계 실패, SLA 위반 |
| 변경 위험도 파악 | 아키텍처 변경 시 영향 범위·위험 사전 식별 | 예상치 못한 연쇄 장애 |
| 트레이드오프 명시화 | 상충하는 품질속성 간 의사결정 근거 마련 | 이해관계자 간 사후 분쟁 |
| 레거시 이해 | 문서화되지 않은 기존 시스템 구조 파악 | 현대화 프로젝트 실패 |
| 아키텍처 부식 방지 | 코드 진화 과정에서 의도된 구조 이탈 방지 | Architecture Drift·Erosion |

```
[아키텍처 부식의 진행]

  초기 설계(As-Designed)      시간 경과(As-Built)
  ┌──────────────────┐        ┌──────────────────┐
  │  계층1 (UI)      │        │  계층1 (UI)      │
  │  ↓ (단방향)      │  →→→   │  ↕ (양방향!)    │ ← 계층 오염
  │  계층2 (비즈)    │        │  계층2 (비즈)    │
  │  ↓ (단방향)      │        │  ↓↑             │
  │  계층3 (데이터)  │        │  계층3 (데이터)  │
  └──────────────────┘        └──────────────────┘
```

- 아키텍처 분석은 설계 품질 유지와 기술 부채 예방을 위한 핵심 활동이다.

2) 정방향 분석과 역방향 분석 비교

| 구분 | 정방향 분석 | 역방향 분석 |
|------|-----------|-----------|
| 방향 | 설계 → 구현 | 구현 → 설계 |
| 목적 | 설계 결정 사전 검증 | 실제 구조 이해·복원 |
| 시점 | 개발 초기~중반 | 유지보수, 현대화 프로젝트 |
| 기법 | ATAM, 리뷰, 시뮬레이션, 형식적 검증 | 정적/동적 분석, 아키텍처 마이닝 |
| 도구 | SparxEA, ArchiMate | SonarQube, ArchUnit, Lattix |
| 결과물 | 위험점 목록, 개선 권고서 | As-Built 아키텍처 다이어그램 |

- 정방향 분석은 예방적, 역방향 분석은 진단적 성격을 가지며 상호 보완적으로 활용해야 한다.

### 3. ATAM 상세 및 방법론 비교

1) ATAM 수행 단계 및 결과물

| 단계 | 활동 | 산출물 |
|-----|------|-------|
| 준비 | 평가팀 구성, 이해관계자 확인 | 평가 계획서 |
| 평가 1단계 | 비즈니스 동인·아키텍처 발표, 유틸리티 트리 생성 | 유틸리티 트리 |
| 평가 2단계 | 시나리오 브레인스토밍, 우선순위화, 재분석 | 시나리오 우선순위 목록 |
| 후속 | 위험점·트레이드오프점 정리, 보고서 작성 | 최종 평가 보고서 |

```
[ATAM의 가치: Boehm 결함 비용 증가 법칙]

  구현 전 ATAM: 아키텍처 문제 수정 비용 = $1,000
  구현 후 발견: 동일 문제 수정 비용     = $100,000
               (코드 재작성 + 재테스트 + 배포 비용)
```

- ATAM 4대 결과물: ①위험점(Risks) ②비위험점(Non-Risks) ③감도점(Sensitivity Points) ④트레이드오프점(Tradeoff Points)

2) SA 평가 방법론 비교

| 구분 | SAAM | ATAM | CBAM | ARID |
|------|------|------|------|------|
| 초점 | 변경용이성 | 다중 QA 트레이드오프 | 경제적 ROI | 설계 적합성 |
| 규모 | 소·중규모 | 대규모 공식 평가 | ATAM 후속 | 간단한 리뷰 |
| 비용 | 낮음 | 높음 (2~3일) | 매우 높음 | 낮음 |
| 정량화 | 불가 | 상대적 우선순위 | ROI 완전 정량화 | 불가 |
| 적용 시점 | 설계 초기·중반 | 설계 완료 후 | ATAM 이후 | 설계 진행 중 |

- 프로젝트 규모와 중요도에 따라 SAAM → ATAM → CBAM 순으로 방법론을 선택하며, 경량 ATAM을 CI/CD에 내재화하는 것이 현대적 접근이다.  "끝"

### 실무 제언

**레거시 시스템 역방향 분석의 전략적 활용**
- **챌린지**: 공공기관·대기업의 핵심 시스템 대부분은 10~20년 된 레거시로, 설계 문서가 실제 코드와 일치하지 않는다. 현대화·클라우드 전환 전에 "지금 시스템이 어떻게 생겼는가"를 파악하는 역방향 분석이 반드시 선행되어야 한다.
- **제언**: JDepend, ArchUnit, SonarQube로 의존성 매트릭스(DSM)를 추출하고 순환 의존성·레이어 위반을 시각화한다. 이를 기반으로 Strangler Fig Pattern(새 기능부터 새 아키텍처, 구 기능은 점진적 대체)을 적용하는 것이 현실적인 현대화 전략이다.

**정방향 분석과 CI/CD 통합**
- **챌린지**: ATAM은 일회성 대규모 평가 이벤트로 끝나는 경우가 많고, 이후 코드 변경 과정에서 아키텍처가 조용히 부식되어도 아무도 감지하지 못한다.
- **제언**: 계층 간 의존성 규칙 위반, 순환 의존성, 팬-인/팬-아웃 임계값 초과 등의 아키텍처 규칙을 ArchUnit 테스트로 작성하고 PR 단계마다 자동 검증하는 지속적 아키텍처 검증(Continuous Architecture Validation)을 CI/CD에 내재화해야 한다.
