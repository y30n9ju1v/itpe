## 문제

소프트웨어 구조의 개량 및 개선을 위한 소프트웨어 아키텍처 분석방법에 대하여 다음 내용을 설명하시오.

1. 소프트웨어 아키텍처 분석의 필요성
2. 소프트웨어 아키텍처 정방향 분석과 역방향 분석 개념
3. 소프트웨어 아키텍처 평가 기법 중 ATAM(Architecture Trade-off Analysis Method)

## 해설

> **아키텍처는 설계하는 것만큼 분석하는 것도 중요하다**  
> 정방향 분석으로 설계를 검증하고, 역방향 분석으로 레거시를 이해하며, ATAM으로 트레이드오프를 평가한다

수십억 원을 투자한 시스템이 운영 1년 만에 성능 문제로 리팩토링(Refactoring, 동작은 유지한 채 내부 코드 구조를 개선하는 작업)을 시작하는 경우가 많습니다. 이러한 비극의 공통점은 "아키텍처 분석"이 체계적으로 수행되지 않았다는 것입니다. 아키텍처 분석은 설계 결정의 결과를 사전에 예측하고(정방향), 이미 구현된 시스템의 실제 구조를 파악하며(역방향), 여러 설계 대안 간의 품질 트레이드오프를 평가합니다(ATAM).

### 1. 출제 배경 및 의도

126회 기술사 시험에서 이 문제가 출제된 것은 레거시 시스템의 현대화(Modernization)가 공공 및 민간 IT 모두에서 핵심 과제로 부상했기 때문입니다. 기존 시스템을 이해하기 위한 **역방향 분석(Reverse Engineering)**과 새로운 아키텍처를 검증하기 위한 **정방향 분석(Forward Analysis)**을 구분하고, 이를 ATAM으로 통합하는 관점이 이 문제의 핵심입니다.

채점관은 단순한 ATAM 설명을 넘어, **왜 아키텍처 분석이 필요한지(필요성)**에서 시작하여 **정방향/역방향 분석의 차이**, 그리고 **ATAM의 실질적인 단계와 결과물**을 논리적으로 연결하는 서술 능력을 평가합니다.

### 2. 개념의 본질과 작동 메커니즘

```text
[소프트웨어 아키텍처 분석 전체 지형]

  요구사항·품질속성
         │
         ▼  ← 정방향 분석(Forward Analysis)
  아키텍처 설계
  (Architecture Design)
  ├── 설계 검토(Design Review)
  ├── 시뮬레이션/프로토타입
  └── ATAM/CBAM 평가
         │
         ▼
  구현·배포된 시스템
  (Implemented System)
         │
         ▼  ← 역방향 분석(Reverse Analysis)
  아키텍처 복원
  (Architecture Recovery)
  ├── 코드 분석 (ArchUnit, SonarQube)
  ├── 의존성 추출 (Dependency Graph)
  └── 아키텍처 재문서화
```

### 3. 핵심 키워드 (Must-Have)

#### 1) 소프트웨어 아키텍처 분석의 필요성

소프트웨어 아키텍처 분석이 필요한 이유는 다음과 같습니다:

| 필요성 | 설명 | 미수행 시 결과 |
|-------|------|-------------|
| **조기 결함 발견** | 구현 전 설계 단계에서 아키텍처 결함 제거 | 운영 단계에서 발견 시 수정 비용 10~100배 증가 |
| **품질속성 충족 검증** | 성능·가용성·보안 등 비기능 요구 충족 여부 사전 확인 | 인수 단계 실패, SLA 위반 |
| **변경 위험도 파악** | 아키텍처 변경 시 영향 범위와 위험 사전 식별 | 예상치 못한 연쇄 장애 |
| **트레이드오프 명시화** | 상충하는 품질속성 간 의사결정 근거 마련 | 이해관계자 간 사후 분쟁 |
| **레거시 이해** | 문서화되지 않은 기존 시스템 구조 파악 | 현대화 프로젝트 실패 |
| **아키텍처 부식 방지** | 코드 진화 과정에서 의도된 구조의 이탈 방지 | Architecture Drift·Erosion |

**아키텍처 부식(Architecture Erosion)**:
```text
[아키텍처 부식의 진행]

  초기 설계                  시간 경과
  ┌─────────────────┐       ┌─────────────────┐
  │  계층1 (UI)     │       │  계층1 (UI)     │
  │  ↓ (단방향)     │  →→→  │  ↕ (양방향!)   │ ← 계층 오염
  │  계층2 (비즈)   │       │  계층2 (비즈)   │
  │  ↓ (단방향)     │       │  ↓↑            │
  │  계층3 (데이터) │       │  계층3 (데이터) │
  └─────────────────┘       └─────────────────┘
  의도된 아키텍처             실제 구현 아키텍처(부식됨)
```

#### 2) 소프트웨어 아키텍처 정방향 분석과 역방향 분석

**정방향 분석(Forward Analysis)**:

```text
[정방향 분석 흐름]

  요구사항 명세 (SRS)
        │
        ▼
  아키텍처 설계 (Architecture Design)
        │
        ▼  분석 수행
  ┌─────────────────────────────────────┐
  │          정방향 분석 기법            │
  │  - 아키텍처 리뷰 (Architecture Review)│
  │  - ATAM/CBAM (품질속성 평가)        │
  │  - 성능 모델링/시뮬레이션           │
  │  - 프로토타이핑                     │
  │  - 형식적 검증 (Formal Verification) │
  └─────────────────────────────────────┘
        │
        ▼
  분석 결과: 위험점, 트레이드오프, 개선 권고
        │
        ▼
  아키텍처 개선 → 구현
```

**역방향 분석(Reverse Analysis)**:

```text
[역방향 분석 흐름]

  기존(레거시) 소스코드·바이너리
        │
        ▼  분석 도구 활용
  ┌─────────────────────────────────────┐
  │          역방향 분석 기법            │
  │  - 정적 분석 (Static Analysis)      │
  │    : 의존성 그래프, 호출 그래프 추출│
  │  - 동적 분석 (Dynamic Analysis)     │
  │    : 런타임 추적, 프로파일링        │
  │  - 메트릭 분석 (CC, LOC, 결합도)   │
  │  - 아키텍처 복원                   │
  │    (Architecture Recovery/Mining)  │
  └─────────────────────────────────────┘
        │
        ▼
  복원된 아키텍처 모델 (As-Built)
        │
        ▼
  설계 의도(As-Designed)와 비교
  → 아키텍처 부식(Erosion) 식별
  → 현대화(Modernization) 전략 수립
```

| 구분 | 정방향 분석 | 역방향 분석 |
|------|-----------|-----------|
| 방향 | 설계 → 구현 | 구현 → 설계 |
| 목적 | 설계 결정 사전 검증 | 실제 구조 이해·복원 |
| 시점 | 개발 초기~중반 | 유지보수, 현대화 프로젝트 |
| 기법 | ATAM, 리뷰, 시뮬레이션 | 정적/동적 분석, 아키텍처 마이닝 |
| 도구 | SparxEA, ArchiMate | SonarQube, ArchUnit, Lattix |
| 결과물 | 위험점 목록, 개선 권고서 | As-Built 아키텍처 다이어그램 |

#### 3) ATAM (Architecture Trade-off Analysis Method)

(이 파일에서는 필요성과 분석 방법의 맥락에서 ATAM을 집중 설명합니다)

**ATAM의 정의**: 아키텍처가 복수의 품질속성 요구사항을 동시에 충족하는지를 평가하고, 품질속성 간의 상충 관계(트레이드오프)를 식별하는 체계적 평가 방법론 (CMU SEI)

**ATAM이 정방향 분석의 핵심인 이유**:

```text
[ATAM의 가치: 조기 위험 식별]

  구현 전 ATAM 수행 시:
  ┌────────────────────────────────────┐
  │  아키텍처 문제 1개 발견·수정 비용  │
  │  = $1,000 (설계 변경 비용)         │
  └────────────────────────────────────┘

  구현 완료 후 발견 시:
  ┌────────────────────────────────────┐
  │  동일 문제 수정 비용               │
  │  = $100,000 (코드 재작성 + 재테스트│
  │              + 배포 비용)          │
  └────────────────────────────────────┘
  ※ Boehm의 결함 비용 증가 법칙 적용
```

**ATAM 수행 단계 (요약)**:

| 단계 | 활동 | 산출물 |
|-----|------|-------|
| 준비 | 평가팀 구성, 이해관계자 확인 | 평가 계획서 |
| 평가 1단계 | 비즈니스 동인, 아키텍처 발표, 유틸리티 트리 생성 | 유틸리티 트리 |
| 평가 2단계 | 시나리오 브레인스토밍, 우선순위화, 재분석 | 시나리오 우선순위 목록 |
| 후속 | 위험점·트레이드오프점 정리, 보고서 작성 | 최종 평가 보고서 |

**ATAM 결과물 4종**:
- **위험점(Risks)**: 잠재적 문제 아키텍처 결정 목록
- **비위험점(Non-Risks)**: 안전하다고 확인된 결정 목록
- **감도점(Sensitivity Points)**: 단일 품질속성에 강한 영향을 주는 결정
- **트레이드오프점(Tradeoff Points)**: 복수 품질속성에 상반된 영향을 주는 결정

### 4. 맥락과 비교

| 구분 | SAAM | ATAM | CBAM | ARID |
|------|------|------|------|------|
| 초점 | 변경용이성 | 다중 QA 트레이드오프 | 경제적 ROI | 설계 적합성 |
| 규모 | 소·중규모 | 대규모 공식 평가 | ATAM 후속 | 간단한 리뷰 |
| 비용 | 낮음 | 높음 (2~3일) | 매우 높음 | 낮음 |
| 정량화 | 불가 | 상대적 우선순위 | ROI 완전 정량화 | 불가 |
| 적용 시점 | 설계 초기·중반 | 설계 완료 후 | ATAM 이후 | 설계 진행 중 |

### 5. 실무 제언

**레거시 시스템 역방향 분석의 전략적 활용**

- **챌린지**: 공공기관과 대기업의 핵심 시스템 대부분은 10~20년 된 레거시로, 설계 문서가 존재하지 않거나 실제 코드와 일치하지 않습니다. 이런 시스템을 현대화하거나 클라우드로 전환하려면 먼저 "지금 시스템이 어떻게 생겼는가"를 파악하는 역방향 분석이 선행되어야 합니다.
- **제언**: JDepend, ArchUnit, SonarQube의 아키텍처 분석 기능으로 **의존성 매트릭스(Dependency Structure Matrix)**를 추출하고, 순환 의존성(Circular Dependency)과 레이어 위반을 시각화해야 합니다. 이를 기반으로 **Strangler Fig Pattern**(새 기능부터 새 아키텍처로, 구 기능은 점진적으로 대체)을 적용하는 것이 현실적인 현대화 전략입니다.

**정방향 분석과 CI/CD 통합**

- **챌린지**: ATAM은 대규모 평가 이벤트(Big-Room Meeting)로 수행되어 일회성으로 끝나는 경우가 많습니다. 이후 지속적인 코드 변경 과정에서 아키텍처가 조용히 부식되어도 아무도 감지하지 못합니다.
- **제언**: 경량 정방향 분석을 **지속적 아키텍처 검증(Continuous Architecture Validation)** 형태로 CI/CD에 내재화해야 합니다. 구체적으로 "계층 간 의존성 규칙 위반", "순환 의존성 발생", "특정 패키지의 팬-인/팬-아웃 임계값 초과" 등의 아키텍처 규칙을 ArchUnit 테스트로 작성하고, 풀 리퀘스트(PR) 단계마다 자동 검증함으로써 아키텍처 부식을 빌드 단계에서 원천 차단하는 체계를 구축해야 합니다.
