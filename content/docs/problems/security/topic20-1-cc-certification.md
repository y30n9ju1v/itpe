## 문제

공공기관에서는 정보통신시스템 구축 시 의무적으로 보안성 검토를 수행해야 한다. 다음 내용에 대하여 설명하시오.
가. 공공기관에서는 의무적으로 수행하는 보안성 검토

다. 보안성 검토에 활용되는 CC 인증 설명

## 해설

### 1. 공공기관 의무적 보안성 검토

- **법적 근거**: 정보통신기반보호법, 국가정보화기본법, 국가정보원 보안업무규정
- **목적**: 정보시스템 도입·개발·운영 시 보안 취약점을 사전에 확인하여 보안사고 예방
- **의무 대상**: 국가·지자체·공공기관의 주요 정보통신기반시설

| 검토 유형 | 내용 |
|-----------|------|
| 사전 보안성 검토 | 정보시스템 구축 전 보안 요구사항 검토 |
| 취약점 분석·평가 | 운영 중 시스템의 취약점 주기적 점검 |
| 보안 컨설팅 | 보안 아키텍처·정책 수립 자문 |

### 2. 보안성 검토 프로세스

```
① 검토 요청 (공공기관)
   정보시스템 구축 계획 제출

② 사전 검토 (국정원 또는 전문기관)
   시스템 설계서·아키텍처 검토

③ 현장 심층 검토
   기술 취약점 분석, 구성 설정 검토

④ 결과 통보
   보안 취약점 목록, 개선 권고

⑤ 조치 및 확인
   취약점 개선 조치 후 재확인

⑥ 완료 확인서 발급
```

### 3. CC(Common Criteria, 공통평가기준) 인증

- **정의**: 정보보호 제품의 보안 기능과 보증 수준을 국제 기준으로 평가하는 제도(ISO/IEC 15408)
- **국내**: 국가정보원이 지정한 평가기관(ITSC 등)에서 평가, 국정원이 인증서 발급

#### CC 인증 구성요소

| 구성요소 | 설명 |
|----------|------|
| **PP(Protection Profile)** | 특정 유형 제품의 보안 요구사항 템플릿 |
| **ST(Security Target)** | 특정 제품의 보안 기능·요구사항 명세 |
| **TOE(Target of Evaluation)** | 평가 대상 제품 |

#### EAL(Evaluation Assurance Level) 등급

| 등급 | 수준 | 특징 |
|------|------|------|
| EAL1 | 기능적 테스트 | 가장 낮은 보증, 기본 기능 확인 |
| EAL2 | 구조적 테스트 | 설계 정보 분석 |
| EAL3 | 방법론적 테스트 | 개발 환경 심사 |
| EAL4 | 방법론적 설계·검토·테스트 | 상업 제품의 일반적 최고 수준 |
| EAL5~7 | 고보증 | 군사·금융 고보안 제품 |

- **국내 공공기관**: 방화벽·IDS·암호화 장비 등 보안 제품은 CC EAL2 이상 인증 의무

### 실무 제언

**CC 인증 제품 도입 프로세스**
- **챌린지**: 공공기관이 CC 미인증 보안 제품을 도입하면 법적 제재 및 보안감사 지적 사항이 된다.
- **제언**: 조달청 나라장터의 CC 인증 제품 목록과 국정원 사이버안전제품 목록을 우선 참조하고, 신기술 보안 제품 도입 시 국정원 보안적합성 검토를 선행하여 법적 리스크를 제거해야 한다.
