## 해설

### 출제 배경 및 의도

LLM 기반 AI 코딩 도구(GitHub Copilot, Cursor, Claude Code 등)의 확산으로 비전문가도 코드를 생성하는 '바이브 코딩' 시대가 열렸다. 이로 인한 보안 취약점과 소프트웨어 품질 문제를 기술사 답안으로 서술할 수 있어야 한다.

### 1. 바이브 코딩의 개념

정 의  • 자연어 프롬프트로 AI(LLM)에게 코드 생성을 요청하고, 결과물의 세부 내용을 완전히 이해하지 않고도 소프트웨어를 개발하는 방식 (Andrej Karpathy, 2025년 제안)
       - 
  - 프로그래밍 전문 지식 없이도 기능 구현 가능
  - "분위기(Vibe)대로" 원하는 것을 설명하면 AI가 구현
  - GitHub Copilot, Cursor, Replit Agent, Claude Code 등이 도구
- **배경**: LLM의 코드 생성 능력 향상으로 비개발자·시민 개발자층이 소프트웨어를 직접 생산

### 2. 소프트웨어 산업에 미치는 영향

| 구분 | 긍정적 영향 | 부정적 영향 |
|------|-------------|-------------|
| 생산성 | 개발 속도 2~10배 향상 | 코드 품질 검증 어려움 |
| 접근성 | 비개발자도 소프트웨어 생산 가능 | 저품질 소프트웨어 양산 |
| 비용 | 개발 인력·비용 절감 | 보안 취약 코드 증가 |
| 혁신 | MVP(최소 기능 제품) 빠른 검증 | 레거시·기술 부채 증가 |
| 직업 | 반복 코딩 업무 자동화 | 주니어 개발자 진입 장벽 변화 |

### 3. 바이브 코딩 보안 취약점 및 고려사항

1) 주요 보안 취약점

| 취약점 유형 | 설명 |
|-------------|------|
| **훈련 데이터 취약점 재현** | LLM이 취약한 코드 패턴으로 학습 → 취약 코드 생성 |
| **컨텍스트 이해 부족** | 보안 요구사항 미반영, 임시방편적 코드 생성 |
| **비밀정보 하드코딩** | API 키·패스워드를 코드에 직접 삽입 |
| **입력 검증 부재** | SQL Injection, XSS 등 미방어 코드 생성 |
| **의존성 취약점** | 취약한 버전의 라이브러리 사용 |
| **과도한 권한** | 최소 권한 원칙 미적용 |
| **환각(Hallucination)** | 존재하지 않는 API·라이브러리 사용 코드 생성 |

2) 고려사항

| 항목 | 고려사항 |
|------|----------|
| 코드 리뷰 | AI 생성 코드도 반드시 인간 보안 리뷰 필요 |
| SAST 적용 | AI 코드에 정적 분석 도구 자동 적용 |
| 시크릿 관리 | .gitignore, Secret Scanner로 하드코딩 방지 |
| 법적 책임 | 오픈소스 라이선스 준수, 저작권 이슈 |
| 의존성 감사 | SCA(Software Composition Analysis) 적용 |
| 프롬프트 보안 | 시스템 프롬프트에 보안 요구사항 명시 |


- 바이브 코딩은 LLM이 자연어 지시만으로 코드를 생성해 개발 속도를 혁신적으로 높이지만, AI 생성 코드의 보안 취약점·라이선스 리스크·기술 부채가 내재되므로 SAST/DAST 자동화 파이프라인과 시큐어 코딩 원칙 기반의 인간 검토를 필수적으로 결합해야 함.  "끝"

### 실무 제언

**AI 코딩 보안 거버넌스 수립**
- **챌린지**: 개발팀이 Copilot·Cursor로 생성한 코드를 충분한 검토 없이 프로덕션에 배포하면 보안 사고가 발생한다.
- **제언**: AI 생성 코드에도 동일한 보안 게이트를 적용하는 "AI-Aware DevSecOps" 파이프라인을 구축하고, Semgrep·GitHub Advanced Security로 AI 생성 코드의 취약 패턴을 자동 탐지하며, 개발자가 생성된 코드를 이해하고 책임지는 "책임있는 바이브 코딩" 문화를 수립해야 한다.
