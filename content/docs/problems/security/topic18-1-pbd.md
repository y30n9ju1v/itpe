## 해설

### 출제 배경 및 의도

PBD(Privacy By Design)는 시스템 설계 단계부터 개인정보 보호를 내재화하는 원칙으로, GDPR 제25조의 법적 기반이 되었다. 7대 원칙과 8대 전략을 서술하고 국내 개인정보보호법 제3조와의 관계를 설명해야 한다.

### 1. PBD 개요

정 의  • Ann Cavoukian(캐나다 온타리오 개인정보보호위원)이 제창한 개념으로, ICT 부야의 프라이버시 보호를 위한 주요방법론으로서 개인정보 보호를 시스템 설계 단계부터 내재화하는 접근법
       - 사후 개인정보 보호가 아닌 설계 단계부터의 선제적 보호
- **법적 근거**: GDPR 제25조(설계에 의한 데이터 보호), 국내 개인정보보호법 제3조

### 2. PBD 7대 원칙

| 원칙 | 내용 |
|------|------|
| **①사전적·예방적** | 프라이버시 침해를 사전에 예방, 사후 대응이 아닌 예방 중심 |
| **②기본값으로서의 프라이버시** | 아무것도 하지 않아도 자동으로 최대 프라이버시 보호 |
| **③설계에 내재화** | 시스템 아키텍처에 프라이버시를 핵심 기능으로 포함 |
| **④완전한 기능** | 보안 vs 프라이버시가 아닌, 보안 AND 프라이버시(제로섬 거부) |
| **⑤전 수명주기 보호** | 수집부터 파기까지 모든 생명주기에서 보호 |
| **⑥가시성·투명성** | 관련 당사자에게 독립적 검증 가능하도록 공개 |
| **⑦사용자 중심** | 이익은 사용자에게, 사용자가 프라이버시를 제어 가능 |

### 3. PBD 8대 전략

| 분류 | 전략 | 내용 |
|------|------|------|
| **데이터 지향** | 최소화(Minimise) | 수집 데이터 최소화 |
| | 분리(Separate) | 개인정보를 다른 데이터와 분리 저장 |
| | 추상화(Abstract) | 식별성이 낮은 집계·추상 수준으로 처리 |
| | 숨기기(Hide) | 개인정보 관련 속성 은폐 |
| **프로세스 지향** | 알리기(Inform) | 정보 처리 내용을 정보주체에게 공지 |
| | 통제(Control) | 정보주체가 자신의 데이터를 직접 제어 |
| | 집행(Enforce) | 프라이버시 정책을 기술적으로 집행 |
| | 시연(Demonstrate) | 프라이버시 준수 여부를 입증 |

### 4. PBD와 국내 개인정보보호법 제3조 비교

| PBD 원칙 | 개인정보보호법 제3조 |
|-----------|---------------------|
| 사전적·예방적 | 제3조 1항: 목적에 필요한 최소한의 개인정보 수집 |
| 기본값으로서의 프라이버시 | 제3조 6항: 개인정보를 익명 또는 가명으로 처리 |
| 설계에 내재화 | 제3조 4항: 안전하게 관리하기 위한 필요한 조치 |
| 전 수명주기 보호 | 제3조 5항: 목적 달성 시 파기 |
| 사용자 중심 | 제3조 3항: 정보주체 권리 보장 |


- PBD는 시스템 설계 초기부터 프라이버시를 기본값으로 내재화하는 방법론으로 GDPR 제25조의 법적 요건이기도 하며, 사후 보완보다 설계 단계 적용이 개인정보 보호 비용을 대폭 절감하고 정보주체 신뢰를 높이는 데 효과적임.  "끝"

### 실무 제언

**AI 개발 단계 PBD 적용**
- **챌린지**: AI 모델 개발 시 대량의 개인정보가 훈련 데이터로 사용되어 프라이버시 침해 위험이 크다.
- **제언**: AI 프로젝트 초기 설계 단계에서 PIA(개인정보 영향평가)를 수행하고, 데이터 최소화(Minimise)·가명처리(Abstract)·연합학습(Federated Learning) 등 PBD 원칙을 기술 아키텍처에 반영하여 '개인정보처리자 중심'이 아닌 '정보주체 중심' AI를 설계해야 한다.
