## 해설

### 출제 배경 및 의도

블록 암호화의 운용 모드(ECB·CBC·CFB·OFB·CTR)와 내부 구조(Feistel vs SPN)를 구분하여 설명할 수 있는지를 검증한다. 각 운용 모드의 특징·장단점과 대표 알고리즘(DES=Feistel, AES=SPN)을 연결하여 서술하면 고득점이 가능하다.

### 1. 블록암호화 운용모드별 동작방식

블록 암호화는 고정 크기(64bit/128bit) 블록 단위로 암호화하며, 동일한 평문 블록이 항상 같은 암호문으로 변환되는 문제(패턴 노출)를 해결하기 위해 운용 모드를 적용한다.

| 운용 모드 | 전체명 | 동작 방식 | 특징 | 병렬처리 |
|-----------|--------|-----------|------|----------|
| **ECB** | Electronic Codebook | 각 블록 독립 암호화 | 동일 평문 → 동일 암호문 패턴 노출, 사용 비권장 | 가능 |
| **CBC** | Cipher Block Chaining | 이전 암호문 블록을 XOR 후 암호화 | 패턴 제거, IV 필요, 직렬 처리 | 복호화만 가능 |
| **CFB** | Cipher Feedback | 이전 암호문을 암호화한 값과 평문 XOR | 스트림 암호화 유사, 오류 전파 | 복호화만 가능 |
| **OFB** | Output Feedback | 키 스트림을 독립적으로 생성 후 XOR | 오류 전파 없음, 전처리 가능 | 키 스트림만 가능 |
| **CTR** | Counter | 카운터 값을 암호화한 값과 평문 XOR | 병렬 처리 가능, 임의 접근 가능 | 완전 가능 |

```
CBC 모드 동작 예시:
IV ──XOR──▶ [암호화] ──▶ C1 ──XOR──▶ [암호화] ──▶ C2
      ↑                         ↑
      P1                        P2
```

### 2. Feistel 구조 vs SPN 구조 비교

| 구분 | Feistel 구조 | SPN(Substitution-Permutation Network) |
|------|-------------|---------------------------------------|
| 개요 | 블록을 L/R 반으로 나눠 라운드 함수 적용 | 전체 블록에 S-Box(치환)와 P-Box(전치) 반복 적용 |
| 역연산 | 암호화·복호화에 같은 구조 사용(키 순서 역전) | 역 S-Box, 역 P-Box가 별도 필요 |
| 속도 | 느림(절반씩 처리) | 빠름(전체 블록 병렬 처리) |
| 구현 복잡도 | 낮음(역함수 불필요) | 다소 높음(역연산 필요) |
| 대표 알고리즘 | DES, 3DES, Blowfish, SEED | AES(Rijndael), PRESENT |
| 안전성 | 라운드 수에 비례 | S-Box 설계가 안전성의 핵심 |

```
Feistel 1라운드:
(L, R) → (R, L XOR F(R, Ki))

SPN 1라운드:
평문 블록 → [AddRoundKey] → [SubBytes(S-Box)] → [ShiftRows] → [MixColumns]
```

- **AES(Advanced Encryption Standard)**: SPN 구조 기반, 128bit 블록, 키 길이 128/192/256bit, NIST 표준
- **DES**: Feistel 구조 기반, 64bit 블록, 56bit 키(현재 브루트포스에 취약, 폐기)

### 실무 제언

**운용 모드 선택 가이드**
- **챌린지**: 잘못된 운용 모드 선택(예: ECB)은 평문 패턴을 노출시켜 보안 취약점이 된다.
- **제언**: 일반 데이터 암호화에는 CBC 또는 GCM(인증 내장) 모드를, 실시간 스트리밍에는 CTR 모드를 사용하고, ECB는 절대 사용하지 말아야 한다.

**AES-GCM 적용**
- **챌린지**: 기밀성(암호화)과 무결성(인증)을 별도 메커니즘으로 구현하면 관리 복잡도가 증가한다.
- **제언**: AES-GCM(Galois/Counter Mode)은 CTR 기반 암호화와 GHASH 기반 인증 태그를 통합 제공하므로, TLS 1.3, IPSec 등 현대 프로토콜에서 선호하는 AEAD(Authenticated Encryption with Associated Data) 방식으로 채택해야 한다.
