## 해설

### 1. 생체인식정보 개념

- **정의**: 지문, 얼굴, 홍채, 정맥, 목소리 등 개인의 신체적·행동적 특성으로부터 추출한 정보
- **민감정보 분류**: 개인정보보호법 제23조의 민감정보로 분류 — 별도 동의 필요
- **처리 목적**: 본인 인증, 출입 통제, 잠금 해제 등

### 2. 생체인식정보 특성

| 특성 | 내용 |
|------|------|
| **불변성** | 변경 불가(유출 시 영구 대체 불가) |
| **유일성** | 개인 고유, 타인과 중복 불가 |
| **직접성** | 개인을 직접 식별 |
| **연동성** | 다른 DB와 결합 시 재식별 용이 |
| **영속성** | 시간이 지나도 동일(일부 노화 제외) |

### 3. 생체인식정보 보호 원칙

| 원칙 | 내용 |
|------|------|
| 목적 명확화 | 명확한 인증 목적으로만 수집 |
| 최소 수집 | 인증에 필요한 최소한의 특징값만 추출 |
| 안전한 저장 | 원본 미저장 원칙(특징값만 암호화 저장) |
| 엄격한 접근 통제 | 최소 권한, 다중 인증으로 접근 제한 |
| 파기 | 목적 달성 시 즉시 파기 |

### 4. 생체인식정보 처리 5단계별 보호 조치

| 단계 | 내용 | 보호 조치 |
|------|------|-----------|
| **① 수집** | 센서로 생체 데이터 취득 | 명확한 동의 취득, 목적 고지, 수집 최소화 |
| **② 특징 추출** | 원본에서 템플릿(특징값) 추출 | 원본 데이터 즉시 파기, 특징값만 보관 |
| **③ 저장** | 템플릿 DB 저장 | 암호화 저장(AES-256), 접근제어, 분리 저장 |
| **④ 비교·인증** | 등록 템플릿과 인증 시 템플릿 비교 | 안전한 비교 환경(TEE), 비교 후 즉시 임시 데이터 파기 |
| **⑤ 파기** | 이용 목적 달성 후 삭제 | 복구 불가능한 방식으로 안전 파기 |

### 실무 제언

**스마트폰 생체 인증 보안 설계**
- **챌린지**: 스마트폰 앱에서 생체 인증을 구현할 때 원본 이미지를 앱 서버에 전송하면 심각한 개인정보 침해가 발생한다.
- **제언**: iOS의 Face ID/Touch ID(Secure Enclave)나 Android의 BiometricPrompt API를 사용하면 생체 특징값이 기기의 TEE(신뢰 실행 환경)에서만 처리되어 앱·서버에 전달되지 않는다. 생체 인증 결과(성공/실패)와 서명된 챌린지만 앱에 전달하는 FIDO2 기반 설계를 채택해야 한다.
