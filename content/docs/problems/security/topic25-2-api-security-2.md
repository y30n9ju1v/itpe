## 문제

디지털 전환이 가속화됨에 따라 기업 내외부의 시스템 연계를 위한 API(Application Programming Interface) 활용이 급증하고 있다. 다음에 대해 설명하시오.

## 해설

### 1. API 개념 및 유형

정 의  • 소프트웨어 구성요소 간 상호작용을 위한 인터페이스로, 기능·데이터를 표준화된 방식으로 노출

| API 유형 | 특징 | 보안 포인트 |
|----------|------|------------|
| **REST** | HTTP 기반, 무상태, JSON 응답 | HTTPS 강제, JWT 검증, CORS 설정 |
| **GraphQL** | 클라이언트가 원하는 데이터 쿼리 선택 | 쿼리 복잡도 제한, 인트로스펙션 비활성화 |
| **gRPC** | Protocol Buffers, 고성능 RPC | mTLS 적용, 서비스 메시(Istio) 연계 |
| **SOAP** | XML 기반, 엔터프라이즈 표준 | WS-Security, 입력 검증 |
| **웹훅(Webhook)** | 이벤트 기반 역방향 API | HMAC 서명 검증, HTTPS 수신 |

### 2. API 보안 위협 심화 분석

#### 가. 인증·인가 취약점

| 위협 | 내용 |
|------|------|
| **BOLA(Broken Object Level Authorization)** | URL의 리소스 ID 조작으로 타 사용자 데이터 접근 |
| **BFLA(Broken Function Level Authorization)** | 일반 사용자가 관리자 기능 API 직접 호출 |
| **토큰 탈취** | JWT 서명 없음/약한 알고리즘, 토큰 유출 |

#### 나. 데이터 노출

| 위협 | 내용 |
|------|------|
| **과도한 데이터 노출** | 클라이언트가 필요한 것보다 많은 데이터 응답 |
| **Mass Assignment** | 요청에 포함된 임의 속성이 객체에 자동 반영 |

#### 다. 비즈니스 로직 위협

| 위협 | 내용 |
|------|------|
| **API Abuse** | 비즈니스 규칙을 우회한 비정상 거래(티켓 매크로 등) |
| **Shadow API** | 문서화되지 않은 API 엔드포인트 노출 |

### 3. API 보안 대응 방안 (심화)

#### 가. 설계 단계

```
API Security by Design:
① API 명세서(OAS) 작성 → 보안 요구사항 명시
② 리소스별 접근 권한 명확히 정의
③ 최소 응답 원칙(필요한 필드만 반환)
④ GraphQL: 쿼리 깊이/복잡도 제한, 인트로스펙션 비활성화
```

#### 나. 구현 단계

| 대응 | 내용 |
|------|------|
| **강력한 인증** | OAuth 2.0 + PKCE, FIDO2 연계 |
| **객체 수준 인가** | 모든 리소스 접근 시 소유자 검증 |
| **입력 유효성 검사** | 스키마 기반 자동 검증, SQL Injection·XXE 방어 |
| **민감 데이터 마스킹** | 응답 필터링, 로그 마스킹 |

#### 다. 운영 단계

| 대응 | 내용 |
|------|------|
| **API 인벤토리** | 자동 API 탐지 솔루션(Salt Security, Noname) |
| **런타임 보호** | API 게이트웨이 + WAF + APIM |
| **제로 트러스트 연계** | 서비스 간 API도 mTLS·JWT로 검증 |
| **행동 분석** | ML 기반 API 이상 사용 탐지 |


- 최근 API 개념·보안 위협·대응 방안 (심화) 관련 보안 위협 증가에 대응하여, 서비스 안전성 확보를 위한 체계적인 통제 및 기술 적용이 증대되는 추세임.  "끝"

### 실무 제언

**오픈뱅킹·마이데이터 API 보안 강화**
- **챌린지**: 금융 오픈뱅킹·마이데이터 API는 핀테크 서드파티가 직접 금융 데이터에 접근하므로, OAuth 토큰 탈취 시 대규모 금융 정보 유출이 발생한다.
- **제언**: OAuth 2.0 Authorization Code + PKCE 방식을 의무화하고, 토큰 수명을 15분 이내로 제한하며, FIDO 기반 사용자 재인증을 고액 거래 시 요구하는 Stepped-Up Authentication을 구현해야 한다. 또한 금융보안원의 오픈뱅킹 API 보안 가이드라인을 준수하는 API 보안 심사 체계를 연간 1회 이상 수행해야 한다.
