## 해설

### 1. 제로 트러스트 아키텍처 개요

- **정의**: "절대 신뢰하지 않고, 항상 검증하라(Never Trust, Always Verify)"는 원칙 하에, 내부 네트워크에 있더라도 모든 사용자·디바이스·트래픽을 지속적으로 인증·인가하는 보안 아키텍처
- **등장 배경**: 경계 기반 보안(Perimeter Security)의 한계 — 내부자 위협, 클라우드 확산, 원격근무로 기존 '내부는 신뢰' 모델 붕괴

### 2. 제로 트러스트 7대 원칙 (NIST SP 800-207)

| 원칙 | 내용 |
|------|------|
| ① **모든 데이터·서비스는 자원으로 간주** | 개인 기기, IoT 포함 모든 자원 |
| ② **모든 통신은 위치 무관 보안 적용** | 내부 네트워크도 동일 수준 검증 |
| ③ **세션별 접근 허가** | 최소 필요 권한, 세션 단위 부여 |
| ④ **동적 정책 기반 접근 통제** | ID·디바이스·행위 기반 실시간 결정 |
| ⑤ **자산 무결성 및 보안 상태 모니터링** | 지속적 자산 취약점·상태 파악 |
| ⑥ **엄격한 인증·인가** | MFA, 행동 분석, 위협 인텔리전스 연계 |
| ⑦ **가시성 및 분석 강화** | 모든 트래픽 로그·분석(SIEM 연계) |

### 3. ZTA 구성요소 (NIST SP 800-207)

```
┌──────────────────────────────────────────────────────┐
│                  제어 평면(Control Plane)             │
│  ┌──────────────┐    ┌──────────────────────────────┐│
│  │정책 결정 지점│←→ │정책 엔진(PE) + 정책 관리자(PA)││
│  │   (PDP)      │    └──────────────────────────────┘│
│  └──────┬───────┘                                    │
│         │ 정책 결정                                   │
├─────────┼────────────────────────────────────────────┤
│         ↓        데이터 평면(Data Plane)              │
│  ┌──────────────┐                                    │
│  │정책 시행 지점│ ←── 모든 트래픽 통과                │
│  │   (PEP)      │                                    │
│  └──────────────┘                                    │
└──────────────────────────────────────────────────────┘
```

| 구성요소 | 역할 |
|----------|------|
| **PE(Policy Engine)** | 접근 허가/거부 결정 (AI 기반 위험 점수 산출) |
| **PA(Policy Administrator)** | PEP에 접근 결정 전달, 세션 토큰 관리 |
| **PEP(Policy Enforcement Point)** | 실제 네트워크 트래픽 차단·허가 집행 |
| **CDM(지속 진단·모니터링)** | 자산 취약점·패치·설정 상태 실시간 파악 |
| **SIEM** | 로그 수집·분석·이상 탐지 |
| **Identity Provider(IdP)** | MFA·SSO·FIDO 기반 신원 검증 |

### 4. 제로 트러스트 전환 방안 (5단계)

| 단계 | 내용 |
|------|------|
| ① **자산 가시성 확보** | 모든 사용자·디바이스·서비스 목록화 |
| ② **마이크로세그멘테이션** | 내부 네트워크를 소단위로 분리, 측면 이동(Lateral Movement) 차단 |
| ③ **강화된 신원 관리** | MFA 전면 도입, 특권 계정 PAM 적용 |
| ④ **지속 모니터링** | 모든 트래픽 로깅, SIEM·UEBA 연동 |
| ⑤ **자동화·오케스트레이션** | SOAR 연계, 이상 탐지 시 자동 격리 |

### 실무 제언

**레거시 환경의 단계적 제로 트러스트 전환**
- **챌린지**: 기존 On-premise 레거시 시스템을 ZTA로 전환하려면 대규모 인프라 교체가 필요하여 수년의 시간과 막대한 예산이 소요된다.
- **제언**: SDP(Software Defined Perimeter) 기술을 우선 적용하여 기존 VPN을 대체하고, IAM(Identity and Access Management) 강화 → 마이크로세그멘테이션 → 전체 SIEM 연동 순서로 3년 로드맵을 수립하여 운영 중단 없이 단계적으로 ZTA를 구현해야 한다.
