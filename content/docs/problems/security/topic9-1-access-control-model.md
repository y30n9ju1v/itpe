## 해설

### 출제 배경 및 의도

접근통제는 정보시스템 보안의 핵심으로, MAC·DAC·RBAC 3가지 모델의 특성과 차이를 명확히 서술하고, 강제적 접근통제의 Bell-LaPadula·Biba·Clark-Wilson 등 보안 모델을 설명할 수 있어야 한다.

### 1. 3가지 접근통제 모델

| 모델 | 전체명 | 접근 결정 주체 | 특징 |
|------|--------|----------------|------|
| **MAC** | Mandatory Access Control (강제적 접근통제) | 시스템(관리자 정의 정책) | 보안 레이블 기반, 군사/정부 시스템 |
| **DAC** | Discretionary Access Control (임의적 접근통제) | 자원 소유자 | ACL 기반, 유연하나 트로이목마 취약 |
| **RBAC** | Role-Based Access Control (역할 기반 접근통제) | 역할(Role) 기반 | 업무 역할에 따라 권한 할당, 관리 용이 |

**비교:**

| 항목 | MAC | DAC | RBAC |
|------|-----|-----|------|
| 보안 강도 | 강 | 약 | 중 |
| 유연성 | 낮음 | 높음 | 높음 |
| 관리 편의 | 불편 | 불편(개별 관리) | 편리(역할 관리) |
| 적용 | 군사, 국가기관 | 일반 OS(Unix chmod) | 기업 ERP, 클라우드 IAM |

### 2. MAC의 대표적 보안 모델

#### 가. Bell-LaPadula 모델 (기밀성 보장)

- **목적**: 군사 기밀 보호, 기밀성 중심
- **규칙**:
  - **No Read Up(단순 보안 규칙)**: 낮은 등급 주체는 높은 등급 객체를 읽을 수 없음
  - **No Write Down(★-속성)**: 높은 등급 주체는 낮은 등급 객체에 쓸 수 없음
- **단점**: 무결성 보장 없음(높은 등급에서 낮은 등급으로 정보 변조 가능)

#### 나. Biba 모델 (무결성 보장)

- **목적**: 데이터 무결성 보호, Bell-LaPadula의 보완
- **규칙**:
  - **No Write Up**: 낮은 무결성 주체는 높은 무결성 객체에 쓸 수 없음
  - **No Read Down**: 높은 무결성 주체는 낮은 무결성 객체를 읽을 수 없음
- **특징**: 기밀성과 반대 방향의 정보 흐름 통제

#### 다. Clark-Wilson 모델 (트랜잭션 무결성)

- **목적**: 상업적 환경에서의 무결성 보장
- **개념**: CDI(제약된 데이터 항목) ↔ IVP(무결성 검증 절차) ↔ TP(변환 절차)
- **특징**: 은행·회계 시스템에서 승인된 절차(TP)를 통해서만 데이터 변경 허용

#### 라. Chinese Wall 모델 (이해충돌 방지)

- **목적**: 금융·컨설팅 업계의 이해충돌 방지
- **규칙**: 경쟁사 정보에 접근한 후에는 다른 경쟁사 정보 접근 금지

#### 마. RBAC (NIST RBAC)

- **구성**: 사용자 → 역할 → 권한(Permission) 계층적 구조
- **특징**: 역할 계층(Role Hierarchy), 역할 제약(Separation of Duty) 지원

### 실무 제언

**클라우드 IAM 최소 권한 적용**
- **챌린지**: 과도한 IAM 권한 부여로 클라우드 계정 침해 시 전체 자원이 노출된다.
- **제언**: RBAC 원칙으로 직무 분리(SoD)와 최소 권한(Least Privilege)을 적용하고, AWS IAM Permission Boundary 또는 Azure RBAC Condition으로 역할별 최대 허용 범위를 명시적으로 제한해야 한다.
