## 해설

### 출제 배경 및 의도

FIDO(Fast IDentity Online)는 패스워드 없는 인증의 표준 프레임워크로, 생체인증·하드웨어 키를 활용한 강력한 인증을 제공한다. 등록·인증의 비대칭키 기반 챌린지-응답 메커니즘을 단계별로 서술해야 한다.

### 1. FIDO 개요

- **정의**: FIDO Alliance(구글·MS·애플 등)가 개발한 패스워드 없는 강력한 인증 표준
- **목적**: 패스워드의 피싱·유출 취약성 해결 + 사용 편의성 향상
- **구성**: FIDO2 = WebAuthn(웹 표준) + CTAP2(디바이스 통신)

| 버전 | 프로토콜 | 특징 |
|------|----------|------|
| FIDO 1.0 | UAF, U2F | 생체인증(UAF), 2단계 인증(U2F) |
| FIDO 2.0 | WebAuthn + CTAP2 | 브라우저 기본 지원, 패스키(Passkey) |

### 2. FIDO 구성요소

| 구성요소 | 역할 |
|----------|------|
| **클라이언트(Authenticator)** | 생체인증·PIN으로 사용자 확인, 개인키 보관(TEE/Secure Enclave) |
| **클라이언트 장치** | 브라우저·앱(WebAuthn API 호출) |
| **FIDO 서버(RP Server)** | 챌린지 발행, 공개키 저장·검증 |

### 3. FIDO 등록(Registration) 절차

```
사용자 브라우저               FIDO 서버(RP)
    │                              │
    │──① 등록 요청────────────────▶│
    │◀─② 챌린지(Challenge) 발행───│
    │                              │
    │ [인증기기에서 생체인증/PIN 확인]│
    │ [공개키/개인키 쌍 생성]       │
    │                              │
    │──③ 공개키 + 서명(챌린지 서명)─▶│
    │   [서명 = 개인키로 챌린지 서명] │
    │◀─④ 등록 완료────────────────│
    │   [서버: 공개키 저장]          │
```

### 4. FIDO 인증(Authentication) 절차

```
사용자 브라우저               FIDO 서버(RP)
    │                              │
    │──① 로그인 요청──────────────▶│
    │◀─② 챌린지(새로운 랜덤값)────│
    │                              │
    │ [인증기기에서 생체인증/PIN 확인]│
    │ [개인키로 챌린지 서명]         │
    │                              │
    │──③ 서명(챌린지 서명) 전송────▶│
    │◀─④ 인증 성공────────────────│
    │   [서버: 공개키로 서명 검증]   │
```

### 5. FIDO의 보안 강점

| 특성 | 설명 |
|------|------|
| **피싱 방지** | 챌린지에 RP ID(도메인) 포함 → 피싱 사이트에서는 서명이 유효하지 않음 |
| **서버 유출 안전** | 서버는 공개키만 저장 → 서버 해킹 시에도 인증기기 없으면 인증 불가 |
| **개인키 안전** | TEE(신뢰 실행 환경)/Secure Enclave에 저장, 외부 추출 불가 |
| **패스워드 불필요** | 패스워드 재사용·유출 문제 원천 해결 |

### 실무 제언

**Passkey 도입 로드맵**
- **챌린지**: 기존 ID/Password 인증 시스템을 FIDO2 Passkey로 전환하면 사용자 경험 변화와 레거시 호환성 문제가 발생한다.
- **제언**: Apple/Google/Microsoft의 Passkey 동기화 기능을 활용하여 크로스 디바이스 인증을 지원하고, 초기에는 패스워드+Passkey 병행, 이후 패스워드리스 단계로 점진 전환하는 로드맵을 수립해야 한다.
