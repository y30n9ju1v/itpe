### 1. 빅데이터 아키텍처의 패러다임, 폴리글랏 프로세싱(Polyglot Processing)

정 의  • 독립적 혹은 서로 다른 프로세싱 엔진을 조합하여 최적의 프로세싱을 수행하는 처리방식(다중언어 프로세싱)
       - 빅데이터 프로세싱은 수집, 저장, 분석, 활용의 파이프라인으로 구성되므로 파이프라인에 가장 적합한 엔진을 사용하여 수행하는 것이 필수적

```
빅데이터 3V 문제점          해결방안
규모(Volume)    →    배치 프로세싱
다양성(Variety) →    NoSQL
속도(Velocity)  →    스트림 프로세싱
```

- 다중언어 프로세싱을 기반으로 한 빅데이터 프로세싱의 아키텍처로 람다 아키텍처(Lambda Architecture)와 카파 아키텍처(Kappa Architecture)가 대표적

---

### 2. 배치 및 스트림 처리 3레이어 구조, 람다 아키텍처

#### 가. 람다 아키텍처(Lambda Architecture)의 개념

- 나단 마르즈(Nathan Marz)가 트위터(Backtype)에서 수행한 분산 데이터 프로세싱 경험을 기반으로 모든 데이터를 대상으로 분석할 수 있도록 배치 레이어(Batch Layer), 스피드 레이어(Speed Layer), 서빙 레이어(Serving Layer)로 구성된 아키텍처
- 어떤 유형의 장애에도 프로세싱이 가능한 강건성 시스템. 지연최소화, 분석결과 일관성, 정확성 제공

#### 나. 람다 아키텍처 구성도

```
신규 데이터 ─┬─→ [배치 레이어(Batch Layer)]
            │    All data → 배치 재계산 → 선계산 정보 → [배치 뷰]──┐
            │                                                      ├→ 정합(Merge)→ 쿼리
            └─→ [스피드 레이어(Speed Layer)]                        │
                 스트림 프로세스→ 실시간 증분→ 증분 정보→ [실시간 뷰]─┘
                                           ↑서빙 레이어(Serving Layer)
```

#### 다. 람다 아키텍처 구성요소

| 레이어 | 설명 | 주요 기술 |
|--------|------|-----------|
| 배치 레이어 | 유입되는 모든 데이터를 master dataset에 저장, 데이터 재계산과 선계산 정보와의 통합 배치뷰 생성 | 맵리듀스, Tez, Spark |
| 스피드 레이어 | 배치 레이어에서 처리하지 못하는 실시간 뷰 생성, 시간 최소화를 통한 증분 방식의 데이터 처리 | Storm, Samza, 플링크 |
| 서빙 레이어 | "배치 뷰 + 실시간 뷰"의 병합결과 제공 | NoSQL DB, Hive |

---

### 3. 스트림 처리 최적화 2레이어 구조, 카파 아키텍처

#### 가. 카파 아키텍처(Kappa)의 개념

- 제이 크렙스(Jay Kreps)가 링크드인(LinkedIn)에서 배치 레이어를 제거하고 모든 계산을 스피드 레이어에서 스트림으로 처리하도록 제안한 모델
- 단일 프레임워크를 활용하여 운영/관리 부분의 단순화와 개발/테스트/디버깅 효율 증가

#### 나. 카파 아키텍처 구성도

```
신규 데이터 ─→ [스피드 레이어(Speed Layer)]──→ [서빙 레이어(Serving Layer)]──→ 쿼리
              작업버전 n-1 → 결과테이블 n-1
              작업버전 n   → 결과테이블 n
              작업버전 n+1 → 결과테이블 n+1
```

#### 다. 카파 아키텍처 구성요소

| 레이어 | 설명 | 특징 |
|--------|------|------|
| 스피드 레이어 | 재작업은 코드 변경에 의해서만 수행, 재작업은 수집된 모든 데이터를 이용하여 현재 수행중인 작업 대신 별도의 스트림 프로세스를 생성한 후 처리 | 단일 프레임워크 활용 |
| 서빙 레이어 | 수행 후 처리 결과 또한 별도의 테이블로 저장(결과 테이블 n+1로 저장), 재작업 프로세스가 완료된 경우 결과값을 새로운 테이블(결과 테이블 n+1)에서 조회하도록 하고 이전 스트림 프로세스를 종료 | 운영·관리 단순화 |

---

### 4. 람다 아키텍처와 카파 아키텍처의 비교 설명

| 비교 항목 | 람다 아키텍처 | 카파 아키텍처 |
|-----------|--------------|--------------|
| 레이어 구조 | 3개 레이어(배치, 스피드, 서빙) | 2개 레이어(스피드, 서빙) |
| 아키텍처 목적 | 전송 지연 최소화, 일관성·정확성 제공 | 레이어간 코드 공유 복잡성 제거 |
| 프로세싱 패러다임 | 배치+스트림 조합 | Only 스트림 |
| 재작업 패러다임 | 배치 사이클마다 | 코드 변경 시 |
| 자원 소비 | 많음 | 적음(Δ: 증분) |
| 신뢰성 | 배치는 신뢰, 스트림은 근사치 제공 | 일관성 있는 스트림 제공으로 신뢰성 높음 |
| 사용자 | 트위터, 라이브퍼슨(Liveperson) | 링크드인, 야후 |

- 람다/카파 선택 기준: 정확성·일관성 중시 → 람다, 운영 단순화·스트림 중심 → 카파
