## 문제

데이터 웨어하우스에 적용되는 온라인 분석처리(OLAP, On-Line Analytical Processing)를 온라인 트랜잭션처리(OLTP, On-Line Transaction Processing)와 비교하여 설명하고, 분석정보 데이터베이스(데이터 웨어하우스)와 운영정보 데이터베이스(운영 데이터베이스)가 분리 운영되어야 하는 이유에 대하여 설명하시오.

## 해설

### 출제 배경 및 의도

기업 정보시스템이 운영계(OLTP)와 분석계(OLAP)로 이원화됨에 따라, 두 처리 방식의 개념·기술적 차이와 DB 분리 운영의 필요성을 논리적으로 서술할 수 있는지 확인하기 위해 출제

### 1. 기업의 통합 데이터 아키텍처 개요

정 의  • 기업의 정보시스템은 거래 데이터를 실시간 처리하는 운영계와, 축적된 데이터를 다차원 분석하는 분석계로 분리 구성하여 각 목적에 최적화된 구조를 갖춤
       - 운영 DB에서 ETL을 통해 데이터 웨어하우스·데이터 마트로 이전되며, OLAP 도구가 분석 시스템에 데이터를 공급

```
[운용시스템] → OLTP → DB --ETL--> Staging ---> DW/DM <-- OLAP <-- [분석시스템]
(SCM,ERP,MES)                                              (EIS, BI)
```

### 2. OLAP과 OLTP의 비교

#### 가. 기능적 측면 비교

| 구분 | OLAP | OLTP |
|------|------|------|
| 개념 | 요청된 조건에 맞는 대량 데이터를 검색하여 다양한 뷰 제공 | 트랜잭션 중심으로 거래 데이터의 정확한 처리 기능 제공 |
| 기능원리 | 중복 및 집계, 소트, 차원별 그룹 | 트랜잭션 ACID |
| 사용목적 | 대용량 데이터의 조회 | 단일 트랜잭션 데이터 저장 |
| 주요기능 | Drill Down/Up, Pivot, Slice, Dice | Insert, Update, Commit, Rollback |
| 구현기술 | MOLAP, ROLAP, DOLAP | TP-monitor, CS, EJB, ODBC |

#### 나. 데이터 측면 비교

| 구분 | OLAP | OLTP |
|------|------|------|
| 데이터 구조 | 중복 및 집계 컬럼 구성, 차원 지원, 목적 테이블/데이터 마트 | 정규화를 통한 중복 최소화, ER모델 |
| 데이터 모델 | 스타스키마, 스노우플레이크, ER모델 | ER모델 |
| 데이터 목적 | 시계열분석, 그룹분석, 실적집계 및 비교 | 데이터의 무결성 유지 |
| 데이터 보호 | Read Only | Repeatable Read 레벨, 락킹, 병행처리 |
| 데이터 접근 | 사용자가 직접 접근, EUC | 응용프로그램을 통한 접근 |

### 3. 분석정보DB와 운영정보DB의 분리 운영 필요성

#### 가. DB 특성 측면에서 분리 운영 필요성

| 구분 | 운영정보 DB | 분석정보 DB |
|------|------------|------------|
| 데이터 특성 | 동일 데이터 동적 변경 처리 | 정적 데이터의 누적(ETL/ETT) |
| 데이터 품질 | 데이터의 무결성, 관계의 정합성 | 데이터 중복과 집계, 다양한 뷰 제공 |
| 데이터의 상태 | 거래발생에 따른 지속 변화 | 이전 자료의 불변성 및 변경 자료 유입 |
| 데이터 처리 | OLTP 사용 | OLAP 사용 |
| 개인정보 관리 | 개인정보의 암호화 활용 | 개인정보의 난독화, 일반화, 마스킹 |

#### 나. 운영 관점에서 분리 운영 이유

| 구분 | 분리 운영 필요성 | 설명 |
|------|----------------|------|
| 서비스 제공 | 응용 시스템 서비스 차별화 | 분석DB 적재 장시간 배치작업으로 인한 운용DB 사용지연 방지 |
| DB 성능 | DB 성능을 고려한 설정 | 병행처리에 따른 성능저하 방지 |
| 데이터 관리 | 데이터 관리 정책의 일관성 유지 | 데이터의 백업 및 복구 방식의 차이 반영 |

### 4. 결론

효율적인 DB 운영을 위하여 운영정보 DB와 분석정보 DB는 목적·특성·처리방식이 본질적으로 상이하므로 분리 운영이 효과적이며, ETL/ETT 파이프라인을 통한 데이터 연계 체계를 명확히 수립하는 것이 핵심이다. **끝**

### 실무 제언

- **ETL 주기 설계**: 운영DB 부하가 낮은 야간 배치로 ETL을 실행하여 서비스 영향 최소화
- **개인정보 처리**: 운영DB의 개인정보는 DW 적재 시 반드시 마스킹·일반화 처리하여 규정 준수
- **성능 모니터링 분리**: OLTP SLA(응답시간)와 OLAP SLA(쿼리 처리량)를 독립적으로 측정·관리
